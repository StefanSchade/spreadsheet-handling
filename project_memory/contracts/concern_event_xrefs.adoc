= Table Contract: concern_event_xrefs
:toc:

== Purpose

`concern_event_xrefs` links concern events to concerns.

The role is relationship-specific: the same event may be a `driver` for one concern and `historical_context` for another.

== Fields

[cols="1,1,3"]
|===
| Field | Required | Contract

| `id`
| yes
| Stable xref id. Existing pattern: `CTSX-<event_id>--<concern_id>`.

| `event_id`
| yes
| References `concern_events.id`.

| `concern_id`
| yes
| References `concerns.id`.

| `event_role`
| yes
| Controlled value. See <<event-roles>>.

| `notes`
| no
| Relationship-specific interpretation. Use this to explain why the role applies to this concern.
|===

== Event Roles [[event-roles]]

[cols="1,3"]
|===
| Value | Meaning

| `driver`
| The event materially explains why the concern exists or is active.

| `trigger`
| The event caused the concern to become visible or actionable.

| `evidence`
| The event supports the concern but is not the main driver.

| `historical_context`
| The event explains the history but is not current steering evidence.

| `watch_item`
| The event keeps the concern visible without demanding immediate action.

| `mitigation`
| The event shows that the concern was reduced, partially addressed, or made safer.

| `implementation_evidence`
| The event shows that a chosen direction was implemented or reinforced.

| `regression_evidence`
| The event shows that the concern worsened, reappeared, or accumulated new risk.
|===

== Role Selection Guidance

Use `driver` only when the event is central to why the concern exists. Use `evidence` when the event merely supports an already-known concern. Use `watch_item` when the event is deliberately not being turned into immediate work.

For `source_type = activity` events, prefer `mitigation` or `implementation_evidence` when the work reduced risk or proved a direction.
