:toc:

== Purpose

`concern_threads` defines coarse steering concerns that cut across individual reviews, findings, FTRs, and commits.

A concern thread is a memory and steering object. It groups signals into a current posture without replacing the detailed tracking tables.

== Granularity Rule

A concern thread should be broad enough to survive multiple implementation slices, but concrete enough to guide prioritization.

Do not create a concern thread for every finding, module, or FTR. Use concern signals and xrefs for detailed evidence.

== Fields

[cols="1,1,3"]
|===
| Field | Required | Contract

| `id`
| yes
| Stable concern id. Existing pattern: `CONC-*`.

| `title`
| yes
| Human-readable concern title.

| `area`
| yes
| Coarse area such as `domain/meta`, `maintainability`, `projection/workbook-editing`, or `backends`.

| `status`
| yes
| Controlled value. See <<statuses>>.

| `posture`
| yes
| Current steering posture. Controlled vocabulary may remain project-specific; see <<postures>>.

| `priority`
| yes
| Controlled value. See <<priorities>>.

| `current_summary`
| yes
| Current-state summary in one sentence.

| `next_action`
| yes
| Steering-level next action. This should not be a detailed task list.

| `last_assessed_date`
| yes
| ISO date `YYYY-MM-DD`.

| `last_assessed_commit`
| no
| Short commit id for the assessment state, if known.

| `notes`
| no
| Scope note, caveat, or modeling note.
|===

== Statuses [[statuses]]

[cols="1,3"]
|===
| Value | Meaning

| `active`
| The concern currently influences near-term work or prioritization.

| `watching`
| The concern remains visible but is not the main current workstream.

| `closed`
| Reserved for future use when a concern is no longer relevant.
|===

== Priorities [[priorities]]

[cols="1,3"]
|===
| Value | Meaning

| `high`
| Strong current steering relevance.

| `medium`
| Relevant, but not currently dominant.

| `low`
| Low current steering relevance. Use sparingly.
|===

== Postures [[postures]]

Existing posture values are intentionally descriptive and may remain project-specific. Examples include:

* `doing_now`
* `after_domain_meta_slice`
* `after_0_3_0`
* `review_after_refactor`
* `no_current_pressure`

When introducing a new posture, prefer a short phrase that explains when or how the concern should be revisited.

== Non-Goals

* No replacement for FTRs.
* No replacement for findings.
* No detailed commit or module tracking.
