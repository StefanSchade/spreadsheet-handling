:toc:

== Purpose

`concern_events` records coarse dated events that explain, shift, strengthen, weaken, mitigate, or contextualize a concern.

An event is intentionally coarser than a finding, FTR, review finding, or individual commit. It is a steering-level memory object.

== Granularity Rule

One row should represent a recognizable historical event, review outcome, finding marker, decision marker, release marker, manual interpretation, or activity wave.

Do not create one event per commit unless the commit itself is the meaningful steering event.

Commit references are anchors, not a complete provenance model. Use Git, FTRs, reviews, and findings for detailed traceability.

== Fields

[cols="1,1,3"]
|===
| Field | Required | Contract

| `id`
| yes
| Stable event id. Existing prefixes include `SIG-CONC-*` and `SIG-REVIEW-*`. Activity events may use `SIG-ACT-*`.

| `event_date`
| yes
| ISO date `YYYY-MM-DD`. For activity waves, use the start date or the most representative date.

| `source_type`
| yes
| Controlled value. See <<source-types>>.

| `source_id`
| yes
| Source identifier. For reviews and findings, use the corresponding review or finding id. For activities, use a coarse activity id such as `ACT-REVIEW-001-CLOSURE-WAVE`.

| `commit_refs`
| no
| Optional comma-separated short commit anchors. Intended primarily for `source_type = activity`. Ranges may be used only when the range is meaningful and reasonably contiguous.

| `weight`
| yes
| Controlled value. See <<weights>>.

| `summary`
| yes
| One-sentence steering summary. Summaries should make the concern impact visible.

| `notes`
| no
| Additional interpretation, scope limitation, or rationale.
|===

== Source Types [[source-types]]

[cols="1,3"]
|===
| Value | Meaning

| `review_set`
| A review set: the full group of reviews under a single storage directory in `docs/cold_storage/reviews/`. `source_id` references `review_sets.id`. Preferred over `review` when the event describes the steering impact of a whole review wave rather than a single document.

| `review`
| Deprecated for new events. A single review document or consolidated review summary. Prefer `review_set` when anchoring to a review wave.

| `finding`
| A concrete finding row.

| `decision`
| A decision record.

| `activity`
| A coarse implementation, cleanup, review-follow-up, migration, or stabilization activity wave.

| `manual_note`
| A manually curated interpretation or steering note.

| `release`
| A release or release-readiness event.
|===

== Weights [[weights]]

[cols="1,3"]
|===
| Value | Meaning

| `high`
| Strong event for current or historical steering.

| `medium`
| Relevant event, but not sufficient by itself to drive current posture.

| `low`
| Weak or contextual event. Use sparingly; weak items often belong only in notes.
|===

== Activity Events

`source_type = activity` is used when a grouped body of work changed the state of one or more concerns.

Activity events normally use `commit_refs`, but commit refs remain coarse anchors. They are not intended to reconstruct commit-level provenance.

Example:

[source,json]
----
{
  "id": "SIG-ACT-001",
  "event_date": "2026-05-12",
  "source_type": "activity",
  "source_id": "ACT-REVIEW-001-CLOSURE-WAVE",
  "commit_refs": "23a38aa3, bd27b57d, a829990f, 8cd88749, 573b428d, 8b2b0585",
  "weight": "high",
  "summary": "REVIEW-001 follow-up wave converted architecture findings into bounded FTR slices, guards, and targeted cleanup.",
  "notes": "Coarse activity event. Commit refs are anchors, not a complete per-commit provenance model."
}
----

== Non-Goals

* No commit-level provenance reconstruction.
* No replacement for findings, FTRs, reviews, decisions, or Git history.
* No nested commit arrays until the model is stable and schema/tooling supports it.
