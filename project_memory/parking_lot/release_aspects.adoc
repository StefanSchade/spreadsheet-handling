= Future Roadmap Modeling Notes
:toc:
:toclevels: 2

== Purpose

This note is a parking place for forward-looking project-memory modeling ideas.

The current `project_memory` model primarily captures observed project history:

* findings
* concerns
* decisions
* review results
* signals
* evidence
* implemented or accepted work

During the v0.2.x stabilization phase it became clear that the model may also
need a way to represent forward-looking planning context without turning every
idea into an active FTR, finding, or commitment.

This document intentionally does not introduce such a model yet.

Its purpose is to preserve early thinking until the project is ready to decide
whether and how prospective elements should become part of `project_memory`.

== Non-Goals

This note is not:

* a committed release plan;
* a replacement for backlog/FTR tracking;
* a canonical project-memory table;
* a promise that all listed release themes will be implemented;
* a reason to normalize or redesign `project_memory` immediately.

The goal is to avoid losing useful roadmap thinking while also avoiding a
premature modeling exercise during the v0.2.x stabilization work.

== Why Prospective Modeling May Be Needed

The existing model is strong at reconstructing what happened and why.

However, some project knowledge is neither historical evidence nor an active
work item. Examples include:

* tentative release-line themes;
* sequencing assumptions;
* planned maturity stages;
* deferred modeling questions;
* candidate 1.0 readiness criteria;
* strategic concerns that are not yet actionable findings.

Without a place for this kind of information, such thoughts remain in chat
history or informal notes. That increases future reconstruction effort.

At the same time, modeling future intent too early can create false authority.
A roadmap entry can easily be misread as a commitment, backlog item, or
review obligation.

The future model therefore needs to distinguish clearly between:

* observed evidence;
* accepted decisions;
* active work;
* watch-only context;
* prospective planning assumptions.

== Candidate Concept: Prospective Project Memory

A future extension could introduce a small prospective layer, for example:

* roadmap themes;
* release-line intentions;
* maturity targets;
* readiness criteria;
* sequencing assumptions;
* explicit non-commitments.

Such elements should be clearly marked as prospective and should not be treated
as evidence that work has been accepted, started, or completed.

Possible lifecycle states might include:

* `idea`
* `candidate`
* `tentative`
* `accepted_direction`
* `superseded`
* `withdrawn`

This is only a sketch. No schema should be introduced until there is enough
repeated use to justify it.

== Example Roadmap Thinking Captured for Later

The following release-line sketch emerged during the v0.2.x project-memory and
Review 004 stabilization work.

It should be treated as a working hypothesis, not as a committed plan.

[cols="1,3,4", options="header"]
|===
| Release line | Candidate theme | Notes

| 0.2.x
| Stabilize by dogfooding
| Technical stabilization, project-memory hygiene, executable sentinel handling,
semantic-contract closure, and small features that improve the maintainer's own
use of the framework. This is not a feature freeze, but new work should reduce
friction, improve steering, or harden existing contracts.

| 0.3.0
| Explain and onboard
| Improve external presentation: demo, tutorial, Pages/reveal.js material,
consumer-oriented explanation, and clear known limitations. The main goal is
understandability and adoption readiness, not broad new feature scope.

| 0.4.0
| Consumer workflow ergonomics
| Improve day-to-day safety and usability for real consumers. Possible themes:
workflow guidance, safer import/export sequences, schema-maintenance ergonomics,
better diagnostics, and the later "Wippe" workflow-management idea.

| 0.5.0
| Semantic contract maturity
| Stabilize central semantic boundaries: canonical vs projection vs runtime
metadata, helper provenance, frame lifecycle, constraint lifecycle, and related
model contracts.

| 0.6.0
| Rich projection features
| Expand higher-level spreadsheet projection patterns once the semantic contracts
are stable enough. Candidate areas include compact xref/crosstable use cases,
legend blocks, dynamic view targeting, and grouped or multi-axis projections.

| 0.7.0
| Backend capability maturity
| Make backend and carrier capabilities explicit. Clarify XLSX/ODS parity,
CSV/JSON as tabular projection carriers, sidecar/convention-based
representations, and unavailable feature diagnostics.

| 0.8.0
| Real adoption hardening
| Incorporate feedback from real consumers and larger workbooks. Improve
migration, diagnostics, documentation gaps, examples, and operational robustness.

| 0.9.0
| 1.0 candidate / contract freeze
| Prepare stable public contracts: CLI, canonical JSON, configuration shape,
public API, deprecation policy, migration notes, and readiness criteria.

| 1.0
| Stable consumer contract
| Not feature completeness, but a reliable, documented, tested, and
audit-understandable core contract for real consumers.
|===

== Important Interpretation

The roadmap sketch above is useful because it captures sequencing intent:

* stabilize and govern first;
* explain and onboard next;
* improve consumer ergonomics;
* mature semantic contracts;
* expand projection richness;
* clarify backend capabilities;
* harden through real adoption;
* freeze contracts before 1.0.

It should not be interpreted as a promise that every listed item belongs to
that exact release line.

The main value is the maturity model, not the numbering itself.

== Open Modeling Questions

Future modeling work should answer:

* Should prospective planning elements live in `project_memory/canonical`, or
  remain in documentation?
* If canonical, which table owns them?
* How can they be linked to concerns, FTRs, findings, and decisions without
  creating false obligations?
* How are prospective items retired, superseded, or promoted?
* How can generated reports distinguish prospective intent from historical
  evidence?
* How can agents use this context without treating it as a current backlog?

== Current Position

For now, prospective roadmap context remains documentation only.

The project should revisit this after the v0.2.x stabilization work, once
`project_memory` has enough operational use to justify a small, explicit
prospective layer.
