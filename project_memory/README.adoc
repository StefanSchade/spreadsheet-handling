= Project Memory

`project_memory` is an experimental dogfood consumer for repository memory.
It exists to keep the project's own truth in a spreadsheet-friendly shape that
can be edited in ODS and exchanged as flat JSON.

== Execution Environment

`project_memory` intentionally uses the repository's current development
environment and the local editable checkout.

The Makefile targets use the repo-local `.venv/bin/sheets-run`.

Do not install or pin a PyPI `spreadsheet-handling` version into this venv for
`project_memory`.

The point of this path is to detect breakage in the current code before
release. If a breaking change affects `project_memory`, update `project_memory`
together with the core change before release.

Testing released-package compatibility is a separate concern and would require
a separate temporary venv or release-smoke setup. That release-smoke setup is
intentionally out of scope for the current dogfood path.

== Layout

* `canonical/` contains curated project truth.
* `canonical/ext_*.json` contains generated candidate tables staged for workbook review.
* `derived/` contains rebuildable query outputs derived from canonical data.
* `extracted/` contains raw generated candidate extraction output.
* `staging/` is the review area produced by ODS import.
* `staging/` must not be promoted blindly.

== Data Files

* `ftrs.json` holds FTR header metadata only.
* `ftr_dependencies.json` is the flat xref replacement for list-valued FTR dependencies.
* `reviews.json` stores review records for FTRs, packages, releases, commits, and architecture areas.
* `findings.json` stores review findings over time and separates historical findings from current risks through `status` and `current_relevance`.
* `decisions.json` stores stable project-level choices.
* `releases.json` stores project-wide release or package pin state.

== Extracted Candidates

`project_memory/extracted/` contains machine-generated candidate JSON produced
from ADOC artifacts.

These files are not curated truth. They are conservative hints for later
review and must not be promoted into canonical truth tables without manual
inspection.

== Staged Candidate Tables

`project_memory/canonical/ext_*.json` contains generated candidate tables copied
from `project_memory/extracted/` so they are visible as sheets in the ODS
workbook.

The `ext_` prefix is mandatory. It marks a table as generated candidate
material, not manually curated truth.

`ext_*` tables are reproducible staging material. They are generated,
gitignored, and overwriteable by `make memory-stage-extracted`.

Manual curation happens by copying or editing selected rows from the `ext_*`
sheets into the real canonical sheets:

* `ext_ftr_candidates` -> `ftrs`
* `ext_review_candidates` -> `reviews`
* `ext_finding_candidates` -> `findings`

No automatic promote or merge into canonical truth tables exists yet.

== Derived Query Views

The derived query pipeline can generate read-only views under `derived/`:

* `current_findings`
* `ftr_dependency_edges`
* `ftr_blockers`

These views are generated from canonical data for review and query
exploration. They are not canonical source, must not be edited manually, and
must not be promoted back into `canonical/`.

The raw canonical export pipeline remains available for the source-only
workbook. Query outputs are separate and rebuildable.

== Generated Context Report

`make memory-context` generates
`docs_generated/project_memory/current_context.adoc` from the derived query
outputs.

That report is generated, gitignored, and intended for LLM/review handoff
context rather than stable user documentation.

== Current Status Vocabulary

Findings use this status vocabulary:

* `open`
* `accepted`
* `resolved`
* `superseded`
* `wontfix`
* `historical`

`current_relevance` uses:

* `current`
* `partial`
* `historical`
* `unknown`

== Finding Assessment Fields

`findings.json` separates original review evidence from later tracking
assessment:

* `summary` is the short finding statement.
* `source_detail` records the review-time detail or rationale from the source
  review. Its time context is the linked `review_id`, `source_path`, and
  `source_line_start`.
* `current_assessment` records the later tracking interpretation: still open,
  resolved, superseded, accepted, or tracked elsewhere.
* `assessment_as_of_commit` and `assessment_as_of_date` identify the repository
  state and date for `current_assessment`.
* `detail` is retained as a legacy compatibility alias for `source_detail`.
* `fix_comment` is retained as a legacy compatibility alias for
  `current_assessment`.

== Notes

Queries are intentionally lightweight and repository-local in this slice.
Future query examples may be added later, but no generic query engine or
automatic ADOC extraction is required here.

`make memory-extract` produces:

* `project_memory/extracted/ftr_candidates.json`
* `project_memory/extracted/review_candidates.json`
* `project_memory/extracted/finding_candidates.json`

`make memory-stage-extracted` copies those files into generated workbook-visible
tables:

* `project_memory/canonical/ext_ftr_candidates.json`
* `project_memory/canonical/ext_review_candidates.json`
* `project_memory/canonical/ext_finding_candidates.json`

`make memory-refresh-workbook` is the convenience path for:

. extract current candidates;
. stage them as `ext_*` canonical-side generated tables;
. export the ODS workbook.
