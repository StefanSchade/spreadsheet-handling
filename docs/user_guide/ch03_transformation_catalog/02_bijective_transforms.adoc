[[_bijective_transforms]]
=== Bijective Transforms

These steps come in expand/contract pairs. The expand direction converts a
compact or matrix representation into an explicit relation (long form). The
contract direction reverses it. Losslessness is conditional — see each entry.

==== `expand_xref` / `contract_xref`

*Category*: primitive · *Inverse*: conditionally bijective

Expand a matrix frame (rows × columns of marker cells) into explicit long-form
relation rows, or contract them back. Typical use: a role × permission table
where each cell marks whether the role has the permission.

See also: concept doc `technical_model/ch04_concepts/crosstab_binary_xref.adoc`
and `compact_transforms.adoc`.

*`expand_xref` parameters*

[cols="2,1,4"]
|===
| Parameter | Required | Description

| `matrix`
| yes
| Name of the input matrix frame.

| `output`
| yes
| Name of the output relation frame to produce.

| `row_keys`
| yes
| Column name (string) or list of column names that form the row identifier.

| `value_columns`
| no
| Columns to expand. Defaults to all non-key columns.

| `column_key`
| no
| *Field name* in the output relation that stores the column-axis address value
  (default: `column_key`).  This is a field name, not a list of axis values —
  see `column_keys` on `contract_xref` for that.

| `value`
| no
| Column name in the output holding the cell value (default: `value`).

| `drop_empty`
| no
| If `true`, omit rows where the cell value is empty or blank.  Default: `false`.
  With `base_relation`: a blank in-scope cell removes the matching base tuple.
  Without `base_relation`: a blank cell either produces a tuple with empty value
  (`false`) or is omitted (`true`).

| `base_relation`
| no
| Optional name of an existing relation frame for scoped recomposition.  When
  absent, `expand_xref` converts the matrix to a new relation (standard mode).
  When present, the matrix cells define the update scope and the matching
  base-relation tuples are replaced or removed; relation rows whose addresses
  fall outside the matrix scope are preserved and appended to the output.
  First-run pipelines with no prior relation omit this parameter.

| `drop_source`
| no
| Assert that the expanded relation fully supersedes the matrix and mark the
  matrix for carrier-neutral final domain cleanup
  (`_meta.pipeline_cleanup`).  Rejected when an explicit `value_columns`
  subset would leave unexpanded matrix columns behind.  Default: `false`.
  Workbook helper/technical columns must be removed by their owning
  projection step (or made explicit) before a full expansion.

| `name`
| no
| Override the step name shown in logs.
|===

*Losslessness condition*: lossless when `drop_empty: false`, or when the
original matrix had no empty cells in the expanded columns.  With
`drop_source: true`, the checkable value-loss condition (an unexpanded
`value_columns` subset) fails fast; shape recovery of empty-only columns
under `drop_empty: true` needs dense column intent or a retained matrix.

[source,yaml]
----
- step: expand_xref
  matrix: role_permissions
  output: permission_rows
  row_keys: role_id
  drop_empty: true
----

*`contract_xref` parameters*

[cols="2,1,4"]
|===
| Parameter | Required | Description

| `relation`
| yes
| Name of the input long-form relation frame.

| `output`
| yes
| Name of the output matrix frame to produce.

| `row_keys`
| yes
| Column name(s) forming the row identifier (must match `expand_xref` config).

| `column_key`
| no
| *Field name* in the relation that holds the column-axis address values
  (default: the literal `column_key`; configure it to match the preceding
  `expand_xref`).  This is a field name, not a list of axis values — see
  `column_keys` below.

| `value`
| no
| Column in the relation holding cell values (default: the literal `value`;
  configure it to match the preceding `expand_xref`).

| `column_keys`
| no
| Explicit ordered list of *column-axis values* that become the visible matrix
  columns.  (`column_key` names the field; `column_keys` lists the values in
  that field.)  When supplied together with `dense_axes.columns_from`, both
  must resolve to the same ordered list — a mismatch fails fast.

| `fill_value`
| no
| Value to place in cells with no matching relation row. Default: empty string.

| `dense_axes`
| no
| Map with optional `rows_from` and/or `columns_from` keys.  Each specifies an
  entity frame and key field whose unique values drive the corresponding axis.
  When `columns_from` is supplied, the resolved value list must match any
  explicit `column_keys` exactly (consistency contract, not precedence).

| `drop_source`
| no
| Assert that the contracted matrix fully supersedes the relation and mark
  the relation for carrier-neutral final domain cleanup
  (`_meta.pipeline_cleanup`).  Rejected when the relation carries fields
  beyond `row_keys` + `column_key` + `value` (the matrix represents nothing
  else), or when the matrix columns do not cover every relation column key
  (a matrix that silently drops relation pairs cannot supersede the
  relation).  Default: `false`.

| `name`
| no
| Override the step name shown in logs.
|===

*Losslessness condition*: lossless when relation rows are unique per
(row_keys, column_key) combination.  With `drop_source: true`, uncovered
relation column keys fail fast instead of silently dropping pairs.

.Persisted XRef intent
****
Both steps record one entry under `_meta.xref_crosstable.<id>` holding the
minimal feature-local inverse intent: `relation` and `matrix` frame names,
`row_keys`, and optional `dense_axes.rows_from` / `columns_from`.  The
`relation` / `matrix` names may legitimately reference a frame that final
domain cleanup has removed — a reverse pipeline needs exactly that intent to
recreate the absent side.  Run-local resolution (`column_keys`,
`dense_axes.resolved`) never survives the persistence boundary.  When intent
is looked up by frame name and more than one entry matches, the lookup fails
(in the transformations, in sparse-default targeting, and in column-role
resolution alike); name the intended transform explicitly — or pass explicit
`key_columns` to role-consuming steps — to disambiguate.
****

.Carrier-stable column identities
****
XRef distinguishes *matrix-axis identities* from *physical frame column
labels*, and applies a different rule to each.

*Matrix-axis identities* — the values that participate in the persisted
matrix/relation roundtrip — must be *unique, non-empty strings*.  Spreadsheet
carriers realize matrix headers as strings, so numeric, missing
(`None`/`NaN`/`pd.NA`), mixed-type (`1` vs `"1"`), duplicate, or unhashable
identities would silently change type, collide, or produce non-scalar cells
on readback.  Both steps enforce this on every axis-identity source —
relation `column_key` values, explicit and metadata-derived `column_keys`,
dense-axis-derived identities (live and stored), scoped `base_relation`
column identities, and the full physical *matrix* header row (matrix headers
are themselves axis identities) — and fail with explicit diagnostics before
any `drop_source` cleanup command is recorded.  Explicit `value_columns` are
validated as axis identities *before* they are compared against `row_keys`,
so a missing, numeric, or array-valued selector fails with the same
diagnostic rather than a raw pandas/NumPy error.

*Physical frame column labels* — the labels used to address a column on a
participating relation or `base_relation` frame — carry a weaker, still
strict contract: they must be non-missing, unique, deterministically
comparable, and safe for scalar column selection (so `row[label]` yields a
scalar, never a `Series`).  They are *not* required to be strings: a unique
numeric label, for example, is accepted.  Missing-like
(`None`/`NaN`/`pd.NA`), ambiguous-equality (for example a multi-element
array), or duplicate physical labels are rejected before any output or
cleanup.  Hierarchical/tuple physical labels and the separation of stable
technical identity from visible header labels remain future design work;
they are neither promised nor prohibited by a global string-only rule.

Malformed field declarations fail the same way, before any output or
cleanup: duplicate physical column labels on the relation, matrix, or
`base_relation` frame; duplicate configured `row_keys`; and explicit
`value_columns` that overlap `row_keys`.
****

[source,yaml]
----
- step: contract_xref
  relation: permission_rows
  output: role_permissions
  row_keys: role_id
----

.`column_key` vs `column_keys`
****
`column_key` and `column_keys` look similar but carry different information:

[cols="1,3", options="header"]
|===
| Parameter | Meaning
| `column_key`
| *Field name* — the name of the field in the long/relation frame that holds
  the column-axis address (e.g. `character_name`, `context_id`, `product_id`).
| `column_keys`
| *Value list* — an explicit, ordered list of concrete axis values that become
  the visible matrix columns (e.g. `[Galli, Donzo, Rexi]`).
|===

Example: `character_name` is the field name; `Galli`, `Donzo`, `Rexi` are the values.

[source,yaml]
----
- step: contract_xref
  relation: story_cast_role_named
  output: story_cast_role_matrix
  row_keys: [story_id]
  column_key: character_name      # field name in the relation
  column_keys: [Galli, Donzo, Rexi]   # concrete axis values
  value: value
----

`column_key` does not have to be a technical primary key.  A configured unique
alternate key (such as `character_name`) is valid when it is declared unique.

*Column-axis key sources* (`contract_xref` only):

. `column_keys` only → list used directly.
. `dense_axes.columns_from` only → list resolved from the entity frame at runtime.
. Both supplied → must match exactly; a mismatch fails fast (consistency
  contract, not precedence).
. Neither → falls back to `column_keys` stored in meta from a prior run, then
  to relation column order.
****

.Sparse-default cells and `expand_xref`
****
`expand_xref` does not interpret sparse-default metadata.  If blank cells in
the matrix represent a compressed default (introduced by `sparse_collapse`),
run `sparse_expand` before `expand_xref` to restore explicit default values.
`sparse_collapse` / `sparse_expand` are separate transformations with their own
registered steps.
****

==== `decode_cell_values` / `encode_cell_values`

*Category*: primitive · *Inverse*: conditionally bijective

Decode a compact cell string (e.g. `"A,B,C"`) into one row per code, or encode
code rows back into a compact cell string. Used inside the compact multiaxis
chain or independently for coded single-column cells.

See also: concept doc `technical_model/ch04_concepts/compact_transforms.adoc`.

Both steps require *explicit codec intent*; there is no metadata fallback
and no `_meta.cell_codecs` entries are written. Two contracts exist:

* *Position-based* (`codec_intent`): several participating columns are
  joined into one compact column and split back by position.
* *Historical code-row* (`mode` + codec parameters): rows of codes are
  joined into a delimited cell and split back. Consumed primarily by the
  compact-multiaxis composition; available directly when `mode` is passed
  explicitly.

.Cell Codec is string-oriented and writes no metadata
****
Cell Codec follows the string substrate
(`ADR-VISIBLE-CELL-TYPING-STRING-SUBSTRATE`): position-based participating
values must be non-empty strings or empty; numeric, boolean, or date values
fail with an explicit diagnostic instead of silently changing type through
the compact-cell roundtrip. Empty and missing values encode as the
configured `absent_value` token and decode back to `""`. There is no
escaping or quoting: a value containing the separator fails encoding, and a
`split_tokens` token containing the delimiter fails serialization, so an
encoded cell can never be ambiguous to decode. Duplicate physical column
labels, duplicate configured columns, and output-column collisions fail
before any output is produced. Codec intent lives entirely in explicit step
configuration; no `_meta.cell_codecs` entries are produced (legacy entries
in old sidecars or workbooks are ignored pass-through sediment).
****

*Position-based `codec_intent`* (same mapping for both steps)

[cols="2,1,4"]
|===
| Key | Required | Description

| `participating_columns`
| yes
| Ordered structured columns joined into (encode) or recreated from
  (decode) the compact cell. Must be unique and must exist on the encode
  source; on decode they must not already exist next to the compact column.

| `compact_column`
| yes
| Name of the compact cell column. Must not be a participating column and
  must not collide with a passthrough column.

| `separator`
| yes
| Token separator inside the compact cell. Values containing it fail
  encoding.

| `absent_value`
| yes
| Token representing an empty/missing attribute. Must not contain the
  separator; a real value equal to it fails encoding; decodes to `""`.
|===

[source,yaml]
----
- step: encode_cell_values
  source: profiles
  output: profiles_compact
  codec_intent:
    participating_columns: [delivery, language, auth]
    compact_column: profile
    separator: " / "
    absent_value: "-"
----

*`decode_cell_values` historical code-row parameters*

[cols="2,1,4"]
|===
| Parameter | Required | Description

| `source`
| yes
| Name of the input frame containing encoded cell strings.

| `output`
| yes
| Name of the output frame with one row per decoded token.

| `mode`
| yes (for this contract)
| `whole_cell_code` (the entire cell is one code; punctuation such as `-`
  stays part of the code) or `split_tokens` (split on `delimiter`).

| `value`
| no
| Column in `source` holding the encoded string. Default: `value`.

| `code`
| no
| Column name in `output` for the decoded token. Default: `code`.

| `passthrough_columns`
| no
| Columns from `source` to copy unchanged into every output row.
  Default: every non-`value` column.

| `drop_empty`
| no
| If `true` (default), empty cells produce no output row; if `false`, an
  empty cell produces one row with an empty token.

| `delimiter`
| no
| Token separator for `split_tokens` mode. Default: `-`.

| `allowed_codes` / `allowed_tokens`
| no
| Explicit list of valid code/token strings. Validation fails on unknown
  values.

| `allowed_from_legend`
| no
| Name of a legend block from which to derive the allowed-value set.

| `normalize_case`
| no
| `upper`, `lower`, or omit to leave case unchanged.

| `strip`
| no
| If `true`, strip whitespace from each token. Default: `false`.

| `name`
| no
| Override the step name shown in logs.
|===

*Losslessness condition*: lossless when `drop_empty: false` and token order is
preserved (or `canonical_order` is set on the encode side).

[source,yaml]
----
- step: decode_cell_values
  source: assignments
  output: assignment_codes
  value: codes
  code: code
  delimiter: ","
  passthrough_columns: [employee_id, project_id]
----

*`encode_cell_values` historical code-row parameters*

[cols="2,1,4"]
|===
| Parameter | Required | Description

| `source`
| yes
| Name of the input frame with one row per code.

| `output`
| yes
| Name of the output frame with encoded cell strings.

| `group_by`
| yes (for this contract)
| Column name(s) to group rows before joining into a cell string. Must be
  unique.

| `mode`
| yes (for this contract)
| Matches `decode_cell_values` mode (`whole_cell_code` / `split_tokens`).

| `code`
| no
| Column containing the token to encode. Default: `code`.

| `value`
| no
| Column name in `output` for the encoded string. Default: `value`.

| `delimiter`
| no
| Separator for `split_tokens` mode. Default: `-`. A token containing the
  delimiter fails serialization (the cell would be ambiguous to decode).

| `canonical_order`
| no
| Explicit ordered list of codes; determines output token order.

| `allowed_codes` / `allowed_tokens`
| no
| Validation whitelist.

| `allowed_from_legend`
| no
| Legend block name for allowed-value derivation.

| `normalize_case`
| no
| `upper`, `lower`, or omit.

| `strip`
| no
| Strip whitespace before encoding.

| `name`
| no
| Override the step name shown in logs.
|===

[source,yaml]
----
- step: encode_cell_values
  source: assignment_codes
  output: assignments
  group_by: [employee_id, project_id]
  code: code
  value: codes
  delimiter: ","
----

==== `expand_compact_multiaxis` / `contract_compact_multiaxis`

*Category*: composite · *Inverse*: conditionally bijective

Compose `expand_xref` and `decode_cell_values` (or their inverses) to handle
compact multi-axis matrices where each cell encodes multiple coded values.
Parameters are a superset of the two constituent steps.

_Full parameter documentation planned._

---

==== `split_by_discriminator` / `merge_by_discriminator`

*Category*: primitive · *Inverse*: conditionally bijective

Split one frame into multiple named frames partitioned by a discriminator column
value, recording frame-name → discriminator mapping in meta. `merge_by_discriminator`
reconstructs the original frame from the split parts.

_Full parameter documentation planned._

---

==== `sparse_collapse` / `sparse_expand`

*Category*: primitive · *Inverse*: conditionally bijective

Replace configured default values in editable crosstable cells with a sparse
blank placeholder (`sparse_collapse`), or restore blanks to the configured
default (`sparse_expand`). Used to make sparse matrices more readable in
spreadsheet form.

_Full parameter documentation planned._

---

==== `flatten_headers` / `unflatten_headers`

*Category*: primitive · *Inverse*: conditionally bijective

Flatten MultiIndex or tuple-like DataFrame column headers into scalar string
labels (`flatten_headers`), or split scalar string headers back into MultiIndex
form (`unflatten_headers`).

_Full parameter documentation planned._
