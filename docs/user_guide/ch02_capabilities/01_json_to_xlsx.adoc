=== JSON → XLSX: Flat and Nested Data

This section walks through what happens when you convert JSON to XLSX
with the current tooling, using three concrete scenarios that illustrate
the current capability boundary.

All commands are run from the project root.
The test files are located under `interactive_test/smoke_nested/`.

==== Setup (Windows / PowerShell)

[source,powershell]
----
cd <project-root>
$env:PYTHONPATH = "src"
----

==== Scenario 1 — Flat Records (fully roundtrip-safe)

The simplest case: an array of flat JSON objects where every value is
a scalar.  No nesting, no arrays.

.`interactive_test/smoke_nested/in_flat/orders.json`
[source,json]
----
[
  {"order_id": "ORD-001", "customer": "Alice",   "product": "Widget A", "quantity": "3", "price": "12.50"},
  {"order_id": "ORD-002", "customer": "Bob",     "product": "Widget B", "quantity": "1", "price": "25.00"},
  {"order_id": "ORD-003", "customer": "Charlie", "product": "Widget A", "quantity": "5", "price": "12.50"}
]
----

.Pack to XLSX, then unpack back to JSON
[source,powershell]
----
.venv_win\Scripts\python -m spreadsheet_handling.cli.apps.sheets_pack `
    interactive_test\smoke_nested\in_flat -o tmp\smoke_flat.xlsx

.venv_win\Scripts\python -m spreadsheet_handling.cli.apps.sheets_unpack `
    tmp\smoke_flat.xlsx -o tmp\smoke_flat_out
----

.Result: `tmp/smoke_flat_out/orders.json`
[source,json]
----
[
  {"order_id": "ORD-001", "customer": "Alice",   "product": "Widget A", "quantity": "3", "price": "12.50"},
  {"order_id": "ORD-002", "customer": "Bob",     "product": "Widget B", "quantity": "1", "price": "25.00"},
  {"order_id": "ORD-003", "customer": "Charlie", "product": "Widget A", "quantity": "5", "price": "12.50"}
]
----

[NOTE]
====
The output is semantically identical to the input (column order and pretty-print
formatting may differ).  This scenario is **fully roundtrip-safe**.
====

==== Scenario 2 — One Level of Nesting: Array in a Field

Here each order has an `items` field that contains an array of objects.

.`interactive_test/smoke_nested/in_nested_1/orders.json`
[source,json]
----
[
  {
    "order_id": "ORD-001",
    "customer": "Alice",
    "items": [
      {"product": "Widget A", "quantity": 3, "price": 12.50},
      {"product": "Widget B", "quantity": 1, "price": 25.00}
    ]
  },
  {
    "order_id": "ORD-002",
    "customer": "Bob",
    "items": [
      {"product": "Widget C", "quantity": 2, "price": 8.00}
    ]
  }
]
----

.Pack and unpack
[source,powershell]
----
.venv_win\Scripts\python -m spreadsheet_handling.cli.apps.sheets_pack `
    interactive_test\smoke_nested\in_nested_1 -o tmp\smoke_nested1.xlsx

.venv_win\Scripts\python -m spreadsheet_handling.cli.apps.sheets_unpack `
    tmp\smoke_nested1.xlsx -o tmp\smoke_nested1_out
----

.Result: `tmp/smoke_nested1_out/orders.json`
[source,json]
----
[
  {
    "order_id": "ORD-001",
    "customer": "Alice",
    "items": "[{'product': 'Widget A', 'quantity': 3, 'price': 12.5}, {'product': 'Widget B', 'quantity': 1, 'price': 25.0}]"
  },
  {
    "order_id": "ORD-002",
    "customer": "Bob",
    "items": "[{'product': 'Widget C', 'quantity': 2, 'price': 8.0}]"
  }
]
----

[IMPORTANT]
====
The `items` array is **serialised as a Python-repr string** and stored in a
single spreadsheet cell.  The data is preserved but the structure is lost on
the round-trip:

* The XLSX sheet has three columns (`order_id`, `customer`, `items`).
* The `items` cell contains the raw list representation — not individual rows.
* The round-trip output differs from the input: the list becomes a string, and
  number types may change (`12.50` → `12.5`).

This is the current MVP behaviour.  Unpacking nested lists into separate sheets
is tracked as _FTR-NESTED-LISTS_ (Phase 4).
====

==== Scenario 3 — Two Levels of Nesting

The deepest test case: nested objects *and* nested arrays within arrays.
The `customer` field is a sub-object; `items[].extras` is an array-of-arrays.

.`interactive_test/smoke_nested/in_nested_2/orders.json`
[source,json]
----
[
  {
    "order_id": "ORD-001",
    "customer": {
      "name": "Alice",
      "address": {"street": "Main St 1", "city": "Berlin"}
    },
    "items": [
      {
        "product": "Widget A",
        "quantity": 3,
        "extras": [
          {"name": "Gift wrap",        "price": 2.00},
          {"name": "Express shipping", "price": 5.00}
        ]
      },
      {"product": "Widget B", "quantity": 1, "extras": []}
    ]
  },
  {
    "order_id": "ORD-002",
    "customer": {
      "name": "Bob",
      "address": {"street": "Oak Ave 7", "city": "Hamburg"}
    },
    "items": [
      {
        "product": "Widget C",
        "quantity": 2,
        "extras": [{"name": "Insurance", "price": 3.50}]
      }
    ]
  }
]
----

.Pack and unpack
[source,powershell]
----
.venv_win\Scripts\python -m spreadsheet_handling.cli.apps.sheets_pack `
    interactive_test\smoke_nested\in_nested_2 -o tmp\smoke_nested2.xlsx

.venv_win\Scripts\python -m spreadsheet_handling.cli.apps.sheets_unpack `
    tmp\smoke_nested2.xlsx -o tmp\smoke_nested2_out
----

.Result: `tmp/smoke_nested2_out/orders.json` (excerpt)
[source,json]
----
[
  {
    "order_id": "ORD-001",
    "customer": "{'name': 'Alice', 'address': {'street': 'Main St 1', 'city': 'Berlin'}}",
    "items": "[{'product': 'Widget A', 'quantity': 3, 'extras': [...]}, ...]"
  },
  ...
]
----

[IMPORTANT]
====
Both `customer` (a sub-object) and `items` (an array) are collapsed to
Python-repr strings.  The XLSX has only three columns.
Everything below the top level is unreadable in the spreadsheet.

This scenario demonstrates the current limit:
**any value that is not a scalar becomes a string cell**.
====

==== Summary: Current Nesting Behaviour

[cols="1,3,1,1"]
|===
| Scenario | Data shape | XLSX usable? | Roundtrip-safe?

| Flat
| Array of scalar records
| ✅ All columns visible
| ✅ Yes

| Nested 1 level
| Array field contains array-of-objects
| ⚠️ Array becomes one text cell
| ❌ Structure lost

| Nested 2 levels
| Sub-objects + nested arrays
| ❌ Sub-objects and arrays all become text
| ❌ Structure lost
|===

The planned feature _FTR-NESTED-LISTS_ (Phase 4) will introduce
`normalize_nested_lists()` / `compose_nested_lists()` domain steps that
unpack list fields into separate sheets (one sheet per list level) and
reconstruct the hierarchy on the way back.
