=== Quickstart: Running an ad-hoc steps file

Once you have tried the predefined shims, the next step is to
**customize the transformation** by describing your own sequence of steps.
This gives you full control over which operations are executed
and in which order.

You do this by writing a small **steps file** in YAML format
and running it with the generic command `sheets-run`.

---

==== 1. Create a steps file

Create a file named `steps.yml` (or any name you like) with this minimal content:

```yaml
io:
  input:  { kind: json, path: data/input }
  output: { kind: xlsx, path: build/out.xlsx }

steps:
  - step: validate
  - step: add_helper_columns
  - step: export
```

* io defines where the data comes from and where it goes.
* steps lists the operations to perform — each line is one step in the pipeline.

You can include any supported step type:
validation, enrichment, extraction, export, and so on.

==== 2. Run it

----
sheets-run --steps steps.yml -v
----

The command will:

. read your `steps.yml`,
. load the input tables from `data/input`,
. apply the listed steps in order,
. and write the result to `build/out.xlsx`.

Use `-v` or `-vv` for more detailed logs,
and `--debug` to show a full Python traceback in case of errors.

==== 3. Modify and rerun

Because the configuration is plain YAML,
you can easily experiment with different step sequences
without touching the code.

For example, to export as JSON instead of Excel:

----
io:
input:  { kind: xlsx, path: build/in.xlsx }
output: { kind: json, path: build/out_dir }

steps:
- step: parse
- step: export
----

and run again:

----
sheets-run --steps steps.yml
----

==== When to use a steps file

Use an ad-hoc steps file when you:

* want to prototype a new workflow quickly,
* need to test individual transformations,
* or prefer to keep your pipeline definition under version control without embedding it in a larger project configuration.

When your collection of steps stabilizes and you start maintaining several reusable workflows, it’s time to move on to the next section and integrate them into a *project-level configuration*.
