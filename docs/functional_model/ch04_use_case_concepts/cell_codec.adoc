== Use Case: Cell Codec (Compact Editable Cell Values)

Business-facing workbook views often need to show several related facts at one
intersection. Splitting every fact into its own worksheet or matrix is easier
for machines, but it makes human review worse: analysts must switch views,
compare scattered cells, and mentally reconstruct relationships that belong
together.

Cell Codec is the functional pattern for this situation. It compresses several
structured attributes into one compact, editable workbook cell and expands the
edited cell back into structured attributes. The goal is information density on
the workbook editing surface, not compact storage.

Source storyline:
`docs/cold_storage/working_context/functional_model_inquiry/cell_codec/cell_codec.adoc`.
The representative use case is the business-facing calculation workbook
roundtrip in `business_calculation_workbook_roundtrip.adoc`, where compact
matrix cells can keep related role or configuration facts visible together.
For shared lifecycle concepts see
`ch05_problem_space_overview/03_meta_lifecycle_framework.adoc`.

=== User Problem

A pipeline author may have a normalized relation such as:

[source,text]
----
product
market
delivery_model
language
authentication_mode
----

For editing, the useful workbook view may be a product-by-market matrix where
each cell contains a compact value such as:

[source,text]
----
SaaS / JP / OIDC
AppStore / EN / LDAP
----

The analyst sees one matrix and can inspect consistency across products and
markets without opening three separate matrices. The structured attributes
remain the source representation; the compact cell is a workbook editing form.

=== Functional Pattern

Cell Codec converts between:

[source,text]
----
structured attributes
<->
compact editable cell value
----

The conversion must be deterministic and reversible within the configured
scope. If a value cannot be decoded without guessing, the workbook surface is
too ambiguous and the configuration must become more explicit.

=== Step: Declare Compact Cell Intent

The pipeline author declares that one workbook-visible cell represents several
structured attributes and names the encoding strategy that makes the
transformation reversible.

==== Information Consumed

* *Structured attribute set* -- essential; the attributes that must survive the
  roundtrip.
* *Compact-cell encoding strategy* -- essential; the configured rule that maps
  attributes to cell text and back.
* *Optional legend identities or allowed value sets* -- optional; used when
  compact tokens should be constrained or resolved through known vocabularies.

==== Information Produced

* *Cell Codec transformation intent* -- durable transformation intent; needed
  so later encode/decode steps do not infer meaning from cell text alone.

==== Semantic Model Reference

Registry entry `cell_codecs`; conceptual piece *Cell Codec Transformation
Intent*. This name is used here to explain the Functional Model; it does not
define registry field paths.

=== Step: Encode Structured Attributes For Workbook Editing

The forward path turns structured attributes into compact cell values for the
workbook surface.

==== Information Consumed

* *Cell Codec transformation intent* -- essential.
* *Structured attribute values* -- essential; absent values must be represented
  according to the declared encoding strategy, not by ad hoc formatting.
* *Legend entry set or allowed value set* -- optional; constrains token choice
  where a controlled vocabulary is configured.

==== Information Produced

* *Compact editable cell value* -- workbook payload, intended for user editing.
* *Optional editing-support structures* -- validation lists, generated
  codebooks, helper ranges, or similar support surfaces. They improve editing
  quality but are not authoritative structured data.

==== Semantic Model Reference

Registry entry `cell_codecs`; existing Semantic Model concepts
*workbook-visible cells*, *configuration*, *metadata attributes*, and
*render directives* apply. Optional support structures use the lifecycle
language from the Functional Model meta lifecycle framework.

=== Step: Decode Edited Compact Cells

The reverse path reads compact workbook values and reconstructs the structured
attributes they represent.

==== Information Consumed

* *Edited compact cell value* -- essential.
* *Cell Codec transformation intent* -- essential; without it, the cell is just
  text and must not be parsed heuristically.
* *Optional legend entry set or allowed value set* -- optional unless the
  declared strategy depends on it.

==== Information Produced

* *Structured attribute values* -- the reconstructed values that rejoin the
  structured source representation.
* *Decode error or validation finding* -- produced when the compact cell cannot
  be interpreted under the declared strategy.

==== Semantic Model Reference

Registry entry `cell_codecs`; existing Semantic Model concepts
*runtime-consumed*, *policy*, *producer*, *consumer*, and *semantic ambiguity*
apply. The cell-codec intent is runtime-consumed because absence or mismatch
changes behavior.

=== Encoding Strategies

The Functional Model does not require one universal encoding mechanism. The
functional requirement is compactness plus reversibility.

* *Position-based encoding* -- meaning comes from position, for example
  `A-B--D-E`.
* *Disjoint value sets* -- meaning comes from the token's value domain, for
  example `SaaS JP OIDC`, where each token can belong to only one category.
* *Category-hinted values* -- meaning is carried in the value, for example
  `delivery:SaaS lang:JP auth:OIDC`.

=== MVP Scope

The MVP concept covers:

* whole-cell codes;
* token splitting under explicit configuration;
* configurable separators;
* deterministic roundtrip behavior.

Category-aware representations, numeric typed values inside compact cells, and
derived validation structures are future scope unless a separate FTR accepts
them.

=== Validation and Editing Support

Validation is adjacent to Cell Codec, not the core concept. Cell Codec answers
how a compact cell value maps to structured attributes. Validation answers
which compact values are acceptable for a particular editing surface.

Simple validation may come from explicitly configured allowed values or from
Legend Blocks. More advanced validation may derive allowed combinations from
category-specific value sets, but that is future scope. Editing-support
structures may be transient inside the system boundary and recreated when the
workbook is rendered again.

=== Distinction From Neighboring Concepts

* *Cell Codec* -- owns compact value transformation inside one workbook-visible
  cell.
* *XRef Crosstable* -- owns matrix projection and recomposition: which tuple a
  matrix cell addresses.
* *Legend Blocks* -- own controlled code vocabularies and optional visible
  explanations for allowed values.
* *Compact Multiaxis* -- composes matrix projection, compact cell values, and
  legend support into a higher-level workbook pattern. It is not the
  motivation for Cell Codec.

In short:

[source,text]
----
XRef Crosstable:  which relation address does this cell represent?
Cell Codec:      how are several attributes represented inside the cell?
Legend Blocks:   what values are allowed or explained?
Compact Multiaxis: a composition that uses the others together.
----

=== Lifecycle Across The System Boundary

The compact cell is an editing surface. The structured attributes remain the
source representation. Generated validation ranges, codebooks, or helper
surfaces may be internal editing aids and do not automatically gain long-term
authority.

When a workbook leaves the system boundary, editing-support structures may be
treated as ephemeral if they can be recreated from durable configuration or
source frames. This follows the boundary-pruning direction in the meta lifecycle
framework: support metadata should not become accidental source truth merely
because it was useful during workbook editing.
