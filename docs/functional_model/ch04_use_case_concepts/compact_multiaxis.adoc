== Use Case: Compact Multiaxis Composition

Compact Multiaxis is a convenience composition for workbook surfaces that need
matrix editing and compact structured cell values at the same time.

It exists to make the pipeline author's configuration and user-facing workflow
smaller and clearer. It does not define independent matrix semantics or
independent cell-code semantics.

Source references:

* `docs/cold_storage/backlog/ftrs_done/FTR-COMPACT-MULTIAXIS.adoc`
* `docs/cold_storage/backlog/ftrs_done/FTR-COMPACT-MULTIAXIS_review.adoc`

=== User Problem

A workbook may need a matrix where each intersection carries one or more short
structured codes.

[source,text]
----
feature_id | Product A | Product B
f1         | E-R       | K
f2         |           | E-K
----

The pipeline author could configure this as separate XRef and Cell Codec steps.
Compact Multiaxis offers a smoother surface for the common case where those
steps are meant to be used together.

=== Functional Boundary

Compact Multiaxis delegates the underlying responsibilities:

* *XRef Crosstable* owns matrix addressing and recomposition: which relation
  tuple a matrix cell represents.
* *Cell Codec* owns compact cell-value interpretation: how one cell represents
  one code or several tokens.
* *Legend Blocks* optionally own allowed-code vocabulary and visible
  explanations.
* *Compact Multiaxis* owns only the composition surface: using those pieces
  together as one configured workflow.

In short:

[source,text]
----
Compact Multiaxis = XRef Crosstable + Cell Codec + optional Legend support
----

This distinction is required. A compact-multiaxis implementation must not grow
parallel rules for matrix addressing, token parsing, code validation, or legend
meaning. Those belong to the delegated concepts.

=== Step: Expand A Compact Matrix

The forward/read path converts a compact matrix into a generic explicit relation
by applying the delegated matrix and cell-code rules.

==== Information Consumed

* *Compact matrix frame* -- essential; the workbook-like matrix being expanded.
* *XRef projection configuration* -- essential; identifies row keys, value
  columns, and the matrix address shape.
* *Cell Codec configuration* -- essential; defines whether each cell is a whole
  code or a tokenized compact value.
* *Legend identity or code groups* -- optional; used only for code validation or
  generic grouping.

==== Information Produced

* *Generic explicit relation* -- rows carrying the row key, column key, compact
  code/token, and optional generic group.
* *Composition trace* -- optional metadata that records the composed operation
  for diagnostics and roundtrip explanation. The trace does not replace the
  delegated XRef or Cell Codec semantics.

==== Semantic Model Reference

Registry anchors: `compact_multiaxis`, `xref_crosstable`, `cell_codecs`,
`legend_blocks`.

=== Step: Contract A Generic Relation To A Compact Matrix

The reverse/write path combines explicit relation rows back into compact matrix
cells by applying the delegated cell-code and matrix rules in the opposite
order.

==== Information Consumed

* *Generic explicit relation* -- essential; the relation to contract.
* *Cell Codec configuration* -- essential; defines how rows become compact cell
  values.
* *XRef projection configuration* -- essential; defines how relation addresses
  become matrix cells.
* *Optional column-key list or fill value* -- optional; controls visible matrix
  shape when explicitly configured.

==== Information Produced

* *Compact matrix frame* -- workbook-facing matrix surface.
* *Composition trace* -- optional metadata for diagnostics and roundtrip
  explanation; not an independent authority for cell or matrix meaning.

==== Semantic Model Reference

Registry anchors: `compact_multiaxis`, `cell_codecs`, `xref_crosstable`.

=== Sparse And Dense Axis Boundary

The current compact composition treats empty cells as sparse by default:
empty cells do not produce explicit relation rows unless configured otherwise.

Dense-axis reconstruction is not a Compact Multiaxis semantic rule. If a future
workflow needs to recreate a complete matrix from configured row and column
sources, that contract belongs to explicit XRef/axis semantics and must not be
inferred from labels, positions, or naming conventions.

=== Non-Goals

Compact Multiaxis does not:

* interpret the business meaning of codes;
* introduce a second matrix projection model;
* introduce a second cell-code parser;
* infer dense axes from workbook shape;
* replace XRef Crosstable, Cell Codec, or Legend Blocks as concepts.

Its value is ergonomic: one clear user surface for a common composition of
already-defined capabilities.
