= Use Case: Header Shape Projection

Header Shape Projection describes the reversible change between structured
column headers and flat workbook-facing column labels.

The concept is narrower than a future multi-row header feature. It only covers
the flattening and unflattening of header structure so that the same frame can
move between structured processing and simple editing or adapter surfaces.

== User Problem

Some workflows carry grouped or path-like attributes in the column header
itself. For processing, validation, or nested output, the structure is useful:

[source,text]
----
order / id
order / date
customer / name
----

For a simple spreadsheet editing surface, a one-line label may be easier to
read, filter, copy, or pass through a flat adapter:

[source,text]
----
order.id | order.date | customer.name
----

Users need this conversion to be explicit and reversible where the configured
rules make reversibility possible. The workbook should not silently lose which
flat label came from which structured header path.

== Functional Intent

The forward direction flattens structured header paths into flat column labels:

[source,text]
----
frame with structured column headers
-> flatten headers
-> frame with flat column labels
----

The reverse direction reconstructs structured header paths from flat column
labels:

[source,text]
----
frame with flat column labels
-> unflatten headers
-> frame with structured column headers
----

This is a frame-shape operation. It changes how column identity is represented,
not the row values, not the domain object identity, and not the workbook sheet
layout by itself.

== Step: Flatten Headers

Flattening converts structured or multi-part column identifiers into ordinary
column labels for a flat editing or adapter surface.

Information consumed:

* source frame;
* selected frame or all frames in the workflow;
* flattening mode, for example joining header parts or selecting the first
  visible/non-empty part;
* separator or naming convention when header parts are joined;
* collision policy for labels that would become identical after flattening.

Information produced:

* frame with flat column labels;
* deterministic column-label mapping when the workflow expects a reverse path;
* diagnostics for duplicate, empty, or ambiguous labels.

The flattening mode matters. Joining all header parts can preserve more
information than selecting only one level. A lossy mode may still be useful for
display, but it should not be treated as a complete reverse contract.

== Step: Unflatten Headers

Unflattening converts flat column labels back into structured header paths
using an explicit separator or mapping convention.

Information consumed:

* source frame with flat column labels;
* selected frame or all frames in the workflow;
* separator or explicit label-to-path mapping;
* policy for labels that do not contain the separator;
* collision and escaping policy.

Information produced:

* frame with structured column headers;
* diagnostics for ambiguous separators, duplicate paths, or unsupported mixed
  header shapes.

Unflattening is deterministic only under declared conventions. If a label can
be interpreted in more than one way, the workflow should fail or require an
explicit mapping.

== Relationship To Neighbor Concepts

Header Shape Projection differs from future multi-row header rendering:
flattening and unflattening change the frame's header representation, while
multi-row rendering decides how a structured header is displayed in a workbook.

Header Shape Projection differs from Hierarchical Config Projection: header
projection changes column identifiers; hierarchical projection changes between
tabular rows/columns and hierarchical configuration structures.

Header Shape Projection differs from Workbook View: workbook view decides sheet
names, placement, and rendering directives. Header projection prepares the
frame's column shape before or after that boundary.

Header Shape Projection differs from FK Helpers: helper columns may use
structured headers as a carrier detail, but header flattening is not an FK
helper requirement by itself.

== MVP Boundary

The supported minimum concept is:

* explicit flattening from structured headers to flat labels;
* explicit unflattening from flat labels to structured headers;
* stable separator or explicit mapping convention;
* diagnostics for non-reversible or ambiguous cases.

Future multi-row workbook rendering, visual merge behavior, style retention,
and nested object/list modeling are outside this chapter.

== Non-Goals

* no new multiheader feature definition;
* no visual layout or merged-cell requirement;
* no automatic hierarchy inference from labels;
* no nested-list modeling;
* no runtime, registry, adapter, production-code, or test change implied by
  this Functional Model chapter.
