= Derived Frame Views: Join And Discriminator Split

Derived frame views are workbook-facing evaluations of already structured
frames. They do not define a separate canonical model. They make a particular
editing or review surface easier to use by reshaping existing frames into a
more convenient view.

The source frames remain the structured pipeline inputs. The derived view may
repeat values from those sources. Because that creates intentional redundancy,
the return path must declare which surface has precedence for overlapping
editable values.

== User Problem

A workbook author often needs a surface that is easier to read or edit than the
canonical frame layout:

* a row should show attributes from a related frame next to the editable record;
* a discriminator such as locale, variant, phase, or context should become
  separate sheets or frames;
* the workbook should show the shape users work with, while the canonical frame
  shape remains suitable for storage and processing.

The transformation is therefore an evaluation for a workbook surface, not a new
business object.

== Step: Join Frames For A Derived View

`join_frames` derives a view frame by combining two frames with explicitly
configured keys, join mode, projected columns, and optional filters.

Information consumed:

* left and right input frames;
* configured join keys, including differently named or composite keys;
* join mode, such as left, inner, or semi;
* selected columns, optional renaming, and collision policy;
* optional pre- or post-join filtering;
* workflow intent for whether the result is read-only projection or an editable
  return surface.

Information produced:

* a materialized derived frame;
* a provenance relation from the derived frame to the source frames;
* a redundancy relationship when source attributes are repeated in the view;
* lifecycle/editability intent for the workbook path.

Semantic Model Reference:

* `join_frames`;
* frame lifecycle / readonly projection;
* workbook view.

`join_frames` is SQL-like in the narrow sense that it combines rows by keys.
It is not a general SQL engine, not an implicit schema inference layer, and not
an automatic synchronization mechanism between redundant frames.

== Step: Split A Frame By Discriminator

`split_by_discriminator` derives multiple frames from one source frame by
partitioning rows by a configured discriminator column.
`merge_by_discriminator` is the corresponding recomposition step.

Information consumed:

* source frame;
* discriminator column;
* target frame naming pattern;
* optional explicit value-to-frame map;
* optional row-order preservation intent.

Information produced:

* one derived frame per discriminator value;
* split metadata that records which frame represents which discriminator value;
* enough recomposition intent for `merge_by_discriminator` to restore the
  discriminator column and, when preserved, original row order.

Semantic Model Reference:

* `split_by_discriminator`;
* `merge_by_discriminator`;
* frame lifecycle / workbook view;
* meta carrier for split recomposition intent.

The split is a workbook layout decision. The discriminator remains canonical
data in the tuple relation; it is only moved out of the visible per-value frame
because the frame name or sheet role already identifies that value for the
editing surface.

== Return Path And Precedence

Derived frame views can intentionally duplicate data that also exists in source
frames. The return path must therefore be explicit about precedence.

A pragmatic default for an editable derived workbook view is:

* the edited derived view has precedence for the overlapping fields that the
  workflow declares as owned by that view;
* original source frames remain authoritative for fields not exposed or not
  owned by the derived view;
* if no ownership or precedence is declared, the derived view should remain a
  read-only projection or the import should fail on conflict.

For joined views this means, for example, that an editable joined table may be
treated as the authoritative return surface for its declared columns, even when
those values also appear in the original input frames.

For discriminator splits this means that edited per-discriminator sheets can be
merged back as the authoritative return surface for their declared rows and
fields, while the discriminator value itself is restored from the split mapping.

This rule is intentionally operational. It avoids hidden reconciliation logic
and keeps the workbook workflow understandable to users.

== Neighbor Concepts

`join_frames` differs from XRef Crosstable: a join combines attributes from
related frames into a row-oriented view; XRef Crosstable pivots a tuple relation
between row keys and a column axis into a matrix editing surface.

`split_by_discriminator` differs from XRef Crosstable: split keeps the row shape
and distributes rows into per-value frames; XRef changes the column shape.

Both steps differ from Compact Multiaxis: Compact Multiaxis is a convenience
composition that delegates to existing expansion/contraction and cell-codec
steps. Join and discriminator split are direct frame derivations.

== Non-Goals

* no new canonical data model;
* no parallel implementation of existing crosstable or codec logic;
* no automatic conflict reconciliation between redundant surfaces;
* no general SQL/query engine;
* no registry, pipeline, or runtime change implied by this Functional Model
  chapter.
