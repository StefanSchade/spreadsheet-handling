== Use Case: FK Helper Columns

FK helpers are readable, derived projection values placed next to a
foreign-key column. They help humans edit or understand a frame but are
not canonical payload.

Example: `departments` references `employees` by `employee_id`. In the
spreadsheet, the `departments` sheet shows `last_name` and `first_name`
next to the FK column so editors do not need to switch sheets. On
reimport, the helper columns are dropped; only `employee_id` enters the
canonical data set.

For the framework concepts that govern these steps (meta carriers, Type
1 / Type 2 events, lifecycle attributes) see
`ch05_problem_space_overview/03_meta_lifecycle_framework.adoc`.

=== Step: Declare FK Relations

Pipeline author declares which column in a source frame is a foreign key
to a target frame, and which fields of that target may serve as helpers.

==== Information Consumed

* *Frame schema* — optional.
  If absent the step has nothing to read; it is a no-op.

==== Information Produced

* *FK relation declaration* — not `drop_at_persistence`; not
  `drop_if_stale`. Durable; survives the sidecar boundary; must be
  present for any later enrichment run.
* *Allowed helper field list* — same lifecycle as FK relation
  declaration. Constrains which target fields may be projected.
  Prevents sensitive fields from being exposed.
* *Default helper field* — same lifecycle. Used when an enrichment step
  receives no explicit helper-field request.

==== Variant: Heuristic Inference

A sub-step that inspects naming conventions (e.g. column name patterns
for primary and foreign keys) and writes the same FK relation
declaration. Must be an explicit, opt-in configuration step — not
silent magic. Produces the same output with the same lifecycle
attributes.

==== Semantic Model Reference

`fk_relations`, `helper_policies`

=== Step: Enrich FK Helpers

Adds read-only helper columns next to FK columns in a source frame,
populated from the target frame according to the declared FK relation.

==== Information Consumed

* *FK relation declaration* — essential.
  No enrichment possible without it; step errors or is a no-op.
* *Allowed helper field list* — essential.
  Determines which target fields may be projected.
* *Target frame* — essential.
  The lookup data source.
* *Requested helper fields* — optional.
  Falls back to the declared default helper field if absent.

==== Information Produced

* *Helper column values* — data added to the source frame; not meta.
  May be literal values or spreadsheet formula (e.g. VLOOKUP).
  Should be rendered as read-only or visually marked.
* *Runtime helper column provenance* — `drop_at_persistence`;
  `drop_if_stale`. Identifies helper columns during same-run processing.
  Dropped at domain egress; never written to sidecar.
* *Durable helper column marker* — `carrier_takes_precedence`.
  Embedded in the workbook carrier on the tabular write path.
  Read by the drop step on reimport when runtime provenance is absent.

==== Semantic Model Reference

`derived.sheets.*.helper_columns` (runtime provenance),
`sheets.*.helper_columns` (durable workbook marker)

=== Step: Drop FK Helpers

On reimport, identifies helper columns by their provenance or durable
marker and removes them so they do not enter the canonical data set.

==== Information Consumed

* *Runtime helper column provenance* — optional; preferred when present
  (same-run scenario).
* *Durable helper column marker* — optional; fallback when provenance
  is absent (cross-run reimport).
* *FK relation declaration* — optional; last-resort reconstruction of
  expected column names from declared relations.

If no identification source is available, the step cannot safely
identify helpers and treats all columns as canonical payload.
This is intentional — the system does not guess silently.

==== Information Produced

* Nothing new. Runtime provenance is cleaned up after the drop.
  The durable marker is preserved (drop step does not own its removal).

==== Semantic Model Reference

`derived.sheets.*.helper_columns` (consumed, then cleared),
`sheets.*.helper_columns` (consumed; persists across reimport),
`fk_relations` (last-resort fallback)

=== Concept Extensions

==== Crosstable Axis Helpers

FK-helper-like projection may also apply to crosstables.

* Current scope: helper columns on the left side of a crosstable row
  axis.
* Future: helper rows above a crosstable column axis; multi-row headers.

The general concept may warrant a name broader than `helper_columns`:
relation helper projection, axis helper projection, or projection helper
marker.

==== Future: Editable Projection Helpers

A possible future feature where projection columns are editable.

* Values copied from source fields into a projection column.
* Spreadsheet users edit the projection copy.
* On reimport, edited values are written back to the original source
  field.

Key distinction:

* FK helper: derived, read-only, dropped on reimport.
* Editable projection helper: derived copy, editable, written back.

These have fundamentally different reimport semantics and would form a
separate step slice.

=== Simplification Note

The current implementation contains historical layering:

* legacy v1 per-target configuration;
* v2 relation policy;
* runtime provenance;
* durable workbook marker;
* multi-source fallback chain in the drop step.

Target direction (one model per concern):

* one durable relation/schema annotation (the FK relation declaration);
* one runtime provenance record (dropped at domain egress);
* one durable workbook marker (carrier-embedded, read on reimport);
* heuristic inference as an explicit optional step only;
* v1 per-target config treated as version mismatch → dropped, not
  migrated; v1/v2 duality resolved by the meta versioning policy.
