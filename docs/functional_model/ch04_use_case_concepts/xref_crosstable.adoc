== Use Case: XRef Crosstable Projection

An XRef crosstable is a scoped editable matrix projection over a durable tuple
relation. The tuple relation is the storage- and processing-friendly
representation. The matrix is the human-friendly editing surface: rows and
columns correspond to entity axis members, and each cell holds a relation entry
for that (row, column) address.

This is not merely a shape conversion. XRef is a scoped editable projection and
recomposition pair: the matrix is generated for editing, then recomposed into a
configured target relation frame, with explicit scoping that limits which
existing relation rows are affected.

Source storylines:
`docs/warm_storage/functional_transformation_concepts/xref_crosstable.adoc` and
`docs/warm_storage/functional_transformation_concepts/xref_002.adoc`.
For the framework concepts these steps build on (meta carriers, lifecycle
attributes) see
`ch05_problem_space_overview/03_meta_lifecycle_framework.adoc`.

=== Pipeline Flow

[source,text]
----
persisted input artifacts
-> transient working model
-> editable crosstable projection
-> edited workbook / projection
-> recomposed relation frame
-> persisted output artifacts
----

The framework does not own permanent state. It reads persisted artifacts,
constructs a transient working model for one run, performs configured
transformations, and writes output artifacts. The persistence mechanism
(filesystem, Git, CI workspace, or other integration) is outside XRef.

=== Terminology Note

"Canonical" is intentionally avoided here. In this project "canonical" can
conflate durable storage, semantic authority, freshness, and consistency --
properties that do not automatically coincide. Preferred terms:

* *durable source structure* -- the long-lived, storage-friendly tuple relation.
* *projection intent* -- the configured mapping: which frames, which key fields,
  which scope.
* *runtime resolution* -- concrete values computed for one run: resolved axis
  keys, materialized column positions, row identities.
* *transient working model* -- the in-memory frame inventory produced for one
  run.

=== Baseline Shape

Baseline XRef requires:

* two existing entity frames;
* a configured axis key field on each entity frame;
* row-axis and column-axis addresses are based on configured key fields;
* resolved axis key lists define the XRef scope; they may come from static
  configured lists or from configured axis sources such as
  `dense_axes.rows_from` / `dense_axes.columns_from`;
* an optional existing relation frame carried through recomposition;
* a generated editable matrix frame;
* a configured target relation frame for recomposition;
* the target relation is created if absent on the first run.

=== XRef Projection Declaration

Pipeline author declares the two entity frames, their axis key fields, the
configured axis key source for each axis, and the target relation frame.

==== Information Consumed

* *Row-axis entity frame* -- essential; absent, the step errors.
* *Column-axis entity frame* -- essential; absent, the step errors.
* *Row-axis key field* -- essential; the configured field used to address
  row-axis members.
* *Column-axis key field* -- essential; the configured field used to address
  column-axis members.
* *Resolved row-axis key list* -- essential; the axis key values that define
  which entity rows enter the matrix. Supplied as a static configured list or
  derived from a configured axis source (e.g. `dense_axes.rows_from`).
* *Resolved column-axis key list* -- essential; the axis key values that define
  which columns appear in the matrix. Supplied as a static configured list or
  derived from a configured axis source (e.g. `dense_axes.columns_from`).
* *Target relation frame name* -- essential for recomposition.
* *Existing relation frame* -- optional; carried through on the reverse path.

==== Information Produced

* *XRef projection intent* -- durable transformation intent; must be present
  for any projection or recomposition run.

==== Semantic Model Reference

Conceptual pieces *XRef Projection Intent*, *Axis Configuration* -- proposals
for the semantic model to formalise; not field paths.

=== Relation to Matrix Projection

Reads the existing relation frame and the resolved axis key lists, then produces
an editable matrix frame. Each cell holds the relation entry for its matrix
address, or is blank if the relation has no entry there.

==== Information Consumed

* *XRef projection intent* -- essential.
* *Row-axis entity frame* -- essential.
* *Column-axis entity frame* -- essential.
* *Resolved axis key lists* -- essential; define which rows and columns appear
  in the matrix.
* *Existing relation frame* -- optional; if absent on the first run, the
  matrix is blank.

==== Information Produced

* *Editable matrix frame* -- workbook payload, not meta; the generated sheet
  for user editing.
* *XRef runtime resolution* -- transient resolution data: resolved axis
  positions and address mappings for the current matrix. Whether stored in
  meta, embedded in workbook metadata, or reconstructed during reimport is
  an implementation question.

==== Semantic Model Reference

Conceptual pieces *Editable Matrix Frame*, *XRef Runtime Resolution* --
proposals; not field paths.

NOTE: Whether XRef runtime resolution is stored temporarily, embedded in workbook
metadata, or reconstructed during reimport is an implementation question. The
functional requirement is that recomposition can resolve present matrix cells
back to row-axis and column-axis addresses.

=== Matrix to Relation Recomposition

Reads the edited matrix frame, resolves each cell address to a
(row-axis-key, column-axis-key) pair, and writes scoped updates to the target
relation frame.

==== Information Consumed

* *XRef projection intent* -- essential; provides the axis key fields and
  scope.
* *Edited matrix frame* -- essential.
* *Existing relation frame* -- optional; out-of-scope rows are preserved.
* *XRef runtime resolution* -- optional; may be available from the same run,
  embedded in workbook metadata, or reconstructed from projection intent.

==== Information Produced

* *Updated relation frame* -- populated from recomposed cells plus carried
  out-of-scope rows from the existing relation.

==== Semantic Model Reference

Conceptual piece *Updated Relation Frame* -- proposal; not a field path.

=== Configured Axis Keys

XRef axes are addressed through configured key fields. A key field declares
which attribute of the entity identifies axis members. Key fields may be:

* the entity primary key field;
* a unique technical code field;
* a unique human-readable name field;
* a unique business identifier field.

A configured axis key field must not be:

* guessed by fuzzy matching from visible labels;
* accepted when non-unique within the axis entity frame;
* silently repaired when ambiguous;
* inferred from workbook header text without configuration.

Primary keys are preferred for long-term machine stability. However, requiring
technical primary keys in the visible matrix is often a usability regression.
Explicitly configured human-readable fields such as `name` are legitimate axis
key fields when they are declared and unique. This is a declared key contract,
not weak typing.

Key fields and key values are distinct. The key field is declared in the axis
configuration; the key values are the actual entity identifiers resolved against
that field. For example, `name` is the key field; `Galli`, `Donzo`, `Rexi` are
key values. Key values may be supplied from static configured lists, from
configured axis sources, or from a future query/filter step.

==== Relation Enrichment Before XRef

A pipeline may enrich a technical relation with a human-readable key field
before creating the crosstable. This is a legitimate pipeline composition
pattern:

[source,text]
----
technical relation (story_id, character_id, value)
-> join with characters (character_id -> characters.id, name -> character_name)
-> named relation (story_id, character_name, value)
-> XRef: column_key = character_name
         columns_from: { frame: characters, key: name }
----

In this pattern:

* `characters.name` is the configured axis source key field.
* `character_name` is the enriched relation column carrying the same address values.
* `contract_xref` uses `character_name` as the relation/matrix `column_key`.

The key domain is explicitly configured, not heuristically parsed from visible
labels. The key field (`characters.name`) is unique in the entity frame. Reverse
recomposition writes to the configured target relation. Any later translation
back to technical IDs is a separate configured step.

This pattern is supported and must be preserved. It is part of the projection
architecture introduced by `FTR-PROJECTED-FRAME-COLUMN-SEMANTICS-P5` (column
role taxonomy) and `FTR-DYNAMIC-FRAME-PROJECTION-P5` (`project_by_role`).

==== Axis Key Value Sources

Axis key values may be supplied from two currently supported sources:

*Static configured lists* enumerate values directly in the pipeline
configuration and are first-class scope declarations, not legacy behavior:

[source,yaml]
----
column_axis:
  frame: characters
  key: name
  keys: [Galli, Donzo, Rexi]
----

*Configured axis sources* derive the full key list from a declared frame and
key field and are used in the current worldbuilding integration:

[source,yaml]
----
dense_axes:
  rows_from: { frame: stories, key: id }
  columns_from: { frame: characters, key: name }
----

In both cases: values are interpreted against the declared key field; duplicates
fail; unknown values fail; no fuzzy matching.

A future *Axis Key List Producer* step may derive key lists via configured
queries or filters; that step is separate from XRef itself.

==== Relationship to Future Axis Mapping

A future richer axis model (`FTR-XREF-AXIS-MAPPINGS-P4A2`) may separate:

[source,text]
----
identity_key:   stable technical identity  (e.g. characters.id)
address_key:    visible workbook axis key  (e.g. characters.name)
display_label:  human-readable header
order_key:      sort field
group_key:      classification / grouping
----

The current baseline uses a single configured key field for both addressing and
display. The baseline wording is broad enough to accommodate this future model
without requiring it now.

=== Axis Scope

XRef does not decide which entity rows are active, current, or relevant. The
baseline accepts resolved axis key lists, whether statically configured or
derived from configured axis sources. XRef consumes the resolved list; it does
not interpret fields such as `active`, `status`, `valid_until`, or
`business_area`.

Axis order is a functional concern. Baseline uses explicit list order. XRef
must not rely on accidental dataframe order.

=== Scoped Update and Carried Relation

XRef owns only its configured update scope.

[source,text]
----
inside the scope:
  visible matrix cells produce update statements

outside the scope:
  existing relation rows are preserved unchanged
----

This is scoped update, not whole-relation replacement. If an existing relation
frame is present, its rows outside the XRef scope pass through unchanged.
Present, addressable cells inside the scope replace or remove their addressed
relation entries. Missing rows or columns produce no update statement for those
addresses.

=== Blank, Missing, and Invalid Address Semantics

[cols="2,3", options="header"]
|===
| Input condition | Meaning

| present cell with value
| Set or replace the relation entry for that matrix address.

| present blank cell
| Remove the relation entry for that matrix address inside the update scope.

| missing row or missing column
| No matrix address exists; no update statement is produced for those addresses.

| invalid or ambiguous row/column identity
| Fail fast.

| relation row outside the XRef update scope
| Preserve unchanged.
|===

XRef does not infer whether the user accidentally deleted a row or column. If
the row or column is absent from the matrix, no update is produced for those
addresses. Detecting renamed headers, corrupted matrix shape, or missing
structural rows belongs to a separate *Projection Structure Integrity* concept,
not to XRef.

=== Error Philosophy

XRef baseline fails fast rather than applying hidden heuristics. Errors for:

* missing row-axis or column-axis entity frame;
* missing configured axis key field;
* non-unique configured alternate key in the axis entity frame;
* relation tuple references an unknown row-axis or column-axis key value;
* duplicate relation entries in scalar baseline mode;
* visible matrix row or column cannot be resolved to a configured axis key;
* ambiguous axis value resolution;
* unparseable cell payload.

No fuzzy matching, silent repair, or automatic renaming in baseline.

=== Value Semantics

XRef owns cell addressing, not token interpretation.

[source,text]
----
XRef:                      which tuple does this matrix cell address?
Cell Codec / Legend Block: what does the visible token mean?
----

Baseline supports simple scalar or boolean-like values (for example `X` for
present, blank for absent). XRef does not bind to any particular token format.
Token parsing, validation, and serialization belong to neighbouring concepts.
See `ch04_use_case_concepts/legend_blocks.adoc` for value resolution via Legend
Blocks.

=== Metadata Pressure

The current implementation exposes classification pressure in
`_meta.xref_crosstable`. Two distinctions need clarification without proposing
a final schema:

* `column_key` is an axis attribute (the configured key field name).
  `column_keys` is the list of concrete resolved axis members for one run.
  These are different kinds of information.

* `dense_axes.columns_from` / `rows_from` represent *projection intent*.
  `dense_axes.resolved.*` represents *runtime resolution*.

The Functional Model records these distinctions. Proposing a final `_meta`
shape is deferred to an implementation concept.

=== Deferred and Neighbouring Concepts

Not part of baseline XRef:

* *Axis Mapping* (`FTR-XREF-AXIS-MAPPINGS-P4A2`) -- a richer future model that
  separates identity key, address key, display label, order key, and group key
  per axis; enables grouped headers and clean separation of technical identity
  from workbook-visible addressing.
* *Axis helper display* -- helper columns showing entity labels next to row-axis
  keys, or helper rows above column-axis keys; read-only display, not identity.
  See `ch04_use_case_concepts/fk_helpers.adoc` (Crosstable Axis Helpers
  extension).
* *Editable helper writeback* -- helper values written back to the source entity
  frame on reimport; fundamentally different reimport semantics.
* *Grouped headers* (`FTR-XREF-GROUPED-COLUMNS-P4A2`) -- structured axis
  display (category → member); requires the clarified axis model above.
* *Compact multi-value cells* -- one cell representing multiple relation tuples;
  Cell Codec / Compact Multiaxis responsibility.
* *Composite cells over several relations* -- combining multiple relation
  projections over shared axes into one visible cell.
* *Legend Block value resolution* -- XRef cells may display legend codes; the
  vocabulary and resolution belong to Legend Blocks.
* *Cell Codec parsing* -- token-level encoding and decoding.
* *Projection Structure Integrity* -- detecting renamed headers, corrupted matrix
  shape, renamed helper columns; cross-cutting concern.
* *Workbook hiding/protection UX* -- whether the tuple relation sheet is visible
  or write-protected.
* *Review/diff/promotion workflow* -- semantic diff of the recomposed relation
  before committing; integration concern.
* *Git-specific lifecycle assumptions* -- persistence mechanism is outside XRef.

=== Open Questions

* How should static and dense-axis key sources be unified in the current YAML
  surface, and what is the preferred configured form when both
  `column_key` / `column_keys` and `dense_axes.columns_from` are present?
* Should relation frame carry-through be modeled as ordinary frame inventory,
  frame lifecycle metadata, or a recomposition-specific working input?
* Should XRef baseline support row-axis helper display immediately, or defer all
  helper display to a later axis-helper feature?
* How should duplicate tuples be handled when the relation has additional fields
  beyond row-axis key, column-axis key, and value?
* How should scalar payload mode distinguish blank, empty string, and absent
  tuple, if at all?
* Does current `drop_empty` already encode the baseline blank-cell behavior, or
  is its meaning narrower in implementation?
* Where should recomposition precedence live if multiple editable projections
  target the same relation?
* Which current `_meta.xref_crosstable` fields are load-bearing for consumers
  and which are historical residue?
