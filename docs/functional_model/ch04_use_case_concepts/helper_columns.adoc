== Use Case: FK Helper Columns for Spreadsheet-Friendly Roundtrip

=== Status

* Draft skeleton
* Intended as one use-case chapter within a larger concept document
* Scope is the first reversible helper-column feature slice, not later formula-driven refinements

=== Purpose

This use case describes how foreign-key helper columns support spreadsheet-friendly editing while preserving a normalized repository representation.

The goal is to let users edit a referencing table in a spreadsheet without losing orientation when a foreign key identifies a related row only by an opaque key.

Example: a `desks` table references an `employees` table by employee number. In spreadsheet form, the `desks` sheet may show additional helper columns such as employee name and surname next to the FK column so the user can edit the sheet without constantly switching to the `employees` sheet.

=== Roundtrip Intent

The roundtrip goal is:

* spreadsheet representation may contain redundant helper columns for usability
* canonical repository representation should not store those helper values as ordinary payload columns
* the enrichment into helper columns must therefore be reversible

In the first feature stage, this is achieved by a pair of domain transformations:

* enrich helper columns
* remove helper columns

=== Carrier Problem

A spreadsheet user needs additional context for foreign keys.

A pure FK value is often insufficient for safe manual editing.

Therefore the spreadsheet-facing representation may contain derived columns that make the FK relation human-readable.

These helper columns are part of the referencing sheet in physical layout terms, but semantically they are derived from the referenced sheet.

=== Current Contract

The first supported contract is intentionally restrictive.

* clean tabular sheets only
* one logical table per sheet
* helper columns are generated in a controlled way
* helper columns are removed again before persisting to normalized repository form
* heuristics may be used to bootstrap missing metadata, but they must not silently contradict existing metadata

At this stage, renaming of relevant sheets or helper columns may be treated as unsupported user action.

=== Logical Responsibilities

==== Referenced Table

The referenced table carries metadata about the source of helper data.

Examples:

* which column is the primary key
* which columns are eligible as helper sources
* which helper fields are defaults

==== Referencing Table

The referencing table carries metadata about concrete helper-column materialization.

Examples:

* this column is a helper column
* this helper belongs to this FK column
* this helper derives from this referenced table field
* this helper uses this naming convention or prefix

This avoids collapsing source-side helper eligibility and target-side helper materialization into one undifferentiated metadata statement.

=== Domain Flow

==== Enrich Step

The enrich step:

* detects or resolves FK relations
* selects helper source fields
* adds helper columns to the referencing sheet
* records enough metadata to identify those helper columns on later reimport

==== Remove Step

The remove step:

* identifies helper columns from metadata
* removes them from payload before persistence into normalized repository form
* removes or updates helper-current-state metadata associated with the materialized helper columns

=== Metadata Needed

==== Source-Side Metadata

Likely source-side metadata includes:

* primary key declaration
* helper-eligible fields
* helper default fields

==== Target-Side Metadata

Likely target-side metadata includes:

* FK declaration or resolved FK association
* helper-column marker
* provenance of each helper column
* relation between helper column and FK column
* relation between helper column and source field

==== Open Metadata Question

It remains to be defined which of these statements are:

* durable workbook intent
* bootstrap output
* current-state operational metadata
* persisted but non-canonical derived metadata

=== Detection and Bootstrap

==== FK Detection

The first version may support heuristic FK detection, for example by naming convention.

This is a bootstrap aid, not the final source of truth.

==== Bootstrap Role

When metadata is absent, a bootstrap step may incubate initial metadata.

In standard operation, bootstrap should fill gaps only.

Overwriting existing metadata should require an explicit opt-in.

==== Stabilization Rule

Once the system has inferred enough structure to operate safely, that interpretation should be persisted so later roundtrips rely less on fragile heuristics.

=== Precedence

The use case requires an explicit precedence model.

Suggested order:

* explicit run overrides
* sheet-local explicit metadata or config
* workbook-wide explicit metadata or config
* persisted inferred or bootstrapped metadata
* fresh heuristics as fallback only

Operational rule:

* heuristics may fill missing gaps
* heuristics must not silently override existing metadata

=== Naming and Prefixes

==== FK Naming

FK detection may use a configurable naming convention in the first stage.

This naming convention is a carrier-facing aid for detection and bootstrap.

==== Helper Naming

Helper columns likely require their own configurable prefix to avoid collisions and to support reversible cleanup.

The helper prefix serves a different purpose from the FK prefix:

* FK prefix supports relation detection
* helper prefix supports disambiguation and reversibility

=== User Interaction Policy

The first version should be conservative.

Suggested initial contract:

* helper columns are derived, not authoritative
* user edits to helper values are not treated as source truth
* helper values may be overwritten by re-enrichment
* unsupported structural actions may be rejected, ignored, or cleaned up conservatively

Likely unsupported in the first version:

* renaming helper columns
* renaming relevant sheets outside dedicated tool support

=== Rendering Possibilities

Helper metadata may later influence rendering.

Possible later behaviors:

* distinct color scheme
* grouping with the FK column
* write protection
* formula-based helper values instead of copied values

These rendering behaviors are separate from the core reversible enrichment concept.

=== First Feature Stage

The first stage should stay narrow.

In scope:

* reversible helper enrichment and removal
* helper provenance metadata
* conditional bootstrap when metadata is missing
* explicit precedence between overrides, metadata, and heuristics
* conservative overwrite policy for user-edited helper values

Out of scope for the first stage:

* formula-driven helper columns
* robust rename support
* sophisticated orphan-metadata repair
* generalized metadata inheritance model beyond what is needed here

=== Open Decisions

* exact FK naming convention
* exact helper naming convention
* exact metadata fields and registry entries
* exact default behavior when no metadata exists
* exact persistence policy for helper-related current-state metadata
* exact scope and inheritance rules for helper-related settings

=== Acceptance Direction

The first concrete acceptance slice should prove at least:

* a referencing sheet can be enriched with helper columns derived from a referenced sheet
* helper columns are distinguishable from ordinary payload columns
* helper metadata survives spreadsheet persistence and reimport
* helper columns can be removed again for normalized repository persistence
* existing metadata is not silently contradicted by fresh heuristics
* bootstrap does not overwrite existing metadata unless explicitly configured to do so

=== Rationale

This use case exists because spreadsheet usability and repository normalization pull in different directions.

The spreadsheet form may be verbose and redundancy-tolerant for human editing.

The repository form should remain normalized, traceable, and low in redundancy.

Helper columns are one concrete mechanism for bridging those two representations in a controlled roundtrip.
