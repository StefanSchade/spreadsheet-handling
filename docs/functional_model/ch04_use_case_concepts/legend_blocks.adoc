== Use Case: Legend Blocks (Controlled Compact Codes)

A pipeline author wants a spreadsheet column or range to accept only a
controlled set of compact codes. The workbook provides dropdown validation from
that code list, optionally shows a visible legend explaining the codes, and on
reverse transformation may resolve each code back into a more explicit
structured value. The spreadsheet surface stays compact while the structured
model stays explicit.

This is the baseline. Richer use cases (matrix value areas, compact multi-token
cells, frame-sourced codebooks) follow by extension and are listed, out of
scope, at the end.

Source storyline:
`docs/cold_storage/working_context/functional_model_inquiry/legend_blocks/legend_blocks.adoc`.
For the framework concepts these slices build on (meta carriers, Type 1 / Type 2

`ch05_problem_space_overview/03_meta_lifecycle_framework.adoc`.

=== Terminology: Code, Label, Long Form

A Legend Block is a named code list. Each entry has up to three value roles:

* *code* (token) — the compact value edited in the cell (`N`); stable.
* *label* — the human-readable display meaning shown in the visible legend or
  dropdown (`Numeric`); display-only, may be localised, not referenced by
  persisted data.
* *long form* — the explicit value written into the structured tuple
  (`numeric`); stable and referenced by persisted data.

In the simplest legends the three collapse (long form and label default to the
code); they are separated only when the cell value, the display text, and the
structured value genuinely differ.

This chapter uses *code* in prose; *token* is the formal synonym, kept in the
combination, codec, and localization passages.

=== Source Modes

A legend's entries have a durable home outside `_meta` (which is gitignored and
may be absent or stale). The home determines the source mode:

* *Config-backed* — entries declared inline in the pipeline config.
* *Frame-backed* — entries read from an existing data frame.

Both are equal functional concepts. Implementation status (a fact about the
code, kept separate from the spec and binary): config-backed is built;
frame-backed is not. The functional model describes both regardless.

=== Step: Define Legend Entries

Declares or provides a named set of codes and their meanings; creates a reusable
code vocabulary and renders nothing by itself.

==== Information Consumed

* *Inline entries* (config-backed), or *source-frame configuration*
  (frame-backed, not built today) — one is required to produce a legend.

==== Information Produced

* *Legend entry set* — durable in its source home (config or frame). The
  durable entry set itself is not dropped at persistence; it lives outside
  `_meta`.
* *Resolved legend working form* (in `_meta`) — run-local; re-derived from the
  durable home each forward run and not trusted from a prior `_meta`;
  `drop_if_stale` and `drop_at_persistence`.
* *Legend identity* — durable; reference-bearing for constraints, render, and
  codecs.

==== Semantic Model Reference

Conceptual pieces *Legend Entry Set*, *Legend Identity* — proposals for the
semantic model to formalise; not field paths.

=== Step: Apply Legend to a Column or Range

Applies a Legend Block to a normal spreadsheet column or range.

==== Information Consumed

* *Legend identity* — essential; absent, the legend cannot be resolved and the
  step is a no-op (or errors).
* *Legend entry set* — essential; absent, there are no allowed values to apply.
* *Target column or range* — essential; absent, there is nothing to constrain.

==== Information Produced

* *Workbook validation / dropdown rule* — durable projection intent;
  `render_consumed`.
* *Visible legend request* — optional, if configured; durable projection intent.
* No payload transformation by default.

==== Semantic Model Reference

Conceptual piece *Target Usage* — proposal; not a field path.

=== Step: Render Visible Legend

The renderer turns a Legend Block into a visible support table.

==== Information Consumed

* *Legend entry set* — essential; absent, there is nothing to render.
* *Visible legend request* — essential to trigger this step; absent, no visible
  legend is rendered.
* *Optional title*, *render / layout context* — optional.

==== Information Produced

* *Rendered legend surface* — a workbook support surface, not ordinary payload;
  must stay distinguishable from payload on readback.
* *Workbook reconstruction hint* — carrier-local; `drop_at_persistence` at the
  sidecar; recomputed on the next render.

==== Semantic Model Reference

Conceptual pieces *Visible Legend Request*, *Rendered Legend Surface*,
*Workbook Reconstruction Hint* — proposals. Detailed layout policy is out of
scope (see Extension Pressure).

=== Step: Reimport with Generated Legend Surface

The parser reads a workbook containing data tables and generated legend
surfaces.

==== Information Consumed

* *Visible workbook tables* — essential; absent, there is nothing to parse.
* *Workbook reconstruction hint* — optional; used to recognise generated legend
  surfaces.

==== Information Produced

* *Payload frames only* — generated legend surfaces do not become ordinary
  payload frames.
* *Parser table classification* — transient read-path result; not project meta.

==== Semantic Model Reference

Conceptual piece *Parser Table Classification* — proposal. This is the safety
step that keeps support surfaces out of the structured model.

=== Step: Resolve Codes on Reverse Transformation (optional)

The reverse pipeline maps compact codes back to more explicit structured values.

==== Information Consumed

* *Payload cell values* — essential.
* *Legend entry set* — essential.
* *Code resolution rule* — essential when resolution is requested; must be
  explicit, never guessed from visible labels.

==== Information Produced

* *Structured values* using long-form values, labels, or configured target
  fields.

==== Semantic Model Reference

Conceptual piece *Code Resolution Rule* — proposal. Resolution differs from
validation: validation says which code is allowed; resolution says what the code
becomes in the structured model.

=== Lifecycle Across the Git Boundary

Essential legend information never depends on surviving in `_meta`. The durable
truth lives in the source home (config or frame); `_meta` carries only the
resolved working form and carrier-local workbook reconstruction hints, both
droppable and re-derived each run.

Neither the resolved legend working form nor the workbook reconstruction hints
are durable source truth; both are re-derived from the legend's durable home.
Where structured data is written across a boundary at which a matching `_meta`
cannot be guaranteed, *boundary pruning* of this transient meta is the safe
conceptual operation (see *Boundary Pruning* in
`ch05_problem_space_overview/03_meta_lifecycle_framework.adoc`).

Feature-level lifecycle pressure recorded but not resolved by this slice:

* *Second roundtrip* — persisted tuples become the next forward run's input and
  must rebuild the view; code resolution must be reversible and the legend
  stable across runs.
* *Legend evolution* — because tuples reference the long form by value, changing
  a long form is a maintenance event; the clean form is a dedicated maintenance
  transformation (peer to table/column rename; cf. `FTR-RENAME-RECONCILIATION`),
  not an in-band edit. Changing a label is harmless (display-only).
* *Allowed combination set* — for combined compact cells, the permitted token
  combinations are their own concept (durable config when supplied or
  constrained, e.g. `N/I` forbidden; or derived but bounded against explosion).

=== Extension Pressure (Out of Scope for This Slice)

Mentioned only to record direction; explicitly out of scope here:

* *Crosstable value areas* — applying a legend to a matrix value area; the first
  extension after the baseline column/range case.
* *Compact multi-token cells (Cell Codec / Compact Multi-Axis)* — packing
  several tokens into one cell; a neighbouring family. Legend Blocks provide the
  vocabulary, Cell Codec the packing/unpacking.
* *Frame-backed (source-frame) legends* — described above as a source mode; not
  built today.
* *Localization / discriminator-dependent legends* — deferred; in the safe near
  term only the label is locale-variable, code and long form stay stable.
* Renderer layout policy, write protection, styling, and editable in-workbook
  codebooks.

=== Open Questions

* Whether the allowed combination set is supplied or derived, and how domain
  constraints (e.g. forbidden combinations) are declared.
* The shape of the maintenance transformation for legend evolution.
* Localization: whether only labels vary by locale, or token identity too
  (deferred; needs its own FTR).
