= Use Case: Validation And Constraints

Validation is a cross-cutting use case for making workbook roundtrips
trustworthy. It answers whether edited or generated data still satisfies the
rules the workflow depends on.

Validation is not one single mechanism. Some rules are rendered into the
workbook to guide editing. Other rules run before or after projection, after
reimport, or before publishing artifacts. The common functional intent is that
invalid data should be visible, reportable, and, where configured, blocking.

== User Problem

Spreadsheet editing is useful because users can inspect and change many values
quickly. That also makes mistakes easy:

* a primary-key column may contain duplicate values;
* a foreign-key column may reference a missing row;
* a tuple relation may violate a uniqueness constraint;
* a matrix or coded cell may contain a value outside an allowed set;
* a graph-shaped model may contain edges whose endpoints do not exist;
* a future workflow may need domain-specific checks, such as numeric ranges,
  prime-number constraints, or consistency with an external source.

The Functional Model treats validation as an explicit workflow concern, not as
an incidental parser behavior.

== Validation Forms

Validation can appear in several forms.

=== Workbook Editing Constraint

A workbook editing constraint helps the user enter valid values before the
workbook is reimported. Examples include dropdown lists, protected columns, or
sheet-level validation ranges.

Information consumed:

* target sheet, frame, column, or column-role declaration;
* allowed values or a source from which allowed values can be resolved;
* whether blank values are allowed;
* render intent for the workbook surface.

Information produced:

* workbook-facing validation or constraint intent;
* optional render directives consumed by the spreadsheet boundary.

This form supports usability. It does not replace data validation after
reimport, because users or tools may still bypass spreadsheet UI constraints.

=== Reference And Uniqueness Validation

Reference and uniqueness validation checks structural integrity of frames.

Information consumed:

* frame and column declarations;
* primary-key or uniqueness rules;
* foreign-key or reference rules;
* optional enablement or row-local conditions;
* severity or mode policy.

Information produced:

* validation findings;
* optional failure when configured as blocking.

This form covers common relational checks such as duplicate primary keys,
duplicate unique tuples, dangling references, and missing referenced rows.

=== Graph Validation

Graph validation applies the same integrity idea to node/edge-shaped data.

Information consumed:

* node frames and node key columns;
* edge frames, source/target columns, and referenced node kinds;
* edge uniqueness rules;
* active checks and severity or mode policy.

Information produced:

* findings for missing endpoints, unknown node references, or duplicate edge
  identities;
* optional failure when configured as blocking.

Graph validation is still validation. It does not introduce a separate graph
model unless a future use case needs one.

== Relationship To Legend Blocks

Legend Blocks and validation must stay separate.

Legend Blocks define or resolve controlled value sets and their human-readable
meaning. A legend may provide:

* compact codes;
* long-form values;
* display labels;
* optional visible support surfaces for workbook users.

Validation may consume a legend as one possible source of allowed values. For
example, a coded matrix cell can be validated against the entries of a legend.

That does not make Legend Blocks a validation subsystem. The legend owns the
controlled vocabulary and its display meaning. Validation owns the rule that a
particular workbook location or frame column must use values from that
vocabulary.

Other validation sources remain possible: configured static lists, reference
frames, uniqueness declarations, external checks, numeric predicates, or future
domain-specific rules. Legend-backed validation is one important special case,
not the whole concept.

== Placement In The Roundtrip

Validation can run at several points:

* before projection, to ensure the source data can be rendered safely;
* during workbook rendering, to attach editing constraints;
* after reimport, to check user edits before reverse transformations;
* after recomposition, to ensure returned frames satisfy structural rules;
* before publishing, to block invalid generated artifacts.

The exact placement is workflow-specific. The Functional Model requirement is
that validation intent is explicit enough for users to know which rules are
checked, when findings are reported, and which findings block the pipeline.

== Relationship To Other Concepts

Validation often uses information produced by other concepts:

* FK Helper and reference concepts identify expected reference relationships;
* XRef Crosstable identifies matrix value areas and relation keys;
* Cell Codec identifies compact values that may need decoding and allowed-value
  checks;
* Legend Blocks provide controlled values for coded cells;
* Workbook View Boundary identifies where a column is rendered when workbook
  constraints need a sheet target.

Those concepts retain their own meaning. Validation records and applies rules
about whether data conforms to the expected shape or value domain.

== Non-Goals

This concept does not:

* make every validation rule part of MVP;
* define a universal validation language;
* replace Legend Blocks;
* replace reference or FK modeling;
* infer constraints from arbitrary spreadsheet formatting;
* require validation to be implemented only as workbook dropdowns;
* require external API checks, numeric predicates, or complex domain rules
  before a concrete workflow asks for them.

== MVP And Future Scope

The current essential validation concepts are:

* uniqueness and primary-key style checks;
* dangling-reference / foreign-key checks;
* workbook-facing allowed-value constraints;
* legend-backed allowed-value checks where coded values are used;
* graph endpoint and edge-identity checks for graph-shaped data.

Future validation concepts may add richer numeric predicates, external-source
checks, conditional rules, or domain-specific rule packs. Those should extend
the validation use case without turning Legend Blocks, Workbook View, or other
neighboring concepts into validation subsystems.
