= Future Concept: Aggregate Planning Reconciliation

Aggregate Planning Reconciliation is a future functional pattern for workbook
surfaces where users edit aggregated planning values and expect the system to
materialize the difference as explicit follow-up data.

This is not MVP. It is recorded so the Functional Model can distinguish this
future pattern from XRef Crosstable, plain `pivot_frame` display views, and
ordinary derived frame projections.

== User Problem

Planning work often starts from detail rows but is reviewed and adjusted at a
coarser level:

* capacity by stream and month;
* effort by product area and quarter;
* budget by cost block and planning cycle;
* demand by market, team, or time period.

A user may want to edit the aggregate value directly. If `API / 2026-07`
changes from `25` to `35`, that edit does not say which existing task, resource,
or source row should change. The transparent interpretation is to compare the
edited aggregate with the baseline and materialize the difference as an
explicit delta, demand, adjustment, or follow-up row.

The source detail rows remain unchanged unless a later allocation or
distribution workflow explicitly changes them.

== Functional Intent

Aggregate Planning Reconciliation has two sides:

* build an aggregate planning projection from configured source data;
* reconcile an edited aggregate projection against a known baseline.

The second side must not silently mutate contributing detail rows. It creates
explicit reconciliation output that a later workflow can allocate, distribute,
approve, or otherwise resolve.

== Step: Build Aggregate Planning Projection

The forward step derives a workbook-facing aggregate planning view from source
detail frames or existing planning frames.

Information consumed:

* source detail frame or planning frame;
* grouping dimensions such as stream, period, team, product, market, or cost
  center;
* value field to aggregate, such as planned effort, capacity, budget, or demand;
* aggregation function, initially most likely sum;
* optional configured axis order or scope declaration;
* lifecycle intent marking the result as an aggregate planning projection.

Information produced:

* workbook-facing aggregate planning frame;
* baseline values needed for later reconciliation;
* trace intent for the source frame, grouping dimensions, value field, and
  aggregation policy.

Semantic Model Reference:

* workbook view;
* frame lifecycle / projection;
* future aggregate reconciliation metadata, if this pattern is implemented.

`pivot_frame` may be a technical view-building primitive for this step. It is
not the whole functional concept.

== Step: Reconcile Edited Aggregate Projection

The reverse step compares the edited aggregate planning frame against the
baseline and materializes the configured interpretation of the difference.

Information consumed:

* edited aggregate planning frame;
* baseline aggregate values from the forward step or from a deterministic
  recomputation using the same source and configuration;
* grouping dimensions, value field, and aggregation function;
* reconciliation policy;
* target frame for generated demand, adjustment, or follow-up rows;
* optional identity-generation policy for new rows.

Information produced:

* explicit delta, demand, adjustment, or follow-up planning rows;
* diagnostics for invalid edits, missing addresses, ambiguous axes, or
  unsupported aggregation functions;
* trace intent linking generated rows to aggregate addresses and edited values.

The reconciliation result is deterministic and user-explainable, but it is not
a strict mathematical inverse of aggregation. More than one detail-state could
explain the same edited aggregate. The configured reconciliation policy chooses
one explicit representation, for example: compute the difference to baseline
and append a new unallocated demand row for the aggregate address.

== Relationship To XRef Crosstable

Aggregate Planning Reconciliation should not be folded into XRef Crosstable.

[cols="1,2,2", options="header"]
|===
|Aspect
|XRef Crosstable
|Aggregate Planning Reconciliation

|Cell meaning
|One addressed relation tuple
|Aggregate over contributing rows

|Edit effect
|Set, remove, or update a relation entry
|Compute delta against baseline

|Reverse path
|Recompose matrix into the target relation
|Materialize adjustment, demand, or follow-up rows

|Blank semantics
|Configured matrix blank/delete/no-update semantics
|Requires an explicit reconciliation policy

|Detail rows
|Usually not contributing aggregate details
|Remain unchanged unless a later workflow edits them
|===

The visible workbook shape may be matrix-like in both cases. The user meaning
is different.

== Relationship To Plain Pivot Views

Plain `pivot_frame` is a bounded projection or reshaping primitive. It can
produce a wide, human-readable display frame from long-form data and may
aggregate duplicate display cells with a configured display policy.

Aggregate Planning Reconciliation is larger than that. It owns edited aggregate
targets, baseline comparison, and explicit delta/materialization semantics.

== Relationship To Derived Frame Views

Derived Frame Views describe projection steps such as `join_frames` and
`split_by_discriminator`. Those views may be editable when a workflow declares
ownership and precedence.

Aggregate Planning Reconciliation is stricter: the edited aggregate does not
overwrite the contributing source rows. The return path materializes a separate
follow-up artifact unless a later workflow explicitly performs allocation or
distribution.

== Non-Goals

This future concept does not:

* implement resource allocation;
* define a scheduling optimizer;
* replace XRef Crosstable;
* turn `pivot_frame` into an editable aggregate workflow by itself;
* infer axes, periods, or grouping dimensions from workbook layout;
* introduce live spreadsheet UI logic;
* imply production, registry, or pipeline changes as part of the Functional
  Model note.

== MVP Disposition

Aggregate Planning Reconciliation is not MVP.

It is retained as a future functional pattern for planning, allocation, or
aggregate-edit workflows where edited workbook aggregates must be reconciled
against a baseline without silently rewriting contributing detail rows.
