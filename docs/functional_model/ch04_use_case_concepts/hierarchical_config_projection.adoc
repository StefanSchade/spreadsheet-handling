= Use Case: Hierarchical Config Projection

Hierarchical Config Projection describes the controlled change between
tabular frame form and hierarchical, configuration-like structure.

The concept is independent of a concrete file format. A hierarchical
configuration tree can later be encoded in different carriers. Likewise, a
file format that is often used for hierarchical configuration can also carry
flat tabular data. The Functional Model therefore treats the hierarchy itself
as the relevant user-facing structure, not the serialization format.

== User Problem

Spreadsheet editing and frame processing work well with tabular data:

* rows identify repeated things;
* columns identify comparable attributes;
* joins, validations, helper columns, and crosstables can operate over explicit
  frame columns.

Many application configuration surfaces, however, are not tabular in their
natural authored form. They often use:

* nested maps;
* ordered lists;
* grouped lists below a key;
* optional fields that disappear when empty;
* stable path-like field names;
* deterministic output order for reviewable diffs.

Users may therefore want to maintain configuration data in spreadsheet-friendly
tables, while downstream systems expect a hierarchical configuration shape.

== Functional Intent

The core functional pattern is:

[source,text]
----
tabular frames
-> configured hierarchical projection
-> hierarchical configuration structure
----

The inverse or companion pattern may also be useful:

[source,text]
----
hierarchical configuration structure
-> configured flattening / extraction
-> tabular frames
----

The important requirement is explicit mapping. The system should not guess a
hierarchy from display labels, visual indentation, column name similarity, or
the file format used to store the result.

== Step: Build Hierarchical Structure From Frames

The forward transformation consumes one or more prepared frames and a
declarative mapping that states how rows and columns become a hierarchical
structure.

Information consumed:

* source frame or frames;
* key columns that identify mapping entries or groups;
* value-column mappings into hierarchical paths;
* list or mapping root intent;
* ordering policy for keys, rows, and list items;
* explicit omission policy for empty optional values;
* required-field policy for values that must not be empty.

Information produced:

* hierarchical configuration structure;
* optional report frame describing generated structures or targets;
* diagnostics for missing columns, missing required values, duplicate targets,
  or unsafe paths.

Semantic Model Reference:

* frame projection intent;
* generated artifact/report intent;
* future hierarchical structure mapping, if this concept is promoted into the
  Semantic Model.

== Step: Flatten Or Extract Hierarchical Structure To Frames

The reverse or companion transformation takes a known hierarchical structure
and extracts configured tabular frames from it.

Information consumed:

* hierarchical source structure;
* configured paths, group keys, list traversal rules, and output frame names;
* required and optional field declarations;
* identity policy for repeated nested entries.

Information produced:

* tabular frames suitable for editing, validation, joins, crosstables, or
  workbook projection;
* diagnostics for missing required paths, duplicate identities, unsupported
  mixed shapes, or ambiguous nesting.

This direction should also be explicit. A nested source should not be flattened
by accidental path conventions if the workflow depends on stable identity.

== Relationship To Publishing

Hierarchical Config Projection may be followed by a publishing step that writes
the resulting structure to files and records a manifest. That publishing step
is not the core concept.

The Functional Model separates:

* transformation of information shape: frames to hierarchical structure, or
  hierarchical structure to frames;
* artifact publishing: writing the structure to one or more target files,
  computing checksums, or recording generated artifact reports.

This separation prevents the hierarchy concept from being coupled to one
adapter or serialization format.

== Relationship To Workbook Roundtrip

This concept is useful when the spreadsheet is the editing surface for
configuration-like data. Users can edit clear tabular frames while the workflow
retains the ability to produce the hierarchical shape expected by downstream
systems.

The tabular frames remain the editable surface. The hierarchy is a configured
projection target or source shape. Validations should run on whichever side is
most appropriate for the rule:

* frame-level uniqueness and reference checks before projection;
* required-path and missing-value checks during hierarchical projection;
* artifact-level checks after publishing, if a workflow publishes files.

== Non-Goals

This concept does not:

* require a particular serialization format;
* replace ordinary flat frame persistence;
* infer hierarchy from spreadsheet formatting or labels;
* define a general templating language;
* allow arbitrary expressions as mapping logic;
* hide domain-specific extraction rules inside a generic publisher;
* require generated artifacts to be part of every workbook roundtrip.

== Classification Notes

Hierarchical Config Projection is a domain transformation concept when it maps
frames to another structured representation, or maps a structured
configuration representation back to frames.

Concrete writers, manifests, checksums, and file paths are publishing concerns
that can use the projected structure, but should not define the functional
concept.
