== Bridge and Workflow

The project is not, at its core, a file-format converter. Converting
JSON to YAML, XML, CSV, XLSX, or ODS and back again describes only a
surface of the work. The center of the project is the controlled
transition of the *same information* between two working forms.

The two working forms are:

* A *machine-near data base*: the structured, long-term, normalized form
  in which information lives between work sessions and travels into
  downstream technical processes. JSON, YAML, XML, CSV directories,
  filesystems, and version-controlled repositories are typical carriers.
* A *domain-near editing view*: a human-friendly working form in which
  someone can read, compare, enrich, and decide. The spreadsheet
  projection is the project's central editing surface, but the project
  treats it as an editing projection, not as the model.

The project's contribution is a controlled crossing between those two
forms. A canonical record is projected into an editing view, a person
edits the view, and the edited view is brought back into the canonical
record without losing structure or determinism:

[source,text]
----
machine-near data base
  -> domain-near editing view
  -> human editing
  -> controlled return
  -> machine-near data base
----

The crossing is what makes the project useful. It is also what creates
the work the rest of the documentation cares about: when an editing view
is richer than the canonical form (helper columns, crosstables,
language-specific projections), the return path has to know how to
reconcile the difference.

The crossing is also broader than a single bidirectional roundtrip. The
following process kinds belong to the same problem space without being
the main case:

* extractions into other views or target structures;
* validations that emit warnings or block the pipeline;
* CI/CD checks;
* preparation of input for downstream tools;
* one-way projections where no return is intended.

These remain attached to the same idea: the project moves information
between machine-near and domain-near forms, or prepares it for the next
process, under explicit rules.

=== Working principles

The German warm-storage concept names a small number of principles that
the project relies on without trying to enforce them as a taxonomy.

* *Heuristics are a comfort, not a foundation.* A heuristic may turn an
  inferred guess into an explicit working assumption. Later steps work
  against the explicit assumption rather than re-guessing on each pass.
* *Collaboration is a requirement.* The machine-near data base lives in
  shared storage. Information that downstream users or processes need
  cannot remain in purely local working state. Information that may
  legitimately be local must be marked as such.
* *The data model itself can change across a roundtrip.* A table can
  become a crosstable on the editing side and a new tuple list on the
  return path. The project treats this as normal, not as an exception.
* *Editing is controlled, not arbitrary.* The project does not promise
  to reconstruct every possible manual change. It promises to support a
  named set of editing patterns and to reject or flag the rest.
* *Downstream uses are downstream.* Generating documentation, AsciiDoc,
  LLM context, or other artifacts from the canonical record is a
  legitimate follow-up process. It is not part of the project's core.

The Semantic Model is the place where these principles are turned into
contract statements where they affect deterministic behavior. The
purpose of this chapter is only to make the principles visible as
problem-space framing, not to encode them.

=== A typical roundtrip, at a glance

A prototypical roundtrip can be sketched as the following sequence. The
exact step set in any specific pipeline is configured declaratively and
may differ from this sketch.

[source,text]
----
1. read the machine-near data base
2. assemble the working tables
3. add or derive structural assumptions
4. produce the domain-near projection
5. render the editing view
6. allow human editing
7. read the edited view back in
8. interpret the changes
9. reverse the projections
10. perform checks and conflict handling
11. write durable results back into the data base
12. discard or separately handle local / transient working information
13. emit the updated machine-near data base
----

This sketch is intentionally not a technical design. It exists so that
later technical and contract decisions can be discussed against a shared
high-level flow. Whenever a step in this sketch produces information
that must outlive the run, the Semantic Model is where the contract for
that information lives.
