= Workbook Presentation Meta Roundtrip

Workbook presentation metadata records user-facing layout choices made on the
spreadsheet editing surface and carries them into the next roundtrip when doing
so is useful.

This is not canonical business data. Losing it must not change source facts,
relations, identifiers, or validation meaning. It is still a Functional Model
concern because it changes whether repeated workbook editing is practical: users
should not have to reapply the same column widths or alignment choices after
every structured-data refresh.

== Functional Purpose

A spreadsheet roundtrip should preserve presentation choices that the user made
in the workbook when those choices can be observed and represented as meta.

Current presentation metadata includes:

* column width;
* horizontal alignment;
* vertical alignment;
* text orientation.

Likely future candidates include:

* cell fill or color;
* word wrap;
* border or line styling.

The purpose is continuity of the editing surface, not enrichment of the
domain data. A workbook that loses presentation metadata remains valid; it is
only less convenient to edit.

== Roundtrip Behavior

On the read path from a spreadsheet carrier, presentation values observed in the
workbook may be written into working meta. On the following structured-data
egress, the persistable subset of that meta may be written beside the structured
data. On the next forward pipeline run, the same meta can be used to recreate a
similar workbook surface.

This means a user can adjust the workbook once, for example by setting useful
column widths or rotating a narrow header label, and the pipeline can preserve
that editing surface through later roundtrips.

The behavior is best-effort and boundary-sensitive:

* if matching meta is present and trusted for the current pipeline run, it may
  be reused;
* if the structured data was refreshed from Git or another external boundary,
  matching meta may be absent or stale;
* if meta is absent, the data remains usable and the workbook can still be
  rendered, but presentation choices may need to be recreated or bootstrapped.

== Bootstrap And Defaults

A fresh clone or boundary refresh often has structured data without matching
presentation meta. The pipeline author therefore needs a way to provide
reasonable default presentation metadata through bootstrap.

Bootstrap should be explicit and configurable. For each presentation datum, the
pipeline design should be able to choose whether the bootstrap declaration:

* fills only when the current meta value is absent; or
* overwrites the current meta value.

The first mode preserves user-observed workbook choices when they already
exist. The second mode lets a project enforce a presentation preset when that
is more important than retaining the incoming carrier's layout.

This mirrors the general meta bootstrap rule: bootstrap attaches internal meta
to working data; it does not turn presentation metadata into canonical payload.

== Precedence Expectations

When the workflow treats an edited workbook as the latest editing carrier,
presentation values observed from that workbook should normally take precedence
over older stored meta. That reflects the user's latest visible editing surface.

Bootstrap should not silently overwrite observed or persisted presentation meta
unless the pipeline explicitly asks it to do so. The meaningful choice belongs
to pipeline design:

* preserve user-authored layout where possible;
* fill missing layout for fresh or reset inputs;
* enforce a project preset when the workflow requires a stable rendered shape.

== Relation To Other Concepts

The workbook view boundary decides which frames and sheets form the workbook
surface. Presentation metadata describes layout properties of that surface.

The meta lifecycle framework owns the carrier behavior: workbook-embedded meta,
working frame meta, sidecar persistence, bootstrap, and boundary pruning.

Presentation metadata should not be modeled as a domain transformation. It does
not split, join, enrich, or reconcile frames. It is a lifecycle attribute of the
editing surface.

== Non-Goals

This concept does not require:

* preserving arbitrary spreadsheet styling;
* storing business semantics only in presentation metadata;
* defining collaboration or merge behavior for layout changes;
* treating Git-persisted meta as fully reliable after external edits;
* deciding exact registry field names, step names, or backend APIs.

== Model Placement

This concept belongs in the problem-space overview because it describes a
cross-boundary lifecycle requirement. It is intentionally separate from the
transformation use-case concepts in chapter 4: the requirement is to preserve
and bootstrap workbook presentation metadata, not to introduce a new
frame-to-frame transformation family.
