= Workbook View Boundary

The workbook view boundary is the point where already prepared frames become a
spreadsheet editing surface, and where an edited spreadsheet is interpreted
back as named frames.

This is not a domain transformation in the same sense as a frame-to-frame
operation. A domain transformation changes, reshapes, enriches, or reconstructs
frames inside the configured data pipeline. The workbook view boundary declares
how prepared frames cross the spreadsheet boundary.

== Functional Purpose

The workbook view boundary answers questions such as:

* which prepared frames should appear in the workbook;
* which visible sheet name should represent each frame;
* in which order sheets should be shown;
* which sheets or columns are intended to be editable or protected;
* which render-near options should be available to the spreadsheet renderer;
* how visible sheet names are associated with logical frame names on the return
  path.

These are workbook-surface decisions. They do not create a new business
relation, infer new domain data, or replace the transformations that prepare
the frames.

== Input Shape

The view declaration may come from explicit workflow configuration or from
metadata that was attached before rendering, for example through a bootstrap
step.

Functionally, both cases mean the same thing: the workflow supplies a
declaration for the workbook-facing surface. The Functional Model does not
require a particular implementation mechanism for supplying that declaration.

== Forward Path

On the forward path, final domain cleanup first resolves explicit drop/keep
commands carrier-neutrally. The workbook view boundary then consumes the
remaining prepared frames and an optional explicit view declaration.

Without `workbook_view.sheets`, every remaining logical frame becomes a sheet.
When `workbook_view.sheets` exists, it is authoritative for frame selection,
sheet names, and order. No role, provenance, canonicality, or frame-name rule
adds or removes a frame.

The prepared frames must already exist. Filtering, joining, pivoting,
crosstable expansion, helper-column generation, legend derivation, and other
data-preparation work belongs before this boundary.

The boundary may carry presentation directives, such as sheet naming, ordering,
visibility, protection, or render placement. Those directives describe how the
editing surface should appear. They should not become a second place for
domain transformation logic.

Presentation values that are observed from or reapplied to the workbook, such
as column widths, alignment, or text orientation, have their own lifecycle
concern. That concern is described in the workbook presentation meta roundtrip
concept rather than in the workbook view boundary itself.

== Return Path

On the return path, the workbook view boundary provides the interpretation
needed to map the edited workbook surface back to logical frame names.

This matters when visible sheet names differ from internal frame names. A user
may see a friendly worksheet name, while the pipeline still needs to recover
the stable frame identity used by later reverse transformations.

The boundary does not decide how edited data is semantically reconciled. Once
the edited workbook has been mapped back to frames, the relevant reverse
transformation or validation step owns the interpretation of changes.

== Interaction With Other Concepts

Other functional concepts may need the workbook view boundary because their
output has a visible workbook shape.

Examples:

* a derived frame view may need a friendly sheet name;
* a crosstable may need to appear as the primary editable sheet for a relation;
* legend blocks may need render placement near the table or matrix they
  explain;
* helper columns may need protection or visible/editable intent;
* validation constraints may need to target the sheet where a column is
  rendered.

In those cases the other concept owns the meaning of the data. The workbook
view boundary owns the workbook-surface routing and presentation declaration.

== Non-Goals

The workbook view boundary does not:

* define a new canonical data model;
* perform joins, pivots, crosstable expansion, or row-producing operations;
* infer workbook layout from arbitrary spreadsheets;
* infer frame inclusion or omission from lifecycle or provenance metadata;
* decide conflict resolution for edited data;
* replace domain transformations that prepare frames before rendering;
* require a particular implementation step name or registry mechanism.

== Model Placement

This concept belongs in the problem-space overview rather than in the
transformation family chapters.

It is still part of the Functional Model because users care about the workbook
surface: sheet names, order, editability, render directives, and return-path
identity all affect whether a roundtrip is understandable and deterministic.
It is separated from transformation concepts because its purpose is boundary
composition, not domain data transformation.
