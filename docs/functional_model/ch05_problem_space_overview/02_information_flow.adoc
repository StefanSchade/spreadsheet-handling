== Information Flow at Process Boundaries

A roundtrip is not only a sequence of steps. It is also a sequence of
*moments at which information changes role*. The same value can, across
one run, be a structural assumption for an early step, working state for
a later step, a presentation aid for the editing view, an input to the
return path, and -- after a successful return -- something to discard.

This chapter sketches those role changes as a problem-space framing. It
does not name semantic sets or define a classification.
The Semantic Model is where the classification work lives (overlapping
semantic sets, lifecycle events, source classes, precedence). The role
of this chapter is to give the Functional Model a place to explain
*why* the contract chapters care about these distinctions.

=== Functional questions at each process boundary

The German archived concept (`docs/cold_storage/concepts/problemraum_konzept/`,
chapter 03) phrases the orientation as a small set of recurring
questions. The English form below preserves the spirit of those
questions without making them load-bearing.

For any piece of information that crosses a process step, it is worth
being able to answer in prose:

* *Who needs it?* The same user later; another user; a downstream
  process; only one current pipeline step.
* *What happens if it is lost?* A subject-matter loss; a wrong return
  path; a loss of working comfort only; no harm because it is
  reconstructible.
* *When is it needed again?* After a fresh checkout of the data base;
  on the next local run; only during this specific return path; only
  during projection, rendering, or write.
* *Where can it be reconstructed from?* The machine-near data base; the
  pipeline configuration; the edited view; only from earlier working
  information; nowhere.
* *Is it shared or local?* Project-wide; user-specific; machine-specific;
  internal to one run.
* *What functional role does it play?* Part of the data base; a
  structural assumption; a decision input for a transformation; an aid
  for editing or display; runtime working memory; a check finding or
  hand-off artifact.

These are deliberately questions, not categories. They orient a reader
who is about to make a decision about where a particular piece of
information belongs. The corresponding *contract* answer -- whether the
information is part of the persistable contract, transient, derived,
runtime-consumed, or roundtrip-relevant -- is the Semantic Model's
responsibility, not this chapter's.

=== Helper columns, as a worked example

Helper columns illustrate why these questions matter without forming a
new taxonomy.

A reference in the machine-near data base, for example
`places.region_id -> regions.id`, is hard to edit by ID alone. The
forward path can therefore add a helper column such as `_regions_name`
to the editing view. The column is derived from existing data; it is
not a new piece of subject-matter information.

For the return path to remain deterministic, the project needs to know
that `_regions_name` is a helper column rather than a normal payload
column. Whether that fact is stored in `_meta`, in a view definition, in
local working state, or rebuilt from configuration at the start of the
next run is a *contract question* the Semantic Model owns. The
*functional* observation is only that the information is needed, that
losing it has consequences (helpers may re-accumulate, payload columns
may multiply across runs), and that the project should make those
consequences visible.

The Semantic Model's FK-helper pilot matrix and the lifecycle chapter
describe how the project actually pins this contract. This chapter does
not duplicate that work.

=== Why this matters for the model boundary

If new pieces of information start being treated by the system, they
will arrive with the same questions: who needs them, what happens if
they are lost, where can they be reconstructed from. Answering those
questions in problem-space terms first -- before introducing a new
contract term, a new semantic set, or a new registry entry -- is the
practical brake on recursive vocabulary growth.

The Functional Model owns the *questions*. The Semantic Model owns the
*contract answers* and may, where the contract is stable, route them
into guards or registry rules. The Technical Model owns the *mechanism*
that realizes those answers in code, persistence, and rendering.
