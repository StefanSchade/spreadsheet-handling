[NOTE]
====
*Reader orientation.*

The Functional Model explains the problem space: working forms,
motivation, usage scenarios, process boundaries, and worked examples.
It is the home of *why* the project does what it does. The Semantic
Model (`docs/semantic_model/`) defines the contract -- vocabulary,
interpretation rules, precedence, lifecycle, guard candidates -- and
the Technical Model (`docs/technical_model/`) describes the
implementation structures.

Preserved input under
`docs/warm_storage/problemraum_konzept/` is the working German concept
that motivated the model-boundary recut under
`FTR-FUNCTIONAL-SEMANTIC-MODEL-SPLIT-P5`. It stays in warm storage
verbatim; chapter 5 absorbs its framing into English without
replacing it.
