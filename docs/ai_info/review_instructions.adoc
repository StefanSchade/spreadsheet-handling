== Review Agent Instructions

This is the mandatory AI-info entry point for review tasks.

For a review task, read this file before starting the review. You may skip the
full general AI-info loading path unless this file, the review prompt, or an
emerging finding requires additional context.

=== Review scope

Reviewers do not silently become implementers. If the prompt says the review is
read-only, do not modify runtime code, tests, registries, backlog items, release
notes, or normal documentation unless the prompt explicitly asks for follow-up
edits.

When a reviewer finds a problem, the default action is to record a finding in
the review artifact, not to fix it.

Assume review artifacts may be committed to the public repository. Avoid
private organization, customer, internal-product, or named-individual context
unless the prompt explicitly says the review is private.

=== Review artifacts

FTR implementation reviews belong in a sibling `_review.adoc` file next to the
FTR under review unless a more specific repository convention is provided, for
example:

[source,text]
----
docs/backlog/FTR-<ID>_review.adoc
docs/cold_storage/backlog/ftrs_done/FTR-<ID>_review.adoc
----

Dedicated code, readiness, architecture, or release reviews use the explicit
review directory or prompt location when one is provided, for example:

[source,text]
----
docs/warm_storage/global_reviews/review_NNN/
----

Trivial edits, such as typo fixes or single-line documentation tweaks, do not
need review artifacts unless the prompt requests one.

The FTR review-note location rules and follow-up update rules are summarized in
link:git_and_workflow.adoc[git_and_workflow.adoc].

=== Required inputs by review type

For an FTR implementation review, load at minimum:

* the FTR under review;
* the diff;
* relevant tests;
* an existing sibling review artifact, if any;
* relevant FTR workflow or checklist rules;
* the release-notes checkpoint when the change is public or release-relevant.

For a broad readiness, architecture, or release review, load at minimum:

* the specific review prompt first;
* stated baseline reviews, if any;
* only the additional AI-info files needed to answer the review question.

For a follow-up review after implementation fixes, load at minimum:

* the original review artifact;
* the follow-up diff;
* validation commands and results;
* enough context to say whether previous findings are closed, downgraded, or
  still open.

=== Findings and prioritization

Review findings should include:

* observation;
* evidence;
* concrete risk if ignored;
* recommended action;
* priority or disposition.

For larger reviews, prefer a disposition pattern that distinguishes:

* blockers;
* before-release items;
* before-low-touch or maintainability items;
* after-release improvements;
* watch-only risks;
* non-findings.

Small reviews may use a lighter format, but findings still need enough evidence
and risk detail to be actionable.

=== Adapter read-path safety checks

When reviewing spreadsheet adapter read paths, treat repeated carrier
structures as a safety concern anywhere they are interpreted, not only in the
main table payload parser.

ODS in particular can represent repeated rows, columns, and cells through
carrier attributes such as `table:number-rows-repeated` and
`table:number-columns-repeated`. Those repeats may appear in presentation
metadata carriers as well as payload cells, for example styled repeated
columns or repeated rotated cells. A parser that avoids expanding repeated
empty payload cells can still reintroduce unbounded materialization if a
separate metadata extractor expands every implied presentation address.

For any new adapter read-path that interprets repeated carrier structures:

* avoid materializing every implied row, column, cell, or metadata entry unless
  the expansion is bounded by the parsed workbook extent and configured limits;
* reuse `ParserLimits` where available, or provide an equivalent bounded
  extraction policy for the adapter path;
* clip display-metadata extraction to the parsed content extent when
  presentation-only filler outside real content should not become canonical
  metadata;
* add pathological repeated-carrier tests for the specific extractor, not only
  for the table payload parser; and
* add carrier-level tests when parser and renderer could otherwise be
  self-consistent while using the wrong carrier shape.

This lesson comes from the `FTR-TEXT-ORIENTATION-ROUNDTRIP-P5` review: ODS
text orientation and column-width metadata needed the same repeated-structure
safety model as table parsing, and ODS rotation needed XML carrier assertions
to prove it was written to the ODF-defined element rather than merely
round-tripping through the project's own parser.

=== Release-notes checkpoint

Release notes are owned by the implementing change.

If an FTR or implementation changes user-facing behavior, public CLI / YAML /
configuration surface, package / import / plugin surface, demo behavior,
documentation entry points, migration semantics, or release-relevant behavior,
the implementer must update `docs/release_notes/release_notes.adoc` or state in
the review artifact why no release note is needed.
This includes adding, changing, or removing *Known limitations* when the FTR
changes a limitation's validity, scope, or release relevance.

Reviewers must verify that checkpoint, including *Known limitations*. During a
read-only review, do not silently take ownership of release notes. If release
notes are missing for a public or release-relevant change, record a finding or
request a follow-up.

Reviewers may propose release-note wording, but should edit release notes only
when the review task explicitly includes follow-up edits.

*Placement rule* (per `FTR-POST-RELEASE-LIFECYCLE-AND-RESET-P5`):
release-note bullets added by an FTR after the most recent release tag
belong under the *active unreleased section*
(`== 0.(X+1) -- planned`), never under a frozen `(released)` section. A
bullet under a frozen section is a release-relevant defect.

See the release-notes checkpoint in link:git_and_workflow.adoc[git_and_workflow.adoc]
and the matching section in
link:../developer_guide/ch08_git_and_workflow/01_git_and_workflow.adoc[the developer-guide workflow]
for the canonical implementation-side rule.
