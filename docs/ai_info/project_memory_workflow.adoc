== Project Memory: Handoff and Workflow

=== What project_memory is

`project_memory` is the repo-local project state model: curated steering memory,
activity trail, and review traceability kept in a spreadsheet-friendly flat JSON
structure. It is not runtime `_meta`. It is not user-facing documentation.
It exists to give human maintainers and AI agents a fast entry point into
current project concerns, open work, and review history without reading raw
Git history or individual review documents.

=== Entry point: generated context report

Before any broad project-memory work, run:

[source,shell]
----
make memory-context
----

Then read:

----
docs_generated/project_memory/current_context.adoc
----

This generated report is the best first-entry project-state document. It is
rebuilt on demand and is always derived from canonical data. Treat it as
read-only.

=== What the report contains

Depending on current data, the report includes:

* *Concerns* — long-lived strategic steering threads (CONC-*) with linked events
* *Concern events* — the activity trail: reviews, findings, decisions, releases, implementation waves
* *Current findings* — open or partial findings filtered for current relevance
* *Active FTRs and blockers* — non-done FTRs with dependency context
* *Review sets and individual reviews* — review waves grouped by storage topology
* *Event–FTR links* — concern events linked to FTRs via shared commit references
* *Diagnostics* — unresolved references between tables (warnings only)

=== How to use the report

* Read `current_context.adoc` to understand the current concern posture and recent project evolution before broad project-memory work.
* Follow `source_id` references back into canonical JSON, `docs/cold_storage/reviews/`, FTR backlog, or findings for more detail.
* Treat generated reports as read-only. Do not edit `docs_generated/` by hand.
* Edit canonical JSON or source docs only when explicitly asked.
* Run `make memory-context` again after making changes to refresh the report.

=== Data layers

----
project_memory/canonical/
  Curated source of truth. These files are the data.

project_memory/canonical/ext_*.json
  Generated staged candidate tables. Visible in the ODS workbook
  as candidate material for review. Not curated truth.
  Gitignored and overwriteable.

project_memory/extracted/
  Raw generated extraction output from ADOC artifacts.
  Not curated. Must not be promoted into canonical without manual inspection.

project_memory/derived/
  Generated query results (current_findings, ftr_blockers, release_note_candidates,
  event_ftr_links, …). Rebuilt by make memory-query. Gitignored.

docs_generated/project_memory/
  Generated AsciiDoc reports. Rebuilt by make memory-context. Gitignored.
----

=== ODS workflow (brief)

`project_memory.ods` is an editable spreadsheet view over canonical data.
`ext_*` sheets are staging/candidate material — not authoritative.
Manual curation moves selected rows from `ext_*` sheets into the real canonical sheets.
Generated ODS and all `ext_*` tables are rebuildable artifacts and must not be
treated as hand-authored documents.

=== Review-set navigation

The project-memory model uses an explicit review-set layer to anchor
review-wave references:

----
Concern / Event
  -> Review Set  (review_sets.json, REVSET-* ids)
     -> Reviews  (reviews.json, REV-* ids)
        -> Review documents  (evidence_path)
----

Review sets mirror the storage topology under `docs/cold_storage/reviews/`.
Concern events that describe the impact of a whole review wave use
`source_type=review_set` and `source_id=REVSET-*` to anchor to the set
rather than an individual review file. This makes navigation deterministic:
a REVSET-* id always resolves to a known directory.

Do not create events that reference individual review documents when the
intent is to describe the steering impact of a whole review wave.

=== Project Memory Maintenance Cycle

==== Purpose

Project Memory stays useful only if canonical data reflects *deliberate, curated
understanding* — not a mechanical mirror of every commit or extraction run.
The maintenance cycle creates a bounded rhythm: extract evidence → inspect
diagnostic reports → curate selectively → return to feature work.

==== When to run

Run the maintenance cycle:

* after completing a meaningful work slice (FTR, review wave, or release)
* before asking an AI agent for a broad project review or context orientation
* before release-note or project-history consolidation work
* when the health report indicates drift that is relevant to current concerns

Do not run after every commit. Do not let maintenance become an ongoing meta-work stream.

==== Commands

[source,shell]
----
make memory-extract         # Extract evidence from ADOC artifacts and Git history
make memory-query           # Rebuild derived query views (project_memory/derived/)
make memory-health-report   # Generate the health/drift diagnostic report
make memory-context         # Rebuild the full generated context report (implies memory-query)
----

`make memory-context` is the highest-level target — it rebuilds query views and the
generated AsciiDoc context report in one invocation. Use it when you need a complete
refresh before a broad review or orientation task.

`make memory-health-report` runs the diagnostic independently; use it when you want
a drift/gap check without regenerating the full context report.

==== Interpreting outputs

[cols="1,3"]
|===
| Output | What it means

| `project_memory/extracted/commit_signals.json`
| Git commit signals extracted from repo history.
  Evidence only — not canonical truth.
  May include noise, ambiguous matches, or commits with no tracked concern.

| `project_memory/extracted/finding_candidates.json`, `ftr_candidates.json`, `review_candidates.json`
| Conservative extraction candidates from ADOC artifacts.
  Not curated. Must not be promoted to canonical without manual inspection.

| `project_memory/derived/memory_health_report.json` / `.adoc`
| Diagnostic: highlights unknown artifact IDs, ambiguous commit links,
  recent unlinked commits, activity events without commit_refs, and FTRs
  without commit evidence.
  Shows likely drift — does not fix drift, does not update canonical data.

| `project_memory/derived/finding_assessment_heatmap.json`
| Derived join of current findings onto the normalized `impact_0_5` scale.
  Shows mapped vs. unmapped findings. Read-only derived output.

| `docs_generated/project_memory/current_context.adoc`
| The primary agent-facing context report. Read-only. Always rebuilt from canonical data.
|===

==== Promotion rule

*Canonical data is curated, not auto-generated.*

Extracted and derived outputs are evidence and interpretation tools — they do not
flow automatically into `project_memory/canonical/`. Before any canonical change:

* Inspect the health report to understand what the gaps represent.
* Decide whether each item is signal (worth tracking) or noise (routine commits, context without impact).
* Edit only the specific canonical JSON table(s) that need the update.
* Cherry-pick deliberate deltas; do not batch-promote all extracted candidates.
* Commit curated changes separately from feature work.

==== Non-goals

* No automatic canonical promotion from extracted outputs.
* No full Git-history canonicalization — canonical memory is not a commit log.
* No automatic event creation from commits; every concern event must have a human-assigned meaning.
* No urgency normalization — the assessment scale covers impact only.
* No snapshot tooling changes during a maintenance pass.
* No canonical schema refactoring during a maintenance pass.

==== Recommended cadence

----
[work slice completes]
  -> make memory-extract
  -> make memory-context      (or make memory-health-report for a quick drift check)
  -> inspect outputs
  -> curate canonical deltas (bounded, deliberate)
  -> commit curated changes
  -> return to feature development
----

A maintenance pass should take minutes to an hour, not days.
If it grows beyond a bounded review it has become meta-work — stop and return to feature development.
