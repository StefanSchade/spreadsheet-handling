== Project Memory: Handoff and Workflow

=== What project_memory is

`project_memory` is the repo-local project state model: curated steering memory,
activity trail, and review traceability kept in a spreadsheet-friendly flat JSON
structure. It is not runtime `_meta`. It is not user-facing documentation.
It exists to give human maintainers and AI agents a fast entry point into
current project concerns, open work, and review history without reading raw
Git history or individual review documents.

=== Entry point: generated context report

Before any broad project-memory work, run:

[source,shell]
----
make memory-context
----

Then read:

----
docs_generated/project_memory/current_context.adoc
----

This generated report is the best first-entry project-state document. It is
rebuilt on demand and is always derived from canonical data. Treat it as
read-only.

=== What the report contains

Depending on current data, the report includes:

* *Concerns* — long-lived strategic steering threads (CONC-*) with linked events
* *Concern events* — the activity trail: reviews, findings, decisions, releases, implementation waves
* *Current findings* — open or partial findings filtered for current relevance
* *Active FTRs and blockers* — non-done FTRs with dependency context
* *Review sets and individual reviews* — review waves grouped by storage topology
* *Event–FTR links* — concern events linked to FTRs via shared commit references
* *Diagnostics* — unresolved references between tables (warnings only)

=== How to use the report

* Read `current_context.adoc` to understand the current concern posture and recent project evolution before broad project-memory work.
* Follow `source_id` references back into canonical JSON, `docs/cold_storage/reviews/`, FTR backlog, or findings for more detail.
* Treat generated reports as read-only. Do not edit `docs_generated/` by hand.
* Edit canonical JSON or source docs only when explicitly asked.
* Run `make memory-context` again after making changes to refresh the report.

=== Data layers

----
project_memory/canonical/
  Curated source of truth. These files are the data.

project_memory/canonical/ext_*.json
  Generated staged candidate tables. Visible in the ODS workbook
  as candidate material for review. Not curated truth.
  Gitignored and overwriteable.

project_memory/extracted/
  Raw generated extraction output from ADOC artifacts.
  Not curated. Must not be promoted into canonical without manual inspection.

project_memory/derived/
  Generated query results (current_findings, ftr_blockers, release_note_candidates,
  event_ftr_links, …). Rebuilt by make memory-query. Gitignored.

docs_generated/project_memory/
  Generated AsciiDoc reports. Rebuilt by make memory-context. Gitignored.
----

=== ODS workflow (brief)

`project_memory.ods` is an editable spreadsheet view over canonical data.
`ext_*` sheets are staging/candidate material — not authoritative.
Manual curation moves selected rows from `ext_*` sheets into the real canonical sheets.
Generated ODS and all `ext_*` tables are rebuildable artifacts and must not be
treated as hand-authored documents.

=== Review-set navigation

The project-memory model uses an explicit review-set layer to anchor
review-wave references:

----
Concern / Event
  -> Review Set  (review_sets.json, REVSET-* ids)
     -> Reviews  (reviews.json, REV-* ids)
        -> Review documents  (evidence_path)
----

Review sets mirror the storage topology under `docs/cold_storage/reviews/`.
Concern events that describe the impact of a whole review wave use
`source_type=review_set` and `source_id=REVSET-*` to anchor to the set
rather than an individual review file. This makes navigation deterministic:
a REVSET-* id always resolves to a known directory.

Do not create events that reference individual review documents when the
intent is to describe the steering impact of a whole review wave.
