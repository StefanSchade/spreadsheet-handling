== Repository Responsibilities and Publishing Boundaries

Three repositories cooperate in this project. They have deliberately separate
responsibilities. Future AI agents working on ACME tutorials, revealjs
material, demo integration, user documentation, or release publishing must not
mix these responsibilities -- especially by committing generated website
artifacts into the demo repository or by treating the demo branches as a
publishing target.

=== The three repositories

[cols="1,3,3",options="header"]
|===
| Repository | Owns | Must NOT contain

| `spreadsheet-handling` (core; exists)
| Source of truth: library code, architecture, ADRs, FTR backlog, release
  workflow and version/release semantics, AI workflow guidance, canonical
  framework documentation, the core user-guide source, transformation catalog
  source, release notes, and release-driven pages-publishing orchestration.
| Demo data/fixtures, consumer-specific config, or generated website/HTML
  artifacts. Core does not host the demo workspace.

| `spreadsheet-handling-demo` (exists)
| A compact, cloneable, runnable demo workspace: the ACME example model,
  pipeline configs, fixtures, example workbook flows, integration-flow
  documentation, and tutorial *source* material (AsciiDoc / revealjs source)
  when it is tightly coupled to those configs and data. Its README is the
  primary entry point for users and links to published versioned docs.
| Generated HTML / revealjs build output / versioned site output; release
  orchestration; core library logic; a documentation framework or CI
  publishing system; any "fluff" that makes the repo harder to clone and
  understand for someone evaluating the library.

| `spreadsheet-handling-pages` (planned by `FTR-PAGES-AND-RELEASE-NOTES-P5`)
| Static published documentation output for GitHub Pages: generated user
  documentation, transformation catalog, release notes, demo walkthroughs, and
  Reveal.js output under release-versioned paths (e.g.
  `/versions/v0.1.0b6/core/`, `/versions/v0.1.0b6/demo/`, and
  `/latest/core/`). Published docs track release tags and validated core/demo
  pairings, not arbitrary demo branches.
| Source-of-truth docs, core implementation, demo configs/fixtures, or
  unreleased scratch output. It is an artifact/publication repository, not a
  fourth development workspace.
|===

=== Artifact placement rules

When deciding where something belongs, apply these rules:

* Tutorial *source* (AsciiDoc / revealjs source) tightly coupled to demo
  configs/data -> `spreadsheet-handling-demo`, next to the configs and
  fixtures it references.
* Framework/user reference source that belongs to a specific core release
  (user guide, transformation catalog, release notes, API/YAML surface notes)
  -> `spreadsheet-handling` (core). This repo is not the primary user entry
  point, but it does produce and version the core user-documentation artifact.
* Integration-flow documentation whose examples depend on demo fixtures,
  scripts, and checked-in configs -> `spreadsheet-handling-demo`.
* Generated HTML / revealjs build output / versioned site -> *not* the demo
  repo and *not* core. It belongs in `spreadsheet-handling-pages` once the
  Pages structure exists. Publishers replace only their owned subtree inside
  shared release roots, such as `latest/core/` or `versions/<tag>/core/`.
  Until then, do not commit generated HTML anywhere.
* Release/version semantics, release notes, and publishing orchestration ->
  `spreadsheet-handling` (core).
* Canonical architecture, AI guidance, and the FTR backlog ->
  `spreadsheet-handling` (core).

=== Rationale

* *Tutorial source stays next to demo configs/data* because each tutorial
  references exact configs and fixtures. Co-location keeps tutorials runnable
  and in sync with the data they describe, and prevents drift between prose
  and the example model.
* *Core documentation source stays in core* because the user guide,
  transformation catalog, and release notes describe the framework surface of a
  specific core release. They must be built from the same release state as the
  package, even though the demo README remains the user entry point.
* *Generated HTML and versioned publishing stay out of the demo repo* so the
  demo stays small and easy to clone. Generated artifacts churn and bloat
  history, and published documentation must track releases/tags for
  reproducible versioned hosting rather than drifting on demo branches.
* *Pages owns published artifacts* so both core-built and demo-built docs can
  be exposed from one static public location after release.
* *Core owns release orchestration* so there is a single source of truth for
  version and release semantics; pages publishing is release-oriented and tied
  to validated core/demo release pairs.

=== Guidance for agents (tutorials / revealjs / demo integration / publishing)

* Writing core user documentation, transformation catalog entries, or release
  notes: source belongs in `spreadsheet-handling`; generated output is a build
  artifact for the Pages repo.
* Writing ACME tutorial source or integration-flow docs: place it in
  `spreadsheet-handling-demo` alongside the configs/data it exercises. Do not
  commit rendered HTML to the demo repo.
* Asked to "publish", "build pages", or "render revealjs to HTML": the pages
  repo structure is owned by `FTR-PAGES-AND-RELEASE-NOTES-P5`. Do not create
  it, generate revealjs/static HTML, or add CI/release publishing automation as
  a side effect of unrelated work.
* Touching release/version/release-notes: that is the core repo's
  responsibility, governed by `git_and_workflow.adoc`.
* Keep the demo repo compact: no documentation framework, no site build
  system, no generated artifacts.

`FTR-PAGES-AND-RELEASE-NOTES-P5` is the dedicated task for moving from this
policy to a concrete Pages repository and directory structure. Later content
FTRs, including the transformation catalog, should build on that structure.

=== Central governance in `spreadsheet-handling` (core)

The following project-wide concerns are centralized in the core repository
even when the work they describe lands elsewhere:

* *FTR backlog and review notes* -- the single backlog under
  `docs/backlog/` (active) and `docs/cold_storage/backlog/` (closed /
  wont-do). Demo and pages do not maintain parallel backlogs; review notes
  authored during demo work live next to their FTR in core cold storage
  once the FTR closes.
* *AI agent instructions* -- the authoritative `CLAUDE.md`, `AGENT.md`,
  and `docs/ai_info/` set lives in core. Sibling repos keep brief entry
  files that point to core; they do not grow parallel agent policy.
* *Developer guide and architecture documentation* --
  `docs/developer_guide/`, `docs/technical_model/`, `docs/functional_model/`
  and related material live in core. Sibling repos describe only their own
  user-facing or artifact-specific surface.
* *Release runbook, release notes, CI policy* -- per
  `developer_guide/ch09_release_management/`. Sibling workflows mirror the
  patterns documented there; they do not invent their own.

Sibling repos remain self-sufficient for what they actually own:

* `spreadsheet-handling-demo` keeps demo pipelines, fixtures, walkthrough
  source, launchers, render scripts, and the demo CI workflow. Its
  `CLAUDE.md` / `AGENT.md` are entry pointers, not policy documents.
* `spreadsheet-handling-pages` keeps published artifacts, the root
  navigation page, and vendored static assets. Its `CLAUDE.md` / `AGENT.md`
  exist mainly to redirect agents away from treating the artifact repo as a
  source-of-truth workspace.

=== Path-reference convention for cross-repo entry docs

Sibling-repo entry files cannot assume a particular local clone name. The
upstream name is `spreadsheet-handling`; some workspaces (including the
maintainer's) check it out as `core/`. Entry docs in sibling repos should
name the *upstream* repo explicitly and list both common local paths, for
example:

----
The core `spreadsheet-handling` repository (sibling on disk; typically at
`../spreadsheet-handling/`, or `../core/` when the workspace uses a short
local clone name).
----

This keeps the docs correct regardless of the local clone layout.
