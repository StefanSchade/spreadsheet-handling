== Conventions

=== Python

* `from __future__ import annotations` in every module
* Type hints required on all public functions and methods
* Dataclasses for value objects; prefer immutability unless mutation is domain-relevant
* Composition over inheritance; use ABCs or explicit extension points when needed
* Naming: `lowercase_with_underscores` functions/variables · `PascalCase` classes · `UPPERCASE` constants
* No nested conditional expressions in production code — use explicit `if`/`elif`/`else`
* Named intermediate variables over dense expressions when the value carries domain, pipeline, or rendering semantics
* Functions small enough to review as one behavior unit; extract meaningful pure helpers
* Do not split code mechanically just to silence a source-style guard

=== Bash

* Strict mode in every script: `set -euo pipefail`
* Externalize complex Makefile logic into `tools/` or `scripts/` (see
  "Helper script placement" below); invoke as a single line from the recipe
* Document environment variables at the top of each script
* Prefer Python for complex logic (better testability); propose the alternative when scope warrants it

=== Helper script placement: `scripts/` vs `tools/`

Two top-level helper directories exist; the split is intentional and matters
for cross-repo reuse:

* `scripts/` -- *repo-local* automation tied to this repository's own build,
  dev, branch, or release helpers (e.g. `build_docs_pages.sh`,
  `merged_to_dev.sh`). Sibling repos do not copy from here.
* `tools/` -- *generic, copyable* helpers intended to be reused across
  sibling repos (e.g. `repo_snapshot.sh`, `check_source_style.py`,
  `pip_install_spec.sh`, `ensure_pip.sh`). Sibling repos copy what they need
  rather than depend on a sibling checkout; see the
  `spreadsheet-handling-demo` Makefile `snapshot` target, which references
  `tools/repo_snapshot.sh` and instructs the user to copy it manually if
  absent.

When adding a helper, place it under `scripts/` if it is meaningful only
inside core's own build or dev loop, or under `tools/` if a sibling repo
could plausibly copy and run it unchanged. If in doubt, prefer `tools/` and
keep it dependency-light.

=== YAML / JSON configs

* 2-space indentation · explicit booleans (`true`/`false`) · avoid `null` unless required
* Comment config keys with expected type and semantics
* Validate syntax before committing

=== Makefile

* Complex multi-line shell logic belongs in `tools/` or `scripts/`, not
  inline recipes -- see "Helper script placement" above for the split
* Provide entire targets when modifying; full file if multiple targets change
* Use `##` comment on every target (auto-generates `make help` output)

=== OS environment and file portability

Three distinct levels apply.  Keep them separate when reasoning about a change.

==== Development and automation environment

The project targets POSIX/Linux for automation: Makefile recipes, `tools/`
scripts, CI commands, and agent workflows may assume a Linux environment unless
an FTR explicitly widens the target.  This is not a claim that the whole project
supports native Windows development.

==== Input file compatibility

The library should not assume that input files were created on Linux.  Accept
Windows-originated files where feasible:

* CRLF line endings in text-based inputs
* Excel-generated XLSX and ODS archives
* Files normalized by Git on any platform

==== Generated artifacts and domain semantics

Generated files and report metadata must be OS-agnostic by default.  Whenever
writing a file or building a report frame:

* *Encoding* — declare explicitly; default UTF-8 unless the FTR specifies otherwise
* *Newlines* — declare explicitly; default LF (`\n`) unless the FTR specifies
  otherwise; use `open(..., newline="\n")` or equivalent, never the platform default
* *Paths in reports* — relative to the configured output root, not absolute
* *Ordering* — fully deterministic: independent of filesystem iteration order,
  input row accidents, dict-insertion order, locale-specific collation, or platform
  defaults; sort by canonical keys before writing or reporting
* *Byte counts and checksums* — derived from the exact bytes written under the
  declared encoding/newline policy, not from a stat call on a differently normalized copy

=== Source-style guards

Run locally with `make source-style-guards` (delegates to `tools/check_source_style.py`).
Exceptions documented in `tools/source_style_allowlist.yaml`.

Active rules:
* Nested conditional expressions in production code → *error*
* German repository-facing comments after cleanup baseline → *error*
* Overlong functions, high branch complexity, dense comprehensions → *warning* (review triggers, not automatic failures)

Allowlist entries must include: `rule`, `path`, `symbol` (if scoped), `rationale`, `owner`, `review_date`.
Temporary entries must name the cleanup condition that removes them.

=== Testing

==== Running tests

[source,console]
----
make test                             # active dev slice: unit + integration + architecture
make test-verbose                     # same with streaming logs
make test-lastfailed                  # re-run last failures
make test-one TESTPATTERN="expr"      # keyword filter
make test-file FILE=path/to/test.py   # single file
make test-node NODE=path::test_name   # single test
make test-unit / test-integ / test-arch / test-ir / test-ods / test-smoke
make test-prehex                      # quarantined pre-hex slice, explicit
make test-all                         # everything including slow
----

Override the marker expression: `make test MARK="not slow"` or `make test MARK=` (no filter).
Pass extra pytest flags: `make test PYTEST_OPTS="--maxfail=1 -x"`.

==== Test naming

* File: `test_<subject>_<aspect>.py` — e.g. `test_ir_writepath_semantics.py`
* Function: `test_<expected_behavior>` — reads like the invariant, e.g. `test_roundtrip_preserves_named_ranges`
* Avoid: `test_misc.py`, and vague names like `happy`, `basic`, `works` when a behavior name is available
* Architecture tests name the boundary directly: `test_rendering_layer_does_not_import_adapters`

Add a short module docstring when a module represents an integration slice or guards an architecture boundary.
State what it protects and why.

==== Test locations and lifecycle

[cols="1,2"]
|===
| Location | Purpose

| `tests/unit/`
| Fast, in-memory; no filesystem unless testing IO

| `tests/integration/`
| End-to-end with real files

| `tests/roundtrip/`
| Invariants over full canonical -> workbook -> canonical pipeline cycles.
  Use when the bug class only appears across an export-import loop
  (cross-step contracts, persisted-meta convergence, pathological-cell
  contracts). Tests are named for invariants, not for implementations
  (`test_helper_columns_absent_after_canonical_reimport`, not
  `test_apply_derived_column_policy_drops_columns`). Run via
  `make test-roundtrip` or `pytest -m roundtrip`. See
  `FTR-ROUNDTRIP-TEST-LAYER-P4A`.

| `tests/architecture/`
| Import-graph and structural guardrails

| `tests/legacy_pre_hex/`
| Quarantined pre-refactor tests; run only with `make test-prehex`
|===

Transitions must be deliberate:
* Quarantined → Active: validate against current architecture, place in correct layer
* Quarantined → Archived: prefer documented deletion; record why behavior was retired
* Active → Quarantined: only during active redesign, with a short written rationale

==== FTR traceability markers

Every test file declares its FTR via module-level `pytestmark` (after imports, before any test functions):

[source,python]
----
import pytest

pytestmark = pytest.mark.ftr("FTR-EXAMPLE-NAME")
----

Multiple FTRs: use a list. Marker registered in `pyproject.toml`.
Collect by name: `pytest -m ftr -k "FTR_NAME"` (note: `-m "ftr('X')"` is not supported — use `-k`).

==== TDD and coverage

* Characterization tests before reshaping dense existing logic
* Prefer small in-memory DataFrames; assert schema, row count, and keys explicitly
* Tests must be hermetic and fast; no network access
* Use filesystem only when the purpose of the test is IO
* Apply `pytest-cov` coverage reporting; call out gaps when reviewing or delivering
