== Git and Workflow

=== Commit format

Conventional Commits: `type(scope): summary`

* Use the active FTR ID as scope for backlog-traceable work: `feat(FTR-IR-WRITEPATH-P1): ...`
* Subject ≤ 72 chars · imperative tone (e.g. "add support for", "fix", "remove")
* Non-trivial commits: short English body explaining the *why*
* Vague summaries (`misc`, `cleanup`, `fix stuff`) are not acceptable
* Reference GitHub issues in footer: `References: #24`; `Closes: #24` when complete
* One logical change per commit

==== Immediate FTR and review follow-up commits

After completing an FTR implementation, commit the resulting changeset
immediately with a Conventional Commit that names the FTR ID.

After applying review findings for an FTR, commit that follow-up immediately as
its own Conventional Commit that names the same FTR ID.

Do not leave FTR implementations, review findings, or follow-up documentation
uncommitted across later FTR work. The goal is a clean history with a clear
mapping from finding to fix, minimized scope mixing, and reviewable evolutionary
architecture steps.

==== FTR review notes

Every FTR implementation review is captured in a review note that records the
verdict, architectural assessment, findings, and final disposition. When review
follow-up changes are made, update the review note to reflect their resolution
*before* committing the follow-up.

Review agents should start from
link:review_instructions.adoc[review_instructions.adoc] before deciding which
additional AI-info files or review artifacts to load.

Location depends on the repo:

* *core* — sibling note next to the FTR file:
  `docs/backlog/FTR-<ID>_review.adoc`
* *demo* (and other consumer repos that do not own FTR files of their own) —
  collected under a `reviews/` subdirectory of the local backlog:
  `docs/backlog/reviews/FTR-<ID>_review.adoc`

The note format is the same in either location. Trivial edits (typo fixes,
single-line doc tweaks) do not need a review note.

==== Release-notes checkpoint

For every FTR that changes user-facing behavior, public CLI / YAML /
configuration surface, package / import / plugin surface, demo behavior,
documentation entry points, migration semantics, or other release-relevant
behavior, the implementer must either:

* update `docs/release_notes/release_notes.adoc`, or
* explicitly state in the FTR review artifact why no release note is needed.

This includes adding, changing, or removing *Known limitations* when the FTR
changes a limitation's validity, scope, or release relevance.

Internal-only refactors, tests, guard-only changes, and documentation-internal
cleanups do not need release-note entries unless they affect public behavior or
release positioning. Do not add release notes for every FTR unconditionally.

Good examples:

[source,console]
----
feat(FTR-IR-WRITEPATH-P1): complete write-path parity with meta persistence

docs(backlog): add parse-contract follow-up note
docs(FTR-REVIEW-001-EXAMPLE-P3): address review findings
----

=== Branching model

* `main` → stable / release-ready
* `dev` → integration branch; default target for all commits

Default: commit directly to `dev`.
Feature branches are only for out-of-the-ordinary work (e.g. a major refactor)
and require coordination with the developer, who must perform `git pull` before the merge.

Out-of-the-ordinary branch flow (developer coordinates):

[source,console]
----
# developer: git pull --rebase  (remote sync before branch)
git switch -c feature/FTR-IR-WRITEPATH-P1
# implement + test
git commit -m "feat(FTR-IR-WRITEPATH-P1): ..."
# developer: git pull --rebase on dev, then:
git switch dev
git merge --ff-only feature/FTR-IR-WRITEPATH-P1
# developer: git push
----

NOTE: Remote sync steps (`git pull`, `git push`) are performed by the developer,
not autonomously by agents — see agent_rules.adoc § Remote sync.

=== Merge policy

* Rebase-only: no merge commits on `dev`
* Rebase feature branch on `dev` before merging; merge only after relevant tests pass
* After rebasing a pushed branch: `git push --force-with-lease` (developer executes)
* GitHub PRs: use "Rebase and merge"

=== Virtual env and dependency stamps

* `.venv` presence does not guarantee a working environment — stamps may be stale
* Safe recovery path:

[source,console]
----
make clean-stamps && make setup
----

* Do not stage environment artifacts; do not infer test runnability from `.venv` alone

=== Tooling sync (spreadsheet-handling-demo)

* `spreadsheet-handling/tools/` is the canonical source for *shared* helper
  scripts (snapshot, source-style guard, pip helpers, etc.). Sibling repos
  copy what they need; the demo Makefile `snapshot` target references
  `tools/repo_snapshot.sh` and instructs the user to copy it manually if
  absent. See `ai_info/conventions.adoc` for the `scripts/` vs `tools/` split
* `spreadsheet-handling/scripts/` is *core-local* (docs build, branch hygiene,
  router debug). Not intended for cross-repo reuse — sibling repos do not
  copy from here
* Changes to `tools/` may be legitimate sync work — keep isolated from feature commits

=== FTR status lifecycle

[cols="1,3"]
|===
| New | Not yet started
| Open | Acknowledged; may be in progress
| Open (Umbrella) | Coordination FTR; delegates to sub-features
| Superseded | Replaced by another FTR — kept for traceability
| Done | All acceptance criteria met and merged to `dev`
|===

FTRs start in `docs/backlog/`.
Move completed FTRs to `docs/cold_storage/backlog/ftrs_done/`.
Move dropped, rejected, or superseded FTRs to `docs/cold_storage/backlog/wont_do/`.
Keep `docs/backlog/_ftrs_open.adoc` focused on active and future work.

==== Documentation artifact roles

Full rule:
`docs/developer_guide/ch08_git_and_workflow/01_git_and_workflow.adoc`
section "Documentation Artifact Roles".

Condensed:

* `docs/backlog/` is the living planning surface for open, postponed, and
  future FTRs / BUGs. It is not a stable archive.
* `docs/warm_storage/` is active working storage for adoption notes, broad
  reviews, assessments, and exploratory concepts. Backlog and other living
  artifacts may reference it when it is the current evidence source.
* `docs/cold_storage/` is the archive. Move completed / historical material
  there instead of deleting it; keep moved bundles discoverable with
  `index.json` where appropriate.
* Project Evolution is the curated phase / milestone narrative, not a
  per-FTR changelog.
* Release Notes are for user- and release-relevant changes, migration notes,
  and known limitations, not every internal FTR.
* `docs/ai_info/` summarizes operational rules for agents and points to the
  Developer Guide rather than duplicating it fully.

==== Documentation cleanup runbook

Full runbook:
`docs/developer_guide/ch08_git_and_workflow/01_git_and_workflow.adoc`
section "Documentation Cleanup Runbook".

Use it after a release or whenever backlog / warm storage drift obscures
current work. Condensed process:

* update Project Evolution at phase / milestone granularity before pruning;
* update Release Notes only for user- or release-relevant changes;
* remove completed items from `_ftrs_open.adoc`;
* move completed FTRs / BUGs and reviews to cold storage;
* move historical warm-storage bundles to structured cold-storage areas;
* keep or add flat-array `index.json` files for moved bundles;
* validate JSON, AsciiDoc entry points, and `git diff --check`.

==== Umbrella FTRs

* Carries integration-level acceptance only meaningful when all sub-features are complete
* Each sub-feature is a normal FTR; sub-features move to archive individually
* Umbrella moves only when all subs are done AND umbrella integration acceptance is met
* Use an umbrella only when sub-features share a coherent theme with dependency ordering — not for loose grouping

==== Superseding FTRs

Add `*Status:* Superseded by <<FTR-NEW#,FTR-NEW>>` to the old FTR.
Move it to `docs/cold_storage/backlog/wont_do/` when the replacement is accepted.

==== Backlog lifecycle — phase completion

Update the project evolution narrative and clean up the open backlog **at phase granularity**, not FTR-by-FTR:

. When all FTRs of a phase are done, write one paragraph or a short section in `docs/developer_guide/ch10_project_evolution/01_project_evolution.adoc`.
. Remove completed FTR includes from `docs/backlog/_ftrs_open.adoc`.
. Move completed FTR files from `docs/backlog/` to `docs/cold_storage/backlog/ftrs_done/`.

*Rules:*

* `_ftrs_open.adoc` contains only genuinely open `include::` directives and future milestones.
* Do not update the narrative per FTR — update it per phase or milestone.
* A phase with some open and some closed FTRs keeps only the open `include::` lines.

=== When to create an FTR

Create an FTR when the work has **acceptance criteria worth defining upfront and checking off later** —
new features, significant architectural changes, anything with user-visible impact.

No FTR needed for: cleanup, refactoring, environment migration, bug fixes, documentation restructuring.
A good commit message is sufficient for these. The story fits in the commit; the FTR would add overhead without value.

=== Per-FTR workflow

[cols="1,2,3"]
|===
| Step | Activity | Owner

| 1. Plan
| Read FTR, clarify scope, raise questions
| Joint (developer + AI)

| 2. Implement
| Code + tests; commit directly to `dev` (feature branch only for out-of-the-ordinary work)
| AI (developer reviews as needed)

| 3. Self-Review
| Verify every AC against actual code; list deviations and deferred items
| AI

| 4. Merge-Ready Report
| `FTR-XXX: ✅ AC 1–4, ⚠️ Item 5 deferred to FTR-YYY` + files changed, test count, open items
| AI → Developer

| 5. Merge
| Fast-forward to `dev` after developer confirms
| Joint
|===

==== Self-review checklist (step 3)

* [ ] Every acceptance criterion met or explicitly deferred with rationale
* [ ] All new code has tests; test count reported
* [ ] Full test suite passes: `pytest tests/unit tests/integration tests/roundtrip tests/architecture` (or `make test`)
* [ ] No unintended changes outside the FTR scope
* [ ] FTR document updated: status set to Done, delivery scope table added
* [ ] FTR moved from `docs/backlog/` to `docs/cold_storage/backlog/ftrs_done/`
* [ ] `docs/backlog/_ftrs_open.adoc` contains no include for the completed FTR
* [ ] `pytestmark = pytest.mark.ftr("FTR-XXX")` set on all new/modified test files
* [ ] Release-notes checkpoint applied: update
  `docs/release_notes/release_notes.adoc` (including *Known limitations* when
  a limitation's validity, scope, or release relevance changes), or record in
  the FTR review artifact why no release note is needed

==== Epic-end consolidated review

When all FTRs of an Epic/Phase are complete:

. Acceptance sweep: re-check each FTR's AC against the current codebase (code may have drifted)
. Cross-FTR integration test: exercise multiple FTRs together (e.g. write → edit → re-import → transform)
. Backlog hygiene: open items → new FTRs; superseded FTRs reviewed; phase-history comments verified
. Coverage check: `pytest -m ftr --co -q` to verify no FTR is missing test coverage
. Docs: user guide and dev guidance reflect the current code state
