== Interfaces and Approval Gates

=== Consumer-facing surface

Treat these as product API even in beta — do not change without explicit user approval.
The project is currently pre-beta, so misleading historical names may be removed when
the developer explicitly accepts the breaking change.

==== User-facing surfaces

CLI entry points::
* `sheets-run` is the generic main entry point.
* `sheets-example-json-to-xlsx` and `sheets-example-xlsx-to-json` are reference
  shortcut commands demonstrating the supported wrapper pattern.

YAML pipeline/config surface::
* registered step names
* step parameter shapes
* top-level and nested `io`, `pipeline`, `pipelines`, and `profiles` fields

Dotted callable extension surface::
* plugin steps may call user-provided Python functions via dotted paths
* registry entries may bind to dotted callables internally
* this protocol is part of the intentional extension surface

Python import surface::
* keep intentionally small
* do not treat every internal module or barrel export as public API
* prefer `orchestrate(...)`, CLI mains, and documented pipeline builders as
  intentional imports

Shortcut commands::
* Shortcut commands are an intended pattern from which real product surfaces may
  later be built.
* The current example shortcut commands are examples/reference implementations,
  not the central product UX.
* The project is pre-beta; removing the misleading historical `pack`/`unpack`
  names is intentional and preferable to carrying compatibility debt.

Consumer-facing examples in `spreadsheet-handling-demo` are also gated.

=== Actions that require explicit approval before implementation

* Any rename or removal of public CLI entry points or flags
* Any structural change to the YAML config schema
* Any reshape of consumer-visible function signatures
* Any behavior change in documented plugin step contracts
* Internal refactors that would silently break any of the above
* Changes to integration assumptions in `spreadsheet-handling-demo`
