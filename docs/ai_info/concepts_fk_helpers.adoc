== FK Helper Columns — Agent Notes

FK helper columns are redundant, spreadsheet-only columns that materialize
fields from a referenced (target) table next to a foreign-key column; they ease
editing but must not become normal payload in the normalized form.

=== Key rules

* Two separate questions, two separate owners:
  ** *helper capability* (which target columns may be exposed, what the
     defaults are) belongs to the *target table* — this is **canonical** intent;
  ** *which concrete columns on the current sheet are helper columns* belongs to
     the *FK-bearing table* — this is **derived** operational metadata.
* Derived helper provenance lives under `_meta["derived"]["sheets"][<sheet>]["helper_columns"]`
  — a list of `{column, fk_column, target, value_field}` entries. This channel
  is registered in `meta_registry.yaml` as transient `meta_rendering` /
  `derived_operational_view` metadata.
* FK detection today is convention-based: column name pattern `id_(<target>)`
  (`FK_PATTERN`). Helper columns are recognized by prefix (e.g. `_employees_…`).
  Both conventions are fallback aids, not the source of truth.
* Helper-field resolution precedence inside `add_fk_helpers` (current):
  `helper_fields_by_fk` → `helper_fields_by_target` (normalized sheet key) →
  `helper_fields_by_target` (original sheet name) → `helper_fields` →
  `label_field` (legacy).
* Precedence is *per step*. There is no universal "user input always wins"
  ladder. `bootstrap_meta` has its own model (`profile_defaults` → persisted →
  `cli_overrides`) and is *not yet* the active source for `add_fk_helpers`.
* General guardrail: heuristics may fill gaps; heuristics must not silently
  contradict existing metadata.
* First conservative contract: manual edits to helper *values* are not source
  truth; helper values may be overwritten on re-enrichment; renaming helper
  columns or relevant sheets is unsupported.
* `remove_fk_helpers` currently removes by prefix; longer-term it should trust
  explicit `helper_columns` provenance first.

=== What breaks this

* Treating helper columns as ordinary payload → they leak into the normalized
  form.
* Relying on heuristics without persisting the result → second-/third-roundtrip
  drift when the carrier state changed.
* Writing helper provenance under canonical `_meta["sheets"]` instead of the
  `derived` namespace → conflates materialized state with model intent.
* Renaming a helper column or its sheet → unsupported; breaks prefix-based
  cleanup.

=== Target model (v2) — transition status

The v2 relation-oriented FK helper policy is *now produced* by
configuration steps; the FK-helper *primitives* still consume v1. This is
the transition state established by
`FTR-INFER-FK-RELATIONS-CONFIGURATION-STEP-P5`; v2 primitive consumption
is owned by `FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5`.

What runs today:

* `infer_fk_relations` writes v2 only (heuristic policy, initial mode
  `naming_convention`).
* `configure_fk_helpers` keeps its v1 / target-oriented input surface
  and writes:
  ** the v1 per-target dict under
     `_meta.helper_policies.fk.<target_key>` (current primitive
     consumers depend on it);
  ** a v2 relation list (`schema_version: 2` plus
     `relations: [...]`) derived through a transitional bridge: for each
     resolved v1 policy entry, the bridge scans current data frames and
     emits one v2 relation per source frame carrying the resolved
     literal `fk_column` header.
* `add_fk_helpers`, `remove_fk_helpers`, `validate_fk_helpers`, and
  `reorder_fk_helpers` still consume v1 + convention-based detection;
  they are unchanged.
* `targets: "auto"` on `configure_fk_helpers` is removed and raises with
  a migration message pointing at `infer_fk_relations`.

Key points to keep in mind when reading or modifying the v2 model:

* Relation identity is per FK column on a specific source frame:
  `(source_frame, source_column)`. Two sheets may reference the same
  target with different helper choices.
* `target_key` (the PK column on the target frame) is mandatory in v2 so
  primitives, validators, and reimport cleanup do not re-derive it from
  `build_registry(...)["id_field"]` defaults.
* `schema_version: 2` lives at the `helper_policies.fk` level, not on
  each relation.
* Two producers, same shape: `configure_fk_helpers` writes manual /
  explicit entries (`produced_by.mode: explicit`); `infer_fk_relations`
  writes inferred entries (`produced_by.mode: naming_convention`, later
  additional modes).
* `apply_v2_relations` merge rules: cross-producer collisions on the
  same `(source_frame, source_column)` key fail; same-producer rewrites
  that are semantically identical are accepted idempotently; any
  meaningful difference (`target_frame`, `target_key`, `helper_fields`,
  `helper_columns`, `helper_prefix`, `produced_by.mode`) fails clearly.
* Derived helper-column provenance under
  `_meta.derived.sheets.*.helper_columns` is materialized state, not the
  policy.
* The cell-level convention `id_(<target>)` (`core/fk.py:FK_PATTERN`)
  remains the wire format; the v2 model removes primitive *inference*,
  not the wire-format convention itself.
* The v1 -> v2 bridge inside `configure_fk_helpers` is *transitional
  compatibility only*. Do not generalize the "frame scan for a literal
  fk_column" pattern into heuristic inference inside primitives or
  elsewhere; heuristic inference belongs to `infer_fk_relations`.
  `FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5` owns narrowing or
  removing the bridge once primitives consume v2.
