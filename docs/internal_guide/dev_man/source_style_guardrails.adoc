== Source-Style Guardrails

Source-style guardrails protect local readability and maintainability. They are
operationally separate from architecture guardrails, which protect boundaries
and semantic contracts.

Run them locally with:

[source,console]
----
make source-style-guards
----

The target delegates to `tools/check_source_style.py` and uses
`tools/source_style_allowlist.yaml` for documented exceptions.

=== Stage A Behavior

The initial rollout is warn-first for most checks:

* overlong functions are warnings;
* high branch complexity is a warning;
* dense comprehensions are warnings;
* possible German repository-facing strings are warnings;
* nested conditional expressions in production code are errors;
* German repository-facing comments are errors after the cleanup baseline.

Warnings are review triggers, not automatic proof that code is wrong.

=== Allowlist Policy

Each allowlist entry must be narrow and include:

* `rule`
* `path`
* `symbol` when the finding is symbol-scoped
* `rationale`
* `owner`
* `review_date`

Temporary entries should name the cleanup slice or review condition that can
remove them. Stale entries should be removed rather than refreshed
mechanically.

=== CI Status

There is no general quality CI workflow in this repository at the time of the
initial source-style guardrail rollout. CI integration is therefore explicitly
deferred; the supported execution path is the separate local Make target until
the project adds a broader quality workflow.
