=== FTR Workflow

==== Per-FTR Workflow

Each FTR follows a five-step process.  Steps 1–4 happen on a feature branch;
step 5 merges to `dev`.

[cols="1,2,3"]
|===
| Step | Activity | Responsibility

| 1. Plan
| Read FTR, clarify scope, raise questions.
| Joint (developer + AI)

| 2. Implement
| Code + tests on `feature/<ftr-slug>` branch.
| AI (developer reviews as needed)

| 3. Self-Review
| Before proposing merge: verify every acceptance criterion from the FTR
  against the actual code.  List any deviations or deferred items.
| AI

| 4. Merge-Ready Report
| Short summary: +
  `FTR-XXX: ✅ Acceptance 1–4, ⚠️ Item 5 deferred to FTR-YYY` +
  Includes: files changed, test count, open items.
| AI → Developer

| 5. Merge
| Fast-forward merge to `dev` after developer confirms (or directly if the
  report is clean and no concerns were raised).
| Joint
|===

==== Self-Review Checklist (Step 3)

The self-review performed before proposing a merge covers at minimum:

* [ ] Every acceptance criterion from the FTR is met (or explicitly deferred
  with rationale).
* [ ] All new code has tests; test count reported.
* [ ] Full test suite passes (`pytest tests/unit tests/integration
  --ignore=tests/unit/legacy_pre_hex --ignore=lib64`).
* [ ] No unintended changes outside the FTR scope.
* [ ] FTR document updated: status set to Done, delivery scope table added.
* [ ] FTR moved from `open/` to `done/`; index files updated.
* [ ] Original phase/wave in `ftrs/open/_ftrs_open.adoc` still contains a
  compact comment naming the moved FTR ID and disposition, so roadmap history
  remains recoverable after old FTR files are archived.
* [ ] `@pytest.mark.ftr("FTR-XXX")` marker set on all new/modified test files.

==== Epic-End Consolidated Review

When all FTRs of an Epic/Phase are complete, perform a consolidated review:

. *Acceptance sweep:* Re-check each FTR's acceptance criteria against the
  current codebase (code may have drifted during later FTRs).
. *Cross-FTR integration test:* Run a scenario that exercises multiple FTRs
  together (e.g., write → edit → re-import → transform).
. *Backlog hygiene:*
  - Open items or deferred scope → new FTRs or notes on existing ones.
  - Superseded FTRs reviewed: still relevant or removable?
  - Completed FTR includes removed from `ftrs/open/_ftrs_open.adoc` still have
    phase-history comments that name the FTR IDs and dispositions.
. *Test coverage check:* `pytest -m ftr --co -q` to list all FTR-marked tests
  and verify no FTR is missing coverage.
. *Documentation:* User guide and dev_man reflect the state of the code.
