=== Git Project Standards

*Commit messages* MUST follow the https://www.conventionalcommits.org/en/v1.0.0/[Conventional Commits Specification].

==== **Commit message format and scope**

- Use the format:

  [source,console]
  ----
  <type>(<scope>): <short description>
  ----

- The `<scope>` field is used for **feature tracking IDs** (e.g. `FTR-IR-WRITEPATH-P1`), rather than technical subsystems.
- This creates a one-to-one mapping between commits, feature specs (FTRs), and GitHub issues.

  [source,console]
  ----
  feat(FTR-IR-WRITEPATH-P1): complete P1 parity path with meta persistence and validations
  ----

- Write commit bodies in **English**, using full sentences and bullet points for clarity.
- Reference related GitHub issues in the footer (e.g. `References: #24`) and use `Closes: #24` once the feature is complete.
- Conventional Commits are required for human and AI-generated commits alike.

==== **Branching model**

- Follow a **main / dev** branching pattern:
  * `main` -> stable, release-ready code.
  * `dev` -> integration branch for active work.

- All new work happens on **feature branches** named after their FTR or purpose:

  [source,console]
  ----
  feature/FTR-IR-WRITEPATH-P1
  ----

- This also applies to AI-assisted implementation work.
- Typical flow:

  [source,console]
  ----
  git switch dev
  git pull --rebase
  git switch -c feature/FTR-IR-WRITEPATH-P1
  # implement + test
  git commit -m "feat(FTR-IR-WRITEPATH-P1): complete write path parity"
  git switch dev
  git merge --ff-only feature/FTR-IR-WRITEPATH-P1
  ----

==== **Merge policy**

- The project follows a **strict rebase-merge policy**:
  * Rebase feature branches on `dev` before merging to keep a linear history.
  * **No merge commits** are permitted (disable `--no-ff` merges).
  * Merge to `dev` only after the relevant tests for the changed area pass.
  * Always use:

    [source,console]
    ----
    git pull --rebase
    git rebase dev
    ----

  * If a remote feature branch must be updated after rebasing, use:

    [source,console]
    ----
    git push --force-with-lease
    ----

  * Pull requests should use **Rebase and merge** in the GitHub UI to avoid merge bubbles.

==== **Commit language**

- Commit messages and pull request descriptions are written in **English**.
- Use an **imperative** tone for the summary line (e.g. "add support for", "fix", "remove").
