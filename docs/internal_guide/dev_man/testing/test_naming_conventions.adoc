==== Test Naming and Intent

Test names should make scope and responsibility visible without requiring a
reader to inspect the implementation first.

===== File Names

Use `test_<subject>_<aspect>.py`.

Examples:

* `test_ir_writepath_semantics.py`
* `test_pipeline_split_join.py`
* `test_xlsx_roundtrip.py`

Avoid generic names such as `test_misc.py`, duplicated prefixes such as
`test_test_...`, and names that only describe the test carrier. The directory
already communicates whether a test is unit, integration, or architecture.

===== Test Function Names

Use `test_<expected_behavior>` so the name reads like the behavior or invariant
that must remain true.

Examples:

* `test_roundtrip_preserves_named_ranges`
* `test_pipeline_produces_expected_frames`
* `test_adapter_does_not_import_domain_modules`

Avoid names that repeat the file name without adding intent, describe only
setup mechanics, or use vague placeholders such as `happy`, `basic`, `works`,
or `smoke` when a concrete behavior name is available.

===== Architecture Tests

Architecture tests should state the boundary or invariant directly:

* `test_rendering_layer_does_not_import_adapters`
* `test_parser_stops_at_workbook_ir`
* `test_meta_canonical_not_used_for_render_decisions`

These tests may assert structure, imports, or registry entries, but their names
should still read like rules.

===== Module Docstrings

Add a short module docstring when a module represents an integration product
slice or enforces an architecture boundary. The docstring should say what the
module protects and why it exists.

Example:

[source,python]
----
"""XLSX metadata roundtrip integration slice.

Guards the contract that canonical metadata survives write/read cycles without
becoming visible data frames.
"""
----

Keep the convention review-owned. Do not rename tests only for cosmetic
uniformity, and avoid adding strict naming tooling unless a later FTR explicitly
calls for it.
