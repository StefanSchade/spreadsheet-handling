== Pipeline Step Semantics, Configuration Contracts, and Frame Lifecycle

:status: draft
:scope: architecture
:updated: 2026-04-28

=== Purpose

This document defines the intended semantics for pipeline steps before the public YAML surface hardens further.

It separates configuration, primitive execution, composition, plugins, frame lifecycle, and render/view decisions so that pipeline behavior remains explicit, inspectable, testable, and documentable.

The document is an architecture artifact. It is not itself a runtime contract. Implementation should happen in small, reviewable slices.

=== Problem Statement

The pipeline already supports useful transformations, but the step surface is becoming harder to reason about.

Several concerns are currently mixed or only implicitly separated:

* primitive data transformations
* composite transformations that call several primitives
* heuristic discovery of configuration
* explicit user configuration
* metadata persistence for inverse or diagnostic use
* plugin extension points
* render/view decisions about which frames should be visible or editable

This creates avoidable risks:

* Users cannot easily tell whether a step executes a transformation, infers configuration, or does both.
* Heuristics can become opaque runtime behavior instead of inspectable configuration.
* Composite steps can hide primitive operations and temporary frames.
* Plugins can materialize workbook views without a common lifecycle language.
* Intermediate and derived frames can remain visible next to canonical frames, creating redundant editable data.
* Inverse operations may rely on metadata that is not documented as part of a step contract.
* User documentation and architecture tests have no single inventory from which to derive expectations.

=== Core Design Principle

Pipeline steps should separate configuration, execution, composition, and extension.

[source,text]
----
configuration step
  -> writes explicit, resolved configuration into _meta

primitive step
  -> reads direct parameters and/or resolved configuration
  -> builds one effective configuration
  -> executes one deterministic operation
  -> writes provenance and lifecycle metadata where applicable

composite step
  -> wraps a documented sequence of primitive and/or configuration steps
  -> declares public outputs, internal frames, and wrapped step trace policy

plugin step
  -> remains an explicit escape hatch for caller-owned behavior
  -> may later be replaced by generic framework behavior if the use case proves reusable
----

Design rule:

[source,text]
----
Heuristics may propose configuration.
They must not silently define execution semantics inside the main primitive transform.
----

=== Step Categories

The step registry should support at least these categories:

[cols="1,4"]
|===
| Category | Meaning

| `primitive`
| Executes one domain-level operation or validation using explicit parameters and/or resolved metadata.

| `configuration`
| Creates or validates configuration in `_meta`; may use heuristics and `auto`.

| `composite`
| Wraps a documented sequence of primitive and/or configuration steps.

| `plugin`
| Calls external, caller-owned code through a deliberate extension point.

| `infrastructure`
| Provides pipeline mechanics such as pass-through, bootstrap, or external override loading; not a domain transformation.
|===

A category is not enough by itself. Registry entries should also carry subtype and inverse information where useful, for example `validation`, `header_shape`, `meta_bootstrap`, `view_materialization`, or `lookup_projection`.

=== Primitive Steps

A primitive step performs exactly one operation.

Examples:

* `expand_xref`
* `contract_xref`
* `decode_cell_values`
* `encode_cell_values`
* `add_lookup_helpers`
* `add_fk_helpers`
* `remove_fk_helpers`
* `validate_fk_helpers`

Rules:

* A primitive step must not interpret `auto` values, except for explicitly documented transitional legacy behavior.
* A primitive step may accept direct YAML parameters for convenience.
* If a primitive step accepts direct parameters, it must be possible to express the same effective behavior as resolved metadata.
* If direct YAML parameters and resolved `_meta` policy both apply and conflict, the step must fail rather than choose one silently.
* A primitive step should write provenance where the result is derived or needed for diagnostics.
* A primitive step should declare its inverse kind.

=== Configuration Steps

A configuration step writes or validates explicit configuration in `_meta`.

Configuration steps are the intended place for heuristics.

Examples:

* `configure_lookup_helpers`
* `infer_lookup_helper_policy`
* `configure_fk_helpers`
* `infer_fk_relations`
* `configure_frame_lifecycle`
* `configure_compact_multiaxis`

Rules:

* Configuration steps may use heuristics.
* Configuration steps may interpret `auto`.
* Configuration steps may combine user-supplied overrides with inferred defaults.
* Configuration steps must write resolved, inspectable metadata.
* Later primitive steps should be able to execute against that resolved metadata.

Example:

[source,yaml]
----
pipeline:
  - step: configure_lookup_helpers
    lookup: variables
    key: ID
    allowed_helpers: [sort_key, value_label_de, business_component, data_type, module]
    default_helpers: [value_label_de]

  - step: add_lookup_helpers
    source: variable_usage_matrix_raw
    lookup: variables
    output: variable_usage_matrix
    on: ID
    helpers: default
----

Partly heuristic variant:

[source,yaml]
----
pipeline:
  - step: configure_lookup_helpers
    lookup: variables
    key: ID
    allowed_helpers: auto
    default_helpers: [value_label_de]
    auto:
      helper_candidates:
        exclude: [ID]

  - step: add_lookup_helpers
    source: variable_usage_matrix_raw
    lookup: variables
    output: variable_usage_matrix
    on: ID
    helpers: default
----

The primitive step does not own the heuristic. It executes the resolved policy.

=== Inline Parameters, Resolved Metadata, and Conflict Rules

Primitive steps may support direct YAML parameters during the transition to richer configuration steps.

This is acceptable only if conflict behavior is explicit.

Rules:

. A primitive step builds an `effective_config` before execution.
. `effective_config` may be built from inline YAML, resolved `_meta` policy, or both.
. If inline YAML and resolved `_meta` policy define the same semantic field with different values, the step fails.
. `auto` is not a valid primitive-step value unless explicitly documented as transitional legacy behavior.
. The registry marks every parameter by source kind.

Recommended parameter source kinds:

[cols="1,3"]
|===
| Source kind | Meaning

| `inline_only`
| Parameter is supplied directly and is not part of a reusable policy.

| `policy_ref`
| Parameter selects or references a resolved policy from `_meta`.

| `resolved_meta`
| Parameter is expected to be read from `_meta` after configuration.

| `inline_or_policy`
| Parameter may come from direct YAML or `_meta`, but conflicts fail.

| `deprecated_alias`
| Transitional name or shape retained only for migration.
|===

=== Composite Steps

A composite step is a convenience wrapper over a documented sequence of lower-level operations.

Examples:

* `expand_compact_multiaxis`
* `contract_compact_multiaxis`
* future workbook-view preparation steps

Rules:

* A composite step must declare `wrapped_steps` in the step registry.
* Every wrapped step must exist as a registered primitive, configuration, or infrastructure step.
* A composite step must declare public outputs.
* A composite step must declare internal frames, if any.
* A composite step must declare whether primitive trace metadata is kept, suppressed, or nested under the composite namespace.
* A composite step must not become a hidden second implementation of the primitive behavior.

Composite provenance policy is mandatory when composite steps use temporary frames internally. If primitive metadata points at frames that are removed before the step returns, later lifecycle, inverse, or documentation logic can misread those frames as public artifacts.

Recommended composite fields:

[source,json]
----
{
  "category": "composite",
  "wrapped_steps": ["encode_cell_values", "contract_xref"],
  "frame_contract": {
    "reads": ["variable_codes"],
    "writes": ["variable_usage_matrix_raw"],
    "internal_frames": ["__compact_multiaxis_<name>_encoded"],
    "public_outputs": ["variable_usage_matrix_raw"]
  },
  "composite_trace": {
    "primitive_meta_policy": "nested_under_composite",
    "suppressed_meta": [],
    "notes": "Internal primitive traces must not be interpreted as public frame contracts."
  }
}
----

Allowed primitive metadata policies for composite steps:

[cols="1,3"]
|===
| Policy | Meaning

| `preserve_public`
| Wrapped primitive metadata is kept because it references public frames.

| `nested_under_composite`
| Wrapped primitive metadata is retained only as internal trace below the composite namespace.

| `suppress_internal`
| Wrapped primitive metadata that refers only to internal frames is not emitted.

| `current_state_unknown`
| Temporary classification until the composite is reviewed.
|===

=== Plugin Steps and Genericization Loop

Plugins are legitimate and must remain integrable.

A plugin is the right place for behavior that is not yet generic, is tied to a concrete consumer model, or is still being explored.

The intended lifecycle for plugin-originated behavior is:

[source,text]
----
Plugin solves a real use case.
Then review asks:
  A) Is this reusable framework behavior?
     -> add a generic primitive/configuration/composite step and migrate controlled plugins.
  B) Is this consumer-specific behavior?
     -> keep it as plugin logic and document the responsibility boundary.
----

The framework must not be designed so tightly that plugins become second-class or impossible.

Rules:

* Plugin steps are explicit extension points, not architecture violations.
* A plugin may create, replace, drop, or materialize frames.
* If a plugin changes the frame set, it should either write lifecycle metadata or declare that it owns lifecycle responsibility for its output.
* Missing lifecycle metadata from a plugin should initially produce diagnostics or review warnings, not a hard failure.
* Genericization should be driven by recurring plugin patterns, not by theoretical purity.

Recommended plugin registry fields:

[source,json]
----
{
  "name": "plugin",
  "category": "plugin",
  "plugin_contract": {
    "may_change_frame_set": true,
    "lifecycle_responsibility": "caller_owned_or_declared",
    "diagnostics": "warn_if_frames_change_without_lifecycle"
  }
}
----

Workbook-view plugins such as consumer-specific `build_*_workbook_views` functions should be treated as current transition paths. They may remain plugins when they are model-specific, or become generic view-materialization steps if the pattern repeats.

=== Infrastructure Steps

Some steps are not domain transformations and should not be forced into the primitive/configuration/composite model.

Examples:

* `identity`
* `bootstrap_meta`
* `apply_overrides`

These are infrastructure or meta-bootstrap steps.

Rules:

* Infrastructure steps still need registry entries.
* Infrastructure steps still need documented parameter surfaces.
* Infrastructure steps may read or write `_meta`, but their role should be classified explicitly as bootstrap, override loading, or pass-through.

=== Step Registry

A JSON-based step registry should become the reviewable inventory of the public pipeline surface.

Initial status:

[source,text]
----
The JSON step registry is a review artifact validated against runtime REGISTRY.
It is not initially the runtime source of truth.
Runtime REGISTRY remains authoritative until a future decision explicitly changes that.
----

Initial purpose:

* document every public step
* classify steps by category and subtype
* document parameters and their source kinds
* document effective metadata contracts
* document frame input/output behavior
* document inverse behavior
* document composite wrappers and plugin extension contracts
* support architecture tests
* later support generated user documentation

Recommended initial location:

[source,text]
----
docs/internal_guide/architecture/pipeline_step_registry.json
----

This location is acceptable while the internal guide is the active architecture staging area. If the registry becomes runtime-bound or directly product-facing, it should later move to a stable product-facing location such as:

[source,text]
----
src/spreadsheet_handling/pipeline/step_registry.json
----

or:

[source,text]
----
docs/reference/pipeline_step_registry.json
----

Repository rule:

* If the JSON registry stays under `docs/`, add a precise `.gitignore` exception for this file.
* Do not rely on a path currently treated as ephemeral by repository ignore rules.

=== Step Registry and Meta Registry Alignment

The step registry must not become a second, independent source of metadata truth.

The existing meta registry remains the review artifact for spreadsheet-relevant metadata entries and their layer/classification behavior.

Rule:

[source,text]
----
Every root referenced by step_registry[*].meta_contract must either:
  A) exist in meta_registry.yaml, or
  B) be explicitly marked as transient / trace-only in the step registry.
----

Recommended additions to the meta registry during follow-up work:

* `helper_policies`
* `frame_lifecycle`
* `derived.sheets.*.helper_columns`
* `derived.sheets.*.lookup_helpers` or the current equivalent lookup provenance path
* composite trace namespaces if retained persistently

Registry `meta_contract` entries should include at least:

* `path`
* `layer`
* `classification`
* `registry_ref` or `transient_reason`
* persistence/roundtrip relevance where applicable

=== Equality and Bijectivity

The project should not equate transformation with byte-identical representation.

A transformation can be relation-preserving even when the concrete representation changes.

Example:

* A normalized relation can become a compact matrix.
* The concrete table shape changes.
* The transformation can still preserve all relevant relations if row keys, column keys, code semantics, and empty-cell semantics are explicit.

Therefore the registry should classify inverse behavior precisely.

Recommended inverse kinds:

[cols="1,3"]
|===
| Kind | Meaning

| `bijective`
| The inverse operation reconstructs the relevant information without additional assumptions.

| `conditionally_bijective`
| The inverse operation is valid only if listed metadata or policy constraints are present.

| `relation_preserving`
| The operation preserves the relevant relation but not the exact representation.

| `projection_only`
| The step adds derived information that should not be treated as canonical ownership.

| `validation_only`
| The step checks consistency but does not transform data.

| `extraction`
| The step intentionally discards information and cannot reconstruct the input relation.

| `unknown`
| Temporary classification for legacy or not-yet-reviewed steps.
|===

Example for compact multiaxis:

[source,json]
----
{
  "name": "contract_compact_multiaxis",
  "category": "composite",
  "wrapped_steps": ["encode_cell_values", "contract_xref"],
  "inverse": {
    "kind": "conditionally_bijective",
    "inverse_step": "expand_compact_multiaxis",
    "requires_meta": [
      "compact_multiaxis.<name>.row_keys",
      "compact_multiaxis.<name>.column_keys",
      "compact_multiaxis.<name>.mode",
      "compact_multiaxis.<name>.fill_value"
    ],
    "notes": "Sparse output is relation-preserving only if empty cells are semantically irrelevant or explicit dense-axis reconstruction metadata is present."
  }
}
----

=== Frame Lifecycle

Many transformations use explicit input and output frames.

Example:

[source,yaml]
----
- step: contract_compact_multiaxis
  relation: variable_codes
  output: variable_usage_matrix_raw
----

Keeping both input and output frames can be useful for debugging and traceability.

But it is dangerous when rendering an editable workbook, because users may see and edit multiple redundant representations of the same information.

The solution should not be an obscure render flag or name-based guess.

The solution should be an explicit frame lifecycle contract in `_meta`.

Example:

[source,yaml]
----
_meta:
  frame_lifecycle:
    variable_codes:
      role: canonical_source
      canonical: true
      editable: false
      render: omit_by_default
      superseded_by: variable_usage_matrix_raw

    variable_usage_matrix_raw:
      role: intermediate
      canonical: false
      editable: false
      render: omit_by_default
      derived_from:
        - variable_codes
      superseded_by: variable_usage_matrix

    variable_usage_matrix:
      role: editable_projection
      canonical: false
      editable: true
      render: visible_by_default
      derived_from:
        - variable_usage_matrix_raw
        - variables
      consistency_policy:
        on_conflict: fail
----

Frame lifecycle should answer:

* Which frames are canonical?
* Which frames are intermediate?
* Which frames are editable projections?
* Which frames are redundant for the current workbook view?
* Which frames should be omitted by default during spreadsheet rendering?
* Which frames participate in inverse reconstruction?
* What happens if redundant representations conflict?

The initial lifecycle model assumes one workbook view per pipeline run. Multi-view scenarios such as debug view versus production view are deferred.

=== Workbook Views, Rendering, and Pipeline Boundaries

Pipeline execution must remain linear.

Render visibility must not become hidden pipeline routing.

Wrong direction:

[source,text]
----
Pipeline inspects _meta and silently skips steps or payload branches.
----

Preferred direction:

[source,text]
----
Pipeline executes declared steps linearly.
Steps write explicit lifecycle and view metadata.
Rendering consumes explicit workbook-view policy to decide visibility/editability.
----

A future rendering option such as `drop_redundant_data` is acceptable only if it means:

[source,text]
----
Do not render frames that are explicitly classified as redundant or omit-by-default in frame_lifecycle.
----

It must not mean:

[source,text]
----
Guess which frames look redundant by name, shape, or position.
----

Workbook-view plugins that physically materialize or suppress frames are allowed, but they should either emit lifecycle metadata or declare caller-owned lifecycle responsibility.

=== Naming Direction for Helper and Lookup Steps

The FK/helper/lookup step names should be normalized while controlled consumer repositories can still be updated cheaply.

Recommended direction:

[cols="1,1,3"]
|===
| Current name | Proposed name | Rationale

| `apply_fks`
| `add_fk_helpers`
| Materializes helper columns based on FK conventions.

| `drop_helpers`
| `remove_fk_helpers`
| Removes FK helper columns using provenance or explicit policy.

| `check_fk_helpers`
| `validate_fk_helpers`
| Expresses validation rather than generic checking.

| `reorder_helpers`
| `reorder_fk_helpers`
| Current behavior is FK-helper specific, not a generic helper concept.

| `enrich_lookup`
| `add_lookup_helpers`
| The source frame is enriched with helper fields from a lookup frame.
|===

Compatibility aliases should be short-lived if used at all.

Registry fields such as `aliases`, `replaced_by`, `deprecated_since`, `warn_until`, and `remove_after` should make the migration contract machine-readable.

If a hard rename is chosen, the corresponding implementation work should list controlled consumer repositories and include smoke checks for their updated YAML pipelines.

=== Guard Strategy

The concept should become guardable in layers. Guards should start with structural registry checks before attempting broad source-code interpretation.

First-wave guard candidates:

[cols="1,4"]
|===
| Guard | Rule

| Registry completeness
| Every public runtime `REGISTRY` entry has exactly one step-registry entry.

| Registry resolution
| Every public step-registry entry resolves to a runtime `REGISTRY` entry, a documented alias, or an explicit external/plugin class.

| Category validity
| Every step has category in `primitive`, `configuration`, `composite`, `plugin`, or `infrastructure`.

| Composite transparency
| Composite steps have non-empty `wrapped_steps`; every wrapped step exists; public outputs and internal frames are declared.

| Composite frame references
| Composite metadata must not reference frames absent from both the final frame set and declared `internal_frames` / lifecycle metadata.

| Meta-registry alignment
| Every persistent `meta_contract` root exists in `meta_registry.yaml`; otherwise the registry entry marks it as transient or trace-only.

| Alias lifecycle
| Deprecated aliases have `deprecated_since` and `remove_after` or equivalent release-bound fields.

| `auto` restriction
| Primitive-step parameter schemas do not allow `auto`, except explicitly marked transitional parameters.

| Inline versus policy conflicts
| Parameters marked `inline_or_policy` require conflict behavior; silent precedence is disallowed.

| Plugin diagnostics
| Plugin steps that change frame sets without lifecycle metadata can produce diagnostics rather than hard failures.

| Render non-guessing
| Render omission features may use lifecycle/view metadata but must not infer redundancy from frame names or shapes.

| Pipeline non-routing
| Pipeline orchestration must not use `_meta` for semantic branching or hidden step skipping.
|===

Not every guard has to be implemented immediately. The first useful guard wave is registry completeness, category validity, composite transparency, meta-registry alignment, and alias lifecycle.

=== Adoption Path

Adoption should be incremental.

==== Step 1: Concept and Decision Record

* Add this architecture document.
* Add a short ADR that records the decision to classify steps, separate heuristics from primitive execution, preserve plugins, and make frame lifecycle explicit.
* No production code changes are required.

==== Step 2: Step Registry Review Artifact

* Add a JSON registry for public pipeline steps.
* Keep runtime `REGISTRY` authoritative initially.
* Validate the JSON registry against runtime `REGISTRY`.
* Add registry completeness and category guards.
* Add a `.gitignore` exception if the registry lives under `docs/`.

==== Step 3: Step Name Normalization

* Rename unclear FK/helper/lookup step names while controlled consumers can still be updated cheaply.
* Decide whether old names are removed immediately or retained only as release-bounded aliases.
* Make alias lifecycle machine-readable in the registry.
* Update controlled consumers in the same integration wave.

==== Step 4: Lookup/FK Configuration Steps

* Introduce explicit configuration steps for lookup helper policy and FK helper policy.
* Support explicit values and controlled `auto` options.
* Write resolved policy to `_meta`.
* Ensure primitive helper steps execute against resolved policy or direct inline config.
* Fail on conflicts between inline config and resolved policy.

==== Step 5: Frame Lifecycle and Workbook Views

* Add frame lifecycle metadata for source, canonical, intermediate, projection, redundant, editable, and diagnostic frames.
* Use lifecycle metadata for workbook view/render decisions.
* Avoid renderer-side guessing.
* Keep plugin-created workbook views supported through lifecycle metadata or explicit caller-owned responsibility.

==== Step 6: Guard Hardening

* Add architecture tests for the stable parts of the concept.
* Start with structural registry guards.
* Add source-level heuristic and routing guards only where the invariant is stable enough.

=== Companion ADR Direction

A companion ADR should be short and decision-focused.

It should record:

* pipeline steps are classified as primitive, configuration, composite, plugin, or infrastructure
* heuristics belong to configuration-producing steps
* primitive execution must be deterministic against direct config and/or resolved metadata
* the step registry starts as a review artifact validated against runtime `REGISTRY`
* metadata paths in step contracts must align with the meta registry
* frame lifecycle is the explicit language for redundant/intermediate/view frames
* plugins remain supported extension points and are candidates for later genericization, not architecture violations

Rejected alternatives:

* keep heuristics hidden inside primitive execution steps
* let composite steps hide primitive behavior without registry trace
* infer render visibility from frame names or shapes
* close the plugin surface in the name of purity

=== Release Strategy

Do not cut a public beta solely as a fixed stake before the naming and registry direction is settled.

If a comparison point is needed, use a git tag or internal checkpoint. The next beta should ideally present the cleaner YAML surface rather than immediately deprecating names it just exposed.

=== Open Questions

* Should the step registry initially live under `docs/internal_guide/architecture/` or directly under `src/spreadsheet_handling/pipeline/`?
* Should old FK/helper/lookup names be removed immediately or retained for exactly one beta with warnings?
* Which direct YAML parameters should remain as convenience syntax once configuration steps exist?
* Should frame lifecycle first be written by selected high-risk transformations only, such as compact multiaxis and lookup helpers?
* How strict should readback conflict handling be when both canonical and editable projection frames are present?
* Which plugin diagnostics should be warnings, and which should eventually become hard failures?

=== Working Decision

Proceed with the pipeline semantics wave, but keep implementation incremental.

Do not stabilize unintuitive step names merely for short-term compatibility.

Do not hide heuristic behavior inside primitive transformation steps.

Do not let composite steps hide internal frame and provenance behavior.

Do not let render visibility become implicit pipeline routing.

Do keep plugins as first-class extension points.

Make configuration, transformation, composition, plugin responsibility, inverse behavior, and frame lifecycle explicit enough that architecture tests and user documentation can be derived from the same reviewed registry.
