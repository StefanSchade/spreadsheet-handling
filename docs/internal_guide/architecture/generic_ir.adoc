== Meta Registry, Behavioral Profiles, and the Separation of `meta_canonical` / `meta_rendering`

=== Current Situation

The current system transports different kinds of information under `meta`.
This is practical in day-to-day usage, but it only partially exposes the role
of individual meta entries.

Typical examples include:

* structurally meaningful statements, such as a column being a helper column
* render-affecting directives, such as `freeze header`
* potentially also information read from a spreadsheet and carried only for
  roundtrip persistence

At transport level, these entries initially look similar, but they do not play
the same role in the system.
The goal is to make that distinction more explicit.

=== Core Problem

Today, very different kinds of statements coexist under `meta`:

* some describe structural or domain-level properties
* some directly or indirectly influence rendering behavior
* some must survive roundtrip persistence without continuing to affect the
  write path

As long as these differences are not described cleanly, central questions
remain only implicitly answered:

* Which information is the source of truth?
* Which information is still allowed to influence rendering behavior in the IR?
* Which information is persisted only?
* Where are conflicts resolved, for example contradictory render-affecting
  directives?
* Which rules belong in the IR layer, and which explicitly do not belong in
  adapters or domain logic?

With a single backend, these questions can still be distributed informally
throughout the code.
As soon as multiple spreadsheet backends exist, however, this becomes
unnecessarily unclear because adapters can otherwise start interpreting implicit
special cases differently.

=== Core Idea

The proposed direction is intentionally pragmatic.
The immediate goal is not to redesign all meta handling or introduce a fully
generic mechanism.

The first step is a registry as structured documentation, initially maintained
manually, for example in JSON form.
For each meta entry, the registry records what role the entry plays in the
system and how it behaves along the write path.

The working term used here is *behavioral profile*.
This does not mean the ontology of an entry, but its operational behavior
inside the system.

Examples include:

* does the entry influence rendering directly, indirectly, or not at all?
* must it be persisted, or is it runtime-only?
* can it be derived from other meta?
* can it inherit across scope boundaries, such as workbook -> sheet?
* how does it behave during merge or conflict situations?
* does it take precedence over other entries when render-affecting conflicts
  occur?

Initially, the registry therefore acts as an explicit clarification mechanism.
Only later may parts of it become executable behavior inside the IR layer.

=== Terminology

==== `meta_canonical`

`meta_canonical` represents canonical meta statements originating from domain,
input, or parse.
They are the source of truth for persistence and roundtrip behavior.

Adapters may transport and persist them.
After derivation into `meta_rendering`, however, they should no longer directly
influence rendering results.

Typical examples:

* structural statements about roles or relationships
* declarative information from which rendering behavior is later derived
* roundtrip-relevant payload that must survive persistence

==== `meta_rendering`

`meta_rendering` represents the operational, render-affecting meta view inside
the IR layer.
It is derived from `meta_canonical` together with IR rules such as inheritance,
prioritization, and conflict resolution, and determines which backend-neutral
spreadsheet behavior enters the `RenderPlan`.

`meta_rendering` is not the canonical persistence form.
It serves the current write path and includes more than layout alone.

Typical examples:

* concrete freeze, filter, or style decisions
* named ranges, validations, or other render-affecting spreadsheet semantics
* prioritized or already-merged information required for rendering

==== Behavioral Profile

A behavioral profile describes in the registry how a meta entry behaves inside
the system.
Its primary purpose is not to answer what an entry “is” ontologically, but what
effect it has along the path Domain -> IR -> Adapter.

A behavioral profile may include statements about:

* scope, such as workbook, sheet, table, column, or range
* origin
* render relevance
* persistence
* roundtrip relevance
* inheritance across scope boundaries, such as workbook -> sheet
* merge behavior
* precedence during render-affecting conflicts
* producers and consumers
* ownership across layers

==== `RenderPlan`

The `RenderPlan` is the concrete but backend-neutral projection of
render-relevant behavior.
It is not the canonical source of meaning, but the executable representation
for spreadsheet adapters.

=== Target Model for the Write Path

The intended write-path model is:

[source,text]
----
Domain/Input
  -> meta_canonical
  -> IR derives meta_rendering from meta_canonical using inheritance,
     prioritization, and conflict resolution
  -> RenderPlan is generated from meta_rendering
  -> Adapter renders the RenderPlan
  -> Adapter persists meta_canonical for roundtrip
----

The responsibilities of the layers are:

* `meta_canonical` originates from domain, input, and eventually also parse.
* The IR layer interprets `meta_canonical` together with its own rules for
  inheritance, prioritization, and conflict resolution and derives
  `meta_rendering`.
* The `RenderPlan` contains the backend-neutral execution view.
* The adapter technically executes the `RenderPlan` and treats
  `meta_canonical` only as persistence payload for roundtrip.

=== Current Parse Direction

As currently defined, the reverse direction exists in limited form:

[source,text]
----
Spreadsheet file
  -> format-specific parser
  -> WorkbookIR
  -> Frames + persisted `_meta` payload
----

This already defines:

* parsers reconstruct spreadsheet structure into `WorkbookIR`
* persisted canonical meta becomes accessible again from hidden `_meta`
  payload
* a general reverse projection from user formatting back into
  `meta_canonical` is not yet defined

=== Naming and Layer Terminology

For generic modules and architecture texts:

* generic layers speak in terms of `WorkbookIR`, `meta_rendering`,
  `RenderPlan`, spreadsheet backends, parsers, and renderers
* format-specific names such as `xlsx` or `OpenPyXL` belong only in
  adapter-specific modules or intentionally format-specific documentation
* generic modules should not use format names if they describe
  spreadsheet-general responsibilities

=== Spreadsheet Backend Contract

The narrow adapter boundary is materialized in code as the spreadsheet
contract.
Its purpose is to prevent spreadsheet adapters from coupling directly to
arbitrary rendering internals.

In the target model:

* adapters use a small generic facade for the path
  `Frames + meta_canonical -> RenderPlan`
* format-specific renderers execute the `RenderPlan`
* format-specific parsers reconstruct `WorkbookIR`
* reverse projection from `WorkbookIR` back to frames plus persisted `_meta`
  payload also runs through a small generic facade

=== Intended Invariants

==== Invariant 1: `meta_canonical` is no longer render-affecting after IR derivation

Once `meta_rendering` has been derived, later rendering stages should no longer
depend directly on `meta_canonical`.

Verifiable architecture test:

* removing `meta_canonical` after IR entry must not change the render result

==== Invariant 2: Roundtrip does not depend on `meta_rendering`

Everything required permanently for reconstruction or re-persistence must exist
inside `meta_canonical` or be stably persisted from it.
`meta_rendering` must not become the sole source for roundtrip behavior.

==== Invariant 3: Adapters are not semantic engines

Adapters should not reinterpret `meta_canonical` semantically in order to
derive rendering behavior.
They should execute the `RenderPlan` and only transport, read, or persist
`meta_canonical` where roundtrip requires it.

==== Invariant 4: Reusable meta rules belong in the IR layer

If generic rules for priority, merge, inheritance, or projection exist, they
belong in the IR layer.
Neither domain logic nor adapters should carry broad meta policy semantics.

=== The Registry as the First Step

The initial expansion is intentionally modest:

* the registry is initially maintained manually
* it first serves as documentation and clarification
* it does not immediately replace typing or specialized runtime logic

At minimum, each entry should record:

* name and meaning
* scope
* origin
* behavioral profile
* producers and consumers
* persistence and roundtrip behavior
* render relevance
* conflict or priority rules where applicable

The primary benefit of this first step is conceptual clarity.
It makes visible which kinds of meta already exist and where they are still
blurred together today.

=== What the Registry May Become Later

Once entries and behavioral profiles are described with sufficient stability,
the IR layer may selectively derive recurring decisions from the registry.

Typical examples:

* priorities for contradictory render-affecting directives
* merge rules
* inheritance rules
* selective projection from `meta_canonical` into `meta_rendering`

The order matters:

* first make behavior explicit
* then stabilize it
* only afterward selectively execute it inside the IR layer

The goal is therefore not a universal meta engine.
Explicit code remains preferable where semantics are too specialized or too
irregular.

=== Explicit Boundary: This Note Describes Only the Write Path

The separation described here initially addresses only the direction from
Domain/Input toward adapters.

It does not yet define the complete reverse direction.
If future requirements demand preserving user-side spreadsheet formatting during
roundtrip, an explicit reverse projection from workbook/format information into
`meta_canonical` will be needed.

Where exactly this reverse projection belongs remains open.
The current model should neither prescribe nor block it.

=== Incremental Path

==== Step 1: Make the write-path separation explicit

Introduce a clear distinction between `meta_canonical` and
`meta_rendering`.

Goals:

* canonical meta remains available for persistence and roundtrip
* render- and spreadsheet-affecting meta is derived separately inside the IR
* rendering behavior and persistence become conceptually decoupled

==== Step 2: Protect invariants

Architecture tests and a few focused integration cases protect the new
separation.

Key checks:

* removing `meta_canonical` after IR entry does not change rendering results
* roundtrip-relevant information does not depend exclusively on
  `meta_rendering`
* adapters execute `RenderPlan` instead of reinterpreting canonical meta as
  rendering semantics

==== Step 3: Build the registry

Capture existing meta entries in structured form.
The initial goal is understanding, not automation.

==== Step 4: Consolidate recurring rules

The registry makes shared patterns visible:

* which entries inherit
* which overwrite each other
* which are only transported
* which directly produce rendering semantics
* which have conflict or priority rules

==== Step 5: Evaluate selective IR execution

Only after the model stabilizes should it be decided which of these rules are
worth executing generically inside the IR layer.

==== Step 6: Add typing and tooling

Once concepts and rules become stable, typed representations, consistency
checks, or generated documentation may follow.

=== Non-Goals of the Initial Expansion Stage

The following are explicitly not goals of this stage:

* fully redesigning all meta handling at once
* immediate full typing
* immediate code generation
* a forced generic mechanism for all meta rules
* finalizing the parse or reverse-projection path

=== Practical Guiding Question

For new meta entries, the most useful question is often not:

“What exactly is this ontologically?”

But instead:

“What behavioral profile does this entry have, and in which layer is it still
allowed to trigger behavior?”

=== Summary

The proposal does not separate concepts for abstraction’s sake, but to make the
different roles of meta more visible.

The intended path is:

* clearly identifying `meta_canonical` as the source for persistence and
  roundtrip
* deriving `meta_rendering` separately as the render- and
  spreadsheet-affecting IR view
* making meta entries explicit through a registry with behavioral profiles
* introducing possible generic decisions only later, and only selectively,
  inside the IR layer

Initially, the registry therefore acts as a clarification instrument.
If stable rules later emerge from it, they can be used inside the IR layer
without overloading domain logic or adapters with broad meta-policy semantics.
