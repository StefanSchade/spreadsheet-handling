=== ARCH-LAYERS — Layer Responsibilities and Boundaries

:updated: 2025-10-28
:status:

This section defines the conceptual layers of the Spreadsheet Handling architecture and their relationships.
It clarifies boundaries, dependencies, and responsibilities to ensure modularity and avoid leakage between concerns.

The layering follows a clean/hexagonal architecture mindset, with *CLI* and *I/O backends* as ports,
and *Pipeline*, *Domain*, and *Render IR* as the inner structure.

'''

==== 1) CLI Layer — Input Port

*Role:*
User- and system-facing entry point. This layer defines how the system is invoked, configured, and orchestrated from the outside.

*Responsibilities:*
- Provide a unified command-line interface (`sh` tool / `cli.py`).
- Collect configuration, input sources, and command arguments.
- Delegate execution to the **Pipeline** layer.
- Handle exit codes, logging, diagnostics, and global error handling.
- Never contain business logic or transformation rules.

*Must NOT:*
- Perform domain transformations.
- Contain serialization or deserialization logic for workbooks.
- Reference concrete I/O backends.

*Dependencies:*
→ Pipeline layer (calls into it)
← none

'''

==== 2) PIPELINE Layer — Application / Orchestration Layer

*Role:*
Coordinates the high-level workflow. In DDD terms, this is the **Application Service Layer**.
It wires together domain operations, rendering, and persistence without embedding business rules.

*Responsibilities:*

- Execute user-defined or system-defined sequences of transformations (domain pipelines).
- Manage lifecycle and context for transformations (e.g., staging, logging, metrics).
- Orchestrate calls into the **Domain** layer and downstream into **Render IR** or other output ports.
- Maintain order and dependency of tasks; assemble results and errors.

*Must NOT:*

- Contain domain-specific logic (e.g., how a transformation computes its output).
- Embed rendering or backend-specific logic.

*Dependencies:*

* Domain (to execute transformations)
* Render IR (for spreadsheet persistence)
* Directly to other I/O backends (for non-spreadsheet formats like JSON, CSV, XML)

'''

==== 3) DOMAIN Layer — Core Model and Business Logic

*Role:*::
The heart of the system. Holds all business rules, transformation semantics, and validation logic.

*Responsibilities:*

- Define the **Domain Model**: workbook, worksheet, and transformation objects.
- Implement domain transformations (`Workbook → Workbook`).
- Produce **semantic meta-information** that describes the workbook (e.g., which columns are ephemeral, linked, calculated, foreign keys, or required).
- Guarantee deterministic results for equivalent inputs.
- Contain all invariants and validation rules.

*Must NOT:*

- Reference user interface or CLI.
- Reference spreadsheet libraries, OpenPyXL, or other persistence details.
- Know about file formats or concrete renderers.

*Dependencies:*

→ none (pure core)
← Pipeline orchestrates it.

'''

==== 4) RENDER IR Layer — Rendering Abstraction / Spreadsheet Adapter

*Role:*::
Adapter layer for spreadsheet-like backends (XLSX, ODS, OOXML).
It translates domain results and meta-information into **backend-agnostic layout directives** (RenderPlan).
This layer exists to avoid duplicating spreadsheet-specific logic across different backends.

*Responsibilities:*::
- Convert Domain Workbook + Meta into **RenderPlan** (abstract representation of layout directives).
- Express universal spreadsheet concepts: headers, filters, validations, named ranges, freeze panes, merges, styles, formula directives.
- Provide passes (RenderPasses) to enrich and normalize the RenderPlan.
- Define and maintain the **Render IR model** (WorkbookIR, SheetIR, TableBlock, DataValidationSpec, etc.).
- Ensure backend-agnostic semantics.

*Must NOT:*::
- Import or reference backend APIs (OpenPyXL, odfpy, etc.).
- Write files or manage serialization formats.
- Duplicate logic that belongs to domain transformations.

*Output:*::
- **RenderPlan** (sequence of RenderOps).
  Examples: `DefineSheet`, `SetHeader`, `ApplyHeaderStyle`, `SetAutoFilter`, `AddValidation`, `MergeCells`, `WriteMeta`.

*Dependencies:*::
→ I/O Backend (via protocol or adapter)
← Pipeline invokes it.

'''
==== 5) IO_BACKEND Layer — Output Port / Technical Persistence

*Role:*
Implements concrete persistence adapters (e.g., `.xlsx`, `.ods`, `.csv`, `.json`, `.yaml`, `.xml`).
Each backend converts abstract render or data directives into a specific file format or stream.

*Responsibilities:*
- Implement a protocol interface (e.g., `XlsxBackend.render(plan: RenderPlan, path: str)`).
- Map RenderOps to concrete APIs (OpenPyXL, odfpy, csv, JSON dumpers).
- Handle format quirks (shared strings, zip ordering, themes, data validation syntax).
- Guarantee deterministic output and normalization for tests.
- Provide backend-specific regression tests.

*Must NOT:*
- Reinterpret or change semantic meaning of RenderOps.
- Contain domain logic.
- Expose backend-specific types to upper layers.

*Dependencies:*
→ External libraries (OpenPyXL, odfpy, Python stdlib)
← Invoked by Render IR for spreadsheet outputs or by Pipeline for simpler formats (CSV, JSON, YAML).

'''
