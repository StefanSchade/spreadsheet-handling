== Spreadsheet Backend Capability Profiles

=== Purpose

A spreadsheet backend capability profile translates the shared spreadsheet
contract into backend-specific support expectations before adapter code is
written.

It exists so that a second spreadsheet adapter does not renegotiate semantics
inside implementation code. Instead, the project can review in advance which
spreadsheet-neutral behaviors are expected to map directly, which remain tied
to backend carriers, which parse results stay hint-like, and which gaps are
accepted for an initial cut.

=== Capability Vocabulary

The mapping uses a small vocabulary:

* direct support: the backend can represent the shared semantic as part of its
  normal spreadsheet model and the project intends to support it
* carrier-backed support: the backend uses a backend-specific carrier or
  expression, but the shared semantic remains the normative meaning
* hint-only parse support: the read path may expose carrier observations, but
  those hints do not become canonical meaning
* accepted initial gap: the first adapter cut does not promise this semantic as
  portable parity

This vocabulary is intentionally about support claims, not about whether a file
format is theoretically capable of more.

=== Why Profiles Exist Even with a Shared Contract

`WorkbookIR`, `RenderPlan`, `meta_canonical`, and the spreadsheet parse
contract are intended to stay backend-neutral.
That does not mean every spreadsheet backend should claim the same support
surface at the same time.

Capability profiles are the reviewable place where the project decides:

* which shared semantics belong in the first support claim of a backend
* which semantics remain bounded by backend-specific carrier choices
* which parse results are canonical reconstructions versus quarantined hints
* which gaps are explicitly accepted instead of being rediscovered during
  implementation

This keeps the shared contract stable while allowing honest first cuts of new
adapters.

=== Initial ODS/Calc Profile

The initial ODS/Calc profile targets LibreOffice Calc / ODS through the same
`Frames -> WorkbookIR -> RenderPlan -> workbook file` seams already established
for spreadsheet-capable backends.

It is intentionally narrower than "full Calc parity".

=== Safe Naming Subset for the First ODS/Calc Cut

The initial ODS/Calc profile assumes that shared named ranges stay within a
backend-safe subset that is already reasonable for cross-spreadsheet use:

* start with a letter or underscore
* continue with letters, digits, or underscores
* avoid blanks and punctuation-heavy names
* avoid names that look like cell references

If the project later needs broader naming behavior, that should become an
explicit shared policy decision or fail-fast validation, not an ODS-only
interpretation hidden inside the adapter.

=== Carrier Choice for Canonical Workbook Metadata

The first ODS/Calc profile keeps canonical workbook metadata on the same side of
the architecture as the XLSX path:

* canonical workbook meaning remains in persisted `_meta`
* the adapter may choose a backend-specific carrier for that payload
* the shared parse contract still expects reconstructible canonical metadata
  rather than silent dependence on backend-private side channels

For the initial ODS/Calc cut, the preferred carrier is therefore a hidden
`_meta` table instead of package-level document metadata.

This keeps the contract aligned with existing roundtrip semantics:

* the metadata stays workbook-internal and reconstructible through the ordinary
  spreadsheet parse path
* hidden carrier content remains subordinate to canonical meaning rather than a
  second document-metadata channel
* parse/write parity with the XLSX path stays easier to reason about

=== Planned Implementation Split for Later Adapter Work

The later ODS/Calc implementation FTR should stay split along the same
boundaries used for XLSX:

* `io_backends/ods/ods_backend.py`: thin adapter entry layer on the spreadsheet
  contract
* `io_backends/ods/odfpy_renderer.py`: execute supported `RenderPlan`
  operations against ODS/Calc carriers
* `io_backends/ods/odfpy_parser.py`: parse workbook data into `WorkbookIR`
  and stop there
* adapter-local interpretation helpers if needed, without creating a second
  generic parser framework

The implementation should not broaden the shared contract just to accommodate
format-local convenience.

=== Consequences for Later Parity Tests

This profile implies a simple test policy for the later parity FTR:

* direct and carrier-backed support claims should become explicit parity tests
* hint-only behavior should stay visible as hints and not be promoted to
  canonical meaning
* accepted initial gaps should be encoded as explicit allowed differences
  instead of ambiguous failures or undocumented skips

That keeps the second adapter honest without forcing false promises of full
format equivalence.
