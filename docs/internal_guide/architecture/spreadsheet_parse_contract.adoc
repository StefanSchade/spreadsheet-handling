== Spreadsheet Parse Contract

=== Purpose

The spreadsheet parse contract defines the intended read-side boundary for
spreadsheet-capable backends.
It describes how a format-specific parser ends at spreadsheet-neutral
structures and where generic projection back into pipeline-facing data begins.

The contract is deliberately about target semantics, not about any one current
adapter implementation.

=== Target Shape

The intended parse direction is:

[source,text]
----
workbook file
  -> format-specific spreadsheet parser
  -> WorkbookIR
  -> generic workbook projection
  -> Frames + persisted _meta payload
----

This means:

* format-specific parser logic is responsible for reading a concrete workbook
  carrier
* `WorkbookIR` is the spreadsheet-neutral target structure of that read path
* projection from `WorkbookIR` into `Frames + _meta` is generic and reusable
  across spreadsheet-capable backends

=== Parser Responsibilities

A spreadsheet parser should be able to reconstruct at least the following
spreadsheet-neutral state:

* visible versus hidden sheets
* table blocks with geometry such as `top`, `left`, `header_rows`, `n_rows`,
  and `n_cols`
* header information in a stable IR-compatible form
* spreadsheet semantics already represented in IR locations such as
  validations or named ranges
* persisted workbook metadata payload in a reconstructible form

The parser may use format-specific heuristics internally, but it should end at
documented `WorkbookIR` semantics rather than exposing carrier-specific logic
as the general contract.

=== Required, Optional, and Carrier-Hint Results

The parse contract should distinguish three classes of results:

* required parse semantics
* optional parse semantics
* carrier-side hints

Required parse semantics are the minimum spreadsheet-neutral facts needed to
support roundtrip-safe data and metadata reconstruction.

Optional parse semantics are spreadsheet features whose reconstruction may be
backend-dependent or incremental, as long as their optional nature is explicit.

Carrier-side hints are format-near observations that may assist diagnostics or
roundtrip behavior, but do not silently replace canonical meaning.

=== Relation to Canonical Meta and IR Semantics

The read path uses the same conceptual vocabulary as the write path:

* `meta_canonical` remains the source of persisted workbook meaning and
  roundtrip relevance
* `WorkbookIR` carries spreadsheet-operational structure and semantics in a
  spreadsheet-neutral form
* carrier-side hints remain explicitly subordinate to canonical metadata and
  spreadsheet-neutral IR semantics

In particular:

* canonical metadata should remain reconstructible from the persisted workbook
  payload
* parser-side hints must not silently override canonical meaning
* format-specific carriers and formulas are not themselves the general parse
  contract

=== Boundary Invariants

The intended parse boundary is characterized by a small set of invariants:

* format-specific spreadsheet parsers end at `WorkbookIR`
* generic projection from `WorkbookIR` to `Frames + _meta` does not depend on a
  specific spreadsheet format
* carrier-side hints remain explicit and quarantined rather than becoming
  implicit canonical semantics
* spreadsheet-capable adapters do not need to infer the parse boundary from one
  concrete backend implementation

=== Non-Goals

The parse contract is not intended to be:

* a full generic parser framework
* a forced unification of all backend heuristics
* a promise that arbitrary user formatting is projected back into
  `meta_canonical`
* a requirement that every parser-side detail already be fully typed
