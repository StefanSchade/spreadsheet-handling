=== ADR-006 — Render Chain: Compose -> Passes -> Plan -> Render

:date: 2026-03-28
:deciders: dr_stefan_schade@yahoo.com

==== Context

ADR-003 introduced the IR layer; ADR-RENDER-FLOW-VS-DOMAIN-PIPELINE separated terminology.
However, the concrete data flow through the spreadsheet rendering layer was only partly documented and the older description drifted away from the current code path.
As the project prepares a spreadsheet-generalized adapter boundary, the active write chain and the currently defined read direction need a single up-to-date reference.

==== Decision

The spreadsheet write chain follows four sequential stages with distinct responsibilities:

[source,text]
----
Frames + meta_canonical
    |
    v
① compose_workbook()                  # rendering/composer/layout_composer.py
    Frames -> WorkbookIR
    one TableBlock per visible sheet
    canonical meta stashed as hidden `_meta` payload
    |
    v
② apply_render_passes()               # rendering/passes/__init__.py
    WorkbookIR -> WorkbookIR  (mutated)
    derives render-driving annotations and specs
    order: MetaPass -> ValidationPass -> StylePass -> FilterPass -> FreezePass -> NamedRangePass
    |
    v
③ build_render_plan()                 # rendering/flow.py
    WorkbookIR -> RenderPlan
    converts render-driving IR state into backend-neutral RenderOps
    |
    v
④ render_workbook()                   # io_backends/xlsx/openpyxl_renderer.py
    RenderPlan -> .xlsx file
    adapter-specific execution via openpyxl
----

The currently defined parse direction is narrower:

[source,text]
----
.xlsx file
  -> parse_ir()
  -> WorkbookIR
  -> workbookir_to_frames()
  -> Frames + persisted `_meta` payload
----

This read path reconstructs persisted canonical meta from the hidden payload and restores selected spreadsheet structure.
It does not yet define a general reverse projection from user formatting back into canonical meta.

==== Key Types and Roles

[cols="30,70"]
|===
| Type / Concept | Purpose

| `WorkbookIR`
| Container for visible `sheets` and hidden `hidden_sheets`.

| `SheetIR`
| Per-sheet container for `tables`, `validations`, `named_ranges`, and pass annotations in `meta`.

| `TableBlock`
| One logical table with geometry, headers, header map, and data block.

| `meta_canonical`
| Canonical payload from domain/input/parse, persisted for roundtrip and used as the source for IR derivation.

| `meta_rendering`
| Render- and spreadsheet-semantics materialized inside `WorkbookIR` by passes and IR rules.

| `RenderPlan`
| Ordered list of backend-neutral `RenderOp` values.

| `RenderOp`
| Frozen dataclasses such as `DefineSheet`, `SetHeader`, `SetFreeze`, `AddValidation`, `WriteMeta`, `DefineNamedRange`.
|===

==== Pass Convention

Passes communicate through explicit IR locations instead of adapter callbacks:

* hidden `_meta.workbook_meta_blob` stores the persisted canonical payload
* `SheetIR.meta["options"]` carries workbook/sheet input options into the IR layer
* dunder keys such as `__style`, `__autofilter`, `__freeze`, `__header_grid`, `__header_merges`, `__helper_cols` carry render-driving annotations
* `SheetIR.validations` and `SheetIR.named_ranges` hold already materialized spreadsheet semantics

`ValidationPass` is the clearest current bridge from canonical to render-driving state: it reads canonical constraints from the hidden payload and produces `DataValidationSpec` entries on the visible sheet IR.

==== Naming Policy

Generic spreadsheet modules and architecture docs use spreadsheet-neutral names:

* generic layers speak about `WorkbookIR`, `RenderPlan`, spreadsheet backends, parsers, and renderers
* format-specific names such as `xlsx` or `openpyxl` belong only in adapter-specific modules or deliberately format-specific documentation
* adapters execute `RenderPlan`; they are not the place where canonical meta is reinterpreted into visible behavior

==== Consequences

* The active XLSX adapter path is documented as `compose -> apply_render_passes -> build_render_plan -> render_workbook`.
* The write-path split between canonical payload and render-driving IR state becomes visible and testable.
* Generic rendering modules can be guarded against accidental imports of XLSX/openpyxl details.
* Future adapter work can target the documented `RenderPlan` boundary instead of inferring contracts from call chains.
