===== Git Primer (Project Standards)

====== Commit Message Conventions
* Follow Conventional Commits: `type(scope): summary`
* Use the FTR ID as the scope (e.g., `feat(FTR-IR-WRITEPATH-P1): ...`)
* Keep subjects <= 72 chars; body optional but encouraged for rationale.
* One logical change per commit.
* Reference GitHub issues or discussions in the footer if relevant.
* For AI-assisted work, Conventional Commit messages are the default.

====== Branching & Merge Policy
* AI agents and human contributors should create feature branches for code-bearing work.
* Preferred naming: `feature/<FTR-ID>` or `feature/<short-purpose>`.
* Docs-only changes may go directly to `dev`, but a feature branch is still preferred when the docs are part of a larger FTR.
* Maintain a `main` (stable) and `dev` (integration) branch.
* `dev` is the integration branch; feature branches are merged only after relevant tests pass.
* Prefer **rebase merges** to keep history linear.
* Rebase your branch on `dev` before opening a PR or merge request.
* Before merging a feature branch into `dev`, run the relevant tests for the changed area and document intentional test gaps in the handover if needed.
* Use signed tags for releases.
