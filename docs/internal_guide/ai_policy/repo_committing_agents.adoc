=== Repo-Committing Agents

This section applies to AI agents that can modify files, run local commands, or
create commits in the repository.

==== Canonical Sources

Root-level helper files such as `CLAUDE.md` or `AGENT.md` are only entry
points. The canonical project instructions for repo-committing agents live in
the internal guide.

At minimum, an editing or committing agent should treat the following as the
source of truth:

* `docs/internal_guide/ai_policy/ai_primer_prompt.adoc`
* `docs/internal_guide/ai_policy/git.adoc`
* `docs/internal_guide/ai_policy/project_specific_conventions.adoc`
* `docs/internal_guide/dev_man/git.adoc`
* `docs/internal_guide/dev_man/ftr_workflow.adoc`
* the relevant FTR, review note, ADR, and nearby architecture docs for the
  active task

==== Safety and Scope Boundaries

Repo-committing agents should optimize for traceability and low-regret actions.

Mandatory boundaries:

* Stay within the workspace repositories unless the user explicitly approves
  something broader.
* Do not modify unrelated scratch or environment state just because it is
  present locally.
* Treat files such as `input.json`, `out.adoc`, `.venv/`, `build/`, `tmp/`, or
  generated helper outputs as non-source state unless the task explicitly says
  otherwise.
* Do not bypass reviewability with destructive history edits merely to improve
  commit aesthetics.

==== Harmful or High-Risk Actions

The following are not normal autonomous actions for repo-committing agents:

* `git push --force`, `git reset --hard`, branch deletion, or other history
  rewriting on shared branches
* publishing releases, pushing tags, or changing CI/CD behavior without explicit
  approval
* deleting files that may be in-progress user work without confirmation
* changing public interfaces, documented config keys, or consumer-facing import
  paths without explicit approval
* touching secrets, credentials, or infrastructure outside the repository task

If a step carries real user-facing or irreversible risk, stop and ask first.

==== Commit Expectations

Commit messages are part of the project archaeology and should make later FTR
tracing easy.

Rules:

* Follow Conventional Commits.
* For backlog-traceable work, use the active FTR ID as the scope.
* Do not use vague summaries such as `misc`, `cleanup`, or `fix stuff`.
* Prefer subjects that describe the concrete delivered unit or architectural
  move.
* Non-trivial commits should include a short English body explaining the why and
  any important scope/disposition.

Good examples:

[source,console]
----
feat(FTR-META-REGISTRY-P3H): add manual spreadsheet meta registry

docs(backlog): add spreadsheet parse contract follow-up
----

==== Working Style

* Prefer small, reviewable commits over large anonymous batches.
* Keep code, tests, backlog notes, and review notes aligned when finishing an
  FTR.
* If a task is intentionally left in an in-between state, record that state in
  the backlog or review note instead of relying on chat history.
