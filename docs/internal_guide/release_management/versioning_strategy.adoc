== Versioning Strategy

=== Current Interpretation

During the current pre-1.0 phase we use beta releases on a target version line,
for example:

* `0.1.0b5`
* `0.1.0b6`
* `0.1.0b7`

The meaning is:

* `0` -- still pre-1.0, so strong compatibility promises are intentionally
  limited,
* `1` -- the current minor line,
* `0` -- the target patch level for that line,
* `bN` -- the beta iteration on the way to that target version.

The Git tag keeps the usual `v` prefix, for example `v0.1.0b6`, while PyPI
shows the package version without that prefix, for example `0.1.0b6`.

=== Practical Project Story

The current project history can be read like this:

* `0.1.0b1` and `0.1.0b2` belonged to the earlier proof-of-concept / tinkering
  stage,
* `0.1.0b3` and `0.1.0b4` already resembled today's CLI direction more closely,
* `0.1.0b5` is the large architecture-heavy beta: hexagonal structure, IR
  layer, generic pipelines, and the second spreadsheet adapter,
* `0.1.0b6` is expected to carry the first larger feature wave needed for
  professional adoption in the author's work context,
* `0.1.0b7` may carry another personally motivated feature wave while still
  staying in the same general pre-1.0 release line.

=== Near-Term Path

The intended simple path from here is:

. `0.1.0b6` -- Phase 4 feature expansion
. `0.1.0b7` -- another feature-driven beta if needed
. Polish phase focused more on documentation, structure, workflow, and release
   surrounding than on major new functional surface
. `0.2.0` -- next intentional pre-1.0 line after the feature-heavy beta period

This keeps the scheme easy to reason about:

* stay on `0.1.0bN` while iterating on the same broad pre-1.0 release story,
* move to `0.2.0` once the next line is intentionally about consolidation and
  a better presented product shape rather than just "another beta build",
* decide later, from a calmer position, what should be required before a true
  `1.0.0`.

=== Keep-It-Simple Rule

Use these defaults unless there is a clear reason to do more:

* no RC tags for now,
* no nightly or milestone tags,
* no extra numbering policy beyond beta releases and eventual stable releases,
* one core release tag per published release,
* one demo tag for the matching public integration state when useful.
