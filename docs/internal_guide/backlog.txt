===  Epics

==== Phase 1 — Stabilize and Modularize the XLSX Backend

*Goal:* Separate spreadsheet logic into clear, testable layers without changing existing behavior.

*Key results:*

- Move all XLSX-specific code into `io_backends/xlsx/` package.
- Introduce the IR (Intermediate Representation) layer with `compose → passes → render` flow.
- Migrate and tag all tests to support `@pytest.mark.xlsx_ir` and `@pytest.mark.xlsx_legacy` dual harness.
- Ensure green CI across both backends.

*Definition of done:* XLSX adapter runs through the IR path; all legacy tests pass or are explicitly tagged; internal APIs stable enough for new features.

*Goal:* Establish a unified domain/meta contract shared between CLI, pipeline, and I/O layers.

*Key results:*

- Introduce `bootstrap_meta` transformation to inject CLI/pipeline config into runtime meta.

- Ensure meta persistence and rehydration across all backends.
- Document and test merge priorities (runtime > persisted > heuristics).

*Definition of done:* All adapters and domain transformations can operate without explicit meta; missing metadata never causes failure; roundtrip preserves meta content.

*Goal:* Achieve feature parity between the new IR backend and the legacy Excel path while enabling extensible rendering logic.

- Move validation, style, and formula logic into dedicated IR passes (`ValidationPass`, `StylePass`, etc.).

- Use IR tests to confirm parity with legacy outputs for known examples.
- Provide extension hooks for future custom passes (e.g., domain-specific formatting).

*Definition of done:* IR backend produces visually and semantically equivalent workbooks for all core scenarios; legacy path officially deprecated in CI.

*Goal:* Introduce new transformation capabilities and richer domain logic leveraging the stable IR and meta foundation.

*Key results:*

*Definition of done:* Transformations are reusable and bijective; IR supports expanded layouts; new tests demonstrate composability of passes and transformations.

=== FTR-META-BOOTSTRAP — Meta Bootstrap and Fault Tolerance

** *Context:*  Workbooks without persisted meta must still roundtrip. All backends should treat missing meta as `{}` and continue.

** *Goal:*  Provide an early domain step (`bootstrap_meta`) that merges configuration inputs into runtime meta, ensuring deterministic defaults even when persisted meta is missing.
- Example for step options are: (e.g., frames.meta.constraints, style.auto_filter, style.freeze_header, style.header_fill_rgb) as we include new features we might have to extend


** *Rationale:*  Prevents `None`-handling errors and gives IR / passes a stable contract without requiring stored meta.

** *Precedence Model:*  Configuration that affects meta may come from several sources. They are applied in the following order (later overwrites earlier unless a step explicitly merges):

1. **Persistence:**  Load existing meta from storage if present; otherwise start with `{}`.
2. **Pipeline Profile Defaults:**  Apply defaults defined in the selected profile.
3. **CLI / Runtime Arguments:**  Overlay any values passed via CLI flags or explicit step configuration. CLI always wins over profile defaults.
4. **Step-Local Behavior:**  Individual steps may choose to *merge* instead of overwrite (e.g., appending to lists of non-essential columns). Such merge semantics are step-specific features, not global rules.

Example — Helper Columns:
- Helper columns are marked in meta as non-essential so they can be safely dropped.
- A later step that adds new helpers should append to that list, not replace it.


** *Acceptance:*
- Missing meta is treated as `{}` by all backends.
- `bootstrap_meta` merges sources according to the precedence model above.
- IR receives the fully merged meta and renders consistently across runs.

=== FTR-MULTIHEADER — Multi-row/column headers (with merges) + Optional Flattening at I/O

** *Purpose:*  Use pandas MultiIndex headers as the single in-memory standard. JSON/XML may emit nested objects by default; provide an optional edge step to flatten/unflatten when a flat representation is needed (including spreadsheets).

** *Scope:*  Applies to all tabular frames; domain/IR stay MultiIndex. Flattening is an optional **edge step** (Frames→Frames) that can be placed before adapters.

** *Solution:*
- Keep MultiIndex headers canonical in domain/IR.
- Spreadsheet renderer honors `headers.mode`:
  - `multi`: render merged multi-row/col headers with correct data offsets and styles.
  - `flat`: consume pre-flattened frames (produced by the edge step) and render single-line headers.
- JSON/XML adapters: default to nested objects from MultiIndex; accept already-flattened frames from the edge step if configured.
- Provide pure pipeline steps: `flatten_headers(mode=join|first_nonempty, sep='.')` and `unflatten_headers(sep='.')` for reversible mapping.
- Document edge cases (non-unique tuples, empty labels, both-axis levels) and the required conventions (placeholders, coercion, `sep`).

** *Acceptance:*
- Domain always operates on MultiIndex headers.
- XLSX: merges when `mode=multi`; renders single-row headers when fed flattened frames.
- JSON/XML: nested output by default; flat output achievable by inserting the flatten step.
- `flatten_headers`/`unflatten_headers` are pure and **lossless** under documented conventions (unique tuples, stable level order, defined `sep`).
- Tests cover: multi→xlsx (merged), multi→json (nested), multi→flatten→xlsx/json (flat), and round-trips with `unflatten_headers`.

=== FTR-NESTED-LISTS — Extract and Recompose Nested Lists from Hierarchical JSON

** *Purpose:*  Enable roundtripping between hierarchical JSON structures (objects containing lists of objects) and tabular frames suitable for spreadsheet or CSV representation.

** *Scope:*  Applies to adapters handling hierarchical formats (JSON, XML). Lists are extracted into separate frames, preserving 1-to-many and nested relationships through synthetic foreign keys.

** *Solution:*
- Introduce a transformation step `normalize_nested_lists(frames, meta)` that:
  - Scans JSON-derived frames for columns containing lists of objects or primitives.
  - Emits new frames for each list field (e.g., `orders.items` → frame `orders__items`).
  - Propagates parent identifiers into child frames to preserve referential integrity.
- Provide the inverse `compose_nested_lists(frames, meta)` to re-embed child tables into their parent JSON objects during export.
- Handle recursive/nested lists by repeating the pattern with deterministic naming (`__` or configurable separator).
- Capture the mapping manifest in `meta` for traceability (parent→child frame map).

** *Acceptance:*
- JSON with nested lists can be unpacked into multiple tables and re-composed without information loss.
- Each emitted table includes a parent-ID column referencing the originating object.
- Roundtrip tests cover simple and multi-level nesting (1-to-many-to-many).
- Spreadsheet outputs show one sheet per extracted list with consistent naming and keys.
- Config allows custom naming templates and maximum recursion depth.

** *Epic:*  Phase 4 — Feature Expansion and Domain Transformations

** *Notes:*  For detailed algorithms and examples (key propagation, naming templates, recursive depth control), see *Concept: Normalizing Nested Structures* in Chapter 2.

** *Rationale*::
Balances usability with technical constraints.

** **Solution design**::
*** Generate allowed dropdown values as all presence/absence combinations across participating layers (order preserved),
with a cap (e.g., 200) to avoid unusable Excel lists.
*** concrete cap value can receive override via config
*** explicit validations can be given (e.g. via a generic validation feature already in place) - in this case the heuristic is not applied

** Acceptance: lists generated; capped with clear warning; applied via DataValidation.
*Acceptance*::
- `excel.validation_cap` default applied; per-sheet override possible.
- Above cap: fallback to free text (no list) with log warning.
- Unit tests assert cap behavior.
