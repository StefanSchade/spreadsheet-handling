== Recommended Structure

*Recommendation: create a new `docs/human_info/concepts/` subfolder; keep the
concept docs as standalone prose files there; add compact distillations only
for the two highest-value-for-agents concepts into `docs/ai_info/`; retire
`docs/internal_guide/concepts/` for this material (but leave `by_use_case/`
untouched — it is out of scope and is a different content type).*

Reasoning:

* `docs/human_info/` is currently thin (only `adr/`). These 8 concept files are
  substantial *prose* explanations aimed at human readers — exactly the charter
  of `human_info/`. They give it real substance.
* Keeping them as **standalone files** (rather than folding them into
  `architecture_primer.adoc`) preserves their individual identity and
  cross-referenceability. The primer is meant to stay compact and agent-facing;
  inlining ~50 KB of concept prose would defeat that.
* Folding into `general_info/` does not fit — that area is status / evolution
  narrative / backlog, not conceptual reference.
* `internal_guide/concepts/` was a reasonable incubator, but the long-term home
  for *human* conceptual reference is `human_info/`. Retire the directory for
  these files (deprecation notices first, deletion after next release).
* `ai_info/` should get **only** compact, rule-shaped extracts where an agent
  genuinely needs them — and the code review already flagged exactly which:
  the canonical/derived metadata model (and its `derived` channel) is
  under-documented for agents (IR-005). Most other concepts are too detailed or
  too domain-specific to belong in `ai_info/`; they get a "Further reading"
  pointer instead.
* Two concepts are **partly transitional** (`pipeline_step_registration`,
  `roundtrip_spreadsheet_layout`) — migrate the timeless core, mark a
  `Status:` line, and move the in-flight implementation notes to a backlog/design
  note rather than freezing them in a "concept" doc.
* One concept (`roundtrip_spreadsheet_layout`) **overlaps `adr-006_render_chain.adoc`**
  and `ADR-RENDER-FLOW-VS-DOMAIN-PIPELINE.adoc` — the concept doc should own the
  *user-facing roundtrip contract + layout modes*; the ADRs own the *decisions*.
  Cross-link, do not duplicate.

== Concept Dependency / Reading Order

_"Understand X before Y" — ordered list for a new reader:_

. `metadata_canonical_vs_derived` — the vocabulary everything else reuses (`canonical` ≠ "input", `derived` ≠ "freshly computed").
. `roundtrip_spreadsheet_layout` — the roundtrip contract (data+metadata, style normalized; flat-table vs. named-range layouts) that every feature serves.
. `fk_helper_metadata` — the most concrete feature; the first place the canonical/derived split bites (helper capability vs. materialized helper columns).
. `compact_transforms` — the bijective transform family (`expand_xref`/`contract_xref`, `expand_compact`/`contract_compact`).
. `crosstab_binary_xref` — the first worked slice of `compact_transforms` (binary relation, `X`-or-empty cells).
. `sparse_relation_dense_axis` — the boundary rule for compact transforms (when may sparse become dense again — only with an explicit axis contract).
. `partition_by_domain` — an independent feature (split/merge by component to reduce Git merge conflicts); no dependency on the compact-transform chain.
. `pipeline_step_registration` — the mechanics: how YAML step names become executable steps. Read last, once you have seen what steps exist.

== Source Language Issues

All 8 concept files are already in English — *no translation needed*. The actual
clean-up items are formatting/consistency, not language:

* Heading levels are inconsistent across files: `==` (`canonical_vs_derived`,
  `fk_helper_metadata`, `pipeline_step_registration`), `===` (`compact_transforms`,
  `partition_by_domain`, `sparse_relation_dense_axis`), `=====` + `Term::`
  definition lists (`roundtrip_spreadsheet_layout`). Normalize each migrated doc
  to a single `== Title` + `=== Section` structure consistent with `human_info/`.
* FTR-style titles (`=== Concept: FTR-COMPACT-TRANSFORMS — …`,
  `==== Concept: FTR-PARTITION-BY-DOMAIN — …`,
  `===== Concept: FTR-ROUNDTRIP-SAFE-P1 — …`) should become plain prose titles;
  keep the FTR ID as a `Status:` / "Tracked as" line, not in the heading.
* `crosstab_edit_binary_xref` is titled "Use Case" but is really a concept —
  rename the title accordingly when migrating.
* Cross-reference anchors (`[[concept-canonical-vs-derived-metadata]]`, etc.)
  must be updated to the new file paths.
