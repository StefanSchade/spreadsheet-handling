== Canonical vs. Derived Metadata

=== Why This Distinction Exists

Several concepts in this project use the terms `canonical` and `derived`.
These words are easy to misread as if they described *when* information was
created:

- canonical = original input
- derived = computed later in the current run

That is *not* the intended meaning.

In this project, the distinction is functional and semantic rather than
temporal.

=== Canonical Metadata

Canonical metadata describes the stable, spreadsheet-neutral meaning that the
system wants to preserve as policy, model semantics, or roundtrip-relevant
intent.

Typical properties of canonical metadata:

- it expresses what the model *means*
- it is the authoritative source for stable semantics
- it should be portable across adapters and carrier formats
- it is a natural candidate for persistence in `_meta` or equivalent sidecar
  payloads

Examples in the current project:

- structured validation rules in `meta_canonical["constraints"]`
- sheet-level options such as `freeze_header` or `auto_filter` when treated as
  stable workbook intent
- future target-side FK helper defaults such as `default_fields`

=== Derived Metadata

Derived metadata describes an operational, materialized, or carrier-near view
of that meaning in a concrete working state.

Typical properties of derived metadata:

- it expresses how the current workbook-shaped state is materialized
- it supports planning, rendering, cleanup, diagnostics, or roundtrip
  continuation
- it may depend on pipeline steps, current sheet state, or adapter/carrier
  conventions
- it is useful and sometimes persist-worthy, but it is not automatically the
  authoritative semantic source

Examples in the current project:

- IR-local operational views such as `meta_rendering`
- carrier-side or parser-side hints
- concrete helper-column provenance like "column `_branch_name` is currently a
  helper for `id_(branch)`"

=== Important Clarification

Derived metadata does *not* mean "freshly computed during this run".

Something may enter the system as input and still be derived metadata if it
describes the current materialized spreadsheet state rather than the stable
model meaning.

This is the key point that often causes confusion.

Example:

- a spreadsheet starts the workflow
- helper columns already exist
- accompanying metadata says `_branch_name` is a helper column belonging to
  `id_(branch)`

Even if that information arrives as input, it still describes the *current
materialized workbook state*. It therefore belongs to the derived/operational
side, not automatically to the canonical side.

=== Input vs. Canonical Is Not the Same Thing

The system may receive several kinds of input:

- canonical intent
- persisted derived operational state
- carrier hints
- mixed payloads

Therefore:

- input is a transport fact
- canonical is a semantic role

These should not be conflated.

=== Persistence Does Not Automatically Make Something Canonical

Persisting derived metadata can still be useful and correct.

For example, derived operational state may be persisted so that a later run can:

- bootstrap working metadata
- continue a roundtrip safely
- clean up materialized helper columns
- preserve workbook editing continuity

But persistence alone does not promote that metadata to canonical semantics.

Persisted derived metadata remains derived if it still describes materialized
working state rather than stable model intent.

=== Practical Rule of Thumb

When deciding whether a piece of metadata is canonical or derived, ask:

1. Does it describe stable model/workbook intent that should remain true even if
   the current spreadsheet materialization changes?
2. Or does it describe how the current workbook-shaped state is materialized,
   projected, or interpreted right now?

If the first question dominates, the metadata is likely canonical.
If the second question dominates, the metadata is likely derived.
