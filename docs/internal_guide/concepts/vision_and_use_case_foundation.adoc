= Vision and Use-Case Foundation

:status: draft
:language: en

== Purpose of this Document

This document is not intended as final customer-facing marketing material.

Its purpose is to establish a coherent conceptual foundation for later:

* demo narratives,
* tutorials,
* README texts,
* website copy,
* positioning discussions,
* and customer-facing explanations.

The project currently exposes many possible usage scenarios. Without a clear conceptual center, external communication risks becoming diffuse and difficult to understand.

This document therefore focuses on:

* the underlying problem space,
* the architectural vision,
* the intended audience categories,
* and the major classes of realistic use cases.

'''

== Core Problem Statement

Modern organizations operate in a persistent tension between:

* flexible human-centric workflows,
* and rigid deterministic software systems.

Human workflows are:

* semantically rich,
* adaptable,
* context-sensitive,
* and comparatively expensive.

Classical software systems are:

* highly deterministic,
* scalable,
* low-cost per transaction,
* but structurally rigid.

Spreadsheets became the de-facto bridge between these worlds because they combine:

* accessibility for non-developers,
* high information density,
* mass-editing capability,
* visual comparability,
* and flexible structure.

As a result, many important business processes are still effectively managed through spreadsheets:

* configuration matrices,
* mapping tables,
* regulatory inventories,
* product variants,
* pricing structures,
* operational checklists,
* reference data,
* and coordination workflows.

However, traditional spreadsheet-centric workflows have major weaknesses:

* hidden semantics,
* duplicated logic,
* inconsistent identifiers,
* uncontrolled redundancy,
* fragile manual processes,
* lack of deterministic roundtrip reconstruction,
* poor versioning behavior,
* and weak integration into structured software pipelines.

'''

== Vision

The project aims to combine:

* structured canonical data models,
* deterministic processing pipelines,
* and spreadsheet-first human workflows.

The goal is not to replace spreadsheets.

The goal is to treat spreadsheets as a legitimate human-facing projection layer on top of structured canonical data.

The project therefore focuses on:

* deterministic projections,
* roundtrip-safe editing,
* explicit semantics,
* normalized canonical structures,
* backend-neutral processing,
* and reproducible transformations.

In simplified form:

[source,text]
----
canonical structured data
        ↓
human-friendly spreadsheet projections
        ↓
editing / review / enrichment
        ↓
deterministic re-import and reconstruction
        ↓
structured downstream systems
----

The project intentionally avoids turning spreadsheets into uncontrolled executable business logic.

Instead, spreadsheets are treated as:

* editable projections,
* review surfaces,
* matrix interfaces,
* enrichment interfaces,
* and structured collaboration artifacts.

'''

== Relevance in the AI Era

The current AI boom changes the technical landscape significantly.

Large Language Models lower the barrier for semi-technical users to interact with:

* APIs,
* automation,
* data pipelines,
* structured information,
* and lightweight software engineering workflows.

At the same time, AI systems frequently generate:

* semi-structured data,
* inconsistent object structures,
* unstable JSON shapes,
* semantic drift,
* and probabilistic outputs.

This creates a growing need for systems that can:

* stabilize semantics,
* normalize structures,
* preserve explicit identifiers,
* support deterministic transformations,
* and remain editable by humans.

The project therefore positions spreadsheets not as legacy artifacts, but as useful human-facing interfaces for structured semantic workflows.

The architectural direction can be understood as a controlled bridge between:

* flexible human reasoning,
* agentic or AI-assisted workflows,
* and deterministic operational systems.

'''

== Intended Audience

The project is not intended primarily for mass-market end users.

It is aimed at technically interested users and organizations operating in the space between:

* business-domain modeling,
* structured data management,
* operational configuration,
* and spreadsheet-centric collaboration.

Typical audience categories include:

* technical business analysts,
* solution architects,
* platform and DevOps engineers,
* configuration-heavy product teams,
* governance and compliance teams,
* technically oriented spreadsheet power users,
* data-centric knowledge workers,
* and advanced AI-assisted workflow users.

The project is especially relevant where organizations already depend heavily on spreadsheets but need stronger structural guarantees and pipeline integration.

'''

== Architectural Direction

The architecture deliberately separates:

* canonical semantic data,
* workbook projections,
* rendering concerns,
* metadata semantics,
* transformation pipelines,
* and backend-specific persistence.

This separation exists to avoid common spreadsheet-system failure modes:

* hidden business logic inside workbook formatting,
* uncontrolled duplication,
* implicit semantics,
* fragile manual conventions,
* and irreversible spreadsheet drift.

The project intentionally emphasizes:

* deterministic roundtrip behavior,
* explicit metadata semantics,
* backend-neutral spreadsheet semantics,
* testability,
* architecture guardrails,
* and reproducibility.

The design philosophy is intentionally pragmatic rather than academically pure.

The project does not attempt to eliminate spreadsheets.

It attempts to make spreadsheet-centric workflows more structured, maintainable, and interoperable with modern software systems.

'''

== Major Use-Case Categories

=== 1. Configuration of Generic Software

One of the primary use cases is the maintenance of complex configuration data for generic software systems.

Examples include:

* feature matrices,
* product variants,
* market-specific configuration,
* validation mappings,
* workflow definitions,
* API mappings,
* regulatory parameterization,
* and pricing structures.

In many organizations these configurations are effectively maintained in spreadsheets already, even when the target systems are highly technical.

The project enables:

* normalized canonical storage,
* deterministic exports,
* business-friendly editing projections,
* validation support,
* and integration into software pipelines.

'''

=== 2. Modern Controlled IDV

Many organizations rely heavily on Individual Data Processing solutions (IDV), often centered around spreadsheets.

These workflows frequently become operationally critical while lacking:

* reproducibility,
* testing,
* governance,
* deterministic processing,
* and maintainable semantics.

The project aims to combine:

* the flexibility of spreadsheet-centric workflows,
* with architectural principles normally associated with structured software systems.

This includes:

* explicit pipelines,
* canonical models,
* roundtrip reconstruction,
* testability,
* and controlled transformations.

'''

=== 3. AI-Friendly Structured Data

AI systems often perform significantly better when operating on structured, explicit, and normalized information.

Potential applications include:

* personal knowledge systems,
* structured journals,
* operational tracking data,
* project memory,
* reference inventories,
* and AI-readable domain models.

The project supports the idea of maintaining structured semantic information in spreadsheet-friendly forms while preserving deterministic machine-readable representations underneath.

'''

=== 4. External Data Integration

External APIs and operational systems frequently expose data in formats that are difficult for non-developers to curate directly.

Examples include:

* fitness platforms,
* operational APIs,
* monitoring systems,
* export feeds,
* and semi-structured integration outputs.

The project allows such data to be:

* normalized,
* projected into spreadsheet-oriented workflows,
* enriched by humans,
* and re-exported into structured downstream pipelines.

'''

=== 5. DevOps and Operational Configuration

Some operational processes are naturally matrix-oriented and spreadsheet-friendly.

Examples include:

* release configurations,
* deployment matrices,
* CVE curation,
* compatibility inventories,
* support mappings,
* and environment-specific parameterization.

The project aims to support such workflows while preserving:

* Git-friendly structures,
* deterministic processing,
* explicit semantics,
* and automation compatibility.

'''

=== 6. Worldbuilding and Structured Creative Domains

A secondary but interesting use case is structured worldbuilding and continuity management.

Potential applications include:

* character registries,
* location inventories,
* timeline consistency,
* species and faction relationships,
* and lore management.

This use case demonstrates the flexibility of the projection and roundtrip model but is not considered the primary architectural driver of the project.

'''

=== 7. Agentic and Semi-Autonomous Workflows

Emerging agentic systems operate in a middle ground between fully manual work and fully deterministic systems.

Such systems often require:

* flexible semantic interpretation,
* human review surfaces,
* structured persistence,
* and deterministic downstream integration.

The project may serve as a semantic stabilization layer between:

* flexible AI-driven processing,
* spreadsheet-oriented human review,
* and structured operational systems.

'''

== Non-Goals

The project is not intended to become:

* a spreadsheet replacement,
* a low-code GUI platform,
* a no-code workflow engine,
* a generic business-process-management suite,
* a SaaS collaboration platform,
* or a fully automatic AI orchestration system.

The project deliberately focuses on:

* structured projections,
* deterministic transformations,
* spreadsheet-centric editing workflows,
* and explicit semantic modeling.

'''

== Strategic Positioning

The project can be summarized conceptually as:

____
Spreadsheet-first editing for structured canonical data.
____

Or more technically:

____
Deterministic, roundtrip-safe spreadsheet projections on top of normalized semantic data models.
____

The project occupies the space between:

* human-centric spreadsheet workflows,
* modern AI-assisted data handling,
* and structured deterministic software systems.

Its central assumption is that spreadsheets are not going away.

Instead, they can become safer, more structured, and more interoperable with modern software architectures.
