= ACME Progressive Tutorial Structure

== Status

* Concept draft
* Intended as bridge between core capabilities, interaction patterns, and consumer-facing tutorials
* Intended to guide upcoming demo-repository integration work
* The authoritative next-beta scope is the `Next Beta Demo Slice vs Future
  Patterns` section. The detailed `Tutorial Progression` section is the
  exploratory cumulative superset and is not all beta-supported.

== Purpose

This document proposes a progressive tutorial structure for the ACME demo universe.

The goal is not only to demonstrate isolated spreadsheet features, but to show how increasingly complex business-facing workbook projections can be derived from canonical structured repository models and roundtripped back safely.

The tutorials are intentionally cumulative.

Each tutorial introduces:

* one or more new interaction patterns
* additional projection complexity
* additional roundtrip requirements
* additional business-facing workbook structures

The tutorials should reuse one coherent ACME business universe instead of unrelated toy datasets.

== Architectural Framing

The tutorial structure should explicitly distinguish three layers:

[cols="1,3"]
|===
|Layer |Meaning

|Core Capabilities
|Low-level transformation primitives and workbook mechanics

|Interaction Patterns
|Reusable business-facing spreadsheet interaction structures built from capabilities

|Use Case Tutorials
|Concrete end-to-end business stories demonstrating patterns in practice
|===

The ACME tutorials belong primarily to the third layer, but should explicitly reference the interaction patterns they exercise.

== ACME Demo Universe

=== Business Theme

ACME Inc. develops and distributes software products globally.

The company operates several product lines:

* ACME Home
* ACME Personal
* ACME Business

Products are offered in different countries and regions.

Some offerings require additional metadata depending on:

* deployment model
* product category
* country or region
* customer or context variation

Business users maintain and review this information through spreadsheet-oriented workbook projections.

Canonical repository data remains normalized and explicit.

=== Beta Demo Slice (authoritative scope)

A coherent showcase that exercises most of the *current* capability set
through supported semantics only -- not a tiny demo, but strictly grounded in
implemented behaviour. Modern pipeline step names only: the legacy `apply_fks`
/ `reorder_helpers` names were renamed in the `0.1.0b6` step registry to
`add_fk_helpers` / `configure_fk_helpers` / `reorder_fk_helpers`; the demo
realignment that depends on this is owned by `FTR-DEMO-REPO-INTEGRATION-P4A`.
This section defines intent only and deliberately does not invent exact demo
YAML.

==== Beta Tutorial 01 -- Normalized product catalog -> readable workbook

* Canonical: products, product_managers, branches (normalized FK relations).
* Capabilities: FK/helper enrichment via `add_fk_helpers` /
  `configure_fk_helpers`, explicit helper-column styling, `reorder_fk_helpers`,
  reference validations / `add_validations`, `flatten_headers`.
* Modern step names only; `apply_fks` must not be presented as current.
* Goal: normalized structures still produce business-readable workbook views.

==== Beta Tutorial 02 -- Product / market availability matrix

* Canonical: products, markets/countries, product offerings (many-to-many).
* Capabilities: `contract_xref` -> editable matrix workbook view,
  `expand_xref` back; dense-axis reconstruction only where implemented (flat
  technical axes, `FTR-XREF-DENSE-AXES-P4A2` minimal slice); declarative
  workbook view.
* Explicit XLSX/ODS disposition recorded as a trust signal.
* Goal: ergonomic many-to-many maintenance via matrix projection.

==== Beta Tutorial 03 -- Edit and reimport

* Forward matrix view written under a renamed visible sheet.
* `apply_workbook_view_sheet_mappings` re-keys the renamed sheet to the
  logical frame; `expand_xref` recomposes canonical relation rows.
* No conflict/precedence claims (see boundary E).
* Grounded in the closed `FTR-WORKBOOK-VIEW-ROUNDTRIP-RECOMPOSITION-P4A`
  configured reimport slice (XLSX + ODS).
* Goal: a safe `canonical -> workbook -> edited -> canonical` roundtrip.

==== Beta Tutorial 04 -- Localized / resource outputs

* Canonical: resources, locales/contexts, localized values.
* Capabilities: resource fallback/override semantics, key-value resource
  writer, generated artifact manifest.
* Goal: deterministic localized resource outputs from canonical tuples.

==== Beta Tutorial 05 / Appendix -- Backend trust story

* The XLSX/ODS parity matrix and explicitly accepted gaps from
  `FTR-ODS-CONSUMER-FEATURE-PARITY-P4A`.
* No fake parity: accepted gaps are shown as accepted gaps, never simulated.
* Goal: portability is a documented trust signal, not an implied guarantee.

=== Future / deferred patterns (must not be shown as supported)

==== A -- Dynamic conditional workbook sections / drilldown detail clusters

Business motivation: a matrix cell value can make additional detail rows
relevant -- e.g. Product × Market says "SaaS in Japan", therefore extra
compliance-contact rows are needed for the affected subset.

This is *not* live workbook UI magic. The workbook is a static projection
artifact. The preferred future architecture is a deterministic
roundtrip-driven follow-up projection: edit matrix -> reimport/recompose ->
explicit configured rule materializes follow-up rows -> a *subsequent*
generated workbook exposes the relevant detail view. Tracked by
`FTR-CONDITIONAL-FOLLOW-UP-PROJECTION-FLOWS-P5` (status New, P5).

Evaluate simpler alternatives first before treating this as a new feature:

* generate all potential detail rows and use helper columns plus autofilter to
  mark relevant rows;
* use lookup/helper formulas to show why a row is relevant;
* split relevant follow-up rows into a separate workbook view via existing
  extraction/join/filter transforms;
* require an explicit second generation pass after reimport.

If these suffice, this stays a documented pattern, not a runtime feature. The
demo may state the motivation but must not imply dynamic conditional sections
exist.

==== B -- Rich structured relation payload matrices

Keep out of the next beta unless the cell payload is a simple scalar/code
(e.g. a deployment-mode marker like `SaaS`). Structured/multi-field cell
payloads are a feature-creep door and may overlap with future nested
object/list work (boundary F). Narrative preview only.

==== C -- Temporal / planning projections

Timeline, release-planning, and gantt-style projection families are a future
domain-specific add-on pattern, not current beta. Narrative preview only.

==== D -- Grouped / structured XRef axes

The current beta uses *flat technical axes* only (the implemented
`FTR-XREF-DENSE-AXES-P4A2` minimal slice). Grouped/structured column axes and
explicit axis mappings are deferred to `FTR-XREF-GROUPED-COLUMNS-P4A2` and
`FTR-XREF-AXIS-MAPPINGS-P4A2` (both status New, Phase 4a2). Grouped axes must
not be presented as available.

==== E -- Conflict / precedence between overlapping editable views

Deferred to `FTR-WORKBOOK-VIEW-PAYLOAD-CONFLICT-PRECEDENCE-P6`. Execution
steps must never guess winners; a conflicting tuple currently surfaces only as
a generic primary-key validation failure after merge. Any future heuristic
assistance may only *propose or materialize deterministic configuration* in a
separate opt-in step; execution/recomposition stays explicit and
configuration-driven. The beta reimport tutorial must make no conflict
resolution claims.

==== F -- Nested object / list mapping

Not supported and not previewed as a workbook capability in this beta. Listed
here only so the boundary is explicit; this concept does not change it.

== Rationale

The ACME tutorial progression is intended to demonstrate that spreadsheet-oriented workbook editing can remain compatible with normalized, testable, repository-driven canonical models.

The tutorials should progressively reveal increasingly complex projection semantics while preserving deterministic roundtrip behavior.
