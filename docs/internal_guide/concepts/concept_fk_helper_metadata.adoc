[[concept-fk-helper-metadata]]
== FK Helper Metadata Concept

=== Purpose

This note explains FK helper columns in terms that are close to the current code
and the current roundtrip goal.

It is not a full metadata model.

It is a working concept for the next small feature slices around helper columns.

The terminology follows
<<concept-canonical-vs-derived-metadata,Canonical vs. Derived Metadata>>.
In particular, `derived` here does not mean "freshly computed during this run".
It means "describing current materialized workbook state rather than stable
model intent".

=== Running Example

In the examples below, the `desks` table contains the FK column and points to
the `employees` table.

* `desks` is the table containing the FK.
* `employees` is the target table referenced by that FK.

If `desks.id_(employees)` points to `employees.id`, the spreadsheet-facing
`desks` sheet may also show helper columns such as `_employees_first_name`,
`_employees_last_name`, or `_employees_birth_date`.

These helper columns make spreadsheet editing easier, but they are redundant and
should not become normal payload columns in the normalized repository form.

=== Why This Is Tricky

Helper columns sit between two concerns:

* spreadsheet usability
* normalized roundtrip-safe data

They are not just rendering decoration.

They also are not ordinary domain payload.

They are redundant columns that are useful in spreadsheet form and must later be
removed again or at least recognized as derived content.

=== Current Implementation Today

The current implementation already supports the basic idea and now also supports
multiple helper columns per FK.

Today the operational helper path is mainly shaped by:

* `add_fk_helpers`
* `remove_fk_helpers`
* prefix- and naming-based recognition in validation and styling

`bootstrap_meta` is already part of the broader workbook metadata direction and
already has its own precedence model, but it is not yet the active source of
helper-field resolution in the current helper implementation.

Today the behavior is roughly this:

* `add_fk_helpers` detects FK columns and adds helper columns
* helper field selection is driven mainly by step defaults such as
  `helper_fields_by_fk`, `helper_fields_by_target`, `helper_fields`, and
  `label_field`
* `remove_fk_helpers` removes helper columns by prefix
* validation and styling still rely heavily on helper prefixes and materialized
  column names

So the current implementation is useful, but still carrier- and naming-driven.

It is not yet a fully metadata-driven helper-column model.

=== Current Detection and Selection Rules

==== FK Detection Today

Today FK detection is still mainly convention-based.

The established convention is currently based on names such as
`id_(employees)`.

This naming convention is not the final model.

It is a practical detection aid for the current phase.

==== Helper Field Selection Today

Today helper field selection already has a concrete precedence chain inside the
helper logic.

Current order:

. `helper_fields_by_fk`
. `helper_fields_by_target` by normalized sheet key
. `helper_fields_by_target` by original sheet name
. `helper_fields`
. `label_field` as legacy fallback

This is important because it means the project already has step-specific
precedence today.

There is not one single universal precedence ladder for all metadata and all
steps.

=== Existing Step-Specific Precedence

==== `add_fk_helpers`

`add_fk_helpers` resolves helper fields from its own step defaults.

Today this is still the active source of helper-field resolution.

==== `bootstrap_meta`

`bootstrap_meta` already has its own precedence model.

Current order:

. `profile_defaults`
. persisted metadata
. `cli_overrides`

This is relevant for the broader metadata direction around helper columns.

It should not be read as saying that `bootstrap_meta` already drives current
helper-field selection inside `add_fk_helpers`.

==== Consequence

Precedence must be described per step.

A single global statement such as "user input always wins" would be too crude
and would already be false for some cases.

=== General Guardrail

The most useful general guardrail is this:

* heuristics may fill gaps
* heuristics must not silently contradict existing metadata

This matters especially on the second or third roundtrip.

If the first run relies on heuristics and the result is not persisted, later
runs may interpret the same workbook differently because the carrier state has
changed.

That creates avoidable inconsistency.

=== Two Different Questions

Helper columns force two different questions that should not be mixed together.

==== Question 1: What may be used as helper source data?

This belongs to the target table.

In the running example, this belongs to `employees`.

Examples:

* which column is the PK of `employees`
* which columns of `employees` may be exposed as helper fields
* which helper fields are the defaults if no FK-specific override is given

==== Question 2: Which concrete columns on the current sheet are helper columns?

This belongs to the table containing the FK.

In the running example, this belongs to `desks`.

Examples:

* `_employees_first_name` is a helper column
* it belongs to FK column `id_(employees)`
* it points to target table `employees`
* it materializes source field `first_name`

So the answer is deliberately split:

* helper capability belongs to the target table
* concrete helper-column materialization belongs to the table containing the FK

=== Minimal Metadata Shapes

==== Target-Side Helper Capability

Example:

[source,yaml]
----
sheets:
  employees:
    fk_helpers:
      available_fields: [first_name, last_name, birth_date]
      default_fields: [first_name, last_name]
----

This says what the target table can offer as helper information.

This is canonical helper intent.

==== Derived Operational Helper Columns on the FK Sheet

Example:

[source,yaml]
----
_meta:
  derived:
    sheets:
      desks:
        helper_columns:
          - column: _employees_first_name
            fk_column: id_(employees)
            target: employees
            value_field: first_name
          - column: _employees_last_name
            fk_column: id_(employees)
            target: employees
            value_field: last_name
----

This says which helper columns are currently materialized on the current working
sheet.

This is derived operational metadata, not canonical sheet intent.

The important distinction is that this lives under the existing `_meta` carrier
in a derived namespace, rather than under canonical sheet intent such as
`_meta["sheets"]`.

That second structure is not fully implemented today.

It is the direction for the next slices.

=== What Is Already True Today

* helper enrichment and helper cleanup exist as real pipeline steps
* multiple helper columns per FK now exist
* helper field order is configurable and stable
* there is already step-specific precedence in helper-field resolution
* there is already step-specific precedence in `bootstrap_meta`

=== What Is Not Yet True Today

* helper cleanup is not yet driven by first-class persisted helper-column
  metadata
* helper identification still depends mainly on prefixes and naming conventions
* helper provenance is not yet modeled as a durable explicit structure in
  working metadata
* sheet rename and helper-column rename are not handled as safe supported
  operations

These limits should be stated explicitly, not hidden behind generic wording.

=== Next Slice

The next slice should stay narrow.

It should not jump directly to formulas, rename support, or a universal
metadata engine.

The next slice should aim for these improvements:

. make the current state explicit in the docs
. define a small helper metadata shape that is concrete enough to guide code
. distinguish clearly between target-side helper capability and current
  helper-column materialization
. persist enough interpreted structure to reduce second-roundtrip drift
. keep the current naming conventions as fallback aids, not as the only source
  of truth

=== First Conservative Policy

For the next slice, a restrictive contract is acceptable.

That means:

* helper columns present redundant information for easier editing
* manual edits to helper values are not treated as source truth
* helper values may be ignored or overwritten on re-enrichment
* renaming helper columns is unsupported
* renaming relevant sheets is unsupported

This is not great UX, but it is a feasible first contract.

=== Cleanup Policy

The current cleanup step `remove_fk_helpers` removes by prefix.

That is good enough for the current implementation phase, but it is also the
main conceptual weakness.

The longer-term direction is:

* prefix may still help with detection
* cleanup should eventually trust explicit helper-column metadata first
* prefixes should remain fallback aids and disambiguation tools

=== Naming Conventions

FK naming and helper naming serve different purposes and should not be merged
into one vague naming rule.

==== FK Naming

FK naming helps the system detect that a column is intended as a foreign key.

Example:

* `id_(employees)`

==== Helper Naming

Helper naming helps distinguish derived helper columns from ordinary payload
columns and helps cleanup avoid collisions.

Example:

* `_employees_first_name`

So even if both are naming conventions, they solve different problems.

=== Current State, Next Slice, Target State

==== Current State

* helper logic is useful and works for controlled conventions
* multiple helper columns per FK are implemented
* the implementation is still mainly naming- and prefix-driven

==== Next Slice

* document the current implementation honestly
* add a minimal concrete metadata shape
* begin shifting from prefix-only recognition toward metadata-backed
  recognition
* keep scope small and conservative

==== Target State

* target-side helper capability is explicit
* concrete helper-column materialization is explicit
* cleanup and reimport rely primarily on metadata, not on prefixes alone
* spreadsheet-first roundtrips become more stable because interpreted structure
  is persisted

=== Open Questions

For the immediate next slice, the follow-up FTR already concretizes two points:

* initial interpreted helper provenance should be written by `add_fk_helpers`
* `remove_fk_helpers` should remove helper columns and synchronize that derived
  provenance during cleanup

Beyond that first provenance-backed cleanup slice, the following questions are
still open and should be kept concrete rather than generic:

* which additional helper-related metadata, beyond the minimal
  `helper_columns` provenance, should survive `remove_fk_helpers`
* which operations remain unsupported in the first safe contract
* how much rename repair or orphan cleanup should ever be attempted

=== Practical Reading Rule

If a sentence in this topic sounds too abstract, rewrite it so that a reader can
map it to one of these concrete things:

* `add_fk_helpers`
* `remove_fk_helpers`
* `bootstrap_meta`
* the `desks` table containing the FK
* the `employees` target table
* a concrete helper column such as `_employees_first_name`

If that mapping is missing, the wording is probably too generic to guide
implementation.
