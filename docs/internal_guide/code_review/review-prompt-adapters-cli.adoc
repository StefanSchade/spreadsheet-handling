== Kontext-Seed (einmalig am Session-Anfang einfügen)

----
Projekt: spreadsheet-handling — Python-Framework zur Transformation von
strukturierten Daten (JSON/XML/YAML) in tabellenorientierte Spreadsheets
(XLS/ODS). Roundtrip-fähig, deterministisch, CLI-Tool. Hexagonale Architektur
mit IR-Layer (rendering/) + Pipeline-Engine (pipeline/).

=== Offene Findings aus Phase 1 + 2 (relevant für diese Session)

D-001 [CRITICAL] — domain/ importiert rendering/formulas.py; Wurzelursache:
  formulas.py liegt im falschen Layer. Fix: formulas.py → core/. Danach
  domain ⊥ rendering-Guard ergänzen.

IR-002 [SHOULD-FIX] — rendering/formulas.py nach core/ verschieben (= Fix D-001).

IR-006 [SHOULD-FIX] — Render-Pfad mutiert Eingaben:
  layout_composer._add_legend_blocks schreibt spec["resolved"] in Aufrufer-meta
  (shallow copy → akkumuliert bei mehrfachem Aufruf);
  frame_selection._resolve_formula_sheet_names mutiert Eingabe-Frames in-place.
  Relevant: falls orchestrator.py dieselben frames/meta mehrfach weitergibt.

IR-008 [SHOULD-FIX] — WriteMeta.kv persistiert meta als str(dict) (Python repr),
  nicht JSON. Read-Path liest per ast.literal_eval. In dieser Session prüfen:
  openpyxl_renderer.py / odf_renderer.py (Schreiben) und openpyxl_parser.py /
  odf_parser.py (Lesen) — ist das der echte Serialisierungspfad?

IR-013 [NICE-TO-HAVE] — MetaPass reserviert exported_at, befüllt es nie.
  In dieser Session prüfen: setzt CLI oder orchestrator.py den Zeitstempel?
  Wenn ja: datetime.now() direkt (Determinismus-Bruch) oder injizierbar?

=== Architekturelle Invarianten (aus Phase 1 + 2 bestätigt)

  pipeline/   ⊥  rendering/   (kein cross-import, bestätigt)
  rendering/  ⊥  domain/      (kein Import von domain in rendering, bestätigt)
  rendering/  ⊥  openpyxl/odf (Backend-Libs nur in io_backends/, bestätigt)

  Formula Boundary: rohe Formel-Strings (=VLOOKUP, A1-Notation) nur in
  io_backends/xlsx und io_backends/ods — rendering/ + domain/ + pipeline/ kennen
  nur FormulaSpec-Wertobjekte.

  Render-Flow-Start: io_backends/spreadsheet_contract.py::
  build_spreadsheet_render_plan() ruft compose_workbook() auf — NICHT die Pipeline.

  Nicht-Spreadsheet-Backends (json, csv, yaml, xml) sollen rendering/
  komplett umgehen (Primer-Behauptung — in dieser Session verifizieren).

Reviewe gegen das Erwartungsmodell aus Schritt 2. Fokus:

1. *Infrastruktur-Klarheit* — Was ist die genaue Rolle von
   `_interfaces.py` vs. `spreadsheet_contract.py` vs. `base.py`?
   Gibt es Überlappungen oder Widersprüche? Ist klar, welches Modul
   ein neues Backend implementieren muss?

2. *router.py Open/Closed* — Wie wird auf Backends dispatcht?
   Statisches Dict, Klassenregistrierung, Entry-Points? Kann ein
   Plugin-Backend hinzugefügt werden ohne `router.py` zu ändern?
   Ist der Nicht-Spreadsheet-Pfad (json/csv/yaml/xml) tatsächlich
   `rendering/`-frei?

3. *Parser-Grenze (IR-008-Kontext)* — Stoppen `openpyxl_parser.py`
   und `odf_parser.py` bei `WorkbookIR`? Oder erzeugen sie direkt
   `Frames`? Wie wird `workbook_meta_blob` aus dem Spreadsheet
   gelesen — als String, als JSON, als `ast.literal_eval`?

4. *Renderer + WriteMeta (IR-008)* — Wie persistiert
   `openpyxl_renderer.py`/`odf_renderer.py` die `WriteMeta`-Op?
   Wird `kv`-Dict als `str()` geschrieben oder als JSON-String?
   Ist der Roundtrip (Write → Read → gleiche Daten) deterministisch?

5. *exported_at (IR-013)* — Setzt `cli/`, `application/orchestrator.py`
   oder ein Backend den Wert? Wenn ja: `datetime.now()` direkt
   (Determinismus-Bruch) oder über injizierbaren Parameter?

6. *Mutation durch IR-006* — Verwendet `application/orchestrator.py`
   dieselben `frames`/`meta`-Objekte nach dem Schreiben weiter
   (z.B. für zweiten Output oder Roundtrip-Prüfung)? Falls ja, ist
   die In-Place-Mutation durch `compose_workbook` ein konkreter Bug.

7. *Backward-Compat-Shim* — Ist `src/spreadsheet_handling/
   orchestrator.py` nur ein Re-Export auf `application/orchestrator.py`?
   Wird er noch von externen Nutzern verwendet (PyPI-public)?
   Kann er entfernt werden?

8. *CLI-Vollständigkeit* — Bildet die CLI alle öffentlichen Features
   ab? Gibt es Funktionen die nur programmatisch, nicht per CLI
   erreichbar sind? Sind Fehler (falsche YAML-Konfig, fehlende
   Eingabedatei) mit nutzbaren Fehlermeldungen abgefangen?

9. *Sind die Files so geschnitten, dass sie gut verständlich durch
   Mensch und KI sind?* Klarer Purpose pro Modul, Vermeidung hoher
   Komplexität oder grosser Länge?
