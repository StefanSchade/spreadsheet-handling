== Context

----
Project: spreadsheet-handling — Python framework, hexagonal architecture,
IR layer, pipeline engine, CLI tool, PyPI beta. Sole dev / evening project.

This session derives Feature/Task Records (FTRs) from a completed
code review. The review produced a consolidated findings document at:

  docs/internal_guide/code_review/review-consolidation.adoc

The review identifier is REVIEW-001. All FTR names derived here must
include REVIEW-001 in their filename so the connection is traceable:

  FTR-REVIEW-001-<SLUG>-<PRIORITY>.adoc

FTRs live flat in: docs/general_info/backlog/

Two FTR styles exist in this project — use the appropriate one:

SURGICAL FTR (most review findings):
  - Precise scope, verified current state, clear acceptance criteria
  - Sections: Purpose, Current State (verified), Scope, Non-Goals,
    Risk, Acceptance
  - Example: FTR-ENGINE-RETIREMENT-P3E.adoc

TRACKING FTR (cross-cutting concerns):
  - Living document, parity matrix, no single fix scope
  - Sections: Purpose, Scope, Non-Goals, Acceptance
  - Example: FTR-ODS-CONSUMER-FEATURE-PARITY-P4A.adoc

Priority scale used in this project (from existing FTRs):
  P3x = near-term, well-scoped
  P4A = medium-term, consumer-visible
  P5  = longer-term / nice-to-have

=== Required sections per FTR (surgical style)

----
===== FTR-REVIEW-001-<SLUG>-<PRIORITY> — <short title>

*Status:* Open
*Priority:* <Px>
*Area:* <layer / component>
*Kind:* Refactoring / Bugfix / Hygiene / Architecture
*Review source:* REVIEW-001, Cluster <Cx>, Finding IDs: <D-0xx, IR-0xx, A-0xx>

==== Purpose

<2-3 sentences: what problem this solves and why it matters>

==== Current State (verified)

<Bullet list of what is actually in the code today — specific module
paths, line numbers where known, concrete symptoms. Use the review
findings as source — do not invent.>

==== Scope

<Numbered list of concrete changes — files to modify/delete/create,
imports to update, tests to add/change. Specific enough that an agent
can execute without additional context.>

==== Non-Goals

<What is explicitly out of scope — prevents scope creep>

==== Dependencies

<Other FTRs that must be completed first — use the REVIEW-001 FTR names>

==== Risk

<One sentence on risk level and why>

==== Acceptance

<Bullet list of verifiable acceptance criteria — grep-able, test-
runnable, or structurally checkable where possible>
----

=== Quick Wins FTR (checklist style)

The Quick Wins FTR uses the same header but replaces the body with
a checklist of self-contained tasks, each with:
- [ ] Task description (1 sentence)
- File(s) to change
- Acceptance: what grep/test confirms it done

=== Content rules

- Language: English throughout.
- "Current State": copy specific evidence from the review (module
  paths, finding IDs, line numbers). Do not paraphrase vaguely.
- "Scope": numbered steps, specific enough for an agent to execute.
  If a step touches a file, name the file.
- "Acceptance": prefer grep-able or test-runnable criteria over
  prose assertions. Model: "All tests pass; `grep -r 'from.*rendering.*
  formulas' src/` returns no matches."
- Do NOT add scope that is not in the review findings — stay faithful
  to what was actually found.
- If a finding references a specific architecture test that must be
  updated, name it in Scope and Acceptance.
