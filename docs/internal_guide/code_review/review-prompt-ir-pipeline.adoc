  rendering/          ← IR-Layer (Intermediate Representation)
    ir.py             — IR-Datenstruktur (Kern-Kontrakt)
    plan.py           — Rendering-Plan (was wird gerendert)
    flow.py           — Ablaufsteuerung des Rendering
    frame_selection.py — Auswahl der zu rendernden Frames
    formulas.py       — Formel-Behandlung
    workbook_projection.py — Projektion auf Workbook-Struktur
    passes/           — Rendering-Passes (Pipeline über IR)
      core.py
      formula_pass.py
      meta_pass.py
      post_pass.py
      style_pass.py
      validation_pass.py
    composer/

  pipeline/           ← Workflow-Engine (Mini-Orchestrator)
    types.py          — Typen/Contracts der Pipeline
    config.py         — Step-Konfiguration
    registry.py       — Step-Registry (YAML-Name -> Factory)
    steps.py          — Step-Implementierungen
    runner.py         — Pipeline-Runner
----

Reviewe gegen das Erwartungsmodell aus Schritt 2. Fokus:

1. *IR als Vertrag* — Ist `ir.py` eine stabile, reine Datenstruktur ohne
   Seiteneffekte? Oder enthält sie Logik die woanders hingehört?
   Kennt die IR ihre eigenen Consumer (Adapter, Writer)?

2. *Passes-Korrektheit* — Folgen alle Passes dem gleichen Muster
   (Input IR → transformierte IR)? Gibt es Passes die direkt in
   Output-Formate schreiben (Grenzverletzung)? Sind die Passes
   unabhängig voneinander oder gibt es implizite Reihenfolge-Abhängigkeiten?

3. *Formula Boundary* — Wo tauchen rohe Formel-Strings auf?
   Hält `formulas.py` die Grenze sauber oder leckt A1-Notation
   in andere Module?

4. *Pipeline/Rendering-Grenze* — Ruft die Pipeline direkt Rendering-
   Funktionen auf? Gibt es eine saubere Übergabe (z.B. Plan-Objekt)
   oder ist die Grenze implizit?

5. *Registry-Konsistenz* — Sind alle Steps in `registry.py` registriert?
   Gibt es Steps in `steps.py` die nicht in der Registry auftauchen
   (oder umgekehrt)? Ist die Registry erweiterbar ohne bestehenden
   Code zu ändern (Open/Closed)?

6. *Determinismus* — Gibt es in Passes oder Pipeline-Steps
   nicht-deterministische Konstrukte (Dict-Iteration, Set-Reihenfolge,
   datetime, uuid ohne seed)?

7. *meta_canonical vs. meta_rendering* — Wird die Trennung im Code
   tatsächlich eingehalten oder gibt es Stellen wo canonical-Meta
   auf dem Rendering-Pfad mutiert wird (oder umgekehrt)?

8. *Sind die Files so geschnitten, dass sie gut verständlich durch
   Mensch und KI sind?* Klarer Purpose pro Modul, Vermeidung hoher
   Komplexität oder grosser Länge?
