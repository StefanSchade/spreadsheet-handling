== Schritt 2 — Priorisierte Gesamt-Übersicht

Ordne alle Findings nach Priorität. Kriterien:

* *CRITICAL:* Korrektheitsproblem, Roundtrip-Bruch, oder
  Release-blockierend für PyPI-Beta-Nutzer.
* *SHOULD-FIX:* Technische Schuld mit konkretem Risiko
  (Determinismus, Mutation, Layer-Verletzung).
* *HYGIENE:* Strukturprobleme die Mensch- und KI-Lesbarkeit
  verschlechtern (tote Dateien, fehlende Header, Modulgröße).
* *NICE-TO-HAVE:* Verbesserungen ohne Dringlichkeit.

Identifiziere ausserdem:

1. *Cluster* — Welche Findings hängen zusammen und sollten im
   selben Commit/PR behoben werden?
   (Beispiel: IR-002 + D-001 sind ein Cluster — formulas.py
   verschieben löst beide.)
2. *Abhängigkeiten* — Welche Findings müssen vor anderen behoben
   werden? (Beispiel: IR-002 vor dem neuen Guard-Test.)
3. *Quick Wins* — Welche Findings sind <30 min Aufwand mit hohem
   Nutzen?

1. *Korrektheit* — Beschreibt der Primer die Architektur so wie sie
   im Code *tatsächlich* ist (Ist), oder so wie sie sein *sollte*
   (Soll)? Wo weicht er ab?
   Konkrete Prüfpunkte aus den Reviews:
   - Beschreibt der Primer die "zwei Flüsse"-Trennung
     (Pipeline ⊥ Rendering) — und stimmt das mit dem Befund überein?
   - Ist `rendering/formulas.py` im Primer als Teil von `rendering/`
     beschrieben, obwohl es nach `core/` gehört?

   - Ist `workbook_meta_blob`-Serialisierung (IR-008) beschrieben?
   - Sind die toten Stub-Dateien in `passes/` (IR-003) im Primer
     als bewusste Struktur dargestellt?

   aufgedeckt haben?

   - Sind die erlaubten Import-Richtungen für alle Layer als
     ausführbare Regel formuliert (oder nur als Prosa)?

3. *Aktualität für KI-Agenten* — Gibt es Stellen in `ai_info/` die

werden soll, sodass ein nachfolgender Agent (oder Mensch) die
Änderungen direkt umsetzen kann.

* *Datei* — welches `docs/ai_info/`-Dokument
* *Abschnitt* — wo genau (Überschrift oder Zeilennummer)
* *Art* — Korrektur / Ergänzung / Streichung
* *Inhalt* — was soll stehen (Stichpunkte, kein fertiger Text)
