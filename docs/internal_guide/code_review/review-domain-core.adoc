=== Hypothesen

_Liste der vermuteten Problemstellen — werden in Schritt 3 verifiziert_

* *H1 — Dreifaches FK-Konzept.* `core/fk.py` (339 Z.), `domain/transformations/fk_helpers.py`
  (440 Z.), `domain/validations/fk_helpers.py` (251 Z.). Der dedizierte Architektur-Test
  `test_architecture_guard_fk_helpers.py` signalisiert, dass diese Grenze historisch
  umkämpft war. Verdacht: überlappende Logik, unklare Verantwortungsteilung, evtl.
  Re-Implementierungen statt geteilter Helfer.
* *H2 — `core/fk.py` enthält Domänenlogik.* Für ein "stateless leaf utility" auffällig
  groß; FK-Erkennung per Namenskonvention/Heuristik ist eine fachliche Regel und gehörte
  ggf. nach `domain/`.
* *H3 — Framework-Leak in Domain via YAML/Cell-Codecs.* `domain/structured_yaml.py`,
  `domain/yaml_overrides.py`, `domain/transformations/cell_codec.py` könnten direkt von
  `yaml`/`openpyxl`-Typen abhängen — hexagonale Grenzverletzung (Domain -> Bibliothek).
* *H4 — Spreadsheet-Vokabular in Domain.* `domain/workbook_views.py` und ggf.
  `workbook`-/`sheet`-Begriffe in `tabular_views.py`/`join_views.py` koppeln Domain an
  Spreadsheet-/IR-Konzepte, die laut Layer-Map erst oberhalb der Domain leben.
* *H5 — Meta-Regeln in der falschen Schicht.* `domain/meta_bootstrap.py` merged Meta;
  der Architektur-Primer sagt aber "generische Meta-Regeln (inheritance, merge, priority)
  gehören in den IR-Layer, nicht in Domain oder Adapter". Grenzfall prüfen.
* *H6 — Große Module schaden Verständlichkeit (Mensch + KI).* `reference_validations.py`
  (725), `cell_codec.py` (631), `join_views.py` (567), `discriminator_split.py` (562),
  `enrich_lookup.py` (562), `compact_multiaxis.py` (468), `helper_policies.py` (455),
  `graph_validations.py` (446), `fk_helpers.py`/transformations (440). Mehrere Dateien
  über der Schwelle, ab der ein Agent nicht mehr "in einem Stück" arbeiten kann.
* *H7 — Determinismus-Risiken.* Transformationen arbeiten auf Dicts/`Frames`; Verdacht auf
  unsortierte Dict-/Set-Iteration, evtl. `datetime.now()` in Provenance-Pfaden. In
  Schritt 3 gegen `semantic_invariants`-Tests prüfen.
* *H8 — Fehlermodell uneinheitlich.* `validations/` liefert `Finding`-Listen (Result-artig),
  `transformations/` wirft vermutlich Exceptions — zu klären, ob Domain-spezifische
  Exception-Typen existieren oder generische `ValueError`/`KeyError` verwendet werden.

=== Antworten auf die Leitfragen (Schritt 1)

*1. Was lebt in `domain/`?* Drei klar getrennte Sub-Pakete — `transformations/`
(Pipeline-Schritte, die `Frames` umformen), `validations/` (Checks, die `Finding`-Listen
liefern, plus `findings.py` als Value Object/Policy), `extractions/` (Teil-Frame-Auswahl).
Daneben Top-Level-Services: `frame_lifecycle`, `helper_policies`, `meta_bootstrap`,
`structured_yaml`, `workbook_views`, `yaml_overrides`. Echte Entities/Value Objects gibt es
kaum — `findings.py` ist der einzige offensichtliche VO-Träger; ansonsten ist die Domain
funktions-/serviceorientiert (Transformationen über dem universellen `Frames`-Carrier).
Ports/Interfaces sind in der Dateiliste nicht sichtbar — Verdacht, dass die Domain keine
eigenen abstrakten Ports definiert (in Schritt 3 verifizieren).

*2. Was lebt in `core/`?* Durchweg kleine, zustandslose Hilfen: `flatten`/`unflatten`,
`df_build`, `indexing`, `paths`, `refs`. Das passt zur "leaf utility"-Beschreibung. Einzige
Ausnahme: `fk.py` (339 Z.) — Umfang und Thema (FK-Erkennung) deuten auf fachliche Logik hin,
die in `domain/` besser aufgehoben wäre (H2).

*3. Welche Architecture-Tests existieren?* Vier Cluster: (a) *dependency_guards* — wer darf
wen importieren, inkl. des FK-Helper-Sonderfalls; (b) *carrier_boundary* /
*semantic_invariants/meta_carrier* — `Frames` und Meta werden durchgereicht, nicht von
Zwischenschichten interpretiert; (c) *formula_boundary* — rohe Formel-Strings bleiben
diesseits einer Grenze; (d) *parse_contract* / *spreadsheet_contract* / *pipeline_semantics*
— Read-Path-, Adapter- und Step-Kontrakte. Für diese Phase zählen vor allem (a) und (b)
sowie `formula_boundary`.

*4. Erste Hypothesen zu Grenzverletzungen:* siehe H1–H5 oben — Schwerpunkt auf
FK-Schichtung, YAML-/Cell-Codec-Framework-Leaks, Spreadsheet-Vokabular in der Domain und
Meta-Merge in der falschen Schicht.

*5. Modularisierung für token-effiziente Entwicklung.* Vorschläge (Reihenfolge = Aufwand/Nutzen):

** *Modul-Header-Konvention erzwingen* — jedes `*.py` startet mit 1–3 Zeilen Docstring
   "Purpose / Layer / darf importieren: …". Ein Agent erfährt per `head -5`, was ein Modul
   tut, ohne es zu lesen. Günstig, sofort wirksam.
** *Pro-Sub-Paket-Index* — `domain/transformations/_ai_info.adoc` (oder `CLAUDE.md`), das
   nur die Schritte dieses Pakets und ihre Kontrakte listet. Ein Agent, der einen
   Transformations-Step ändert, lädt diesen Index statt des ganzen Domain-Verzeichnisses.
** *Stabile öffentliche Sub-Paket-API* — `domain/transformations/__init__.py` und
   `domain/validations/__init__.py` exportieren die wenigen öffentlichen Symbole; Aufrufer
   (und Agenten) importieren von dort statt tiefer Pfade. Reduziert die Oberfläche, die ein
   Agent verstehen muss, und macht Refactorings innerhalb des Pakets unsichtbar.
** *Große Dateien in Pakete zerlegen* — `cell_codec.py` -> `cell_codec/` mit
   `encode.py`/`decode.py`/`types.py`; analog `reference_validations.py`, `join_views.py`,
   `discriminator_split.py`, `enrich_lookup.py`. Ziel: keine Datei > ~300 Zeilen, damit ein
   Agent ein Modul in einem Read-Window vollständig erfasst. Höchster Aufwand, höchster
   Langzeit-Nutzen.
** *FK-Konzepte disambiguieren* — entweder unter einem Namespace bündeln (`fk/detect.py`,
   `fk/helper_columns.py`, `fk/checks.py`) oder eindeutig umbenennen
   (`core/fk_detect.py`, `domain/transformations/fk_helper_columns.py`,
   `domain/validations/fk_checks.py`). Beseitigt die wiederkehrende Verwechslung dreier
   gleichnamiger Module — sowohl für Menschen als auch für die KI-Suche.
** *Architektur-Grenzen als ausführbare Regel sichtbar machen* — die `dependency_guards`-Tests
   so dokumentieren/benennen, dass ein Agent vor einer Änderung weiß, welche Imports erlaubt
   sind, ohne den Test zu lesen.

== Schritt 3 — Findings

=== Zusammenfassung

`domain/` ist überwiegend sauber: funktional/serviceorientiert, deterministisch (keine
`datetime.now()`/`random`/`uuid` in `domain` oder `core`; Dict-Iteration konsequent
first-seen oder explizit `sorted(...)`), gut testabgedeckt, ein klarer Zweck pro Modul.
Es gibt aber *eine echte hexagonale Verletzung* — zwei Domain-Module importieren
`spreadsheet_handling.rendering.formulas` — die von keinem Architektur-Guard erkannt wird
(Lücke L1). `core/` ist bis auf `core/fk.py` echtes Blatt-Utility; `core/fk.py` trägt die
FK-Namenskonvention (eine Domänenregel) und die FK-Helper-Primitive — genau das, was der
FK-Guard selbst "gehört nach `domain.transformations.fk_helpers`" nennt. Daneben:
mehrere überlange Dateien, dreifache `fk_helpers`/`fk`-Namensüberlappung, zwei parallele
Finding-Modelle, ein paar mislabelte Module und Rest-Kompat-Shims.
