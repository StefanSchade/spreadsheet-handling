= Finding-Model Consolidation Notes (FTR-FINDING-MODEL-CONSOLIDATION-P5)

Date: 2026-05-18

== Decision: one shared record + one serializer, not one universal model

The inventory shows the duplication is *finding-frame serialization*, not a
single uniform record. The smallest behavior-preserving consolidation is:

. Extract the genuinely shared mechanic â€” the column constant, the record
  shape that two producers share verbatim, the `pd.DataFrame([...],
  columns=...)` serializer, the empty-frame contract, and the flat failure
  message â€” into one small domain module.
. Leave genuinely different records (`ReferenceFinding`,
  `findings.py::Finding`) as distinct, but route the *serialization* of
  `ReferenceFinding` through the one shared serializer so the mechanic is
  single-sourced.

This deliberately rejects a single "universal Finding" or a diagnostics engine
(explicit FTR non-goal and REVIEW-002 anti-framework constraint).

== Placement: `domain/finding_frame.py`

Domain-level, beside `frame_lifecycle.py`, `helper_policies.py`,
`workbook_views.py`. Rationale:

* The shared model belongs in the domain layer (FTR design constraint).
* Both `domain/transformations/*` and `domain/validations/*` consume it;
  a neutral domain-level module avoids a `transformations â†’ validations`
  cross-subpackage import question entirely.
* `frame_lifecycle.py` already proves this exact placement + import pattern is
  guard-clean (the domain-transformation boundary/cross-family guards only
  constrain the `fk_helpers` / `discriminator_split` packages; `findings.py`
  was rejected as the home because its `Finding` is a different concern and
  the FTR forbids touching severity behavior).

== Shared API (final)

* `FINDING_COLUMNS` â€” the 7-col base (`rule_type, frame, columns, row_index,
  value, severity, message`).
* `Finding` â€” frozen dataclass, 7-col shape, `to_record()` = the
  resource-override/derived-column rendering (`", ".join` columns, labelled
  `row_index`, `" | ".join` value). `severity` defaults to `"warn"`,
  behaviourally inert (all call sites pass it explicitly).
* `row_index_label(row_index)` â€” scalarâ†’`str`, tupleâ†’`", ".join(map(str,â€¦))`.
* `findings_to_frame(findings, *, columns=None)` â€” generic over any
  `to_record()`-bearing record; `columns=None` â†’ `FINDING_COLUMNS`. Empty
  input still yields a zero-row frame with exactly `columns` (schema-stable
  empty-frame contract preserved for *both* the 7-col and 9-col schemas).
* `simple_failure_message(findings, *, heading)` â€” the flat, uncapped
  `"{heading}:\n  - {rule_type}: {message}"` block.

`FindingRecord` is a `Protocol` (just `to_record`) so the serializer accepts
`ReferenceFinding` without importing it (no domain cycle).

== Per-module conversion

| `resource_overrides`

  `findings_to_frame` now: `return _serialize_findings(findings,
  columns=FINDING_COLUMNS)` where `_serialize_findings` is the shared
  serializer (imported aliased to avoid shadowing the public name).

| `findings.py`
| Untouched. Documented rationale: different semantic model, no serialization
  to consolidate, collapsing it = severity-behavior change (forbidden).
