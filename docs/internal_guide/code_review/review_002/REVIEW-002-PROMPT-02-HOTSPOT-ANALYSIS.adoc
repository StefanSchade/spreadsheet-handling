== Purpose

Perform a focused architectural hotspot review for large or semantically dense
domain modules that may become future refactoring or package-split candidates.

The goal is NOT to immediately refactor code.

The goal IS to determine:

* which modules are healthy despite size,
* which modules are accumulating unrelated responsibilities,
* which seams are already stable enough for behavior-preserving extraction,
* which modules require additional conceptual clarification before any split,
* and which modules should explicitly remain cohesive despite internal complexity.

This review is intended to generate:

* architectural guidance,
* hotspot prioritization,
* future FTR candidates,
* and explicit non-goals / protected seams.

'''

== Scope

Primary review targets:

* `domain/transformations/xref_crosstable.py`
* `domain/transformations/reference_validations.py`
* `domain/transformations/cell_codec.py`
* `domain/transformations/join_views.py`
* `domain/transformations/enrich_lookup.py`
* `domain/workbook_views.py`
* `domain/transformations/resource_overrides.py`
* `domain/transformations/compact_multiaxis.py`
* `domain/transformations/derived_column_policy.py`
* `domain/transformations/tabular_views.py`
* `domain/transformations/helper_policies.py` (optional)

Secondary references allowed where needed:

* architecture guard inventory
* ADR documents
* existing package-split analyses
* workbook-view / projection FTRs
* meta registry
* lifecycle metadata
* current spreadsheet contract guards

'''

== Required Output Structure

For EACH reviewed module:

. Public/runtime surface inventory
. Responsibility map
. Internal coupling assessment
. External dependency and registry coupling
. Candidate package seams (if any)
. Architectural risk assessment
. Recommended disposition
. Potential future FTR outline
. Explicit non-goals / forbidden anticipatory abstractions

'''

== Mandatory Classification

Every reviewed module MUST receive exactly one classification:

[cols="1,4",options="header"]
|===
| Classification | Meaning

| PROCEED
| Large but mechanically separable with low semantic risk.

| PROCEED WITH CONSTRAINTS
| Split feasible, but only under explicit architectural restrictions.

| ANALYZE FIRST
| Current responsibilities or semantics are insufficiently understood.

| DEFER
| Refactoring likely useful later, but blocked by unresolved upstream design work.

| DO NOT SPLIT
| Complexity is inherently cohesive and package splitting would likely increase architectural drift.
|===

'''

== Review Constraints

* Prefer behavior-preserving analysis over redesign proposals.
* Do NOT propose new frameworks unless the existing architecture clearly fails.
* Avoid introducing abstract "engines", "managers", or orchestration layers without demonstrated need.
* Avoid speculative package decomposition.
* Avoid extracting tiny utility packages that weaken locality.
* Avoid anticipatory conflict-resolution seams unless conflict handling already exists.
* Respect existing architectural ownership boundaries:
** Pipeline orchestrates.
** Domain owns semantics.
** Rendering owns spreadsheet semantics.
** IO adapters own backend syntax.

'''

== Important Architectural Questions

The review should explicitly assess:

* Is the module large because it is overloaded, or because the domain itself is intrinsically complex?
* Are responsibilities merely adjacent, or truly independent?
* Does the module contain hidden bidirectional invariants?
* Are there implicit symmetry assumptions (`merge(split(x)) == x` style)?
* Are there canonical metadata contracts that would make splitting dangerous?
* Are ordering semantics or normalization semantics distributed across helpers?
* Would splitting create premature framework seams?
* Would splitting accidentally pre-empt future FTR ownership?
* Is there hidden coupling through tests, registries, or docs?

'''

== Explicit Non-Goals

* No implementation changes.
* No production-code refactors.
* No registry rewrites.
* No YAML-surface redesign.
* No new orchestration framework.
* No generalized plugin architecture.
* No automatic migration tooling.
* No conflict-resolution framework unless already implemented.

'''

== Expected Deliverables

* One architectural assessment section per reviewed module.
* One consolidated hotspot ranking.
* One consolidated future-FTR candidate list.
* One consolidated "avoid splitting these" section.
* Explicit identification of modules where cohesion is preferable to modularization.

'''

== Suggested Final Summary

The final summary should answer:

* Which modules are currently the highest-value refactor targets?
* Which modules are deceptively complex and should remain cohesive?
* Which seams are already mature enough for package splits?
* Which areas still lack architectural clarity?
* Which future FTRs are likely worthwhile versus architectural churn?

'''

== Recommended Tone

* skeptical
* architecture-first
* behavior-preserving
* anti-framework
* anti-speculative
* explicit about risks and hidden coupling

The review should optimize for long-term maintainability and conceptual integrity, not for maximal decomposition.
