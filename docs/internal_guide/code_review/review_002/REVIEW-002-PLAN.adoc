= REVIEW-002 Plan

== Purpose

REVIEW-002 is a longitudinal follow-up review after the major stabilization and
maintainability wave completed after REVIEW-001.

The goal is not another broad “find problems everywhere” pass.
The goal is to reassess architectural maturity, maintainability trajectory,
remaining structural risks, and refactoring readiness after:

* workbook roundtrip/recomposition work;
* ODS parity consolidation;
* prototype consumer integration;
* demo integration;
* transformation package modularization;
* architecture guardrails;
* registry and metadata stabilization work.

The review should explicitly distinguish:

* accidental complexity vs. inherent complexity;
* healthy modularization vs. fragmentation;
* real structural risks vs. theoretical architecture purity concerns.

== Review Structure

REVIEW-002 is intentionally split into several focused passes instead of one
large review.

Reasoning:

* reduces context dilution;
* improves depth per topic;
* avoids broad speculative recommendations;
* keeps findings traceable to a specific review focus.

== Planned Passes

=== Pass 1 -- Architecture Follow-up

Focus:

* reassess REVIEW-001 findings against the current repository state;
* evaluate architectural maturity trend;
* evaluate recent maintainability refactors and architecture guards;
* identify remaining macro-level risks;
* identify areas where complexity is likely inherent rather than accidental.

Primary question:

Has the project become structurally more stable, coherent, and safely
refactorable since REVIEW-001?

Expected deliverables:

* updated architectural assessment;
* resolved vs unresolved REVIEW-001 findings;
* maintainability trajectory assessment;
* remaining high-level structural concerns;
* architectural confidence assessment approaching beta.

No production edits.

=== Pass 2 -- Maintainability Hotspots and Refactorability

Focus:

* large/high-density modules;
* maintainability hotspots;
* coupling and semantic density;
* refactorability assessment;
* package-split suitability;
* risk assessment for future maintainability work.

Special interest areas include:

* `xref_crosstable.py`
* `reference_validations.py`
* `cell_codec.py`
* `tabular_views.py`
* workbook-view and resource-override areas
* long validation and transformation modules

Primary question:

Which remaining hotspots are good candidates for behavior-preserving
maintainability refactors, and which should explicitly NOT be aggressively
modularized?

Expected deliverables:

* hotspot inventory;
* risk/benefit assessment;
* suggested future maintainability slices;
* explicit “do not refactor aggressively” recommendations where appropriate.

No production edits.

=== Pass 3 -- Spreadsheet Adapter and IO Health

Focus:

* XLSX/ODS adapter architecture;
* parser/renderer separation;
* backend-neutral contract discipline;
* roundtrip robustness;
* hidden semantic policy in adapters;
* duplication and drift risks;
* IR-boundary discipline.

Special interest areas include:

* `openpyxl_renderer.py`
* `odf_renderer.py`
* `odf_parser.py`
* render-plan and parse-contract boundaries.

Primary question:

Are spreadsheet adapters still acting as thin execution/parsing layers, or is
semantic complexity drifting into backend-specific code?

Expected deliverables:

* adapter-health assessment;
* duplication/drift assessment;
* backend-boundary assessment;
* maintainability and extensibility outlook.

No production edits.

== Review Principles

The review should remain:

* technically critical where appropriate;
* calibrated against the actual maturity of the project;
* realistic rather than purity-driven;
* focused on maintainability and architectural stewardship;
* resistant to speculative framework proposals.

The review should NOT:

* propose broad rewrites;
* invent generic frameworks without demonstrated need;
* optimize for theoretical purity over operational maintainability;
* assume that all large modules are automatically bad;
* treat inherent transformation complexity as accidental by default.

== Expected Outcome

REVIEW-002 should answer:

* how much architectural maturity has improved since REVIEW-001;
* where remaining maintainability risks still exist;
* which refactoring trajectories are healthy;
* where boundaries are now stable enough to protect;
* and how close the repository feels to a stable beta architecture rather than
  a rapidly shifting prototype.
