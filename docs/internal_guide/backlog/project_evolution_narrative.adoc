== Era 0 — Proof of Concept (Aug–Sep 2025)

The project started on 2025-08-29 as a quick Python proof of concept:
bidirectional conversion between JSON configuration data and Excel
spreadsheets.  The original use case was "world building" for bedtime stories
with a dinosaur theme — structured data about species, habitats, and story
arcs, maintained in spreadsheets and consumed as JSON.  The initial commits
(`d90fc75`..`c0c632a`) landed within a single weekend and covered basic
pack/unpack, column ordering, a custom writer to suppress pandas index columns,
and first tests.

Key characteristics of this era:

* single-file scripts evolving into a small module structure
* no formal architecture — direct pandas-to-openpyxl calls
* CLI via argparse with positional subcommands
* no IR layer; the pandas DataFrame _was_ the intermediate representation
* CSV and XML backends explored alongside XLSX (`3f847a5`..`9afba61`)

By mid-September the code handled foreign-key helper columns, multi-page
spreadsheets, validation logic, and basic logging — but the codebase had
accumulated organic complexity without clear layer boundaries.

'''

== Era 1 — Pipeline Introduction and First Product Release (Sep 2025)

Once the utility of the tool for other domains became clear (professional
configuration management, not just dinosaur world-building), the decision was
made to generalize.  A `pipeline` abstraction was introduced in `1fa7677`
(2025-09-19): `BoundStep`, `run_pipeline`, and YAML-driven step definitions.
This allowed domain-specific transforms to be declared rather than hard-coded.

The "Entdinofizieren" refactor (`b5eed8f`, 2025-09-24) — literally
"de-dinosaur-ify" — stripped all domain-specific naming, restructured the
folder layout, and established the project as a generic spreadsheet-handling
tool.  A first PyPI release (`0.1.0b3`) was tagged at `3bd76f7` (2025-09-29)
after adding XLSX autofilters and header styling.

'''

== Era 2 — Hexagonal Architecture and IR Layer (Oct 2025)

=== The hexagonal pivot

Commit `1517360` (2025-10-01) introduced the hexagonal architecture: a
runner/router core with thin CLI adapters (`5ede1ce`) replacing direct
function-to-backend coupling.  The following two weeks were spent stabilizing
integration issues (`3c63550`..`9149fca`) and tightening test coverage.

=== IR layer introduction

The intermediate rendering layer (IR) landed in `5967ccf` (2025-10-16).  This
decoupled the logical spreadsheet model (sheets, blocks, styles, named ranges)
from the physical openpyxl write path.  The IR made future backend
diversification (ODS, CSV rendering) structurally possible.

=== Backlog and documentation formalization

Between late October and early November 2025 the project received its formal
internal documentation structure: FTR-based backlog (`fcb2633`), AI policy
(`537a56b`), and the internal development guide.  The `docs/internal_guide/`
directory was chosen as a pragmatic catch-all for any information not explicitly
targeted at the end user — architecture notes, backlog, AI policy, development
conventions — with the explicit plan to separate into proper standalone
documents once enough content existed to justify a richer structure.  This
avoided premature stub hierarchies in a project that was still finding its
shape.

== Backend Landscape (as of 2026-04-30)

The project's I/O surface is asymmetric by design:

* *JSON* — the primary serialization target for generated configuration.
  Most real pipelines today produce or consume JSON.
* *YAML* — structured YAML writer for nested config trees (Phase 4a);
  flat YAML for frame exchange.
* *XLSX* — the primary spreadsheet backend, fully IR-driven since Phase 3a.
* *ODS/Calc* — second spreadsheet backend, landed in Phase 3j, hardened in 3k.
* *CSV* — read/write exists but is structurally limited: no layout, no meta,
  no multi-sheet.  CSV is list-oriented rather than object-oriented, making it
  closer to a minimal spreadsheet adapter than to JSON/YAML.  Its long-term
  role is unclear as features grow more layout-aware.
* *XML* — legacy backend from the PoC era; multi-header parity was added in
  Phase 2 but the backend has seen no further investment.  Likely candidate
  for deprecation or explicit scope-limiting.
