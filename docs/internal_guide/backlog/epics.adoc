===  Epics

==== Phase 1 — Stabilize and Modularize the XLSX Backend

*Goal:*::
Separate spreadsheet logic into clear, testable layers without changing existing behavior.

*Key results:*::

- Move all XLSX-specific code into `io_backends/xlsx/` package.
- Introduce the IR (Intermediate Representation) layer with `compose → passes → render` flow.
- Migrate and tag all tests to support `@pytest.mark.xlsx_ir` and `@pytest.mark.xlsx_legacy` dual harness.
- Ensure green CI across both backends.

*Definition of done:* XLSX adapter runs through the IR path; all legacy tests pass or are explicitly tagged; internal APIs stable enough for new features.

==== Phase 2 — Domain Consistency and Meta Integration

*Goal:*::
Establish a unified domain/meta contract shared between CLI, pipeline, and I/O layers.

*Key results:*::

- Introduce `bootstrap_meta` transformation to inject CLI/pipeline config into runtime meta.
- Standardize fault-tolerant meta handling (empty `{}` treated as valid).
- Ensure meta persistence and rehydration across all backends.
- Document and test merge priorities (runtime > persisted > heuristics).

*Definition of done:*::
All adapters and domain transformations can operate without explicit meta; missing metadata never causes failure; roundtrip preserves meta content.

==== Phase 3 — Functional Parity and IR-driven Features

*Goal:*::
Achieve feature parity between the new IR backend and the legacy Excel path while enabling extensible rendering logic.

*Key results:*::

- Move validation, style, and formula logic into dedicated IR passes (`ValidationPass`, `StylePass`, etc.).
- Re-implement complex behaviors (multi-sheet layouts, hidden meta sheets, data validation lists).
- Use IR tests to confirm parity with legacy outputs for known examples.
- Provide extension hooks for future custom passes (e.g., domain-specific formatting).

*Definition of done:*::
IR backend produces visually and semantically equivalent workbooks for all core scenarios; legacy path officially deprecated in CI.

==== Phase 4 — Feature Expansion and Domain Transformations

*Goal:*::
Introduce new transformation capabilities and richer domain logic leveraging the stable IR and meta foundation.

*Key results:*::

- Implement `FTR-COMPACT-TRANSFORMS` (cross-table ↔ compact-table bijection).
- Implement `FTR-PARTITION-BY-DOMAIN` (partitioning by component attribute).
- Extend helper-column logic to support multiple enriched columns (name, category, description).
- Improve style/validation passes to handle these structures correctly.

*Definition of done:*::
Transformations are reusable and bijective; IR supports expanded layouts; new tests demonstrate composability of passes and transformations.

==== Phase 5 — Integration and Demonstration (Demo Repository)

*Goal:*::
Connect the refactored core library to the `spreadsheet-handling-demo` repository to demonstrate production usage patterns.

*Key results:*::

* Integrate new IR backend in the demo pipelines and CLI shims.
* Provide example datasets (e.g., product/feature matrix) exercising cross-table and compact transformations.
* Verify JSON ↔ Excel ↔ JSON roundtrip with persisted meta and formatting.
* Document all example workflows in the User and Developer Guides.

*Definition of done:*::
Demo repository runs fully on refactored library; examples reflect all core features; public-facing documentation is aligned and reproducible.
