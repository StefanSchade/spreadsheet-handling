===== FTR-LEGEND-BLOCKS -- Legend/Codebook blocks for compact spreadsheet values

*Status:* New

*Purpose:*::
Provide a small spreadsheet-facing legend block that documents compact codes
or tokens used in visible table values.
+
The first slice treats a legend as a codebook:

* a stable code/token
* a human-readable explanation
* an optional token group/category
* optional validation metadata

The legend makes compact values understandable to users without making the
legend block itself the first source of domain truth.
+
In the first slice, a full cell value such as `E-R-K` may be treated as one
legend code. Splitting such values into smaller tokens is a later codec concern,
not a requirement of this FTR.

*Context:*::
Later compact transformations may build, split, or merge token strings across
several steps, for example in a cross-table or multi-axis representation.
Those later transforms need a clear place where token meanings can be rendered
and, where configured, validated.
+
This FTR intentionally covers only the legend/codebook representation and its
placement in a sheet. It does not implement cross-table transforms or compact
cell decoding.

*Scope:*::

* Represent a legend as a `TableBlock` placed alongside data blocks.
* Support entries with at least `token` and `label`; `token` may also represent
  a whole-cell code.
* Optionally support `group`, `description`, and validation-oriented fields.
* Persist enough metadata to identify the block as a legend/codebook rather
  than as ordinary data.
* Keep placement deterministic through top-left coordinates or an equivalent
  resolved layout contract.
* Apply distinct but simple styling so legend blocks are readable and visibly
  separate from data tables.

*Non-Goals:*::

* No cross-table generation or contraction.
* No compact multi-axis transformation.
* No parsing of compact cell values into relations.
* No editable legend-as-source-of-truth workflow.
* No inference of token semantics from free-form spreadsheet text.
* No hierarchical axis/header model.

*Example Configuration:*::

[source,yaml]
----
legend_blocks:
  product_market_codes:
    title: "Legend"
    placement:
      sheet: product_market_matrix
      anchor: right_of_table
      target: product_market_matrix
    entries:
      - token: A
        group: product
        label: Advanced SaaS
      - token: SO
        group: deployment
        label: On premises
      - token: D
        group: channel
        label: Direct sales
----

*Rendered Shape:*::

[source,text]
----
Legend
Token | Meaning       | Group
A     | Advanced SaaS | product
SO    | On premises   | deployment
D     | Direct sales  | channel
----

*Validation Role:*::
If a later transform or codec refers to a legend block, its tokens may be used
as an allowed-token set.
+
For this FTR, validation is limited to the codebook itself:

* codes/tokens are unique within a legend block
* empty codes/tokens are rejected
* required labels are present
* optional groups are carried through without interpretation

Validation of actual compact cell values belongs to `FTR-CELL-CODEC` or later
composed transforms.

*Resolved Placement Metadata:*::
During composition, the selected placement is resolved to concrete sheet
coordinates and written back under the corresponding
`legend_blocks.<name>.resolved` metadata entry.
+
This is intentional in the first slice: the write path persists the exact
legend anchor so the read path can distinguish rendered legend/codebook tables
from ordinary data tables. Workbooks without this persisted resolved metadata
are not required to be heuristically recognized as containing legend blocks.
+
Longer term, the composer may switch from mutating the caller-supplied meta
object to persisting a normalized copy, but the observable contract remains the
same: resolved legend placement is part of the persisted workbook metadata.

*Relationship to Later Compact Transforms:*::

This feature provides the visible codebook that later steps can reference.
The expected longer chain is:

. `FTR-LEGEND-BLOCKS` renders and persists token explanations.
. `FTR-CELL-CODEC` parses and serializes compact token strings.
. `FTR-XREF-CROSSTABLE` transforms between matrix and relation shape.
. `FTR-COMPACT-MULTIAXIS` composes matrix axes and cell-token axes into richer
  explicit structures.

The important boundary is that this FTR makes token meanings explicit and
portable, but does not yet perform the transformation that gives those tokens
relational meaning.

*Acceptance:*::

* A workbook can contain a normal data table and a legend `TableBlock` on the
  same sheet without treating the legend as a separate data frame.
* The legend block has deterministic placement and distinct styling.
* Legend metadata is persisted so read/import paths can identify the block as
  legend/config content.
* Duplicate or empty tokens in one legend block produce a clear validation
  error.
* The implementation does not introduce cross-table, cell-codec, or multi-axis
  semantics.
