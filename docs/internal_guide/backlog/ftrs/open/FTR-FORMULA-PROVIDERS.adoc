'''

====== Purpose

Introduce a backend-neutral formula representation (`FormulaSpec`) and keep
backend-specific formula syntax isolated to adapter translation.

This FTR enables formula providers and FK-linking behavior without reintroducing
XLSX-shaped semantics into generic Domain/Pipeline/Rendering layers.

'''

====== Problem statement

Raw formula strings are an easy short-term path but create long-term coupling:

* generic layers become backend-specific
* tests validate syntax strings instead of semantics
* ODS/XLSX behavior drifts

Formula support must preserve architecture boundaries.

'''

====== Goals

. Represent formula intent structurally above adapter boundaries.
. Translate formula specs to XLSX/ODS syntax only in adapter-facing modules.
. Keep references stable via named ranges or symbolic references.
. Provide tests that assert semantic equivalence at generic layers.

'''

====== Scope

This FTR includes:

* add a minimal `FormulaSpec` representation for the required formula use cases
* route formula generation through formula providers/translators
* keep backend syntax emission in concrete adapter translation points
* update tests to separate semantic assertions from backend string assertions

'''

====== Non-goals

This FTR does not:

* define a complete end-user formula language
* guarantee support for all spreadsheet formula functions
* expose backend syntax as a generic API

'''

====== Acceptance criteria

. References resolve correctly for supported formula use cases.
. No hard-coded worksheet coordinates are required in generic layers.
. Generic-layer tests assert semantic intent (structure/equivalence), not raw
  backend formula strings.
. Backend-specific string assertions are limited to adapter/translator tests.
. A violation is detectable if raw backend syntax is introduced above the
  adapter/formula-translation boundary.

'''
