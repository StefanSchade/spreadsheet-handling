===== FTR-EDITABLE-COLUMNS-AND-PROTECTION-P4A -- Distinguish editable and derived columns in rendered workbooks

*Status:* Open
*Priority:* P4A
*Area:* Rendering / editability metadata

==== Purpose

Make editability explicit at column or view level and render that intent into
spreadsheet protection where the backend supports it.

Generated workbooks often mix editable business data with derived helper
columns, labels, lookup context, or generated views. Users need to see and edit
the intended cells without accidentally changing derived context.

==== Consumer Trigger

In a prototypical consumer matrix sheet, the code matrix cells are the
business-editable area. The helper/context columns next to the primary key
should be visible but treated as derived.

==== Goals

* Define metadata for editable, read-only, and derived columns or ranges.
* Allow pipeline/view steps to mark helper/context columns as non-editable.
* Render protection intent for XLSX.
* Evaluate and implement ODS/Calc protection support where feasible.
* Keep workbook protection optional and explicit.
* Preserve metadata through parse/render roundtrips where supported.

==== Non-goals

* Do not make spreadsheet protection a security boundary.
* Do not require password management in this slice.
* Do not lock entire workbooks by default.
* Do not block parsing edited values from unlocked cells.

==== Acceptance

* A generated workbook can distinguish editable matrix/code columns from
  read-only helper columns.
* XLSX output locks or protects the configured read-only cells while leaving
  editable cells usable.
* ODS support is either implemented with an adapter test or documented as an
  explicit capability gap.
* Reimport still reads user-authored data cells correctly.
* Tests cover at least one mixed helper/editable sheet.
