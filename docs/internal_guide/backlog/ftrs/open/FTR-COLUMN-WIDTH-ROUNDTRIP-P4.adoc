*Status:* Open
*Epic:* Phase 4 -- Feature Expansion and Domain Transformations
*Priority:* Medium

*Related-to:*

'''

====== Purpose

Preserve user-maintained spreadsheet column widths when a workbook is read through an adapter and written again.

The first useful behavior is roundtrip fidelity: if a user adjusts column widths in a workbook, the adapter should parse those widths into metadata and the writer should reconstruct them when rendering the workbook again.

'''

====== Context

Consumer workbooks are not only data containers. Column widths are part of the practical editing experience, especially for generated helper columns, compact matrices, descriptive labels, and review-oriented sheets.

Today the framework preserves data and selected workbook-level metadata, but column widths are not represented as explicit roundtrip metadata.

This means a workbook can be structurally correct after a roundtrip while still losing important human-facing layout work.

'''

====== Goals

. Read column width information from supported spreadsheet adapters.
. Persist width information in metadata using a documented, registry-backed shape.
. Reapply stored column widths during spreadsheet rendering.
. Keep widths sheet-local and column-local.
. Preserve existing behavior when no width metadata exists.
. Add focused roundtrip tests for at least the XLSX adapter.
. Document backend capability gaps, especially for ODS/Calc if exact semantics differ.

'''

====== Scope

This FTR includes:

* define a metadata key such as `__column_widths`
* add a meta-registry entry for the chosen metadata key
* parse XLSX column dimensions into that metadata key
* render XLSX column widths from that metadata key
* add at least one parse-write-parse roundtrip test that verifies width preservation
* inspect ODS/Calc support and either implement equivalent behavior or document an accepted capability gap

Suggested metadata shape:

[source,yaml]
----
_meta:
  sheets:
    Orders:
      __column_widths:
        A:
          width: 18.0
          source: workbook
        B:
          width: 42.5
          source: workbook
----

The exact storage location should follow the current meta carrier conventions. If the implementation uses numeric column indexes instead of Excel letters internally, the metadata must still be stable across parse/render for the same sheet layout.

'''

====== Non-goals

This FTR does not:

* implement a general spreadsheet layout engine
* infer ideal widths from cell content
* add YAML configuration for auto-sizing columns
* preserve every possible backend-specific column style flag
* solve row-height roundtrip
* require exact parity between XLSX and ODS if the backends expose materially different width semantics

Row heights, auto-fit behavior, and authoring-time width policies can be separate follow-up FTRs.

'''

====== Design notes

Column widths should be treated as explicit workbook presentation metadata, not as semantic table data.

Important boundaries:

* Missing width metadata means the writer keeps its current default behavior.
* Parsed widths should not change frame data or table detection.
* Rendering should only apply widths for columns that still exist in the rendered sheet.
* If a transformation changes column order or drops columns, width metadata should either follow stable column identity where available or be conservatively ignored for ambiguous columns.
* Hidden columns are related but should not be smuggled into this FTR unless the adapter already exposes them with the same column dimension object and the behavior is trivial.

'''

====== Tests

Required tests:

. An XLSX workbook with non-default column widths is parsed and the widths appear in metadata.
. Rendering the parsed workbook reconstructs the stored column widths.
. A parse-write-parse roundtrip preserves at least two distinct non-default widths.
. Missing width metadata does not fail rendering and preserves current defaults.
. The meta-registry contains the column-width metadata key.

Optional tests:

* ODS width parse/render behavior if supported in the same slice.
* Width metadata is ignored safely when it references columns outside the rendered table extent.

'''

====== Acceptance criteria

. Column widths survive an XLSX adapter roundtrip.
. Column-width metadata is documented in the meta registry.
. Rendering remains backward compatible when metadata is absent.
. Unsupported or lossy backend behavior is documented explicitly.
. Existing active tests continue to pass.

'''

====== Review guidance

When reviewing this FTR, check:

* Is the metadata shape stable enough for roundtrip use?
* Does the adapter preserve user-authored widths without affecting data semantics?
* Are backend-specific units and defaults handled deliberately?
* Does the implementation avoid accidental auto-sizing behavior?
* Are ambiguous transformed columns handled conservatively?

'''

====== Rationale

Column widths are small metadata, but they have a large effect on whether generated and roundtripped workbooks remain pleasant to edit.

Preserving them keeps the framework closer to the user's authored workbook instead of treating the spreadsheet as only tabular data.
