===== Review: FTR-PIPELINE-STEP-REGISTRATION-NORMALIZATION

*Status:* Implemented in the current working tree, not yet committed.

*Outcome:*::
The first normalization slice is in place.

The registry now supports a normalized descriptor form via `StepRegistration`,
while still accepting legacy bare factories for unchanged step kinds.

This establishes the new normal path:

* registry entry
* -> binding factory
* -> optional domain target
* -> `BoundStep`

*What Changed:*::
- Added `StepRegistration` to the pipeline type layer.
- Added generic pipeline binders for the two targeted callable shapes:
  - builder-style `(...config) -> Step`
  - frames-first `(frames, **config) -> Frames`
- Migrated the simple clearly domain-owned registry entries to the normalized
  descriptor path:
  - `flatten_headers`
  - `unflatten_headers`
  - `reorder_helpers`
  - `bootstrap_meta`
  - `apply_overrides`
  - `add_validations`
- Kept legacy/special step paths unchanged for now:
  - `identity`
  - `validate`
  - `apply_fks`
  - `drop_helpers`
  - `check_fk_helpers`
  - `plugin`
- Added a domain-facing convenience entry point
  `load_and_apply_overrides(...)` so the `apply_overrides` step can use the
  same generic frames-first binding path as the other migrated steps.
- Added focused registry tests in
  `tests/unit/pipeline/test_step_registration.py`.

*Architectural Assessment:*::
This slice achieved the intended boundary improvement without forcing broader
feature refactors.

The main architectural win is not new behavior in `run_pipeline()`, but a
cleaner seam one layer earlier:

* YAML step name resolution is less tightly coupled to bespoke per-step pipeline
  factories
* simple domain-owned steps now advertise their domain target explicitly
* future extractions such as FK-helper domain extraction now have a cleaner
  registration model to land on

*Local Test Limitations:*::
- Tests using `tmp_path` could not be run reliably in the local Windows setup
  because `pytest` hit permission errors around temporary directory management.
  This affected file-path-based override tests and one plugin smoke test.
- `tests/unit/test_orchestrator_meta_contract.py` could not be collected in the
  local Windows venv because the import path pulled in the ODS backend and the
  `odf` package was not installed there.

These limits do not indicate a regression in the implemented slice, but they do
mean the local verification is not a full suite pass.

*Residual Risk:*::
- The registry now mixes normalized descriptor entries and legacy bare factory
  entries by design. That is the intended incremental state for this slice, not
  a bug.
- `apply_fks`, `drop_helpers`, and validation-related orchestration still need
  their own follow-up tickets to migrate away from fat factories.
- The new generic binders intentionally cover only the two dominant callable
  shapes. Additional step kinds can be added later if and when the architecture
  really needs them.
