====== Purpose

Create a structured inventory of existing architecture guardrails and make gaps explicit before adding further guard logic.

This FTR prevents blind expansion of guard tests and ensures that new guardrails are based on actual risk, not assumptions.

'''

====== Context

After introducing a dedicated architecture test layer, guardrails now exist but are distributed across different categories:

* dependency guards
* spreadsheet contract guards
* parse contract guards
* semantic invariants
* package-specific fitness guards
* current-state checks

Additionally, targeted guards such as `FTR-ARCHITECTURE-FITNESS-GUARDS-P4X` already address specific drift areas (e.g. pipeline/FK-helper coupling).

However, there is no single overview answering:

* which architectural seams are already protected
* which are only partially covered
* which are currently unguarded

Without this inventory, further guard work risks duplication, gaps, or over-constraining the system.

'''

====== Problem statement

Architecture guardrails currently exist in an implicit and fragmented form.

This leads to:

* unclear coverage of critical boundaries
* redundant or overlapping tests
* missing protection in high-risk areas (e.g. formula handling)
* confusion between durable invariants and current-state checks

Without a clear inventory, guard development is reactive and inconsistent.

'''

====== Goals

. Identify and list all existing architecture guard tests.
. Map each test to the architectural seam or invariant it protects.
. Distinguish durable invariants from current-state checks.
. Identify high-risk gaps in guard coverage.
. Classify gaps to guide follow-up FTRs.

'''

====== Scope

This FTR includes:

* scanning `tests/architecture/` for existing guard tests
* identifying guard-like tests outside that directory if relevant
* classifying each test into one of:
** dependency guard
** adapter contract guard
** parse contract guard
** semantic invariant
** current-state check
** package-specific fitness guard
* mapping each test to the seam it protects
* identifying missing guards in key architectural areas
* producing a compact inventory document

'''

====== Non-goals

This FTR does not:

* introduce new guard tests
* modify existing tests
* enforce architectural rules directly
* redesign architecture boundaries

All enforcement work belongs to follow-up guard FTRs.

'''

====== Key architectural seams to assess

The inventory must explicitly evaluate at least the following areas:

* Domain vs Pipeline separation
* Pipeline vs payload semantics
* Render IR vs adapters
* Spreadsheet contract boundaries
* Parse contract (`WorkbookIR` boundary)
* Meta layers (`meta_canonical`, `meta_rendering`, carrier hints)
* Formula boundary (backend-neutral vs backend-specific syntax)
* Adapter isolation (XLSX, ODS)

'''

====== Gap classification

Each identified gap must be classified as:

[cols="1,3"]
|===
| Category | Meaning

| immediate
| Guard can be implemented immediately with clear invariant

| needs-design
| Risk is known but invariant is not yet stable

| observe
| Monitor area; no guard yet

| covered
| Existing guards are sufficient
|===

'''

====== Deliverable

Create a document:

[source,text]
----
docs/internal_guide/architecture/architecture_guard_inventory.adoc
----

Content should include:

* table of existing guards
* mapping to architectural seams
* durability classification
* identified gaps
* recommended follow-up actions

The document must remain concise and focused.

'''

====== Acceptance criteria

. All architecture guard tests are listed and classified.
. Each guard is mapped to a clear architectural seam.
. Durable invariants are distinguished from current-state checks.
. Key architectural seams are evaluated for coverage.
. Gaps are identified and classified.
. Follow-up guard work is clearly scoped.
. No production code changes are introduced.
. Existing tests remain unchanged and passing.

'''

====== Review guidance

Reviewers should verify:

* Is each listed guard actually protecting an architectural invariant?
* Are current-state checks clearly separated?
* Are important seams missing coverage?
* Are gaps correctly classified (not over- or under-specified)?
* Does the inventory enable targeted follow-up work?

'''

====== Rationale

Architecture guardrails are only effective if they are deliberate.

Without an inventory, the system risks either under-protection (drift) or over-protection (rigidity).

This FTR provides the necessary map to guide all subsequent guard work, especially in high-risk areas such as formula handling and adapter boundaries.
