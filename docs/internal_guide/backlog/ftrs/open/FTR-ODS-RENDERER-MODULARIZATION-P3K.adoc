===== FTR-ODS-RENDERER-MODULARIZATION-P3K — Modularize the ODS renderer along the same seam pattern as XLSX

*Depends-on:* Phase 3j completion

*Context:* The Phase-3j ODS adapter landed a working renderer in a single `odf_renderer.py` file (~440 lines). This was a deliberate first-cut decision: get the adapter working against the contract before splitting internal structure. Now that the adapter is proven and the parity tests are green, the renderer mixes several concerns that would benefit from the same seam pattern already applied to the XLSX side.

*Problem statement:* If the ODS renderer grows with future ODS-specific extensions (additional validation types, conditional formatting, richer style support), the current single-file layout will become harder to maintain. The XLSX side already went through a modularization pass in Phase 3i; the ODS side should reach a comparable internal clarity before significant feature work exercises it further.

*Goals:*

. Separate ODS address/reference helpers from rendering logic.
. Extract validation formula conversion and style registration into focused helpers.
. Keep the `_collect_sheets` / `_build_table` / `render_workbook` pipeline readable as the main flow.
. Improve navigability so contributors can tell where ODS-specific carrier decisions live versus where the generic `RenderPlan` dispatch happens.

*Acceptance criteria:*

. The renderer's internal structure has at least one additional focused helper module (e.g., address utilities, validation emission, or style registration).
. The main rendering entry point `render_workbook` remains clear and its dispatch over `RenderPlan` ops is easy to follow.
. Existing ODS tests and parity tests continue to pass without behavior changes.
. The resulting module layout makes future ODS extensions (new validation types, conditional formats) easier to add without growing the main renderer file.

*Non-goals:*

- No new ODS features in this FTR.
- No attempt to create a shared renderer framework across XLSX and ODS.
- No changes to the `RenderPlan` contract or generic rendering layers.

*Why now:* Phase 4 feature work will exercise both adapters. Cleaning up the ODS renderer structure before that work begins is cheaper than modularizing under feature pressure.
