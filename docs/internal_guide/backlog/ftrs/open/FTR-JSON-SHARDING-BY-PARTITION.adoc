===== FTR-JSON-SHARDING-BY-PARTITION — Split JSON/YAML output into per-partition files

*Status:* Open (deferred — advanced feature, post-beta)
*Epic:* Phase 6
*Priority:* Low

*Related-to:*

* <<FTR-SPLIT-BY-DISCRIMINATOR-P4A#,FTR-SPLIT-BY-DISCRIMINATOR-P4A>>
* <<FTR-PARTITION-BY-DOMAIN#,FTR-PARTITION-BY-DOMAIN>>

'''

====== Purpose

Split a single logical table into multiple JSON or YAML files on disk, keyed by
a configurable partition column (e.g. `fachkomponente`). This avoids Git merge
conflicts when multiple BAs edit different partitions concurrently, because each
partition lives in its own file.

'''

====== Context

In the BA config workflow, a single logical resource like `businessrules` is
edited across multiple Fachkomponenten. Today this produces one JSON file. When
two BAs commit changes to different Fachkomponenten simultaneously, Git reports
a merge conflict on the shared file even though the logical changes are
disjoint.

Sharding the output along the partition column produces:

[source]
----
data/businessrules/
  fk_leben.json
  fk_sach.json
  fk_kranken.json
----

Each BA touches only their file — no conflicts.

'''

====== Problem statement

Without sharding:

* concurrent BA edits on the same logical table create avoidable merge conflicts
* manual workarounds (communication, serialized editing) slow teams down

With sharding:

* each partition is an independent file
* Git merges are conflict-free for disjoint partitions
* reassembly on read produces the original logical table

'''

====== Design sketch (indicative)

[source,yaml]
----
- step: shard_to_files
  frame: businessrules
  partition_column: fachkomponente
  output_dir: data/businessrules/
  filename_pattern: "{value}.json"

- step: reassemble_from_shards
  target_frame: businessrules
  partition_column: fachkomponente
  input_dir: data/businessrules/
  filename_pattern: "{value}.json"
----

====== Key decisions (to be refined later)

* Filename derivation: slugified partition value vs. explicit mapping
* Sort order within shard files: preserve original or sort by key columns
* Reassembly order: sorted by partition value (deterministic)
* Empty partitions: omit file or write empty array
* Schema consistency: all shards must share the same column set

'''

====== Rationale for deferral

* Currently not many BAs work in parallel on the same table
* The base workflow (Discriminator-Split, XRef, Sparse, etc.) must stabilize
  first
* Premature optimization — revisit when concurrent editing becomes a real pain
  point

'''

====== Scope (when executed)

* `shard_to_files` pipeline step: partition frame → N files
* `reassemble_from_shards` pipeline step: N files → single frame
* roundtrip guarantee: `reassemble(shard(df)) == df`
* integration with existing JSON/YAML I/O router

'''

====== Non-goals

* Excel-side sharding (multiple workbooks) — separate concern
* Automatic conflict resolution — out of scope, Git handles this
* Real-time collaboration — not a use case
