= Review — FTR-MULTI-HELPER-COLUMN

== Ergebnis

Das Ticket ist im aktuellen Zuschnitt umgesetzt.

- Ein FK kann nun mehrere Helper-Spalten erzeugen.
- Die Reihenfolge der Helper-Spalten ist konfigurierbar und bleibt beim Reorder-Schritt erhalten.
- `drop_helpers` und `clean_aux_columns` entfernen Helper-Spalten nun auch dann robust, wenn die Spalten als Tuple-/MultiIndex-Header vorliegen.
- FK-Validierungen arbeiten mit mehreren Helper-Spalten, ohne denselben `unresolvable_fk` mehrfach fuer dieselbe FK-Spalte zu melden.

== Konfigurationsform

Primärer Einstieg fuer dieses Ticket ist:

[source,yaml]
----
defaults:
  helper_fields_by_fk:
    id_(branch): [category, name]
----

Semantik:

- `helper_fields_by_fk` konfiguriert die gewuenschte Helper-Reihenfolge pro FK-Spalte.
- `helper_fields` bleibt als globaler Fallback moeglich.
- Ohne explizite Helper-Feld-Konfiguration bleibt das bisherige Verhalten erhalten und es wird weiterhin nur das `label_field` als einzelner Helper erzeugt.

== Architekturentscheidung

Der Change fuehrt keine neue umfangreiche FK-DSL ein. Stattdessen bleibt das Modell bewusst klein:

- `detect_fk_columns(...)` erzeugt nun mehrere `FKDef`-Eintraege pro FK-Spalte, jeweils mit eigenem `value_field`.
- `build_id_value_maps(...)` liefert feldspezifische Lookup-Maps pro Zielblatt.
- Styling bleibt praefixbasiert; deshalb war kein eigener neuer Style-Pfad noetig. Mehrere Helper-Spalten werden automatisch konsistent behandelt, solange ihr sichtbarer Spaltenname mit dem Helper-Praefix beginnt.
