===== FTR-CELL-CODEC -- Cell Value <-> Token/Code Transformation

*Status:* Implemented (first generic slice)

*Purpose:*::
Provide a generic mechanism to validate, parse, and serialize compact cell
values without assigning domain-specific business meaning to them.

*Context:*::
Some compact spreadsheets use a single code per cell, for example `S`, `E`,
`x`, or `E-R-K`. Others use delimited token strings where one cell contains
several separately meaningful tokens.
+
The first implementation should support the safe case where the whole cell
value is treated as one code. Token splitting is an explicit mode, not the
default assumption.

*Scope:*::

* Validate whole-cell codes against a configured value set.
* Optionally parse delimited cell values into token lists.
* Serialize token/code values back into a deterministic cell representation.
* Support configurable delimiters and normalization rules.
* Allow `allowed_tokens` / `allowed_codes` to reference a legend block.

*Non-Goals:*::

* No domain-specific meaning assigned to tokens or codes.
* No hardcoded token semantics.
* No cross-table or relation transformation.
* No inference of token structure from punctuation such as `-`.

*Solution:*::

* Default mode: treat the entire non-empty cell value as a single code.
* Split mode: split cell values using configured delimiters.
* Normalize only where explicitly configured.
* Validate against optional allowed code/token lists.
* Serialize into a canonical string representation.

*First Implementation Slice:*::

The first implementation provides a domain codec module plus two frames-first
pipeline steps:

[source,yaml]
----
pipeline:
  - step: decode_cell_values
    source: feature_product_codes
    output: decoded_codes
    value: value
    code: code
    passthrough_columns: [feature_id, column_key]
    mode: whole_cell_code
    allowed_from_legend: status_codes

  - step: encode_cell_values
    source: decoded_codes
    output: feature_product_codes_roundtrip
    group_by: [feature_id, column_key]
    code: code
    value: value
    mode: whole_cell_code
    allowed_from_legend: status_codes
----

Low-level helpers are also available for direct use:

* `parse_cell_value(...)`
* `serialize_cell_value(...)`
+
`parse_cell_value` reads compact cell strings. `serialize_cell_value` expects an
already structured code/token value and writes the compact cell string. In
`split_tokens` mode, passing `"K-E"` to `serialize_cell_value` means one
structured token named `K-E`, not a compact cell value that should be split.

`decode_cell_values` expands one compact cell-value column into explicit rows.
In `whole_cell_code` mode each non-empty cell produces at most one output row.
In `split_tokens` mode one row is produced per configured token.
+
`encode_cell_values` groups explicit code/token rows and writes the compact cell
value back. In `split_tokens` mode `canonical_order` can force deterministic
token ordering. Without `canonical_order`, input row order is preserved.
+
Empty cells are sparse by default for decoding (`drop_empty: true`). They can be
preserved as explicit empty-code rows by setting `drop_empty: false`.
+
The sparse default is not lossless for empty-only groups in an XRef roundtrip:
`expand_xref -> decode_cell_values(drop_empty: true) -> encode_cell_values ->
contract_xref` can lose a row identity when all compact cells for that identity
are empty. Use `drop_empty: false` for lossless roundtrips where empty matrix
cells or empty-only rows must survive.

*Configuration (Meta):*::

* `mode`: `whole_cell_code` or `split_tokens`
* `delimiter` for split-token mode, for example `-` or `,`
* `allowed_codes` or `allowed_tokens`, optionally sourced from a legend block
* `canonical_order` for split-token mode
* case normalization rules

The implementation records resolved step configuration under workbook metadata
key `cell_codecs`, keyed by the step `name` or output frame name. This metadata
is a generic transformation contract, not domain meaning. It is persisted
documentation of the resolved contract; the first slice does not use
`cell_codecs` as an implicit runtime configuration resolver.

*Implemented Boundary:*::

The codec validates configured code/token sets and handles deterministic
parse/serialize behavior. It does not infer structure from punctuation. For
example, `E-R-K` remains a single code unless `mode: split_tokens` is explicitly
configured.
+
Legend integration is read-only for this slice: `allowed_from_legend` uses
`legend_blocks.<name>.entries[*].token` as the allowed value set.
+
If explicit `allowed_codes` / `allowed_tokens` are combined with
`allowed_from_legend`, duplicate values are treated as configuration errors.

*Acceptance:*::

* Whole-cell codes such as `E-R-K` can be validated without being split.
* Parsing and serialization are deterministic.
* Roundtrip `cell value -> structured representation -> cell value` yields the
  configured canonical representation.
* Invalid codes/tokens can be detected if configuration is provided.
* A codec can use a `FTR-LEGEND-BLOCKS` codebook as its allowed-value source.
* Behavior is fully driven by configuration.
