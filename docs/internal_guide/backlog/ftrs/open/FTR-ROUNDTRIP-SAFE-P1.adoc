===== FTR-ROUNDTRIP-SAFE-P1 — Controlled round-trip for spreadsheet layouts

Purpose:::
Ensure that workbooks produced by our renderer can be edited in a controlled way and then re-imported into the IR/domain without structural drift. The goal is predictable round-tripping for our exported shapes, not arbitrary spreadsheets.

Scope:::
- Define two supported scenarios and their constraints:
** Scenario A — Flat table(s) without named ranges: one or more tables on a sheet, discoverable by start anchors; growth by appending rows/columns until a stopping condition.
** Scenario B — Multiple tables with named ranges: each table is delimited by a named range and can safely co-exist with other tables on the same sheet.
- Define metadata bootstrap strategies for imports with and without embedded meta.
- Provide a minimal parser (parse_ir) that reconstructs a canonical IR from supported layouts.

Design Principles:::
- Stable anchors over heuristics: prefer explicit coordinates or named ranges to infer boundaries.
- Explicit meta over guessing: when available, embedded meta (_meta sheet or named meta range) wins over heuristics.
- Controlled edit surface: allow appending rows, inserting columns inside table ranges, and editing values; disallow structural ops that break anchors (e.g., merges over headers) unless covered in a later phase.
- Lossless data, lossy style: guarantee round-trip for data and metadata; styling may be normalized.

Supported Scenarios (P1):::
.Scenario A — Flat tables
 - Each table has a start anchor (top-left cell).
 - Table grows down/right until the first fully empty row/column (stop sentinel).
 - Guideline: Prefer one table per sheet in A to avoid accidental cross-contamination when users insert empty rows/columns. If multiple tables are needed, strongly consider Scenario B.

.Scenario B — Multi-table with named ranges
 - Each table is defined by a named range (e.g., tbl_customers, tbl_orders).
 - Ranges strictly define the import/export area.
 - Writer ensures named range creation; reader resolves tables by names.

Metadata Bootstrap (when importing):::
Order of precedence to discover structure:
 1) Embedded meta (hidden _meta sheet or named range __sh_meta) → JSON/YAML with table ids, range names, and typing.
 2) Named ranges (Scenario B).
 3) Anchors config (Scenario A) provided via CLI/args (e.g., --tables S!A1, --tables S!A1,S!J1).
 4) Sidecar file (workbook.xlsx.meta.json) colocated with the file.
 5) Heuristics fallback (discouraged): first non-empty header row + sentinel empties; can be disabled by flag.

CLI & Pipeline Extensions:::
- New optional flags for the import step:
 * --meta-source=embedded|sidecar|none
 * --table=Sheet1!A1[:NamedRange] (repeatable)
 * --stop-on-empty-row / --stop-on-empty-col (Scenario A controls)
- In pipeline YAML, mirror the same options.

Implementation Sketch:::
- parse_ir(xlsx_path, options) -> WorkbookIR
 * Reads embedded meta (if present) and resolves tables.
 * If Scenario B, enumerate named ranges and materialize table blocks.
 * If Scenario A, use provided anchors + sentinel rules.
 * Populate IR with blocks, headers, data, validations (where resolvable).
- Renderer continues to write _meta and named ranges.

Acceptance:::
- Given a workbook produced by our renderer:
 * Edits limited to appending rows, inserting columns within a table range, and changing values are round-trip safe.
 * parse_ir reconstructs the same logical table set (ids, order, and shapes) for both Scenario A and B.
- A test fixture set covers A (single-table per sheet) and B (multi-table with named ranges).
- Docs include a 'How to edit safely' section.

Out of Scope (P1):::
- Complex merges in headers (covered by Multi-header/P2).
- Arbitrary user-inserted shapes/charts/pivots.
- Structural changes that shift anchors without meta updates.
