===== FTR-SPREADSHEET-PARSE-CONTRACT-P3I — Specify the spreadsheet parse contract above format-specific parsers

*Depends-on:* <<FTR-XLSX-ADAPTER-ALIGNMENT-P3H#,FTR-XLSX-ADAPTER-ALIGNMENT-P3H>>, <<FTR-SPREADSHEET-BACKEND-CONTRACT-P3H#,FTR-SPREADSHEET-BACKEND-CONTRACT-P3H>>, <<FTR-SPREADSHEET-META-SEMANTICS-P3H#,FTR-SPREADSHEET-META-SEMANTICS-P3H>>

*Context:* The write-side spreadsheet contract is now substantially clearer: format-specific renderers execute `RenderPlan`, and generic spreadsheet logic is no longer mixed with XLSX-specific implementation details. The read side has also been improved by moving the OpenPyXL parser into the XLSX adapter package and separating generic `WorkbookIR -> Frames` projection. However, the parser-side contract is still mostly implicit in the current spreadsheet parser shape rather than stated as a spreadsheet-neutral boundary for future adapters.

*Problem statement:* Without an explicit parse contract, a future spreadsheet parser such as ODS/Calc would likely either copy the current XLSX parser shape too literally or rediscover parser responsibilities ad hoc while implementation is already underway. That would risk hardening current parser accidents instead of turning the read path into a deliberate spreadsheet-neutral boundary.

*Goals:*

. Define the spreadsheet-neutral parser boundary from workbook file to `WorkbookIR` and from `WorkbookIR` to `Frames + persisted _meta payload`.
. Clarify which parse results are required, which are optional, and which remain carrier-side hints.
. Align parser terminology with `meta_canonical`, `meta_rendering`, `WorkbookIR`, and the spreadsheet contract vocabulary already used on the write side.
. Make the eventual ODS/Calc adapter target a stated read-path contract instead of inferring one from `openpyxl_parser.py`.
. Add a small set of focused tests guarding the intended parse-path boundary and invariants.

*Acceptance criteria:*

. The architecture docs state a spreadsheet-neutral parse contract analogous to the already clarified write-side contract.
. The intended boundary between format-specific parser logic, `WorkbookIR`, and generic `WorkbookIR -> Frames + _meta` projection is explicitly documented.
. The docs distinguish required parse semantics, optional parse semantics, and carrier-side hints.
. At least a small set of focused tests guards parse-path dependency direction and the central roundtrip/payload invariants.
. The resulting contract is clear enough that a future spreadsheet parser can target it without inferring its shape from `openpyxl_parser.py`.

*Non-goals:*

- No ODS/Calc implementation in this FTR.
- No full parity guarantee across all spreadsheet formats.
- No generic parser framework driven by registry data.
- No final reverse-projection policy for arbitrary end-user formatting into canonical meta.
- No forced deep split of the current XLSX parser unless the contract work reveals a real boundary problem.

*Why now:* The XLSX alignment work has made the parser boundary visible enough to describe deliberately. Freezing that seam before concrete second-adapter work starts is cheaper than discovering it only while implementing the next parser. At the same time, this should remain lighter-weight than a full second-adapter FTR.
