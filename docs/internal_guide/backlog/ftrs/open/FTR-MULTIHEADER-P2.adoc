===== FTR-MULTIHEADER-P2 — IR Renderer “Parity Slice v2” (P2) Multi-row/column headers (with merges) + Optional Flattening at I/O

*Purpose:*::
Use pandas MultiIndex headers as the single in-memory standard. JSON/XML may emit nested objects by default; provide an optional edge step to flatten/unflatten when a flat representation is needed (including spreadsheets).

*Scope:*::
Applies to all tabular frames; domain/IR stay MultiIndex. Flattening is an optional **edge step** (Frames→Frames) that can be placed before adapters.

*Context:*::
Slice P2 of the Roadmap towards IR Parity <<RM-IR-PARITY: Parity between IR renderer and legacy renderer, RM-IR-PARITY>>

*Solution:*

* Keep MultiIndex headers canonical in domain/IR.
* Spreadsheet renderer honors `headers.mode`:
  - `multi`: render merged multi-row/col headers with correct data offsets and styles.
  - `flat`: consume pre-flattened frames (produced by the edge step) and render single-line headers.
* JSON/XML adapters: default to nested objects from MultiIndex; accept already-flattened frames from the edge step if configured.
* Provide pure pipeline steps: `flatten_headers(mode=join|first_nonempty, sep='.')` and `unflatten_headers(sep='.')` for reversible mapping.
* Document edge cases (non-unique tuples, empty labels, both-axis levels) and the required conventions (placeholders, coercion, `sep`).

*Acceptance:*

* Domain always operates on MultiIndex headers.
* XLSX: merges when `mode=multi`; renders single-row headers when fed flattened frames.
* JSON/XML: nested output by default; flat output achievable by inserting the flatten step.
* `flatten_headers`/`unflatten_headers` are pure and **lossless** under documented conventions (unique tuples, stable level order, defined `sep`).
* Tests cover: multi→xlsx (merged), multi→json (nested), multi→flatten→xlsx/json (flat), and round-trips with `unflatten_headers`.
