===== FTR-SPREADSHEET-META-SEMANTICS-P3H — Specify spreadsheet-agnostic workbook metadata semantics

*Depends-on:* <<FTR-IR-ARCHITECTURE-CLARITY-P3H#,FTR-IR-ARCHITECTURE-CLARITY-P3H>>

*Context:* The rendering path already carries workbook and sheet semantics such as validations, named ranges, freeze panes, autofilters, helper-column highlighting, and hidden meta payload. In practice, parts of this are currently expressed through conventions that emerged from the XLSX/OpenPyXL implementation, especially around hidden `_meta` sheets and validation formulas.

*Problem statement:* A second spreadsheet adapter needs a precise statement of what spreadsheet semantics belong to `WorkbookIR` and `RenderPlan`, what must be reconstructible from a workbook file, and what may remain format-specific. Without that contract, XLSX and ODS would risk drifting into subtly different interpretations of the same IR.

*Goals:*

. Define spreadsheet semantics at the IR/plan boundary in a format-neutral way.
. Clarify which metadata is canonical in `WorkbookIR` and which persistence strategy each adapter may use.
. Make hidden workbook metadata an explicit cross-backend convention rather than an XLSX accident.
. Prepare validation and named-range handling for multiple spreadsheet formats.
. Improve maintainability by stating which semantics belong in IR, which belong in `RenderPlan`, and which must remain adapter-specific.

*Acceptance criteria:*

. Document the canonical spreadsheet semantics carried by `WorkbookIR` and `RenderPlan`, including at least sheet visibility, table anchors, freeze panes, autofilter regions, validations, named ranges, and workbook/sheet metadata.
. Define how hidden meta payload is represented conceptually, even if concrete storage differs by file format.
. Identify any current render-plan fields whose meaning is Excel-specific and either generalize or explicitly quarantine them.
. Document invariants for what `WorkbookIR` may contain directly versus what is derived later when building `RenderPlan`.
. Add a small set of focused unit tests for those invariants so the semantic split remains executable, not only documented.
. Add or update architecture docs so a future ODS adapter has a normative target behavior.

*Non-goals:*

- No implementation of a new adapter in this FTR.
- No expansion of spreadsheet feature scope beyond clarifying existing semantics.
- No requirement that every file format support every spreadsheet feature identically on day one.

*Why now:* The project is at the point where spreadsheet semantics should be treated as a first-class contract rather than reverse-engineered from the current XLSX implementation.
