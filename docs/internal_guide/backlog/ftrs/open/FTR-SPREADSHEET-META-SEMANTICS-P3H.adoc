===== FTR-SPREADSHEET-META-SEMANTICS-P3H — Specify spreadsheet-agnostic workbook metadata semantics

*Depends-on:* <<FTR-IR-ARCHITECTURE-CLARITY-P3H#,FTR-IR-ARCHITECTURE-CLARITY-P3H>>

*Context:* The rendering path already carries workbook and sheet semantics such as validations, named ranges, freeze panes, autofilters, helper-column highlighting, and hidden meta payload. In practice, parts of this are currently expressed through conventions that emerged from the XLSX/OpenPyXL implementation, especially around hidden `_meta` sheets, validation formulas, and an only partly explicit distinction between canonical meta and render-driving spreadsheet semantics.

*Problem statement:* A second spreadsheet adapter needs a precise statement of what belongs in `meta_canonical`, what is derived into `meta_rendering`, what must be reconstructible from a workbook file, and what may remain format-specific. Without that contract, XLSX and ODS would risk drifting into subtly different interpretations of the same IR.

*Goals:*

. Define spreadsheet semantics at the IR/plan boundary in a format-neutral way.
. Clarify the write-path split between `meta_canonical` and `meta_rendering`.
. State explicitly that `meta_rendering` may carry render- and spreadsheet-semantics such as named ranges or validations, not only visual layout.
. Make hidden workbook metadata an explicit cross-backend convention rather than an XLSX accident.
. Improve maintainability by stating which semantics belong in IR, which belong in `RenderPlan`, and which must remain adapter-specific.

*Acceptance criteria:*

. Document the canonical spreadsheet semantics carried by `meta_canonical` and `meta_rendering`, including at least sheet visibility, table anchors, freeze panes, autofilter regions, validations, named ranges, and workbook/sheet metadata.
. Define how hidden meta payload is represented conceptually, even if concrete storage differs by file format.
. Document which metadata belongs only in `meta_canonical`, which is derived into `meta_rendering`, and which semantics are expressed only in `RenderPlan`.
. Document how scope inheritance, precedence, and conflict resolution fit conceptually into the IR layer, even if no generic resolver mechanism is implemented yet.
. Record the write-path invariants that `meta_canonical` may be persisted but must not affect the render result after IR derivation, and that roundtrip-relevant information must not live only in `meta_rendering`.
. Identify any current render-plan fields whose meaning is Excel-specific and either generalize or explicitly quarantine them.
. Add a small set of focused unit tests for those invariants so the semantic split remains executable, not only documented.
. Add or update architecture docs so a future ODS adapter has a normative target behavior.

*Non-goals:*

- No implementation of a new adapter in this FTR.
- No expansion of spreadsheet feature scope beyond clarifying existing semantics.
- No immediate generic resolver framework driven by registry data.
- No final decision yet on reverse projection from workbook formatting back into canonical meta.

*Why now:* The project is at the point where spreadsheet semantics should be treated as a first-class contract rather than reverse-engineered from the current XLSX implementation.
