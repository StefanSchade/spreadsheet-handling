===== FTR-DEMO-REPO-INTEGRATION-P4A -- Realign the demo repository after consumer feedback

*Status:* Open
*Priority:* P4A
*Area:* Demo integration / beta readiness

==== Purpose

Bring the demo repository up to the current library capability set after the
prototype consumer pass has either validated the core behavior or exposed the
next library gaps.

The demo repository should remain cloneable and approachable for a new user,
but it now needs a little more structure than an ad-hoc pipeline collection.
Its detailed coordination lives in the demo repository's `DEMO-*` backlog; this
FTR tracks the main-library beta dependency.

==== Scope

* Use the demo repository's `docs/internal/backlog/DEMO-*` items as the working
  plan.
* Refresh maintained demo pipelines so they exercise the current Phase-4
  capabilities on the supported run path.
* Add or adjust automated demo smoke/integration checks for the curated
  journeys.
* Keep public documentation oriented around user journeys rather than internal
  feature names.
* Tag the aligned demo state for the beta release once the main library beta is
  ready.

==== Non-goals

* Do not move the main FTR backlog into the demo repository.
* Do not commit generated walkthrough HTML to the demo `main` branch.
* Do not showcase every internal transform separately if a smaller journey set
  tells the story better.

==== Acceptance

* The demo repository has a clear public entry point for the maintained
  journeys.
* The demo repository has `DEMO-*` backlog items for showcase structure,
  integration harness, walkthroughs, versioned Pages publishing, and capability
  showcases, with references back to the relevant main-library FTRs where
  useful.
* The curated demo flows run against the beta-candidate library checkout.
* Each maintained journey maps to pipeline/config, fixture data, expected
  output or verification, and user-facing documentation.
* The demo release/tag plan is clear enough for the next beta.
