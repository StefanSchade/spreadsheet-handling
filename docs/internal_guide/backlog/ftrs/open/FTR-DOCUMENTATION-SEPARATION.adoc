===== FTR-DOCUMENTATION-SEPARATION — Split "internal guide" to get the usual set of documents

*Context*::
The "internal guide" is a pragmatic place to collect pieces of documentation that need a place
(compare <<Motivation and Scope of the Internal Guide, Motivation and Scope>>).
Towards a release we want to provide a structure that is more consumer oriented and less dependent
on one catch-all document tree.

Today, `docs/internal_guide/` still mixes several documentation concerns that should eventually live
in more conventional homes: architecture explanation, developer workflow guidance, AI/collaboration
instructions, backlog/project planning, and publication-oriented user documentation.

There is also a practical AI-maintenance problem in the current arrangement: detailed FTR history,
review notes, and older planning context tend to remain in the active documentation tree forever.
That makes repository snapshots and fresh-context continuation heavier than necessary, even when most
of that detail is historical background rather than active working context.

*Goal*::
Create an adequate set of documents you would expect as an observer, and redistribute the current
internal-guide content into those destinations without losing information.

*Rationale*::
Project documentation should become more accessible, more maintainable, and easier to navigate for
both humans and AI-assisted contributors. It should also distinguish better between active working
context and long-tail historical detail so snapshot-based AI context stays leaner.

*Acceptance*::
* Provide a minimal set of documents you would expect:
- Developer Guide
- Architecture Documentation (for example a lightweight arc42-style structure)
- Project Planning / Backlog documentation
- AI / collaboration instructions where still useful
- Project history documentation with a condensed historical narrative
- Archive material for detailed retired history that should remain accessible but not stay in the active path
- Public-facing user and project documentation
* Each document should have the content and structure an observer would expect.
* Information currently collected under `docs/internal_guide/` is retained, but moved to an adequate
  long-term home instead of remaining in one pragmatic catch-all area.
* The target structure is explicit enough to guide migration, for example:
- `docs/architecture/`
- `docs/dev_guide/`
- `docs/ai_guide/`
- `docs/backlog/`
- `docs/project_history/`
- `docs/archive/`
* The migration explicitly reviews where current sections belong, including at least:
- architecture material from `docs/internal_guide/architecture/`
- developer workflow material from `docs/internal_guide/dev_man/`
- AI guidance and repo-context instructions
- backlog/planning material
* Historical FTR detail is no longer all carried in the active contributor context by default:
- active backlog and current architecture/dev/AI guidance stay in the primary doc paths
- compressed phase/history summaries remain available in `project_history/`
- older detailed FTR/review material can move to `archive/` where appropriate
* Snapshot-oriented tooling such as `tools/repo_snapshot.sh` is adjusted so archival material can be
  filtered out of the default AI/review snapshot path.
* Cross-links and entry points remain good enough that contributors can still find things during the transition.
