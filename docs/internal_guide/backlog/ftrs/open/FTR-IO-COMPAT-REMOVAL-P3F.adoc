===== FTR-IO-COMPAT-REMOVAL-P3F -- Remove remaining backward-compat hacks from I/O layer

*Status:* Open +
*Epic:* Phase 3f -- I/O Infrastructure Cleanup

*Purpose:*::
Phase 3e removed dead modules and zombie imports, but several defensive patterns survive inside the active I/O layer: dual-order function signatures, TypeError fallbacks in `BackendBase`, and duplicated `_coerce_options()` helpers. These add branching complexity without protecting against real scenarios -- all active backends already conform to the current signatures.

*Findings (verified):*

[cols="1,2,1"]
|===
| Location | Issue | Action

| `json_backend.py` `write_json_dir()` (line 177)
| Accepts `(path, frames)` and `(frames, path)` via runtime type-sniffing -- same pattern we already fixed in `write_xml_dir()` during P3E.
| Standardize to `(frames, path)` matching router convention.

| `base.py` `BackendBase.write_multi()` (line 51)
| Wraps `self.write()` in `try/except TypeError` to tolerate backends that don't accept `options`. All 4 active backends (JSON, XML, XLSX, CSV) accept `options`.
| Remove TypeError fallback; call `self.write()` directly.

| `base.py` `BackendBase.read_multi()` (line 60)
| Same `try/except TypeError` pattern for `self.read()`.
| Remove TypeError fallback; call `self.read()` directly.

| `json_backend.py` `_coerce_options()` (line 67) + `xml_backend.py` `_coerce_options()` (line 24)
| Identical function duplicated in two backends. Converts `None\|dict\|BackendOptions` to BackendOptions with defensive `is_dataclass` check.
| Move to `base.py` as single shared utility, or inline since `BackendOptions` is already a dataclass.
|===

*Scope:*::
. `write_json_dir()`: single signature `(frames, path, *, options)`.
. `BackendBase.write_multi/read_multi`: remove `try/except TypeError`.
. `_coerce_options()`: deduplicate (move to `base.py` or inline).
. Update all callers (router.py registers savers as `(frames, path)` -- already correct).

*Non-Goals:*::
- No changes to the Backend class hierarchy itself (that is FTR-BACKEND-ABC in Phase 6).
- No changes to router registration mechanism (that is FTR-ROUTER-DECLARATIVE-P3F).

*Acceptance:*::
- No `try/except TypeError` in `base.py`.
- No dual-order parameter signatures in any backend convenience function.
- `_coerce_options` exists in exactly one place.
- All tests pass.
