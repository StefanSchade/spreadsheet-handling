===== FTR-XLSX-ADAPTER-ALIGNMENT-P3H — Align the XLSX adapter with the generalized spreadsheet contract

*Depends-on:* <<FTR-SPREADSHEET-BACKEND-CONTRACT-P3H#,FTR-SPREADSHEET-BACKEND-CONTRACT-P3H>>, <<FTR-SPREADSHEET-META-SEMANTICS-P3H#,FTR-SPREADSHEET-META-SEMANTICS-P3H>>

*Context:* Once the spreadsheet-backend contract and spreadsheet-neutral metadata semantics are defined, the current XLSX path becomes the reference implementation that should embody those decisions cleanly. Today, the XLSX adapter already uses the IR path, but naming and module boundaries still leak some XLSX-specific assumptions into generic areas.

*Problem statement:* If the current XLSX implementation is not first aligned to the new abstractions, an ODS/Calc adapter would either copy transitional design debt or force simultaneous refactoring while the new adapter is being built.

*Goals:*

. Make the XLSX implementation a clean instance of the generalized spreadsheet contract.
. Isolate OpenPyXL-specific details to XLSX-facing modules.
. Ensure the read and write paths both terminate at the same spreadsheet-neutral abstractions.
. Create a stable reference adapter before introducing ODS/Calc.
. Improve navigability by making the insertion point between generic spreadsheet flow and XLSX-specific code obvious.

*Acceptance criteria:*

. XLSX rendering and parsing modules implement the new spreadsheet contract without requiring XLSX-specific names in generic packages.
. Generic rendering and parsing layers no longer import or describe XLSX concepts unless the code is intentionally format-specific.
. The module layout and naming make it clear where `WorkbookIR` ends, where `RenderPlan` begins, and where OpenPyXL-specific logic starts.
. Adapter-side tests verify that the XLSX adapter uses the narrow rendering/parsing API rather than deep, ad-hoc imports into generic rendering internals.
. Existing XLSX regression coverage continues to pass with no intended behavior changes.
. The resulting module layout makes the insertion point for an ODS/Calc adapter obvious.

*Non-goals:*

- No ODS/Calc behavior in this FTR.
- No attempt to redesign the full feature set of the XLSX renderer.
- No product-surface CLI changes beyond any minimal new backend-kind plumbing needed later.

*Why now:* It is cheaper to turn the existing XLSX path into a clean baseline than to develop two spreadsheet adapters against a moving abstraction boundary.
