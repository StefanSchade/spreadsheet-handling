===== FTR-YAML-SURFACE-NORMALIZATION-P5 -- Normalize step names and parameters after beta

*Status:* Open
*Priority:* P5
*Area:* Pipeline API / documentation polish

==== Purpose

Review and normalize the public YAML configuration surface after the next beta
has exercised the current feature wave through real consumer and demo flows.

The library now has strong mechanics, but the YAML surface still reflects
several development waves. Some steps use `source`, others `source_frame`,
`relation`, `matrix`, `output`, or `target_frame`. Those names are often
locally sensible, but users need a calmer and more predictable interface before
the project moves toward a broader public documentation story.

==== Trigger

Run this FTR after the next beta has been cut and the transformation catalog
has exposed enough of the surface to make inconsistencies visible.

The catalog is not just documentation. It is the review tool for this FTR:
every awkward table entry is a signal that the YAML API may need an alias,
rename, or clearer category boundary.

==== Scope

* Review maintained pipeline steps by category:
** transforms
** view preparation
** workbook view composition
** validations
** output writers
** metadata/configuration steps
* Normalize input and output parameter naming within each category.
* Decide where terms such as `source`, `source_frame`, `relation`, `matrix`,
  `output`, `target_frame`, `column_key`, `column_keys`, `value`,
  `value_column`, and `value_columns` should be preferred.
* Add compatibility aliases where needed instead of breaking beta users
  abruptly.
* Mark deprecated spellings in the pipeline step registry and docs.
* Update examples and concept snippets to the preferred spelling.
* Identify any step whose shape should remain intentionally different because
  the underlying concept is different.

==== Goals

* Reduce cognitive load for YAML authors.
* Make related inverse pairs feel symmetrical.
* Keep the registry as the source of truth for public step names, aliases, and
  parameter contracts.
* Preserve backwards compatibility through a documented deprecation window.
* Use the transformation catalog to validate that the resulting surface can be
  explained compactly.
* Avoid making plugin/dotted-path steps look like the preferred path for normal
  workbook composition.

==== Non-goals

* Do not redesign pipeline execution semantics.
* Do not remove plugin support.
* Do not break existing beta pipelines without an alias/deprecation path.
* Do not add new transformation capability unless a tiny alias/helper is needed
  to make the normalized surface coherent.
* Do not freeze a 1.0 API contract prematurely.

==== Acceptance

* The transformation catalog and step registry agree on preferred public names.
* Existing examples use the preferred parameter spelling.
* Legacy spellings that remain supported are documented as aliases or
  deprecated forms.
* At least the xref, compact multiaxis, discriminator split, extraction,
  validation, workbook view, and writer families have been reviewed.
* Tests cover representative alias compatibility where renamed parameters are
  accepted.
* The resulting YAML surface can be summarized by category without repeatedly
  explaining accidental historical naming differences.
