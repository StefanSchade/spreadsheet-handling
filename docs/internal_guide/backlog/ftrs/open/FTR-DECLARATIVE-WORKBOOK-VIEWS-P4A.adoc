===== FTR-DECLARATIVE-WORKBOOK-VIEWS-P4A -- Configure workbook views without consumer plugins

*Status:* Open
*Priority:* P4A
*Area:* Pipeline / workbook view composition

==== Purpose

Make the BA-config workbook shape configurable with generic pipeline steps
instead of requiring a consumer-local `workbook_views` plugin for ordinary view
composition.

Plugins should remain available for domain-specific extraction, validation, or
generation logic. They should not be required just to select columns, order
sheets, rename visible sheets, sort rows, and assemble an editable workbook view
from generic frames.

==== Consumer Trigger

The BA-config Elmar workbook currently uses
`plugins.nxt_config.workbook_views:build_elmar_workbook_views` to create
friendly tabs such as `Variablen`, `Produkte`, `Variablen Matrix`, and KS3.0
mapping views.

Some of that logic is genuinely consumer-specific naming and presentation
choice, but the shape is not credit-specific: it is a declarative view layer on
top of normalized frames.

==== Goals

* Add one or more generic steps for declaring workbook-visible views.
* Support sheet renaming and output sheet order.
* Support column selection and ordering.
* Support row sorting by configured columns.
* Support copying a frame as a visible view without mutating the canonical
  source frame.
* Preserve frame lifecycle metadata so canonical sources, derived views,
  editable projections, and omitted intermediates remain explicit.
* Keep ODS and XLSX behavior aligned at the adapter contract level.

==== Non-goals

* Do not implement a SQL engine in this slice.
* Do not move BA-specific NXT generation logic into the core library.
* Do not require the generic view layer to express every possible pandas
  transformation.
* Do not remove plugin support.

Parameterized joins, pivots, or richer query-like view definitions can become
later FTRs if repeated consumers need them.

==== Acceptance

* A consumer can replace the simple BA `workbook_views` plugin cases with
  declarative YAML for view selection, sheet naming, column ordering, and
  sorting.
* The selected view frames render in the configured order.
* View frames carry lifecycle metadata describing whether they are canonical,
  derived, editable, visible, or omitted.
* Tests cover at least one XLSX and one ODS render/read smoke path for the
  configured workbook views.
* The BA-config Elmar workbook can use the generic view path for the parts that
  are not domain-specific.
