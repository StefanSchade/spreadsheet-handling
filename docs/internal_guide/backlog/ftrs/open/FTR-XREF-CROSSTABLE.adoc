===== FTR-XREF-CROSSTABLE -- Cross-Table (Matrix) <-> Relation Transformation

*Status:* Implemented (first generic slice)

*Purpose:*::
Provide a generic transformation between matrix/cross-table representations and
explicit relational representations.

*Context:*::
A common compact spreadsheet pattern uses one axis for row entities and another
axis for column entities. The cell value represents the configured state for
that row/column pair.
+
Neutral example:

* rows: features or configuration items
* columns: product variants, optionally grouped by product family
* cells: compact codes such as `S`, `E`, `E-R-K`, `A-V-Z`, or `x`

*Scope:*::

* Transform matrix layout into a relation of the form
  `(row_key, column_key, value)`.
* Transform such relations back into matrix form.
* Preserve all cell values without interpreting their semantics.
* Preserve row/column identity through configured or persisted axis mappings.

*Identity Contract:*::

Visible labels are display-only.
+
The transform must not reconstruct identity by parsing visible labels, grouped
headers, or flattened helper text. Import must use configured keys or persisted
axis mappings.
+
Sparse relations must not be densified by heuristic reconstruction. If a
relation does not contain a row/column pair, `contract_xref` may fill that pair
only when the target axis space is explicitly configured, provided directly, or
available from persisted contract metadata. It must not guess missing axes from
visible labels, table positions, or name similarity.

*Non-Goals:*::

* No interpretation of cell values.
* No validation beyond structural consistency.
* No cell-code parsing; that belongs to `FTR-CELL-CODEC`.
* No domain-specific product, market, or business semantics.
* No heuristic reconstruction of missing row/column combinations.

*Solution:*::

* `expand_xref`: matrix -> list of `(row_key, column_key, value)` tuples.
* `contract_xref`: tuple list -> matrix.
* Support configurable row and column key detection.
* Persist axis mappings when the rendered matrix cannot carry stable keys
  visibly.

*First Implementation Slice:*::

The first implementation provides two frames-first pipeline steps:

[source,yaml]
----
pipeline:
  - step: expand_xref
    matrix: product_matrix
    output: feature_product_codes
    row_keys: [feature_id]
    value_columns: [P-001, P-002]

  - step: contract_xref
    relation: feature_product_codes
    output: product_matrix_roundtrip
    row_keys: [feature_id]
----

The expanded relation uses these default output columns:

[source,text]
----
feature_id | column_key | value
----

The first slice treats matrix cell values as opaque. A value such as `E-R-K`
is copied as one value and is not parsed into tokens.
+
The default is lossless rather than sparse: `drop_empty` defaults to `false`,
so empty cells are preserved as explicit long-form rows. Sparse expansion is
available by setting `drop_empty: true` and is intentionally opt-in.
+
The first slice supports flat column labels only. MultiIndex/tuple columns are
rejected because their identity and YAML configuration shape are not yet part
of this FTR.
+
`expand_xref` writes `xref_crosstable` metadata containing the row-key names
and matrix column-key order. `contract_xref` uses that metadata when explicit
`column_keys` are not supplied, so `matrix -> relation -> matrix` can preserve
the original column order.

*Future Dense Axis Contract:*::

A future mode may support explicit dense-axis reconstruction from sparse
relations. This is a contract shape to prepare for, not part of the first
implementation slice.
+
Example sketch:

[source,yaml]
----
dense_axes:
  rows_from:
    frame: Products
    key: product_id
  columns_from:
    frame: Markets
    key: market_id
----

Semantics:

* the relation may be sparse
* the full matrix axis space is read from explicit axis sources
* missing `(row, column)` combinations receive `fill_value`
* axis identity is based on configured keys
* no axis may be guessed from visible labels, grouped headers, table position,
  or frame-name similarity

Until such a mode is explicitly implemented, strict shape-preserving roundtrips
must keep empty cells in the relation or pass the target `column_keys`
explicitly where the existing first-slice API supports it.

*Acceptance:*::

* Transformation is lossless for non-ambiguous inputs.
* Empty and missing values are handled consistently.
* Roundtrip `matrix -> relation -> matrix` produces identical structure.
* Display labels are not used as identity during reimport.
* Implementation is independent of value semantics.
