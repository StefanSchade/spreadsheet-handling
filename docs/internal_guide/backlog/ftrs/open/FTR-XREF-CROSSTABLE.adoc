===== FTR-XREF-CROSSTABLE -- Cross-Table (Matrix) <-> Relation Transformation

*Status:* New

*Purpose:*::
Provide a generic transformation between matrix/cross-table representations and
explicit relational representations.

*Context:*::
A common compact spreadsheet pattern uses one axis for row entities and another
axis for column entities. The cell value represents the configured state for
that row/column pair.
+
Neutral example:

* rows: features or configuration items
* columns: product variants, optionally grouped by product family
* cells: compact codes such as `S`, `E`, `E-R-K`, `A-V-Z`, or `x`

*Scope:*::

* Transform matrix layout into a relation of the form
  `(row_key, column_key, value)`.
* Transform such relations back into matrix form.
* Preserve all cell values without interpreting their semantics.
* Preserve row/column identity through configured or persisted axis mappings.

*Identity Contract:*::

Visible labels are display-only.
+
The transform must not reconstruct identity by parsing visible labels, grouped
headers, or flattened helper text. Import must use configured keys or persisted
axis mappings.

*Non-Goals:*::

* No interpretation of cell values.
* No validation beyond structural consistency.
* No cell-code parsing; that belongs to `FTR-CELL-CODEC`.
* No domain-specific product, market, or business semantics.

*Solution:*::

* `expand_xref`: matrix -> list of `(row_key, column_key, value)` tuples.
* `contract_xref`: tuple list -> matrix.
* Support configurable row and column key detection.
* Persist axis mappings when the rendered matrix cannot carry stable keys
  visibly.

*Acceptance:*::

* Transformation is lossless for non-ambiguous inputs.
* Empty and missing values are handled consistently.
* Roundtrip `matrix -> relation -> matrix` produces identical structure.
* Display labels are not used as identity during reimport.
* Implementation is independent of value semantics.
