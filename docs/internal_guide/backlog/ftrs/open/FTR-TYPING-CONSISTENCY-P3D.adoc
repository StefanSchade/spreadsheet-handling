===== FTR-TYPING-CONSISTENCY-P3D — Establish consistent typing after architectural stabilization

*Status:* Open
*Epic:* Phase 3d — Code Quality & Maintainability

*Purpose:*::
Introduce consistent and meaningful type annotations across the codebase after Phase 3b/3c have stabilized architectural boundaries, ensuring that typing reflects real responsibilities rather than transitional structure.

*Context:*::
During earlier phases, architectural responsibilities were still shifting, making strict typing premature. With domain, pipeline, and rendering boundaries clarified, type annotations can now serve as a reliable form of documentation and error prevention.

*Scope:*::
- Add type hints to core data flows (e.g., Frames, Meta, Step interfaces).
- Ensure consistent typing across domain, pipeline, and rendering layers.
- Introduce or standardize shared types where beneficial (e.g., aliases, TypedDict, dataclasses).
- Improve IDE support and static analysis without introducing excessive complexity.

*Non-Goals:*::
- No premature or speculative typing of unstable APIs.
- No requirement for full mypy/strict compliance across the entire codebase.
- No redesign of architecture driven by typing constraints.

*Design Principle:*::
Typing follows architecture. Types should reinforce clear boundaries and responsibilities, not compensate for unclear structure.

*Acceptance:*::
- Core modules have consistent and readable type annotations.
- Type annotations align with actual responsibilities (domain vs pipeline vs rendering).
- No misleading or placeholder types remain.
- Typing improves readability without significantly increasing cognitive overhead.
