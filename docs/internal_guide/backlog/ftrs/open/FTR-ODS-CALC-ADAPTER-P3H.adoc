===== FTR-ODS-CALC-ADAPTER-P3H — Add a full LibreOffice Calc / ODS adapter on top of the spreadsheet-neutral IR path

*Depends-on:* <<FTR-XLSX-ADAPTER-ALIGNMENT-P3H#,FTR-XLSX-ADAPTER-ALIGNMENT-P3H>>

*Context:* After the spreadsheet-backend contract is explicit, metadata semantics are spreadsheet-neutral, and the XLSX path has been aligned to those boundaries, the project is ready for a second full spreadsheet backend.

*Problem statement:* The system currently offers one full spreadsheet file format with integrated IR read/write behavior. That limits interoperability and keeps spreadsheet-neutral architectural claims unproven until a second implementation exists.

*Goals:*

. Add an ODS/Calc adapter that participates in the same `Frames <-> WorkbookIR <-> RenderPlan <-> workbook file` architecture as XLSX.
. Support both write and read paths through the spreadsheet-neutral abstractions.
. Preserve the existing project model in which spreadsheet formats are backend choices, not alternate product modes.
. Confirm through a second implementation that the generalized spreadsheet IR model is understandable and usable beyond the original XLSX path.

*Acceptance criteria:*

. Introduce an `ods` backend kind that can be selected through the existing I/O routing model.
. ODS write path renders workbook structure from the shared IR/render-plan pipeline.
. ODS read path reconstructs `WorkbookIR` and returns frames through the same high-level contract as XLSX.
. Core spreadsheet semantics defined in Phase 3h are supported to the degree the target format/library allows, with any gaps documented explicitly.
. Add focused tests that exercise ODS round-trip behavior and compare semantics against the shared spreadsheet contract rather than byte-level file identity.
. Add lightweight dependency-guard coverage so ODS-specific library details stay inside the ODS adapter boundary.
. Any semantic mismatch that remains format-specific is documented at the adapter boundary rather than leaking back into generic IR terminology.

*Non-goals:*

- No GUI automation of LibreOffice itself.
- No promise of perfect parity for every Excel-specific behavior if the ODS format cannot represent it directly.
- No expansion of unrelated domain features.

*Why now:* With the architecture stabilized, a second spreadsheet adapter becomes an implementation exercise rather than an architectural gamble.
