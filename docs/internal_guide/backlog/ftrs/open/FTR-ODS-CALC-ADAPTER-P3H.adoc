===== FTR-ODS-CALC-ADAPTER-P3H — Add a full LibreOffice Calc / ODS adapter on top of the spreadsheet-neutral IR path

*Status:* Deferred for reformulation after Phase 3h / 3i

*Context:*::
This ticket was drafted when Phase 3h was first outlined and still assumed that the ODS/Calc adapter
could already be specified concretely enough to live inside the same phase.

After reassessment, that is too early. Phase 3h now explicitly focuses on clarifying spreadsheet
contracts, IR semantics, narrow adapter APIs, and lightweight architectural guardrails. Phase 3i is
reserved to turn those clarified boundaries into drift protection. The exact shape of a full ODS/Calc
adapter should be derived from those outcomes, not frozen beforehand.

*Decision:*::
Keep this ticket only for traceability.
Do not implement it as a concrete Phase 3h FTR.
A later post-3h phase will define the actual ODS/Calc implementation work once the spreadsheet
architecture and guardrails have settled.
That follow-up backlog now lives in Phase 3j via
<<FTR-ODS-CALC-CONTRACT-MAPPING-P3J#,FTR-ODS-CALC-CONTRACT-MAPPING-P3J>>,
<<FTR-ODS-CALC-ADAPTER-IMPLEMENTATION-P3J#,FTR-ODS-CALC-ADAPTER-IMPLEMENTATION-P3J>>,
and <<FTR-ODS-CALC-PARITY-TESTS-P3J#,FTR-ODS-CALC-PARITY-TESTS-P3J>>.

*Original intent (retained for traceability):*::

- Add an `ods` backend kind that participates in the same `Frames <-> WorkbookIR <-> RenderPlan <-> workbook file` architecture as XLSX.
- Support both write and read paths through the spreadsheet-neutral abstractions.
- Use a second spreadsheet implementation to validate the generalized IR model.

*Why deferred:*::

- The adapter scope depends on Phase 3h decisions about spreadsheet contracts and semantics.
- It also depends on Phase 3i decisions about how those boundaries are guarded against drift.
- Freezing the adapter FTR now would risk specifying the wrong seam too early.
