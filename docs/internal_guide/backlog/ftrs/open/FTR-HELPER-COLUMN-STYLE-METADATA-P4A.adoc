===== FTR-HELPER-COLUMN-STYLE-METADATA-P4A -- Style helper columns by explicit metadata

*Status:* Open
*Priority:* P4A
*Area:* Rendering / helper metadata

==== Purpose

Allow derived helper columns to be marked and styled explicitly, independent of
column-name prefixes.

The current renderer can highlight helper columns by prefix convention. That is
useful for internal helper names, but consumer-facing workbooks often need
readable helper columns such as `value_label_de`, `business_component`, or
`data_type`. Those columns should still be visibly distinct from editable data
columns.

==== Consumer Trigger

In the BA-config `Variablen Matrix`, columns projected from `variables` are
helper/context columns. They are not the relation's editable matrix values, but
their names should remain meaningful for a BA user.

==== Goals

* Add a canonical or documented metadata shape for helper/display columns.
* Allow pipeline steps such as `add_lookup_helpers` and declarative workbook
  views to mark projected helper columns explicitly.
* Render helper-column styling from metadata rather than only by prefix.
* Keep the existing prefix behavior as a compatibility/default path.
* Support per-sheet or per-view helper fill color configuration.
* Keep behavior backend-neutral and implement support for both XLSX and ODS
  where styling support exists.

==== Non-goals

* Do not infer helper columns from arbitrary column names unless configured.
* Do not implement full cell-level style authoring.
* Do not make helper styling a data-semantic property.

==== Acceptance

* Helper columns can be marked explicitly in metadata or step output.
* XLSX rendering highlights explicitly marked helper columns.
* ODS rendering highlights the same explicitly marked helper columns, or an
  adapter gap is recorded with a focused test.
* Prefix-based helper highlighting continues to work.
* BA-config can mark matrix context columns as helpers without renaming them to
  prefixed technical names.
