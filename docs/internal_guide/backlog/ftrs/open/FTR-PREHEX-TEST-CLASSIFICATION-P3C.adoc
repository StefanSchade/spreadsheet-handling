===== FTR-PREHEX-TEST-CLASSIFICATION-P3C — Classify and resolve legacy pre-hex tests

*Status:* Open
*Epic:* Phase 4 — Feature Expansion and Domain Transformations
*Priority:* Medium

*Depends-on:*

* <<FTR-TEST-SUITE-TOPOLOGY-P3C#,FTR-TEST-SUITE-TOPOLOGY-P3C>>

*Related-to:*

* <<FTR-ARCHITECTURE-TEST-LAYER-P3C#,FTR-ARCHITECTURE-TEST-LAYER-P3C>>
* <<FTR-TEST-MARKERS-AND-MAKE-TARGETS-P3C#,FTR-TEST-MARKERS-AND-MAKE-TARGETS-P3C>>
* <<FTR-INTEGRATION-SLICE-CONSOLIDATION-P3C#,FTR-INTEGRATION-SLICE-CONSOLIDATION-P3C>>

'''

====== Purpose

Resolve the ambiguous status of legacy pre-hex tests by explicitly classifying each test as retained behavior, rewritten behavior, or obsolete implementation artifact.

The goal is to eliminate the current situation where historical tests are simultaneously ignored in execution but implicitly treated as a hidden source of requirements.

This FTR ensures that all still-relevant behavior expressed in pre-hex tests is either promoted into the active test suite or consciously replaced, while obsolete tests are removed or archived with intent.

'''

====== Context

The repository contains a set of historical tests under `tests/legacy_pre_hex/` (or equivalent), originating from earlier architecture phases before the current Domain / Pipeline / Render IR / Spreadsheet Contract separation was established.

These tests fall into three categories:

* tests that still describe valid product behavior
* tests that describe behavior tied to obsolete internal structure
* tests that mix both and are therefore unclear

Currently, these tests are quarantined and excluded from the default test run. This avoids immediate breakage but creates long-term ambiguity:

* relevant behavior may only be protected by ignored tests
* obsolete behavior may still influence design decisions indirectly
* developers cannot easily tell whether a failing pre-hex test indicates a regression or expected evolution

The backlog already acknowledges this situation and requires an explicit classification step.

'''

====== Problem statement

The project carries a non-trivial amount of legacy tests whose status is undefined.

This leads to:

* hidden requirements that are not part of the active test suite
* duplicated or conflicting specifications between new and legacy tests
* uncertainty during refactoring (is this behavior still required?)
* reluctance to delete outdated logic because tests still exist somewhere

Without explicit classification, the legacy test area becomes a permanent gray zone.

'''

====== Goals

. Classify every pre-hex test into a small set of explicit categories.
. Ensure that all still-relevant behavior is protected by active tests.
. Remove or archive obsolete tests with clear intent.
. Make the remaining legacy area minimal and well-understood.
. Avoid mixing historical structure with the new architecture.

'''

====== Classification categories

Each test in `tests/legacy_pre_hex/` must be assigned exactly one of the following categories:

====== promote

The test expresses still-valid behavior that fits the current architecture.

Action:

* move the test (possibly with minimal adaptation) into `tests/unit/`, `tests/integration/`, or `tests/architecture/`
* remove it from the pre-hex directory

Typical cases:

* stable domain transformations
* valid spreadsheet contract behavior
* still-relevant rendering semantics expressed in older form

====== rewrite

The test expresses relevant behavior but is tightly coupled to obsolete structure.

Action:

* write a new test in the active suite that captures the same intent using the current architecture
* remove or archive the original test after replacement

Typical cases:

* tests relying on deprecated intermediate structures
* tests mixing domain logic with adapter details
* tests asserting behavior at the wrong abstraction level

====== retire

The test describes behavior that is no longer required or was tied to an obsolete design.

Action:

* remove the test or move it to an archive location with minimal documentation

Typical cases:

* tests verifying internal data layout that no longer exists
* tests tied to deprecated APIs or representations
* tests whose behavior contradicts current architectural decisions

'''

====== Scope

This FTR includes:

* iterating over all files in `tests/legacy_pre_hex/`
* assigning one classification (`promote`, `rewrite`, `retire`) per test file or per logical test group
* documenting the classification decision (inline comment, short note, or tracking file)
* promoting or rewriting tests where straightforward
* removing or archiving clearly obsolete tests

This FTR may be executed incrementally, but each touched file must be fully classified.

'''

====== Non-goals

This FTR does not:

* guarantee that all legacy tests are processed in a single pass
* rewrite large complex tests in one step if they require deeper design decisions
* introduce new product features
* change production behavior to satisfy legacy tests
* merge legacy tests blindly into the new suite without evaluation

'''

====== Execution strategy

. Start with small, clearly understandable test files.
. Prefer `promote` when behavior is obviously still valid.
. Use `rewrite` when intent is clear but structure is outdated.
. Use `retire` when the test no longer reflects intended system behavior.
. Avoid long debates per test; if unclear, mark temporarily and continue.
. Track progress explicitly (e.g. checklist or lightweight index file).

The goal is steady reduction of the legacy surface, not perfect classification in one iteration.

'''

====== Documentation

For each classified test (file or group), record:

* chosen category (`promote`, `rewrite`, `retire`)
* short rationale (1–2 lines)

This can be done via:

* inline comments in the test file
* a simple tracking file under `tests/legacy_pre_hex/`

Documentation should be minimal but sufficient to avoid re-discussion later.

'''

====== Acceptance criteria

. Each test in `tests/legacy_pre_hex/` has an explicit classification or is part of a clearly marked remaining subset.
. All behavior considered still relevant is covered by tests in the active suite (`unit`, `integration`, or `architecture`).
. No critical behavior depends solely on pre-hex tests.
. Obsolete tests are removed or archived intentionally.
. The size of the pre-hex directory is measurably reduced or clearly partitioned.
. The default test run remains unchanged (pre-hex tests still excluded unless explicitly run).
. Developers can determine for any remaining pre-hex test whether it is pending classification or intentionally retained.

'''

====== Rationale

Keeping legacy tests indefinitely without classification creates hidden coupling to past architecture decisions.

Explicit classification forces decisions:

* what behavior still matters
* what must be expressed in the new architecture
* what can be safely discarded

This reduces uncertainty during refactoring and prevents legacy tests from silently constraining future design.

The effort is moderate but pays off by removing a persistent source of ambiguity in the codebase.
