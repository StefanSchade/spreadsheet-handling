===== FTR-META-SCHEMA-VALIDATION-P5 -- Lightweight validation for _meta contracts

*Status:* Open
*Epic:* Phase 5 -- Stabilization and Internal Quality
*Priority:* Medium

*Origin:*::
External architecture critique (2026-04-29) raised the concern that `_meta`
paths are documented in `meta_registry.yaml` but not validated at runtime.
The observation is directionally useful: as the number of configuration steps
grows, the risk of undocumented or conflicting `_meta` namespaces increases
without a lightweight safety net.

The related `FTR-META-SCHEMA-V2` (Phase 6) addresses schema consolidation:
defining canonical shapes for specific meta fields. This FTR addresses a
smaller Phase-5 concern: explicit, opt-in structural validation where the meta
registry work already provides a machine-readable contract.

'''

====== Purpose

Add a lightweight validation mechanism that checks actual `_meta` namespaces
and declared shapes against an explicit validation contract derived from the
meta registry work.

The goal is to catch typos, top-level path collisions, undocumented meta
namespaces, and simple type mismatches early without turning `_meta` into a
heavy schema system.

'''

====== Scope

This FTR includes:

* a validation function that can inspect a dataframe's `_meta` dictionary
  without mutating it
* an explicit validation contract for selected registry entries; this may live
  in `meta_registry.yaml` as dedicated machine-readable fields or in a generated
  artifact derived from it
* a validator that uses only explicit validation fields and does not infer
  runtime rules from prose fields such as `meaning`, `scope`, or narrative
  producer descriptions
* diagnostics for unknown top-level `_meta` namespaces
* diagnostics for type mismatches where a declared validation shape exists
* integration as an opt-in diagnostic pipeline step (`validate_meta`) or as a
  guard/test utility used in test/CI pipelines

Validation is structural only:

* known namespace
* declared type family
* declared nested path, where a validation shape exists

'''

====== Non-goals

This FTR does not:

* replace or redesign `meta_registry.yaml`
* enforce semantic invariants within meta values; that remains the
  responsibility of individual steps or later dedicated guards
* introduce JSON Schema or a heavy validation framework
* block production pipelines by default; the validator is opt-in or test-only
* overlap with `FTR-META-SCHEMA-V2`, which focuses on schema consolidation for
  specific meta fields
* create a direct runtime dependency on the current manual-review registry
  format
* infer producer/provenance validity unless such provenance is explicitly
  present in the metadata itself

'''

====== Acceptance criteria

. A `validate_meta` function or optional step exists that checks `_meta`
  against explicit validation entries.
. Unknown top-level `_meta` namespaces produce a clear diagnostic.
. Type mismatches produce a clear diagnostic for at least one registry entry
  with a declared validation shape.
. Entries without a declared validation shape are reported as uncovered rather
  than guessed from prose.
. The validation can be invoked as a pipeline step, architecture guard, or test
  utility.
. Existing pipelines are not affected unless they opt in.
. The meta registry remains the source of truth for reviewed meta ownership; the
  validation contract is either stored there explicitly or generated from it,
  not maintained as an unrelated parallel schema.

'''

====== Implementation notes

Not yet implemented.

Suggested sequence:

. Add explicit validation metadata for a small initial set of high-value
  entries, for example FK helper policies and compact transform metadata.
. Add a loader that builds a validation map only from those explicit fields.
. Walk actual `_meta` and report unknown namespaces or incompatible declared
  paths.

The current `meta_registry.yaml` is marked as a manual review artifact. This
FTR should therefore either add explicit validation fields to that contract or
use a generated validation artifact whose maintenance path is documented. It
must not treat the current prose-oriented registry as a runtime schema by
accident.

If the implementation starts needing a general schema engine, defer that scope
to `FTR-META-SCHEMA-V2` instead.

'''

====== Relation to existing work

* `meta_registry.yaml` -- the existing review artifact for meta ownership.
* `FTR-META-SCHEMA-V2` (Phase 6) -- consolidates meta field shapes. This FTR
  validates structural contracts for selected fields.
* `FTR-PIPELINE-SEMANTICS-GUARDS-P4` -- guards for pipeline-level invariants.
  Meta validation is complementary but operates at the data level, not the
  pipeline level.
