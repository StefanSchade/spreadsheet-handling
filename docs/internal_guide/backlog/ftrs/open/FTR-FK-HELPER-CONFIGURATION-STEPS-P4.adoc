====== Purpose

Add `configure_fk_helpers` as the FK-convention counterpart to `configure_lookup_helpers`.

The lookup-helper configuration slice introduced the shared `_meta.helper_policies` root and the rule that configuration steps resolve policy before primitive helper execution.

This FTR applies that rule to FK-convention helper columns.

'''

====== Scope

This FTR includes:

* `configure_fk_helpers` runtime step
* resolved `_meta.helper_policies.fk` policy shape
* explicit target policies with key, allowed helpers, default helpers, and FK-column convention
* selected `auto` inference for FK targets and helper defaults
* `add_fk_helpers` support for consuming resolved FK policy where available
* validation of default helpers against allowlists
* conflict detection between inline `add_fk_helpers` defaults and resolved FK policy
* runtime step-registry and JSON step-registry updates
* tests for explicit FK policy, policy-backed materialization, invalid helper requests, and legacy inline behavior

'''

====== Non-goals

This FTR does not:

* redesign FK column naming globally
* remove direct inline `add_fk_helpers` usage
* remove helper provenance based cleanup
* implement formula-driven helper cells
* implement frame lifecycle or workbook view materialization

'''

====== Acceptance criteria

. `configure_fk_helpers` exists and is registered as a public configuration step.
. `configure_fk_helpers` writes concrete `_meta.helper_policies.fk` metadata.
. `add_fk_helpers` can consume resolved FK helper policy.
. `auto` is interpreted only by `configure_fk_helpers`, not by primitive execution.
. Default FK helpers are validated against allowed helpers.
. Inline FK helper config conflicting with resolved policy fails.
. Existing inline `add_fk_helpers` behavior remains supported.
. Step registry and meta registry stay aligned.
. Tests cover policy writing, policy-backed execution, conflict behavior, and legacy behavior.
