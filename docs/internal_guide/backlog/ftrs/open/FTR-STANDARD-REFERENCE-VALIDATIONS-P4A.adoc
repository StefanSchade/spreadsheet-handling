===== FTR-STANDARD-REFERENCE-VALIDATIONS-P4A -- Add declarative key and reference validation steps

*Status:* Open
*Priority:* P4A
*Area:* Domain validations

==== Purpose

Provide standard declarative validations for the table invariants that recur in
consumer workbooks: primary-key uniqueness, foreign-key referential integrity,
duplicate reference tuples, and configurable uniqueness constraints.

Consumer repositories should be able to validate common relational rules
without writing a local model-specific plugin for every table pair.

==== Validation Types

* `unique`: configured columns must be unique within a frame.
* `primary_key`: configured key columns must be present, non-empty, and unique.
* `foreign_key`: source key columns must reference existing target key columns.
* `unique_reference`: configured reference tuple columns must not contain
  duplicates unless explicitly allowed.

==== Example Shape

[source,yaml]
----
- step: validate_references
  mode: fail
  rules:
    - type: primary_key
      frame: variables
      columns: [ID]
    - type: foreign_key
      frame: variable_codes
      columns: [ID]
      target: variables
      target_columns: [ID]
    - type: unique_reference
      frame: variable_codes
      columns: [ID, transaction_type_short]
----

==== Goals

* Return or attach a findings frame in `warn` mode.
* Raise a clear error in `fail` mode.
* Support single-column and composite keys.
* Report table, columns, row numbers, offending values, and target table where
  useful.
* Keep validation rules data-model neutral.

==== Non-goals

* Do not implement full schema validation in this slice.
* Do not infer every foreign key automatically.
* Do not replace domain-specific validations that depend on business rules.

==== Acceptance

* The pipeline registry exposes standard reference validation steps.
* Tests cover primary-key uniqueness, arbitrary uniqueness, dangling FK
  detection, composite FK detection, duplicate tuple detection, warn mode, and
  fail mode.
* The findings shape is documented and stable enough for user-facing reports.
* Existing domain-specific validation plugins can be reduced to business rules
  rather than generic table integrity checks.
