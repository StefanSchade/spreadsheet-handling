===== FTR-VALIDATION-FINDINGS-P3B — Structured validation findings and configurable severity policy

*Status:* Open
*Epic:* Phase 3b — Pipeline & Orchestration Consolidation

*Purpose:*::
Replace ad-hoc validation behavior with structured findings and configurable severity handling so that validation can remain expressive without forcing future control-flow choices today.

*Context:*::
Validation behavior currently tends to be embedded implicitly in orchestration code. For the target architecture, validations should produce explicit findings while configuration determines whether a finding is ignored, logged as warning, or treated as fatal.

This keeps the current system simple while preserving the option to support richer orchestration later without redesigning every validation feature.

*Scope:*::
- Introduce a structured validation result/finding concept.
- Support configurable severity handling such as `ignore`, `warn`, and `fail`.
- Apply this model to representative validation cases, including FK/helper-related findings.

*Non-Goals:*::
- No workflow engine.
- No conditional branching/DAG execution in pipelines.
- No requirement that all existing validations be migrated at once.

*Design Principle:*::
Validations report findings; orchestration applies configured policy. This preserves a simple pipeline model while avoiding a hard dependency on exceptions as the only control mechanism.

*Acceptance:*::
- Validation output can carry machine-readable findings.
- Severity policy can be specified via configuration/meta without hardcoding every behavior path.
- Existing fatal validation behavior remains representable.
- The design does not block future guarded/conditional orchestration, but does not require it now.
