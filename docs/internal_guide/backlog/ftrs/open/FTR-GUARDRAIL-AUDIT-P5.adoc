===== FTR-GUARDRAIL-AUDIT-P5 -- Review architecture guards for continued value

*Status:* Open
*Epic:* Phase 5 -- Stabilization and Internal Quality
*Priority:* Low

*Origin:*::
External architecture critique (2026-04-29) raised the concern that
architecture guards may freeze design decisions that should still be revisable.
The broad claim ("reduce by 50%") is not substantiated, but the underlying
question is legitimate: after the intensive Phase-4 feature wave, which guards
still protect active contracts and which have become obsolete, too broad, or
too expensive during intentional refactoring?

This is a targeted audit, not a reduction campaign.

'''

====== Purpose

Systematically review all architecture guards after Phase 4 is complete. For
each guard, assess whether it protects an active invariant, caught or would
catch real drift, duplicates another mechanism, or primarily creates friction
during intentional refactoring.

Document the findings and decide per guard:

* *keep* -- still blocks an active architecture, domain, or integration
  contract
* *narrow* -- useful invariant, but current assertion is too broad
* *move to advisory* -- useful signal, but should not fail the default suite
* *retire* -- invariant is obsolete or covered by a stronger mechanism

'''

====== Scope

This FTR includes:

* inventory of all architecture guards under `tests/architecture/` and related
  semantic-invariant test modules; `architecture_guard_inventory.adoc` is the
  starting point
* for each guard: documented invariant, owner/FTR if known, enforcement mode,
  and current value
* review of recent true positives, false positives, and changes required during
  intentional feature work
* classification into keep, narrow, move to advisory, or retire
* execution of agreed retire/narrow/advisory decisions
* updated guard inventory document

'''

====== Non-goals

This FTR does not:

* introduce new guards beyond tiny coverage checks needed to keep the inventory
  honest
* change the guard framework or test infrastructure
* set a numeric reduction target; the outcome is driven by evidence, not by a
  quota
* question the value of architecture guards in general; the mechanism is sound,
  the question is whether each specific guard still earns its place
* weaken high-value guards solely to make feature work faster

'''

====== Acceptance criteria

. Every guard in the inventory has a documented assessment with rationale.
. The inventory records at least: invariant, current test location, enforcement
  mode, decision, rationale, and last-reviewed date.
. Retired guards are removed from the test suite only after confirming that the
  protected invariant is obsolete or covered elsewhere.
. Narrowed guards keep failing for the intended invariant while dropping
  obsolete or overbroad assertions.
. Advisory guards are moved to a clearly non-blocking category or documented
  command; they must not silently disappear from developer visibility.
. `architecture_guard_inventory.adoc` is updated to reflect the post-audit
  state.

'''

====== Implementation notes

Not yet implemented.

Suggested approach:

. Run the architecture test suite against the post-Phase-4 codebase.
. Compare the actual guard files with `architecture_guard_inventory.adoc`.
. Review `git log` for guards that required repeated adjustment during feature
   FTRs.
. Update the inventory first, then make retire/narrow/advisory edits in small
   commits so the audit trail stays readable.

New guards from `FTR-PIPELINE-SEMANTICS-GUARDS-P4` may not yet have much
history. For those, judge whether the invariant is still an active design
contract and whether the assertion is scoped tightly enough.

'''

====== Relation to existing work

* `architecture_guard_inventory.adoc` -- the existing inventory document.
* `FTR-PIPELINE-SEMANTICS-GUARDS-P4` -- adds new semantic guards. This audit
  reviews all guards including the new ones after they have landed.
* Phase 3i FTRs -- originally introduced most architecture guards.
