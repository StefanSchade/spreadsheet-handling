===== FTR-ODS-CONSUMER-FEATURE-PARITY-P4A -- Track ODS parity for BA-driven workbook features

*Status:* Open
*Priority:* P4A
*Area:* Spreadsheet adapters / parity

==== Purpose

Keep ODS/Calc support visible while BA-driven workbook features are added.

The ODS adapter is not just a secondary output format. It is also the practical
test that the adapter architecture carries the spreadsheet-neutral contract
rather than hard-coding XLSX assumptions into new features.

==== Scope

For the pre-beta BA-derived feature set, maintain a small parity matrix and
focused tests for:

* declarative workbook views
* explicit helper-column styling
* editable/read-only column protection
* lookup helper formula mode
* validation dropdowns from legend metadata, if not already covered by existing
  ODS parity tests
* column-width behavior, including accepted XLSX-only gaps

==== Non-goals

* Do not force fake ODS parity where the backend model is materially different.
* Do not block a feature forever if the correct outcome is an explicit
  documented adapter gap.
* Do not duplicate every XLSX integration test one-to-one.

==== Acceptance

* Every BA-derived pre-beta workbook feature has an explicit ODS disposition:
  implemented, not applicable, or accepted gap.
* At least one ODS integration or roundtrip test exercises the maintained
  declarative workbook path.
* Any accepted ODS gap is documented in the relevant FTR and in a compact
  adapter capability note.
* No new generic workbook feature lands with only implicit XLSX behavior.
