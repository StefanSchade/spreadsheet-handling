===== FTR-SIMPLE-JOIN-VIEWS-P4A -- Add bounded declarative join views when extraction is not enough

*Status:* Open
*Priority:* P4A
*Area:* Domain transformations / view preparation
*Deferrable:* Yes, unless the maintained consumer path needs tuple-driven view
filtering before beta.

==== Purpose

Provide a bounded generic join step for workbook view preparation when a simple
single-frame extraction is not enough.

The core already has `add_lookup_helpers` for the common helper-enrichment
case. This FTR is narrower than a query engine: it should only cover explicit,
deterministic joins that consumers would otherwise implement as small local
plugins.

`FTR-DECLARATIVE-TABULAR-VIEW-OPS-P4A` owns bounded display pivots and
higher-level view preparation. This FTR owns explicit joins, especially
semi-join / existence-filter behavior over expanded tuple relations.

==== Ownership Boundary

This FTR owns join semantics and produces materialized, named output frames.
Workbook-view steps may reference these frames as visible sheets but must not
redefine, inline, or duplicate join semantics. If a workbook view needs a
joined frame, the join must be declared as a preceding pipeline step that
produces an explicit output frame.

==== Consumer Trigger

Some business workbooks derive visible row sets from a matrix or crosstable.
The matrix can be expanded into tuples such as:

* entity key
* axis key
* marker/code

To build a product-specific or axis-specific view, a consumer may need to join
the canonical entity table against the expanded tuple relation and keep only
rows whose marker matches the configured view condition.

That is a generic view-preparation need: the core should support the explicit
join/filter mechanics without encoding the business meaning of the marker.

==== Goals

* Support explicit left and inner joins.
* Support semi-join / existence-filter behavior for "keep rows with matching
  tuple" view preparation.
* Support single-column and composite join keys.
* Support selecting and renaming columns from both sides.
* Support clear collision handling through configured suffixes or explicit
  output names.
* Support simple post-join marker filtering by reusing the bounded extraction
  predicate vocabulary where possible.
* Mark output frames as derived view frames by default.
* Keep `add_lookup_helpers` as the preferred helper-column API when that
  semantic fit is good enough.

==== Non-goals

* Do not implement arbitrary SQL joins.
* Do not support implicit join inference in this slice.
* Do not add aggregations or pivots here.
* Do not attach domain semantics to marker/code values.
* Do not replace `add_lookup_helpers`.

==== Acceptance

* The pipeline registry exposes a simple join view step if extraction plus
  `add_lookup_helpers` cannot express the maintained prototype-consumer view
  path.
* Tests cover left join, inner join, semi-join, composite keys, missing keys,
  duplicate target keys, marker filtering, and column collisions.
* At least one test covers a crosstable/matrix-expanded tuple relation used to
  filter an entity view.
* The FTR can be deferred without blocking beta only if the current consumer
  path no longer needs tuple-driven view filtering.
