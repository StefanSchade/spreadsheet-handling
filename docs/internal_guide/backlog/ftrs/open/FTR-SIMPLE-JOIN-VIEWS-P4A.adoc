===== FTR-SIMPLE-JOIN-VIEWS-P4A -- Add bounded declarative join views when extraction is not enough

*Status:* Open
*Priority:* P4A
*Area:* Domain transformations / view preparation

==== Purpose

Provide a bounded generic join step for workbook view preparation when a simple
single-frame extraction is not enough.

The core already has `add_lookup_helpers` for the common helper-enrichment
case. This FTR is narrower than a query engine: it should only cover explicit,
deterministic joins that consumers would otherwise implement as small local
plugins.

==== Goals

* Support explicit left and inner joins.
* Support single-column and composite join keys.
* Support selecting and renaming columns from both sides.
* Support clear collision handling through configured suffixes or explicit
  output names.
* Mark output frames as derived view frames by default.
* Keep `add_lookup_helpers` as the preferred helper-column API when that
  semantic fit is good enough.

==== Non-goals

* Do not implement arbitrary SQL joins.
* Do not support implicit join inference in this slice.
* Do not add aggregations or pivots here.
* Do not replace `add_lookup_helpers`.

==== Acceptance

* The need is rechecked against the prototype consumer after extraction and
  tabular view ops land.
* If still needed, the pipeline registry exposes a simple join view step.
* Tests cover left join, inner join, composite keys, missing keys, duplicate
  target keys, and column collisions.
* The FTR can be deferred without blocking beta if `add_lookup_helpers` and
  extraction steps cover the current consumer path.
