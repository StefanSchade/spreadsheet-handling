===== FTR-LEGEND-VALIDATION-LISTS-P4A — Excel dropdown lists derived from legend block tokens

*Status:* Open
*Epic:* Phase 4 — Feature Expansion and Domain Transformations
*Priority:* High (pre-beta, driven by BA config crosstable editing UX)

*Depends-on:*

* <<FTR-LEGEND-BLOCKS#,FTR-LEGEND-BLOCKS>> (legend metadata must be present)

*Related-to:*

* <<FTR-XREF-CROSSTABLE#,FTR-XREF-CROSSTABLE>>
* <<FTR-CELL-CODEC#,FTR-CELL-CODEC>>
* <<FTR-COMPACT-MULTIAXIS#,FTR-COMPACT-MULTIAXIS>>
* <<FTR-SPARSE-CROSSTABLE-COLLAPSE-P4A#,FTR-SPARSE-CROSSTABLE-COLLAPSE-P4A>>

'''

====== Purpose

Automatically derive Excel data-validation dropdown lists from legend block
tokens, so that editors of crosstables see a constrained value picker in each
cell rather than typing free-form codes.

This is a small bridge between two existing subsystems (legend blocks and the
validation pass) that significantly improves the editing UX for compact
crosstable sheets.

'''

====== Context

The project already has:

* *Legend blocks* that define the valid token vocabulary per domain
  (e.g. `E = Editable, R = Read-only, ...`)
* A *ValidationPass* that materializes `in_list` constraints as Excel
  `DataValidation` dropdowns
* *Crosstable rendering* (`xref_crosstable`) that places compact token values
  into matrix cells

What is missing: the ability to say "the allowed values in this column/area
come from legend X" so that the ValidationPass can look up the tokens and
produce the dropdown automatically.

'''

====== Problem statement

Without legend-derived dropdowns:

* editors must remember or look up valid codes manually
* typos in compact codes create silent data errors
* the legend block is a visual reference only, not an active constraint
* manual `constraints` entries with duplicated value lists drift from the
  legend over time

'''

====== Goals

. Allow constraints to reference a legend block by name instead of listing
  values explicitly.
. The ValidationPass resolves the reference at render time, extracting tokens
  from the named legend.
. Rendered Excel sheets show dropdown lists in the target cells with exactly
  the legend's token set.
. Works naturally with crosstable sheets where legend-coded values dominate.
. Roundtrip-safe: dropdowns are re-applied on re-render; they do not interfere
  with parse.

'''

====== Design sketch

====== YAML surface (indicative)

[source,yaml]
----
constraints:
  - sheet: product_matrix
    column: "*"          # or specific columns, or area: "data_body"
    rule:
      type: from_legend
      legend: status_codes
      # optional: include_empty: true  (add blank as valid choice)
----

Alternative: implicit mode where any column rendered by `compact_multiaxis`
that has `allowed_from_legend` automatically gets a dropdown. This could be
a boolean flag on the compact-transform config:

[source,yaml]
----
- step: compact_multiaxis
  # ...
  apply_legend_validation: true   # auto-generate dropdowns from legend
----

Both approaches can coexist; explicit `from_legend` constraints are the
minimal viable path.

====== ValidationPass changes

The pass already iterates `_meta.constraints`. For `rule.type == "from_legend"`:

1. Look up `legend_blocks[rule.legend]` from workbook meta.
2. Extract the `token` values from entries.
3. Optionally prepend empty string if `include_empty` is set.
4. Construct `ListLiteralFormulaSpec(values=tokens)`.
5. Proceed with existing `DataValidationSpec` materialization.

This is a ~20-line extension to the existing pass.

====== Cap / large-legend guard

If a legend has many tokens (>50), Excel dropdown UX degrades. A configurable
warning or cap (`max_dropdown_values: 30`) avoids unusable menus. For the
initial implementation, a warning log at >50 values is sufficient; hard caps
can follow in `FTR-VALIDATION-COMPACT`.

'''

====== Scope

This FTR includes:

* new constraint type `from_legend` in the ValidationPass
* token extraction from named legend block
* dropdown materialization via existing `DataValidationSpec` path
* `include_empty` option
* warning when legend exceeds 50 tokens
* unit test: `from_legend` constraint resolves to correct token list
* integration test: rendered Excel file contains dropdown with legend values

'''

====== Non-goals

* Named-range-based validation (pointing dropdowns at the rendered legend
  table range) — possible optimization for large legends, deferred
* Automatic constraint generation without explicit config (convenience, later)
* Hard validation caps (`FTR-VALIDATION-COMPACT`)
* ODS data-validation (ODS format support for dropdowns is limited; xlsx-first)

'''

====== Acceptance criteria

. `from_legend` constraint resolves tokens from the named legend block.
. Rendered Excel cells show a dropdown with exactly the legend tokens.
. Include-empty option adds a blank choice when configured.
. Warning logged for legends with >50 tokens.
. Existing `in_list` constraints continue to work unchanged.
. Roundtrip: dropdown presence does not corrupt data on re-import.
. All existing tests continue to pass.
