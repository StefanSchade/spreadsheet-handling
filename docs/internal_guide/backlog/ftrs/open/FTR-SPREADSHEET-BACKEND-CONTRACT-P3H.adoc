===== FTR-SPREADSHEET-BACKEND-CONTRACT-P3H — Define a spreadsheet-backend contract above format-specific adapters

*Depends-on:* <<FTR-IR-ARCHITECTURE-CLARITY-P3H#,FTR-IR-ARCHITECTURE-CLARITY-P3H>>

*Context:* The current IR write path is already close to spreadsheet-agnostic: frames are composed into `WorkbookIR`, passes enrich the IR, and `build_render_plan()` emits backend-agnostic render operations. However, the adapter boundary is still named and shaped around XLSX. Current interfaces and helper modules implicitly treat `xlsx` as the only full spreadsheet backend, even though the next planned extension is a second spreadsheet target for LibreOffice Calc / ODS.

*Problem statement:* Without a clear spreadsheet-backend contract, a new Calc adapter would likely duplicate the current XLSX structure with different file extensions and library calls. That would harden today’s accidental `XLSX = spreadsheet` assumption instead of turning IR into the intended spreadsheet-agnostic boundary.

*Goals:*

. Define an explicit contract for spreadsheet-capable backends.
. Separate generic spreadsheet concepts from format-specific implementation details.
. Make XLSX one implementation of that contract rather than the implicit default model.
. Prepare the codebase for a second spreadsheet backend without committing to ODS parsing/rendering details yet.
. Improve the readability of adapter boundaries so contributors can tell which responsibilities are generic and which are format-specific.
. Introduce a deliberately narrow API surface from generic rendering into concrete spreadsheet adapters.

*Acceptance criteria:*

. Introduce a clearly named spreadsheet-backend contract that covers the render and parse responsibilities needed by spreadsheet file formats.
. Remove or rename XLSX-specific protocol/interface names in generic modules where they describe a broader spreadsheet concern.
. Document the intended boundaries between `WorkbookIR`, `RenderPlan`, spreadsheet renderers, and spreadsheet parsers.
. Record the allowed dependency direction between generic rendering modules and format-specific adapters.
. Provide a small rendering/parsing facade or equivalent narrow API that spreadsheet adapters are expected to use instead of importing arbitrary internal rendering modules.
. Add focused tests that guard the intended dependency direction and keep adapters on the narrow API surface.
. Existing XLSX functionality continues to work unchanged after the contract extraction.

*Non-goals:*

- No ODS/Calc implementation in this FTR.
- No behavior change to current XLSX output.
- No rewrite of unrelated text backends such as JSON, CSV, YAML, or XML.

*Why now:* A second spreadsheet adapter is the right forcing function to harden these boundaries. Doing this before implementation avoids baking more XLSX-specific assumptions into the architecture.
