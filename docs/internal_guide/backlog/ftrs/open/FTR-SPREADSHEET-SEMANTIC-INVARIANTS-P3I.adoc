===== FTR-SPREADSHEET-SEMANTIC-INVARIANTS-P3I -- Separate smoke checks from durable spreadsheet-semantics invariants

*Depends-on:* <<FTR-SPREADSHEET-META-SEMANTICS-P3H#,FTR-SPREADSHEET-META-SEMANTICS-P3H>>, <<FTR-META-REGISTRY-HARDENING-P3I#,FTR-META-REGISTRY-HARDENING-P3I>>, <<FTR-SPREADSHEET-PARSE-CONTRACT-P3I#,FTR-SPREADSHEET-PARSE-CONTRACT-P3I>>

*Context:* The Phase-3h semantics work made several important spreadsheet invariants explicit, but the corresponding tests still mix different levels of intent: some are presence/smoke checks, some describe the current carrier representation, and some are closer to durable architectural policy. That is acceptable for discovery work, but it is too blurry for long-term guardrails.

*Problem statement:* Without a clearer taxonomy and stronger semantic tests, later adapter work may accidentally optimize for the current XLSX carrier shape instead of the shared spreadsheet contract. Reviewers would also have a harder time distinguishing "the path exists today" from "this semantic invariant must remain true across adapters."

*Goals:*

. Distinguish more clearly between smoke tests, current-state documentation tests, and durable spreadsheet-semantics invariants.
. Harden the tests around canonical versus carrier-derived semantics such as hidden workbook meta payload, `meta_rendering` derivatives, freeze/filter intent, and roundtrip-relevant metadata.
. Make explicit where current implementation details are tolerated transitional state rather than normative contract.
. Ensure the shared spreadsheet semantics are expressed in a form that a second adapter can target without inheriting XLSX-specific accidental behavior.

*Acceptance criteria:*

. The test suite and/or nearby docs clearly differentiate smoke/presence checks from stronger semantic-invariant checks.
. At least the central canonical-vs-carrier cases have semantically phrased tests rather than only representation-oriented ones.
. The tests make current-state behavior versus intended cross-adapter contract more explicit where the two still differ.
. The resulting guardrails are useful for later parity work and do not silently encode XLSX-only behavior as universal semantics.

*Non-goals:*

- No exhaustive parity matrix across all future adapters in this FTR.
- No attempt to settle every currently transitional detail as permanent policy.
- No full typed meta model or registry-driven semantic engine.

*Why now:* Before Phase 3j starts proving the architecture with a second adapter, the suite should say more clearly which spreadsheet semantics are real contract and which checks are only smoke or current-state documentation.
