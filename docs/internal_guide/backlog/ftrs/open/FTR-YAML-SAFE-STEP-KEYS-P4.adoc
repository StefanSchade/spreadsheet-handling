===== FTR-YAML-SAFE-STEP-KEYS-P4 -- YAML-safe aliases for pipeline step parameters

*Status:* Open
*Epic:* Phase 4 -- Feature Expansion and Domain Transformations
*Priority:* Medium

*Discovered-in:*

* `ba-config` integration of `FTR-EXPLICIT-HELPER-LOOKUP-POLICY-P4`

*Related-to:*

* <<FTR-EXPLICIT-HELPER-LOOKUP-POLICY-P4#,FTR-EXPLICIT-HELPER-LOOKUP-POLICY-P4>>
* <<FTR-COMPACT-TRANSFORM-API-ERGONOMICS-P4#,FTR-COMPACT-TRANSFORM-API-ERGONOMICS-P4>>

'''

====== Purpose

Reduce YAML syntax friction in consumer-facing pipeline configuration.

The `enrich_lookup` step naturally wants a join-key parameter. The first API used
`on`, which is familiar from joins, but plain `on:` is a YAML 1.1 boolean-like
bareword. In practice, a pipeline must write `"on": ID` to avoid the key being
parsed as a boolean by common YAML loaders.

That is technically correct, but it is a poor consumer experience.

'''

====== Problem statement

Pipeline YAML should be easy to write and hard to subtly misparse.

Requiring users to remember that `on` must be quoted creates avoidable friction:

[source,yaml]
----
- step: enrich_lookup
  source: variable_usage_matrix_raw
  lookup: variables
  output: variable_usage_matrix
  "on": ID
----

Without quotes, some YAML parsers interpret `on` as boolean `true`, which can
lead to confusing binding errors such as non-string keyword arguments.

'''

====== Goals

. Add YAML-safe aliases for `enrich_lookup` join key configuration.
. Prefer the safe aliases in documentation and examples.
. Keep the existing quoted `"on"` form backward-compatible.
. Emit clear diagnostics if both old and new key forms are supplied.
. Scan current public/internal pipeline examples for other YAML 1.1
   boolean-like parameter names.

'''

====== Candidate API

Preferred single-key form:

[source,yaml]
----
- step: enrich_lookup
  source: variable_usage_matrix_raw
  lookup: variables
  output: variable_usage_matrix
  key: ID
----

Preferred multi-key form:

[source,yaml]
----
- step: enrich_lookup
  source: source_frame
  lookup: lookup_frame
  output: enriched_frame
  keys: [region, product_id]
----

Compatibility form:

[source,yaml]
----
- step: enrich_lookup
  source: variable_usage_matrix_raw
  lookup: variables
  output: variable_usage_matrix
  "on": ID
----

Alternative naming such as `join_key` / `join_keys` is acceptable if the
implementation review prefers the more explicit wording.

'''

====== Acceptance criteria

. `enrich_lookup` accepts a YAML-safe alias for a single join key.
. `enrich_lookup` accepts a YAML-safe alias for multiple join keys.
. Existing `"on"` configurations continue to work.
. Supplying conflicting key forms raises an actionable configuration error.
. FTR and usage examples use YAML-safe aliases by default.
. A focused test covers unquoted YAML-safe configuration through the pipeline
   registry or YAML loader path.
. A small documentation note explains why `on` is not the preferred YAML spelling.

'''

====== Non-goals

This FTR does not:

* remove support for `"on"`
* redesign the pipeline config language
* change `enrich_lookup` join semantics
* implement a custom YAML loader
* solve unrelated YAML syntax pitfalls outside current consumer-facing step APIs

'''

====== Review guidance

When reviewing this FTR, ask:

* Does the preferred spelling avoid YAML 1.1 boolean-like barewords?
* Is the compatibility story clear?
* Are examples copy-pasteable without quoting surprises?
* Does the implementation avoid adding multiple equally recommended spellings?
