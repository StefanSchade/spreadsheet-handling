'''

====== Purpose

Add architecture guards that keep the pipeline step semantics concept enforceable over time.

The guards should prevent the public YAML step surface, runtime registry, step registry, metadata contracts, composite wrappers, plugin boundaries, and render/view decisions from drifting back into implicit or opaque behavior.

This FTR turns the pipeline semantics concept from documentation into a set of mechanically checked invariants.

'''

====== Context

`FTR-PIPELINE-SEMANTICS-CONCEPT-P4` established the target semantics:

* configuration steps may infer or declare policy
* primitive steps execute deterministic operations against direct config and/or resolved metadata
* composite steps must expose wrapped primitives and internal-frame policy
* plugin steps remain supported extension points
* infrastructure steps are classified explicitly
* render/view decisions must use explicit lifecycle/view metadata rather than frame-name or shape heuristics

`FTR-PIPELINE-STEP-REGISTRY-P4` introduces a JSON registry as the reviewed inventory of public pipeline steps.

That registry is the natural anchor for guards. The first guards should validate the registry itself. Deeper guards should then check that runtime, metadata, and selected source-code behavior do not bypass the registered semantics.

'''

====== Problem statement

Without guards, the pipeline semantics concept can be bypassed gradually:

* a new runtime step may be added without a registry entry
* a step may be exposed under an undocumented alias
* a composite step may hide primitive behavior or internal frames
* a persistent metadata path may be introduced without a meta-registry entry
* a primitive step may start interpreting `auto` values directly
* inline YAML config and `_meta` policy may silently conflict
* rendering may start omitting frames by name or shape heuristics
* pipeline orchestration may drift toward `_meta`-driven routing
* plugins may mutate frame sets without any lifecycle responsibility being visible

These are not hypothetical risks. They are the normal failure modes of a framework whose public surface grows through small feature additions.

'''

====== Goals

. Add guards for pipeline step registry completeness.
. Add guards for registry structure and enum validity.
. Add guards for canonical public step names and release-bounded aliases.
. Add guards for composite step transparency.
. Add guards for meta-registry alignment of persistent metadata paths.
. Add guards for plugin registry semantics without closing the plugin extension point.
. Add guards for helper-configuration boundaries where stable.
. Add guards for lifecycle/render non-guessing where stable.
. Add guards that prevent pipeline orchestration from becoming payload-driven routing.
. Keep all guards narrow, reviewable, and tied to documented invariants.
. Document unavoidable current-state exceptions with sunset or review conditions.

'''

====== Scope

This FTR includes architecture tests and supporting test helpers for the pipeline semantics concept.

The implementation should proceed in guard tiers.

Tier 1 guards should be implemented as soon as the step registry exists:

* registry completeness
* registry structural validity
* category validity
* inverse-kind validity
* composite wrapped-step resolution
* alias lifecycle shape
* plugin entry shape

Tier 2 guards should be implemented once naming and helper-configuration steps land:

* canonical public names in runtime registry
* no undocumented public aliases
* `auto` limited to configuration step schemas unless explicitly transitional
* inline-vs-policy conflict behavior documented for relevant primitive parameters
* helper policy metadata paths aligned with meta registry

Tier 3 guards should be implemented once frame lifecycle and workbook views land:

* lifecycle metadata paths aligned with meta registry
* render omission uses lifecycle/view metadata
* render omission does not infer redundancy from frame names or shapes
* selected lifecycle-aware transformations declare frame contracts

Tier 4 guards may be source-level drift checks where the invariant is stable enough:

* pipeline orchestration does not branch or skip semantic steps based on `_meta`
* primitive helper execution does not implement hidden policy inference
* composite metadata does not leak references to undeclared internal frames

'''

====== Non-goals

This FTR does not:

* introduce new pipeline semantics
* add new transformation behavior
* make the JSON step registry the runtime source of truth
* rename public steps
* implement helper configuration steps
* implement frame lifecycle behavior
* remove plugin support
* create a broad repository-wide import linter
* statically prove every semantic property of dynamic plugin code
* block intentionally documented current-state exceptions

Implementation work for naming belongs to `FTR-PIPELINE-STEP-NAMING-P4`.

Implementation work for helper configuration steps belongs to `FTR-LOOKUP-FK-CONFIGURATION-STEPS-P4`.

Implementation work for frame lifecycle and workbook views belongs to `FTR-FRAME-LIFECYCLE-AND-WORKBOOK-VIEWS-P4`.

'''

====== Guard principles

The guards should follow these rules:

. Guard documented invariants, not preferred implementation aesthetics.
. Prefer registry- and metadata-based guards before brittle source scanning.
. Keep source-level scans conservative and explain every forbidden pattern.
. Distinguish durable invariants from current-state checks.
. Every allowlist entry needs a rationale and sunset or review condition.
. Plugins remain allowed; guards should require visible responsibility, not forbid extension.
. A guard should fail with a message that tells the maintainer which registry, meta, or architecture rule was violated.

'''

====== Guard tier 1: step registry structural guards

These guards become mandatory as soon as `pipeline_step_registry.json` exists.

[cols="1,4,2"]
|===
| Guard | Rule | Type

| registry completeness
| Every public runtime `REGISTRY` key has exactly one registry entry or a documented alias relationship.
| durable

| registry resolution
| Every registry entry resolves to a runtime step, a structured alias, or an explicit plugin/infrastructure exception.
| durable

| unique names
| Registry step names are unique; aliases do not collide with canonical names unless intentionally marked as current runtime aliases.
| durable

| category validity
| Every step category is one of `primitive`, `configuration`, `composite`, `plugin`, or `infrastructure`.
| durable

| required fields
| Every public entry has name, category, status, purpose, parameters, frame contract, inverse classification, and reference fields.
| durable

| inverse-kind validity
| Every inverse kind is one of the documented enum values.
| durable

| parameter source-kind validity
| Every parameter source kind is one of the documented enum values.
| durable

| plugin entry shape
| The `plugin` entry declares that plugins may change frame sets and how lifecycle responsibility is handled.
| durable
|===

These guards should be fast and purely local.

'''

====== Guard tier 2: naming and alias lifecycle guards

These guards become mandatory after `FTR-PIPELINE-STEP-NAMING-P4` lands.

[cols="1,4,2"]
|===
| Guard | Rule | Type

| canonical runtime names
| Runtime `REGISTRY` uses the normalized public step names.
| durable

| alias structure
| Aliases are structured objects, not plain strings.
| durable

| alias lifecycle
| Deprecated aliases carry `deprecated_since` and `remove_after` or equivalent release-bound fields.
| durable

| no undocumented aliases
| Runtime aliases may exist only if represented in the step registry.
| durable

| maintained example scan
| Maintained examples and test fixture YAML do not use replaced names except in explicit migration notes or alias tests.
| durable
|===

If the project chooses hard renames without runtime aliases, the alias lifecycle guard should still ensure that old names are not silently reintroduced.

'''

====== Guard tier 3: composite transparency guards

Composite steps are allowed, but they must not hide semantics.

[cols="1,4,2"]
|===
| Guard | Rule | Type

| wrapped steps declared
| Every composite entry has non-empty `wrapped_steps`.
| durable

| wrapped steps resolve
| Every wrapped step name resolves to a registry entry.
| durable

| public outputs declared
| Every composite entry declares public output frames or an explicit reason why outputs are dynamic/plugin-owned.
| durable

| internal frames declared
| Composite entries that use temporary frames declare `internal_frames` or explicitly state that no internal frames are used.
| durable

| primitive metadata policy declared
| Composite entries declare whether wrapped primitive metadata is preserved, nested, suppressed, or still current-state unknown.
| durable with current-state escape
|===

A deeper guard may later verify that composite metadata does not reference frames absent from final frames plus declared internal frames. That should wait until lifecycle/composite metadata is stable enough.

'''

====== Guard tier 4: meta-registry alignment guards

The step registry must not become a second metadata authority.

[cols="1,4,2"]
|===
| Guard | Rule | Type

| meta contract shape
| Every `meta_contract` entry declares path, persistence class, and layer/classification or transient reason.
| durable

| persistent roots known
| Every persistent metadata root referenced by the step registry exists in `meta_registry.yaml`.
| durable

| transient entries explicit
| Transient or trace-only entries carry a reason and do not pretend to be canonical metadata.
| durable

| helper policy roots
| Helper policy roots introduced by configuration steps are represented in the meta registry.
| enabled after helper config FTR

| lifecycle roots
| `frame_lifecycle` and `workbook_view` are represented in the meta registry.
| enabled after lifecycle FTR
|===

The first implementation may allow documented current-state gaps, but each gap needs a follow-up note or review condition.

'''

====== Guard tier 5: configuration-versus-primitive guards

These guards enforce the rule that heuristics belong to configuration steps and primitive steps execute resolved policy.

They should be introduced only where the registry and implementation provide stable signals.

[cols="1,4,2"]
|===
| Guard | Rule | Type

| `auto` restriction
| Primitive step parameter schemas do not allow `auto`, except explicitly marked transitional parameters.
| durable after config FTR

| transitional auto expiry
| Transitional primitive `auto` support carries a review or removal condition.
| durable

| conflict behavior required
| Parameters marked `inline_or_policy` declare conflict behavior.
| durable

| helper conflict tests
| Lookup/FK helper primitives fail when inline config conflicts with resolved `_meta` policy.
| behavior guard

| no hidden helper inference
| Primitive helper execution does not contain broad target/helper inference that belongs to configuration steps, except documented transitional code.
| source-level conservative
|===

The source-level guard should be conservative. It should not block ordinary parameter resolution or policy lookup.

'''

====== Guard tier 6: frame lifecycle and render non-guessing guards

These guards become meaningful after lifecycle/view behavior exists.

[cols="1,4,2"]
|===
| Guard | Rule | Type

| lifecycle registry entries
| Lifecycle-aware steps declare frame contract and lifecycle metadata effects in the step registry.
| enabled after lifecycle FTR

| render uses lifecycle
| Render omission uses explicit lifecycle/view metadata.
| durable after lifecycle FTR

| no name-based omission
| Render omission logic does not use suffixes such as `_raw`, `_view`, `_debug`, or similar as lifecycle decisions.
| durable

| no shape-based omission
| Render omission logic does not infer redundancy from frame shape or similar columns.
| durable

| pipeline non-routing
| Pipeline orchestration does not skip or branch semantic steps based on `_meta.frame_lifecycle` or workbook view metadata.
| durable
|===

Render guards should focus on generic render/view selection code, not on backend-specific mechanics.

'''

====== Guard tier 7: plugin responsibility guards

Plugins are supported extension points and should not be blocked by purity rules.

The guard goal is visible responsibility, not plugin prohibition.

[cols="1,4,2"]
|===
| Guard | Rule | Type

| plugin entry present
| The step registry contains a `plugin` entry with extension and lifecycle responsibility semantics.
| durable

| plugin may change frames
| The registry explicitly allows plugins to change the frame set.
| durable

| plugin lifecycle mode
| Plugin contract documents `declared_lifecycle` or `caller_owned_lifecycle` as accepted modes.
| durable after lifecycle FTR

| plugin diagnostics
| If implemented, diagnostics warn when plugin frame-set changes lack lifecycle metadata or caller-owned declaration.
| behavior guard
|===

Do not add a guard that forbids plugin frame creation or plugin workbook-view materialization.

'''

'''

====== Suggested implementation direction

. Add registry loading helper for tests.
. Add enum and required-field validation tests.
. Add runtime registry completeness tests.
. Add composite transparency tests.
. Add alias lifecycle tests once naming aliases exist or are intentionally absent.
. Add meta-registry alignment tests for stable persistent roots.
. Add helper-configuration boundary tests after configuration steps land.
. Add lifecycle/render non-guessing tests after lifecycle rendering lands.
. Keep allowlists small and documented with review conditions.

Prefer simple Python tests over a heavy external architecture-testing framework.

Use AST/source scanning only for narrow, stable patterns where registry checks are insufficient.

'''

====== Current-state exceptions

Some guards may need transitional exceptions while the pipeline semantics wave is in progress.

Acceptable exception format:

[source,text]
----
Exception:
  path or step:
  violated target rule:
  why it is currently tolerated:
  linked follow-up FTR:
  removal or review condition:
----

Unstructured allowlists are not acceptable.

Every exception must either be:

* removed during the FTR that introduces the relevant behavior
* linked to a named follow-up FTR
* explicitly classified as durable because the exception is part of the intended architecture

'''

====== Acceptance criteria

. Registry completeness guard exists.
. Registry structural validity guard exists.
. Category, inverse-kind, and parameter source-kind enums are validated.
. Composite transparency guard exists.
. Plugin contract guard exists and preserves plugin extensibility.
. Alias lifecycle guard exists or is explicitly not applicable because hard renames removed aliases.
. Meta-registry alignment guard exists for stable persistent metadata roots.
. Current-state exceptions are structured and linked to follow-up review conditions.
. No guard forbids legitimate plugin extension behavior.
. No guard makes the JSON registry the runtime source of truth.
. Guards are documented with clear failure messages.
. Existing active tests continue to pass.

If helper configuration or frame lifecycle has not landed when this FTR is first implemented, the corresponding deeper guards may be documented as deferred sections with explicit activation conditions.

'''

====== Review guidance

When reviewing this FTR, check:

* Does each guard enforce a documented invariant?
* Is the guard too brittle for the current implementation stage?
* Are current-state exceptions explicit and reviewable?
* Does the guard preserve plugin extensibility?
* Does the guard distinguish runtime authority from descriptive registry authority?
* Are source-level scans narrow and justified?
* Would the guard catch a realistic drift scenario?
* Does the guard failure message explain the architectural violation clearly?

'''

====== Rationale

The pipeline semantics concept introduces useful distinctions, but documentation alone will not keep them intact.

A step registry without guards can drift. Naming decisions without alias lifecycle checks can become compatibility debt. Metadata contracts without meta-registry alignment can create parallel truth sources. Lifecycle rules without render guards can be bypassed by name-based omissions.

The guard strategy should therefore start with the most stable source of truth: the reviewed registry. Deeper source-level and behavior guards should follow only where the underlying concepts have become stable enough.

This keeps the architecture enforceable without turning the test suite into a brittle policy maze.

'''
