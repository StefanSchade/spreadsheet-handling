===== FTR-COMPACT-XREF-USE-CASE-SLICE-P4A2 -- Neutral complex XRef usage slice

*Status:* New

*Purpose:*::
Add a neutral, end-to-end usage slice that exercises the complex XRef shape
needed for real configuration work without committing private/business-specific
vocabulary to the repository.

*Context:*::
After Phase 4a finishes, the project should first try the flat-axis path:

* legend/codebook blocks
* XRef matrix <-> relation
* whole-cell code validation
* compact multi-axis composition

If that proves the core model, Phase 4a2 can add grouped column axes before the
project moves into Phase 4b nested JSON work.

*Neutral Domain Model:*::

Use generic product/configuration vocabulary:

* `features` or `config_items`
* `product_families`
* `product_variants`
* `feature_product_codes`
* `legend_blocks`

Avoid private or customer-specific terminology in fixtures, docs, and tests.

*Target Spreadsheet Shape:*::

[source,text]
----
Feature       Fixed                  Variable
              FZ-AD      FZ-TD       VAR-AD     VAR-TD
currency      E          E           E          E
amount        E-R-K      E-R-K       E-R-K      E-R-K
obsolete      x          x           x          x
----

*Target Explicit Shape:*::

[source,text]
----
feature_id | product_family | variant_id | code | group?
----

*Scope:*::

* Provide fixture data for products/features/variants and codebook entries.
* Exercise export to a human-maintainable grouped crosstab.
* Exercise import back into explicit relation form.
* Verify that code semantics still come from legend/codec configuration.
* Document which parts are generic library behavior and which are only example
  fixture data.

*Non-Goals:*::

* No private domain vocabulary.
* No customer-specific validation rules.
* No nested list/object JSON transformation.
* No attempt to make grouped-column XRef mandatory for the first Phase 4a
  complex-use-case trial.

*Acceptance:*::

* The slice demonstrates a neutral grouped XRef workbook shape.
* Roundtrip verifies that feature identity and product variant identity survive.
* Codes such as `E-R-K` can remain whole-cell codes.
* The fixture can be reused by later demo-repository integration.
* The documentation clearly states that Phase 4a2 is optional follow-up after
  the flat-axis Phase 4a trial.
