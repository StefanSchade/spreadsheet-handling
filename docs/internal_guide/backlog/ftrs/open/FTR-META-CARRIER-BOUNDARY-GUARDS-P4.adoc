= FTR-META-CARRIER-BOUNDARY-GUARDS-P4

== Status

Proposed

== Context

Spreadsheet metadata and carrier-specific hints are a separate boundary from
test-suite topology.

This FTR is reserved for guards around spreadsheet meta/carrier semantics such
as:

* `meta_canonical`
* `meta_rendering`
* carrier hints
* persisted `_meta`
* backend-specific metadata introduced by formula providers or adapters

Test directories, pytest markers, Make targets, and support test roots are
covered by `FTR-TEST-CARRIER-BOUNDARY-GUARDS-P4`.

Backlog note: this FTR remains an open guard implementation item. Earlier
mentions of "deferred" only meant "not completed by the Test-Carrier Boundary
slice"; they did not mean that this item should be skipped in backlog order.
The current implementation should be narrow and based on existing meta seams
unless new carrier fields or formula-provider metadata require broader design.

== Problem

The current meta/carrier seam is partly protected by semantic-invariant and
current-state tests, but it does not yet have dedicated boundary guards.

As new carrier fields, backend-specific hints, or formula-provider metadata are
introduced, the project needs explicit checks that prevent accidental coupling
between canonical spreadsheet meaning and backend/rendering-specific carrier
details.

== Goal

Define and implement dedicated guards for the current spreadsheet
meta/carrier boundary without inventing a new metadata model.

Potential guard topics:

* canonical metadata must not depend on backend-specific rendering hints
* rendering metadata must not leak into persisted canonical state unless
  explicitly modeled
* backend carrier hints must stay scoped to their owning adapter or provider
* persisted `_meta` must remain stable across supported roundtrips

== Non-Goals

* No test-suite topology guards
* No marker or Make-target policy
* No archive, quarantine, or support-carrier rules

== Approach

Add focused architecture guards that:

* keep carrier-shaped and render-local registry entries out of
  `meta_canonical`
* keep canonical registry entries free of carrier-shaped names
* verify that projection back to `frames["_meta"]` prefers the canonical
  workbook payload over carrier hints

Broaden these guards only when new carrier fields, backend-specific meta hints,
or formula-provider metadata are introduced.

== Deliverables

* Meta/carrier boundary guards under `tests/architecture/semantic_invariants/`
* Backlog wording that distinguishes "not closed by Test-Carrier" from
  "deprioritized"
* Guard inventory entry for the dedicated meta/carrier guard

== Risks

* Closing this FTR with test-topology work would leave the real spreadsheet
  meta/carrier seam untracked.
* Adding dedicated guards too early could freeze metadata shapes before the
  domain model has stabilized.

== Dependencies

* FTR-FORMULA-PROVIDERS
* FTR-RAW-FORMULA-BOUNDARY-GUARDS-P4
* FTR-ADAPTER-CONTRACT-BOUNDARY-GUARDS-P4
* FTR-ARCHITECTURE-GUARD-INVENTORY-P4

== Acceptance Criteria

* Carrier-shaped registry entries remain classified as `meta_rendering`, not
  `meta_canonical`.
* Canonical registry entries do not use carrier-shaped names.
* Projection to persisted `frames["_meta"]` does not promote carrier hints over
  the canonical workbook payload.
* Existing active architecture tests continue to pass.

== Decision

Implement as a narrow current-seam guard now. This FTR was not closed by the
Test-Carrier Boundary work; that work only moved test-topology concerns into
`FTR-TEST-CARRIER-BOUNDARY-GUARDS-P4`.

Broader meta/carrier guard expansion should wait for new carrier fields,
backend-specific meta hints, or formula-provider metadata.
