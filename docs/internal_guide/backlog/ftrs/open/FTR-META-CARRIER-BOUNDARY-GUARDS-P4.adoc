= FTR-META-CARRIER-BOUNDARY-GUARDS-P4

== Status

Proposed

== Context

Mit zunehmender Modularisierung der Testsuite und der Einführung klarer Topologien (siehe FTR-TEST-SUITE-TOPOLOGY-P3C) entsteht ein strukturelles Risiko:

* Test-Carriers (Verzeichnisse, Marker, Make-Targets) beginnen implizite fachliche oder architektonische Bedeutung zu tragen.
* Grenzen zwischen Testtypen (z.B. Architekturtests, Integrationstests, Featuretests) werden unscharf oder versehentlich verletzt.
* Änderungen an der physischen Struktur können unbeabsichtigt semantische Auswirkungen haben.

Bisher existieren keine systematischen Guards, die diese Grenzen explizit absichern.

== Problem

Ohne explizite Boundary Guards entstehen langfristig folgende Probleme:

* Vermischung von Testtypen innerhalb eines Carriers
* schleichende Erosion der Architekturregeln
* schwer nachvollziehbare Testausführung (insbesondere bei selektiven Runs)
* steigende kognitive Last bei Wartung und Erweiterung

Diese Probleme treten typischerweise nicht sofort, sondern graduell auf und sind daher schwer zu erkennen.

== Goal

Einführung klarer, überprüfbarer Boundary Guards für Test-Carriers, um:

* semantische Konsistenz sicherzustellen
* strukturelle Drift zu verhindern
* implizite Regeln explizit und überprüfbar zu machen

Nicht Ziel dieses FTR ist die vollständige Definition aller Testtypen selbst – dies gehört in FTR-ARCHITECTURE-TEST-LAYER-P3C.

== Non-Goals

* Keine Einführung neuer Testtypen
* Keine Änderung bestehender Testlogik
* Keine umfassende Refaktorierung bestehender Tests

== Approach

=== 1. Definition von Carrier-Typen

Explizite Klassifikation von Test-Carriern, z.B.:

* architecture
* integration
* feature
* support / helpers

Diese Klassifikation erfolgt primär über:

* Verzeichnisstruktur
* Marker
* Namenskonventionen

=== 2. Formulierung von Boundary-Regeln

Beispiele:

* Architekturtests dürfen keine produktiven Imports mit Seiteneffekten ausführen.
* Featuretests dürfen keine Architektur-Assertions enthalten.
* Helper-Module dürfen keine Tests enthalten.

Die Regeln müssen klar formuliert und technisch überprüfbar sein.

=== 3. Technische Guards

Einführung automatischer Checks, z.B.:

* pytest collection checks
* Marker-Validierung
* statische Analyse (Import-Regeln)
* optionale CI-Gates

Ziel ist es, Regelverletzungen frühzeitig und deterministisch zu erkennen.

=== 4. Fail-Fast-Strategie

Boundary-Verletzungen führen zu:

* klaren Fehlermeldungen
* Abbruch des Testlaufs (konfigurierbar)

Kein "best effort" – Regelverletzungen sind strukturelle Fehler.

== Deliverables

* Dokumentierte Liste von Carrier-Typen
* Satz expliziter Boundary-Regeln
* Implementierte technische Guards (mindestens minimaler Satz)
* Integration in bestehenden Testlauf (pytest / Make)

== Risks

* Überregulierung → zu starre Struktur
* falsche Klassifikation bestehender Tests
* initialer Mehraufwand bei Einführung

Diese Risiken sind akzeptabel, da sie strukturelle Klarheit langfristig erhöhen.

== Dependencies

* FTR-TEST-SUITE-TOPOLOGY-P3C
* FTR-ARCHITECTURE-TEST-LAYER-P3C
* FTR-TEST-MARKERS-AND-MAKE-TARGETS-P3C
* FTR-GUARDRAIL-DECISION-RECORD-P4

== Acceptance Criteria

* Für jeden Test-Carrier ist ein Typ definiert
* Mindestens 3 zentrale Boundary-Regeln sind technisch enforced
* Verletzungen werden im Testlauf sichtbar und reproduzierbar erkannt
* Dokumentation ist ausreichend, um neue Tests korrekt einzuordnen
