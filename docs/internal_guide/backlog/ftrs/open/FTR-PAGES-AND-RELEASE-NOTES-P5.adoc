===== FTR-PAGES-AND-RELEASE-NOTES-P5 -- Versioned Pages and release notes trail

*Status:* Open
*Priority:* P5
*Area:* Documentation / release management

==== Purpose

Define the lightweight publishing path for versioned documentation, demo
walkthroughs, and release notes once the next beta has proven the integration
story.

The immediate beta can still be handled with repository docs and tags. This FTR
captures the step toward a more professional public trail without forcing
generated HTML into source branches.

==== Scope

* Decide the Pages ownership model for the main repository and demo repository.
* Define the split between the core library site and the demo walkthrough site.
* Define how release notes are authored in source form and published.
* Define versioned public paths for core docs, demo walkthroughs, and release
  notes.
* Keep generated HTML and Reveal.js output as build artifacts, not committed
  source.
* Connect this with the conservative release/promotion policy rather than
  duplicating release automation decisions.

==== Site Roles

Main library Pages site:

* release notes and changelog trail
* user guide entry points
* transformation catalog
* API and YAML surface notes when stable enough
* links to matching demo walkthrough versions

Demo Pages site:

* journey index for new users
* generated Reveal.js walkthrough decks
* downloadable or linked demo inputs/outputs where useful
* capability matrix for the demo repository
* links back to the core version the demo was built against

The demo repository should own its own `DEMO-*` backlog item for the concrete
Pages publishing workflow. This core FTR owns the cross-repository policy and
release trail.

==== Preferred Direction

Use GitHub Pages from the owning repository where possible:

* main library site: release notes, user guide entry points, project status
* demo site: generated walkthrough decks and current demo journey index

Prefer stable public paths such as:

* `/latest/`
* `/versions/v0.1.0b6/` for core documentation
* `/versions/v0.1.0b6-demo/` for demo walkthroughs

A separate Pages repository is only needed if repository policy requires it or
if the generated artifact history becomes too noisy to maintain in the owning
repository.

==== Versioning Policy

Every published beta or release should have a stable documentation snapshot.
`latest` can move, but versioned paths should remain stable.

For coupled beta/demo releases, the release notes should say which demo tag was
validated against which core tag. The pages index should make that pairing
visible so a user can follow a walkthrough that matches the installed library
version.

Generated HTML may live in a Pages artifact or generated publishing branch, but
not on the normal source branch. Source branches keep AsciiDoc, slide sources,
release note sources, and build configuration only.

==== Non-goals

* No requirement to publish the full Pages setup before the next beta.
* No generated HTML on normal source branches.
* No large documentation site framework before the AsciiDoc and Reveal.js
  source layout has proven itself.

==== Acceptance

* The repository policy for generated Pages artifacts is documented.
* Release notes have a source-of-truth location and a repeatable generation or
  publishing path.
* Demo walkthrough publishing supports versioned snapshots, not only a single
  overwritten latest state.
* The core and demo sites have distinct ownership and link to each other by
  version.
* Reveal.js walkthroughs are treated as generated tutorial artifacts, not as
  the source of truth for reference documentation.
* The next beta can link to either source docs or published Pages output without
  ambiguity.
