===== FTR-BA-CONFIG-INTEGRATION-P4A -- Run the first real BA config consumer before beta

*Status:* Open
*Priority:* P4A
*Area:* Consumer integration / beta readiness

==== Purpose

Use the `ba-config` repository as the first real consumer integration pass
before the next beta.

The goal is not to turn private or company-specific configuration into a core
library fixture. The goal is to run the current library against the concrete
usage path, discover missing general capabilities early, and keep those gaps
visible before the demo repository and beta release are polished.

==== Scope

* Bind `ba-config` to the local `spreadsheet-handling` checkout for the
  integration pass.
* Identify the maintained BA spreadsheet flow that should represent the beta
  readiness check.
* Run the selected flow end to end where local data and permissions allow it.
* Capture any missing generic library behavior as separate FTRs in this
  backlog.
* Keep consumer-specific changes, fixtures, and generated artifacts in the
  consumer repository.

==== Non-goals

* No company data is copied into the main library repository.
* No BA-specific assumptions are hard-coded into generic transforms.
* No broad demo rewrite is bundled into this FTR; the demo follows after this
  consumer pass.

==== Acceptance

* The selected `ba-config` flow can run against the local library checkout, or
  the blocker is documented with a concrete next FTR.
* Any main-library behavior needed by `ba-config` is either implemented as a
  general capability or recorded as a follow-up FTR.
* Scratch outputs, snapshots, and local run artifacts are ignored or removed
  rather than committed accidentally.
* The next demo-integration pass can start from the capabilities proven or
  newly queued here.
