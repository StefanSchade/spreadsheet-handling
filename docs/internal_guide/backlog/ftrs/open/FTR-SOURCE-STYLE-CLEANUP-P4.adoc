====== Purpose

Reduce source-level readability debt that is not architectural in nature but still makes the code harder to review, maintain, and safely extend.

This FTR is deliberately limited to code style and local maintainability. It does not change architecture, public APIs, documented YAML shapes, CLI behavior, adapter contracts, or domain semantics.

====== Context

A source-level review found no material nested-ternary problem in the Python code. The more relevant shortcomings are conventional maintainability issues:

* German comments, maintainer-facing diagnostics, or documentation snippets in repository source despite the project convention that repository code, comments, and documentation should be English.
* Overlong functions that mix parsing, branching, data shaping, and error handling in one procedural block.
* Dense transformation code where intermediate names would make the intent easier to verify.
* Some compact inline fallback expressions and comprehensions that are acceptable individually but become harder to review when embedded in already large functions.
* Backend rendering helpers that accumulate imperative detail and would benefit from narrower helper functions.

The affected areas should be identified mechanically during implementation rather than frozen in this ticket. Initial candidates from the review include compact transform expansion, spreadsheet rendering helpers, ODS rendering helpers, lookup enrichment, and selected transformation modules.

====== Goals

. Remove German comments and repo-facing messages from source files unless they describe localized business terminology intentionally.
. Split overlong functions where the split reveals stable sub-responsibilities.
. Replace dense expressions with named intermediate variables where this improves reviewability.
. Keep behavior unchanged unless a bug is discovered and explicitly covered by tests.
. Avoid broad reformat-only churn that makes later archaeology harder.
. Keep each cleanup commit small and traceable.

====== Language scope clarification

For this FTR, "repo-facing messages" means text that maintainers read directly in repository context, for example comments, exception text intended for maintainers, diagnostics in developer tooling, and developer-facing docs snippets near code.

User-visible CLI or report wording can still be handled here when it is clearly developer-oriented or incidental, but any wording change that alters product semantics, compatibility expectations, or intentionally localized business vocabulary should be tracked separately.

====== Non-goals

* No architecture redesign.
* No public API cleanup.
* No CLI option rename or deprecation.
* No adapter contract reshaping.
* No documentation rewrite except small references required by changed code comments or messages.
* No attempt to optimize performance unless an extracted helper would otherwise regress noticeably.

====== Scope

This FTR includes source files under `src/` and directly related tests under `tests/` when needed to pin behavior before refactoring.

Recommended implementation order:

. Inventory source-style findings with a script or `ruff`/AST helper and produce a ranked result that includes file, symbol, line count, finding category, and short rationale. Persist the inventory as a committed artifact only if it remains useful as review evidence; otherwise summarize the ranking in the implementation notes or commit body.
. Fix German comments and repo-facing strings first, because this is low-risk and directly violates the existing project convention.
. Refactor the worst overlong functions one at a time.
. For each refactor, add or confirm characterization tests before changing structure.
. Prefer extracting pure helpers over adding classes or framework abstractions.
. Leave intentionally complex functions alone if splitting would only scatter local logic without improving names or tests.

====== Suggested issue categories

[cols="1,3,3"]
|===
| Category | Description | Preferred treatment

| German source text
| German comments, maintainer-facing diagnostics, or developer documentation snippets in repository source.
| Translate to English or mark as intentional localized domain vocabulary.

| overlong function
| A function is too large to review as one behavior unit.
| Extract named helpers around stable sub-steps; add tests before changing behavior.

| dense expression
| Inline conditional, comprehension, or chained expression hides the reason for a branch.
| Introduce intermediate variables or expand to normal control flow.

| mixed responsibility helper
| A local helper performs parsing, normalization, rendering, and error handling together.
| Split by responsibility if the resulting functions have meaningful names.

| weak naming
| Temporary or generic names make data shape or semantic role unclear.
| Rename locally; avoid public symbol changes unless explicitly approved.
|===

====== Acceptance

* No unintended behavior change is introduced; relevant existing tests pass.
* German comments and repo-facing messages in touched source files are translated or explicitly justified.
* A ranked cleanup inventory exists for current findings and documents the assessment basis, either as a committed artifact or as traceable implementation notes.
* At least the top five currently worst overlong functions from that inventory are assessed, and each is either refactored or explicitly documented as intentionally left alone with rationale.
* Refactored functions have characterization tests or existing tests that cover the moved behavior.
* No public API, CLI flag, YAML key, plugin field, or documented import path changes without separate approval.
* Cleanup commits reference the FTR ID using the project's normal commit-message convention.
