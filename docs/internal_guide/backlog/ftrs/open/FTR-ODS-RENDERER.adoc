===== FTR-ODS-RENDERER — Parity adapter for OpenDocument Spreadsheet

*Status:* Historical placeholder; concrete ODS/Calc scope to be reformulated after Phases 3h / 3i

*Context*::
This older proposal captured the early intuition that an ODS backend should exist once the IR path
was mature enough. Its original wording still assumes a comparatively simple renderer-first addition.

That is no longer the preferred planning shape. With the newer spreadsheet-generalization work in
Phase 3h and the planned architectural guardrails in Phase 3i, the eventual ODS/Calc work should be
specified against the clarified spreadsheet contract rather than against the older "stub now, parity
later" idea.

*Decision*::
Keep this ticket only for historical traceability.
Do not treat it as the current implementation plan for ODS/Calc support.
A later follow-up phase should define the concrete adapter scope after the spreadsheet architecture
has stabilized.

*Original intent (retained for traceability)*::
- Provide an `ODSBackend` stub and register it in the router.
- Reuse IR passes with a different renderer.
- Implement rendering later via `odfpy` with documented fallbacks.
