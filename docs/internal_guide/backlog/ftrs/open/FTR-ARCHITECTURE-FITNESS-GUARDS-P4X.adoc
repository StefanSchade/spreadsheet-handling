===== FTR-ARCHITECTURE-FITNESS-GUARDS-P4X -- Add lightweight fitness guards for layer boundaries

*Status:* Open
*Epic:* Phase 4 -- Feature Expansion and Domain Transformations
*Depends-on:*:: `FTR-FK-HELPER-DOMAIN-EXTRACTION`

*Purpose:*::
Protect the intended responsibility cuts between `pipeline`, `domain`, `core`,
`rendering`, and concrete backends with a small set of executable guardrails so
that feature work cannot silently erode the architecture.

*Context:*::
The architecture documentation already states clear layer responsibilities:
`pipeline` orchestrates and applies generic execution policy, `domain` owns
feature semantics and invariants, `core` provides stateless leaf utilities,
`rendering` owns spreadsheet-generic render preparation, and concrete backends
execute technical persistence. In practice, feature work can still drift across
these cuts because the boundaries are documented but not sufficiently defended
by tests.

The repository already contains spreadsheet-boundary guardrails from
`FTR-SPREADSHEET-BOUNDARY-GUARDS-P3I`. This ticket should extend that existing
line of executable protection with a few new seams; it must not introduce a
second competing architecture-guard regime.

Recent FK-helper work shows the risk clearly: once a step factory already
exists in `pipeline`, feature-specific defaults, metadata handling, or helper
resolution logic can accumulate there even though the intended ownership lies in
`domain`.

This FTR is intentionally sequenced *after*
`FTR-FK-HELPER-DOMAIN-EXTRACTION`: the goal is to protect the intended target
cut after that refactor has landed, not to freeze the current drift into place.

The intended `pipeline` model is deliberately simple: it is a sequencing and
transport layer. The pipeline runs a linear configured step sequence and does
not inspect payload contents for routing decisions. If orchestration-level
routing metadata is ever introduced in the future, it must be an explicit
control envelope outside the domain payload, not an excuse for pipeline to
inspect workbook payload or `_meta`.

*Scope:*::
- Add lightweight architecture guard tests for the documented layer boundaries.
- Add targeted drift protection for the representative FK-helper slice.
- Extend the existing spreadsheet-boundary guard suite with the new seams needed
  for Phase 4 work.
- Document the guarded boundaries in concise implementation-facing form if
  needed.

*Non-Goals:*::
- No heavyweight architecture framework such as import-linter or a Python
  ArchUnit analogue.
- No large-scale package reorganization in this FTR.
- No full typed meta model.
- No attempt to make all boundaries perfectly airtight.
- No general governance framework or second parallel guard harness.

*Solution Direction:*::
- Extend the existing spreadsheet-boundary guard suite rather than creating a
  second independent guard structure.
- Add import-boundary tests that enforce at least the following:
.. `domain` must not import from `pipeline`
.. `core` remains leaf-only and must not import from `domain`, `pipeline`,
   `rendering`, or concrete backend / adapter modules
.. generic `rendering` modules must not import format-specific backend modules
.. spreadsheet adapters must stay on the narrow spreadsheet contract instead of
   importing arbitrary rendering internals directly
- Add a representative source-level or AST-level guard for `pipeline/steps.py`
  so that feature-specific FK-helper semantics do not reappear there after
  `FTR-FK-HELPER-DOMAIN-EXTRACTION`.
- Add a small representative guard for the broader pipeline principle: pipeline
  must not inspect `_meta` or other payload semantics to make feature-specific
  routing or business decisions.
- Keep severity-policy application and step sequencing in `pipeline`, but
  require FK-helper field resolution, helper materialization, helper cleanup,
  and helper-provenance mutation to live in `domain`.
- Add or update one concise architecture note if the guarded rules are not yet
  stated in implementation-facing language.

*Acceptance:*::
- A small automated test suite fails when `domain` imports from `pipeline`.
- A small automated test suite fails when `core` imports from `domain`,
  `pipeline`, `rendering`, or concrete backend / adapter modules.
- A small automated test suite fails when generic `rendering` modules import
  format-specific backend modules.
- A small automated test suite fails when spreadsheet adapters bypass the
  intended spreadsheet contract and import arbitrary rendering internals
  directly.
- After `FTR-FK-HELPER-DOMAIN-EXTRACTION`, FK-helper-specific field resolution
  and helper-provenance mutation no longer live in `pipeline/steps.py`; a guard
  test protects that cut.
- The guard suite documents and protects the simpler orchestration model:
  pipeline sequences configured steps, but does not inspect payload semantics
  such as `_meta` for routing decisions.
- At least one representative feature slice test demonstrates the intended
  division: domain tests prove FK-helper semantics, pipeline tests prove
  orchestration and policy only.
- The guard implementation remains lightweight, readable, and runnable in
  normal CI without introducing a new external architecture-tool dependency.

*Notes:*::
- Prefer a few explicit and understandable tests over a broad but opaque rule
  engine.
- Treat the existing spreadsheet-boundary guard suite as the base to extend, not
  as legacy to duplicate beside a new guard harness.
- This FTR protects the existing architecture; it does not redesign it.
