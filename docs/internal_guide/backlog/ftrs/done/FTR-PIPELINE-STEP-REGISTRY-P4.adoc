====== Purpose

Introduce a JSON-based registry of public pipeline steps as the reviewed inventory of the pipeline YAML surface.

The registry should make each step discoverable, classifiable, testable, and later documentable without changing runtime execution semantics in this first slice.

The registry starts as a review artifact validated against the existing runtime `REGISTRY`. It is not initially the runtime source of truth.

'''

====== Context

`FTR-PIPELINE-SEMANTICS-CONCEPT-P4` established the target model for pipeline step semantics:

* primitive steps execute one deterministic operation
* configuration steps write or validate resolved configuration in `_meta`
* composite steps wrap documented primitive/configuration sequences
* plugin steps remain first-class extension points
* infrastructure steps provide pass-through, bootstrap, override loading, or similar mechanics

The current runtime registry already resolves YAML step names into executable `BoundStep` instances. Earlier step registration normalization introduced a cleaner descriptor/binding path for domain-addressable steps, while keeping special legacy step entries working.

What is still missing is a single reviewable registry that describes the public step surface independently from executable binding code.

Without such a registry, later work on step naming, configuration steps, lifecycle metadata, generated documentation, and architecture guards has no stable inventory to check against.

'''

====== Problem statement

The public pipeline step surface is currently discoverable mostly by reading Python registry code, tests, and scattered FTRs.

This causes several problems:

* Step categories are implicit.
* Step parameter surfaces are not centrally documented.
* It is unclear which steps are primitive, composite, configuration, plugin, or infrastructure.
* It is unclear which metadata paths a step reads or writes.
* It is unclear which frames a step reads, writes, preserves, drops, or treats as internal.
* Inverse behavior is not visible in one place.
* Deprecated names and future aliases cannot be guarded mechanically.
* User documentation cannot yet be generated from a single reviewed source.
* Architecture tests cannot verify the intended public surface without re-encoding it in test logic.

'''

====== Goals

. Add a JSON pipeline step registry as a committed review artifact.
. Cover every current public runtime pipeline step.
. Classify every step as `primitive`, `configuration`, `composite`, `plugin`, or `infrastructure`.
. Document each step's purpose, status, runtime name, target or factory shape, and parameter surface.
. Document each step's metadata contract at least at root-path level.
. Document each step's frame contract at least at input/output/preserve/drop/internal-frame level.
. Document each step's inverse classification.
. Represent composite steps with explicit wrapped-step declarations.
. Represent plugin semantics without constraining plugins out of the architecture.
. Add initial architecture tests that validate registry completeness and structural consistency.
. Keep runtime behavior unchanged.

'''

====== Scope

This FTR includes:

* choose and add the initial registry file
* add a `.gitignore` exception if the registry lives under `docs/`
* add a minimal schema or schema-like validation helper for registry structure
* inventory current runtime `REGISTRY` entries
* create one registry entry for each public runtime step
* classify all current steps
* add structural tests for registry completeness and category validity
* add basic composite transparency checks
* add basic alias lifecycle checks if aliases are already represented
* add basic meta-registry alignment checks where stable enough
* document intentional `unknown` or transitional fields explicitly

Suggested initial registry path:

[source,text]
----
docs/internal_guide/architecture/pipeline_step_registry.json
----

This path keeps the registry near the architecture concept while it is still a review artifact.

If the registry later becomes runtime-bound or directly product-facing, a later FTR may move it to a stable product-facing location such as:

[source,text]
----
src/spreadsheet_handling/pipeline/step_registry.json
----

or:

[source,text]
----
docs/reference/pipeline_step_registry.json
----

'''

====== Non-goals

This FTR does not:

* make the JSON registry the runtime source of truth
* generate runtime `REGISTRY` from JSON
* redesign YAML syntax
* rename steps
* introduce configuration steps
* implement frame lifecycle behavior
* change plugin execution semantics
* change pipeline execution order
* introduce routing, branching, DAG execution, or payload-driven orchestration
* generate user documentation yet
* fully validate JSON Schema as a published external contract

Step renaming belongs to `FTR-PIPELINE-STEP-NAMING-P4`.

Configuration-producing steps belong to `FTR-LOOKUP-FK-CONFIGURATION-STEPS-P4`.

Frame lifecycle behavior belongs to `FTR-FRAME-LIFECYCLE-AND-WORKBOOK-VIEWS-P4`.

Broader guard hardening belongs to `FTR-PIPELINE-SEMANTICS-GUARDS-P4`.

'''

====== Registry status and authority

The initial registry authority model is:

[source,text]
----
runtime REGISTRY
  = executable source of truth

pipeline_step_registry.json
  = reviewed descriptive inventory
  = checked against runtime REGISTRY
  = not used to build steps at runtime
----

This avoids coupling the first documentation/guard slice to runtime loader behavior.

The first implementation should make drift visible, not make the registry executable.

'''

====== Registry schema direction

The exact file schema may be adjusted during implementation, but each public step entry should carry at least the following concepts:

[source,json]

  "factory_shape": "frames_target",
  "aliases": [
    {
      "name": "enrich_lookup",
      "status": "current_runtime_name",
      "replaced_by": "add_lookup_helpers",
      "warn_until": null,
      "remove_after": null
    }
  ],
  "parameters": {
    "source": {
      "required": true,
      "type": "string",
      "source_kind": "inline_only"
    },
    "lookup": {
      "required": true,
      "type": "string",

    },

    }
  },

        "classification": "canonical_meta",

    "reads": ["source", "lookup"],
    "writes": ["output"],

    "drops": [],

    "user_reference": "future_generated"
  }
}
----

The first implementation may use less detail for legacy or not-yet-reviewed steps, but missing detail must be explicit through values such as `unknown`, `current_state`, or `todo_review` rather than silently omitted.

====== Meta-registry alignment

The step registry must not become a second independent metadata authority.

Every persistent root referenced by `meta_contract` should either exist in `docs/internal_guide/architecture/meta_registry.yaml` or be explicitly marked as transient / trace-only in the step registry.

The first implementation may start with a conservative alignment check:

* require each `meta_contract` entry to declare whether it is `persistent`, `transient`, or `trace_only`
* check persistent roots against `meta_registry.yaml` where the current naming is stable
* allow explicitly marked current-state gaps with a follow-up note

Follow-up work may add or refine meta registry entries for:

* `helper_policies`
* `frame_lifecycle`
* `derived.sheets.*.helper_columns`
* `derived.sheets.*.lookup_helpers`
* composite trace namespaces if retained persistently

'''

====== Architecture guard scope in this FTR

This FTR should add only the first low-risk registry guards.

Required first-wave guards:

. Runtime registry completeness: every public runtime `REGISTRY` entry has a JSON registry entry.
. Registry resolution: every JSON entry resolves to a runtime step name, current alias, or explicit plugin/infrastructure exception.
. Category validity: every entry uses a valid category.
. Required-field validation: every entry has name, category, status, purpose, parameter contract, frame contract, inverse classification, and references.
. Composite transparency: composite entries have non-empty `wrapped_steps`, and wrapped step names resolve to registry entries.
. Alias lifecycle shape: alias entries cannot be unstructured strings; deprecated aliases carry `warn_until` and `remove_after`.

Optional if cheap in this slice:

. Meta-registry alignment for stable persistent roots.
. Inverse-kind enum validation.
. Parameter source-kind enum validation.

Deferred to `FTR-PIPELINE-SEMANTICS-GUARDS-P4`:

* source-level checks that primitive steps do not interpret `auto`
* source-level checks that pipeline orchestration does not branch on `_meta`
* source-level checks that rendering does not infer redundancy by frame name or shape
* deeper composite metadata/frame reference validation

'''

====== Acceptance criteria

. `pipeline_step_registry.json` exists and is committed.
. The registry covers every current public runtime pipeline step.
. The registry clearly marks itself as a review artifact, not the runtime source of truth.
. Every step entry has a valid category.

. Composite step entries declare wrapped steps.
. Plugin step behavior is represented without constraining plugin use out of the architecture.
. Infrastructure steps are represented explicitly rather than forced into primitive/configuration categories.
. Any unknown or transitional classifications are explicit.
. A test verifies runtime `REGISTRY` completeness against the JSON registry.

. If the registry lives under `docs/`, `.gitignore` contains a precise exception for it.
. No production pipeline behavior changes are introduced.
. Existing tests continue to pass.

'''

====== Review guidance

When reviewing this FTR, check:

* Does the registry describe the current public surface honestly?
* Are transitional and unknown areas explicit rather than hidden?
* Are composite steps transparent enough to support later lifecycle and inverse reasoning?
* Does the plugin entry preserve the plugin escape hatch?
* Does the registry avoid becoming an executable runtime contract prematurely?
* Are guard tests checking registry structure rather than duplicating implementation details?
* Are metadata paths aligned with the meta registry or clearly marked as follow-up gaps?

'''

====== Rationale

The pipeline is becoming a user-facing framework surface. Without a step registry, users and maintainers must infer behavior from Python factory code, FTR prose, and examples.

A reviewed registry provides a single inventory for:

* architecture tests
* naming cleanup
* configuration-step design
* frame lifecycle design
* plugin responsibility boundaries
* later generated user documentation

Keeping the registry descriptive in the first slice avoids overcommitting to a runtime-loader design before the schema has proven useful.

This is the lowest-risk next step after the pipeline semantics concept.

'''
