===== FTR-GENERIC-FRAME-EXTRACTIONS-P4A -- Add declarative frame extraction and filtering steps

*Status:* Done
*Priority:* P4A
*Area:* Domain transformations / view preparation

==== Purpose

Provide simple declarative extraction steps for deriving named frames from
existing frames without consumer-local Python code.

The first useful shape is intentionally small: select columns from one input
frame, optionally filter rows with a bounded predicate, optionally rename
columns, optionally add configured constant columns, sort rows, and write the
result as a new frame.

==== Example Shape

[source,yaml]
----
- step: extract_frame
  source: variables
  output: visible_variables
  columns: [ID, label, component, data_type]
  where:
    column: active
    equals: true
  rename:
    ID: variable_id
  constants:
    resource_type: field_label
  sort_by: [component, variable_id]
----

==== Goals

* Support `select * from <frame>` by omitting `columns`.
* Support explicit column projection and output ordering.
* Support optional column renaming.
* Support optional constant output columns so split/view recomposition can
  restore discriminator, locale, type, or aspect values explicitly; those
  columns are derived payload columns, not metadata annotations.
* Support deterministic sorting.
* Support a small safe `where` vocabulary, such as equality, membership,
  non-empty, and null/non-null checks.
* Write lifecycle metadata that marks outputs as derived frames unless
  configured otherwise.

==== Non-goals

* Do not implement arbitrary SQL.
* Do not evaluate Python expressions from YAML.
* Do not implement cross-frame joins here; use the bounded join-view FTR for
  predicates that depend on another frame.
* Do not replace `add_lookup_helpers`; use lookup helpers when columns must be
  enriched from another frame by key.
* Do not expand the first predicate vocabulary beyond equality, membership,
  non-empty, and null/non-null checks without a follow-up FTR.
* Do not infer business meaning from column names.

==== Acceptance

* The pipeline registry exposes a generic extraction step.
* A consumer can express common view preparation cases such as `select * from
  table` and selected-column filtered views in YAML.
* Missing columns and unsupported predicates fail with clear messages.
* Tests cover full-frame copy, selected columns, rename, sort, and at least two
  `where` predicate types.
* Tests cover adding one or more constant columns without mutating the source
  frame.
* Output lifecycle metadata marks the result as a derived projection by default
  unless configuration explicitly says otherwise.
* The resulting frames render identically through XLSX and ODS paths.
