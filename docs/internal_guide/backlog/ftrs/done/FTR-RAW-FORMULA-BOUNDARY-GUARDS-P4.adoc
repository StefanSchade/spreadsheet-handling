'''

====== Purpose

Add focused architecture guards that prevent raw backend-specific spreadsheet formula syntax from leaking above the adapter or formula-translation boundary.

This FTR protects the future formula-provider work from starting with stringly typed, backend-specific shortcuts that would make XLSX, ODS, and later spreadsheet backends diverge.

'''

====== Context

`FTR-FORMULA-PROVIDERS` raised a core architectural concern: formula support is valuable, but raw formula strings can easily cross the wrong layer boundary.

The project already distinguishes backend-neutral spreadsheet semantics from backend-specific rendering and parsing details. `RenderPlan` is the adapter-facing language, and concrete backends execute it. Formula syntax is more dangerous than ordinary values because it mixes several concerns:

* semantic meaning, such as list validation or lookup behavior
* reference model, such as named ranges, table ranges, or cell coordinates
* backend syntax, such as XLSX formula strings or ODS formula expressions
* locale and escaping rules
* parser and roundtrip behavior

If raw formula strings are allowed too high in the stack, the architecture will silently become XLSX-shaped again.

This FTR does not implement the formula DSL. It only establishes initial guards against the most obvious drift mode: backend-specific formula strings above the layer where backend translation belongs.

'''

====== Problem statement

Without guards, contributors can easily solve formula-related tasks by inserting strings such as `=XLOOKUP(...)`, `=VLOOKUP(...)`, `=Sheet1!A1:A10`, or backend-specific validation formulas into generic rendering, domain, pipeline, or meta code.

That would create several problems:

* formula behavior would become backend-specific before the formula abstraction exists
* tests would start asserting string equality instead of semantic equivalence
* ODS and XLSX support would drift
* FK-helper formula work could reintroduce domain knowledge into rendering or adapters
* later formula-provider work would have to undo accidental public assumptions

The project needs a small drift-protection layer before formula-provider implementation resumes.

'''

====== Goals

. Prevent obvious raw backend formula syntax above the adapter or formula translator boundary.
. Make formula-boundary violations visible in `tests/architecture/`.
. Establish a clear allowed zone for backend-specific formula strings.
. Keep the guards narrow enough that they do not block existing validation behavior unnecessarily.
. Prepare for `FTR-FORMULA-PROVIDERS` without implementing it here.

'''

====== Boundary rule

Above the adapter or dedicated formula translation layer, formulas must be represented as structured semantics, not backend-specific formula strings.

Allowed above the boundary:

* canonical validation rules in `meta_canonical`, such as `{"type": "in_list", "values": [...]}`
* backend-neutral formula specifications introduced by future formula-provider work
* named-range or symbolic references represented as structured values
* current quarantined adapter-facing fields where explicitly documented as current-state

Not allowed above the boundary:

* direct XLSX formulas such as `=VLOOKUP(...)`, `=XLOOKUP(...)`, `=SUM(A1:A10)`
* direct ODS formulas such as `of:=...` or OpenFormula-specific syntax
* hard-coded A1-style range strings used as semantic source above the adapter boundary
* tests that assert formula string equality when the intended assertion is semantic equivalence

Backend-specific formula strings are allowed only in:

* concrete spreadsheet renderers
* concrete spreadsheet parsers where they are read as carrier-side facts
* dedicated formula translator modules once introduced
* narrowly documented current-state tests under `tests/architecture/current_state/` or equivalent

'''

====== Scope

This FTR includes:

* add architecture tests under `tests/architecture/` that scan selected generic modules for raw formula syntax
* define an allowlist for modules where backend formula strings are currently legitimate
* check for obvious forbidden formula patterns in generic layers
* document the formula-boundary rule in the architecture guard inventory or test README
* mark any existing formula-string checks as current-state if they cannot yet be made semantic

The initial guard should be conservative. It should detect obvious violations without pretending to be a complete formula parser.

'''

====== Suggested guard locations

Place new tests under:

[source,text]
----
tests/architecture/formula_boundary/
----

or, if the architecture-test taxonomy has not yet introduced that subdirectory:

[source,text]
----
tests/architecture/dependency_guards/
tests/architecture/semantic_invariants/
----

The preferred long-term location is `tests/architecture/formula_boundary/`.

'''

====== Candidate forbidden patterns

The initial guard may scan for obvious backend formula markers such as:

[source,text]
----
=VLOOKUP(
=XLOOKUP(
=SUM(
=IF(
"=Sheet"
"!A"
"!$A"
"of:="
"table:formula"
"openformula"
"oooc:"
"msoxl:"
----

The exact pattern list should be deliberately small. The goal is not to reject every string containing `=`. The goal is to catch obvious backend-formula leakage.

'''

====== Candidate scanned areas

The guard should scan generic or higher-level project areas such as:

[source,text]
----
src/spreadsheet_handling/core/
src/spreadsheet_handling/domain/
src/spreadsheet_handling/pipeline/
src/spreadsheet_handling/rendering/
src/spreadsheet_handling/io_backends/spreadsheet_contract.py
----

The guard should not scan concrete backend implementation modules as forbidden areas, because backend syntax is expected there:

[source,text]
----
src/spreadsheet_handling/io_backends/xlsx/
src/spreadsheet_handling/io_backends/ods/
----

A future formula translator package should be explicitly classified. If it translates backend-neutral formula specs into backend syntax, backend formula strings may be allowed there but should remain isolated.

'''

====== Non-goals

This FTR does not:

* implement `FormulaSpec`
* implement a formula DSL
* change validation semantics
* convert existing validation formulas to a new representation
* introduce FK-helper XLOOKUP or formula-driven helper cells
* provide a complete static analysis of all possible formula strings
* forbid backend-specific formula syntax inside concrete adapters or dedicated translators
* make ODS and XLSX formulas fully equivalent

The formula-provider design belongs to `FTR-FORMULA-PROVIDERS` and related follow-up FTRs.

'''

====== Current-state exceptions

Some current code or tests may legitimately contain formula strings before the formula-provider design is complete, especially around data validations.

Such exceptions must be handled explicitly:

* keep them in backend modules if they are backend execution details
* move tests that assert current formula strings into a current-state or quarantine area
* add a comment explaining why the string is not yet represented structurally
* do not let exceptions become implicit permission for new generic formula strings

If an exception is needed in a generic module, this FTR should document it as technical debt and link it to a follow-up formula-provider task.

'''

====== Acceptance criteria

. Architecture tests exist that scan selected generic modules for obvious raw backend formula syntax.
. The scanned areas and allowlisted areas are explicit in the test code.
. Concrete backend modules are not falsely treated as violations merely because they contain backend syntax.
. Any existing generic formula-string usage found by the guard is either removed, moved behind a backend/translator boundary, or documented as a current-state exception.
. Tests distinguish backend-specific formula strings from canonical validation rules.
. At least one test would fail if a raw XLSX-style formula such as `=XLOOKUP(...)` were added to generic rendering, domain, pipeline, or spreadsheet-contract code.
. `tests/README.adoc` or the architecture guard inventory documents the formula-boundary rule.
. No production behavior changes are introduced unless required to remove an obvious boundary violation.
. Existing active tests continue to pass.

'''

'''

====== Review guidance

When reviewing formula-related changes after this FTR, reviewers should ask:

* Is this formula represented as backend-neutral semantics or as backend-specific syntax?
* Is the syntax allowed in this layer?
* Would this work for ODS without changing generic code?
* Is the test asserting semantic behavior or just an XLSX string?
* Is a current-state exception documented, or is it accidental?

'''

====== Rationale

Formula support is one of the easiest ways to undo the spreadsheet architecture work.

A string such as `=XLOOKUP(...)` looks harmless, but it can smuggle backend syntax, reference policy, domain assumptions, and render behavior through several layers at once.

This FTR adds a small mechanical guard before the project resumes formula-provider work. It does not solve formulas. It makes the wrong shortcut visible.
