===== FTR-META-REGISTRY-HARDENING-P3I -- Systematize and harden the manual spreadsheet meta registry

*Status:* Done

*Depends-on:* <<FTR-META-REGISTRY-P3H#,FTR-META-REGISTRY-P3H>>, <<FTR-XLSX-ADAPTER-ALIGNMENT-P3H#,FTR-XLSX-ADAPTER-ALIGNMENT-P3H>>, <<FTR-SPREADSHEET-PARSE-CONTRACT-P3I#,FTR-SPREADSHEET-PARSE-CONTRACT-P3I>>

*Context:* Phase 3h deliberately introduced `meta_registry.yaml` as a manual, review-oriented seed artifact rather than as runtime machinery. That was the right cut, but the reviews also showed that the registry is now important enough to deserve stronger internal structure, clearer maintenance rules, and better drift detection.

*Problem statement:* Without a Phase-3i hardening pass, the registry risks remaining informative but fragile: important fields may drift, carrier-specific entries may be misread as canonical model, producer/consumer references may silently go stale, and tests may continue to prove only that a YAML file exists rather than that it stays semantically useful.

*Goals:*

. Define a clearer internal classification for registry entries such as canonical meta, derived rendering meta, carrier artifacts, and read-path hints.
. Tighten the minimum required profile fields and consistency checks so the registry protects meaning, not just file presence.
. Add lightweight maintenance tooling or tests that make editorial drift, duplicate names, unsupported layer labels, and obviously stale structure easier to catch.
. Keep the registry explicitly manual and review-oriented while making it easier to maintain accurately.
. Align the registry docs with the Phase-3h semantics vocabulary and the parse/write contracts used elsewhere in the spreadsheet architecture.

*Acceptance criteria:*

. `meta_registry.yaml` has a documented, intentional classification scheme for the current entry kinds and their expected fields.
. Tests or equivalent lightweight checks guard duplicate names, required common fields, allowed layer/classification values, and a stronger minimum level of seed completeness than the current existence-oriented checks.
. The architecture docs explain how new entries should be added, how carrier/quarantined entries should be read, and what the registry still does not promise.
. The checks stay out of the product runtime path; they remain review- and maintenance-oriented.
. The resulting registry is strong enough to serve as a dependable review artifact for later adapter work without already pretending to be a typed runtime contract.

*Non-goals:*

- No runtime loader or runtime validator wired into product code.
- No code generation from the registry.
- No full typed meta model.
- No registry-driven IR resolver framework in this FTR.

*Why now:* `meta_registry.yaml` is the most concrete architectural insight that emerged from Phase 3h. Before a second adapter or stronger automation builds on it, the registry itself should be critically reviewed and systematized as a guarded maintenance artifact.
