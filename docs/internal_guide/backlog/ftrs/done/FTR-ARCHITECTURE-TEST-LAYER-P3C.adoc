====== Purpose

Make architecture guardrails and semantic invariants visible as a dedicated test layer instead of letting them remain scattered among ordinary unit and integration tests.

This FTR establishes `tests/architecture/` as the home for tests that protect layer boundaries, dependency direction, spreadsheet contracts, parse contracts, and durable semantic invariants.

The goal is not to add a heavy architecture-testing framework. The goal is to make existing and near-term guardrails explicit, discoverable, and easy to run before feature work expands the surface area again.

'''

====== Context

The project has invested significant effort in separating responsibilities between Domain, Pipeline, Render IR, Spreadsheet Contract, and concrete spreadsheet adapters.

Recent and planned feature work, especially around formula providers, FK-helper evolution, and spreadsheet-facing rendering behavior, can easily reintroduce drift if architecture tests remain implicit or scattered.

Several current tests already behave like architecture guardrails:

* dependency-direction checks between generic rendering modules and format-specific adapters
* spreadsheet contract checks around narrow adapter entry points
* parse contract checks around `WorkbookIR` and projection boundaries
* semantic invariant checks around `meta_canonical`, `meta_rendering`, carrier hints, and `RenderPlan`

These tests are conceptually different from ordinary unit tests. They protect the shape of the system, not only one function's output.

After `FTR-TEST-SUITE-TOPOLOGY-P3C`, the test tree should have a physical place for these tests. This FTR populates and sharpens that architecture layer.

'''

====== Problem statement

Architecture guardrails are currently too easy to overlook.

If guardrail tests are mixed into ordinary unit or integration folders, several problems follow:

* reviewers may not notice when a change weakens an architectural invariant
* feature work may add tests at the wrong level
* backend-specific behavior may slowly leak upward into generic rendering or domain code
* adapter tests may start checking semantics that should have been resolved earlier
* durable invariants may be confused with current-state or quarantine checks

This is especially risky before formula-related work, where raw strings, backend-specific formula syntax, coordinate references, and FK-helper semantics could easily cross the wrong boundary.

'''

====== Goals

. Establish `tests/architecture/` as the explicit home for architecture guardrails.
. Move existing architecture-style tests into this layer where appropriate.
. Separate durable invariants from smoke checks and current-state quarantine checks.
. Define a small taxonomy for architecture tests so future guardrails are placed consistently.
. Keep the approach lightweight and compatible with the current pytest-based suite.
. Prepare the suite for feature work that must preserve render, adapter, parse, and formula boundaries.

'''

====== Proposed architecture test layout

[source,text]
----
tests/
  architecture/
    dependency_guards/
    spreadsheet_contract/
    parse_contract/
    semantic_invariants/
    current_state/
----

The exact subdirectory names may be adjusted during implementation if existing tests suggest a better split, but the distinction between dependency guards, contract guards, semantic invariants, and current-state checks should remain visible.

'''

====== Test categories

====== Dependency guards

Dependency guards protect import direction and package boundaries.

Examples:

* generic rendering modules must not import XLSX, ODS, OpenPyXL, or odfpy adapter internals
* spreadsheet adapters must use the narrow spreadsheet contract instead of importing arbitrary rendering internals
* Domain modules must not import spreadsheet adapters
* formula provider abstractions must not depend on concrete backend libraries

Dependency guards may use AST inspection, import graph inspection, or small focused runtime checks. They should stay narrow and should avoid becoming broad lint replacements.

====== Spreadsheet contract guards

Spreadsheet contract guards protect the write-side adapter boundary.

Examples:

* adapters receive or execute backend-neutral `RenderPlan` operations
* backend-specific renderers do not reinterpret canonical meta as domain semantics
* spreadsheet adapters call the generic spreadsheet contract facade instead of duplicating compose/pass/plan orchestration
* render-facing formula specifications remain backend-neutral before adapter translation

====== Parse contract guards

Parse contract guards protect the read-side boundary.

Examples:

* format-specific parsers produce `WorkbookIR`
* generic projection from `WorkbookIR` to `Frames + persisted _meta` remains outside format-specific parsers
* carrier-side hints such as `__freeze`, `__autofilter_ref`, or equivalent parser hints are not promoted to canonical meta
* parser-specific interpretation helpers remain local to the adapter package

====== Semantic invariants

Semantic invariant tests protect durable architectural statements rather than current implementation details.

Examples:

* `meta_canonical` may be persisted, but must not drive visible render behavior after IR derivation
* roundtrip-relevant information must not live only in `meta_rendering`
* workbook and sheet option precedence is resolved before adapter execution
* non-layout spreadsheet semantics such as validations and named ranges reach the backend-neutral plan layer
* formula semantics above the adapter boundary are represented structurally, not as raw backend formula strings

====== Current-state and quarantine checks

Some tests intentionally describe current carrier forms or transitional implementation details.

These tests are allowed, but they must be clearly separated from durable invariants.

Examples:

* current `WriteMeta` carrier representation
* current parser hint names
* current backend-specific normalization details
* temporary formula string compatibility during migration

Such tests should live under `tests/architecture/current_state/` or carry an explicit module docstring explaining why the checked behavior is transitional.

'''

====== Scope

This FTR includes:

* create or populate the `tests/architecture/` subtree introduced by the test topology FTR
* move existing architecture-style tests into the appropriate architecture subdirectory
* add module docstrings that state which invariant or boundary a test file protects
* mark architecture tests consistently, if marker normalization already exists or can be done locally without broad Makefile changes
* distinguish durable invariants from current-state or quarantine checks
* update `tests/README.adoc` with a short section describing architecture tests
* keep behavior and coverage equivalent while moving tests

'''

====== Non-goals

This FTR does not:

* implement formula providers
* add a full architecture testing framework
* replace ruff, mypy, or ordinary linting
* redesign the entire Makefile or CI matrix
* classify all legacy pre-hex tests
* broaden every existing guardrail into a complete package-wide rule
* turn current-state implementation checks into durable product contracts

Broader marker and Make target normalization should remain in `FTR-TEST-MARKERS-AND-MAKE-TARGETS-P3C`. Broader architecture fitness expansion should remain in `FTR-ARCHITECTURE-FITNESS-GUARDS-P4X` or a dedicated follow-up.

'''

====== Migration rules

. Move tests into `tests/architecture/` only if they protect a boundary, contract, dependency direction, or semantic invariant.
. Do not move ordinary unit tests into the architecture layer merely because they touch rendering, parsing, or adapters.
. Do not move integration roundtrip tests into the architecture layer unless the test's primary assertion is an architectural invariant.
. If a test mixes product behavior and architecture assertions, split it only when the split is straightforward and low-risk.
. If the split is not straightforward, place the file according to its dominant purpose and leave a TODO for follow-up consolidation.
. Use module docstrings to state the protected boundary or invariant.
. Keep current-state or carrier-shape checks visibly separate from durable invariants.
. Do not change production code behavior as part of this FTR.

'''

====== Acceptance criteria

. Existing architecture-style tests are collected under `tests/architecture/` or explicitly left elsewhere with a justified reason.
. `tests/architecture/` contains at least the categories needed for dependency guards, spreadsheet contract guards, parse contract guards, semantic invariants, and current-state checks.
. Each architecture test module has a short docstring explaining the protected boundary or invariant.
. Durable invariant tests are not mixed silently with current-state or quarantine checks.
. At least one existing guardrail from the spreadsheet contract, parse contract, or semantic invariant area is moved into the new architecture test layer.
. The default test command still passes.
. A focused architecture test command can be introduced now or is explicitly deferred to `FTR-TEST-MARKERS-AND-MAKE-TARGETS-P3C`.
. `tests/README.adoc` explains when to add a test under `tests/architecture/` instead of `tests/unit/` or `tests/integration/`.
. No production behavior changes are introduced.

'''

====== Review guidance

When reviewing changes under `tests/architecture/`, reviewers should ask:

* Which boundary or invariant does this test protect?
* Is the test checking durable architecture or only current implementation shape?
* Would this assertion still make sense for both XLSX and ODS unless explicitly scoped?
* Is the test preventing drift, or is it just duplicating a product behavior test?
* Is a backend-specific detail being promoted into a generic contract by accident?

If these questions cannot be answered clearly, the test probably belongs under `tests/unit/`, `tests/integration/`, or `tests/architecture/current_state/` instead.

'''

====== Rationale

The project is now in Phase 4 feature work, but that feature work depends on the architectural boundaries established in the previous phases. The backlog phase labels are partly historical, yet the technical risk remains current.

Formula providers, FK-helper changes, and spreadsheet-facing enhancements will all put pressure on the same seams: Domain versus Render IR, RenderPlan versus adapter, canonical meta versus carrier hints, and backend-neutral formula semantics versus backend-specific syntax.

A dedicated architecture test layer makes these seams visible and reviewable. It also avoids overloading ordinary unit and integration tests with responsibilities they cannot communicate clearly by location.

This FTR should be implemented early in the test cleanup sequence, after the physical topology exists and before larger formula or rendering feature work resumes.
