===== FTR-FK-HELPER-REFACTOR-P3B — Re-slice FK helper behavior across domain, validations, and rendering

*Status:* Open
*Epic:* Phase 3b — Pipeline & Orchestration Consolidation

*Purpose:*::
Use FK-helper behavior as a representative feature slice to separate data transformation, validation, and spreadsheet presentation concerns.

*Context:*::
FK-helper behavior currently spans multiple concerns: detection of FK-based enrichments, materialization of helper columns, consistency checks, and optional spreadsheet-specific treatment such as visual distinction or protection. In the transitional architecture these concerns are still too tightly grouped.

*Scope:*::
- Move FK-helper column materialization and cleanup toward pure domain transformations.
- Introduce explicit validation points for FK-helper consistency problems.
- Keep spreadsheet-specific helper rendering concerns outside the transformation core.
- Preserve bijective roundtrip expectations between maintenance views and canonical representations.

*Examples of validation findings:*::
- unexpected helper column present
- expected helper column missing
- helper value differs from canonical FK-derived value
- referenced FK target cannot be resolved

*Non-Goals:*::
- No domain-specific business semantics.
- No mandatory formula/VLOOKUP-based implementation.
- No full UI/process policy system.

*Solution Direction:*::
- Domain transformations own helper-column add/remove logic and helper-related meta.
- Domain validations own detection of helper inconsistencies and FK-related integrity problems.
- IR/rendering owns optional visual treatment such as helper styling or protection.
- Backends only execute rendering instructions.

*Acceptance:*::
- FK-helper behavior is expressible without embedding spreadsheet-specific logic into the core transformation path.
- Helper-column generation and removal are testable as pure data transformations.
- Consistency problems are detectable independently from rendering.
- Spreadsheet-specific presentation can be layered on top without changing canonical data semantics.
