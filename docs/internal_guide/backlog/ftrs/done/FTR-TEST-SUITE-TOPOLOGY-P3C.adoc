===== FTR-TEST-SUITE-TOPOLOGY-P3C — Establish a responsibility-based active test suite topology

*Status:* Implemented
*Epic:* Phase 3c — Test Consolidation after Architecture Stabilization
*Priority:* High

*Depends-on:*

* <<FTR-PREHEX-TEST-CONSOLIDATION-P3C#,FTR-PREHEX-TEST-CONSOLIDATION-P3C>>
* <<FTR-SPREADSHEET-BACKEND-CONTRACT-P3H#,FTR-SPREADSHEET-BACKEND-CONTRACT-P3H>>
* <<FTR-SPREADSHEET-PARSE-CONTRACT-P3I#,FTR-SPREADSHEET-PARSE-CONTRACT-P3I>>
* <<FTR-SPREADSHEET-SEMANTIC-INVARIANTS-P3I#,FTR-SPREADSHEET-SEMANTIC-INVARIANTS-P3I>>

'''

====== Purpose

Establish a clear physical topology for the active test suite so that test files communicate their responsibility by location before markers or file names have to be inspected.

The test suite should distinguish ordinary unit behavior, cross-layer integration behavior, architecture and contract guardrails, and historical quarantined tests. This makes future feature work safer because contributors can place new tests in the correct layer instead of extending the current historically grown structure.

'''

====== Context

The `src/` package structure has been cleaned up and now more clearly reflects the architectural responsibilities of the project. The test tree has not yet received the same treatment.

Recent work introduced a number of focused guardrails and semantic invariant tests around the spreadsheet contract, parse contract, `meta_canonical` / `meta_rendering`, adapter boundaries, and backend-neutral `RenderPlan` behavior. These tests are conceptually different from ordinary unit tests, but they are not yet consistently visible as a dedicated test layer.

The project also still carries historical tests from earlier architecture phases. Some of them describe still-relevant behavior, while others protect obsolete implementation structure. These tests must remain quarantined until explicitly classified, promoted, rewritten, or retired.

Without a clear test topology, new features such as formula providers, FK-helper evolution, or additional rendering semantics would likely add more tests to arbitrary locations and make future drift harder to detect.

'''

====== Problem statement

The current test layout does not make the role of each test obvious enough.

This creates several risks:

* architecture guardrails may be mistaken for ordinary unit tests
* integration tests may be scattered across module-oriented paths
* historical tests may remain a hidden source of requirements without a clear disposition
* new contributors may add tests near the code they touched instead of near the behavior or boundary they protect
* marker-based slicing may diverge from the physical structure of the suite

The result is not only cosmetic disorder. It weakens reviewability and makes architecture drift easier to reintroduce.

'''

====== Goals

. Introduce a responsibility-based top-level structure for active tests.
. Make architecture and contract guardrails visible as a first-class test category.
. Keep historical pre-hex tests quarantined and clearly separated from the active suite.
. Preserve current behavior and CI confidence while moving files.
. Provide a short, explicit placement guide for future tests.
. Prepare the suite for follow-up work on markers, Make targets, architecture guardrails, and formula-related FTRs.

'''

====== Proposed test topology

The active test suite should be organized under these top-level directories:

[source,text]
----
tests/
  unit/
    core/
    domain/
    pipeline/
    rendering/
    io_backends/

  integration/
    pipeline/
    spreadsheet_contract/
    io_backends/
    roundtrip/

  architecture/
    dependency_guards/
    semantic_invariants/
    spreadsheet_contract/
    parse_contract/

  legacy_pre_hex/
----

This FTR establishes the topology and moves existing active tests into the closest matching category. It does not require rewriting the test logic unless minimal import-path adjustments are needed after moving files.

'''

====== Classification rules

Tests should be placed according to the behavior they protect, not according to the implementation file that happened to motivate the test.

====== Unit tests

Unit tests belong under `tests/unit/`.

A test is a unit test if it primarily verifies one module, function, class, or small local collaboration without exercising a full product path.

Typical examples:

* pure Core helpers
* Domain transformations
* Domain validations
* small rendering helpers
* formula translator units once formula provider work begins
* parser helper functions where no full workbook read path is exercised

Unit tests should avoid real file I/O unless the file system itself is the behavior under test.

====== Integration tests

Integration tests belong under `tests/integration/`.

A test is an integration test if it verifies a real product path across multiple layers or a concrete backend boundary.

Typical examples:

* `Frames + meta_canonical -> WorkbookIR -> RenderPlan`
* `read_spreadsheet_frames(...)` over a parser boundary
* adapter write/read paths
* XLSX or ODS roundtrips
* cross-adapter parity checks
* pipeline execution involving multiple configured steps

Integration tests may use temporary files when they test adapter, parser, renderer, or roundtrip behavior.

====== Architecture tests

Architecture tests belong under `tests/architecture/`.

A test is an architecture test if it protects a boundary, dependency direction, semantic invariant, or long-term architectural contract rather than a single product behavior.

Typical examples:

* generic rendering modules do not import format-specific adapters
* spreadsheet adapters use the narrow spreadsheet contract instead of importing rendering internals
* carrier-side hints are not promoted to canonical meta
* `meta_canonical` is not render-driving after IR derivation
* backend-neutral `RenderPlan` operations remain the adapter-facing language
* FormulaSpec or future formula providers do not leak raw backend formulas above the adapter boundary

Architecture tests may inspect imports, AST structure, dependency direction, or representative semantic invariants. They should remain focused and should avoid becoming broad end-to-end product tests.

====== Legacy pre-hex tests

Historical pre-hex tests remain under `tests/legacy_pre_hex/`.

They are not part of the active topology until explicitly classified as one of:

* `retire` — obsolete implementation detail
* `rewrite` — still-relevant behavior expressed through obsolete structure
* `promote` — behavior already suitable for the active suite after relocation or minor adjustment

This FTR does not complete that classification. It preserves the quarantine boundary so that the follow-up pre-hex classification work can proceed deliberately.

'''

====== Scope

This FTR includes:

* create the top-level active test directories `tests/unit/`, `tests/integration/`, and `tests/architecture/` if they do not already exist
* move active test files into the closest responsibility-based location
* keep `tests/legacy_pre_hex/` outside the default active suite
* add or update a concise `tests/README.adoc` describing the topology and placement rules
* adjust imports, relative fixture paths, and conftest placement as needed after moving files
* keep existing test behavior unchanged except for path-related adjustments
* keep marker changes minimal unless required to preserve current test execution

'''

====== Non-goals

This FTR does not:

* introduce new formula behavior
* rewrite existing tests for better assertion style unless relocation requires a minimal adjustment
* classify every pre-hex test as `retire`, `rewrite`, or `promote`
* introduce a full architecture enforcement framework such as import-linter
* redesign CI completely
* decide final marker policy
* remove tests only because they are old
* reduce coverage by consolidating tests aggressively

Marker normalization, Make target cleanup, pre-hex classification, and broader guardrail expansion are follow-up FTRs.

'''

====== Migration rules

. Move tests by responsibility, not by package name alone.
. Prefer the smallest truthful category. A pure helper test should stay in `unit` even if the helper belongs to a backend package.
. Put cross-layer or real file-format tests under `integration`.
. Put import, dependency, semantic invariant, and contract boundary tests under `architecture`.
. Do not mix historical pre-hex tests into the active suite during this FTR.
. If a file contains tests from multiple categories, split it only where the split is straightforward and low-risk. Otherwise move it to the dominant category and leave a TODO comment for follow-up cleanup.
. Do not change expected behavior while moving tests.
. Run the same default test command before and after the move and treat behavior changes as regressions unless explicitly justified.

'''

====== Documentation

Add or update `tests/README.adoc` with:

* the purpose of each top-level test directory
* the classification rules from this FTR in concise form
* examples of where to place common new tests
* a warning that `tests/architecture/` is for boundary and invariant protection, not a dumping ground for difficult tests
* the current status of `tests/legacy_pre_hex/` as quarantined historical material

The README should be practical and short enough to be useful during code review.

'''

====== Acceptance criteria

. The active test suite has the top-level directories `tests/unit/`, `tests/integration/`, and `tests/architecture/`.
. Active tests are moved into the closest responsibility-based directory according to the classification rules in this FTR.
. Architecture and contract guardrail tests are no longer scattered only among ordinary unit or integration tests; they are visible under `tests/architecture/`.
. `tests/legacy_pre_hex/` remains quarantined and outside the default active test run.
. A concise `tests/README.adoc` documents the test topology and placement rules.
. The default test command still passes after the move.
. Existing specialized test commands or CI jobs continue to pass, or any intentional command changes are explicitly documented as follow-up work.
. No production code behavior changes are introduced by this FTR.
. No tests are removed unless they are exact duplicates created by the move itself.

'''

====== Suggested follow-up FTRs

This FTR intentionally creates the physical test topology first. The following work should remain separate:

* `FTR-TEST-MARKERS-AND-MAKE-TARGETS-P3C` — normalize markers and Make targets against the new topology
* `FTR-ARCHITECTURE-TEST-LAYER-P3C` — strengthen architecture guardrail taxonomy and coverage
* `FTR-PREHEX-TEST-CLASSIFICATION-P3C` — classify, promote, rewrite, or retire historical pre-hex tests
* `FTR-INTEGRATION-SLICE-CONSOLIDATION-P3C` — consolidate cross-layer integration slices and reduce accidental duplication
* `FTR-TEST-NAMING-AND-CONVENTIONS-P3C` — normalize naming, docstrings, and review conventions

'''

====== Rationale

The project has already invested heavily in clarifying boundaries between Domain, Pipeline, Render IR, Spreadsheet Contract, and concrete backends. The test suite should now reflect the same boundaries.

A marker-only strategy would not be enough. Markers are useful for execution slices, but they do not make the suite legible when browsing files or reviewing a change. Physical topology should communicate the primary responsibility of a test; markers should remain a secondary execution and filtering axis.

This FTR is therefore a low-risk structural cleanup with high leverage. It reduces future drift before larger formula and rendering features add more tests.
