'''

====== Purpose

Use the first `spreadsheet-handling-demo` integration of the Phase-4 compact-transform feature chain to reduce avoidable consumer-facing friction in the main repository.

This FTR does not add new transform semantics. It hardens the API, diagnostics, examples, and run-path support around the already implemented compact-transform primitives.

'''

====== Context

The demo repository successfully exercised the local working tree of `spreadsheet-handling` with the new Phase-4 compact-transform chain:

* legend/codebook blocks
* XRef matrix/relation expansion and contraction
* cell-codec parsing and serialization
* compact-multiaxis composition
* XLSX workbook output
* ODS/Calc workbook output

The integration did not reveal a fundamental architecture problem. It did reveal consumer-facing friction:

* local-library setup versus the pinned PyPI demo version
* `drop_empty: true` as a sparse-roundtrip footgun
* choosing `whole_cell_code` versus `split_tokens`
* repeated YAML boilerplate in compact-transform flows
* row-key identity semantics not being obvious enough
* legend `group` projection being easy to over-interpret
* ODS/Calc routing through `sheets-run` needing explicit support

This FTR captures those findings as a small main-repo hardening slice.

'''

====== Problem statement

The compact-transform feature chain is functionally usable, but a consumer currently needs to understand too many details at once:

* matrix versus relation shape
* XRef row and column identity
* codec mode selection
* legend-based validation
* sparse versus strict-shape roundtrip behavior
* workbook output routing

This is acceptable for first implementation tests, but too fragile for demo-facing or early user-facing flows.

The main risk is not missing capability. The risk is that users or future agents compensate for unclear configuration by adding heuristics, aliases, or domain-specific shortcuts.

'''

====== Goals

. Reduce observed compact-transform YAML friction where a small change improves actual demo usage.
. Make sparse behavior explicit and hard to misunderstand.
. Improve diagnostics around `drop_empty`, `whole_cell_code`, `split_tokens`, missing legend blocks, and unsupported `dense_axes`.
. Clarify that `row_keys` define identity.
. Clarify that display labels only roundtrip when explicitly included as row keys.
. Clarify that `group` is a generic legend/codebook projection, not business semantics.
. Ensure `sheets-run` supports the intended XLSX and ODS/Calc demo routes consistently.
. Keep all behavior deterministic and configuration-driven.

'''

====== Design constraints

* Configuration must remain explicit and deterministic.
* Defaults may be convenient, but they must not obscure lossiness.
* If a mode can drop empty-only groups, documentation and diagnostics must say so.
* Missing rows, columns, or row/column combinations must not be reconstructed heuristically.
* Any future dense reconstruction must remain behind an explicit `dense_axes` contract.
* Demo ergonomics must not weaken the separation between `XREF-CROSSTABLE`, `CELL-CODEC`, `LEGEND-BLOCKS`, and `COMPACT-MULTIAXIS`.
* Generic transforms must not acquire domain-specific meaning.

'''

====== User-facing semantics to clarify

====== `row_keys`

`row_keys` define identity. A display label is preserved through a compact-transform roundtrip only if it is explicitly part of `row_keys` or otherwise carried by the configured flow.

The transform must not infer identity from visible labels, similar column names, grouped headers, or table positions.

====== `drop_empty`

`drop_empty: true` is a valid sparse default. It means:

* empty compact cell values disappear from the decoded relation
* empty-only row/column groups may disappear
* strict matrix-shape roundtrip is not guaranteed

For strict matrix-shape roundtrip without later explicit dense-axis reconstruction, users must choose `drop_empty: false`.

====== `whole_cell_code` versus `split_tokens`

`whole_cell_code` treats a value such as `E-R-K` as one code.

`split_tokens` splits only when explicitly configured with a delimiter.

Punctuation must not imply token structure by itself.

====== `group`

`group` is a generic projection from legend/codebook metadata or explicit code-group configuration.

It must not become a domain classification mechanism inside the generic compact-transform layer.

'''

====== Scope

This FTR includes:

* review the Phase-4 demo YAML and identify repeated or confusing compact-transform configuration fragments
* add small API conveniences only where they reduce observed demo friction
* improve diagnostics for missing legend blocks, invalid codec mode, missing value columns, unsupported `dense_axes`, and likely sparse-roundtrip misunderstandings
* document sparse and strict-shape behavior in compact-transform examples
* ensure `ods` and `calc` input/output routing works consistently through `sheets-run`
* add focused orchestrator coverage for ODS/Calc routing if not already present
* add one demo-shaped main-repo test for the intended compact-multiaxis consumer flow if it provides useful coverage

'''

====== Candidate API improvements

The implementation may choose one or more of the following if they demonstrably reduce friction:

* clearer examples for `expand_compact_multiaxis` and `contract_compact_multiaxis`
* grouped configuration in documentation, such as a conceptual `codec` section, without necessarily changing the runtime API
* clearer naming or diagnostics for sparse versus strict-shape roundtrip modes
* better error messages when `allowed_from_legend` points to a missing or malformed legend block
* better error messages when `split_tokens` is configured without a usable delimiter
* better error messages when `dense_axes` is supplied before implementation
* a small helper or alias only if it removes repeated demo boilerplate without hiding semantics

Any new convenience must remain a thin configuration aid. It must not introduce a second transform model.

'''

====== ODS/Calc run-path hardening

The demo integration showed that workbook output is not only an adapter concern. The application orchestration path must route the configured output kind correctly.

This FTR includes the small main-repo hardening needed so that `sheets-run` can route XLSX and ODS/Calc flows consistently.

Acceptance for this part should be covered by focused orchestration tests rather than only by demo-repo scripts.

'''

====== Non-goals

This FTR does not:

* implement `dense_axes`
* infer row or column axes from visible labels, names, or layout
* introduce domain-specific compact-transform semantics
* replace the primitive `expand_xref`, `contract_xref`, `decode_cell_values`, or `encode_cell_values` steps
* hide sparse behavior behind a magical roundtrip mode
* require a new package release before demo iteration continues
* move demo-specific files into the main repository

'''

====== Example documentation target

The compact-transform examples should make the following distinction obvious:

[source,yaml]
----
# Sparse relation: empty compact cells disappear.
- step: expand_compact_multiaxis
  matrix: product_matrix
  output: feature_product_codes
  row_keys: [feature_id]
  value_columns: [P-001, P-002]
  mode: whole_cell_code
  allowed_from_legend: status_codes
  drop_empty: true
----

[source,yaml]
----
# Strict matrix-shape roundtrip: empty compact cells remain explicit.
- step: expand_compact_multiaxis
  matrix: product_matrix
  output: feature_product_codes
  row_keys: [feature_id]
  value_columns: [P-001, P-002]
  mode: whole_cell_code
  allowed_from_legend: status_codes
  drop_empty: false
----

The examples should also show that a value such as `E-R-K` remains one code in `whole_cell_code` mode.

'''

====== Acceptance criteria

. The Phase-4 compact-multiaxis demo flow can be expressed with clearer documentation or reduced repeated configuration where justified by observed friction.
. `row_keys` identity semantics are documented clearly.
. Documentation states that display labels roundtrip only when explicitly included in the configured identity or carried by the configured flow.
. Sparse behavior is documented in compact-transform examples.
. Strict-shape roundtrip guidance explicitly points to `drop_empty: false` unless a future explicit dense-axis reconstruction mode is configured.
. `whole_cell_code` and `split_tokens` are documented with a clear example showing that punctuation alone does not imply splitting.
. `group` is documented as a generic legend/codebook projection, not business semantics.
. `sheets-run` routes XLSX and ODS/Calc demo workbook flows consistently.
. ODS/Calc routing is covered by a focused main-repo test.
. Error messages for the main demo friction cases are clear enough to diagnose without reading implementation code.
. No heuristic dense-axis reconstruction is introduced.
. Existing compact-transform unit, pipeline, architecture, and integration tests remain green.

'''

====== Review guidance

When reviewing this FTR, ask:

* Does this make the demo easier to configure without hiding semantics?
* Is the behavior still deterministic?
* Would a user understand when empty rows or cells are lost?
* Does this preserve the separation between XRef, CellCodec, Legend, and CompactMultiAxis?
* Is this an actual API improvement, or merely a new alias for existing complexity?
* Are ODS/Calc changes isolated to orchestration and adapter routing rather than leaking into transform semantics?

'''

====== Rationale

The first demo integration shows that the Phase-4 feature chain is viable.

The remaining issue is not architecture. It is early consumer ergonomics.

A small hardening slice now is cheaper than allowing every demo or consumer repository to invent its own wrapper conventions around compact-transform primitives.

'''

====== Implementation notes

Delivered as a focused ergonomics hardening slice around the existing compact
transform chain.

Implemented behavior:

* clearer diagnostics for unsupported cell-codec modes, missing
  `allowed_from_legend` blocks, empty `split_tokens` delimiters, configured
  `value_columns`, and unsupported `dense_axes`
* a user-guide capability page with compact-transform examples for identity
  (`row_keys`), sparse versus strict-shape roundtrips, `whole_cell_code` versus
  `split_tokens`, generic `group` projection, and XLSX/ODS/Calc routing through
  `sheets-run`
* focused tests for display labels roundtripping only when configured as
  `row_keys`
* focused ODS/Calc orchestration coverage for the `calc` backend alias
* CLI help text updated so `sheets-run` advertises `ods` and `calc` as supported
  input/output kinds

No new transform semantics or dense-axis reconstruction were introduced.
