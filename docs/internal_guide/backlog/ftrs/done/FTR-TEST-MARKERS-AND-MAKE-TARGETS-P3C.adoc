===== FTR-TEST-MARKERS-AND-MAKE-TARGETS-P3C — Normalize test markers and Make targets

*Status:* Implemented
*Epic:* Phase 4 — Feature Expansion and Domain Transformations
*Priority:* High

*Depends-on:*

* <<FTR-TEST-SUITE-TOPOLOGY-P3C#,FTR-TEST-SUITE-TOPOLOGY-P3C>>
* <<FTR-ARCHITECTURE-TEST-LAYER-P3C#,FTR-ARCHITECTURE-TEST-LAYER-P3C>>

*Related-to:*

* <<FTR-PREHEX-TEST-CLASSIFICATION-P3C#,FTR-PREHEX-TEST-CLASSIFICATION-P3C>>
* <<FTR-ARCHITECTURE-FITNESS-GUARDS-P4X#,FTR-ARCHITECTURE-FITNESS-GUARDS-P4X>>
* <<FTR-FORMULA-PROVIDERS#,FTR-FORMULA-PROVIDERS>>

'''

====== Purpose

Normalize pytest markers and Make targets so the reorganized test suite can be executed by clear, stable slices.

The physical test topology should explain what a test is. Markers and Make targets should explain how to run useful subsets of the suite.

This FTR makes those two axes consistent without turning marker usage into a substitute for responsibility-based test placement.

'''

====== Context

`FTR-TEST-SUITE-TOPOLOGY-P3C` introduces a responsibility-based test layout with `tests/unit/`, `tests/integration/`, and `tests/architecture/`.

`FTR-ARCHITECTURE-TEST-LAYER-P3C` makes architecture guardrails visible as a dedicated test layer.

After those structural changes, the execution model must be aligned. Developers need predictable commands for common slices such as fast unit tests, integration tests, architecture guardrails, IR-related spreadsheet tests, legacy tests, and optional pre-hex checks.

The existing test harness already uses markers and Make targets in parts of the project, but the marker language should be normalized so it does not drift from the new topology.

'''

====== Problem statement

Markers and Make targets are currently at risk of becoming inconsistent with the physical test layout.

If markers remain ad hoc:

* developers cannot reliably run the slice they intend
* architecture guardrails may be skipped accidentally
* legacy or pre-hex tests may leak into default execution
* backend-specific test slices may become unclear
* CI jobs may not reflect the suite's conceptual structure

The project needs a small, explicit marker vocabulary and matching Make targets.

'''

====== Goals

. Define a compact marker vocabulary aligned with the test topology.
. Provide Make targets for the most important test slices.
. Keep the default test run safe and useful for normal development.
. Keep historical and expensive slices explicit rather than accidental.
. Make marker usage discoverable through documentation and `make help`.
. Avoid marker proliferation.

'''

====== Marker policy

Markers are a secondary execution axis. They do not replace physical placement under `tests/unit/`, `tests/integration/`, or `tests/architecture/`.

Use markers only when they add execution value.

====== Required markers

[cols="1,3"]
|===
| Marker | Meaning

| `unit`
| Fast isolated tests, normally located under `tests/unit/`.

| `integ`
| Cross-layer or backend-facing integration tests, normally located under `tests/integration/`.

| `arch`
| Architecture, dependency, contract, or semantic invariant tests, normally located under `tests/architecture/`.

| `guardrail`
| Tests whose primary purpose is to prevent architectural or semantic drift. Usually combined with `arch`.

| `xlsx_ir`
| Tests specifically targeting the XLSX IR-based spreadsheet path.

| `ods`
| Tests specifically targeting the ODS / Calc spreadsheet path.

| `legacy`
| Historical or deprecated tests excluded from the default run.

| `prehex`
| Tests from the pre-hex historical suite, always excluded from the default run.
|===

====== Optional markers

[cols="1,3"]
|===
| Marker | Meaning

| `slow`
| Tests that are valid but too slow for the normal default slice.

| `smoke`
| Small end-to-end or consumer-facing confidence checks.

| `current_state`
| Tests that intentionally document current implementation shape or transitional carrier behavior rather than durable invariants.
|===

New markers should not be added casually. If a proposed marker does not correspond to a useful execution slice, prefer a directory, filename, or docstring instead.

'''

====== Make targets

The Makefile should expose a small set of predictable test commands.

[cols="1,3"]
|===
| Target | Expected behavior

| `make test`
| Run the normal development slice. Include unit, integration, and architecture tests that are not marked `legacy`, `prehex`, or `slow`.

| `make test-unit`
| Run unit tests only.

| `make test-integ`
| Run integration tests only.

| `make test-arch`
| Run architecture and guardrail tests only.

| `make test-ir`
| Run XLSX IR-specific tests.

| `make test-ods`
| Run ODS / Calc-specific tests.

| `make test-smoke`
| Run smoke tests if such tests exist.

| `make test-all`
| Run all active non-legacy tests, including slow tests if practical.

| `make test-prehex`
| Explicitly run quarantined pre-hex tests. This target must not be part of the default run.
|===

If some targets already exist with equivalent meaning, prefer preserving their names and aligning their behavior rather than renaming for cosmetic consistency.

'''

====== Default test behavior

`make test` should remain safe for everyday development.

It should:

* run the active test suite
* include architecture guardrails unless they are explicitly marked slow or current-state-only
* exclude `legacy` and `prehex` tests
* exclude `slow` tests unless the project already treats slow tests as part of the default run
* fail fast enough to be useful during normal development

Architecture tests should not be silently excluded from the default test run merely because they are not product behavior tests. Drift protection is part of the active quality gate.

'''

====== Scope

This FTR includes:

* define the supported marker vocabulary in pytest configuration
* update existing tests to use the normalized markers where useful
* add or normalize Make targets for the key slices
* update `make help` or equivalent developer help output
* document marker usage in `tests/README.adoc`
* ensure default test execution excludes quarantined historical tests
* keep compatibility with existing CI where practical

'''

====== Non-goals

This FTR does not:

* reorganize the physical test tree
* decide the final fate of pre-hex tests
* rewrite integration tests
* add new formula behavior
* introduce a new test runner
* replace pytest markers with a custom selection mechanism
* redesign the entire CI matrix unless required to preserve existing behavior

Physical topology belongs to `FTR-TEST-SUITE-TOPOLOGY-P3C`. Pre-hex disposition belongs to `FTR-PREHEX-TEST-CLASSIFICATION-P3C`. Broader integration consolidation belongs to `FTR-INTEGRATION-SLICE-CONSOLIDATION-P3C`.

'''

====== Marker placement rules

. A test under `tests/unit/` should normally carry `unit` only if marker-based execution needs it.
. A test under `tests/integration/` should normally carry `integ` only if marker-based execution needs it.
. A test under `tests/architecture/` should normally carry `arch`; use `guardrail` for drift-prevention tests.
. Backend markers such as `xlsx_ir` and `ods` should be added only to tests that specifically exercise that backend path.
. `legacy` and `prehex` markers must exclude tests from the default run.
. `current_state` must not be used for durable invariants.
. Do not add FTR-specific markers unless the project already follows such a convention and the marker is useful for execution.

'''

====== Acceptance criteria

. The supported marker vocabulary is declared in pytest configuration.
. Existing active tests use the normalized markers where marker-based execution is useful.
. `make test` runs the normal active development slice and excludes `legacy`, `prehex`, and intentionally slow tests.
. `make test-unit`, `make test-integ`, and `make test-arch` are available and run the intended slices.
. Backend-specific targets such as `make test-ir` and `make test-ods` are available if corresponding tests exist.
. `make test-prehex` is available only as an explicit quarantined historical slice or is explicitly deferred if the pre-hex suite is not runnable.
. `make help` documents all test-related targets.
. `tests/README.adoc` explains the marker policy and the relationship between physical topology and marker-based execution.
. CI continues to run the intended active test slices without accidentally including pre-hex tests.
. No production behavior changes are introduced.

'''

====== Review guidance

When reviewing marker or Make target changes, reviewers should ask:

* Does this marker correspond to a useful execution slice?
* Is physical test placement still the primary classification mechanism?
* Does the default run include architecture guardrails?
* Are legacy and pre-hex tests excluded deliberately rather than accidentally?
* Does `make help` tell a new contributor which command to run?
* Is this change reducing ambiguity, or adding another naming layer?

'''

====== Rationale

A clean test tree is necessary but not sufficient. Developers also need reliable execution slices.

Markers should be used sparingly and deliberately. Too few markers make targeted execution difficult. Too many markers create a second taxonomy that drifts away from the directory structure.

This FTR establishes a pragmatic middle ground: physical topology expresses test responsibility, while markers and Make targets provide stable ways to run the slices that matter during Phase 4 feature work.
