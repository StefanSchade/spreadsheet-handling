====== Purpose

Preserve the first architecture concept for pipeline step semantics as a
reviewable project-archaeology baseline.

The concept records the intended direction before the public YAML pipeline
surface hardens further.
It is intentionally a design artifact, not a runtime implementation slice.

'''

====== Context

Recent pipeline work added generic transformations such as XRef contraction,
cell-code codecs, compact multiaxis transforms, and explicit lookup helper
enrichment.
Together, these features made the public step surface more powerful but also
more semantically dense.

Several concerns are now close together:

* primitive data transformations
* configuration and heuristic resolution
* composite steps that wrap lower-level primitives
* `_meta` contracts for inverse behavior and diagnostics
* workbook-view decisions about visible, editable, redundant, and canonical
  frames

The first concept draft captures a direction for separating these concerns
before a second version or implementation wave rewrites the design.

'''

====== Deliverables

This FTR adds the concept document:

[source,text]
----
docs/internal_guide/architecture/pipeline_semantics.adoc
----

It also preserves two first-pass reviews under this FTR:

[source,text]
----
docs/internal_guide/backlog/ftrs/done/FTR-PIPELINE-SEMANTICS-CONCEPT-P4_review_claude_code.adoc
docs/internal_guide/backlog/ftrs/done/FTR-PIPELINE-SEMANTICS-CONCEPT-P4_review_codex.adoc
----

'''

====== Scope

The concept covers:

* primitive, configuration, composite, and plugin step categories
* heuristic-to-metadata resolution
* a future pipeline step registry
* inverse and bijectivity categories
* frame lifecycle metadata
* rendering/workbook-view separation
* step naming direction for FK/helper/lookup steps
* implementation slices and proposed follow-up FTRs

'''

====== Non-goals

This FTR does not:

* implement the step registry
* rename runtime pipeline steps
* add configuration steps
* change pipeline execution behavior
* change rendering behavior
* decide the final schema for frame lifecycle metadata

Those items belong to follow-up FTRs after the concept has been revised.

'''

====== Acceptance criteria

. The first pipeline semantics concept is stored in the architecture docs.
. The first review artifacts are retained next to this FTR for future
  archaeology.
. The reviews remain clearly associated with this FTR and the first concept
  draft.
. No production code changes are introduced.
. The repository has a commit boundary before the next concept revision starts.

'''

====== Review notes

The first review pass is intentionally preserved as two separate artifacts:

* `FTR-PIPELINE-SEMANTICS-CONCEPT-P4_review_claude_code.adoc`
* `FTR-PIPELINE-SEMANTICS-CONCEPT-P4_review_codex.adoc`

Together they document the main follow-up risks for the next design iteration:

* complete classification of existing runtime steps
* relationship between JSON step registry and runtime `REGISTRY`
* alignment between step registry contracts and `meta_registry.yaml`
* composite-step provenance for internal primitive metadata and temporary
  frames
* precedence between inline YAML parameters and resolved `_meta`
* lifecycle handling for workbook-view plugins
* registry-backed migration contracts for step renames

'''

====== Rationale

This FTR exists mainly to create a deliberate archival boundary.

The first concept is useful even if the next version changes substantially:
it records why the project started separating configuration, transformation,
composition, inverse semantics, and frame lifecycle.
Keeping the reviews near the FTR makes the first design conversation
discoverable without leaving transient review files in the active architecture
folder.
