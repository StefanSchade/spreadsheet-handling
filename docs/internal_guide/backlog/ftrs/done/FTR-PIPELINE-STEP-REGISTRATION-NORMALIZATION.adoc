===== FTR-PIPELINE-STEP-REGISTRATION-NORMALIZATION -- Normalize registry binding for domain-addressable steps

*Status:* Implemented
*Epic:* Phase 4a -- First feature wave on top of the new architecture

*Purpose:*::
Normalize how YAML step names are resolved into executable `BoundStep`s so that
the pipeline registry can point to generic binding behavior plus a domain target
instead of requiring bespoke pipeline factory code for every ordinary step.

*Context:*::
`run_pipeline()` is already close to the intended architecture: it executes a
linear list of steps and does not inspect payload semantics.

The current weakness is one level earlier in the pipeline seam:

* `REGISTRY` maps most step names directly to per-step factories
* `build_steps_from_config()` therefore resolves user-facing step names into
  bespoke binding code rather than into a more generic registration model

Because no generic binding path exists, every domain-addressable step must
either embed its domain call inline in the pipeline layer or build a bespoke
wrapper there.  The result is that thematically domain-owned behavior
accumulates in `pipeline.steps` not by design but because there is no better
place to put it yet.

This is the root cause of both problems visible today: steps like
`flatten_headers` are thematically wrong in `pipeline` even though they look
short, and factories like `make_apply_fks_step` have grown fat with domain
logic precisely because the generic delegation path was missing.

Establishing the generic path is the prerequisite for both the FK-helper
extraction and the later architecture fitness guards.  Once the generic
mechanism exists, domain-owned steps can be migrated to it, and fat factories
can be extracted to domain one at a time without leaving a worse mess behind.

Note that fat factories are not a reason to delay this ticket.  The extraction
of fat factories is a mechanical operation: move the domain logic to a domain
function, then the factory becomes a thin delegate.  Whether the newly
extracted domain code then needs further cleanup is a separate question.

The target model is described in
<<concept-pipeline-step-registration,Pipeline Step Registration Concept>>.

*Goal:*::
Introduce a new normal path for pipeline-addressable domain steps:

* registry stays factory-based
* registry entries may additionally declare the intended target callable
* a small generic binding factory handles ordinary domain step shapes
* bespoke legacy factories remain temporarily allowed where needed

*Scope:*::
- Introduce a `StepRegistration` descriptor or equivalent registration shape in
  the registry layer.
- Allow registry entries to carry a binding factory plus an optional target
  callable or target reference.
- Introduce one or more small generic binding factories for the common internal
  domain step shapes already present in the codebase.
- Keep YAML step syntax unchanged.
- Keep `run_pipeline()` unchanged in behavior.
- Migrate the registry entries whose thematic home is clearly domain, starting
  with `flatten_headers`, `unflatten_headers`, `reorder_helpers`,
  `bootstrap_meta`, `apply_overrides`, and `add_validations`.  The criterion
  for migration is thematic ownership, not line count.
- Keep `apply_fks`, `drop_helpers`, `validate`, `check_fk_helpers`, and
  `plugin` working even if they remain legacy for now.  `apply_fks` and
  `drop_helpers` are thematically wrong in `pipeline` too, but their
  migration requires extracting domain logic first; that is the job of
  `FTR-FK-HELPER-DOMAIN-EXTRACTION` and is intentionally out of scope here.

*Non-Goals:*::
- No routing, branching, DAG execution, or process-engine features.
- No payload-driven orchestration decisions.
- No YAML syntax redesign.
- No requirement to migrate every existing step in this ticket.
- No forced unification of all future step kinds to one permanent callable
  signature.
- No immediate removal of the plugin step or its dedicated dotted-callable
  behavior.

*Solution Direction:*::
- Keep the registry as the extension point for allowed step names.
- Replace the current "registry value is only a bare factory callable" model
  with a small descriptor or equivalent normalized registration structure.
- Make the normalized registration support at least `factory` and optional
  `target`.
- Introduce a generic binding path for ordinary domain-addressable steps.
- The first slice should support at least the two dominant internal callable
  shapes: builder-style `(...config) -> Step` and frames-first
  `(frames, **config) -> Frames`.
- Prefer direct callable references for internal registrations, but allow a
  target reference form that can later support lazy resolution if useful.
- Migrate the step registrations whose thematic home is clearly domain, starting
  with those already listed under Scope.
- Leave `apply_fks`, `drop_helpers`, and validation-specific orchestration to
  later follow-up tickets; their extraction is a separate mechanical step that
  builds on the generic binding path this ticket establishes.

*Implementation Direction:*::
- Add a small registration type such as `StepRegistration` or an equivalent
  mapping representation in the registry layer.
- Update `build_steps_from_config()` so it can consume both normalized
  descriptor entries and temporary legacy bare factory entries.
- Add a generic binder in the pipeline layer for ordinary domain step targets.
- Move migrated simple steps so their registry entry names the domain target
  explicitly instead of hiding it behind bespoke per-step pipeline code.
- Keep the change incremental: old and new registration forms may coexist during
  the migration.

*Acceptance:*::
- YAML pipeline files remain valid without syntax changes.
- `run_pipeline()` remains a simple linear step runner with no new orchestration
  semantics.
- The registry supports a normalized descriptor form that can express "factory
  plus optional target".
- The simple migrated step names no longer require bespoke registry entries that
  exist only to bind one domain function.
- The migrated simple registry entries expose their intended domain target
  explicitly.
- `apply_fks`, `drop_helpers`, `validate`, `check_fk_helpers`, and `plugin`
  continue to work even if they remain legacy in this slice.
- The change is covered by focused tests in the registry/building layer and by a
  small set of pipeline smoke tests for migrated steps.

*Suggested Test Areas:*::
- a focused registry test module such as
  `tests/unit/pipeline/test_step_registration.py`
- existing pipeline smoke/config tests:
  - `tests/unit/pipeline/test_pipeline_basic.py`
  - `tests/unit/pipeline/test_pipeline_yaml.py`

*Relation to Existing Work:*::
- This ticket prepares the ground for
  `FTR-FK-HELPER-DOMAIN-EXTRACTION`.
- It also gives the later
  `FTR-ARCHITECTURE-FITNESS-GUARDS-P4X`
  a cleaner target architecture to protect.
- It does not replace the helper-extraction ticket; it makes the pipeline seam
  cleaner before that extraction happens.
