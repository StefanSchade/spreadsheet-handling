===== FTR-STRUCTURED-YAML-WRITER-P4 -- Structured YAML writer for generated config trees

*Status:* Done
*Epic:* Phase 4 -- Feature Expansion and Domain Transformations
*Priority:* High

*Discovered-in:*

* `ba-config` Elmar/NXT spreadsheet-flow prototype

*Related-to:*

* <<FTR-COMPACT-MULTIAXIS#,FTR-COMPACT-MULTIAXIS>>
* <<FTR-COMPACT-TRANSFORM-API-ERGONOMICS-P4#,FTR-COMPACT-TRANSFORM-API-ERGONOMICS-P4>>
* <<FTR-JSON-BUNDLER#,FTR-JSON-BUNDLER>>

'''

====== Purpose

Add a deterministic, declarative writer for nested YAML configuration trees.

The current YAML backend can persist frames as flat list-of-records files. That
is useful for canonical tabular exchange, but it is not enough for generated
application configuration such as `nxt-config`, where the target shape is a
directory tree of nested mappings and lists.

Consumer repositories should not need long-lived domain plugins whose main job
is writing structured YAML side effects.

'''

====== Context

The `ba-config` prototype models the business-owned spreadsheet as canonical
relations and then generates files below `nxt-config-generated/`, for example:

* `variable/variable-registry.yml`
* `variable/business-component-registry.yml`
* `calculation/input-variables.yml`
* `calculation/mappings/request-mappings.yml`
* `calculation/mappings/response-mappings.yml`
* `calculation/calculation-registry.yml`
* `calculation/calculation-operations.yml`

The generated files are not flat frame dumps. They contain nested maps, ordered
lists, grouped entries, optional fields, and stable file paths.

The prototype can keep a repository-local writer for the meeting/demo slice, but
the capability is generic enough to belong in the main library once the target
shape has been validated with the business model.

'''

====== Problem statement

Without a structured writer, consumers have two weak options:

* flatten generated YAML into shapes that do not match the owning application
* write repository-local side-effect plugins for every generated config tree

The first option makes the generated output less useful. The second option
duplicates plumbing, makes behavior harder to test consistently, and hides
important output semantics outside the pipeline contract.

'''

====== Goals

. Provide a declarative way to write nested YAML files from one or more frames.
. Keep output deterministic: stable paths, stable key ordering, stable list
  ordering, and explicit handling of missing optional fields.
. Support mapping-shaped and list-shaped YAML roots.
. Support grouping rows by one or more configured keys.
. Support nested field paths such as `request.nominaldaten.darlehensart` without
  guessing structure from labels or names.
. Support explicit omission rules for empty optional values.
. Return or persist a generated-file report frame for CI visibility.
. Make path handling safe and explicit.

'''

====== Design principles

* Configuration over inference.
* No heuristic nesting from column labels, display labels, or name similarity.
* No implicit relation reconstruction.
* Missing required fields fail clearly.
* Empty optional values are omitted only when the writer configuration says so.
* Generated output should be reproducible from the same input frames and config.

'''

====== Candidate configuration shape

The exact API can still change, but the contract should feel like a normal
pipeline writer rather than an arbitrary Python callback.

[source,yaml]
----
- step: write_structured_yaml
  output_dir: ./nxt-config-generated
  files:
    - path: variable/variable-registry.yml
      frame: variables
      root: mapping
      key: variable_id
      value:
        label: value_label_de
        type: data_type
        businessComponent: business_component
      omit_empty: [businessComponent]

    - path: calculation/mappings/request-mappings.yml
      frame: request_mappings
      root: mapping
      key: transaction_type_id
      sequence:
        sort_by: [sort_key]
        value:
          source: source_path
          target: target_path
          variable: variable_id
----

'''

====== Acceptance criteria

. A pipeline can generate a nested YAML directory tree from configured frames.
. The writer supports at least one mapping-root file and one list-root or
   grouped-sequence file.
. Generated files are deterministic across repeated runs.
. Optional empty values can be omitted explicitly.
. Missing required fields produce actionable diagnostics.
. The writer records which files were generated.
. Output paths cannot escape the configured output directory.
. The implementation includes a demo-shaped test based on the NXT config
   generation use case.
. Existing flat YAML backend behavior remains unchanged.

'''

====== Implementation notes

Delivered as the public pipeline step `write_structured_yaml`.

Implemented behavior:

* mapping-root YAML files keyed by one or more configured frame columns
* grouped-sequence files using `sequence.value` and optional `sequence.sort_by`
* list-root YAML files using explicit row value mappings
* dotted output paths for nested YAML mappings
* deterministic ordering via `sort_by`, mapping keys, and stable row tiebreakers;
  typed numeric sort values are ordered numerically
* explicit `omit_empty` handling for optional output paths
* required-value diagnostics for empty required fields
* safe relative output paths that cannot escape `output_dir`, with duplicate
  output paths rejected before writing
* generated-file report frame, defaulting to `structured_yaml_files`
* JSON pipeline step registry entry and runtime `REGISTRY` binding
* focused tests for NXT-shaped mapping/grouped-sequence output, list output,
  determinism, composite keys, numeric sort keys, duplicate paths, path safety,
  missing values, and missing columns

The existing flat YAML backend remains unchanged and separate from this writer.

'''

====== Non-goals

This FTR does not:

* infer nested output structure from visible spreadsheet labels
* replace the flat YAML frame backend
* implement a full templating language
* implement arbitrary Python expressions in YAML configuration
* solve dense-axis reconstruction
* require the `ba-config` prototype to remove its local writer before the
  business model has settled

'''

====== Review guidance

When reviewing this FTR, ask:

* Is the writer generic enough to remove repeated consumer plumbing?
* Is the output shape still explicit and reviewable?
* Can a generated file be traced back to configured frames and fields?
* Does the design avoid hiding domain-specific extraction logic in the generic
  library?
* Are path safety and deterministic ordering covered by tests?
