===== FTR-EXPLICIT-HELPER-LOOKUP-POLICY-P4 -- Explicit helper lookup and helper-field policy

*Status:* Implemented
*Epic:* Phase 4 -- Feature Expansion and Domain Transformations
*Priority:* High

*Discovered-in:*

* `ba-config` Elmar/NXT spreadsheet-flow prototype

*Related-to:*

* <<FTR-MULTI-HELPER-COLUMN#,FTR-MULTI-HELPER-COLUMN>>
* <<FTR-FK-HELPER-PROVENANCE-CLEANUP#,FTR-FK-HELPER-PROVENANCE-CLEANUP>>
* <<FTR-FK-HELPER-DOMAIN-EXTRACTION#,FTR-FK-HELPER-DOMAIN-EXTRACTION>>

'''

====== Purpose

Add a generic, explicit helper lookup/enrichment transform and a small helper
field policy model.

The existing FK-helper feature supports multiple helper fields and derived
provenance, but it is still centered on FK column detection by naming convention,
for example `id_(Products)`.

The `ba-config` prototype needs a more explicit projection:

* source relation: `variable_usage_matrix_raw`
* lookup table: `variables`
* join key: `ID`
* helper fields: `sort_key`, `value_label_de`, `business_component`,
  `data_type`
* output relation: `variable_usage_matrix`

That is not domain-specific logic. It is a generic table lookup operation with
helper-field governance.

'''

====== Current behavior

The current FK-helper implementation already supports:

* configurable multiple helper columns
* selection precedence via `helper_fields_by_fk`, `helper_fields_by_target`,
  global `helper_fields`, and fallback `label_field`
* derived helper provenance for materialized helper columns
* helper cleanup using provenance with prefix fallback
* domain ownership of FK-helper behavior after extraction from `pipeline.steps`

This is useful for classic spreadsheet FK columns where the reference is encoded
in the source column name.

'''

====== Gap

The current implementation does not provide:

* a declarative lookup/enrichment step that joins arbitrary frames by explicit
  keys
* a canonical allowlist of helper fields per lookup table
* a default helper-field list per lookup table
* validation that explicitly requested helpers are allowed for that lookup table
* helper enrichment that does not depend on column names like `id_(Target)`
* deterministic output column ordering for explicitly projected helper columns

Without this, consumer repositories need small local plugins for simple helper
projection. The plugin itself is often only a `merge`, but the repeated local
code is a signal that the generic library is missing the right primitive.

'''

====== Goals

. Introduce a generic pipeline step for explicit lookup enrichment.
. Allow a source frame to be enriched from a lookup frame through configured
   join keys.
. Support requested helper fields, lookup-table defaults, and lookup-table
   allowlists.
. Enforce that requested helper fields are allowed when an allowlist is present.
. Preserve deterministic column ordering.
. Record derived helper provenance for materialized helper columns where useful.
. Keep the current FK-helper path working unchanged.
. Avoid heuristic inference from labels, positions, or similar names.

'''

====== Candidate configuration

[source,yaml]
----
- step: enrich_lookup
  source: variable_usage_matrix_raw
  lookup: variables
  output: variable_usage_matrix
  "on": ID
  helpers:
    fields: [sort_key, value_label_de, business_component, data_type]
    allowed: [sort_key, value_label_de, business_component, data_type, module]
    default: [value_label_de]
  order:
    helper_position: before_key       # before_key | after_data (default)
    sort_by: [sort_key]
  missing: fail                       # fail (default) | empty
----

For repeated use, helper policy may also be declared once in metadata or
pipeline defaults:

[source,yaml]
----
helper_policies:
  variables:
    key: ID
    allowed_helpers: [sort_key, value_label_de, business_component, data_type, module]
    default_helpers: [value_label_de]
----

Then a step can request defaults:

[source,yaml]
----
- step: enrich_lookup
  source: variable_usage_matrix_raw
  lookup: variables
  output: variable_usage_matrix
  "on": ID
  helpers: default
----

Multi-key joins are supported by passing a list to `on`:

[source,yaml]
----
"on": [region, product_id]
----

'''

====== Behavioral contracts

*Duplicate lookup keys:* Rejected. If the lookup frame contains duplicate rows
for the join key(s), the step raises a `ValueError`. Helper projection must not
silently multiply source rows.

*Column conflicts:* Rejected. If a requested helper field already exists as a
column in the source frame (excluding join key columns), the step raises a
`ValueError`. This prevents silent shadowing of lookup values.

*Missing matches (``missing`` parameter):*

* `fail` (default) -- raises `ValueError` when any source row has no match in
  the lookup table. This is the safe default because a missing match usually
  indicates a broken reference.
* `empty` -- fills unmatched helper values with `""`. Useful for demo or
  tolerant workflows.

*Invalid configuration:* Unrecognized `helper_position` values and missing
`sort_by` columns are treated as configuration errors (`ValueError`).

*Helper provenance:* Materialized helper columns are recorded in
`_meta["derived"]["sheets"][<output>]["enrich_lookup"]` with `lookup`, `on`,
and `helper_columns` keys.

'''

====== Helper field selection

The intended precedence is:

. Explicit `helpers.fields` on the step.
. Lookup-table `default_helpers`.
. All `allowed_helpers`, only when the step explicitly opts into that behavior.
. No implicit helper projection.

If `allowed_helpers` is present, every requested helper must be listed there.
An unknown or disallowed helper field is a configuration error.

'''

====== Non-goals

This FTR does not:

* remove the existing FK-helper naming-convention path
* infer references from visible labels or similar column names
* implement formula-driven helper values
* implement Excel dropdowns or data-validation lists
* solve reverse synchronization from edited helper columns back into lookup
  tables
* introduce dense-axis reconstruction

'''

====== Acceptance criteria

. A pipeline can enrich one frame from another via explicit `source`, `lookup`,
   `on`, and `helpers` configuration.
. The transform supports single-column and multi-column key joins.
. Requested helper fields are copied from the lookup frame in configured order.
. Output column ordering is deterministic and configurable enough for workbook
   helper views.
. A helper allowlist rejects disallowed requested fields.
. A helper default list can be used without repeating the field list in every
   step.
. Duplicate lookup keys are rejected.
. Column conflicts between source and requested helpers are rejected.
. Unmatched source rows raise by default (`missing: fail`); `missing: empty`
   fills with `""`.
. Invalid `helper_position` values and missing `sort_by` columns are rejected.
. Helper provenance is written to `_meta`.
. The behavior is covered by domain-level and pipeline-level tests.
. Existing FK-helper tests remain green.
. The `ba-config` local enrichment plugin can be replaced by the generic step in
   a follow-up integration change.

'''

====== Review guidance

When reviewing this FTR, ask:

* Does this remove a real consumer-plugin pattern rather than add an alias?
* Is helper selection explicit and governed by allowlists/defaults?
* Does the feature stay deterministic?
* Does it avoid reviving heuristic FK detection under a different name?
* Can the old FK-helper path and this explicit lookup path coexist cleanly?
