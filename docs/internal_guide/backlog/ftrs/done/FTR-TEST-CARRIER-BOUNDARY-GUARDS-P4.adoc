= FTR-TEST-CARRIER-BOUNDARY-GUARDS-P4

== Status

Implemented

== Context

With increasing test-suite modularization and clearer test topology, test
carriers such as directories, markers, and Make targets can start to carry
implicit semantics.

That creates a structural risk:

* test types are mixed within one carrier
* support directories silently become active test roots
* path-owned marker policy is bypassed by manual marker declarations
* physical test layout changes unexpectedly affect selective test runs

This FTR covers the **test-carrier boundary**. It does not cover spreadsheet
metadata, `meta_canonical`, `meta_rendering`, carrier hints, or persisted
`_meta`; those remain in `FTR-META-CARRIER-BOUNDARY-GUARDS-P4`.

== Problem

Without explicit test-carrier boundary guards, topology drift is gradual and
hard to notice:

* tests can appear outside known active roots
* helper or data carriers can start collecting pytest tests
* manually written topology markers can conflict with path-owned markers
* selective marker-based runs become harder to reason about

== Goal

Introduce clear, checkable guards for test carriers so that:

* active test roots stay explicit
* support carriers stay non-collecting
* path-owned topology markers remain path-owned
* topology drift fails early and deterministically

== Non-Goals

* No new test types
* No changes to production behavior
* No broad test refactoring
* No spreadsheet meta/carrier semantic guards

== Approach

=== 1. Define carrier roots

Active carriers:

* `tests/unit/`
* `tests/integration/`
* `tests/architecture/`
* `tests/legacy_pre_hex/`

Support carriers:

* `tests/utils/`
* `tests/data/`
* `tests/experimental/`

`tests/experimental/` is temporary parking only. It must not contain collected
tests and should not be used as a long-term archive.

=== 2. Enforce boundary rules

Initial rules:

* pytest-style test files must live under known active carrier roots
* support carriers must not define pytest-collected test symbols
* topology markers such as `unit`, `integ`, `arch`, `current_state`, `legacy`,
  and `prehex` must not be written manually in test modules

=== 3. Technical guards

The initial implementation lives in
`tests/architecture/carrier_boundary/test_carrier_boundary_guards.py`.

It uses static file and AST checks. The marker guard intentionally catches the
normal explicit form, such as `pytest.mark.unit`; it is not a full Python
interpreter for dynamic marker construction.

== Deliverables

* documented test-carrier roots
* documented support-carrier policy
* implemented architecture guard tests
* README guidance for placement and marker ownership

== Risks

* Over-constraining scratch or migration work
* Misclassifying existing tests during topology cleanup
* Treating `tests/experimental/` as a long-term archive

These risks are acceptable if support-carrier exceptions remain explicit and
temporary.

== Dependencies

* FTR-TEST-SUITE-TOPOLOGY-P3C
* FTR-ARCHITECTURE-TEST-LAYER-P3C
* FTR-TEST-MARKERS-AND-MAKE-TARGETS-P3C
* FTR-GUARDRAIL-DECISION-RECORD-P4
* FTR-TEST-ARCHIVE-AND-QUARANTINE-RULES-P3C

== Acceptance Criteria

* Active and support test carriers are documented
* At least 3 central test-carrier boundary rules are technically enforced
* Violations are visible and reproducible in architecture test runs
* Documentation is sufficient to place new tests correctly
