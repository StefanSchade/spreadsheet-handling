'''

====== Purpose

Normalize unintuitive public pipeline step names before controlled consumer repositories and examples harden around the current YAML surface.

The immediate focus is the FK/helper/lookup step cluster, because this is where current naming most strongly obscures the distinction between FK-convention helper materialization, helper validation, helper reordering, helper removal, and explicit lookup-helper projection.

This FTR should make the public step names match the pipeline semantics concept and the reviewed step registry.

'''

====== Context

`FTR-PIPELINE-SEMANTICS-CONCEPT-P4` defines the target distinction between primitive, configuration, composite, plugin, and infrastructure steps.

`FTR-PIPELINE-STEP-REGISTRY-P4` introduces a JSON step registry as the reviewed inventory of the public pipeline surface.

The registry makes naming inconsistencies visible. This FTR fixes the most harmful names while the known consumer repositories are still controlled and can be updated together.

The current FK/helper/lookup names are historically grown:

* `apply_fks` sounds like a generic FK operation, but it materializes FK helper columns.
* `drop_helpers` sounds dangerously broad, but should mean removing derived FK helper columns.
* `check_fk_helpers` uses informal "check" terminology rather than validation terminology.
* `reorder_helpers` sounds generic, but currently means reordering FK helper columns next to FK columns.
* `enrich_lookup` is grammatically ambiguous and does not clearly say that helper columns are added to a source frame from a lookup frame.

The next beta should not publish these unclear names as the recommended YAML surface if they can still be changed cheaply.

'''

====== Problem statement

The current step names make it harder to understand and teach the framework:

* They blur the difference between FK-convention helpers and explicit lookup-helper projection.
* They hide whether a step adds, removes, validates, or reorders helper columns.
* They are less suitable for generated user documentation.
* They force users to infer intent from implementation details or examples.
* They risk becoming permanent compatibility debt if published in a beta and copied into consumers.

Because the currently known consumer repositories are under project control, delaying the rename would increase cost without buying meaningful compatibility.

'''

====== Goals

. Rename the current FK/helper/lookup pipeline steps to clearer public names.
. Keep the new names aligned with the step categories in the pipeline semantics concept.
. Update the runtime pipeline registry.
. Update the JSON step registry introduced by `FTR-PIPELINE-STEP-REGISTRY-P4`.
. Update tests, examples, demo configs, and controlled consumer configs.
. Decide explicitly whether old names are removed immediately or retained as short-lived aliases.
. If aliases are retained, make their lifecycle machine-readable in the step registry.
. Avoid behavior changes except for the public step name surface and alias handling.

'''

====== Scope

This FTR includes the following public step renames:

[cols="1,1,4"]
|===
| Current name | New name | Rationale

| `apply_fks`
| `add_fk_helpers`
| The step materializes helper columns derived from FK conventions. The new name states the visible effect.

| `drop_helpers`
| `remove_fk_helpers`
| The step removes FK helper columns, preferably using provenance or explicit policy. The new name avoids the overly broad `helpers` term.

| `check_fk_helpers`
| `validate_fk_helpers`
| The step is validation-only. The new name aligns with validation terminology used elsewhere.

| `reorder_helpers`
| `reorder_fk_helpers`
| The current behavior is FK-helper specific rather than a generic helper-column operation.

| `enrich_lookup`
| `add_lookup_helpers`
| The step adds helper columns from a lookup frame to a source frame. The new name avoids the ambiguous verb-object shape of `enrich_lookup`.
|===

This FTR includes updates to:

* runtime pipeline step registry
* pipeline step JSON registry
* unit and integration tests
* architecture tests that refer to step names
* documentation examples
* demo repository pipeline YAML, if updated in the same workspace wave
* company / consumer repository pipeline YAML, where under current project control

'''

====== Non-goals

This FTR does not:

* change transformation behavior
* introduce new configuration steps
* change FK detection heuristics
* change lookup-helper policy semantics
* implement frame lifecycle behavior
* make the JSON step registry the runtime source of truth
* rename unrelated steps unless an implementation-time review finds a trivial and clearly beneficial correction
* remove plugin support

Configuration-producing steps belong to `FTR-LOOKUP-FK-CONFIGURATION-STEPS-P4`.

Frame lifecycle behavior belongs to `FTR-FRAME-LIFECYCLE-AND-WORKBOOK-VIEWS-P4`.

Deeper semantics guards belong to `FTR-PIPELINE-SEMANTICS-GUARDS-P4`.

'''

====== Alias strategy

Preferred strategy:

[source,text]
----
Hard rename the public YAML names now.
Update controlled consumers in the same integration wave.
Do not carry long-lived aliases.
----

Rationale:

* The known consumer repositories are controlled.
* Integration is still early.
* The next beta should expose the cleaned names.
* Long-lived aliases make documentation and registry guards noisier.

Fallback strategy if immediate consumer update is impractical:

[source,text]
----
Keep old names as one-beta aliases.
Emit an explicit deprecation warning when old names are used.
Represent the alias lifecycle in the step registry.
Remove aliases no later than the next beta after the cleaned YAML surface is published.
----

Alias registry entries must be structured, not plain strings.

Example:

[source,json]
----
{
  "name": "add_lookup_helpers",
  "runtime_name": "add_lookup_helpers",
  "aliases": [
    {
      "name": "enrich_lookup",
      "status": "deprecated_alias",
      "deprecated_since": "0.1.0b-next",
      "warn_until": "0.1.0b-next",
      "remove_after": "0.1.0b-next"
    }
  ]
}
----

If the hard rename strategy is chosen, the registry may still record `replaces` / `replaced_by` for project archaeology, but runtime aliases are not required.

'''

====== Runtime behavior

The new names should bind to the same implementation behavior as the old names.

Expected mapping:

[cols="1,1,3"]
|===
| New name | Existing implementation area | Expected behavior

| `add_fk_helpers`
| FK helper materialization
| Same behavior as current `apply_fks`.

| `remove_fk_helpers`
| FK helper removal / provenance-based cleanup
| Same behavior as current `drop_helpers`.

| `validate_fk_helpers`
| FK helper consistency validation
| Same behavior as current `check_fk_helpers`.

| `reorder_fk_helpers`
| helper column ordering transformation
| Same behavior as current `reorder_helpers`.

| `add_lookup_helpers`
| explicit lookup-helper projection
| Same behavior as current `enrich_lookup`.
|===

Any accidental behavior change should be treated as a bug unless explicitly called out and approved.

'''

====== Registry updates

The pipeline step registry should reflect the new names as the public names.

For each renamed step, the registry should include:

* `name`: new public name
* `runtime_name`: new runtime name
* category and subtype
* prior name in `replaces` or structured alias metadata
* unchanged parameter contract unless the implementation intentionally cleans up obvious wording only
* unchanged frame contract
* unchanged inverse classification
* FTR reference to this FTR

Example direction:

[source,json]
----
{
  "name": "add_lookup_helpers",
  "runtime_name": "add_lookup_helpers",
  "category": "primitive",
  "subtype": "lookup_projection",
  "status": "public",
  "replaces": ["enrich_lookup"],
  "ftr": "FTR-PIPELINE-STEP-NAMING-P4"
}
----

The registry should not list both old and new names as independent public steps unless aliases are intentionally supported at runtime.

'''

====== Documentation updates

All maintained examples should use only the new step names.

Update at least:

* internal guide examples
* user-facing examples if present
* demo pipeline YAML
* test fixture YAML
* README snippets if they mention old step names
* generated or hand-written pipeline step references if already present

Avoid documenting old names except in a short migration note if aliases are retained.

Migration note example:

[source,text]
----
`enrich_lookup` was renamed to `add_lookup_helpers` before the next beta to align the YAML surface with the pipeline step semantics model.
----

'''

====== Consumer repository updates

Known controlled consumers should be updated in the same integration wave where practical.

Expected controlled consumers:

* `spreadsheet-handling-demo`
* current company / application repository using the core package

The implementation should either:

* update those consumers directly in companion commits, or
* record explicit handover commands / patches if they are outside the current workspace

At minimum, run or document smoke checks for the maintained consumer pipelines that use renamed steps.

'''

====== Tests

Tests should verify the public YAML surface after renaming.

Required tests:

. New step names resolve through the runtime registry.
. Old names do not appear in maintained example YAML after the migration.
. Pipeline smoke tests use the new names.
. Step registry completeness still passes after renaming.
. Registry alias lifecycle guard passes if aliases are retained.

If the hard rename strategy is chosen:

* Add a negative test that an old name fails with a clear error, if the current registry behavior supports useful error assertions.

If the one-beta alias strategy is chosen:

* Add tests that old names still run.
* Add tests that old names emit deprecation warnings.
* Add tests that alias lifecycle fields are present in the JSON registry.

'''

====== Suggested implementation direction

. Decide hard rename versus one-beta aliases.
. Rename runtime registry keys.
. Keep implementation function names unchanged unless a local rename is safe and clearly improves readability.
. Update the step registry public names.
. Update tests and fixtures.
. Grep the repo for old names and classify remaining occurrences as migration notes, review artifacts, or unintended leftovers.
. Update controlled consumer YAML.
. Run focused pipeline and registry tests.

The implementation should prefer whole-step registry updates over ad hoc compatibility glue.

'''

====== Acceptance criteria

. The public runtime step names use the normalized FK/helper/lookup names.
. Maintained pipeline examples use `add_fk_helpers`, `remove_fk_helpers`, `validate_fk_helpers`, `reorder_fk_helpers`, and `add_lookup_helpers` where applicable.
. The JSON step registry uses the normalized names as canonical public names.
. Old names are either removed or represented only as release-bounded aliases.
. Alias lifecycle is machine-readable if aliases exist.
. No behavior changes are introduced beyond step-name resolution and alias/deprecation behavior.
. Tests cover the new names.
. Registry completeness tests pass.
. Controlled consumer pipelines are updated or explicit update instructions are recorded.
. Existing active tests continue to pass.

'''

====== Review guidance

When reviewing this FTR, check:

* Are the new names actually clearer for a user reading YAML?
* Do the names distinguish FK-convention helpers from explicit lookup helpers?
* Are old names avoided in maintained examples?
* Is alias handling deliberately chosen rather than accidental?
* Does the step registry treat old names as aliases/replacements, not independent concepts?
* Did any behavior change sneak in under the rename?
* Are consumer YAML files updated before the next beta baseline?

'''

====== Rationale

Public YAML step names are product surface.

The project still controls the known consumers, so unclear names should be fixed now rather than defended later with compatibility aliases, migration notes, and duplicated documentation.

The proposed names are deliberately plain:

* `add_*` for materialization/projection
* `remove_*` for removal
* `validate_*` for validation
* `reorder_*` for ordering
* `fk_helpers` for FK-convention helpers
* `lookup_helpers` for explicit lookup projections

This naming does not solve configuration policy or lifecycle by itself, but it removes avoidable ambiguity before those follow-up slices build on the public step surface.

'''
