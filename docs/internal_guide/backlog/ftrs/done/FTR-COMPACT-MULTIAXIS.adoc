===== FTR-COMPACT-MULTIAXIS -- Multi-Axis Compact Representation

*Status:* Implemented (first generic slice)

*Purpose:*::
Compose cross-table expansion and cell-code handling into a generic explicit
representation for compact multi-axis spreadsheets.

*Context:*::
Some spreadsheet views combine several structural ideas at once:

* a row axis, for example features or configuration items
* a column axis, for example product variants grouped by product family
* compact cell codes, for example `E-R-K` or `A-V-Z`
* legend/codebook blocks that explain and optionally validate those codes

The repository example should stay neutral, using products, markets, product
variants, or configuration items rather than private domain vocabulary.

*Scope:*::

* Combine cross-table expansion with cell-code validation or token decoding.
* Produce an explicit generic long form.
* Build on top of `FTR-XREF-CROSSTABLE` and `FTR-CELL-CODEC`.
* Keep domain interpretation outside this generic feature.
* Keep sparse handling deterministic and explicitly configured.

*First Explicit Output Form:*::

The first explicit form should be generic long form:

[source,text]
----
row_key | column_key | code | group?
----

Where:

* `row_key` comes from the row-axis entity.
* `column_key` comes from the column-axis entity.
* `code` is either the whole-cell code or one decoded token.
* `group` is optional metadata from a legend/codebook or codec config.

Later application layers may derive richer domain-specific tables from this
generic long form.

*Non-Goals:*::

* No assumption about business meaning.
* No fixed domain schema.
* No hardcoded encoding strategy.
* No private/professional use-case vocabulary in repo fixtures.
* No heuristic reconstruction of dense axes from labels, positions, or names.

*Solution:*::

* Provide a composed pipeline step that chains:
  . XREF cross-table expansion
  . cell-code validation or token decoding
  . optional legend/codebook lookup for group/category metadata
* Allow configuration of how token/code groups map to abstract dimensions.
* Preserve sparse handling: empty cells produce no explicit rows by default.

Sparse-by-default output is intentional: empty compact cells disappear from the
explicit relation unless configured otherwise. That is strict-lossless only when
empty cells are irrelevant for the caller's semantics, or when a later explicit
dense-axis reconstruction step rebuilds the matrix shape from configured axis
sources.

*First Implementation Slice:*::

The first implementation provides two frames-first pipeline steps:

[source,yaml]
----
pipeline:
  - step: expand_compact_multiaxis
    matrix: product_matrix
    output: feature_product_codes
    row_keys: [feature_id]
    value_columns: [P-001, P-002]
    mode: whole_cell_code
    allowed_from_legend: status_codes
    group: group

  - step: contract_compact_multiaxis
    relation: feature_product_codes
    output: product_matrix_roundtrip
    row_keys: [feature_id]
    mode: whole_cell_code
    allowed_from_legend: status_codes
----

The expanded explicit relation uses the generic shape:

[source,text]
----
feature_id | column_key | code | group?
----

`expand_compact_multiaxis` composes `expand_xref` and `decode_cell_values`.
`contract_compact_multiaxis` composes `encode_cell_values` and `contract_xref`.
+
The first slice supports flat row and column labels only. It inherits the
`whole_cell_code` / `split_tokens` modes from `CELL-CODEC`.
+
`drop_empty: true` is the default for the composed expansion, so empty compact
cells produce no explicit rows. Use `drop_empty: false` when empty cells must
remain explicit for strict matrix-shape roundtrips without later reconstruction.
+
Optional `group` output is a projection from legend entry `group` values or
explicit `code_groups`; it is metadata, not business semantics.
+
`code_groups` keys should be string-like compact codes/tokens. Persisted
metadata normalizes them through the same strip/case policy as lookup.
+
The implementation writes composed transform metadata under `compact_multiaxis`
and continues to write the underlying `xref_crosstable` and `cell_codecs`
metadata for the delegated primitives.

*Dependencies:*::

* <<FTR-XREF-CROSSTABLE#,FTR-XREF-CROSSTABLE>>
* <<FTR-CELL-CODEC#,FTR-CELL-CODEC>>
* <<FTR-LEGEND-BLOCKS#,FTR-LEGEND-BLOCKS>>

*Configuration (Meta):*::

* `axes`: list of axis definitions
* `codec`: reference to the cell-codec configuration
* `crosstab`: reference to the cross-table configuration
* optional legend/codebook reference
* optional token/code categories
* optional constraints on valid combinations

Future configuration may include an explicit dense-axis reconstruction contract:

[source,yaml]
----
dense_axes:
  rows_from:
    frame: Products
    key: product_id
  columns_from:
    frame: Markets
    key: market_id
----

The meaning would be: the relation may be sparse, but contraction can recreate
the complete matrix from configured row and column sources, filling missing
combinations with `fill_value`. This contract must be explicit or come from
persisted contract metadata. It must not be inferred from visible labels,
spreadsheet positions, or naming conventions.

This FTR does not start a `dense_axes` implementation. It only reserves the
architectural direction for a later, separately scoped mode.
+
If `dense_axes` is supplied to the first-slice implementation, the step raises a
clear "not implemented" error instead of attempting partial reconstruction.

*Acceptance:*::

* Given a matrix where columns represent one dimension and cell values encode
  another, the step produces generic long form.
* Whole-cell codes can roundtrip without being split.
* A fixture with at least two axes round-trips under an explicitly chosen mode:
  sparse output where empty cells are irrelevant, explicit empty preservation,
  or future explicit dense-axis reconstruction.
* Empty cells produce no rows in the explicit output by default.
* Dense reconstruction, when later implemented, is driven only by explicit
  axis contracts and never by heuristics.
* Configuration errors raise clear diagnostics.
* The composed step is usable as a standard pipeline step via
  `build_steps_from_config`.
