====== Purpose

Add focused architecture guards that keep concrete spreadsheet adapters on their intended contract boundaries.

Concrete XLSX and ODS adapters may contain backend-specific implementation details, but they must not become alternate orchestration layers, domain interpreters, or generic rendering pipelines.

This FTR protects the adapter boundary before additional formula, validation, FK-helper, and spreadsheet-facing features expand the surface area again.

'''

====== Context

The project now separates spreadsheet handling into several layers:

* Domain and Pipeline produce or transform frames and canonical metadata.
* Generic rendering builds backend-neutral IR and `RenderPlan` structures.
* Spreadsheet contract helpers orchestrate the generic write/read paths.
* Concrete adapters execute backend-specific read/write mechanics for formats such as XLSX and ODS.

This separation is fragile. It is easy for an adapter to reach backward into rendering internals, inspect canonical metadata directly, or duplicate orchestration logic because that seems simpler for one backend.

Recent work already introduced contract and parse-boundary tests. This FTR generalizes the protection: adapters should remain thin and format-specific, while generic semantics stay outside them.

'''

====== Problem statement

Without explicit guards, concrete spreadsheet adapters can gradually accumulate responsibilities that belong elsewhere:

* calling render composer or render passes directly instead of using the spreadsheet contract boundary
* interpreting domain or pipeline semantics
* depending on generic rendering internals beyond `RenderPlan` consumption
* duplicating parse projection logic from `WorkbookIR` to `Frames + _meta`
* handling formula semantics instead of only backend syntax execution or parsing
* promoting carrier hints to canonical metadata

This would recreate the coupling that earlier architecture work tried to remove.

'''

====== Goals

. Keep adapter write paths behind the spreadsheet contract boundary.
. Keep adapter read paths behind the parse contract boundary.
. Ensure concrete renderers consume `RenderPlan` rather than rebuilding or mutating generic render semantics.
. Ensure concrete parsers produce `WorkbookIR` and do not project to frames directly.
. Prevent adapters from importing Domain or Pipeline modules.
. Make violations visible as architecture tests under `tests/architecture/`.

'''

====== Boundary rules

====== Write path

Concrete backend entry points such as `xlsx_backend.py` and `ods_backend.py` should delegate generic write orchestration to the spreadsheet contract layer.

Allowed:

* call a spreadsheet contract function that builds a backend-neutral render plan
* call the concrete renderer with that plan
* handle backend-specific file paths, options, and IO details

Not allowed:

* import render composers directly
* import render passes directly
* import render flow internals directly unless explicitly part of the contract
* inspect domain-specific metadata to decide rendering semantics
* duplicate generic write orchestration inside each backend

====== Renderer boundary

Concrete renderers may consume `RenderPlan` and use backend libraries such as OpenPyXL or odfpy.

Allowed:

* import concrete backend libraries
* translate `RenderPlan` operations to backend-specific API calls
* emit backend-specific formula syntax only when executing an already adapter-facing operation

Not allowed:

* call render composer or render passes
* reconstruct `WorkbookIR` from frames
* reinterpret canonical metadata
* reach into Domain or Pipeline packages

====== Read path

Concrete parsers should parse backend files into `WorkbookIR` and stop there.

Allowed:

* use backend-specific parser libraries
* populate `WorkbookIR`, sheet IR, tables, validations, named ranges, and carrier hints
* use backend-local parser interpretation helpers

Not allowed:

* project `WorkbookIR` to frames directly inside the parser
* call spreadsheet contract projection helpers from parser internals
* promote carrier hints such as freeze panes, autofilter refs, or backend anchors into canonical `_meta`
* import Domain or Pipeline modules

====== Backend contract facade

The generic spreadsheet contract layer may connect generic rendering/projection functions to adapter entry points.

Allowed:

* import generic rendering and projection modules
* expose narrow write/read functions used by concrete adapters

Not allowed:

* import concrete XLSX or ODS adapter modules
* contain backend-specific syntax handling
* depend on OpenPyXL or odfpy

'''

====== Scope

This FTR includes:

* add or strengthen architecture tests under `tests/architecture/` for adapter boundary imports
* verify that XLSX and ODS backend entry points use the spreadsheet contract facade
* verify that XLSX and ODS renderers consume `RenderPlan` without reaching back into earlier render layers
* verify that XLSX and ODS parsers stop at `WorkbookIR` and do not call projection or contract layers directly
* verify that concrete adapters do not import Domain or Pipeline modules
* document any intentional current-state exceptions

The first implementation should focus on the high-value concrete spreadsheet adapters. It does not need to scan every file in every backend package.

====== Current-state exceptions

Some current modules may still import or inspect more than the ideal boundary allows.

If such cases exist, this FTR should not hide them. Each exception must be one of:

* removed during this FTR if the fix is small and safe
* documented as `current-state` in an architecture test
* deferred to a named follow-up FTR

Unexplained allowlists are not acceptable. Every allowlist entry should explain the intended removal condition or why it is durable.

'''

====== Non-goals

This FTR does not:

* rewrite the XLSX or ODS adapters
* introduce formula providers
* change spreadsheet rendering behavior
* change parse or roundtrip semantics
* implement a full repository-wide import linter
* decide all future backend extension rules
* remove all current-state exceptions in one step

Broader formula-specific guards belong to `FTR-RAW-FORMULA-BOUNDARY-GUARDS-P4`. Meta/carrier-specific guards belong to `FTR-META-CARRIER-BOUNDARY-GUARDS-P4`.

'''

====== Acceptance criteria

. Architecture tests exist that protect the XLSX and ODS adapter contract boundaries.
. XLSX and ODS backend entry points are checked for delegation through the spreadsheet contract facade.
. XLSX and ODS renderers are checked for consuming `RenderPlan` without reaching back into composer, passes, flow, or domain/pipeline layers.
. XLSX and ODS parsers are checked for stopping at `WorkbookIR` and staying out of projection and spreadsheet contract layers.
. The generic spreadsheet contract facade is checked to remain free of concrete backend imports and backend libraries.
. Adapters are checked not to import Domain or Pipeline modules.
. Any necessary exceptions are documented as current-state or linked to follow-up FTRs.
. Existing active tests continue to pass.
. No production behavior changes are introduced unless required to remove a small, obvious boundary violation.

'''

====== Implementation notes

Initial implementation adds focused guards for the current XLSX and ODS adapter
surface:

* `tests/architecture/spreadsheet_contract/test_ir_call_chain.py` now covers
  ODS write/read delegation through `spreadsheet_contract`, matching the
  existing XLSX handoff checks.
* `tests/architecture/spreadsheet_contract/test_adapter_contract_boundary_guards.py`
  checks that backend entry points, concrete renderers, concrete parsers,
  parser-interpretation helpers, and the spreadsheet contract facade stay on
  their intended import boundaries.
* The ODS spreadsheet-contract handoff parity gap in the architecture guard
  inventory is considered resolved by this implementation.

No production behavior change was required.

====== Review guidance

When reviewing adapter changes after this FTR, reviewers should ask:

* Is this adapter executing backend mechanics, or is it reintroducing generic rendering orchestration?
* Does this parser stop at `WorkbookIR`?
* Does this renderer consume `RenderPlan` only?
* Is any Domain or Pipeline semantic being interpreted inside an adapter?
* Would the same generic behavior be available to both XLSX and ODS without duplicating logic?
* Is an allowlist entry durable, or is it hiding technical debt?

'''

====== Rationale

Spreadsheet adapters are a natural place for architecture drift because they sit at the boundary between generic semantics and concrete file formats.

A small shortcut inside one adapter can silently become a second implementation of the rendering or parsing contract. That increases backend divergence and makes future formula-provider work more fragile.

This FTR adds mechanical checks at the adapter seam. The goal is not to make adapters abstract or powerless. The goal is to keep them concrete, format-specific, and narrow.
