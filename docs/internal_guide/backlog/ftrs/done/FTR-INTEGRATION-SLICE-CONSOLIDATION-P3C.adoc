===== FTR-INTEGRATION-SLICE-CONSOLIDATION-P3C — Consolidate and clarify integration test slices

*Status:* Implemented
*Epic:* Phase 4 — Feature Expansion and Domain Transformations
*Priority:* Medium

*Depends-on:*

* <<FTR-TEST-SUITE-TOPOLOGY-P3C#,FTR-TEST-SUITE-TOPOLOGY-P3C>>
* <<FTR-ARCHITECTURE-TEST-LAYER-P3C#,FTR-ARCHITECTURE-TEST-LAYER-P3C>>

*Related-to:*

* <<FTR-PREHEX-TEST-CLASSIFICATION-P3C#,FTR-PREHEX-TEST-CLASSIFICATION-P3C>>
* <<FTR-ARCHITECTURE-GUARD-INVENTORY-P4#,FTR-ARCHITECTURE-GUARD-INVENTORY-P4>>

'''

====== Purpose

Reduce ambiguity, duplication, and accidental overlap in integration tests by consolidating them into clearly defined product slices.

Integration tests should describe real product paths across layers, not act as a mixed collection of semi-unit tests, duplicated roundtrips, and implicit architecture checks.

'''

====== Context

After `FTR-TEST-SUITE-TOPOLOGY-P3C`, integration tests are physically separated from unit and architecture tests. However, the integration layer itself may still contain:

* duplicated roundtrip scenarios
* tests that differ only in minor variations of the same pipeline
* tests that mix product behavior with architecture assertions
* legacy patterns carried forward from pre-hex structure

Without consolidation, integration tests become noisy and lose their value as high-signal product verification slices.

'''

====== Problem statement

The current integration test layer risks:

* duplication of the same product paths with slight variations
* unclear test intent (is this a product slice, a regression test, or an invariant?)
* slow test runs due to redundant coverage
* difficulty selecting meaningful subsets for development or CI

Integration tests should represent a small number of well-understood end-to-end or cross-layer scenarios, not an accumulation of historical cases.

'''

====== Goals

. Identify and group integration tests by product slice.
. Remove or merge redundant tests that cover the same behavior.
. Ensure each integration test has a clear, documented purpose.
. Separate product-path tests from architecture or invariant checks.
. Keep the integration suite lean but representative.

'''

====== Scope

This FTR includes:

* review all tests under `tests/integration/`
* group tests into meaningful slices such as:
** roundtrip (XLSX, ODS, JSON)
** spreadsheet contract read/write paths
** pipeline execution scenarios
** adapter-specific integration paths
* identify duplicate or near-duplicate tests
* merge or remove redundant tests where safe
* move architecture assertions out of integration tests into `tests/architecture/`
* document the intent of remaining integration slices in `tests/README.adoc` or module docstrings

'''

====== Non-goals

This FTR does not:

* redesign the architecture
* introduce new product features
* rewrite all integration tests from scratch
* enforce strict naming conventions (handled separately)
* classify pre-hex tests

'''

====== Consolidation strategy

====== Step 1 — Identify slices

Typical slices should be explicitly named, for example:

* `roundtrip/xlsx`
* `roundtrip/ods`
* `spreadsheet_contract/read`
* `spreadsheet_contract/write`
* `pipeline/basic_flow`
* `pipeline/split_join`

The exact naming is less important than consistency and clarity.

====== Step 2 — Detect duplication

Look for tests that:

* execute the same pipeline with minor input variations
* differ only in data values but not behavior
* assert identical outcomes through different helper functions

These should be:

* merged into parametrized tests, or
* reduced to a single representative case

====== Step 3 — Separate concerns

If an integration test contains assertions such as:

* import-direction checks
* meta-carrier structure checks
* adapter boundary assertions

then those assertions belong in `tests/architecture/`.

Integration tests should focus on:

* observable product behavior
* cross-layer correctness

====== Step 4 — Preserve signal

Do not over-reduce the suite.

Each retained integration test should answer a clear question, for example:

* "Does a full XLSX roundtrip preserve canonical meta?"
* "Does the pipeline produce correct frames for split/join scenarios?"
* "Does the spreadsheet contract correctly bridge rendering and backend execution?"

'''

====== Documentation

Each integration test module should include a short docstring describing:

* the product slice it represents
* the layers involved
* why the test exists

Optionally, `tests/README.adoc` can include a short section listing the main integration slices.

'''

====== Acceptance criteria

. Integration tests are grouped into clearly identifiable slices.
. Duplicate or near-duplicate tests are removed or consolidated.
. Each remaining integration test has a clear purpose documented in a module docstring.
. Architecture or invariant assertions are not mixed into integration tests.
. The integration test suite remains green.
. Test runtime is not increased and should ideally decrease.
. No production code changes are introduced.

'''

====== Implementation notes

The current implementation keeps the existing integration product slices and
documents them explicitly:

* `tests/integration/README.adoc` lists the active integration slices and the
  consolidation rules for future additions.
* Remaining integration test modules have module docstrings that describe the
  product slice they represent.
* Empty placeholder modules under `tests/integration/io_backends/xlsx/` were
  removed because they did not represent executable integration slices:
  `test_roundtrip_json_xlsx.py` and `test_xlsx_backend_write_multi.py`.

No production code changes were introduced. No executable integration scenario
was removed.

'''

====== Review guidance

When reviewing integration test changes, reviewers should ask:

* What product slice does this test represent?
* Is this behavior already covered by another test?
* Is this test asserting product behavior or architecture?
* Could this be expressed as a parametrized case instead?
* Does removing this test reduce coverage in a meaningful way?

'''

====== Rationale

Integration tests are expensive and high-value. If they are duplicated, unclear, or overly numerous, they lose signal and slow down development.

A small, well-structured set of integration slices provides better confidence than a large, unstructured collection of overlapping tests.

This FTR ensures that integration tests remain focused on what they are meant to protect: real product behavior across layers.
