===== FTR-GRAPH-REFERENCE-VALIDATIONS-P4A -- Validate simple node/edge configuration networks

*Status:* Done
*Priority:* P4A
*Area:* Domain validations / relational graph checks
*Prerequisite:* FTR-STANDARD-REFERENCE-VALIDATIONS-P4A

==== Purpose

Add bounded graph-shaped validation for configuration models that describe
operations, routes, dependencies, or other node/edge networks.

Standard reference validations cover the table mechanics: keys exist, foreign
keys point to known rows, and relation tuples are unique. Some consumer models
also need graph-level diagnostics: which edge is broken and whether duplicate
edges exist.

This first slice is intentionally basic. Topology checks such as cycle
detection and reachability are deferred to
`FTR-GRAPH-TOPOLOGY-VALIDATIONS-P5`.

==== Example Shape

[source,yaml]
----
- step: validate_graph
  mode: fail
  graph: calculation_operation_network
  nodes:
    - name: operations
      frame: calculation_operations
      key: operation_id
    - name: variables
      frame: variables
      key: variable_id
  edges:
    - name: operation_outputs
      frame: operation_outputs
      source_node: operations
      source_column: operation_id
      target_node: variables
      target_column: variable_id
      unique: true
  checks:
    - endpoints_exist
    - unique_edges
----

==== Goals

* Provide graph-shaped findings over configured node and edge frames.
* Reuse primary-key and foreign-key validation mechanics where possible.
* Support duplicate-edge checks for single-column and composite edge identity.
* Keep endpoint-existence and unique-edge checks as the first implemented
  contract.
* Keep graph configuration explicit: no inference from column names.
* Make diagnostics readable for workbook users by naming node, edge, frame, and
  offending row values.

==== Non-goals

* Do not implement a full graph query language.
* Do not implement cycle detection, reachability, or other topology checks in
  this slice.
* Do not infer domain meaning from operation, route, or marker names.
* Do not decide whether a business route is professionally correct.
* Do not replace consumer plugins for semantic validations that require domain
  rules.

==== Acceptance

* A pipeline can validate a simple node/edge network from configured frames.
* Tests cover endpoint existence, duplicate edges, and composite edge keys.
* Findings include enough frame/row/key context to trace the issue back to the
  workbook.
* The FTR explicitly composes with standard reference validations instead of
  duplicating their public contract.
