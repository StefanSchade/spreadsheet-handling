'''

====== Purpose

Define one explicit, lightweight decision record for architecture guardrails so
that new guards are consistent, reviewable, and intentionally scoped.

This mini-FTR prevents hidden policy drift and avoids creating competing guard
regimes across test files.

'''

====== Context

Phase-4 planning now includes multiple guardrail tickets and an architecture
guard inventory. Without one shared decision template, teams can still diverge
on:

* what is a durable invariant vs. a current-state check
* where exceptions are allowed
* who owns each guard
* when temporary allowlists must be removed

The project needs a single minimal record that each new guard ticket can use.

'''

====== Problem statement

Guard tickets can be individually reasonable but collectively inconsistent if
their decisions are not recorded in one normalized format.

Typical failure modes:

* unclear invariant durability
* permanent "temporary" allowlists
* duplicated or contradictory guard logic
* no explicit removal condition for exceptions

'''

====== Goals

. Introduce a compact Guardrail Decision Record (GDR) template.
. Require each new guard to document scope, rationale, and sunset conditions.
. Separate durable invariants from current-state checks.
. Keep overhead low so the process remains practical.

'''

====== Scope

This mini-FTR includes:

* add a small GDR template to the architecture inventory document
* require each new guardrail FTR to contain at least one filled GDR entry
* require explicit ownership and review cadence for each guard
* require explicit exception handling with a removal trigger

'''

====== Non-goals

This mini-FTR does not:

* add new runtime guard tests by itself
* redesign architecture boundaries
* introduce heavyweight governance tooling

'''

====== Guardrail Decision Record (GDR) template

Use this minimal template per guard:

[source,text]
----
Guard ID:
Boundary / Invariant:
Type: durable-invariant | current-state
Protected modules / paths:
Forbidden patterns / imports:
Allowed exceptions:
Exception sunset condition:
Owner:
Review cadence:
Failure mode in CI:
Linked FTR(s):
----

'''

====== Acceptance criteria

. A GDR template is documented and referenced from architecture guard work.
. Each active guardrail FTR includes at least one concrete GDR entry or an
  explicit statement that the entry will be created in implementation.
. Each documented exception has a sunset condition.
. Durable invariants and current-state checks are explicitly distinguished.
. Guard ownership is explicit for all new guard tickets in this wave.

'''

====== Rationale

The project already has strong architectural work and guard intent. This
mini-FTR adds the missing decision discipline with minimal process overhead.
