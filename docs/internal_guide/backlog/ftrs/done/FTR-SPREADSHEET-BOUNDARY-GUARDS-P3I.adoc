===== FTR-SPREADSHEET-BOUNDARY-GUARDS-P3I -- Expand executable guards for spreadsheet adapter boundaries

*Status:* Done

*Depends-on:* <<FTR-SPREADSHEET-BACKEND-CONTRACT-P3H#,FTR-SPREADSHEET-BACKEND-CONTRACT-P3H>>, <<FTR-XLSX-ADAPTER-ALIGNMENT-P3H#,FTR-XLSX-ADAPTER-ALIGNMENT-P3H>>, <<FTR-SPREADSHEET-PARSE-CONTRACT-P3I#,FTR-SPREADSHEET-PARSE-CONTRACT-P3I>>, <<FTR-XLSX-PARSER-MODULARIZATION-P3I#,FTR-XLSX-PARSER-MODULARIZATION-P3I>>

*Context:* Phase 3h clarified the spreadsheet architecture and introduced initial guardrails, but several of them are still consciously minimal. Current tests prove the desired direction in selected places, yet package coverage, adapter-API enforcement, and read-path protection are still lighter than the architectural intent they are meant to defend.

*Problem statement:* If the current minimal guards remain the only executable protection, future changes can easily reintroduce broad imports, bypass narrow adapter seams, or mix format-specific knowledge back into generic spreadsheet layers without immediate feedback.

*Goals:*

. Broaden dependency and import guards so both write-path and read-path seams are covered more systematically.
. Enforce that spreadsheet adapters consume the intended narrow boundary APIs rather than internal rendering/parser details.
. Protect the separation between format-specific parser/renderer logic and generic `WorkbookIR` / projection logic.
. Make boundary tests read as durable architecture guards rather than one-off checks tied to the current XLSX implementation shape.

*Acceptance criteria:*

. The protected module/package set for spreadsheet boundary checks is broader and more systematic than the current hand-picked minimal list.
. There are executable guards for the intended dependency direction on both the write path and the now-documented parse path.
. The XLSX path demonstrably uses the intended narrow boundary modules/APIs instead of reaching through them into internal implementation details.
. The resulting tests are documented as architectural guardrails, including where they are intentionally conservative or partial.
. The guards are strong enough that future adapter work can treat the spreadsheet boundary as an enforced seam rather than as documentation only.

*Non-goals:*

- No broad rename or module-shuffle campaign for cosmetic cleanliness.
- No general-purpose architecture linter for the whole repository.
- No forced deep refactor of adapter-internal code where the existing cut is already boundary-safe.

*Why now:* Phase 3i is the right point to turn the now-visible spreadsheet seams into stronger drift protection before Phase 3j starts depending on them.
