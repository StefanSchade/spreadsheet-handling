==== Implementation Quality

*Strengths:*

* **Layered resolution** in `StylePass` is clean and well-ordered:
  1. Explicit `sheets.<sheet>.helper_columns` from options
  2. FK helper provenance from `derived.sheets.*.helper_columns`
  3. Lookup helper provenance from `derived.sheets.*.enrich_lookup.helper_columns`
  4. `helper_prefix` fallback (existing behavior preserved)
* **`_sheet_source_candidates`** correctly resolves the frame-name→sheet-name
  indirection introduced by `configure_workbook_view` (e.g., frame
  `variables_view` rendered as sheet `Variables`) by looking up the
  `workbook_view.sheets` mapping
* **`_helper_column_names_from_value`** is defensively coded: handles string,
  list-of-strings, list-of-mappings (`column`/`name` keys), mapping-with-columns
  — accommodating both the simple list form and structured provenance shapes
* **Conflict detection** in `_merge_sheet_options`: if `helper_columns` is
  configured both as a top-level sheet spec key and inside `options` with
  different values, a clear `ValueError` is raised
* **Parser option hints** correctly propagate `helper_columns` through
  both XLSX and ODS `OPTION_HINT_KEYS` tuples, ensuring roundtrip preservation
* **`_helper_column_indices`** merges explicit names and prefix detection into a
  single sorted index list — no duplicate indices possible

*Side fix:*

* `join_views.py`: nested ternary for `right_selected` replaced with
  `if/elif/else` — improves readability, no behavioral change

*Minor observations (non-blocking):*

* `_workbook_meta(doc)` is called once per `StylePass.apply()` and passed to
  `_derived_helper_column_names` per sheet — efficient; no concern
* The `_helper_column_names_from_value` function is quite forgiving in what it
  accepts. This is appropriate given that provenance metadata may come from
  various sources (FK helpers, lookup helpers, user config)
* `_non_empty_string` appears again in `frame_selection.py` — same shared
  utility refactoring candidate noted in previous reviews

==== Architectural Observations

This FTR cleanly fills a gap in the rendering metadata stack:

* **Frame-level lifecycle** (`frame_lifecycle`): owned by preceding steps
* **Sheet-level options** (`sheets.<sheet>.*`): now includes `helper_columns`
* **Derived rendering metadata** (`__helper_cols`): transient IR, produced by
  `StylePass`, consumed by backends

The metadata resolution stays within the existing `ApplyColumnStyle` render-op
path — no new backend-specific branches were needed.

==== Verdict

Implementation correctly adds explicit helper-column metadata support while
preserving prefix compatibility, documents the metadata namespace relationship,
and covers both XLSX and ODS render paths. The layered resolution strategy is
sound. Accepted without modification.
