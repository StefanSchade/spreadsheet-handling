====== Purpose

Establish a minimal, consistent set of naming and structural conventions for tests so that their intent, scope, and responsibility are immediately understandable.

The goal is not stylistic perfection but reducing ambiguity during development and review.

'''

====== Context

After introducing a responsibility-based test topology (`unit`, `integration`, `architecture`), the physical structure already conveys the primary role of a test.

However, inconsistencies in file names, test function names, and module documentation still create friction:

* unclear distinction between behavior tests and invariant tests
* inconsistent naming patterns across modules
* missing or misleading test names
* difficulty understanding intent during code review

A lightweight convention layer is needed to stabilize the suite without over-engineering it.

'''

====== Problem statement

Without conventions, the test suite tends to drift toward:

* inconsistent naming styles
* redundant or vague test names
* implicit assumptions about test scope
* increased cognitive load during maintenance

This does not break functionality, but it slows down development and increases the risk of misunderstanding test intent.

'''

====== Goals

. Define clear naming conventions for test files and test functions.
. Ensure test names communicate behavior or invariant explicitly.
. Require module-level docstrings where they add clarity.
. Keep conventions minimal and enforceable through review, not heavy tooling.
. Avoid churn where no clarity improvement is achieved.

'''

====== Naming conventions

====== Test file names

* Use `test_<subject>_<aspect>.py`

Examples:

* `test_ir_writepath_semantics.py`
* `test_pipeline_split_join.py`
* `test_xlsx_roundtrip.py`

Avoid:

* generic names like `test_misc.py`
* overly long descriptive names

The directory already conveys the test type (`unit`, `integration`, `architecture`). File names should describe the subject and focus.

====== Test function names

* Use `test_<expected_behavior>`

Examples:

* `test_roundtrip_preserves_named_ranges`
* `test_pipeline_produces_expected_frames`
* `test_adapter_does_not_import_domain_modules`

Avoid:

* names that describe implementation steps instead of behavior
* names that repeat the file name without adding information

====== Architecture test naming

Architecture tests should explicitly state the invariant or boundary:

* `test_rendering_layer_does_not_import_adapters`
* `test_parser_stops_at_workbook_ir`
* `test_meta_canonical_not_used_for_render_decisions`

These names should read like rules, not scenarios.

'''

====== Module docstrings

Each test module should include a docstring when:

* the test represents a product slice (integration tests)
* the test enforces an invariant or boundary (architecture tests)

The docstring should briefly explain:

* what is being tested
* why the test exists

Example:

[source,python]
----
"""
Ensures that the XLSX roundtrip preserves canonical metadata.
This test guards against regressions in the spreadsheet contract.
"""
----

Avoid long explanations; focus on intent.

'''

====== Assertion style

* Prefer assertions that express behavior or invariants clearly.
* Avoid asserting intermediate implementation details unless necessary.

Examples:

* Prefer: `assert result.frames == expected_frames`
* Avoid: checking internal intermediate objects unless they are part of the contract

Architecture tests may assert structure or imports, but should still reflect the intended rule.

'''

====== Use of parametrization

* Use `pytest.mark.parametrize` when multiple inputs test the same behavior.
* Avoid duplicating nearly identical test functions.

This is especially relevant for integration tests and roundtrip scenarios.

'''

====== Scope

This FTR includes:

* align obviously inconsistent file names with the conventions
* improve unclear test function names where beneficial
* add missing module docstrings for architecture and integration tests
* remove clearly misleading names

'''

====== Non-goals

This FTR does not:

* rename all tests for cosmetic consistency
* enforce conventions via heavy tooling
* change test behavior
* restructure the test suite (handled by other FTRs)
* introduce strict linting rules for naming

'''

====== Acceptance criteria

. New tests follow the defined naming conventions.
. Obvious inconsistencies in existing tests are corrected.
. Architecture and integration test modules include meaningful docstrings.
. Test names clearly express behavior or invariants.
. No unnecessary churn is introduced.
. All tests continue to pass.

'''

====== Delivery scope

Delivered as a lightweight convention baseline:

* added developer-manual guidance for test file names, test function names,
  architecture-test invariants, and when module docstrings are expected
* renamed obvious file-name inconsistencies:
  `test_test_carrier_boundary_guards.py`,
  `test_validate_colums_xls_effect.py`, and
  `test_run_select_io_config_unknow_profile.py`
* clarified a focused set of vague test function names in touched CLI,
  pipeline, and integration modules
* added the missing module docstring to the legend-block integration slice
* marked modified test files with this FTR for traceability

No strict naming linter or broad cosmetic rename pass was introduced.

'''

====== Review guidance

When reviewing tests, reviewers should ask:

* Does the file name describe the subject and aspect?
* Does the test name describe expected behavior or an invariant?
* Is the intent clear without reading the full implementation?
* Is the test name redundant or misleading?
* Would renaming improve clarity, or just create noise?

'''

====== Rationale

Naming is a low-cost way to improve clarity across the test suite.

Clear, consistent naming reduces the need to read test implementations during review and makes it easier to understand what is protected.

This FTR intentionally keeps the scope small: it targets the most obvious inconsistencies and establishes a stable baseline for future work.
