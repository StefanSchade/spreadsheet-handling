===== FTR-PARITY-IR-P3 — IR Renderer Parity with Legacy OpenPyXL

*Context*::
During migration, the IR-backed XLSX renderer must match legacy outputs for core scenarios (style, validations, sheet naming).
A short-lived CI safeguard verifies parity on a controlled set of fixtures while refactors are still landing.

*Goal*::
Achieve parity in layout and key behaviors; provide test fixtures that compare legacy vs IR outputs for selected cases.
Prove correctness under real CI conditions, then prepare to flip to IR-only.

*Rationale*::
Temporary parity verification prevents silent regressions from transitive changes (deps, locale/ordering, shared utils) during the migration window.
Confidence for deprecating legacy path; documents expected equivalence.

*Acceptance*::
* A dedicated CI job **parity-ir** runs a small fixture suite with **both** backends and asserts **normalized structural equivalence** (sheets/order, styles, filters, validations, meta) — not byte-zip equality.
* Job triggers only on:
** Changes under `ir/`, `xlsx/`, `compose/`, `passes/`, CI config, or dependency manifests.
** A nightly run during migration.
* Environment is hardened and pinned in the job: `TZ=UTC`, `LC_ALL=C`, fixed `PYTHONHASHSEED`, pinned Python minor + XLSX deps.
* Parity suite covers (minimum):
** Header styles (bold/fill), AutoFilter, optional freeze first row.
** Validations: in-list and basic numeric (min/max).
** Sheet naming and order.
** Hidden `_meta` sheet with minimal fields (e.g., `version`, `exported_at`, `author`).
* Stability gate: **10 consecutive green runs** across the supported CI matrix over **7 calendar days**, with **zero flakes** recorded.
* Exit outcome of P3: open a prepared PR (or branch) that flips adapters to IR by default and is blocked on P4 acceptance.

IMPORTANT: No product-surface flag: the IR/legacy switch exists **only in tests/CI**; adapters in production code remain unchanged in P3.
