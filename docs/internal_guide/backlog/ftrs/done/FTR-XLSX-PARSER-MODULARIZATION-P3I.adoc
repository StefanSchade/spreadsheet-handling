===== FTR-XLSX-PARSER-MODULARIZATION-P3I -- Sharpen the XLSX parser seam before second-adapter work

*Status:* Done

*Depends-on:* <<FTR-SPREADSHEET-PARSE-CONTRACT-P3I#,FTR-SPREADSHEET-PARSE-CONTRACT-P3I>>, <<FTR-XLSX-ADAPTER-ALIGNMENT-P3H#,FTR-XLSX-ADAPTER-ALIGNMENT-P3H>>

*Context:* The outer XLSX adapter cut is already in reasonable shape: `xlsx_backend.py` is a thin entry layer over the spreadsheet contract, and the renderer mainly executes `RenderPlan` via OpenPyXL. The remaining structural pressure sits on the read side, where `openpyxl_parser.py` still mixes format-specific workbook extraction with spreadsheet-semantic interpretation and `WorkbookIR` assembly.

*Problem statement:* If the current parser stays monolithic, a future ODS/Calc adapter will face the wrong choice: either duplicate the XLSX parser shape too literally or improvise a second parser architecture under delivery pressure. Both outcomes would make cross-adapter parse consistency harder to reason about and would hide the real parse contract behind one format's current implementation.

*Goals:*

. Separate XLSX-/OpenPyXL-specific extraction concerns from spreadsheet-semantic interpretation concerns inside the current parser path.
. Improve the parser's internal modularization where that sharpens the read-path contract, without turning this into a broad cosmetic refactor of the whole XLSX adapter.
. Identify parser logic that is merely XLSX carrier handling versus logic that is a candidate for shared spreadsheet interpretation or at least shared contract tests.
. Reduce the risk that ODS/Calc work begins by copying internal XLSX parser structure instead of targeting the clarified parse contract.

*Acceptance criteria:*

. The parser-side code has a clearer internal seam between format extraction and `WorkbookIR` construction/interpretation than today.
. The resulting structure makes it easier to see which responsibilities are XLSX-only and which ones are intentionally spreadsheet-level.
. Any extracted/shared logic is justified by real contract value or cross-adapter relevance, not by premature abstraction alone.
. Existing behavior remains covered by focused parse-path tests, and any newly clarified seam is reflected in those tests or nearby architecture docs.
. The work stays narrower than a broad adapter rewrite and does not reopen the already-settled boundary-first disposition for the XLSX adapter as a whole.

*Non-goals:*

- No broad rename or package shuffle of the XLSX adapter for aesthetic reasons.
- No requirement that every current parser helper move into the generic IR layer.
- No premature shared parser framework for arbitrary future spreadsheet formats.
- No ODS/Calc implementation in this FTR.

*Why now:* Before a second spreadsheet adapter is implemented, it is worth cleaning up the one place where the current reference adapter still hides too much contract shape in one file: the parser boundary between carrier extraction and spreadsheet interpretation.
