'''

====== Purpose

Define clear, minimal rules for how archived and quarantined tests are handled so that the test suite does not accumulate ambiguous, semi-active test code over time.

The goal is to avoid a persistent gray zone where tests are neither trusted nor removed.

'''

====== Context

The project already distinguishes active tests from historical or transitional ones, especially with the introduction of `tests/legacy_pre_hex/`.

However, without explicit rules:

* archived tests may remain indefinitely without decision
* quarantined tests may silently influence behavior expectations
* developers may not know whether a failing test indicates a regression or intentional change

A small set of rules is required to make these states explicit and manageable.

'''

====== Problem statement

Test code that is not clearly active tends to accumulate in ambiguous states:

* ignored but still present
* partially outdated but still referenced
* duplicated across active and legacy areas

This creates:

* hidden coupling to past behavior
* confusion during refactoring
* unnecessary maintenance burden

Without rules, this problem grows over time.

'''

====== Goals

. Define clear categories for non-active tests.
. Ensure quarantined tests are explicitly marked and isolated.
. Prevent new ambiguous test states from being introduced.
. Provide a clear lifecycle for archived tests.
. Keep the active test suite clean and trustworthy.

'''

====== Test states

Tests must belong to exactly one of the following states:

====== Active

* Located under `tests/unit/`, `tests/integration/`, or `tests/architecture/`
* Executed in the default test run
* Considered authoritative for current behavior

====== Quarantined

* Located under `tests/legacy_pre_hex/` or explicitly marked as quarantined
* Not executed in the default test run
* Represent behavior that is under evaluation or migration

====== Archived

* Removed from the active tree or stored in a clearly labeled archive location
* Not executed and not intended to be reactivated without explicit decision
* Retained only for historical reference if necessary

'''

====== Quarantine rules

* Quarantined tests must be physically separated from the active suite.
* They must not run as part of the default test command.
* They must be explicitly runnable (e.g. via `make test-prehex`).
* Each quarantined test or group must have a short note explaining its status.

Acceptable reasons for quarantine:

* pending classification (`promote`, `rewrite`, `retire`)
* depends on incomplete architecture decisions
* expresses behavior tied to deprecated structures

Unacceptable:

* leaving tests quarantined indefinitely without review

'''

====== Archive rules

* Tests classified as `retire` should be removed or archived.
* Archival should be explicit (e.g. dedicated archive directory or documented deletion).
* Archived tests must not be referenced by active tests or documentation as normative behavior.

Archiving is preferred over indefinite quarantine.

'''

====== Transition rules

Movement between states must be explicit:

* Quarantined → Active:
  - test is validated against current architecture
  - moved into the appropriate active directory

* Quarantined → Archived:
  - behavior is no longer required

* Active → Quarantined:
  - only allowed when behavior is under active redesign or investigation
  - must include a short rationale

Implicit transitions are not allowed.

'''

====== Scope

This FTR includes:

* define and document the test states and rules
* ensure `tests/legacy_pre_hex/` is clearly treated as quarantined
* document how to run quarantined tests
* add minimal documentation (README or inline notes)

'''

====== Non-goals

This FTR does not:

* classify all pre-hex tests (handled in `FTR-PREHEX-TEST-CLASSIFICATION-P3C`)
* remove large amounts of test code
* introduce complex tooling for lifecycle management
* enforce rules automatically beyond basic structure

'''

====== Acceptance criteria

. Test states (active, quarantined, archived) are clearly defined.
. `tests/legacy_pre_hex/` is explicitly documented as quarantined.
. Default test runs exclude quarantined tests.
. There is a documented way to run quarantined tests.
. New tests do not introduce ambiguous states.
. Developers can determine the status of any test without ambiguity.

'''

====== Delivery scope

Delivered as a lightweight test-lifecycle baseline:

* documented active, quarantined, and archived states in the Developer Manual
* updated `tests/README.adoc` with lifecycle states and transition rules
* clarified `tests/legacy_pre_hex/README.adoc` as the explicit pre-hex
  quarantine carrier and documented `make test-prehex`
* aligned the Make-target developer snippet with the current active default
  (`not legacy and not prehex and not slow`)
* added a small carrier-boundary guard requiring quarantine carriers to declare
  status and an explicit run target

No broad pre-hex classification, archive migration, or complex lifecycle
tooling was introduced.

'''

====== Review guidance

When reviewing changes involving non-active tests, reviewers should ask:

* Is this test active, quarantined, or archived?
* Is that status clearly visible from its location or documentation?
* Does this test still represent required behavior?
* Should this test be promoted, rewritten, or retired instead?
* Is quarantine being used as a temporary state or as a permanent hiding place?

'''

====== Rationale

A clean test suite depends not only on active tests but also on how inactive tests are handled.

Without clear rules, legacy and transitional tests accumulate and create long-term ambiguity.

By defining explicit states and transitions, the project avoids hidden coupling to past behavior and keeps the active test suite focused and reliable.
