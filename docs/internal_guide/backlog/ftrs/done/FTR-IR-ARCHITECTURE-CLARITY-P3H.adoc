===== FTR-IR-ARCHITECTURE-CLARITY-P3H — Clarify IR-layer concepts, invariants, and architecture documentation

*Status:* Done

*Context:* Phase 3h aims to generalize the spreadsheet architecture beyond the current XLSX implementation. That work will improve extensibility, but it should also leave the IR layer easier to understand and maintain. At the moment, the core ideas are sound, yet some terminology, module boundaries, and write-path invariants still have to be inferred from code and historical migration context.

*Problem statement:* If the spreadsheet-generalization work changes contracts and boundaries without explicitly documenting the conceptual model, the code may become more flexible while still remaining hard to understand. Future contributors would then have to reverse-engineer what belongs in `WorkbookIR`, what belongs in `RenderPlan`, how `meta_canonical` and `meta_rendering` relate to each other, and what belongs only in format-specific adapters.

*Goals:*

. Make the IR layer easier to understand for future maintenance work.
. Document the conceptual flow `Frames -> WorkbookIR -> meta_rendering -> RenderPlan -> Spreadsheet Backend` and the corresponding parse direction where it is already defined.
. Define explicit invariants for `WorkbookIR`, `RenderPlan`, and the write-path split between `meta_canonical` and `meta_rendering` so they do not become catch-all transport objects.
. Harmonize terminology across code, backlog FTRs, and architecture docs.
. Add lightweight drift protection so the clarified boundaries do not immediately erode again.

*Acceptance criteria:*

. Add or update architecture documentation to describe the spreadsheet flow and the role of `WorkbookIR`, `meta_canonical`, `meta_rendering`, `RenderPlan`, parsers, and renderers.
. Document a concise set of invariants for `WorkbookIR`, `RenderPlan`, and the write path, including at least: dropping `meta_canonical` after IR entry does not change the render result; roundtrip does not depend solely on `meta_rendering`; adapters execute `RenderPlan` instead of reinterpreting canonical meta as render semantics.
. Record a naming policy that distinguishes generic spreadsheet concepts from format-specific implementation names.
. Add lightweight architecture-guard tests or equivalent dependency checks for the spreadsheet flow and the intended write-path invariants, without introducing heavy external tooling.
. Update the relevant Phase 3h tickets and/or architecture docs so maintainability and readability are explicit desired outcomes rather than implied side effects.

*Non-goals:*

- No new spreadsheet file format in this FTR.
- No large-scale code rewrite driven only by style preferences.
- No duplicate architecture handbook beyond targeted updates to the existing internal architecture docs.
- No mandatory adoption of a heavyweight ArchUnit/import-linter-style framework in this phase.
- No final typed meta model in this step.

*Why now:* The architectural boundaries are in motion anyway. This is the cheapest point to make the spreadsheet IR model legible before a second spreadsheet backend turns implicit assumptions into permanent complexity.
