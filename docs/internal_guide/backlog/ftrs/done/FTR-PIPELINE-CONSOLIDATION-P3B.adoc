===== FTR-PIPELINE-CONSOLIDATION-P3B — Consolidate pipeline/application responsibilities after IR stabilization

*Status:* Open
*Epic:* Phase 3b — Pipeline & Orchestration Consolidation

*Purpose:*::
Establish a clear application-layer structure after the IR rendering path has been stabilized. Reduce responsibility overlap between `engine`, `pipeline`, and `orchestrator` so that new features can be implemented on a simpler and more maintainable control flow.

*Context:*::
The current architecture already separates spreadsheet rendering concerns through the IR path, but the application-side flow still contains transitional overlap. In particular, `engine` still acts as a compact orchestration node for validation sequencing, FK-helper preparation/application, and related registry/mapping setup.

This overlap is acceptable during migration, but it should not become the long-term structure for Phase 4 feature work.

*Scope:*::
- Clarify and stabilize the responsibilities of `pipeline`, `orchestrator`, and transitional `engine` code.
- Ensure that pipeline sequencing lives in the application layer rather than in feature-specific helper code.
- Prevent new features from increasing the architectural role of `engine`.

*Non-Goals:*::
- No full removal of `engine` in a single step.
- No user-facing feature expansion.
- No workflow/process-engine style branching model.

*Target Structure:*::
- `domain/` holds pure feature logic and validations.
- `pipeline/` holds step composition and sequencing.
- `orchestrator.py` remains the unified application entry point.
- `engine/` is reduced to a temporary bridge only where still necessary.

*Acceptance:*::
- No new feature logic is introduced into `engine/`.
- Pipeline sequencing responsibilities are explicit and readable.
- The intended ownership of application flow is documented through code structure and tests.
- At least one representative feature slice is routed through the target structure without expanding `engine`.
