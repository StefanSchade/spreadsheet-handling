===== FTR-DEMO-REINTEGRATION -- Re-align the demo repository with the post-Phase-3 core and prepare the next beta

*Context:*::
The original demo-reintegration ticket was written against a much earlier core
milestone. Since then, Phases 3g-3k have substantially changed the core
library: the spreadsheet contract was generalized, the XLSX path was refactored
into a framework-like shape, ODS landed as a second adapter, and the remaining
pre-beta work now focuses on run-path consistency plus demo alignment. The demo
repository should therefore be treated as a real integration surface for the
current beta candidate, not as a one-time cleanup after orchestrator
consolidation.

*Problem statement:*::
The demo repository still reflects an older core baseline (`0.1.0b4`,
XLSX-centric assumptions, and partially stale integration expectations). Before
the next beta is published, the demo must be verified and updated against the
current core build so that:

* local editable development works again end-to-end,
* the documented demo flows reflect the actual current CLI/config model,
* and the demo can act as a release-facing smoke consumer for the next beta.

*Goals:*::

. Verify the current demo repository against the local post-Phase-3 core
   checkout before changing behavior.
. Update demo dependency wiring, build targets, tests, and pipeline assumptions
   so the demo runs cleanly against the next beta candidate.
. Re-center the maintained demo entry points on `sheets-run` plus checked-in
   YAML config/pipeline files instead of the older `sheets-pack` /
   `sheets-unpack`-style shims.
. Decide pragmatically whether workbook output format stays demo-fixed (`xlsx`)
   or becomes an exposed demo-level choice (`xlsx` / `ods`) where that improves
   reuse without overcomplicating the demo.
. Leave the demo repository in a state that can be committed on `main`
   together with the new beta release and still produce a working, reproducible
   integration story.

*Non-goals:*::

- No new core-library feature work here unless the demo exposes a real
  integration defect that belongs back in the core repo.
- No requirement to preserve legacy demo conventions if they conflict with the
  current CLI/config model.
- No promise that every demo path must exercise both adapters if that adds
  complexity without real user value.

*Acceptance criteria:*::

* The demo repo installs and runs cleanly against the current local core
  checkout and the next beta release target.
* At least the key maintained demo pipelines and tests execute successfully
  end-to-end with the current core build.
* The maintained demo generation and readback flows run through `sheets-run`
  with checked-in YAML config/pipeline files; any retained `pack`/`unpack`
  helpers are clearly secondary.
* Obsolete assumptions from the older beta baseline are either updated or
  explicitly removed.
* For the upcoming beta, the demo's maintained workbook path is intentionally
  `xlsx`-first unless `ods` can be externalized cleanly through the same run
  path and verified without extra demo complexity.
* The resulting demo state is suitable as the companion integration surface for
  the next beta release.
