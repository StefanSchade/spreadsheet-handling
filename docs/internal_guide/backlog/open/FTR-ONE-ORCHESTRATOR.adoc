* FTR-008: Orchestrator Unification — Single shared orchestrate(...) for sheets-run, sheets-pack, and sheets-unpack

** Context: Historically, pack/unpack duplicated flow logic. We now prefer a single execution engine and have wrappers construct pipelines.

** Goal: Provide a shared function orchestrate(pipeline=None, profile=None, io_overrides=None, ...) used by all CLIs. sheets-run and the wrappers call this engine instead of delegating CLI→CLI.

** Rationale: One engine removes drift, simplifies testing, and enables reuse in services (e.g., FastAPI) and batch jobs without re-implementing control flow.

** Solution Steps
*** Extract current run-logic into spreadsheet_handling.orchestrator:orchestrate(...) (resolve pipeline/profile, route I/O via backends, apply passes, collect results).
*** Make sheets-run parse args → call orchestrate(...) (no extra logic).
*** Make sheets-pack/sheets-unpack build their pipelines from flags as today → call orchestrate(pipeline=..., io_overrides=..., ...).
*** Keep steps pure (Frames→Frames); ensure router backends own all I/O. Preserve profile override semantics from CLI.
*** (Optional) Add a lightweight deprecation note in pack/unpack help pointing users to sheets-run + profiles for advanced flows.

** Acceptance::
- A single module spreadsheet_handling/orchestrator.py exposes orchestrate(...) with a stable signature.
- sheets-run calls orchestrate(...) directly (no CLI-to-CLI delegation).
- sheets-pack and sheets-unpack construct pipelines and then call orchestrate(...).
- Pipelines remain pure; all I/O is handled by the router backends; profiles can override I/O and options.
- Docs show both flag-only and profile-based runs for sheets-run, and wrapper usage that yields identical artifacts.
- Unit tests patch orchestrate and assert it is invoked by all three CLIs with the expected arguments.
- Integration tests confirm outputs are unchanged vs. pre-refactor for representative pack/unpack scenarios.
- The engine is importable and suitable for reuse in a web service without invoking any CLI layer.
