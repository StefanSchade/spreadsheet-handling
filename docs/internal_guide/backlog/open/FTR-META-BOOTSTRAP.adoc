=== FTR-META-BOOTSTRAP — Meta Bootstrap and Fault Tolerance

** *Context:*  Workbooks without persisted meta must still roundtrip. All backends should treat missing meta as `{}` and continue.

** *Goal:*  Provide an early domain step (`bootstrap_meta`) that merges configuration inputs into runtime meta, ensuring deterministic defaults even when persisted meta is missing.
- Example for step options are: (e.g., frames.meta.constraints, style.auto_filter, style.freeze_header, style.header_fill_rgb) as we include new features we might have to extend


** *Rationale:*  Prevents `None`-handling errors and gives IR / passes a stable contract without requiring stored meta.

** *Precedence Model:*  Configuration that affects meta may come from several sources. They are applied in the following order (later overwrites earlier unless a step explicitly merges):

1. **Persistence:**  Load existing meta from storage if present; otherwise start with `{}`.
2. **Pipeline Profile Defaults:**  Apply defaults defined in the selected profile.
3. **CLI / Runtime Arguments:**  Overlay any values passed via CLI flags or explicit step configuration. CLI always wins over profile defaults.
4. **Step-Local Behavior:**  Individual steps may choose to *merge* instead of overwrite (e.g., appending to lists of non-essential columns). Such merge semantics are step-specific features, not global rules.

Example — Helper Columns:
- Helper columns are marked in meta as non-essential so they can be safely dropped.
- A later step that adds new helpers should append to that list, not replace it.


** *Acceptance:*
- Missing meta is treated as `{}` by all backends.
- `bootstrap_meta` merges sources according to the precedence model above.
- IR receives the fully merged meta and renders consistently across runs.
