==== Context

The Functional Model promotion work has made the user-facing transformation
space smaller and clearer than the current domain/meta implementation.

The current implementation grew through many locally reasonable feature slices:
FK helpers, lookup helpers, derived columns, metadata carriers, workbook views,
render directives, validations, plugins, and cleanup paths. Several of these
paths now overlap or encode historical implementation strategy instead of a
simple user-facing requirement.

At the same time, the project is still young:

* there is no significant external adoption to protect;
* breaking internal changes are acceptable when they reduce long-term
  complexity;
* the main consumers are known and controllable: `personal_data`, repository
  internal Project Memory, and Worldbuilding;
* adapters, parsers, renderers, CLI behavior, and fixtures contain useful
  boundary knowledge that should not be discarded casually.

The primary risk is therefore not disruption. The primary risk is preserving
old domain/meta complexity under new names.

The domain/IO seam is not a loose boundary where both sides only exchange
primitive frames. It is a bridge: frames plus metadata form a shared contract
language that domain transformations, workbook configuration, renderers,
parsers, and adapters all need to understand.

The work therefore is not a greenfield rewrite. The work is to stabilize that
bridge language, derive it from Functional Model use cases, and then cut the
implementation back to that contract.

==== Decision

Adopt a *contract-first domain/meta reset around a stabilized bridge DSL* as
the architectural direction for simplifying the domain and metadata core.

The term "rewrite" in this ADR's identifier means reset of the domain/meta
interior around an explicit contract. It does not mean a greenfield rebuild of
the product or reflexive replacement of adapters, renderers, parsers, CLI
entry points, or useful fixtures.

Future domain/meta work should follow this direction:

[source,text]
----
Functional Model intent
-> bridge DSL: meta family, writer, reader, lifecycle, conflict rule
-> integration use-case contracts
-> simplified domain/meta implementation around that bridge
-> semantic consistency checks
-> implementation consistency checks
----

The reverse direction is rejected:

[source,text]
----
existing implementation sediment
-> inferred model
-> renamed sediment
----

The rewrite should:

* build executable integration contracts from the Functional Model before
  replacing behavior;
* derive the bridge DSL from Functional Model use cases, not from historical
  registry shape alone;
* define, for each metadata family or datum, who may write it, who may read it,
  whether it is intent, observation, or derived state, how long it persists, how
  conflicts are resolved, and when it is deleted;
* treat domain transformations and IO/render/parse code as producers and
  consumers of the same explicit bridge contract;
* treat `personal_data`, Project Memory, and Worldbuilding as staged adoption
  coverage checks, not as golden masters for current workarounds;
* preserve adapters, parsers, renderers, CLI entry points, and useful fixtures
  where their boundary behavior remains valid;
* allow breaking changes in the domain/meta interior;
* allow targeted in-place deletion of obvious dead or duplicate historical
  paths when the deletion is low risk and does not need a new contract first;
* delete old domain/meta paths once replacement contracts pass;
* avoid introducing new public concepts merely because an old module or step
  exists.

This ADR records architectural direction only. Concrete work still requires
normal FTRs/backlog items.

==== Bridge DSL Scope

The bridge DSL is the explicit language shared by domain/meta code and the
IO/render/parse boundary.

For every bridge datum, the model should answer:

[cols="1,3", options="header"]
|===
|Question
|Meaning

|Writer
|Which surface may create or update it: bootstrap, domain transformation,
workbook-view configuration, parser readback, adapter, or consumer extension.

|Reader
|Which surface may consume it: renderer, parser, validation, domain cleanup,
reimport, adapter, or consumer extension.

|Kind
|Whether the datum is user/domain intent, carrier observation, or derived
workflow state.

|Lifecycle
|Whether it is persistent, render-consumed, dropped at persistence, dropped if
stale, required, or temporary.

|Conflict rule
|What wins when runtime config, persisted meta, derived meta, and workbook
carrier observation disagree.

|Deletion rule
|When the datum must be removed instead of carried forward.
|===

The expected outcome is a much smaller, lifecycle-aware contract than the
current registry surface. If the Functional Model use cases cannot justify a
metadata family, that family is a deletion or migration candidate.

==== First Contract Sequence

The expected contract order is:

. `personal_data`: minimal JSON/spreadsheet/workbook-view/validation roundtrip.
. Project Memory: XRef matrix editing, ODS, projection cleanup, and plugin
  boundary behavior.
. Worldbuilding: broader real-consumer composition across FK helpers, XRef,
  legends, compact cells, workbook views, discriminator split/merge, and
  validation.

The first domain/meta rewrite slice should be small enough to prove the
bridge. FK-helper roundtrip is the preferred first candidate because it exposes
the current overlap between domain behavior, metadata behavior, workbook view,
helper cleanup, and validation.

Each contract must cite the Functional Model use case it encodes and list
current implementation or consumer workaround behavior that is intentionally
not preserved. Consumers provide coverage evidence; they do not define the
contract by themselves.

Each slice must also have a convergence gate. If the contract cannot be stated
without importing most of the old implementation shape, the slice stops and is
re-scoped before implementation continues.

==== Alternatives Considered

. *Hard reset / greenfield core.* Rejected as the main path.
  It maximizes design freedom but discards too much adapter, parser, renderer,
  CLI, and fixture knowledge.

. *Classic strangler.* Rejected as the main path.
  It minimizes compatibility risk, but this project has little external
  adoption. A long parallel implementation would likely preserve the old
  complexity too long.

. *In-place simplification only.* Rejected as the main path.
  It remains useful for obvious deletions, but the existing domain/meta shape is
  likely to resist the simpler Functional Model rather than express it cleanly.

. *Contract-first domain/meta reset around a stabilized bridge DSL, with adapter
  preservation and targeted deletion.* Accepted.
  It uses the Functional Model as intent source, preserves valuable boundary
  knowledge, allows low-risk in-place deletions, and accepts disruption where
  compatibility is less valuable than deleting accumulated complexity.

==== Consequences

* Breaking changes in domain/meta internals are acceptable and expected.
* Compatibility with historical step names or metadata shapes is not a primary
  design constraint unless an integration contract requires it.
* Adapter and rendering boundaries should be protected as assets, not rewritten
  reflexively.
* The bridge DSL must be made explicit before broad rewrite FTRs begin, but it
  is grown slice by slice. Each slice specifies the part of the bridge it uses;
  the project must not recreate a complete registry-scale DSL up front.
* Functional Model chapters and inquiry notes become inputs to contract tests,
  not passive documentation.
* The lifecycle-attribute model (`drop_at_persistence`, `drop_if_stale`,
  `carrier_takes_precedence`, `required`, `render_consumed`) is a keeper
  hypothesis, not presumed sediment. It should be validated against contracts
  before being kept, simplified, or retired.
* Obvious dead or duplicate historical paths may be deleted in place when the
  risk is low and the deletion does not depend on a new bridge contract.
* The first rewrite FTR must define a test-layer policy: which tests are
  contract-level and should survive the reset, and which tests pin old
  implementation shape and may churn or disappear with that path.
* Coexistence of old and new paths must be short per slice. Once a slice's
  contract passes, the replaced old path should be removed immediately rather
  than left for a later cleanup wave.
* Follow-up work must be represented through FTRs/backlog items. This ADR does
  not itself authorize a broad rewrite without scoped execution tickets.
* Old implementation concepts should not be promoted into the Functional Model
  or Semantic Model solely because they exist in code.

==== Revisit Criteria

Revisit this decision if:

* an external adopter appears whose compatibility needs outweigh internal
  simplification;
* the contract suite cannot cover current consumers without preserving most of
  the old implementation shape;
* the bridge DSL cannot be derived from Functional Model use cases without
  recreating most of the current registry surface;
* adapter preservation proves more expensive than rebuilding the relevant
  boundary behavior;
* the Functional Model expands in a way that makes the domain/meta complexity
  proportionate again.
