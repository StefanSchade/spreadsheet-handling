[[concept-compact-transforms]]
== Compact Transforms — Multi-Layer ⇄ Compact Representation

_A family of bijective transforms between normalized relations (tuple lists,
cross-tables) and a single compact human-editable table where multiple logical
layers are encoded as short tokens inside one cell._

Status: feature concept; the generic base (`XREF-CROSSTABLE`, `CELL-CODEC`,
`COMPACT-MULTIAXIS`) is partially implemented. Quantitative and non-bijective
extensions in the "Future directions" section are not implemented. Related
backlog: `FTR-COMPACT-XREF-USE-CASE-SLICE-P4A2`, `FTR-VALIDATION-COMPACT`.

=== Why this matters

Normalized link tables are good for storage and merging but painful to edit by
hand. A compact matrix view (`U-EU-US` in one cell) is far easier for a human,
but only worth supporting if the transform back to the normalized form is
deterministic and lossless. This concept defines that bijective bridge.

=== The two bijective steps

. *XRef ⇄ Cross-Table* (`expand_xref` / `contract_xref`) — converts tuple
  lists into simple two-dimensional tables; the n = 1 case. Supports both
  2-tuple (flag) and 3-tuple (value) forms. See
  <<concept-crosstab-binary-xref,Crosstab for Editing Binary Cross References>>
  for the binary first slice.
. *Multi-Layer ⇄ Compact Table* (`expand_compact` / `contract_compact`) —
  introduces compact encoding for n ≥ 2 logical layers, each represented by
  short, unique tokens concatenated into a single cell value.

Layer cardinality: n = 1 is the natural case for step 1 (simple cross-tables);
n ≥ 2 triggers step 2 (multi-layer compact encoding); n ≫ 2 is still supported
— usability may degrade, but there is no algorithmic limit.

=== Requirements

Every layer must have a unique short code, a descriptive source name, and a
configurable value set. Both 2-tuple (flag) and 3-tuple (value) representations
are supported. Separators are configurable — `token_sep` between tokens (default
`-`), `kv_sep` between a layer code and its value (optional, default `=`),
`multi_sep` between multiple values within a layer (default `+`). All resolved
configuration values are persisted in `meta` to maintain bijectivity.

=== Heuristic defaults vs. explicit configuration

The system follows a two-level design: basic conventions are inferred
automatically (a column named `ID` is the primary key; `ID (other_table)` is a
foreign-key reference; a sheet name can serve as the default short code for that
layer), and users can override or complement these via declarative YAML, e.g.:

[source,yaml]
----
table:
  name: table_a
  id_col: y
  foreign_ref:
    column: z
    origin: table_b
----

These settings are read once, stored in `meta`, and reused for the inverse
transform.

=== Meta schema for compact encoding

[source,yaml]
----
compact_encoding:
  token_sep: '-'
  kv_sep: '='
  multi_sep: '+'
  sort_order: ['nature', 'edition', 'market']
  derivations:
    - name: availability_implied_by_any_market
      if_any_of_groups: ['market']
      then_add_token: A
      write_policy: omit
  tokens:
    - { code: U,    group: nature,       label: User-driven,  cardinality: 'exactly-one-of:nature' }
    - { code: R,    group: nature,       label: Regulatory,   cardinality: 'exactly-one-of:nature' }
    - { code: Std,  group: edition,      label: Standard,     cardinality: 'exactly-one-of:edition' }
    - { code: Ext1, group: edition,      label: Extension 1,  cardinality: 'exactly-one-of:edition' }
    - { code: EU,   group: market,       label: Europe,       cardinality: 'zero-or-more-of:market' }
    - { code: US,   group: market,       label: United States,cardinality: 'zero-or-more-of:market' }
    - { code: AMEA, group: market,       label: AMEA,         cardinality: 'zero-or-more-of:market' }
    - { code: A,    group: availability, label: Available,    cardinality: 'zero-or-one-of:availability' }
----

=== Concrete example

[cols="1,1,1,1", options="header"]
|===
| feature | product2 | product3 | product1
| abc | U-EU-US | R-EU-AMEA | Std-EU
| def | R-US | U-AMEA | Ext1-EU
|===

Each token is globally unique, so the cell string is a deterministic, reversible
representation of the same tuple set.

=== Edge-case rules

* *Normalization*: sort tokens, deduplicate, apply a uniform case policy.
* *Cardinality*: enforce group constraints; violations produce `meta.issues`.
* *Unknown tokens*: preserved but flagged; never silently dropped.
* *Derived tokens*: applied on read; omitted on write per `write_policy`.

=== Validation and IR integration

Validation lists are materialized only when the total number of allowed token
combinations is ≤ a hard cap (default 64). Beyond that, the IR layer switches to
a decomposed UX with per-layer helper columns and dropdowns. All canonical rules
(token uniqueness, sort order, cardinality) are enforced by meta-driven passes
in the domain layer.

=== Future directions

* A single scalar numeric value per intersection (e.g. `W=0.75`) can later be
  added as an additional token family without breaking bijection.
* Aggregated values (sums, averages, percentages) are *not* bijective and will
  be handled by future non-bijective extraction/aggregation transforms — not by
  the compact-transform family.
* See <<concept-sparse-relation-dense-axis,Sparse Relation / Dense Axis
  Reconstruction>> for the rule governing when a sparse relation may be
  reconstructed back into a dense matrix.
