== Canonical vs. Derived Metadata

_The distinction between `canonical` and `derived` metadata is functional, not
temporal: it is about **what a piece of metadata means**, not **when it was
created**._

=== Why this matters

Several features in this project carry metadata in two roles, and the words
`canonical` and `derived` are easy to misread as "original input" vs. "computed
later". That reading is wrong, and getting it wrong leads to persisting the
wrong things, promoting working state to authoritative semantics, and producing
inconsistent results on the second or third roundtrip. This concept fixes the
vocabulary that every other feature concept reuses.

=== The two roles

*Canonical metadata* describes the stable, spreadsheet-neutral *meaning* the
system wants to preserve — policy, model semantics, roundtrip-relevant intent.
It expresses what the model *means*, it is the authoritative source for stable
semantics, it should be portable across adapters and carrier formats, and it is
a natural candidate for persistence in `_meta` or an equivalent sidecar.

*Derived metadata* describes an operational, materialized, carrier-near *view*
of that meaning in a concrete working state. It expresses how the
current workbook-shaped state is materialized; it supports planning, rendering,
cleanup, diagnostics, or roundtrip continuation; it may depend on pipeline
steps, current sheet state, or adapter/carrier conventions; and it is useful and
sometimes persist-worthy, but it is not automatically the authoritative semantic
source.

=== The clarification that prevents most confusion

Derived metadata does *not* mean "freshly computed during this run". Something
can enter the system as input and still be derived metadata if it describes the
current materialized spreadsheet state rather than the stable model meaning.

Consequently, two things must not be conflated:

* **input is a transport fact** — the system may receive canonical intent,
  persisted derived operational state, carrier hints, or mixed payloads;
* **canonical is a semantic role** — it is not a synonym for "what arrived as
  input".

And persistence alone does not promote derived metadata to canonical: persisted
derived metadata remains derived as long as it still describes materialized
working state rather than stable model intent. Persisting it can still be
correct and useful — e.g. so a later run can bootstrap working metadata,
continue a roundtrip safely, clean up materialized helper columns, or preserve
editing continuity.

=== Concrete example

A spreadsheet starts the workflow with helper columns that already exist, and
accompanying metadata says `_branch_name` is a helper column belonging to
`id_(branch)`. Even though that information *arrived as input*, it describes the
*current materialized workbook state*, not stable model meaning — so it belongs
on the derived/operational side, under a derived namespace in the `_meta`
carrier, not under canonical sheet intent.

By contrast, structured validation rules in `meta_canonical["constraints"]`,
sheet-level intent such as `freeze_header` or `auto_filter` when treated as
stable workbook intent, or a target table's `fk_helpers.default_fields` are
canonical: they remain true even if the current spreadsheet materialization
changes.

=== Practical rule of thumb

When deciding whether a piece of metadata is canonical or derived, ask:

. Does it describe stable model/workbook intent that should remain true even if
  the current spreadsheet materialization changes? → likely *canonical*.
. Or does it describe how the current workbook-shaped state is materialized,
  projected, or interpreted right now? → likely *derived*.

=== How it connects

* The foundational vocabulary for the meta layer (the strict five-criterion
  definition of *canonical*; the orthogonal *model* vs *presentation* vs
  *workbook_identity* classification; the data-layer terms *original* vs
  *derived*) lives in
  `docs/technical_model/ch05_registries/registry_semantics/`. This concept
  doc gives the functional-vs-temporal framing; the ch5 foundation refines
  it into the layer and classification axes the registry rests on.
* Maps onto the existing project split around `meta_canonical`,
  `meta_rendering`, `derived_operational_view`, and read-path carrier hints —
  see `docs/ai_info/architecture_primer.adoc` (§ "meta_canonical vs
  meta_rendering") and `docs/technical_model/ch08_architecture_details/generic_ir.adoc`.
* The same distinction is reused by <<concept-fk-helper-metadata,FK Helper
  Metadata>> (helper capability is canonical; materialized helper columns are
  derived) and by <<concept-sparse-relation-dense-axis,Sparse Relation /
  Dense Axis Reconstruction>> (axis-contract metadata is canonical; a
  materialized sparse relation is derived).
* The `_meta["derived"]["sheets"]` channel that domain transforms write and IR
  render passes read is a concrete derived channel; it is registered as
  transient `meta_rendering` / `derived_operational_view` metadata in
  `registries/meta_registry.yaml`.
