[[concept-fk-helper-metadata]]
== FK Helper Metadata

_Helper columns are redundant, spreadsheet-only columns that materialize fields
from a referenced table next to a foreign-key column; they make editing easier
but must not become normal payload in the normalized repository form._

Status: working concept for the next small helper-column slices. It is *not* a
full metadata model. Some of the metadata shapes below ("target-side helper
capability", first-class persisted helper provenance) are the *direction*, not
yet the current implementation. Related backlog: `FTR-DOMAIN-CONSOLIDATION`
(the FK-helper domain extraction) and the consolidation review clusters C2
(D-002/D-003) and C6 (IR-005).

=== Why this matters

Helper columns sit between two concerns — spreadsheet usability and
normalized, roundtrip-safe data. They are neither pure rendering decoration nor
ordinary domain payload, and treating them as either causes drift: on the second
or third roundtrip, a workbook whose helper columns were materialized by
heuristics (and not persisted) can be interpreted differently because the
carrier state changed. The fix is to make the distinction explicit.

This concept uses the vocabulary of
<<concept-metadata-canonical-vs-derived,Canonical vs. Derived Metadata>>: here
`derived` does not mean "freshly computed this run" — it means "describing the
current materialized workbook state rather than stable model intent".

=== Running example

The `desks` table contains a foreign-key column `id_(employees)` pointing to
`employees.id`. The spreadsheet-facing `desks` sheet may then also show helper
columns such as `_employees_first_name`, `_employees_last_name`,
`_employees_birth_date`. These help a human editing `desks`, but they are
redundant and should not survive into the normalized form unchanged.

=== Two questions that must not be mixed

. *What may be used as helper source data?* — this belongs to the **target
  table** (`employees` in the example): which column is its PK, which columns
  may be exposed as helper fields, which fields are the defaults if no
  FK-specific override is given. This is *canonical* helper intent.
. *Which concrete columns on the current sheet are helper columns?* — this
  belongs to the **table containing the FK** (`desks`): `_employees_first_name`
  is a helper column, it belongs to FK column `id_(employees)`, it points to
  target `employees`, it materializes source field `first_name`. This is
  *derived* operational metadata, not canonical sheet intent.

=== Minimal metadata shapes

Target-side helper capability (canonical):

[source,yaml]
----
sheets:
  employees:
    fk_helpers:
      available_fields: [first_name, last_name, birth_date]
      default_fields: [first_name, last_name]
----

Derived operational helper columns on the FK sheet:

[source,yaml]
----
_meta:
  derived:
    sheets:
      desks:
        helper_columns:
          - column: _employees_first_name
            fk_column: id_(employees)
            target: employees
            value_field: first_name
          - column: _employees_last_name
            fk_column: id_(employees)
            target: employees
            value_field: last_name
----

The important point: the derived shape lives under the `_meta` carrier in a
*derived* namespace, not under canonical sheet intent such as `_meta["sheets"]`.
The canonical `fk_helpers` shape under `_meta["sheets"][...]` is the *direction*
and is not fully implemented today.

=== What is true today, and what is not

Already true: helper enrichment and cleanup exist as real pipeline steps;
multiple helper columns per FK are supported; helper-field order is configurable
and stable; there is step-specific precedence in helper-field resolution; and
`bootstrap_meta` has its own precedence model.

Not yet true: helper cleanup is not yet driven by first-class persisted
helper-column metadata; helper identification still depends mainly on prefixes
and naming conventions; helper provenance is registered only as transient
working metadata, not as durable canonical intent; sheet rename and
helper-column rename are not supported operations.

Current behaviour (operational path): `add_fk_helpers` detects FK columns and
adds helper columns; helper-field selection is driven by step defaults
(`helper_fields_by_fk`, `helper_fields_by_target` by normalized sheet key, then
by original sheet name, `helper_fields`, `label_field` as legacy fallback);
`remove_fk_helpers` removes helper columns by prefix; validation and styling
rely on helper prefixes and materialized column names. `bootstrap_meta` is part
of the broader workbook-metadata direction (precedence: `profile_defaults` →
persisted metadata → `cli_overrides`) but is *not yet* the active source of
helper-field resolution inside `add_fk_helpers`.

=== Precedence is per step

There is no single universal precedence ladder. `add_fk_helpers` resolves from
its own step defaults; `bootstrap_meta` has its own model. A blanket statement
like "user input always wins" would be too crude and is already false for some
cases. The most useful *general* guardrail: heuristics may fill gaps, but
heuristics must not silently contradict existing metadata.

=== Naming conventions: two different jobs

* *FK naming* (`id_(employees)`) helps the system detect that a column is
  intended as a foreign key.
* *Helper naming* (`_employees_first_name`) helps distinguish derived helper
  columns from ordinary payload and helps cleanup avoid collisions.

Both are conventions, but they solve different problems and should not be merged
into one vague rule. Neither is the final model; both are practical detection
aids for the current phase.

=== First conservative policy

For the next slice a restrictive contract is acceptable: helper columns present
redundant information for easier editing; manual edits to helper *values* are
not treated as source truth; helper values may be ignored or overwritten on
re-enrichment; renaming helper columns is unsupported; renaming relevant sheets
is unsupported. Not great UX, but a feasible first contract.

Cleanup direction: `remove_fk_helpers` removing by prefix is good enough for
now but is the main conceptual weakness; longer term, cleanup should trust
explicit helper-column metadata first and keep prefixes as fallback aids.

=== Target model (v2): relation-oriented FK helper policy

Status: *partially live during transition*. The v2 relation policy shape
is *now produced* by configuration steps:

* `infer_fk_relations` writes v2 only (heuristic policy);
* `configure_fk_helpers` writes the v1 per-target shape that current
  primitives still consume, *and* additionally writes v2 relation
  entries through a deterministic transitional bridge (see "Transitional
  v1 -> v2 bridge" below).

The v2 shape is *not yet consumed* by primitives: `add_fk_helpers`,
`remove_fk_helpers`, `validate_fk_helpers`, and `reorder_fk_helpers`
continue to read the v1 per-target shape and convention-based detection.
Producing v2 from configuration steps is owned by
`FTR-INFER-FK-RELATIONS-CONFIGURATION-STEP-P5`; consuming v2 in
primitives is owned by `FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5`.
The shape itself was defined by
`FTR-FK-RELATION-POLICY-MODEL-P5`.

The v2 shape sits under the existing root `_meta.helper_policies.fk`
(no new adjacent root) and carries a top-level `schema_version` marker so
consumers can detect the shape unambiguously. Relation identity is per
FK column on a specific source frame -- the `(source_frame,
source_column)` pair -- because two sheets may reference the same target
frame with different helper choices.

[source,yaml]
----
_meta:
  helper_policies:
    fk:
      schema_version: 2
      relations:
        - source_frame: Products
          source_column: id_(product_manager)
          target_frame: ProductManagers
          target_key: id
          helper_fields:
            - name
          helper_columns:
            - column: _product_manager_name
              target_field: name
          helper_prefix: "_"
          produced_by:
            step: infer_fk_relations
            mode: naming_convention
----

An explicit / manual relation written by `configure_fk_helpers` (target
state) uses the same shape and records `produced_by.step:
configure_fk_helpers`, `produced_by.mode: explicit`.

==== Field schema

`target_key` is mandatory: it is the PK column on `target_frame` against
which `source_column` joins. Storing it in policy means primitives,
validators, and reimport cleanup never need to re-derive it from
`build_registry(...)["id_field"]` defaults.

`helper_columns[].column` records the literal helper-column header that
materialization writes. `helper_columns[].target_field` records the
target column it reads. Pairing them in policy lets validation check
helper values against the target without consulting either the FK column
convention or the helper-prefix convention.

`produced_by.step` plus `produced_by.mode` make the entry's origin
explicit so consumers can distinguish manual configuration from inferred
configuration; lifecycle FTRs use the same fields to decide ownership
when a later run wants to rewrite an entry.

For the full per-field table (required / optional, type, semantics, and
which step produces each field) see the *Relation entry: field schema
(v2)* subsection of
`docs/backlog/FTR-FK-RELATION-POLICY-MODEL-P5.adoc`.

==== Four distinctions that must stay separate

The v2 model only earns its keep if these four concerns are not
collapsed. They are listed in the order metadata flows through the
pipeline:

. *Configuration / inference produces resolved relation policy.* Two
  configuration steps write `_meta.helper_policies.fk.relations`:
  `configure_fk_helpers` (manual / explicit policy) and
  `infer_fk_relations` (heuristic policy). Both are now implemented;
  both write the same shape under the same root; both behave as
  `configuration` steps in the pipeline registry sense; a pipeline that
  runs both must fail clearly when they collide on the same relation
  key (`(source_frame, source_column)`). `apply_v2_relations` also
  refuses *same-producer* rewrites that diverge in any meaningful field;
  same-producer rewrites that are semantically identical are accepted
  idempotently.
. *Primitive FK-helper execution consumes resolved relation policy.*
  The primitive steps (`add_fk_helpers`, `validate_fk_helpers`,
  `remove_fk_helpers`, `reorder_fk_helpers`) read policy and derived
  helper provenance; they do not run inference. Missing policy is a
  clear error whose message names the configuration / inference step
  that should have run earlier. Encoding this is owned by
  `FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5`.
. *Derived helper provenance records materialized helper-column
  state.* The per-run record under
  `_meta.derived.sheets.*.helper_columns` (registered as the `derived`
  entry in `meta_registry.yaml`) is *not* the policy. It is what the
  primitive wrote when it last enriched the source frame. The
  producer-side enrichment of that record with `target_key` is owned
  by `FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5`; its lifecycle
  (pruning, stale / orphan handling, schema validation) is owned by
  `FTR-DERIVED-PROVENANCE-LIFECYCLE-P4A`.
. *Wire-format FK column naming is independent of inference and
  execution.* The cell-level convention `id_(<target>)`
  (`core/fk.py:FK_PATTERN`) is how cross-sheet references survive XLSX /
  ODS roundtrips. It is consumed by the parser / writer and by
  `infer_fk_relations` as one heuristic mode among future modes. It is
  *not* consumed by primitive FK-helper execution for relation
  inference. Removing primitive inference is not the same as removing
  the wire format.

==== Transition state: what is now live and what still consumes v1

As of `FTR-INFER-FK-RELATIONS-CONFIGURATION-STEP-P5` the runtime
produces v2 from configuration steps, but the FK-helper *primitives*
have not yet been refactored. Concrete state:

* `infer_fk_relations` writes v2 only (`schema_version: 2` plus a
  `relations` list under `_meta.helper_policies.fk`); the
  `targets: "auto"` heuristic mode on `configure_fk_helpers` is removed
  and now raises with a migration message.
* `configure_fk_helpers` writes both:
+
** the v1 per-target shape under
   `_meta.helper_policies.fk.<target_key>` for current primitive
   consumers:
+
[source,yaml]
----
_meta:
  helper_policies:
    fk:
      <target_key>:
        target: <target_key>
        target_sheet: <sheet_name>
        key: <id_field>
        allowed_helpers: [...]
        default_helpers: [...]
        helper_prefix: "_"
        fk_column: id_(<target>)
        label: <optional>
----
+
** the v2 relation list (one entry per `(source_frame, source_column)`
   pair carrying the resolved `fk_column`) for v2-aware consumers.
* `domain.transformations.fk_helpers.enrich_helpers` (the
  `add_fk_helpers` primitive) still reads the v1 per-target shape;
  primitive detection of FK columns still uses `FK_PATTERN`;
  `remove_fk_helpers` prefers derived provenance and falls back to the
  helper prefix.

==== Transitional v1 -> v2 bridge inside `configure_fk_helpers`

`configure_fk_helpers` keeps its v1 / target-oriented *input surface*
(per-target `key`, `allowed_helpers`, `default_helpers`, `fk_column` /
`fk_column_pattern`, etc.). To populate v2 it runs a bounded,
deterministic *bridge*: for each resolved v1 policy entry, it scans
current data frames and emits one v2 relation per source frame that
carries the resolved literal `fk_column` header.

Bridge invariants:

* No source frame is inferred from naming heuristics. The bridge only
  matches data frames whose column header equals the configured
  `fk_column` exactly.
* If no current data frame carries the configured `fk_column`, the
  bridge emits no v2 entry for that policy (v1 is still written).
* If multiple current data frames carry the configured `fk_column`, the
  bridge emits one v2 relation per source frame; relation identity is
  `(source_frame, source_column)`.

This bridge is *transitional compatibility* for the period in which
configuration steps already produce v2 but primitives still consume v1.
`FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5` owns *narrowing or
removing* the bridge: once primitives consume v2, the natural
endpoints are to either drop the bridge entirely and require explicit
v2 source identity on `configure_fk_helpers` callers, or keep it only as
a backwards-compat shim behind an opt-in flag. The bridge is *not* a
sanctioned mechanism for heuristic inference inside primitives or
elsewhere; that responsibility belongs to `infer_fk_relations`.

The registry artifacts
(`docs/technical_model/ch05_registries/meta_registry/meta_registry.yaml`,
`docs/technical_model/ch05_registries/pipeline_step_registry/pipeline_step_registry.json`)
reflect this transitional truth: v1 is retained as the *current
primitive consumer* contract, v2 is the *target producer shape* now
written by configuration steps, and removing or narrowing the v1+v2
coexistence is owned by `FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5`.
