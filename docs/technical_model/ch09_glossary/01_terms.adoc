== Terms

The *category* column groups entries by which subsystem the term belongs to.
The *meta-system* category covers the foundation vocabulary for the meta
registry; full definitions live in
`docs/technical_model/ch05_registries/registry_semantics/`.

[cols="2,2,10", options="header"]
|===

|
term
|
category
|
explanation

|
pipeline
|
pipeline
|
User-configured sequence of frame transformations and validations. The
pipeline operates on frames and metadata; it does not own spreadsheet
rendering.

|
flow
|
render
|
Sequence of rendering steps in the Render IR layer that turns selected
frames and canonical metadata into backend-neutral render operations.

|
frame
|
data
|
Domain-level tabular data object carried in `Frames`, normally a pandas
`DataFrame` keyed by frame name. A frame is not a workbook sheet.

|
canonical tuple frame
|
data
|
Long-form source or target frame whose configured key columns identify
each semantic tuple. Matrix/crosstable views should be recomposable into
this shape.

|
matrix frame / crosstable frame
|
data
|
Domain-level pivoted frame where row identity remains in columns and a
configured matrix axis becomes payload columns. It may later be projected
to a sheet, but it is not itself a sheet.

|
discriminator
|
data
|
Configured column or value axis used to partition a canonical tuple frame
into discriminator-specific frames, for example locale, product, customer,
deployment mode, or software variant.

|
discriminator-specific frame
|
data
|
Frame produced by applying a discriminator split. It carries one configured
discriminator value implicitly through split metadata or recomposition
config.

|
matrix axis / pivot axis
|
data
|
Configured values that become columns in a matrix/crosstable frame and
become the `column_key` values again when expanded back to tuples.

|
payload value
|
data
|
Editable semantic cell value carried in the canonical tuple frame and
displayed in matrix-frame payload cells. Helper/display columns are not
payload unless explicitly configured as such.

|
workbook view
|
render
|
Rendering policy that selects, orders, names, and annotates frames for
spreadsheet projection. It belongs at the projection boundary, not inside
domain transformation semantics.

|
sheet
|
render
|
Concrete spreadsheet tab in a workbook. Sheets are render/readback
artifacts; domain transformations should refer to frames unless they are
explicitly configuring workbook projection.

|
render IR
|
render
|
Backend-neutral spreadsheet intermediate representation (`WorkbookIR`,
`SheetIR`, `TableBlock`, `RenderPlan`) used between selected frames and
concrete spreadsheet backends.

|
backend
|
render
|
I/O adapter for a concrete storage format such as JSON, CSV, XLSX, or ODS.
Spreadsheet backends render or parse through the spreadsheet contract and
Render IR.

|
original (data)
|
data
|
Data-layer status: the cell content is the authoritative source of its
information. Editing it changes the truth the system holds about the model.
See `ch05_registries/registry_semantics/01_data_and_meta.adoc`.

|
derived (data)
|
data
|
Data-layer status: the cell content is a projection or copy of information
that lives elsewhere. Editing it does not change the underlying truth — the
system either ignores the change, restores the projected value on the next
render, or treats the edit as a signal that the original source should be
reviewed. Distinct from "derived" at the meta layer; see
`ch05_registries/registry_semantics/01_data_and_meta.adoc`.

|
canonical (meta)
|
meta-system
|
Meta-layer role: a meta claim that expresses intent or property, is the
authoritative source for its information, is spreadsheet-neutral,
persistable, and lifecycle-decoupled from any single run. *Canonical*
applies only to meta claims, never to data. Full definition in
`ch05_registries/registry_semantics/02_meta_layer.adoc`.

|
derived (meta)
|
meta-system
|
Meta-layer role: the current-run realization of a canonical claim or
inferred state. Not authoritative; recomputable from canonical sources.
Examples: `__freeze`, `__autofilter`, `__style`, the
`_meta["derived"]["sheets"]` namespace. Distinct from "derived" at the data
layer.

|
operational-carrier
|
meta-system
|
Meta-layer role: the carrier-shaped representation of canonical (or
derived) meta — the form in which meta is written to a concrete backend or
read back from it. Examples: `workbook_meta_blob`, `_hidden`,
`__autofilter_ref`, `__legend_blocks`. Not a second canonical source.

|
model meta
|
meta-system
|
Meta classification: claims about durable structural properties of the data
— FK relationships, helper status, validation rules, lifecycle, codec
configuration, transformation configuration. Losing model meta is a data
integrity issue. Full definition in
`ch05_registries/registry_semantics/03_meta_classification.adoc`.

|
presentation meta
|
meta-system
|
Meta classification: claims about display intent — column widths, freeze
panes, fill colors, autofilter. Persistable and roundtrip-relevant, but
distinct from model meta because the consequences of loss are aesthetic
rather than structural; conflict resolution and adapter portability
requirements are also weaker than for model meta.

|
workbook_identity meta
|
meta-system
|
Meta classification: claims about the workbook itself as a named, versioned
artifact (`version`, `author`, `exported_at`). Neither a model claim nor
display intent.

|
carrier
|
meta-system
|
The concrete backend persistence target — for the current XLSX and ODS
adapters, the hidden `_meta` sheet plus the visible table cells. The
foundational definition (what counts as a carrier; where carrier ends and
IR begins) is still open; see
`ch05_registries/registry_semantics/04_carrier.adoc`.

|===
