== Carrier

A *carrier* is the concrete backend persistence file produced or consumed
by an I/O adapter. For the current XLSX and ODS adapters, the carrier is
the workbook file itself (`.xlsx`, `.ods`). Each adapter materializes the
carrier through its own object model (`openpyxl`, `odfpy`), but the object
model is not the carrier — it is the in-memory representation of the
carrier during write or parse.

The narrow definition is deliberate. Earlier project usage applied
"carrier" loosely to several distinct things, and that conflation is part
of what made the meta vocabulary hard to use.

=== Carrier-shape — the small layer above the carrier

A meta entry is in *carrier-shape* when it has been translated into the
serialized form that the carrier can store. For the current adapters this
form is primarily the deterministic JSON written into the hidden `_meta`
sheet, plus a small set of carrier-specific control artifacts (the
`_hidden` marker, the read-path reconstruction hints).

Carrier-shape is a thin layer that sits *between* the canonical meta and
the carrier itself. It exists because canonical meta is spreadsheet-neutral
by definition, while the carrier has format constraints — a hidden sheet
expects rows of strings, an ODS settings element expects XML. Carrier-shape
is the form in which canonical meta gets reduced to something the carrier
file format can hold.

The *operational-carrier* layer in the meta system (`02_meta_layer.adoc`)
refers to meta entries that exist in carrier-shape form. The
operational-carrier layer *is* carrier-shape; the carrier itself is the
file that the carrier-shape is written into.

=== What carrier is not

* *Not WorkbookIR / SheetIR / RenderPlan.* The IR is backend-neutral and
  exists independently of any carrier. The IR can be materialized into any
  number of carriers without being itself a carrier.
* *Not the canonical role.* Canonical is a role a meta claim plays in the
  authority graph (`02_meta_layer.adoc`); it is not tied to any one
  carrier. Canonical meta exists logically before it is reduced to
  carrier-shape.
* *Not the visible cell data.* Cell content is written through the carrier
  but is a data-layer concern (`01_data_and_meta.adoc`), not a carrier
  concern. The same data can be materialized into many carriers.
* *Not the openpyxl / odfpy object model.* That is the adapter's in-memory
  representation of the carrier during read or write; the carrier proper
  is the file on disk.

=== Historical drift (why the narrow definition is needed)

"Carrier" has been used in current docs and code comments for at least
five distinct things:

. The backend file (`.xlsx`, `.ods`) — the narrow definition above.
. The hidden `_meta` sheet inside that file.
. The JSON-serialized payload stored in the hidden `_meta` sheet.
. The `openpyxl` / `odfpy` in-memory object model used during read or
  write.
. Any backend-near hint or artifact, including read-path reconstructions.

The narrow foundation collapses (2) and (3) into *carrier-shape* (the
serialized form prepared for the carrier), treats (4) as the adapter's
working representation rather than the carrier itself, and treats (5) as
meta in the operational-carrier layer — the role being distinct from the
carrier being a concrete file.

=== Mapping to the live registry

The live registry uses `classification: carrier_artifact` and
`classification: read_path_hint` for entries that, under the foundation
vocabulary, are in carrier-shape:

[cols="2,3,3", options="header"]
|===
| Live entry | Foundation reading | Notes

| `workbook_meta_blob`
| operational-carrier layer; carrier-shape representation of the canonical
  workbook-meta payload.
| The JSON blob is *not* the carrier; it is the form in which canonical
  meta is reduced for writing into the carrier (the hidden `_meta` sheet
  inside the `.xlsx` or `.ods` file).

| `_hidden`
| operational-carrier layer; carrier-shape control flag.
| A marker that a `SheetIR` belongs to the hidden meta-carrier set, used
  by plan emission to keep carrier-shape sheets separate from visible
  sheets.

| `__autofilter_ref`
| operational-carrier layer (read-path); carrier-shape reconstructions
  parsed from an existing carrier.
| Read-path counterparts of carrier-shape; reconstructed from the carrier
  on parse, not emitted on write.
|===

Re-encoding the YAML to use *carrier* and *carrier-shape* as distinct
terms is deferred to the same future FTR that handles the
layer/classification relabeling. The foundation here is the conceptual
model.

=== Remaining open question

The foundation here treats *write-shaped carrier artifacts* (e.g.
`workbook_meta_blob` on the write path) and *read-path reconstructions*
(e.g. `__autofilter_ref`) as a single operational-carrier layer with two
sub-modes. The live registry separates them as `carrier_artifact` vs
`read_path_hint`. Whether the foundation should also separate them — i.e.
whether "carrier-shape on write" and "carrier-shape on read" deserve
distinct foundation terms — is left open. It is a question about whether
the read/write asymmetry is structural or merely directional, and nothing
in the current architecture forces an answer yet.
