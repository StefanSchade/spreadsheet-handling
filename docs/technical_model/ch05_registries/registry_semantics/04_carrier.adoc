== Carrier

A *carrier* is the concrete persistence or transport shape in which meta is
written and read back. The term is used inconsistently in current docs
("persistence carrier", "carrier-near", "carrier-side", "carrier hint",
"carrier artifact") and needs a foundational definition.

The carrier-semantics foundation is *not yet written*. The open questions
identified during the REVIEW-002 follow-up are tracked in the discussion
that produced the rest of this subsection and will be folded into this file
once the underlying decisions have been made.

=== Provisional working definition

Until the foundation is written, this file uses the following provisional
reading:

* The *carrier* is the concrete backend persistence target — for the
  current XLSX and ODS adapters, the hidden `_meta` sheet plus the visible
  table cells.
* The *operational-carrier* layer (see `02_meta_layer.adoc`) refers to
  meta entries that exist in carrier shape — most prominently
  `workbook_meta_blob` for the canonical payload and the various
  `__*_ref` read-path hints.
* Where existing documentation says "carrier-near" or "carrier-side", read
  it as "in the operational-carrier layer".

=== Known open questions

. *What counts as a carrier?* The hidden `_meta` sheet itself, the
  JSON-serialized blob inside it, the openpyxl object model, or all
  three at different levels? A clean foundation will likely treat the
  carrier as a *logical persistence interface* that each adapter
  materializes differently (a hidden sheet for XLSX/ODS, a sidecar file
  for a hypothetical JSON adapter, an XML element for a hypothetical
  ODS-settings adapter).
. *Where does carrier end and IR begin?* WorkbookIR/SheetIR is the
  backend-neutral intermediate representation; the carrier is the
  backend-specific persistence shape. The seam is enforced by the
  spreadsheet-contract boundary, but the meta vocabulary has not yet
  consistently named which side of that seam each meta entry sits on.
. *Are read-path hints carrier artifacts or a separate role?* The live
  registry distinguishes `carrier_artifact` (write-shaped) from
  `read_path_hint` (read-shaped). The foundation here folds both into
  `operational-carrier`; whether that conflation is right or whether the
  read-write asymmetry deserves its own foundation term is open.
