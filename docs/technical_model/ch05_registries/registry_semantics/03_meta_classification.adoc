== Meta Classification

The *classification* axis classifies meta by its *subject matter* — what the
meta claim is about. The classification is orthogonal to the layer axis
(canonical vs derived vs operational-carrier).

Three classification values are recognized:

* `model`
* `presentation`
* `workbook_identity`

=== Model

*Model meta* makes claims about *durable structural properties of the data*
— relationships between columns and frames, validation rules, helper
relationships, transformation configuration,
codec parameters. Losing a model-meta claim is a *data integrity* issue: the
next parse may interpret the workbook incorrectly, or a downstream
transformation may produce wrong results.

Model meta is the canonical case of "canonical": the historical informal
notion was almost always about model claims. Examples in the live registry:

* `helper_policies`, `helper_columns`, `helper_prefix` — what is a helper,
  which columns, how to resolve them.
* `workbook_view`, `workbook_view.sheet_mappings` — which prepared frames are
  exposed as sheets and how visible sheets map back to logical frames.
* `constraints` — structured validation rules.
* `protection` — editability declarations.
* `legend_blocks` — codebook definitions.
* `xref_crosstable`, `split_by_discriminator`, `sparse_defaults`,
  `compact_multiaxis` — transformation configuration. (The former
  `cell_codecs` family is retired: Cell Codec intent lives in explicit
  step configuration; legacy payloads are tolerated pass-through
  sediment.)

=== Presentation

*Presentation meta* makes claims about *display intent* — how the user
wants the workbook to look when rendered. Losing a presentation-meta claim
is an *aesthetic* issue: the workbook still parses correctly and the model
is intact, but the result does not match the user's display preferences.

Presentation claims can still be canonical on the layer axis (durable,
authoritative, persistable, adapter-portable, lifecycle-decoupled). The
distinction from model is the *consequence* of losing them, which in turn
affects conflict-resolution rules, roundtrip strictness, and adapter
portability requirements.

The motivating case for treating presentation as canonical-layer at all: a
user who explicitly sets a column width expects that width to survive a
roundtrip, even though the width is not a property of the data. To survive
the roundtrip, the read path must write the width back into the meta
payload that persists across cycles — which puts it on the canonical
layer. The classification axis (presentation rather than model) then
records the fact that the claim is about display intent, not data
structure. Conflating the two led to widths being treated by the same
conflict-resolution and adapter-portability rules as `helper_policies`,
which is too strict for one and too loose for the other.

Examples in the live registry:

* `column_widths` — user-authored column display widths.
* `freeze_header` — frozen-pane request.
* `auto_filter` — autofilter request.
* `header_fill_rgb`, `helper_fill_rgb` — header and helper fill colors.

==== Why this matters operationally

Treating model and presentation meta as a single bucket masks four real
behavior differences:

. *Conflict-resolution policy*: two conflicting `helper_policies` are a
  real semantic conflict requiring careful resolution. Two conflicting
  `header_fill_rgb` values are last-writer-wins, no questions asked.
. *Roundtrip strictness*: losing model meta breaks the parse contract.
  Losing presentation meta degrades visual fidelity but preserves the
  model.
. *Adapter portability*: model meta must be portable across all spreadsheet
  backends or model meaning is lost. Presentation meta can degrade
  gracefully — a CSV adapter that drops fills is acceptable; a CSV adapter
  that drops FK-helper structure is not.
. *Producer profile*: model meta is typically produced by configuration
  steps and domain transformations. Presentation meta is typically produced
  by user authoring, bootstrap defaults, or read-path reconstruction.

The foundation vocabulary names these as separate classifications so that
future work on conflict/precedence, adapter graceful-degradation, and
roundtrip policy can apply different rules to each without re-deriving the
distinction every time.

=== Workbook_identity

*Workbook_identity meta* makes claims about the *workbook itself* as a
named, versioned artifact — not about the data inside it and not about how
to display it. These are durable, persistable, and roundtrip-relevant, but
they are neither model claims nor display intent.

Workbook_identity is a small bucket. It exists because folding these
entries into `model` would mislabel them ("`version` is not a structural
property of the data") and folding them into `presentation` would be wrong
outright.

Examples in the live registry:

* `version` — workbook-level version marker.
* `author` — workbook-level author marker.
* `exported_at` — workbook-level export timestamp.

=== Open questions

. The live `sheets` namespace under which per-sheet workbook meta lives is
  borderline. The namespace itself does not make a claim; entries under it
  do. The foundation here proposes treating namespaces as *structural*
  rather than classified — only the entries inside the namespace carry a
  classification. Under that treatment, `sheets` is not a
  workbook_identity entry but a structural carrier with no classification
  of its own.
. Some entries genuinely straddle two classifications. `protection`
  declares both editability semantics (model) and lock styling
  (presentation). The current proposal treats `protection` as model because
  the editability semantics dominate, and treats any presentation aspect
  as a downstream rendering concern. Entries that straddle two
  classifications should be revisited case-by-case rather than introducing
  a "multi-classification" mechanism.

=== Mapping to the live registry

The live registry does not yet encode the classification axis described
here; today's `classification` YAML field encodes the layer axis instead.
The foundation classification is currently *implicit* in the live registry
— it can be read off the entry's subject matter — but it is not a YAML
field. Re-encoding the YAML to expose the foundation classification
explicitly is deferred to a separate FTR.
