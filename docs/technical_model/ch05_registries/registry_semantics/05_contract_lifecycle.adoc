== Contract Lifecycle

The *contract lifecycle* axis describes how a registry entry's *contract*
evolves across producer / consumer migrations. It separates three things
that the project has historically blurred when contracts were in motion:

. *Current state* -- what runtime producers and consumers actually do
  today.
. *Target state* -- what the contract is meant to be after the planned
  change.
. *Transition bridge* -- the temporary mechanism that lets current and
  target shapes coexist during the migration, and the condition under
  which the bridge is removed.

This axis is *opt-in*. Most registry entries describe a single stable
contract and do not need lifecycle fields. Lifecycle fields are declared
only on entries that genuinely span a current / target / transition
ambiguity. Absence of `contract_lifecycle` on an entry is the default and
does not imply non-compliance.

[NOTE]
====
This axis is *orthogonal* to the existing meta vocabulary:

* It is distinct from the *layer* axis (`canonical`, `derived`,
  `operational-carrier`; see `02_meta_layer.adoc`). A claim remains
  canonical / derived / operational-carrier whatever its lifecycle state
  is; a canonical claim can be canonical *and* in transition between two
  shapes.
* It is distinct from the *classification* axis (`model`, `presentation`,
  `workbook_identity`; see `03_meta_classification.adoc`).
* It is distinct from *backend capability profiles* (the
  `accepted initial gap` vocabulary in
  `spreadsheet_backend_capability_profiles.adoc`). Lifecycle status names a
  producer / consumer contract migration state; capability-profile
  vocabulary names a backend's gap relative to an IR operation. The two
  are loosely analogous but live on different axes and must not share a
  status field.
* It is distinct from *workbook `_meta` payload lifecycle* (e.g. derived
  helper provenance pruning, owned by
  `FTR-DERIVED-PROVENANCE-LIFECYCLE-P4A`). The contract lifecycle
  describes the *shape of the contract a registry entry documents*, not
  the runtime lifetime of a `_meta` payload.

Do not use `canonical` as a synonym for `target_state`, and do not use
`accepted_gap` (or any other capability-profile term) as a bridge status.
====

=== Field model

A registry entry that opts into the lifecycle model carries a single
`contract_lifecycle` block. The block uses the smallest field set that
remains truthful, guardable, and readable by a fresh agent.

[source,yaml]
----
contract_lifecycle:
  scope: <optional namespace inside the entry the lifecycle applies to>
  notes: <optional prose; not parsed by guards>

  current_state:
    shapes:
      <shape_token>: <short prose meaning; not parsed by guards>
      ...
    runtime_writes:
      - <shape_token>
    runtime_reads:
      - <shape_token>
    producers:
      - <code reference or registered step name>
    consumers:
      - <code reference or registered step name>

  target_state:
    preferred_shape: <shape_token>
    owning_ftr: FTR-<...>
    target_consumers:
      - <code reference or registered step name>

  transition_bridge:
    status: none | planned | active | retired
    bridge_type: dual_write | dual_read | adapter | mapping |
                 rejection_with_migration_error | manual_review_only
    source_shape: <shape_token>
    target_shape: <shape_token>
    reason: <prose; not parsed by guards>
    bridge_rule: <prose; not parsed by guards>
    identity_derivation: explicit_new_fields |
                         deterministic_normalization |
                         temporary_bridge |
                         rejection_with_migration_error |
                         no_migration |
                         not_applicable
    identity_derivation_notes: <optional prose; not parsed by guards>
    removal_ftr: FTR-<...>
    removal_condition: <prose; not parsed by guards>
----

=== Field semantics

==== Shape tokens

A *shape token* is a short symbolic identifier for a producer / consumer
contract shape (for example `v1_per_target_policy`, `v2_relation_policy`).
Tokens are entry-local and need not be globally unique. Each token used
under `runtime_writes`, `runtime_reads`, `target_state.preferred_shape`,
`transition_bridge.source_shape`, or `transition_bridge.target_shape`
must appear under `current_state.shapes` so the entry is internally
consistent. The `shapes` map is the place where prose explains what a
token means.

==== `current_state`

Records the *runtime truth* at the time the entry is reviewed. Lists
producers and consumers as dotted Python module / symbol paths relative
to `spreadsheet_handling` when possible, or registered pipeline step
names when the producer / consumer is the step itself. Producers that
write more than one shape are listed once; the shapes they write are
captured in `runtime_writes`. Target-only consumers belong under
`target_state.target_consumers`, not here.

==== `target_state`

Records the *contract intent* the project is moving toward. The
`preferred_shape` is the shape the target consumers will read; the
`owning_ftr` is the FTR id that is expected to make that target true at
runtime. Listing `target_consumers` separately from `current_state.consumers`
prevents target-only consumers from being read as current readers of the
target shape.

==== `transition_bridge`

Records the *temporary mechanism* that lets current and target shapes
coexist. `status` is a small closed enum:

[cols="1,3"]
|===
| Status | Meaning

| `none`
| No bridge is declared; the entry is single-shape today. Allowed for
  entries that opted in for documentation reasons only.

| `planned`
| A target contract is documented; no bridge has shipped yet.

| `active`
| The bridge is live at runtime.

| `retired`
| The bridge has been removed; the entry is single-shape again. Retired
  bridges must not still describe current consumers as reading the
  retired source shape.
|===

`bridge_type` is also a small closed enum (`dual_write`, `dual_read`,
`adapter`, `mapping`, `rejection_with_migration_error`,
`manual_review_only`).

`identity_derivation` records *how new identity is derived from an old
shape* when the contract shape changes (for example when v1 is keyed
per-target and v2 is keyed by source frame + source column). It uses a
small closed enum (`explicit_new_fields`, `deterministic_normalization`,
`temporary_bridge`, `rejection_with_migration_error`, `no_migration`,
`not_applicable`). Naming the mechanism in structured form keeps future
agents from silently widening a bounded bridge into general heuristic
inference.

`removal_ftr` names the FTR that owns narrowing or removing the bridge;
the id must match the project's `FTR-...` pattern. `removal_condition`
explains in prose when the bridge should be removed (a closed runtime
contract, a test demonstration, a registry update). Date-only
removal conditions are intentionally not modeled here; FTR-id-based,
consumer-state-based, or test-based conditions fit this repository
better.

`reason`, `bridge_rule`, `identity_derivation_notes`, and
`removal_condition` are prose fields and are *not* parsed by guards. They
exist for reviewers and future agents.

=== Documented public surface

The contract lifecycle model deliberately covers *runtime producer /
consumer state* only. It does not model *documented public surface* state
(what the user-guide transformation catalog, README, or pipeline-step
registry `purpose` claims about a contract). Public-surface state is
handled separately:

* The pipeline-step registry's `purpose` and per-meta-ref `notes` fields
  carry transition wording when a step's documented contract is in
  motion.
* The user-guide transformation catalog is realigned by an explicit
  realignment FTR after the runtime change has landed.
* The public-surface approval gate in
  `docs/ai_info/interfaces_and_gates.adoc` is the discipline that keeps
  documented-public-surface changes deliberate.

If a future entry has documented-public-surface state that drifts
independently from runtime state, that entry's FTR must state how the
drift is recorded; the lifecycle model above is not the right place for
it.

=== Pipeline step registry

The lifecycle model above is *not* propagated to
`pipeline_step_registry.json`. Step-surface transition state already
lives there as prose under per-step `purpose` plus per-meta-ref `notes`
and is governed by the same FTR that owns the runtime change. Adding a
parallel lifecycle block to step entries would duplicate that wording
without reducing ambiguity. If a future step transition needs structured
lifecycle fields, it should reuse the field model defined here rather
than invent a parallel one.

=== When to opt in

Add `contract_lifecycle` to a registry entry when *all* of the following
are true:

. The entry's contract is in a real, already-approved transition between
  a current shape and a target shape.
. Current runtime and target intent can be confused by a fresh agent
  reading the entry without surrounding FTR prose.
. A bridge or migration mechanism exists and has a named owning FTR.

Do not add `contract_lifecycle` to entries with a stable single-shape
contract, and do not add it speculatively for transitions that have not
been authorized. The lifecycle model is a truthfulness tool, not a
governance framework, and it must never become a compatibility
straightjacket that prevents an explicitly approved "rename and migrate"
decision.

=== Guard scope

A narrow architecture guard (see
`tests/architecture/semantic_invariants/test_meta_registry_contract_lifecycle.py`)
validates the structure of `contract_lifecycle` blocks where they are
declared. The guard:

* validates only entries that opt in to `contract_lifecycle`;
* fails on target state without distinguishable current state;
* fails on an active bridge that omits any of `bridge_type`,
  `source_shape`, `target_shape`, `reason`, `removal_ftr`, or
  `removal_condition`;
* fails when `runtime_reads` and `runtime_writes` reference different
  shapes without a non-retired bridge;
* fails when a retired bridge still lists its `source_shape` under
  `runtime_reads`;
* fails on bridge status values outside the closed lifecycle enum
  (including any capability-profile term such as `accepted_gap`);
* fails when `removal_ftr` does not match the `FTR-...` id pattern;
* fails when a shape token used under `runtime_writes`, `runtime_reads`,
  target, or bridge is not defined under `current_state.shapes`.

The guard reads only structured fields. It does *not* parse `reason`,
`bridge_rule`, `identity_derivation_notes`, or `removal_condition`. It
does *not* require lifecycle fields on entries that have not opted in,
and it does *not* validate `pipeline_step_registry.json`.
