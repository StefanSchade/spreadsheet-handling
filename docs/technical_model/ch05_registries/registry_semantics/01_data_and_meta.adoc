== Data and Meta

The vocabulary in this chapter applies at two distinct levels. Confusing them
is the root cause of most semantic drift in the project's metadata language.

=== The data layer

The *data layer* is the cell content carried by frames and rendered into
spreadsheet cells: numbers, strings, dates, the values a user actually edits.
At the data layer the relevant distinction is *original* versus *derived*.

*Original data* is the authoritative source of its information content.
Editing an original cell changes the truth that the system holds about the
model. The row in `id_(branch)` that defines branch 17 is original; if the
user changes branch 17's label from "Berlin" to "Hamburg" in that row, the
model now says branch 17 is Hamburg.

*Derived data* is a projection or copy of information that lives elsewhere.
Editing a derived cell does *not* change the underlying truth — the system
either ignores the change, restores the projected value on the next render,
or treats the edit as a signal that the original source should be reviewed.
The label "Berlin" appearing in a `_branch_name` helper column next to a
foreign key is derived data: it is a redundant rendering of information
whose authoritative form lives on the branch row.

The words *canonical* and *non-canonical* do not apply at the data layer.
They have historically been used informally to mean "original" and "derived"
data, but that conflates the data-layer distinction with a separate
distinction at the meta layer. The two distinctions are related but not
identical.

=== The meta layer

The *meta layer* consists of claims the system holds *about* the data, the
workbook, or its presentation. Examples:

* "The `_branch_name` column is a helper, derived from the FK `id_(branch)`."
* "The header row of sheet `customers` should be rendered with fill #4472C4."
* "This workbook is version 2.3, authored on 2026-04-01."

Meta claims are subject to the *layer* and *classification* axes described in
`02_meta_layer.adoc` and `03_meta_classification.adoc`. The word *canonical*
applies to meta, not to data — a meta claim can be canonical, derived, or
operational-carrier; data is original or derived.

=== Why the levels must stay separate

The original/derived distinction at the data layer and the canonical/derived
distinction at the meta layer are *related but orthogonal*:

* The data in a helper column is derived. The meta claim "this column is a
  helper" is canonical (it is a durable, authoritative statement about the
  model).
* The data in an `id_(branch)` column is original. The meta claim "the
  `id_(branch)` column is the primary key of the `branch` frame" is also
  canonical.
* A computed-from-canonical rendering payload (e.g. `__freeze`) is a derived
  *meta* claim, not derived data — it is meta describing the realization of
  a canonical intent.

Mixing the two levels in a single sentence ("the helper column is
non-canonical") obscures which level is being asserted and creates exactly
the conflations the rest of this subsection seeks to remove.
