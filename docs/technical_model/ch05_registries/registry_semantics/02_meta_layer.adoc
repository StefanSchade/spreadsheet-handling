== Meta Layer

The *layer* axis classifies meta by its *role in the meta lifecycle* —
whether a piece of meta is the authoritative statement of intent, a
realization derived for the current run, or a carrier-shaped representation
prepared for persistence or transport.

Three layer values are recognized:

* `canonical`
* `derived`
* `operational-carrier`

The layer is orthogonal to the classification axis (model vs presentation vs
workbook_identity) — every meta entry has a value on both axes.

=== Canonical

A meta claim is *canonical* when it satisfies all of the following:

. It expresses *intent or property* — a statement the author wants to hold,
  not a record of what the system happened to do.
. It is the *authoritative source* for its information: in conflict with
  derived or operational-carrier forms of the same fact, the canonical form
  wins.
. Its information content is *spreadsheet-neutral* in the sense that it can
  be expressed without referring to any one backend's representation. The
  same canonical claim can be honored, ignored, or approximated by each
  adapter independently.
. It is *persistable* — it can survive a parse/write/parse cycle and be the
  form trusted on read-back.
. Its *lifecycle is decoupled from any single run* — it remains true across
  runs, edits, and re-rendering until the user changes it.

If a meta claim fails any of these tests, it is not canonical. The strict
five-criterion definition is deliberate, and the reason it is needed is the
historical drift described below.

==== Historical drift (why the strict definition is needed)

"Canonical" entered the project as an informal property of data columns,
distinguishing original data from derived/helper columns. It then expanded
in three quiet steps, each of which imported a new criterion without
replacing the old one:

. It transferred from data to meta — "the helper column" became "the meta
  claim about the helper column".
. It picked up the *intent-vs-realization* criterion — `freeze_header` was
  "canonical" because it expressed user intent, while `__freeze` was the
  derived realization.
. It picked up the *adapter-portability* criterion — a value could not be
  "canonical" if it could not be expressed across both XLSX and ODS.

Each expansion was reasonable in isolation, but the criteria were never
named explicitly. As a result, the word "canonical" today silently asserts
all five properties at once. The strict definition makes those assertions
explicit so that edge cases (presentation intent that is backend-restricted;
reconstructed hints that become de-facto sources on read-back; persistable
derived caches) can be named and placed deliberately rather than absorbed
into the canonical bucket by accident.

==== "Canonical" applies only to meta

By this definition, *data* is never canonical. A column is original or
derived; canonical/non-canonical does not apply to cell content. The word
applies exclusively to *claims about* data, the workbook, or its
presentation — see `01_data_and_meta.adoc`.

=== Derived

A meta claim is *derived* when it is the *current-run realization* of a
canonical claim or of an inferred state. Derived meta records what the
system has materialized for this run; it is not authoritative and does not
survive a parse/write/parse cycle as the source of truth.

Examples in the live registry:

* `__freeze` — derived from `freeze_header` (canonical) plus current table
  geometry, prepared for a `SetFreeze` plan op.
* `__autofilter` — derived from `auto_filter` (canonical) plus current table
  geometry.
* `__style` — derived from `header_fill_rgb` / `helper_fill_rgb` (canonical)
  plus column resolution, prepared for style plan ops.
* `_meta["derived"]["sheets"].helper_columns` — derived from FK helper
  enrichment for the current run; consumed and removed by `drop_helpers`.

Derived meta is *recomputable*. If lost, the next run can regenerate it from
the canonical sources plus the current frame state. A derived claim is not
the *carrier* shape — carrier shape is a separate role described below.

=== Operational-carrier

A meta entry is *operational-carrier* when it is the *carrier-shaped
representation* of canonical (or, more rarely, derived) meta — the form in
which meta is written to a concrete persistence target or read back from
it.

Examples in the live registry:

* `workbook_meta_blob` — the deterministic-JSON serialization of canonical
  workbook meta written into the hidden `_meta` sheet for XLSX/ODS.
* `_hidden` — the marker that a sheet belongs to the hidden meta-carrier set
  (control flag for plan emission, not a claim).
* `__autofilter_ref` — read-path reconstruction of the concrete autofilter
  reference parsed from an existing carrier; used by tests and diagnostics
  but never the authoritative canonical form.

Operational-carrier entries are *not* a second canonical source. They exist
because canonical meta must be written into and read out of a concrete
backend that has its own constraints (a hidden sheet, an XML element, a
sidecar file). Treating an operational-carrier entry as if it were canonical
silently grants authority to the carrier, defeating the carrier-neutrality
that the canonical role is supposed to guarantee.

A foundational definition of *carrier* itself (what is and is not a carrier;
where carrier ends and IR begins) is the subject of `04_carrier.adoc`.

=== Mapping to the live registry

The live `meta_registry.yaml` uses an older two-value layer axis
(`meta_canonical` / `meta_rendering`) and a four-value classification axis
that conflates layer-role with subject-matter. The foundation layer values
map onto the live encoding as follows:

[cols="2,3,3", options="header"]
|===
| Foundation layer | Live registry encoding | Examples

| canonical
| `layer: meta_canonical` + `classification: canonical_meta`
| `helper_policies`, `freeze_header`, `column_widths`, `version`,
  `constraints`, `frame_lifecycle`.

| derived
| `layer: meta_rendering` + `classification: derived_operational_view`
| `__freeze`, `__autofilter`, `__style`, `__helper_cols`, `options`,
  `derived`.

| operational-carrier
| `layer: meta_rendering` + `classification: carrier_artifact` or
  `read_path_hint`
| `workbook_meta_blob`, `_hidden`, `__autofilter_ref`.
|===

Re-encoding the YAML to expose the foundation layer directly is deferred to
a separate FTR; the foundation here is the conceptual model, not a renaming
proposal.
