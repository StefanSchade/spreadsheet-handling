===== FTR-DEMO-USER-LAUNCH-FLOWS-AND-REVEAL-WALKTHROUGHS-P5 -- User-facing demo launch flows and walkthrough structure

*Status:* Open
*Priority:* P5
*Area:* Demo integration / user guidance

==== Purpose

Define the next user-facing demo slice for `spreadsheet-handling-demo`: simple
cross-platform launch flows and a repeatable Reveal.js walkthrough structure
for the main ACME journeys.

This FTR is planning only in the core backlog. It does not implement launchers,
pipelines, demo repository files, or generated presentation artifacts.

==== Context

The demo repository remains the main user entry point. `make` may remain
available for developers and CI, but it must not be the primary end-user
interface for trying demo flows.

End users should be able to run the curated demo flows without WSL and without
installing `make`. The demo should provide small launchers that prepare the
local Python environment and call the maintained Python module entry point
directly.

The architecture deliberately separates developer/CI orchestration from
end-user launch flows:

* `make` remains developer and CI tooling,
* shell and PowerShell launchers are the user-facing wrappers,
* launchers invoke the maintained Python module entry point directly, for
  example `python -m spreadsheet_handling.cli.apps.run`,
* no hidden orchestration layer or GUI shell is introduced in this slice.

==== Scope

Define and later implement in `spreadsheet-handling-demo`:

* simple shell launchers for Linux/macOS,
* simple PowerShell launchers for Windows,
* idempotent `.venv` creation/reuse inside the demo repository,
* direct calls to the maintained Python module entry point instead of `make`
  as the user-facing path,
* paired pack/unpack pipeline flows for each curated demo scenario,
* reverse-direction outputs under `tmp/target` or an equivalent generated
  target directory,
* user documentation that explains repeated roundtrip trials,
* AsciiDoc source structure for future Reveal.js walkthroughs.

==== Launcher Requirements

* Launchers must be intentionally small and readable.
* They are lightweight, inspectable wrappers around explicit files and
  pipelines.
* They create or reuse a local, idempotent `.venv`.
* They install pinned demo dependencies when the environment is missing or
  stale.
* Normal demo runs must avoid uncontrolled dependency upgrades; the goal is a
  reproducible demo state, not the latest available package state.
* They call the maintained Python module entry point directly, for example
  `python -m spreadsheet_handling.cli.apps.run`.
* `make` remains valid for developer/CI automation, but user instructions
  should present the shell and PowerShell launchers first.
* Generated outputs must not overwrite source data.
* They must not introduce Electron, Tauri, full GUI tooling, or any other
  heavyweight hidden orchestration layer in this slice.

==== Roundtrip Flow Model

Every demo flow is a pair:

* a forward pack/projection pipeline that creates a spreadsheet from source
  data,
* a reverse unpack/reimport pipeline that reads the spreadsheet and writes
  returned structured data.

The reverse direction writes to `tmp/target` or an equivalent generated target
location. This target area is a safe sandbox for user experiments. It must
never overwrite checked-in source data, canonical fixtures, or source pipeline
inputs.

Documentation must explain how a user can re-point the next forward run to the
latest `tmp/target` output for repeated roundtrip/edit trials. Users should be
able to iterate safely: edit the generated spreadsheet, unpack it into the
sandbox, then choose whether the next run starts from the original fixtures or
from the generated sandbox output.

==== Walkthrough Structure

Reveal.js output is generated later from AsciiDoc sources. The first slice
should define source structure and narrative shape only; generated Reveal.js
HTML belongs in `spreadsheet-handling-pages`, not in the demo repository.

Each walkthrough follows the same sequence:

. Run the pipeline.
. Inspect the pipeline config.
. Inspect and edit the generated spreadsheet.
. Run the reverse pipeline.
. Inspect returned data.

==== Initial ACME Flow Sequence

Implement the launch/walkthrough model first for these ACME flows:

. Plain JSON to spreadsheet and back.
. FK helper enrichment and back.
. Relation/crosstable editing and back.

The sequence is intentional. It starts with structurally simple data that still
produces spreadsheet representations that are hard to read relationally, then
progresses toward increasingly human-oriented editing representations. Each
step should make the value of the next projection pattern visible before the
walkthrough adds more capability.

==== Optional Later Bonus

A creative-writing example named "Pirates of Python" may be added later if it
stays clearly secondary to the ACME path and does not distract from the primary
business/demo story. Its role is to showcase flexible structured-content
handling, not to redefine creative writing as the primary product narrative.

==== Non-goals

* Do not implement launcher code in this core-repo FTR.
* Do not generate Reveal.js output in this FTR.
* Do not make `make` disappear from developer or CI workflows.
* Do not require WSL for Windows users.
* Do not overwrite source fixture data during reverse demo flows.
* Do not publish presentation artifacts before the later demo/presentation
  publishing slice.
* Do not introduce Electron, Tauri, full GUI tooling, or hidden orchestration
  assumptions.

==== Acceptance

* The demo backlog or implementation plan defines shell and PowerShell launchers
  as the primary user-facing execution path.
* The launchers are specified to create/reuse an idempotent `.venv`, install
  pinned demo dependencies only when missing/stale, and avoid uncontrolled
  dependency upgrades during normal demo runs.
* The launchers invoke the maintained Python module entry point directly, for
  example `python -m spreadsheet_handling.cli.apps.run`.
* The implementation plan preserves small transparent wrappers around explicit
  files and pipelines, without GUI or hidden orchestration assumptions.
* Demo flow documentation treats each scenario as a paired pack/unpack flow.
* Reverse-flow outputs are specified to land under `tmp/target` or an
  equivalent generated target directory.
* The documentation explains `tmp/target` as a safe sandbox and shows how to
  re-point later runs to generated output for repeated roundtrip trials without
  mutating canonical checked-in fixtures.
* Reveal.js walkthrough source structure is defined in AsciiDoc, with generated
  HTML deferred to Pages publishing.
* The initial ACME sequence covers plain JSON, FK helper enrichment, and
  relation/crosstable editing, with an explicit didactic progression toward
  more human-oriented spreadsheet editing representations.
