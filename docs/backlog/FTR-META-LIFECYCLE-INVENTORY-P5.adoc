===== FTR-META-LIFECYCLE-INVENTORY-P5 -- Complete `_meta` lifecycle inventory

*Status:* New

*Type:* Lifecycle / contract clarification

*Area:* Pipeline / metadata lifecycle / persistence boundary

==== Context

``BUG-RUNTIME-META-PERSISTENCE-BOUNDARY-P4A`` established a narrow
orchestrator-owned persistence boundary that strips the concrete
runtime-only carriers known at the time: top-level ``derived``,
top-level ``__*``-prefixed keys, and runtime-produced FK-helper v2
relations.

That fix does not classify the rest of ``_meta``. Multiple top-level
keys observed in adoption ``_meta.yaml`` files
(``frame_lifecycle``, ``workbook_view``, ``compact_multiaxis``,
``split_by_discriminator``, ``xref_crosstable``, ``cell_codecs``,
``constraints``, ``legend_blocks``, ``sheets``, ...) are currently
classified as ``canonical_meta`` in the registry, but it is not yet
verified whether each entry's persistence behavior matches its actual
producer/consumer contract under repeated runs.

==== Why this FTR

A complete inventory protects future producers and reviewers from
silently introducing new runtime-only carriers, and gives the persistence
boundary a documented source of truth.

==== Scope

Build a contract matrix covering every observed ``_meta`` root and
sub-root. For each entry, classify:

* persisted by default? (yes / no / conditional)
* runtime-only?
* user-authored?
* derived?
* carrier-specific?
* roundtrip-relevant?
* reconstructed on read?
* consumed by primitives, by passes, by both?

Cross-check the classification against:

* the runtime producer/consumer code,
* ``registries/meta_registry.yaml``,
* the lifecycle chapter in ``docs/semantic_model/``,
* the contract matrices chapter in ``docs/semantic_model/``,
* the persistence boundary rules in
  ``src/spreadsheet_handling/pipeline/persistence_boundary.py``.

Where the runtime, the registry, and the docs disagree, decide which
wins and update the others.

==== Non-Goals

* Do not implement before the upcoming release. This FTR exists to make
  the deferred work explicit.
* Do not redesign individual producers/consumers as part of the
  inventory. The inventory may surface follow-ups, but each follow-up
  is its own FTR.
* Do not bind the runtime to the registry. The registry remains a
  manual review artifact unless a separate ADR upgrades it.

==== Acceptance

* Every observed ``_meta`` root and sub-root has a classification entry.
* Every classification is consistent across runtime, registry, and docs,
  or its disagreement is explicitly named with a chosen resolution path.
* The persistence boundary's pruning rules cite the inventory rather
  than hard-coded keys.
* New producers can find a clear rule for "may my output survive
  persistence?".

==== Validation

* The contract matrix renders in ``docs/semantic_model/`` without gaps.
* The persistence boundary tests cover at least one entry from each
  classification bucket.
* No top-level ``_meta`` key exists at runtime that is missing from the
  inventory.
