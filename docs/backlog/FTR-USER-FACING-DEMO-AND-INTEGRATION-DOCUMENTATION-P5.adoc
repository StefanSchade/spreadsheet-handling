= FTR-USER-FACING-DEMO-AND-INTEGRATION-DOCUMENTATION-P5

== Status

Open

== Type

Documentation / Adoption Experience

== Context

The project currently provides:

* versioned core documentation
* release notes
* user guide
* Reveal.js demo material

The v0.2.0 release and the first external consumer adoption (Adoption-001, worldbuilding) highlighted that the project now needs a clearer user-facing adoption surface.

The current demo material primarily serves as an internal exploration and review vehicle. As additional adoption scenarios grow beyond Adoption-001, the project needs a more structured presentation layer for future users.

At the same time, not all documentation artifacts require historical preservation.

A distinction should be made between:

* product documentation
* tutorials and walkthroughs
* integration patterns

== Open Questions

The current direction is not yet explicitly documented.

Open questions include:

* which artifacts are versioned and archived
* which artifacts primarily represent the latest recommended workflow
* how demo content should be organized
* how integration patterns should be presented
* how future users discover relevant adoption examples

Without a documented direction, documentation and Pages content may evolve inconsistently.

== Desired Direction

=== Core Documentation

Core documentation remains release-oriented.

Examples:

* User Guide
* Release Notes
* Functional Model
* Technical Model

These artifacts are archived per release.

=== Demo Documentation

Demo documentation acts primarily as a bridge between the library and users.

Its purpose is to demonstrate workflows rather than preserve historical states.

Demo content should therefore be primarily latest-oriented.

Selected milestone snapshots may be retained when they provide long-term value.

=== Integration Patterns

Integration patterns describe recommended usage.

Examples:

* Linux setup
* Windows setup
* local development workflows
* automation examples

These are considered living guidance and should normally follow the current recommended approach rather than preserving every historical variation.

=== Pages Structure

Pages should evolve toward a clear user-facing navigation structure covering:

* product documentation
* tutorials and walkthroughs
* adoption examples
* integration patterns

The goal is to make successful adoption easier to discover and reproduce.

Pages serves as a navigation and discovery surface.
It is not a second artifact repository and should not replicate the archival
role already fulfilled by versioned release snapshots, Git tags, and PyPI history.

== Long-Term Goal

Support a future user experience where a business analyst or domain expert can:

* discover relevant examples
* follow guided workflows
* understand common modeling patterns
* adopt the library with limited maintainer assistance

This direction aligns with the broader 0.3.0 vision of AI-assisted adoption and improved user-facing documentation.

== Candidate Follow-Ups

Potential future work may include:

* tutorial landing pages
* release-aware demo navigation
* adoption-example catalogues
* integration-pattern catalogues
* AI-oriented onboarding material
* reusable documentation templates

These are intentionally not required by this FTR.

== Non-Goals

* preserve every historical tutorial revision
* archive every integration-pattern revision
* duplicate Git history in Pages
* create a second artifact repository
* redesign the existing release process

== Validation

A future implementation should demonstrate:

* a clear distinction between archived product documentation and living adoption guidance
* discoverable entry points for tutorials and integration patterns
* alignment between Pages structure and the project's adoption strategy
