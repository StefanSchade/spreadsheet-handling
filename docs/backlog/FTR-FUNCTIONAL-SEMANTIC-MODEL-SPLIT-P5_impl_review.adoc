==== Findings

===== Before merge: Semantic Model stubs contain non-functional xrefs

*Observation:* The two Semantic Model stubs preserve their original anchors,
but their intended links to the relocated Functional Model content are written
as backticked literal text:

* `docs/semantic_model/03_problem_statement.adoc` contains
  ``(`xref:functional-model-problem-statement`)``.
* `docs/semantic_model/10_roundtrip_walkthroughs.adoc` contains
  ``(`xref:functional-model-roundtrip-walkthroughs`)``.

Rendered output confirmed that both appear as inline code, not links:

* `/tmp/semantic_model.html` rendered
  `<code>xref:functional-model-problem-statement</code>`.
* `/tmp/semantic_model.html` rendered
  `<code>xref:functional-model-roundtrip-walkthroughs</code>`.

*Risk:* The FTR's navigation-stub strategy depends on readers being able to
follow relocation pointers. Because these are not valid AsciiDoc link macros,
the docs build does not catch the issue and readers see non-clickable pseudo-
xrefs.

*Recommended action:* Replace the literal pseudo-xrefs with valid AsciiDoc
links or remove the xref wording. If cross-document xrefs are desired, use a
form that resolves from `docs/semantic_model/semantic_model.adoc` to the
Functional Model output. At minimum, do not present `xref:...` as a link when
it is rendered as code.

*Priority:* Before merge / documentation navigation.

==== Review Questions

*Did the implementation stay within the FTR?*

Yes. The changes are documentation-only and stay inside Functional Model,
Semantic Model, and the boundary-review follow-up. No runtime source, registry
schema/content, tests, or model implementation files were changed.

*Did it avoid creating a Functional Model ontology/taxonomy?*

Yes. The new Functional Model problem-space overview is explicitly bounded.
`docs/functional_model/ch05_problem_space_overview/_chapter.adoc` carries a
visible *Scope limit* note stating that the chapter is not an ontology,
taxonomy, semantic-set model, lifecycle classification, registry vocabulary,
or rule-ready contract model. The content remains explanatory and repeatedly
pushes classification work back to the Semantic Model.

*Are ch03/ch10 stubs acceptable and are anchors preserved?*

Mostly yes. The original Semantic Model anchors are preserved:

* `[[semantic-model-problem-statement]]`
* `[[semantic-model-roundtrip-walkthroughs]]`

The stubs are acceptable as relocation pointers, but the malformed xrefs noted
above need correction.

*Is ch05 split note sufficient?*

Yes. The note clearly distinguishes "Relevant ideas" as explanatory context
from "Project interpretation" as contract-adjacent. That is sufficient for
this FTR because it does not require physically moving chapter 05 content.

*Is ch08 persistence-boundary note correctly scoped?*

Yes. It keeps the Semantic Model contract rule in place, identifies the module
and mechanism details as Technical Model material, and explicitly defers
physical relocation to a follow-up FTR because no paired Technical Model
chapter exists yet.

*Is adding `ch06_preserved_problem_space_context` under Functional Model appropriate?*

Yes. It is appropriate as preserved explanatory context. The chapter note says
it is not a contract surface and adds no new examples, sets, or rules. This is
preferable to keeping the full problem narrative and walkthroughs in the
Semantic Model contract path.

*Is the boundary review follow-up verdict wording justified?*

Yes. "GREEN on the boundary recut itself; YELLOW carried on the persistence-
boundary mechanism relocation" is justified. The recut conditions are met, and
the implementation deliberately preserves the known persistence-boundary
relocation deferral.
