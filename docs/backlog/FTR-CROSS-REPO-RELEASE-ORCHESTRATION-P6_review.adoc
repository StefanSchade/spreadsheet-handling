== Orchestration model added

The runbook now exposes a deliberate two-axis model of the release flow.

**Axis 1: per-repo stages, in order.**
Stage 1 (core: pre-tag check → tag → PyPI publish → core docs publish)
hands off to Stage 2 (demo: pin bump → demo CI → demo tag → demo slides
publish) hands off to Stage 3 (pages: artifact reception). Each stage
has an explicit owner and a single verification command; the table is
the canonical place to look when an operator is mid-release and unsure
which step is next.

**Axis 2: per-question helper.**
Three read-only helpers compose:

* "is this tag topologically safe to push?" → `make release-check`
  (already existed; carries through the cross-repo flow).
* "did the publish leave the expected files on Pages?" → `make
  pages-check` (added by the previous FTR; carries through).
* "what is the current state across all three repos, and what should I
  do next?" → `make release-status` (added by this slice; the
  orientation helper).

Operators can pick the helper that matches the question they have,
without rederiving the flow each time. None of the three helpers makes
network calls, none fetches, none publishes.

== Recovery scenarios covered

All five cases from the FTR's `Common recovery cases` checklist are
documented under §"Cross-Repo Recovery Scenarios":

. Demo tag pushed before main alignment — confirm slide artifact,
  fast-forward `main` to the tag commit, do *not* retag.
. Stale workflow rerun — recognize that tag-triggered runs use the
  workflow file at the tag, accept historical action pins if
  re-publishing is the goal, cut a new patch tag if current workflow
  shape is the goal.
. PyPI succeeded but docs failed — re-run the failed `publish-docs`
  job from the UI; do not retag (PyPI rejects same-version
  re-upload); validate with `pages-check` after the re-run.
. Demo publish green but CI hygiene red — recognize that the publish
  was the release-blocking signal; non-publish CI noise is classified
  per §"Blocking vs Non-Blocking CI Signals"; do *not* re-run the
  publish workflow.
. Pages publish green but public site not yet visibly updated — wait
  for the GitHub Pages edge cache to refresh (typically a few minutes);
  cache-bust in the browser; inspect Pages Settings only if the stale
  state persists past ~10 minutes; do *not* retag or hand-edit pages.

The sixth FTR-listed case, "tag created from non-`main` branch",
already had a full section §"Troubleshooting: Accidental Release
Topology" prior to this slice; the new recovery section
cross-references it rather than duplicating it.

== Residual risks

* *Helper's "next action" hint is heuristic.* It compares the demo
  pin to the core latest tag and the pages versioned snapshots
  against the core/demo latest tags. It does not understand release
  intent (e.g. a deliberate decision to defer the demo bump for two
  releases). When the heuristic conflicts with the operator's plan,
  the operator overrides; the hint is *advisory*, not authoritative.
  The closing line of the helper output makes this explicit.
* *Latest-tag detection uses `git describe --match 'v*'`.* This finds
  the most recent annotated-or-lightweight tag whose name starts with
  `v` and is reachable from HEAD. If a tag exists on a branch other
  than HEAD's history, it will not be picked up — for the release
  flow this is the right default (a release tag must be reachable
  from `main`), but a one-off feature branch could surprise the
  operator. The cross-correlation surfaces this as a "no" in the
  "at HEAD" field.
* *Pages versioned-snapshot detection is structural only.* The helper
  reports that `versions/<tag>/core/` exists; it does not verify
  contents. `make pages-check CORE_TAG=<tag>` is the deeper structural
  check operators should run after `release-status` flags a missing
  snapshot. The runbook's per-question helper table makes this
  composition obvious.
* *Recovery scenarios are written for the current workflow shape.* If
  the publish workflows are reshaped meaningfully (different jobs,
  different secrets, etc.), the recovery prose will need a refresh.
  Cross-references to the workflow source files are kept loose so a
  rename does not silently invalidate the runbook.
* *No CI invocation of `release-status`.* The helper is operator-only
  on purpose; running it from CI would add a sibling-checkout
  requirement to the release pipeline that does not exist today.
  This is intentional and aligns with the FTR's "favor clarity and
  observability over aggressive automation" note.

== Follow-up skeptical review (2026-05-23)

Scope: independent review of implementation commit `809e0f7` against
the FTR goals, with no assumed prior context.

Findings resolved:

* *Medium: explicit invalid checkout paths rendered as plausible
  state.* `bash tools/release_status.sh --core /missing` exited 0 and
  reported `branch=DETACHED (clean) HEAD=?`, which could mislead a
  release operator. The helper now validates any non-empty `--core`,
  `--demo`, `--pages`, `CORE_DIR`, `DEMO_DIR`, or `PAGES_DIR` value
  against the repo markers before collecting Git state. Invalid paths
  exit 3 with a setup error.
* *Low: incomplete local views overclaimed coherence.* With only
  `CORE_DIR` available, the helper reported "cross-repo state appears
  coherent" even though demo and pages were not checked. Missing
  optional siblings now appear in cross-correlation as visibility gaps,
  and the next-action hint says to locate the missing checkouts before
  deciding release action.
* *Low: pages snapshot path stripping used a raw path as a `sed`
  regex.* This could mis-strip snapshot names for checkout paths
  containing regex metacharacters. The helper now strips the validated
  absolute prefix with shell parameter expansion instead.
