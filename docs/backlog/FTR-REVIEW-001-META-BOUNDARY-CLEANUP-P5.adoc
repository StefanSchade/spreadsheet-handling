===== FTR-REVIEW-001-META-BOUNDARY-CLEANUP-P5 -- Close REVIEW-001 meta and domain-boundary cleanup residues

*Status:* Open
*Priority:* P5
*Area:* domain / rendering / meta contract
*Kind:* Refactoring / Hygiene
*Review source:* REVIEW-001 findings D-008, D-011, D-013, IR-013

==== Purpose

Close the remaining REVIEW-001 cleanup findings where domain code still blurs
runtime metadata mutation, carrier/file I/O, and workbook-identity metadata.

These findings are low-risk individually, but they keep recurring in reviews as
signals that meta ownership and domain boundaries are still partly implicit.
Handle them as one small cleanup program instead of scattering one-off edits.

==== Current State

* `domain/validations/validate_columns.py` exposes `add_validations`, but the
  operation writes workbook constraint metadata rather than validating data.
  This is the D-008 naming / ownership issue.
* `domain/structured_yaml.py`, `domain/yaml_overrides.py`, and related domain
  helpers still perform file I/O or import `yaml`. This keeps D-011 open rather
  than closed.
* `domain/meta_bootstrap.py` and `domain/validations/validate_columns.py` still
  support both `frames.meta` and `frames["_meta"]`; `validate_columns.py` also
  still has the temporary sidecar branch. This is the D-013 dual-interface
  residue.
* `rendering/passes/meta_pass.py` still reserves `exported_at` with an empty
  default, while no package path fills it. Later meta work treats
  `version` / `author` / `exported_at` as workbook-identity metadata, but the
  original IR-013 decision remains unresolved.

==== Scope

. Decide whether `add_validations` should stay under validations as a metadata
  producer or move/rename to a clearer constraint-configuration home.
. Decide whether domain YAML/file-writing helpers are intentional domain
  services or should be moved behind an application / I/O boundary.
. Remove or explicitly retain the dual `frames.meta` / `frames["_meta"]`
  compatibility convention with a documented owner and tests.
. Decide `exported_at`: either remove it from the default MetaPass shell or
  document and implement the owner that sets it using an injectable clock.

==== Non-goals

* Do not redesign the whole `_meta` model in this FTR.
* Do not change user-facing YAML surfaces unless the chosen boundary decision
  requires a compatibility note.
* Do not remove `exported_at` silently without checking workbook meta
  roundtrip expectations and registry semantics.

==== Acceptance

* D-008, D-011, D-013, and IR-013 each have an explicit closure decision:
  fixed, retained-by-design, or split into a narrower follow-up.
* Any retained compatibility path has a named rationale and a characterization
  test.
* Domain file-I/O / `yaml` usage is either removed from domain or documented as
  an intentional domain service with guard coverage.
* `exported_at` no longer exists as an unexplained empty placeholder.
