===== FTR-FK-HELPER-COLUMN-GROUP-META-P5 -- FK helper configuration emits column-group presentation intent

*Status:* New

*Purpose:*::
Allow FK helper configuration to request collapsible helper-column groups by
writing explicit presentation intent into metadata, which the Render IR can
later translate into backend-neutral column-group operations.

*Context:*::
After FK helper primitives become policy-driven, the system will know exactly
which columns are materialized helper columns and where they belong. That makes
helper-column grouping straightforward: grouping is no longer based on prefixes
or naming heuristics, but on resolved helper policy and derived-helper
provenance.
+
The FK-helper layer must not render spreadsheet outlines directly. It should
only declare presentation intent in metadata. Render IR and adapters remain
responsible for turning that intent into concrete spreadsheet output.

*Dependencies:*::

* `FTR-FK-RELATION-POLICY-MODEL-P5`
* `FTR-INFER-FK-RELATIONS-CONFIGURATION-STEP-P5`
* `FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5`
* `FTR-COLUMN-GROUP-RENDER-CAPABILITY-P5`
* Existing FK-helper policy/provenance model.
* Existing metadata bootstrap and render metadata flow.

*Scope:*::

* Add FK-helper configuration for optional helper-column grouping.
* Support at least:
  ** grouping enabled/disabled,
  ** collapsed by default enabled/disabled,
  ** optional outline level if needed,
  ** default behavior for all FK helpers,
  ** per-relation override if reasonably small.
* Decide where the grouping intent lives:
  ** in helper policy,
  ** in canonical presentation metadata,
  ** or as a derived render hint produced from helper policy.
* Ensure helper grouping intent is explicit and inspectable in `_meta`.
* Ensure grouping intent is based on resolved FK helper policy/provenance, not
  on helper column prefixes.
* Add or update the Render IR pass that turns helper grouping metadata into
  backend-neutral column-group RenderOps.
* Keep helper grouping optional and disabled by default unless the FTR review
  explicitly decides otherwise.

*Configuration sketch:*::

[source,yaml]
----
pipeline:
  - step: configure_fk_helpers
    relations:
      - source_frame: products
        source_column: product_manager_id
        target_frame: product_managers
        target_key: id
        helper_fields: [name]
    presentation:
      group_helpers: true
      collapsed_by_default: true

  - step: add_fk_helpers
----

Or, if grouping belongs directly to the materialization step:

[source,yaml]
----
pipeline:
  - step: add_fk_helpers
    group_helpers: true
    collapsed_by_default: true
----

The final placement should prefer the cleaner semantic ownership:
configuration/policy owns intent; primitive materialization owns output columns.

*Metadata sketch:*::

[source,yaml]
----
_meta:
  helper_policies:
    fk:
      presentation:
        group_helpers: true
        collapsed_by_default: true
  derived:
    sheets:
      Products:
        helper_columns:
          - column: _product_manager_name
            source_column: product_manager_id
            target_frame: product_managers
            target_key: id
            target_field: name
            presentation:
              group_with_source: true
              collapsed_by_default: true
----

The final shape may differ, but it must be clear whether the grouping metadata
is canonical presentation intent or derived render intent.

*Non-Goals:*::

* No new spreadsheet adapter capability; that belongs to
  `FTR-COLUMN-GROUP-RENDER-CAPABILITY-P5`.
* No FK relation inference.
* No helper-column materialization redesign beyond the metadata needed for
  grouping.
* No prefix-based helper grouping.
* No readback of user-expanded/collapsed state.
* No preservation of a user's manual outline changes after reimport.
* No demo or tutorial update; that belongs to the docs/demo FTR.

*Behavior:*::

* If `group_helpers` is false or absent, helper columns render as before.
* If `group_helpers` is true, materialized helper columns are marked for
  grouping next to their source FK column.
* If `collapsed_by_default` is true, grouped helper columns should be rendered
  collapsed/hidden according to backend capability.
* If a backend does not support column grouping, the render layer follows the
  documented capability policy.
* Grouping metadata must not change canonical data or reimport behavior.

*Acceptance:*::

* FK helper grouping is configurable.
* The default behavior is explicit.
* Grouping metadata is written without using column-name heuristics.
* Grouping metadata is separated from FK relation identity.
* Render IR receives enough information to create column-group RenderOps.
* `add_fk_helpers` still materializes the same data columns regardless of
  grouping setting.
* Reimport cleanup still removes helper columns based on provenance/policy, not
  grouping state.
* Tests cover:
  ** grouping disabled,
  ** grouping enabled,
  ** collapsed-by-default enabled,
  ** multiple FK helper groups in one sheet,
  ** no grouping for non-helper columns,
  ** deterministic metadata ordering.
* Existing architecture guards still pass.

*Validation:*::

* Run FK-helper unit tests.
* Run Render IR tests for metadata-to-column-group conversion.
* Run XLSX/ODS targeted tests from the render capability FTR if affected.
* Run relevant pipeline integration tests.
* Render modified technical documentation.
* Run `git diff --check`.

*Residual risks:*::

* The final metadata shape may need later adjustment if a generic presentation
  metadata model emerges.
* Some spreadsheet applications may display grouped helper columns differently,
  even when the carrier metadata is valid.
* Readback of collapsed/expanded UI state remains out of scope.
