= Cell Codec Projection-Family Slice: FTR-META-ONTOLOGY-REMOVAL-WORKBOOK-PROJECTION-EPIC-P4A
:revdate: 2026-07-20
:status: Implemented
:type: Projection-family slice record (cell_codec)

== Purpose

Second projection-family slice, independent of `compact_multiaxis` (which
remains blocked pending XRef acceptance and is treated strictly as a
consumer here). Applies the established method: explicit transformation
parameters, minimal persisted feature-local inverse intent, run-local
Resolution only where a concrete consumer exists, explicit cleanup only
when mechanically safe, and no generic lifecycle, canonicality, provenance,
or Workbook View semantics.

== Field inventory result: no persisted intent at all

The complete producer/consumer trace found that **`_meta.cell_codecs` had
no runtime reader anywhere** — not the codec steps themselves (decode and
encode both *require* explicit `codec_intent` or `mode` and never fall
back to metadata), not `compact_multiaxis` (which passes parameters
explicitly and persists its own `compact_multiaxis` family), not
rendering, schema maintenance (generic convention scan never matched the
real payload shapes), sparse defaults, column roles, or any consumer
repository. Every field in both payload shapes (`operation`, `source`,
`output`, `value`, `code`, `passthrough_columns`, `drop_empty`,
`group_by`, `mode`, `delimiter`, `strip`, `allowed_*`, `canonical_order`,
`normalize_case`, `participating_columns`, `compact_column`, `separator`,
`absent_value`) was descriptive duplication of explicit configuration.

Under the family rule (retain only with a concrete decision consumer) the
producer was removed entirely:

* Cell Codec intent lives in explicit step configuration only. Decode
  without explicit intent fails (pre-existing behavior, now pinned with a
  legacy-payload-present test proving no metadata fallback resurrects).
* There is no run-local Resolution facet: nothing consumed one.
* Legacy `_meta.cell_codecs` payloads are tolerated pass-through sediment
  (persistence boundary passes them unchanged; nothing reads them).
* No cleanup commands are involved: neither codec direction supersedes its
  source frame by contract (encode keeps the structured frame; decode
  keeps the compact frame); frame-set decisions stay with the pipeline
  builder via the generic cleanup contract.

This also removes the `compact_multiaxis`-composed sediment: the wrapper's
inner codec calls no longer persist entries naming
`+__compact_multiaxis_*+` temp frames (question 10 of the slice mandate —
resolved as a side effect of the producer removal, without touching the
wrapper).

== Forward/inverse contract

Two explicit contracts, both symmetric:

* *Position-based* (`codec_intent` on both steps): `participating_columns`
  join into `compact_column` with `separator`; `absent_value` represents
  empty/missing attributes and decodes back to `""`. Encode and decode
  take the identical intent mapping.
* *Historical code-row* (`mode` + codec parameters, consumed primarily by
  `compact_multiaxis`): code rows group into a delimited cell and split
  back. Symmetric parameter vocabulary; `mode` is required explicitly.

Declaration guards (fail before any output):

* duplicate `participating_columns` / `group_by`; missing participating
  columns; `compact_column` that is also participating or collides with a
  passthrough column; `absent_value` containing the separator; duplicate
  physical source labels (equality-based detection, codec diagnostic);
  decode rejects participating columns that already exist beside the
  compact column (silent overwrite).

== Carrier/type contract (string-oriented, enforced)

Cell Codec is intentionally string-oriented
(`ADR-VISIBLE-CELL-TYPING-STRING-SUBSTRATE`); this slice states and
*enforces* that contract instead of adding a typed-value system:

* position-based participating values must be non-empty strings or empty;
  numeric, boolean, date, and mixed values fail with an explicit
  diagnostic (`1` vs `"1"` ambiguity cannot arise: the typed value fails);
* empty string and null/missing are equivalent: both encode as
  `absent_value` and decode to `""` (not null);
* there is no escaping/quoting: separator-in-value fails encoding, and
  `split_tokens` serialization now rejects delimiter-containing tokens
  (previously it silently produced cells that decoded into the wrong
  token count — the one behavioral defect found in the scalar layer);
* `whole_cell_code` cells may contain delimiter punctuation by design;
* XLSX/ODS roundtrip proven for compact cells including absent handling
  and numeric-looking string tokens (`"1"` stays a string across both
  carriers under the string substrate).

The historical code-row path stringifies typed code values (`7` → `"7"`);
that wrapper-era behavior is characterized by test and documented, with
tightening deferred to the `compact_multiaxis` composite slice.

== Schema maintenance

`cell_codecs` moved from the generic blocked roots to the ignored
(out-of-scope) surfaces: a retired family with no producer and no consumer
has no live references, so legacy sediment must not block renames or
drops. `ReferenceRoot.CELL_CODECS` was removed; the meta-reference-surfaces
row records the ignored role with the uninventoried marker. A
characterization test proves a realistic legacy payload blocks nothing and
passes through unchanged.

== Composed-consumer findings (`compact_multiaxis`, consumer only)

* The wrapper passes all codec parameters explicitly; nothing in it reads
  codec metadata. Its behavior is unchanged by this slice (full compact
  suite green).
* Wrapper-owned and deferred to its composite slice: legacy stringification
  of typed codes in the code-row path; the wrapper's own
  `compact_multiaxis` metadata family (including its codec-parameter
  duplication) was not inventoried here.

== Registries and documentation

* Step registry: `codec_intent` parameter documented on both steps;
  `cell_codecs` write refs and the (already false) `encode` read ref
  removed; the compact entries' transitive `cell_codecs` refs pruned to
  match actual behavior (description-only change, no wrapper refactor).
* Meta registry: `cell_codecs` entry removed (with pinned-test updates);
  its consumer list had claimed `layout_composer` / `workbookir_to_frames`
  consumption that never existed.
* User guide: the codec section was substantially wrong (invented
  `delimited`/`positional` modes, wrong `,` delimiter default, wrong
  `drop_empty` default, `codec_intent` undocumented); rewritten to the
  actual contract with the string-contract note.
* Functional model: durable intent clarified as the explicit declaration;
  implementation reference updated. Technical model classification list
  and the string-substrate ADR carry retirement notes (the ADR decision
  itself is unchanged).

== Tests

23 new tests (suite 1552 → 1578): no-metadata pinning for all four paths
plus legacy-payload no-fallback; declaration guards (duplicates, missing,
collisions, absent/separator conflicts, physical duplicates, decode
overlap); string-contract enforcement (numeric/boolean/date/mixed, empty
vs missing, separator-in-value); legacy path characterization
(delimiter-in-token rejection, whole-cell punctuation, stringification,
duplicate group_by/labels); input and metadata non-mutation; XLSX and ODS

Validation: full Core suite 1578 passed / 1 xfailed; ruff clean;
architecture/registry suites green; demo ACME XRef roundtrip green;
worldbuilding XLSX, ODS, and workbook-view workflows green (the dino
pipelines exercise the code-row codec through `compact_multiaxis` in both
directions).

== Compatibility

* `_meta.cell_codecs` is no longer written (pre-1.0; no consumer read it;
  legacy payloads tolerated; release-noted).
* New failure modes only where output was previously corrupt or ambiguous:
  typed participating values (silent type change), delimiter-in-token
  serialization (undecodable cells), duplicate labels/fields (non-scalar
  cells), decode overwrite collisions.
* No shipped consumer pipeline uses the removed metadata or the newly
  rejected inputs (demo and worldbuilding verified green).

== Deferred (wrapper-owned or out of scope)

* `compact_multiaxis` composite slice: its own metadata family inventory,
  code-row typed-value tightening, temp-frame naming.
* Category-hinted/typed compact values: future scope per the functional
  model; would be a new explicit contract (see ADR status note).
* Side-effecting writers, metadata pileup, M:N Workbook View, facet
  taxonomy: unchanged deferrals.

== Human decisions

None required. The string-oriented contract is the functional model's MVP
scope plus the accepted string-substrate ADR; the metadata removal follows
mechanically from the no-consumer rule. The removal is prominently
flagged for the independent review: if the review establishes a concrete
future consumer for persisted codec intent, reintroduction would be a
narrow, explicit contract addition.

== Correction record (after independent Review 001)

The independent implementation review
(`_cell_codec_projection_slice_review_001.adoc`, commit `d5ca05e`) accepted
the explicit-only direction (no `_meta.cell_codecs` producer/consumer, no
metadata fallback, inert legacy sediment, schema-maintenance retirement,
`compact_multiaxis` as explicit consumer) but *withheld acceptance* for two
blocking defects and four important gaps. All are corrected; the explicit-only
direction is unchanged.

Shared physical-frame label boundary (architectural)::
XRef and Cell Codec required the same physical table-boundary contract, so it
is extracted into a small shared module `domain/tabular`
(`ensure_unique_physical_column_labels`, `ensure_unique_field_declaration`,
`is_scalar_addressable_label`). The contract: physical frame column labels
must be non-missing, hashable, uniquely and deterministically comparable, and
safe for scalar pandas addressing. It is distinct from XRef matrix-axis
identities (unique non-empty strings) and Cell Codec participating cell values
(string-oriented). XRef was migrated to the shared helper with no behavior
change (its full regression suite is unchanged and green); no generic
object-equality, table, or serialization framework was introduced.

B1 -- physical-label boundary across all four paths::
Position encode, position decode, historical encode, and historical decode all
run the shared physical-label boundary before any pandas
membership/selection/grouping/row-indexing. Duplicate ordinary strings,
duplicate `NaN`, duplicate `pd.NA`, selected list labels, and selected
multi-element NumPy-array labels now receive a deterministic Cell Codec
`ValueError` instead of a raw pandas/Python exception; unique numeric labels
remain valid; tuple/MultiIndex surfaces remain rejected by the flat-frame
guard. The Review 001 corruption case (historical encode + duplicate `NaN`
physical labels + `group_by`/`code` addressing the duplicate label) now fails
deterministically with no output frame, no `_meta.cell_codecs`, no input or
nested-metadata mutation, and no serialization of a pandas `Series` as cell
data. Configured selector lists are also validated for scalar-addressability
before membership.

B2 -- position grammar soundness::
`_position_intent` now rejects an overlapping separator/absent-marker
declaration in *either* direction (`separator in absent_value` or
`absent_value in separator`). This closes the review counterexample
(`separator="--"`, `absent_value="-"`, values `["A", "", "B"]`) that encoded to
`A-----B` and could not be decoded. A confined local analysis confirms that,
with overlap rejected and per-row rejection of values containing the separator
or equal to the absent marker, every accepted encode output decodes back to
the same tokens under the identical intent (no token contains the separator
and none is empty).
+
NOTE: This soundness claim was *incomplete for multi-character separators*
and was corrected by Review 002 (see the Review 002 correction record below):
a token or absent marker ending in a separator *prefix* can form a separator
occurrence across the join boundary. The grammar is now narrowed to a
one-character separator, which restores full soundness.

I1 -- historical delimiter validation before data shortcuts::
`split_tokens` mode validates the delimiter before the empty-cell / empty-group
fast paths in both `parse_cell_value` and `serialize_cell_value`, so
`delimiter=""` fails regardless of cell contents. `whole_cell_code` delimiter
behavior is unchanged.

I2 -- duplicate passthrough declarations::
Historical decode rejects duplicate `passthrough_columns` before output, so a
duplicate declaration never creates duplicate output columns. The shared
field-declaration check is reused.

I3 -- position/historical mutual exclusion::
The public `encode_cell_values` / `decode_cell_values` entry points enforce
the documented mutual exclusion: supplying `codec_intent` together with any
explicitly supplied historical parameter (`mode`, `group_by`, `delimiter`,
`allowed_*`, `canonical_order`, `normalize_case`, `strip`, `drop_empty`,
`passthrough_columns`, `code`/`value`) fails deterministically instead of
silently selecting the position path. A per-argument sentinel distinguishes an
explicitly supplied historical argument from an untouched default, so untouched
defaults are not treated as caller intent. The pipeline registry now marks
`encode_cell_values.group_by` conditionally required (historical encode only)
and records the mutual exclusion.

I4 -- documentation and registry realignment::
The Functional Model no longer references a live `cell_codecs` registry entry;
the release-notes known-limitation no longer instructs configuring codecs
through the retired `cell_codecs` workbook-metadata family;
`pipeline_cleanup.py` no longer claims inverse transforms need
`_meta.cell_codecs`; the pipeline registry corrects the conditional `group_by`
requirement and mutual exclusion; and the meta reference-surface identifier is
renamed `MRS-BLOCKED-CELL-CODECS` -> `MRS-IGNORED-CELL-CODECS` (its substantive
ignored/retired classification was already correct). Superseded ADR material
that is clearly dated as historical is left intact.

Legacy code-row path::
The historical code-row path is retained only for `compact_multiaxis`
compatibility and is treated explicitly as a compatibility path, not an
equal-status permanent contract. Its removal-versus-extraction decision is
deferred to `FTR-CELL-CODEC-LEGACY-CODE-ROW-RETIREMENT-P4A` (not implemented
here).

== Correction record (after independent Review 002)

Independent Review 002 (`_cell_codec_projection_slice_review_002.adoc`,
commit `1494512`) accepted the shared physical-label helper, the
behavior-preserving XRef migration, the explicit-only metadata retirement,
and Review 001 findings I1 and I2, but *withheld acceptance*: both Review 001
blockers were only partially closed, and the mutual-exclusion sentinel leaked
into the public API. All four residuals are now closed. No accepted decision
was reopened.

R002-B1 -- generated historical output labels::
The Review 001 correction validated the four *source* selectors but not the
two *generated output* selectors. Historical decode's generated `code` output
label and historical encode's generated `value` output label now pass through
the shared `is_scalar_addressable_label` boundary before any collision
membership, record-dict key construction, or DataFrame output-label
construction. `decode_cell_values(..., code=[1, 2])` and
`encode_cell_values(..., value=[1, 2])` (and the `pd.NA` / multi-element
ndarray variants) now raise a deterministic Cell Codec `ValueError` instead of
a raw `TypeError`/ambiguous-truth error; the `None` variant, which previously
produced a pandas-normalized `NaN` output label, is now rejected. Ordinary
string and numeric-scalar generated labels remain valid. Rejections produce no
output frame, no `_meta.cell_codecs`, no cleanup, and no input/metadata
mutation.

R002-B2 -- one-character separator contract::
Review 002 showed the earlier containment-only proof was invalid for
multi-character separators: a token ending in a separator *prefix*
(`separator="--"`, values `["A-", "B", "C"]` -> `A---B--C` -> `["A", "-B",
"C"]`) forms a separator across the join boundary. The grammar is narrowed to
the accepted no-escaping contract: `codec_intent.separator` must be *exactly
one Unicode character*. With a one-character separator (no proper non-empty
prefix), the retained per-row and marker checks guarantee that every accepted
token contains no separator character, so joining cannot create a separator
occurrence across a boundary and decode recovers the same fields. Retained
rules: non-empty separator/marker, separator and marker differ, marker does
not contain the separator, real values equal to the marker or containing the
separator fail, empty/missing encode to the marker, marker decodes to `""`.
No quoting, escaping, or automata were introduced. No shipped configuration
used a multi-character position separator; only test fixtures were updated.

R002-I3 -- public signatures restored::
The private `_UNSET` sentinel no longer appears in the public
`encode_cell_values` / `decode_cell_values` signatures, `inspect.signature`,
`help()`, or generated tooling. The functions carry ordinary typed defaults.
Mutual exclusion is now an *effective-value* rule: a historical parameter
supplied alongside `codec_intent` conflicts only when its value differs from
its documented default (structurally meaningful parameters default to `None`
and so conflict on any real value; a value equal to the default carries no
distinct historical intent and is accepted on the position path). Meaningful
mixed contracts (`mode`, `group_by`, non-default `delimiter`, `strip=True`,
allowed collections, legend reference, canonical order, passthrough) still
fail deterministically. `compact_multiaxis` is unchanged and green.

R002-I4 -- documentation and records::
The User Guide states the one-character separator rule, drops the incomplete
asymmetric overlap wording, and adds the required explicit `mode` to both
historical YAML examples; the Compact Multiaxis Functional Model no longer
lists `cell_codecs` as a live registry anchor and describes Cell Codec as an
explicit compatibility primitive; the pipeline registry codec_intent notes
state the one-character separator soundness, the effective-value mutual
exclusion, and generated-output-label validation; this slice record and the
epic no longer claim the multi-character grammar was proven sound; and the
legacy follow-up FTR's safety and XRef-sequencing statements are corrected.
Review 001 and Review 002 artifacts are left untouched as historical
evidence.

The independent Review 001 artifact is left untouched as historical evidence.
