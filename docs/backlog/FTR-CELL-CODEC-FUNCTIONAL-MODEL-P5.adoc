===== FTR-CELL-CODEC-FUNCTIONAL-MODEL-P5 -- Promote cell_codec into the Functional Model

*Status:* New

*Purpose:*::
Promote the already-green `cell_codec` transformation family into the Functional
Model so it is modeled, not only implemented. Modeling and documentation only.

*Context:*::
`cell_codec` is implemented and tested but unmodeled (architecture health review:
"Unmodeled (impl Green)"). The sequencing review notes `compact_multiaxis`
imports `cell_codec`, so `cell_codec` must be modeled before the composition
capstone. Promotion should be use-case-anchored (e.g. under
`business_calculation_workbook_roundtrip`), not a mechanical per-family catalog
entry.

*Dependencies:*::
* `FTR-CONCEPT-DOCS-CLEANUP-P5` -- run the `ch04` editorial pass first.
* Legend Blocks is already modeled (`ch04/legend_blocks.adoc`) and is the
  neighbour concept to reference.

*Scope:*::
* Add a use-case-anchored `ch04` step-slice chapter for `cell_codec`.
* Clarify its relation to Legend Blocks and to XRef value semantics.
* Use a Semantic Model Reference that names the registry entry rather than
  embedding field paths.

*Non-Goals:*::
* No runtime code changes.
* No mechanical standalone catalog chapter.
* No new cell-codec features or token surfaces.

*Acceptance:*::
* `cell_codec` has a `ch04` home anchored to an existing use case.
* The chapter states the relation to Legend Blocks and XRef value semantics.
* The chapter names the registry entry (`cell_codecs`) instead of embedding
  producer/consumer field paths.
* No runtime code is changed.
