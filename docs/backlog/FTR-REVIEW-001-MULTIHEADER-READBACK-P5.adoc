===== FTR-REVIEW-001-MULTIHEADER-READBACK-P5 -- Close JSON and CSV MultiIndex readback gaps

*Status:* Open
*Priority:* P5
*Area:* io_backends / multiheader roundtrip
*Kind:* Backend hardening
*Review source:* REVIEW-001 findings A-008, A-009
*Builds on:* FTR-MULTIHEADER-P2

==== Purpose

Close the remaining MultiIndex readback asymmetries that REVIEW-001 kept as
nice-to-have backend gaps after the original multiheader work.

`FTR-MULTIHEADER-P2` established pandas `MultiIndex` headers as the in-memory
standard and delivered XLSX / JSON write support plus flatten / unflatten edge
steps. The remaining gaps are narrower: JSON directory readback does not rebuild
MultiIndex from nested records, and CSV directory `read_multi` still ignores
`header_levels`.

==== Current State

* `JSONBackend.write_multi` emits nested JSON objects for MultiIndex columns.
  `JSONBackend.read_multi` reads JSON records with `pd.read_json(...)` and does
  not use `header_levels` to reconstruct MultiIndex columns.
* `CSVBackend.read` respects `header_levels` by passing a header range to
  pandas.
* `CSVBackend.read_multi` still reads each CSV file with `header=0` and then
  pads flat column names into fake MultiIndex tuples. Real multi-row CSV headers
  are therefore not preserved in directory mode.

==== Scope

. Define the supported JSON readback contract for nested objects:
  reconstruct MultiIndex columns directly, or document that callers must use an
  explicit unflatten step.
. Make CSV directory readback honor `header_levels` consistently with the single
  CSV read path, or document an intentional capability gap.
. Add focused tests for JSON nested MultiIndex readback and CSV `read_multi`
  multi-header directory input.

==== Non-goals

* Do not introduce nested-list JSON normalization here.
* Do not change spreadsheet adapter behavior.
* Do not make CSV carry metadata beyond tabular headers.

==== Acceptance

* A-008 has a clear closure decision backed by tests or an explicit documented
  non-goal.
* A-009 is linked to the `FTR-MULTIHEADER-P2` lineage and either fixed or
  documented as an intentional CSV capability gap.
* Existing flat JSON and CSV directory workflows remain compatible.
