= FTR-RELEASE-PROCESS-AUTOMATION-P5

== Status

Open

== Type

Process Improvement

== Context

During the v0.2.0 release the publication completed successfully:

* main was merged and pushed
* tag v0.2.0 was created and published
* Pages artifacts were generated
* versioned documentation links resolved correctly

However, a post-release observation showed an inconsistency on the Pages landing page:

* "Latest release" correctly displayed v0.2.0
* "Browse by version" still showed v0.1.0b6

The root cause is currently unknown.

Possible explanations:

* manual archive maintenance step missing from the release process
* incomplete archive regeneration during publication
* intentionally manual archive ownership that is insufficiently documented

The release itself was successful and no broken links were observed.

== Goal

Clarify and improve ownership of release archive updates.

Future releases should make it obvious whether version archive updates are:

* generated automatically
* maintained manually
* or validated by release tooling

== Desired Outcome

For a future release:

* version archive entries are updated consistently
* release runbook documents the required behavior
* release validation detects obvious archive/version mismatches

== Non-Goals

* No changes to the v0.2.0 release
* No retroactive repair required
* No redesign of Pages publishing

== Possible Directions

* Generate archive entries from tags/releases
* Add archive validation to release-check
* Add an explicit release-runbook step if manual maintenance remains the preferred approach
* Replace the version archive table with a pointer to PyPI release history and
  archived Pages versions, eliminating manual maintenance entirely
  (PyPI already provides a concise authoritative list)

== Validation

Perform a future release and verify that:

* Latest release banner
* Browse by version archive
* versioned Pages links

all reference the same release line.
