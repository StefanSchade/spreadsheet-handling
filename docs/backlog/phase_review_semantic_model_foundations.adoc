* purpose: carry FK-helper review lessons into a structured semantic-model path;

== FTR Split Assessment

The seven-FTR split is coherent:

* skeleton: stabilizes the semantic-model document structure;
* problem statement: elaborates ambiguity/friction and ambiguity policy;
* inspirations: captures external influences as project interpretations;
* sets/lifecycle model: defines overlapping sets, lifecycle, precedence, and
  update concepts;
* registry role: separates description, test oracle, and possible future rule
  input;
* adoption strategy: defines no-big-bang transformation-family slices;
* FK-helper contract-matrix pilot: applies the matrix narrowly to FK helpers.

Dependencies are represented through recommended order and FTR references. The
registry-role FTR precedes contract-matrix guard work, and the FK-helper pilot
defers lifecycle policy to the existing Phase 4 derived-provenance FTR.

No FTR appears too large for a planning stub. The sets/lifecycle FTR is the
broadest, but its scope is vocabulary and rule-ready documentation only, with
explicit exclusion of applying the model across transformations.

== Acceptance And Stub-Quality Assessment

All seven FTRs are stubs, not full implementations. Each has:

* `Status: New`;
* Goal;
* Scope;
* Out of scope;
* Seed sections elaborated;
* rough acceptance criteria;
* code/guard allowance note.

The code/guard boundaries are appropriate. Six FTRs are documentation-only. The
FK-helper pilot permits only narrowly scoped FK-helper guard refinement where
the contract is already stable, and explicitly excludes new FK-helper runtime
behavior.

== Design Decision Preservation

The plan preserves the key design decisions:

* semantic ambiguity is not always a bug;
* deterministic roundtrip behavior requires explicit semantic contracts;
* semantic classes are overlapping sets, not disjoint buckets;
* conflict should preferably be resolved at metadata write/update time;
* versioning and evolution are part of the model;
* adoption proceeds transformation-family by transformation-family;
* registries are first-class operational artifacts but are not automatically a
  runtime rule input;
* FK helpers are the first pilot because the contract is stable enough for a
  narrow matrix/guard slice.

== Residual Risks

* The semantic-model seed is intentionally long and draft-like. Later FTRs must
  avoid treating the current scaffold as final contract text.
* The FK-helper pilot may uncover drift between registries, docs, guards, and
  runtime behavior. The current plan handles this by preferring documentation or
  registry tightening and forbidding broad runtime rewrites.
* The sets/lifecycle vocabulary must stay practical. If it becomes too abstract,
  later adoption slices may lose the intended transformation-family focus.
* Registry-as-rule-input remains explicitly future-facing. Any move from
  description/test oracle to runtime input should require a separate
  architectural decision.
