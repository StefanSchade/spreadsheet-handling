== 2. Verdict

The two exact findings from `5350b22` are closed:

* missing-like relation / `base_relation` labels are rejected before pandas
  selection or cleanup;
* explicit `value_columns` are validated by the shared matrix-axis identity
  validator before overlap membership.

The documented matrix-axis / physical-label distinction is sound in intent,
and the Tuple/Multiheader design remains correctly deferred. However, the
new physical-label implementation does not yet establish its complete stated
boundary. A deterministic but unhashable physical label such as `[1, 2]`
passes `_ensure_unique_physical_labels` even though pandas cannot use it as a
scalar column selector. A live contraction or base-relation expansion can
then raise raw `TypeError: unhashable type: 'list'` during membership rather
than the promised deterministic XRef diagnostic.

This is one blocking acceptance defect in the newly introduced
scalar-addressability contract. It is cleanup-safe and does not affect a
shipped consumer, but XRef is not yet accepted as the stated foundation for
later composed families.

== 4. Physical-label-boundary assessment

The correction orders its intended checks correctly for the reviewed
counterexamples:

* `_require_frame` resolves the source without selecting columns;
* `_ensure_unique_physical_labels` runs before configured-field membership,
  row selection, metadata output, or cleanup scheduling;
* missing-like labels are detected before equality-based deduplication;
* labels whose self-equality has an ambiguous truth value are rejected before
  deduplication;
* ordinary duplicate string labels are rejected before scalar indexing.

Unique numeric and tuple labels pass the boundary primitive. The operation
surface still rejects tuple / `MultiIndex` columns through its existing
first-slice flat-label guard, so that primitive-level acceptance does not
promise present Tuple/Multiheader support.

`_is_missing_like` and `_has_deterministic_equality` are small, readable,
XRef-local boundary helpers. They do not create a generic arbitrary-object
identity or equality framework.

=== Blocking counterexample: scalar addressability is not established

The helpers test missingness and `bool(value == value)`, but do not establish
that a remaining label is usable as a pandas scalar column key. For example:

[source,python]
----
labels = np.empty(3, dtype=object)
labels[:] = ["r", "k", [1, 2]]
frames = {
    "rel": pd.DataFrame(
        [["r1", "A", "x"]],
        columns=pd.Index(labels, dtype=object),
    )
}

contract_xref(
    frames,
    relation="rel",
    output="matrix",
    row_keys="r",
    column_key="k",
    value=[1, 2],
    drop_source=True,
)
----

`[1, 2]` is non-missing and its self-equality converts to `True`, so
`_ensure_unique_physical_labels` accepts it. `_ensure_columns` then reaches
pandas membership and raises raw `TypeError: unhashable type: 'list'`.
The equivalent `base_relation` expansion counterexample raises the same raw
error. Both leave the input frames unchanged and write no `_meta`, output,
or cleanup, but they violate the declared requirement that every physical
label be scalar-addressable and validated before pandas membership.

This counterexample does not imply a global string rule. Numeric labels and
hashable tuple labels can remain valid at the boundary; the missing check is
the local rejection of labels that pandas cannot address as scalar columns.

== 5. Matrix-axis validation-order assessment

The matrix-axis contract remains coherent: identities participating in the
persisted matrix/relation roundtrip are unique, non-empty strings. Explicit
`value_columns` now reach `_ensure_column_identity_list` before overlap
comparison. Independent probes produced the XRef identity diagnostic for:

* `value_columns=[pd.NA]`;
* `value_columns=[np.array([1, 2])]`;
* numeric and empty selectors;
* duplicate string selectors.

A valid string selector succeeds. With `value_columns` omitted, every
non-row-key matrix column is still inferred as a value column, with no change
to ordering or output semantics.

== 6. Cleanup-safety assessment

All specifically requested rejected cases were cleanup-free and did not
mutate frames or metadata: duplicate NaN labels, duplicate `pd.NA` labels,
missing-like base labels, ambiguous-equality labels, ordinary duplicate
strings, and invalid configured value-column selectors.

A valid full contraction writes exactly one source cleanup command,
`drop_frames: [rel]`. A separately executed valid full expansion writes
exactly one source cleanup command, `drop_frames: [matrix]`. The inferred
expansion path remains valid.

The newly found unhashable-label counterexample is also cleanup-free and
produces no output or XRef metadata. Its defect is the raw pandas error and
the failure to enforce the stated boundary, not silent corruption or unsafe
cleanup.

== 7. Tuple/Multiheader compatibility assessment

The correction makes no architectural decision about Tuple/MultiIndex
operation support, stable technical column ids, or visible display/header
paths. The documentation consistently says:

* string identity is required only for persisted XRef matrix-axis identity;
* physical relation / `base_relation` labels have a weaker boundary and are
  not globally string-only;
* the first-slice operation surface still requires flat labels;
* tuple/Multiheader operation support and identity-versus-display design are
  future work, neither promised nor prohibited.

No typed spreadsheet-header encoding or generic identity metadata is
introduced.

== 8. Documentation and registry assessment

The Functional Model, Technical Model, User Guide, meta registry,
pipeline-step registry, release notes, epic record, and third XRef correction
record consistently distinguish the two contracts and preserve the deferred
future design. None claims that every physical DataFrame label must be a
string, that Tuple/MultiIndex operation support currently exists, or that the
future stable-id/display-label model has been decided.

Those surfaces do, however, uniformly claim that physical labels are
scalar-addressable and rejected before pandas selection/membership. The
unhashable-list counterexample makes that completion claim too broad until
the local boundary is completed.

== 11. Human decisions

No new human decision is required. The accepted distinction already decides
the expected behavior: matrix-axis identities stay unique non-empty strings;
physical labels may be non-string but must be scalar-addressable; future
Tuple/Multiheader and stable-id/display-label design stays unresolved.

Closing the remaining defect requires only completing the existing local
boundary and characterizing the deterministic-unhashable relation and
`base_relation` cases. It does not require typed header encoding or a generic
equality framework.

== 12. XRef acceptance and composed-family readiness

The two findings in `5350b22` are closed, but the newly documented physical
label boundary is not fully enforced. Therefore the XRef projection-family
slice is not yet accepted, and later composed-family work must not rely on
the boundary as complete. Independent primitive-family work remains
unaffected.
