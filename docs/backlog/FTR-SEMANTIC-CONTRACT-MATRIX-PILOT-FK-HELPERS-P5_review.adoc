== Semantic Consistency Assessment

The matrix captures the current FK-helper semantic split correctly:

* `infer_fk_relations` and `configure_fk_helpers` are policy producers.
* FK-helper primitives consume v2 policy and / or transient derived provenance.
* Primitives do not perform convention-based FK relation inference.
* `FK_PATTERN` / `id_(<target>)` remains a wire-format and producer-side inference convention, not primitive execution logic.
* `_meta.derived.sheets.*.helper_columns` is treated as transient derived runtime metadata, not canonical roundtrip truth.

The runtime code matches the matrix on the key behavioral points. `add_fk_helpers`, `remove_fk_helpers`, `reorder_fk_helpers`, and `validate_fk_helpers` resolve FK identity from v2 policy / provenance and raise clear missing-policy errors when neither source is available.

* Establish: producer steps write resolved policy.
* Transform: primitives consume policy / provenance and may write or clean transient derived provenance.
* Reimport / cleanup: helper removal uses provenance first, policy fallback second.

The matrix correctly avoids defining the full lifecycle policy for `_meta.derived.sheets.*.helper_columns`; that remains deferred to `FTR-DERIVED-PROVENANCE-LIFECYCLE-P4A`.

=== Minor

==== M-01: Residual v1 bridge removal ownership is ambiguous

`registries/pipeline_step_registry.json:1159` says a follow-up clean-up FTR will trim the residual v1 per-target write. `registries/meta_registry.yaml:427` still records `FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5` as the `removal_ftr`, and `docs/semantic_model/12_contract_matrices.adoc:337` says the owning FTR is recorded in the lifecycle block.

Those statements are not the same ownership model. Either the existing FTR still owns removal, or a follow-up FTR does. The runtime state is clear, but the lifecycle-owner signal is not fully reconstructable from artifacts alone.

Recommended fix: make the registry / matrix wording use one owner model. If the cleanup FTR is intentionally future and unnamed, say that explicitly instead of implying a concrete existing owner.

==== M-02: Guard coverage assessment overstates the reorder guard

`docs/semantic_model/12_contract_matrices.adoc:296` says `test_fk_helper_primitives_consume_policy.py` pins that the `fk_helpers` package, `reorder_fk_helpers`, and `validate_fk_helpers` do not import `FK_PATTERN`, `detect_fk_columns`, or `build_registry`. The actual reorder-specific guard at `tests/architecture/dependency_guards/test_fk_helper_primitives_consume_policy.py:104` only asserts that `reorder_fk_helpers` does not import `FK_PATTERN`.

The implementation is currently clean, and `reorder_helpers_next_to_fk` imports none of those `core.fk` convention helpers. The issue is that the matrix claims stronger guard coverage than the test provides.

Recommended fix: narrow the prose to the actual guard coverage, or extend the reorder guard to check the full forbidden set.

==== M-03: Pipeline-step wrapper docstrings still describe old primitive behavior

`src/spreadsheet_handling/pipeline/steps.py:159` says `make_apply_fks_step` detects FK columns, and `src/spreadsheet_handling/pipeline/steps.py:178` says `make_drop_helpers_step` removes helper columns starting with a prefix. The wrapper implementations delegate to the policy-driven domain functions, so runtime behavior is correct, but the source docstrings contradict the semantic contract.

This was not introduced by `13bd123`, but it is in the FK-helper runtime surface inspected for this pilot. An agent reconstructing semantics from repository artifacts could treat these docstrings as stale truth.

Recommended fix: update the docstrings in a small docs-only/source-comment follow-up; no runtime change is needed.
