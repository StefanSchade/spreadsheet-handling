== 2. Exact review scope

The review tested only whether an ordinary deterministic but unhashable
physical column label, specifically a Python list `[1, 2]`, is rejected by
the XRef physical-label boundary before pandas membership or selection.

It also checked the explicitly requested boundary characterizations,
validation order, cleanup/non-mutation properties, changed documentation,
and shipped-consumer validation. It did not reopen matrix-axis semantics,
cleanup design, metadata design, Multiheader design, or adversarial custom
`__hash__` / `__eq__` behavior.

== 3. Verdict

The Review 004 defect is only partially closed.

`contract_xref` now rejects the exact selected-list relation label through
the deterministic XRef physical-label `ValueError`. The direct boundary also
correctly rejects list labels, while retaining numeric and hashable tuple
labels and preserving the missing-like / ambiguous-equality diagnostic order.

The exact selected-list `base_relation` path remains defective.
`expand_xref` calls `_ensure_output_names_do_not_collide` before it loads and
validates `base_relation`. When the configured `value` field is the list
label `[1, 2]`, that helper evaluates `{column_key, value}` and raises raw
`TypeError: unhashable type: 'list'`. The new base-relation test uses the
list only as an extra, unselected column, so it does not cover this Review
004 counterexample.

The remaining failure is cleanup-free and non-mutating, but it violates the
required deterministic XRef diagnostic and validation-before-membership
contract. XRef therefore remains unaccepted as the foundation for later
composed families.

== 5. Relation and base-relation probe results

=== Relation contraction

[source,python]
----
labels = np.empty(3, dtype=object)
labels[:] = ["r", "k", [1, 2]]
frames = {
    "rel": pd.DataFrame(
        [["r1", "A", "x"]],
        columns=pd.Index(labels, dtype=object),
    ),
    "_meta": {"sentinel": {"keep": ["unchanged"]}},
}

contract_xref(
    frames,
    relation="rel",
    output="out",
    row_keys="r",
    column_key="k",
    value=[1, 2],
    drop_source=True,
)
----

Result: deterministic XRef `ValueError` identifying the unhashable physical
label. No `out` frame was created, no XRef or cleanup metadata was written,
and deep frame/meta comparisons showed no input mutation.

=== Base-relation expansion

[source,python]
----
labels = np.empty(3, dtype=object)
labels[:] = ["r", "k", [1, 2]]
frames = {
    "matrix": pd.DataFrame([{"r": "r1", "A": "x"}]),
    "base": pd.DataFrame(
        [["r9", "B", "kept"]],
        columns=pd.Index(labels, dtype=object),
    ),
    "_meta": {"sentinel": {"keep": ["unchanged"]}},
}

expand_xref(
    frames,
    matrix="matrix",
    output="out",
    row_keys="r",
    column_key="k",
    value=[1, 2],
    base_relation="base",
    drop_source=True,
)
----

Result: raw `TypeError: unhashable type: 'list'`, not the XRef physical-label
`ValueError`. No `out` frame, XRef metadata, or cleanup command was produced,
and deep frame/meta comparisons showed no input mutation.

== 6. Validation-order assessment

Inside `_ensure_unique_physical_labels`, the implemented order is correct:

. missing-like rejection;
. ambiguous-equality rejection;
. unhashable rejection;
. duplicate detection.

For relation contraction, this boundary runs before configured-column
membership and selection, so the full required order holds.

For `base_relation` expansion, the effective operation order is not yet the
required one. Matrix validation and output-name collision detection occur
before the base frame is loaded. A selected unhashable base label therefore
reaches Python set construction before the physical-label boundary. It does
not reach pandas or write state, but it still produces the exact raw
`TypeError` that Review 004 required the boundary to replace.

The statement that no reviewed path reaches pandas with an unhashable
physical label is true after `9ea4b5a`; the stronger required statement that
the label receives the deterministic XRef boundary diagnostic before any
membership/selection-like use is false on the selected base path.

== 8. Cleanup and non-mutation assessment

Both exact selected-list rejection probes produced no output frame, XRef
metadata, or `_meta.pipeline_cleanup` command and left input frames and
pre-existing sentinel metadata unchanged. The remaining base-path defect is
therefore a diagnostic/validation-order failure, not silent corruption or
cleanup loss.

A valid full contraction still writes exactly `drop_frames: [rel]`; a
separate valid full expansion still writes exactly `drop_frames: [matrix]`.

== 9. Documentation consistency assessment

The Functional Model, Technical Model, User Guide, meta registry,
pipeline-step registry, release notes, epic record, and fourth correction
record consistently preserve the intended distinction:

* matrix-axis identities remain unique, non-empty strings;
* physical labels are non-missing, hashable, uniquely and deterministically
  comparable, and scalar-addressable, but not globally string-only;
* valid numeric and hashable tuple labels remain possible at the boundary;
* current Tuple/MultiIndex operation support is not promised;
* stable-id-versus-display-label modelling remains open.

Their completion claim is not yet accurate for the selected unhashable
`base_relation` field, which still receives a raw `TypeError` before the base
boundary. The fourth correction record specifically claims that the rejected
unhashable cases produce "no raw `TypeError`"; the probe above is a direct
counterexample.

There is also one minor historical wording error introduced in the fourth
correction record: "Review 004 ... accepted the slice" and the heading
"after Review 004 acceptance" contradict Review 004's explicit
`XREF PROJECTION-FAMILY SLICE NOT ACCEPTED` verdict.

== 11. Remaining findings by severity

=== Blocking

. `expand_xref` still raises raw `TypeError: unhashable type: 'list'` when an
  unhashable physical `base_relation` label is the configured `value` field.
  `_ensure_output_names_do_not_collide` constructs a set from that label
  before the base frame reaches `_ensure_unique_physical_labels`. This is the
  exact Review 004 base-path contract and is not characterized by the new
  test.

=== Important

None.

=== Minor

. The fourth correction record incorrectly says Review 004 accepted the
  slice, although Review 004 explicitly withheld acceptance.

== 12. Human decisions

No new human decision is required. The accepted boundary already determines
the behavior: ordinary unhashable physical labels must receive the XRef
diagnostic before membership/selection-like use. Closing the remaining path
is a local validation-order correction and focused regression test; it does
not require a string-only physical-label rule, Tuple/MultiIndex support, or
stable-id/display-label design.

== 13. XRef acceptance and composed-family readiness

The Review 004 defect is not fully closed because the exact selected-list
`base_relation` path still raises a raw `TypeError`. XRef remains unaccepted,
and later composed families must not yet rely on its physical-label boundary
as complete.
