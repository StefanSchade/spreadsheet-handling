===== FTR-SCHEMA-EVOLUTION-OPERATIONS -- Explicit one-shot schema maintenance operations

*Status:* New

*Priority:* P5

*Type:* FTR definition / pre-`0.3.0` maintenance workflow slice.

*Assessment:*::

This FTR is based on the accepted assessment in
`docs/backlog/FTR-SCHEMA-EVOLUTION-OPERATIONS-ASSESSMENT.adoc`.
The assessment concluded that a narrow pre-`0.3.0` FTR is justified.
This document does not re-open that decision.

==== Re-anchoring note

This revision supersedes earlier wording that conflated authoritative model
data with `json_dir` carrier storage.  The first tested workflow may use
`json_dir` plus `_meta.yaml` because that is the currently declared
authoritative storage workflow for the consumer shape under discussion, but
that storage choice is not a domain rule.  Git history is sufficient for the
previous wording; no separate active `_v1` copy is kept.

==== Purpose

Define explicit one-shot schema maintenance operations for authoritative frame
columns:

* `add_column`
* `drop_column`
* `rename_column`
* `reorder_columns`

The goal is to give consumers a controlled workflow for intentional
model-shape maintenance without treating arbitrary workbook edits, carrier
details, or dynamic projections as schema evolution.

==== Problem

The current architecture intentionally separates domain frames, workbook
projection, spreadsheet sheets, concrete carriers, and metadata roles.
Ordinary roundtrip editing is correct when the model shape is stable: users
edit values through a declared projection and the system reverses known
projections back into frames.

That leaves a maintenance gap.  When a user intentionally changes the shape of
an authoritative frame, manual storage edits and ad-hoc spreadsheet header
edits are both weak workflows.

Manual storage edits are possible, but they do not help with metadata
references, review reports, or repeated updates across many records.  Workbook
header edits are worse: they blur value editing with schema mutation and can
silently drift from validation rules, helper policies, workbook-view metadata,
projection configuration, and other model intent.

The project needs an explicit, reviewable way to perform small schema
maintenance operations over `Frames`.

==== Goals

* Define a small operation vocabulary for authoritative frame column
  maintenance.
* Keep schema maintenance separate from ordinary workbook roundtrip editing.
* Keep domain operation semantics carrier-neutral and expressed over `Frames`.
* Keep carrier kind, artifact role, data role, and meta role distinct.
* Use the current `json_dir` plus optional `_meta.yaml` workflow as the first
  tested storage workflow without making it a domain rule.
* Provide a dedicated maintenance CLI app that builds a private `BoundStep`
  programmatically and delegates execution to `orchestrate()`.
* Keep the private maintenance step out of the public pipeline registry,
  registry review artifact, and normal `sheets-run` YAML surface.
* Prefer fail-fast behavior over silent metadata drift.
* Update, prune, or fail recognized metadata references only under explicit
  rules; block ambiguous or unsupported references.
* Treat `_meta.derived` and other derived/transient metadata as
  non-authoritative.
* Require dry-run/report semantics so changes can be reviewed before writing.
* Keep the first slice small enough for pre-`0.3.0`.

==== Non-Goals

This FTR does not introduce:

* automatic schema inference from workbook edits;
* arbitrary spreadsheet table discovery;
* named ranges as table anchors;
* multiple logical tables per sheet;
* automatic table-region detection;
* visible cell typing;
* automatic migration discovery;
* migration history;
* migration versioning;
* migration graphs;
* compatibility layers;
* automatic schema evolution during normal roundtrip;
* a generic migration framework;
* a generic metadata rewriting framework;
* a generic dry-run framework;
* dynamic matrix-value column handling;
* treating dynamic matrix axis members as schema fields;
* ordinary roundtrip editing changes;
* FK relation redesign;
* `split_column`;
* `merge_columns`;
* `move_column_between_frames`;
* `change_fk_relation`.

Unknown plugin-owned metadata is not automatically rewritten in this slice.
Plugin-owned metadata blocks only when the operation detects a structured
frame/column reference that targets the affected column.  Generic string-search
detection and heuristic rewrite are out of scope.  Unsupported metadata
references must fail clearly.

==== Terminology

Frame::
A domain-level tabular data object carried in `Frames`, keyed by frame name.
A frame is not a workbook sheet and is not a persistence carrier.

Authoritative frame::
A frame whose data role is to hold the durable model data being maintained.
The first implementation may load such frames from `json_dir`, but the
operation semantics are defined for `Frames`, not for JSON files.

Schema field::
A named column of an authoritative frame.  Examples include
`characters.name`, `characters.persona.motto`, `stories.theme`, or
`story_locations.place_id`.

Schema maintenance operation::
An explicit user-declared operation that changes the column shape or column
order of an authoritative frame.  It is a maintenance action over the
machine-near model data, not a normal workbook edit.

Carrier kind::
The concrete I/O backend kind used in an `orchestrate()` descriptor, such as
`json_dir`, `yaml_dir`, `xlsx`, or `ods`.  Carrier kind is an I/O choice and
must not decide domain semantics.

Artifact role::
The role a concrete artifact plays in a workflow, for example input storage,
output storage, or an operation report.  Artifact role is distinct from carrier
kind: a `json_dir` may be an input or output artifact, and the report is a
review artifact rather than frame data.

Data role::
The role a frame or column plays in the data model, for example authoritative
frame, relation frame, key column, FK column, discriminator column, or dynamic
projection column.

Meta role::
The role of metadata according to the meta-system vocabulary: canonical meta
claims express authoritative intent; derived meta describes recomputable
current-run or carrier-near state; operational-carrier shape is a storage form
for meta.  These roles do not make carrier storage itself authoritative data.

Ordinary roundtrip editing::
Editing values in a declared workbook projection while the configured model
schema stays stable.  Reimport reverses known projections and validates the
result.

Dynamic projection column::
A visible workbook column that represents an axis value or projected data value
rather than a schema field.  Example: a per-character matrix column in a
story-cast matrix view.

Recognized metadata reference::
A metadata or configuration reference whose target frame and target column are
known to the operation.  Only recognized references may be updated or pruned
automatically.

Unsupported reference::
A column reference in metadata, configuration, or plugin-owned state that the
operation cannot update with a clear frame/column target and an explicit rule.
Unsupported references are failures, not warnings.

SchemaMaintenanceResult::
The domain operation result shape:
`SchemaMaintenanceResult(frames, report)`.  `frames` is the updated `Frames`
payload.  `report` is a structured maintenance report.  The report is not a
frame and must not be stored in `_meta`.

==== Architecture

Schema maintenance is a one-shot maintenance workflow, not ordinary pipeline
behavior.

The operation semantics live in domain-level code and operate on `Frames`.
They do not read workbook layout, sheet geometry, named ranges, spreadsheet
headers, backend objects, or carrier files directly.  Domain operations return
`SchemaMaintenanceResult(frames, report)`.

The first user-facing surface is a dedicated maintenance CLI app.  The CLI app
must:

* parse maintenance-specific user arguments;
* build normal `orchestrate()` input and output descriptors;
* build a private `BoundStep` programmatically;
* call `orchestrate(input=..., output=..., steps=[...])`;
* collect the domain report through a local collector owned by the CLI call;
* write or print the report as a review artifact outside `Frames` and outside
  `_meta`.

The private `BoundStep` adapts the domain result to the pipeline contract:
`Frames -> Frames`.  Its closure calls the domain operation, stores the
returned report in the CLI-local collector, and returns only
`SchemaMaintenanceResult.frames` to the orchestrator.

The private maintenance step must not be registered in
`pipeline.REGISTRY`, must not be added to
`registries/pipeline_step_registry.json`, and must not be addressable from
normal `sheets-run` YAML.  The dedicated CLI is the public maintenance surface
for this slice.

The first tested workflow may use:

* `input: {kind: json_dir, path: ...}`;
* optional `_meta.yaml` sidecar loaded by the JSON-directory backend;
* `output: {kind: json_dir, path: ...}` for write mode.

Those are workflow and adapter choices.  They do not change the domain rule:
schema maintenance operations are carrier-neutral operations over `Frames`.

Spreadsheet layout, visible table geometry, named ranges, sheet order,
workbook header edits, and backend carrier details are not inputs to schema
maintenance.

==== User-Visible Behavior

Every schema maintenance operation must be explicit.

The user identifies:

* the target frame;
* the operation kind;
* the source and/or target column names required by that operation;
* default value or placement options where the operation requires them;
* whether to run dry-run or write mode;
* where the report should be emitted, if the CLI supports report output paths.

By default, the operation runs safely:

* no silent inference;
* no hidden metadata repair;
* no best-effort rename matching;
* no destructive update if known references cannot be handled;
* no change to ordinary roundtrip behavior.

The operation produces a structured report that separates at least:

* requested operation and target frame;
* frame data changes;
* metadata/configuration changes;
* reference updates, prunes, and blockers;
* skipped or unchanged items;
* blocking failures;
* warnings.

The domain report records frame, metadata, and reference changes.  The emitted
CLI report may enrich that domain report with carrier, input, and output
artifact descriptors.  The report is a maintenance review artifact.  It is not
a frame, not a sheet, not model data, and not persisted under `_meta`.

Dry-run is implemented by routing the orchestrator output to an output-only
null/discard saver or an equivalent orchestrator-compatible output sink.  This
is an implementation direction for this CLI path, not a generic dry-run
framework.  Dry-run must still execute the private step and produce the same
report class without writing changed storage artifacts.

Write mode must detect blocking failures before writing to the user-selected
output.  "No partial write" means no user-selected output write after blocking
failures are detected.  Filesystem-transactional replacement is not required
by this FTR unless separately scoped.

==== Operation Definitions

===== `add_column`

Purpose: add a new schema field to one authoritative frame.

Required behavior:

* Target frame must exist.
* Target column must not already exist, unless an explicit idempotent mode is
  included in the implementation slice.
* The operation must write a declared default value or empty value for every
  existing row.
* Existing row order is preserved.
* Existing column order is preserved except for the inserted column.
* Placement is explicit: append by default, or insert before/after a named
  existing column if the user declares placement.
* No validation, FK helper, workbook-view, or presentation metadata is inferred
  for the new column.

Metadata behavior:

* No metadata change is required by default.
* Optional metadata additions are out of the first slice unless they are
  explicitly scoped by the implementation FTR.
* If the new column should become part of validation or helper policy, that is
  a separate explicit configuration change.

===== `drop_column`

Purpose: remove an existing schema field from one authoritative frame.

Required behavior:

* Target frame must exist.
* Target column must exist.
* Values in the dropped column are removed from every record.
* Existing row order is preserved.
* Remaining column order is preserved.
* Dropping key columns, FK columns, discriminator columns, validation columns,
  or configured projection columns must fail unless the implementation slice
  defines a deliberately explicit force/prune mode.

Metadata behavior:

* The operation must scan recognized metadata/configuration references.
* If recognized references can be pruned safely under an explicit prune mode,
  the report must list every pruned reference.
* If no such mode is active, references to the dropped column are blocking
  failures.
* FK helper-policy references are blocking failures in the first slice unless
  an implementation FTR explicitly scopes a safe update.
* Unknown plugin-owned references are blocking failures only when detected as
  structured frame/column references to the target column.
* `_meta.derived` is not an authority and must not be used to justify the
  operation.

===== `rename_column`

Purpose: rename one schema field in one authoritative frame while preserving
values.

Required behavior:

* Target frame must exist.
* Source column must exist.
* Target column must not already exist.
* Values are preserved.
* Relative column order is preserved: the target name occupies the source
  column's position.
* Recognized metadata references must be updated atomically or the operation
  must fail.
* The report must list every updated metadata reference.

Metadata behavior:

* Validation references are updated only when the frame and column target are
  unambiguous.
* FK helper-policy references are updated only when the frame/column role is
  unambiguous and within first-slice scope; otherwise they are blocking
  failures.
* Workbook-view helper/editability/presentation references are updated only
  when they clearly target the same authoritative frame.
* Frame lifecycle and projection metadata are updated only for recognized
  frame/column references.
* `_meta.derived` is treated as transient materialized-state metadata.  It may
  be discarded, invalidated, or ignored per the implementation contract, but it
  must not become the authoritative rename source.
* Similarity matching, fuzzy rename detection, and "old header changed in the
  workbook" inference are forbidden.

===== `reorder_columns`

Purpose: change the stored/reviewed column order of one authoritative frame
without changing values or schema field names.

Required behavior:

* Target frame must exist.
* Requested column names must be unique.
* Unknown column names are blocking failures.
* The operation must either require a complete column order or support a
  bounded partial-order mode:
  ** `listed_first`: listed columns move to the front; omitted columns retain
     their prior relative order.
  ** `listed_last`: listed columns move to the end; omitted columns retain
     their prior relative order.
* Existing row order and all values are preserved.
* This operation does not change dynamic matrix axis order unless the target
  frame is itself the authoritative axis frame.

Metadata behavior:

* Most semantic metadata is unchanged.
* Explicit column-order metadata may be updated only where it is recognized as
  referring to the same authoritative frame.
* Unknown explicit ordering metadata is a failure only if it would become
  inconsistent; otherwise it is reported as unaffected.

==== Metadata Rules

The first implementation slice must define a concrete list of recognized
metadata references.  At minimum it must account for:

* validation rules (`columns`, `target_columns`);
* FK helper policies under `_meta.helper_policies.fk`;
* workbook-view and sheet helper-column references;
* frame lifecycle metadata;
* projection metadata that stores row keys, value columns, source columns, or
  column order;
* `_meta.derived` transient helper provenance.

Rules:

. Canonical/persistent metadata is in scope for validation, update, prune, or
  fail-fast behavior.
. Derived/transient metadata is not authoritative.  It must not be used to
  infer user intent.
. `_meta.derived` may be dropped, invalidated, or ignored only under explicit
  implementation rules; it is never the source of truth for a schema operation.
. Unknown metadata roots are not rewritten.
. Ambiguous references fail.
. Detected plugin-owned structured frame/column references fail unless a
  plugin-specific reference contract is explicitly registered in a later slice.
  Generic string-search detection and heuristic rewrite are forbidden.
. Every metadata mutation appears in the operation report.
. Every metadata reference that blocks the operation appears in the operation
  report with enough context for the user to fix it manually.

==== Failure Behavior

Failures are preferred over silent drift.

Blocking failures include at least:

* missing target frame;
* missing source column;
* existing target column for rename/add;
* duplicate requested column names;
* unknown requested column names in reorder;
* key/FK/discriminator/validation column drop without explicit safe mode;
* known metadata reference that cannot be updated or pruned safely;
* ambiguous metadata reference;
* unknown plugin-owned structured frame/column reference that targets the
  affected column;
* malformed `_meta` shape that prevents safe reference analysis.

Warnings are allowed only for non-blocking observations that cannot change the
result, for example an unaffected metadata root.

No operation may write to the user-selected output after a blocking failure has
been detected.  This FTR does not require filesystem-transactional replacement
for writes unless a later implementation slice explicitly scopes it.

==== Dry-Run and Report Expectations

Dry-run is required for this FTR.

The report must be machine-readable or otherwise structured enough for tests to
assert:

* operation kind;
* target frame;
* source/target columns where applicable;
* write mode vs dry-run mode;
* frame changes that would be applied or were applied;
* metadata changes that would be applied or were applied;
* metadata references updated;
* metadata references pruned;
* blocking failures;
* warnings.

These are domain report expectations.  The CLI may enrich the emitted report
with review-safe carrier, input, output, and storage artifact descriptors.

The report should be usable in consumer review workflows before committing
schema changes to the authoritative storage repository.

The report must not be encoded as a frame, injected into `Frames`, stored as a
sheet, or persisted under `_meta`.

==== Validation Strategy

The implementation FTR must include tests in these categories:

* Operation unit tests on small `Frames` fixtures:
  ** success cases for all four operations;
  ** missing source column;
  ** duplicate target column;
  ** duplicate requested order;
  ** unknown ordered column;
  ** idempotent mode only if explicitly included.
* Metadata-reference tests:
  ** rename updates known validation references or fails;
  ** drop fails or prunes according to explicit mode;
  ** FK policy references are not silently broken;
  ** workbook-view helper references are updated or rejected;
  ** derived/transient metadata is not treated as authoritative.
* `json_dir` integration tests for the first declared workflow:
  ** operation over frames loaded from JSON-directory storage;
  ** `_meta.yaml` sidecar handling;
  ** dry-run uses a null/discard output sink or equivalent
     orchestrator-compatible saver and does not write storage artifacts;
  ** write mode produces stable output.
* Dedicated maintenance CLI tests:
  ** CLI delegates to `orchestrate()`;
  ** CLI builds a private `BoundStep` programmatically;
  ** local collector captures the `SchemaMaintenanceResult.report`;
  ** report is emitted outside `Frames` and outside `_meta`.
* Registry and YAML guardrail tests:
  ** private schema-maintenance step is not in `pipeline.REGISTRY`;
  ** private schema-maintenance step is not in
     `registries/pipeline_step_registry.json`;
  ** private schema-maintenance step is not addressable from normal
     `sheets-run` YAML.
* Worldbuilding-shaped fixture tests:
  ** dotted field names such as `persona.motto` and
     `story.learning_goal`;
  ** relation fields such as `home_place_id`, `story_id`,
     `character_id`;
  ** validation and helper-policy references.
* Guardrail tests:
  ** operation does not read workbook layout or sheet geometry;
  ** ordinary roundtrip editing behavior is unchanged;
  ** dynamic projection columns are not treated as schema fields;
  ** advanced table-discovery concepts are not introduced.

==== Architectural Placement and Gates

This FTR is expected to affect a user-visible maintenance CLI surface.  Any
implementation slice that adds CLI commands, flags, or documented public
behavior must pass the public-surface gate in
`docs/ai_info/interfaces_and_gates.adoc`.

This FTR does not authorize a public pipeline step.  The implementation must
not register the private schema-maintenance step in `pipeline.REGISTRY`, must
not add it to `registries/pipeline_step_registry.json`, and must not expose it
through normal `sheets-run` YAML.

If a later FTR wants a YAML-addressable maintenance pipeline form, that later
FTR must explicitly re-open the registry and public YAML surface question.

If an implementation adds or changes persistent metadata roots, it must follow
the metadata/registry slicing rules and update the relevant metadata registry
artifacts in that implementation slice.

No migration-history or compatibility-layer gate is opened by this FTR because
those topics are non-goals.

==== Acceptance Criteria

* The implementation defines the four in-scope operations:
  `add_column`, `drop_column`, `rename_column`, and `reorder_columns`.
* Domain operations are carrier-neutral and operate on `Frames`.
* Domain operations return `SchemaMaintenanceResult(frames, report)`.
* The schema maintenance report is not a frame and is not stored in `_meta`.
* The private `BoundStep` adapts `SchemaMaintenanceResult` to `Frames ->
  Frames`, and the CLI captures the report via a local collector.
* The maintenance surface is a dedicated CLI app that builds the private
  `BoundStep` programmatically and calls `orchestrate()`.
* The private schema-maintenance step is not registered in `pipeline.REGISTRY`.
* The private schema-maintenance step is not added to
  `registries/pipeline_step_registry.json`.
* The private schema-maintenance step is not addressable from normal
  `sheets-run` YAML.
* The first tested storage workflow may use `json_dir` plus optional
  `_meta.yaml`, but carrier kind is not a domain rule.
* Ordinary roundtrip editing remains unchanged and is explicitly documented as
  separate from schema maintenance.
* Dynamic projection columns are explicitly excluded from schema maintenance
  semantics.
* Dry-run uses an output-only null/discard saver or equivalent
  orchestrator-compatible output sink and writes no changed storage artifacts.
* Write mode produces a reviewable report of frame and metadata changes.
* Blocking failures prevent writes to the user-selected output after failure
  detection; filesystem-transactional replacement is not required unless
  separately scoped.
* Metadata handling is fail-fast for unknown, ambiguous, or unsupported
  references.
* Recognized references are updated, pruned, or failed according to explicit
  rules.
* `_meta.derived` and other transient metadata are not treated as
  authoritative schema intent.
* Tests cover operation behavior, metadata-reference behavior, the first
  `json_dir` workflow, dedicated CLI behavior, registry/YAML guardrails,
  worldbuilding-shaped fixtures, and the guardrails listed in the validation
  strategy.
* Public-surface and release-note updates are included in the implementation
  slice if the dedicated CLI surface is added.
