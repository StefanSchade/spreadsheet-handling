This review intentionally uses the existing review context from:

The review question was whether the FTR accurately captures the architectural

normalisation, render-op vocabulary, and backend carrier details remain

keeps the scope focused on authoritative-carrier semantics for existing
presentation metadata families and avoids drifting into a generic styling
framework.

I would approve implementation work from this FTR as written.

== Findings

Blocking:: None.

Before implementation:: None.

Watch-only:: The proposed helper name includes `cell_addressed` while the FTR
also intentionally migrates `column_widths`, which are dimension-addressed.
The FTR mitigates this by repeatedly stating that the helper stores whatever
the extractor returns and does not unify address keying. This is not a
blocker, but implementers should not let the name imply cell-address
validation or address-shape abstraction.

== Positive Observations

The FTR correctly captures M1.

It identifies the actual shared concern as the duplicated
embedded-workbook-meta update policy, not carrier extraction, vocabulary,
normalisation, render-op construction, or ODS parser safety. The statement
that extraction and carrier logic are backend-local is explicit in the Problem,
Goal, Backend Strategy, and Non-Goals sections.

The FTR preserves the canonical-vocabulary boundary.

It explicitly rejects a central canonical-vocabulary module or enum across
families and keeps render-op vocabulary ownership family-local. That matches
the concept analysis conclusion that vocabulary duplication is not the same
shared concern as replace-or-clear metadata authority.

The helper / retrofit bundling is architecturally coherent.

The FTR preserves both rationales from the concept analysis:

* a helper used only by horizontal alignment would be incomplete and would not
  prove the consolidation value; and
* retrofitting `text_orientations` and `column_widths` without a helper would
  harden the same duplication M1 asked the next slice to avoid.

The FTR is strict about backend boundaries.

It keeps ODS `ParserLimits`, carrier placement, style-map walking, and
normalisation in the ODS parser, while keeping openpyxl extraction/default
handling in the XLSX parser. The shared helper operates only on the already
parsed `workbook_meta` dict.

The non-goals are unusually strong.

They explicitly exclude vertical alignment, wrap, new render ops, new
metadata families, registry changes, generic styling, flow-builder
consolidation, `_merge_xlsx_alignment` refactoring, address-keying
unification, and canonical-vocabulary centralisation.

== Scope Risks

The largest risk is accidental cleanup around the helper. The FTR correctly
calls this out: vocabulary constants, `_*_ops` flow builders,
`_merge_xlsx_alignment`, and carrier extraction are tempting adjacent edits
but outside the FTR.

The second risk is over-reading `column_widths` as "cell-addressed" because of
the helper name. The FTR's substance keeps this safe by defining the helper as
family-payload storage, not address-shape logic.

No scope appears unnecessarily large. Bundling stale-meta retrofit for
`text_orientations` and `column_widths` with the helper is the smallest
coherent change that resolves both the known latent defect and M1.

== Missing Acceptance Criteria

No blocking acceptance criteria are missing.

Optional tightening:

* Add an explicit helper-level acceptance criterion for the full return-value
  truth table already listed in Scope: no-op/equal returns `False`,
  replace/different returns `True`, clear/existing returns `True`, and
  empty/absent returns `False`.
* Add an explicit assertion that the shared helper does not create
  `workbook_meta["sheets"]` or a sheet entry on an empty clear/no-op path.

These are refinements, not approval blockers, because the Scope section
already defines the four helper cases clearly.

== Recommendation

Approve implementation work from this FTR as written.

Minimum changes required before implementation:: None.

Implementation review should specifically verify:

* all six parser call sites invoke the helper unconditionally;
* the helper is the only home for replace-or-clear logic;
* stale-meta tests exist for `text_orientations` and `column_widths` on both
  backends;
* helper tests prove unrelated metadata is preserved; and
* no vocabulary or styling abstraction is introduced under the cover of this
  maintenance FTR.
