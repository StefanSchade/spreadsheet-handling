= FTR-POST-RELEASE-LIFECYCLE-AND-RESET-P5

== Status

Open

== Type

Process Improvement

== Context

The v0.2.0 release exposed a missing post-release activity.

The release process currently describes how a release is prepared and published,
but does not clearly define the transition back into normal development mode.

Several release-specific artifacts remain in a frozen state after publication,
including:

* version-baked README links
* release notes describing the just-published release
* release-specific documentation state

Without an explicit post-release procedure, development branches may continue
from a release-oriented state.

== Decision Direction

Release notes are not intended to become a permanent historical archive.

The project prefers:

* concise release notes focused on the current release line
* project history documented elsewhere
* archived documentation versions for historical reference
* Git tags and repository history as the authoritative release record

Release notes should therefore transition to the next development cycle after a successful release.

Example:

[source,text]
----
0.2.0 released → 0.3.0 planned
----

rather than accumulating an indefinitely growing release history.

== Goal

Define and document the activities required after a successful release.

== Candidate Post-Release Activities

* restore README links from versioned URLs back to latest
* open the next release-notes section
* transition release notes from released to planned state
* verify Pages publication
* verify package publication
* record release observations and process improvements

== Non-Goals

* redesign of release notes structure
* migration of historical release information
* changes to existing published releases

== Validation

After a future release:

* release notes are reset for the next development cycle
* README references latest documentation again
* release runbook contains an explicit post-release section
* release state and development state are clearly separated
