===== FTR-MODEL-DEFAULT-CONFIGURATION-P5 -- Derive common configuration from compact model declarations

*Status:* New

*Purpose:*::

Reduce configuration overhead for consumers by allowing common model
characteristics to be declared once and expanded into existing framework
configuration structures.

The goal is not to introduce hidden framework behavior.

The goal is to generate explicit existing configuration artifacts from a more
compact model-oriented declaration.

*Context:*::

Consumer validation revealed that many pipeline configurations repeatedly
express information already implied by the model.

Examples include:

* primary key columns,
* label columns,
* foreign key relations,
* helper definitions,
* workbook view defaults,
* validation rules,
* common projection settings.

In many cases the framework already possesses the required capabilities, but
consumers must repeatedly describe them in low-level configuration.

This creates adoption friction and increases maintenance effort when models
grow.

*Example:*::

Instead of repeatedly configuring:

[source,yaml]
----
validations:
  ...
helpers:
  ...
views:
  ...
projection_settings:
  ...
----

a consumer could declare:

[source,yaml]
----
model:
  frames:
    places:
      id: id
      label: name
    regions:
      id: id
      label: name
  relations:
    - source: places.region_id
      target: regions.id
      helpers: [name]
----

and derive existing framework configuration from that declaration.

*Non-Goals:*::

This FTR does not introduce:

* implicit runtime heuristics,
* automatic hidden behavior,
* special-purpose business semantics,
* replacement of existing low-level configuration.

Generated configuration must remain inspectable and understandable.

*Acceptance Criteria:*::

* A compact model-oriented declaration format is defined.
* Existing framework concepts remain the execution model.
* Generated configuration is explicit and reviewable.
* A consumer can reduce repetitive configuration without losing control.
* The feature is demonstrated on at least one real consumer repository.
* The approach is evaluated against model growth scenarios where new attributes
  are introduced without extensive pipeline modifications.

*Discussion Topics:*::

* Which concepts are suitable for derivation?
* Which concepts should remain explicit?
* How should generated configuration be inspected?
* Should generation occur as a dedicated configuration step?
* How does the feature interact with Dynamic Frame Projection?
* How does the feature interact with workbook presentation defaults?
