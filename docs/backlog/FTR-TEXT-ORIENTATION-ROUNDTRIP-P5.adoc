= FTR-TEXT-ORIENTATION-ROUNDTRIP-P5

== Status

Open

== Type

Feature

== Context

Column widths survive XLSX roundtrips since FTR-COLUMN-WIDTH-ROUNDTRIP-P4
(done, cold storage).

User-authored text orientation (vertical text, rotated headers) does not.
During the worldbuilding adoption (Adoption-001) this was a recurring pain
point: header cells with vertical or rotated text revert to horizontal on
re-export, forcing the user to reapply orientation by hand after every
roundtrip.

Text orientation belongs to the same `presentation` classification class as
column widths per the meta classification model
(`docs/technical_model/ch05_registries/registry_semantics/03_meta_classification.adoc`).
Losing it is an aesthetic issue, not a model integrity issue, but for
actively edited workbooks it is a real friction source.

The `FTR-COLUMN-WIDTH-ROUNDTRIP-P4` delivery explicitly deferred text
orientation and row heights as separate follow-up FTRs.

== Goal

Preserve user-authored cell text orientation through XLSX roundtrips,
following the established `column_widths` pattern:

. Parse text orientation from XLSX into canonical presentation meta.
. Persist it under a registry-backed key.
. Derive an internal view via a rendering pass.
. Emit backend-neutral render operations.
. Apply them in the XLSX renderer.

== Scope

* Define a metadata key (e.g. `text_orientations`) in the meta registry,
  classified as `presentation`, canonical layer.
* Parse column- or cell-level text orientation from XLSX
  (openpyxl `Alignment.text_rotation`).
* Emit a backend-neutral render operation (e.g. `SetTextOrientation`).
* Apply the operation in the XLSX renderer.
* Add focused roundtrip tests (parse, render apply, parse-write-parse
  preservation, missing-metadata safety).

== Non-Goals

* ODS text orientation parity is an accepted gap in this slice if the
  backend semantics differ materially; document the gap.
* No row-height roundtrip (separate follow-up FTR).
* No general cell-style preservation.
* No auto-detection or inference of orientation from cell content.
* No new YAML configuration surface for authoring orientation from scratch.

== Validation

* An XLSX workbook with vertical or rotated header cells is parsed and
  orientation appears in metadata under the registered key.
* A parse-write-parse roundtrip preserves at least two distinct orientation
  values.
* Missing orientation metadata does not fail rendering; existing defaults
  are preserved.
* The meta registry carries the new key with `presentation` classification.
* Existing tests continue to pass.
