== Executive summary

`FTR-DYNAMIC-VIEW-COLUMN-TARGETING-P5` was written before the column-role
foundation existed. It uses the word *payload* as a primary category, frames
target sets by *exclusion* (`exclude_helper_columns`, `exclude_key_columns`),
and lists *editable-state rendering* among its motivating downstream
concerns. Each of those choices is now misaligned with either the foundation
FTR or the projection decision, but none of them is load-bearing for the
FTR's actual question.

The targeting FTR's real question -- *how do post-projection concerns
address dynamic projected column sets without enumerating column names?* --
is unchanged by the foundation and the projection decision. Both upstream
FTRs deliberately stop at "the role classification exists and is queryable"
and at "the projection step consumes that classification to shape the
view"; neither extends into the rule-addressing problem. So the targeting
FTR is *not* superseded.

The right disposition is *refine*, not *split*, and not *supersede*. The
refinements are wording- and scope-level: declare dependencies on both
upstream FTRs, replace "matrix payload columns" with `matrix_value`,
recast exclusion-vocabulary targeting as positive role-based targeting,
narrow Option B from "selector semantics" to "targeting by role", clarify
the "specific logical class" hint, and add the standard role-taxonomy and
editability/merge/lifecycle Non-Goals. No new section is added; no scope
beyond rule-target addressing is introduced.

After refinement, the FTR remains an *evaluation* FTR with three viable
landings (A keep plugins / B add a small role-based targeting surface /
C reuse existing metadata). The decision itself is not made by this
review; a future targeting decision artifact -- analogous to
`FTR-DYNAMIC-FRAME-PROJECTION-P5_decision.adoc` -- is the appropriate
home for it.

== Concern separation

The three FTRs partition the projection / addressing / classification
problem cleanly. The targeting FTR must stop at its own border.

[cols="1,3"]
|===
| Concern | Owned by

| *Column-role taxonomy* (the names `row_identity`, `display_helper`,
  `matrix_value`; their definitions; the reimport-survival rule; the
  metadata-source expectations).
| `FTR-PROJECTED-FRAME-COLUMN-SEMANTICS-P5`.

| *Projection mechanism* (how a generated view's *shape* is produced --
  outbound ordering / retention by role, and the symmetric inbound
  retention). One step `project_by_role` per the decision artifact.
| `FTR-DYNAMIC-FRAME-PROJECTION-P5` (+ its decision artifact).

| *Post-projection rule addressing* (how validation, presentation, or
  workbook-constraint rules address *projected* column sets by role
  without enumerating column names).
| `FTR-DYNAMIC-VIEW-COLUMN-TARGETING-P5` -- this FTR.

| *Selector grammar / wildcards / class system beyond three roles.*
| *Not owned by any current FTR.* If a richer class system proves
  necessary, it is a future FTR. Targeting must not silently absorb it.

| *Editability, merge-back, collaboration semantics, lifecycle.*
| Not owned by any of the three FTRs above. Touched by
  `FTR-PRESENTATION-DEFAULTS-SEMANTIC-CLASSIFICATION-P5`,
  `FTR-WORKBOOK-VIEW-PAYLOAD-CONFLICT-PRECEDENCE-P6`, and the
  collaboration-roundtrip concept. Targeting must not adopt them as
  motivating scope.

== Current overlaps with the foundation FTR and the projection FTR

Every place where the targeting FTR currently re-derives a concept that
upstream FTRs now own, or now better own.

[cols="1,2,2"]
|===

  all non-helper columns, all non-key columns, all projected columns
  derived from a specific logical class")
| Names the targetable sets by *exclusion* ("non-helper", "non-key") and

| Foundation FTR: target sets are nameable positively as
  `display_helper`, `row_identity`, `matrix_value`. "Payload" is

| Uses "payload columns" again as the recurring quantity.

| *Representative Example*, YAML using `exclude_helper_columns: true`,
  `exclude_key_columns: true`
| Defines targets by what they are *not*.
| With roles named, the same target is `roles: [matrix_value]` or

  parameter-restatement smell the driver review §4 flagged across both
  consumer FTRs.

| *Architectural Options*, Option C "Reuse Existing Metadata:
  Investigate whether workbook-view metadata, helper-column metadata,
  or future projection metadata can already expose enough information"
| Treats reuse of metadata as one option among three, still speculative.
| Foundation FTR names the classification; projection decision already
  records that the classification is the consumption baseline. Option C

The two FTRs do *not* overlap with the targeting FTR on:

* projection mechanism (how the view is shaped at runtime),
* role-source resolution (where projection reads role classification

* the project decision on plugins-vs-declarative for projection
  (already recorded; not re-evaluated here).

Those remain in their respective upstream FTRs.

== Hidden assumptions now owned upstream

Points where the targeting FTR's text presupposes concepts that
upstream FTRs now define. Left unrefined, they would either re-derive
the upstream content or contradict it.

. *"Payload" as a primary category.* The targeting FTR uses "matrix
  payload columns" as if "payload" were a defined role. Foundation FTR
  explicitly disclaims that use: "this is a deliberate disambiguation
  against `FTR-PAYLOAD-SEMANTICS-RECOVERY-P5`, which uses 'payload' in
  the unrelated sense of 'visible spreadsheet cells whose meaning
  depends on metadata'." The correct name is `matrix_value`.
. *Target sets defined by exclusion.* `exclude_helper_columns: true`
  and `exclude_key_columns: true` presuppose that there is a privileged
  "rest" set that is named negatively. Foundation FTR names that rest
  positively as `matrix_value`, scoped to projected matrix / view
  surfaces only, and explicitly states that rest-by-exclusion *is not*
  a global rule. The exclusion vocabulary should not survive
  refinement.
. *Helpers and keys appear in positional locations.* The targeting
  FTR's Context lists `title` and `story_id` by name and treats them
  as fixed. Foundation FTR *Future Design Pressure (Non-Scope)* states
  that `display_helper` is positional-neutral; targeting must not
  hardwire helpers to a leading position.
. *A "specific logical class" exists beyond the three roles.* The
  targeting FTR's Purpose hints at "projected columns derived from a
  specific logical class". This implies a class system richer than the
  three roles. Foundation FTR names exactly three roles and lists
  "Monolithic column meta model" as a Non-Goal: "It names exactly three
  roles. Future axes are explicitly out of scope." Targeting must
  either reduce this hint to "by role" or explicitly defer the richer
  class question to a separate future FTR.
. *Editable-state rendering belongs in scope.* The targeting FTR
  lists editable-state as a downstream consumer. Both foundation FTR
  and projection decision exclude editability. Targeting may *cite*
  editable-state rendering as a future consumer of role targeting
  (since rendering rules will eventually need targets) but must not
  treat it as a motivating in-scope use case in its current form.
. *Selector semantics is the natural Option B.* The targeting FTR's
  Option B proposes a "small declarative selector model". With roles
  named upstream, the simplest sufficient Option B is *targeting by
  role*, not a selector grammar. A selector grammar is a much larger
  commitment and is not required by the worldbuilding case.

== Findings

Findings use the disposition pattern from
`docs/ai_info/review_instructions.adoc` § "Findings and prioritization":
observation, evidence, risk, recommended action, priority.

=== R1 -- Declare dependencies on the foundation FTR and the projection FTR (priority: required for refinement)

* *Observation.* The targeting FTR does not cite
  `FTR-PROJECTED-FRAME-COLUMN-SEMANTICS-P5` or
  `FTR-DYNAMIC-FRAME-PROJECTION-P5` anywhere.
* *Evidence.* Purpose, Context, Representative Example, and
  Architectural Options all describe role-classification concepts
  inline. Driver review §6 / §7 recommend recording both as
  dependencies.
* *Risk.* Without explicit dependency declarations, future readers may
  rederive the role taxonomy here or introduce a parallel selector
  vocabulary that converges with the projection FTR only at
  implementation time -- exactly the failure mode the driver review
  warned about.
* *Recommended action.* Add a top-level *Depends on:* block citing:
  - `FTR-PROJECTED-FRAME-COLUMN-SEMANTICS-P5` (foundation classification);
  - `FTR-DYNAMIC-FRAME-PROJECTION-P5` (the projected surfaces this FTR
    addresses), with the note that the projection decision artifact
    has selected the role-based projection mechanism this FTR consumes
    target sets from.

=== R2 -- Replace "payload" wording with `matrix_value` and disclaim against payload-semantics-recovery (priority: refinement)

* *Observation.* The targeting FTR uses "matrix payload columns", "set
  of visible payload columns", and "payload column" in Purpose, Context,
  and Representative Example.
* *Evidence.* Foundation FTR Scope, `matrix_value` row, explicitly
  disclaims this usage of "payload" and states the disambiguation
  against `FTR-PAYLOAD-SEMANTICS-RECOVERY-P5`.
* *Risk.* "Payload" silently conflates the projection's *value cells*
  with the payload-semantics-recovery rule's *visible cells whose
  meaning depends on metadata*. The two concepts are unrelated; the
  conflation already caused parallel FTRs.
* *Recommended action.* Replace "matrix payload columns" with
  `matrix_value` columns (or "matrix value cells") in all occurrences.
  Add an explicit one-sentence disclaimer in the targeting FTR's
  Relationship / Discussion Note section pointing at the foundation
  FTR's disambiguation paragraph.

=== R3 -- Recast exclusion-vocabulary targeting as positive role-based targeting (priority: refinement)

* *Observation.* Representative Example uses
  `exclude_helper_columns: true` and `exclude_key_columns: true`.
  Purpose and Context use the equivalent prose ("non-helper", "non-key").
* *Evidence.* Foundation FTR provides positive names; projection
  decision artifact's Option B surface uses `direction`/`role_order`
  with positive role names. Driver review §4 identifies the exclusion
  vocabulary as the parameter-restatement smell.
* *Risk.* Negative-form selectors silently presuppose the privileged
  three-role partition and re-introduce the foundation's narrowed
  rest-by-exclusion as a global rule. They also create gratuitous
  vocabulary divergence with the projection step's positive-role
  surface.
* *Recommended action.* Recast targets in role terms. The YAML example
  should be illustrative only (the FTR commits to no syntax) but should
  read like:
+
[source,yaml]
----
- step: add_validations
  rules:
    - target:
        sheet: story_groups
        frame: story_group_matrix_view
        roles: [matrix_value]   # positive; foundation role
      rule:
        type: from_legend
        legend: story_group_codes
----
+
Keep the original exclusion form as a *contrast example* if useful,
labeled as the pre-foundation phrasing.

=== R4 -- Clarify the "specific logical class" hint (priority: refinement)

* *Observation.* Purpose includes "all projected columns derived from
  a specific logical class". This hints at a class system richer than
  the three foundation roles.
* *Evidence.* Foundation FTR Non-Goals lists "Monolithic column meta
  model. This FTR does not define a generic column-meta schema covering
  provenance, lifecycle, editability, formatting, validation, and i18n.
  It names exactly three roles. Future axes are explicitly out of
  scope."
* *Risk.* If the targeting FTR is read as implicitly committing to a
  richer class system, it tacitly enlarges the foundation's
  responsibilities. The smell is identical to the one the foundation
  was created to remove.
* *Recommended action.* Either (a) collapse the bullet into "by role
  (e.g. `matrix_value`)" with no other class language, or (b) keep it
  but explicitly defer: "Richer class systems beyond the foundation's
  three roles are explicitly out of scope here; if a use case requires
  one, it is a separate future FTR." Option (a) is preferred; option
  (b) is acceptable if the maintainer wants to keep the door visibly
  open.

=== R5 -- Remove editable-state rendering from the motivating downstream-concern list (priority: refinement)

* *Observation.* Purpose lists "editable-state rendering" as one of
  four downstream concerns (alongside validation, presentation,
  workbook constraints).
* *Evidence.* Foundation FTR *Future Design Pressure (Non-Scope)*:
  "Editability and merge-back policy remain explicitly out of scope
  here." Projection decision artifact also excludes editability.
* *Risk.* Listing editable-state in Purpose presents the targeting FTR
  as a vehicle for editability decisions, which it is not. Reviewers
  may treat the FTR as the right home for those decisions.
* *Recommended action.* Remove editable-state from the Purpose's
  downstream-concern list. *Optionally*, mention it in the Discussion
  Note as a *future* downstream consumer once editability is itself
  defined elsewhere, with an explicit cross-reference to the FTRs that
  own that question. Do not list it in scope.

=== R6 -- Reframe Option C from speculative to concrete consumption (priority: refinement)

* *Observation.* Option C is phrased as "Investigate whether ...
  metadata can already expose enough information".
* *Evidence.* Foundation FTR names the classification and the
  reimport-survival rule. Projection decision artifact records the
  consumption surface.
* *Risk.* If Option C is left as speculative investigation, the
  decision documented under *Acceptance Criteria* may revert to the
  pre-foundation framing and accidentally re-open the modelling
  question.
* *Recommended action.* Reframe Option C as "*Consume the foundation
  classification.* Address target sets by foundation role
  (`matrix_value`, `display_helper`, `row_identity`) on projected
  surfaces produced by the projection FTR's mechanism. The remaining
  design question is *where the consumption surface lives* -- on the
  existing validation / constraint step inputs, on a new dedicated
  targeting step, or by extending existing rule shapes -- and whether
  any additional declarative ergonomics are needed."

=== R7 -- Reduce Option B from "selector semantics" to "targeting by role" (priority: refinement)

* *Observation.* Option B proposes a "small declarative selector model
  for dynamic workbook-view columns" with "selector surface".
* *Evidence.* Foundation FTR plus projection decision narrow the
  requirement: addressing by role is the simplest sufficient surface
  for the worldbuilding case. A selector grammar is a larger
  commitment.
* *Risk.* A "selector model" framing invites grammar design, wildcards,
  set algebra, and composition rules -- exactly the surface area the
  driver review §5 warned against ("start at ergonomics and work
  down"). The targeting FTR's Non-Goal "The goal is not to introduce a
  second projection framework" is then quietly violated.
* *Recommended action.* Recast Option B as: "*Add a small targeting-by-
  role surface.* Allow validation / presentation / constraint rules to
  name a foundation role (e.g. `roles: [matrix_value]`) instead of
  enumerating column names. No wildcard grammar, no set-algebra, no
  composition operators are introduced." Selector grammar can be
  reopened as a separate later FTR if needed.

=== R8 -- Add Non-Goals for role-taxonomy redefinition and adjacent-domain exclusions (priority: refinement)

* *Observation.* Current Non-Goals list does not exclude role-taxonomy
  redefinition or the standard adjacent-domain concerns.
* *Evidence.* Same shape as projection FTR R7 (the role-taxonomy
  exclusion) and the projection FTR's projection-side adjacent-domain
  exclusions.
* *Risk.* Without explicit exclusions the targeting FTR may quietly
  expand scope on review.
* *Recommended action.* Add to *Non-Goals*:
  - "This FTR does not define or redefine the column-role taxonomy.
    The taxonomy and the reimport-survival rule are owned by
    `FTR-PROJECTED-FRAME-COLUMN-SEMANTICS-P5`."
  - "This FTR does not re-decide the projection mechanism; that
    decision is recorded in
    `FTR-DYNAMIC-FRAME-PROJECTION-P5_decision.adoc`."
  - "No selector-grammar design, no wildcard / set-algebra design, no
    editability or merge-back semantics, no collaboration semantics,
    no lifecycle modelling, no presentation-classification work.
    Those belong to other FTRs."

=== R9 -- Add the foundation as a third axis in the Discussion Note (priority: refinement)

* *Observation.* Discussion Note correctly separates two questions:
  "How is a generated view shaped?" (projection) vs "How do later
  concerns address the generated columns of that view?" (targeting).
* *Evidence.* The boundary statement is the original driver review's
  framing and remains correct.
* *Risk.* The two-axis framing now omits the third axis -- where the
  role classification lives. A reader could conclude that targeting
  defines the classification rather than reads it.
* *Recommended action.* Add a third bullet: "How is the column-role
  classification *defined*? -- answered by
  `FTR-PROJECTED-FRAME-COLUMN-SEMANTICS-P5`. The targeting FTR *reads*
  the classification; it does not define it." Keep the existing two
  bullets verbatim.

=== R10 -- Update Acceptance Criteria to reflect role-based addressing and the projection decision (priority: refinement)

* *Observation.* Current Acceptance Criteria are generic ("The
  worldbuilding use case is evaluated explicitly. Transferability ...
  A project decision is recorded. ...").
* *Evidence.* The worldbuilding case is now expressible in role terms;
  the projection decision artifact has formalized that style for the
  sibling FTR.
* *Risk.* Generic criteria allow the targeting decision to drift into
  selector-grammar design without anyone noticing.
* *Recommended action.* Sharpen the criteria:
  - "The worldbuilding use case is evaluated explicitly, expressed in
    terms of the foundation FTR's role taxonomy."
  - "The evaluation explicitly states that targeting reads the same
    role classification consumed by `FTR-DYNAMIC-FRAME-PROJECTION-P5`
    (no parallel taxonomy)."
  - "A project decision is recorded in a dedicated decision artifact
    analogous to `FTR-DYNAMIC-FRAME-PROJECTION-P5_decision.adoc`."
  - "If declarative support is preferred, the design direction names
    only role-based addressing as the in-scope surface; any wildcard /
    set-algebra surface is recorded as a separate follow-up question."
  - "It is documented whether consumer plugins remain the preferred
    solution."

== Suggested scope reductions

The targeting FTR can shrink along three axes without losing meaning:

* *"Payload" framing removed.* The single word change ripples through
  Purpose, Context, Representative Example, and Discussion Note.
  Net effect: ~6-8 phrase replacements; no structural change.
* *Selector-language ambition cut back.* Option B narrows from "small
  declarative selector model" to "small targeting-by-role surface".
  Net effect: removes implicit wildcard / set-algebra commitments.
* *Editable-state demoted.* One bullet removed from Purpose; optional
  cross-reference added in Discussion Note. No new content required.

The targeting FTR's *Motivation* paragraph (the "many spreadsheet-driven
maintenance surfaces follow the same pattern" passage) survives intact.
That paragraph is the FTR's load-bearing value proposition and remains
accurate post-foundation -- maintenance-surface ergonomics is the
business question role-based targeting answers.

== Remaining responsibilities of FTR-DYNAMIC-VIEW-COLUMN-TARGETING-P5

After refinement, the targeting FTR is responsible for exactly the
following, none of which is owned by upstream FTRs:

* *The decision.* Should validation / presentation / constraint rules
  be able to address projected column sets by role declaratively, or
  should this remain a consumer-side plugin concern? (Options A, B, C
  in their refined form.)
* *The mechanism.* If declarative, how do rules cite a role? Is it a
  field on each rule's existing target schema, a new dedicated
  targeting step, or a shared addressing helper consumed by multiple
  rule shapes? Coordination with the existing validation/constraint
  surfaces is required.
* *The surface.* If declarative, what does the YAML look like? The
  refined Option B narrows the surface to role-based addressing; a
  richer wildcard / set-algebra surface is out of scope and remains
  the subject of a separate later FTR if needed.
* *Coordination with existing validation and constraint steps.*
  Whether `add_validations` or equivalent already takes a target
  shape that can be extended is a targeting-FTR design question.
* *The recommendation* on whether plugin support remains the preferred
  solution.

The targeting FTR does *not* own:

* the column-role taxonomy (`row_identity`, `display_helper`,
  `matrix_value`) or its names,
* the reimport-survival rule for `row_identity` and `display_helper`,
* the metadata-source expectations,
* the projection mechanism or its decision,
* selector grammars, wildcards, set algebra,
* editability, merge-back, lifecycle, collaboration semantics,
* presentation classification.

== Verdict

* *Verdict:* *Refine.*
* *Not* keep-as-is: the FTR carries the "payload" framing, exclusion
  vocabulary, the editable-state inclusion, and the "selector
  semantics" Option B framing -- all of which conflict with the
  foundation FTR or the projection decision.
* *Not* split: the FTR's open question (declarative-vs-plugin for
  rule-target addressing) is a single well-shaped concern.
  Splitting "address by role" from "address by richer logical class"
  is premature; the richer class system is not motivated by any
  recorded use case and should remain out of scope.
