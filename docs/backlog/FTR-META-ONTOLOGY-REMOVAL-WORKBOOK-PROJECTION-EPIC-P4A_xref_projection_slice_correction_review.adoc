The correction closes the ordinary additional-relation-field defect, the

and the stale persistence-boundary docstring. It also correctly validates the
common relation, explicit-list, metadata-list, stored-dense-list, and physical

The shipped configurations remain compatible and the normal-path suites are
green. The remaining defects are narrow XRef validation defects; they do not
reopen the accepted lifecycle, Workbook View, or carrier architecture.

== 4. Remaining findings by severity

=== 4.1 Blocking -- duplicate and overlapping field declarations can produce an invalid representation and still schedule cleanup

The new unrepresented-field check uses a set of configured names and label
membership (`operation.py:232-235`). `_ensure_columns` only checks presence;
it does not validate uniqueness of the configured field list or uniqueness of
physical relation columns (`primitives.py:100-113`). This leaves several
mechanically detectable cases:

* `contract_xref(row_keys=['r', 'r'], drop_source=True)` succeeds, produces
  matrix columns `['r', 'r', 'A']`, and writes
  `pipeline_cleanup.drop_frames=['rel']`;
* a relation with physical columns `['r', 'k', 'v', 'v']` succeeds, indexes
  `source_row['v']` as a `Series`, stores that non-scalar object as a matrix
  cell, and writes cleanup;
* a relation with duplicate physical row-key columns reaches
  `_ensure_unique_pairs` and fails with raw `TypeError: unhashable type:
  'Series'`, rather than a deterministic contract error;
* `expand_xref(row_keys=['r'], value_columns=['r', 'A'],
  drop_source=True)` accepts the row-key column as a value column, emits a
  relation row whose column identity is `r`, and schedules the matrix for
  cleanup. Contracting that result naturally wants duplicate `r` columns.

The last case also shows why the subset guard alone is insufficient: every
non-row-key physical column is covered, but the explicit value set improperly
overlaps the row-key set. The inferred path remains coherent: in the absence
of explicit `value_columns`, every non-row-key matrix column is intentionally a
value column, and helper/technical columns remain the owning projection's
responsibility.

Required correction is local and mechanical: before any pandas selection,
set construction, metadata write, or cleanup write, reject duplicate physical
relation labels, duplicate `row_keys`, and overlap between explicit
`value_columns` and `row_keys`. Keep the existing output-name collision rule
and explicit/inferred ordering. Add both operations' regression cases and
assert that rejected input metadata contains no cleanup command.

=== 4.2 Blocking -- the unique non-empty string identity contract is not enforced on every source

The shared validators in `primitives.py:56-97` correctly reject `None`, NaN,
`pd.NA`, empty/whitespace-only strings, Python and NumPy numeric scalars,
mixed `1`/`"1"`, unhashable values, and duplicate identity lists _when those
values reach the validators_. Three paths do not satisfy the documented
contract:

. `expand_xref` validates the selected matrix labels but does not validate
  `base_relation[column_key]` before appending out-of-scope rows
  (`operation.py:125-143`). A base row with numeric identity `1` is accepted;
  the output relation contains that identity and the matrix cleanup command is
  written. The produced relation therefore violates the accepted roundtrip
  vocabulary and cannot later be contracted under the same contract.
. Live dense-column resolution inserts values into `seen: set[Any]` before the
  shared list validator runs (`dense_axes.py:200-215`). An unhashable list
  identity fails with raw `TypeError: unhashable type: 'list'`. Numeric live
  dense values happen to reach the later validator, but the path is not
  consistently validation-first.
. Expansion calls `source.columns.duplicated()` before validating selected
  physical matrix identities (`operation.py:73-94`). An unhashable physical
  label fails there with the same raw `TypeError` rather than the promised
  XRef contract diagnostic.

The first case is cleanup-unsafe because invalid output is accepted and the
source is marked for deletion. The latter two reject before cleanup, but
violate the explicit-diagnostic and validation-before-pandas requirements.

Validate scoped base-relation column identities, and route live dense and
physical matrix labels through the shared string validator before hash-based
duplicate detection. This requires no new identity encoding or human policy.

=== 4.3 Important -- sparse-default ambiguity is payload-filtered, while documentation claims all frame matches fail

Transformation lookup counts all mapping entries whose relation or matrix
identity matches (`primitives.py:125-168`), and column-role lookup counts all
mapping entries claiming the matrix. Sparse-default lookup instead adds a
matching entry only when its `column_keys` is a list
(`sparse_defaults.py:300-315`). With two entries for one matrix, one partial
and one carrying `column_keys`, it silently uses the latter. With two partial
entries it reports that explicit columns are required rather than ambiguity.

This is not insertion-order selection and it does not choose between two
usable payloads; two list-bearing candidates do fail. It is nevertheless not
the uniform "multiple matching entries fail" policy stated by the User Guide,
release notes, correction record, and meta registry. Either count all mapping
frame matches before inspecting `column_keys`, or narrow those claims to the
consumer's usable-payload rule and add partial/malformed-entry tests. No
generic precedence rule should be introduced.

=== 4.4 Important -- pipeline registry was not realigned with the correction

`registries/pipeline_step_registry.json` still contains only the original
subset/coverage guard wording:

* `expand_xref.drop_source` mentions unexpanded explicit subsets but not the
  string-identity or duplicate-physical-label contract
  (`pipeline_step_registry.json:2450-2455`);
* `contract_xref.drop_source` mentions uncovered relation column identities
  but not rejection of fields outside `row_keys + column_key + value`, nor the
  carrier-stable identity contract (`:2555-2560`);
* `expand_xref.meta_contract.reads` remains empty (`:2457-2459`), although
  expansion reads a prior XRef entry to carry durable dense-axis intent
  (`operation.py:148-169`).

The slice, epic, User Guide, release notes, and meta registry were updated, but
the explicitly required pipeline registry was not. Update it after the runtime
guards are complete so registry semantics do not overstate or omit behavior.

=== 4.5 Test completeness -- non-blocking once the runtime blockers are fixed

The dedicated schema predicate is shared by rename and drop, so the
implementation covers every required reference shape. The current tests do
not directly exercise both actions for every shape: top-level `column_keys`
drop and both `columns_from.key` / `columns_from.keys` forms are inferred from
the common loop rather than independently characterized. Add those cases,
plus explicit input-meta non-mutation, while touching the suite for the
blocking corrections.

Producer replacement and untouched sediment are also verified by simple
assignment and an empirical probe, but have no focused producer regression:
`_write_xref_meta` replaces `configs[config_id]` wholesale, so old
`operation`/`column_key`/`value`/`drop_empty` fields disappear for that id,
while other entries remain untouched. A small pinning test would protect this
intentional compatibility boundary.

== 6. Cleanup-safety assessment

The valid full contraction still writes exactly the explicit relation drop;
the valid full expansion writes exactly the matrix drop. Ordinary additional
relation fields, uncovered relation identities, unexpanded matrix fields, and
all common rejected scalar identity cases fail before `_write_xref_meta` and
`mark_frames_for_cleanup`. Input metadata is not mutated.

Cleanup safety is not complete because the accepted duplicate-row-key,
duplicate-physical-value, overlapping-value-column, and numeric-base-relation
probes write a cleanup command. These are observable counterexamples to the
correction record's "before any cleanup command" claim, not speculative later
carrier behavior.

`drop_empty=True` remains an accepted feature-local semantic: empty-only
column shape needs durable dense intent or a retained matrix. Inferred
expansion continues to treat all non-row-key physical columns as values; no
Workbook View/helper ontology is inferred.

== 7. Ambiguity assessment

`_xref_config` correctly gives an exact mapping id precedence and otherwise
collects every mapping matching the queried relation or matrix. Multiple
relation matches and multiple matrix matches fail independently. No live call
queries both relation and matrix, so its OR-union cannot create a current
false ambiguity. A partial exact mapping remains authoritative and consumers
fall back locally when its optional facet is absent; this is deterministic.

The strengthened exact-id test is behaviorally meaningful: different
candidate `column_keys` produce different column sets, and the assertion proves
which payload was consumed. Non-mapping entries are skipped consistently.

Column-role ambiguity and explicit `key_columns` bypass are correct. Sparse
defaults contain no first-match-by-insertion-order path, but their
payload-filtered partial-entry behavior differs from the broadly documented
policy (Finding 4.3).

== 8. Schema-maintenance assessment

The dedicated XRef handler now covers all concrete consumers:

* `row_keys` on relation and matrix;
* top-level run-local `column_keys` on matrix;
* `dense_axes.rows_from.key` and `.keys` on the configured row-axis frame;
* `dense_axes.columns_from.key` and `.keys` on the configured column-axis
  frame;
* `dense_axes.resolved.column_keys` on matrix.

Both rename and drop use the same feature-aware predicate and block rather
than rewriting because `row_keys` is shared between two frames and the other
identities cannot be safely changed one-sidedly. Unrelated frame names,
fabricated `frame + value_column` convention shapes, and legacy descriptive
fields do not block. A malformed root fails; an individual non-mapping entry
is deliberately ignored, consistently with runtime tolerant readers.

The handler only reads metadata, and schema operations preserve the input on
blocked paths; no input mutation was found. The minor docstring phrase
"exactly three places" precedes four bullets, but the implementation and
bullets are correct.

== 9. Persistence and carrier assessment

Current producers write minimal feature-local intent plus run-local
Resolution:

* durable: `relation`, `matrix`, `row_keys`, optional
  `dense_axes.rows_from` / `columns_from`;
* run-local: top-level `column_keys` and `dense_axes.resolved`.

The persistence boundary strips both Resolution facets for every carrier and
preserves the durable fields. Current producers no longer write `operation`,
`column_key`, `value`, or `drop_empty`. Same-id assignment replaces the entire
old entry, so removed fields do not survive a rewrite. Differently keyed,
untouched legacy entries pass through as tolerated sediment, correctly
deferred to metadata-pileup work.

The XLSX/ODS integration test proves identical minimal intent, absence of the
source relation, and recreation of the relation for contractual string
identities. Worldbuilding independently passed both carrier reimports and the
physical workbook-view flow. Workbook View has no XRef semantic branch.

== 10. Documentation and registry assessment

The slice record, epic record, User Guide, meta registry, and release notes all
adopt the human-decided vocabulary and explicitly reject typed header
preservation. Their carrier recreation claim is appropriately limited to that
vocabulary.

They currently overstate that every source is validated with an explicit
diagnostic before cleanup (Findings 4.1/4.2), and broadly state uniform sparse
ambiguity failure despite partial-entry behavior (Finding 4.3). The pipeline
registry remains stale (Finding 4.4). These claims should be corrected to the
final implemented behavior rather than weakened around defects that can be
mechanically fixed.

== 11. Consumer compatibility

Demo ACME XRef pipeline and Worldbuilding Dino-Insel pipeline (apart from
tests, registries, historical fixtures, and project-memory examples).

* Demo `product_market.market_id` contains only non-empty strings. Repeated
  relation values are valid; the resulting distinct matrix identities are
  strings.
* Worldbuilding column axes are `groups.name` and `characters.name`; all 9
  group names and all 30 character names are unique, non-empty strings.
* Worldbuilding supplies explicit transformation parameters and removes
  workbook helper columns through `project_by_role` before expansion.
* No live consumer configuration supplies numeric, missing, mixed, or
  unhashable XRef column identities, and none reads removed descriptive

The accepted string contract therefore does not break a shipped consumer.
Ambiguity failure and stricter `drop_source` validation remain intentional
compatibility changes. Untouched legacy entries remain readable sediment.

== 13. Accepted limitations

The following remain accepted and are not findings:

* no typed spreadsheet-header identity encoding;
* `drop_empty=True` shape recovery requires durable dense intent or a retained
  matrix;
* inferred expansion treats every non-row-key matrix column as a value column;
  helper/technical-column removal is feature-owner responsibility;
* untouched legacy descriptive fields may remain as sediment;
* malformed individual XRef entries are tolerated when they have no valid
  reference shape;
* compact-composed temporary relation-name sediment belongs to the
  `compact_multiaxis` family slice;
* side-effecting-writer layering, generic metadata-pileup tooling, M:N Workbook
  View, named ranges, and audit history remain out of scope.

No generic lifecycle, provenance, precedence, Workbook View coupling, or new
metadata ontology was introduced by the correction.

== 14. Blockers, human decision, and readiness

Blockers before acceptance:

. reject duplicate physical relation fields, duplicate configured row keys,
  and row-key/value-column overlap before producing output or cleanup;
. validate scoped base-relation identities and route live dense and physical
  matrix identities through the shared string validator before hash/pandas
  operations;
. add adversarial tests proving every rejected case leaves cleanup absent and
  valid full projections remain accepted;
. realign the pipeline registry and overbroad completion claims with the
  corrected runtime contract.

No new human decision is required. The existing unique, non-empty string
decision determines every remaining correction. These are local validation,
test, and registry tasks.

The XRef primitive is not yet a settled foundation for
`compact_multiaxis`. Do not start that family slice until the blockers above
are corrected and independently re-reviewed. This conclusion does not expand
the work into Workbook View, generic lifecycle, writer layering, or metadata
pileup.
