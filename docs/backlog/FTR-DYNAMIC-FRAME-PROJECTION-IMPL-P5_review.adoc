== Verdict

Yellow.

The implementation matches the reviewed slice on core behavior and on the
worldbuilding YAML migration shape. The new `project_by_role` step is
backend-neutral, operates at the Frame layer, delegates role resolution to the
shared resolver, enforces explicit `direction`, implements the outbound and
inbound role-retention rules, is registered in the pipeline registry, and has
the expected public-gate and release-note entries.

Worldbuilding migrated all 10 projection call sites using the reviewed
upstream-rename strategy, without adding `source:` / `output:` to
`project_by_role`. XLSX and ODS export/reimport validation passed.

The remaining issue is test adequacy: the reviewed implementation slice
requires migration-parity tests covering all 10 surfaces, frame-name parity,
row-order parity, empty-cell behavior, and resolver-input availability. The
current tests and make validations prove the main end-to-end path, but they do
not independently assert row-order parity across the migrated outbound
surfaces.

== Executive Summary

Core implementation is aligned with `FTR-DYNAMIC-FRAME-PROJECTION-IMPL-P5`.
`project_by_role` imports and calls
`spreadsheet_handling.domain.column_roles.resolve_column_roles` and does not
define a second role taxonomy. The resolver consumes the interim sources named
by the FTR: explicit overrides, `contract_xref` row-key metadata for the same
frame, workbook-view helper columns, FK-helper derived provenance, and
matrix-value rest-by-exclusion within the frame.

The worldbuilding migration is also structurally correct. All five outbound
surfaces now project the final `*_matrix_view` frame after `join_frames`, and
all five inbound surfaces project the `*_matrix_view` frame before downstream
`expand_*` consumers read it. The retained plugin functions were not changed
except for one-line retention comments.

== Core Findings By Priority

=== Medium-1: Required migration-parity test fixture is missing

The FTR requires more than a fabricated `project_by_role` roundtrip. It states
that migration-parity tests must cover every projection surface, frame-name
parity, row-order parity, empty-cell behavior, resolver-input availability,
and all 10 migrated call sites
(`docs/backlog/FTR-DYNAMIC-FRAME-PROJECTION-IMPL-P5.adoc`, lines 657-715 and
880-884).

Current core integration coverage is meaningful but minimal:
`tests/integration/pipeline/test_project_by_role_roundtrip.py` uses one
fabricated `story_group_matrix_view` fixture and asserts the YAML surface,
column order, retained data values, and invalid direction handling (lines
33-87). It does not enumerate the five outbound and five inbound dino_insel
surfaces, compare pre/post plugin behavior, or assert outbound row-order
parity.

Impact: the implementation could still miss a migration-only regression,
especially because the removed outbound plugin sorted by `story_id` while
`project_by_role` intentionally performs no row operation.

Recommendation: add a focused migration-parity test fixture before commit, or
explicitly amend the slice if the make-based consumer validation is now the
accepted substitute. The former is preferable and does not require reopening
architecture.

== Test Adequacy

Core unit tests are meaningful. They assert resolver precedence and source
rules, warn-only missing `row_identity`, role-order defaults, custom
`role_order`, invalid roles, invalid direction, missing frame, inbound helper
dropping, outbound no-drop behavior, and value preservation. The architecture
guard checks that `project_by_role` imports and calls the shared resolver and
does not define role-name constants locally.

The integration test verifies the public YAML surface through the registry and
pipeline execution path, not just direct Python mechanics.

Gap: migration-parity coverage is incomplete for the reviewed slice. The
worldbuilding make checks are strong end-to-end validation, but because the
comparison normalizes ordering, they are not a substitute for the explicit
row-order parity check required by the FTR.

  `display_helper`.

* The resolver is the single role-detection path used by `project_by_role`.
* The implementation does not introduce `source:` / `output:` or targeting
  parameters.
* Empty-cell / NaN-vs-empty-string compatibility is acceptable in observed
  consumer validation: both XLSX and ODS reimports matched canonical data.
* No unrelated worldbuilding plugins were migrated.
