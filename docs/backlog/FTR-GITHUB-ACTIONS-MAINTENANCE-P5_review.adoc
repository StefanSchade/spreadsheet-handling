== Scope reviewed

This review covered the CI-maintenance cleanup slice after the successful
`v0.1.0b6` beta release. The review checked whether the implementation
actually hardened operational CI behavior, whether the GitHub Actions version
cleanup was complete, whether the new workflow permission blocks were
minimal, whether the Dependabot classification was justified from repository
state, whether the release runbook distinguishes release-blocking CI from
maintenance noise, and whether the sibling demo-CI cleanup FTR was complete
enough to archive.

This was a review-only pass. No workflow behavior was changed during the
review.

== Action-major-version finding from Codex

Codex verified the current upstream action majors during review and found that
the repository-local claims are stale:

* `actions/checkout`: workflows use `@v4`; current upstream documentation
  advertises `@v6`.
* `actions/setup-python`: workflows use `@v5`; current upstream documentation
  and releases advertise `@v6`.
* `actions/setup-java`: workflows use `@v4`; current upstream documentation
  and releases advertise `@v5`.
* `pypa/gh-action-pypi-publish@release/v1` remains appropriate for the PyPI
  Trusted Publisher job.

This is not merely wording drift. The FTR acceptance criteria include current
supported GitHub Action major versions and Node.js runtime deprecation
handling. Because current workflow pins lag upstream majors that target the
newer Node runtime, the implementation is *not yet fully accepted* and should
not be pushed from `dev` until the pins are updated or the maintainer records
a conscious decision to defer the bump.

== Residual risks

* The outdated action majors remain unresolved in the workflows. Until fixed,
  Node runtime deprecation warnings may continue and the documentation is
  still ahead of actual CI behavior.
* Bumping `actions/checkout`, `actions/setup-python`, and `actions/setup-java`
  to current majors should be watched on the first GitHub-hosted run for runner
  compatibility and checkout credential behavior changes.
* Dependabot 403 handling still requires manual GitHub UI inspection. The code
  review can prove there is no repo-local `dependabot.yml`, but it cannot see
  or fix repository Security / Code-security settings.
* The workflow `permissions:` blocks harden current workflows, but the
  repository-level Actions default should still be checked in the GitHub UI so
  future workflows do not inherit write access by omission.

== Disposition

The current implementation is *not yet fully accepted*. It should not be
pushed from `dev` in either `core` or `demo` until one of the following is
true:

. The workflows are updated to current upstream action majors and the release
  docs/review note are corrected accordingly.
. The maintainer consciously accepts the older action majors for this slice,
  records the reason, and keeps the Node runtime warning as an explicit
  follow-up risk.

Permission scoping, Dependabot classification, release-blocking vs
maintenance-noise separation, and the cold-storage/archive move are acceptable.
The blocking gap is specifically the stale action-major-version state and the
incorrect documentation that described those old pins as current.
