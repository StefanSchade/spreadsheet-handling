== Verdict

Ship. Every first-party GitHub Action already pinned to its current
published major before this FTR; the operational gap was that workflow
permissions and the Dependabot story were both implicit, and that the
runbook gave operators no way to separate release-blocking red signals
from infrastructure-maintenance noise. The change closes those three
gaps without touching the release pipeline's runtime behavior.

== CI-maintenance warnings addressed

[cols="1,2,3", options="header"]
|===
| Signal | Pre-state | Post-state

| GitHub Actions Node.js 20 deprecation warnings
| Visible during `v0.1.0b6` release; cause not classified.
| Classified as upstream-driven. All first-party actions
  (`actions/checkout@v4`, `actions/setup-python@v5`,
  `actions/setup-java@v4`) are on the latest published majors and still
  run on Node 20. The warning disappears when action maintainers ship
  the next major targeting the next Node LTS; nothing in our YAML can
  silence it today. Documented in CI/CD chapter and reclassified as
  non-blocking in the runbook.

| Implicit `GITHUB_TOKEN` permissions
| Demo `ci.yml`, demo `publish-docs.yml` `test` job, and core
  `release.yml` top-level + `test` job all used the repository-default
  permission set. Subtle but real risk: a future workflow edit that
  reads/writes through `GITHUB_TOKEN` would silently inherit
  contents-write on `push:`-triggered runs in a public repo.
| Every workflow now declares a top-level `permissions: contents: read`
  default; every job either inherits that read-only baseline or
  explicitly elevates (only `publish` does, for PyPI OIDC).

| Ambiguity between release-blocking and maintenance signals
| Operators had to remember which red statuses to ignore at tag time;
  not written down.
| Runbook § Blocking vs Non-Blocking CI Signals lists both categories
  with concrete job names so a new operator can triage a tag run
  without recall from memory.
|===

== Dependabot classification

[cols="1,2", options="header"]
|===
| Question | Finding

| Is there a checked-in `dependabot.yml` in any repo?
| No. Verified by `find . -name dependabot.yml` across `core`, `demo`,
  and `pages` (excluding `.venv` and `build`). Zero results.

| What is the source of the failing Dependabot run reported during
  `v0.1.0b6` (`403 Token invalid`)?
| GitHub repository-level Security / Code-security settings, not
  workflow configuration. The two GitHub-side surfaces that can
  surface Dependabot signals without a checked-in config are
  *Dependabot alerts* (passive vulnerability scanning) and
  *Dependabot security updates* (automated PRs). The `403 Token invalid`
  shape is consistent with the *security updates* path: the bot
  attempts to author a PR and the per-repo token boundary rejects the
  action.

| Is the failure repo misconfiguration, GitHub service issue,
  permission issue, or intentional-disable candidate?
| Inconclusive from a code-side review alone. Most likely a
  per-repo token / permission boundary in the GitHub UI; requires
  manual inspection of Settings → Code security and analysis on each
  affected repository to confirm whether to fix or disable.

| Do release-critical workflows depend on Dependabot?
| No. Verified by reading all three workflows end-to-end. No `needs:`,
  `if:`, env, or external action consumes Dependabot output, run
  status, or PR state. Dependabot failures cannot block PyPI publish
  or Pages publish.

| Should this FTR add a `dependabot.yml`?
| No. The FTR scope explicitly excludes "dependency policy redesign"
  and "replacing Dependabot with Renovate". Introducing a
  `dependabot.yml` where none exists is a policy decision (cadence,
  ecosystems, group strategy, target branch). Defer to a future
  small FTR if the manual UI inspection shows the maintainer wants
  to keep Dependabot updates on and pin the configuration.
|===

== Required manual GitHub UI follow-up

The agent cannot read or change repository Settings. The following must
be done by the maintainer in a browser session:

. For each of `spreadsheet-handling`, `spreadsheet-handling-demo`, and
  `spreadsheet-handling-pages`, open
  `Settings → Code security and analysis` and record the current
  Dependabot state (alerts on/off, security updates on/off).
. For any repo where security updates are on and runs show
  `403 Token invalid`, capture the failing run ID before changing
  anything. Then either (a) fix the underlying token boundary in the
  UI (most often: re-grant Dependabot access after a repo-permission
  change) or (b) intentionally disable Dependabot security updates and
  record that decision in the next iteration of this FTR series.
. Confirm `Settings → Actions → General → Workflow permissions` is
  *Read repository contents and packages permissions* (the
  most-restrictive default). The top-level workflow `permissions:`
  blocks added in this FTR depend on that repo-default not being
  *Read and write permissions*; otherwise a workflow without an
  explicit `permissions:` block would still inherit write access.

A short follow-up FTR should capture whatever the UI inspection
reveals; do not silently bake the result back into this slice.

== Residual risks

* *UI follow-up still owed.* The Dependabot 403 is reclassified and
  reframed, but not actually fixed. If left untouched, the same red
  signal will reappear on every push. The manual UI step above is what
  closes that loop.
* *Top-level workflow permissions assume the repo default is also
  most-restrictive.* If `Settings → Actions → General → Workflow
  permissions` is set to *Read and write*, a future workflow that
  forgets to declare `permissions:` will still inherit write contents
  on `push:` runs. Mitigated by the UI follow-up step that records the
  setting.
* *Node-20 deprecation will eventually start emitting hard-fail
  notices*, not just warnings. The CI/CD chapter says the fix is
  upstream-driven; track upstream `actions/*` major releases (v5/v6)
  and bump pins when they ship.
* *Single-job demo `ci.yml`*. `permissions:` at top level and at the
  one job are intentionally redundant — they encode the same
  read-only baseline both as "workflow-wide intent" and "job-scoped
  guarantee against a future second job inheriting wrong defaults".
  Cheap; no churn on rename or split.
