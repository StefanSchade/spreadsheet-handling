== Scope

Reviewed the drafted FTR itself, not implementation code.

The concept survey has already been reviewed as ready for FTR derivation in
`docs/backlog/FTR-VERTICAL-ALIGNMENT-ROUNDTRIP-P5_ftr_readiness.adoc`. This
assessment checks whether the FTR faithfully captures that survey and is ready
to guide implementation.

Primary focus:

* first-slice boundary;
* canonical vocabulary (`top`, `center`, `bottom`);
* ODS `middle` to canonical `center` mapping;
* default suppression and unsupported-value handling;
* reuse of `apply_cell_addressed_presentation_meta`;
* ODS carrier placement on `style:table-cell-properties`;
* ODS style-cache-key migration;
* pass ordering and registry strategy;
* acceptance criteria and test strategy; and
* non-goal protection against new shared infrastructure or generic styling
  abstractions.

== Verdict

Green.

Would I approve implementation work from this FTR as written?:: Yes.

The FTR accurately derives from the survey, is bounded enough for a focused
implementation, and makes the important backend-carrier, normalisation,
default-suppression, stale-meta, and composition risks explicit.

The FTR is likely to support a single focused implementation commit, assuming
the implementation follows the listed code paths and does not expand into wrap
text or generic styling infrastructure.

== Findings

Blocking:: None.

Before implementation:: None.

Watch-only:: The FTR's acceptance criteria are intentionally strict for
carrier placement, ODS repeated-structure safety, stale-meta behaviour, and
composition. Implementers should resist trimming those tests during
implementation, because they encode the exact failure modes found in the
preceding presentation-metadata slices.

== Positive Observations

The FTR accurately captures the survey conclusions.

It preserves the survey's key decisions:

* vertical alignment is the next cell-addressed presentation-metadata family;
* no new shared infrastructure is needed;
* the first slice covers only `top`, `center`, and `bottom`;
* XLSX uses `Alignment.vertical` with identity mapping;
* ODS uses `style:vertical-align` on `style:table-cell-properties`;
* ODS `middle` maps to canonical `center`;
* defaults are suppressed rather than materialised;
* unsupported values are dropped on read; and
* carrier-authority semantics are applied from day one through
  `apply_cell_addressed_presentation_meta`.

The first slice is appropriately bounded.

The in-scope list is concrete and mechanical: one registry entry, one render
op, one pass, one flow builder, one extractor per backend, one write-path
extension per backend, one carrier-authority call site per backend, focused
tests, and release notes. The non-goals exclude wrap text, generic styling,
plugin registries, vocabulary consolidation, `_merge_xlsx_alignment`
refactoring, render-flow refactoring, and carrier-authority redesign.

The canonical vocabulary is correct for the first slice.

`top`, `center`, and `bottom` are the right canonical values. They are
XLSX-shaped, match the established horizontal-alignment pattern, and avoid
promoting implicit defaults or backend-only values into canonical metadata.

The ODS mapping is correctly specified.

The FTR states both directions:

* ODS read: `middle` -> canonical `center`;
* ODS write: canonical `center` -> ODS `style:vertical-align="middle"`.

It also requires tests for both directions, which is important because read
normalisation and write emission are separate code paths.

The deferred values are appropriate.

Dropping XLSX `justify` / `distributed` is a good first-slice decision because
they are XLSX-specific and not needed to preserve the common cross-backend
alignment intent. Dropping ODS `automatic` is also correct because it is a
default/no-value sentinel. Excluding `baseline` is correct because it belongs
to inline text vertical alignment, not cell content vertical alignment on
`style:table-cell-properties`.

Backend-local responsibilities stay local.

The FTR keeps:

* XLSX extraction in `openpyxl_parser`;
* XLSX write composition in `openpyxl_renderer` through
  `_merge_xlsx_alignment`;
* ODS style-map walking and repeated-carrier safety in `odf_parser`;
* ODS carrier placement and style-cache changes in `odf_renderer`; and
* workbook-meta replace-or-clear semantics in the existing shared helper.

The ODS carrier location is correct.

The FTR explicitly identifies the carrier as `style:vertical-align` on
`style:table-cell-properties`, and it requires an XML assertion proving the
attribute is not placed on `style:paragraph-properties` or
`style:text-properties`. That directly addresses the copy/paste trap from
horizontal alignment.

The ODS style-cache-key migration is sufficiently specified.

The FTR names every part that must change:

* `CellStyleKey` grows from four fields to five;
* the early-exit sentinel grows to `(False, None, 0, None, None)`;
* `_register_cell_style` gains `vertical_alignment`;
* `_BufferedCell` gains `vertical_alignment`;
* the collection step sets the addressed cell; and
* the `_build_table` call site passes the value through.

The focused cache test requirement is enough: two styles that differ only in
vertical alignment must receive distinct style names.

The pass-ordering decision is reasonable.

Placing `VerticalAlignmentPass` immediately after `HorizontalAlignmentPass`
and before `NamedRangePass` keeps presentation alignment passes together.
The order is not semantically load-bearing because the passes use distinct
metadata keys and backend renderers compose independent carrier attributes.

The registry strategy is consistent.

The FTR requires a `vertical_alignments` `canonical_meta` entry sibling to
`horizontal_alignments`, with `workbookir_to_frames` listed from day one. That
matches the post-M2 convention for presentation-metadata families.

== Missing Acceptance Criteria

No blocking acceptance criteria are missing.

The FTR already covers:

* XLSX and ODS parse/render/roundtrip;
* ODS carrier XML placement;
* ODS `middle`/`center` normalisation in both directions;
* ODS `automatic` and XLSX `None` default suppression;
* XLSX `justify` / `distributed` and unmapped ODS value dropping;
* ODS repeated-structure clipping and `ParserLimits`;
* XLSX composition across text orientation, horizontal alignment, and
  vertical alignment;
* stale-meta clearing on both backends;
* helper parametrisation;
* registry hardening;
* pass ordering; and
* release notes.

Optional tightening, not required before implementation:

* In the out-of-vocabulary acceptance criterion, the phrase "on both
  backends" follows a mixed list of XLSX-only and ODS-only values. The intent
  is clear from the rest of the FTR, but implementation reviewers should read
  it as "unsupported values are dropped by the backend on which they appear",
  not as requiring impossible cross-backend fixtures.
* The release-notes heading should match the repository's actual spelling if
  it differs (`behavior` vs `behaviour`). This is editorial, not semantic.

== Scope Risks

The main risk is accidental expansion during implementation, not missing
scope in the FTR.

Risks to watch:

* implementing wrap text while touching alignment code;
* adding a generic styling abstraction instead of one small presentation
  family;
* centralising vocabulary constants despite the explicit non-goal;
* redesigning carrier authority despite the existing helper;
* refactoring `_merge_xlsx_alignment` instead of adding a dispatch arm;
* weakening ODS repeated-carrier tests because they look heavier than XLSX
  tests; and
* using the wrong ODS carrier element by copying horizontal alignment too
  literally.

The FTR already names these boundaries clearly enough for implementation
review.

== Recommendation

Approve implementation work from this FTR as written.

No prerequisite FTR edits are required.

Implementation should remain a single focused commit unless unexpected local
test infrastructure work is needed. If implementation discovers a need for new
shared infrastructure, a generic styling framework, or carrier-authority
changes, that should be treated as a scope break and brought back for a new
concept/FTR decision rather than folded into this slice.
