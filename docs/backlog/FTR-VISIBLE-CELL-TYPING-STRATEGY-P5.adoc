===== FTR-VISIBLE-CELL-TYPING-STRATEGY-P5 -- Decide visible spreadsheet cell typing strategy for 0.2.0 and beyond

*Status:* New

*Purpose:*::
Establish an explicit project position regarding the typing of visible
spreadsheet payload cells and determine the contractual behavior for release
0.2.0.

The goal of this FTR is not to implement a typing system but to make the
current behavior explicit, evaluate alternatives, and decide how the project
will proceed.

*Context:*::
Current spreadsheet roundtrips preserve visible workbook content primarily as
textual cell values.

During consumer validation of a real-world worldbuilding repository,
a JSON -> workbook -> JSON roundtrip revealed that numeric values such as:

## [source,json]

{
"sequence_no": 1
}
-

may be reimported as:

## [source,json]

{
"sequence_no": "1"
}
-

This behavior is consistent with a spreadsheet-centric editing model but raises
an architectural question:

* Should visible spreadsheet payload remain string-based?
* Should explicit typing become part of the long-term model?
* Should the decision be deferred until additional production experience has
  been collected?

The project currently has no explicit contract describing the intended
behavior.

*Motivation:*::
The absence of a documented decision creates uncertainty for both users and
future maintainers.

Without a documented strategy:

* users may assume type preservation that is not guaranteed,
* future contributors may implement incompatible solutions,
* the roadmap for typed payload support remains unclear.

A release-quality framework requires an explicit position.

*Scope:*::

* Evaluate current visible-cell behavior.
* Define the contractual behavior for release 0.2.0.
* Document the behavior for users.
* Decide whether typing is:
  ** intentionally out of scope,
  ** planned for the future,
  ** or deferred pending operational experience.
* Create follow-up work if required.

*Non-Goals:*::

* No implementation of typed roundtrips.
* No schema system.
* No metadata type registry.
* No parser redesign.
* No locale-aware numeric conversion.
* No automatic type inference.

*Options Under Consideration:*::

Option A -- String Payload Contract
+
Visible workbook payload is defined as textual data.
+
Advantages:
+

* simple mental model,
* deterministic spreadsheet behavior,
* avoids type inference ambiguity,
* aligns with spreadsheet editing workflows.

-

Disadvantages:
+

* numeric and boolean types are not preserved automatically.

Option B -- Typed Payload Roadmap
+
Release 0.2.0 documents current behavior but explicitly commits to future
typed roundtrip support.

Option C -- Deferred Decision
+
Release 0.2.0 documents current behavior and intentionally postpones any
long-term commitment until more production experience has been gathered.

*Acceptance Criteria:*::

* A release decision for version 0.2.0 is made and documented.
* User-facing documentation clearly explains the behavior of visible
  spreadsheet payload cells.
* Users can determine whether type preservation is guaranteed.
* The project records a long-term position:
  ** accepted permanently,
  ** planned as future work,
  ** or intentionally deferred.
* If future evaluation is required, a dedicated follow-up FTR or equivalent
  backlog item is created.
* The topic cannot silently disappear from project planning.

*Validation:*::

* Review release notes and user documentation.
* Verify consistency between documented behavior and actual implementation.
* Verify that any deferred decision has a visible tracking mechanism.

*Discussion Note:*::

A likely short-term outcome is that release 0.2.0 formally adopts a
string-based visible-cell contract while collecting operational experience 
from
real consumer repositories.

This FTR intentionally leaves the final decision open until review and
discussion are complete.

*Risk*::

* A premature adoption of a typing strategy could become a large functional endeavor and endanger stability ahead of the upcoming release.
* Without real adoption experience, the practical value of typed library values cannot be judged reliably. The feature may turn out to be unnecessary, too costly, or inappropriate for the project's current string-only simplicity.
