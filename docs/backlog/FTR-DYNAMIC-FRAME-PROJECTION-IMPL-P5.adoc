===== FTR-DYNAMIC-FRAME-PROJECTION-IMPL-P5 -- Implementation slice for `project_by_role`

*Status:* New

*Priority:* P5

*Type:* Implementation slice (code + tests + worldbuilding YAML migration + release notes).

*Depends on:*::

* `FTR-PROJECTED-FRAME-COLUMN-SEMANTICS-P5` (foundation; provides the
  column-role taxonomy this step consumes -- `row_identity`,
  `display_helper`, `matrix_value` -- and the reimport-survival rule).
* `FTR-DYNAMIC-FRAME-PROJECTION-P5` (parent evaluation FTR) and

This FTR is the *implementing* slice of the projection decision. It
does not redefine the role taxonomy, the projection mechanism, or the
targeting surface.

*Purpose:*::

Implement a backend-neutral pipeline step `project_by_role` that
performs role-based ordering and retention on a projected matrix /
view surface, replacing the worldbuilding consumer plugins
`build_story_cast_matrix_view` and `build_story_cast_matrix_payload`.
Introduce the shared column-role resolver that this step and the
future targeting implementation slice both consume, so that role
detection cannot diverge between consumers.

*Proposed YAML surface:*::

The YAML surface below is the slice's commitment, derived from the
decision artifact's *Architectural direction* section. The
implementation slice owns its own design review; minor refinements
during review are acceptable as long as none of the non-goals above
is touched.

*`project_by_role` is an in-place transform.* It mutates the named
frame's column sequence; it does not produce a new output frame.
There is no `source:` / `output:` parameter. The migration plan
below (*Worldbuilding migration plan*) is responsible for ensuring
that the frame to be projected already carries the intended name
when the step runs.

The illustrative examples below use the concrete dino_insel frame
names from the worldbuilding migration; `story_cast_cells` is *not*
used as an example because it is the canonical long-form source,
not a projected matrix surface (see review finding High-1).

Outbound:

[source,yaml]
----
- step: project_by_role
  frame: story_group_matrix_view
  direction: outbound
  helper_columns: [title]      # required at this call site (see Role-resolution strategy)
  key_columns: [story_id]      # required at this call site
  # Optional:
  # role_order: [display_helper, row_identity, matrix_value]
----

Inbound:

[source,yaml]
----
- step: project_by_role
  frame: story_group_matrix_view
  direction: inbound
  helper_columns: [title]      # required at this call site
  key_columns: [story_id]      # required at this call site
  # Optional:
  # role_order: [row_identity, matrix_value]
----

Parameter contract:

[cols="1,1,3"]
|===
| Parameter | Required | Meaning

| `frame`
| yes
| Name of the frame on which projection runs. Must reference an
  existing frame in the current pipeline state; otherwise a
  validation error is raised before the step runs.

| `direction`
| yes
| One of `outbound` or `inbound`. The YAML surface *requires*
  explicit direction; there is no silent default, because the two
  directions have materially different retention semantics. Any
  other value raises a validation error.

| `helper_columns`
| no
| Explicit override for the `display_helper` role's column list.
  Intended primarily as an interim affordance until the foundation
  FTR's metadata channel ships. When omitted, the resolver reads
  the configured sources (see *Role-resolution strategy*).

| `key_columns`
| no
| Explicit override for the `row_identity` role's column list.
  Same status as `helper_columns`.

| `role_order`
| no
| Optional ordering for the roles within the projected output. Must
  be a sequence of foundation role names from
  `{row_identity, display_helper, matrix_value}`. Defaults are
  defined per direction (see *Outbound / Inbound semantics*). Any
  other value raises a validation error. For `direction: inbound`,
  including `display_helper` in `role_order` is a validation error
  (the inbound direction drops `display_helper`).
|===

*Role-resolution strategy:*::

The shared resolver computes role classification for a frame as
follows. Both `project_by_role` (this slice) and the future
targeting implementation slice consume it; the resolver is the
single source of truth in the current code base.

* `row_identity`:
** `key_columns` step-local override if present, *else*
** the columns named in `row_keys` declared on `contract_xref` for
   the same frame, *else*
** an empty set (the step runs but logs a warning that no row
   identity could be resolved; useful for early consumer wiring).
* `display_helper`:
** `helper_columns` step-local override if present, *else* the
   union of:
*** `helper_columns` declared on `configure_workbook_view` for the
    same frame, and
*** FK-helper `derived` provenance entries under
    `_meta.derived.sheets.<frame>.helper_columns`.
* `matrix_value`:
** Every column on the frame that is not in `row_identity` and not
   in `display_helper`. This is the foundation FTR's narrowed
   rest-by-exclusion, scoped to *this projected matrix / view
   surface only*.

*This resolver is an interim implementation strategy.* The
foundation FTR's longer-term direction is a unified column-role
metadata channel (e.g.
`_meta.derived.sheets.<frame>.column_roles`). When that channel
ships in a future slice owned by the foundation FTR (or a child of
it), the resolver switches its source of truth in one module
without changing the YAML surface and without changing the
contract of `project_by_role`.

*Important consequence for the worldbuilding migration:* the
scattered-source strategy presupposes that, at the moment
`project_by_role` runs, the resolver can reach the upstream
declarations *for the same frame name*. In the current
worldbuilding outbound pipeline, this is not the case:

* `configure_workbook_view` runs *after* the outbound projection
  call sites (`json_to_spreadsheet.yaml` lines 94/240-262 vs line
  296+), so view-declared `helper_columns: [title]` is not yet
  visible to the resolver at outbound projection time.
* `contract_xref` declares `row_keys: [story_id]` on the upstream
  matrix frame (e.g. `story_group_matrix`), not on the projected
  surface (e.g. `story_group_matrix_view`). The resolver does not
  trace frame ancestry in this slice.

Therefore the outbound *and* inbound worldbuilding call sites must
pass explicit `helper_columns: [title]` and `key_columns:
[story_id]` overrides until either the foundation channel ships or
the pipeline ordering changes. The migration plan below records
this requirement and the *Tests and validation* section asserts
the resolver inputs are reachable in the migrated YAML.

The resolver module must:

* expose a single public function (e.g.
  `resolve_column_roles(frames, *, frame, overrides=None) ->
  ColumnRoleMap`) that returns a mapping or three-set object the
  step can consume.
* be importable from the targeting slice's call sites without
  cycle.
* be documented as the interim resolver, with the migration path to
  the unified channel called out in a one-line module header.

*Outbound semantics:*::

For `direction: outbound`, the step:

. Resolves the three role sets for `frame` via the shared resolver.
. Determines the role ordering:
** if `role_order` is provided, use it verbatim; it must contain
   exactly the three role names `display_helper`, `row_identity`,
   `matrix_value` in some permutation (validation error otherwise);
** otherwise the default order is
   `[display_helper, row_identity, matrix_value]`.
. Constructs the output column sequence by concatenating, for each
  role in `role_order`, the columns of that role *in their input
  order on the frame*. Within-role order is *not* re-sorted.
. Replaces `frame`'s column sequence with the constructed sequence.
  Data values are preserved; only column order may change, and no
  columns are dropped on the outbound side.
. Returns the updated `frames` (per the existing
  `(Frames) -> Frames` step contract).

Edge cases:

* Empty `matrix_value` set (no dynamic columns yet): outbound
  produces a frame containing only `display_helper + row_identity`,
  with input-order preservation. No error.
* Empty `row_identity` (resolver could not find any): warn-only;
  produce the rest of the projection. This case is observable in
  worldbuilding's early consumer wiring and is not a hard error in
  this slice.
* Column not classified by the resolver: should not happen if the
  resolver is correct, but if it does, the step raises a clearly
  named internal error pointing at the resolver. Defensive only;
  not a documented user-facing error.

*Inbound semantics:*::

For `direction: inbound`, the step:

. Resolves the three role sets for `frame` via the shared resolver.
. Determines the role ordering:
** if `role_order` is provided, it must contain exactly
   `row_identity` and `matrix_value` in some permutation
   (validation error otherwise; including `display_helper` is
   rejected);
** otherwise the default order is
   `[row_identity, matrix_value]`.
. Drops the columns classified as `display_helper`.
. Constructs the output column sequence by concatenating, for each
  role in `role_order`, the columns of that role in their input
  order on the frame.
. Replaces `frame`'s column sequence with the constructed sequence.
  Data values are preserved for the retained columns.
. Returns the updated `frames`.

Edge cases:

* Empty `display_helper` set: inbound is a no-op for the drop part;
  ordering is still applied. No error.
* All columns are `display_helper` (no row identity, no value
  cells): inbound produces an empty-column frame. Warn-only.
* `role_order: [matrix_value, row_identity]`: accepted; output
  order is `matrix_value` first, then `row_identity`. The step
  does not impose downstream-step requirements; whether
  `expand_xref` can still consume that order is a pipeline-author
  responsibility documented in the YAML surface notes.

Symmetry note: outbound followed by inbound (without intervening
edits) restores the *retained* column set's order to the inbound
default, not necessarily to the original. Roundtrip symmetry of
data values is preserved; column-order roundtrip symmetry is not
guaranteed because outbound and inbound have intentionally
different default orderings.

*Plugin behavior parity (gap analysis).*

The current plugins do three things that `project_by_role` does
*not* do, by design. The migration plan addresses each explicitly.

. *Row sorting (outbound only).* The plugin sorts the result by
  `story_id` with a stable mergesort. `project_by_role` performs
  no row-level operation; it only reorders columns. The upstream
  `contract_xref` declares `dense_axes.rows_from: { frame: stories,
  key: id }`, which produces deterministic row order keyed by
  `stories.id`. The migration plan *relies on this upstream
  determinism* and removes the plugin-side sort.
  Migration-parity tests (*Tests and validation*) assert that the
  observed row order on each migrated outbound surface matches the
  pre-migration observed order on the same fixture.
. *Empty-cell normalization (outbound and inbound).* The plugins
  apply `result.where(pd.notnull(result), "")`, converting NaN to
  empty string. `project_by_role` performs no value transformation
  (its contract is "data values for retained columns are
  unchanged"). The migration may therefore present NaN where the
  plugin presented `""`. The migration plan accepts this as a
  *conscious behavior change* and treats normalization as the
  responsibility of separate value-shape steps if any downstream
  consumer relies on `""` semantics. Migration-parity tests assert
  end-to-end roundtrip with `expand_compact_multiaxis drop_empty:
  true` / `expand_xref drop_empty: true` semantics; if those tests
  fail because of the NaN-vs-"" distinction, the implementation
  slice adds an explicit normalization step in the worldbuilding
  YAML using an existing primitive (e.g. `sparse_defaults` or
  equivalent). The slice does *not* add normalization to
  `project_by_role`.
. *Frame renaming.* The plugins rename by virtue of `source:` /
  `output:`. The upstream-rename strategy above absorbs this work
  into existing `join_frames` `output:` (outbound) and downstream
  consumer `matrix:` (inbound), with no new primitive.

*Fallback path (consciously not chosen here).*

If implementation review concludes the upstream-rename strategy is
unworkable (e.g. because the `*_matrix_enriched` intermediate name
acquires a downstream consumer outside the dino_insel pipeline, or
because the inbound downstream `matrix:` parameter rename touches
more than the five named consumers), the fallback is a *conscious
surface change*: add optional `source:` / `output:` parameters to
`project_by_role`, mutually exclusive with `frame:`, with the
in-place behavior preserved when only `frame:` is given. That
fallback would reopen the projection decision's "no source / output"
commitment and would require a one-line amendment to
`FTR-DYNAMIC-FRAME-PROJECTION-P5_decision.adoc` recording the
deliberate surface refinement. This implementation slice does *not*
take that path; it is recorded here only so future implementers
know it is an option of last resort, not silent invention.

*Tests and validation:*::

Test layout follows the existing patterns surveyed in the
projection decision artifact's reconnaissance (unit tests under
`core/tests/unit/pipeline/`, integration tests under
`core/tests/integration/pipeline/`).

Unit tests (resolver):

* Resolves `row_identity` from declared `row_keys` on
  `contract_xref` for the same frame.
* Resolves `row_identity` from explicit override when override is
  provided (override wins).
* Resolves `display_helper` from
  `configure_workbook_view.helper_columns` when present.
* Resolves `display_helper` from FK-helper `derived` provenance
  under `_meta.derived.sheets.<frame>.helper_columns` when present.
* Union of view-declared and FK-helper-derived helpers when both
  present; no duplicates.
* Resolves `display_helper` from explicit override when override
  is provided (override wins).
* `matrix_value` is the rest-by-exclusion within the frame.
* Warn-only on missing `row_identity`; no exception.
* Frame not in pipeline state raises a clear error.

Unit tests (outbound):

* Default `role_order` produces
  `[display_helper, row_identity, matrix_value]` with within-role
  input-order preservation.
* Custom valid `role_order` rearranges roles as specified.
* Custom `role_order` containing an unknown role name raises a
  validation error naming the offending value.
* Empty `matrix_value` produces a frame with only the other two
  roles, no error.
* Data values for retained columns are unchanged.

Unit tests (inbound):

* Default `role_order` produces `[row_identity, matrix_value]` with
  within-role input-order preservation.
* `display_helper` columns are dropped.
* `role_order` containing `display_helper` raises a validation
  error.
* `role_order` containing an unknown role name raises a validation
  error.
* Data values for retained columns are unchanged.

Unit tests (validation):

* Missing `direction` raises a clear validation error before the
  step runs.
* `direction: "sideways"` raises a clear validation error naming
  accepted values.
* Missing `frame` raises a clear validation error.

Integration test (roundtrip):

* Mirrors the worldbuilding outbound -> inbound flow on a minimal
  fixture: produce a workbook with `project_by_role(outbound)`,
  re-read it, and run `project_by_role(inbound)`. Data values for
  retained columns must roundtrip identically.

Migration-parity tests (worldbuilding-specific):

The slice migrates 10 plugin call sites (5 outbound + 5 inbound). A
single illustrative `project_by_role` test on a fabricated frame is
*not* sufficient to certify the migration. The slice adds a
migration-parity test fixture that exercises every current
projection surface and asserts parity with pre-migration plugin
behavior on the dimensions where parity is required:

. *Frame-name parity.* For each of the 10 surfaces, the frame name
  consumed by the next downstream step after the migration must be
  exactly the frame name the migrated YAML produces. Concretely:
** Outbound: each `configure_workbook_view` `frame:` entry on lines
   304-308 of `json_to_spreadsheet.yaml` (`story_group_matrix_view`,
   `story_cast_role_matrix_view`,
   `story_cast_notes_matrix_view`,
   `story_cast_character_development_matrix_view`,
   `story_cast_crystal_matrix_view`) must still resolve after
   migration.
** Inbound: each downstream `expand_compact_multiaxis` /
   `expand_xref` `matrix:` parameter must reference a frame that
   exists at the point of consumption after the rename from
   `*_matrix_payload` to `*_matrix_view`.
. *Row-order parity (outbound only).* For each of the 5 outbound
  surfaces, the observed row order on a fixed dino_insel fixture
  must equal the pre-migration observed row order on the same
  fixture. The expectation is that `contract_xref`'s
  `dense_axes.rows_from: { frame: stories, key: id }` provides
  deterministic row order keyed by `stories.id`, making the
  plugin-side `sort_by=("story_id",)` redundant. If the test fails,
  the migration is rejected; the implementation must not paper over
  the difference by adding a sort inside `project_by_role`.
. *Empty-cell-normalization parity (end-to-end).* For each of the 5
  inbound surfaces, exercise the inbound roundtrip through
  `expand_compact_multiaxis drop_empty: true` (for
  `story_group_*`) and `expand_xref drop_empty: true` (for the four
  `story_cast_*` surfaces). The reimported payload must equal the
  pre-migration reimported payload byte-for-byte (or via the
  established roundtrip-equality comparator). If the test fails due
  to the NaN-vs-`""` distinction documented in the migration plan's
  *Plugin behavior parity (gap analysis)*, the implementation slice
  adds an explicit normalization step in the worldbuilding YAML
  using an existing primitive -- *not* by adding normalization to
  `project_by_role`.
. *Resolver-input availability.* For each of the 10 surfaces, the
  test fixture confirms that the explicit `helper_columns: [title]`
  and `key_columns: [story_id]` overrides are reachable by the
  resolver at the point `project_by_role` runs (i.e. the parameters
  are read at step invocation, not deferred). This guards against
  silent regression if the resolver's override-precedence rules are
  inadvertently weakened.
. *All current projection surfaces covered.* The test fixture
  must exercise all 10 surfaces, not only the
  `story_group_matrix_view` example used in *Proposed YAML
  surface*.

The migration-parity test fixture may live under
`core/tests/integration/pipeline/` or under a worldbuilding-tagged
test path; the implementation review chooses the placement.

Backend parity:

* The step is backend-neutral (Frame layer). No XLSX / ODS code
  paths are introduced. Per the FTR-slicing rules, no asymmetry
  between adapters is created.

Shared-resolver guard:

* A repository-level test (or a `tests/architecture/` guard)
  asserts that `project_by_role` does not implement its own role
  detection -- it calls the shared resolver module. This is the
  forcing function that prevents projection and targeting from
  diverging when targeting later lands.

*Risks:*::

* *Role-source resolver edge cases.* Multiple `contract_xref`
  declarations for a single frame, or overlapping FK-helper and
  view-declared `helper_columns`, may yield resolution ambiguity.
  Mitigation: define union semantics for overlapping sources in
  the resolver (no duplicates); document the precedence of the
  explicit override; surface a clear warning when the resolved
  sets overlap.
* *Worldbuilding regression at migration time.* If the core step
  has a bug, the worldbuilding consumer breaks at the same commit.
  Mitigation: the plugin functions remain importable; rollback is
  per-surface YAML edits documented in the migration plan's
  *Rollback* subsection. The release notes document the rollback
  path.
* *Row-order drift on the migrated outbound surfaces.* The plugin
  enforced `sort_by=("story_id",)` with stable mergesort. The
  migration plan relies on upstream `contract_xref dense_axes`
  for deterministic order. If that assumption is wrong on any of
  the 5 outbound surfaces, the migrated row order changes.
  Mitigation: migration-parity row-order test (item 2 of the
  *Migration-parity tests* list) is a *blocking* test for this
  slice; failure rejects the migration.
* *NaN-vs-empty-string distinction on inbound payload roundtrip.*
  The plugins normalized NaN to `""` via `.where(pd.notnull(...),
  "")`. `project_by_role` does not. If `expand_compact_multiaxis
  drop_empty: true` or `expand_xref drop_empty: true` treats `""`
  and NaN differently, the reimported payload diverges.
  Mitigation: migration-parity end-to-end test (item 3 of the
  *Migration-parity tests* list) is a blocking test; if it fails,
  the implementation slice adds an explicit normalization step in
  the worldbuilding YAML using an existing primitive, *not* by
  adding normalization to `project_by_role`.
* *Upstream-rename collateral.* The migration plan retires the
  `*_matrix_enriched` intermediate frame names (outbound) and the
  `*_matrix_payload` intermediate frame names (inbound) by folding
  their work into adjacent steps. If any external consumer (a
  downstream pipeline file, a test, a doc example) references
  those intermediate names, the rename breaks them silently.
  Mitigation: a pre-merge grep for the 10 retiring frame names
  across the worldbuilding repo and any worldbuilding-tagged
  fixtures is part of the implementation slice's review checklist.
* *Conscious fallback temptation.* If the implementation review
  hits unexpected friction with the upstream-rename strategy, the
  fallback (adding `source:` / `output:` to `project_by_role`)
  reopens the projection decision artifact. Mitigation: the
  fallback path requires an explicit one-line amendment to
  `FTR-DYNAMIC-FRAME-PROJECTION-P5_decision.adoc` and an
  acknowledgement in this FTR's *Worldbuilding migration plan*
  *Fallback path* subsection. The fallback is not silent.
* *Hidden dependence on the consumer plugins.* Tests or demos
  outside the worldbuilding consumer may import the plugin
  functions directly. Mitigation: a one-time grep for plugin
  symbol references is part of the implementation slice's
  pre-merge review.
* *Premature foundation-channel implementation.* If a reviewer
  pushes to ship the unified
  `_meta.derived.sheets.<frame>.column_roles` channel in this
  slice, the slice scope balloons and the foundation FTR's
  text-only acceptance is silently violated. Mitigation: the
  *Non-Goals* list above forbids it; the resolver is structured
  so the channel can be added later in a separate slice.
* *Default-direction surprise.* Earlier sketches in the decision
  artifact noted a Python-side default of `outbound`. The YAML
  surface in this slice intentionally *requires* explicit
  `direction:`. Mitigation: validation error is clear and the
  release-notes entry calls this out.

*Acceptance Criteria:*::

The slice is complete when *all* of the following are demonstrably
true on a single PR / branch:

. `core/registries/pipeline_step_registry.json` lists
  `project_by_role` as a registered step.
. The step lives in a backend-neutral location under
  `core/src/spreadsheet_handling/domain/...` and operates as a
  `(Frames) -> Frames` callable, consistent with the existing
  pipeline-step contract.
. A shared role resolver module exists, exposes a single public
  function, and is the *only* role-detection code path used by
  `project_by_role`. A repository guard or test asserts the
  single-source-of-truth property.
. The resolver implements the three role rules per the
  *Role-resolution strategy* section. Explicit overrides win over
  scattered upstream sources. Missing `row_identity` is
  warn-only; missing `display_helper` is silent (empty set).
. `direction` is required in the YAML. `direction:
  outbound | inbound` are accepted; any other value raises a
  validation error named in the *Tests and validation* section.
. Outbound default order is
  `[display_helper, row_identity, matrix_value]`; inbound default
  order is `[row_identity, matrix_value]`; within-role order is
  the input order. Custom `role_order` is respected within the
  per-direction constraints documented under *Outbound semantics*
  / *Inbound semantics*.
. Invalid `role_order` values (unknown role name; including
  `display_helper` for inbound; missing roles for outbound) raise
  validation errors before the step runs.
. Unit tests for the resolver, outbound, inbound, and validation
  exist and pass. The list in *Tests and validation* is the
  minimum.
. An integration test exercises the worldbuilding outbound ->
  inbound roundtrip end-to-end and passes.
. The worldbuilding YAMLs at
  `worldbuilding/pipelines/dino_insel/json_to_spreadsheet.yaml`
  and `worldbuilding/pipelines/dino_insel/spreadsheet_to_json.yaml`
  call `project_by_role` instead of the two projection plugins on
  *all 10* current call sites (5 outbound + 5 inbound) enumerated

  matrix:` renamed from `*_matrix_payload` to `*_matrix_view`. No
