===== FTR-SEMANTIC-SETS-AND-LIFECYCLE-MODEL-P5 -- Define overlapping sets, lifecycle, and precedence

*Status:* New
*Priority:* P5
*Area:* Semantic model / documentation

==== Goal

Turn the overlapping-semantic-set view and the lifecycle / update model
into a stable, rule-ready vocabulary. The chapter must let later
adoption slices classify metadata attributes and transformations without
re-inventing terminology each time.

==== Scope

* Elaborate the Project Semantic Model, Overlapping Semantic Sets,
  Lifecycle And Update Semantics, and Conflict And Precedence Questions
  chapters in `docs/semantic_model/semantic_model.adoc`.
* Name and define the candidate sets (Canonical, Derived, Persistent,
  Transient, RoundtripRelevant, RenderConsumed, RuntimeConsumed,
  UserEditable, ProducerOwned, Configurable, HeuristicallyInferred,
  Policy, Provenance, PublicSurface, InternalOnly, Deprecated,
  TransitionBridge) with one-sentence definitions.
* Record lifecycle events, write-time conflict resolution, precedence as
  update semantics, and the role of age / generation / run identity.
* Mark which questions remain open and feed them into the Open Questions
  chapter.

==== Out of scope

* No application of the vocabulary to existing transformations beyond
  the seed's worked examples. Adoption is owned by
  `FTR-SEMANTIC-ADOPTION-STRATEGY-P5` and the FK-helper pilot.
* No new metadata fields written to `_meta`.
* No new registry entries or changes to existing ones.

==== Seed sections elaborated

* Project Semantic Model
* Overlapping Semantic Sets
* Lifecycle And Update Semantics
* Conflict And Precedence Questions

==== Acceptance (rough)

* Each named semantic set has a one-sentence definition.
* Lifecycle events, write-time conflict resolution, and precedence
  rules are described as rule-ready concepts (not executable).
* The chapter explicitly distinguishes age / generation from semantic
  precedence, consistent with the seed.
* The document still renders cleanly via `make docs-html`.

==== Code and guards

* No production code. No tests. No guards. Documentation only.
