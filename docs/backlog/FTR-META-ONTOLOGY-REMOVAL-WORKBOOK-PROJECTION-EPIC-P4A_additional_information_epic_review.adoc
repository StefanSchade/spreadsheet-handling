== 1. Executive conclusion

The target model is *supported with amendments*. The repository is already much
closer to the target architecture than the epic's problem statement implies:

* The explicit Workbook View branch
  (`workbook_view.sheets` in
  `src/spreadsheet_handling/rendering/frame_selection.py::_select_configured_sheets`)
  already implements authoritative, ontology-free artifact projection with
  missing-frame and duplicate-target failure. Every real consumer pipeline uses
  this branch.
* The no-Workbook-View default already is the epic's target: without
  `workbook_view` in `_meta`, `select_render_frames` returns all frames and
  `compose_workbook` maps each frame to a same-named worksheet.
* The lifecycle-ontology selection branch (`_should_render_frame`,
  `DEFAULT_OMIT_ROLES`, `DEFAULT_INCLUDE_ROLES`, `mode`, `drop_redundant_data`,
  `include_roles`, `omit_roles`, `include_debug_frames`,
  `include_omit_by_default`, `unknown_frame_policy`) is reachable only when
  `_meta.workbook_view` exists *without* a `sheets` list. No shipped producer,
  example, consumer pipeline, or documented user-guide surface creates that
  state. It is exercised exclusively by hand-built meta in unit tests.
* There is no generic carrier-neutral frame cleanup in core. The one real
  consumer that needs it (`worldbuilding` dino_insel reverse pipeline)
  implements it as a private plugin
  (`worldbuilding/plugins/frame_sets.py::keep_frames`) — direct evidence that
  the epic's Responsibility 2 fills a real, currently consumer-patched gap.

The amendments (Section 12) concern four evidence-backed corrections, none of
which invalidates the target model:

. Cleanup directives must be contributable by the *pipeline builder* as well as
  by transformations, because in the flagship pipeline most redundant
  intermediates are produced by generic steps (`join_frames`, `extract_frame`,
  `expand_xref`) that cannot know their outputs are intermediate.
. "Fail on contradictory retained references" must be scoped narrowly:
  transformation intent metadata (`xref_crosstable.*`, `compact_multiaxis.*`,
  `split_by_discriminator.*`) legitimately references frames that are absent
  from the persisted state and *must survive* cleanup, because inverse
  transformations need it to re-create those frames.
. Cleanup directives must be *consumed* (removed from `_meta`) when executed,
  because `_meta` roundtrips through the workbook blob carrier and stale
  directives would otherwise re-trigger drops in reverse pipelines.
. The reverse-mapping field `sheet_mappings[*].canonical_frame` is currently
  *derived from* lifecycle ontology (`canonical: true` + singleton
  `derived_from`); removing lifecycle producers wholesale without an explicit
  replacement would silently degrade reimport mapping metadata.

== 3. Confirmed current architecture

=== 3.1 Selection path (spreadsheet only)

`select_render_frames(frames, meta)` in
`src/spreadsheet_handling/rendering/frame_selection.py` has exactly three
branches:

. *No `workbook_view` mapping in meta* → returns `dict(frames)` unchanged
  (line 26–28). Combined with
  `rendering/composer/layout_composer.py::compose_workbook` ("naive
  1-table-per-sheet IR", line 256–296), this is already the epic's target
  default: every remaining frame becomes one same-named worksheet; `_meta` is
  excluded via `RESERVED_FRAME_KEYS` and persisted as the hidden `_meta` sheet
  by `MetaPass`.
. *`workbook_view.sheets` present* → `_select_configured_sheets` is
  authoritative: only listed frames are materialized, in declared order, under
  declared sheet names; missing frames raise `KeyError`, duplicate sheet names
  raise `ValueError`, reserved names are rejected, and
  `LookupFormulaSpec.lookup_sheet` references are rewritten from frame names to
  physical sheet names (`_resolve_formula_sheet_names`). Lifecycle metadata is
  *entirely ignored* in this branch.
. *`workbook_view` present without `sheets`* → the ontology branch
  `_should_render_frame` interprets `frame_lifecycle.<frame>.render`
  (`never`, `debug_only`, `omit_by_default`), `role` against
  `DEFAULT_OMIT_ROLES = {"intermediate", "redundant", "system"}` and
  `DEFAULT_INCLUDE_ROLES`, plus the view-level knobs `mode`,
  `drop_redundant_data`, `include_roles`, `omit_roles`, `include_debug_frames`,
  `include_omit_by_default`, and `unknown_frame_policy`.

The only call site is
`src/spreadsheet_handling/io_backends/spreadsheet_contract.py::build_spreadsheet_render_plan`,
used exclusively by the XLSX backend
(`io_backends/xlsx/xlsx_backend.py:39`) and ODS backend
(`io_backends/ods/ods_backend.py:36`). *Render selection is not used outside
spreadsheet rendering.* JSON, YAML, XML, and CSV backends persist every frame
plus the `_meta.yaml` sidecar (e.g. `io_backends/json_backend.py:126–153`).

=== 3.2 The ontology branch is dead in every produced flow

`domain/workbook_views.py::configure_workbook_view` — the only shipped producer
of `_meta.workbook_view` — *always* writes the explicit `sheets` list
(line 90–103). Because branch 2 above short-circuits before any lifecycle
lookup, the `mode`, `drop_redundant_data`, and `unknown_frame_policy` values
that `configure_workbook_view` simultaneously writes are never read back by the
renderer in the flow that produced them. Observable inconsistency:
`configure_workbook_view` defaults `unknown_frame_policy="fail"` while
`frame_selection._unknown_frame_decision` defaults `"visible"` — the divergence
is invisible today precisely because the field is dead in the produced flow.

The ontology branch is reachable only from hand-built `_meta` (unit tests
`tests/unit/rendering/test_frame_selection.py` tests 1–5 and
`tests/unit/domain/transformations/test_lookup_formula_mode.py`).

Classification: *dead or obsolete behavior* (ontology branch and its knobs, in
all shipped flows); *confirmed supporting evidence* for epic Slice 4.

=== 3.4 Frame drop/prune behavior today

* `domain/transformations/compact_multiaxis.py:89,191` — pops its own
  `__compact_multiaxis_*` temp frames (feature-local, correct under the target
  model).
* `domain/workbook_views.py::apply_workbook_view_sheet_mappings` — readback
  direction: output contains *only* mapped frames plus `_meta` (a de-facto
  frame selection on the reverse path).
* `worldbuilding/plugins/frame_sets.py::keep_frames` — consumer plugin,

  intermediate frames before the JSON write.
* There is *no generic drop/prune step* in
  `pipeline/registry.py::REGISTRY`. The epic's Review Question 6 is answered:
  nothing reusable exists in core; the nearest artifact is the consumer
  plugin, with inverted (keep-list) semantics.

=== 3.5 Orchestration and writers

`application/orchestrator.py::orchestrate` (line 67–121): one input load →
linear `run_pipeline` → *orchestrator-owned persistence boundary*
(`pipeline/persistence_boundary.py::project_meta_to_persistable_contract`,
deliberately not a YAML-visible step) → one output save. This is an existing,
documented precedent for an implicit carrier-neutral end-stage between the last
configured step and persistence.

Writers can also appear mid-pipeline as ordinary steps
(`write_structured_yaml`, `write_key_value_resources`,
`write_artifact_manifest` in `pipeline/registry.py:84–164`), so one run can
emit multiple carriers. These steps add *report frames* (e.g.
`structured_yaml_files`) with no lifecycle marking; without a Workbook View
they are rendered into the workbook and persisted to structured outputs —
another concrete candidate for explicit cleanup or view exclusion.

The architecture guard
`tests/architecture/pipeline_semantics/test_pipeline_semantics_guards.py::test_pipeline_orchestration_does_not_route_on_workbook_payload_metadata`
forbids the string literals `_meta`, `frame_lifecycle`, `workbook_view` in
`pipeline/{__init__,build,execution,registry,runner}.py`. The persistence
boundary module and `application/orchestrator.py` are *not* in the guarded set.
A final cleanup implemented as a domain/boundary module invoked by the
orchestrator (mirroring `persistence_boundary.py`) or as an explicit registry
step does not collide with the guard; embedding meta interpretation in the
guarded orchestration modules would.

`pipeline/execution.py` traces per-step `_meta` diffs at DEBUG
(`_meta_change_trace`). Frame-*set* changes are not traced. Deferred cleanup
directives written to `_meta` would appear in the trace; the executed drop
would be visible only via the directive-consumption meta change (see
amendment A3) unless frame-set tracing is added.

== 5. Counterexamples and semantic blockers

No semantic blocker was found. Four falsification attempts produced material
findings that require amendments rather than rejection:

=== 5.1 Transformations alone cannot originate all cleanup decisions

Attempted falsification of "functional transformations decide logical
retention": in the dino reverse pipeline the frames that must be cleaned up
(`story_cast_role_named`, `story_cast_role_matrix_view`, …) are outputs of
*generic* steps (`join_frames`, `expand_xref`, `project_by_role`,
`extract_frame`). A generic join cannot know whether its output is a terminal
domain frame or a disposable intermediate; only the pipeline builder knows.
The current `keep_frames` whitelist encodes exactly that builder knowledge.

Consequence: the cleanup contract is sound, but directive *origination* must be
open to the pipeline builder (per-step opt-in parameters such as
`drop_source: true` / `cleanup_output: true`, or a generic
`mark_frames_for_cleanup` step), not restricted to feature-internal decisions.
The epic's Responsibility 1 text ("a transformation must decide, based on its
own contract and parameters") is compatible with this if "parameters" is read
to include explicit builder-supplied cleanup parameters on generic steps —
the review recommends stating this explicitly.
Classification: *semantic finding requiring amendment* (A1).

=== 5.2 References to absent frames are often required intent, not contradictions

Attempted falsification of Slice 2's "fail deterministically on contradictory
retained references": in the dino forward flow, the persisted workbook meta
contains `xref_crosstable.story_group_matrix.relation = "story_group_named"`
and equivalent `compact_multiaxis.*` entries although `story_group_named` is
absent from the workbook artifact. The reverse pipeline *depends* on this
metadata to re-create the relation
(`expand_compact_multiaxis` → `output: story_group_named`). The
persistence boundary already codifies this distinction as Intent (preserved)
versus Resolution (stripped) in `pipeline/persistence_boundary.py:52–59`.

A cleanup validator that treats every metadata reference to a dropped frame as
contradictory would break the flagship pipeline. The failure rule must be
scoped to an explicit allow/deny list per metadata family (Section 8).
Classification: *semantic finding requiring amendment* (A2); also *confirmed
supporting evidence* for the additional-information document's initial-create
case — the framework already lives with "reference to a frame that does not
currently exist".

=== 5.3 Directive persistence through the workbook blob carrier

`_meta` roundtrips in full through the hidden workbook sheet
(`tests/integration/io_backends/xlsx/test_workbook_meta_blob_carrier.py`
asserts `mode`/`drop_redundant_data`/`unknown_frame_policy` survive today). If
cleanup directives are persisted like other canonical meta, a reverse pipeline
reading the workbook would inherit forward-run drop directives against frames
the inverse transformations are about to re-create — the epic's "cleanup
directives surviving after the frame was dropped earlier" conflict, arising
automatically rather than through builder error. Directives must either be
consumed on execution or stripped at the persistence boundary.
Classification: *implementation risk requiring amendment* (A3).

=== 5.4 Binary drop contract: no counterexample found

Searched for cases needing more than `drop: true` / absent: every existing
transformation either keeps its source (join, enrich_lookup, pivot, xref,
split_by_discriminator, extract) or removes private temp frames locally
(compact_multiaxis). The shadow-state (preserved-source) case needs *no*
directive. The initial-create case has no frame to mark. The only richer
semantic found anywhere is `keep_frames`' whitelist, which is a different
authoring ergonomic, not a different runtime semantic — it compiles to the
same binary per-frame decision. The binary contract holds.
Classification: *confirmed supporting evidence*.

=== 5.5 Revocation (deliberately open in the epic)

No current code produces competing cleanup decisions for one frame; there is no
shared-source dual-projection example in any repository. The additional
information document's YAGNI deferral is consistent with the evidence.
Recommendation: duplicate *identical* directives are idempotent and legal;
anything that would need revocation fails as a conflict until a concrete
shared-source case exists. Classification: *speculative complexity* (correctly
deferred).

== 6. Hidden implementation dependencies

. *Selection, renaming, and formula rewriting are fused.*
  `_select_configured_sheets` re-keys frames to *sheet* names before
  composition, so `compose_workbook`, all render passes, and per-sheet options
  (`_meta.sheets.<sheet-name>`, written by
  `workbook_views._merge_sheet_options`) operate on physical sheet names.
  A projection-plan refactor must preserve or deliberately migrate this
  rekeying point. (implementation risk)
. *Cross-sheet formula references assume target materialization.*
  `LookupFormulaSpec.lookup_sheet` names a *frame*; `frame_selection`
  rewrites it only when the frame is mapped under a different sheet name. If
  the lookup frame is omitted from the view (or later cleaned up), the
  rendered `XLOOKUP` silently references a nonexistent sheet. No validation
  exists today (`_resolve_formula_sheet_names` skips unmapped names).
  Slice 5 should validate content-level sheet references, not only mapping
  targets. (implementation risk)
. *`sheet_mappings[*].canonical_frame` is computed from lifecycle ontology* —
  see Section 8.2. (implementation/compatibility risk)
. *Non-render consumers of `workbook_view.sheets` mappings*:
  `domain/validations/validate_columns.py:205–235` (maps rule `sheet:` targets
  to frames), `domain/column_roles/resolver.py:236–249` (`_sheet_for_frame`),
  `rendering/passes/_base.py:55` (sheet-source candidates), and
  `domain/schema_maintenance/meta_refs.py::build_sheet_resolver`. These consume
  the *mapping* (compatible with the target model) but hard-code today's
  `frame`/`sheet` mapping shape; a future M:N mapping model must version or
  extend this shape for four additional readers. (implementation risk)
. *Architecture guard on orchestration modules* (Section 3.5) constrains where
  an auto-appended cleanup may be implemented. (implementation constraint,
  benign if respected)
. *`mark_source_if_unclassified` defaults `canonical: true`* for any
  unclassified source, and `write_frame_lifecycle(preserve_existing_canonical=True)`
  then locks that classification. Removal of render inference does not remove
  this write path; the canonical-flag semantics survive into `canonical_frame`
  computation. (implementation sediment to review in Slice 4)
. *Vocabulary collision:* frame-lifecycle `role` versus column roles
  (`project_by_role`, `column_roles/`) and versus `produced_by.mode` /
  validation `mode` are unrelated namespaces. Slice 4's "review `mode`" must
  scope strictly to `workbook_view.mode`. (documentation debt)

== 7. Carrier and roundtrip risks

* Confirmed asymmetry mechanism: only XLSX/ODS pass through
  `select_render_frames`; JSON/YAML/XML/CSV persist everything. With an
  explicit view this asymmetry is *intended artifact projection*; with the
  (dead) ontology branch it would be the unintended "logically redundant but
  still in JSON" state the epic describes. Today the visible asymmetry in
  practice is the *inverse* problem: redundant frames persist in *all*
  carriers (acme_03) unless a consumer plugin intervenes (dino). Final domain
  cleanup resolves both directions. (confirmed supporting evidence)
* `_meta` blob carrier roundtrip of directives — see 5.3. (implementation
  risk)
* Mid-pipeline writers (`write_structured_yaml` etc.) run before any
  end-stage cleanup; their report frames feed `write_artifact_manifest`.
  Cleanup of report frames must therefore be placeable *after* manifest
  aggregation — natural with builder-placed cleanup, impossible with a
  hard-wired "immediately after last step" auto-append that also targets
  report frames produced late. Keep placement pipeline-builder-owned as the
  epic states, or scope the orchestrated end-stage to directives only.
  (implementation risk, low)
* Deterministic tracing: meta-diff tracing covers directive writes; add
  frame-set change logging to the cleanup step's diagnostics to satisfy the
  epic's "deterministic and visible in traces" requirement. (implementation
  note)

== 8. Metadata-reference cleanup risks

Metadata families that can reference a frame name, with the recommended
cleanup policy for a dropped frame:

[cols="3,2,4"]
|===
| Path | Recommended policy | Evidence / rationale

| `frame_lifecycle.<frame>` (entry keyed by dropped frame)
| remove
| Entry describes a frame that no longer exists; retaining it re-creates the
  stale-ontology problem.

| `frame_lifecycle.*.derived_from` / `superseded_by` mentioning the frame
| preserve as historical provenance (or drop with the field, see Section 10)
| `superseded_by` has no consumer; `derived_from` feeds only
  `canonical_frame` computation at configure time, which runs before cleanup.

| `workbook_view.sheets[*].frame`
| *fail* — this is the epic's core conflict (cleanup vs. explicit mapping)
| Both declarations are explicit; silent precedence is excluded by the epic.

| `workbook_view.sheet_mappings[*].frame` / `.canonical_frame`
| fail if it names a dropped frame at write time; recompute on configure
| `resolve_workbook_view_sheet_mappings` already fails on unknown logical
  frames when given `logical_frames` (`domain/workbook_views.py:339–355`).

| `sheets.<sheet-name>` (per-sheet options)
| preserve; keyed by sheet name, orphaned entries are inert noise
| Options are looked up only for materialized sheets
  (`layout_composer.py:324–328`).

| `xref_crosstable.<name>.*`, `compact_multiaxis.<name>.*`,
  `split_by_discriminator.<name>.*`, `cell_codecs`
| *preserve* — inverse-transformation Intent
| Section 5.2; persistence boundary Intent/Resolution rule.

| `helper_policies.fk.relations[*]` (source/target frames)
| preserve; validate lazily at materialization
| Relations are durable v2 declarations
  (`persistence_boundary.py:81–105`); `enrich_helpers` and reverse cleanup
  need them even when a frame is temporarily absent.

| `legend_blocks[*].placement/target`
| preserve; targets sheets/tables, validated at compose time
| `layout_composer.py::_target_table_for_legend` already fails loudly.

| `validate_references` / `add_validations` rules
| out of scope — rules live in pipeline config, not `_meta`
| Rule targets failing on missing frames is existing step behavior.

| `derived.*` (runtime provenance)
| no action — already pruned at the persistence boundary
| `RUNTIME_ONLY_TOP_LEVEL_META_KEYS`.
|===

Key risk if this table is not made explicit in Slice 2: either over-aggressive
failure (breaking dino, Section 5.2) or silent stale references
(`frame_lifecycle` keys). Classification: *implementation risk*, resolved by
the per-family policy above.

=== 8.2 `canonical_frame` derivation

`configure_workbook_view` → `_sheet_mappings` → `_explicit_canonical_frame`
derives the persisted reverse-mapping `canonical_frame` from
`frame_lifecycle.<frame>.canonical == true` or a singleton `derived_from`
pointing at a canonical entry
(`domain/workbook_views.py:393–429`; characterized by
`tests/unit/domain/test_workbook_views.py::test_configure_workbook_view_persists_reverse_mapping_from_explicit_lifecycle`).
This is a *justified, distinct consumer* of `canonical` and `derived_from` in
the epic's sense — it feeds reimport identity, not render selection. Slice 4
must either retain these two fields for this consumer or replace the
derivation with an explicit `canonical_frame` declaration in the Workbook View
DSL. Silent removal would degrade `sheet_mappings` without failing any
currently-passing selection test. Classification: *hidden implementation
dependency / compatibility risk*.

== 9. Workbook View and future M:N implications

The explicit branch already provides the target's authoritative behavior
(Section 3.1). Hard 1:1 assumptions found:

* `_select_configured_sheets` — one `selected[sheet] = frame` dict; duplicate
  *frames* rejected in `configure_workbook_view`
  (`Duplicate workbook view frame`, line 152–153), so 1 frame → N sheets is
  blocked at the DSL today.
* `_resolve_formula_sheet_names` — `frame_to_sheet: dict[str, str]` is
  single-valued; 1 frame → N sheets makes formula target choice ambiguous
  (a genuine future design question, not a refactor blocker).
* `compose_workbook` — one `TableBlock` at A1 per sheet; but `SheetIR.tables`
  is already a *list* and legend blocks already place secondary structures at
  computed offsets on existing sheets
  (`layout_composer.py::_resolve_legend_position`) — the IR is M:N-ready even
  though composition is not.
* `workbookir_to_frames` — reads `sh.tables[0]` only
  (`rendering/workbook_projection.py:136`); the *read* side is the strongest
  1:1 assumption and is untouched by the epic's slices — fine, but M:N export
  without M:N readback would break roundtrip symmetry; note in Slice 7.
* Per-sheet options and four non-render mapping consumers keyed by single
  sheet name (Section 6.4).

Conclusion: the proposed refactor (projection plan, explicit mappings,
conflict validation) does not entrench 1:1 further and the IR substrate plus
legend precedent leave a credible path to 1→N sheets, N→1 composition, and
named-range targets. Requirement for Slice 3/7: model the plan as a list of
`(source frame, target)` mappings rather than a frame-keyed dict, and do not
promote the current rekey-then-compose fusion into the new contract.
Classification: *confirmed supporting evidence*, with the read-side note as

== 12. Recommended epic-slice changes (amendments)

*A1 — Slice 1:* extend directive origination — generic transformation steps
gain explicit opt-in cleanup parameters (e.g. `drop_source: true` on
`expand_xref`/`contract_xref`/`join_frames`-class steps where the builder
knows the source is fully consumed), and/or a generic
`mark_frames_for_cleanup` step. Also specify duplicate-directive semantics
(identical → idempotent; contradictory forms impossible while the contract is
a bare `drop: true`).

*A2 — Slice 2:* replace the blanket "fail on contradictory retained
references" with the per-family policy table of Section 8; explicitly preserve
transformation Intent metadata.

*A3 — Slice 2:* define directive *consumption* on execution so directives
cannot leak through the workbook blob carrier (Section 5.3); emit frame-set
diagnostics for trace visibility.

*A4 — Slice 2 placement:* implement as an explicit registry step *and/or* an
orchestrator-invoked standard end-stage in the same pattern as
`persistence_boundary` (outside the guarded orchestration modules), whichever
the open decision lands on — both are compatible with the existing
architecture guard.

*A5 — Slice 3:* add the `canonical_frame` decision (retain lifecycle-derived
computation vs. explicit DSL field, Section 8.2) and fix or remove the dead
`unknown_frame_policy` default divergence (Section 3.2) instead of carrying it
over.

*A6 — Slice 4:* is mostly deletion (Section 10); include removal of the
corresponding `configure_workbook_view` keyword parameters, and keep the two
negative-inference guard tests alive against the new selector.

*A7 — Slice 5:* extend conflict validation to content-level references —
rendered lookup formulas targeting unmapped/dropped frames (Section 6.2) and
legend targets — not only mapping-level duplicates and missing frames.

*A8 — Slice 7:* record the read-side 1:1 assumption
(`workbookir_to_frames` → `tables[0]`) as an explicit M:N constraint.

*A9 — New (small) work item:* decide whether a core `keep_frames`-style
whitelist step is offered alongside drop directives, so the worldbuilding
plugin can be retired; otherwise document the plugin as the sanctioned
pattern.
