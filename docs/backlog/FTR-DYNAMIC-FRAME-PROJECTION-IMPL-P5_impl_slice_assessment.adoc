== Verdict

Yellow.

The implementation-slice FTR is architecturally aligned and mostly executable:
it consumes the foundation role taxonomy, implements the recorded projection
decision without reopening it, keeps targeting out, assigns the public YAML
gate and release-note obligations to the implementation slice, and scopes the
shared resolver narrowly enough for the sibling targeting slice to reuse.

One bounded correction is needed before implementation: the worldbuilding
migration plan is not yet executable as written against the current
worldbuilding YAML. The FTR commits to an in-place `frame:` step, but the
current plugins are source-to-output transforms at several call sites. The
migration section also uses `story_cast_cells` as the example frame even though
the actual projected workbook surfaces are the `*_matrix_view` frames. This
does not reopen the architecture; it needs a precise wording/wiring correction.

== Executive Summary

The FTR is a good implementation contract for the core `project_by_role` step
and the shared role resolver. Requiring explicit `direction:` in YAML is an
improvement over the earlier Python sketch default because outbound and inbound
drop/order different role sets. The resolver rules are coherent as an interim
computed-role strategy and are properly framed as replaceable by a future
unified role channel. Warn-only missing `row_identity` is acceptable for this
slice, provided tests assert the warning path.

The main gap is the consumer migration. The current outbound YAML invokes
`build_story_cast_matrix_view` with `source` and `output`, for example
`story_group_matrix_enriched -> story_group_matrix_view` and four story-cast
matrix variants. The current inbound YAML likewise invokes
`build_story_cast_matrix_payload` with distinct source and output frames. The
new step's contract intentionally has only `frame`, so the migration plan must
state how the existing pipeline is rewired without adding `source` / `output`
to `project_by_role`.

== Main Strengths

* The FTR treats `FTR-DYNAMIC-FRAME-PROJECTION-P5_decision.adoc` as the source
  of truth and does not reopen Option C + minimal Option-B.
* The role vocabulary is exactly the foundation vocabulary:
  `row_identity`, `display_helper`, `matrix_value`.
* The YAML surface is small: `frame`, required `direction`, optional interim
  overrides, and constrained `role_order`.
* The shared resolver is scoped as a single interim module, not as a new
  column-role lifecycle or `_meta.column_roles` implementation.
* Targeting remains out of scope: no `roles:` target field, selector syntax,
  wildcard grammar, set algebra, or targeting parameter on `project_by_role`.
* The plugin-retention rule is explicit: the two projection plugin functions
  remain importable and behavior-compatible for one release cycle.
* Public-surface gate and release-note responsibilities are assigned to the
  implementation slice, which matches `docs/ai_info/interfaces_and_gates.adoc`
  and the release-note checkpoint.

== Findings By Priority

=== High-1: Worldbuilding migration is not executable as written

Observation::
The FTR's committed YAML surface is an in-place step:
`project_by_role(frame=..., direction=...)`. The migration plan says to replace
`plugin:build_story_cast_matrix_view` / `plugin:build_story_cast_matrix_payload`
with `frame: story_cast_cells`.

Evidence::
The FTR defines only `frame` and says the step replaces that frame's column
sequence in place (`docs/backlog/FTR-DYNAMIC-FRAME-PROJECTION-IMPL-P5.adoc`,
lines 169-173, 277-281, 312-316). Its migration examples use
`frame: story_cast_cells` (lines 337-339 and 356-358).
+
The current outbound worldbuilding YAML uses source-to-output plugin calls,
for example `source: story_group_matrix_enriched` and
`output: story_group_matrix_view`, plus four story-cast matrix view calls
(`../worldbuilding/pipelines/dino_insel/json_to_spreadsheet.yaml`, lines
94-98 and 240-262). The current inbound YAML does the same for payload frames
(`../worldbuilding/pipelines/dino_insel/spreadsheet_to_json.yaml`, lines
49-53 and 77-99).
+
The helper metadata currently names the final view frames, not
`story_cast_cells`: `story_group_matrix_view`,
`story_cast_role_matrix_view`, `story_cast_notes_matrix_view`,
`story_cast_character_development_matrix_view`, and
`story_cast_crystal_matrix_view`
(`../worldbuilding/pipelines/dino_insel/json_to_spreadsheet.yaml`, lines
296-308).

Risk::
An implementer following the FTR literally can either project the wrong frame
(`story_cast_cells`, the canonical long-form source), add unapproved
`source` / `output` parameters to `project_by_role`, or invent ad hoc pipeline
rewiring. Any of those outcomes would make the slice brittle and would blur
the public YAML contract.

Recommended action::
Revise only the migration-plan wording. Keep the `project_by_role` API
in-place unless the project deliberately reopens that surface in a separate
decision. Enumerate the actual worldbuilding surfaces/call sites and specify
the intended wiring, for example: produce the final `*_matrix_view` /
`*_matrix_payload` frame names before calling `project_by_role`, then call the
step in place on those frames. If existing primitives are used to copy/rename
frames, name them. Also state whether `helper_columns` / `key_columns`
overrides are required for these call sites, given the current order of
`configure_workbook_view`.

=== Medium-1: Migration parity checks should cover current plugin behavior, not only role order

Observation::
The tests require a minimal outbound/inbound fixture and data-value roundtrip,
but the migration risk is broader: the existing plugins also create new output
frames, normalize missing values to `""`, and the outbound plugin sorts by
`story_id` by default.

Evidence::
The FTR's integration-test requirement says to mirror the worldbuilding flow on
a minimal fixture and assert retained values roundtrip (`docs/backlog/FTR-DYNAMIC-FRAME-PROJECTION-IMPL-P5.adoc`,
lines 440-445). The current plugin functions perform source/output projection,
`where(pd.notnull(...), "")`, and outbound `sort_by=("story_id",)`.

Risk::
The core step can satisfy its narrow column-role tests while the migrated
worldbuilding pipeline changes row order, frame names, or empty-cell behavior.
That would make rollback necessary even though the implementation appeared to
meet the written FTR.

Recommended action::
Add a migration-parity test requirement, still bounded to this slice: after
rewiring the worldbuilding YAML, exercise the actual dino_insel outbound and
inbound projection surfaces or a fixture that covers all current projection
call-site shapes. The assertion should include frame names consumed by the
next step, row order if it is observable, empty-cell normalization if it
changes, and data equality for retained columns. If row sorting is intentionally
left to an upstream/downstream step, say so in the migration plan.

== Scope Check

The FTR stays inside the projection implementation slice.

In scope and appropriate:

* `project_by_role` as a backend-neutral pipeline step.
* Role-based outbound ordering and inbound helper dropping.
* A shared resolver module consumed first by projection and later by targeting.
* Registration of the new step and documentation of the public YAML surface.
* Worldbuilding YAML migration for the two named projection plugin functions.
* One-release-cycle importability for the two plugin functions.

Out of scope and successfully excluded:

* Targeting `roles:` fields on validation / constraint / presentation rules.
* Selector syntax, wildcard grammar, set algebra, or composition operators.
* Targeting parameters on `project_by_role`.
* Editability, merge-back, collaboration, lifecycle modelling, or presentation
  classification semantics.
* A unified `_meta.derived.sheets.<frame>.column_roles` channel.
* Refactoring `contract_xref`, `expand_xref`, `join_frames`, or
  `configure_workbook_view`.
* Migration of `add_from_legend_constraints_for_matrix` or other unrelated
  worldbuilding plugins.

== Test Adequacy / Missing Test Concerns

The proposed unit-test set is strong for resolver precedence, default role
orders, invalid `direction`, invalid `role_order`, and missing `frame`.

The missing coverage is migration-specific, not architecture-specific:

* actual current source/output plugin call-site shapes;
* all current dino_insel projection surfaces, not only one illustrative
  `story_cast_cells` example;
* preservation or deliberate relocation of the outbound plugin's row sorting;
* empty-cell normalization parity, if the new step changes it;
* confirmation that resolver inputs are available at the point where the
  migrated YAML calls `project_by_role`.

The shared-resolver guard is appropriate. A repository-level or architecture
test that `project_by_role` calls the shared module is a reasonable forcing
function and does not overgeneralize the foundation.

== Commit Readiness

Not commit-ready as-is.

This is a Yellow planning review, not a Red architecture rejection. The FTR can
become commit-ready with limited wording changes to the worldbuilding migration
plan and test checklist. No architecture redesign is required.

== Recommended Changes Before Implementation

. Rewrite the *Worldbuilding migration plan* to enumerate the actual current
  outbound and inbound projection frames/call sites.
. State exactly how the in-place `frame:` surface replaces current
  source/output plugin invocations without adding `source` / `output` to
  `project_by_role`.
. Replace the `frame: story_cast_cells` migration examples with real projected
  surface examples, or label `story_cast_cells` as illustrative only and add
  concrete dino_insel examples.
. Clarify whether `project_by_role` runs before or after
  `configure_workbook_view` in the outbound pipeline, since helper resolution
  depends on that metadata unless explicit overrides are supplied.
. Add migration-parity test wording for frame names, row order, empty-cell
  behavior, and all current projection surfaces.

== Non-Findings Worth Recording

* Explicit `direction:` in YAML is acceptable and preferable. The earlier
  `outbound` default was a Python sketch, not the final public YAML contract;
  requiring direction avoids silent inbound/outbound semantic drift.
* Warn-only missing `row_identity` is acceptable for this slice. It keeps early
  consumer wiring possible and is explicitly tested. The implementation should
  make the warning visible and should still fail on missing `frame` or invalid
  configuration.
* The resolver rules are coherent as an interim computed-role strategy. The
  FTR avoids the decision artifact's more speculative natural-key fallback and
  keeps explicit overrides as narrow transitional affordances.
* The shared resolver module is correctly scoped. It is a small single
  source-of-truth helper for projection and future targeting, not a premature
  implementation of the foundation metadata channel.
* Keeping `build_story_cast_matrix_view` and
  `build_story_cast_matrix_payload` importable for one release cycle is well
  documented and rollback-safe in principle.
* Release notes and public-surface gate requirements are correctly assigned to
  the implementation slice.
* `_ftrs_open.adoc` registration and ordering are appropriate: the
  implementation slice appears after the projection evaluation/decision path
  and before the later targeting implementation work in Phase 3d.
* The FTR does not accidentally pull in targeting, selector syntax, lifecycle,
  editability, merge-back, or unified `_meta.column_roles`.

=== Verdict

Green.

The revised FTR closes both prior findings. It is now a bounded, executable

=== High-1 Status

Closed.

The FTR now explicitly states that `project_by_role` is an in-place transform
and has no `source:` / `output:` parameters (`FTR-DYNAMIC-FRAME-PROJECTION-IMPL-P5.adoc`,
lines 137-142). It also removes the misleading `story_cast_cells` example and
uses real projected matrix surface names instead (lines 144-170).

The source/output mismatch with the current plugins is reconciled by a concrete
upstream-rename strategy:

* outbound: rename the immediately preceding `join_frames output:` from
  `*_matrix_enriched` to `*_matrix_view`, then run `project_by_role` in place
  on the `*_matrix_view` frame (lines 408-466);
* inbound: run `project_by_role` in place on the existing `*_matrix_view`
  frame, then retarget the downstream `expand_compact_multiaxis` /
  `expand_xref matrix:` from `*_matrix_payload` to `*_matrix_view`
  (lines 471-518).

The 10 actual current call sites are enumerated correctly: five outbound
plugin calls in `json_to_spreadsheet.yaml` and five inbound plugin calls in
`spreadsheet_to_json.yaml` (FTR lines 363-395). This matches the reviewed YAML.

The strategy is executable with existing pipeline primitives. Outbound uses
the existing `join_frames output:` parameter; inbound uses existing downstream
`matrix:` parameters. The FTR explicitly forbids silently adding `source:` /
`output:` to `project_by_role` and treats that as an unchosen fallback that
would require an explicit decision amendment (lines 397-406 and 556-571).

The `helper_columns: [title]` and `key_columns: [story_id]` overrides are now
correctly justified. The FTR records that `configure_workbook_view` runs after
the outbound projection sites, and that `contract_xref` row keys are declared
on upstream matrix frames rather than the projected `*_matrix_view` frames
(lines 252-270 and 428-431). The same explicit overrides are required for all
10 migration call sites in the acceptance criteria (lines 860-864).

=== Medium-1 Status

Closed.

The revised FTR adds worldbuilding-specific migration-parity tests covering
the exact dimensions requested:

* downstream frame-name availability for outbound `configure_workbook_view`
  entries and inbound `expand_* matrix:` consumers (lines 666-679);
* all current projection surfaces, not only the illustrative
  `story_group_matrix_view` example (lines 657-664 and 708-711);
* outbound row-order parity against pre-migration plugin behavior, with test
  failure rejecting the migration rather than adding sorting to
  `project_by_role` (lines 680-688);
* empty-cell normalization parity through the inbound end-to-end flow, with
  any needed normalization assigned to a separate existing YAML primitive
  rather than to `project_by_role` (lines 689-700);
* resolver-input availability for explicit overrides at each of the 10
  surfaces (lines 701-707).

The risk section and acceptance criteria now reinforce those checks as
blocking implementation obligations (lines 772-789 and 865-868).

=== New Findings

No new findings introduced by the correction.

The fallback paragraph mentions a possible future `source:` / `output:`
surface only as an explicit last-resort path requiring a decision amendment.
It does not change this implementation slice's committed surface and does not
reopen the projection architecture decision within this FTR.

=== Commit Readiness

Commit-ready as a planning/FTR artifact.

No further wording changes are required before implementation. The future
