== Verdict

*Yellow -- keep with required follow-up before release.*

There are no blocking findings and no evidence of scope creep into the actual
BUG fixes. The roundtrip layer is doing the right kind of work: full
`canonical -> workbook -> canonical` cycles through the public CLI entry path,
with open invariants represented as visible `xfail`s.

The required follow-up is test-architecture hygiene: the new `roundtrip`
topology marker is auto-applied by path, but the carrier-boundary guard does
not yet treat `roundtrip` as a path-owned topology marker. That leaves an
escape hatch where a test outside `tests/roundtrip/` could be manually marked
`roundtrip` and join `pytest -m roundtrip tests/` without a guard failure.

== Blocking Findings

== Non-blocking Findings / Follow-ups

=== F1: Roundtrip marker ownership is under-guarded

*Severity:* Medium follow-up
*Location:* `tests/architecture/carrier_boundary/test_carrier_boundary_guards.py`

`ACTIVE_CARRIER_ROOTS` correctly includes `roundtrip`, but
`PATH_OWNED_TOPOLOGY_MARKERS` remains:

----
("unit", "integ", "arch", "current_state", "legacy", "prehex")
----

The new tests manually apply `pytest.mark.roundtrip` while
`tests/conftest.py` also auto-applies the same marker for files under
`tests/roundtrip/`. The explicit mark is defensible as selection redundancy,
but the architecture guard should either:

* include `roundtrip` in the path-owned topology marker set and encode the
  allowed duplicate marker pattern for tests under `tests/roundtrip/`; or
* explicitly document that `roundtrip` is intentionally not path-owned.

The current state works for this commit's tests, but it weakens the stated
topology rule by allowing manual `roundtrip` markers outside the carrier root.

=== F2: Makefile and AI-info quick references are slightly stale

*Severity:* Low follow-up
*Locations:* `Makefile`, `docs/ai_info/conventions.adoc`

`ACTIVE_TEST_PATHS` now includes `tests/roundtrip`, and the functional target
works. Nearby descriptive text still says the default active slice is only
unit + integration + architecture:

* `Makefile` quick reference: `make test # unit + integ + arch, no slow/prehex`
* `Makefile` target help: `test: ... Active suite: unit + integ + arch`
* `docs/ai_info/conventions.adoc` running-tests quick reference:
  `make test # active dev slice: unit + integration + architecture`

This is not a behavior bug, but it undercuts the documentation update's goal
of making roundtrip a clearly named active layer. `test-roundtrip` is also not
listed in the Makefile `.PHONY` declaration; the target works today, but it
should be made consistent with the other test targets.

=== F3: Make dependency bootstrap adds noisy network retry output

*Severity:* Low follow-up
*Location:* `make test-roundtrip`, inherited from `deps-dev`

In this sandbox, `make test-roundtrip` first attempted to upgrade
`pip`, `setuptools`, and `wheel`, hit DNS failures for `pypi.org`, then
continued and ran the roundtrip tests successfully. This appears inherited
from the existing Make dependency flow rather than introduced by this commit.
For the new target's clarity, consider whether dependency-refresh network
noise should be suppressed, cached, or made opt-in in local/offline runs.

== Marker / Makefile Behavior

Marker registration is correct, and `tests/conftest.py` auto-marks files under
`tests/roundtrip/`. The explicit module-level `pytest.mark.roundtrip` markers
also make the new tests robust under `pytest -m roundtrip` regardless of hook
ordering.

`make test-roundtrip` behaves clearly at the pytest layer: it runs only
`tests/roundtrip` under the existing default marker filter
`not legacy and not prehex and not slow`, and the expected xfails are visible.
The target's external clarity is slightly reduced by inherited dependency
bootstrap network retry noise and by stale adjacent Makefile/docs descriptions
of the active suite.

Adding `tests/roundtrip` to `ACTIVE_TEST_PATHS` is appropriate now: the layer
contains one green invariant test and two intentional xfails, so it raises the
visibility of open roundtrip invariants without turning the default suite red.

== Xfail Discipline

The cleanup invariant is marked:

----
strict=True
reason="BUG-HELPER-CLEANUP-CONTRACT-P4A: ..."
----

That is appropriate because the expected transition after the cleanup BUG fix
is deterministic: an unexpected pass should fail the suite and force removal
or update of the xfail.

The unresolved-helper-value invariant is marked:

----
strict=False
reason="BUG-UNRESOLVED-HELPER-VALUE-NAN-SEMANTICS-P4A: ..."
----

That is acceptable for this slice because the final representation contract is
explicitly undecided. The candidate assertion keeps the current `"nan"` failure
visible without treating an intermediate XPASS as a release blocker.

No skips were found for the open BUG invariants.

== Invariant Orientation

The tests are named for invariants, not implementation details:

* `test_helper_columns_absent_after_canonical_reimport`
* `test_meta_stable_from_second_roundtrip`
* `test_empty_fk_source_helper_representation`

The assertions operate on full-cycle outputs rather than step internals. The
shared fixture builds YAML configs and invokes
`spreadsheet_handling.cli.apps.run.main(["--config", ...])`, which is a
sufficient public CLI path for this layer. The tests do not call production
domain transformation functions directly.

The metadata-convergence test compares `_meta` from cycle 2 to cycle 3 after
scrubbing an explicit drift whitelist. This is narrow enough for the current
minimal fixture because it does not compare to a large static expected payload
and it does not require first-cycle equality. Future legitimate drift should be
added to the whitelist with an explicit reason, not hidden behind approximate
equality.

== Fixture Assessment

`minimal_fk_dataset` is appropriately synthetic. It contains two small frames,
one FK relation, one helper materialization path, and one empty FK source row.
It avoids copying worldbuilding data while reproducing the relevant failure
class: helper materialization, workbook reimport, cleanup intent, and
unresolved helper values.

== Architecture Guard Assessment

Adding `roundtrip` to `ACTIVE_CARRIER_ROOTS` is correct because
`tests/roundtrip/` is now an active test carrier.

The guard update is incomplete for marker ownership, as described in F1. The
test topology rule should explicitly cover how `roundtrip` markers are owned,
especially because this commit intentionally combines path auto-marking with
manual module-level marks.

== Scope / Docs Assessment

No production code was changed. The commit does not fix
`BUG-HELPER-CLEANUP-CONTRACT-P4A` or
`BUG-UNRESOLVED-HELPER-VALUE-NAN-SEMANTICS-P4A`; it exposes them with xfails.

The AI-info docs are broadly sufficient and do not claim the BUGs are fixed.
They record the new layer, public CLI expectation, invariant naming, and
xfail-not-skip rule. The only documentation concern is stale quick-reference
wording around the default active test slice.

== Final Recommendation

Keep the commit. Before release, tighten the marker-ownership guard for
`roundtrip` and refresh the stale Makefile / AI-info quick references so the
new layer is represented consistently. No implementation fixes are needed in
this review slice.
