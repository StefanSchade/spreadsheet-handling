===== FTR-PRESENTATION-DEFAULTS-SEMANTIC-CLASSIFICATION-P5 -- Classify workbook presentation defaults and define their architectural path

*Status:* New

*Purpose:*::
Clarify the architectural role of workbook-wide presentation defaults such as
header coloring, freeze panes, and autofilters, and decide whether these
capabilities remain direct renderer options, become Render IR capabilities, or
are modeled as metadata-driven semantics.

*Context:*::
The project currently contains a mixture of presentation-related mechanisms.

Some presentation behavior originates from workbook-level configuration, for
example:

[source,yaml]
----
excel:
auto_filter: true
freeze_header: true
header_fill_rgb: D9EAD3
----


Other presentation behavior originates from semantic metadata and derived
workbook structure, for example helper-column rendering, editable/read-only
classification, validation rendering, and future column-group presentation.

This creates an architectural question:

* Are workbook-wide presentation defaults a distinct semantic class?
* Should they continue to bypass metadata entirely?
* Should they be normalized into Render IR before backend execution?
* Which capabilities require backend parity and which are accepted backend
  differences?

The goal of this FTR is classification and direction, not immediate feature
implementation.

*Motivation:*::

The current behavior is historically evolved and partially predates the
Render IR architecture.

The project should explicitly decide whether concepts such as:

* autofilters,
* freeze panes,
* default header styling,

are:

* renderer defaults,
* presentation capabilities,
* metadata-driven semantics,
* or a hybrid model.

Without a documented decision, future features risk following inconsistent
paths.

*Scope:*::

* Inventory existing workbook-wide presentation defaults.
* Classify them as a coherent semantic category.
* Define their intended architectural flow.
* Define whether workbook-wide presentation defaults remain configuration-only
  or become Render IR concepts.
* Define expectations for backend parity (XLSX vs ODS).
* Update technical documentation if a classification is accepted.

*Non-Goals:*::

* No immediate migration of existing configuration surfaces.
* No removal of existing `excel:` configuration.
* No mandatory movement of all presentation concerns into `_meta`.
* No implementation of new rendering features.
* No redesign of helper-column semantics.
* No UI or application-level theming framework.

*Architectural Options:*::

Option A -- Configuration Defaults
+
[source,text]
----
config -> backend
----

* Keep workbook-wide presentation defaults as renderer configuration.
Simple but architecturally separate from newer Render IR concepts.

Option B -- Render Capability Path
+
[source,text]
----
config -> Render IR -> backend
----

* Workbook-wide presentation defaults become backend-neutral render
capabilities while remaining configuration-driven.

Option C -- Metadata Driven
+
[source,text]
----
config -> metadata -> Render IR -> backend
----

* Presentation defaults become explicit semantic model elements.

*Acceptance:*::

* A documented classification exists for workbook-wide presentation defaults.
* The intended architectural path is explicitly recorded.
* The relationship between configuration, metadata, Render IR, and adapters
  is clarified.
* The status of existing features (`auto_filter`, `freeze_header`,
  `header_fill_rgb`) is documented.
* Backend parity expectations are documented.
* Future presentation features have a clear precedent to follow.

*Validation:*::

* Render technical documentation.
* Review architecture consistency with existing Render IR concepts.
* Verify that the chosen classification does not contradict current
  metadata-boundary rules.

*Residual Risks:*::

* Different spreadsheet backends may continue to expose different native
  presentation capabilities.
* Excessive semanticization of purely cosmetic features may create
  unnecessary complexity.
* Excessive reliance on backend-specific configuration may weaken Render IR
  consistency.

*Discussion Note:*::

A likely outcome is that workbook-wide presentation defaults form a separate
architectural category from domain-derived presentation semantics.

Examples:

* Workbook-wide autofilter.
* Workbook-wide freeze-header behavior.
* Default header styling.

These differ fundamentally from semantics derived from business meaning,
such as helper-column classification, validation intent, editable-state
annotation, or future helper-column grouping.
