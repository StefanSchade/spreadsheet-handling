===== FTR-FUNCTIONAL-SEMANTIC-MODEL-SPLIT-P5 -- Stabilize the Functional/Semantic model boundary

*Status:* New
*Priority:* P5
*Area:* Documentation / models

==== Purpose

Stabilize the model documentation around an explicit boundary rule:

* The *Functional Model* explains the problem space -- working forms,
  motivation, usage scenarios, process boundaries, fachliche risks, and
  worked examples.
* The *Semantic Model* defines the contract -- externally visible
  vocabulary, guarantees, interpretation rules, producer/consumer
  responsibility, precedence, lifecycle, and guard candidates.
* The *Technical Model* describes the implementation structures in which
  these contracts are realized and protected.

This FTR does not introduce a new model layer and does not invent new
runtime concepts. Its scope is *navigational* documentation work, with a
single, narrow editorial pass on the four chapters identified by the
boundary review as misplaced.

==== Context

Two artifacts motivate this FTR:

* The German working concept under
  `docs/warm_storage/problemraum_konzept/` lays out the working hypothesis
  ("Functional explains, Semantic contracts, Technical implements") and
  proposes this FTR in its closing chapter
  (`06_restrukturierung_und_ftr_vorschlag.adoc`).
* The read-only boundary review at
  `docs/warm_storage/reviews/semantic_boundary_review.adoc` (verdict
  YELLOW) classifies every Semantic Model chapter against the working
  hypothesis, identifies four bounded misplacements, and recommends this
  FTR as the single owner of the recut.

The review's headline observations are taken as the entry state:

* Of the 18 Semantic Model chapters, 11 are clearly contract, 2 are
  clearly explanatory (03 Problem Statement, 10 Roundtrip Walkthroughs),
  3 are mixed (01 Management Summary, 05 External Inspirations,
  08 Lifecycle And Update Semantics), and 2 are process/adoption material
  (15 Adoption Strategy, 16 Planned FTRs).
* The semantic-growth mechanism the German concept names is real but
  currently bounded; the Adoption Strategy chapter is the project's
  existing brake on it.
* The Functional Model has structural room to host more problem-space
  content but has identifiable gaps (no ontology chapter; no
  information-flow chapter equivalent to problemraum_konzept ch03).
* The hypothesis's binary cut has a blind spot for adoption/process
  material; the recut must carve out a third zone explicitly.

==== Background -- four load-bearing decisions

The review names four boundary cases that this FTR must decide. Each is
listed with the current state, the proposed direction, and the open
question the FTR must answer.

. *Problem Statement (`docs/semantic_model/03_problem_statement.adoc`)*
+
Current state: ~165 lines of problem-space narrative (FK-helper review
friction, early/growing/legacy ambiguity handling, identity-keyed-mode
anti-pattern, roundtrip promise).
+
Proposed direction: relocate the framing into the Functional Model and
have the Semantic Model entry path reference it. Concrete FK-helper
examples that *also* serve as evidence for contract chapters (Ambiguity
Policy, Lifecycle) should be referenced from those chapters rather than
duplicated.
+
Open question: whether the chapter is *moved* in a single editorial step
or whether a stub remains under chapter 03 as a navigation pointer.

. *Roundtrip Walkthroughs (`docs/semantic_model/10_roundtrip_walkthroughs.adoc`)*
+
Current state: forward, reimport, and repeated-roundtrip process flows
plus question lists. Self-described as eventually containing PlantUML
sequence diagrams.
+
Proposed direction: relocate the walkthroughs into the Functional Model
(operating model / use-case chapters). The contract chapters that
constrain behavior at the process boundaries reference the walkthroughs
rather than reprint them.

. *External Inspirations (`docs/semantic_model/05_external_inspirations.adoc`)*
+
Current state: each subsection pairs an inspirational "relevant ideas"
block (explicitly non-normative) with a "project interpretation" block
(contract-adjacent).
+
Proposed direction: separate the two halves. The "relevant ideas" half
moves into the Functional Model as background; the "project
interpretation" sentences either fold inline into Ambiguity Policy /
Lifecycle / Versioning where they state real contract, or are kept as
cross-referenced notes.

. *Persistence boundary in Lifecycle And Update Semantics*
+
Current state: chapter 08's "Persistence boundary" subsection reads as a
mechanism description: it names a concrete module
(`pipeline/persistence_boundary.py`), enumerates current rules,
operational scope, and known deferrals.
+
Proposed direction: the *rule* ("transient runtime keys must not be
persisted as canonical metadata") stays in the Semantic Model; the
*mechanism description* candidates as Technical Model content. Because
the Technical Model has no paired chapter today, this case stays
physically in place for this FTR and is only annotated with a navigation
note. A follow-up FTR owns the technical relocation.

==== Scope

In scope for this FTR:

* Add a Semantic Model navigation entry that explicitly classifies the
  18 chapters by role: *Contract Core*, *Explanatory Context*,
  *Adoption / Planning*, *Appendix*. The classification follows the
  review's Chapter Classification Matrix; new vocabulary or new chapters
  are out of scope.
* Carve out an explicit *Adoption / Planning* zone for chapter 15 and
  chapter 17, so the binary Functional/Semantic cut does not silently
  absorb process material.
* Integrate the four load-bearing decisions above as a single editorial
  pass on the affected chapters, with cross-references replacing
  duplication where possible.
* Add a Functional Model entry path that names the problem-space role
  explicitly and links out to the German problemraum_konzept as
  preserved input.
* Add a lightweight *problem-space vocabulary / information-flow
  overview* to the Functional Model that absorbs the *framing* of
  problemraum_konzept ch02 and ch03 into English. Size cap: *one concise
  chapter, or two short subchapters*; explanatory prose only, no
  classification model. The German source stays verbatim in warm
  storage.
* Land the work as one or more narrow FTR slices under the existing FTR
  Sizing Preflight (`docs/developer_guide/ch08_git_and_workflow/01_git_and_workflow.adoc`).
  The single-FTR landing is the goal; a split into a navigational slice
  plus an editorial slice is acceptable if the preflight requires it.

Out of scope for this FTR:

* No runtime behavior changes.
* No registry schema or registry content changes; no `meta_registry.yaml`
  edits; no `pipeline_step_registry.json` edits.
* No model implementation changes.
* No new semantic sets in chapter 07. The candidate additions named in
  chapter 17 (`AuditOnly`, `DiagnosticOnly`) are deferred until after
  this FTR ships, because new vocabulary added during the recut would
  have to be re-classified anyway.
* No broad rewrite of the Functional, Technical, or Semantic Model.
* No retroactive rewrite of historical FTRs or review artifacts.
* No relocation of the persistence-boundary mechanism description out of
  chapter 08 -- that is a separate Technical Model FTR, filed as a
  follow-up if and when a paired Technical Model chapter exists.
* No translation of the German problemraum_konzept itself. The German
  source stays in warm storage verbatim; only its framing is absorbed
  into English Functional Model prose.

==== Non-goals (explicit)

* This FTR is not policy ratification of the working hypothesis as
  project doctrine. It is the *first* operational application of the
  hypothesis; subsequent reviews may sharpen or revise it.
* This FTR does not introduce a documentation governance framework.
* This FTR does not invent a "fourth model" for adoption material. The
  Adoption / Planning zone is a Semantic Model sub-section, not a new
  model document.
* This FTR must not create a Functional Model ontology, taxonomy
  expansion point, semantic set model, lifecycle taxonomy, registry
  vocabulary, or rule-ready contract model. The new problem-space
  overview is *explanatory prose only*. If the editorial pass discovers
  that classification material is needed, the right answer is to file a
  follow-up FTR, not to expand scope here. The point of this FTR is to
  *contain* semantic inflation, not to relocate it from the Semantic
  Model into the Functional Model.

==== Acceptance

* Every current Semantic Model chapter is classified in the entry page
  as Contract Core, Explanatory Context, Adoption / Planning, or
  Appendix, matching the boundary review's Chapter Classification
  Matrix.
* The Functional Model has a stated problem-space entry path that
  references the German problemraum_konzept as preserved input.
* The Functional Model gains a lightweight problem-space vocabulary /
  information-flow overview (one concise chapter, or two short
  subchapters) that absorbs the framing of problemraum_konzept ch02-ch03
  into English. The new material is explanatory prose only and contains
  no classification model, set definitions, lifecycle taxonomy, or
  registry vocabulary. The German source remains verbatim in
  `docs/warm_storage/problemraum_konzept/`.
* The new Functional Model material carries a visible *Scope limit* note
  stating that the overview is explanatory only, that it is not an
  ontology or taxonomy, and that any future classification work belongs
  in the Semantic Model behind the Adoption Strategy's slice rules.
* Chapters 03 (Problem Statement) and 10 (Roundtrip Walkthroughs) are
  either relocated into the Functional Model or replaced with single-line
  navigation stubs that point there. No content is silently dropped.
* Chapter 05 (External Inspirations) has a clear internal split: the
  "relevant ideas" halves are either moved to the Functional Model or
  marked as explanatory; the "project interpretation" sentences are
  either folded inline into Ambiguity Policy / Lifecycle / Versioning
  where they state contract, or remain as cross-referenced notes.
* Chapter 08's persistence-boundary subsection carries a navigation note
  flagging the mechanism description as Technical Model material
  pending a follow-up FTR. No content is moved in this FTR.
* The FK-helper pilot matrix (chapter 12) is unchanged.
* The Adoption Strategy chapter (15) and Open Questions chapter (17) are
  unchanged in substance and are surfaced under an explicit
  "Adoption / Planning" navigation grouping.
* Chapter 16 (Planned FTRs) has an archival note for any item that is
  already done (per `_ftrs_open.adoc` and the cold-storage backlog).

* The boundary review at
  `docs/warm_storage/reviews/semantic_boundary_review.adoc` is closed by
  a follow-up note recording verdict movement (target: GREEN).

==== Residual risks

* *Reference rot.* Moving Problem Statement and Roundtrip Walkthroughs
  out of the Semantic Model increases cross-reference surface. Mitigated
  by using AsciiDoc xrefs (caught by the docs build) rather than raw
  paths.
* *Language-distance loss.* The German problemraum_konzept exists in
  part because its language distance from established English vocabulary
  is itself an asset. The English absorption may erode that distance.
  Mitigated by preserving the German source verbatim and treating the
  English chapter as a *framing summary*, not a translation.
* *Adoption-zone misclassification.* If chapter 15 (Adoption Strategy)
  is silently folded into either Contract Core or Functional Model, the
  project loses its existing brake on recursive semantic growth.
  Mitigated by making the Adoption / Planning zone a first-class
  navigation grouping.
* *Persistence-boundary fragmentation.* Splitting chapter 08's
  persistence-boundary subsection between Semantic and Technical Model
  prematurely risks losing a single coherent design description.
  Mitigated by leaving it physically in place under this FTR and
  deferring the relocation to a dedicated follow-up.
* *Hypothesis premature.* The working hypothesis has not been ratified
  as project doctrine. If this FTR's outcome makes the navigation worse,
  it must be reversible. Mitigated by the navigational-first design: the
  entry-page classification and chapter-level cross-references are
  cheap to revert.

==== Discussion note

The boundary review (verdict YELLOW) found that the misplacement is
*localized*, not pervasive, and that the contract vocabulary itself
(chapters 04, 06, 07, 09, 11, 12, 13, 14, 17, 18) is in good shape. This
FTR's value therefore comes from clarifying *where the next piece of
explanatory pressure lands* -- the Functional Model -- so that new
semantic vocabulary continues to have to defend itself against a stated
functional need rather than against another piece of semantic
vocabulary.

The FTR deliberately does not let the Functional Model become a second
classification surface. The new problem-space overview is a *short
explanatory entry path*, not an ontology. If a future need for
classification emerges, it travels through the Adoption Strategy's slice
rules and lands in the Semantic Model under the existing semantic-set,
lifecycle, and precedence chapters -- not under a parallel Functional
Model taxonomy.

A successful landing should move the boundary review's verdict from
YELLOW to GREEN. The review names two concrete preconditions for that
movement: (a) chapters 03 and 10, and the "relevant ideas" halves of
chapter 05, are no longer in the Semantic Model entry path; (b) the
Functional Model has a short problem-space vocabulary / information-flow
overview (with a visible Scope limit note) that absorbs the
problemraum_konzept's framing.
