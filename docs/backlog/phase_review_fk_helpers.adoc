== Review Verdict

*Ship with required edits.*

The phase is architecturally on-message and slot-fit is good: it
operationalizes the "heuristics propose, primitives execute" split
that `pipeline_semantics.adoc` already promises but does not yet
deliver. However, several FTRs misdescribe current state in ways that
will cause downstream rework (`configure_fk_helpers` already exists
and already writes `helper_policies.fk`; `remove_fk_helpers` already
prefers provenance and uses prefix only as fallback; the actual FK
convention is `id_(<target>)` not `<target>_id`). The phase also
silently grows a *parallel* metadata-shape question (relations vs.
the existing `helper_policies.fk` policy entry) without saying
whether the new shape *extends*, *replaces*, or *renames* what
`helper_policies.fk` produces today. That ambiguity must be resolved
in `FTR-FK-RELATION-POLICY-MODEL-P5` before anything else lands.

The collapsible-column-group block (the last three FTRs) is
defensible but is a *second feature* sharing the phase. It should
either be split into Phase 3a / Phase 3a-bis or be explicitly named
"optional, deferrable" so it cannot bleed back into the semantic
slice's critical path.

== Executive Summary

* The phase belongs after Phase 2. The current backlog had nothing
  between Phase 2 (public entrypoints) and Phase 4 (metadata
  stabilization). FK helpers are the most user-visible example of
  the convention-vs-policy gap that Phase 4 wants to fix in general,
  so resolving them first is a good "model case" choice and prevents
  Phase 2's new public docs from immediately going stale against
  Phase 4 changes.
* The split into seven FTRs is *roughly* the right shape but
  contains current-state errors and an ambiguity that the model FTR
  must resolve up front.
* The grouping/presentation block is correctly downstream of the
  semantic block; its dependency arrows are right. It is *not*
  obviously worth bundling into the same phase: it shares no code
  path with the semantic cleanup and its public value is much
  smaller. Calling the phase "Phase 3a FK semantics" and moving the
  grouping FTRs to a clearly optional "Phase 3a-bis" (or to Phase 7
  as a transformation-extension capability) would protect the
  semantic slice.
* Several non-goals are well chosen (composite keys, generic join
  inference, schema-v2, public BC). One non-goal is missing on the
  semantic side: rename/reconciliation interplay. The phase intends
  to make provenance authoritative for cleanup; that bumps directly
  against `FTR-RENAME-RECONCILIATION` in Phase 4 and the FTRs are
  silent about it.

== Roadmap Fit

=== Does the phase belong after Phase 2?

Yes. Three concrete reasons:

. *Public docs alignment.* Phase 2 just landed the maintained
  product-business-slice tutorial as the canonical first-hour user
  surface. The tutorial currently teaches FK helpers as a
  convention-based primitive (the demo pipeline uses
  `add_fk_helpers` with `detect_fk: true` and no explicit relation
  policy). If Phase 4 later changes the underlying mental model,
  Phase 2's freshly-stabilized tutorial becomes misleading. Putting
  3a here keeps the public surface and the runtime in sync before
  the next public artifact is released.
. *Concrete model case for Phase 4.* Phase 4's first three FTRs
  (`FTR-CONCEPT-DOCS-CLEANUP-P5`,
  `FTR-DERIVED-PROVENANCE-LIFECYCLE-P4A`,
  `FTR-META-SCHEMA-VALIDATION-P5`) all reason in the abstract about
  "derived provenance lifecycle", "schema validation", "conflict
  semantics". Phase 3a forces those abstractions to be *implemented*
  in one place first, which makes Phase 4 less prone to becoming a
  second metadata-schema project that designs in a vacuum.
. *Existing infrastructure is ready.* `configure_fk_helpers` already
  writes `_meta.helper_policies.fk`; `_validate_fk_policy_usage`
  already exists; the pipeline-step-registry already classifies
  `configure_fk_helpers` as `configuration` and the FK primitives as
  `primitive`. The phase tightens an in-flight refactor rather than
  starting a greenfield model.

=== Does it logically precede Phase 4?

Mostly yes. But the boundary with `FTR-DERIVED-PROVENANCE-LIFECYCLE-P4A`
is uncomfortably close. That P4A FTR already talks about
"`fk_helpers._write_helper_provenance`", "stale or orphaned
provenance", and the lifecycle of `_meta.derived.sheets.*.helper_columns`.
The Phase 3a primitives FTR will *itself* expand the derived
provenance shape (richer entries to support reimport cleanup and
later consistency checks). Either:

* P3a should be explicitly named the producer-side change and P4A
  the lifecycle-policy change, or
* the two FTRs should be ordered with an explicit handoff note.

The current phase descriptions are silent about this. See finding
*High-1*.

=== Useful model case for "heuristics write resolved config; primitives consume config"?

Yes -- this is the strongest argument for the phase. The current
`add_fk_helpers` is the cleanest counterexample to the rule the
project has already documented in `pipeline_semantics.adoc`:
heuristics live in `configuration` steps; `primitive` steps consume
resolved metadata. Right now `add_fk_helpers` is registered as
`primitive` but contains the FK-detection heuristic itself (via
`detect_fk_columns`). Closing this gap legitimizes the published
architecture, and reusing the resulting pattern for
`add_lookup_helpers` later is straightforward.

=== Slicing -- too many / too few / right?

Seven is on the high end but defensible. The three smallest
possible mergers are not improvements:

* Merging the *model* FTR into the *infer* FTR would couple a pure
  model change to runtime behaviour and lose the "model first,
  approved, then build" cadence the project already follows.
* Merging the *primitives* FTR into the *docs/demo* FTR would create
  one giant slice that touches almost every FK file in core and
  demo. Bad commit hygiene per `git_and_workflow.adoc`.
* Merging the *render capability* into the *helper grouping meta*
  FTR would re-couple "domain owns intent; render IR/adapters own
  output" -- which is exactly the boundary the capability FTR was
  carved out to protect.

Possible *split* that may be worth it: separating the render
capability work entirely into its own phase or into Phase 6 (backend
hardening). See *Medium-2*.

=== Are docs/demo at the right point?

Yes, in both blocks. One docs FTR per semantic block, after the
runtime change has landed, is the right cadence. The risk is they
become a single big batch if the upstream FTRs slip; that is
worth flagging in residual risks but not changing the structure.

=== Should render capability be before or after FK-helper policy?

Technically: independent. Roadmap-wise: *after* the semantic block
is right, because the grouping meta FTR (sixth) is the only
internal consumer of the render capability, and that FTR depends on
the policy block. If render capability lands first and then the
semantic block slips, the capability sits unused. Order is defensible
but state the dependency direction explicitly in the phase preamble.

=== Is the grouping block correctly downstream of explicit helper identity?

Yes. The phase's own "Semantic split" paragraph already says
"Column grouping must not be used as a substitute for helper
identity." Good. The risk is *practical* -- collapsible columns are
the more user-visible feature and may attract attention away from
the semantic cleanup. Keep them firmly second.

== Architecture Boundary Review

=== Configuration vs. primitive

Direction is right and matches `pipeline_semantics.adoc` (which
already lists both `configure_fk_helpers` and `infer_fk_relations`
as `configuration`-step examples). However, the FTR set conflates
two distinct configuration steps:

* `configure_fk_helpers` (already exists, *manual* policy)
* `infer_fk_relations` (proposed, *heuristic* policy)

The phase text occasionally treats them as a single thing
("`infer_fk_relations` followed by policy-driven primitives" in
docs realignment, but also "`configure_fk_helpers` as explicit
policy configuration"). The model FTR must clarify whether they
write the *same shape* or two adjacent shapes. See *Blocking-1*.

=== Heuristic inference vs. deterministic execution

The intent is clear: primitives consume resolved policy; heuristics
live in configuration steps. The current `add_fk_helpers` is the
canonical violation today. The proposed split is correct in
direction. One subtle problem: the *FK column convention*
(`id_(<target>)`) is *not just inference*. It is also the *naming
rule* the parser/writer use as cell-level FK identity. Removing
convention detection from the primitive does not remove the
convention from the system. The FTRs should say so explicitly, or
they invite a follow-up "where did the FK syntax go?" question.

=== Canonical metadata vs. derived provenance vs. presentation metadata

The phase introduces three metadata channels that *might* be the
same channel viewed differently:

* `helper_policies.fk.*` -- canonical, persistent (today)
* `helper_policies.fk.relations` -- canonical, persistent (proposed
  by the model FTR; shape is sketched but the registry placement
  decision is deferred)
* `helper_policies.fk.presentation` and/or
  `derived.sheets.*.helper_columns[].presentation` -- canonical or
  derived? (`FTR-FK-HELPER-COLUMN-GROUP-META-P5` correctly *flags*
  the question and refuses to decide it now)

The grouping meta FTR's hedge ("decide where the grouping intent
lives") is acceptable for now, but it must not survive the model
FTR. Whichever shape is chosen, all three slots must be in the
*meta_registry.yaml* before primitives land.

=== Domain/helper logic vs. Render IR vs. spreadsheet adapters

The column-group capability FTR draws this boundary correctly:
domain writes intent; Render IR produces backend-neutral ops;
adapters render. The proposed `SetColumnGroup` op is consistent
with the existing op family in `rendering/plan.py` (frozen
dataclass, sheet name, integer column indices, no carrier
specifics). Good.

One concrete risk: the FTR says "implement XLSX rendering" via
openpyxl column-outline support, and "ODS rendering if
straightforward". This is consistent with the
`spreadsheet_backend_capability_profiles.adoc` vocabulary. Make
sure the FTR adds the row to the ODS profile (`accepted initial
gap` is a documented vocabulary item there); otherwise the gap
gets carried in code comments instead of in the canonical
capability matrix.

=== XLSX vs. ODS carrier concerns

Well handled. The non-goal "No cross-adapter shared carrier helper
that imports XLSX assumptions into ODS or ODS assumptions into
XLSX" is exactly right. Recommend keeping that as written.

=== Runtime registry vs. documentation/registry artifacts

The FTRs correctly mention that `pipeline_step_registry.json` and
`meta_registry.yaml` must be updated. The architecture guard
`test_pipeline_step_registry` is mentioned in
`architecture_primer.adoc` and enforces parity with `REGISTRY`.
This guard should be named explicitly in the infer-step FTR's
validation block (currently it just says "Run pipeline registry /
architecture guard tests" -- name the test).

=== Public tutorial/docs vs. internal technical model

`FTR-FK-HELPER-DOCS-DEMO-REALIGNMENT-P5` correctly targets both
the user guide transformation catalog and the technical concept
docs (`docs/technical_model/ch04_concepts/fk_helper_metadata.adoc`).
That concept doc is currently the most authoritative prose source
and will need rewriting. Worth listing it by path. See
*Medium-3*.

== Current-State Accuracy

This is the section with the most blocking errors.

=== `configure_fk_helpers` already exists

The model FTR says: "the policy lives under `_meta.helper_policies.fk`
or under a new adjacent root, and document the decision." It does
not mention that *`helper_policies.fk` is already a real, populated
registry root, produced by the existing `configure_fk_helpers`
step*, and consumed by `enrich_helpers` via
`fk_helpers/policy.py`. The current shape per target is roughly:

[source,yaml]
----
helper_policies:
  fk:
    <target_key>:
      target: <target_key>
      target_sheet: <sheet_name>
      key: <id_field>
      allowed_helpers: [...]
      default_helpers: [...]
      helper_prefix: "_"
      fk_column: "<id>_(<target>)"
      label: <optional>
----

The model FTR's sketch introduces a *different* shape under
`helper_policies.fk.relations` (a list keyed by `(source_frame,
source_column)`), without saying it would *replace* or *augment*
the existing per-target dict. That is *Blocking-1*.

=== `infer_fk_relations` overlap with `configure_fk_helpers`

`configure_fk_helpers` already supports a `targets: "auto"` mode and
an `auto:` block (see `_auto_fk_policy_specs` in
`domain/helper_policies.py`). That is, *manual policy with bounded
inference is already implemented*, although it currently iterates
over target frames and writes per-target shapes.

The proposed `infer_fk_relations` step has overlapping
responsibility: it also infers using a `naming_convention` mode and
also writes `helper_policies`. The infer-step FTR must say:

* whether `configure_fk_helpers(targets: "auto")` is *deprecated*,
  *renamed*, or *kept* alongside the new step;
* whether the auto path in `configure_fk_helpers` is migrated into
  `infer_fk_relations` (which would be cleaner -- a single inference
  surface -- and matches the "one job per step" model).

This is *High-2* and must be answered before the infer step is
implemented; otherwise the project ships *two* heuristic FK
configuration steps.

=== `remove_fk_helpers` -- prefix fallback already de-emphasized

The phase preamble says "Remove the remaining convention-driven
FK-helper behavior". `FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5`
says "Refactor `remove_fk_helpers` so helper removal uses derived
provenance and/or explicit policy, not prefix fallback."

Current code (`fk_helpers/drop.py`):

[source,python]
----
sheet_prov = (derived_sheets.get(sheet) or {}).get("helper_columns")
if sheet_prov:
    # Metadata-backed removal: drop exactly the columns listed in provenance
    ...
else:
    # Prefix-based fallback
    ...
----

So the *provenance path is already primary*; the prefix path is
*fallback* for frames that arrive without `_meta.derived.sheets`
(reimport from an externally edited workbook is the obvious case).
This is *Medium-1*: the FTR should describe the current state
accurately, define exactly *when* the fallback fires today, and
specify whether the new behavior is "remove the fallback entirely"
or "narrow the fallback to a guarded compatibility mode for read-
path roundtrips."

=== Existing FK helper provenance lacks target-key information

The current provenance entry shape (`fk_helpers/provenance.py`):

[source,python]
----
{
  "column": fk.helper_column,
  "fk_column": fk.fk_column,
  "target": fk.target_sheet_key,
  "value_field": fk.value_field,
}
----

There is no `target_key` (the ID column on the target sheet). For
"helper value consistency checks" -- which the policy-driven
primitives FTR claims to enable -- you need to know which target
column the FK is supposed to point at. Today this is reconstructed
via `build_registry(...)["id_field"]` from defaults. The model FTR
should explicitly call out target-key as a required provenance
field; the primitives FTR should add it. This is *Medium-4*.

=== Existing guards already enforce some of the proposed rules

`tests/architecture/dependency_guards/test_architecture_guard_fk_helpers.py`
already enforces that FK-helper domain logic does not leak back
into `pipeline.steps`. The proposed FTRs add a stricter guard
(primitives cannot run inference). That is a *new* guard; it does
not replace the existing one. Both must coexist. The primitives FTR
should say "add a new architecture guard X; existing guard Y stays
unchanged" rather than "Existing architecture guards pass or are
updated", which is vague.

=== Proposed tests are realistic

For the most part yes. Specific notes:

* The infer-step FTR lists "deterministic ordering of written
  policy entries" -- this is necessary because the current
  `_auto_fk_policy_specs` iterates over `iter_data_frames(frames)`,
  which is dict-iteration order in Python; deterministic per CPython
  3.7+ but tests should pin it. Good.
* The primitives FTR lists "helper cleanup after reimport without
  prefix fallback" -- realistic, but only after *Medium-1* is
  resolved.
* The grouping-meta FTR lists "deterministic metadata ordering" --
  realistic given the rest of the meta registry uses ordered
  shapes.
* No FTR lists a test that *both* `configure_fk_helpers` and
  `infer_fk_relations` can coexist in one pipeline if both are
  retained (or one specifically forbidding it). Once *High-2* is
  resolved, the matching test must be added.

=== FK column convention is `id_(<target>)`, not `<target>_id`

The model FTR's policy sketch uses example column names like
`product_manager_id`. The actual project convention (in
`core/fk.py:FK_PATTERN` and in the demo data
`product.json` -- `id_(product_manager)`) is the *reverse*. This
won't change runtime correctness because the FTR's sketches are
illustrative, but it makes the FTRs look like they were written
against an imagined codebase rather than this one. Fix the
examples. This is *Low-1* on its own, but it amplifies the
impression that the model FTR was drafted without checking
`core/fk.py`.

=== `reorder_fk_helpers` is fully heuristic today

`domain/transformations/helpers.py:reorder_helpers_next_to_fk`
matches FK columns via `FK_PATTERN` and helper columns via
`f"{helper_prefix}{target}_"`. It uses *no* provenance at all
today. The primitives FTR scopes "Refactor `reorder_fk_helpers` so
helper placement uses explicit policy or helper provenance, not FK
name patterns." That is correct and matches the current state.
Good. (No correction needed; this is included only for completeness
since the prompt asked about it.)

== Findings

=== Blocking

==== Blocking-1: Model FTR must resolve the relationship to the existing `helper_policies.fk` shape

*Evidence:* `domain/helper_policies.py:configure_fk_helpers` writes
`_meta.helper_policies.fk[<target>] = { target, target_sheet, key,
allowed_helpers, default_helpers, helper_prefix, fk_column,
label? }`. `fk_helpers/policy.py:_fk_policies` reads exactly that
shape. The model FTR's sketch instead places relations under
`helper_policies.fk.relations` as a *list* keyed by source frame +
source column, with fields `source_frame, source_column,
target_frame, target_key, helper_fields, helper_columns,
produced_by`. These are different shapes and different keying.

*Why it matters:* If both shapes coexist, the
primitives FTR has to read two formats; if the new shape replaces
the old, the FTR is silently a breaking change to the public YAML
surface (`configure_fk_helpers` is listed in the user guide
transformation catalog and is registered in the runtime registry).
That collides with `interfaces_and_gates.adoc` which says step
parameter shapes and registered step names require explicit
approval. Either way, leaving this implicit will cause a redesign
mid-FTR-3.

*Recommended change:* In `FTR-FK-RELATION-POLICY-MODEL-P5`, add an
explicit subsection "Relationship to existing
`helper_policies.fk`" and choose one of:

. *Extend in place* -- keep the per-target shape; add a
  `relations: [...]` sub-list inside each target entry. No public
  shape removal.
. *Add adjacent root* -- introduce `helper_policies.fk_relations`
  as a separate registry root with the new shape; leave
  `helper_policies.fk` for helper *capability* (allowed/default
  helper fields, helper_prefix).
. *Rename and migrate* -- promote the new shape to be the canonical
  `helper_policies.fk` and remove the old shape. This is a public
  surface change and requires the gate in
  `interfaces_and_gates.adoc`; the FTR must say so and the developer
  must explicitly approve.

State the decision and the rationale; do not defer.

==== Blocking-2: Phase preamble must distinguish "remove convention" from "remove convention as primitive default"

*Evidence:* The phase preamble says "Remove the remaining
convention-driven FK-helper behavior before it becomes part of the
public mental model." Read literally that includes `FK_PATTERN`
itself, the convention `id_(<target>)`, and the helper-prefix
convention `_<target>_<field>`. These conventions are *not* just
inference: they are the cell-level wire format. The FK column
*name* is how cross-sheet references survive an XLSX/ODS roundtrip
without a schema layer.

*Why it matters:* A reader (human or AI) following the preamble
literally would propose removing `FK_PATTERN` from `core/fk.py`,
breaking parse and roundtrip. The non-goals section of
`FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5` already excludes "No
generic relation engine beyond FK helper use cases", but it does
not exclude "do not remove the FK column convention itself".

*Recommended change:* In `_ftrs_open.adoc` Phase 3a preamble, change
the third bullet to:

[quote]
"Remove the remaining convention-driven FK-helper *primitive
behavior* (inference inside `add_fk_helpers`, prefix-based default
in `remove_fk_helpers`) before it becomes part of the public mental
model. The FK column convention (`id_(<target>)`) remains as the
cell-level reference shape; that is a separate question owned by
later FTRs."

And add a non-goal in `FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5`:
"No removal of `FK_PATTERN` or the `id_(<target>)` cell-level
convention; this FTR only changes primitive *behavior*, not the
serialized FK shape."

=== High

==== High-1: Boundary with `FTR-DERIVED-PROVENANCE-LIFECYCLE-P4A` is undefined

*Evidence:* `FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5` says
"Write derived provenance rich enough to support reimport cleanup
and later helper value consistency checks", which changes the
shape of `_meta.derived.sheets.*.helper_columns`. The Phase 4 FTR
`FTR-DERIVED-PROVENANCE-LIFECYCLE-P4A` is *about* the lifecycle of
exactly that key. No FTR says who owns what.

*Why it matters:* Phase 3a may end up redefining the producer side
while Phase 4 redefines the consumer/lifecycle side, with both
under-specified about how they meet. Worst case: Phase 4 has to
re-do the producer work.

*Recommended change:* Add a "Coordination with Phase 4" note in the
Phase 3a preamble. Specify: Phase 3a owns *what fields the
producers write* (incl. richer provenance for value checks); Phase
4 owns *when entries are pruned, who is allowed to read them, and
schema validation*. Add a cross-reference both ways in the
respective FTRs.

==== High-2: Overlap of `infer_fk_relations` with `configure_fk_helpers(targets: "auto")`

*Evidence:* `configure_fk_helpers` already supports `targets:
"auto"` (`domain/helper_policies.py:_auto_fk_policy_specs`). The
proposed `infer_fk_relations` covers heuristic inference of the
same FK structure from naming conventions. The two would coexist
unless the FTR says otherwise.

*Why it matters:* Two configuration steps writing overlapping
shapes is exactly the "two heuristic surfaces" problem the phase
sets out to fix. Worse, with the model FTR's sketch they would
write *different* shapes.

*Recommended change:* In
`FTR-INFER-FK-RELATIONS-CONFIGURATION-STEP-P5` add a "Relationship
to `configure_fk_helpers`" subsection. Choose one of:

. The auto path of `configure_fk_helpers` is migrated into
  `infer_fk_relations`; `configure_fk_helpers` keeps only the
  manual/explicit path.
. `infer_fk_relations` is a thin wrapper that calls into
  `configure_fk_helpers(targets: "auto")` with the new policy
  shape.
. Both coexist; the public guidance is "use `infer_fk_relations`
  for new pipelines, `configure_fk_helpers(targets: "auto")` is
  deprecated".

Option 1 is cleanest. Whichever is chosen, list the public-surface
impact and confirm whether it triggers the
`interfaces_and_gates.adoc` approval gate.

==== High-3: "No public backward compatibility" is a public-surface assertion

*Evidence:* `FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5` includes
in non-goals: "No public backward compatibility. No legacy mode
hidden behind defaults. No automatic migration of old user YAML."

*Why it matters:* The library is on PyPI as v0.1.0b6 with a
maintained demo and a published Pages site. `interfaces_and_gates.adoc`
explicitly says: *"YAML pipeline/config surface (registered step
names, step parameter shapes...) -- Treat these as product API
even in beta -- do not change without explicit user approval."*
The non-goal is therefore *not* unilaterally available to an
implementing agent. It is a developer-approval decision.

*Recommended change:* In the FTR, add an *Approval gate* subsection:
"This FTR proposes a non-backwards-compatible change to the
`add_fk_helpers` parameter contract (`detect_fk` and the implicit
relation-detection path are removed). Per
`interfaces_and_gates.adoc`, the developer must approve this
breaking change before implementation begins." Phase 3a's preamble
should also mention that block-A FTRs collectively require this
approval, so the gate is visible at planning time, not buried in
FTR 3.

=== Medium

==== Medium-1: "Prefix fallback" mischaracterized in current state

*Evidence:* `FTR-FK-HELPER-DOCS-DEMO-REALIGNMENT-P5` validation
includes: "Check README and docs for stale convention-oriented
wording such as: ... `prefix fallback` ... where those phrases
describe primitive behavior rather than explicit configuration or
inference."

*Why it matters:* The user-guide transformation catalog *currently*
says, accurately, that `remove_fk_helpers` "Remove materialized
FK helper columns using derived provenance or prefix fallback."
That description is *correct* today (see `fk_helpers/drop.py`).
The FTR should specify what the new sentence will be -- not just
"remove prefix fallback wording". If the fallback is kept as a
read-path roundtrip safety net, the doc still needs to mention it.

*Recommended change:* Add an *Expected new wording* example to
`FTR-FK-HELPER-DOCS-DEMO-REALIGNMENT-P5`, e.g.:

[quote]
"`remove_fk_helpers` -- Remove materialized FK helper columns using
the provenance written by `add_fk_helpers`. Frames without
provenance metadata are an error in normal pipelines; the read
path may apply a guarded prefix fallback for externally-edited
workbooks."

(Adjust depending on the Block A decision; the point is to write
the *destination* phrasing, not just enumerate banned phrases.)

==== Medium-2: Column-group block is a second feature, not an extension

*Evidence:* The render capability FTR explicitly says it "does not
itself depend on FK-helper semantics" and that the operation
"`SetColumnGroup`" must contain no FK-specific terminology. The
helper-column-group meta FTR is the only internal consumer.

*Why it matters:* Bundling them inside Phase 3a invites three
risks:

. Implementation effort competes with the semantic block.
. If the semantic block slips, the grouping block may land first
  with no internal consumer (since the meta FTR depends on the
  semantic block) -- a render capability without a producer.
. The phase title "FK Helper Semantics and Helper Presentation"
  signals to readers that they are one project; future readers
  will trace the entire arc together.

*Recommended change:* Rename the phase "Phase 3a: FK Helper
Semantics" and move the three grouping FTRs into a separate
"Phase 3a-bis: Helper-Column Presentation" with a one-line note
that 3a-bis only unlocks after block A is done. Alternatively,
keep them in 3a but rename the phase title and add an explicit
"Optional / deferrable" tag on the three grouping FTRs.

==== Medium-3: Concept doc rewrite is implied but not scoped

*Evidence:* `docs/technical_model/ch04_concepts/fk_helper_metadata.adoc`
currently describes the "first conservative policy" -- helper
values may be overwritten, prefix-based cleanup is acknowledged as
the conceptual weakness, and so on. Most of that becomes wrong
after Phase 3a. `FTR-FK-HELPER-DOCS-DEMO-REALIGNMENT-P5`

this file by path.

*Why it matters:* The concept doc is the canonical narrative
referenced by `docs/ai_info/concepts_fk_helpers.adoc` and likely
to be loaded by future agents. If it falls out of sync, the
agent-facing rules diverge from the runtime within one phase.

*Recommended change:* In `FTR-FK-HELPER-DOCS-DEMO-REALIGNMENT-P5`
*Scope*, list specifically:

* `docs/technical_model/ch04_concepts/fk_helper_metadata.adoc`
* `docs/ai_info/concepts_fk_helpers.adoc`
* `docs/technical_model/ch06_architectural_layers/architectural_layers.adoc`
  (the FK-helper feature-slice trace)

These three are the agent-facing surfaces and they must change in
the same commit as the user-guide table.

==== Medium-4: `target_key` missing from proposed provenance

*Evidence:* Current `_write_helper_provenance` emits `{column,
fk_column, target, value_field}`. The model FTR's example
provenance is similar:
`{column: _product_manager_name, fk_column: id_(product_manager),
target: product_managers, value_field: name}`. Neither includes
the *target key column* (the PK column on the target sheet) that
the helper value was looked up against.

*Why it matters:* The primitives FTR claims "helper value
consistency checks" as an outcome. Verifying that
`_product_manager_name = lookup(product_manager.id, P-100,
name)` requires knowing that the join column was `id`, not just
the target sheet. Today the validation code reconstructs this from
`build_registry(...).id_field` -- which is exactly the "primitive
re-infers configuration" pattern Phase 3a is removing.

*Recommended change:* In `FTR-FK-RELATION-POLICY-MODEL-P5` policy
sketch, add `target_key: <id_field>` to the per-relation entry. In
`FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5`, list "extend derived
helper provenance with `target_key`" as a concrete change. Update
the meta registry note for `derived.sheets.*.helper_columns`
accordingly.

==== Medium-5: Architecture guard updates under-specified

*Evidence:* `FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5` says
"Existing architecture guards pass or are updated to reflect the
stricter split." The infer-step FTR says "Architecture guards
continue to enforce that auto/heuristic behavior belongs to
configuration steps, not primitive FK-helper execution."

*Why it matters:* "Pass or are updated" is too soft. The current
guard
(`tests/architecture/dependency_guards/test_architecture_guard_fk_helpers.py`)
only forbids domain logic in `pipeline.steps`. The new "primitives
must not infer relations from column names" rule is a *different*
guard. Without a concrete spec, the FTR can ship without adding
it.

*Recommended change:* In `FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5`
acceptance, add: "A new architecture guard
`test_fk_helper_primitives_consume_policy` exists. It asserts
that `domain.transformations.fk_helpers.enrich.enrich_helpers`
does not call `detect_fk_columns` and does not import
`FK_PATTERN`, and that any FK structure used by the primitive is
read from `_meta.helper_policies` (or whichever root the model FTR
chose)."

==== Medium-6: ODS capability gap needs canonical record, not just docstring

*Evidence:* `FTR-COLUMN-GROUP-RENDER-CAPABILITY-P5` says: "If ODS

*Why it matters:* The project has a canonical capability table at
`docs/technical_model/ch08_architecture_details/spreadsheet_backend_capability_profiles.adoc`
with the vocabulary `direct support / carrier-backed support /
hint-only parse support / accepted initial gap`. If a "gap" is

*Recommended change:* Add to `FTR-COLUMN-GROUP-RENDER-CAPABILITY-P5`

`spreadsheet_backend_capability_profiles.adoc` with profile

== Residual Risks

These remain acceptable if documented:

* *Two-step rollout drift.* If the four block-A FTRs land but the
  three grouping FTRs slip, the user guide will reference helper
  grouping behavior that is not implemented. Document that grouping
  docs land *only* in `FTR-FK-HELPER-GROUPING-DOCS-DEMO-P5`, not
  earlier.
* *Policy shape may need revision for composite keys later.* The
  model FTR already notes this; acceptable since composite keys
  are an explicit non-goal here.
* *Reimport from externally edited workbooks.* If the prefix
  fallback in `remove_fk_helpers` is removed entirely, an XLSX
  that loses its `_meta.derived` payload (corruption, foreign
  editor, sidecar removal) cannot clean helpers anymore. The
  primitives FTR should choose explicitly: error out, or keep a
  guarded read-path fallback. Either is acceptable; the choice is
  not made today.
* *ODS partial capability.* If ODS does not support outline groups
  in the initial cut, demo screenshots taken in LibreOffice Calc
  may visibly differ from XLSX screenshots. The grouping-docs FTR
  must mention this honestly.
* *Phase 4 churn.* `FTR-DERIVED-PROVENANCE-LIFECYCLE-P4A` and
  `FTR-META-SCHEMA-VALIDATION-P5` will revisit the same `derived`
  and `helper_policies` shapes. Some Phase 3a code may need a
  small follow-up. Acceptable if scoped.
