===== FTR-FUNCTIONAL-MODEL-SUPPORTING-CAPABILITY-FAMILIES-P5 -- Triage Functional Model coverage for supporting capabilities

*Status:* New

*Type:* Functional Model documentation / triage.

*Purpose:*::
Catch up or explicitly defer Functional Model coverage for implemented
supporting capability families that are not primary reversible transformations.

*Context:*::
Derived from the June 2026 transformation/capability family inventory:
`docs/cold_storage/working_context/transformation_family_inventory_2026_06/transformation_family_inventory.adoc`.
The inventory identifies partial coverage for `validation` /
`graph_validation`, `resource_overrides`, and `writer` / `artifact_manifest`.
It also flags infrastructure-oriented or classification-ambiguous families such
as `meta_bootstrap` / `yaml_overrides`, `pipeline_extension` /
`identity_plugin`, and `header_shape`.

This FTR keeps the catch-up work explicit without forcing every implemented
capability into the Functional Model.

*Scope:*::
* Review whether validation and graph validation need direct Functional Model
  prose beyond existing workflow/problem-space references.
* Review whether resource override normalization should be modeled inside the
  localized-resource workbook roundtrip use case or left as technical support
  detail.
* Review whether writer/output capabilities and generated artifact manifests
  need Functional Model coverage as part of repository-output workflows.
* Record a bounded decision for infrastructure-oriented families:
  `meta_bootstrap` / `yaml_overrides`, `pipeline_extension` / `identity_plugin`,
  and `header_shape`.
* If a family is better treated in the Technical Model or user guide, record
  that routing instead of adding Functional Model content.

*Non-Goals:*::
* No runtime implementation.
* No validation, writer, resource, or pipeline infrastructure code changes.
* No new FTRs for each supporting capability unless this triage identifies a
  bounded need.
* No Project Memory entities.
* No copy of the inventory table, coverage matrix, or implementation evidence.

*Acceptance Criteria:*::
* Each supporting family named in Scope has a Functional Model routing decision:
  promote now, defer, or route to Technical Model/user-guide documentation.
* Any promoted material is concise and use-case-oriented.
* Infrastructure-oriented families are not forced into Functional Model coverage
  merely because they are implemented.
* The June 2026 inventory is cited as provenance without duplicating inventory
  contents.
* The FTR remains documentation-only; no runtime implementation is performed
  unless separately approved.

*Relationship to the inventory and Functional Model:*::
The inventory is the canonical June 2026 evidence source. This FTR is the
bounded backlog item that decides which supporting capability families deserve
Functional Model promotion and which should remain technical/infrastructure
documentation topics.
