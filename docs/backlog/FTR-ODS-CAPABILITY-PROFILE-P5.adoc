===== FTR-ODS-CAPABILITY-PROFILE-P5 -- Tighten the ODS capability profile

*Status:* New

*Purpose:*::
Tighten the ODS capability profile so each support claim is explicit and each
accepted gap is linked to evidence. Documentation / small data constant only.

*Context:*::
The io_backends / carrier-boundary review found parity gaps (freeze readback,
protection) scattered across a parity test and ODS renderer no-op branches, with
no central record. A small doc plus an optional tiny constant the parity test can
import is the right altitude; an enforcement framework is not.

*Dependencies:*::
* None. Precedes the ODS-internals stabilization phase.

*Scope:*::
* Provide a capability table classifying each ODS family / feature as one of:
  direct support, carrier-backed support, hint-only parse support, accepted
  initial gap, or not applicable.
* Link each accepted gap to a parity test or a documented non-goal.
* Optionally single-source the gap list as a small constant the parity test
  imports.

*Non-Goals:*::
* No runtime implementation.
* No capability-profile enforcement engine.

*Acceptance:*::
* The profile names each ODS capability with one of the five support classes.
* Each accepted gap is visible from the backend comparison surface and links to a
  parity test or documented non-goal.
* The ODS freeze-readback status is named explicitly.
* No runtime code is changed.
