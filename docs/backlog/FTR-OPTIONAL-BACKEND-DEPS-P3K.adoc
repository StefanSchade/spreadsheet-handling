===== FTR-OPTIONAL-BACKEND-DEPS-P3K — Decide on optional vs. mandatory backend dependencies before stable release

*Context:* Phase 3j moved `odfpy` from `extras_require["ods"]` into the base `dependencies` list in `pyproject.toml`. This was a deliberate choice: the ODS adapter is a first-class backend with factory and router registration, so it should be available in every installation.

*Problem statement:* With `openpyxl`, `XlsxWriter`, and `odfpy` all as mandatory dependencies, the install footprint grows. For an internal beta this is fine, but before a stable 1.0 release the project should make an explicit decision about whether spreadsheet backend libraries remain mandatory or move behind extras with lazy loading.

*Goals:*

. Record an explicit decision on the dependency strategy for spreadsheet backend libraries.
. If the decision favors optional extras, implement lazy backend loading so that missing dependencies produce clear error messages rather than import crashes.
. If the decision favors mandatory dependencies, document the rationale and ensure the install footprint is acceptable.

*Acceptance criteria:*

. The `pyproject.toml` dependency layout reflects the recorded decision.
. If backends become optional: `make_backend("ods")` or `make_backend("xlsx")` raises a clear, user-friendly error when the corresponding library is not installed.
. If backends remain mandatory: a brief rationale is documented in the project README or architecture docs.
. The factory and router work correctly under the chosen strategy.

*Non-goals:*

- No new backend support.
- No changes to the spreadsheet contract or rendering pipeline.
- No removal of any backend's functionality.

*Why now:* This decision should be made before a stable release but does not need to block current beta work. It is best revisited after Phase 4 features have settled and before the final 1.0 polish phase.
