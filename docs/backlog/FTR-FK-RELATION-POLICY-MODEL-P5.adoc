===== FTR-FK-RELATION-POLICY-MODEL-P5 -- Explicit FK relation policy model

*Status:* New

*Purpose:*::
Define an explicit metadata model for foreign-key relations used by FK helper
steps, so primitive FK-helper transformations no longer need to infer relation
semantics from column names or table names.

*Context:*::
The current FK-helper path still carries an older convention-based model:
primary keys are commonly inferred from configured/default ID column names, and
foreign keys are commonly detected from naming patterns such as an ID prefix or
target-table suffix.
+
That was acceptable during early bootstrap, but it conflicts with the current
architecture direction:
+
* heuristics may propose configuration;
* primitive transformations should execute resolved configuration;
* hidden naming rules must not become implicit domain semantics.
+
The project already distinguishes configuration steps from primitive steps in
the pipeline registry model. This FTR defines the FK relation policy shape that
future FK configuration and primitive steps will use.

*Dependencies:*::

* Existing FK-helper behavior and tests.
* Existing `_meta.helper_policies` metadata root.
* Existing pipeline step categories: `configuration` and `primitive`.
* Existing derived-helper provenance channel under `_meta.derived.sheets`.

*Scope:*::

* Define the canonical FK relation policy shape stored in `_meta`.
* Represent at least:
  ** source frame,
  ** source FK column,
  ** target frame,
  ** target key column,
  ** target display/helper fields,
  ** materialized helper column names or naming policy,
  ** provenance fields needed for reimport cleanup and later consistency checks.
* Clarify whether the policy lives under `_meta.helper_policies.fk` or under a
  new adjacent root, and document the decision.
* Update the meta registry and pipeline step registry documentation so the
  policy has one clear authority.
* Mark convention-based FK detection as legacy current-state behavior to be
  removed by the follow-up primitive refactor.
* Define the failure principle for future primitive FK-helper execution:
  missing FK relation policy is an error unless a configuration/inference step
  wrote one earlier in the pipeline.

*Non-Goals:*::

* No implementation of `infer_fk_relations`.
* No rewrite of `add_fk_helpers`, `remove_fk_helpers`, `validate_fk_helpers`, or
  `reorder_fk_helpers`.
* No demo pipeline migration.
* No backward-compatibility layer for public users.
* No runtime schema validation framework.
* No generic relation model for all possible joins; this FTR only covers FK
  helper semantics.

*Policy sketch:*::

[source,yaml]
----
_meta:
  helper_policies:
    fk:
      relations:
        - source_frame: products
          source_column: product_manager_id
          target_frame: product_managers
          target_key: id
          helper_fields:
            - name
          helper_columns:
            - column: _product_manager_name
              target_field: name
          produced_by:
            step: infer_fk_relations
            mode: naming_convention
----

The exact shape may change during implementation, but the final policy must be
explicit enough that helper materialization does not need to derive relation
identity from visible column names.

*Acceptance:*::

* The FK relation policy shape is documented in the technical model.
* The metadata authority is clear: the policy is a canonical configuration
  claim, not derived helper provenance.
* The relation policy contains enough information for:
  ** helper materialization,
  ** helper validation,
  ** helper reordering,
  ** provenance writing,
  ** later reimport cleanup.
* The policy distinguishes relation identity from materialized helper columns.
* The policy does not rely on visible workbook labels as identity.
* Existing convention-based FK detection is documented as current-state behavior
  to be removed by follow-up FTRs.
* No primitive FK-helper step is changed in this FTR except for documentation
  comments or registry descriptions required to explain the transition.

*Validation:*::

* Render modified AsciiDoc documents.
* Validate modified JSON/YAML registry artifacts, if any.
* Run targeted tests only if registry guards are touched.
* Confirm no runtime behavior changes are introduced by this FTR.

*Residual risks:*::

* The first implementation FTR may discover that the policy needs a slightly
  different shape for MultiIndex, workbook-view, or readback use cases.
* Existing tests and demo flows will remain convention-backed until the
  follow-up primitive refactor lands.
