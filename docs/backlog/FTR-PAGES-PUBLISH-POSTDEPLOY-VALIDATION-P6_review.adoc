== Repository-role discipline

* The helper lives in *core* `tools/`. That matches the canonical
  split (`tools/` is for copyable cross-repo helpers, `scripts/` for
  core-local automation) and keeps release-governance documentation
  with the release-governance owner.
* The Pages repository is untouched. The helper inspects a Pages
  working tree but does not mutate it, does not require a particular
  pages workflow file, and tolerates the documented absence of any
  pages CI surface.
* The demo repository is untouched. The helper happens to validate
  demo subtrees when `--demo-tag` is supplied, but the demo repo
  owns the publishing of those subtrees, not the validation contract.

== Residual risks

* *Structural ≠ semantic.* The helper certifies that the expected
  HTML files exist and are nonzero. It does not verify the rendered
  content is correct, that internal links resolve, that the AsciiDoc
  attributes baked in match the intended tag, or that GitHub Pages
  is actually serving the files at the public URL. Operators should
  still do a quick manual eyeball of `https://stefanschade.github.io/spreadsheet-handling-pages/`
  for the first release after a publishing change. The runbook makes
  this explicit ("does not validate page content or external
  accessibility").
* *Pull-before-check ordering.* The helper is intentionally
  fetch-free (matches the `release_check.sh` read-only contract). If
  the operator forgets `git -C ../pages pull --ff-only` first, the
  check reports the previous publish state -- which surfaces as a
  *missing* classification for a tag that should be present. The
  runbook step ordering and the exit-2 cause hint both point at this.
* *`latest/` semantics are opaque to the helper.* The check confirms
  that `latest/core/*` exists and is nonzero; it does not confirm
  that those files correspond to the tag passed via `--core-tag`.
  Detecting that mismatch would require parsing the published HTML
  for a version string, which is meaningfully more invasive and
  belongs in a follow-up FTR if the cross-tag-leak risk ever
  materializes. The PEP 440 latest-tag guard inside the publisher
  workflows already prevents older tags from clobbering `latest/`,
  so this gap is acceptable for the current slice.
* *Helper is bash-only.* Matches the existing `release_check.sh`
  pattern and the project's POSIX/Linux automation target. Operators
  on Windows would need to run via WSL or Git Bash (same constraint
  as `release-check`).
* *No CI invocation.* The FTR asks for the validation to be useful
  "both locally, after pulling the Pages repo; and potentially from
  CI/release troubleshooting later". The helper meets the local-use
  case directly. Wiring it into the publisher workflows is
  intentionally deferred -- that would add a network step to
  release-time CI for a check that the operator can run more reliably
  from a local pulled checkout. A follow-up FTR can add a CI step if
  the operator-side check proves insufficient.
