== 3. Findings ordered by severity

=== 3.1 Minor — legacy Workbook View knob fields repersist as inert sediment

Legacy `frame_lifecycle` is stripped at the persistence boundary and can
never repersist. Legacy knob fields *inside* `workbook_view` (`mode`,
`drop_redundant_data`, `unknown_frame_policy`, `omit_roles`,
`include_debug_frames`, `include_omit_by_default`, `include_roles`) are
ignored by every consumer but are **not** stripped: an empirical probe
confirmed they roundtrip through XLSX and back into a `json_dir`
`_meta.yaml` unchanged.

This is behaviorally harmless (the negative-guard test
`test_legacy_ontology_and_view_knobs_do_not_infer_selection` protects
selection, and no consumer reads the keys) and is consistent with the
tolerant-reader posture and the deliberately deferred metadata-pileup work
(appraisal Section 12). It is recorded here so the pileup/inventory FTR
inherits it explicitly: either add a `workbook_view` inner-key projection
rule at the boundary or document the keys as permanently ignored sediment.

Classification: accepted limitation, assigned to the metadata-pileup /
`FTR-META-LIFECYCLE-INVENTORY-P5` follow-up. Not a defect of this slice.

=== 3.2 Minor — stale AI-guidance line (corrected during this review)

`docs/ai_info/agent_rules.adoc` still instructed agents that pruning of
`frame_lifecycle.{derived_from, produced_by}` was "deliberately deferred to
their owning FTRs" — actively wrong after this slice strips the whole
family at the boundary. Corrected in this review commit (tiny, doc-only,
agent-facing guidance).

=== 3.3 Nit — stale test name

`tests/unit/domain/transformations/test_tabular_views.py::test_pivot_frame_marks_output_as_readonly_projection_by_default`
now (correctly) asserts that `pivot_frame` writes *no* `_meta` at all; the
name still describes the removed lifecycle behavior. Rename in passing
during the next slice. Behavior asserted is correct; name only.

== 5. Runtime-contract assessment

The frame-selection contract is now exactly:

[source,text]
----
no workbook_view                      -> all remaining logical frames
workbook_view without "sheets"        -> all remaining logical frames
workbook_view.sheets present          -> authoritative explicit selection,
                                         naming, ordering; missing frame,
                                         reserved name, duplicate sheet fail
----

verified in `rendering/frame_selection.py` (the only lifecycle branch is
deleted; `explicit_sheets is None -> dict(frames)`) and by
`test_render_plan_selects_every_remaining_frame_without_workbook_view` and
`test_legacy_ontology_and_view_knobs_do_not_infer_selection`.

Empirical end-to-end probe (this review): a `json_dir` input whose sidecar
carried a full legacy payload — `frame_lifecycle` with `role: system`,
`render: never`, `superseded_by`, plus `workbook_view` with
`mode: editable`, `drop_redundant_data: true`, `unknown_frame_policy: fail`,
`omit_roles: [system]` — rendered **all** frames to XLSX, and the reread
workbook meta contained **no** `frame_lifecycle` (stripped before the blob
was written). The formerly `fail`-valued `unknown_frame_policy` raised
nothing. No renderer pass, adapter, schema-maintenance path, or generic
scanner interprets lifecycle, role, provenance, or canonicality; the
remaining "role" vocabulary in the repository is the separate column-role
concept (`project_by_role`, `domain/column_roles`), which is a distinct,
legitimate contract.

Legacy-payload strip consequences (question 6): the top-level key
`frame_lifecycle` is now reserved-and-silently-dropped at persistence. A
hypothetical caller using that name for their own metadata would lose it
without an error. This is documented in the boundary module and acceptable:
the name was this project's own family, and silent stripping (rather than
loud failure) is the same posture already applied to `derived`. No naming
collision exists in the repository or consumers.

== 8. Registry and documentation assessment

* `registries/meta_registry.yaml`: `frame_lifecycle` entry removed;
  `workbook_view` / `sheet_mappings` entries frame-only; `pipeline_cleanup`
  reclassified `meta_canonical / canonical_meta` with `lifecycle: transient`
  (question 11, below).
* `registries/pipeline_step_registry.json`: lifecycle parameters and
  `frame_lifecycle` meta refs removed from all step entries; the plugin
  `lifecycle_responsibility` field and its guard enum removed;
  `meta_reference_surfaces.json` `MRS-FRAME-LIFECYCLE` removed and the
  gap-findings wording updated to drop the family from the exempt list.
* Question 11 — is `meta_canonical / canonical_meta` for `pipeline_cleanup`
  safe despite being semantically imperfect? *Yes.* The registry declares
  `runtime_binding: forbidden`; no `src` module reads it; its only
  consumers are pinning tests (`test_meta_registry_hardening` profile map,
  step-registry cross-reference checks). The classification schema forces
  the pair (non-canonical entries must be `meta_rendering`, which was the
  previously recorded and semantically *worse* choice flagged by Codex);
  `lifecycle: transient` records the true behavior. No runtime or
  architecture decision follows from the label. Revisit when
  `FTR-META-FACET-CLASSIFICATION-P5` / the lifecycle inventory introduce a
  proper transient-command classification.
* Documentation: the Technical Model `pipeline_semantics.adoc` lifecycle/
  render sections are rewritten (zero residual ontology vocabulary); the
  Semantic Model and User Guide now state the refined conflict rule
  (explicit drop + explicit mapping -> conflict; keep-mode implied removal +
  stale mapping -> allowed, later spreadsheet write fails loudly);
  functional-model concept pages and the glossary were realigned; release
  notes carry the removal bullet. The only stale live-doc line found was
  `agent_rules.adoc` (finding 3.2, corrected).
