===== FTR-NATIVE-CELL-VALUE-ROUNDTRIP-P5 -- Preserve native scalar values across spreadsheet roundtrips

*Status:* New

*Priority:* P5

*Purpose:*::
Preserve basic spreadsheet value behavior across structured-data and
spreadsheet roundtrips without introducing a strict application data model.

Values that are already known to be text, boolean, or numeric should retain
that scalar identity instead of being unconditionally converted to text.

*Context:*::
The current visible-cell contract deliberately uses a string substrate. This
keeps parsing and roundtrip behavior simple, but it also means that native
spreadsheet values may lose functional behavior.

For example:

[source,text]
----
JSON false
  -> spreadsheet text "False"
  -> JSON string "False"
----

or:

[source,text]
----
JSON number 12.5
  -> spreadsheet text "12.5"
  -> formulas, sorting, and filtering may treat the value as text
----

This FTR does not propose a general type system. It records the smaller product
requirement that spreadsheets should remain functionally usable after a
roundtrip.

*Problem Statement:*::
When booleans or numbers become text during a roundtrip, spreadsheet behavior
can change:

* formulas may no longer calculate as expected;
* numeric sorting may become lexical sorting;
* numeric and boolean filters may behave differently;
* validations and spreadsheet functions may no longer recognize the value;
* subsequent exports reproduce text cells instead of native cells.

Preventing this loss does not require strict column schemas, homogeneous
columns, typed value objects, or a new canonical data model.

*Design Principle:*::
Prefer the least invasive mechanism that preserves spreadsheet-relevant scalar
behavior.

Preserve known value identity. Do not introduce new value identity merely from
textual appearance.

In particular:

* a known boolean should remain a boolean;
* a known numeric value should remain numeric;
* known text should remain text;
* text such as `"00123"` must not automatically become a number;
* metadata or explicit type declarations should only be introduced where the
  participating carriers cannot otherwise preserve the distinction.

*Initial Scope:*::

* text values;
* boolean values;
* numeric values;
* JSON -> spreadsheet -> JSON roundtrips;
* spreadsheet-authored values -> JSON -> spreadsheet roundtrips;
* XLSX and ODS behavior.

The initial capability only needs to preserve the broad category `numeric`.
It does not need to establish a strict distinction between integer,
floating-point, and decimal domain types.

*Expected Behavior:*::

[source,text]
----
JSON false
  -> native spreadsheet boolean
  -> JSON false
----

[source,text]
----
JSON number 12.5
  -> native spreadsheet number
  -> JSON number 12.5
----

[source,text]
----
JSON string "00123"
  -> spreadsheet text "00123"
  -> JSON string "00123"
----

A value entered directly as a native boolean or numeric spreadsheet cell should
be imported as the corresponding JSON scalar where the carrier exposes that
identity unambiguously.

*Implementation Direction:*::
The first implementation should reuse native type information already available
from JSON, frame values, and spreadsheet carriers.

The implementation should not begin by adding:

* mandatory column schemas;
* structured `{value, type}` envelopes;
* global lexical prefixes;
* broad type-inference rules;
* a general-purpose coercion framework.

Such mechanisms remain possible later if a concrete value category cannot be
preserved reliably through the simpler path.

*Presentation Boundary:*::
Value identity and visible formatting are separate concerns.

For example:

[source,text]
----
value: 12.3
display: 12.30
----

The visible number of decimal places does not normally change the canonical
numeric value. Number formats, date display patterns, percentage formatting,
and similar concerns belong primarily to presentation metadata.

Loss of presentation formatting may reduce workbook fidelity, but it must not
silently change a numeric value into text.

*Deferred Concerns:*::
Dates, times, and date-time values are related but require an additional
contract because JSON has no native date or time scalar.

Formula preservation is also related to spreadsheet functionality but requires
a separate contract covering at least:

* formula expression;
* calculated or cached value;
* XLSX and ODS formula syntax;
* behavior when no calculation engine is available.

These concerns should be handled in reviewed follow-up slices rather than
expanding the initial scalar-preservation implementation.

*Non-Goals:*::

* No strict type system for canonical data.
* No requirement for homogeneous column types.
* No schema language for JSON documents.
* No integer-versus-float-versus-decimal domain model in the initial slice.
* No implicit conversion of text based only on its appearance.
* No locale-dependent parsing of ordinary text cells.
* No domain-specific units or value objects.
* No formula preservation in the initial slice.
* No date/time representation decision in the initial slice.
* No requirement that presentation formats become part of canonical values.

*Open Questions:*::

* Which current parser or frame-normalization boundary collapses native scalar
  values to strings?
* Can native JSON, pandas, XLSX, and ODS scalar types be preserved directly
  without additional metadata?
* Which distinctions are exposed consistently by both spreadsheet backends?
* How should blank cells and null values interact with native booleans and
  numbers?
* Are mixed text and numeric values in one column preserved without coercing
  the whole column?
* Which backend differences are acceptable for the initial implementation?

*Acceptance:*::

* JSON booleans survive XLSX and ODS roundtrips as JSON booleans.
* JSON numeric values survive XLSX and ODS roundtrips as JSON numeric values.
* Text values survive roundtrips as text.
* Numeric-looking text such as `"00123"` is not converted automatically.
* Native boolean and numeric cells authored in XLSX or ODS import as the
  corresponding JSON scalar when the carrier type is unambiguous.
* Writing the imported JSON recreates equivalent native spreadsheet cells.
* The implementation does not require a strict canonical schema.
* The implementation does not introduce structured typed-value envelopes
  without a separately reviewed need.
* Roundtrip tests cover boolean, numeric, text, blank, and mixed-value cases for
  both XLSX and ODS.
* Existing string-substrate behavior is updated explicitly where this FTR
  changes it rather than being contradicted implicitly.
