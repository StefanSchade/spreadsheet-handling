= FTR-WORKBOOK-VIEW-WORLDBUILDING-ADOPTION-P5 -- Adopt explicit sheet names and ordering in the worldbuilding consumer

*Status:* New
*Priority:* P5
*Area:* Consumer ergonomics / workbook view / roundtrip
*Kind:* Adoption and validation slice

== Purpose

Apply the existing workbook-view capability to the worldbuilding consumer so
visible spreadsheet sheet names may be shorter and more user-oriented than the
canonical JSON frame names, while preserving explicit sheet ordering and safe
roundtrip reconstruction.

The worldbuilding repository is the first realistic proof point. The public
demo follows only after the behavior and configuration shape are understood.

== Context

Canonical frame names are optimized for stable data modeling and configuration.
Visible spreadsheet sheet names are a user-interface concern and may benefit
from shorter labels and an intentional order.

The core already provides:

* `configure_workbook_view`;
* explicit sheet-to-frame mappings;
* workbook-view sheet ordering;
* `apply_workbook_view_sheet_mappings` for reimport;
* XLSX and ODS metadata carriers.

This slice validates that these capabilities work together in a real consumer.

== Scope

* Select a representative set of worldbuilding frames.
* Configure explicit visible sheet names.
* Configure a deliberate sheet order that differs from lexical frame ordering.
* Export the consumer workbook as XLSX and ODS.
* Reimport both carriers through the maintained consumer workflow.
* Verify that visible sheet names resolve back to canonical frame names.
* Verify that canonical JSON frame names remain unchanged.
* Record any configuration, documentation, backend, or roundtrip gaps found.

== Non-Goals

* No new workbook-view model unless the existing contract proves insufficient.
* No heuristic inference of frame names from visible sheet names.
* No renaming of canonical JSON frame keys.
* No demo-repository update in the first slice.
* No addition of new presentation-meta families.
* No unrelated worldbuilding pipeline cleanup.

== Acceptance

* The exported workbook uses the configured visible sheet names.
* Sheet order follows explicit configuration rather than lexical frame order.
* Canonical JSON and internal frame names remain unchanged.
* XLSX reimport restores the declared logical frame names.
* ODS reimport restores the declared logical frame names or exposes a narrowly
  documented backend gap.
* Reimport does not depend on the current physical order of sheets.
* Existing worldbuilding roundtrip checks remain green.
* Any discovered defect is tracked as a separate `BUG-*` item rather than being
  silently absorbed into this adoption FTR.
* The resulting configuration and lessons are suitable as input for a later
  demo-oriented documentation slice.

== Follow-up

After this FTR closes:

. decide whether the current workbook-view API is sufficiently ergonomic;
. close any surfaced defects;
. specify one new presentation-meta roundtrip family;
. adapt the public demo with simpler, didactic vocabulary.
