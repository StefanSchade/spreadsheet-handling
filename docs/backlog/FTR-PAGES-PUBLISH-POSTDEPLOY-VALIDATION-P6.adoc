===== FTR-PAGES-PUBLISH-POSTDEPLOY-VALIDATION-P6 -- Verify Pages publish output structure after release

*Status:* Proposed
*Priority:* P6
*Area:* Documentation publishing / observability

==== Goal

Improve confidence in documentation publishing by validating the observable
post-deploy state of the Pages repository after release publication.

The `v0.1.0b6` release successfully published documentation and demo slides,
but validation remained largely manual and operator-driven.

The release process should provide a lightweight confirmation that the
expected published structure actually exists after deployment.

==== Motivation

Observed during beta release execution:

* publish jobs completed successfully;
* operators still manually inspected Pages repository contents, generated
  paths, the `latest/` structure, and the versioned release structure;
* it was unclear whether a green publish job alone guaranteed externally
  usable documentation;
* the first successful end-to-end release depended heavily on manual
  inspection and interpretation.

The release technically succeeded, but validation confidence relied on human
exploration rather than explicit confirmation.

==== Scope

This FTR covers:

* lightweight post-publish validation;
* Pages structure verification;
* release-path sanity checks;
* documentation publication observability.

This FTR does not cover:

* visual regression testing;
* browser automation;
* full site integrity validation;
* uptime monitoring;
* CDN correctness testing.

==== Acceptance

* Release documentation defines the expected Pages structure after publish.
* At least one automated or semi-automated validation step verifies expected
  output paths.
* Validation distinguishes missing publish, partial publish, and structurally
  correct publish.
* Operators can quickly confirm that the versioned publish exists, the
  `latest/` aliases exist, and the expected core/demo paths are present.
* Publish failures become easier to localize: CI failure vs deploy failure vs
  Pages structure mismatch.

==== Candidate Validation Targets

Examples:

* `versions/vX.Y.Z/core/`
* `latest/core/`
* `versions/vX.Y.Z-demo/demo/slides/`
* `latest/demo/slides/`

Possible implementation styles:

* shell verification script;
* lightweight CI verification step;
* post-publish Pages-repo inspection helper;
* release checklist helper command.

==== Notes

The `v0.1.0b6` release demonstrated that the underlying publishing
architecture works end-to-end. This FTR focuses on confidence and
observability improvements rather than functional repair.
