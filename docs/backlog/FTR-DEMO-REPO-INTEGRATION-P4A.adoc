===== FTR-DEMO-REPO-INTEGRATION-P4A -- Realign the demo repository after consumer feedback

*Status:* Open
*Priority:* P4A
*Area:* Demo integration / beta readiness

==== Purpose

Bring the demo repository up to the current library capability set after the
prototype consumer pass. That pass (recorded in
`FTR-PROTOTYPE-CONSUMER-INTEGRATION-P4A`) showed the demo repo binds cleanly to
the local core checkout, but one maintained demo flow still uses pre-`0.1.0b6`
step IDs. This FTR turns that into deliberate demo realignment toward the
current canonical interface, driven by the beta tutorial strategy in
`acme_progressive_tutorial_guide.adoc`.

The demo repository should remain cloneable and approachable for a new user,
but it now needs a little more structure than an ad-hoc pipeline collection.
Its detailed coordination lives in the demo repository's `DEMO-*` backlog; this
FTR tracks the main-library beta dependency.

==== Demo journey priorities

* The ACME progressive tutorial is the *primary* beta demo journey. It should
  showcase the current broad capability set strictly according to the ACME
  concept's `Next Beta Demo Slice vs Future Patterns` section
  (`acme_progressive_tutorial_guide.adoc`): beta slice only, modern step
  names, flat axes, closed reimport slice.
* The Worldbuilding / Storyforge trail
  (`by_use_case/worldbuilding_continuity_workbook_roundtrip.adoc`) is an
  *optional, smaller, clearly bounded* second tutorial trail -- add it only if
  cheap and well bounded.
* The Worldbuilding first slice is limited to characters/species FK-helper
  enrichment and a readable workbook projection -- *no matrix, no reimport*
  yet (later slices stay future per that note's `Demo Trail Scope`).
* Worldbuilding must not displace ACME as the main integration path or expand
  beyond its first slice in this FTR.

==== Interface stability stance

* The public user surface is still beta / pre-stable; the demo repo realigns
  to the *current canonical library surface* rather than pinning to old demo
  conventions.
* Pre-`0.1.0b6` demo step names and config patterns may be *changed* rather
  than supported indefinitely. The upcoming UX/user-surface smoothing phase
  may still adjust step/config/CLI/docs conventions and demo structure.
* This FTR must not turn old demo config drift into a core
  backwards-compatibility requirement. There are currently no known external
  users depending on the old demo config surface, and sensible interface
  improvements are not blocked by compatibility concerns at this stage.
* Compatibility aliases in the core library are out of scope unless a separate
  explicit product decision introduces them. Do not propose legacy aliases
  just to keep old demo configs running.

==== Known drift (from the prototype consumer/demo pass)

* Some maintained demo configs (e.g. `demo_product_business_slice.yaml`) still
  reference pre-`0.1.0b6` step IDs `apply_fks` / `reorder_helpers`.
* The current canonical surface uses `add_fk_helpers` /
  `configure_fk_helpers` / `reorder_fk_helpers` where applicable.
* The demo integration migrates the affected configs and tutorial docs to the
  current canonical names.
* This is *expected pre-stable demo realignment*, not a user-breaking
  regression: the capability still exists under the current step names; only
  the demo config text lagged the `0.1.0b6` expansion.

==== Repository and documentation boundaries

See `docs/ai_info/repository_boundaries.adoc`. For this FTR:

* The demo repo contains runnable examples, pipeline configs, fixtures,
  expected outputs / verification, smoke checks, and tutorial *source* tightly
  coupled to demo data.
* Generated RevealJS / static HTML must *not* be committed to the demo `main`
  branch.
* `spreadsheet-handling-pages` is future / planned only; first beta publishing
  may still be manual.
* Do not create the pages repo or any publishing automation in this FTR.

==== Future-pattern boundaries

The demo integration must not implement or imply support for patterns the ACME
concept marks as future/deferred. It may mention them only as future/deferred,
exactly as framed there:

* ACME conditional detail clusters / dynamic workbook sections (boundary A)
* rich structured payload matrices (boundary B)
* temporal / planning projections (boundary C)
* grouped / structured XRef axes (boundary D)
* conflict / precedence handling between overlapping editable views
  (boundary E; `FTR-WORKBOOK-VIEW-PAYLOAD-CONFLICT-PRECEDENCE-P6`)
* nested object / list mapping (boundary F)

==== Scope

* Use the demo repository's `docs/internal/backlog/DEMO-*` items as the
  working plan.
* Realign the maintained ACME journey to the current canonical
  pipeline/config surface (modern step names) on the supported run path.
* Optionally add the bounded Worldbuilding first slice as a secondary trail.
* Add or adjust automated demo smoke/integration checks for the curated
  journeys against the beta-candidate local library checkout.
* Keep public documentation oriented around user journeys rather than internal
  feature names.
* Tag the aligned demo state for the beta release once the main library beta
  is ready.

==== Non-goals

* Do not move the main FTR backlog into the demo repository.
* Do not commit generated walkthrough HTML to the demo `main` branch.
* Do not showcase every internal transform separately if a smaller journey set
  tells the story better.
* Do not add core compatibility aliases for old demo step/config names.
* Do not implement or imply any ACME future/deferred pattern (A-F above).
* Do not create the pages repo or publishing automation here.

==== Acceptance

* The ACME maintained journey maps to pipeline/config, fixture data,
  verification, and user-facing documentation, following the ACME concept's
  beta slice.
* Maintained demo configs use the current canonical step names / config
  surface (no `apply_fks` / `reorder_helpers`; no core compatibility aliases).
* Demo smoke / integration checks run against the beta-candidate local library
  checkout.
* The demo repository has a clear public entry point that guides users through
  the curated journeys rather than a raw pile of configs.
* If present, the Worldbuilding trail is clearly secondary and bounded to its
  first slice (characters/species FK-helper / readable workbook; no
  matrix/reimport).
* The demo repository has `DEMO-*` backlog items for showcase structure,
  integration harness, walkthroughs, versioned Pages publishing, and
  capability showcases, with references back to the relevant main-library FTRs
  where useful.
* The demo release/tag plan is clear enough for the next beta.
* No generated HTML artifacts are committed to the demo `main` branch.
