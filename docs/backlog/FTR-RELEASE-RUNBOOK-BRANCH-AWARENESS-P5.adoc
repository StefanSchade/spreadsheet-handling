===== FTR-RELEASE-RUNBOOK-BRANCH-AWARENESS-P5 -- Make release tag branch topology explicit and verifiable

*Status:* Proposed
*Priority:* P5
*Area:* Release process / runbook

==== Goal

Reduce release-process fragility caused by tagging and publishing from
non-`main` branches without explicit awareness.

The `v0.1.0b6` release succeeded technically, but both core and demo releases
were initially performed from `dev` branches. This did not corrupt published
artifacts -- the tags pointed to the intended commits -- but it caused
avoidable confusion, CI noise, and post-release branch hygiene repair.

The release process should make the branch state explicit and fail early for
unintended release topology.

==== Motivation

Observed during the `v0.1.0b6` release:

* release tags created from `dev`;
* release commits not initially reachable from `main`;
* ambiguity whether release artifacts represented canonical history;
* uncertainty whether CI reruns used repaired workflow definitions;
* manual post-release fast-forward repair of `main`.

The actual release artifacts were valid, but operator confidence was
unnecessarily reduced.

==== Scope

This FTR covers:

* release runbook hardening;
* branch-awareness checks;
* optional automated guardrails;
* release topology visibility.

This FTR does not cover:

* semantic-version policy;
* multi-stage staged rollout;
* automated orchestration between repositories.

==== Acceptance

* The release runbook explicitly requires verification that release tags are
  reachable from `main`.
* A documented pre-tag checklist exists for:
** current branch,
** divergence from `main`,
** expected tag target commit.
* A lightweight automated check exists before tagging or publishing.
* Release troubleshooting documentation covers:
** how to repair accidental release-from-`dev` situations,
** when retagging is safe vs unsafe.
* Release operators can determine the canonical release topology of a tag
  without sifting through workflow logs.

==== Candidate Enhancements

Possible implementations:

* `make release-check`;
* a git pre-push hook;
* CI validation for release tags;
* a script verifying clean working tree, branch alignment, and tag ancestry.
