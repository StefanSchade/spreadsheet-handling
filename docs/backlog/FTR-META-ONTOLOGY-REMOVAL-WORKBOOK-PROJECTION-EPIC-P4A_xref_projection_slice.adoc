= XRef Projection-Family Slice: FTR-META-ONTOLOGY-REMOVAL-WORKBOOK-PROJECTION-EPIC-P4A
:revdate: 2026-07-19
:status: Implemented
:type: Projection-family slice record (xref_crosstable)

== Purpose

First projection-family slice after the accepted ontology removal: reduce
`xref_crosstable` metadata to the minimal feature-local inverse contract,
harden `drop_source` with checkable lossless conditions, make intent
discovery fail on ambiguity, and make schema maintenance handle only real
XRef references. No generic lifecycle, canonicality, provenance, or render
concept was reopened. Workbook View required no change: it carries no XRef
semantics (verified; the only XRef/View touchpoint is the generic
frame-to-sheet mapping that the column-role resolver reads for helper
columns, which is view-owned, not XRef-owned).

== Minimal retained intent (field inventory result)

Every retained field has a concrete producer, consumer, and operational
decision; everything else was removed from the producers.

[cols="2,3,4", options="header"]
|===
| Field | Status | Concrete consumer / decision

| `relation`, `matrix`
| retained (persisted intent)
| Intent discovery by frame identity (`_xref_config` fallback when the
  transform id differs across runs); column-role resolution
  (`column_roles.resolver` reads `matrix` + `row_keys` for
  `row_identity`); `sparse_defaults` matrix lookup. May legitimately
  reference frames removed by final domain cleanup: a reverse pipeline
  needs exactly this intent to recreate the absent side.

| `row_keys`
| retained (persisted intent)
| Column-role resolution (`row_identity` for `project_by_role` and
  role-targeted validations); schema-maintenance reference detection.

| `dense_axes.rows_from` / `columns_from`
| retained (persisted intent)
| Dense-axis re-resolution on a later `contract_xref`
  (`_dense_axes_from_config_or_meta`), including reimport re-export runs.

| `column_keys` (entry top level)
| retained (run-local Resolution; stripped at the persistence boundary,
  unchanged)
| Same-run matrix column reuse in `contract_xref`
  (`_column_keys_from_meta_or_relation`) and `sparse_defaults` target
  columns (`_xref_column_keys_for_frame`).

| `dense_axes.resolved`
| retained (run-local Resolution; stripped at the boundary, unchanged)
| Axis-snapshot fallback when a later same-run `contract_xref` operates on
  a frame subset that no longer contains the axis frames
  (`test_contract_xref_reuses_stored_dense_axes_without_source_frames`).
  An initial removal attempt during this slice broke that shipped
  dense-axes capability and was reverted — the snapshot is
  Resolution-with-a-consumer, not sediment.

| `operation`
| *removed* from producers
| No reader anywhere; descriptive only.

| `column_key`, `value`
| *removed* from producers
| No reader anywhere. Inverse operations are parameterized by explicit
  step configuration, not by the intent entry; the intent entry is
  discovery/validation state, not a parameter store. (Two user-guide
  claims that these defaulted from meta were incorrect and are fixed.)

| `drop_empty`
| *removed* from producers
| No reader anywhere.
|===

Legacy payloads carrying the removed fields pass through untouched until the
producing transformation replaces the entry; schema maintenance ignores the
legacy fields (they are not references).

== Forward/inverse behavior

* Forward (`contract_xref`) and inverse (`expand_xref`) now write
  symmetrical minimal intent: `{relation, matrix, row_keys, column_keys}`
  plus dense intent (contract resolves it; expand carries the prior entry's
  dense intent forward for a later re-contract).
* Intent discovery by frame identity fails explicitly when more than one
  entry matches (`Ambiguous xref_crosstable metadata ...`); an exact
  transform id (`name=...`) always wins and disambiguates.
* Roundtrips are covered relation→matrix→relation and matrix→relation→matrix
  (unit and dense variants), plus carrier-level recreation from persisted
  intent on both XLSX and ODS.

== Cleanup interaction (`drop_source`)

`drop_source` remains an explicit caller assertion feeding
`_meta.pipeline_cleanup`, now with the *checkable* value-loss conditions
enforced:

* `contract_xref(drop_source=True)` fails when the matrix columns do not
  cover every relation column key (silent pair loss cannot supersede the
  relation). Dense-axis validation already rejected out-of-axis tuples;
  the guard closes the explicit/meta-derived `column_keys` subset path.
* `expand_xref(drop_source=True)` fails when an explicit `value_columns`
  subset would leave unexpanded matrix columns behind.
* Shape recovery for empty-only columns under `drop_empty=True` remains
  the caller's feature-local judgement (dense column intent or a retained
  matrix provides it); documented in the user guide.

Scoped (non-superseding) projections without `drop_source` are unchanged.

== Workbook View interaction

No Workbook View change was necessary: after the ontology removal it holds
no XRef-specific behavior. Cleanup-versus-view conflict semantics are
unchanged (explicit drop command of a view-mapped frame fails; verified by
the existing cleanup suite).

== Schema maintenance

`xref_crosstable` moved from the generic convention-scanned blocked roots to
a dedicated feature-aware handler (`meta_update._handle_xref_crosstable`)
that blocks rename/drop only for *real* XRef references:

* `row_keys` on the `relation`/`matrix` frames (one intent list serves both
  sides, so a one-sided rename cannot be rewritten safely — blocking is the
  correct policy, matching the established feature-family posture);
* run-local `column_keys` on the `matrix` frame (present in-run or in
  hand-authored meta);
* `dense_axes.rows_from`/`columns_from` `key`/`keys` on the configured axis
  frame.

Fabricated convention-style shapes (`frame` + `value_column` keys) and
legacy descriptive fields no longer block; previously the scanner blocked
the fabricated shapes while real references dangled silently. A malformed
(non-mapping) root fails as `malformed_meta`.

== Compatibility

* Persisted `xref_crosstable` entries lose the descriptive
  `operation`/`column_key`/`value`/`drop_empty` fields (pre-1.0, release-
  noted; no consumer existed, legacy payloads are tolerated).
* Intent discovery ambiguity now fails instead of silently picking the
  first matching entry.
* `drop_source` combinations that silently lost values now fail.
* Schema maintenance blocking behavior for the xref root changed from
  convention-shape detection to real-reference detection.

== Deferred (recorded, out of this slice's scope)

* `compact_multiaxis`-composed flows persist xref entries whose `relation`
  field names the popped `+__compact_multiaxis_*+` temp frame — harmless
  sediment with no consumer, owned by the compact_multiaxis family slice.
* YAML `name:` binds to the pipeline step, not the transform id, so
  YAML-configured entries are always keyed by the output/relation name;
  the frame-identity discovery fallback covers this. A future ergonomics
  decision may expose the transform id in YAML (any projection family).
* Side-effecting writers, metadata pileup, M:N Workbook View, facet
  classification: unchanged deferrals.

== Correction record (after Codex implementation review)

The independent Codex review
(`_xref_projection_slice_implementation_review.adoc`) found two blocking
local defect classes and two completeness gaps in the slice above. All are
corrected; the sections above should be read together with this record.

Adopted human decision::
XRef matrix column identities that participate in the persisted
matrix/relation roundtrip are contractually *unique, non-empty strings*.
No typed header identity encoding is introduced. A shipped-consumer scan
(worldbuilding dino_insel, demo acme_03, core examples) found only string
identities, so no consumer relies on the previously accepted arbitrary
flat scalars.

Corrections::

. `contract_xref(drop_source=True)` now rejects relation fields outside
  `row_keys + column_key + value` — the matrix represents nothing else, so
  scheduling the relation for cleanup would silently lose those fields.
  Scoped projections without `drop_source` are unchanged.
. One column-identity validation contract
  (`_ensure_column_identity_values` / `_ensure_column_identity_list` in
  `primitives.py`) is enforced across relation `column_key` values,
  explicit / metadata-derived / dense-derived `column_keys`, physical
  matrix labels, and expansion selection: string, non-empty, unique where
  the values form an identity list, hashable by construction, no missing
  values, no post-header collisions (string-only excludes `1` vs `"1"`).
  Duplicate physical matrix columns are rejected before expansion. Identity
  validation runs before the duplicate-pair check so unhashable values get
  the contract diagnostic, not a pandas hashtable error.
. Column-role resolution (`_row_keys_from_xref_meta`) now fails on multiple
  entries claiming the same matrix frame, matching the transformation and
  sparse-default ambiguity policy; an explicit `key_columns` override
  bypasses the lookup entirely.
. Schema maintenance additionally blocks matrix-frame renames/drops
  referenced by stored `dense_axes.resolved.column_keys` snapshots (the
  consumed resolved-only hand-authored configuration), and the
  non-mapping-entry tolerance is now characterized by a test.
. The stale `_project_xref_crosstable` boundary docstring no longer calls
  removed legacy fields "Intent fields".
. The exact-id precedence test now uses candidate entries with observably
  different `column_keys` payloads and asserts which entry was consumed;
  matrix-side fallback ambiguity is tested directly.

The cross-carrier recreation claim in this document is accordingly
*qualified*: it holds for the contractual identity vocabulary (unique,
non-empty strings), which is now enforced rather than assumed. Typed
column identities are explicitly unsupported across spreadsheet headers.

Validation: full Core suite 1531 passed / 1 xfailed; ruff clean on `src`
and changed files; architecture/registry suites green; worldbuilding XLSX
and ODS reimports and workbook-view tests pass.

== Second correction record (after independent correction re-review)

The correction re-review
(`_xref_projection_slice_correction_review.adoc`, commit `c2bcc19`)
accepted the first correction's closures but found two remaining blocking
validation-gap classes and two completeness gaps. All are closed; the
identity contract is unchanged (unique, non-empty strings, no typed
encoding).

Finding 4.1 — duplicate and overlapping declarations::
Both operations now reject, before any pandas selection, metadata write,
or cleanup write: duplicate physical column labels on the relation (and
`base_relation`) frame, duplicate configured `row_keys`, and explicit
`value_columns` overlapping `row_keys`. Duplicate physical row-key fields
now fail with the XRef contract diagnostic instead of a raw pandas
`TypeError` from the duplicate-pair check. The inferred expansion rule is
unchanged: without explicit `value_columns`, every non-row-key matrix
column is a value column. Regression tests assert cleanup absence on
every rejection.

Finding 4.2 — every identity source validated before hash/pandas::
Scoped `base_relation[column_key]` identities are validated before
out-of-scope rows are appended (the cleanup-unsafe case). Live dense
column identities are validated before insertion into the duplicate-
detection set. Expansion now validates the *entire physical matrix header
row* (including row-key headers, which are equally spreadsheet headers of
the roundtrip artifact) through the shared string validator before any
pandas hashing, replacing the pandas `duplicated()` pre-check; unhashable
labels therefore get the contract diagnostic.

Finding 4.3 — sparse-default ambiguity::
`sparse_defaults._xref_column_keys_for_frame` now counts every mapping
entry claiming the matrix identity before inspecting payloads: one
complete plus one partial entry, two partial entries, and two complete
entries all fail as ambiguous. The exact preferred config id still wins
first, consistent with the family policy. The documented uniform
ambiguity claim is now accurate.

Finding 4.4 — pipeline registry::
`expand_xref` / `contract_xref` `drop_source` notes now state the full
guard set and the identity contract; `expand_xref.meta_contract.reads`
records the prior-intent read that carries durable dense-axis intent
forward.

Finding 4.5 — test completeness::
Added: duplicate physical relation/base labels, duplicate configured
row keys (both operations), value/row-key overlap, numeric scoped base
identities (fail) and valid string base identities (pass, with cleanup),
unhashable live dense identities, unhashable physical matrix labels,
sparse partial-entry ambiguity, cleanup absence on every rejection,
matrix `column_keys` drop action, `columns_from.key` rename and
`columns_from.keys` drop, blocked-path schema input non-mutation,
successful-path xref input-meta non-mutation, and same-id producer
replacement with untouched legacy-entry preservation. The schema
docstring "exactly three places" phrasing was corrected.

== Third correction record (after independent final review)

The final independent review
(`_xref_projection_slice_final_review.adoc`, commit `5350b22`) accepted the
second correction's closures but found two remaining blocking local
validation defects. Both are closed. The accepted human decision is
unchanged and is now stated as a distinction between two label contracts,
not one: matrix-axis identities remain unique, non-empty strings; physical
frame column labels are a separate, weaker boundary and are *not* globally
required to be strings.

Finding 4.1 — missing-like duplicate physical labels escaped detection and
could schedule cleanup around a `Series` cell::
`_ensure_unique_physical_labels` compared labels with `_values_equal`, for
which `NaN != NaN` and `pd.NA` comparison is ambiguous, so duplicate
missing-like physical relation / `base_relation` labels were not detected: a
selected duplicate `NaN` value field produced a `Series` cell and still
wrote a `drop_frames` cleanup command. The physical-label boundary now
rejects missing-like (`None`/`NaN`/`pd.NA`) and ambiguous-equality labels
(for example a multi-element ndarray) *before* the equality-based duplicate
check, and only then de-duplicates the remaining labels. The check stays
missing-aware and ambiguity-safe without hashing, so valid tuple labels are
not rejected and no generic arbitrary-object equality utility is introduced.

Finding 4.2 — selected expansion identities compared before the shared
validator::
`expand_xref` compared explicit `value_columns` against `row_keys` before
validating them, so `value_columns=[pd.NA]` and `value_columns=[np.array([1,
2])]` raised raw Python/NumPy ambiguity errors instead of the XRef
diagnostic. Explicit `value_columns` are now materialized and validated
through `_ensure_column_identity_list` *before* any overlap membership; the
inferred path is unchanged because its columns are the already-validated
physical matrix header row.

Physical-label / matrix-axis distinction (documentation)::
The functional model
(`functional_model/ch04_use_case_concepts/xref_crosstable.adoc`, "Axis
Identity Versus Physical Column Label"), the technical model
(`technical_model/ch04_concepts/crosstab_binary_xref.adoc`, "Physical
column-label validation boundary"), the User Guide
(`ch03_transformation_catalog/02_bijective_transforms.adoc`), the pipeline
and meta registries, and the release notes now distinguish matrix-axis
identities (string vocabulary) from physical frame column labels
(non-missing, unique, deterministically comparable, scalar-addressable, not
string-only), and record that hierarchical/tuple labels remain open future
work rather than being decided here.

Cleanup safety::
Every rejected physical-label or configured-identity case produces no output
frame, writes no XRef metadata, schedules no cleanup command, and leaves
input frames and input metadata unchanged. Valid full contraction and
expansion still write exactly their intended `drop_frames` cleanup command
when `drop_source=True`.

Tests: duplicate `NaN` / `pd.NA` relation labels, duplicate missing-like
`base_relation` labels, `value_columns=[pd.NA]`,
`value_columns=[np.array([1, 2])]`, cleanup absence and input non-mutation
on every rejection, valid tuple and numeric physical labels accepted,
ambiguous-equality labels rejected, and valid string full projection still
scheduling cleanup (`TestMissingLikePhysicalLabelBoundary`).

Validation: full Core suite 1588 passed / 1 xfailed; ruff clean on `src` and
changed test; architecture (134) and domain-contracts suites green; demo
ACME XRef roundtrip green; worldbuilding XLSX, ODS, and workbook-view
workflows green.

== Fourth correction record (after Review 004 acceptance)

Review 004 (`_xref_projection_slice_review_004_acceptance.adoc`) accepted the
slice but found one remaining local boundary defect: the physical-label
validator did not yet establish its complete stated boundary.

Finding — deterministic unhashable physical labels bypass the boundary::
A physical label such as a Python `list` (`[1, 2]`) is non-missing and its
self-equality converts to `True`, so `_ensure_unique_physical_labels`
accepted it; `_ensure_columns` then reached pandas membership and raised a
raw `TypeError: unhashable type: 'list'` rather than the deterministic XRef
physical-label diagnostic. The equivalent `base_relation` expansion
counterexample raised the same raw error. Both were already cleanup-free and
non-mutating, but violated the requirement that every physical label be
scalar-addressable and validated before pandas membership.

Correction::
The physical-label boundary now requires labels to be *hashable* before
duplicate detection or any pandas operation. The stated boundary is
therefore complete: physical labels must be non-missing, hashable, uniquely
and deterministically comparable, and scalar-addressable — and are still not
globally required to be strings. Valid numeric and hashable tuple labels stay
accepted; lists and other unhashable labels are rejected with the XRef
`ValueError`. The hashability check runs after the ambiguity check, so a
multi-element ndarray (both unhashable and ambiguous) keeps its existing
"ambiguous equality" diagnostic. No generic arbitrary-object equality or
serialization utility was introduced.

Cleanup safety::
The rejected unhashable-label cases produce no output frame, no XRef
metadata, no cleanup command, and no raw `TypeError`, and leave input frames
and input metadata unchanged. Valid full contraction and expansion still
write exactly their intended `drop_frames` cleanup command.

Documentation::
The functional model, technical model, User Guide, pipeline and meta
registries, release notes, and epic status now state the physical-label
contract as non-missing, hashable, uniquely and deterministically comparable,
and scalar-addressable. The matrix-axis string contract, Multiheader
deferral, XRef metadata, and cleanup semantics are unchanged.

Tests: unhashable relation and `base_relation` list labels (contract/expand,
asserting no raw `TypeError`, no output, no `_meta`, no cleanup, input
unchanged), the boundary primitive rejecting a list label, and valid tuple
labels still passing (`TestMissingLikePhysicalLabelBoundary`).

Validation: full Core suite 1591 passed / 1 xfailed; ruff clean on `src` and
changed test; architecture (134) and domain-contracts suites green; demo
ACME XRef roundtrip green; worldbuilding XLSX, ODS, and workbook-view
workflows green.
