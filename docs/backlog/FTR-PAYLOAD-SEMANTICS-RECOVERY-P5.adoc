===== FTR-PAYLOAD-SEMANTICS-RECOVERY-P5 -- Classify payload-semantics-bearing metadata and define the recovery / pinning contract

*Status:* New (exploratory)

*Priority:* P5

*Area:* Metadata model / registry / lifecycle / contract surface

*Type:* Contract clarification (rule-ready, not rule engine)

*Driver review:* `docs/warm_storage/payload_semantics_recovery_review.adoc`

[NOTE]
====
This FTR is exploratory. It does not implement a typing system, a
metadata encoder, a NaN normalization fix, or a new registry root.
It does not preempt the per-attribute slice decisions that the
adoption strategy already requires. Its job is to name an invariant
the project already follows in part, classify the known cases, and
hand the per-attribute work to existing FTRs.
====

==== Purpose

Define an explicit project-level contract for *metadata that carries
payload semantics*, so that:

* the meaning of visible spreadsheet payload cells does not silently
  depend on metadata that may be lost, gitignored, treated as
  generated state, or pruned;
* metadata entries on which payload meaning depends are explicitly
  classified in the registry;
* when payload meaning can be surfaced into canonical data without
  semantic loss, the project prefers that path ("recovery
  preference");
* when it cannot, the entry's canonical-layer status is explicitly
  load-bearing and cannot be silently weakened by an unrelated
  later contract change ("pinning fallback").

The motivating case is the literal `"nan"` collapse documented in
`BUG-UNRESOLVED-HELPER-VALUE-NAN-SEMANTICS-P4A`, but the concept is
general -- the literal `"nan"` is one instance of a pattern that
also covers compact codecs, sparse defaults, matrix cells, sheet
identity, and visible-cell typing.

==== Motivation

The companion review of the collaborative-repository-roundtrip
concept identified that the proposed collaboration-default rule
("treat persisted `_meta` as regenerable working state") is safe
*only* if no visible payload meaning depends solely on metadata.
`BUG-UNRESOLVED-HELPER-VALUE-NAN-SEMANTICS-P4A` is a live
counter-example: under the current contract, "FK source empty / FK
target missing / formula error / in-domain string nan" collapse to
the literal `"nan"` on canonical JSON. The visible cell value alone
cannot recover the meaning.

Several existing canonical-meta entries in
`registries/meta_registry.yaml` already imply this rule in their
precedence notes:

* `cell_codecs`: "domain interpretation remains outside this key";
* `sparse_defaults`: "domain meaning of the default value remains
  outside this key";
* `xref_crosstable`: "cell value semantics remain outside this key";
* `compact_multiaxis`: "dense axis reconstruction remains a future
  explicit contract and must not be inferred";
* `column_widths`: "Width metadata records presentation state
  only; it must not alter table shape or cell semantics".

The registry does not yet have a *field* that names the payload-
semantics dependency. Naming it would:

* give the unresolved-helper BUG a principled rather than ad-hoc
  answer;
* give the visible-cell typing FTR a backbone;
* give the lifecycle inventory FTR a new column;
* prevent the collaborative-roundtrip concept from being applied
  unsafely.

==== Background -- grounded in code and registry

The current canonical-vs-derived model
(`docs/technical_model/ch04_concepts/metadata_canonical_vs_derived.adoc`,
`docs/technical_model/ch05_registries/registry_semantics/02_meta_layer.adoc`)
distinguishes:

* *canonical* meta: authoritative, persistable, spreadsheet-neutral,
  lifecycle-decoupled from any single run;
* *derived* meta: recomputable current-run realization;
* *operational-carrier* meta: carrier-shape of canonical claims.

Within the canonical layer the registry currently encodes a flat
classification: model / presentation / workbook_identity. The
proposed contract refines this with an orthogonal axis: whether the
entry *carries payload meaning*.

==== Scope

In-scope:

* Document the *load-bearing form* of the rule
  (see "Contract statement" below).
* Catalogue the *currently observable cases* by registry entry.
* Propose a *registry attribute*
  (e.g. `payload_semantics_bearing: yes | no | conditional`) with a
  short description field
  (`payload_semantics_recovery: <prose | reference>`).
* State the relationship to:
  ** the canonical-layer definition (this FTR refines, not
     replaces);
  ** the persistence-boundary rule (orthogonal axis);
  ** the collaborative-roundtrip concept (precondition);
  ** workbook-identity exceptions
     (`version`, `author`, `exported_at`);
  ** regenerable presentation defaults (not payload-semantics-
     bearing);
  ** transient derived meta (rule applies only to its canonical
     projection, if any).
* Cross-reference the rule from the relevant concept documents,
  registry vocabulary chapters, and AI-info architecture primer.
* Establish a *case survey* artifact in
  `docs/technical_model/ch04_concepts/` (e.g.
  `payload_semantics_recovery.adoc`) that future per-attribute work
  can reference.

Out of scope:

* Implementation of any encoder, decoder, or typing system.
* Reclassification of individual canonical-meta entries (handed off
  to `FTR-META-LIFECYCLE-INVENTORY-P5`).
* Resolution of `BUG-UNRESOLVED-HELPER-VALUE-NAN-SEMANTICS-P4A`
  (the BUG remains its own owner; this FTR provides the
  framework).
* Visible-cell typing decision
  (`FTR-VISIBLE-CELL-TYPING-STRATEGY-P5` remains its own owner;
  this FTR reframes the options but does not decide them).
* New `_meta` roots.
* Runtime rule evaluation.
* Guards on individual payload-semantics dependencies (per-slice
  work).

==== Contract statement (proposed)

____
*Payload semantics recovery rule (load-bearing form).*

Every metadata entry on which the *meaning of visible payload
cells depends* must be:

. *Classified.* The entry is explicitly tagged in the registry
   as payload-semantics-bearing, with a short statement of which
   payload meaning depends on it.
. *Recovered if possible.* When the dependent meaning can be
   surfaced into canonical data or another non-metadata channel
   without semantic loss, the project prefers that path. This is
   a design *preference*, not a per-slice mandate; individual
   slices retain the right to leave meaning in metadata when
   recovery would be more harmful than helpful.
. *Pinned otherwise.* When recovery is not possible, the entry's
   canonical-layer + persistent + roundtrip-relevant status is
   *load-bearing*: it cannot be silently weakened by a later
   collaboration-default rule, persistence-boundary refinement,
   metadata-lifecycle decision, or registry housekeeping change.
   Any such weakening must explicitly state how the dependent
   payload meaning is preserved or accept the loss as a
   contract change.
____

Explicit exclusions:

* *Workbook-identity meta* (`version`, `author`, `exported_at`):
  not payload-semantics-bearing.
* *Bootstrap-default presentation* (`header_fill_rgb`,
  `helper_fill_rgb`, `freeze_header`, `auto_filter` when
  configuration-default): not payload-semantics-bearing.
* *Operational-carrier shape* (`workbook_meta_blob`, `_hidden`,
  `__autofilter_ref`, `__legend_blocks`): not
  payload-semantics-bearing per se; their underlying canonical
  entries may be.
* *Transient derived meta* (`_meta.derived.sheets.*`,
  `__*`-prefixed render-pass keys): rule applies to the canonical
  projection (e.g. `_meta.sheets.<frame>.helper_columns`), not to
  the runtime channel.
* *Pure presentation* without semantic consequence (e.g.
  `text_orientations`): not payload-semantics-bearing.

Explicit inclusions (per the case survey; non-exhaustive, to be
refined by the lifecycle inventory):

* `cell_codecs`
* `sparse_defaults`
* `xref_crosstable`
* `compact_multiaxis`
* `split_by_discriminator`
* `workbook_view.sheet_mappings`
* `helper_columns` (durable list under `_meta.sheets.<frame>`)
* `helper_prefix`
* `helper_policies` (the v2 `relations` content; v1 residual is
  producer-side only)
* `frame_lifecycle`
* `constraints`
* `legend_blocks`
* `protection` (editability semantics, not lock-styling
  presentation)

The list above is a *survey calibration set*, not a definitive
catalogue.

==== Solution directions (recorded, not chosen)

Direction A -- *Registry attribute only (lightest):*

* Add `payload_semantics_bearing` (yes / no / conditional) and
  `payload_semantics_recovery` (prose) fields to
  `registries/meta_registry.yaml`.
* Populate via `FTR-META-LIFECYCLE-INVENTORY-P5`.
* No code change. No new guards.
* This direction's value is documentation and reviewability.

Direction B -- *Registry attribute + concept doc:*

* Direction A plus a new concept document
  `docs/technical_model/ch04_concepts/payload_semantics_recovery.adoc`
  that states the rule, lists exceptions, and documents the case
  survey.
* Cross-link from the canonical-vs-derived concept doc, the
  registry semantics chapter, and `agent_rules.adoc`.

Direction C -- *Direction B plus a narrow guard:*

* A registry-only guard that fails when a payload-semantics-
  bearing entry's other classification fields are inconsistent
  (e.g. classified as canonical-meta + payload-semantics-bearing
  but marked non-persistent).
* No runtime rule; check is in `tests/architecture/`.

Direction D -- *Direction C plus a recovery review for the worst
offenders:*

* Pick one or two entries (likely `cell_codecs` or
  `sparse_defaults`) and document the recovery decision per the
  rule. Establishes the *pattern* without committing to the full
  inventory.

The recommended starting point is *Direction B*. Direction C is a
natural follow-up once the registry attribute exists. Direction D
is a separate slice and not part of this FTR.

==== Interaction with existing items

* *`BUG-UNRESOLVED-HELPER-VALUE-NAN-SEMANTICS-P4A`*: this FTR
  provides the *framework*; the BUG provides the *contract
  decision*. The rule's recovery preference favors options 1
  (empty / null with a normalization layer) or 3 (preserve
  spreadsheet errors as typed tokens), because those surface the
  meaning into the data layer. The rule accepts option 2
  (structured finding) as a pinning fallback when recovery is
  not possible without semantic loss elsewhere. The BUG keeps
  ownership of the final choice and the implementation.

* *`FTR-META-LIFECYCLE-INVENTORY-P5`*: this FTR contributes the
  classification *axis*; the inventory FTR does the per-attribute
  classification *work*. Acceptance of this FTR depends on the
  inventory FTR existing but not yet on it being executed.

* *`FTR-VISIBLE-CELL-TYPING-STRATEGY-P5`*: visible-cell typing is
  one instance of "payload meaning in metadata vs canonical
  data". The rule reframes the FTR's three options as
  instances of a general pattern. The release-timing decision
  remains owned by the typing FTR; a future typed-payload move
  would consume this FTR's classification.

* *`FTR-PRESENTATION-DEFAULTS-SEMANTIC-CLASSIFICATION-P5`*: the
  rule sharpens the split between presentation entries that *do*
  carry payload meaning (e.g. the editability part of `protection`)
  and those that do not (e.g. `header_fill_rgb`). Coordinated
  read; no direct dependency.

* *Collaborative-roundtrip concept (warm storage)*: the rule is a
  *precondition*. The companion review explicitly recommended this
  rule as the way to make the collaborative-roundtrip rule safe.

* *`FTR-FK-HELPER-POLICY-LIFECYCLE-P4A`*: touches the
  `produced_by` provenance question (case L-13 in the review).
  The lifecycle FTR may classify the `produced_by` fields under
  the new axis.

* *`FTR-DERIVED-PROVENANCE-LIFECYCLE-P4A`*: clarifies that the
  rule applies only to the *durable* projection of helper
  identity, not to the transient `_meta.derived` channel.

* *`FTR-WORKBOOK-VIEW-PAYLOAD-CONFLICT-PRECEDENCE-P6`*: compatible.
  Conflict precedence already rejects heuristic resolution; the
  rule sits one layer up ("what *is* payload semantics in the
  first place").

* *`FTR-META-CONTRACT-VERSIONING-P4A`*: if the registry gets a
  `payload_semantics_bearing` attribute, its schema is in scope
  there.

==== Non-goals

* Do not implement a typing system.
* Do not implement a metadata encoder, decoder, or finding
  framework.
* Do not resolve `BUG-UNRESOLVED-HELPER-VALUE-NAN-SEMANTICS-P4A`.
* Do not produce per-attribute classification (this is the
  inventory FTR's scope).
* Do not introduce new `_meta` roots.
* Do not introduce a generic semantic-rule engine.
* Do not change runtime behavior.
* Do not preempt the visible-cell typing decision.

* The concept document renders in the technical-model build
  without gaps.
* The case survey reproduces the entries in section 2 of
  `docs/warm_storage/payload_semantics_recovery_review.adoc`.
* If Direction B's registry fields are added, the

* No code changes are required to validate the FTR; review and
  reading are sufficient.

==== Residual risks

* *Over-classification* -- too many entries tagged as
  payload-semantics-bearing reduces the discriminating power. The
  bar is "the meaning of a visible cell changes if this entry is
  lost".
* *Under-classification* -- obvious cases excluded. The case
  survey in
  `docs/warm_storage/payload_semantics_recovery_review.adoc`
  is the calibration set.
* *Confusion with visible-cell typing* -- typing is *one
  instance*, not the whole pattern.
* *Confusion with the collaborative-roundtrip rule* -- this rule
  is a *precondition* for that one, not a variant.
* *Per-attribute work growing under this FTR's scope* -- the
  per-attribute reclassification belongs to
  `FTR-META-LIFECYCLE-INVENTORY-P5`.
* *Aspirational recovery preference* -- "surface into canonical
  data" is a preference, not a mandate; misreading it as a mandate
  would force unnecessary canonical-data redesigns. The FTR text
  must make the soft/hard split explicit.

==== Coordination notes

* The FTR is *exploratory*. Promotion to *Active* or *Accepted*
  requires the case survey to be reviewed by the maintainer and
  the relationship to the unresolved-helper BUG to be agreed.
* The FTR's slicing follows
  `docs/developer_guide/ch08_git_and_workflow/01_git_and_workflow.adoc`
  "FTR Slicing for Metadata and Registry Changes": this FTR
  introduces a *new registry classification axis*, which is the
  "extending vs replacing vs introducing" boundary -- this is
  *extending* (a new optional field), not *introducing* (a new
  root). The lifecycle FTR remains the introducer if it lands
  first.
* Per
  `docs/ai_info/interfaces_and_gates.adoc`, adding a new optional
  registry field is *not* a gated public-surface change. Adding a
  new top-level `_meta` root would be; this FTR does not.
