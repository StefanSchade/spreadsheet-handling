= Human Review Appraisal: FTR-META-ONTOLOGY-REMOVAL-WORKBOOK-PROJECTION-EPIC-P4A
:revdate: 2026-07-19

:reviewed-artifact: FTR-META-ONTOLOGY-REMOVAL-WORKBOOK-PROJECTION-EPIC-P4A_additional_information_epic_review.adoc

== Purpose

This document records human appraisal of the Fable architecture review before
implementation begins.

It is intended to prevent implementation from proceeding on plausible but
incorrect assumptions that arose from ambiguous wording in the review.

This appraisal does not reject the Fable verdict:

[source,text]
----
TARGET MODEL SUPPORTED WITH AMENDMENTS
----

It accepts the broad result that the repository is closer to the target model
than initially feared and that the cleanup gap is real.

It corrects several interpretations and resolves the human-decision questions
needed for the first implementation pass.

The implementation shall follow the epic, the additional-information document,
the Fable review, and this appraisal together. Where this appraisal contradicts
an interpretation in the review, this appraisal is authoritative.

== 1. Core architectural principle

The target architecture is operational and contract-oriented, not ontological.

Feature transformations understand their own semantics.

They may emit narrow instructions for generic downstream mechanisms, for
example:

* drop this frame during final domain cleanup;
* keep this frame because the inverse transformation needs shadow state;
* apply protection to this workbook projection;
* highlight or otherwise annotate an editable area;
* map a feature-local projection into a workbook target.

Generic downstream components execute these instructions mechanically.

They do not need to understand:

* why the frame is redundant;
* whether a frame is canonical;
* whether a frame is an editable projection;
* whether a projection is lossless;
* how reconstruction works;
* which business meaning the transformation implements.

The corresponding inverse transformation belongs to the same transformation
family and understands the feature-local forward contract.

It reconstructs or merges according to that contract.

It must not depend on generic classifications such as `canonical_frame` to
discover what the forward transformation meant.

== 2. Human decision on `canonical_frame`

=== 2.1 The review question was based on a misleading premise

The Fable review asks whether `sheet_mappings[*].canonical_frame` should:

. continue to be derived from `frame_lifecycle.canonical` and `derived_from`; or
. become an explicit Workbook View DSL field.

This is not the primary decision.

Before preserving or relocating `canonical_frame`, the implementation must
identify the concrete runtime decision currently hidden behind that term and
determine whether the field is needed at all.

The intended model does not require a generic canonical identity to connect a
forward transformation to its inverse.

=== 2.2 Why `canonical` is misleading

Consider an unrestricted XRef Crosstable transformation.

Initial state::

A classical relation/xref frame is read from JSON and may be the representation
persisted in Git. Calling it canonical is intuitive.

Forward projection::

The transformation creates a complete crosstable projection.

Lossless condition::

Because no filter restricts the projection, the crosstable is sufficient to
reconstruct the complete relation.

Cleanup decision::

The original relation frame is marked for deletion to avoid redundant logical
state.

The frame that appeared canonical at the input boundary is therefore the frame
that is legitimately removed after a lossless projection.

This demonstrates that `canonical` is not a stable, preservation-implying frame
property.

A frame may be:

* authoritative at one persistence boundary;
* the source of a projection;
* redundant after a lossless transformation;
* absent during the editable phase;
* reconstructed later by the inverse transformation.

Generic canonicality imports rules that conflict with the actual feature
contract, especially the assumption that canonical data must always be
retained.

=== 2.3 Decision

Do not automatically replace lifecycle-derived `canonical_frame` with an
explicit `canonical_frame` field in Workbook View.

First:

. inventory the real consumers of `sheet_mappings[*].canonical_frame`;
. identify the exact operational decision each consumer makes;
. determine whether the responsible inverse transformation already knows that
  decision from its feature-local contract;
. remove the generic field if it is not independently required;
. introduce a narrower feature-local mapping only if a concrete consumer
  requires it.

The implementation must not preserve ontology merely by moving it from
`frame_lifecycle` into Workbook View DSL.

== 3. Human decision on provenance fields

The fields under review include:

* `derived_from`;
* `superseded_by`;
* `produced_by`;
* related generic provenance carried in `frame_lifecycle`.

=== 3.1 Decision

Prefer radical removal from generic persistent metadata.

These fields may appear valuable for traceability or debugging, but they create
three risks:

* persistent metadata ballast;
* accumulation across roundtrips;
* renewed invitation to derive generic behavioral rules from broad provenance
  terms.

The cleanup should be approached like clearing a cluttered storage space:
retain only information with a concrete, demonstrated consumer.

Note: derived_from currently has a demonstrated consumer through the canonical_frame computation described in Section 2. The removal bias stated above applies only after that consumer has been fully inventoried and either removed or replaced by a narrower contract.

=== 3.2 Feature-local metadata remains valid

Removing generic provenance does not mean removing transformation state.

A transformation may require precise feature-local data for its inverse.

For example, XRef Crosstable may need to persist the exact columns or value
domain shown in the editable projection. If a user deletes a displayed column
and that deletion is interpreted as removing all corresponding references, the
inverse transformation must know which columns were originally presented.

Such data belongs in the transformation family's own metadata contract.

It should be:

* named in the vocabulary of the feature;
* produced by the forward transformation;
* consumed by the corresponding inverse transformation or another explicit
  member of the same transformation family;
* persisted only for as long as the feature contract requires;
* documented with its exact meaning and lifecycle.

Only narrow commands that are genuinely needed by another generic component
should diffuse into a shared generic metadata family.

Examples include an explicit cleanup instruction or a concrete workbook
projection directive.

=== 3.3 Implementation rule

For every generic provenance field:

. identify a concrete consumer;
. identify the exact decision made by that consumer;
. identify the required persistence lifetime;
. retain only if those answers establish current value;
. otherwise remove the producer, consumer assumptions, registry entry, and
  tests together.

Do not retain generic provenance merely for possible future debugging.

== 4. Human decision on final cleanup execution

=== 4.1 Decision

For the first implementation, final domain cleanup shall be implicit at the
orchestrator-owned persistence boundary.

Conceptually:

[source,text]
----
configured domain pipeline
-> implicit final domain cleanup
-> persistence-boundary projection
-> selected output adapter
----

The cleanup must remain a visible, deterministic architectural boundary even
though it is not declared as a normal YAML step.

=== 4.2 Rationale

The cleanup is part of producing the final logical state for persistence.

Requiring every pipeline builder to remember an explicit final cleanup step
would make correctness depend on repeated boilerplate.

The project is still free to introduce configuration or an explicit step later
if real use cases require alternate placement.

That later extension would be a deliberate contract change.

=== 4.3 Constraints

The implicit cleanup boundary must:

* execute only explicit cleanup directives;
* never infer deletion from lifecycle roles or frame categories;
* consume the directives it executes;
* be deterministic and idempotent for the current invocation;
* run before the final orchestrator-owned adapter;
* remain separate from spreadsheet-specific Workbook View projection;
* not retroactively affect artifacts already emitted by side-effecting
  mid-pipeline steps.

The existence of side-effecting mid-pipeline steps is itself considered a
separate architecture problem, recorded in Section 10.

=== 4.4 Practical consequence for the first implementation

Because implicit final cleanup runs only at the orchestrator-owned persistence
boundary, it cannot affect artifacts that mid-pipeline side-effecting steps
(`write_structured_yaml`, `write_key_value_resources`, and possibly parts of
`write_artifact_manifest`) have already written to the filesystem earlier in
the same run.

This means that, for as long as those steps exist in their current form, a
real carrier asymmetry remains between:

* the final adapter output, which reflects the cleaned logical state produced
  by the implicit cleanup boundary; and
* any artifact already written by a mid-pipeline step before that boundary
  runs, which reflects the uncleaned state at the time it was written.

This is a known and accepted limitation of the first implementation, not an
oversight. It follows directly from the decision in Section 4.1 and the
layering problem recorded in Section 10, and it is called out here explicitly
so it is not missed by only reading Section 4 or only reading Section 10 in
isolation.

Resolving this asymmetry requires the separate layering refactor described in
Section 10 (separating projection from persistence for these steps). It is out
of scope for this epic. Until that refactor lands, pipeline builders who use
mid-pipeline side-effecting writers should be aware that those outputs do not
benefit from final domain cleanup and may still contain frames that the final
adapter output no longer does.


== 5. Human decision on drop and keep authoring modes

=== 5.1 No generic decision algorithm

There shall be no generic algorithm that decides whether a frame should be
dropped or kept and then allows transformations or users to override that
decision in either direction.

That model is explicitly rejected.

The decision belongs to:

* the feature transformation when its contract establishes the result; or
* the pipeline builder when only the concrete pipeline composition establishes
  whether a generic intermediate is disposable.

The cleanup mechanism only executes the declared decision.

There is no ontology-derived default that can override it.

=== 5.2 Two authoring modes may be supported

Core may support both:

Drop mode::

The producer or pipeline builder names frames to remove.

Keep mode::

The pipeline builder names the final frame set to retain and all other frames
are removed.

Both modes are useful, especially when a large transformation pipeline produces
many intermediates but only a small export set should remain.

=== 5.3 Mutual exclusivity

Drop and keep are alternative configuration modes for one cleanup decision.

They must not be mixed in the same cleanup contract or cleanup execution.

The valid shapes are conceptually:

[source,yaml]
----
pipeline_cleanup:
  drop_frames:
    - intermediate_a
    - intermediate_b
----

or:

[source,yaml]
----
pipeline_cleanup:
  keep_frames:
    - stories
    - characters
    - places
----

A configuration that declares both must fail.

This is not an override or precedence system.

It is a choice between two equivalent ways of specifying the intended final
logical frame set.

=== 5.4 Transformation-produced instructions

A feature transformation that knows a source frame is fully reconstructable may
contribute a drop instruction.

A feature transformation that requires shadow state contributes no drop
instruction.

There is no `omit_by_default`.

There is no generic cleanup heuristic that later changes the transformation's
decision.

== 6. Human decision on `enrich_lookup`

`enrich_lookup` is a feature/domain transformation outside
`domain/workbook_views.py`.

The Fable review identified that it currently writes lifecycle metadata such as:

* `role: intermediate`;
* `render: omit_by_default`;
* `superseded_by`.

Those values are inert in real produced Workbook View flows and participate in
the ontology that this epic intends to remove.

=== Decision

`enrich_lookup` shall follow the same general rule as other feature
transformations:

* stop writing misleading render/lifecycle classifications where no concrete
  independent consumer remains;
* if its feature contract can establish that a source frame may be removed,
  contribute an explicit cleanup instruction under an explicit transformation
  parameter or feature-local condition;
* if the source is required as shadow or later pipeline input, do not contribute
  a drop instruction;
* do not write `omit_by_default`;
* do not rely on a generic cleanup algorithm to infer a default.

The same principle applies to `xref_crosstable`, `compact_multiaxis`, and other
affected transformations.

The exact drop/keep decision remains feature-local or builder-supplied.

== 7. Human decision on cleanup directive persistence

Cleanup directives are commands, not history.

=== Decision

Consume cleanup directives when final domain cleanup executes.

After execution, the command must not remain in persistable `_meta`.

This prevents:

* re-execution after workbook blob roundtrip;
* accidental deletion of frames reconstructed by the inverse pipeline;
* metadata pileup;
* confusion between pending commands and historical records.

If a cleanup audit or diagnostic record is later required, it must use a
separate non-executable report or trace contract.

It must not remain in the command namespace.

== 8. Correction of architecture-guard interpretation

The existing architecture guard does not prohibit:

* metadata interpretation by pipeline steps;
* generic steps understanding metadata;
* cross-family metadata access;
* a feature transformation writing into a shared cleanup metadata family;
* a cleanup component consuming that family.

The guard is intended to prevent generic pipeline infrastructure from:

* treating `_meta` as an ordinary frame;
* routing pipeline configuration through payload metadata;
* making undeclared orchestration decisions from workbook-specific or
  lifecycle-specific payload metadata.

Cross-family metadata interaction is allowed.

It must be:

* explicit;
* minimal;
* directed;
* documented as a producer/consumer contract;
* tested where the interaction changes behavior.

For this epic, the expected interaction is:

[source,text]
----
feature transformation or pipeline builder
-> shared cleanup command family
-> final cleanup consumer
----

That interaction is intentional and not a guard violation.

== 9. Logging and privacy correction

The Fable review suggests that frame-set changes might be added to cleanup
diagnostics because current DEBUG tracing records only `_meta` path changes.

Do not add frame-set tracing as part of this epic.

This omission is intentional.

Frame names and frame sets may contain domain-specific, business-sensitive, or
personal information. Logs can have different access and retention properties
from the persisted data artifacts.

Logging frame-set contents may therefore create unnecessary information-
protection or GDPR risk.

The resulting frame set is visible in the actual output artifact.

Allowed diagnostics should remain narrow, for example:

* count of frames removed, if safe;
* cleanup command consumed;
* success/failure status;
* non-sensitive error identifiers.

Any richer frame-name logging requires a separate privacy and logging decision.

== 10. Side-effecting pipeline steps are a separate layer violation

The Fable review correctly found that:

* `write_structured_yaml`;
* `write_key_value_resources`;
* possibly parts of `write_artifact_manifest`;

perform filesystem I/O as ordinary pipeline/domain steps.

This behavior contradicts the intended layering.

The intended responsibility split is:

[source,text]
----
domain transformation / projection
-> transforms Frames and Meta into a structured representation

application / I/O boundary
-> writes external artifacts
----

Direct filesystem persistence inside a domain transformation is considered a
likely layer violation, not a desirable precedent for cleanup placement.

This issue is outside the implementation scope of the current epic.

It must be recorded as a separate FTR candidate covering:

* inventory of side-effecting pipeline steps;
* separation of projection from persistence;
* artifact handoff representation;
* failure and transaction semantics;
* dry-run behavior;
* report-frame behavior;
* an architecture guard against unapproved filesystem writes from domain
  transformation modules.

Until that refactor lands, implicit final cleanup cannot change artifacts that
those steps already emitted.

== 11. Interpretation of worldbuilding evidence

The many steps and private plugins in the `worldbuilding` consumer are evidence
of missing generic core capabilities and accumulated workarounds.

They are not a reference architecture to preserve.

In particular:

[source,text]
----
currently consumer-owned
----

means:

[source,text]
----
the consumer currently compensates for functionality that should be available
as a generic core building block
----

Migration from `worldbuilding/plugins/frame_sets.py::keep_frames` to the new
core cleanup mechanism is a useful acceptance proof.

It must not be interpreted as evidence that consumer-specific plugins are the
desired long-term contract.

== 12. Absent-frame intent and metadata pileup

The review correctly finds that transformation intent metadata may legitimately
reference frames absent from the current logical state.

Such references must not be rejected generically.

However, legitimate absent-frame intent can accumulate across repeated
roundtrips and become stale metadata sediment.

This epic must preserve currently required feature-local intent.

It does not need to solve generic metadata pileup.

A separate analysis/FTR candidate shall investigate:

* detection of metadata references to absent frames;
* growth across repeated roundtrips;
* warning thresholds or health reports;
* classification of persistent intent versus likely sediment;
* explicit pipeline-builder cleanup tools;
* an optional end-of-roundtrip prune before persistence of the final JSON or
  other authoritative data state.

The framework cannot generally decide when an absent-frame reference is
obsolete without knowledge of the pipeline's intended future steps.

Its responsibility is to provide analysis and controlled cleanup primitives.

The pipeline builder supplies the domain knowledge.

== 13. Implementation guidance for Fable

Before implementation, Fable shall amend its working assumptions as follows:

. Do not introduce an explicit `canonical_frame` field merely to replace the
  current lifecycle derivation.
. Audit whether `canonical_frame` is needed at all.
. Remove generic provenance aggressively unless a concrete consumer proves
  current value.
. Implement cleanup as an implicit orchestrator-owned final domain boundary for
  the first slice.
. Execute only explicit drop or keep declarations.
. Do not implement ontology-derived defaults or override precedence.
. Treat drop mode and keep mode as mutually exclusive authoring alternatives.
. Remove `omit_by_default` and equivalent render-inference behavior where no
  independent consumer remains.
. Consume cleanup commands on execution.
. Allow and document cross-family metadata contracts.
. Do not add frame-set logging.
. Do not treat side-effecting pipeline steps as evidence for desired layering.
. Preserve required feature-local absent-frame intent using explicit per-family
  policies.
. Treat worldbuilding plugins as migration targets and workaround evidence.

== 14. Remaining implementation questions

The following details may still be resolved during implementation without
reopening the architecture:

* exact shared cleanup metadata path;
* exact YAML spelling of mutually exclusive drop and keep modes;
* whether feature transformations contribute commands directly or via small
  shared helpers;
* error behavior when a drop-mode frame is already absent;
* idempotence semantics for duplicate identical drop contributions;
* exact per-family metadata-reference cleanup table;
* whether a non-sensitive cleanup count is logged;
* exact migration sequence for `worldbuilding`;
* whether a separate cleanup report is useful.

Any proposal that reintroduces generic frame ontology, implicit drop/keep
heuristics, or canonicality-based preservation requires renewed human review.
