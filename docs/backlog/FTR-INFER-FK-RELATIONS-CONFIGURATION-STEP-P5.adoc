===== FTR-INFER-FK-RELATIONS-CONFIGURATION-STEP-P5 -- FK relation inference as explicit configuration step

*Status:* New

*Purpose:*::
Introduce an explicit `infer_fk_relations` configuration step that may use
bounded heuristics to derive FK relation policy and write the resolved result
to `_meta`, without materializing helper columns.

*Context:*::
The project direction is to isolate heuristics from primitive transformations.
A heuristic may inspect frames and propose configuration, but the later
primitive step must consume explicit resolved metadata.
+
FK helpers are the first concrete case where this split should become visible
in runtime behavior. The current primitive path still infers relation identity
from naming conventions. This FTR introduces the configuration step that owns
that inference.

*Relationship to `configure_fk_helpers`:*::
+
`configure_fk_helpers` already exists and today carries *both* an explicit
per-target path and a heuristic `targets: "auto"` path
(`domain.helper_policies._auto_fk_policy_specs`). Phase 3a removes the
heuristic from `configure_fk_helpers`:
+
* `configure_fk_helpers` remains the *explicit / manual* policy step. It
  writes the v2 relation-oriented `_meta.helper_policies.fk` shape from
  user-supplied configuration only. The `targets: "auto"` mode is removed
  from `configure_fk_helpers` together with the related `auto:` parameter
  shapes.
* `infer_fk_relations` is the *heuristic* policy step. It owns the inference
  modes (initially `naming_convention`) that previously lived inside
  `configure_fk_helpers(targets: "auto")` and `add_fk_helpers`'s implicit
  detection. It writes the same v2 shape.
+
Both steps write to the same root and the same shape; pipelines pick one
according to whether the relation set is explicit or inferred. A pipeline
that uses both must fail clearly on conflicting writes (a relation key may
be produced by at most one step in a given pipeline run).
+
This is a documented YAML / step-surface change to `configure_fk_helpers`
and therefore goes through the public-surface gate in
`docs/ai_info/interfaces_and_gates.adoc` (already named in the Phase 3a
preamble and in `FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5`).

*Dependencies:*::

* `FTR-FK-RELATION-POLICY-MODEL-P5`
* Existing pipeline registry infrastructure.
* Existing `BoundStep` / step factory pattern.
* Existing metadata bootstrap and `_meta` propagation behavior.

*Scope:*::

* Add a new pipeline step named `infer_fk_relations`.
* Register it as a `configuration` step in the runtime registry and the
  pipeline step registry; ensure the registry-parity guard
  (`tests/architecture/test_pipeline_step_registry.py`) still passes after
  the addition.
* The step reads current frames and configured inference rules.
* The step writes resolved FK relation policy into the v2 shape under
  `_meta.helper_policies.fk` (see `FTR-FK-RELATION-POLICY-MODEL-P5`).
* The step must not create helper columns.
* The step must not validate helper column values.
* The step must not write derived helper provenance.
* The step must make its heuristic mode explicit in configuration and in the
  produced policy metadata (`produced_by.step`, `produced_by.mode`).
* Support at least one initial bounded inference mode based on the existing
  `id_(<target>)` naming convention.
* Migrate the heuristic / `targets: "auto"` path out of `configure_fk_helpers`
  into `infer_fk_relations`; `configure_fk_helpers` keeps only its explicit /
  manual policy path. Update the pipeline step registry descriptions
  accordingly.

*Configuration sketch (illustrative):*::

The example uses the project's existing FK column convention
(`id_(<target>)`); other patterns may be added by future inference modes.

[source,yaml]
----
pipeline:
  - step: infer_fk_relations
    mode: naming_convention
    id_columns: [id]
    fk_patterns:
      - "id_({target})"
    target_label_fields: [name, label]
    on_ambiguous: fail
    on_missing_target: fail
----

Alternative naming may be chosen during implementation, but the step must keep
heuristic policy explicit and opt-in, and the produced shape must match the
v2 relation policy defined in `FTR-FK-RELATION-POLICY-MODEL-P5`.

*Behavior:*::

* If a configured FK pattern maps one source column to exactly one target frame
  and one target key, the step writes a resolved relation policy entry.
* If multiple targets match and `on_ambiguous: fail`, the step raises a clear
  error.
* If no target matches and `on_missing_target: fail`, the step raises a clear
  error.
* If no relation is inferred and policy allows ignore/warn behavior, the output
  must still be deterministic and visible in logs or findings.
* The output metadata must be inspectable without re-running the heuristic.

*Non-Goals:*::

* No helper column materialization.
* No FK helper value validation.
* No prefix-based helper cleanup.
* No automatic default inference hidden inside `add_fk_helpers`.
* No attempt to infer arbitrary business relations without configured rules.
* No compatibility promise for old implicit FK conventions.

*Acceptance:*::

* `infer_fk_relations` exists as a registered configuration step.
* The step writes the v2 FK relation policy shape defined by
  `FTR-FK-RELATION-POLICY-MODEL-P5` under `_meta.helper_policies.fk`.
* The step is opt-in; FK relation inference does not happen unless the
  pipeline asks for it.
* The produced policy records enough provenance to identify that it was
  inferred and which inference mode produced it (`produced_by.step`,
  `produced_by.mode`).
* Ambiguous inference fails or reports deterministically according to explicit
  configuration.
* The step has unit tests for:
  ** one clean inferred relation,
  ** multiple target ambiguity,
  ** missing target,
  ** no relation found,
  ** deterministic ordering of written policy entries,
  ** failure when both `configure_fk_helpers` and `infer_fk_relations` write
     the same relation key in one pipeline run (conflicting writers).
* The heuristic / `targets: "auto"` path is removed from `configure_fk_helpers`;
  a single integration test exercises `infer_fk_relations` followed by an
  explicit-config `configure_fk_helpers` call writing disjoint relations
  (composition allowed when keys do not collide).
* The pipeline step registry classifies the step as `configuration` and the
  registry-parity guard (`test_pipeline_step_registry`) still passes.
* Architecture guards continue to enforce that auto / heuristic behavior
  belongs to configuration steps, not primitive FK-helper execution; see
  `FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5` for the new primitive-side
  guard.

*Validation:*::

* Run targeted tests for the new inference step.
* Run `pytest tests/architecture/test_pipeline_step_registry.py` to confirm
  registry parity between `pipeline.registry.REGISTRY` and
  `docs/technical_model/ch05_registries/pipeline_step_registry/pipeline_step_registry.json`.
* Run the FK-helper architecture guard
  (`tests/architecture/dependency_guards/test_architecture_guard_fk_helpers.py`)
  and any new primitive-side guard added by
  `FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5`.
* Run `make test-one` or equivalent targeted test command for FK-helper
  related tests.
* Render updated documentation.
* Validate modified JSON / YAML registry artifacts.

*Residual risks:*::

* The first inference mode may only cover simple naming conventions.
* More complex models may still require explicit hand-written relation policy.
* The follow-up primitive refactor must remove the old implicit detection path;
  otherwise this step only adds a cleaner alternative without enforcing it.
