===== FTR-FK-HELPER-DOCS-DEMO-REALIGNMENT-P5 -- Realign demo, tutorial, and user docs with explicit FK helper policy

*Status:* New

*Purpose:*::
Update the public-facing demo, tutorial, and user documentation after FK helper
semantics move from implicit naming conventions to explicit relation policy and
configuration-step inference.

*Context:*::
Phase 2 created a clearer public entry surface: README flow, Pages root/archive
navigation, and a maintained product-business-slice tutorial path.
+
The FK-helper refactor will change how users should understand and configure
FK helper behavior. If the tutorial and user guide remain convention-oriented,
the freshly improved public surface will become misleading.
+
This FTR realigns the demo and documentation after the FK-helper runtime changes
land.

*Dependencies:*::

* `FTR-FK-RELATION-POLICY-MODEL-P5`
* `FTR-INFER-FK-RELATIONS-CONFIGURATION-STEP-P5`
* `FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5`
* Existing product-business-slice demo and walkthrough.
* Existing user guide transformation catalog.
* Existing Pages documentation build.

*Scope:*::

* Update demo pipelines to use the new FK-helper sequence.
* Decide for the maintained product-business-slice path whether to show:
  ** fully explicit FK relation policy, or
  ** `infer_fk_relations` followed by policy-driven primitives.
* Update demo README tutorial steps if command output, YAML explanation, or
  verification expectations change.
* Update walkthrough text for the product business slice.
* Update user guide transformation catalog:
  ** add `infer_fk_relations`,
  ** clarify `configure_fk_helpers`,
  ** remove wording that says `add_fk_helpers` discovers helpers from FK
     conventions,
  ** remove or correct prefix-fallback language for `remove_fk_helpers`.
* Update technical model / AI info references that still describe FK helper
  behavior as convention-driven primitive logic.
* Re-render relevant local docs and confirm the Pages/public path still makes
  sense.
* Update screenshots only if visible workbook output changes in a way that
  makes existing screenshots misleading.

*Non-Goals:*::

* No further FK-helper runtime behavior changes.
* No new public tutorial path beyond the maintained product-business-slice path.
* No new release.
* No Pages publishing workflow change.
* No README version-baking solution.
* No attempt to document all possible FK inference modes exhaustively.

*Expected demo pipeline shape:*::

The maintained demo pipeline should show the intended architectural split
clearly. For example:

[source,yaml]
----
pipeline:
  - step: infer_fk_relations
    mode: naming_convention
    id_columns: [id]
    fk_patterns:
      - "{target}_id"
    target_label_fields: [name]

  - step: add_fk_helpers

  - step: configure_workbook_view
    # existing workbook view configuration
----

Or, for a fully explicit tutorial path:

[source,yaml]
----
pipeline:
  - step: configure_fk_helpers
    relations:
      - source_frame: products
        source_column: product_manager_id
        target_frame: product_managers
        target_key: id
        helper_fields: [name]

  - step: add_fk_helpers
----

The final form should prefer the clearest user-facing story.

*Acceptance:*::

* The maintained demo product-business-slice pipeline runs successfully with the
  new FK-helper model.
* The reimport pipeline removes helper columns without relying on prefix
  fallback.
* The demo README no longer implies that primitive FK helper steps discover
  relations by naming convention.
* The user guide transformation catalog accurately describes:
  ** `infer_fk_relations` as configuration / heuristic,
  ** `configure_fk_helpers` as explicit policy configuration,
  ** `add_fk_helpers` as policy-driven primitive materialization,
  ** `remove_fk_helpers` as provenance/policy-driven cleanup.
* The product-business-slice walkthrough remains executable.
* Existing screenshots are either still accurate or are updated with a review
  note explaining the change.
* Local Pages build succeeds.
* The public entrypoint flow remains coherent after the terminology change.

*Validation:*::

* In core:
  ** render the user guide,
  ** render affected technical docs,
  ** run `make docs-pages`,
  ** run relevant FK helper tests.
* In demo:
  ** run the maintained product-business-slice forward pipeline,
  ** perform one workbook edit,
  ** run the reimport pipeline,
  ** verify canonical JSON does not contain helper columns,
  ** run the demo roundtrip editing test.
* Check README and docs for stale convention-oriented wording such as:
  ** "detect FK columns",
  ** "from conventions",
  ** "prefix fallback",
  ** "id_<table>",
  ** "ID column identifies primary keys",
  where those phrases describe primitive behavior rather than explicit
  configuration or inference.
* Run `git diff --check`.

*Residual risks:*::

* The tutorial may become more abstract if it shows too much metadata policy.
  The final wording should show only enough YAML to explain the split.
* If screenshots are updated, the visual walkthrough FTR history remains valid
  but the image assets need a clear follow-up review.
* The README version-baking problem remains separate and must not be solved in
  this FTR.
