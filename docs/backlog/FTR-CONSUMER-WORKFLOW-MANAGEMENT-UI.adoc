===== FTR-CONSUMER-WORKFLOW-MANAGEMENT-UI-P5

*Status:* Idea / Post-0.3.0 candidate

====== Problem

The spreadsheet-handling workflow is powerful but exposes several command-line
operations whose safe order matters:

* export canonical data to spreadsheet projection;
* edit projection;
* reimport projection into staging;
* check and promote reimport;
* perform schema maintenance on canonical data.

For technical users this order can be managed through Make targets and CLI
commands. For business analysts, however, the workflow contains avoidable
pitfalls. A typical dangerous sequence is:

. export canonical JSON to a spreadsheet;
. keep the spreadsheet projection open/outstanding;
. perform schema maintenance on the canonical JSON;
. later reimport the older spreadsheet projection;
. accidentally overwrite or conflict with the schema-maintained canonical state.

The problem is not only command ergonomics. The problem is missing workflow
state.

====== Core Idea: The Wippe Model

Introduce a simple workflow state model with two sides:

* `canonical_active`
* `projection_active`

The model behaves like a rocker / "Wippe":

[source,text]
----
canonical_active
  -- export spreadsheet -->
projection_active
  -- reimport/promote or discard projection -->
canonical_active
----

Schema maintenance is allowed only while the workflow is on the canonical side.

When `projection_active` is set, schema maintenance must be disabled or blocked
by the management UI with a clear explanation:

[source,text]
----
Schema maintenance is locked because a spreadsheet projection is currently
active. Review/promote the projection or discard it before changing the
canonical schema.
----

====== Proposed Solution

Add a small, deliberately simple, cross-platform management UI.

Initial implementation candidate:

* Python
* Tkinter
* ugly but functional
* no spreadsheet grid
* no duplicate schema logic
* no core behaviour changes

The UI is an ergonomic workflow layer over existing commands:

* export spreadsheet;
* check reimport;
* promote reimport;
* discard projection;
* run schema maintenance dry-run;
* run schema maintenance write-to-staging;
* show JSON report.

The UI should collect arguments and invoke existing command surfaces, not
reimplement transformation, metadata, or schema-maintenance logic.

====== Project Profiles

The UI should support multiple consumer repos through profiles, for example:

[source,yaml]
----
worldbuilding:
  data_dir: daten/dino_insel
  projection_path: tmp/dino_stories.xlsx
  reimport_stage_dir: tmp/dino_stories_reimported
  check_command: make check-reimport
  promote_command: make promote-reimport-checked

ssdp:
  data_dir: structured/data
  projection_path: tmp/structured/ssdp_catalog.xlsx
  reimport_stage_dir: tmp/structured/reimported
  check_command: make check-structured
  promote_command: make promote-structured
----

====== Workflow State

The first version may store repo-local UI state in a generated file, for
example:

[source,json]
----
{
  "state": "projection_active",
  "projection_path": "tmp/dino_stories.xlsx",
  "canonical_data_dir": "daten/dino_insel",
  "exported_at": "2026-06-10T20:55:00"
}
----

Later versions may add hashes of the canonical data at export time to detect
whether canonical data changed while a projection was outstanding.

====== MVP Scope

The first version should support:

* selecting a repo profile;
* showing current workflow state;
* export projection;
* check/promote reimport;
* discard projection;
* schema maintenance only when `canonical_active`;
* dry-run schema maintenance;
* write schema maintenance to staging;
* display report output.

Supported schema maintenance operations:

* `add_column`
* `drop_column`
* `rename_column`
* `reorder_columns`

====== Non-Goals

* no new core schema-maintenance semantics;
* no public pipeline step;
* no spreadsheet-grid editor;
* no automatic schema inference from Excel;
* no replacement for existing CLI/Make workflows;
* no complex GUI framework in the first slice;
* no hidden writes to canonical data.

====== Design Principle

The UI should not make the workflow magical. It should make the safe order
obvious.

The purpose is to guide business analysts through the workflow and prevent
dangerous sequencing errors, especially the accidental overwrite of canonical
schema changes by stale spreadsheet projections.
