==== Findings

===== Before implementation: "ontology / information-flow chapter" is too broad and under-capped

*Observation:* The FTR states that the recut is "navigational documentation
work" and a "single, narrow editorial pass" (`FTR...adoc`, lines 20-23), but
then makes it in-scope to add "an ontology / information-flow chapter (or a
chapter pair)" to the Functional Model (`FTR...adoc`, lines 132-135), and
repeats that as an acceptance criterion (`FTR...adoc`, lines 178-181). The
same scope section also says "new vocabulary or new chapters are out of
scope" for the Semantic Model navigation entry (`FTR...adoc`, lines 118-122),
which is directionally right but does not cap the new Functional Model chapter.

*Risk:* "Ontology" is broad enough to invite a second large vocabulary model:
problem-space classes, subtypes, information locations, process roles,
relationships, lifecycle states, and edge-case taxonomies. That would defeat
the FTR's purpose. The semantic-inflation risk would be moved from the
Semantic Model into the Functional Model instead of being contained.

*Evidence from source review:* The boundary review describes the desired
containment rule as keeping explanatory pressure in the Functional Model so
semantic vocabulary must defend itself against a stated functional need
(`semantic_boundary_review.adoc`, lines 360-364). It also identifies only a
partial "ontology / vocabulary" gap and an "information-flow explanations" gap
(`semantic_boundary_review.adoc`, lines 388-416), not a need for a full
ontology.

*Recommended action:* Replace "ontology" with narrower wording and add an
explicit size/scope constraint to scope and acceptance.

Recommended replacement for the scope bullet:

[source,adoc]
----
* Add a lightweight problem-space vocabulary / information-flow overview to the
  Functional Model. It may absorb only the framing of problemraum_konzept ch02
  and ch03: working forms, process boundaries, information roles, and loss /
  reconstruction questions. It must not introduce a formal ontology, new
  semantic set taxonomy, lifecycle taxonomy, registry vocabulary, or rule-ready
  contract model.
----

Recommended replacement for the acceptance bullet:

[source,adoc]
----
* The Functional Model gains a bounded problem-space vocabulary /
  information-flow overview, limited to one concise chapter or two short
  subchapters. It summarizes the ch02-ch03 framing without expanding it into a
  new ontology. The German source remains verbatim in
  `docs/warm_storage/problemraum_konzept/`.
----

Recommended additional non-goal:

[source,adoc]
----
* Do not create a Functional Model ontology. This FTR may name problem-space
  vocabulary only where it explains existing workflows and process boundaries;
  contract vocabulary remains owned by the Semantic Model.
----

Recommended additional acceptance criterion:

[source,adoc]
----
* The new Functional Model material has an explicit "Scope limit" note stating
  that it is an overview, not a taxonomy expansion point. Any future expansion
  requires a separate FTR with a stated functional need and a size cap.
----

*Priority:* Before implementation / maintainability.

==== Focus Questions

*Is the proposed Functional Model ontology/information-flow material clearly bounded?*

Not yet. The FTR says the German source remains verbatim and only "framing" is
absorbed, which helps, but "ontology / information-flow chapter (or chapter
pair)" is not bounded by length, allowed terms, forbidden terms, or output
shape.

*Does the FTR define a lightweight target, or could it become a new large model?*

It intends a lightweight target, but the current wording could become a large
model. The safest interpretation is "short problem-space overview"; the
current phrase also permits "new Functional Model ontology."

*Should acceptance criteria include an explicit size/scope constraint?*

Yes. Add a cap such as "one concise chapter or two short subchapters" and say
the result must summarize ch02-ch03 framing rather than expand it into a new
taxonomy.

*Is the phrase "ontology" too broad for this slice?*

Yes. In this project context it is especially risky because the whole FTR is
about preventing terminology from generating more terminology.

*Would "problem-space vocabulary" or "information-flow overview" be safer?*

Yes. "Problem-space vocabulary / information-flow overview" is the safer
phrase because it keeps the target explanatory and tied to workflows.

*Does the FTR sufficiently prevent future semantic inflation, or merely move
the inflation risk from Semantic Model to Functional Model?*

It partially prevents inflation through the no-runtime/no-registry/no-new-
semantic-sets non-goals and the adoption/planning zone. It does not yet
sufficiently prevent Functional Model inflation because the new Functional
Model material lacks a scope cap and uses the broad term "ontology."

==== Non-findings

* The FTR correctly preserves the German problemraum_konzept verbatim and
  treats the English Functional Model material as absorbed framing, not a full
  translation.
* The FTR correctly keeps runtime, registry, implementation, and new semantic
  set changes out of scope.
* The explicit Adoption / Planning grouping is a useful brake and should stay.

==== Release Notes

No release-note change is required for this FTR assessment. The reviewed item
is documentation planning and does not itself change user-facing behavior,
CLI/YAML/API surface, demo behavior, or runtime semantics.
