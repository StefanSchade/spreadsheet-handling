===== FTR-DYNAMIC-VIEW-COLUMN-TARGETING-P5 -- Declarative targeting of dynamic workbook view columns

*Status:* New

*Purpose:*::

Allow declarative rules to target dynamic workbook-view columns without
explicitly enumerating their concrete names.

The goal is not to introduce a second projection framework.

The goal is to let consumers address projected column sets such as:

* all matrix payload columns,
* all non-helper columns,
* all non-key columns,
* all projected columns derived from a specific logical class,

for downstream concerns such as:

* validation,
* editable-state rendering,
* presentation,
* workbook constraints.

*Context:*::

Consumer validation in the `worldbuilding` repository exposed a recurring
pattern:

1. A canonical relation is projected into a workbook-facing matrix view.
2. The set of visible payload columns is dynamic and depends on domain data.
3. Post-projection concerns must then be applied to exactly those columns.

Examples:

* apply `from_legend` validation to every matrix payload column,
* exclude helper columns like `title`,
* exclude key columns like `story_id`,
* keep the rules stable while the dynamic axis grows with new entities.

Today this is possible, but not declaratively.

The validating consumer had to use a local plugin to inspect the projected
frame, derive the effective payload column list, and emit matching metadata for
each column.

This suggests a real product question:

Should the framework support declarative targeting of dynamic workbook-view
column sets after projection has already happened?

*Motivation:*::

This is strategically meaningful because many spreadsheet-driven maintenance
surfaces follow the same pattern:

* classification matrices,
* permission matrices,
* localization matrices,
* feature-toggle matrices,
* configuration matrices with helper columns.

In these scenarios, the important abstraction is not the literal column name
but the semantic class of the column within the generated view.

If the framework cannot address such column classes declaratively, consumers are
forced to reintroduce procedural glue precisely at the point where the product
should provide stability.

This is therefore not only a convenience issue.

It is a question of whether dynamic workbook views can become a first-class
maintenance surface in the product.

*Scope:*::

* Evaluate declarative ways to target dynamic workbook-view column sets.
* Evaluate how targeting should compose with existing projection steps.
* Evaluate whether targeting should operate on view metadata, frame metadata,
  or explicit selectors.
* Evaluate applicability to validation, editable-state semantics, and
  presentation concerns.
* Record the intended long-term direction.

*Non-Goals:*::

* No redesign of `contract_xref`, `expand_xref`, or `compact_multiaxis`.
* No replacement of existing explicit column targeting.
* No inference of business semantics from cell values.
* No mandatory removal of plugin extensibility.
* No implementation commitment in this FTR.

*Representative Example:*::

The consumer need is structurally similar to:

[source,yaml]
----
- step: add_validations
  rules:
    - target:
        sheet: story_groups
        columns:
          selector:
            frame: story_group_matrix_view
            exclude_helper_columns: true
            exclude_key_columns: true
      rule:
        type: from_legend
        legend: story_group_codes
----

or an equivalent selector language.

The exact syntax is open.

The relevant point is that the consumer expresses intent against a dynamic
column class rather than enumerating concrete column names.

*Architectural Options:*::

Option A -- Keep Plugins
+
Treat dynamic column targeting as a consumer-side concern.
+
Advantages:
+
* maximum flexibility,
* no additional selector surface in the pipeline language.

-

Disadvantages:
+
* recurring procedural glue in consumers,
* poor reuse across common matrix-style use cases,
* weakens the product story for dynamic workbook maintenance views.

Option B -- Add Selector Semantics
+
Introduce a small declarative selector model for dynamic workbook-view columns.
+
Advantages:
+
* reduces repeated consumer glue,
* aligns well with matrix-based maintenance surfaces,
* composes naturally with validation and presentation features.

-

Disadvantages:
+
* introduces new declarative surface area,
* requires careful architectural placement.

Option C -- Reuse Existing Metadata
+
Investigate whether workbook-view metadata, helper-column metadata, or future
projection metadata can already expose enough information to derive target
column sets without a new dedicated selector model.

*Acceptance Criteria:*::

* The worldbuilding use case is evaluated explicitly.
* Transferability to future business-oriented consumers is assessed.
* A project decision is recorded.
* It is documented whether plugins remain the preferred solution.
* If declarative support is preferred, a concrete follow-up design direction is
  identified.

*Validation:*::

* Review the `worldbuilding` consumer implementation.
* Review the `generic_loan_config` adoption notes for matrix-heavy business
  scenarios.
* Verify whether proposed targeting concepts would eliminate the current
  consumer-side metadata plugin.
* Verify consistency with workbook-view metadata and existing validation flows.

*Discussion Note:*::

This topic is adjacent to `FTR-DYNAMIC-FRAME-PROJECTION-P5` but not identical.

Dynamic frame projection answers:

* How is a generated view shaped?

Dynamic view column targeting answers:

* How do later concerns address the generated columns of that view?

Both questions matter for product strategy, but they operate at different
stages of the workbook-view pipeline.
