===== FTR-FK-HELPER-GROUPING-DOCS-DEMO-P5 -- Document and demonstrate collapsible FK helper columns

*Status:* New

*Purpose:*::
Update the demo, tutorial, and user-facing documentation to explain optional
collapsible FK helper columns after helper grouping is supported by metadata,
Render IR, and spreadsheet adapters.

*Context:*::
The public documentation entrypoint work made the product-business-slice demo
the maintained first-hour tutorial. Once FK helper columns can be grouped or
collapsed, the tutorial and user guide should explain the feature in the same
intent-based style:
+
* helper columns improve readability;
* helper columns remain derived;
* users may collapse them when they want a cleaner workbook view;
* collapsing is presentation only and does not affect canonical data or
  reimport semantics.
+
This FTR keeps public docs aligned with the new helper behavior without mixing
documentation changes into the runtime implementation FTRs.

*Dependencies:*::

* `FTR-FK-HELPER-COLUMN-GROUP-META-P5`
* `FTR-COLUMN-GROUP-RENDER-CAPABILITY-P5`
* Existing product-business-slice demo pipeline.
* Existing demo README and walkthrough.
* Existing user guide transformation catalog.
* Existing screenshot / walkthrough asset policy.

*Scope:*::

* Update the maintained demo pipeline to enable or deliberately mention
  helper-column grouping.
* Update the demo README tutorial steps if the workbook inspection step changes.
* Update the product-business-slice walkthrough to explain collapsible helper
  columns.
* Update the user guide transformation catalog:
  ** document the grouping configuration parameter,
  ** state that grouping is presentation-only,
  ** state that grouping does not affect reimport cleanup.
* Update technical/user docs that still describe helper columns only as visible
  adjacent columns.
* Decide whether existing screenshots remain accurate.
* Replace screenshots only if the visible workbook output materially changes or
  the old screenshots would mislead users.
* If screenshots are left unchanged, document a local re-render against the
  new grouping pipeline and a visual confirmation that the relevant areas of
  the workbook still match the existing image.
* Validate normal AsciiDoc and Reveal.js rendering behavior if screenshots are
  touched.
* If ODS column grouping is an accepted capability gap per
  `FTR-COLUMN-GROUP-RENDER-CAPABILITY-P5`, the user-facing docs must state
  the limitation honestly (e.g. "collapsible groups are available when the
  workbook is opened in Excel / openpyxl-supported tools; LibreOffice Calc /
  ODS may display the columns ungrouped") rather than implying parity.

*Non-Goals:*::

* No runtime implementation changes.
* No new RenderOp.
* No adapter changes.
* No FK relation policy changes.
* No new public tutorial path.
* No release.
* No README version-baking solution.
* No readback documentation for collapsed/expanded state.

*Expected documentation message:*::

The docs should say, in substance:

[source,text]
----
FK helper columns are derived display columns. They can be grouped and
collapsed to reduce visual noise in spreadsheet editors that support column
outlines. This is presentation metadata only: canonical data, helper
provenance, and reimport cleanup do not depend on whether the group is
expanded or collapsed in the UI.
----

*Acceptance:*::

* Demo pipeline still runs end to end.
* Demo workbook visibly contains helper columns that are grouped/collapsible
  when opened in a supported spreadsheet editor.
* Demo README and walkthrough no longer imply that helper columns are always
  plain visible columns only.
* User guide documents the grouping configuration option.
* User guide clearly states that helper grouping is presentation-only.
* Reimport tutorial still verifies that helper columns do not leak into
  canonical JSON.
* Screenshots are either:
  ** explicitly left unchanged after a documented local re-render against
     the new grouping pipeline and a visual confirmation that the relevant
     workbook areas still match the existing image, or
  ** updated with validation proving normal HTML includes the images and
     Reveal.js behavior remains intentional.
* Pages build succeeds.
* No stale wording remains that contradicts policy-driven FK helper behavior or
  optional helper grouping.

*Validation:*::

* In core:
  ** render the user guide,
  ** render affected walkthrough/docs files,
  ** run `make docs-pages`,
  ** run `git diff --check`.
* In demo:
  ** run the product-business-slice forward pipeline,
  ** open or inspect the generated workbook enough to confirm grouping metadata
     is present,
  ** perform one workbook edit,
  ** run the reimport pipeline,
  ** verify canonical JSON does not contain helper columns,
  ** run the demo roundtrip editing test.
* If screenshots are touched:
  ** verify PNG files are nonzero and within documented asset budget,
  ** render normal AsciiDoc output and count expected images,
  ** render Reveal.js output and confirm expected image behavior.

*Residual risks:*::

* Some users may not notice spreadsheet outline controls immediately. The
  tutorial should mention the feature without making it the central concept.
* LibreOffice Calc and Excel may expose collapsed groups with slightly
  different UI affordances.
* If ODS grouping is an accepted capability gap, the docs must state the
  backend limitation honestly rather than implying false parity.
