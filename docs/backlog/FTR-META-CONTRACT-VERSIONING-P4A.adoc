===== FTR-META-CONTRACT-VERSIONING-P4A -- Establish meta contract versioning direction before 0.2.0

*Status:* New
*Priority:* P4A
*Area:* Metadata lifecycle / compatibility / release evolution

==== Purpose

Establish a minimal compatibility direction for persisted `_meta` before the
`0.2.0` release line, without introducing a full metadata migration framework
prematurely.

The recent runtime-meta persistence bug showed that persisted `_meta` cannot be
treated as timelessly valid runtime input. Metadata is not only descriptive; it
can carry executable semantic contracts, projection state, user-authored
presentation intent, or runtime-produced compiled policy. Release and schema
evolution therefore need at least a minimal contract boundary.

This FTR defines the problem space and the intended direction. It should not
be expanded into a broad pre-release migration project.

==== Context

The project currently persists `_meta` in sidecars and workbook metadata
carriers, but persisted `_meta` is not yet tagged with a stable meta-contract
or schema version.

Some internal namespaces already have local schema markers, for example FK
helper policy v1/v2 shapes. These are useful local evolution markers, but they
do not define the compatibility contract of the persisted `_meta` document as
a whole.

The `BUG-RUNTIME-META-PERSISTENCE-BOUNDARY-P4A` fix introduced a write-side
persistence boundary to prevent selected runtime-produced metadata from leaking
into future persisted artifacts. That fix does not solve the more general
question of how a future runtime should interpret `_meta` written by older
library versions.

Before `0.2.0`, the user base is intentionally small and the consumer-facing
surface is still stabilizing. The project should therefore avoid investing
large effort into migration support for pre-contract beta metadata. At the same
time, `0.2.0` should start a path that can later support a stable long-term
user experience once the application has matured.

==== Problem Statement

Persisted `_meta` needs a versioned compatibility contract.

Without such a contract, future runtimes cannot reliably distinguish:

* current compatible metadata;
* older metadata that requires migration;
* unsupported legacy metadata;
* pre-contract beta metadata that should not be trusted;
* metadata written by a newer runtime that the current runtime may not
  understand.

Assuming that persisted `_meta` will continue to work unchanged across release
boundaries is unsafe. The recent metadata persistence issue demonstrated that
stale metadata can be replayed as active runtime semantics and cause incorrect
behavior.

==== Design Direction

Introduce a top-level meta contract marker, conceptually similar to API schema
versioning.

Example shape:

[source,yaml]
----
meta_contract:
  schema_version: 1
  written_by:
    package: spreadsheet-handling
    version: 0.2.0
----

The schema version, not the package release version, is the primary
compatibility discriminator.

The package release version is useful for diagnostics, release archaeology, and
future migration decisions, but compatibility should not depend directly on
every package release. Multiple package releases may share the same
`meta_contract.schema_version`.

==== Intended Compatibility Model

The read path should eventually behave like this:

[source,text]
----
read raw frames + raw _meta
detect meta_contract.schema_version
if current schema:
    accept
elif supported older schema:
    migrate to current schema
elif untagged pre-contract metadata:
    apply legacy policy
elif unsupported or newer schema:
    fail clearly or drop unsafe metadata according to explicit policy
run configured pipeline steps
project runtime meta to persistable meta contract
write current schema
----

The write path should always write the current supported meta contract version.

Migration should be explicit. The project should not attempt best-effort
migration unless a concrete migration rule is implemented and tested.

==== Pre-0.2.0 Policy

For metadata without `meta_contract.schema_version`, treat it as pre-contract
legacy metadata.

Because there is currently no broad stable user base and no long-term guarantee
for pre-0.2.0 beta metadata, the project should minimize compatibility effort
for untagged metadata.

Default policy for pre-0.2.0 untagged `_meta`:

* do not blindly trust it as current semantic contract;
* discard or quarantine unsafe metadata before configured pipeline steps;
* preserve only explicitly safe metadata if a narrow rule exists;
* emit a clear diagnostic where appropriate;
* do not build a broad migration system for historical beta artifacts.

This is a deliberate scope decision, not a general long-term stance toward user
data.

==== Future Policy

For future tagged metadata, the project should keep a path open toward
long-term stable user data.

Possible future behavior:

* support selected older `meta_contract.schema_version` values;
* migrate supported old schema versions stepwise to the current schema;
* write only the current schema version;
* when a schema version falls out of support, treat it as explicit legacy input;
* do not silently consume unsupported metadata as if it were current.

Skipped releases should be handled by schema-version migration chains, not by
package-release-specific branching.

Example:

[source,text]
----
v1 -> v2 -> v3
----

If a user opens metadata written with schema `v1` using a runtime whose current
schema is `v3`, the runtime may apply `v1 -> v2` and then `v2 -> v3`, if both
migrations exist.

If a required migration step does not exist, the runtime should fail clearly or
drop only explicitly unsafe metadata according to documented legacy policy.

==== Scope For This FTR

* Define the top-level meta contract direction.
* Decide that `0.2.0` is the first release line that writes a meta contract
  marker.
* Define untagged `_meta` as pre-contract legacy metadata.
* Define a minimal read-side compatibility boundary concept.
* Document that schema-version compatibility is separate from package release
  versioning.
* Document that full migration support is intentionally deferred until concrete
  schema evolution requires it.

==== Non-Goals

* No full `_meta` lifecycle inventory in this FTR.
* No full migration framework before `0.2.0`.
* No guarantee to migrate all pre-0.2.0 beta metadata.
* No release-by-release compatibility matrix.
* No attempt to preserve every historical beta `_meta` shape.
* No broad redesign of FK-helper metadata.
* No registry-as-runtime-rule-engine.
* No commitment to long-term support windows before the user-facing surface has
  matured.

==== Acceptance

* The backlog records a clear meta-contract versioning direction before
  `0.2.0`.
* The project distinguishes package release version from metadata schema
  version.
* Untagged `_meta` is explicitly classified as pre-contract legacy metadata.
* The intended post-`0.2.0` path leaves room for stable user data and explicit
  migrations.
* The FTR makes clear that compatibility effort for pre-0.2.0 beta metadata is
  intentionally minimized.
* Documentation does not imply that all persisted `_meta` written by older beta
  versions remains valid forever.
* The planned approach is compatible with the existing persistence-boundary
  model:
  ** read-side compatibility boundary before configured pipeline steps;
  ** write-side persistence boundary before saving.

==== Validation Ideas

Initial validation can stay small:

* Unit test that newly written `_meta` receives `meta_contract.schema_version`.
* Unit test that untagged `_meta` is detected as pre-contract legacy metadata.
* Integration test that a read/write cycle writes current meta-contract metadata.
* Documentation check that release notes and semantic model describe the
  contract-versioning boundary.

Full migration-chain validation is explicitly deferred until a real schema
version change exists.

==== Relationship To Existing Work

* `FTR-META-LIFECYCLE-INVENTORY-P5`

This FTR does not replace those items. It defines the release/schema
compatibility axis that future metadata lifecycle work should respect.

==== Open Questions

* Should pre-contract untagged `_meta` be fully discarded, or should a very
  small safe subset be preserved?
* Should the read-side compatibility boundary live in the same module family as
  the write-side persistence boundary?
* Should diagnostics be warnings, findings, or hard errors for legacy metadata?
* Which metadata roots, if any, are safe enough to preserve from untagged
  pre-contract beta artifacts?
* At which maturity point should the project define support windows for older
  meta schema versions?
