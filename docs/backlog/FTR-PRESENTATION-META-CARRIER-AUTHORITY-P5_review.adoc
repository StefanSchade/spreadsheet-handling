=== B1 Disposition

Resolved.

Commit `c7effc3` fully eliminates the stale XLSX
`SheetIR.meta["__column_widths"]` path identified in the previous review.

The fix removes the embedded-meta seeding of XLSX `column_widths` from
`build_visible_sheet_ir`. That function now deliberately leaves
`__column_widths` unset; `openpyxl_parser.parse_workbook` remains the only XLSX
parser path that writes `sheet_ir.meta["__column_widths"]`, and it does so only
when visible carrier extraction returns non-empty column widths.

Evidence:

* `src/spreadsheet_handling/io_backends/xlsx/parser_interpretation.py`, lines
  90-99.
* `src/spreadsheet_handling/io_backends/xlsx/openpyxl_parser.py`, lines
  53-73.

when the visible carrier extraction is empty, so the in-memory IR cache and
hidden `_meta.workbook_meta_blob` now agree.

Manual reproduction of the previous failure now returns:

== Findings

=== B1: XLSX `column_widths` stale metadata can survive in `SheetIR.meta`

Severity:: High
Category:: Defect / carrier-authority semantics
Disposition:: Must fix before closing this FTR.

Observation::

The XLSX parser can still return stale `__column_widths` in the visible
`SheetIR.meta` even after the new shared helper clears
`_meta.sheets.<sheet>.column_widths` from the embedded workbook-meta blob.

Evidence::

`parse_workbook` builds `meta_hints` from embedded workbook metadata before
the carrier-authority helper runs:

* `src/spreadsheet_handling/io_backends/xlsx/openpyxl_parser.py`, lines
  63-69.

`build_visible_sheet_ir` then copies embedded `column_widths` hints into the
visible sheet IR:

* `src/spreadsheet_handling/io_backends/xlsx/parser_interpretation.py`, lines
  90-92.

The new helper is invoked later and clears the embedded workbook-meta payload,
but it does not remove the stale in-memory `sheet_ir.meta["__column_widths"]`:

* `src/spreadsheet_handling/io_backends/xlsx/openpyxl_parser.py`, lines
  72-99.

Manual reproduction with an XLSX workbook containing:

* a visible `Data` sheet with default column dimensions; and
* hidden `_meta.workbook_meta_blob` carrying stale
  `sheets.Data.column_widths`

produced this parser state:

[source,text]
----
ir.sheets["Data"].meta == {
    "__column_widths": {"A": {"width": 18.0, "source": "workbook"}}
}

ir.hidden_sheets["_meta"].meta["workbook_meta_blob"] == '{"sheets":{"Data":{}}}'
----

The render-flow consequence is concrete: `build_render_plan(ir)` emitted:

[source,text]
----
SetColumnWidth(sheet='Data', col=1, width=18.0)
----

Risk::

A direct XLSX parse-to-render path can still reapply a custom column width
that the user cleared from the visible workbook carrier. This violates the
FTR's authoritative-carrier goal for `column_widths` and leaves one stale-meta
bug shape open.

Recommended action::

Fix the XLSX parse path so an empty extracted `column_widths` removes stale
`sheet_ir.meta["__column_widths"]`, or stop seeding XLSX `SheetIR.meta` from
embedded `column_widths` hints before visible carrier extraction decides
authority.

Add a regression that builds the stale XLSX column-width fixture, calls
`parse_workbook()`, asserts `__column_widths` is absent from
`ir.sheets["Data"].meta`, and preferably asserts that `build_render_plan(ir)`
emits no `SetColumnWidth`.
