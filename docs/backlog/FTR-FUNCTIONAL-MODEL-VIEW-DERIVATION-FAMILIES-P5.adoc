===== FTR-FUNCTIONAL-MODEL-VIEW-DERIVATION-FAMILIES-P5 -- Model view and projection derivation families

*Status:* New

*Type:* Functional Model documentation.

*Purpose:*::
Catch up the Functional Model for implemented view/projection derivation
families that are currently only partially or indirectly represented.

*Context:*::
Derived from the June 2026 transformation/capability family inventory:
`docs/cold_storage/working_context/transformation_family_inventory_2026_06/transformation_family_inventory.adoc`.
The inventory identifies `extract_frame`, `join_views`, `tabular_views` /
`pivot_frame`, `workbook_views`, and `dynamic_projection` / `project_by_role`
as implemented families with partial or missing Functional Model coverage.

This FTR records actual Functional Model catch-up work. The inventory remains in
cold storage; Project Memory remains a navigation/intent anchor only.

*Scope:*::
* Add or revise Functional Model material that explains view derivation as a
  family of human-oriented working forms over canonical frames.
* Cover the implemented roles of:
  `extract_frame`, `join_frames`, `pivot_frame`, `configure_workbook_view`,
  `apply_workbook_view_sheet_mappings`, and `project_by_role`.
* Clarify boundaries between generic frame derivation, declarative joins,
  pivoted display views, workbook sheet/view mapping, and role-based dynamic
  projection.
* Align terminology with existing Functional Model problem-space chapters under
  `docs/functional_model/ch05_problem_space_overview/`.
* Reference the inventory and existing implementation FTRs for provenance
  rather than duplicating implementation evidence.

*Non-Goals:*::
* No runtime implementation.
* No changes to projection, workbook-view, join, extract, or pivot code.
* No new projection mechanism, selector grammar, or role taxonomy.
* No reopening of completed dynamic-frame or dynamic-view targeting decisions.
* No Project Memory entities.
* No copy of the inventory table or coverage matrix.

*Acceptance Criteria:*::
* The Functional Model has a clear view/projection derivation explanation that
  includes the listed implemented families.
* The material distinguishes frame derivation, workbook-view declaration,
  readback mapping, and role-based projection.
* Stale terminology such as historical `join_view` examples is reconciled with
  current registered step names where appropriate.
* The June 2026 inventory is cited as the evidence source without duplicating
  its contents.
* The FTR remains documentation-only; no runtime implementation is performed
  unless separately approved.

*Relationship to the inventory and Functional Model:*::
The inventory provides implemented-family coverage status. This FTR turns the
view/projection gap into bounded Functional Model work and keeps concrete
implementation details out of the Functional Model unless they are necessary to
explain user-facing semantics.
