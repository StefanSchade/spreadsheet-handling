== Verdict

*CELL CODEC PROJECTION-FAMILY SLICE NOT ACCEPTED.*

The shared tabular helper is a sound small primitive, XRef's migration is
behavior-preserving, the four Cell Codec paths now reject unsafe *physical
source* labels before pandas operations, and Review 001 findings I1 and I2 are
closed. The explicit-only metadata retirement also remains correct.

Acceptance is nevertheless withheld because both Review 001 blockers remain
partially open:

. B1 does not validate the historical paths' generated output selectors
  (`decode.code` and `encode.value`), allowing missing output labels and raw
  `TypeError`/ambiguous-truth failures; and
. B2's containment check does not prevent a multi-character separator from
  being formed across a token/marker boundary, so accepted encode can still
  decode to different fields silently.

I3's runtime precedence defect is closed, but the private sentinel now leaks
through the public Python signatures. I4 is only partially closed because live
User Guide and Compact Multiaxis Functional Model statements remain inaccurate.
No accepted architectural decision needs to be reopened.

== Shared tabular helper assessment

`domain/tabular/physical_labels.py` retains the exact intended physical-frame
contract:

* non-missing;
* deterministically comparable;
* hashable;
* unique and therefore scalar-addressable.

The effective order is exactly missing-like, ambiguous equality, unhashable,
then duplicate detection. Independent probes produced deterministic domain
`ValueError` diagnostics for `None`, `NaN`, `pd.NA`, duplicate `NaN`, duplicate
`pd.NA`, a Python list, a multi-element NumPy array, duplicate strings, and
duplicate hashable tuples. Unique strings, numeric labels, and hashable tuple
labels pass the boundary.

The module is a narrow 187-line domain validation leaf. It imports pandas only
to recognize scalar/missing labels and reuses the existing local equality
primitive. It introduces no DataFrame wrapper, table abstraction,
serialization, metadata, identity ontology, or arbitrary-object framework.

`ensure_unique_field_declaration` is deliberately only a uniqueness check. It
does not by itself establish scalar addressability: for example two `pd.NA`
selectors and two NumPy arrays are not diagnosed as duplicates because their
equality is not a usable boolean. That is safe only when a caller subsequently
applies `is_scalar_addressable_label` before membership/selection. Cell Codec
does so for participating, group, and passthrough lists. XRef continues to
apply its family-specific physical/matrix-axis boundaries and uses the shared
function only for ordinary `row_keys` uniqueness. No raw error or behavior
change was found in the accepted XRef paths.

This separation is coherent: configured-list uniqueness is not falsely made a
new generic physical-label or matrix-axis identity policy. The remaining Cell
Codec B1 defect is a missing call-site validation for scalar *generated output
labels*, not a defect in the shared primitive.

== XRef behavior-preserving migration assessment

The XRef implementation replaces the previously local physical-label and
field-declaration bodies with thin aliases to the shared helpers. The removed
bodies and shared implementations retain the same validation order,
diagnostics, and value rules. The only call-site syntax change is the shared
helper's keyword-only `field_name`.

Focused XRef tests and independent probes verify:

* missing-like, ambiguous, unhashable, duplicate, numeric, and tuple physical

* matrix-axis identities remain unique non-empty strings;
* ordinary duplicate `row_keys` still fail;
* valid contraction with `drop_source=True` still writes exactly
  `pipeline_cleanup.drop_frames: [rel]`;
* invalid cases remain output-, metadata-, cleanup-, and mutation-free; and
* the Review 006 selected-unhashable `base_relation` regression still fails at
  physical validation before output-name collision construction, with no
  output or input mutation.

No XRef diagnostic, cleanup, metadata, carrier identity, or validation-order
regression was introduced by `715eeed`.

== B1 closure across the four Cell Codec paths

All four implementations now load the source frame and immediately call the
shared physical-label boundary before configured-field membership, pandas
selection, grouping, row indexing, passthrough construction, or output
installation. Independent probes on position encode, position decode,
historical encode, and historical decode confirm deterministic failure for:

* duplicate ordinary strings;
* `NaN` / duplicate `NaN`;
* `pd.NA` / duplicate `pd.NA`;
* an unhashable list label; and
* a multi-element NumPy-array label.

Unique numeric physical labels remain valid where the family path uses them.
Tuple/MultiIndex surfaces remain rejected earlier by Cell Codec's explicit
flat-frame guard, as intended.

The exact Review 001 corruption regression is closed. Historical encode over a
frame with duplicate physical `NaN` columns, with `group_by`/`code` selecting
that label, raises the shared missing-like physical-label `ValueError`. It
returns no output, writes no codec metadata or cleanup command, and leaves the
source frame and nested sentinel metadata unchanged. No pandas `Series`
representation is serialized.

Configured position `participating_columns`, historical `group_by`, and
historical decode `passthrough_columns` are validated as unique and
scalar-addressable before pandas membership. Historical encode's *source*
`code` selector and historical decode's *source* `value` selector also use the
single-label boundary.

B1 is not fully closed because the inverse-role scalar selectors are omitted:
historical decode does not validate the generated `code` output label, and
historical encode does not validate the generated `value` output label. Exact
counterexamples are recorded under R002-B1 below.

== B2 position-grammar assessment

The correction properly rejects declarations where:

* `separator in absent_value`;
* `absent_value in separator`; or
* separator and absent marker are equal.

The exact Review 001 example (`separator="--"`, `absent_value="-"`, values
`["A", "", "B"]`) now fails before row processing and output. A valid
single-character separator with a non-overlapping marker roundtrips ordinary,
empty, and missing values. Real values equal to the absent marker and real
values containing the complete separator still fail before output.

The stated local proof is incomplete for multi-character separators. A token
may contain no *complete* separator while ending with a proper separator
prefix. Joining then forms an earlier separator occurrence across the token /
delimiter boundary. For example:

[source,python]
----
intent = {
    "participating_columns": ["a", "b", "c"],
    "compact_column": "packed",
    "separator": "--",
    "absent_value": "~",
}
values = {"a": "A-", "b": "B", "c": "C"}
----

Encode accepts the declaration and row because `"--" not in "A-"`; it emits
`A---B--C`. Decode succeeds but recreates `{"a": "A", "b": "-B", "c":
"C"}`. The structure is silently different.

The same problem affects an absent marker ending in a separator prefix. With
`separator="--"`, `absent_value="x-"`, and values `["A", "", "B"]`, encode
emits `A--x---B` and decode recreates `["A", "x", "-B"]` rather than the
empty middle value.

This is an ordinary-string counterexample inside the current no-escaping
grammar. It does not require quoting, escaping, typed values, or pathological
objects. B2 therefore remains blocking.

== I1 historical delimiter assessment

I1 is closed. Both scalar entry points run `_ensure_delimiter` immediately
after mode validation when `mode="split_tokens"`, before conversion or empty
fast paths.

The following all reject `delimiter=""` deterministically with no output,
metadata, cleanup, or input mutation:

* encode with an all-empty group;
* direct serialization with no structured values;
* decode of an empty cell with `drop_empty=False`; and
* decode of an empty cell with `drop_empty=True`.

Valid non-empty split-token encode/decode remains green. `whole_cell_code`
does not consume the delimiter and its punctuation behavior is unchanged.

== I2 passthrough and output-shape assessment

I2 itself is closed. Historical decode materializes `passthrough_columns`,
checks declaration uniqueness, then checks scalar addressability before
membership or output. Duplicate ordinary passthrough fields, `pd.NA`, a list,
and a NumPy-array selector all receive deterministic Cell Codec errors.

Ordinary collisions between passthrough fields and the generated `code` name
still fail. Reusing the compact source `value` name as generated `code` is
deterministic when that source field is not also retained: the compact field is
replaced by the decoded field rather than duplicated. Explicitly retaining a
same-named passthrough is rejected.

No accepted *ordinary-string* declaration tested produces duplicate output
labels. The missing/unhashable generated-label residual in R002-B1 is separate
from duplicate passthrough closure.

== I3 mutual-exclusion and sentinel assessment

Runtime mutual exclusion is correctly enforced. Untouched historical defaults
do not count as supplied intent. Every explicitly supplied historical argument
alongside `codec_intent` fails, including values equal to semantic defaults:

* encode: `group_by`, `code`, `value`, `mode=None`, `delimiter="-"`, both
  allowed sets, legend reference, canonical order, normalization, and
  `strip=False`;
* decode: `value`, `code`, `passthrough_columns=None`, `drop_empty=True`,
  `mode=None`, `delimiter="-"`, both allowed sets, legend reference,
  normalization, and `strip=False`.

Position-only and historical-only direct calls pass. Pipeline binding omits
untouched parameters, position configuration binds successfully, historical
encode binds with conditionally required `group_by`, and an explicitly
configured historical default alongside position intent reaches the same
mutual-exclusion error. `compact_multiaxis` supplies no `codec_intent` and
continues down the historical branch.

The sentinel does not leak into error messages, registries, YAML binding, or
wrapper calls. It does leak through public Python introspection. The signatures
now expose every historical parameter as `Any = <unset>` rather than its
documented type and semantic default. For example:

[source,text]
----
encode_cell_values(..., group_by: Any = <unset>, code: Any = <unset>,
                   value: Any = <unset>, mode: Any = <unset>, ...)
----

That is an important public-API regression for `inspect.signature`, `help()`,
type tooling, and generated API documentation. The sentinel is a reasonable
runtime mechanism, but the public surface must either expose the conditional
contract intentionally or restore useful typed/introspection metadata without
weakening the explicit-default conflict rule. This does not require a pipeline
binding redesign.

== Remaining findings by severity

=== Blocking defects

*R002-B1 -- Historical generated output labels bypass scalar-addressability
validation.*

Exact calls:

[source,python]
----
decode_cell_values(
    {"s": pd.DataFrame([{"id": "r", "value": "A"}])},
    source="s", output="out", mode="whole_cell_code", code=[1, 2],
)

encode_cell_values(
    {"s": pd.DataFrame([{"g": "G", "code": "A"}])},
    source="s", output="out", group_by="g", mode="whole_cell_code",
    value=[1, 2],
)
----

Observed: each raises raw `TypeError: unhashable type: 'list'`. Replacing the
selector with `pd.NA` raises raw `TypeError: boolean value of NA is ambiguous`.
Replacing it with `None` succeeds and returns an output whose generated column
label is pandas-normalized to `NaN`.

Expected: `decode.code` and `encode.value` must reach the same deterministic
Cell Codec scalar-addressability boundary as source selectors before collision
membership, record-dict construction, or DataFrame output construction.
Output/cleanup/metadata/mutation: list/`pd.NA` cases return no output and leave
input, metadata, and cleanup unchanged; `None` returns an invalid output frame,
with no metadata or cleanup and no input mutation. Affected code paths:
`_decode_legacy_code_rows` after `value`/passthrough validation and
`_encode_legacy_code_rows` after `code`/group validation. Severity: blocking.
No human decision is required.

*R002-B2 -- Multi-character separator occurrences can be formed across token
boundaries.*

Exact input is the `separator="--"`, `absent_value="~"`, values
`["A-", "B", "C"]` position call shown above. Encode returns `A---B--C`;
decode succeeds with `["A", "-B", "C"]`.

Expected: every accepted position encode must decode to the same ordered
fields under identical intent. Observed: silent structural corruption.
Output/cleanup/metadata/mutation: encoded and decoded outputs are returned;
there is no codec metadata or cleanup, and the input remains unchanged.
Affected path: `_position_intent` containment proof ->
`_encode_compact_value` complete-separator check -> join/split boundary in
`_decode_compact_value`. Severity: blocking. No human decision or escaping
design is required; the existing no-escaping contract needs a complete
boundary-safety validation.

=== Important findings

*R002-I3 -- Private sentinel leaks into the public Python API signature.*
`inspect.signature` / `help()` show `Any = <unset>` for every historical
parameter instead of the documented types and defaults. Runtime and pipeline
semantics are correct, but direct API introspection and generated tooling see a
private implementation detail. Keep the explicit-default conflict rule while
restoring or explicitly documenting a usable public signature.

*R002-I4 -- Live documentation and registry claims remain inconsistent.* The
User Guide's grammar rule and historical examples, Compact Multiaxis registry
anchors, registry/slice/epic soundness claims, and follow-up FTR safety/sequence
claims are inaccurate as detailed above. Correct them only after runtime
closure so the documents state the verified boundary.

=== Minor findings

None separate from the stale XRef sequencing sentence included in R002-I4.
