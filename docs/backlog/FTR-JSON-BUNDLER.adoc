===== FTR-JSON-BUNDLER -- Bundle nested 1:n JSON structures into spreadsheet-facing child tables

*Context:*::
Nested JSON is not uniformly difficult. Nested singleton objects can usually be
flattened into dotted paths or multi-header columns without breaking the basic
tabular model. The harder case starts when objects contain lists, especially
lists of objects, because a single parent record then expands into multiple
child records.

*Problem statement:*::
For nested 1:n structures we need a spreadsheet-facing layout strategy that
turns hierarchical JSON into multiple related tables. The resulting workbook
should expose these as normal tabular structures linked by synthetic foreign
keys, rather than pretending that one row can directly represent the whole
hierarchy.

The open design questions are exactly what makes this feature risky:

* whether child tables always get their own sheet or may share a sheet,
* how aggressively cascading nested structures are allowed to fan out into many
  tables,
* how naming, scoping, and traceability are represented in meta,
* and how far the first implementation should generalize beyond the immediate
  use cases.

*Scope:*::

* define the spreadsheet-side bundling strategy for nested lists / child tables,
* record the parent/child linkage model in meta,
* provide a bounded first policy for sheet placement and table naming,
* prepare the ground for roundtrip-safe recomposition by
  <<FTR-NESTED-LISTS#,FTR-NESTED-LISTS>>.

*Non-goals:*::

- No attempt to treat nested singleton objects as a separate major feature;
  those remain in the flattening / multi-header design space.
- No requirement that arbitrary recursive structures are rendered in a single
  aesthetically optimal workbook layout.
- No promise that the first version solves every placement strategy (same sheet
  vs dedicated sheet, grouping, collapsing, etc.).

*Acceptance:*::

* For JSON structures containing lists of objects, the system can emit multiple
  related tabular outputs linked by synthetic keys.
* The initial layout strategy is explicit and documented rather than emergent.
* Meta records enough linkage information to support later recomposition.
* A bounded policy exists for limiting or configuring cascade fan-out.

*Relationship to other FTRs:*::

* nested singleton objects remain largely in the flattening / multi-header
  space and are not the main difficulty here,
* this FTR deals with the spreadsheet/table layout strategy,
* <<FTR-NESTED-LISTS#,FTR-NESTED-LISTS>> deals with the normalization and
  recomposition logic built on top of that strategy.
