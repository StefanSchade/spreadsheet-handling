===== FTR-COLUMN-GROUP-RENDER-CAPABILITY-P5 -- Backend-neutral collapsible column groups

*Status:* New

*Purpose:*::
Add a backend-neutral Render IR capability for collapsible column groups, so
spreadsheet backends can render optional column outlines without receiving
domain-specific helper semantics.

*Context:*::
FK helper columns are useful for business-readable workbooks, but they can also
add visual noise. Spreadsheet users should be able to collapse helper columns
when they want to focus on canonical business fields.
+
This must be modeled as a rendering/presentation capability, not as FK-helper
logic inside spreadsheet adapters. The domain/helper layer may later declare
that a set of columns should be grouped, but adapters must only execute a
backend-neutral render instruction.
+
The architecture boundary is important:
+
* Domain/configuration writes intent.
* Render IR turns intent into backend-neutral render operations.
* Spreadsheet adapters render those operations locally.
* XLSX and ODS implementations must not depend on each other's carrier model.

*Dependencies:*::

* Existing Render IR / RenderPlan architecture.
* Existing XLSX and ODS spreadsheet adapters.
* Existing render-op execution model.
* Existing backend capability documentation and architecture guards.
* Should be scheduled after FK helper relation/policy cleanup, but does not
  itself depend on FK-helper semantics.

*Scope:*::

* Add a backend-neutral column grouping / outline render operation.
* Add the minimum IR/pass support needed to carry column-group instructions
  from composed workbook metadata to the RenderPlan.
* Define the operation without FK-specific terminology.
* Support at least:
  ** sheet name,
  ** first grouped column,
  ** last grouped column,
  ** outline level,
  ** collapsed-by-default flag,
  ** hidden-by-default flag if needed by backend semantics.
* Implement XLSX rendering of the operation in the XLSX adapter.
* Implement ODS rendering if straightforward.
* If ODS support is not straightforward, document an explicit capability gap
  without changing the generic RenderOp to match ODS carrier quirks.
* Add tests proving the RenderPlan carries the operation and that backend
  renderers handle it without crossing abstraction boundaries.

*Capability matrix:*::

[cols="1,1,4", options="header"]
|===
| Backend | Status | Notes

| XLSX
| Required
| Implement through adapter-local XLSX/openpyxl column-outline support.

| ODS
| Required if straightforward; otherwise accepted explicit gap
| Must not distort the generic RenderOp to mimic ODS-specific carrier shape.
  If unsupported in this slice, the adapter must ignore/fail according to the
  documented backend capability policy.

| CSV / JSON / YAML / XML
| Not applicable
| Non-spreadsheet backends do not render presentation outlines.
|===

*RenderOp sketch:*::

[source,python]
----
@dataclass(frozen=True)
class SetColumnGroup:
    sheet: str
    first_col: int
    last_col: int
    level: int = 1
    collapsed: bool = False
    hidden: bool = False
----

The final name and exact field names may differ, but the operation must remain
backend-neutral and column-index based by the time it reaches adapters.

*Non-Goals:*::

* No FK-helper-specific behavior.
* No automatic grouping of helper columns.
* No domain or pipeline step changes.
* No readback/parsing of column-group state from XLSX or ODS.
* No public demo or tutorial update.
* No cross-adapter shared carrier helper that imports XLSX assumptions into
  ODS or ODS assumptions into XLSX.
* No attempt to support row groups.
* No generic UI-state framework.

*Acceptance:*::

* A backend-neutral column-group RenderOp exists.
* Render IR can produce column-group operations without referencing XLSX or ODS
  APIs.
* XLSX rendering writes a workbook where the configured columns are grouped.
* ODS rendering either supports the same operation or records a clear accepted
  capability gap.
* Adapter implementation remains adapter-local; no XLSX/ODS carrier-model
  dependency is introduced between backends.
* Architecture guards still pass.
* Tests cover:
  ** RenderPlan operation creation,
  ** XLSX adapter execution,
  ** ODS adapter execution or documented gap behavior,
  ** absence of FK-helper terminology in the generic RenderOp.
* Existing workbooks without column-group metadata render unchanged.

*Validation:*::

* Run targeted Render IR tests.
* Run targeted XLSX adapter tests.
* Run targeted ODS adapter tests or the explicit ODS gap test.
* Run architecture guards for render/backend boundaries.
* Run source-style guards if production code changes are substantial.
* Render modified technical documentation.
* Run `git diff --check`.

*Residual risks:*::

* ODS support may be less capable than XLSX. That must be documented as a
  backend capability difference, not hidden behind false parity.
* Spreadsheet applications may display collapsed/hidden outline state slightly
  differently. Tests should verify carrier structure rather than pixel-perfect
  UI behavior.
* Readback of column-group state is intentionally deferred.
