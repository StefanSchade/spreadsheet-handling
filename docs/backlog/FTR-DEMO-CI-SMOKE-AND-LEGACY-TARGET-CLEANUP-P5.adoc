===== FTR-DEMO-CI-SMOKE-AND-LEGACY-TARGET-CLEANUP-P5 -- Separate maintained demo smoke from deprecated legacy targets

*Status:* Proposed
*Priority:* P5
*Area:* Demo CI / cross-repo release governance

Tracked in the core backlog because the strategic decision -- what does demo
CI's intent represent against the maintained core surface? -- is a cross-repo
governance question. Implementation work lands in the demo repository, as it
did for the closed `FTR-DEMO-REPO-INTEGRATION-P4A`.

==== Goal

Stabilize the demo repository CI by separating maintained integration smoke
paths from deprecated legacy helper targets.

The `v0.1.0b6` release exposed that CI still depended on historical
compatibility shims (`pack`, `unpack`, `extract`) whose underlying CLI
entrypoints had already drifted or disappeared.

The CI intent of the demo repository must be clarified and aligned with the
repository's actual role.

==== Motivation

Observed during beta release stabilization:

* CI failed because `sheets-pack` no longer existed;
* CI later failed because `sheets.yaml` no longer existed;
* the failures were not caused by broken core functionality;
* the failures came from stale operational glue in the demo workflow;
* CI intent had become ambiguous between partial integration verification,
  legacy smoke testing, and workflow residue from earlier architecture
  phases.

The demo repository is not the authoritative correctness test surface for the
core implementation.

Its primary CI responsibility is verifying:

* installability;
* CLI availability;
* maintained demo-path viability;
* basic integration wiring.

==== Scope

This FTR covers:

* demo CI simplification;
* maintained smoke-path definition;
* cleanup or explicit deprecation of legacy targets;
* alignment between Makefile, workflows, and maintained documentation.

This FTR does not cover:

* deep correctness testing of the core library;
* performance testing;
* large-scale integration testing;
* semantic regression testing.

==== Acceptance

* Demo CI no longer depends on deprecated CLI entrypoints.
* CI workflow intent is explicitly documented.
* A maintained smoke path is defined and used consistently.
* Legacy targets are either removed, isolated, or explicitly documented as
  compatibility-only.
* CI failures clearly indicate whether the cause is installation, entrypoint,
  or maintained-flow.
* CI no longer contains stale workflow residue from abandoned flows.

==== Candidate Directions

Possible maintained smoke approaches:

* CLI availability smoke: `sheets-run --help`;
* maintained pipeline smoke: render one curated workbook;
* minimal import smoke: verify installed package and entrypoints.

Possible cleanup targets:

* `pack`
* `unpack`
* `roundtrip`
* `extract`

==== Notes

The pragmatic hotfix during `v0.1.0b6` replaced stale legacy smoke behavior
with a minimal CLI availability check. This FTR formalizes the long-term
direction after the emergency stabilization.
