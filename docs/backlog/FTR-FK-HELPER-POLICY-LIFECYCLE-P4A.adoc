===== FTR-FK-HELPER-POLICY-LIFECYCLE-P4A -- Classify FK-helper policy lifecycle

*Status:* New

*Type:* Improvement / Lifecycle

*Area:* Pipeline / metadata lifecycle / FK helpers

==== Context

``BUG-RUNTIME-META-PERSISTENCE-BOUNDARY-P4A`` established a narrow
persistence boundary in the orchestrator and removed the concrete carrier
that broke the worldbuilding repeated-run scenario: ``helper_policies.fk
.relations`` entries produced at runtime by ``configure_fk_helpers``.

That fix is intentionally narrow. It identifies *one* runtime-produced
shape and prunes it; it does not classify the full FK-helper policy
surface and does not decide which other parts of ``helper_policies`` may
or may not survive a roundtrip.

==== Why this FTR

``helper_policies`` mixes several lifecycle classes today:

* user-authored / model intent (lookup helpers, FK columns declared in
  YAML),
* explicit step configuration (e.g. ``configure_fk_helpers`` invocations
  with declarative arguments),
* runtime-produced compiled v2 relations (current narrow carrier; partly
  pruned by the boundary),
* inferred / heuristic relations from ``infer_fk_relations``,
* helper materialization provenance (``derived.sheets.*.helper_columns``;
  already runtime-only),
* residual v1 per-target dicts written by ``configure_fk_helpers`` as a
  producer-side compatibility shim that no runtime consumer reads.

The current fix treats ``produced_by.step == configure_fk_helpers`` as the
single discriminator. That is sufficient to close the concrete
worldbuilding failure but does not capture the broader lifecycle.

==== Scope

* Enumerate every shape under ``helper_policies`` (``lookup`` and ``fk``,
  including v1 per-target dicts, v2 relations envelope, schema_version
  semantics, provenance under ``derived``).
* Decide for each shape: persistable canonical, runtime-only, inferred,
  derived, or carrier-specific.
* Decide whether ``produced_by`` is sufficient as a runtime/canonical
  discriminator or whether an explicit lifecycle marker is required (e.g.
  a typed ``origin`` field or a separate runtime namespace).
* Align ``registries/meta_registry.yaml``, the lifecycle chapter, the
  agent rules, and any guards.
* Decide handling of legacy ``_meta.yaml`` containing pre-fix shapes.
* If needed, migrate or quarantine such legacy entries on load.

==== Non-Goals

* Do not redesign the entire ``_meta`` model -- that is the broader
  ``FTR-META-LIFECYCLE-INVENTORY-P5``.
* Do not extend the runtime fact base into a new typed config surface for
  consumers without a clear migration path.
* Do not block the upcoming release.

==== Acceptance

* The persistence boundary rules under ``helper_policies`` are derived
  from a documented classification, not from the single
  ``configure_fk_helpers`` discriminator.
* Each ``helper_policies`` shape carries a clear lifecycle label, either
  via ``produced_by`` semantics or an explicit lifecycle marker.
* Legacy ``_meta.yaml`` files written before the new classification
  either round-trip cleanly or fail with a clear diagnostic.
* The semantic model, registry, agent rules, and tests agree on the new
  classification.

==== Validation

* Repeated-run tests covering each classified shape (persistable
  survives; runtime is pruned).
* Legacy fixtures from the bug period are exercised; outcomes are
  documented.
* Registry classifications match the runtime rules.
