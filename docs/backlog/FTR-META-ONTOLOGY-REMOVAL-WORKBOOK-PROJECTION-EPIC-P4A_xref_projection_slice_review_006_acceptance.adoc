== 4. Selected-list base probe

The exact required input was reproduced:

[source,python]
----
labels = np.empty(3, dtype=object)
labels[:] = ["r", "k", [1, 2]]

frames = {
    "matrix": pd.DataFrame([{"r": "r1", "A": "x"}]),
    "base": pd.DataFrame(
        [["r9", "B", "kept"]],
        columns=pd.Index(labels, dtype=object),
    ),
    "_meta": {"sentinel": {"keep": ["unchanged"]}},
}

expand_xref(
    frames,
    matrix="matrix",
    output="out",
    row_keys="r",
    column_key="k",
    value=[1, 2],
    base_relation="base",
    drop_source=True,
)
----

Observed result:

* exception type: `ValueError`, not `TypeError`;
* diagnostic: `Frame 'base' has unhashable physical column label(s) [[1,
  2]]; physical labels must be hashable to address a scalar column`;
* no `out` frame;
* no `_meta.xref_crosstable`;
* no `_meta.pipeline_cleanup`;
* matrix and base frames equal to deep snapshots;
* sentinel metadata equal to its deep snapshot.

This directly closes the counterexample recorded by Review 005.

== 5. Validation-order assessment

Source inspection and the runtime probe confirm the effective order in
`expand_xref`:

. load and validate the matrix frame and its string header identities;
. materialize, validate, and resolve `row_keys` against the matrix;
. load `base_relation`, when configured;
. run `_ensure_unique_physical_labels` on the base frame;
. run `_ensure_output_names_do_not_collide`;
. continue value-column validation, selection, expansion, base recomposition,
  metadata production, and optional cleanup scheduling.

The selected unhashable base field therefore cannot reach set construction,
membership, pandas selection, metadata writes, or cleanup before the
physical-label boundary rejects it.

`_ensure_output_names_do_not_collide` was not modified or generalized. The
correction changes call order only and introduces no arbitrary-object
comparison helper.
