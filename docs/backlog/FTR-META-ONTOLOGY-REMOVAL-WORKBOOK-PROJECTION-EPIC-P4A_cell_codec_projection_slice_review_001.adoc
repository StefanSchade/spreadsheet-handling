== Verdict

*CELL CODEC PROJECTION-FAMILY SLICE NOT ACCEPTED.*

The architectural direction is coherent: Cell Codec is explicit-only, no
runtime producer or consumer of `_meta.cell_codecs` remains, legacy payloads
are inert pass-through sediment, schema maintenance ignores that retired root,
and `compact_multiaxis` continues to call the historical primitive explicitly.

Acceptance is withheld for two blocking implementation defects:

. the equality-only physical-label guard does not establish scalar
  addressability before pandas membership and selection; and
. the position grammar permits a separator/absent-marker combination for
  which successful encode cannot be decoded.

There are also important declaration-validation and documentation/registry
gaps. None requires reopening the accepted explicit-only metadata direction,
adding a typed codec, or refactoring `compact_multiaxis`.

== Position-based encode assessment

The normal flat-string path is substantially correct:

* `codec_intent` supplies the entire position contract;
* duplicate participating declarations, missing columns, helper participation,
  compact-column participation, and compact-column/passthrough collision are
  checked before row output;
* non-string participating values fail explicitly;
* `None`, `NaN`, `pd.NA`, and empty strings used as *cell values* map to the
  absent marker;
* a real token equal to the absent marker, or containing the separator, fails;
* records are accumulated locally and the output frame is installed only after
  every row succeeds; and
* source frames and nested metadata are not mutated.

The path is not yet safe for every accepted physical-label category. A selected
unhashable list label reaches `column not in df.columns` and raises raw
`TypeError: unhashable type: 'list'`. A selected multi-element NumPy-array label
raises the raw ambiguous-truth `ValueError`. Duplicate missing-like labels are
not recognized reliably. These are part of blocking finding B1.

The position grammar is also not closed under its own accepted declarations.
With:

[source,python]
----
codec_intent = {
    "participating_columns": ["a", "b", "c"],
    "compact_column": "packed",
    "separator": "--",
    "absent_value": "-",
}
row = {"a": "A", "b": "", "c": "B"}
----

encode succeeds with `A-----B`, but decode splits that value into empty tokens
and fails with `compact value contains empty token for column 'b'`. The current
guard rejects `separator in absent_value`, but not the inverse overlap
`absent_value in separator`. This is blocking finding B2.

== Position-based decode assessment

Decode correctly requires explicit `codec_intent` even when realistic legacy
metadata is present. It validates the compact-column existence, rejects a
participating output column already present as passthrough, enforces token
count, maps the absent marker to `""`, preserves numeric-looking tokens as
strings, and constructs output only after all rows parse.

Malformed ordinary compact strings fail before output and without mutation or
metadata writes. The same physical-label boundary defect applies before compact
column selection, and the accepted separator/absent overlap means decode cannot
consume all values emitted by encode.

== Historical code-row encode assessment

The intended historical behavior remains present:

* `mode` and `group_by` are explicit;
* duplicate ordinary `group_by` declarations fail;
* typed codes retain historical stringification (`7` becomes `"7"`);
* `split_tokens` rejects a token containing its delimiter;
* `whole_cell_code` permits delimiter punctuation;
* canonical ordering, explicit/legend-derived allowed values, normalization,
  stripping, and ordinary empty handling retain their existing tests; and
* no Cell Codec metadata is written.

The physical-label problem is more severe here. A source with two physical
`NaN` columns and both `group_by` and `code` addressing `NaN` is accepted.
`row[np.nan]` yields a `Series`; encode then emits the string representation of
that Series as the compact cell. This is silent structural corruption, not a
diagnostic.

There is also an important validation-order issue: in `split_tokens` mode with
`delimiter=""`, a group whose codes are all empty returns an output cell `""`
without ever running `_ensure_delimiter`. The same configuration fails only
when non-empty data reaches the delimiter branch. Explicit declaration validity
is therefore data-dependent.

== Historical code-row decode assessment

Explicit mode/configuration remains required; `_meta.cell_codecs` is never a
fallback. Ordinary passthrough/output-name collision, allowed code/token and
legend validation, `drop_empty`, normalization, stripping, and parsing remain
functional. No Cell Codec metadata is written.

Two important declaration-boundary gaps remain:

* `passthrough_columns=["id", "id"]` is accepted and creates an output with
  columns `["id", "id", "code"]`; this is a non-scalar/ambiguous output shape
  and produces a pandas warning when converted to records; and
* `mode="split_tokens", delimiter=""` with an empty compact cell and
  `drop_empty=False` returns an empty code row instead of rejecting the invalid
  delimiter. As with encode, validation depends on cell contents.

== Physical-label and declaration-boundary assessment

`_ensure_unique_physical_labels` compares labels through `_values_equal`, which
converts `left == right` to `bool` and treats `TypeError` or `ValueError` as
"not equal". That does not prove that a label is hashable, deterministically
comparable, unique under pandas semantics, or scalar-addressable.

[cols="2,4",options="header"]
|===
|Probe |Reviewed-source result

|Duplicate strings
|Correct deterministic Cell Codec duplicate-label `ValueError`.

|Duplicate `NaN`
|Not recognized by the guard. Position encode receives a `Series` and fails
late with a misleading participating-value diagnostic; historical encode can
emit the Series representation as data.

|Duplicate `pd.NA`
|Not recognized by the guard; public position encode raises raw
`TypeError: boolean value of NA is ambiguous`.

|Selected Python-list label
|Raw `TypeError: unhashable type: 'list'` before a deterministic codec
diagnostic.

|Selected multi-element NumPy-array label
|Raw ambiguous-truth `ValueError` before a deterministic codec diagnostic.

|Accepted and scalar-addressable in the tested position path.

|Unique hashable tuple label
|Rejected deliberately by `_require_frame`; the first slice advertises flat
labels and does not claim Tuple/MultiIndex operation support.
|===

The evidence does not establish a string-only *field-label* contract. It
establishes a flat, scalar-addressable-label intent: numeric labels work, tuple
labels are explicitly outside this slice, while the string-only rule applies to
participating *cell values*. A local boundary guard can enforce what subsequent
Cell Codec addressing requires without importing the XRef matrix-axis identity
contract.

The runtime also accepts `codec_intent` together with historical `mode`,
`group_by`, and delimiter settings, silently selecting the position path. The
pipeline registry says those parameter families are mutually exclusive. Either
the existing documented mutual exclusion must be enforced or the registry must
document the actual precedence; the current combination is inconsistent.

== Explicit-only and no-fallback assessment

The explicit-only metadata contract itself is verified:

* position encode/decode with empty metadata work and write no codec root;
* decode with realistic legacy metadata but no explicit intent fails;
* explicit intent conflicting with legacy metadata controls behavior and leaves
  the sediment unchanged;
* historical mode with legacy metadata present uses only its arguments;
* no configuration field is recovered from old metadata;
* no Cell Codec operation mutates the input DataFrame or nested metadata in the
  successful and rejected ordinary cases; and
* Cell Codec has no cleanup integration, so it does not schedule source drops.

Legacy sediment is retained because returned frame mappings preserve the input
`_meta` object unchanged. This is pass-through compatibility, not live intent.

== Schema-maintenance retirement assessment

The retirement is coherent in runtime code:

* `ReferenceRoot.CELL_CODECS` has no remaining source reference;
* the root was removed from blocked-root dispatch and added to the explicit
  ignored/out-of-scope set;
* realistic stale payloads do not block rename or drop and are not rewritten;
* blocked behavior for `compact_multiaxis`, `legend_blocks`,
  `sparse_defaults`, and `split_by_discriminator` is unchanged; and
* domain-contract validation accepts the canonical classification.

The canonical reference-surface row has the correct substantive fields
(`maintenance_role: ignored`, both policies `ignore`, and no meta-registry
entry), but its identifier remains `MRS-BLOCKED-CELL-CODECS`. That stale name is
misleading even though it does not change checker behavior.

== Carrier-roundtrip assessment

The integration test crosses a real application boundary:

`JSON directory -> explicit encode -> XLSX/ODS write -> carrier read -> explicit
decode -> JSON directory`.

It is not merely an in-memory encode/decode repetition. Both carriers preserve
ordinary strings, multiple rows, passthrough fields, empty values, and
numeric-looking strings. An independent source-forced probe additionally used
`"pre-post"` (contains the absent-marker character without equalling it),
`"1"`, `"2"`, `"003"`, empty string, and `None`; XLSX and ODS recreated the
same strings, with empty and missing both intentionally normalized to `""`.
Neither loaded carrier metadata nor the reimported sidecar contained
`cell_codecs`.

Real values equal to the absent marker and values containing the separator are
correctly rejected in memory. Carrier stability is therefore proven for the
ordinary contractual vocabulary, but the separator/absent overlap in B2 means
the declared position grammar is not generally sound yet.

== `compact_multiaxis` consumer compatibility

`expand_compact_multiaxis` passes the value column, generated code column,
passthrough columns, drop-empty behavior, mode, delimiter, allowed sets,
legend reference, normalization, and stripping explicitly to decode.
`contract_compact_multiaxis` passes group columns, code/value columns, mode,
delimiter, allowed sets, legend reference, canonical ordering, normalization,
and stripping explicitly to encode.

The wrapper removes each inner temporary relation, and the Cell Codec primitive
now persists no inner entry that could retain a temporary name. The wrapper's
own `_meta.compact_multiaxis` contract remains unchanged and deferred. Focused
compact tests plus both Worldbuilding carrier reimports pass. The slice creates
no dependency on compact metadata and does not refactor the composite family.

== Cleanup, metadata, and non-mutation assessment

Cell Codec imports neither pipeline cleanup nor cleanup command helpers. Valid
operations preserve their source and add one output frame; invalid operations
do not schedule cleanup. No reviewed path writes `_meta.cell_codecs`.

For the raw list/array/`pd.NA` failures, separator-overlap decode failure, and
invalid-delimiter probes, the input frame and sentinel nested metadata remained
unchanged and no output key was added to the input mapping. The duplicate-`NaN`
historical counterexample is worse: it returns a separate mapping containing a
corrupt output frame, while still leaving the input and metadata unchanged.

== Documentation and registry assessment

The slice record, epic status, User Guide, Technical Model classification, ADR
status note, pipeline meta contracts, and meta-registry removal correctly
describe the explicit-only direction in their newly aligned sections. They
also correctly distinguish position intent, historical explicit mode,
retired sediment, string cell values, and deferred `compact_multiaxis` work.

The following current surfaces remain inconsistent:

* the Functional Model still says `Registry entry cell_codecs` in both the
  encode and decode semantic-model references even though that registry entry
  was removed;
* the release notes' current known-limitations section still directs consumers
  to declare codecs through the `cell_codecs` *workbook metadata family*;
* `pipeline_cleanup.py` states that `cell_codecs` is preserved because inverse
  transformations need it, which is false under the explicit-only contract;
* the pipeline registry marks `encode_cell_values.group_by` universally
  required although position-based encode neither needs nor accepts it as part
  of its intent; and
* the pipeline registry calls position and historical arguments mutually
  exclusive while runtime silently applies position precedence.

The ADR's older body still discusses registry-declared Cell Codec metadata, but
its prominent dated status note explicitly classifies those statements as
historical and says a future typing layer would be a new contract. That is
adequate historical preservation rather than a live contradictory claim.

No documentation reviewed promises preservation of the original empty-versus-
missing distinction, typed position values, or a completed
`compact_multiaxis` refactor.

== Remaining findings by severity

=== Blocking defects

*B1 -- Physical-label validation does not protect pandas addressing.*

Exact counterexamples and results:

* position encode with a selected physical list label `[1, 2]` raises raw
  `TypeError: unhashable type: 'list'` in pandas membership;
* a selected NumPy array label raises raw ambiguous-truth `ValueError`;
* duplicate `pd.NA` labels raise raw `TypeError: boolean value of NA is
  ambiguous`; and
* historical encode over duplicate `NaN` labels returns a compact value equal
  to the string representation of a pandas `Series`.

Expected contract: every physical source label used by Cell Codec must reach a
local deterministic boundary that guarantees subsequent membership/selection
is scalar-addressable, and duplicate labels must fail before row processing.
Observed state: raw exceptions or corrupt output. Cleanup: none. Metadata:
unchanged, with no codec root. Input frames: unchanged. Affected path:
`_ensure_unique_physical_labels` -> `_ensure_columns` / row selection.

*B2 -- A permitted position declaration can encode a value that cannot be
decoded.*

Exact input: `separator="--"`, `absent_value="-"`, participating values
`["A", "", "B"]`. Encode returns `A-----B`; decode with the identical intent
raises `compact value contains empty token for column 'b'`.

Expected contract: every value accepted by position encode must have one
unambiguous decode under the identical explicit intent. Observed state: encode
succeeds and decode fails. Cleanup: none. Metadata and input: unchanged.
Affected path: `_position_intent` checks only `separator in absent_value`, then
`_encode_compact_value` joins tokens and `_decode_compact_value` splits them.

=== Important findings

*I1 -- Historical `split_tokens` declaration validation is data-dependent.*
Both encode of an all-empty group and decode of an empty compact cell accept
`delimiter=""` and return output because the empty fast path precedes
`_ensure_delimiter`. The explicit parameter must be validated before cell-data
shortcuts.

*I2 -- Historical decode accepts duplicate passthrough declarations.*
`passthrough_columns=["id", "id"]` creates duplicate physical output columns
instead of failing before output.

*I3 -- The two explicit contracts are documented as mutually exclusive but
runtime applies silent precedence.* Supplying `codec_intent` together with
historical `mode`, `group_by`, and delimiter settings succeeds on the position
path and ignores the historical declarations. The registry also marks
`group_by` universally required despite the supported position-based pipeline
shape.

*I4 -- Documentation and registry realignment is incomplete.* The Functional
Model, current release-note limitation, cleanup contract comment, and reference
surface identifier retain live-sounding or contradictory `cell_codecs`
statements detailed above.

=== Minor findings

None beyond the misleading `MRS-BLOCKED-CELL-CODECS` identifier already
included in I4.

== Human decisions

No new human decision is required.

The explicit-only metadata direction, string-oriented position values,
empty/missing normalization, historical typed-code behavior, and deferred
composite work are already decided. B1 needs a local physical-label boundary
appropriate to Cell Codec's actual flat scalar addressing; B2 needs the
declared no-escaping grammar to reject overlapping separator/absent markers.
I1-I4 follow mechanically from existing validation and documentation claims.

== Acceptance and composed-family readiness

The retirement of `_meta.cell_codecs` is accepted in principle and is not a
reason to retain or recreate generic metadata. The current runtime slice as a
whole is not accepted until B1 and B2 are closed and the important declaration
and documentation gaps are reconciled.

`compact_multiaxis` remains compatible with the changed primitive on shipped
data, but later composed-family work should not treat Cell Codec as a fully
validated standalone foundation yet.
