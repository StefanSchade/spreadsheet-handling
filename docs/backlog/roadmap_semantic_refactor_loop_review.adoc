== Semantic-Refactor Dependency Assessment

Pairs where semantics and refactoring depend on each other (in
descending order of coupling):

. *Phase 3a / `FTR-DOMAIN-CONSOLIDATION` (REVIEW-001 Supplement, C2).*
  The consolidation FTR's supplement still proposes a `domain/fk/`
  sub-package (`columns.py`, `helpers.py`, `checks.py`) and "FK-specific
  migration ... should be executed first". Phase 3a, however, refactors
  FK code *in place* at `domain/transformations/fk_helpers/` and
  introduces the v2 `helper_policies.fk` shape. The two plans now
  describe overlapping work on the same files with different target
  layouts. Phase 3a's outcome must win (the consolidation FTR was
  written before Phase 3a existed); the consolidation FTR's supplement
  needs a one-paragraph re-anchor.

. *Phase 3a / `FTR-DERIVED-PROVENANCE-LIFECYCLE-P4A`.* Already
  coordinated in `28d0a00` (Phase 3a owns producer fields incl.
  `target_key`; P4A owns pruning / orphan / schema-validation). No
  further action.

. *Phase 3a / `FTR-META-SCHEMA-VALIDATION-P5`.* Validation needs at
  least one *concrete* meta shape to validate. The Phase 3a v2
  `helper_policies.fk` shape is the natural pilot. Without a pilot,
  this FTR risks becoming a validator with nothing to validate.

. *Phase 4 semantics / Phase 5 `FTR-GUARDRAIL-AUDIT-P5`.* An audit of
  guards is only meaningful once the semantic invariants the guards
  protect have settled. Doing the audit before the Phase 3a / 4
  primitive refactors closes risks tearing out guards that are about
  to become correct again.

. *Phase 4 `FTR-CONCEPT-DOCS-CLEANUP-P5` / every other Phase 4 FTR.*
  The cleanup FTR is currently first in Phase 4 and is scoped as
  pure edit-only ("Do not add new product behavior or architecture
  changes"). Done first, it will re-produce the staleness it is
  trying to fix as soon as the other Phase 4 FTRs land.

Pairs that are *safe to do linearly* (low semantic coupling):

* Phase 5 `FTR-REVIEW-001-MODULE-HEADERS-P5` -- one-line header
  convention; no semantic dependency.
* Phase 5 `FTR-REVIEW-001-MODULARIZATION-P4A` -- structural splits
  with stable `__all__`; semantic-neutral.
* Phase 6 backend FTRs -- ODS hardening, packaging.
* Phase 7 transformation FTRs -- new features, not refactors of
  existing semantics.
* Phase 8 / 9.

== Recommended Execution Model

For the metadata / registry workstream only (Phase 3a, the Phase 4
producer-shape FTRs, and any Phase 5 FTR that touches the same
producers / primitives), adopt the following loop as documented
practice:

[source,text]
----
clarify minimal semantics  (one root, one decision, no governance framework)
  -> add guard / test       (the rule that the new semantics imposes)
    -> narrow refactor      (only the code that crosses the clarified boundary)
      -> docs / demo update  (only if a public surface changed)
        -> stop
----

Boundary conditions:

* A loop is one FTR or a tightly-coupled FTR pair, not a phase.
* A loop ends when the guard passes, the demo runs, and no further
  refactor is justified by the *same* semantic clarification. New
  refactor work needs a new loop with a new minimal semantic
  clarification.
* If a loop spans a phase boundary (e.g. Phase 3a producer +
  Phase 4 lifecycle), name the handoff in both FTRs.
* For workstreams that do *not* touch metadata / registry roots,
  keep the existing linear phase model. The loop pattern is the
  exception for high-semantic-coupling work, not a new default.

This pattern is *already operationalized* by:

* "FTR Slicing for Metadata and Registry Changes" in
  `docs/developer_guide/ch08_git_and_workflow/01_git_and_workflow.adoc`;
* the copy-paste checklist in the same section;
* the Phase 3a preamble in `docs/backlog/_ftrs_open.adoc` (worked
  example).

Naming it explicitly in two more places (Phase 4 preamble +
Current Direction note) is the only doc work needed; no new
chapter, no new file.

== Findings

=== High

==== High-1: `FTR-DOMAIN-CONSOLIDATION` REVIEW-001 Supplement collides with Phase 3a

*Evidence:* The supplement (lines 39-69 of
`docs/backlog/FTR-DOMAIN-CONSOLIDATION.adoc`) proposes:

[source,text]
----
domain/fk/columns.py   - FK_PATTERN, detect_fk_columns
domain/fk/helpers.py   - build_registry, apply_fk_helpers
domain/fk/checks.py    - FK validations (from domain/validations/fk_helpers.py)
Dissolve core/fk.py entirely; trim core/__init__.py accordingly.
----

and states "FK-specific migration ... should be executed first" and
"Complete FTR-REVIEW-001-META-REGISTRY-DERIVED-P3 first -- `fk_helpers.py`
is a `derived`-channel producer, and the contract should be on record
before the file is moved."

Phase 3a refactors FK code in place at
`src/spreadsheet_handling/domain/transformations/fk_helpers/` (already
exists) and changes the producer contract (v2 `helper_policies.fk`,
`target_key` in derived provenance). It does *not* move the code to
`domain/fk/`, and it explicitly does *not* remove `FK_PATTERN`
(non-goal in `FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5`).

*Why it matters:* When Phase 3a closes, the consolidation FTR will
either drag the just-refactored FK code into a *different* sub-package
(churn for no semantic gain) or be silently outdated. Either way, the
next agent who reads the supplement will get the wrong target.

*Recommended change:* Add a one-paragraph note at the top of the
`REVIEW-001 Supplement` section explaining that Phase 3a's FK refactor
landed at `domain/transformations/fk_helpers/`, that the v2
`helper_policies.fk` shape is now in `_meta`, and that the
`domain/fk/` target structure should be re-evaluated only after Phase
3a closes -- and only if the move still earns its keep. Keep the
non-FK remainder of the FTR (coverage gate, non-FK `core/` cleanup,
deprecation shims) unchanged.

=== Medium

==== Medium-1: Phase 4 preamble does not name the loop pattern

*Evidence:* Phase 4 preamble in `_ftrs_open.adoc` lists FTRs in
"Recommended order" but says nothing about how semantic and
refactoring work should interleave with Phase 3a's outcome or with
each other.

*Why it matters:* Without a short note, the natural reading is "all of
Phase 4 happens before any Phase 5 refactor". That contradicts the
Phase 3a precedent and will produce another mid-phase re-plan.

*Recommended change:* Add ~5 lines to the Phase 4 preamble:

[quote]
"For FTRs in Phase 4 that touch existing producers / primitives /
registry roots, use the semantic-refactor loop documented in
`docs/developer_guide/ch08_git_and_workflow/01_git_and_workflow.adoc`
(`FTR Slicing for Metadata and Registry Changes`): clarify minimal
semantics, add a guard, narrow refactor, docs / demo update, stop.
Phase 3a was the first worked example."

==== Medium-2: Current Direction note implies strict linear order

*Evidence:* `docs/developer_guide/ch10_project_evolution/01_project_evolution.adoc`,
"Current Direction" reads:

[quote]
"stabilize terms and metadata semantics (Phase 4: semantic and
metadata stabilization, built on the registry-semantics foundation);
repay maintainability debt once the semantic model is clear (Phase 5)"

*Why it matters:* The narrative encoded here is the strict
horizontal model that the FK-helper review just refuted. Future
agents reading this as orientation will not see the loop pattern
even if they read the developer guide.

*Recommended change:* Add one sentence after the "repay
maintainability debt ... (Phase 5)" bullet:

[quote]
"For workstreams where semantics and refactoring depend on each
other (chiefly FK helpers, derived provenance, meta schema), this
order is *interleaved*: small semantic-refactor loops, not a strict
phase sweep. Phase 3a is the worked example; see the developer
guide for the rule set."

No restructuring; one bullet's worth of edit.

==== Medium-3: `FTR-META-SCHEMA-VALIDATION-P5` has no pilot subject

*Evidence:* The FTR scopes "a validation function ... unknown
top-level `_meta` namespaces ... type mismatches where a declared
validation shape exists" but does not pick a first shape to
validate. The acceptance criterion "Type mismatches produce a clear
diagnostic for *at least one registry entry* with a declared
validation shape" requires picking one and is silent on which.

*Why it matters:* Specifying a validator without a concrete first
target is exactly the kind of "abstract metadata modeling with no
code pressure" the FK-helper review warned about. The Phase 3a v2
`helper_policies.fk` shape is the obvious pilot and makes the
validation FTR cheaper and reviewable.

*Recommended change:* Add to the FTR an "Initial pilot" subsection
naming the v2 `helper_policies.fk` shape (Phase 3a) as the first
registry entry to carry a declared validation shape. No new tests
required at planning time; just name the target.

==== Medium-4: `FTR-CONCEPT-DOCS-CLEANUP-P5` is first in Phase 4

*Evidence:* Phase 4 "Recommended order" begins with concept-doc
cleanup. The FTR itself is edit-only ("Do not add new product
behavior or architecture changes").

*Why it matters:* The concept docs that this FTR cleans up
(notably `fk_helper_metadata.adoc`) are about to be rewritten by
`FTR-FK-HELPER-DOCS-DEMO-REALIGNMENT-P5` and by the Phase 4
provenance lifecycle FTR. Cleaning first then re-rewriting later
is wasted motion.

*Recommended change:* Move `FTR-CONCEPT-DOCS-CLEANUP-P5` to the
end of Phase 4's recommended order (after
`FTR-DERIVED-PROVENANCE-LIFECYCLE-P4A` and
`FTR-META-SCHEMA-VALIDATION-P5`). Or split it: per-concept
cleanup happens as the *last step* of each Phase 4 loop, with the
old FTR retained only for concepts no other Phase 4 FTR touches.
Both options are acceptable; the second is closer to the loop
model and probably cheaper.

==== Medium-5: `FTR-GUARDRAIL-AUDIT-P5` scheduled before semantic settling

*Evidence:* Phase 5 begins with `FTR-GUARDRAIL-AUDIT-P5`. Its own
text already notes "after the intensive Phase-4 feature wave",
suggesting the author saw the dependency.

*Why it matters:* If guards are audited before Phase 3a / Phase 4
finish moving the producer / primitive boundary, the audit will
either lock in guards that are about to be obsoleted or recommend
removing guards that are about to become correct again.

*Recommended change:* Add one line to the FTR: "Do not start until
`FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5` and
`FTR-DERIVED-PROVENANCE-LIFECYCLE-P4A` are closed; the new
primitive-side architecture guard and the lifecycle contract are
both audit inputs." This is a sequencing comment, not a scope
change.

==== Medium-6: No backlog-level stop rule

*Evidence:* The backlog has no documented signal for "when to stop
reviewing / replanning and return to implementation". The last
month produced three review artifacts (FK-helper phase review;
this one; the implicit pre-Phase-3a review embedded in
ce9336f's commit message) and four review-related commits. None
explicitly closes the planning loop.

*Why it matters:* Process can become its own deliverable. Without a
stop rule, future agents may write a fourth review of the same
material instead of starting implementation.

*Recommended change:* Add a stop-rules subsection (or three short
bullets) to the developer-guide "FTR Slicing" section pointing at
the criteria in this artifact's *Stop Rules* section below. Single
short addition; no new file.

== Stop Rules

Stop reviewing / replanning when:

* the High-1 re-anchor has been applied;
* the Phase 4 preamble carries the loop note (Medium-1); and
* the next concrete implementation target has been chosen (almost
  certainly `FTR-FK-RELATION-POLICY-MODEL-P5`, since it is
  pure-doc and unlocks the rest of Phase 3a).

Return to implementation when:

* the developer has approved the v2 `helper_policies.fk` shape
  (public-surface gate in Phase 3a);
* OR the developer chooses to defer Phase 3a and pick a different
  starting point. In that case the smallest viable path is to
  start `FTR-DERIVED-PROVENANCE-LIFECYCLE-P4A` (documentation-only,
  no producer change) and let Phase 3a queue.

Do *not* write a follow-up roadmap review unless implementation
reveals a roadmap contradiction this review missed. Repeated
review without implementation is itself a smell.

== Residual Risks

* *Phase 4 may still grow.* If new producers / consumers of
  `_meta.derived` arise during Phase 3a implementation, Phase 4's
  scope expands. Acceptable: the lifecycle FTR is the documented
  catch-all.
* *`FTR-DOMAIN-CONSOLIDATION` may shrink to almost nothing after
  Phase 3a.* The non-FK remainder (coverage gate, non-FK `core/`
  cleanup, deprecation shims) may not justify a standalone FTR.
  Acceptable: re-evaluate at re-anchor time; a smaller FTR is
  fine.
* *The loop pattern may not transfer to non-metadata work.* It is
  scoped to metadata / registry workstreams in this review; other
  Phase 5+ work stays linear. Acceptable: explicit in the
  recommended execution model above.
* *Approval-gate fatigue.* Phase 3a already names a public-surface
  gate. The pattern may collide with the developer's available
  attention. Acceptable: gates are per FTR, not per loop.
