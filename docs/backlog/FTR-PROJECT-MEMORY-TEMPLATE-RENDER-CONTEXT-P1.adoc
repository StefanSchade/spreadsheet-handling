= FTR-PROJECT-MEMORY-TEMPLATE-RENDER-CONTEXT-P1

== Status

Implemented

== Type

Project Memory / Dev Tooling

== Problem

`project_memory` has grown from simple review, finding, and FTR metadata into a
steering model with concern threads, concern signals, and concern signal xrefs.

The existing `render_context` implementation mixed data loading and AsciiDoc
layout in Python code. That made the generated handoff context harder to
review, and the new concern steering layer was not first-class in the report.

== Goal

Separate Python context building from AsciiDoc document layout:

* Python builds a structured render context.
* Jinja templates own the AsciiDoc layout.
* Concern threads and linked signals are visible before raw finding/FTR lists.
* Unresolved project-memory references are surfaced as generated-context
  warnings.

== Dependency Rule

Jinja2 is an optional `project-memory` tooling dependency only.

It must not be added to normal `[project].dependencies`, and it must not be
imported under `src/spreadsheet_handling/`.

== Makefile Rule

Project-memory tooling is installed through an explicit non-phony stamp
dependency, not silently by the render script. `make setup` remains the normal
development setup; `make memory-setup` and first-use `make memory-context`
ensure `.venv/.stamp/project-memory.ok`.

Install command for this tooling slice:

[source,shell]
----
pip install -e '.[project-memory]'
----

== Scope

* Add a `project-memory` optional dependency extra containing Jinja2.
* Render `docs_generated/project_memory/current_context.adoc` via external
  `.adoc.j2` templates.
* Keep data loading, joins, grouping, sorting, and diagnostics in Python.
* Render concern threads, concern metadata, and linked concern signals.
* Render current findings, active FTRs, FTR dependency context, review
  takeaways, and diagnostics.
* Warn about unresolved concern signal xrefs and unresolved FTR dependency
  endpoints.
* Install project-memory tooling through a Makefile stamp dependency.
* Guard that product/runtime code under `src/spreadsheet_handling/` does not
  import Jinja2.

== Non-Goals

* No semantic rule engine.
* No automatic concern-thread extraction.
* No automatic canonical mutation.
* No broad mining of historical review ADOC files.
* No package/runtime dependency on Jinja2.
* No list-valued canonical JSON fields.

== Residual Risks / Follow-Ups

* The generated context is still intentionally lightweight. It is a handoff
  artifact, not a complete project-memory query/report engine.
* Template structure can be split further if the report grows, but that is not
  required for this slice.
* The diagnostics are warnings. They intentionally do not block rendering while
  the project-memory model is still being dogfooded.
