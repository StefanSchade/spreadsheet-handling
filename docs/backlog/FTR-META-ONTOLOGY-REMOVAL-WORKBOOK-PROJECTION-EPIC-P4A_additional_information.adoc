== Purpose of this document

This file does not change or override the epic. It captures clarifications
that emerged from a discussion about the epic before the Fable architecture
review, in three areas where the epic's wording could otherwise trigger
avoidable review round-trips. Fable should treat this as background context,
not as an amendment to the target model.

== 1. The three transformation-side cases, and why the cleanup contract stays binary

Feature-local transformations that build a spreadsheet projection over
relational data (e.g. `workbook_view`, `xref_crosstable`) and later reverse
that projection encounter exactly three cases on the source-frame side:

Reconstructable-source case::
The projection alone is sufficient to fully reconstruct the source frame on
inverse transformation. No shadow state is required. The source frame can be
marked `drop: true` for final domain cleanup.

Preserved-source case::
The projection is lossy with respect to the source (e.g. value ranges were
restricted, or the projection otherwise cannot carry enough information back).
A shadow copy of the source must be retained so the inverse step can merge
edited projection data back into it. The source frame is not marked for
cleanup.

Initial-create case::
On a first-ever roundtrip, no shadow state exists yet (e.g. there was no prior
relation/xref at all). The inverse step must therefore branch internally
between two strategies:

* `reconstruct_from_projection` -- used when the reconstructable-source case
  applies, or
* `merge_canonical_and_projection` -- used when a shadow bestand already
  exists and must be merged with edited projection data.

This branching is fachlich (feature-local) logic that belongs entirely inside
the transformation's own inverse implementation. It is not a case the generic
cleanup contract needs to represent.

At the point where the generic domain-cleanup contract is consulted (i.e. when
the spreadsheet/workbook artifact is written), all three cases above collapse
to the same binary question the epic already specifies:

[source,text]
----
drop: true  -> frame is removed during final domain cleanup
no directive / drop: false -> frame is not removed through this contract
----

In the initial-create case there is no source frame present at all, so no
drop decision applies -- the question is simply moot, not a fourth state.

Recommendation: this explanation may be folded near-verbatim into
"Transformation and Roundtrip Responsibility" in the epic, since it directly
pre-empts a question Fable is likely to raise (Review Questions 8-11).

== 2. Concrete worked example: xref / crosstable transformation

A concrete instance of the three cases above, useful as a characterization
scenario for Slice 0 and for Fable's evidence-gathering:

Unrestricted value domain::
A classical xref (tuple/relation representation) is projected into a
crosstable. If the value domain is not restricted, the crosstable fully
determines the xref on inverse transformation. The xref source frame is
reconstructable and may be marked for domain cleanup (`drop: true`).

Restricted value domain::
If the crosstable projection restricts the value range (e.g. only a subset of
possible row/column value combinations is shown), the projection is lossy. The
xref source frame must be preserved as shadow state, and the inverse
transformation must merge edited crosstable data back into it.

Initial / no prior relation::
On first creation, there may be no pre-existing xref at all. This is the
initial-create case from section 1: the inverse step creates the xref from the
projection, since no shadow state and no drop decision exist yet.

This scenario should be usable directly as a Slice 0 characterization case and
as an illustrative example in "Transformation and Roundtrip Responsibility."

== 3. Scope: existing transformations vs. future transformations

The epic's framing ("This is expected to be the first of several similar
reformation epics") could read as if the target model is purely greenfield.
In practice:

* `workbook_view` and (planned/partially implemented) `xref_crosstable` are
  existing transformations whose current implementation is considered more
  complex than necessary. This epic's scope is to simplify/realign these
  specific transformations with the target model described in the epic.
* Some comparable transformations are not yet implemented. These are expected
  to follow the pattern established by this epic once it lands, rather than
  being retrofitted.
* Future FTRs referenced in the epic's "first of several" framing concern
  *other* transformations/metadata areas following this same pattern -- not a
  second pass over `workbook_view` / `xref_crosstable` itself.

This distinction matters for Fable's Review Questions 1-2 and Slice 0: the
characterization work should treat `workbook_view` and `xref_crosstable` as
in-scope existing code to inventory and realign, not as out-of-scope prior art
to merely observe.

== Note: revocation of cleanup directives (intentionally not resolved here)

The epic already lists, under "Open Decisions for Review", whether multiple
transformations may revoke or override an earlier cleanup directive. This
question arose from a real (if currently hypothetical) scenario: two
projections sharing the same source frame, where one projection's
transformation considers the source reconstructable and the other does not.

This is deliberately left open rather than pre-decided here. The concrete
merge semantics for such a shared-source case are not yet on the table, and
resolving revocation logic in the abstract risks over-engineering an untested
case (YAGNI). This note simply confirms that the open question is a known,
intentionally deferred one -- not an oversight -- and that a concrete
shared-source example (should one arise during Slice 0 characterization)
should surface this to Fable's attention rather than being decided ad hoc.
