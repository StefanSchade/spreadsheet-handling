==== Verdict

*Accept with deletions.* The promotion is faithful to the inquiry, anchored to a
real use case, and keeps implementation content quarantined at the end. The
chapter's weakness is not omission but *excess*: it coins more semantic
vocabulary than the inquiry warrants and carries a self-administered
semantic-model reconciliation that the established ch04 convention defers. The
corrective direction is removal, not addition. No blocker.

==== Answers to the review questions

*1. Derives from the inquiry?* Yes. Every load-bearing claim of the inquiry is
present: density-not-storage, reversibility, structured attributes remain the
source, the three encoding strategies (verbatim), MVP scope (verbatim),
validation-as-adjacent, ephemeral editing aids. The chapter additionally maps
the inquiry onto the Declare/Encode/Decode step-slice form, which is the correct
ch04 adaptation. The only inquiry content dropped is the *Non-Goals* list
("not a parser utility / string-manipulation feature / matrix projection /
business-semantics engine / metadata-authority framework"); that loss is benign
and consistent with simplification.

*2. User problem clear before implementation terminology?* Yes. The opening
three paragraphs state the human-review problem in business terms; the registry
name `cell_codecs` first appears only in the per-step *Semantic Model Reference*
blocks, well after *User Problem* and *Functional Pattern*. Minor: the
source-storyline / cross-reference scaffolding (lines 14-20) sits between the
motivation and the *User Problem* heading and slightly delays it, but the
problem is already stated above it.

*3. Cell Codec / XRef Crosstable / Legend Blocks / Compact Multiaxis cleanly
separated?* Yes, and consistently with the neighbour chapters. The
*Distinction From Neighboring Concepts* table is crisp and matches XRef's
"Value Semantics" split and Legend Blocks' "Extension Pressure". The separation
is sound; the problem is that it is *restated* in several places (see finding
R3).

*4. Unnecessary semantic vocabulary introduced?* Yes -- this is the main finding
(R1). See below.

*5. Drifts into implementation-level thinking?* Largely no. `cell_codecs` is
named without field paths (an explicit FTR scope allowance) and the
*Implementation Reference* is correctly last and secondary. The real altitude
slip is downward into the *Semantic Model*, not into code: the
*Semantic Consistency Review* section performs concept reconciliation that the
convention (and the Legend Blocks precedent) defers to a later semantic pass
(finding R2).

*6. Belongs in a Functional Model?* Yes for the core. Two sections lower the
altitude: *Semantic Consistency Review* (semantic-model bookkeeping) and, to a
lesser degree, the redundant tail of *Implementation Reference*. Trimming these
restores the problem-space altitude.

*7. What to remove rather than add?* See *Recommended deletions*.

==== Recommended deletions (in priority order)

*R1 -- Drop the coined concept "Compact editable cell contract".*
Evidence: introduced as an *Information Produced* item (l.83-84), repeated in the
*Semantic Model Reference* (l.88-90) and in the new-terms list (l.237-238). It
only restates the inquiry's plain sentence "the structured attributes remain the
canonical representation; the compact cell value is a presentation-oriented
representation." It is exactly the kind of new metadata-flavoured concept the
prompt asks to avoid. Risk if kept: an unearned noun that the later semantic
pass must now reconcile against the registry. Action: delete the term; keep the
plain statement already present in the intro and *User Problem*.

*R2 -- Remove (or reduce to one sentence) the "Semantic Consistency Review"
section (l.218-245).* Evidence: it enumerates which Semantic Model concepts are
reused and which terms are "introduced", duplicating the per-step
*Semantic Model Reference* blocks. The Legend Blocks chapter deliberately
*defers* this ("reconciling them with the registry is deferred ... not performed
in this promotion"); the cell_codec chapter instead performs it inline, pulling
the chapter below Functional-Model altitude. Action: delete the section; if any
trace is wanted, a single deferral sentence mirroring Legend Blocks suffices.

*R3 -- Collapse the repeated neighbour-distinction statements.* Evidence: the
four-concept boundary is asserted in *Functional Pattern*, *Validation and
Editing Support*, *Distinction From Neighboring Concepts*, and again inside
*Semantic Consistency Review*. Keep the single table under *Distinction From
Neighboring Concepts*; drop the echoes.

*R4 (watch-only) -- "Cell Codec transformation intent".* Borderline. Unlike R1
it has a defensible parallel in the established *XRef projection intent*, so it
buys cross-chapter symmetry. Keep it, but if a deletion pass touches the
vocabulary list, this is the next candidate after R1. The third "introduced"
term, *Encoding strategy*, is not new -- the inquiry already uses it -- so the
claim that it is "introduced here" (l.239) should simply be dropped.

*R5 (minor) -- Trim the tail of "Implementation Reference" (l.247-253).* It
re-states the opening motivation and re-introduces the registry family already
named in every step block. One line naming `cell_codecs` as the implementation
anchor is enough.
