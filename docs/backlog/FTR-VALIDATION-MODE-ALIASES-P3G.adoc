===== FTR-VALIDATION-MODE-ALIASES-P3G — Accept friendly aliases for validation severity modes

*Context:* Demo-driven usage exposed a small but realistic configuration stumble: users naturally try values such as `error` for validation severity, while the current pipeline expects `fail`. The current behavior is internally consistent enough to work once known, but it is not forgiving and can be confusing at the configuration boundary.

*Problem statement:* Validation-related config currently depends on a narrow set of string values such as `warn`, `ignore`, and `fail`. A nearby synonym like `error` is easy to assume, but silently wrong for the current pipeline configuration surface. This creates avoidable friction in YAML authoring.

*Goals:*

. Make validation severity configuration more user-friendly.
. Preserve one canonical internal representation while accepting a small set of aliases.
. Document the canonical values clearly so config remains readable and predictable.

*Acceptance criteria:*

. Validation mode parsing accepts at least:
.. `fail` and `error` as equivalents
.. `warn` and `warning` as equivalents
.. `ignore` unchanged
. Internally the system normalizes these inputs to a canonical set.
. Invalid values still fail clearly with a helpful error message listing accepted values.
. Update documentation and at least one consumer-facing example to reflect the canonical spelling.

*Non-goals:*

- No redesign of the validation finding model itself.
- No change to the semantic meaning of existing severities.
- No broad config-schema overhaul beyond this small normalization layer.

*Why now:* This is a small paper-cut, not a blocker. It fits well as follow-up cleanup once the current demo reintegration thread settles, because it improves configuration ergonomics without changing feature scope.
