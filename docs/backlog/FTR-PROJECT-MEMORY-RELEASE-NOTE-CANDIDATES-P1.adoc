= FTR-PROJECT-MEMORY-RELEASE-NOTE-CANDIDATES-P1

== Status

Implemented

== Type

Project Memory / Dev Tooling

== Problem

Curated activity/action events in `concern_events` (ids with prefix `SIG-ACT-*`,
`source_type = activity`) capture grouped bodies of work that changed the state of one
or more concerns.

These events are not yet surfaced as release-note candidate material.
The existing release notes (`docs/release_notes/release_notes.adoc`) are manually
maintained and are the authoritative public record.
There is no generated path from curated activity events to reviewable release-note
candidates.

== Goal

Prove a technical path from curated activity signals to generated release-note candidate
artifacts.

* Derive `project_memory/derived/release_note_candidates.json` from activity events.
* Render `docs_generated/project_memory/release_note_candidates.adoc` as candidate
  material for manual editorial review.
* Provide traceability from each candidate back to its source event, commit refs, and
  linked concerns.

== Source

Activity events in `project_memory/canonical/concern_events.json`:

* selected by `source_type == "activity"` or id prefix `SIG-ACT-*`
* linked to concerns via `concern_event_xrefs`

== Dependency Rule

This FTR is project-memory/dev tooling only.

Jinja2 rendering reuses the existing `project-memory` optional extra.
No product runtime dependency is added.
No import of optional dependencies under `src/spreadsheet_handling/`.

== Scope

* Add a `release_note_candidates` query function to
  `project_memory/plugins/queries.py`.
* Register the step in
  `project_memory/pipelines/memory/json_to_derived_queries.yaml`.
* Add a Jinja2 template
  `project_memory/templates/release_note_candidates.adoc.j2`.
* Add `render_release_note_candidates()` to
  `project_memory/plugins/render_context.py`; invoke from `__main__`.
* The derived JSON includes: candidate id, source event id, event date,
  source type, source id, commit refs, weight, section, audience, status,
  summary, notes, linked concern ids.
* Optional per-event metadata (`release_note_section`, `release_note_audience`,
  `release_note_status`, `release_note_summary`) is supported via defaults when
  absent.

== Non-Goals

* Do not replace `docs/release_notes/release_notes.adoc`.
* Do not generate public release notes automatically.
* Do not solve final consumer-facing release-note editorial style.
* Do not infer candidates from historical_context, watch_item, evidence,
  manual_note, or review/finding events.
* No new product/runtime dependency on Jinja2.

* verified `docs/release_notes/release_notes.adoc` was not modified

== Residual Risks / Follow-Ups

* The generated candidates are intentionally lightweight candidate material.
  Editorial style, audience segmentation, and promotion workflow for public
  release notes are deferred to a follow-up FTR.
* `release_note_section`, `release_note_audience`, `release_note_status`, and
  `release_note_summary` optional fields are defined but not populated in the
  first data slice; all candidates use the defaults.
* Commit refs on activity events remain blank in the first data slice.
  Population is deferred to the next activity-event data wave.
