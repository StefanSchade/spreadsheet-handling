the FTR.* The Acceptance criteria require 3-5 PNG screenshots that

plus a real drift fix in the visual walkthrough source that the task

== Residual risks

* **Pending PNG capture is a release-blocking gap for the FTR but
  *not* a release-blocking gap for users.** Until the PNGs land,
  the demo journey has the Reveal.js deck as its visual artifact
  (rendered from the now-corrected walkthrough source) and the
  text-only first-hour tutorial in the demo `README.md`. Both are
  internally consistent; neither references a missing PNG. The cost
  of a delayed capture is "no inline images on the README", not
  "broken docs".

* **The currently-published slide deck at
  `/latest/demo/slides/product_business_slice.html` still carries
  the old "helpers stay understandable on reimport" wording until
  the next demo `publish-docs` run.** This slice deliberately does
  not edit generated Pages output. A new demo tag (or a manual
  `make publish-docs` rerun on the demo repo) is required to
  republish the corrected deck. The CI/CD posture is unchanged; the
  drift simply cleared from source.

* **Per-image content specification assumes the workbook layout
  stays roughly as it is today.** If a future pipeline change
  renames columns or reshuffles helper placement, the specification
  block in screenshots `README.md` will drift before any captured
  PNG does. Mitigation: the release-checklist bullet in core asks
  the releaser to review *the screenshot set* but should be read as
  including the spec itself; if a captured image needs new framing
  notes, the spec gets updated in the same commit.

* **Publishing-model decision still deferred.** The screenshots
  currently live only in the demo repo's working tree. The FTR's
  *Risks / open questions* says a later FTR must pick a publication
  model that matches the frozen-context policy of
  `FTR-RELEASE-README-VERSION-BAKING-P5`. Until that later FTR
  lands, screenshots are visible only via the demo repo's GitHub
  web view, which is fine for the current beta line but does mean
  PyPI-page readers do not see them yet.

* **Drift on the walkthrough's embedded YAML block.** The
  walkthrough now quotes the literal `remove_fk_helpers` step from
  `pipelines/demo_workbook_reimport.yaml`. If the pipeline shape
  changes (for example a different prefix, or a follow-up step),
  the walkthrough block becomes stale. Acceptable for now: the
  reimport pipeline has been stable across the last several
  releases and the embedded block is a single two-line `step:`,
  not a full pipeline reproduction.
