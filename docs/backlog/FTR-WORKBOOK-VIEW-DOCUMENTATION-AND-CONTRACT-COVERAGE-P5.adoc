= FTR-WORKBOOK-VIEW-DOCUMENTATION-AND-CONTRACT-COVERAGE-P5 -- Document the existing Workbook View workflow and contract placement

*Status:* New

*Priority:* P5

*Area:* Workbook View / public documentation / contract coverage

*Kind:* Documentation and contract-alignment slice

== Purpose

Make the already-implemented Workbook View capability practical to adopt from
the User Guide, while keeping its established boundary semantics and registry
classification intact.

This slice documents an existing public workflow. It does not design a new
Workbook View model or change runtime behavior.

== Context and adoption evidence

The completed
`FTR-WORKBOOK-VIEW-WORLDBUILDING-ADOPTION-P5` consumer slice exercised the
existing contract in the maintained Dino-Insel roundtrip. The adoption proved
that a consumer can:

* keep canonical JSON filenames and frame keys unchanged;
* configure shorter, user-facing sheet labels;
* declare a physical sheet order that differs from lexical frame order;
* roundtrip the configured view through both XLSX and ODS;
* restore configured logical frame names from visible sheet names with
  persisted explicit mappings and, where applicable, restore canonical source
  frames through the existing inbound steps;
* reorder sheets physically before reimport without associating logical frames
  with the wrong visible sheets;
* preserve representative payload data across the roundtrip.

The adoption closed without a core, carrier-backend, cross-carrier, or missing-
capability defect. Its remaining platform-level observation is documentation:
the generic User Guide catalogs the two public steps but does not yet show the
complete forward and inbound workflow that the consumer successfully used.

Evidence is recorded in:

* `docs/cold_storage/backlog/ftrs_done/FTR-WORKBOOK-VIEW-WORLDBUILDING-ADOPTION-P5.adoc`;
* `docs/cold_storage/backlog/ftrs_done/FTR-WORKBOOK-VIEW-WORLDBUILDING-ADOPTION-P5_review.adoc`.

== Current documentation and contract state

=== Functional Model

`docs/functional_model/ch05_problem_space_overview/04_workbook_view_boundary.adoc`
already gives the correct conceptual coverage. It defines Workbook View as a
spreadsheet-boundary capability that selects, orders, names, and annotates
prepared frames and maps returned visible sheets back to logical frames. It
also distinguishes that boundary policy from frame-to-frame domain
transformations.

The implementation slice must preserve that chapter as the conceptual source
of truth and must not duplicate it in the User Guide.

=== User Guide

`docs/user_guide/ch03_transformation_catalog/03_supporting_steps.adoc` currently
lists:

* `configure_workbook_view` as the configuration-only step that declares sheet
  names, order, lifecycle, and rendering options; and
* `apply_workbook_view_sheet_mappings` as the inbound step that rekeys visible
  sheets to logical frame names from persisted mappings.

That catalog coverage is accurate but incomplete for adoption. It lacks a
small generic forward-and-reverse pipeline example, placement guidance, and a
plain distinction among the configured logical frame, its canonical source
frame where applicable, the visible sheet name, and physical sheet position.

=== Semantic and Technical Models

The Semantic Model has general lifecycle and persistence-boundary material but
does not need a second definition of Workbook View merely for completeness.

The Technical Model already records relevant aspects in:

* the pipeline-boundary discussion in
  `docs/technical_model/ch08_architecture_details/pipeline_semantics.adoc`;
* the Workbook View and sheet-mapping classification in
  `docs/technical_model/ch05_registries/registry_semantics/03_meta_classification.adoc`;
* the Workbook View and sheet terminology in
  `docs/technical_model/ch09_glossary/01_terms.adoc`.

Implementation should add at most a concise cross-reference where it improves
navigation to the Functional Model or practical User Guide workflow. No
Semantic or Technical Model expansion is presumed by this FTR.

=== Registries and domain-contract representation

`registries/meta_registry.yaml` already classifies `workbook_view` as persisted
canonical render policy and `workbook_view.sheet_mappings` as the authoritative
visible-sheet-to-logical-frame mapping. The mapping contract explicitly rules
out reconstruction from labels, physical order, or columns.

`registries/pipeline_step_registry.json` already represents both public steps,
including the configuration-only nature of `configure_workbook_view` and the
relation-preserving reverse remap performed by
`apply_workbook_view_sheet_mappings`.

The canonical domain-contract registry already has implementation-unit entries
for both steps and a supported meta-reference surface for Workbook View sheet
mappings. Its transformation and requirement tables are deliberately small and
transformation-centric.

The absence of a Workbook View row from
`registries/domain_contracts/canonical/transformations.json` is therefore not,
by itself, a coverage defect. A transformation row would conflict with the
current Functional Model if it classified Workbook View as a frame-to-frame
domain transformation. The implementation review must record the final
placement decision, but the preliminary assessment is that the existing
implementation-unit, meta-reference, meta-registry, and pipeline-step entries
are the appropriate machine-readable surfaces.

=== Tests and guards

Current focused coverage already exercises:

* Workbook View declaration validation, configured order, visible names,
  lifecycle, mappings, and order-independent reverse rekeying;
* frame selection, render planning, formulas, and source immutability;
* XLSX and ODS configured-name and configured-order parity;
* carrier persistence and recovery of Workbook View mappings;
* end-to-end configured Workbook View reimport;
* meta-registry classification and reference integrity;
* the rule that pipeline orchestration does not branch on Workbook View
  metadata.

Documentation presence alone is not a reason to add an architecture guard. A
new guard is justified only if implementation identifies a specific stable,
machine-checkable contract that is not already protected and has credible drift
risk. Any proposed guard must name that contract and the failure it prevents.

== Scope

The implementation slice shall:

* add a compact, generic Workbook View workflow to the supporting-steps section
  of the User Guide;
* show forward configuration with explicit `frame` and visible `sheet`
  declarations in the intended declaration order;
* explain that declaration order controls the produced physical sheet order;
* show inbound application of persisted sheet mappings before downstream steps
  that address the configured logical frame or, where applicable, restore its
  canonical source frame;
* distinguish the configured logical frame, its canonical source frame where
  applicable, the visible sheet name, and physical sheet position;
* explain those as practical relationships among concrete frames, names, and
  order, not as four newly invented identity types;
* state that visible names and positions are not reverse-mapping heuristics;
* link to the existing Functional Model boundary instead of reproducing its
  conceptual treatment;
* decide whether any one-line or short-paragraph Semantic or Technical Model
  cross-reference is genuinely needed for navigation;
* confirm and record the correct domain-contract registry placement; and
* inventory the existing focused tests and guards before deciding whether any
  additional guard is warranted.

The example should use neutral frame and sheet names and only parameters that
the current public steps support. It must demonstrate the sequence, not create
a second exhaustive parameter reference.

== Explicit assessment questions

Implementation and review must answer:

. What is the smallest generic YAML example that makes both the forward and
  inbound workflow unambiguous?
. Which wording best distinguishes the configured logical frame, its canonical
  source frame where applicable, the visible sheet name, and physical order
  while remaining consistent with current code and documentation and without
  introducing a new identity ontology?
. Where must `apply_workbook_view_sheet_mappings` appear relative to inbound
  steps that address the configured logical frame or restore its canonical
  source frame where applicable?
. Is a Semantic or Technical Model cross-reference needed, and what navigation
  gap would it close?
. Do the existing implementation-unit and meta-reference-surface entries fully
  represent Workbook View for the current domain-contract registry's purpose?
. Would adding a transformation row misclassify a supporting boundary
  capability? The default answer is yes unless implementation finds contrary
  repository evidence.
. Is any stable contract currently unguarded, or would a new guard merely check
  for apparent registry or documentation completeness?
. Does this documentation-only change merit a release-note entry under the
  current release workflow? The implementation review must record the answer.

== Non-goals

This FTR does not include:

* runtime Python changes;
* YAML schema, public parameter, or pipeline-step signature changes;
* changes to the `_meta.workbook_view` shape or lifecycle;
* heuristic reconstruction from visible sheet labels or physical positions;
* changes in the worldbuilding or demo repositories;
* a new presentation feature or presentation-metadata family;
* domain-contract ontology or registry-schema expansion;
* a new registry table or transformation row for apparent completeness;
* a new guard without a specific uncovered contract and drift risk;
* speculative architecture or model rewrites;
* duplication of the existing Functional Model chapter.

If implementation discovers a need for any of these, it must stop and propose
a separately approved slice rather than broadening this FTR silently.

== Public-surface gate and approval status

The intended User Guide change documents existing public YAML pipeline steps
and therefore crosses the public-documentation gate described in
`docs/ai_info/interfaces_and_gates.adoc`. It does not authorize a change to the
YAML contract itself.

This planning artifact is approved by the request that created it. The FTR's
implementation is *not* approved by that request. Explicit human approval is
required before editing the public User Guide or any other implementation
surface.

Preflight assessment:

* Roundtrip: both directions must be documented; current consumer and core
  tests already supply runtime evidence.
* Metadata registry: existing roots are reused; this slice does not extend,
  replace, or introduce a meta root.
* Carrier parity: the documented contract is carrier-neutral and has existing
  XLSX and ODS evidence.
* Public documentation: gate applies and implementation approval is pending.
* Domain contracts: placement must be assessed, but no registry mutation is
  presumed.

Any implementation proposal that changes a signature, schema, registry schema,
metadata shape, precedence rule, or runtime behavior requires a new gate
assessment and separate approval.

== Acceptance criteria

The implementation may be reviewed Green only when:

* the User Guide contains a concise generic forward-and-reverse Workbook View
  workflow using the existing public steps;
* the workflow explains configured visible names and declared physical order;
* the inbound example applies explicit persisted mappings before steps that
  address the configured logical frame or, where applicable, restore its
  canonical source frame;
* the configured logical frame, its canonical source frame where applicable,
  the visible sheet name, and physical position are clearly distinguished as
  practical concepts rather than new identity types;
* the text explicitly rejects inference from a visible name or sheet position;
* the User Guide links to the existing Functional Model boundary and does not
  duplicate it;
* any Semantic or Technical Model change is a narrow, justified cross-reference
  recorded in the review, or the review records that no such change was needed;
* the review records whether the transformation-centric domain-contract
  registry is an appropriate representation surface and cites current registry
  evidence;
* no transformation row, registry table, runtime behavior, or guard is added
  merely to make coverage appear symmetrical;
* any new guard names the stable contract and demonstrated drift risk that
  justify it;
* documentation terminology and examples agree with current source, registry,
  and test behavior;
* no runtime, YAML schema, Workbook View metadata-shape, consumer, demo, or new
  presentation-feature change is included; and
* the release-note checkpoint is explicitly resolved in the implementation
  review.

== Validation expectations

For the expected documentation-only implementation, run at least:

* strict AsciiDoc rendering of every changed standalone document;
* strict rendering of the affected assembled User Guide;
* strict rendering of `docs/backlog/_ftrs_open.adoc` and
  `docs/backlog/backlog.adoc` while the FTR remains active;
* `git diff --check`.

Registry syntax and registry-guard checks are required only if an explicitly
approved implementation changes a registry. Runtime tests are not required for
a documentation-only implementation unless source or guard code changes. Any
additional command claimed by the review must have been executed and recorded
with its result.
