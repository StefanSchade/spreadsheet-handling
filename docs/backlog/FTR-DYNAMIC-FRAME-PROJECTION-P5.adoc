===== FTR-DYNAMIC-FRAME-PROJECTION-P5 -- Evaluate declarative support for dynamic frame projections

*Status:* New

*Purpose:*::
Evaluate whether common projection patterns that currently require consumer-side
plugins should be expressible declaratively within the pipeline model.

The goal is not to remove consumer extensibility but to determine whether a
small set of frequently occurring projection patterns deserves first-class
support.

*Context:*::
During validation of the worldbuilding consumer repository, a workbook matrix
representation was created for story-to-character relations.

The desired workbook view required a column layout of the form:

[source,text]
----
title | story_id | <dynamic character columns>
----

The set of character columns is not known in advance and grows dynamically as
new characters are added.

Current pipeline primitives allow:

* explicit column selection,
* explicit column ordering,
* workbook view mapping,
* matrix contraction and expansion,

but do not directly express:

[source,text]
----
fixed leading columns
+
all remaining columns
----

or similar dynamic projection patterns.

As a result, a small consumer-side plugin was introduced to:

* place helper columns first,
* keep key columns in a fixed position,
* append all remaining dynamic columns,
* reconstruct the corresponding payload view during reimport.

The solution works correctly but raises the question whether this projection
pattern should become a reusable pipeline capability.

*Motivation:*::
The worldbuilding consumer demonstrated a real use case that is not
artificially constructed.

The current plugin is small and maintainable, but the same projection pattern
may reappear in:

* relation matrices,
* localization matrices,
* configuration workbooks,
* FK helper views,
* future crosstable use cases.

The project should determine whether this represents:

* a consumer-specific concern,
* a reusable projection pattern,
* or a broader gap in frame composition ergonomics.

*Scope:*::

* Evaluate recurring dynamic projection patterns.
* Assess whether existing primitives are sufficient.
* Evaluate declarative alternatives.
* Determine whether consumer plugins remain the preferred solution.
* Define the intended long-term direction.

*Non-Goals:*::

* No removal of plugin support.
* No redesign of workbook views.
* No redesign of contract_xref or expand_xref.
* No automatic migration of existing consumers.
* No implementation commitment.

*Examples:*::

Current consumer plugin behavior:

[source,text]
----
title
story_id
all remaining columns
----

Potential declarative forms:

[source,yaml]
----
columns:
  - title
  - story_id
  - "*"
----

or

[source,yaml]
----
projection:
  leading_columns:
    - title
    - story_id
  keep_remaining: true
----

or equivalent alternatives.

*Architectural Options:*::

Option A -- Keep Plugins
+
Treat this as a consumer concern.
+
Advantages:
+

* maximum flexibility,
* no additional pipeline complexity.

-

Disadvantages:
+

* recurring projection logic may be duplicated across consumers.

Option B -- Add Projection Ergonomics
+
Introduce a small declarative mechanism for common projection patterns.
+
Advantages:
+

* reduces consumer boilerplate,
* improves readability of pipeline definitions.

-

Disadvantages:
+

* increases pipeline surface area.

Option C -- Reuse Existing Metadata
+
Investigate whether workbook-view metadata, helper-column metadata, or
xref metadata can provide sufficient information to eliminate the need for
custom projection code.

*Acceptance Criteria:*::

* The worldbuilding use case is evaluated.
* A project decision is recorded.
* The preferred architectural direction is documented.
* It is explicitly documented whether consumer plugins remain the
  recommended solution.
* If additional functionality is desired, a concrete follow-up item is
  created.
* The topic cannot silently disappear from future planning.

*Validation:*::

* Review the worldbuilding consumer implementation.
* Verify whether proposed declarative solutions would eliminate the plugin.
* Verify consistency with existing pipeline and workbook-view concepts.

*Discussion Note:*::

The worldbuilding consumer demonstrated that the core framework already
contains the required transformation primitives.

The question is not whether the use case can be solved, but whether the
current solution requires more consumer-side boilerplate than is desirable for
a recurring pattern.

This FTR intentionally focuses on ergonomics and composability rather than
missing functionality.

*Additional Consumer Validation Observation:*::

The same consumer validation also exercised the canonical FK-helper path on a
simple `places.region_id -> regions.id` relation, including formula-backed
helper cells and reimport cleanup.

That path works correctly and should remain the demonstration path in the
consumer repository for now.

One ergonomics observation from that slice is that explicit FK helper names are
currently generated mechanically as:

`{helper_prefix}{target_frame}_{field}`

For example:

* `_regions_name`
* `_regions_kind`

These names are technically stable and appropriate for canonical FK-helper
materialization, but they are less domain-friendly than aliases such as:

* `region_name`
* `region_kind`

A projected lookup surface could provide those aliases without changing the
underlying canonical FK-helper capability.

This is not a blocker and does not indicate a missing core feature. It is an
ergonomics / composability question about how much aliasing or projection
preparation should be expressible declaratively around the existing helper
machinery.
