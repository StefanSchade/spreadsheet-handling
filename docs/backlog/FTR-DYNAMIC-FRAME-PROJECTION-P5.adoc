===== FTR-DYNAMIC-FRAME-PROJECTION-P5 -- Evaluate declarative support for dynamic frame projections

*Status:* New

*Depends on:* `FTR-PROJECTED-FRAME-COLUMN-SEMANTICS-P5` (foundation;
provides the column-role taxonomy this FTR consumes --
`row_identity`, `display_helper`, `matrix_value` -- and the
reimport-survival rule for those roles). This FTR is a *consumer*
of that classification, not a co-owner.

*Purpose:*::
Evaluate whether common projection patterns that currently require consumer-side
plugins should be expressible declaratively within the pipeline model.

The goal is not to remove consumer extensibility but to determine whether a
small set of frequently occurring projection patterns deserves first-class
support.

*Context:*::
During validation of the worldbuilding consumer repository, a workbook
matrix representation was created for story-to-character relations.

Mapped to the column-role taxonomy defined in
`FTR-PROJECTED-FRAME-COLUMN-SEMANTICS-P5`, the desired projection is:

* `display_helper`: `title`,
* `row_identity`: `story_id`,
* `matrix_value`: the dynamic per-character columns, whose set is not
  known in advance and grows as new characters are added.

One possible rendering of that projection -- positional, with the
`display_helper` column at the leading edge -- is:

[source,text]
----
title | story_id | <dynamic character columns>
----

The leading-edge placement is a *render* choice, not part of the role
definition. The foundation FTR keeps `display_helper` positional-neutral
(see its *Future Design Pressure (Non-Scope)* section); this FTR must
not foreclose that.

Current pipeline primitives allow explicit column selection, explicit
column ordering, workbook view mapping, and matrix contraction /
expansion, but do not directly express *ordering or retention by role*
over the foundation's classification. As a result, a small consumer-side
plugin was introduced to:

* order columns by role on the outbound (render) side, and
* reconstruct the equivalent role-based retention on the inbound
  (reimport) side.

The solution works correctly but raises the question whether this
projection pattern should become a reusable pipeline capability that
consumes the role classification declaratively.

*Motivation:*::
The worldbuilding consumer demonstrated a real use case that is not
artificially constructed.

The current plugin is small and maintainable, but the same projection pattern
may reappear in:

* relation matrices,
* localization matrices,
* configuration workbooks,
* FK helper views,
* future crosstable use cases.

The project should determine whether this represents:

* a consumer-specific concern,
* a reusable projection pattern,
* or a broader gap in frame composition ergonomics.

*Scope:*::

This FTR *consumes* the column-role classification owned by
`FTR-PROJECTED-FRAME-COLUMN-SEMANTICS-P5`; it does not define or
redefine that classification. Within that constraint:

* Evaluate recurring dynamic projection patterns expressed as
  *ordering and retention by role* (`row_identity`, `display_helper`,
  `matrix_value`).
* Assess whether existing primitives are sufficient to consume the
  foundation classification, or whether a small declarative
  projection surface is warranted.
* Evaluate declarative alternatives over the role classification.
  Positional formulations (e.g. "leading columns + remaining") may
  appear as illustrative examples but are not the underlying
  semantic model; the underlying model is the role partition.
* Cover *symmetric* projection through reimport:
** *Outbound:* retain `row_identity + display_helper + matrix_value`
   in a role-determined order.
** *Inbound:* retain `row_identity + matrix_value`; drop
   `display_helper`. This is the role-based statement of what the
   consumer-side `build_*_matrix_payload` plugin does today.
* Determine whether consumer plugins remain the preferred solution.
* Define the intended long-term direction.

*Non-Goals:*::

* This FTR does not define or redefine the column-role taxonomy.
  The taxonomy (`row_identity`, `display_helper`, `matrix_value`),
  the reimport-survival rule, and the metadata-source expectations
  are owned by `FTR-PROJECTED-FRAME-COLUMN-SEMANTICS-P5`.
* No removal of plugin support.
* No redesign of workbook views.
* No redesign of `contract_xref` or `expand_xref`.
* No automatic migration of existing consumers.
* No implementation commitment.
* No selector-language design, no targeting-mechanics design, no
  editability or merge-back semantics, no collaboration semantics,
  no lifecycle modelling. Those belong to other FTRs.

*Examples:*::

These examples illustrate *one possible declarative surface* over the
column-role classification owned by the foundation FTR. The role
taxonomy is the underlying semantic model; the YAML below is one
rendering of it. This FTR does not commit to any specific syntax.

Current consumer plugin behavior, restated in role terms:

* outbound order: `display_helper`, then `row_identity`, then
  `matrix_value` (rest within the projected matrix surface);
* inbound retention: `row_identity` + `matrix_value`; drop
  `display_helper`.

Worldbuilding case, with the worldbuilding column-name mapping shown
inline for illustration only:

[source,text]
----
display_helper(title) | row_identity(story_id) | matrix_value(<dynamic character columns>)
----

Potential declarative forms over the role classification (illustrative
only; not a proposed syntax):

[source,yaml]
----
columns:
  - title       # display_helper
  - story_id    # row_identity
  - "*"         # matrix_value (rest-by-exclusion on this projected surface)
----

or

[source,yaml]
----
projection:
  leading_columns:
    - title       # display_helper
    - story_id    # row_identity
  keep_remaining: true   # matrix_value
----

or an equivalent surface that names the roles directly. Whether the
surface uses column names, role names, or both is a design question
to be settled if Option B is selected.

*Architectural Options:*::

Option A -- Keep Plugins
+
Treat this as a consumer concern.
+
Advantages:
+

* maximum flexibility,
* no additional pipeline complexity.

-

Disadvantages:
+

* recurring projection logic may be duplicated across consumers.

Option B -- Add Projection Ergonomics
+
Introduce a small declarative mechanism for common projection patterns.
+
Advantages:
+

* reduces consumer boilerplate,
* improves readability of pipeline definitions.

-

Disadvantages:
+

* increases pipeline surface area.

Option C -- Consume the Foundation Classification
+
Consume the column-role classification provided by
`FTR-PROJECTED-FRAME-COLUMN-SEMANTICS-P5` directly. The *what* is no
longer speculative -- the foundation FTR names the roles and the
reimport-survival rule. The remaining design question is *where the
consumption surface lives* (on the existing `configure_workbook_view`
step, on a new dedicated projection step, or on both), and whether
any declarative ergonomics are needed beyond consuming the role
classification. This option is now the baseline against which
Options A and B are compared.

*Acceptance Criteria:*::

* The worldbuilding use case is evaluated, expressed in terms of the
  foundation FTR's role taxonomy.
* The evaluation explicitly covers *both* outbound projection and the
  inbound mirror (reimport-side retention), since both consume the
  same role classification.
* A project decision is recorded.
* The preferred architectural direction is documented.
* It is explicitly documented whether consumer plugins remain the
  recommended solution.
* If additional functionality is desired, a concrete follow-up item is
  created.
* The topic cannot silently disappear from future planning.

*Validation:*::

* Review the worldbuilding consumer implementation.
* Verify whether proposed declarative solutions would eliminate the
  plugin on *both* the outbound and inbound sides.
* Verify consistency with existing pipeline and workbook-view concepts.
* Verify alignment with `FTR-PROJECTED-FRAME-COLUMN-SEMANTICS-P5`:
  this FTR introduces no parallel role taxonomy and treats role
  semantics as consumed, not redefined.

*Discussion Note:*::

The worldbuilding consumer demonstrated that the core framework already
contains the required transformation primitives.

The question is not whether the use case can be solved, but whether the
current solution requires more consumer-side boilerplate than is desirable for
a recurring pattern.

This FTR intentionally focuses on ergonomics and composability rather than
missing functionality.

*Additional Consumer Validation Observation:*::

The same consumer validation also exercised the canonical FK-helper
path on a simple `places.region_id -> regions.id` relation, including
formula-backed helper cells and reimport cleanup. That path works
correctly and should remain the demonstration path in the consumer
repository for now.

FK-helper *name aliasing* (e.g. `_regions_name` -> `region_name`) was
also noted during validation. That aliasing question is a separate
projection-ergonomics concern; the foundation FTR's *Non-Goals*
explicitly classifies it as outside the column-role-classification
scope. It is recorded here as a pointer only and is not part of this
FTR's evaluation.
