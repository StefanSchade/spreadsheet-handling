===== FTR-FUNCTIONAL-MODEL-VOCABULARY-AND-CONVENTIONS -- Functional-model glossary and authoring conventions

*Status:* New

*Type:* Documentation / conventions

*Area:* Functional model / documentation

*Sequencing:* Runs after the legend-blocks thematic is captured (warm-storage
concept -> ch04 promotion). Not a blocker for ongoing slices.

==== Context

The legend-blocks lifecycle slice surfaced two cross-cutting needs that affect
the whole functional model, not just one feature:

* the functional model uses terms (`structured data`, `canonical`, `payload`,
  `carrier`, `app boundary`, `durable home`, ...) functionally and
  situationally, without a defined, bounded vocabulary;
* there is no convention for how implementation status is expressed, so
  not-yet-built behavior risks reading as a future/now roadmap gradient.

Both were captured provisionally in the legend-blocks warm-storage concept
(its "Terminology Status and Working Vocabulary" appendix) and deferred here.

==== Why this FTR

* Loaded terms became problematic by accreting many concepts; using them
  undefined re-imports that baggage and is part of what made the meta model
  self-referential. The functional model needs its own bounded glossary.
* Functional-model and semantic-model vocabularies overlap. Without explicit
  delimitation, a reader silently imports a semantic-model contract into
  functional-model wording (e.g. reading "structured data" as "not modified by
  layer X").
* The functional model is a specification and legitimately describes behavior
  not yet built. Without a convention, "future" vs "now" blurs into a gradient
  that sounds like a commitment.

==== Scope

* Glossary pass over the functional model: span its own terms, give each a
  bounded definition, and delimit every overlap with the semantic model.
* Settle the `canonical` vs `structured data` question once, with a definition.
  `canonical` is acceptable once defined; the undefined import was the problem.
* Authoring convention: the functional model states target behavior in the
  present tense; implementation status is annotated separately and binary
  (built / not built), never as a future/near gradient; the semantic model and
  its registry are the place that tracks the current implementation, because the
  registry is part of the implementation.
* Decide where these live (ch01 management summary, ch05 framework, or a
  dedicated glossary appendix) and apply across existing chapters.
* Fold in the per-note terminology appendix from the legend-blocks concept.

==== Non-Goals

* Not a semantic-model glossary; that model owns its own vocabulary.
* Do not rewrite feature content; only vocabulary definitions and status
  annotations.
* Do not retroactively rename existing FTR files for naming consistency; that is
  a separate backlog-hygiene concern.

==== Acceptance

* The functional model has a defined glossary; each load-bearing term has a
  bounded definition.
* Every term that overlaps the semantic model carries an explicit delimitation
  note.
* The `canonical` vs `structured data` choice is made and applied consistently
  across promoted chapters.
* An authoring convention for spec-vs-implementation-status exists and is applied
  to at least the legend-blocks and FK-helper chapters.

==== Validation

* Docs build; no undefined load-bearing term remains in promoted functional-model
  chapters.
* Spot-check: no functional-model sentence silently implies a semantic-model
  rule.
