===== FTR-RELEASE-README-VERSION-BAKING-P5 -- Versioned README links for frozen PyPI contexts (DRAFT)

*Status:* Draft -- design only; not implementation-ready
*Priority:* P5
*Area:* Release tooling (design)

[IMPORTANT]
====
This FTR is a *design draft*. It documents the link-versioning policy
and a candidate release-tooling shape, but the implementation path
needs separate review and explicit approval before any code or tooling
work begins. For the next beta, the interim solution is a documented
manual release-checklist step (see "Interim manual checklist" below).
====

==== Purpose

Make sure that frozen public contexts -- PyPI long descriptions, GitHub
release pages, and any release-bound HTML under
`versions/<tag>/...` -- carry versioned deep links of the form
`/versions/<tag>/...` rather than `/latest/...`, so that pages frozen
at release time keep matching the version they describe even as the
project moves on.

==== Context

The beta publish exposed a gap: the `README.md` that becomes the PyPI
long description at tag time uses `/latest/...` deep links. PyPI keeps
old release pages online indefinitely, so as soon as a second release
ships, the previous PyPI release page silently starts pointing
visitors at documentation that no longer matches their installed
version.

This FTR records the policy and a candidate implementation but does
not move ahead of design review.

==== Link-versioning policy (the part that holds regardless of
implementation)

* *Frozen contexts use `/versions/<tag>/...`*. Frozen means the text
  is captured at a specific moment and lives there forever:
** PyPI long descriptions for any release tag;
** the contents of `versions/<tag>/core/release-notes/` and any other
   document inside a `versions/<tag>/...` tree;
** GitHub release pages that surface the tagged `README.md`.
* *Living contexts may use `/latest/...`*, *and* the link label must
  carry the lifetime explicitly:
** GitHub `README.md` viewed on `main`;
** Pages root `index.html` "latest release" banner;
** demo `README.md` viewed on `main`;
** good labels: *Read the user guide (latest release)*, *User guide
   for v0.1.0b6 (current release)*; bad labels: *User guide*,
   *Latest core docs*.
* *The Pages root is also a version archive*. Anyone arriving at the
  bare root URL must be able to find documentation for their installed
  version. That part is owned by
  `FTR-PAGES-ROOT-ARCHIVE-NAV-P5`.
* The policy is independent of how it is enforced. The candidate
  tooling below is one possible enforcement; the interim manual
  checklist is another.

==== Scope (design phase)

* Document the policy above in
  `developer_guide/ch09_release_management/`.
* Describe two candidate enforcement paths and the trade-offs:
** a *manual checklist step* in the Core Release Checklist that the
   maintainer performs before tagging (no new tooling);
** an *automated release-tooling step* (e.g.
   `make prepare-release VERSION=vX.Y.Z`) that creates a release-only
   commit replacing `/latest/...` with `/versions/vX.Y.Z/...` in
   `README.md` (and demo `README.md` for demo tags), then optionally a
   `make finalize-release` that restores `/latest/...` on `main`.
* Record an interim decision for the *next* beta release (manual or
  automated).
* Out-of-scope work the design must not absorb is enumerated under
  *Non-goals*.

==== Interim manual checklist (use if this FTR is not promoted before
the next tag)

Before tagging a release on the core repository:

. Branch off `main` to a short-lived release branch.
. In `README.md`, replace every occurrence of
  `https://stefanschade.github.io/spreadsheet-handling-pages/latest/`
  with
  `https://stefanschade.github.io/spreadsheet-handling-pages/versions/<vX.Y.Z>/`.
. Commit the substitution as a single Conventional Commit (suggested:
  `docs(release): bake versioned README links for vX.Y.Z`).
. Tag from that commit and push the tag.
. Restore `/latest/...` on `main` via a follow-up commit (or by
  letting the next `prepare-release` run rebake from a known baseline).

The same approach applies to the demo `README.md` for demo tags. The
checklist substitution surface is intentionally narrow: only Pages
URLs that begin with the `…/latest/` prefix are rewritten.

==== Non-goals (design boundary)

* No CI automation of *tagging* itself. The maintainer continues to
  push tags by hand.
* No template engine, no Jinja-style preprocessing, no general-purpose
  documentation pipeline.
* No rewriting of URLs outside `README.md` (release notes content
  already lives under `versions/<tag>/...` and uses relative
  references; walkthrough HTML is published per version by the demo
  workflow).
* No rewriting of in-code URLs, docstrings, or asciidoc cross-
  references.
* No expansion of scope to cover sibling repositories beyond core and
  demo READMEs.
* No implementation work until the design path (manual vs automated)
  is explicitly approved.

==== Acceptance (design-phase, current)

* The policy above is documented in
  `developer_guide/ch09_release_management/`.
* Both candidate paths (manual checklist; automated
  `prepare-release` / `finalize-release`) are described with explicit
  trade-offs.
* The interim choice for the next beta release is recorded (default:
  manual checklist unless automation is explicitly approved).
* The substitution surface (which URL prefixes are rewritten) is
  documented and intentionally narrow.

==== Acceptance (to be expanded when promoted from Draft to New)

When the FTR moves out of Draft, the acceptance criteria are
extended with:

* the chosen enforcement path is implemented (manual: runbook update
  only; automated: tooling commits land under `tools/` since demo
  READMEs also need rewriting for demo tags);
* idempotency of any automated substitution is verified (re-running
  with the same version produces the same file);
* CI or a Make target asserts that the resulting tagged README
  contains no remaining `…/latest/...` Pages URLs;
* the `developer_guide/ch09_release_management/02_release_runbook.adoc`
  Core Release Checklist references the chosen path explicitly.

==== Dependencies

* `FTR-PUBLIC-DOCUMENTATION-ENTRYPOINTS-P5` provides the READMEs that
  carry the links being versioned. That FTR may ship first using
  `/latest/...` with explicit labels; this FTR (or the manual
  checklist) upgrades the links for the *next* tag.
* Conceptually related to
  `FTR-RELEASE-RUNBOOK-BRANCH-AWARENESS-P5`: both add steps to the
  pre-tag part of the release runbook. If both ship, consider a
  combined `make release-check` umbrella so the maintainer runs one
  command rather than two.

==== Implementation notes (if promoted)

* If the automated path is chosen, the substitution helper belongs
  under `tools/` (see `ai_info/conventions.adoc` *Helper script
  placement*) because the demo repository also needs it for demo tags
  and tools/ is the documented cross-repo copy point.
* Substitution allowlist: only `…/spreadsheet-handling-pages/latest/`
  URLs are rewritten; other `latest`-shaped strings (e.g. in
  unrelated prose) are left alone.
* Two-commits-per-release vs single-drift-commit is the most
  contentious design point and must be decided before implementation.

==== Risks / open questions

* Whether to handle the "two commits per release" workflow
  (`prepare-release` + `finalize-release`) or accept one drift commit
  on `main` between releases. Decision needed before promoting.
* Whether to also rewrite links inside published HTML (release notes
  file copies, walkthrough output). Current assumption: no, those
  already sit under `versions/<tag>/...` and their internal references
  are naturally version-correct.
* Whether the demo repository (which is not on PyPI) needs the same
  baking. Demo `README.md` is surfaced on GitHub release pages for
  demo tags; recommended to apply the same rule to keep demo release
  pages consistent.
* Whether to fold this into `FTR-RELEASE-RUNBOOK-BRANCH-AWARENESS-P5`
  if both are implemented together. Currently kept separate because
  the policy is independent of the branch-topology check.
