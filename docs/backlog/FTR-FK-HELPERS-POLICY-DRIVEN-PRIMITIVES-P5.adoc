===== FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5 -- Make FK helper primitives consume explicit relation policy

*Status:* New

*Purpose:*::
Refactor FK helper primitive steps so they consume explicit FK relation policy
from `_meta` instead of inferring relation semantics from frame, table, or
column naming conventions.

*Context:*::
After FK relation policy and `infer_fk_relations` exist, the primitive FK-helper
steps should become deterministic executors of resolved metadata.
+
The current primitive path still uses convention-based detection to decide
which columns are FKs and which target frames they reference. That makes the
primitive step both an inference engine and a transformation, which conflicts
with the architecture rule that heuristics belong in configuration steps.
+
Backward compatibility is intentionally not a goal for this project stage.
There are no relevant external users whose pipelines must keep running with the
old implicit behavior.

*Dependencies:*::

* `FTR-FK-RELATION-POLICY-MODEL-P5`
* `FTR-INFER-FK-RELATIONS-CONFIGURATION-STEP-P5`
* Existing FK helper implementation.
* Existing derived-helper provenance implementation.
* Existing FK-helper validation tests.

*Scope:*::

* Refactor `add_fk_helpers` so it reads FK relation policy from `_meta`.
* Remove normal-path dependency on convention-based FK detection from
  `add_fk_helpers`.
* Refactor `validate_fk_helpers` so validation checks relation policy and
  materialized helper values, not naming conventions.
* Refactor `remove_fk_helpers` so helper removal uses derived provenance and/or
  explicit policy, not prefix fallback.
* Refactor `reorder_fk_helpers` so helper placement uses explicit policy or
  helper provenance, not FK name patterns.
* Write derived provenance rich enough to support reimport cleanup and later
  helper value consistency checks.
* Remove or quarantine old convention helpers that are no longer used by
  primitive FK-helper runtime paths.
* Update tests to use either:
  ** explicit relation policy, or
  ** `infer_fk_relations` before FK helper primitives.

*Behavior:*::

* `add_fk_helpers` fails clearly if no FK relation policy is available.
* `add_fk_helpers` materializes only helpers declared by policy.
* `validate_fk_helpers` checks declared FK relations and helper values.
* `remove_fk_helpers` removes only columns identified by provenance or policy.
* `reorder_fk_helpers` moves only declared helper columns.
* No primitive FK-helper step may interpret `auto`, infer target frames, or
  guess target key columns.

*Non-Goals:*::

* No public backward compatibility.
* No legacy mode hidden behind defaults.
* No automatic migration of old user YAML.
* No generic relation engine beyond FK helper use cases.
* No redesign of workbook-view readback.
* No screenshot or tutorial update; that belongs to the docs realignment FTR.

*Acceptance:*::

* FK helper primitives no longer infer FK relation identity from column names.
* Missing policy produces a clear actionable error that names the expected
  configuration or inference step.
* Tests cover a fully explicit policy path.
* Tests cover an inferred-policy path using `infer_fk_relations` followed by
  `add_fk_helpers`.
* Tests cover failure when `add_fk_helpers` is run without policy.
* Tests cover helper cleanup after reimport without prefix fallback.
* Tests cover validation mismatch using target key and helper field information
  from policy/provenance.
* `remove_fk_helpers` no longer needs a helper prefix fallback in normal
  operation.
* Registry docs classify FK-helper primitives as primitive execution steps with
  no heuristic responsibility.
* Existing architecture guards pass or are updated to reflect the stricter
  split.

*Validation:*::

* Run FK helper unit tests.
* Run relevant pipeline integration tests.
* Run demo product-business-slice tests after the docs realignment FTR updates
  the demo pipeline, or temporarily update a fixture pipeline inside this FTR.
* Run architecture guards.
* Run source-style guards if production code is changed substantially.
* Render affected technical and user documentation.

*Residual risks:*::

* Removing convention fallback may break local scratch pipelines not represented
  in tests. This is acceptable before external users depend on the package.
* Some old core utilities may become dead code but not be removed immediately.
  Dead-code cleanup may be handled in a later maintainability FTR if removing
  them would expand this slice too far.
* The relation policy shape may need later extension for composite keys or
  grouped workbook views.
