===== FTR-META-FACET-CLASSIFICATION-P5 -- Classify remaining transform-config metadata facets

*Status:* New

*Purpose:*::
Classify the remaining transform-config metadata entries by Intent / Resolution /
Provenance and by payload-semantics-bearing relevance. Registry, documentation,
and persistence-boundary preparation only.

*Context:*::
The architecture health review flags `canonical_meta` over-use: entries such as
`sparse_defaults`, `cell_codecs`, `compact_multiaxis`, and
`split_by_discriminator` are one undifferentiated `canonical_meta` blob, while
`legend_blocks` and `xref_crosstable` already carry facet-level
`persistence_behavior`. The persistence boundary preserves Intent and drops
Resolution; classifying the remaining entries as data is the prerequisite for any
later attribute-driven handling.

*Dependencies:*::
* `FTR-META-LIFECYCLE-INVENTORY-P5` -- this is the umbrella inventory; this slice
  populates part of it.
* `FTR-PAYLOAD-SEMANTICS-RECOVERY-P5` -- sibling / input: it names which metadata
  carries payload semantics. Treat it as a classification input, not as scope to
  absorb or discard.

*Scope:*::
* For each remaining transform-config registry entry, record an Intent /
  Resolution / Provenance facet classification and a payload-semantics-bearing
  flag.
* Mirror the wording already used for `legend_blocks` / `xref_crosstable`.
* Feed the result into the `FTR-META-LIFECYCLE-INVENTORY-P5` table.

*Non-Goals:*::
* No generic / attribute-driven persistence engine.
* No broad `_meta` redesign and no new registry root.
* No flip of `runtime_binding: forbidden`.

*Acceptance:*::
* Each remaining transform-config entry distinguishes Intent vs Resolution vs
  Provenance and is flagged for payload-semantics relevance.
* The relationship to `FTR-PAYLOAD-SEMANTICS-RECOVERY-P5` is recorded.
* `tests/architecture/semantic_invariants` guards stay green.
* No runtime behavior change.
