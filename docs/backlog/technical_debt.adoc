= Technical Debt Register

This document tracks active technical debt that is intentionally not
resolved in the current change set. Each entry names a follow-up FTR or
a containment strategy.

Entries listed here are observed gaps with a clear owner and a known cost,
not speculative ideas. Speculative or exploratory items belong in the
phase-grouped FTR backlog under ``docs/backlog/_ftrs_open.adoc``.

== TD-META-LIFECYCLE-INVENTORY -- _meta lifecycle classification incomplete

*Status:* Open

*Severity:* Medium / High

*Area:* Metadata lifecycle / persistence boundary

*Origin:* ``BUG-RUNTIME-META-PERSISTENCE-BOUNDARY-P4A``

*Tracking FTRs:*

* `FTR-META-LIFECYCLE-INVENTORY-P5` -- full inventory
* `FTR-FK-HELPER-POLICY-LIFECYCLE-P4A` -- FK-helper-specific classification

*Summary:*

The current fix establishes a narrow persistence boundary in the
orchestrator and removes the known runtime-produced FK-helper v2 relation
carrier under ``helper_policies.fk.relations``. It does not complete a
full lifecycle classification of all ``_meta`` roots.

Until the inventory is done, new ``_meta`` producers must be conservative
and document whether their output is persistable, runtime-only, derived,
carrier-specific, or user-authored. Reviewers should treat any new
top-level ``_meta`` key as a flag for "needs classification" and either:

* extend the projection rules in
  ``src/spreadsheet_handling/pipeline/persistence_boundary.py`` with an
  explicit named rule, or
* document the key as a persistable canonical entry with a citation in
  the lifecycle chapter and the registry.

*Why not fix now:*

The complete classification is a structural change that touches the
registry, the lifecycle chapter, the architecture guards, and several
producers. Doing it before the next release would extend scope beyond
what the worldbuilding adoption surfaced and would conflict with the
deliberate "release-near" framing of the bug fix.

*Containment until resolved:*

* The orchestrator-owned projection in
  ``pipeline/persistence_boundary.py`` is the single chokepoint.
* New runtime-only top-level keys must be added to
  ``RUNTIME_ONLY_TOP_LEVEL_META_KEYS`` (or covered by an equally
  explicit rule).
* The ``__*`` root-level filter catches accidental new
  rendering/carrier artifacts.
* Consumer-side meta-stripping plugins are explicitly out of policy --
  the boundary belongs in core.
