== Findings ordered by severity

=== Important follow-up: `canonical_frame` is not completely removed from live behavior

Workbook View production, mapping readback, and sheet resolution are correctly
frame-only. Legacy `sheet_mappings[*].canonical_frame` values are ignored by
the Workbook View reader and are not sheet-resolution candidates.

However,
`domain/schema_maintenance/meta_refs.py::FRAME_KEYS` still contains
`canonical_frame`. The generic convention scanner therefore continues to
interpret that key as a frame reference in blocked and unknown metadata roots.

A direct characterization using:

[source,yaml]
----
plugin_state:
  canonical_frame: source
  column: name
----

showed that schema maintenance still blocks renaming `source.name`.

Therefore `canonical_frame` is removed from Workbook View identity but not yet
fully removed from all live behavior.

Recommended correction:

* remove `canonical_frame` from `FRAME_KEYS`;
* add a regression test proving that a legacy `canonical_frame` key does not
  create a generic structured frame reference.

Classification: *important follow-up*.

=== Important follow-up: documentation and registry drift

The refined runtime rule is correct: only explicitly drop-commanded frames
conflict with explicit Workbook View mappings. Keep-mode implied removals may
coexist with reimported mappings.

The User Guide and Semantic Model still use the broader wording that a frame
"scheduled for cleanup" while mapped is a conflict. They must distinguish:

[source,text]
----
explicit drop command + explicit Workbook View mapping -> conflict
keep-mode implied removal + existing Workbook View mapping -> allowed
----

The metadata registry also classifies `pipeline_cleanup` as
`meta_rendering / derived_operational_view` and describes its scope as
workbook-level. The command is instead explicit, transient, carrier-neutral,
and consumed by final domain cleanup. It is not a rendering realization.

No runtime layering violation was introduced, but the structured registry
currently records the wrong architectural layer and its architecture test pins
that classification.

Classification: *important follow-up*.

== Behavioral answers

. *Is `canonical_frame` fully removed from live behavior?*
+
No. It is fully removed from Workbook View identity, readback decisions, and
sheet routing, but remains a generic schema-reference key.

. *Are legacy payloads handled safely?*
+
Yes for Workbook View payloads: `sheet` and `frame` remain authoritative and
validated, while `canonical_frame` is ignored. The generic schema scanner
residue must still be removed for complete compatibility semantics.

. *Is frame-only schema-maintenance routing coherent?*
+
Yes. Sheet-scoped metadata belongs to the frame actually mapped to the sheet.
Renaming a former projection source no longer mutates metadata owned by the
projected frame, and conflicting explicit frame declarations remain ambiguous
and block.

. *Are characterization tests sufficient and meaningful?*
+
They are meaningful and cover the principal routing and readback decisions.
They need targeted additions for the generic `canonical_frame` scanner and
stale-mapping structured-output to later-spreadsheet failure.

. *Is the explicit-drop versus keep-implied-removal distinction correct?*
+
Yes. It accurately distinguishes two explicit contradictory declarations from
a reverse-pipeline final-set declaration applied to reimported metadata.

. *Can stale Workbook View metadata cause silent corruption?*
+
No silent frame substitution or data corruption was found within the current
boundary. Structured output may persist stale view mappings. A later
spreadsheet write fails explicitly on its first missing mapped frame before
creating an artifact. No configured step runs after the final cleanup boundary.

. *Does cleanup obey the no-precedence and no-ontology rules?*
+
Yes. It executes only explicit commands and never inspects lifecycle roles or
canonicality. Builder-owned drop and keep modes remain mutually exclusive,
while transformation-produced drops compose mechanically with builder keep
mode: intersection is a contradiction and exclusion is compatible redundancy.
No override precedence is introduced.

. *Is the Worldbuilding migration behaviorally equivalent?*
+
Yes for the actual JSON export contract. The nine data JSON files are
equivalent, the declaration is correctly ordered, and XLSX and ODS reverse
flows pass. The expected metadata difference is that cleanup commands do not
persist and `frame_lifecycle` entries keyed by removed frames are pruned, while
required intent and stale Workbook View mappings survive.

. *Were new layering violations or generic metadata concepts introduced?*
+
No runtime layering violation or new ontology was introduced. The shared
cleanup command is a justified narrow producer/consumer contract. The metadata
registry misclassifies it as rendering-derived and must be corrected.

. *Is the next ontology-removal slice ready?*
+
Yes. The important follow-ups can be handled at its start without reopening the
architecture.

. *What must the next slice remove or rewrite?*
+
The exact scope is recorded below.

. *Are human decisions still required?*
+
No. The later authoritative Human Decision settles cleanup composition:
builder-owned drop and keep modes are mutually exclusive, while explicit
transformation drops may compose with builder keep mode under contradiction-
or-redundancy semantics. The appraisal also decides canonicality removal,
provenance-removal bias, cleanup placement, and command consumption.

== Test assessment

Focused Core validation passed: *202 tests* across cleanup, Workbook View,
schema maintenance, XRef producers, renderer selection, carrier integration,
XLSX/ODS parity, persistence-boundary behavior, workbook meta-blob transport,
registry invariants, and pipeline architecture guards.

Worldbuilding validation passed: *2 cross-carrier workflow tests* covering
declared workbook names/order and unchanged-data reimport through both XLSX and
ODS.

Direct probes additionally established:

* generic schema maintenance still interprets `canonical_frame` as a frame
  key;
* the real Dino reverse result contains exactly the nine intended JSON files;
* `pipeline_cleanup` is absent from the persisted `_meta.yaml`;
* persisted `sheet_mappings` contain only `sheet` and `frame`;
* stale Workbook View mappings remain in structured metadata; and
* writing that structured result to XLSX fails with a missing-frame error and
  creates no workbook artifact.

Recommended test changes:

. prove `canonical_frame` no longer participates in convention-based metadata
  reference scanning;
. add an integration sequence covering keep mode, structured persistence of
  stale view mappings, and later explicit spreadsheet failure before artifact
  creation;
. assert the intended Worldbuilding `_meta` delta: commands consumed, removed-
  frame lifecycle entries pruned, intent preserved, stale frame-only mappings
  retained.

== Compatibility assessment

Legacy persisted Workbook View mappings remain readable. Any
`canonical_frame` value is ignored; malformed values under that legacy key do
not affect Workbook View resolution. Required `sheet` and `frame` values,
duplicates, missing sheets, and unknown logical frames still fail
deterministically.

Removing `WorkbookViewSheetMapping.canonical_frame` is a Python API break. It is
intentional, release-noted, and acceptable under the project's pre-1.0
compatibility posture.

The Worldbuilding commit requires a Core revision that provides
`configure_pipeline_cleanup`; deployment must therefore update Core before or
together with Worldbuilding.

== Accepted documented limitations

The following are accepted and are not blockers:

* final cleanup cannot alter artifacts already emitted by side-effecting mid-
  pipeline writers;
* keep-mode structured output may retain stale Workbook View mappings and
  legitimate absent-frame intent;
* generic metadata-pileup detection and pruning are deferred;
* general M:N workbook projection remains out of scope; and
* lifecycle-driven rendering remains until the next ontology-removal slice.

These limitations do not justify retaining or recreating `canonical_frame`,
`derived_from`, or another generic preservation ontology.
