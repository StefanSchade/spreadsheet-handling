== 4. Remaining findings by severity

=== 4.1 Blocking -- missing-like duplicate physical labels escape detection and can schedule cleanup around a `Series` cell

`_ensure_unique_physical_labels` deliberately avoids hashing and compares each
label using `_values_equal` (`primitives.py:56-78`). `_values_equal` converts
`left == right` to `bool` and returns `False` on ambiguous results. Therefore:

* two `float('nan')` labels are not recognized as duplicates because NaN is
  unequal to itself;
* two `pd.NA` labels are not recognized because converting their comparison to
  `bool` raises and is treated as unequal.

Direct contraction probe:

[source,python]
----
n = np.nan
frames = {
    "rel": pd.DataFrame(
        [["r1", "A", "x", "y"]],
        columns=["r", "k", n, n],
    )
}
contract_xref(
    frames,
    relation="rel",
    output="matrix",
    row_keys="r",
    column_key="k",
    value=n,
    drop_source=True,
)
----

Observed result: the call succeeds; the `A` matrix cell is a pandas `Series`
rather than a scalar; and `_meta.pipeline_cleanup.drop_frames == ['rel']`.
The input metadata remains unchanged, but final cleanup is instructed to
delete the only scalar source representation.

The same counterexample exists in expansion with a scoped base relation:
duplicate `NaN` physical value fields escape base-frame duplicate detection,
the appended out-of-scope cell is a `Series`, and cleanup drops the matrix.
With duplicate `pd.NA` fields, a raw `TypeError: boolean value of NA is
ambiguous` occurs later instead of the promised duplicate-label diagnostic.

Although pipeline configuration declares field selectors as strings, the
domain functions do not enforce that annotation, YAML can supply non-string
scalars, and the correction's explicit contract is stronger: _all_ duplicate
physical relation/base labels are to be rejected before selection, metadata,
or cleanup. This is an exact counterexample to Finding 4.1, the User Guide,
release notes, registry wording, epic record, and second correction record.

The required correction is mechanical and requires no new policy: duplicate
physical-label comparison must treat missing-like values consistently (and
must remain safe for unhashable/ambiguous equality) before any selection.

=== 4.2 Blocking -- selected expansion identities are compared before the shared identity validator

Expansion constructs `value_cols`, then evaluates:

[source,python]
----
row_key_overlap = [col for col in value_cols if col in row_key_cols]
----

Only afterwards does it call `_ensure_column_identity_list`
(`operation.py:90-104`). This reverses the required order for invalid selected
identities.

Two direct probes against a valid string-header matrix establish the defect:

[cols="3,4", options="header"]
|===
|Call
|Observed result

|`expand_xref(..., value_columns=[pd.NA], drop_source=True)`
|Raw `TypeError: boolean value of NA is ambiguous` from list membership.

|`expand_xref(..., value_columns=[np.array([1, 2])], drop_source=True)`
|Raw `ValueError: The truth value of an array with more than one element is
ambiguous` from equality during list membership.
|===

Both failures occur before `_write_xref_meta` and
`mark_frames_for_cleanup`, and input metadata is unchanged, so they are
fail-safe with respect to deletion. They nevertheless violate the explicit
requirements that selected expansion value columns reach the unique,
non-empty string validator before equality/hash/pandas behavior and that
`pd.NA` and unhashable values receive a deterministic XRef diagnostic.

The fix is local: validate/materialize selected `value_cols` through the
shared identity validator before computing overlap. No human decision or new
metadata is needed.

=== 4.3 Important -- registries and records overstate the implementation while the blockers remain

The User Guide says every identity source fails explicitly and every duplicate
physical relation/matrix/base declaration fails before output or cleanup
(`02_bijective_transforms.adoc:178-195`). The release notes make the same
claims (`release_notes.adoc:81-90`); the epic record says contract diagnostics
are used "never raw TypeError"; the second slice correction record says all
findings are closed. Findings 4.1 and 4.2 disprove those statements.

The pipeline registry is now structurally complete, including expansion's
prior-intent read. Its wording should ultimately distinguish repeated uses of
a relation/base `column_key` value (valid across different row identities)
from uniqueness of the resulting matrix header vocabulary; relation values
are intentionally checked with `_ensure_column_identity_values`, which allows
repetition. This is a wording clarification, not a request to reject valid
repeated relation identities.

Do not weaken the contract around the defects. Once the runtime order and
missing-like duplicate comparison are corrected, realign the completion
claims and add the exact counterexample tests.

== 7. Cleanup-safety assessment

Valid full contraction writes exactly
`pipeline_cleanup.drop_frames == ['relation-source']`; valid full expansion
writes exactly `['matrix-source']`. Input metadata remains unchanged. Ordinary
unrepresented relation fields, uncovered relation identities, unexpanded
matrix fields, duplicate string fields, duplicate row-key declarations,
string overlap, and invalid identities fail before `_write_xref_meta` and
`mark_frames_for_cleanup`.

Scoped non-superseding projections remain supported:

* contraction with an explicit representable column subset and
  `drop_source=False` succeeds without cleanup;
* expansion with an explicit subset and `drop_source=False` succeeds without
  cleanup;
* valid scoped base-relation recomposition preserves out-of-scope rows.

Finding 4.1 is a remaining cleanup-safety violation: a duplicate missing-like
physical field can evade the guard, produce a non-scalar cell, and write the
single explicit drop. Finding 4.2 is fail-safe for cleanup but violates the
diagnostic/validation-order contract.

The documented `drop_empty=True` dense-shape limitation and the rule that
inferred expansion treats all non-row-key columns as values remain accepted
feature-local semantics, not defects.

== 8. Ambiguity assessment

Uniform ambiguity behavior is now coherent across live consumers:

* transformation `_xref_config` gives an exact mapping config id precedence,
  otherwise counts every mapping matching the relation or matrix;
* relation-side and matrix-side fallbacks both fail on multiple matches;
* sparse-default targeting now counts complete and partial mapping entries
  before looking for a usable `column_keys` payload;
* column-role resolution counts every matrix mapping; an explicit
  `key_columns` override bypasses metadata lookup.

Complete+partial, two partial, and two complete frame matches fail without
insertion-order selection. Exact config-id selection remains authoritative and
is proven with observably different payloads. One partial entry falls back
locally as documented. Non-mapping/malformed individual entries are ignored by
tolerant readers; a non-mapping root behaves as absent for runtime lookup.

No live call supplies both relation and matrix to `_xref_config`; its OR-union
therefore creates no current false ambiguity. No usable-payload or generic
override precedence remains.

== 10. Producer replacement and persistence assessment

Both producers still write only minimal feature-local intent plus run-local
Resolution:

* durable: `relation`, `matrix`, `row_keys`, optional
  `dense_axes.rows_from` / `columns_from`;
* run-local: top-level `column_keys` and `dense_axes.resolved`.

`_write_xref_meta` assigns the new payload to `configs[config_id]`, replacing
the complete old entry. The new regression proves that obsolete
`operation`/`column_key`/`value`/`drop_empty` fields disappear for the rewritten
id while a differently keyed legacy entry remains untouched sediment. The
input metadata is unchanged on success.

The carrier-neutral persistence boundary strips top-level `column_keys` and
`dense_axes.resolved`, preserves durable minimal intent, passes untouched
legacy fields tolerantly, and does not mutate input. XLSX and ODS integration
tests prove identical persisted intent and recreation of an absent relation
for the supported string vocabulary. Workbook View remains free of XRef
semantics.

== 11. Registry and documentation assessment

The correction adds the missing `expand_xref` prior-intent read to the
pipeline-step registry and expands both drop-source notes. The meta registry,
User Guide, release notes, epic record, and second XRef correction record all
describe the intended string identity, duplicate declaration, uniform
ambiguity, and cleanup contracts.

They are not yet fully accurate because they state that all duplicate physical
labels are rejected and every identity source receives an explicit diagnostic
before cleanup. Findings 4.1 and 4.2 are direct counterexamples. The final
runtime correction must be accompanied by regression tests and realignment of
these completion claims; this review does not edit them.

No generic metadata concept, typed identity preservation, or precedence rule
was introduced.

== 14. Blockers and human decision

Blockers before XRef acceptance:

. make equality-based duplicate physical-label detection missing-aware and
  ambiguity-safe, preserving deterministic handling of unhashable labels;
. validate selected expansion `value_columns` through the shared string/list
  validator before any overlap membership/equality operation;
. add exact regression tests for duplicate `NaN`/`pd.NA` relation and base
  labels, the cleanup-writing `Series` counterexamples, and invalid selected
  `pd.NA`/NumPy-array identities;
. correct registry/documentation completion claims after runtime behavior
  actually matches them.

No new human decision is required. The existing unique, non-empty string
decision and the already accepted duplicate-declaration rule fully determine
the fixes. No typed header encoding or new metadata is needed.

== 15. Readiness

XRef is not accepted as a safe foundation for a later composed family while a
duplicate physical field can yield a `Series` and cleanup. Work on an
independent primitive family may proceed separately, but it must not treat
XRef as finally accepted or inherit these validation assumptions.

`compact_multiaxis` and any other XRef-composed family must remain blocked on
the two local corrections and a final focused re-review. This review does not
assess or constrain unrelated Cell Codec work in its separate clone/branch.
