== Phase 3a: FK Helper Semantics and Helper Presentation

Purpose:

* Turn FK helper handling into the first concrete model case for explicit,
  metadata-driven transformation semantics.
* Separate heuristic inference from primitive execution: heuristics may write
  resolved configuration, but helper primitives must consume explicit policy.
* Remove the remaining convention-driven FK-helper behavior before it becomes
  part of the public mental model.
* Add optional collapsible helper-column presentation only after helper identity
  and provenance are explicit.
* Keep spreadsheet adapter concerns behind Render IR operations; FK-helper code
  must not know XLSX or ODS carrier details.

Recommended order:

* Define the explicit FK relation policy model.
* Add `infer_fk_relations` as an opt-in configuration step that writes resolved
  policy.
* Refactor FK-helper primitives to consume policy instead of detecting relations
  from names.
* Realign demo pipelines, tutorial, and user documentation with the
  policy-driven helper model.
* Add backend-neutral column-group rendering capability.
* Let FK-helper configuration emit optional helper-column grouping intent.
* Realign demo and documentation for collapsible helper columns.

Semantic split:

* The first group owns relation semantics and helper provenance.
* The second group owns presentation of already-known helper columns.
* Column grouping must not be used as a substitute for helper identity.
