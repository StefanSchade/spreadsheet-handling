== Phase 3a: FK Helper Semantics

Purpose:

* Turn FK helper handling into the first concrete model case for explicit,
  metadata-driven transformation semantics.
* Separate heuristic inference from primitive execution: heuristics write
  resolved configuration; primitive FK-helper steps consume explicit policy or
  resolved metadata.
* Remove convention-driven *primitive behavior* (inference inside
  `add_fk_helpers`, prefix-based default in `remove_fk_helpers`, FK-pattern
  matching inside `reorder_fk_helpers` / `validate_fk_helpers`) before it
  becomes part of the public mental model.
* Replace the existing per-target `helper_policies.fk` shape with a v2,
  relation-oriented shape under the same root. No adjacent FK-helper policy
  root is introduced; the project keeps a single FK-helper policy root.

Out of scope here (kept deliberately):

* The FK column convention `id_(<target>)` (`core/fk.py:FK_PATTERN`) is *not*
  removed. It remains the cell-level wire format for cross-sheet references on
  XLSX/ODS roundtrips. This phase changes primitive *behavior*, not the
  serialized FK shape.
* Composite keys and arbitrary join inference.

Recommended order:

* `FTR-FK-RELATION-POLICY-MODEL-P5` defined the v2 relation-oriented FK
  policy shape as a model-only slice and is archived in
  `docs/cold_storage/backlog/ftrs_done/`.
* `FTR-INFER-FK-RELATIONS-CONFIGURATION-STEP-P5` added `infer_fk_relations`
  as a configuration step that writes the v2 shape, migrated the heuristic
  / `targets: "auto"` path out of `configure_fk_helpers` into the new step,
  and kept `configure_fk_helpers` as the explicit / manual policy step
  (writing v1 for current primitive consumers and v2 through a transitional
  bridge). The FTR is archived in `docs/cold_storage/backlog/ftrs_done/`.
* Refactor FK-helper primitives (`add_fk_helpers`, `remove_fk_helpers`,
  `reorder_fk_helpers`, `validate_fk_helpers`) to consume the v2 policy and
  derived provenance. Missing policy is a clear error; the prefix fallback in
  `remove_fk_helpers` is removed from normal operation.
* Realign demo pipelines, tutorial, user-guide transformation catalog, the
  FK-helper concept doc, and the agent-facing FK-helper notes.

Public surface gate:

* The v2 policy shape replaces the documented per-target `helper_policies.fk`
  shape and removes inferred / `detect_fk` behavior from `add_fk_helpers`. Per
  `docs/ai_info/interfaces_and_gates.adoc` these are gated YAML / step-surface
  changes. The developer must approve the block before
  `FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5` is implemented.

Coordination with Phase 4:

* Phase 3a owns *producer-side* metadata shape (the v2 `helper_policies.fk`
  shape and the fields written into
  `_meta.derived.sheets.*.helper_columns`, including `target_key`).
* `FTR-DERIVED-PROVENANCE-LIFECYCLE-P4A` owns lifecycle policy: pruning,
  stale / orphan handling, read-side rules, and schema validation of the
  derived channel.
* Phase 3a should not pre-empt the lifecycle decisions; Phase 4 must respect
  the producer shape Phase 3a writes.

== Phase 3.5: Semantic Model Foundations & Adoption Plan

Purpose:

* Carry the design intent that emerged from the FK-helper review into a
  structured, incremental adoption path for an explicit project semantic
  model.
* Preserve the seed scaffold under `docs/semantic_model/semantic_model.adoc`
  as the source of design intent for this phase, and elaborate it section by
  section through small, reviewable FTRs.
* Treat metadata attributes and transformations as overlapping semantic sets
  with explicit producer/consumer, lifecycle, precedence, and roundtrip
  contracts.
* Make registry roles explicit (description, test oracle, possibly future
  rule input) without committing to a rule engine.

Scope:

* This phase produces documentation, classification, and a pilot contract
  matrix. Code- or guard-level work is allowed only in the FK-helper
  contract-matrix pilot, and only where the contract is already stable.
* Each FTR is a slice that elaborates one or two sections of the seed
  scaffold. The seed itself is preserved as draft text and not rewritten
  into polished prose ahead of those slices.

Out of scope:

* No general semantic rule engine.
* No big-bang adoption of existing transformations.
* No runtime refactoring driven by this phase.
* No movement or renaming of registry artifacts; the root-level registries
  established in the previous phase remain in place.
* No closing, archiving, or cold-storing of existing FTRs from this phase.

Coordination:

* Phase 3a (FK Helper Semantics) supplies the worked example consumed by
  `FTR-SEMANTIC-CONTRACT-MATRIX-PILOT-FK-HELPERS-P5`. That pilot must align
  with the producer-side metadata shape Phase 3a writes and with the
  derived-provenance lifecycle owned by Phase 4.
* Phase 4 (Semantic and Metadata Stabilization) should consume the slice
  questions and exit criteria defined by
  `FTR-SEMANTIC-ADOPTION-STRATEGY-P5` when sequencing its own FTRs.
* The semantic model document is an internal/agent-facing artifact under
  `docs/semantic_model/`. It is not part of the Pages publishing surface
  defined in `scripts/build_docs_pages.sh`; updating Pages navigation is
  explicitly out of scope here.

Recommended order:

* `FTR-SEMANTIC-MODEL-SKELETON-P5` -- integrate the seed scaffold into the
  docs build/navigation and stabilize the chapter structure.
* `FTR-SEMANTIC-PROBLEM-STATEMENT-P5` and `FTR-SEMANTIC-INSPIRATIONS-P5` --
  elaborate the motivating prose; can run in parallel.
* `FTR-SEMANTIC-SETS-AND-LIFECYCLE-MODEL-P5` -- define overlapping sets,
  lifecycle events, and precedence questions as rule-ready concepts.
* `FTR-SEMANTIC-REGISTRY-ROLE-P5` -- pin the registry-role distinctions
  before any contract matrix proposes guards.
* `FTR-SEMANTIC-ADOPTION-STRATEGY-P5` -- define the no-big-bang slice
  template and adoption order.
* `FTR-SEMANTIC-CONTRACT-MATRIX-PILOT-FK-HELPERS-P5` -- pilot the matrix on
  FK helpers, the only family whose runtime contract is already stable
  enough to support targeted guards.

== Phase 3b: Helper Column Presentation

*Status:* Optional / deferrable. Unlocks only after Phase 3a is complete.

Purpose:

* Add an optional, backend-neutral collapsible helper-column presentation
  capability so spreadsheet users can fold helper columns out of the way.
* Keep presentation strictly downstream of helper identity and provenance.
  Column grouping must never be used as a substitute for helper identity.
* Keep spreadsheet adapter concerns behind Render IR operations; FK-helper
  code must not know XLSX or ODS carrier details.

  XLSX rendering, ODS rendering or an explicit accepted capability gap in the
  backend capability profile).

* The render capability is generic; no FK-specific terminology in the
  RenderOp.

  ODS or ODS assumptions into XLSX.
* No readback / persistence of user-expanded / collapsed UI state.
