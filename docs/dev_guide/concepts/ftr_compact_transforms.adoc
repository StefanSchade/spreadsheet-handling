=== Concept: FTR-COMPACT-TRANSFORMS — Multi-Layer ↔ Compact Representation

==== Purpose
This feature enables the bijective transformation between cross-tables or tuple lists that represent multiple logical layers (e.g., availability, market, edition) and a single compact table where these layers are encoded as tokens within one cell.

==== Overview of the Two Bijective Steps
*Step 1 — XRef ⇄ Cross-Table*::
  Converts tuple lists into simple two-dimensional tables (n = 1 case).
  - Function names: `expand_xref(frames, meta)` and `contract_xref(frames, meta)`
  - Supports both 2-tuple (flag) and 3-tuple (value) forms.

*Step 2 — Multi-Layer ⇄ Compact Table*::
  Introduces compact encoding for n ≥ 2 logical layers.
  - Function names: `expand_compact(frames, meta)` and `contract_compact(frames, meta)`
  - Each layer is represented by short, unique tokens concatenated into a single cell value.

==== Layer Cardinality and Usability
* `n = 1` – Natural use case for Step 1 (simple cross-tables).
* `n ≥ 2` – Triggers Step 2 (multi-layer compact encoding).
* `n ≫ 2` – Still supported; usability may degrade but there is no algorithmic limit.

==== Requirements
* Every layer must have:
  ** a unique short code,
  ** a descriptive source name, and
  ** a configurable value set.
* Both 2-tuple (flag) and 3-tuple (value) representations are supported.
* Provide configurable separators:
  ** `token_sep` between tokens (default '-')
  ** `kv_sep` between layer code and value (optional, default '=')
  ** `multi_sep` between multiple values within a layer (default '+').
* All resolved configuration values are persisted in `meta` to maintain bijectivity.

==== Default-Heuristic vs. Explicit Configuration
The system follows a two-level design philosophy:

*Heuristic defaults*::
  Basic conventions are inferred automatically, for example:
  - A column named `ID` is interpreted as a primary key.
  - A column named `ID (other_table)` is treated as a foreign key reference.
  - A sheet name can serve as the default short code for that layer.

*Explicit configuration*::
  Users can override or complement heuristics via declarative YAML:
  [source,yaml]
  ----
  table:
    name: table_a
    id_col: y
    foreign_ref:
      column: z
      origin: table_b
  ----
  These settings are read once, stored in `meta`, and reused for inverse transformations.

==== Meta Schema for Compact Encoding
[source,yaml]
----
compact_encoding:
  token_sep: '-'
  kv_sep: '='
  multi_sep: '+'
  sort_order: ['nature','edition','market']
  derivations:
    - name: availability_implied_by_any_market
      if_any_of_groups: ['market']
      then_add_token: A
      write_policy: omit
  tokens:
    - code: U
      group: nature
      label: User-driven
      cardinality: 'exactly-one-of:nature'
    - code: R
      group: nature
      label: Regulatory
      cardinality: 'exactly-one-of:nature'
    - code: Std
      group: edition
      label: Standard
      cardinality: 'exactly-one-of:edition'
    - code: Ext1
      group: edition
      label: Extension 1
      cardinality: 'exactly-one-of:edition'
    - code: EU
      group: market
      label: Europe
      cardinality: 'zero-or-more-of:market'
    - code: US
      group: market
      label: United States
      cardinality: 'zero-or-more-of:market'
    - code: AMEA
      group: market
      label: AMEA
      cardinality: 'zero-or-more-of:market'
    - code: A
      group: availability
      label: Available
      cardinality: 'zero-or-one-of:availability'
----

==== Example Encodings
|===
| feature | product2 | product3 | product1
| abc | U-EU-US | R-EU-AMEA | Std-EU
| def | R-US | U-AMEA | Ext1-EU
|===

Each token is globally unique, ensuring the cell string remains a deterministic, reversible representation of the same tuple set.

==== Validation and IR Integration
* Validation lists are materialized only when the total number of allowed token combinations ≤ hard cap (default 64).
* Beyond that, the IR layer switches to a decomposed UX with per-layer helper columns and dropdowns.
* All canonical rules (token uniqueness, sort order, cardinality) are enforced by meta-driven passes in the domain layer.

==== Edge-Case Rules
* Normalization: sort tokens, deduplicate, apply uniform case policy.
* Cardinality: enforce group constraints; violations produce `meta.issues`.
* Unknown tokens: preserved but flagged; never silently dropped.
* Derived tokens: applied on read; omitted on write per `write_policy`.

==== Notes on Quantitative Extensions
* Single scalar numeric values per intersection (e.g. `W=0.75`) can later be supported as an additional token family without breaking bijection.
* Aggregated values (sums, averages, percentages) are not bijective and will be handled by future non-bijective extraction/aggregation transformations.

==== Summary
The FTR-COMPACT-TRANSFORMS concept defines a deterministic, reversible, and user-friendly encoding for complex multi-layer relations inside spreadsheet-style data. It adheres to a clear separation between domain semantics, IR-level presentation, and configurable heuristics, ensuring reproducible round-trips between tuple lists, cross-tables, and compact human-editable sheets.
