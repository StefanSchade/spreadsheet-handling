= Review: Collaborative Repository Roundtrip Concept

== Status

* Review of: `docs/warm_storage/collaborative_repo_roundtrip.adoc`
  (the source draft currently lives next to this file under the
  duplicated filename
  `collaborative_repo_roundtrip.adoccollaborative_repo_roundtrip.adoc` --
  see "Editorial / filename observation" below).
* Type: architectural / consistency / placement review of an
  exploratory concept draft.
* Scope: read-only review. No official documentation is modified. The
  draft itself is not rewritten. No implementation work is performed.
* Reviewer constraint: review-only per
  `docs/ai_info/review_instructions.adoc`. Findings recorded; default
  action is *do not fix*.

== Executive summary

The draft articulates a collaboration pattern -- "persisted `_meta`
should behave as generated state, reproducible from canonical data and
checked-in configuration" -- that is *strongly congruent with the
existing semantic model and architectural direction*, while also
introducing one genuinely new framing: an explicit *collaboration-grade
reproducibility contract* for persisted metadata.

Specifically:

* The draft echoes, in adoption language, a position the project has
  already taken structurally in the persistence boundary
  (`pipeline/persistence_boundary.py`), in the
  canonical-vs-derived distinction
  (`docs/technical_model/ch04_concepts/metadata_canonical_vs_derived.adoc`),
  and in the registry's three-value layer axis
  (`docs/technical_model/ch05_registries/registry_semantics/02_meta_layer.adoc`).
* It also says something the existing docs do *not* yet say plainly: that
  the right *collaboration default* -- as opposed to the right *runtime
  default* -- is "treat persisted `_meta` as reconstructable working
  state whenever possible".
* That additional framing is useful, but it cuts across at least three
  load-bearing contracts (canonical-layer presentation meta, the
  `_meta.sheets.<frame>.helper_columns` durable channel, and the
  workbook_identity bucket) that *cannot* be reconstructed from
  canonical data plus checked-in configuration without losing
  information. Adopting the concept naively would silently demote them
  from canonical to derived, which would be a contract change, not a
  collaboration tweak.
* Section 7 (payload semantics boundary) exposes the most important
  hidden assumption: the concept treats "metadata" as a single shape
  ("merge noise"), but the project already has cases
  (`BUG-UNRESOLVED-HELPER-VALUE-NAN-SEMANTICS-P4A`) where the
  distinction between *visible empty cell* and *unresolved /
  error-bearing helper* lives partly in metadata. A reproducibility
  rule that licenses metadata loss must explicitly forbid payload
  semantics from depending on it.

Recommended posture for the concept: keep as an *adoption-guidance*
draft (warm storage), do not promote it to the functional model, the
technical model, the registry, or `_ai_info`, and -- if it is later
adopted -- attach it to the existing meta lifecycle inventory work
(`FTR-META-LIFECYCLE-INVENTORY-P5`) rather than to a new
architectural axis.

== 1. Consistency review

Each subsection lists *direct contradictions* (would require changing
an existing contract to accept the concept), *tensions* (would require
new constraints to avoid breaking an existing contract), and
*alignments* (the concept restates an existing position).

=== 1.1 Functional model

Reference: `docs/functional_model/ch04_use_case_concepts/`
(`business_calculation_workbook_roundtrip.adoc`,
`worldbuilding_continuity_workbook_roundtrip.adoc`,
`helper_columns.adoc`,
`localized_resource_workbook_roundtrip.adoc`).

Alignments::

* The business-calculation use case explicitly distinguishes
  authoritative canonical relations from workbook *views* derived from
  them via `compose_workbook_views`, and treats the workbook as a
  presentation/editing layer over canonical frames. That is the same
  story the draft tells from the Git-collaboration angle.
* The same document explicitly asks (Open Design Questions) "How much
  lifecycle metadata should be persisted in `_meta` versus supplied by
  pipeline configuration?" The draft proposes "as little as possible";
  the functional model has only posed the question.

Tensions::

* The functional model's reverse-path stories
  (`apply_workbook_view_sheet_mappings`) rely on
  `_meta.workbook_view.sheet_mappings` being *persisted* and
  *authoritative* -- precisely because sheet identity must not be
  inferred from display labels. That is a persistent-`_meta` claim
  that *cannot* be reconstructed from canonical data + checked-in
  pipeline configuration alone (the visible sheet labels are
  user-editable). The draft's "should be reproducible" rule needs an
  explicit carve-out for workbook-view sheet mappings, or it
  contradicts the recomposition pattern that is already shipping.

Direct contradictions::

* None. The functional model is mostly compatible because its
  examples already declare which columns are canonical and which are
  helper context; the draft does not propose changing the canonical
  set, only changing how persisted `_meta` is treated.

=== 1.2 Technical model / registry semantics

References:
`docs/technical_model/ch05_registries/registry_semantics/01_data_and_meta.adoc`
through `05_contract_lifecycle.adoc`;
`docs/technical_model/ch04_concepts/metadata_canonical_vs_derived.adoc`;
`docs/ai_info/architecture_primer.adoc` (`meta_canonical` vs
`meta_rendering`);
`docs/semantic_model/08_lifecycle_and_update_semantics.adoc`
(persistence boundary).

Alignments::

* The draft's "Recommended Collaboration Pattern" -- canonical data
  plus checked-in pipeline configuration as the *shared truth*, with
  persisted `_meta` treated as generated state -- is the exact
  framing of the *derived* meta layer:
  "Derived meta is *recomputable*. If lost, the next run can
  regenerate it from the canonical sources plus the current frame
  state." (`02_meta_layer.adoc`).
* The persistence boundary in
  `pipeline/persistence_boundary.py` already removes runtime-only
  carriers (top-level `derived`, `__*` keys, runtime-produced FK-helper
  v2 relations) before any saver runs. That is a *load-bearing*
  encoding of the draft's intent for the subset of meta the project
  has already classified.
* `BUG-RUNTIME-META-PERSISTENCE-BOUNDARY-P4A` and
  `FTR-META-LIFECYCLE-INVENTORY-P5` together state the project's
  current position: narrow runtime-only filter today, full
  classification work deferred. The draft is consistent with that
  trajectory.

Tensions::

* The registry's *canonical* layer is defined by five strict criteria
  (`02_meta_layer.adoc`), one of which is "lifecycle is decoupled
  from any single run -- it remains true across runs, edits, and
  re-rendering until the user changes it". A blanket policy that
  treats persisted `_meta` as regenerable would collapse canonical
  and derived back into one bucket, which is the exact drift the
  five-criterion definition was created to prevent. The draft must be
  read as applying *within the derived layer* and *within the
  Presentation classification's reconstructable subset*, not
  uniformly across `_meta`.
* The presentation-meta motivating case in `03_meta_classification.adoc`
  is `column_widths`: "a user who explicitly sets a column width
  expects that width to survive a roundtrip". A user-authored column
  width is canonical-layer + Presentation-classification, and is *not*
  reconstructable from canonical data plus checked-in pipeline
  configuration -- the user's keystroke *is* the canonical source.
  The draft's wording ("workbook styling restored, presentation
  defaults re-applied") elides this case.
* Operational-carrier entries (e.g. `workbook_meta_blob`) are not
  "merge surface" in the draft's sense -- they are the serialized
  form of canonical claims, written through a deterministic
  serializer. Repository merge conflicts on operational-carrier
  payloads are an *encoder-determinism* concern, not a
  "treat as derived" concern. The draft conflates the two.

Direct contradictions::

* The draft's "Losing persisted `_meta` should not normally destroy a
  collaborative project" *as a project-wide invariant* directly
  contradicts the canonical-layer definition. As a *guidance posture
  for the Presentation-classification, reconstructable subset* it is
  fine. The unconditional phrasing is the contradiction; a scoped
  phrasing would not be.

=== 1.3 Glossary (semantic model)

Reference: `docs/semantic_model/18_glossary.adoc`.

Alignments::

* The draft's central distinction matches the glossary's
  *canonical* / *derived* / *persistent* / *transient* axes.

Tensions::

* The draft uses "metadata" as a flat category. The glossary already
  distinguishes *Policy*, *Provenance*, *Configurable*,
  *RenderConsumed*, *RoundtripRelevant*, *Persistent*, *Transient*,
  *UserEditable*, *Canonical*, *Derived* (`07_overlapping_semantic_sets.adoc`).
  The draft would gain a great deal of precision by stating *which
  overlapping sets* the proposed "treat as generated state" rule
  applies to. Without that, the rule is unenforceable.

Direct contradictions::

* None at vocabulary level; the draft is consistent with the
  glossary when read as scoped to derived / transient meta.

=== 1.4 Registry semantics

Reference: `docs/technical_model/ch05_registries/registry_semantics/`
and `registries/meta_registry.yaml`.

Alignments::

* The registry's *operational-carrier* layer already documents
  carrier-shape entries (`workbook_meta_blob`, `_hidden`,
  `__autofilter_ref`, `__legend_blocks`) as not authoritative -- the
  draft's instinct that "the workbook itself is not problematic" is
  the same observation.
* `contract_lifecycle` blocks are the project's only sanctioned way
  to record dual-shape coexistence. If the draft's recommendation
  ever lands as a normative rule for a specific entry, that entry's
  lifecycle block is the right home for the transition wording.

Tensions::

* The draft does not yet say *which registry entries* are in scope.
  Without that, a reader could legitimately conclude that
  `helper_policies`, `frame_lifecycle`, `workbook_view.sheet_mappings`,
  or `constraints` are in scope -- all of which are model-classified
  canonical meta and must not be silently treated as regenerable.
* The registry's classification axis (model / presentation /
  workbook_identity) already encodes the consequence-of-loss
  distinction (`03_meta_classification.adoc`):
  * losing *model* meta is a data-integrity issue;
  * losing *presentation* meta is an aesthetic issue;
  * losing *workbook_identity* meta is a versioning / provenance issue.
  The draft's "losing persisted `_meta` should not normally destroy a
  collaborative project" is only safely true for *presentation* meta
  *and only when that presentation meta is reproducible from
  checked-in configuration*. For model meta it is false; for
  workbook_identity meta (`version`, `author`, `exported_at`) it is
  also false, because reconstruction is not even meaningful.

Direct contradictions::

* As written, the draft would have to either:
  ** scope itself to the Presentation + Reproducible-from-config
     intersection, or
  ** propose changing the canonical-layer definition.
  The first is consistent with the registry; the second is a
  registry-level change of the kind that
  `agent_rules.adoc` requires to be routed through an explicit
  registry-roots checklist.

=== 1.5 Metadata lifecycle concepts

Reference: `docs/semantic_model/08_lifecycle_and_update_semantics.adoc`;
`FTR-DERIVED-PROVENANCE-LIFECYCLE-P4A`;
`FTR-META-LIFECYCLE-INVENTORY-P5`;
`FTR-FK-HELPER-POLICY-LIFECYCLE-P4A`.

Alignments::

* The lifecycle chapter explicitly distinguishes "the runtime view"
  from "the persistable contract" and names the orchestrator-owned
  projection point. The draft restates the same intent in
  collaboration terms.
* "Producers may freely write into the runtime `_meta` view -- including
  under `derived` and under `__*`-prefixed keys -- when the value is
  only needed for the duration of the current run" is already the
  project's rule. The draft's stronger claim ("required metadata
  should be reproducible") generalises it as adoption guidance.
* "Repeated roundtrip risk" is exactly the failure mode the draft
  cares about, viewed through the lens of Git history rather than
  runtime accumulation. They are the same phenomenon at different
  timescales.

Tensions::

* The lifecycle chapter explicitly treats persistence as
  *post-pipeline, write-side only*. The draft proposes a guidance
  rule that operates at *project-organisation* time (deciding what to
  check in). Those are two different governance surfaces. The draft
  does not yet acknowledge that distinction; it should, to avoid
  being read as "the persistence boundary should strip more keys",
  which is *not* what it is asking for.
* The draft's "fresh checkout should be able to reconstruct the
  required workbook behavior through pipeline execution" implies a
  bootstrap-from-zero invariant. The current bootstrap model
  (`bootstrap_meta`, ADR-004) does not yet provide that as a checked
  invariant; it provides merge precedence between config and
  persisted meta. Promoting "fresh-checkout reconstructibility" to an
  invariant would need its own FTR.

Direct contradictions::

* None. The draft is a *strict subset* of what the lifecycle chapter
  contemplates as long as it is scoped to derived / transient /
  reproducible-presentation entries.

=== 1.6 Adoption documents

Reference: `docs/semantic_model/15_adoption_strategy.adoc`;
`docs/warm_storage/adoption_progress/adoption_plan.adoc`;
`docs/warm_storage/concepts/acme_progressive_tutorial_guide.adoc`.

Alignments::

* `adoption_plan.adoc` already nominates
  `FTR-PRESENTATION-DEFAULTS-SEMANTIC-CLASSIFICATION-P5` as
  strategically valid but lower-leverage. The draft's concern lives
  inside that FTR's question space (which presentation behaviors are
  configuration-driven, which are derived, which are
  metadata-driven).
* The semantic model's no-big-bang adoption stance ("adoption proceeds
  transformation family by transformation family, in narrow vertical
  slices") is consistent with the draft remaining a concept until a
  concrete slice consumes it.

Tensions::

* The draft is a *project-organisation guideline*, but the project's
  adoption mechanism is the *transformation-family slice*. There is
  no current slice that owns "collaborative repository hygiene". The
  draft cannot be adopted until a slice claims it.

Direct contradictions::

* None.

=== 1.7 AI guidance

References: `docs/ai_info/agent_rules.adoc`,
`architecture_primer.adoc`, `concepts_fk_helpers.adoc`,
`interfaces_and_gates.adoc`.

Alignments::

* `agent_rules.adoc` already states: "Presentation intent (styles,
  column groups, outlines) is downstream of identity / provenance.
  Presentation must never substitute for identity." That is the same
  asymmetry the draft is feeling out: presentation meta loss is
  recoverable; identity / model meta loss is not.
* `architecture_primer.adoc` already pins the invariant: "Roundtrip
  must not depend on `meta_rendering` alone -- everything needed for
  reconstruction must be in `meta_canonical` or stably persisted from
  it."

Tensions::

* `agent_rules.adoc` requires that any "new pipeline-step categories"
  or new registry roots go through the registry / step-surface
  checklist. The draft does not propose new roots, but its phrase
  "Should the framework provide explicit support for local generated
  `_meta` artifacts?" (Open Questions) would, if accepted, require
  exactly such a registry move. That should be called out as a
  gated change.

Direct contradictions::

* None.

=== 1.8 Existing FTRs and BUGs

Already-named FTRs the draft brushes against:

* `FTR-META-LIFECYCLE-INVENTORY-P5` -- the natural home for any
  future normative statement of the draft's rule per `_meta` root.
* `FTR-DERIVED-PROVENANCE-LIFECYCLE-P4A` -- already enforces
  "deterministic vs heuristic boundary"; the draft must not be read
  as licensing heuristic reconstruction during ordinary pipeline runs.
* `FTR-FK-HELPER-POLICY-LIFECYCLE-P4A` -- governs the FK-helper
  v1/v2 bridge; explicit residual writes are an example of "metadata
  that is *not* reproducible from canonical + config" because they
  carry producer-side history.
* `FTR-PRESENTATION-DEFAULTS-SEMANTIC-CLASSIFICATION-P5` -- the
  classification question the draft implicitly assumes is already
  settled. It is *not* settled; that FTR is still in candidate state.
* `FTR-WORKBOOK-VIEW-PAYLOAD-CONFLICT-PRECEDENCE-P6` -- enforces:
  "Execution steps must not use heuristics to decide winners. ...
  Precedence must be explicit and configuration-driven." The draft
  must be read as *not* relaxing this rule.
* `BUG-UNRESOLVED-HELPER-VALUE-NAN-SEMANTICS-P4A` -- the most
  important interaction. See section 7.
* `BUG-HELPER-CLEANUP-CONTRACT-P4A` -- helper-column leak twin;
  illustrates that "treat as derived" is correct *only* if the
  derivation pipeline actually cleans up before the artifact is
  written.

== 2. Existing support

The draft's recommendation is not new at architectural level. It is
new at the *adoption-guidance* level.

What already supports the direction::

* The persistence boundary
  (`pipeline/persistence_boundary.py`) materially encodes "derived
  channels do not survive into the persisted contract".
* The registry's three-value layer axis treats `derived` as
  recomputable by definition.
* The canonical-vs-derived concept document
  (`metadata_canonical_vs_derived.adoc`) explicitly says: "Derived
  meta is *recomputable*. If lost, the next run can regenerate it
  from the canonical sources plus the current frame state."
* The lifecycle chapter names the "Repeated roundtrip risk" failure
  mode and the persistence boundary as the existing mitigation.
* `agent_rules.adoc` already states "presentation must never
  substitute for identity".

What assumes the opposite::

* The canonical-layer definition explicitly requires that canonical
  meta be *persistable* and *roundtrip-relevant* -- not regenerable
  from configuration. Canonical entries (`helper_policies`,
  `constraints`, `frame_lifecycle`, `workbook_view.sheet_mappings`)
  are not in the draft's scope and the draft does not yet say so.
* The Presentation-classification motivating case (`column_widths`)
  *requires* persisted-meta survival: the user's keystroke is the
  canonical source, not the pipeline configuration. Treating it as
  generated would lose information.
* The workbook_identity bucket (`version`, `author`, `exported_at`)
  is not reconstructable from canonical data + checked-in
  configuration, by definition.
* `ADR-004`'s merge priority (Runtime Config -> Persisted Meta ->
  Heuristics/Defaults) places persisted meta *above* heuristics, not
  below configuration. The draft's "should be reproducible" rule
  would, if naively read, invert that.

Is the concept new?::

The *architectural conclusion* is not new; the *adoption posture* is.
Concretely, the new piece is: "in a collaborative-Git context, prefer
configurations where persisted `_meta` is the regenerable side rather
than the authoritative side, because that produces fewer merge
conflicts and fewer hard-to-review diffs".

That is a guideline about *how to organise a consumer project*, not
about *how the framework treats metadata*. The framework already
treats derived meta as derived. The draft adds: "when you have a
choice, prefer the derived path".

== 3. Metadata analysis

The draft distinguishes four sources:

. canonical data,
. pipeline/configuration,
. workbook-embedded metadata,
. repository-persisted metadata.

The registry already distinguishes:

* layer: `canonical`, `derived`, `operational-carrier`;
* classification: `model`, `presentation`, `workbook_identity`;
* sets: `Canonical`, `Derived`, `Persistent`, `Transient`,
  `RoundtripRelevant`, `RenderConsumed`, `RuntimeConsumed`,
  `UserEditable`, `ProducerOwned`, `Configurable`,
  `HeuristicallyInferred`, `Policy`, `Provenance`, `PublicSurface`,
  `InternalOnly`, `Deprecated`, `TransitionBridge`.

=== 3.1 Mapping the draft's four sources onto existing axes

[cols="1,2,2",options="header"]
|===
| Draft source | Existing-model mapping | Notes

| canonical data
| frame data, layer-independent of meta
| Maps to "the data layer" in `01_data_and_meta.adoc`. `canonical`
  at the meta layer is *not* the same word; the draft uses
  "canonical" data-layer-style and that is fine, but the review
  should flag it.

| pipeline/configuration
| `Configurable` source class; YAML/CLI
| Source class with its own authority posture (`11_conflict_and_precedence_questions.adoc`).

| workbook-embedded metadata
| `operational-carrier` layer when carrier-shaped; otherwise
  canonical claims carried in the hidden `_meta` sheet
| Per `04_carrier.adoc`, the workbook *is* the carrier; embedded
  meta is the carrier-shape of canonical claims. The draft treats
  it as relatively benign, which is correct *if and only if* it is
  a single-user roundtrip surface; in multi-author Git
  collaboration the embedded form is just another version-controlled
  payload.

| repository-persisted metadata
| Sidecar payloads (`_meta.yaml`, `*.meta.json`) +
  workbook hidden-meta sheet when the workbook is committed
| The draft's "repository-persisted" category is the *commit-surface
  view* of the persistable contract, not a new layer.
|===

=== 3.2 Per-classification analysis

==== Model meta

The draft's rule cannot apply to model meta without a contract change.
Model meta is by definition "claims about durable structural
properties of the data". Losing it is a data-integrity issue.
Examples in the live registry: `helper_policies`, `frame_lifecycle`,
`constraints`, `workbook_view.sheet_mappings`, `legend_blocks`,
transformation configuration roots (`xref_crosstable`,
`split_by_discriminator`, `compact_multiaxis`, `cell_codecs`,
`sparse_defaults`).

Some of these are arguably reconstructable from checked-in pipeline
configuration -- e.g. transformation configuration roots if the
pipeline YAML names them explicitly. Others, notably
`workbook_view.sheet_mappings` and user-authored `constraints`, are
*not* reproducible from canonical data + configuration alone, because
they carry information the canonical data does not contain.

Recommendation: the draft should explicitly exclude Model meta from
its rule, or scope itself to the strict subset that is *demonstrably*
reconstructable.

==== Presentation meta

The Presentation classification is the natural home for the draft's
rule. But even here the rule is *not* uniform:

* `header_fill_rgb`, `helper_fill_rgb` -- reconstructable from
  checked-in configuration; safe to treat as generated.
* `freeze_header`, `auto_filter` -- declaratively settable;
  reconstructable from checked-in configuration; safe to treat as
  generated.
* `column_widths` -- the motivating case for treating Presentation as
  canonical-layer. User-authored widths *cannot* be reconstructed
  from canonical data + configuration; the user's keystroke is the
  source.

Recommendation: the draft's rule applies cleanly only to *bootstrap-
default* presentation (fills, freeze, autofilter) and breaks for
*user-authored* presentation (column widths, hand-set styling
overrides). This split deserves an explicit sentence.

==== Derived meta

Already governed by the existing rule (derived = recomputable).
The draft is restating the status quo for this classification.

==== Operational-carrier meta

Not a draft concern. The draft's "workbook-embedded metadata" is, in
the live encoding, an operational-carrier shape of canonical meta.
The draft's claim that "the workbook itself is not problematic" is
correct under single-user roundtrips and *incorrect* under
collaborative Git when the workbook itself is committed -- the
operational-carrier payload is then just as much a merge surface as
a `_meta.yaml` sidecar.

Recommendation: clarify the scope -- *carrier-as-commit* and
*carrier-as-handoff* are different problems.

==== Lifecycle concepts

The draft is fully consistent with the lifecycle chapter when scoped
to derived + transient + reconstructable-presentation entries. It is
inconsistent with the lifecycle chapter if applied to canonical or
operational-carrier entries.

=== 3.3 Hidden assumptions

. *"Metadata" is a flat category.* The biggest hidden assumption.
   The project already classifies meta along two axes; treating
   "persisted `_meta`" as one merge surface is an over-flattening
   that the registry has been built specifically to avoid.
. *Fresh-checkout reproducibility is already achievable.* It is the
   *direction*, but not the *current state* for several roots
   (e.g. `workbook_view.sheet_mappings`, user-authored
   `column_widths`).
. *Workbook-embedded metadata is benign.* True for single-user
   roundtrip, not necessarily true when the workbook file itself is
   committed -- in that case binary XLSX/ODS files create their own
   review-and-merge problems independent of `_meta`.
. *Pipeline determinism is already strong enough to support
   regeneration.* For derived meta this is largely true. For
   presentation defaults the determinism depends on whether
   `bootstrap_meta` / `add_lookup_helpers` ordering is stable across
   environments; the project does not yet treat that as a checked
   invariant.

== 4. Adoption perspective

=== 4.1 Git-based collaboration

The draft does describe a real problem:

* `_meta.yaml` and the hidden `_meta` sheet in committed XLSX/ODS
  files generate diffs that include keys whose value the author did
  not consciously set (e.g. inferred FK relations,
  configure_fk_helpers residuals, bootstrap defaults).
* These diffs create merge conflicts whose resolution is
  semantically opaque -- the reviewer cannot easily tell whether
  a `helper_policies.fk` line is the user's intent or a producer
  residual.
* `BUG-RUNTIME-META-PERSISTENCE-BOUNDARY-P4A` was opened exactly
  because such a residual was leaking into sidecars from prior
  runs.

The draft's recommendation -- keep the canonical data + checked-in
configuration as the *shared truth*, treat `_meta` as
working/generated state -- is genuinely useful collaboration advice
in that environment.

What it does *not* solve::

* Binary workbook files committed to Git are *themselves* an
  unmergeable surface, independent of `_meta`. Two analysts editing
  the same XLSX still produce a conflict whose only resolution is
  "pick one". The draft does not address this.
* User-authored presentation (widths, deliberate overrides) is
  exactly the data that benefits from being persisted, not
  regenerated. The draft does not exclude this case explicitly.
* Workbook_identity meta (`version`, `author`, `exported_at`) is
  precisely *not* reproducible and is precisely the kind of metadata
  a multi-author Git workflow wants to see in diffs.

=== 4.2 Multiple business analysts

The use cases the framework already targets (business calculation
roundtrip, worldbuilding continuity, generic loan configuration) are
multi-analyst by design. The draft's intuition is right that "fewer
merge surfaces" is a real adoption win.

However, the dominant multi-analyst risk in the existing model is not
`_meta` merge conflicts; it is *editable-column overlap*
(`FTR-WORKBOOK-VIEW-PAYLOAD-CONFLICT-PRECEDENCE-P6`). That FTR
records the project's position: detection + explicit
configuration-driven precedence, never heuristic merging. The draft
should not be read as relaxing it.

=== 4.3 Review-oriented roundtrip workflows

The draft's framing -- "what does a reviewer see in a PR diff?" --
is *new* in the docs and is genuinely valuable. The project's review
culture (per `review_instructions.adoc`, `git_and_workflow.adoc`) is
heavily artifact-based; reducing the volume of incidental meta in
diffs would materially help reviewers.

This is the draft's strongest contribution. It deserves to be
preserved even if every other claim is descoped.

=== 4.4 New risks introduced

. *Demoting a canonical claim by accident.* The biggest risk.
   Without an explicit "what is in scope" list, an over-eager reading
   would treat user-authored `column_widths` or
   `workbook_view.sheet_mappings` as regenerable and silently lose
   information across collaborative runs.
. *Hidden heuristics re-entering the pipeline.* If "regenerable"
   ever means "the pipeline guesses from data shape", the project's
   stated boundary
   (`FTR-DERIVED-PROVENANCE-LIFECYCLE-P4A`,
   `FTR-WORKBOOK-VIEW-PAYLOAD-CONFLICT-PRECEDENCE-P6`,
   `infer_fk_relations` as a *configuration* step) is breached.
. *Encoding-determinism debt.* "Fewer diffs" assumes the persisted
   form is byte-deterministic across runs and environments. Today
   the JSON `_meta.yaml` serializer is not under a checked
   determinism contract; cross-environment YAML/JSON ordering or
   formatting drift would defeat the draft's promise even if every
   semantic claim were stable.
. *Payload semantics loss.* See section 7.

== 5. Placement recommendations

=== 5.1 Recommended placement

The concept is *adoption guidance*, not a model, an architecture
spec, or an AI rule.

If it is adopted in any form, the right home is:

* a chapter in `docs/warm_storage/concepts/` (alongside
  `acme_progressive_tutorial_guide.adoc`) while still exploratory; or
* a section in `docs/developer_guide/` once stabilised, in the
  collaboration-and-workflow area, beside the existing
  `ch08_git_and_workflow/` content.

Reasoning: the existing warm-storage `_concepts.adoc` already
declares the area as "concept drafts that may still affect Phase 3
planning". The draft fits that purpose precisely.

=== 5.2 Where it should *not* be placed

* *Not* `docs/functional_model/`. The concept does not describe a
  use case, a domain transformation, or a data model. Placing it
  there would mislead readers into thinking the framework provides
  a new capability.
* *Not* `docs/technical_model/ch04_concepts/` or
  `ch05_registries/`. The draft proposes a *posture*, not a
  mechanism. The technical-model concepts are about how metadata
  works; this is about how a *project* should be organised.
* *Not* `docs/technical_model/ch03_architecture_decisions/`. It is
  not an ADR -- it does not record a binding architectural
  decision, and adopting it as one would conflict with the
  canonical-layer definition.
* *Not* `docs/ai_info/`. AI guidance constrains agents working
  inside the repository. This is project-level adoption advice and
  belongs in human-facing docs.
* *Not* the semantic model
  (`docs/semantic_model/`). The semantic model classifies the
  *framework's* attributes; this concept is about how a *consumer
  project* should be organised. Mixing them would make the
  semantic model larger and less coherent.

=== 5.3 Classification

Primary category: *adoption guidance*.

Secondary category: *future contributor to
`FTR-META-LIFECYCLE-INVENTORY-P5` and
`FTR-PRESENTATION-DEFAULTS-SEMANTIC-CLASSIFICATION-P5`* -- both of
which need the project to take a position on per-root
"persist vs regenerate" defaults. The draft can feed those FTRs
without being adopted as doctrine in its own right.

It is *not* a new concept category. The project already has the
axes; the draft just argues for a default along one of them.

== 7. Payload semantics boundary

This is the most important technical observation in the review.

=== 7.1 The concrete case

`BUG-UNRESOLVED-HELPER-VALUE-NAN-SEMANTICS-P4A` documents a live
contract gap: when an FK source cell is empty, the materialised
helper value reaches canonical JSON as the literal string `"nan"`. A
consumer of that JSON *cannot distinguish*:

* a helper that resolved to the in-domain string `"nan"`;
* a helper that failed to resolve because the FK source was empty;
* a helper that failed to resolve because the FK target was missing;
* a helper that failed to resolve for some other reason (type
  coercion, formula error, parser fallback).

That is a *payload semantics* failure. The visible cell value alone
does not carry the meaning; the missing context is currently nowhere.

The BUG documents three candidate contracts (empty/null, structured
finding, spreadsheet-error preservation). None has been chosen.

=== 7.2 Why this matters for the draft

The draft's central claim is that persisted `_meta` should be
treatable as regenerable. That claim is *only* safe if no payload
semantics depend on metadata.

If, for example, the project ever chose the *structured finding*
contract for unresolved helper values, the "this cell was unresolved"
fact would live in metadata, not in the visible cell. Discarding
that metadata would silently convert "unresolved" into "the empty
string", which is a different in-domain meaning.

Symmetrically, the project should *guarantee* that payload semantics
*never* depend solely on metadata, precisely so that the draft's
rule can be applied safely. The draft does not articulate this
guarantee, but it depends on it.

=== 7.3 Direct answers to the section-7 questions

==== "Can repository-persisted metadata safely be removed if payload semantics depend on it?"

No. If a payload's meaning is recoverable only through metadata, the
metadata is *part of* the canonical payload contract regardless of
how the registry classifies it. Removing it would be semantic data
loss, not just metadata loss.

==== "Which kinds of information must remain recoverable from canonical data or explicit rendered data rather than metadata?"

At minimum (provisional list; this would be formalised in
`FTR-META-LIFECYCLE-INVENTORY-P5`):

* The *resolved-vs-unresolved* status of a helper value (today
  ambiguous; see the BUG).
* The *formula-error* status of a cell (`#NAME?`, `#N/A`, `#REF!`).
  Today XLSX/ODS preserve these in the carrier; the project has no
  documented contract that they survive in JSON output.
* The *NaN-vs-empty* distinction (today both surface as the literal
  `"nan"`).
* The *user-edited-vs-default* status when a presentation value
  was explicitly authored. If we want to preserve user intent
  across roundtrips, that intent has to be reified somewhere that
  the draft's rule does not delete.
* The *explicit-policy-vs-inferred-heuristic* origin of any
  `_meta.helper_policies.fk.relations` entry (already encoded in
  `produced_by.step` + `produced_by.mode`; the draft must not
  license its loss).

==== "Does the current metadata model already define this boundary?"

Partially.

* `agent_rules.adoc` states: "Presentation must never substitute
  for identity." That is the closest existing statement.
* `metadata_canonical_vs_derived.adoc` distinguishes "transport
  fact" from "semantic role" but does not yet say "payload
  semantics must not live in metadata".
* The Conflict-and-Precedence chapter states that user-visible
  workbook content overrides hidden metadata for Canonical content,
  but does *not* address the converse case where the *meaning* of
  a visible cell is encoded in metadata.

A short explicit statement -- "payload semantics must not exist
solely in metadata" -- is not yet in the docs. The BUG above is a
live counterexample to its absence.

==== "Would the collaborative repository concept require an explicit rule that payload semantics must never exist solely in metadata?"

Yes. This is the review's strongest single recommendation. If the
draft is adopted in any form, it must be paired with an explicit
project rule of the form:

____
*Payload semantics must not exist solely in metadata.*

A visible cell whose *meaning* depends on a metadata claim that is
not also derivable from the cell content + canonical data must
either:

* surface the missing meaning into canonical data (e.g. a sentinel
  value, a separate column, an explicit "unresolved" representation), or
* preserve the metadata as canonical-layer and never as regenerable
  derived state.

The rule applies regardless of any collaboration or merge-safety
preference.
____

Without that rule, the draft's "metadata loss is normally harmless"
posture is unsafe in the project's current state (as
`BUG-UNRESOLVED-HELPER-VALUE-NAN-SEMANTICS-P4A` demonstrates) and
would silently encode a contract regression.

This rule is also useful *independently* of the draft, because it
gives the unresolved-helper BUG and the visible-cell-typing FTR
(`FTR-VISIBLE-CELL-TYPING-STRATEGY-P5`) a shared backbone.

== Findings

Findings use the project's review-instructions disposition shape.
Priorities reflect the *concept's status as exploratory*; no finding
implies blocked release work, since the draft is not yet shipping
anywhere.

=== Watch-only

[cols="1,3"]
|===
| F-1 | The concept's substance is broadly congruent with existing
        project direction; the existing persistence boundary,
        canonical-vs-derived concept, and registry layer axis
        already implement most of what the draft asks for in the
        derived/transient subset. Keeping the draft as
        exploratory adoption guidance is the right disposition.
|===

=== Before-low-touch / maintainability

[cols="1,3"]
|===
| F-2 | The draft uses "metadata" as a flat category. The project
        already classifies meta on two orthogonal axes. The draft
        should be scoped to the Presentation-classification +
        regenerable subset, *or* it should be read as a project-
        organisation guideline that explicitly does not redefine
        registry semantics.
        Recommended action: add a "scope" subsection naming which
        registry entries the rule applies to and which it does
        not. Priority: low; do this when (or if) the draft moves
        toward adoption.

| F-3 | The Presentation classification is not internally uniform.
        Bootstrap-default presentation (`header_fill_rgb`,
        `freeze_header`, `auto_filter`) is regenerable; user-authored
        presentation (`column_widths`) is not. The draft needs a
        sentence on this split; otherwise a reader could apply the
        rule to the wrong subset.

| F-4 | "Workbook-embedded metadata is not problematic" is true
        for single-user roundtrip and *not necessarily* true under
        committed-binary-workbook workflows. The draft should
        distinguish *carrier-as-handoff* (XLSX exchanged outside
        Git) from *carrier-as-commit* (XLSX committed to Git).
|===

=== After-release improvements

[cols="1,3"]
|===
| F-5 | The Open Question about "explicit support for local
        generated `_meta` artifacts" would be a new registry root
        or a new operational layer if adopted, and would be a
        gated change per `agent_rules.adoc`. The draft should
        flag this explicitly so a reader does not interpret the
        Open Question as a small refinement.

| F-6 | "Required metadata should be reproducible from canonical
        data, checked-in configuration, deterministic transformation
        logic" is a useful direction, but the project does not yet
        have a checked determinism contract on the
        `_meta.yaml` / JSON sidecar serialization path. The
        promise the draft is making is currently aspirational; the
        draft should say so.
|===

=== Blockers for any *normative* adoption

(These are not blockers for the draft remaining in warm storage;
they are blockers for promoting it into binding doctrine.)

[cols="1,3"]
|===
| F-7 | The canonical-layer definition is explicitly broken by an
        unscoped reading of the draft. Promoting the concept must
        either reword it to apply within layers (derived,
        regenerable-presentation), or open an explicit FTR to
        change the canonical-layer definition. The latter is a
        registry-roots-class change.

| F-8 | The unresolved-helper-value contract gap
        (`BUG-UNRESOLVED-HELPER-VALUE-NAN-SEMANTICS-P4A`) is a
        live counterexample to "metadata loss is harmless". Any
        normative adoption of the draft must be paired with the
        explicit "payload semantics must not exist solely in
        metadata" rule recommended in section 7, *and* the BUG's
        contract must be decided.

| F-9 | `workbook_view.sheet_mappings` is canonical model meta
        whose value is not reconstructable from canonical data +
        pipeline configuration alone (the visible sheet labels are
        user-editable and the mapping is the
        ground truth). The draft's rule must not apply to it. Any
        normative wording must exclude it explicitly.
|===

== Residual risks

. The biggest residual risk is *the draft being read as more
   normative than it is*. The draft itself flags this ("not currently
   a mandatory architectural rule"); this review preserves that
   posture.
. The draft's Open Questions remain genuinely open. This review
   does not answer them; it places them.
. The "payload semantics must not exist solely in metadata" rule
   recommended in section 7 is, on its own, a project-level
   architectural commitment that deserves its own consideration. It
   is recommended here because it is the cheapest way to make the
   draft's posture safe; it has independent merit and would not be a
   bad rule in any case.
