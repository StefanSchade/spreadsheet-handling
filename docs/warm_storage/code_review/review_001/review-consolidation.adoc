=== Beobachtungen zur Rohliste

* *Doppelungen / Cluster bereits erkennbar:*
  ** D-001 ≡ IR-002 — derselbe Sachverhalt: D-001 ist der *Symptom-Ort* (`domain` importiert `rendering`), IR-002 der *Fix-Ort* (`rendering/formulas.py` ist falsch platziert).
  ** A-004 ≡ IR-008 — identisches Problem, einmal aus IR-Sicht (Phase 2), einmal aus Adapter-Sicht (Phase 3). In Schritt 2 als *ein* Finding behandeln.
  ** D-003 + D-002 — FK-Namensüberlappung und FK-Domänenlogik im Core hängen zusammen (gemeinsamer Fix: `domain/fk/`-Paket).
  ** IR-003 + IR-004 — `passes/`-Aufräumen löst auch die doppelte Pass-Pipeline.
* *Schichten-Verteilung:* Phase 1 (`domain`/`core`) trägt die einzige echte hexagonale Verletzung + den Großteil der Hygiene-Schuld (überlange Module, FK-Wildwuchs, leere `__init__`). Phase 2 (`rendering`/`pipeline`) ist architektonisch sauber — fast nur Struktur/Hygiene. Phase 3 (`io_backends`/`cli`/`application`) trägt die Funktions-Bugs (A-002, A-003) und die Dispatch-Triplikat-Schuld (A-001).
* *Querschnitt-Themen:* (a) Modul-Header-Konvention fehlt überall (D-012, IR-014); (b) `meta_registry.yaml` ist unvollständig (`derived` fehlt, IR-005); (c) `exported_at`-Platzhalter zieht sich durch alle drei Phasen (IR-013, A-? in Phase 3).

=== Fehlende Dokumentation

_Was in `ai_info/` fehlt und ergänzt werden sollte:_

* *Der `derived`-Meta-Kanal (IR-005)* — `domain/transformations/{enrich_lookup,fk_helpers}.py` schreiben `_meta["derived"]["sheets"]`, `rendering/passes/core.py` liest es; weder im Primer noch in `meta_registry.yaml` erwähnt. Sollte als transienter `meta_rendering`/`derived_operational_view`-Kanal beschrieben *und* registriert werden (C6).
* *`workbook_meta_blob`-Serialisierungsformat (IR-008/A-004)* — der Primer nennt die `WriteMeta`-Op, aber nicht, dass der canonical-Blob heute als Python-`repr` (`str(dict)`) persistiert und per `ast.literal_eval` zurückgelesen wird (fragil bei NaN/Nicht-Literalen). Verdient eine eigene ADR ("Meta-Blob-Persistenz: JSON, kein `repr`") + einen Satz im Primer/`generic_ir.adoc`.
* *`exported_at`-Status (IR-013)* — der canonical-Meta-Key wird von `MetaPass` reserviert, aber nie befüllt. Der Primer enumeriert die `meta_canonical`-Keys nicht (das tut `meta_registry.yaml`), aber wenn `exported_at` bleibt, gehört dokumentiert *wer* ihn setzt (CLI/Orchestrator) und *dass über einen injizierbaren Clock-Wert* — Determinismus.
* *`meta_registry.yaml` als Pflicht-Artefakt* — der Primer verweist nur indirekt (`generic_ir.adoc`-Pointer). Es fehlt die explizite Regel in "Where new code belongs": jede gelesene/geschriebene Meta-Wurzel braucht einen Registry-Eintrag mit *importierbaren* `producer`/`consumer`-Refs, Layer (`meta_canonical`/`meta_rendering`) und Klassifikation.
* *Lazy-Import der optionalen Spreadsheet-Adapter* — `router._lazy_callable` / `io_backends/__init__.__getattr__` importieren `openpyxl`/`odfpy` erst bei Bedarf; das ist *Absicht* (Umgebungen ohne einen der beiden Adapter sollen die Lib nicht zwangsimportieren), steht aber nur implizit im Code. Verdient eine ADR ("Optionale Spreadsheet-Adapter werden lazy importiert"), damit eine spätere Registry-Konsolidierung (C5) das nicht versehentlich zurückbaut.
* *Modul-Header-Konvention (D-012/IR-014)* — `conventions.adoc` regelt Funktions-Größe, aber nicht Modul-Header (Purpose/Layer/erlaubte Imports) und nicht Modul-Größe. Sollte ergänzt werden (C8).
* *Erlaubte Import-Richtungen als ausführbare Regel* — aktuell Prosa (Layer Map) + verstreute Tests. Es fehlt ein konsolidierter "wer-darf-wen"-Block (idealerweise generiert aus den `dependency_guards`-Tests), den ein Agent vor einer Änderung liest, ohne die Tests zu öffnen.
* *`sheets-run` als gegateter Surface?* — `interfaces_and_gates.adoc` listet `sheets-pack`/`sheets-unpack`, nicht aber `sheets-run` (die Haupt-CLI-App, 178 Z.). Klären, ob `sheets-run` + seine Flags ebenfalls Produkt-API sind, und ergänzen.

=== Irreführende Stellen für KI-Agenten

_Stellen, die einen Agenten zu falschen Entscheidungen verleiten — Kurzliste, Details in der Tabelle oben:_

. *"New render behavior → Pass in `rendering/passes/`"* — ein neu angelegtes `passes/<name>.py` wird nicht verdrahtet (alles läuft über `core.py`). Höchstes Irreführungspotenzial.
. *Layer Map `Pipeline/Application → Render IR`* — verleitet zu der Annahme, ein Pipeline-Step dürfe `compose_workbook` aufrufen. Tut keiner; soll auch keiner.
. *`core/` "FK detection"* — endorst eine bekannte Fehlplatzierung; ein Agent würde neue FK-Heuristik nach `core/` legen statt nach `domain/`.
. *"New IR datatype → `rendering/ir.py`"* — für einen strukturellen `*FormulaSpec` falsch (gehört nach `core/formulas.py` nach C1).
. *"register kind in `router.py`"* (Backends) — unvollständig wegen des Dispatch-Triplikats (A-001); das Backend wäre über zwei der drei Pfade unsichtbar.
. *`orchestrator.py` "backward-compatibility shim"* — ein Agent könnte es als entfernbaren Cruft einstufen; es ist gegateter Public-Surface.
. *"Architecture tests … enforce layer boundaries automatically"* — suggeriert Vollständigkeit; ein Agent könnte annehmen, ein `domain → rendering`-Import sei ok, weil "die Tests würden's ja fangen" (tun sie nicht — L1).

=== Nicht betroffen

_Was der Primer korrekt und vollständig beschreibt — gegen die Reviews bestätigt:_

* Die *"zwei Flüsse"-Trennung als Konzept* (Pipeline ⊥ Render-Flow): im Code strikt eingehalten — `pipeline/` ⊥ `rendering/`, der Render-Flow startet im Backend-Saver. (Nur das *Diagramm* hat die Phantomkante; der erläuternde Text stimmt.)
* *`meta_canonical → meta_rendering`-Invarianten* ("nach IR-Derivation beeinflusst `meta_canonical` das Render-Ergebnis nicht mehr direkt"; "Adapter führen den `RenderPlan` aus, re-interpretieren `meta_canonical` nicht"): test-verifiziert sauber (Phase 2 — `__`-Keys-Ableitung, `test_visible_render_result_is_stable_without_hidden_meta_payload…`). Einzige Trübung: der nicht-registrierte `derived`-Kanal (IR-005) — der bricht keine Invariante, ist aber undokumentiert.
* *CLI "no business logic"* und *Pipeline "passes payloads through without interpreting semantics"*: bestätigt — `pipeline/runner.py` ruft nur `step(out)` in einer Schleife; `pipeline/__init__.py`/`registry.py` enthalten keine Workbook-Meta-Routing-Tokens.
* *"Steps execute in declared order; no branching"*: bestätigt.
* *"JSON/CSV/YAML backends bypass [the rendering] layer entirely"*: bestätigt — `application/orchestrator.py` und die einfachen Backends importieren nichts aus `rendering/`.
* *Parser stoppt bei `WorkbookIR`; Projektion (`workbookir_to_frames`) ist generisch*: bestätigt + dicht geguarded.
* *Key-Types-Tabelle* (`Frames`, `WorkbookIR`, `SheetIR`, `TableBlock`, `RenderPlan`, `meta_canonical`/`meta_rendering`): inhaltlich korrekt.
* *`conventions.adoc`* insgesamt: konsistent mit den Reviews — einzige *Lücke* (kein *Fehler*) ist die fehlende Modul-Header-/Modul-Größen-Regel; einzige *Spannung* ist "prefer immutability" vs. der Render-Pfad-Mutation (IR-006), aber das ist ein Code-Bug gegen die Konvention, kein Doku-Fehler.
* *`project_context.adoc`*: korrekt; minimal unvollständig ("JSON/CSV ↔ XLSX/ODS" — XML- und YAML-Backends existieren ebenfalls), aber nicht irreführend.

* *(a) Sofort korrigieren* — die Doku beschreibt den **Ist**-Zustand *falsch*; die Änderung ist eine reine Korrektheits-Reparatur, unabhängig von jeder Code-Arbeit.
* *(b) Soll-Zustand + Ist-Vermerk* — die Doku beschreibt den **Soll**-Zustand *korrekt*, aber der Code weicht ab. Soll-Beschreibung *behalten*, einen Hinweis `Status: ausstehend — tracked als <Finding-ID>` ergänzen; nach Umsetzung des Findings den Hinweis entfernen.
* *(c) Neu dokumentieren* — fehlt komplett.

_Kein fertiger Text — Stichpunkte, sodass ein nachfolgender Agent/Mensch es direkt umsetzen kann._

== Gesamtfazit

=== Architektur-Qualität in 3 Sätzen

Der Kern (`domain/`, `rendering/`-IR, `pipeline/`, Spreadsheet-Adapter) ist hexagonal sauber geschnitten und ungewöhnlich dicht durch Architektur-Guards abgesichert — die zentralen Layer-Invarianten (`pipeline ⊥ rendering`, `rendering ⊥ Backend-Libs`, Formula-Boundary, Parser stoppt bei `WorkbookIR`, `meta_canonical → meta_rendering`) halten im Code. Es gibt genau *eine* echte hexagonale Verletzung (`domain → rendering.formulas`, D-001), deren Wurzel eine Fehlplatzierung ist und deren Fix mechanisch ist; der Rest sind Hygiene-/Struktur-Themen (FK-Wildwuchs, `passes/`-Stubs, drei Backend-Registries, überlange Module) und zwei reale Funktions-Bugs in den *Rand*-Schichten (YAML/XML-Writer crashen bei `_meta`; `sheets-pack --input-kind csv_dir`). Die echten Risiken sind nicht "kaputte Architektur", sondern (1) die fragile `repr`-Serialisierung der Meta-Payload, die einen stillen Roundtrip-Bruch bei NaN/Nicht-Literalen erlaubt, und (2) die Doku, die an mehreren Stellen den Ist-Zustand falsch beschreibt und einen KI-Agenten in die Irre führt.

=== Die 5 wichtigsten nächsten Schritte

. *C1 — `rendering/formulas.py` → `core/formulas.py` + `domain ⊥ rendering`-Guard.* Behebt die einzige echte Layer-Verletzung (D-001), ist mechanisch (~½ Tag), und schließt Lücke L1. Höchster Architektur-Hebel pro Aufwand.
. *Quick Wins bündeln (1 PR, <2 h):* `_interfaces.py` löschen (A-006), `coerce_backend_options(None) → BackendOptions()` (A-007), `_meta`-Skip in `xml_backend`/`yaml_backend` (A-003-Teil — behebt einen echten Crash), tote Step-Factories löschen (IR-007), `_deep_merge` & Co. öffentlich machen (D-007). `csv_dir`-Choice aus `sheets-pack` (A-002) *nur nach Nutzer-Freigabe* (gegateter CLI-Flag).
. *C4 — `workbook_meta_blob` auf JSON umstellen* (~2 h + NaN-Roundtrip-Test) und die zugehörige ADR schreiben. Beseitigt das stille-Roundtrip-Bruch-Risiko.
. *Doku-Sofortkorrekturen (a)-Kategorie* in `architecture_primer.adoc` + der falsche ADR-Pfad in `agent_rules.adoc` (~1 h). Verhindert, dass KI-Agenten weiter auf Basis falscher Layer-/Pfad-Angaben arbeiten — billig, hoher Hebel für die KI-gestützte Weiterentwicklung.
. *C3 — `passes/core.py` entlang der Dateinamen aufspalten + tote Stubs löschen + eine `default_passes()`-Quelle* (~½–1 Tag). Größter Lesbarkeits-Hebel im Render-Layer (für Mensch *und* KI) und beseitigt die irreführendste Primer-Stelle ("Pass in `rendering/passes/`").

_(Danach, mittelfristig: C2 FK-Konsolidierung, C5 Backend-Registry, C6 `meta_registry.yaml`-Vervollständigung, C7 Modularisierung/`__all__`, C8 Modul-Header.)_

=== Zustand für PyPI-Beta

Vertretbar für Beta-Nutzer. Die hexagonale Architektur ist real und getestet; die offene Layer-Verletzung (D-001) ist *intern* und für Konsumenten unsichtbar. Was Beta-Nutzer treffen *könnte*: A-003 (`sheets-run --out-kind yaml_dir|xml_dir` crasht bei meta-erzeugenden Pipelines) und A-002 (`sheets-pack --input-kind csv_dir` → `ValueError`) — beides schmale, schnell behebbare Funktions-Bugs auf wenig genutzten Pfaden. Vor einem *Stable*-Release sollten behoben sein: A-003 + A-002 (echte Bugs), C4/IR-008 (Roundtrip-Robustheit — Kernversprechen der Bibliothek), C1/D-001 (sonst zementiert sich der Layer-Verstoß), und die A-005-Inkonsistenz (CLI-Fehler-UX greift im installierten Pfad nicht). Die Hygiene-Cluster (C2/C3/C7/C8) sind kein Release-Blocker, aber je länger sie offen sind, desto teurer wird KI-gestützte Weiterentwicklung.

=== Empfehlung für KI-gestützte Weiterentwicklung

* *`architecture_primer.adoc` ist das wichtigste Steuerdokument für Agenten — und enthält die meisten irreführenden Stellen.* Die (a)-Korrekturen (Layer-Map-Kante, `core/` "FK detection", `orchestrator.py`-Rolle, Pass-Reihenfolge, Guard-Vollständigkeit) sofort umsetzen; die (b)-Vermerke (`Status: ausstehend — tracked als …`) bei "New render behavior", "New IR datatype", "New … backend" ergänzen, damit ein Agent den Soll-Zustand kennt *und* weiß, dass der Code noch abweicht.
* *Den falschen ADR-Pfad in `agent_rules.adoc` (Z. 44) korrigieren* — ein Agent, der den "canonical sources"-Block befolgt, findet die ADRs sonst nicht.
* *Modul-Header-Konvention (C8) einführen und in `conventions.adoc` festschreiben* — der mit Abstand günstigste Hebel für token-effizientes Arbeiten: `head -5` statt "ganzes Modul lesen". Kombiniert mit stabilen `__all__` pro Sub-Paket (D-010) wird ein großer Teil der "Agent muss zu viel Kontext laden"-Last erschlagen.
* *Die nicht-geguardeten Layer-Kanten (L1–L8) in `architecture_guard_inventory.adoc` als bekannte Lücken führen* — damit ein Agent nicht fälschlich annimmt, "die Tests fangen das schon". Nach C1 (L1 geschlossen), und idealerweise zusätzlich ein `domain ⊥ third-party-libs`-Guard (L2) und ein Determinismus-Guard (L3).
* *Beim Schreiben neuer Steps/Meta-Keys den Doppel-Eintrag (`pipeline_step_registry.json` + `meta_registry.yaml`) explizit in die "Where new code belongs"-Tabelle aufnehmen* — die Guards erzwingen ihn, der Primer erwähnt ihn aber nicht; ein Agent läuft sonst in einen Test-Fehler, dessen Ursache er erst suchen muss.
