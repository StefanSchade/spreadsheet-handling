= Current Working Method
:toc: left
:toclevels: 2

====
This document is intentionally descriptive.

It records current practice.

It does not define project policy, governance, or mandatory workflow.
====


== Purpose

The project increasingly combines:

* architectural exploration,
* implementation work,
* independent reviews,
* FTR-driven evolution,
* AI-assisted development.

As the project grows, successful practices tend to become habits.

This document records those habits.

== Current Delivery Flow

Typical work currently follows this pattern:

[source,text]
====

Idea
↓
Concept exploration
↓
FTR (or review of an existing FTR)
↓
Implementation
↓
Independent review
↓
Fixes
↓
Final review
↓
Release
↓
Cold storage
====

Not every change follows every step.

The larger or more architectural a change becomes, the more of this flow tends
to appear.

== Current Review Pattern

A common pattern is separation of responsibilities:

[cols="1,3"]
|===
| Activity | Typical role

| Concept challenge
| Questions scope, assumptions, architecture and long-term consequences.

| Implementation
| Produces the change and its tests.

| Independent review
| Attempts to find defects, missing tests, contract violations or regressions.
|===

The important property is role separation, not tool choice.

The same tool may perform different roles in different contexts.

== Current Documentation Pattern

A recurring structure has emerged:

[cols="1,3"]
|===
| Location | Typical purpose

| Warm Storage
| Exploration, open questions, reviews, draft concepts and future directions.

| Backlog
| Candidate work items and accepted planning artifacts.

| Active Documentation
| Current project understanding and accepted contracts.

| Cold Storage
| Historical decisions, completed FTRs and retired material.
|===

Movement between these areas is intentionally conservative.

Material is usually reviewed before becoming part of the active model.

== Current Release Pattern

Typical release preparation includes:

* implementation completion,
* targeted testing,
* independent review,
* review-driven fixes,
* release notes,
* release tag,
* backlog cleanup.

The exact sequence may evolve over time.

== Open Questions

The project has not yet standardized:

* when a second independent review is required;
* when completed work should move to cold storage;
* naming conventions for all review artifacts;
* when process descriptions should become governance;
* how AI-specific workflows should be documented.

These questions remain intentionally open.

== Non-Goals

This document is not:

* a governance model;
* a contributor contract;
* a maintainer policy;
* a release process specification;
* a mandatory workflow definition.

If the project later requires such documents, they should be created
explicitly rather than inferred from this one.
