=== 1.4 New architectural risks since REVIEW-001

. *Module-size debt grew with feature velocity.* The P4A wave delivered real
capability (xref crosstables, joins, pivots, workbook views, resource
overrides, compact multi-axis, ODS) as large single files. This is the one
axis where the absolute state is worse than at REVIEW-001. It is *accidental*
complexity (feature-first, split-later), now repayable -- see Section 4.
. *Guard-inventory drift.* `architecture_guard_inventory.adoc` (dated
2026-04-24) no longer enumerates the live suite: it omits the
`tests/architecture/domain_transformations/` family
(`test_no_cross_family_internal_imports.py`,
`test_package_public_api_boundaries.py`,
`test_pipeline_registry_uses_public_transform_apis.py`) and the
`carrier_boundary/` guard. The anti-sprawl mechanism is sound, but the
inventory must track the suite or it loses its value.
. *`odfpy` promoted from optional extra to core runtime dependency* (ODS
adapter). A deliberate, accepted product/distribution decision, but it sits in
mild tension with the REVIEW-001 "lazy optional adapters" direction and should
be reflected wherever that ADR-style guidance lives.
. *Layer-map doc lag.* `ARCH-LAYERS.adoc` (2026-03-31) still describes
`pipeline/pipeline.py`, `core/fk.py` as the FK home,
`domain/transformations/helpers.py` for reorder, and a 5-pass order. ADR-006
and `generic_ir.adoc` are current and accurate; ARCH-LAYERS is the stale
steering doc. REVIEW-001 flagged "docs misdescribe the Ist-state" as a top
agent-facing risk; for the layer-map document specifically that risk is *not
fully resolved*.
. *Recurring tiny-helper duplication* (`_non_empty_string` / `_string_list` /
`_is_empty_cell`) across `frame_extract` / `tabular_views` / `join_views` /
`workbook_views` -- flagged independently in three FTR reviews. Minor,
fragmentation-adjacent, non-blocking; a watch item, not a defect.

== 3. Maintainability Trajectory

The project moved *toward* better modularity (for new code), markedly safer
refactoring, executable boundaries, and clearer ownership; accidental coupling
is down (single dispatch owner, single config dialect, formula intent
relocated, derived channel registered).

It did *not* move toward smaller modules -- the opposite, in absolute terms.
The erosion risk is therefore localized and specific: the large transformation
modules are where future feature work will concentrate, and they are exactly
the files an AI agent (or human) cannot hold in one read-window. This is the
single most important maintainability watch-item, and it is *accidental*
complexity, not inherent.

Net trajectory: *positive*, because the boundary and discipline gains are
structural and durable while the module-size regression is a known,
now-repayable debt rather than a design flaw.

== 4. Refactoring Discipline (the headline maturity gain)

The `fk_helpers` and `discriminator_split` package splits are the strongest
single piece of evidence in this review:

* *Behavior-preserving*: AST-equivalence checking of moved bodies;
roundtrip/symmetry witnesses (`merge(split(df)) == df`).
* *Surface-preserving*: package `__init__` re-exports the public/registry/meta
symbols; colon-target and dotted meta-registry paths resolve unchanged with
*zero edits outside the package*.
* *Guard-protected*: per-family re-export guards
(`test_fk_helpers_package_reexports.py`,
`test_discriminator_split_package_reexports.py`), a cross-family
private-import guard, a package-public-API guard, and a "registry uses public
transform APIs" guard.
* *Analysis-driven*: the 2026-05-18
`discriminator_split_package_split_analysis.adoc` reasons explicitly about
*whether* to split, derives boundaries from the code's real coupling DAG
(`split/merge/naming/values/framecheck/metadata`, acyclic), caps the shared
core at one tiny `values.py`, and *forbids* an anticipatory
`conflict.py`/`precedence.py` because that design space is owned by a future
FTR. The live package matches the analysis exactly.

Were the boundaries chosen well? Yes -- they map to real responsibility
clusters and an acyclic dependency graph, not to cosmetic size targets.

Over-engineering / fragmentation signals: *minimal, and actively policed.* The
GDR template, the single guard inventory, the explicit refusals ("no
import-linter", "no second governance document", "not a universal meta
engine", the rejected over-aggressive CLI rewrite, the localized-resource
matrix solved through existing primitives instead of a framework) show a
project that resists generalization. The one genuine fragmentation incident
(meta-carrier guard built as the wrong family) was caught and split correctly.
The recurring tiny-helper duplication is the only mild, persistent
fragmentation-adjacent signal -- and it is the *opposite* of over-abstraction
(under-consolidation of trivia).

Conclusion: refactoring is now a *repeatable, documented, reviewed, and
guard-backed process*. This is the most important architectural advance since
REVIEW-001 and the reason the module-size regression is a manageable debt
rather than an accelerating one.

*Inherent vs accidental complexity:*

* *Inherent*: spreadsheet roundtrip fidelity (canonical/rendering/carrier meta
split, NaN/nested serialization, multi-backend parity); bijective transforms
(split/merge, xref, compact_multiaxis) and their symmetry invariants; meta
conflict/precedence. The layered model + registry is a *proportionate*
response here, not over-design.
* *Accidental*: module size from feature-first delivery; tiny-helper
duplication; doc lag (ARCH-LAYERS, guard inventory); finding-model duality;
compat residue; empty top-level `__init__`. All cheaply addressable; none
require architecture change.

== 6. Architectural Confidence

The project is *close to a stable beta architecture* on every load-bearing
axis: layering, the adapter/contract boundary, IR and metadata contracts,
executable guard enforcement, and -- decisively -- a safe, repeatable
refactoring process. Changes route through FTR + review + GDR; the seams that
matter are explicit and tested; a second backend has validated the central
abstraction.

It is *not* a rapidly shifting prototype: the structural vocabulary
(WorkbookIR, RenderPlan, meta_canonical/meta_rendering, spreadsheet_contract,
the pass chain, the backend router) has been stable across the wave and is
guard-pinned.

It is *not yet fully stable*: a known latent input-mutation bug (IR-006)
persists untouched; the largest modules are larger than at REVIEW-001; and the
primary steering doc for layers (ARCH-LAYERS) lags the code. These are
localized prototype-grade residues on an otherwise beta-grade skeleton -- they
do not undermine the architecture, but they should be visible and owned rather
than discovered later.

Confidence: *stable-beta architecture with localized, named, non-architectural
residue.*
