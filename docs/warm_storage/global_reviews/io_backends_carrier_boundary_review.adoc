== 1. Executive Summary

`io_backends` is a *Mild hotspot* -- not Healthy-and-bounded, not Structural,
and emphatically not an Immediate blocker. The carrier-boundary *architecture*
is in good shape: the seams are fenced by AST import-graph guards, the
rendering layer is provably carrier-neutral, domain coupling is a single
deliberate exception (`legend_blocks`), and the safety-critical read paths are
well characterized. What earns the "mild hotspot" label is *concentration and
duplication inside the ODS backend*, plus a handful of carrier-neutral helpers
that are copied across the two backends instead of shared.

The dominant fact: `ods/odf_parser.py` is 1147 LOC and *~48% of it (≈555 LOC)
is presentation-metadata extraction* organized as four families, three of which
(`horizontal_alignments`, `vertical_alignments`, `text_orientations`) share
~250 LOC of *near-duplicated cell-walk* code that differs only in which style
map it consults and which value key it writes. This is the single clearest
piece of accidental growth. Importantly, *most* of the ODS/XLSX size gap is
*not* accidental: ODF stores presentation through named styles the parser must
resolve, while `openpyxl` resolves them into per-cell objects -- so ODS is
intrinsically heavier, and that part should be left alone.

The right operating mode here matches the baseline reviews: *small,
capability-driven slices*, and -- a refinement this review adds -- the *safest
first slices are zero-runtime hygiene that is even lower-risk than ODS
modularization*: an architecture guard locking `legend_blocks` as the sole
permitted domain crossing, and consolidating helpers that are currently copied
verbatim across backends (a 5-copy blob decoder, and three byte-identical
legend/meta helpers). These reduce the blast radius of the eventual ODS
modularization without touching the fragile core.

*Bottom line:* the carrier boundary is architecturally ready to receive its
next wave of preparation. Do the cheap cross-cutting hygiene first, then a
*within-ODS* presentation-extractor consolidation (its characterization already
exists), and only after that consider a presentation submodule. Do *not* start
grouped headers or nested lists, and do *not* touch the repeated-cell safety
core (`_parse_table_grid`) or the deliberately divergent ODS/XLSX renderer
architectures.

== 4. Domain Coupling Assessment

*Method:* grep for the 13 transformation-family tokens listed in the prompt
across `io_backends/` and `rendering/`, plus a check for `domain/` *imports*.

[source,text]
----
# domain-family tokens in io_backends  →  ONLY legend_blocks
ods/odf_parser.py:871:        legend_blocks
xlsx/openpyxl_parser.py:351:   legend_blocks

# domain/ imports inside io_backends OR rendering  →  <none>
----

*Fact -- the boundary is data-shaped, not code-shaped.* Neither `io_backends`
nor `rendering` imports any `spreadsheet_handling.domain` module. All
interaction with domain output is through the opaque `_meta` dict and documented
string keys. The "coupling" that exists is *knowledge of the `_meta` schema*,
which is exactly what `registries/meta_registry.yaml` is for.

[cols="2,2,3", options="header"]
|===
| Reference | Layer | Verdict

| `legend_blocks`
| `io_backends` (both parsers)
| *Legitimate carrier-boundary concern.* A rendered legend is a *visible* table; the parser must classify it (`table.kind = "legend"`) so it is not treated as payload. Reads `_meta.legend_blocks[*].resolved.{sheet,top,left,frame_name}`. This is the accepted exception.

| `legend_blocks`
| `rendering` (`composer`, `passes/_base`, `workbook_projection`)
| *Legitimate render intent.* Composer places legend tables and freezes `resolved` coordinates; `_base` reads legend `entries` to build `from_legend` validation dropdowns; `workbook_projection` passes the key through `canonicalize_workbook_meta`.

| `workbook_view`
| `rendering` (`frame_selection`, `composer`, `passes/_base`)
| *Legitimate render intent.* Drives sheet selection, frame→sheet mapping, and sheet ordering. This *is* the projection spec.

| `frame_lifecycle`
| `rendering/frame_selection`
| *Legitimate render intent.* Read to decide `render`/`role` for each frame. (The baseline rule "pipeline execution must not branch on `frame_lifecycle`" is about the *pipeline* layer; rendering is the intended consumer of lifecycle.)

| `enrich_lookup`, `derived.*.helper_columns`
| `rendering/passes/_base`
| *Legitimate render intent.* Read to know which columns are helpers, for helper styling. Reads the `derived` operational view; never imports `enrich_lookup` code.

| (everything else: `fk_helpers`, `xref_crosstable`, `cell_codecs`, `compact_multiaxis`, `sparse_defaults`, `split_by_discriminator`, `join_views`, `tabular_views`)
| --
| *Absent.* These ride `_meta` as opaque blobs through the carrier and rendering layers. No smell.
|===

*Architectural smell?* None active. The one residual risk is that the accepted
`legend_blocks` crossing is *not protected by a guard* -- nothing prevents a
future change from reading, say, `xref_crosstable` directly in a parser. The
existing AST guards fence *imports* (no `domain` import), but a string-key read
of a new domain family would pass.

*Recommendation (guard):* add a small architecture guard asserting that
`legend_blocks` is the *only* domain-family token permitted in
`io_backends/**`. This locks the documented exception and converts the baseline
prose finding into an enforced invariant. Low risk, test-only. (§13 slice 2.)

== 5. `_meta` / Carrier-Boundary Assessment

The `_meta` crossing is *well separated and registry-classified*. The registry
(`meta_registry.yaml`, 919 lines) uses two layers (`meta_canonical`,
`meta_rendering`) and four classifications, and
`tests/architecture/semantic_invariants/test_meta_carrier_boundary_guards.py`
enforces that the carrier/rendering keys are never `canonical_meta` and that
canonical entries never use carrier-shaped (`_`-prefixed) names.

[cols="2,2,3", options="header"]
|===
| `_meta` area | Where it lives | Classification / handling

| Sidecar / persistence egress
| `pipeline/persistence_boundary.py` (pure function, carrier-neutral)
| Preserves *Intent*, drops *Resolution* (`legend_blocks[*].resolved`, `xref_crosstable[*].column_keys` + `dense_axes.resolved`); prunes top-level `derived` and `__*`. Same rules for json/yaml/xml/csv and spreadsheet.

| Workbook-carrier embedded payload
| `workbook_meta_blob` in hidden `_meta` sheet (JSON)
| `carrier_artifact` / `meta_rendering`. Written by composer/passes; read by both parsers + `workbook_projection`. The current authoritative read-back carrier.

| Hidden metadata carriers
| XLSX: `ws.sheet_state == "hidden"`. ODS: table style `display:false` (`_parse_hidden_style_names`/`_table_is_hidden`) or name `_meta`.
| Different carrier mechanisms, identical logical role.

| Presentation families (widths, orientations, h/v alignments)
| Extracted per backend; policy via `presentation_meta`
| `canonical_meta` / `meta_canonical`, but *carrier-authoritative* (replace-or-clear). Persisted unchanged at the boundary (explicit non-goal to prune the `source: workbook` envelope).

| Read-path reconstruction hints
| `__autofilter_ref` (`read_path_hint`), `__freeze` (`derived_operational_view`)
| Carrier-side; stripped from canonical `_meta` on readback (verified by `test_spreadsheet_parse_contract`).
|===

*Is `io_backends` reading domain intent or preserving/classifying carrier
artifacts?* Overwhelmingly the latter. The single exception where it interprets
*intent* is `legend_blocks` (it must, to classify a visible table). Everything
else it either preserves opaquely (the blob) or extracts from the carrier
(presentation families) under a carrier-authoritative policy.

*Are presentation meta, workbook meta, hidden meta, and support-surface
classification sufficiently separated?* Yes, conceptually and by guard. The one
*implementation* blemish is duplication, not conflation: the blob decoder
appears 5 times (Finding F4) and the legend/store-meta helpers are copied
(Finding F3). Separation of *concerns* is clean; *deduplication* of helpers is
the gap.

* *Presentation extraction is ~48% (≈555 LOC, lines 106-661) of the file.* This
  is a real concentration.
* *The three cell-addressed extractors* (`_extract_ods_horizontal_alignments`,
  `_extract_ods_vertical_alignments`, `_extract_ods_text_orientations`) are
  *~250 LOC of near-duplicated cell-walk*: identical row/col iteration,
  identical repeated-structure clipping (`effective_row_repeat`/
  `effective_col_repeat`), identical `ParserLimits` enforcement, identical
  per-column `default-cell-style-name` fallback. They differ only in (a) which
  map they consult, (b) the value key (`horizontal`/`vertical`/`rotation`), and
  (c) text-orientation's extra `_ods_rotation_to_xlsx` + `>0` filter. *This is
  the one piece of genuinely accidental growth* and the prime consolidation
  target.

  presentation through *named styles*: the parser must build `{style_name →
  value}` maps from `<style:style>` elements (`_build_*_map(doc)`), resolve a
  cell's `table:style-name`, *and* fall back to the column's
  `table:default-cell-style-name`. `openpyxl` resolves all of this into per-cell
  `cell.alignment` objects, so the XLSX extractors read a value directly. The
  `_build_*_map` + default-style-fallback machinery has no XLSX analogue *by
  carrier necessity*. Leave it.

It is the fix for `BUG-ODS-REIMPORT-REPEATED-CELL-MEMORY-EXPLOSION`, is densely
commented about why it must *not* materialize repeated fillers, and is pinned by
a thorough characterization suite (§10). Touching it for cosmetic reasons is
pure downside.

== 7. XLSX vs ODS Comparison

*Is XLSX structurally cleaner, or merely smaller?* Both -- and the "smaller" is
mostly intrinsic. `openpyxl` provides a *live, resolved object model*:
presentation values arrive per-cell, so the parser needs no style-map
resolution; and the renderer mutates cells directly via `_execute_render_op`,
so it needs no buffered intermediate. ODF requires the opposite on both sides
(resolve named styles on read; buffer-then-emit a whole XML document on write,
with a style cache).

[cols="3,2,2", options="header"]
|===
| Responsibility | XLSX | ODS

| Hidden-sheet meta
| `sheet_state="hidden"`
| table style `display:false` / name `_meta`

| Presentation extraction (4 families)
| Direct from `cell.alignment` / `column_dimensions`
| Resolve named styles + column default-style fallback (heavier *by carrier*)

| Legend classification
| `_legend_table_hints` / `_apply_legend_table_hints`
| *byte-identical* copies

| Workbook-meta blob
| `_store_workbook_meta`
| *byte-identical* copy

| Validations (list literal)
| `"a,b,c"` comma form (`_parse_xlsx_list_literal_formula`)
| `cell-content-is-in-list("a";"b")` semicolon form (`_parse_validation_formula`)

| Renderer architecture
| Live model, `_execute_render_op` dispatch (405 LOC)
| Buffered `_collect_sheets` + `_build_table` + style cache (675 LOC)

| Freeze panes
| `ws.freeze_panes`; parsed back (`__freeze`)
| `settings.xml` config items; *not parsed back* (documented gap)

| Lookup formula
| `=XLOOKUP(...)` A1 refs
| `of:=XLOOKUP(...)` `[.A1]` refs
|===

*Legitimate (do not force symmetry):* the renderer architectures (live vs
buffered), the style-resolution asymmetry, the formula/validation string
formats, and the freeze-readback gap are all *correct* carrier differences.

*Accidental (normalizable):* the byte-identical `_legend_table_hints`,
`_apply_legend_table_hints`, `_store_workbook_meta`; the 5-copy blob decoder;
the *within-each-parser* 3× cell-walk; and the near-duplicated
`parser_interpretation` modules (Finding F2). Of these, the cross-backend
*byte-identical* helpers are the safest to share; the `parser_interpretation`
unification is higher-value but higher-risk (the cell accessors genuinely
differ, so it needs an accessor abstraction).

== 8. Rendering-Layer Boundary Assessment

*Is `rendering/` truly carrier-neutral?* Yes, and it is *guard-enforced*.
`test_adapter_alignment.test_generic_rendering_package_stays_free_of_openpyxl_and_xlsx_adapter_imports`
walks every module under `rendering/` and asserts no `openpyxl` /
`io_backends.xlsx` import; a fresh grep confirms zero `openpyxl`/`odf` imports
anywhere in `rendering/`.

*Are RenderOps / WorkbookIR / RenderPlan sufficiently separated from carrier
implementation?* Yes. `plan.py` defines ~17 frozen `RenderOp` dataclasses;
notably `SetHorizontalAlignment`/`SetVerticalAlignment` *enforce the canonical
vocabulary at construction* (`__post_init__` raises `ValueError` on out-of-vocab
values), so a backend cannot smuggle a carrier-specific alignment token through
the plan. `WriteMeta`'s docstring explicitly anticipates non-hidden-sheet
carriers ("other carriers remain possible").

*Where would future carrier-visible features land first?* *Rendering before
parsers.* The write path (compose → passes → flow → plan) is the producer of all
carrier-visible structure; parsers only need to recognize it on readback. For
grouped headers specifically, the path already exists (§2), so the work is
*extending* `layout_composer._build_header_grid_and_merges` and the matching
parser recognition -- in three places, because the header machinery is copied.

*Guards protecting this boundary:* `test_adapter_contract_boundary_guards`
(backends delegate to contract; renderers consume only `plan`; parsers stop at
`ir` + local interpretation; contract stays neutral), `test_adapter_alignment`,
`test_spreadsheet_parse_contract`, and the two `*_parser_modularization` guards.
This is a *strong* fence and a real asset.

== 9. Capability-Profile Recommendation

*Are parity gaps explicit?* Yes, but *scattered* across two representations:

* *Documented-gap tests with positive assertions* (not skips):
  `test_freeze_parse_hint_difference_is_an_explicit_accepted_gap` (XLSX parses
  `__freeze` back, ODS does not) and `test_protection_is_xlsx_only_ods_documented_gap`.
* *Renderer no-op branches:* `odf_renderer` `continue  # ODS protection not
  implemented in this slice` and `# ODS cell locking not implemented in this
  slice`.

There is *no central record*. To answer the gap question today you must read
both the parity test and the ODS renderer.

*Recommendation -- smallest useful form:* a *documentation table* (a single
`*.adoc` listing family × {xlsx, ods} × {full, partial, none} with a one-line
basis and a pointer to the asserting test), optionally single-sourced as a small
Python constant that the parity test imports so the doc and the tests cannot
drift. That is enough to make the gaps reviewable *before* grouped headers /
nested lists -- which will add capability *unevenly* across backends and need a
place to declare "ODS: not yet."

*Do not* build a capability-profile *framework* / enforcement engine. There is
no current pressure, and (mirroring the persistence-boundary reasoning in the
sequencing review) two data points -- freeze and protection -- do not define a
schema worth freezing. A doc + tiny constant is the right altitude now.

== 14. Specific Questions -- Explicit Answers

. *Is `io_backends` healthy, mildly hot, structurally risky, or a blocker?*
  *Mild hotspot.* Architecture and guards are healthy; the heat is ODS internal
  concentration + cross-backend helper duplication. Bounded and well-tested.
. *Is ODS the main concern, or the whole carrier-boundary layer?* *ODS* -- and
  within it, specifically the presentation-extraction cell-walk. The broader
  carrier boundary (contract, rendering neutrality, guards) is a strength.
. *Does `legend_blocks` in parser code represent acceptable classification or
  leakage?* *Acceptable carrier classification.* A visible legend must be
  recognized so it is not treated as payload. *Recommendation:* add a guard
  locking it as the *sole* permitted domain crossing (§13 slice 2).
. *Is `presentation_meta.py` a good extraction pattern to imitate?* *Yes* -- with
  the nuance that it consolidated the *replace-or-clear policy*, not the
  *extraction walk*. Imitate it for the next move: a carrier-neutral policy/
  helper plus carrier-specific thin callers (slice 6 follows the same shape).
. *Should `spreadsheet_contract.py` become more central?* It already *is* the
  central seam, and it is correctly *thin* (two functions). Keep new carriers
  routing through it; do *not* grow it into a god-facade. "More central" only in
  the sense of remaining the single entry -- which it is.
. *Is a backend capability profile useful now or premature?* A *small doc/data*
  profile is useful *now* (before grouped headers/nested lists). A *framework*
  is premature.
. *Which ODS split first?* The safest first behavioral move is the *within-ODS
  cell-walk extraction* (slice 6), because its characterization already exists.
  But the *overall* safest first steps are the *cross-cutting hygiene* (slices
  2, 4, 5) which carry even lower risk and shrink the blast radius of slice 6/7.
. *Which tests before any ODS modularization?* Mostly already present
  (`test_ods_parser_repeated_cells.py` covers the extractors). *Add* before
  touching code: an ODS *renderer* characterization for `_collect_sheets` /
  `_build_table` (before slice 7 or any renderer work) and a *blob-decode* unit
  test (before slice 4).
. *What should absolutely not be touched yet?* `_parse_table_grid` (repeated-cell
  safety core); the divergent ODS *buffered* renderer model; the ODS↔XLSX
  vocabulary-conversion split (no codec concept is justified yet); and the three
  dispatch mechanisms (no pressure).
. *How does this affect the `next_program_sequencing_review` roadmap?* It
  *confirms* Phase 5 (adapter preparation, bounded + guard-backed) and supplies
  its concrete slice order. It *adds* that Phase 5 can *open* with zero-runtime
  cross-cutting hygiene (guard + blob-decode + legend hoist) that is safer than,
  and should precede, ODS modularization. It *reinforces* the "do not start
  grouped headers / nested lists" gate, while *refining* that grouped headers
  extend an existing (duplicated) multi-row-header path rather than building one
  from scratch.
