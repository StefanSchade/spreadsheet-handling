== 1. Executive Summary

The active backlog is *mildly-to-structurally stale*: the planning *structure*
(`_ftrs_open.adoc` phased include surface) is sound and was deliberately built,
but it currently *mixes completed and open work*, and the directory holds a
second, *orphaned* population of files that no include surface references at all.

Two distinct staleness classes exist, and they matter for different reasons:

. *Completed work still presented as open.* `_ftrs_open.adoc` includes
  `FTR-LEGEND-BLOCKS-LIFECYCLE-P6` (*Status: Done*) and two *Fixed* P4A bugs
  (`BUG-GENERATED-META-CONSTRAINT-ACCUMULATION-P4A`,
  `BUG-REIMPORT-PROMOTION-HELPER-COLUMN-LEAKAGE-P4A`). A future agent reading the
  active surface would treat closed work as open. *(Fact: status lines + include
  lines both present.)*
. *Orphaned files in the active directory.* 27 `.adoc` files in `docs/backlog/`
  are *not* referenced by `_ftrs_open.adoc`. Most are completed-workstream
  artifacts (XRef inventory/impl-concept, Legend-Blocks companion docs) or
  historical phase/roadmap reviews that belong in cold storage. A few are live
  companions of *open* tickets (keep them). Two are *whole workstreams in an
  undefined state* -- `FTR-SCHEMA-EVOLUTION-OPERATIONS` (Slices 1-4 implemented in
  `domain/schema_maintenance/`, but the FTR still says *New* and sits in no
  include surface) and `FTR-PAYLOAD-SEMANTICS-RECOVERY-P5` (exploratory, orphaned).

*The dominant risk is the second class for AI agents specifically.* The
`FTR-XREF-CROSSTABLE_inventory.adoc` "Gaps in Test Coverage" section, and the
schema-evolution FTR's *New* status, both describe work that the code and the
three baseline reviews show is already done. An agent that trusts these as
ground truth would re-implement or re-test closed surfaces. This is precisely the
"stale inventory/review artifacts could mislead future agents" hazard the prompt
flags, and it is *present and concrete*, not hypothetical.

*This is a backlog-governance problem, not an architecture problem.* All three
baseline reviews independently conclude the codebase is healthy and ready for the
next wave; they each list backlog cleanup as the cheap "do-first" hygiene with
*zero runtime risk*. A prior `semantic_backlog_reconciliation_review.adoc`
(2026-05-27) already flagged the same drift class and was *partially* acted on
(e.g. `FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5` has since been archived). This
triage extends that work to the post-consolidation state.

*Recommendation:* one small archival commit (completed workstreams), one small
archival commit (historical reviews), then a *decision-gated* pass on the two
orphaned workstreams, then reword-in-place of ~10 still-valid-but-broad tickets.
Defer all review-derived *new* tickets until the archival noise is gone. Prefer
archival over deletion throughout; preserve the `include::` planning surface.

== 2. Backlog Health Classification

*Classification: structurally stale (recoverable), trending misleading-for-agents
in two named spots.* Not "clean enough"; worse than "mildly stale" because the
staleness reaches the *include surface itself* (Done/Fixed items are included),
not only loose files.

Evidence and reasoning (Fact / Judgment):

* *Fact.* `docs/backlog/` holds 87 ticket/review `.adoc` files; `_ftrs_open.adoc`
  references 60 via `include::`. 27 are orphaned (on disk, not included).
* *Fact.* Three included items are already closed: `FTR-LEGEND-BLOCKS-LIFECYCLE-P6`
  (*Done*), `BUG-GENERATED-META-CONSTRAINT-ACCUMULATION-P4A` (*Fixed*),
  `BUG-REIMPORT-PROMOTION-HELPER-COLUMN-LEAKAGE-P4A` (*Fixed*).
* *Fact.* `domain/schema_maintenance/` (model/columns/operations/meta_refs/
  meta_update) and `tests/architecture/test_schema_maintenance_private_step.py`
  exist, and `FTR-SCHEMA-EVOLUTION-OPERATIONS_review.adoc` reviews "Slices 1-4" --
  yet `FTR-SCHEMA-EVOLUTION-OPERATIONS.adoc` is *New* and in no include surface.
* *Fact (cross-checked against baselines).* `architecture_health_review` §8 and
  `next_program_sequencing_review` §9/Phase-0 both name the XRef inventory's stale
  "gaps" (e.g. "No test for out-of-scope relation row preservation", which Slice D
  closed with `test_out_of_scope_base_row_preserved`).
* *Judgment.* The phased structure (Phases 0-9 by capability/maturity) is good and
  should be *kept*. The problem is lifecycle hygiene (closed items not evicted)
  and orphan accumulation, not the taxonomy.

*Why this specifically misleads AI agents:* an agent recalls the active backlog
as authoritative open work. A *Done* FTR on that surface, or a stale "gaps"
section in an inventory the agent treats as ground truth, produces confident
re-work of closed scope. Human readers tolerate this (they read git log and
release notes); cold-start agents do not. The fix is cheap and high-leverage.
