== 1. Executive Summary

*The dominant mode for the next program is "process the next family through a
lightweight pipeline and learn" -- not "stop and fix a chokepoint first."* The
domain transformation layer is good enough (the prior review's strongest layer),
and the FK/Legend/XRef waves proved that meta/lifecycle work can be done at the
registry + persistence-boundary level *without* refactoring domain modules.

There is exactly one cheap "do first" block, and it is hygiene, not engineering:
archive the completed workstream artifacts and classify the transform-config
registry entries. Both are days of docs/registry work with zero runtime risk,
and both remove false signal and unblock everything downstream.

Three facts from the code decide the capability order:

. *`compact_multiaxis` imports `cell_codec`, `xref_crosstable`, and the shared
  `_legend_blocks` reader* -- it is a composition capstone. `cell_codec` *must*
  be modeled before `compact_multiaxis`. (Fact, not preference.)
. *`cell_codec` depends only on `_legend_blocks`*, which is already modeled
  (`legend_blocks`) -- so `cell_codec` is unblocked today.
. *`sparse_defaults` has no cross-family imports* -- it is independent and can be
  modeled at any time, or left green-but-unmodeled.

The Functional Model is already use-case-organized (`ch04` is a set of roundtrip
narratives plus per-family step-slice chapters). `compact_multiaxis` and
`cell_codec` already have a home: `business_calculation_workbook_roundtrip`
references them. So promotion should be *use-case-anchored*, not mechanical
per-family chapters -- which is also how the project's own slicing discipline
works.

On the `persistence_boundary.py` question: *do neither extreme.* Do not keep
hand-coding projectors forever, and do not build the attribute-driven engine
speculatively now. The correct intermediate step is to put Intent/Resolution
*facets in the registry as data* (Phase 1), then let the next capability
(`compact_multiaxis`, Phase 3) surface the *third* Resolution-bearing family --
and that third projector is the evidence threshold that justifies building the
generic engine. There are two hand-coded projectors today (legend, xref); build
the engine when the third is needed, not before.

*Special attention -- `persistence_boundary.py` (the question posed).* Keep the
two existing hand-coded projectors (`_project_legend_blocks`,
`_project_xref_crosstable`); they are correct and cheap. The right intermediate
step is **registry facet classification as data (Phase 1)** -- this is the
prerequisite for *any* generic engine and is useful even if the engine is never
built. The generic, facet-driven projector should be built only when the *third*
Resolution-bearing family arrives, because: (a) two data points do not define a
pattern, (b) `compact_multiaxis` is the most likely third and it is two phases
out, and (c) building the engine now would freeze a facet schema before the
third family can stress-test it. *Evidence to escalate:* a 3rd named projector
becomes necessary in `persistence_boundary.py`. Until then, one more per-family
projector is cheaper than a speculative engine.

== 4. The Meta / Domain-Refactor Hypothesis

*Earlier hypothesis:* "Improving `_meta` and improving the domain transformation
layer must proceed together."

*Verdict: this no longer holds in general. The waves weakened it.*

*Evidence.* The Legend and XRef persistence-boundary projectors were added in
`pipeline/persistence_boundary.py` *without refactoring* the `legend_blocks` or
`xref_crosstable` domain modules -- the domain modules simply write `_meta`, and
the boundary projects it. FK v1 retirement classified `helper_policies` via the
registry `contract_lifecycle` block, again without a structural domain rewrite
(the one runtime change was a containment fix in `enrich_helpers`, not a
re-architecture). So meta/lifecycle classification is now demonstrably
*separable* from domain refactoring.

*Can the next capability go through FM + meta classification without further
domain refactoring? Yes* -- for `cell_codec` and `sparse_defaults` especially.
Both are already adequately modular (`cell_codec/` is a package; `sparse_defaults`
is a standalone 339-line module with no cross-family imports). Their meta is
simple enough to classify by reading, not refactoring.

*The five change classes, separated:*

[cols="2,3", options="header"]
|===
| Class | Current need

| Global domain refactor
| *Not needed.* No evidence of a cross-cutting structural problem. Avoid.

| Local package cleanup
| *Only `compact_multiaxis`* may warrant it -- 438 LOC composing three families;
  decide after its inventory (Phase 3), not before.

| Meta / lifecycle classification
| *The main near-term work.* Proceeds independently of domain code (Phase 1).

| Runtime behavior change
| *Capability-driven only* (workbook_view precedence; validations extensions;
  compact_multiaxis only if its inventory reveals a real gap).

| Adapter / carrier-layer change
| *Deferred* to the adapter-prep phase; needed only for grouped headers / nested
  lists.
|===

*Which domain packages still need modularization before meta work is meaningful?*
None block meta work. `compact_multiaxis` is the only candidate, and only its
*inventory* (reading) is a prerequisite to classifying its meta -- not a refactor.

*Which meta cleanup can proceed without touching domain code?* All of Phase 1
(registry facet classification), the `frame_lifecycle` persistence audit, and the
XREF inventory de-staling.

*Which domain refactors should be avoided unless a capability exposes pressure?*
Any split of `compact_multiaxis`, `join_views` (479 LOC), `derived_column_policy`
(461 LOC), or `resource_overrides` (436 LOC). They are large but green and
guard-backed; size alone is not pressure.

*Is the domain layer good enough to support the next wave without broad
refactoring? Yes.*

== 5. Recommended Default Process for Capability Work

The project has converged on a chain:
`warm concept -> FM promotion -> inventory -> impl concept -> characterization ->
runtime -> docs -> review/archive`. *This should be documented as the project
process -- but as a tiered decision tree, not a mandatory waterfall.* Forcing
every family through the full chain would cause exactly the FM inflation and
inventory overhead the prior review warned about.

*Tiered method (recommended default):*

[cols="1,3,2", options="header"]
|===
| Tier | When | Chain steps applied

| T1 -- Classify only
| Implemented, green, meta has intent/resolution facets to label (split, sparse,
  cell_codecs, compact configs)
| meta/lifecycle classification + (optional) lightweight FM mention. No inventory,
  no runtime.

| T2 -- Lightweight FM promotion
| Implemented, green, unmodeled, no behavior change wanted (cell_codec, sparse)
| FM promotion under an *existing* use case + T1 classification + docs. No
  inventory, no runtime.

| T3 -- Inventory-first
| Implemented but composes other families or has a deferred contract
  (compact_multiaxis)
| fresh inventory -> impl concept -> (only if a real gap) characterization +
  runtime. This is the XRef pattern.

| T4 -- Full chain
| Genuinely new/uncertain semantics, real runtime gap (what XRef *was*)
| warm concept -> ... -> review/archive (the complete chain).
|===

*Answers to the sub-questions:*

* *Every implemented-but-unmodeled family through the full chain?* No -- most are
  T1/T2.
* *Lightweight FM promotion only?* `cell_codec`, `sparse_defaults`,
  `enrich_lookup`, `join_views`, `tabular_views`.
* *Fresh inventory before any change?* Only `compact_multiaxis` (T3).
* *Meta/lifecycle classification only, skip runtime?* The transform-config
  families whose only gap is `canonical_meta` over-use (T1).
* *Warm-storage concept useful vs overkill?* Useful when exploring new/uncertain
  semantics (XRef was). Overkill for an already-green, already-understood family
  (cell_codec) -- skip straight to T2.
* *When to archive a workstream?* When all its slices are Done and its
  inventory/impl-concept docs are no longer planning surfaces -- XREF and
  discriminator_split now qualify.
* *Document the method?* Yes, as the tiered tree above, ideally in the developer
  guide workflow chapter. Emphasize tier selection over ceremony.

== 6. Functional Model -- What It Has Achieved

*Judgment: the FM is working as a problem-space filter, and the evidence is in
the XRef chapter.* `ch04/xref_crosstable.adoc` cleanly separates the baseline
obligation (scoped editable projection over a tuple relation) from a long,
explicit *Deferred and Neighbouring Concepts* list (axis mapping, grouped
headers, compact multi-value cells, cell-codec parsing, projection structure
integrity) and a frank *Open Questions* section. That structure is what kept
Slice D (scoped recomposition) narrow and prevented it from absorbing axis
mappings or grouped headers. So:

* *Useful problem-space filter?* Yes.
* *Preventing over-implementation?* Yes -- demonstrably, via XRef's explicit
  deferral lists and the impl-concept's "Explicit Non-Goals."
* *Separating baseline from deferred?* Yes -- this is its strongest function.
* *Right entry point for future capability work?* Yes, *when a capability is new
  or uncertain* (T3/T4). For green families it is a documentation endpoint, not
  an entry point.

*Risk of FM inflation: real.* `ch04/_chapter.adoc` self-labels as "Exploratory
use-case concepts ... pending editorial cleanup under `FTR-CONCEPT-DOCS-CLEANUP-P5`
... no semantic review performed during the move." Promoting all ~7 unmodeled
families as standalone chapters now would inflate an already-unreviewed chapter.

*Promote one-by-one or only when a change touches them?* Neither extreme.
Promote *use-case-by-use-case*: `business_calculation_workbook_roundtrip` already
references `compact_multiaxis` + `legend` + `lookup_helper`, so promoting
`cell_codec`/`compact_multiaxis` belongs under that use case, not as two isolated
family chapters. This matches the project's use-case-driven slicing discipline.

*Keeping chapters concise (avoid becoming impl concepts):* enforce the existing
step-slice convention (`_chapter.adoc`): one sentence per step, Information
Consumed/Produced with lifecycle attributes, and a *Semantic Model Reference*
that *names* registry entries rather than embedding field paths or
producer/consumer lists. The XRef chapter's "Metadata Pressure" section is right
at the edge of this line -- it is the model to imitate but not exceed. Run
`FTR-CONCEPT-DOCS-CLEANUP-P5` *before* adding chapters, so new promotions land in
a cleaned chapter.

== 7. Dependency Map & Sequencing Rationale

[source,text]
----
                _legend_blocks reader (shared)
                      │
   legend_blocks ─────┤ (MODELED)
        │             ▼
        │         cell_codec ───────────────┐
        │             ▲                      ▼
   xref_crosstable ───┴──────────────► compact_multiaxis  (capstone composition)
   (MODELED)                                 ▲
                                             │ deferred "dense-axis reconstruction" contract
   sparse_defaults  (standalone, no cross-family imports)

   frame_lifecycle ──► workbook_view precedence        (small enables larger)
   XRef axis model ──► grouped headers ◄── render capability ◄── ODS modularization
   (core flatten + adapters) ──► nested lists
----

*Answers to the posed sequencing tests (reasoned from code/docs, not assumed):*

* *cell_codecs before compact_multiaxis?* *Yes -- forced.* `compact_multiaxis.py`
  imports `decode_cell_values`/`encode_cell_values` from `cell_codec` and
  `contract_xref`/`expand_xref` from `xref_crosstable`.
* *sparse_defaults before or after cell_codecs?* *Independent.* No code
  dependency (`sparse_defaults.py` has no cross-family imports). Can run in
  parallel; it is the safest first promotion if a warm-up is wanted.
* *workbook_view/sheet_mappings before nested lists?* *Yes, but not as a direct
  prerequisite* -- nested lists are blocked by adapters regardless. Complete
  workbook_view on its own merits (the open precedence FTR), not to unblock
  nested lists.
* *frame_lifecycle persistence before workbook_view precedence?* *Yes, mildly.*
  The precedence question (`FTR-WORKBOOK-VIEW-PAYLOAD-CONFLICT-PRECEDENCE-P6`)
  interacts with `frame_lifecycle` classification; the persistence audit is the
  smaller slice, so do it first as the enabler.
* *ODS modularization before grouped headers?* *Yes -- strongly.* Grouped headers
  add multi-row header structure that lands in the ODS/XLSX parsers+renderers;
  modularizing the 1147-LOC ODS parser first bounds the blast radius.
* *XRef axis mappings before grouped headers?* *Yes -- dependency.* The XRef FM
  states grouped headers "requires the clarified axis model"
  (`FTR-XREF-AXIS-MAPPINGS-P4A2`). Grouped headers therefore has three
  prerequisites: axis model + render capability + adapter prep.
* *backlog cleanup before meta lifecycle work?* *No dependency*, but cleanup is
  trivial and removes stale-signal risk, so it goes first by cost, not by
  dependency.
