credible first-hour trial use; the step 1 cleanup landed three of the

== First-Hour User Journey Assessment

The target reader -- technically inclined non-developer, LLM-assisted,
comfortable with files/spreadsheets/scripts -- is most likely to enter
through the GitHub homepage of `spreadsheet-handling` and read the
README at `core/README.md`.

The README does three things well for that reader:

* a one-paragraph "what this is" framing (JSON / CSV / Excel round-trip
  with FK / index / hierarchy preservation, YAML-configured pipelines);
* an explicit `pip install spreadsheet-handling` block with the Python
  3.10+ floor;
* a "Where to go next" section that names exactly three onward links --
  the demo repository, the latest-release user guide on Pages, and the
  per-release version archive portal. The order matches the maintained
  first-hour intent (try the demo first; consult the user guide for
  reference).

It does *not* invent a parallel quickstart. That is the right call: the
demo is the maintained first-hour path and the README correctly defers
to it rather than duplicating it.

What the reader does next forks on which "next" link they click.

=== Branch A -- demo-first reader

If the reader clicks the demo link, they land on the demo's README which
walks through eight concrete steps: clone, `make setup`,
`make setup-lib-local` (or `make setup-lib-pypi`),
`make run PIPELINE=./pipelines/demo_product_business_slice.yaml`, open
the generated workbook, inspect helpers and validations, edit a cell,
reimport, and `diff` the round-tripped JSON. The pipeline YAML behind
`make run` uses the canonical `- step:` dialect and the current
`infer_fk_relations` -> `add_fk_helpers` flow.

This path produces a successful first-hour round-trip without private
maintainer knowledge. It also exposes the user to FK-helper policy
naturally (the demo pipeline runs `infer_fk_relations` before
`add_fk_helpers`), which prepares them to read the 0.2.0 release notes
section on FK migration. Branch A is the maintained happy path and it
works.

=== Branch B -- user-guide-first reader

If the reader clicks the published user guide link instead -- a curious
reader, a reader who does not want to clone a second repository for a
"reference", an LLM agent that needs structured catalog text -- they
land on the rendered user guide. The introduction (ch01) is short and
honest about the two-repository split. The workflow chapter (ch02)
introduces "Pipeline YAML Structure" and gives this concrete example:

[source,yaml]
----
pipeline:
  - name: expand_xref
    params:
      matrix: role_permissions
      output: permission_rows
      row_keys: role_id
      drop_empty: true
  - name: write_structured_yaml
    params:
      source: permission_rows
      output_path: out/permissions.yaml
----

Every YAML example in the transformation catalog (ch03) follows the same
shape -- `- name: <step> / params: {...}` -- for `expand_xref`,
`contract_xref`, `decode_cell_values`, and `encode_cell_values`. A
reader who copies any of these into a file and runs

[source,bash]
----
sheets-run --steps my_pipeline.yaml --in-path ... --out-path ...
----

hits

[source,text]
----
ValueError: Step spec missing 'step': {'name': 'expand_xref',
'params': {'matrix': 'role_permissions', ...}}
----

The runtime error names the missing key (`step`) explicitly, so a
technically inclined reader can recover, but they then discover that
parameters are also expected inline (the runtime does not consume a
`params:` sub-key; the `params` key is passed through as an unknown
keyword argument to the step factory and raises `TypeError`). The
two-step recovery -- swap `name:` for `step:`, then unnest `params:` --
is doable in 15-30 minutes with LLM help, but it is the first thing the
reader is taught after the introduction and it teaches the wrong
dialect.

An LLM-assisted reader is particularly vulnerable: an LLM faithfully
asked to "write a pipeline that uses the catalog steps" will reproduce
the catalog dialect verbatim, since that is what the published reference
shows. This is the realistic onboarding failure mode the prompt asks
about.

=== Branch C -- curious-repo-browser reader

If the reader clones the `spreadsheet-handling` core repository to
inspect it (a reasonable thing for a "technically inclined" reader to
do), the root listing is now considerably cleaner than at step 1
(`examples/` is gone, `input.json` is gitignored), but two visibly
top-level YAML files remain: `core/sheets.yaml` (12 lines) and
`core/pipeline.yml` (4 lines). Both reference obsolete dotted module
paths:

* `spreadsheet_handling.domain.validations.unique:make_step` -- this
  module does not exist (`src/spreadsheet_handling/domain/validations/`
  contains `findings.py`, `fk_helpers.py`, `graph_validations.py`,
  `reference_validations.py`, `validate_columns.py`; there is no
  `unique.py`);
* `spreadsheet_handling.domain.extractions.select:make_step` -- this
  module does not exist
  (`src/spreadsheet_handling/domain/extractions/` contains only
  `frame_extract.py`).

A reader copying either file as a "working starting point" will fail.
Neither file is referenced from any current documentation file. They are
not gitignored (`git ls-files` returns both as tracked). The same step 1
finding -- F-0.2-1 -- explicitly named them, but the cleanup commit
addressed only `examples/`.

=== Edit-and-reimport path

For the reader who reached the workbook through Branch A, the
edit-and-reimport path is well-explained: change a single business value
in the workbook (status from `active` to `pilot`), save, run the
reimport pipeline, and `diff -ru` the regenerated JSON against the
original data. The demo README explicitly excludes `overrides.yaml` and
`_meta.yaml` from the diff with a clear rationale (workbook rendering /
configuration sidecars, not business payload). This is the right level
of explanation for the target audience.

The 0.2.0 release notes also explain the migration semantics that
matter for an editor: FK helper columns are derived from policy, the
`id_(<target>)` wire format is preserved on roundtrip, and editing the
helper column itself is discouraged because it is derived. The demo
README echoes this with "Leave the helper column alone; it is derived."

This part of the journey is coherent. The friction is concentrated in
the user-guide branch and in the curious-browser branch, not in the
edit-and-reimport step itself.

==== F-0.2-2-B1 -- User-guide YAML examples teach the wrong step-spec dialect

*Observation.* Every YAML example in the user guide
(`docs/user_guide/ch02_workflow/01_workflow.adoc` lines 22-33;
`docs/user_guide/ch03_transformation_catalog/02_bijective_transforms.adoc`
lines 62-69, 115-121, 203-212, 274-283) uses the form

[source,yaml]
----
- name: <step_name>
  params:
    <param>: <value>
----

The canonical runtime dialect, enforced by
`src/spreadsheet_handling/pipeline/build.py::build_steps_from_config`,
is

[source,yaml]
----
- step: <registry_key_or_dotted_path>
  <param>: <value>
----

with parameters *inline* under the step entry. There is no
`params:` indirection. The runtime explicitly raises
`ValueError: Step spec missing 'step'` when the `step` key is absent
(`build.py:27`) and the runtime's own error message for `pipeline:`
input directs the reader to "the canonical step: dialect" (`config.py:46`).

*Evidence.*

* Six occurrences of `- name: <step>` followed by `params:` across two
  user-guide files (grepped):
+
[source,text]
----
docs/user_guide/ch02_workflow/01_workflow.adoc:23:  - name: expand_xref
docs/user_guide/ch02_workflow/01_workflow.adoc:24:    params:
docs/user_guide/ch02_workflow/01_workflow.adoc:29:  - name: write_structured_yaml
docs/user_guide/ch02_workflow/01_workflow.adoc:30:    params:
docs/user_guide/ch03_transformation_catalog/02_bijective_transforms.adoc:63:- name: expand_xref
docs/user_guide/ch03_transformation_catalog/02_bijective_transforms.adoc:64:  params:
docs/user_guide/ch03_transformation_catalog/02_bijective_transforms.adoc:116:- name: contract_xref
docs/user_guide/ch03_transformation_catalog/02_bijective_transforms.adoc:117:  params:
docs/user_guide/ch03_transformation_catalog/02_bijective_transforms.adoc:204:- name: decode_cell_values
docs/user_guide/ch03_transformation_catalog/02_bijective_transforms.adoc:205:  params:
docs/user_guide/ch03_transformation_catalog/02_bijective_transforms.adoc:275:- name: encode_cell_values
docs/user_guide/ch03_transformation_catalog/02_bijective_transforms.adoc:276:  params:
----
* `src/spreadsheet_handling/pipeline/build.py:25-27` -- the loader pops
  the `step` key and raises if it is missing; there is no `name:`
  alternative for the step identifier (the `name:` key is a separate,
  optional "rename in logs" override on the same dict).
* `src/spreadsheet_handling/pipeline/config.py:46-48` -- the error
  message for a non-list `pipeline:` value explicitly says "must be a
  step list using the canonical step: dialect. Example:
  `pipeline: - step: identity`".
* `src/spreadsheet_handling/examples/pipeline_basic.yml`,
  `pipeline_pack.yml`, `pipelines.yml`, plus
  `demo/pipelines/demo_product_business_slice.yaml` -- all use the
  canonical `- step:` dialect with inline parameters.
* Reproduction: a copy of the ch02 workflow example saved as
  `/tmp/test_userguide_yaml.yaml` and run with
  `sheets-run --steps /tmp/test_userguide_yaml.yaml --in-path ... --out-path ...`
  fails with `ValueError: Step spec missing 'step': {'name':
  'expand_xref', 'params': {'matrix': 'role_permissions', ...}}` (see
  Validation Performed).

*Realistic early-adopter failure mode.* A non-developer reader, or any
LLM consumer, lands on the published user guide -- the canonical
reference for the transformation catalog -- copies the first concrete
YAML example, runs it, and is greeted by a Python `ValueError` referring
to a "step" key that does not appear anywhere in the example they just
copied. With LLM assistance, the reader can usually recover by
discovering that `step:` is the right key and parameters are inline, but
the recovery requires reading a third file
(`pipeline_basic.yml`, the demo pipeline, or the loader source) to
infer the canonical shape. An LLM asked to extend a pipeline from
catalog text will reproduce the broken dialect on every new step,
because the catalog is the structured reference it was given.

*Recommended action.* Rewrite every YAML example in
`docs/user_guide/ch02_workflow/01_workflow.adoc` and
`docs/user_guide/ch03_transformation_catalog/02_bijective_transforms.adoc`
to use the canonical `- step: <name>` dialect with parameters inline,
mirroring `src/spreadsheet_handling/examples/pipeline_basic.yml`. This
is a mechanical edit (one substitution per example) and does not
require new content. The transformation catalog tables themselves
(parameter columns, inverse kinds, losslessness conditions) are
correct and do not need to change. No runtime change is needed.

*Priority rationale.* This is the most likely first-hour failure for a
user-guide-first reader and the most likely failure for any
LLM-assisted user, because LLMs preferentially reproduce the published
reference's surface vocabulary. Mechanical fix; no architectural risk;
visible on day one; directly aimed at the prompt's named target
audience.

==== F-0.2-2-LT2 -- The user guide ch01 says "shipped with a specific library release" but does not pin which release the reader is reading

*Observation.* `docs/user_guide/ch01_introduction/01_introduction.adoc`
line 17-18 says: "This guide documents the framework surface shipped
with a specific library release." The Asciidoctor build pipeline
substitutes `{project-version}` into the header
(`docs/attributes.adoc` line 7, default `DEV-SNAPSHOT`), and the Pages
publishing pipeline overrides this with the release tag for versioned
snapshots. So a `versions/v0.2.0/core/user-guide/` snapshot will show
`0.2.0` in the header. However, a reader on `/latest/core/user-guide/`
sees whichever release the latest pointer currently advertises, and the
intro's "specific library release" prose does not point them to a
visible version-identification cue. The README-side guard for this
(`tools/readme_links_check.sh`) protects only the README's
`/latest/` -> `/versions/<tag>/` substitution; it does not protect a
user-guide reader from accidentally consulting the wrong release's
catalog during a low-touch period when the demo and core may have
drifted.

*Evidence.*

* `docs/attributes.adoc:7` -- `:project-version: DEV-SNAPSHOT`
  (overridden at build time by `scripts/build_docs_pages.sh`).
* `docs/user_guide/ch01_introduction/01_introduction.adoc:18` --
  "This guide documents the framework surface shipped with a specific
  library release."
* `tools/readme_links_check.sh` covers the README only; there is no
  matching helper for the user-guide-side version-identification cue.
* `docs/developer_guide/ch09_release_management/01_ci_cd_and_publishing.adoc`
  documents that `latest/core/` is a moving pointer and that
  `versions/<tag>/core/` is immutable -- so the Pages infrastructure is
  designed for both audiences, but the user-guide text does not direct
  the reader at the choice.

*Realistic early-adopter failure mode.* During a low-touch period, the
demo on `/latest/demo/` and the core user guide on `/latest/core/`
both advance with releases. A reader who cross-references their

=== Watch Only

Risks to observe during real use; do not act on them yet.

==== W-0.2-2-1 -- The CLI exposes overlapping shapes (`--config` with `pipelines:`/`pipeline:`/`profiles:`, `--steps` ad-hoc) without a documentation summary

*Observation.* `sheets-run --help` lists `--config`, `--pipeline`,
`--steps`, `--profile`, plus IO overrides. The interaction between
these is implemented carefully in `cli/apps/run.py` (steps file wins;
`--pipeline` selects from `pipelines:`; `--profile` may bind both IO
and a default pipeline). The user guide does not name any of these
shapes. A reader who later goes beyond the demo into custom pipelines
has to read the CLI source to understand the precedence rules.

Watch whether real-use feedback finds this surface ergonomically
confusing. If F-0.2-2-B3 lands (the user-guide CLI bridge), the natural
follow-up is to extend that bridge with a small precedence table; do
not pre-empt that decision.

==== W-0.2-2-2 -- Asymmetry between `io.input` (singular) at the runtime boundary and `io.inputs` (plural, dict-shaped) in the dataclass loader

*Observation.* `cli/apps/run.py` and the in-package examples use
`io.input: {kind: ..., path: ...}` (singular). The dataclass
`pipeline/config.py::IOConfig` carries an `inputs: Dict[str, IOEndpoint]`
field (plural, mapping). The dataclass loader appears unused by the CLI
runner (the runner builds its IO dict directly), and the dataclass
shape is what surviving step 1 stale `sheets.yaml` accidentally
reflects (`io.inputs: {primary: ...}`). The dual shape is not on the
public surface today (the documented shape is singular), but the
dataclass class still exists.

Watch whether any future doc, example, or fixture leaks the plural
shape onto the public surface. If F-0.2-2-B2 closes (the two root
configs disappear), the only remaining `io.inputs` use is the
dataclass internals.

==== W-0.2-2-3 -- The README's "Try it locally" pointer assumes a sibling on-disk checkout for `make setup-lib-local`

*Observation.* The demo's first-hour tutorial uses
`make setup-lib-local` to bind the local core checkout (sibling on
disk). The README's "Try it locally" link does not pre-empt this; the
fallback `make setup-lib-pypi` is documented inside the demo README.
This is correct, but a reader who installs only the demo without a
sibling core may briefly try `make setup-lib-local` before reading
step 2.

Watch whether the demo's "If you do not have a local core checkout"
note is sufficient; do not pre-empt the maintained demo wording.
