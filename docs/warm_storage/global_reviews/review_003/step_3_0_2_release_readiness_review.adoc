== Findings

*None.*

A finding is release-blocking only if it would make a non-beta
`0.2.0` *misleading*, *broken*, *unreproducible*, or *unsafe* for
careful early adopters. The runtime is not broken. The release-notes

early adopter who reads the release notes' explicit pre-1.0
stability hedge.

=== Must Fix Before Tagging 0.2.0

These are issues that should be closed before the `v0.2.0` tag is
created. They are not fundamental blockers in the strict sense above,
but they would either contradict an explicit project-side commitment
(F-RR-B-1) or leave a mechanical pre-tag step undone (F-RR-B-2). The
maintainer's own runbook already names the latter as a tagging

==== F-RR-B-1 -- The 0.2 cutline FTR for visible-cell typing has no recorded release-notes outcome

*Observation.* `docs/backlog/_ftrs_open.adoc` was reorganized
(`946ef08`, 2026-05-30) to introduce a new section
"Cutline 0.2: Semantic Decisions". That section currently contains
exactly one open FTR:
`FTR-VISIBLE-CELL-TYPING-STRATEGY-P5` ("Decide visible spreadsheet
cell typing strategy for 0.2.0 and beyond"), status `New`. Its
*Acceptance Criteria* explicitly require:

* "A release decision for version 0.2.0 is made and documented.";
* "User-facing documentation clearly explains the behavior of visible
  spreadsheet payload cells.";
* "Users can determine whether type preservation is guaranteed.";
* "The project records a long-term position: accepted permanently,
  planned as future work, or intentionally deferred.";
* "The topic cannot silently disappear from project planning."

The current `docs/release_notes/release_notes.adoc` 0.2.0 draft does
not mention visible-cell typing, string-only payload contracts, or
the `sequence_no: 1` -> `sequence_no: "1"` reimport example the FTR
describes. The user guide also does not name this contract.

*Evidence.*

* `docs/backlog/_ftrs_open.adoc` lines 19-30 -- the cutline section
  is *new* in this review's window and explicitly states it is
  "for decisions that shape future phase execution but are not
  themselves phase-owned delivery slices".
* `docs/backlog/FTR-VISIBLE-CELL-TYPING-STRATEGY-P5.adoc` lines
  3, 60-69, 80-110, 111-124 -- status `New`; scope explicitly
  includes "Define the contractual behavior for release 0.2.0";
  acceptance criteria explicitly include "A release decision for
  version 0.2.0 is made and documented" and "User-facing
  documentation clearly explains the behavior of visible
  spreadsheet payload cells".
* `grep -rn "visible-cell\|cell typing\|string-only\|string payload"
  docs/release_notes/ docs/user_guide/` -- no matches.
* `docs/backlog/FTR-VISIBLE-CELL-TYPING-STRATEGY-P5.adoc` line 131-136
  Discussion Note: "A likely short-term outcome is that release
  0.2.0 formally adopts a string-based visible-cell contract while
  collecting operational experience from real consumer
  repositories." This is the maintainer's own articulated
  preferred default outcome; if accepted, it is a one-paragraph
  release-notes / user-guide insertion.

*Concrete release risk if ignored.* Two failure modes:

* a user reading the release notes assumes type preservation
  (because the notes do not name a contract), encounters the
  `sequence_no` string round-trip in their first real workbook, and
  concludes the project silently downgrades types. The release
  notes' explicit pre-1.0 hedge does not name this specific
  contract, so it is the wrong kind of honest;
* the FTR itself goes stale during the low-touch period because its
  acceptance criteria depend on a 0.2.0 release that has already
  happened without the decision.

Neither failure mode is *broken-runtime* severity. Both are *0.2.0
should not silently skip a decision the project just publicly
declared a 0.2.0 cutline*.

*Recommended action.* Close the FTR by adopting Option A ("String
Payload Contract") as the documented 0.2.0 default, per the FTR's
own Discussion Note, OR adopt Option C ("Deferred Decision") with a
visible tracking entry. Either resolution is one paragraph in the
release notes (Known limitations or a new "Visible-cell typing"
section) and a one-sentence user-guide note in ch01 introduction or
the workflow chapter. Then move the FTR to
`docs/cold_storage/backlog/ftrs_done/` per the documented backlog
convention.

*Priority rationale.* Self-declared cutline. Cheap to close
(one-paragraph release-note edit + one-sentence user-guide edit +
FTR archival). The maintainer has already done the difficult part
(identifying the question, naming the options, articulating the
likely outcome); only the act of recording the decision remains.

*Observation.* Two mechanical pre-tag steps named in the existing
release runbook have not yet been performed. Both are *runbook
items*, not findings -- this is recorded so a single artifact
captures the complete "before tagging 0.2.0" checklist:

  runbook §"Branch and Topology Awareness" / §"Core Release
  Checklist" requires the operator to be on `main`, clean, in sync

  reports one `https://stefanschade.github.io/spreadsheet-handling-pages/latest/`

  §"Release-bound README links" documents the manual sed
  substitution and the `make readme-check TAG=v0.2.0` enforcement
  step. CI also enforces this through `make readme-check
  TAG=$GITHUB_REF_NAME` before `python -m build`, so an unbaked
  tag would fail at publish time, not after PyPI publication.

*Evidence.*

* `tools/release_check.sh` output (this review) -- 4 FAILs as
  above, exactly the pattern the script exists to surface.
* `tools/readme_links_check.sh` output (this review) -- 1

*Priority rationale.* Mechanical. The maintainer's own helpers
flag both items today. Not a quality finding; recorded only so
the readiness artifact lists the full set of before-tag actions
in one place.

== Release Mechanics Check

The repository carries enough release mechanics for a manual
`0.2.0` cut:

* *Version bump expectations.* setuptools-scm reads the latest
  `v*` tag; the next `v0.2.0` tag automatically produces
  `_version.__version__ == "0.2.0"`. No `__version__` literal
  needs to be edited in source.
* *Tag naming.* `v<version>`, no extra punctuation, per the runbook
  §"Tagging Convention". `v0.2.0` matches.
* *README link baking.* Documented in
  `02_release_runbook.adoc` §"Release-bound README links"; enforced
  at lint time (`make readme-check`) and at publish time (CI runs
  `make readme-check TAG=$GITHUB_REF_NAME` before `python -m
  build`).
* *Pages publication.* Documented in
  `01_ci_cd_and_publishing.adoc` (concurrency group, rebase-with-
  retry push policy, three repositories topology) and verified
  post-deploy through `make pages-check`.
* *PyPI publication assumptions.* PyPI Trusted Publisher; the
  publish job sequences `test -> publish -> publish-docs`; PyPI is
  the irreversible side effect (the runbook §"Troubleshooting:
  Accidental Release Topology" is explicit about this and is the
  documented basis for "no retag after PyPI success").
* *Release checklist / helper scripts.* `release-check`,
  `readme-check`, `pages-check`, `release-status` exist, are
  documented in `02_release_runbook.adoc`, and are read-only safe.
  Maintainer-recorded recovery scenarios (`Tag created from dev`,
  `Stale workflow rerun`, `PyPI succeeded but docs failed`,
  `Demo publish succeeded but CI hygiene failed`, `Pages publish
  green but public site not yet visibly updated`) are an unusually
  thorough operator-runbook layer for a project of this size.
* *Commands that should be run before tagging.*
  `make release-check` (after the merge to main and push), then
  bake the README link, then `make readme-check TAG=v0.2.0`,
  then create the tag, then `make release-check TAG=v0.2.0`,
  then push. The runbook gives this in sequence; the helpers
  enforce each step that can be enforced.

This review does *not* perform any release step. The mechanics are
considered sufficient as documented.
