== 1. Executive Summary

The recent consolidation waves (FK Helper v1 retirement, Legend Blocks
lifecycle clarification, XRef Crosstable functional-model promotion +
characterization + scoped recomposition) have left the project in a *healthy
and net-improved* state. The domain transformation layer is the strongest part
of the codebase: it is package-split, guard-enforced, and orchestrated by a
genuinely thin registry.

The single most important reframing this review offers: *the gap between
Functional Model and implementation runs the opposite direction from what one
might fear.* There is no evidence of Functional-Model-described runtime behavior
that is unimplemented (no "Red"). Instead, the *Functional Model lags the
implementation* -- only 3 of ~13 transformation families have been promoted to
`ch04_use_case_concepts`, while families such as `cell_codec`,
`compact_multiaxis`, `sparse_defaults`, `discriminator_split`, `enrich_lookup`,
`join_views`, and `tabular_views` are implemented and tested but not modeled.

`_meta` complexity is *better controlled but has moved, not vanished*. The
registry now has a real classification taxonomy and per-entry Intent/Resolution
wording, and the persistence boundary is a single chokepoint -- but that
chokepoint still contains hand-coded, per-family field knowledge, and the
registry is explicitly *not* a runtime contract. The functional model's own
"target state" (attribute-driven Type 2 events) is not yet reached, and this is
honestly tracked as `TD-META-LIFECYCLE-INVENTORY`.

The next architectural gravity point is the *spreadsheet adapter layer* -- not
because it is broken, but because it concentrates the most lines
(`ods/odf_parser.py` at 1147 LOC), holds the one remaining domain-semantic
coupling (`legend_blocks`), and is exactly where the tempting next features
(grouped headers, nested lists) would land.

*Bottom line:* the project is architecturally ready for the next backlog wave,
provided that wave is chosen for *leverage* (close the model-coverage and
meta-classification gaps) rather than *novelty* (grouped headers / nested
lists, which are not yet prepared).

== 4. `_meta` Health Assessment

*Fact -- the taxonomy now exists.* `registries/meta_registry.yaml`
(`registry_version: 2`) defines four classifications -- `canonical_meta`,
`derived_operational_view`, `carrier_artifact`, `read_path_hint` -- and two
layers (`meta_canonical`, `meta_rendering`). This is a real improvement: the
five conceptual kinds the prompt asks about are distinguishable, but via *three
different mechanisms* rather than one:

* *Carrier-local reconstruction hints* and *derived/generated meta* are
  distinguished by the `classification` field (`read_path_hint`, e.g.
  `__autofilter_ref`; `carrier_artifact`, e.g. `workbook_meta_blob`, `_hidden`;
  `derived_operational_view`, e.g. `options`, `__style`, `derived`).
* *Presentation meta* is distinguished by family + carrier-authority policy
  (`column_widths`, `text_orientations`, `horizontal/vertical_alignments`,
  consolidated by `io_backends/presentation_meta.py`).
* *Intent vs Runtime Resolution* is distinguished only in *prose*, inside the
  per-entry `persistence_behavior` text for `legend_blocks` and `xref_crosstable`,
  and re-encoded in the persistence-boundary code -- *not* in the classification
  taxonomy itself.

*Fact -- the registry is explicitly non-binding.* The maintenance contract
states `maintenance_mode: manual_review_artifact` and `runtime_binding:
forbidden`. The FM's stated *target* ("Type 2 event code reads only meta
registry attributes; a new domain step declares its attributes at registration
time", `ch05/03_meta_lifecycle_framework.adoc`) is therefore *not reached*.

*Fact -- field knowledge moved into one chokepoint.*
`pipeline/persistence_boundary.py` hand-codes the Type-2 egress rules:
`RUNTIME_ONLY_TOP_LEVEL_META_KEYS = {"derived"}`, a `__`-prefix filter, and
two named per-family projectors (`_project_legend_blocks`,
`_project_xref_crosstable`). This is acknowledged debt:
`TD-META-LIFECYCLE-INVENTORY` (Medium/High), tracked by
`FTR-META-LIFECYCLE-INVENTORY-P5`.

*Assessment (Specific Q2):* `_meta` is *better controlled, but complexity has
relocated rather than dissolved.* What used to be scattered, duplicated
field-level handling is now (a) labeled in a registry and (b) funneled through
one pure projection function. That is a genuine maintainability win. But the
registry cannot enforce anything, and the projection still grows one named
branch per family that has Resolution facets. Net: contained and documented, not
eliminated.

*Consistency check -- FK / Legend / XRef use `_meta` consistently?* Largely yes,
and the three are the most disciplined:

* FK Helpers: durable v2 `helper_policies.fk.relations` + transient
  `derived.sheets.*.helper_columns`; v1 retired and inert.
* Legend Blocks: durable entries/placement/identity vs droppable `resolved`
  working form + carrier reconstruction hints.
* XRef: Intent fields preserved; `column_keys` + `dense_axes.resolved` stripped
  at the sidecar boundary, retained in workbook carrier for cross-run fallback.

*Hidden control flow?* Mostly absent. Registry `precedence_notes` repeatedly and
correctly assert that pipeline execution must not branch on `frame_lifecycle` or
`derived`. The one deliberate `_meta`-drives-behavior path is XRef's cross-run
dense-axis fallback (read `dense_axes.resolved` when source frames are absent),
which is documented as a conscious cross-run persistence choice, not leakage.

*`canonical_meta` over-use (Specific):* yes, residual. The transform-config
entries `split_by_discriminator`, `sparse_defaults`, `cell_codecs`,
`compact_multiaxis` (and to a degree `frame_lifecycle`, `workbook_view`) are all
classified `canonical_meta` with `persistence_behavior: persisted ... unchanged`,
even though, like XRef, they plausibly carry mixed durable-intent and
runtime-resolution facets. Only FK (via `contract_lifecycle`), Legend, and XRef
have received the facet-level treatment. This is the clearest concrete residue.

*Stale / residue paths:* v1 `helper_policies.fk.<target>` (retired, inert);
`frame_lifecycle.{derived_from, produced_by}` (flagged Resolution but strip
deferred pending consumer audit -- see `persistence_boundary.py` out-of-scope
list); `sheets[*].helper_columns` declared-vs-resolved question deferred to
`FTR-FK-HELPER-POLICY-LIFECYCLE-P4A`.

== 5. Domain Package Health Assessment

*Assessment (Specific Q3): healthiest layer in the codebase; clearly improved.*

*Separation -- Fact.* `domain/transformations/` is package-split where
complexity warranted it: `fk_helpers/`, `discriminator_split/`,
`xref_crosstable/`, `cell_codec/`, `enrich_lookup/` are packages; single-module
families (`compact_multiaxis`, `sparse_defaults`, `tabular_views`,
`join_views`, `project_by_role`, `resource_overrides`, `derived_column_policy`)
remain flat, appropriately. `transformations/__init__.py` is empty -- families
are reached via their own subpackages, not a god-module.

*Discoverability + cohesion -- Fact.* Helper placement follows ownership:
`xref_crosstable/primitives.py` and `dense_axes.py` sit beside `operation.py`;
FK helper policy/provenance sit inside `fk_helpers/`. The pipeline registry
maps each YAML step name to a dotted `target`, so the family that owns behavior
is unambiguous.

*Registry as thin orchestration -- Fact (Specific Q: "thin or leaking?").*
`pipeline/registry.py` is purely declarative: `REGISTRY` maps strings to
`StepRegistration(factory=..., target="module:function")`. No transformation
logic lives there. Domain behavior is *not* leaking into orchestration.

*Cross-family dependencies -- one intentional, guards partial.*

* `transformations/_legend_blocks.py` is a *shared private reader* for the
  `_meta.legend_blocks` schema, consumed by both `cell_codec.scalar` and
  `compact_multiaxis`. Its docstring states the intent (remove a duplicated
  schema walk; not a public API). This is a deliberate shared helper, low risk.
* *Guard gap (Risk).* `tests/architecture/.../test_no_cross_family_internal_imports.py`
  enforces `GUARDED_INTERNAL_PREFIXES` for only `fk_helpers` and
  `discriminator_split`. The newer packages (`xref_crosstable`, `cell_codec`,
  `enrich_lookup`) are *not* in the guarded set, so an accidental cross-family
  import into their internals would not be caught. The guard mechanism exists;
  its coverage simply lags the package splits.

*Old compatibility paths -- mostly retired.* FK v1 dual-write bridge is retired
(`contract_lifecycle.transition_bridge.status: retired`); the
`__legend_blocks` read-path hint was removed (commit `e9a8361`).

*Newest XRef changes consistent with package + test style -- Yes.* Slice D
(`base_relation` scoped recomposition) lives in `operation.py`, reuses
`primitives.py`/`dense_axes.py`, and is preceded by characterization tests
(Slice A) before the runtime change -- matching the established
semantic-refactor-loop discipline.

== 6. Spreadsheet Adapter / Carrier Health Assessment

*Assessment (Specific Q4): yes, the adapter layer is the next likely hotspot --
but it is a known, bounded one, not a fire.*

*Fact -- domain transforms do not leak into adapters.* A grep for transform-config
meta keys across `io_backends/` and `rendering/` finds *only* `legend_blocks`.
`xref_crosstable`, `sparse_defaults`, `cell_codecs`, `compact_multiaxis`, and
`split_by_discriminator` do not appear in the adapter or rendering layers at all
-- they ride through `_meta` as opaque blobs. This is strong evidence that
carrier-specific behavior is isolated from domain transformations.

*Fact -- legend_blocks is the one legitimate crossing.* Both parsers contain
legend recognition (`ods/odf_parser.py::_legend_table_hints` /
`_apply_legend_table_hints`; `xlsx/openpyxl_parser.py` legend read). This is
inherent to the feature: a rendered legend is a *visible* support surface that
the parser must classify so it does not become payload (the FM "Parser Table
Classification" safety step). It is correctly the exception, not the rule.

*Fact -- the adapter weight is presentation metadata, and it has been
consolidated.* `ods/odf_parser.py` (1147 LOC, largest file in the tree) is
dominated by presentation extraction: column widths, h/v alignment, text
rotation, cell-style maps. The replace-or-clear policy was recently unified
across both backends into `io_backends/presentation_meta.py`
(`FTR-PRESENTATION-META-CARRIER-AUTHORITY-P5`), eliminating four near-duplicate
functions (three of which had missed the clear branch). Carrier authority is now
explicit.

*Fact -- alignment guards hold.* `test_adapter_alignment.py` confirms the
generic rendering package imports no `openpyxl`/`xlsx`, and
`spreadsheet_contract` uses only the generic `workbook_projection`. The
repeated-carrier memory-explosion class of bug is closed
(`BUG-ODS-REIMPORT-REPEATED-CELL-MEMORY-EXPLOSION` done; `ParserLimits` exists;
the lesson is codified in `review_instructions.adoc`).

*Risks / hotspots.*

* `ods/odf_parser.py` (1147) and `ods/odf_renderer.py` (675) are the two
  heaviest modules; `FTR-ODS-RENDERER-MODULARIZATION-P3K` and
  `FTR-ODS-FREEZE-READBACK-P3K` already sit in Phase 6.
* *Future-feature stress (Specific Q):* grouped/multiline headers and nested
  lists would stress the adapters precisely because they add *new visible
  structure* to the carrier -- exactly the place where `legend_blocks` already
  shows the coupling cost (new RenderOps + parser recognition on both backends).
  These features cannot be added cleanly until the adapter has a generic
  multi-row-header / nested-range capability.

== 7. Test Safety-Net Assessment

*Scale -- Fact.* ~1167 test functions, 103 `Test` classes; `pytest --co`
collects with no errors. Targeted runs during this review:

* `tests/architecture` + `test_xref_crosstable.py` + `test_persistence_boundary.py`
  + `test_legend_blocks.py` -> *185 passed*.
* `tests/architecture/semantic_invariants` (meta-registry guards) -> *28 passed*.

*Well covered:* FK helpers (policy-driven primitives, drop/enrich/provenance,
edge cases); XRef (unit + dense axes + scoped-recomposition characterization +
A1 boundary + A2 resolver); Legend Blocks (roundtrip, persistence, classification);
presentation-meta roundtrips (width, h/v align, orientation); architecture guards
(dependency, carrier boundary, parse contract, semantic invariants, meta-registry
lifecycle); validations.

*Characterized but not fully specified:* XRef blank-cell / `drop_empty`
semantics -- the characterization tests lock *current* behavior, but the
implementation concept (C.4/C.5) notes the mapping to the FM's "present blank ->
remove relation entry" is an interpretation, not a fully closed spec. This is
the right state for now (locked, then deliberately evolved), but it is not a
finished contract.

*Critical lifecycle boundaries lacking tests:* the *full* `_meta` lifecycle
inventory is uncharacterized (by design -- `FTR-META-LIFECYCLE-INVENTORY-P5`);
`frame_lifecycle.{derived_from, produced_by}` persistence behavior is not pinned
by a test (it is on the persistence-boundary "deferred" list).

*Brittle implementation-detail tests:* a handful of dense-axis tests assert the
exact `_meta` dict shape (e.g. `dense_axes.resolved.row_identities`). These are
implementation-coupled but acceptable as *characterization* of a meta contract;
the volume is not excessive.

*Consumer-like integration tests:* present
(`test_consumer_smoke`, `test_cli_orchestrator_end_to_end`,
`test_localized_resource_matrix_roundtrip`,
`test_configured_workbook_view_reimport_roundtrip`). The larger consumer surface
lives in the demo repo, out of scope here.

*XLSX / ODS parity risk -- visible and bounded.*
`tests/integration/io_backends/test_ods_xlsx_parity.py` exists and *documents*
the capability gap (ODS skips protection ops) rather than hiding it. Parity risk
is therefore explicit, not silent.

*Did XRef work improve the net in the right places? -- Yes.* The
characterization-first sequencing (Slice A locks behavior; Slice D changes it)
added exactly the boundary and resolver tests the inventory flagged as missing.
Both proposed tests (A1 `test_drops_xref_crosstable_column_keys_facet`, A2
`test_resolves_row_identity_from_contract_xref_row_keys`) are present.

persistence boundary can become attribute-driven (the FM target). It is not a
flashy "feature," but it unblocks safe extension of *every* transformation family
and retires the per-family hand-coding in `persistence_boundary.py`. Cleanup
slices 1 and 3 above are its natural first steps.

Among genuine *feature* families, the next major wave should be *workbook_view /
sheet_mappings / frame-lifecycle completion* (resolve
`FTR-WORKBOOK-VIEW-PAYLOAD-CONFLICT-PRECEDENCE-P6`): it is "ready after small
cleanup," carries direct consumer value, and adds *no* adapter risk -- in
contrast to grouped headers / nested lists, which do. Promoting and extending
`cell_codecs` / `compact_multiaxis` in the FM is a strong parallel modeling track.

*Explicitly do not pick grouped headers or nested lists as the next wave* -- they
land in the least-prepared layer (the adapters) and depend on capabilities that
do not yet exist.
