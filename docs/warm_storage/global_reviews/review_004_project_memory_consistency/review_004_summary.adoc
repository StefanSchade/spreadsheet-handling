== Executive summary

Project_memory is *structurally sound where it matters most and dirty in a few
localized, fixable places*. It is a usable steering model today, but two issues
slightly compromise trusting it to *sequence the next slice on its own*: the
single most decision-relevant BUG mis-states itself as implemented, and FTR/finding
status is not a controlled vocabulary.

*What is reliable.* Identity integrity is clean (no duplicate ids in any table).
The concern / event / xref / review-set layer has zero dangling references and
zero orphans, and the context-report Diagnostics confirm it. The
assessment-scale mapping is a genuine derived view (config table +
derived join); no canonical finding stores a numeric impact, so there is *no
pseudo-truth* and the mapping layer reduces rather than adds ambiguity. Derived
reports remain derived and gitignored.

*What is dirty (all localized field hygiene, not entity confusion).*

* PMC-002 (high): `ftrs.status` is free text -- 24 distinct values, some full
  sentences, 58 empty. The model's core "what is open?" question is not queryable.
* PMC-007 (medium): BUG-UNRESOLVED-HELPER-VALUE-NAN-SEMANTICS-P4A reads as
  implemented (status=`new` but `implementation_commits` set + "Implemented in a
  single commit"), while it is actually an *unresolved contract guarded by a live
  xfail*. The commit cited introduced the sentinel layer, not a fix.
* PMC-001 (medium): findings use undocumented `done` and `in_arbeit` statuses.
* PMC-003/004/005 (medium-low): `superseded_by` is list-valued and sometimes
  points to FTRs not findings; `resolved_by_ftr_id` is multi-valued; `tracking_ftr_ids`
  contains a prose token.
* PMC-006 (medium): `reviews.json` has duplicate competing columns
  (`Verdict`/`verdict`, `set`/`review_set_id`), out of sync.
* PMC-008 (medium): the health/diagnostics layer does not check findings'
  reference fields -- the model is blind to its own dirtiest data.

== Recommended next engineering slice

*Slice A -- project_memory consistency hardening, including a structured
executable-sentinel representation*, sequenced immediately before *Slice C --
the FK-helper unresolved-value contract decision (BUG-...P4A)*.

A is the safest (data + diagnostics only, near-zero runtime risk), discharges
this review's findings, and makes C's xfail->green transition observable as a
real state change rather than a silent header edit. C is the best-prepared,
highest-severity product slice and is independent of ODS and of any meta
redesign. The minimal meta-boundary contract (B) should follow C, using the
FK-helper case as its worked example; ODS decomposition (D) stays gated prep.

Full comparison, prerequisites, and risks: `next_slice_recommendation.adoc`.
