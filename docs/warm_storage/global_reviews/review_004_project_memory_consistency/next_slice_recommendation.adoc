== Assessment of each candidate

=== A -- project_memory sentinel + consistency hygiene

*State today.* The xfail sentinel is real and green (`1 xfailed`), but
project_memory represents it only narratively (one `manual_note` event with a
`source_id` that resolves to no table) and the BUG header record contradicts it
(status=`new` yet `implementation_commits` + "Implemented in a single commit" --
PMC-007). Project_memory cannot currently express the three distinct states the
prompt asks about: *test-green*, *known-xfail*, *unresolved-semantic-contract*.
On top of that sit PMC-001..PMC-008 (status-vocabulary drift, free-text FTR
status, list-valued/type-mixed reference fields, duplicate `Verdict`/`verdict`,
diagnostic blind spot).

*Risk.* Very low. Data + diagnostics only; no runtime code path touched.
`make memory-context` validates the result. Worst case is a re-run.

*Prerequisites.* None. Bounded by the maintenance-cycle rules (curate
deliberately, commit separately).

*Leverage.* High *for the steering model*: it repairs exactly the drift that
would otherwise make the model untrustworthy for sequencing B/C/D, and it makes
C's eventual "green" transition observable as a state change rather than a
silent header edit.

=== B -- minimal meta-boundary contract

*State today.* Three current findings name concrete gaps (FIN-TOPIC-META-PSR-001
load-bearing metadata; FIN-TOPIC-MODEL-CMB-005 `derived_from`/`produced_by`
ambiguity; FIN-TOPIC-MODEL-SBR-004 persistence-boundary subsection). An open
FTR host exists (FTR-REVIEW-001-META-BOUNDARY-CLEANUP-P5; FTR-PAYLOAD-SEMANTICS-RECOVERY-P5).

*Risk.* Medium. The boundary is genuinely under-specified; without tight
scoping this slice tends to grow toward a meta-model redesign (explicitly out of
scope). The FK-helper unresolved-value case (C) is the cleanest *concrete
instance* of the payload-semantics question, so doing B before C risks
theorizing the contract in the abstract.

*Prerequisites.* A decided FK-helper representation (C) gives B a worked example
to generalize from.

=== C -- FK-helper unresolved-value contract (BUG-...P4A)

*State today.* The best-prepared product slice: 3 options documented, maintainer
leans option 1 (empty/null), an executable xfail already pins the open contract,
the twin cleanup BUG is done, and it is post-0.2 non-blocking. One *major*
current finding (FIN-TOPIC-META-PSR-002) is the steering signal.

*Risk.* Low-medium. Bounded to the FK-helper family on the canonical persistence
path; the BUG's non-goals explicitly fence out framework-wide NaN/null
normalization. Main residual risk is choosing option 1 vs option 2 (structured
finding) -- a decision, not a technical unknown.

*Prerequisites.* Confirm the producing carrier (`core/fk.py` `_norm_id` NaN
path) before coding; make the option decision. Does *not* depend on ODS or on a
meta redesign.

=== D -- ODS behavior-preserving decomposition

*State today.* Gated prep per the io_backends review (Mild hotspot). Valuable but
lower-leverage now; no current/major finding drives it.

*Risk.* Low technically, but low payoff this cycle. It is preparation for a
*future* visible-structure feature, not a response to a live finding.

== Recommendation

*Recommended next slice: A -- project_memory consistency hardening, including a
structured executable-sentinel representation.* Sequence C immediately after, as
the first product slice.

Rationale:

. *Safety.* A is the only candidate with essentially zero runtime risk
  (data + diagnostics), and this review is itself a project-memory reliability
  review -- A directly discharges its findings.
. *Leverage / ordering.* The model currently mis-states the single most
  decision-relevant BUG (PMC-007) and cannot distinguish test-green / known-xfail
  / unresolved-contract. Until that is fixed, the model is a weak basis for
  choosing between B/C/D. Doing A first means C's progress (xfail -> green) is
  recorded as a real state transition, and the next reviewer can trust the
  triage.
. *Then C.* C is the best-prepared, highest-severity product slice (one major
  current finding, decision-ready, sentinel in place, ODS-independent,
  meta-redesign-independent). It also yields the concrete worked example that
  makes B (the broader meta-boundary contract) tractable instead of abstract.
. *B after C; D stays gated prep.*

=== Concrete content of slice A (bounded)

* Make the BUG-...P4A record stop reading as implemented: keep status open /
  unresolved; record `88e4b78` as the sentinel/invariant-layer commit, not an
  implementation (PMC-007).
* Give the executable sentinel a queryable representation distinguishing
  *test-green* / *known-xfail* / *unresolved-contract* (a small `sentinels`
  notion or explicit fields on the BUG; the prompt forbids a large ontology, so
  prefer minimal fields over a new model).
* Normalize the status vocabularies (PMC-001/PMC-002) and flatten the
  finding reference fields (PMC-003/PMC-004/PMC-005); collapse the duplicate
  `reviews` columns (PMC-006).
* Extend the health report to validate findings cross-references (PMC-008).
* Run `make memory-context` before and after; commit curated deltas separately
  from any product change.

=== Risks of the recommendation

* *Meta-work creep.* Slice A must stay a bounded hygiene pass (minutes-to-hours,
  per the maintenance-cycle guidance), not a schema redesign. The biggest risk is
  over-engineering the sentinel representation -- keep it to minimal fields.
* *Deferring C.* C carries the only major current finding; if release pressure on
  the FK-helper contract rises, run A and C in the same short sequence rather than
  letting A expand. A is explicitly the *enabler* for C, not a substitute.
