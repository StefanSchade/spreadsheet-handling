Reflective analysis, read-only. No runtime code, tests, registries, or backlog
items were modified. Claims are grounded in concrete project artifacts and cited
by path; where the evidence is thin or qualitative, that is stated rather than
smoothed over. This document is an opinion piece backed by evidence, not a
neutral specification and not a universal claim about AI-assisted development.
====

== Purpose

This document extracts the working method this repository has actually used for
AI-assisted development, so the maintainer can decide what to keep, what to
tighten, and what to drop as the project continues. It is written first for
internal steering of future work in this repository, and second as a
professional argument the maintainer can use to explain this development style
to colleagues or reviewers who have only seen the "vibe coding" caricature.

It is deliberately not a method guide for other projects. The evidence is local;
the conclusions are local. Where a practice is likely to generalise, that is
flagged as a hypothesis, not a result.

== Executive Summary

The project's own history records a transition "from architecture creation to
architecture comprehension"
(`docs/developer_guide/ch10_project_evolution/03_domain_comprehension_and_semantic_refactoring_wave.adoc`).
That single sentence captures why AI assistance has been effective here: the
work that AI did most of -- inventory, comparison, contradiction detection,
structured drafting, review -- is exactly the work that "comprehension" requires,
and the project externalised that work into durable, versioned artifacts rather
than leaving it in chat.

The effective ingredients, in order of leverage:

. *Artifacts carry context, not chat memory.* FTRs, functional/semantic model
  docs, warm-storage reviews, the `audit/` corpus, and architecture guards mean
  a cold-start agent reconstructs intent from files, not from a conversation it
  never had.
. *Reviews are first-class, committed artifacts* (the `audit/` directory holds
  ~20 of them for the FK-helper and domain-cleanup wave alone). They exist to
  find semantic drift, and they repeatedly did.
. *Modeling precedes refactoring where behavior carries domain meaning.* The
  project paused refactoring after REVIEW-002 specifically because "metadata
  meaning must be modeled before more refactoring"
  (`.../ch10_project_evolution/01_project_evolution.adoc`).
. *Guards are executable memory.* Architecture tests pin decisions an agent would
  otherwise re-litigate or silently violate.
. *Backlog truth is maintained deliberately,* because stale artifacts mislead the
  next agent more than they mislead a human.

The strongest counter-evidence to the "vibe coding" framing is the FK-helper
wave's own diagnosis: the initial symptom was "high token consumption during bug
analysis"
(`docs/cold_storage/working_context/domain_cleanup_fk_helper_2026_06/domain_cleanup_procedure_lessons_fk_helper_slice.adoc`), and the response
was *not* to spend more tokens or grant more autonomy. It was to write a
functional model, inventory the metadata paths, and reason from evidence. Token
spend was treated as a symptom of artifact ambiguity, not as a resource to
maximise.

The method is real but incomplete. The Functional Model still covers only 3 of
~13 transformation families (`architecture_health_review.adoc`), the meta
registry is explicitly non-binding, and the persistence engine is deliberately
unbuilt. The method has earned trust on the slices where it was applied; it has
not been proven at scale, and this document does not claim it has.

== Working Thesis

[quote]
____
This project does not practice vibe coding. It practices artifact-guided,
review-driven, model-conscious AI-assisted engineering.
____

The thesis holds on the evidence, with one important refinement: the project did
not arrive at this method by design. It arrived by *failure recovery*. The
domain-cleanup wave began "as a conventional package-refactoring exercise" and
only became a method when the FK-helper review "exposed a different class of
problem" -- that the dominant source of complexity "was not code structure" but
"uncertainty about metadata meaning"
(`.../03_domain_comprehension_and_semantic_refactoring_wave.adoc`). The method is
therefore better described as *earned discipline* than as an a-priori
philosophy, and that origin matters: it is why the practices are concrete and
load-bearing rather than aspirational.

== What This Project Did Differently

Three structural choices distinguish this repository from a chat-driven workflow:

* *Context lives in the tree, addressable by path.* `docs/ai_info/` is a
  purpose-built agent context surface (`agent_rules.adoc`,
  `review_instructions.adoc`, `conventions.adoc`, `interfaces_and_gates.adoc`,
  `architecture_primer.adoc`, `project_context.adoc`,
  `repository_boundaries.adoc`, `concepts_fk_helpers.adoc`). An agent is told
  what to load, in what order, and what is authoritative.
* *Work units are contracts, not requests.* FTRs carry acceptance criteria,
  scope boundaries, non-goals, and traceability markers
  (`docs/developer_guide/ch05_ai_friendly_development/01_ai_friendly_development.adoc`),
  so "done" is checkable rather than felt.
* *The history is curated for the next reader.* The project evolution chapter
  ends with explicit "Reading Rules for Future Agents" that say to "treat cold
  storage as archaeology, not default context" and to "prefer review summaries
  over raw review prompts"
  (`.../ch10_project_evolution/01_project_evolution.adoc`). The project assumes
  its future reader is an agent with no memory and optimises for that reader.

== Practices That Worked

=== Artifact-Guided Context Instead of Chat Memory

The project does not rely on a long conversation to reconstruct intent. The
FK-helper investigation explicitly ran a "documentation-blind domain audit"
(`docs/cold_storage/working_context/domain_cleanup_fk_helper_2026_06/domain_cleanup_procedure_lessons_fk_helper_slice.adoc`) *and* a
documentation-driven model pass, then compared them. That only works because both
the code-derived view and the intended view exist as artifacts that can be set
against each other.

*Why it helped AI specifically:* an agent reconstructing intent from a stale or
contradictory conversation guesses; an agent comparing two written artifacts
detects the contradiction. The project plays to the second mode and avoids the
first.

*Counterpoint / nuance:* more context is not automatically better. The
"Reading Rules" deliberately cap context ("load the detailed FTR only if the
short context is insufficient"). The discipline is *curation*, not accumulation
-- a point the io_backends review reinforces by noting the prompt named files
(`io_backends/backend.py`, `options.py`) that "do not exist"
(`io_backends_carrier_boundary_review.adoc`), a reminder that even well-intended
context can be wrong and must be checked against the tree.

=== FTRs as Scope and Lifecycle Contracts

An FTR fixes scope (`Purpose` / `Scope` / `Non-Goals`) and now also fixes
lifecycle. The recently tightened Definition of Done states that "a workstream is
not complete until the active planning surface is true again"
(`docs/developer_guide/ch08_git_and_workflow/01_git_and_workflow.adoc`,
"FTR Workstream Completion / Lifecycle Cleanup"). This distinguishes
*implementation complete* from *review green* from *workstream complete* -- three
states a chat-driven workflow usually collapses into one.

*Concrete evidence it was needed:* the backlog triage found completed work still
presented as open -- `FTR-LEGEND-BLOCKS-LIFECYCLE-P6` (Status `Done`) and two
`Fixed` P4A bugs were still on the active include surface, and
`FTR-SCHEMA-EVOLUTION-OPERATIONS` was marked `New` despite its slices being
implemented (`backlog_triage_review.adoc`). The lifecycle contract exists because
its absence produced observable drift.

=== Functional Model Before Implementation Where Semantics Matter

The project does not model everything. It models where code behavior carries
domain meaning and silence would cause drift. The pivot is dated: after
REVIEW-002 "the project paused broader refactoring work and started explicit
model documentation," with the lesson that "not every surprising field is
architectural debt; some fields encode reconstruction authority"
(`.../ch10_project_evolution/01_project_evolution.adoc`).

The FK-helper lessons make the mechanism explicit: stop reasoning from metadata
categories ("is this canonical? is this derived?") and start from feature
behavior ("what is the feature trying to do? where is information created,
consumed, persisted, removed?")
(`docs/cold_storage/working_context/domain_cleanup_fk_helper_2026_06/domain_cleanup_procedure_lessons_fk_helper_slice.adoc`, "Functional Model
First"). The result "was substantially easier to reason about than the existing
metadata structures."

*Counterpoint:* modeling has its own failure mode -- inflation. The architecture
review warns that `ch04` "self-labels as Exploratory ... pending editorial
cleanup," and the sequencing review explicitly rejects mass-promotion of all
families to avoid "FM inflation"
(`next_program_sequencing_review.adoc`). Modeling is applied as a scalpel
(use-case-anchored, one family at a time), not as a mandate.

=== Reviews as First-Class Engineering Artifacts

Reviews here are committed files with findings, dispositions, and residual-risk
sections, not chat summaries. The `audit/` directory alone holds the FK-helper
lifecycle review, meta-path inventory, functional-model-to-implementation review,
v1-retirement review, consolidation closure, and consumer-verification -- the
full "artifacts" list the wave narrative enumerates
(`.../03_domain_comprehension_and_semantic_refactoring_wave.adoc`). The
`docs/ai_info/review_instructions.adoc` rules require findings to carry
observation, evidence, concrete risk, recommended action, and disposition.

*Why reviews mattered beyond bureaucracy:* they repeatedly discovered *semantic*
drift that structure-level checks miss. The legend-reconstruction follow-up
"showed that some fields looked render-local but also carried reconstruction
meaning" -- a class of bug a purely structural refactor would have introduced by
"cleaning up" a field whose authority was unmodeled. The "Reading Rules"
codify the resulting caution: "avoid cleanup that removes fields before their
authority is modeled."

=== Architecture Guards as Executable Memory

Decisions that would otherwise erode are pinned as tests. The cross-family
internal-import guard, the adapter contract/alignment guards, and the
meta-carrier-boundary semantic-invariant guards all encode "do not do this again"
as CI (`architecture_health_review.adoc`,
`io_backends_carrier_boundary_review.adoc`). The "Reading Rules" instruct agents
to "add or update guards with the change," so memory is extended as the system
changes.

*Honest limitation the reviews themselves surface:* guard coverage lags reality.
The cross-family guard covers only `fk_helpers` and `discriminator_split`, not
the newer `xref_crosstable` / `cell_codec` / `enrich_lookup` packages
(`architecture_health_review.adoc`); and `legend_blocks` is the one legitimate
domain crossing into `io_backends` but is "not protected by a guard"
(`io_backends_carrier_boundary_review.adoc`). Executable memory is only as good
as its coverage, and the project tracks the gap rather than claiming completeness.

=== Backlog Truth and Lifecycle Cleanup

Backlog cleanup is treated as correctness work, not cosmetics, with a specific
stated reason: stale inventory artifacts mislead future agents more than humans.
The triage found the `FTR-XREF-CROSSTABLE_inventory` "Gaps in Test Coverage"
section still listing gaps that Slice D had already closed
(`backlog_triage_review.adoc`), and `_ftrs_open.adoc` now carries an explicit
open-only contract: "Only open work belongs here. Done, Fixed, Superseded, or
historical workstream artifacts must leave this file."

*Why this is an AI-specific concern:* a human reading a backlog applies social
context ("that's old"); a cold-start agent treats the active surface as ground
truth. The project's own "Reading Rules" -- "treat cold storage as archaeology,
not default context" -- exist precisely to stop an agent from acting on
historical work as if it were current.

=== Prompting as Scoped Work Design

Prompting in this repository is not phrasing tricks; it is work design. The task
prompts that produced the recent reviews specify inputs, non-goals, allowed and
forbidden changes, validation commands, and required output shape. The backlog
triage prompt, for instance, fixed the one file the agent could write and listed
nine "Do Not Touch Yet" items. This is why the model could be given real latitude
without drifting: the boundary was explicit, so autonomy inside it was safe.

*The honest version:* good prompting mattered because it set scope and
validation, but it mattered *less* than artifact discipline. A perfectly phrased
prompt against a stale backlog still produces wrong work -- which is exactly why
the lifecycle and backlog-truth practices exist upstream of the prompt.

=== Small Vertical Loops Instead of Big-Bang Refactors

The project explicitly refused a global `_meta` redesign and adopted a repeatable
per-family loop: "inventory metadata paths; document the functional problem
space; derive a functional model; compare model against implementation; identify
historical residue; remove only complexity that has no demonstrated consumer;
preserve proven contracts; repeat"
(`.../03_domain_comprehension_and_semantic_refactoring_wave.adoc`). The XRef
crosstable promotion is the cleanest instance: characterization tests (Slice A)
locked current behavior *before* the scoped-recomposition runtime change (Slice
D), and the FM's explicit "Deferred and Neighbouring Concepts" list kept Slice D
from absorbing axis mappings or grouped headers
(`next_program_sequencing_review.adoc`).

The sequencing review generalises this into a tiered method (classify-only /
lightweight-FM / inventory-first / full-chain) precisely to avoid forcing every
family through the full ceremony.

== What This Is Not

=== Not Vibe Coding

Vibe coding trusts the generated result because it looks right. This project
characterises behavior with tests *before* changing it (XRef Slice A), models
meaning before refactoring (post-REVIEW-002), and reasons from a metadata
inventory rather than intuition. The FK-helper inventory's key result -- that
"several suspicious structures were discovered to be genuinely load-bearing"
while "several other structures turned out to have no active consumers"
(`audit/...fk_helper_slice.adoc`) -- is the opposite of vibe: it is the explicit
refusal to act on appearance.

=== Not Token Maximization

The triggering symptom of the entire domain wave was "high token consumption
during bug analysis." The response was to *reduce* the reasoning surface by
writing a functional model, not to spend more. The lessons document names this
directly under "Evidence Instead of Ontology": rich metadata categories
"encouraged reasoning through abstraction before understanding the feature
itself," and the fix was to trace one concrete feature's information flow. Token
spend was a diagnostic signal, not a budget to fill.

=== Not Blind Agent Autonomy

Read-only reviews are required to stay read-only; the review instructions state
"Reviewers do not silently become implementers"
(`docs/ai_info/review_instructions.adoc`). The new Lifecycle cleanup check
distinguishes recommending cleanup (read-only tasks) from performing it
(authorised tasks). Autonomy is bounded by task scope, and the bound is written
down.

=== Not a Generic Agent-Orchestration Framework

There is no runtime that schedules agents, no shared mutable agent state, no
graph of tool-calling nodes. The "orchestration" is the artifact set: FTRs
sequence work, reviews gate it, guards enforce it, git records it. It is
orchestration expressed as version-controlled documents rather than as a running
system.

== Why No LangGraph-Style Orchestration Was Needed So Far

The project externalises orchestration into versioned artifacts -- FTRs, reviews,
guards, model docs, and git history -- and that has so far covered the
coordination needs that a framework like LangGraph would otherwise serve.
Concretely:

* *State is the repository.* A framework's value is largely managing state and
  control flow across steps. Here, state is the working tree plus the FTR/review
  artifacts, and control flow is the human-sequenced phase plan in
  `_ftrs_open.adoc`. Both are diffable and reviewable; an opaque orchestration
  runtime is neither.
* *Re-entrancy is already solved by artifacts.* The whole point of the FTR/review
  design is that a fresh-context agent can resume from files. That is the same
  problem persistent-graph frameworks solve, solved instead with documents.
* *The unit of work is one reviewable slice,* not a long autonomous multi-step
  run. The tiered method keeps slices small enough that per-task agent invocation
  plus human review is sufficient.

*This is not a claim that such frameworks are useless.* The calculus changes if
the project needs genuinely concurrent long-running agents, multi-agent
coordination over shared mutable state, or automated fan-out/fan-in that a human
cannot sequence by hand. None of those pressures has appeared yet. The honest
statement is "not needed *so far,* given the current slice size and human-in-the-
loop cadence," not "never needed."

== Failure Modes Observed

These are drawn from the project's own records, not hypotheticals:

* *Stale artifacts as false truth.* Done FTRs and closed gaps remained on the
  active surface and in inventories (`backlog_triage_review.adoc`). Mitigated by
  the lifecycle DoD; the mitigation is new and unproven over time.
* *Reasoning through ontology before evidence.* Rich metadata categories invited
  abstract reasoning that increased cost without understanding
  (`audit/...fk_helper_slice.adoc`). Mitigated by functional-model-first.
* *Implementation sediment masquerading as architecture.* v1/v2 representations,
  compatibility layers, and persistence-boundary exceptions made FK helpers look
  architecturally broken when "the feature itself was smaller than its
  implementation history." Mitigated by inventory-before-refactor.
* *Fields with hidden authority.* Render-local-looking fields that carried
  reconstruction meaning -- a refactor hazard
  (`.../ch10_project_evolution/01_project_evolution.adoc`).
* *Tempting false abstraction.* The io_backends review explicitly refused to
  unify the XLSX (live object model) and ODS (buffered, named-style carrier)
  renderers, classifying that divergence as "a correct carrier difference," and
  warned that most of the ODS/XLSX size gap "is mostly intrinsic, not accidental"
  (`io_backends_carrier_boundary_review.adoc`). The failure mode here is the
  abstraction an agent *would* propose because the two look similar.
* *Prompt/reality mismatch.* The io_backends review found prompt-named files that
  did not exist and mapped "the layer as built" instead -- a reminder that the
  agent must verify context, not trust it.
* *Guard coverage lag.* Real decisions (legend-only crossing, newer package
  internals) were unguarded at review time.

== Practical Rules for Future Work

Keep, because the evidence supports them:

. Characterise before you change behavior; model before you refactor meaning.
. One transformation family at a time; let abstractions "emerge from repeated
  observations," not from an up-front ontology.
. Treat a metadata inventory (producer / consumer / persistence / tests /
  runtime dependency) as a prerequisite, not a nicety, when touching `_meta`.
. Keep the persistence boundary's "preserve Intent, drop Resolution" rule, and
  *do not* build the generic attribute-driven engine until a third
  Resolution-bearing family demands it (`next_program_sequencing_review.adoc`,
  `docs/ai_info/agent_rules.adoc` -- Persistence boundary).
. Add or update a guard with every boundary decision; close the known guard-
  coverage gaps before the features that would exploit them.
. Finish the workstream: status, includes, archival, residual scope -- the
  planning surface must be true before a workstream is "done."
. Write prompts as work design: inputs, non-goals, allowed/forbidden changes,
  validation, output shape.
. Verify prompt context against the tree before acting on it.

== Professional Framing

For a workplace audience that equates "AI-assisted" with "unreviewable
autocomplete," the defensible framing is narrow and concrete:

* AI was used where it is strong -- *inventory, comparison, contradiction
  detection, structured drafting, and review* -- and these produced committed
  artifacts that a human and the next agent can both audit.
* Human judgment remained responsible for the decisions AI is weak at:
  priorities, model boundaries, and accepting trade-offs (e.g. the explicit
  decision to defer the persistence engine, and to refuse the ODS/XLSX renderer
  unification).
* The safety property is not "the agent is trusted." It is "every consequential
  change leaves a reviewable artifact and, where behavior matters, an executable
  guard." That is auditable in a way a chat transcript is not.

This framing is honest because it concedes the limits: the method is most
defensible *because* it does not depend on trusting the model, and it is most
useful on work that is analysis-heavy and artifact-shaped. It would be weaker
evidence for, say, green-field exploratory UI work where there is no existing
contract to compare against.

== Open Questions

* *Does the lifecycle DoD hold over time?* It was added after drift was already
  observed; whether it prevents recurrence is not yet demonstrated.
* *Will the tiered modeling method scale to the remaining ~10 unmodeled
  families* without becoming the ceremony it was designed to avoid?
* *At what point does artifact-as-orchestration stop scaling?* If slice
  parallelism or autonomous multi-step runs increase, the cost-benefit of a real
  orchestration layer changes -- and the project has no measured threshold for
  that yet.
* *Is the method transferable, or is it co-dependent with this maintainer's
  review habits?* The evidence cannot separate the two.
* *How much of the observed "AI did well here" is selection bias* -- the project
  kept the artifacts where the method worked and may under-record where it did
  not?

== Conclusion

The thesis survives scrutiny: this is artifact-guided, review-driven,
model-conscious AI-assisted engineering, and the project's own narrative -- "from
architecture creation to architecture comprehension" -- names the same thing from
the inside. The practices are load-bearing because they were earned by recovering
from a concrete failure (the FK-helper wave), not adopted as doctrine.

The appropriate confidence level is "this worked, repeatedly, on the slices where
it was applied, and is worth continuing," not "this is the way to do AI-assisted
development." The method is incomplete by its own accounting, depends on a human
who still owns the hard judgments, and has not been tested under the conditions
(scale, parallelism, autonomy) that would justify heavier machinery. Its central,
transferable insight is modest and well-evidenced: when code behavior carries
domain meaning, the scarce resource is not model capability or context volume --
it is *truthful, comparable, version-controlled artifacts*, and the discipline of
keeping them true.
