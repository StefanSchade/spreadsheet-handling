== Executive Summary

The active backlog has not fully caught up with the Phase 3.5 semantic-model foundations work. The most visible mismatch is around FK-helper policy semantics: implementation, registries, and review artifacts now show policy-driven primitive behavior, while related active FTRs still appear as new or unreconciled.

The backlog also contains older modularization and consolidation guidance that predates newer REVIEW-002 outcomes. Those entries should not be executed literally without re-anchoring to current implementation boundaries, glossary vocabulary, and observed risks.

This is mostly a backlog-governance problem, not evidence that Phase 3.5 requires a new semantic phase. The practical next step is a cleanup pass that closes stale lifecycle state, updates terminology, and groups related xref/compact/grouped-column work around existing transformation families.

== Terminology Drift

=== Frame vs sheet

The glossary now distinguishes in-memory frames from workbook sheets. Backlog and older architecture text still sometimes use sheet language for semantic frame behavior. Public workbook APIs may still legitimately use sheet-oriented terms, but semantic backlog text should use `frame` for data units and `workbook view` for projected workbook layout.

=== Canonical vs derived

The technical model now treats canonical/non-canonical primarily as metadata classification. Data should generally be described as original or derived. Older backlog language that implies canonical data frames should be updated when touched.

=== Model metadata vs presentation metadata

Column widths, workbook view state, helper fills, protections, and sheet mappings need vocabulary that separates model metadata from presentation metadata and workbook-identity metadata. The XLSX parser review especially depends on this distinction.

=== Operational-carrier semantics

`carrier`, `carrier-shape`, and `operational-carrier` are now separate terms. Backlog text that treats every adapter-derived fact as a carrier leak should be narrowed.

=== Tuple, matrix, and crosstable terminology

The user guide currently describes `expand_xref` and `contract_xref` directions consistently with implementation. At least one older concept document reverses this direction. Backlog items that build on tuple/matrix conversion should align with the current implementation and glossary before execution.

== Semantic-Model Alignment Issues

The semantic-model documents still read partly as a draft/adoption scaffold, while parts of their pilot content are already implemented. This creates a mild mismatch:

* `docs/semantic_model/semantic_model.adoc` still presents the model as a scaffold.
* `docs/semantic_model/12_contract_matrices.adoc` records the FK-helper pilot and resolved drift.
* `docs/semantic_model/15_adoption_strategy.adoc` frames registry use as review/drift detection, not runtime input.
* Active backlog state does not consistently reflect which semantic-model pilot work has already landed.

Assessment::
This is acceptable as a transitional state, but it should be made explicit during backlog cleanup so future work does not treat completed pilots as aspirational work.

== Backlog Fragmentation Observations

The backlog is starting to fragment around related transformation-family concerns.

XRef/grouped/compact fragmentation::
Axis mappings, grouped columns, compact xref use cases, dense axes, and matrix-to-relation conversion are closely related. They can remain separate implementation slices, but should share vocabulary and acceptance criteria.

Semantic-model vs active-backlog mismatch::
Some semantic-model work is documented as a pilot foundation, while active backlog still carries older FTRs as if the pilot has not happened.

Registry validation fragmentation::
Meta schema validation, lifecycle guards, pipeline registry review, and contract matrices can reinforce one another, but should not be merged into a broad registry governance project.

Module backlog fragmentation::
Generic modularization guidance is less useful than small module-specific reconciliation tied to semantic ownership, adapter boundaries, or guard gaps.

== Recommended Cleanup Categories

=== Must-fix

These items should be reconciled before related backlog execution:

* Update lifecycle state for FK-helper FTRs whose core work has already been implemented and reviewed.
* Re-anchor `FTR-REVIEW-001-MODULARIZATION-P4A` before using it to justify module splits.
* Correct tuple/matrix direction drift in older architecture text before using it as acceptance criteria.
* Name a pilot for `FTR-META-SCHEMA-VALIDATION-P5` before expanding schema-validation work.

=== Should-fix

These items should be handled during the next backlog/doc cleanup pass:

* Harmonize `frame`, `sheet`, `workbook view`, and `workbook_identity_meta` terminology in active FTRs.
* Review `FTR-DOMAIN-CONSOLIDATION` supplemental guidance against current FK-helper package reality.
* Group xref, grouped-column, compact, and dense-axis backlog items under shared vocabulary.
* Inventory architecture guards against stabilized Phase 3.5 semantics.
* Align compact-transform documentation with current implementation naming and behavior.

=== Acceptable Transitional State

These areas can remain transitional if they are not treated as final architecture:

* Registry semantics as review/drift truth rather than runtime input.
* Opt-in lifecycle classification instead of mandatory lifecycle status for every contract.
* Partial semantic-model adoption where FK helpers are the first completed pilot.
* Separate FTRs for xref and compact work, provided future cleanup prevents contradictory terminology.
* Older backlog entries remaining active temporarily, provided they are not executed literally without re-review.
