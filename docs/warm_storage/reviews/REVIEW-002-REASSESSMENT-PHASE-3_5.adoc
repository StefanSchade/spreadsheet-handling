= REVIEW-002 Reassessment After Phase 3.5
:toc:
:toclevels: 2

Review date:: 2026-05-27
Scope:: Reassessment of REVIEW-002 after the Phase 3.5 semantic-model foundations work.
Status:: Warm-storage review artifact. This document records current interpretation only; it does not change runtime behavior, registries, or active backlog state.

== Executive Summary

Phase 3.5 materially changes the status of several REVIEW-002 concerns. The most important change is that FK-helper primitive semantics are now explicit enough to treat policy-driven helper behavior as stabilized at the primitive boundary. The current implementation no longer relies on convention inference inside `add_fk_helpers` or `remove_fk_helpers`; those primitives consume configured helper policy and provenance.

The broader semantic posture is also clearer. The registries under `registries/` are now best understood as semantic review truth and drift-detection artifacts, not as runtime configuration inputs. The runtime source of truth remains the implementation, while the registries and guards document and check the intended contracts.

Several REVIEW-002 concerns are therefore resolved or reframed rather than still open in their original form. Finding-frame duplication is no longer simply a duplicate-framework problem: `domain/finding_frame.py` provides shared serialization while `domain/validations/findings.py` retains validation-specific finding semantics. XLSX column-width handling is no longer only a carrier-leak concern; it is now a presentation-meta classification question.

The remaining risks are operational and governance-oriented. IR-006 mutable-path concerns remain open, adapter-boundary constants still lack a single ownership point, guard coverage needs inventory against the newer semantic vocabulary, and active backlog lifecycle state has not fully caught up with implemented Phase 3.5 reality.

== Resolved Findings

=== FK-helper primitive semantics

REVIEW-002 concerns about FK-helper primitive inference are resolved at the primitive layer.

Implemented reality::
`src/spreadsheet_handling/domain/transformations/fk_helpers/policy.py`, `enrich.py`, and `drop.py` now make helper primitives consume configured policy/provenance rather than infer helper intent from prefixes or naming conventions.

Registry/documentation state::
`registries/meta_registry.yaml` records `helper_policies` as an active lifecycle bridge with a v2 relation policy as runtime truth and a v1 compatibility shape as residual producer-side compatibility. `docs/semantic_model/12_contract_matrices.adoc` records the FK-helper pilot and the drift that was resolved.

Current interpretation::
The semantic issue is stabilized for primitive consumption. Remaining work is not a new semantic model; it is lifecycle cleanup around stale FTR status, registry/schema guard coverage, and eventual compatibility-bridge retirement when the project chooses to do so.

=== Registry-as-semantic-truth posture

REVIEW-002 ambiguity about where semantic truth lives is substantially resolved.

Implemented/documented reality::
`registries/meta_registry.yaml` and `registries/pipeline_step_registry.json` are manual semantic review artifacts. The pipeline registry explicitly points back to `src/spreadsheet_handling/pipeline/registry.py::REGISTRY` as runtime source of truth.

Current interpretation::
The registries are authoritative for review vocabulary, drift detection, lifecycle notes, and guard alignment. They are not runtime rule engines. This distinction is important because it keeps Phase 3.5 from becoming a hidden runtime semantic framework.

=== Lifecycle and contract semantics

The lifecycle vocabulary introduced during Phase 3.5 resolves part of REVIEW-002's earlier uncertainty about contract maturity.

Documented reality::
`docs/technical_model/ch05_registries/registry_semantics/05_contract_lifecycle.adoc` defines contract lifecycle as an opt-in semantic-review axis and distinguishes it from runtime `_meta` payload lifecycle and backend capability profiles.

Implemented reality::
`helper_policies` now has an explicit active dual-write bridge in `registries/meta_registry.yaml`, and architecture tests cover lifecycle classification for the pilot.

Current interpretation::
Lifecycle semantics are stabilized as a governance vocabulary. They should remain opt-in for contracts where lifecycle state affects maintenance decisions, not a mandatory classification axis for every registry entry.

=== Frame, workbook-view, and meta vocabulary baseline

The glossary and technical model now provide enough vocabulary to resolve several recurring REVIEW-002 ambiguities.

Stabilized terms include:

* `frame` rather than `sheet` for the in-memory data unit;
* `workbook view` for projected workbook layout and sheet mappings;
* `canonical tuple frame`, `matrix frame`, and `crosstable frame`;
* `model_meta`, `presentation_meta`, and `workbook_identity_meta`;
* `carrier`, `carrier-shape`, and `operational-carrier`;
* `canonical_meta` and `derived_meta`, while data itself is better described as original or derived.

These terms are not fully propagated across all backlog and older architecture text, but the target vocabulary is now stable enough for cleanup.

== Partially Resolved or Reframed Findings

=== Finding-frame duplication

Original concern::
REVIEW-002 treated finding-related model duplication as a possible unresolved semantic split.

Current state::
`src/spreadsheet_handling/domain/finding_frame.py` provides shared finding-frame serialization and metadata handling. `src/spreadsheet_handling/domain/validations/findings.py::Finding` remains a validation-domain finding object.

Reassessment::
This is reframed from "duplicate framework" to "two related semantics with a shared interchange frame." The remaining concern is documentation clarity: future backlog cleanup should avoid treating every finding-shaped payload as the same domain concept.

=== XLSX presentation-meta handling

Original concern::
XLSX column-width extraction and promotion looked like a carrier semantic violation.

Current state::
`src/spreadsheet_handling/io_backends/xlsx/openpyxl_parser.py` extracts and merges column-width metadata. The registry classifies `column_widths` as canonical presentation metadata when user-authored workbook presentation is preserved.

Reassessment::
This is now a presentation-meta classification question rather than a simple carrier-pollution finding. The useful remaining question is whether the project wants sharper documentation for when imported presentation facts are canonical presentation meta versus operational carrier facts.

=== Module and family consolidation concerns

Original concern::
REVIEW-002 and adjacent backlog items identified several modules as possible consolidation or split candidates.

Current state::
Phase 3.5 clarified semantic families, but it did not make broad modularization instructions automatically safe. Some older FTRs still predate the semantic vocabulary and later module-specific review outcomes.

Reassessment::
The concern is no longer "split large modules because they are large." It is "re-anchor module and family backlog items to the implemented semantic boundaries and current risk evidence."

== Still-Open Findings

=== IR-006 mutable-path concerns

IR-006 remains open. The review identified mutable metadata paths in rendering/composition code, including paths around workbook projection and layout composition where cloned or shallow-copied metadata can still be mutated.

Representative references:

* `src/spreadsheet_handling/rendering/composer/layout_composer.py`
* `src/spreadsheet_handling/rendering/workbook_projection.py`

Current interpretation::
This is an operational/runtime correctness concern, not a missing semantic vocabulary concern. The vocabulary now helps name the risk, but it does not resolve the mutability behavior.

=== Adapter-boundary constants ownership

Adapter-boundary constants remain under-owned.

Representative references:

* `src/spreadsheet_handling/rendering/composer/layout_composer.py::_WORKBOOK_OPTION_KEYS`
* header flattening and splitting around the literal separator `" / "`
* `src/spreadsheet_handling/rendering/workbook_projection.py`

Current interpretation::
The risk is that workbook projection, parser readback, and presentation metadata semantics can drift because boundary constants are not clearly owned. This should be handled as a small reconciliation/guard problem, not as a broad framework change.

=== Guard inventory and backlog audit

The guard set has improved but still needs inventory against the new vocabulary.

Current state::
There are guards for FK-helper primitive policy consumption and for selected registry lifecycle invariants. There are also transformation package-boundary guards for some families.

Remaining gap::
The review did not establish that all stabilized semantic contracts have matching guards, or that all old backlog assumptions have been re-audited after Phase 3.5.

=== Active backlog lifecycle state

Some active FTRs still describe work as new or pending even where implementation and review have already advanced.

Current interpretation::
This is not a runtime defect. It is a governance risk: future work may follow stale instructions unless the backlog is reconciled.

=== Derived provenance lifecycle

The vocabulary for derived metadata is clearer, but the review did not close all derived-provenance lifecycle questions.

Current interpretation::
The semantic model now distinguishes canonical and derived metadata more cleanly. Operational rules for every derived path, especially across adapters and composed transformations, remain an incremental cleanup area.

== Obsolete Findings

The following original interpretations should no longer be treated as active REVIEW-002 findings in their old form:

* "FK-helper primitives infer helper semantics from naming conventions" is obsolete for primitive consumption after the policy-driven changes.
* "Finding-frame and validation finding objects are simply duplicated competing frameworks" is obsolete; the current issue is documentation and boundary clarity.
* "XLSX column-width metadata is necessarily invalid carrier promotion" is obsolete as an absolute statement; it has been reframed as canonical presentation-meta versus operational-carrier classification.

== Current Interpretation of Remaining Risks

The remaining REVIEW-002 risks are mostly about maintaining coherence while the repository evolves incrementally.

Must-fix before relying on related backlog execution::
Active backlog entries that contradict implemented FK-helper behavior or current module-review outcomes should be reconciled before they are executed.

Should-fix as semantic cleanup::
Older architecture text should be aligned with the glossary where it reverses transformation directions, uses stale compact transform names, or treats `sheet` and `frame` interchangeably in semantic contexts.

Acceptable transitional state::
The registry-as-review-truth posture, opt-in lifecycle vocabulary, and partial guard coverage are acceptable while the project is still using Phase 3.5 as a foundation. They become riskier only if treated as complete runtime governance.
