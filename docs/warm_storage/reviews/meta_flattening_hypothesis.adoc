= Meta Flattening Hypothesis
:toc: left
:toclevels: 3
:sectnums:
:icons: font

== Purpose

This document records the current working hypothesis reached during the
post-domain-cleanup analysis of `_meta`.

It is intended as a handover note for human and LLM reviewers before they enter
the existing Semantic Model, registry, FK-helper, persistence-boundary, or
metadata lifecycle documents.

This is not an ADR and not a final architectural decision. It is a review
orientation artifact.

== Context

The project originally models transformations as mappings from frames to
frames. Spreadsheet export, spreadsheet reimport, JSON directory IO, workbook
projection, and transformation chains all fit this basic mental model.

`_meta` was introduced as a pragmatic sidecar frame / sidecar payload to keep
additional information inside the same frames-to-frames universe.

The original idea was intentionally simple:

## [source,text]

frames -> frames

payload frames:
normal data tables

sidecar:
_meta
-----

This allowed the project to keep the transformation model closed while still
carrying additional information required for roundtrip, rendering, validation,
projection, and reconstruction.

== New Working Hypothesis

The current concern is not that `_meta` exists.

The concern is that `_meta` may have gradually shifted from a sidecar carrier
into a central semantic ontology.

Working hypothesis:

## [source,text]

_meta is a carrier, not a central ontology.
Feature semantics should be owned by feature-local contracts.
Global metadata categories should be introduced only when several features
prove that the shared abstraction reduces complexity.
-----------------------------------------------------

This suggests a direction called *meta flattening*:

## [source,text]

Do not build a better global meta taxonomy first.
Instead, reduce global semantic centralization and describe each feature's
additional data locally.
------------------------

== What Changed In The Assessment

Earlier reviews tended to treat metadata stability and backward compatibility
as high-priority defaults.

That stance is appropriate for a mature product with unknown external users and
uncontrolled historical data.

The current project situation is different:

* the project is still owner-controlled;
* current consumer repositories are few and controlled;
* breaking metadata changes can often be migrated deliberately;
* maintainer understandability is a first-class architecture goal;
* the test suite and consumer roundtrips provide strong safety nets.

Therefore, the default risk balance should not be:

## [source,text]

## keep legacy metadata unless removal is proven safe

but rather:

## [source,text]

simplify legacy metadata unless current code, tests, or controlled consumers
prove that the layer is still needed
------------------------------------

== Why The Existing Meta Model Became Suspicious

Several necessary questions are valid:

* Who produces this metadata?
* Who consumes it?
* Does it survive roundtrip?
* Is it runtime-only?
* Is it user-authored?
* Does it have precedence over regenerated information?
* Should it be deleted after reimport?
* Is it carrier-specific?
* Is it migration residue?

The problem begins when these questions become a global category system.

A dozen lifecycle or ownership questions can create a large combinatorial space
of possible classifications. Naming and guarding all combinations risks making
the metadata model more complex than the features it is supposed to support.

The current working preference is:

## [source,text]

Do not classify every metadata item into a global ontology.
For each feature-local datum, make only the operational decisions that matter:
keep, transport, persist, ignore, validate, clean up, or remove.
----------------------------------------------------------------

== Feature-Local Contracts

A feature-local contract should describe only the additional data owned by that
feature.

For each datum, prefer a small disposition table over a global semantic class:

[cols="1,3"]
|===
| Field | Meaning

| Name
| Local name of the datum.

| Owner feature
| Feature that owns the semantics.

| Producer
| Step or component that writes it.

| Consumer
| Step or component that reads it.

| Persistence
| Whether it may cross a run / roundtrip / storage boundary.

| Cleanup
| Whether it should be removed after reimport or after a materialization phase.

| Authority
| Whether it may act as truth for later runs.

| Status
| Target contract, runtime helper, carrier detail, or legacy residue.
|===

This is intentionally not a global ontology.

== FK Helpers As Pilot

FK Helpers are the proposed first pilot for meta flattening.

The functional feature is small:

## [source,text]

A foreign-key column contains a stable identity.
A helper column shows human-readable values from the referenced target frame.
The helper column is a projection, not canonical payload.
Reimport must not promote helper columns into canonical data.
Cleanup may use feature-local provenance or declared policy.
------------------------------------------------------------

FK Helpers currently carry historical layers:

* v2 relation policy under `_meta.helper_policies.fk.relations`;
* `schema_version: 2`;
* residual v1 per-target policy shape under `_meta.helper_policies.fk.<target>`;
* runtime helper provenance under `_meta.derived.sheets.*.helper_columns`;
* workbook-view or sheet-local helper column declarations;
* `produced_by` markers;
* registry lifecycle language;
* persistence-boundary pruning rules.

The working suspicion is not that all of these are wrong.

The suspicion is that several of them exist because FK Helpers were an early
proof-of-concept feature and survived multiple architecture migrations.

The pilot should determine which layers are still needed by current runtime
behavior and which are historical sediment.

== Important Boundary: Bootstrap Is Not FK Helper

Primary-key annotation and relation discovery are not intrinsically part of FK
Helper materialization.

FK Helpers depend on already-known relation information.

That relation information may come from:

* explicit configuration;
* bootstrap defaults;
* naming-convention inference;
* future schema/key registry features;
* validation features;
* consumer-specific preprocessing.

Those producers may feed relation payload to FK Helpers, but they are not the
FK Helper feature itself.

This distinction matters because otherwise FK Helpers inherit responsibility
for schema discovery, uniqueness validation, dangling-reference validation, and
bootstrap semantics.

== Materialization Mode

FK helper columns can currently be materialized as literal values or as
spreadsheet formulas.

This is a materialization strategy, not a separate feature semantics.

## [source,text]

same FK helper meaning
different spreadsheet carrier
-----------------------------

A literal helper cell and a formula-based helper cell should still be treated as
the same kind of helper projection for cleanup, validation, and reimport
purposes.

== Review Stance For LLM Agents

When reviewing metadata changes in this area, do not treat existing
documentation as proof of an active runtime contract.

Prefer code evidence.

For each metadata path, determine:

* current runtime readers;
* current runtime writers;
* persistence behavior;
* controlled consumer dependency;
* tests that actually rely on it;
* whether it is target contract, runtime helper, carrier detail, or legacy
  residue.

Actively look for evidence that contradicts the meta-flattening hypothesis.

Do not answer only with "risk outweighs benefit" unless the risk is tied to a
current reader, writer, test, consumer, or explicit public contract.

Maintainer understandability is part of the risk model.

== Falsification Questions

The hypothesis should be challenged with concrete code questions:

* What current runtime behavior breaks if the residual v1 FK-helper policy
  shape disappears?
* Which tests rely on v1 as an active consumer contract rather than as legacy
  compatibility?
* Which current consumer repository cannot be migrated?
* Which metadata paths are required by current code rather than by historical
  documentation or registry prose?
* Which guards protect active behavior, and which guards protect terminology?
* Which registry entries describe current runtime truth, and which document
  historical transition state?
* Where does `_meta` act only as a carrier, and where does it currently impose
  cross-feature semantics?

== Proposed Next Work

=== Step 1: FK Helper Metadata Inventory

Create a focused inventory of FK-helper-related metadata paths.

Candidate paths:

## [source,text]

_meta.helper_policies.fk.relations
_meta.helper_policies.fk.schema_version
_meta.helper_policies.fk.<target>
_meta.derived.sheets.*.helper_columns
_meta.sheets.<frame>.helper_columns
produced_by
helper_columns
helper_prefix
-------------

For each path, record:

* producer;
* consumer;
* persistence behavior;
* whether it is target model, runtime helper, carrier detail, or legacy residue;
* whether removal or narrowing is plausible.

=== Step 2: Documentation-Blind Code Review

Run a code-first review that reads the FK-helper implementation, tests,
registry entries, and persistence boundary, but not the broader Semantic Model
prose.

Goal:

## [source,text]

## Find the metadata layers actually required by current runtime behavior.

=== Step 3: Documentation-Aware Contradiction Review

Run a second review that compares the code-first result with existing
documentation, registries, and guards.

Goal:

## [source,text]

Find where existing documentation or registry language pulls reviewers back
toward unnecessary global meta ontology.
----------------------------------------

=== Step 4: First Controlled Cleanup Slice

Likely first candidate:

## [source,text]

Stop writing, gate, or explicitly deprecate the residual v1 FK-helper
per-target policy shape if no current runtime consumer requires it.
-------------------------------------------------------------------

Do this only after the inventory and falsification review.

== Non-Goals

This document does not propose:

* deleting `_meta`;
* replacing the Semantic Model immediately;
* rewriting all metadata documentation;
* removing all registry entries;
* weakening active roundtrip safety;
* ignoring actual runtime dependencies;
* turning FK Helper cleanup into a broad metadata rewrite;
* introducing a new global classification system under another name.

== Working Summary

The likely problem is not `_meta` itself.

The likely problem is the ontologization of `_meta`.

The next direction is not "better meta".

The next direction is:

## [source,text]

less central meta theory
more feature-local contracts
controlled removal of historical metadata sediment
code-first falsification before cleanup
---------------------------------------
