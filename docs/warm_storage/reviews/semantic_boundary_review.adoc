== Executive Summary

The Semantic Model is large, internally consistent, and has clearly absorbed
material that, on a strict reading of the working hypothesis, is
problem-space material rather than contract material. The misplacement is
*localized* (a small number of chapters and section blocks), not pervasive.

The growth mechanism the German problemraum_konzept names -- semantic
terminology generating further terminology -- is observable in the current
model but is not yet pathological. Its main carrier today is not the
contract vocabulary (overlapping sets, lifecycle events, source classes) but
the *explanatory framing* that surrounds the vocabulary: the Problem
Statement, External Inspirations relevance blocks, and Roundtrip
Walkthroughs.

The Functional Model has structural room to host more problem-space
content but currently does not -- it is intentionally narrow (management
summary, usage scenarios, landscape, exploratory use cases) and one chapter
(`ch04`) is explicitly pending an editorial cleanup FTR.
A Functional/Semantic recut is therefore plausible and small, but it is
*not free*: the lifecycle-and-persistence-boundary content resists clean
splitting, and the project-interpretation blocks of External Inspirations
are contract-adjacent rather than pure background.

Recommendation: adopt the working hypothesis as a *navigational* policy
and defer the actual relocation to the FTR the German concept already
proposes (`FTR-FUNCTIONAL-SEMANTIC-MODEL-SPLIT-P5`). Do not redesign now.

== Verdict

*YELLOW.*

The boundary is misplaced in identifiable, bounded places. Continued growth
without a navigational rule is the main risk. No emergency action is
warranted; no documentation rewrite is warranted. A minimal recut, owned
by a single planning FTR, is sufficient.

== Findings

=== F1. Problem Statement is almost entirely problem-space material

`03_problem_statement.adoc` runs roughly 165 lines and contains, in order:
project structural-discipline summary, FK-helper semantic-friction
examples, a three-stage early/growing/legacy project taxonomy, the
identity-keyed-mode anti-pattern, and the canonical roundtrip promise.
None of this binds runtime behavior; almost all of it motivates the rest of
the Semantic Model.

Evidence: every example in the chapter is illustrative (FK-helper review
findings, customer-X branches), and the chapter's own headings are
narrative ("Architecture reduced structural friction, but semantic
friction remains"; "Human teams, AI agents, and semantic ambiguity";
"Why this matters more here than in ordinary CRUD systems").

If the working hypothesis is taken literally, this chapter sits in the
Functional Model.

Risk if untouched: the chapter remains *useful*, but it sets a precedent
that the Semantic Model is the natural home for any new "why this matters"
content. That precedent is what the German concept warns about.

=== F2. Roundtrip Walkthroughs is process-flow material, not contract

`10_roundtrip_walkthroughs.adoc` is three flow diagrams (forward, reimport,
repeated) and three question lists. The chapter has an explicit note that
it "should eventually contain PlantUML sequence diagrams". This is a
description of *what happens*, not a description of *what is allowed*.

Under the working hypothesis, walkthroughs belong in the Functional Model
(operating model, scenarios) and are referenced -- not duplicated -- by
the contract chapters that constrain behavior at process boundaries.

=== F3. External Inspirations is structurally bi-modal

`05_external_inspirations.adoc` consistently pairs a "relevant ideas"
block (inspirational, explicitly non-normative) with a "project
interpretation" block (the project's adopted reading of the idea). The
two blocks have *different* roles:

* "relevant ideas" blocks fit the Functional Model's
  data-transformation-landscape framing.
* "project interpretation" blocks are partial contract material -- they
  state what the project does and does not adopt.

The current chapter conflates both. Under the hypothesis, the "relevant
ideas" half is explanatory; the "project interpretation" sentences should
either become inline contract statements in Ambiguity Policy / Lifecycle
/ Versioning, or be referenced as cross-links.

=== F4. Lifecycle persistence boundary is implementation-leaning

`08_lifecycle_and_update_semantics.adoc` § "Persistence boundary" reads
more like an architecture-decision narrative than a contract: it explains
which step deletes which key under which conditions, names a concrete
module (`pipeline/persistence_boundary.py`), enumerates current rules and
known deferrals, and describes the operational scope (post-pipeline,
write-side only).

This is necessary content; the question is *whose* content it is. Under
the working hypothesis it belongs in the Technical Model (with an entry
in the registry's `contract_lifecycle` blocks summarizing the contract
side). The Semantic Model can state the *rule* ("transient runtime keys
must not be persisted"); the *mechanism* belongs to the technical model.

The remainder of chapter 08 (lifecycle events, write-time conflict
resolution, precedence-as-update, repeated-roundtrip risk) is contract.

=== F5. The semantic-growth mechanism is real but currently bounded

The German concept identifies a recursive pattern:

[source,text]
----
contradiction -> new semantic term -> new edge case -> further terms
----

Direct evidence in the current Semantic Model:

* Chapter 07 names ~17 candidate sets (Canonical, Derived, Persistent,
  Transient, RoundtripRelevant, RenderConsumed, RuntimeConsumed,
  UserEditable, ProducerOwned, Configurable, HeuristicallyInferred,
  Policy, Provenance, PublicSurface, InternalOnly, Deprecated,
  TransitionBridge).
* Chapter 17 Open Questions explicitly asks whether *additional* sets
  (`AuditOnly`, `DiagnosticOnly`) are needed -- the recursive step.
* Chapter 08 introduces secondary vocabulary: historical residue,
  semantic drift, lifecycle events, persistence boundary, semantic
  precedence vs age/generation, candidate carrier fields
  (`produced_by.step`, `generation`, `valid_for_frame_schema_hash`,
  `lifecycle_stage`, `supersedes`, `stale_when`, `prune_on`).
* Chapter 11 adds source classes (a parallel taxonomy alongside the
  semantic sets).

The mechanism is therefore not theoretical. *However*, the present
vocabulary is still descriptive rather than enforced, and only one
family (FK helpers) has actually been wired through. The growth is in
the "rule-ready" phase, which the project has consciously chosen.

So: the mechanism the German concept names is real; the current state
is *near* but not *past* the point where the vocabulary becomes
self-generating. The Adoption Strategy chapter (15) is the project's
explicit brake on this -- it requires operational symptoms before a
family produces new contract material.

The boundary problem is therefore better described as *containment*:
keep the explanatory pressure that drives new terminology in the
Functional Model so that new semantic vocabulary has to defend itself
against a stated functional need rather than against another piece of
semantic vocabulary.

=== F6. The Functional Model is structurally narrow

The Functional Model today is:

* ch01: Management Summary -- three short sub-sections (Goals, Non Goals,
  Relation to other documents). Total content under ~20 lines.
* ch02: Usage Scenarios -- operating model, primary/secondary scenarios,
  non-goals. ~80 lines total.
* ch03: Data Transformation Landscape -- positioning vs adjacent tools.
  ~80 lines total.
* ch04: Use Case Concepts -- exploratory, explicitly pending
  `FTR-CONCEPT-DOCS-CLEANUP-P5`; "No semantic review performed during
  the move".

Mapping against the review's question list:

=== F7. The working hypothesis's binary cut has a blind spot

The hypothesis splits content into "problem space" and "contract /
interpretation / guarantees". The current Semantic Model contains a
third category that the hypothesis does not address:

* *Adoption / process* material (chapter 15 Adoption Strategy, the
  pilot-selection criteria, the slice questions, the rule-ready vs
  rule-engine distinction). This is operationally important and is
  neither problem-space nor contract.

Forcing chapter 15 into either side would distort it:

* Pushing it into the Functional Model conflates "what we're solving"
  with "how we adopt the solution shape".
* Keeping it in the Semantic Model alongside contract chapters
  silently expands what "contract" means.

Practical effect: any recut should explicitly carve out an
"Adoption / process" zone, not pretend the binary cut is complete.

== Risks

|===
| ID | Risk | Mitigation

| R1
| If the recut is treated as a relocation exercise, the FK-helper
  concrete examples (currently load-bearing for several contract
  chapters via *evidence*) become orphaned references in the Semantic
  Model and lose context-of-use in the Functional Model.
| Reference instead of move. Keep the FK-helper examples where they
  illustrate a contract rule, and have the Functional Model link out to
  them as worked examples of the problem space.

| R2
| The persistence boundary content (chapter 08) is naturally
  implementation-leaning. Splitting it cleanly between Semantic and
  Technical models risks fragmenting a single coherent design
  description.
| Leave the persistence boundary section *physically* where it is until
  the Technical Model has a paired chapter to receive it. Add a one-line
  navigation note that the deeper mechanism description belongs there
  long-term.

| R3
| Cross-references multiply. Stale links between Functional and Semantic
  Model become a new drift surface.
| Use AsciiDoc xrefs (not raw paths) so doc builds catch broken links;
  rely on existing Make build to fail on link errors.

| R4
| Converting the German problemraum_konzept into English Functional
  Model prose may erase the *language distance* it was written for.
  The concept explicitly states that distance is the point.
| The proposed FTR already says the German source stays in warm
  storage. Preserve it verbatim. The English Functional Model absorbs
  *the framing*, not the prose.

| R5
| Chapter 15 (Adoption Strategy) gets misclassified into either side and
  loses its role as a brake on recursive semantic growth.
| Carve out a third zone explicitly ("Adoption / planning") in the
  Semantic Model entry navigation. Do not move chapter 15.

| R6
| The hypothesis is treated as project policy before it is tested.
  Premature policy will produce its own churn.
| Adopt as *navigational* policy only. Defer the relocation to the
  proposed FTR. Re-evaluate after that FTR ships.

| R7
| The semantic growth mechanism continues to operate while a recut is
  pending. Chapter 17 already asks whether *more* sets (AuditOnly,
  DiagnosticOnly) are needed.
| Add a navigation note that new terms require a stated functional need
  before they enter chapter 07. Until the recut, the Adoption Strategy
  pilot-selection criteria already function as this brake; rely on
  them rather than inventing a new one.

== Minimal Viable Recut

The recut described here is the *smallest* boundary clarification consistent
with the working hypothesis. It does not move files, does not redesign
chapters, and does not modify content.

== Recommended Next Step

. *Do not* rewrite documentation now. The hypothesis is plausible, the
  misplacement is bounded, and a rewrite under an unaccepted hypothesis
  would itself be drift.

. *Do* file (or accept) the FTR the German concept already proposes --
  `FTR-FUNCTIONAL-SEMANTIC-MODEL-SPLIT-P5` -- and use it as the single
  owner of the recut. The proposed scope, non-goals, acceptance, and
  validation in `docs/cold_storage/concepts/problemraum_konzept/06_restrukturierung_und_ftr_vorschlag.adoc`
  are already aligned with this review's minimal-recut classification.

. *Inside that FTR*, treat the four boundary cases as the load-bearing
  decisions:

  .. Problem Statement (chapter 03) -- relocate or reference into the
     Functional Model.
  .. Roundtrip Walkthroughs (chapter 10) -- relocate or reference into
     the Functional Model.
  .. External Inspirations (chapter 05) -- split "relevant ideas" from
     "project interpretation".
  .. Persistence boundary block in chapter 08 -- decide whether the
     mechanism description is contract or technical; only the rule
     remains in the Semantic Model either way.

. *Preserve* the German problemraum_konzept verbatim in cold storage.
  Its language distance is itself a project asset.

. *Defer* any vocabulary expansion under chapter 07 (the candidate
  additional sets `AuditOnly`, `DiagnosticOnly` mentioned in chapter 17)
  until the FTR above ships. Rationale: new vocabulary added during a
  pending boundary recut would have to be re-classified anyway.

. *Re-evaluate* this review after the FTR lands. The verdict should
  move to GREEN if (a) chapters 03, 05-relevant-ideas, and 10 are no
  longer in the Semantic Model entry path, and (b) the Functional Model
  has a stated ontology / information-flow chapter that absorbs the
  problemraum_konzept's framing.

== Follow-up: implementation under FTR-FUNCTIONAL-SEMANTIC-MODEL-SPLIT-P5

The FTR derived from this review
(`docs/backlog/FTR-FUNCTIONAL-SEMANTIC-MODEL-SPLIT-P5.adoc`) has been
implemented as a single editorial pass. The implementation:

* Added a Semantic Model entry-page classification grouping every
  current chapter as Contract Core, Explanatory Context, Adoption /
  Planning, or Appendix.
* Relocated the chapter 03 Problem Statement and chapter 10 Roundtrip
  Walkthroughs content into the Functional Model (under the new
  `ch06_preserved_problem_space_context/`). The Semantic Model retains
  navigation stubs at the original locations with their original
  anchors so that incoming references resolve.
* Added a lightweight problem-space vocabulary / information-flow
  overview to the Functional Model (under the new
  `ch05_problem_space_overview/`) that absorbs the framing of the
  German problemraum_konzept (chapters 02 and 03) into English. The
  German source is preserved verbatim in cold storage.
* Marked chapter 05 (External Inspirations) with an internal-split
  note distinguishing the "relevant ideas" half (explanatory) from the
  "project interpretation" half (contract-adjacent). No content is
  moved in this FTR.
* Annotated chapter 08's persistence-boundary subsection with a
  Technical Model navigation note. Mechanism content stays in place
  pending a follow-up FTR.
* Added an archival note to chapter 16 pointing at the cold-storage
  archive for the seven planned FTRs that have shipped.

Verdict-movement check against this review:

* Precondition (a) -- chapter 03, chapter 10, and the "relevant ideas"
  halves of chapter 05 are no longer in the Semantic Model entry path
  as Contract Core. Chapter 03 and chapter 10 are stubs that classify
  themselves as Explanatory Context and point to the Functional Model.
  Chapter 05's "relevant ideas" halves are marked as Explanatory and
  point to the Functional Model overview. *Met.*
* Precondition (b) -- the Functional Model has a stated problem-space
  vocabulary / information-flow overview, with a visible *Scope limit*
  note, that absorbs the problemraum_konzept's framing without becoming
  an ontology. *Met.*

The review's residual hold on the lifecycle persistence-boundary
relocation (Risk R2 / Finding F4) is preserved unchanged: the mechanism
description stays physically in chapter 08, flagged for a follow-up
Technical Model FTR. The verdict for that residual remains YELLOW.

Overall verdict after implementation: *GREEN* on the boundary recut
itself; *YELLOW (carried)* on the persistence-boundary mechanism
relocation, which is deferred by design.
