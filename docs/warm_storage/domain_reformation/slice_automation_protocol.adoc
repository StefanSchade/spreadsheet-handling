= Domain Reformation Slice Automation Protocol
:revdate: 2026-06-24

== Purpose

This note records the intended lightweight automation shape for repeated
domain/meta reformation slices.

It is a warm-storage working note, not an ADR, backlog item, or binding project
policy. It exists to reduce repeated prompt noise while keeping the actual
contract decisions explicit.

== Motivation

The contract-first domain/meta reset will likely repeat the same slice shape
many times:

. clarify functional intent;
. formulate a small `Frames + Meta` bridge contract;
. add contract or integration tests;
. implement a narrow vNext path;
. delegate from or delete old implementation sediment;
. record only the decisions that need durable memory.

Running that flow through many one-off prompts would create process noise. The
repeatable parts should be scaffolded and checked, while the functional and
semantic decisions remain reviewed by a human.

== Automation Boundary

Automate:

* slice scaffolding;
* file and heading conventions;
* reminder checklists;
* references to Functional Model sources;
* test-command capture;
* simple consistency checks before a slice is considered ready.

Do not automate:

* the functional contract decision itself;
* semantic vocabulary promotion;
* Project Memory changes;
* deletion of old paths without contract-test coverage;
* broad backlog generation.

== Slice Template Shape

Each slice should have one compact working document with these sections:

* *Intent* -- the user-facing requirement and why the slice exists.
* *Functional Model Source* -- source chapters or inquiry documents.
* *Bridge DSL Contract* -- what `Frames + Meta` information enters and exits.
* *Lifecycle Semantics* -- which meta is durable, transient, consumed, deleted,
  or recreated.
* *Contract Tests* -- tests that express the requirement rather than the old
  implementation.
* *Implementation Shape* -- the smallest vNext path that satisfies the tests.
* *Old-Path Handling* -- delegate, delete, or leave untouched with rationale.
* *Acceptance* -- observable completion conditions.
* *Memory / Backlog Impact* -- explicit statement whether PM, ADR, or backlog
  changes are needed.

== Proposed Command Shape

The repository-local helper lives under `scripts/` because it is specific to
the core repository's warm-storage and reformation workflow. Per
`docs/ai_info/conventions.adoc`, `tools/` is reserved for generic helpers that
can plausibly be copied to sibling repositories.

Helper:

[source,bash]
----
make reformation-slice NAME=fk-helper-unresolved-values
----

Expected behavior:

* create a working folder under warm storage;
* create a slice document from the template;
* create or suggest a contract-test file location;
* print the expected targeted test command.

Check:

[source,bash]
----
make reformation-check SLICE=fk-helper-unresolved-values
----

Expected checks:

* slice document exists;
* Functional Model or inquiry source is named;
* bridge contract section is present;
* contract-test reference is present;
* non-goals or old-path handling are explicit;
* Project Memory/backlog impact is stated;
* verification command is recorded.

== First Candidate Slice

The likely first slice is `fk-helper-unresolved-values`.

Reason:

* the release keeps one accepted xfail sentinel;
* the open issue is a contract gap, not a mechanical implementation failure;
* it is tied to FK helper / helper-column sediment already identified during
  Functional Model promotion;
* it is small enough to prove the reformation workflow before broader domain
  and meta replacement work.

The current sentinel is:

* `tests/roundtrip/test_pathological_cell_contracts.py`
* `test_missing_fk_key_helper_representation_without_cleanup`
* `BUG-UNRESOLVED-HELPER-VALUE-NAN-SEMANTICS-P4A`

The first slice should decide and test the unresolved-helper-value contract
before changing the implementation.

== Operating Principle

Use one prompt per slice where possible:

[source,text]
----
Start reformation slice fk-helper-unresolved-values using the reformation
protocol.
----

The agent should then scaffold, inspect sources, propose the contract decision
points, implement the narrow path only after the contract is clear, run targeted
checks, and report any required durable documentation updates.
