= Reformation Slice: Cell Codec Frames Meta Contract
:revdate: 2026-06
:slice-name: cell-codec-frames-meta-bridge

== Status

Draft. Human-approved decisions consolidated. No implementation yet.

== Purpose

Cell Codec solves a frame-shape problem: selected columns are contracted into
one compact column, and that compact column is expanded back into the same
selected columns under explicit codec intent. Compact values are not
self-describing and must not carry business meaning.

== Maintainer Decisions

* Cell Codec is domain-neutral; example values are example data only.
* Cell Codec maps selected expanded columns into one compact column and
  reverses that mapping.
* Non-participating columns pass through unchanged.
* Position-based encoding is sufficient for the MVP.
* The MVP contract includes ordered participating columns, compact column name,
  separator or encoding rule, absent-value representation, and error behavior.
* Decoding requires matching codec intent, supplied by persisted meta, pipeline
  configuration, or explicit bootstrap.
* Without matching codec intent, compact values are plain text.
* Decode failures are hard errors by default.
* Helper or derived columns must not participate by default; default behavior is
  a hard error unless a later explicit contract allows it.
* Validation lists, helper ranges, codebooks, and workbook tables matter only
  through the role assigned by the pipeline.
* Resting-data backward compatibility is not a default requirement. Current
  repo fixtures, tests, demos, and documented workflows must stay consistent if
  affected.
* XRef handles tuple or matrix addressing; Cell Codec handles compact value
  encoding.

== Current-State Observations

The current implementation is already domain-neutral. It provides scalar
parse/serialize helpers plus frame-level `decode_cell_values` /
`encode_cell_values`. Existing tests cover current code/token mechanics,
allowed values, empty-cell behavior, and a small XRef composition path.

The registry currently describes `cell_codecs` with historical vocabulary such
as `meta_canonical` / `canonical_meta`. That is current-state evidence only,
not proposed contract wording.

Open implementation question: can current code/token mechanics express the
Functional Model case cleanly with explicit configuration and no heuristic
parsing? The next phase should prove that with reference tests before adding
structure.

== Position-Based MVP Contract

* one logical frame shape is transformed;
* passthrough columns are copied unchanged;
* ordered participating columns are contracted into one compact column;
* reverse expansion restores those configured participating columns;
* codec intent names participating columns, compact column, separator or
  encoding rule, absent-value representation, and error behavior;
* codec intent may come from persisted meta, pipeline configuration, or
  explicit bootstrap;
* missing codec intent means no decode;
* wrong token count, invalid absent-value use, helper/derived-column
  participation, or disallowed value is a hard decode failure by default;
* no delimiter, column-name, or neighboring-workbook-content heuristic may
  create codec intent.

Disjoint value sets and category-hinted values are later scope.

== Boundary Rules

Codec intent must be available at decode time through meta, configuration, or
bootstrap. Generated spreadsheet support is not codec intent unless the
pipeline assigns that role explicitly.

Crossing a boundary without carrying codec intent leaves only plain compact
text. Reattaching codec intent after a boundary is explicit work, not inference.

Old resting `_meta`, JSON, YAML, or workbook artifacts do not impose a
compatibility tax. Avoid avoidable crashes where reasonable, but do not add
compatibility shims, API versioning, or migration layers merely to preserve old
resting shapes.

== Composition Boundary

XRef crosstable and Cell Codec remain separate primitives.

XRef owns tuple and matrix addressing. It expands or contracts the position of
a value in a matrix-shaped workbook view. It does not interpret the internal
meaning of a compact cell value.

Cell Codec owns compact value encoding for configured semantic slots. It may
contract several ordered slot columns into one compact cell value and expand the
compact value back into the same ordered slot columns. The participating columns
are positional semantic slots, not repeated values of the same type.

Compact Multiaxis is the convenience composition layer. It may orchestrate XRef,
Cell Codec, and related support such as legends, but it must not make XRef or
Cell Codec depend on each other internally.

For the MVP, each participating slot is atomic. Cell Codec does not support
lists inside a slot, cartesian expansion, or association semantics between
multiple values inside different slots. If a use case needs a combined value
such as `SaaS + OnPrem`, that combined string must be modeled as one configured
allowed value of the slot.

A later feature may allow repeated values of the same type inside a compact
cell, for example token-list disassembly and reassembly. That is a different
capability from the MVP slot contract and is explicitly out of scope here.


[source,text]
----
XRef crosstable
  = Matrixposition <-> Tupelposition

Cell Codec
  = positionsgebundene Slot-Spalten <-> kompakter Zellwert

Compact Multiaxis
  = Convenience-Komposition aus XRef + optional Cell Codec + ggf. Legend
----


== Contract Tests

The next implementation phase should add tests before code changes.

Unit tests:

* encode ordered columns into a compact column;
* decode a compact column into ordered columns;
* separator handling;
* absent-value representation;
* wrong token count;
* hard decode failure;
* helper/derived column rejection.

Frame-level integration tests:

* expanded frame shape -> compact frame shape -> expanded frame shape;
* passthrough columns remain unchanged;
* no heuristic delimiter guessing without codec intent.

Roundtrip / pipeline-level reference test:

* representative Functional Model case;
* codec intent via meta, configuration, or bootstrap;
* intended behavior, not just current implementation mechanics.

XRef + Cell Codec integration test:

* XRef handles tuple or matrix addressing;
* Cell Codec handles compact value encoding;
* composition works without merging responsibilities.

== Implementation Shape

No implementation is approved yet.

Next phase order:

. write reference tests;
. check whether current code/token mechanics satisfy them with explicit
  configuration;
. add structure only if the tests show the current mechanics cannot express the
  approved contract clearly.

No public YAML/CLI shape change is approved by this slice.

== Old-Path Handling

No old-path deletion, delegation, compatibility shim, or vNext wrapper is
approved by this slice.

== Acceptance

This planning slice is acceptable when it states the frame-shape contract,
records the maintainer decisions, names required test layers, and leaves code,
tests, runtime behavior, Project Memory, ADRs, backlog, semantic vocabulary, and
old paths unchanged.

== Out of Scope

* disjoint value sets;
* category-hinted values;
* new semantic vocabulary;
* Project Memory / ADR / backlog changes;
* API versioning or compatibility shims;
* resting-data migration;
* public YAML/CLI changes unless explicitly approved;
* old-path deletion;
* runtime behavior changes in this pass.
* lists or repeated values inside one participating slot;
* cartesian products between slot values;
* association semantics for multi-valued slot contents;
* token-list disassembly as a separate feature;
* making XRef depend on Cell Codec or Cell Codec depend on XRef;
* moving XRef/Cell-Codec composition out of Compact Multiaxis into the primitives.
