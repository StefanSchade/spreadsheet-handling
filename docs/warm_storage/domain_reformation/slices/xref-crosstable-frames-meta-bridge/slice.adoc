= Reformation Slice: XRef Crosstable Frames Meta Bridge
:revdate: 2026-06
:slice-name: xref-crosstable-frames-meta-bridge

== Status

Draft. Human-approved functional decisions recorded. No implementation or test
rewrite yet.

== Intent

Define the narrow XRef contract as a reversible transformation pair:

* forward transformation: relation-shaped rows -> xref crosstable matrix
  projection;
* return transformation: edited matrix projection -> relation-shaped rows.

XRef solves tuple/matrix addressing for editable crosstables. It does not
interpret relation values and does not own Cell Codec, Legend Block, validation,
or Compact Multiaxis semantics.

== Bridge DSL Contract

The MVP contract is operational:

* Relation-shaped form has row-axis key column(s), column-axis key column(s),
  and one relation value string whose internal meaning is not interpreted by
  XRef.
* Matrix-shaped form has row-axis key column(s), one editable matrix cell per
  configured row/column address, and relation values as cell content.
* Forward projection maps each non-empty relation tuple to exactly one matrix
  cell address.
* Return recomposition maps each present non-empty matrix cell to exactly one
  relation row.
* Empty matrix cells mean no relation and do not produce relation rows.
* Duplicate tuple coordinates for the same row-axis and column-axis address are
  hard errors.
* Row-axis and column-axis member domains are represented independently from
  currently filled relation cells, so an all-empty matrix can still be rendered.
* Axis identity is explicit: row-axis key column(s) and column-axis key
  column(s) are configured and must identify matrix addresses unambiguously.
* Axis members may be supplied explicitly or dynamically from configured source
  frames / columns, including an "all current values from this source" style.
  The axis-domain source must be explicit; the member list itself does not have
  to be manually enumerated.
* No fuzzy matching, implicit label parsing, silent repair, aggregation, value
  parsing, boolean interpretation, delimiter logic, or grouped/MultiIndex
  extension belongs to this MVP.

XRef owns addressing only:

* Addressing includes resolving which configured axis member maps to which
  matrix row or column.
* Addressing does not include interpreting the relation value stored at that
  address.
* XRef does not parse, type, split, validate, or constrain relation strings.
* Cell Codec may later process relation strings as a separate step.
* Legend Blocks or validation may later explain or constrain relation strings
  as separate concerns.
* Compact Multiaxis is the convenience composition layer for XRef plus optional
  neighbouring transformations.

== Lifecycle Semantics

Projection intent must identify the relation frame, matrix frame, row-axis key
column(s), column-axis key column(s), value column, and axis-member sources or
lists needed to render the matrix and recompose it later.

Runtime resolution, such as resolved axis members and materialized column
positions, is support data for the current projection/readback. The slice does
not approve a new `_meta.xref_crosstable` schema. Any public step parameter,
registered meta shape, persistence-boundary, or carrier behavior change remains
a human gate.

Current registry language classifies `_meta.xref_crosstable` as
`meta_canonical` / `canonical_meta`; this is current-state vocabulary only, not
new contract wording for the slice.

== Current-State Classification

Target XRef contract evidence:

* `src/spreadsheet_handling/domain/transformations/xref_crosstable/operation.py`
  already avoids interpreting cell values and implements matrix/relation
  conversion.
* `tests/unit/domain/transformations/test_xref_crosstable.py` covers basic
  matrix -> relation, relation -> matrix, duplicate coordinate rejection, flat
  axis constraints, dense axis domains, and scoped recomposition.
* `tests/unit/pipeline/test_step_registration.py::test_xref_crosstable_steps_are_config_addressable`
  proves the registered steps can execute the basic pair.

Composition behavior belonging outside XRef:

* `tests/unit/domain/transformations/test_compact_multiaxis.py` covers
  orchestration of XRef with value interpretation, legend/code grouping, cell
  codec behavior, and lifecycle side effects.
* `tests/integration/pipeline/test_sparse_crosstable_pipeline.py` covers XRef
  composed with sparse defaults.
* `tests/integration/pipeline/test_localized_resource_matrix_roundtrip.py`
  covers discriminator split/merge, helpers, workbook IO, and XRef together.

Value interpretation belonging to Cell Codec / Legend / Validation:

* allowed code lists, legend-sourced groups, token splitting, whole-cell code
  semantics, and validation of relation strings are not XRef contract tests.

Historical sediment or retarget candidates:

* `drop_empty=False` default behavior currently preserves blank rows in some
  direct `expand_xref` calls. The target contract says empty matrix cells mean
  no relation; whether migration needs a temporary legacy switch or explicit
  configuration is a gate.
* tests that only characterize historical blank-cell defaults should be
  rewritten or removed during the implementation pass after the blank-cell
  surface decision.
* stored `column_keys` and `dense_axes.resolved` are current reconstruction
  support; their durable lifecycle must not be inferred from current metadata.

Deferred extensions:

* grouped columns / MultiIndex axes;
* richer axis mappings with separate identity, address, display, order, and
  grouping fields;
* FK-helper or axis-helper lifecycle;
* value parsing, Cell Codec, Legend Block, validation, Sparse Defaults, Compact
  Multiaxis, review/diff/promotion workflow, and workbook protection UX.

== Implementation Shape

Do not introduce a parallel vNext path. After human approval of the remaining
gates, update the existing XRef primitive path narrowly so the contract tests
express the Functional Model behavior.

Prefer direct frame-level tests before implementation. Preserve old paths until
contract-test coverage and explicit human approval justify rewrites or deletion.
Do not change XRef to call Cell Codec, Legend Blocks, validation, Sparse
Defaults, or Compact Multiaxis.

== Old-Path Handling

No old path is deleted in this slice. Existing implementation remains
current-state evidence.

The next pass may rewrite tests that preserve historical blank-cell or
metadata-shape behavior only after the relevant human gate is approved.
Composition tests stay with their owning layers unless they are needed to prove
that XRef continues to compose without absorbing their responsibilities.

== Remaining Human Decisions

Implementation-blocking gates:

* Blank-cell migration surface: decide whether a temporary legacy switch or
  explicit config is needed to protect current public behavior while moving to
  the approved target behavior that empty matrix cells mean no relation.
* Axis-domain carrier: approve which existing or new shape carries the explicit
  axis-domain source cleanly, whether current fields are sufficient, and
  whether a narrow public/meta shape change is needed before implementation.
* Composite column-axis scope: confirm the MVP stays with one flat column-axis
  key column and flat matrix headers; grouped/MultiIndex/composite column-axis
  keys remain deferred.
* Public surface: approve any required registered step parameter, YAML, carrier,
  or `_meta.xref_crosstable` persistence-boundary change before implementation.
* Old-path handling: approve any removal or rewrite of historical tests/paths
  that preserve behavior outside the target XRef contract.
