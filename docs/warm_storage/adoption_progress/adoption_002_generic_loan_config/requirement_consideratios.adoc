= Adoption 002 – Generic Loan Configuration
:toc:
:toclevels: 3

== Purpose

This document captures requirements considerations for a generic loan
configuration use case.

The objective is not to reproduce an existing spreadsheet solution but to
explore whether the spreadsheet-handling framework can support a class of
business configuration problems characterized by:

* large variable catalogs,
* product-dependent classifications,
* multiple maintenance views,
* controlled reimport workflow.

The use case is intentionally generic and avoids references to specific
organizations, products, or implementations.

== Status

* Early adoption exploration
* Not an implementation plan
* Not a committed roadmap
* Input for future consumer validation

== Central Observation

The primary business artifact is not a spreadsheet.

The primary artifact is a canonical configuration model.

Spreadsheets act as editable projections over that model.

The central pattern can be summarized as:

[source,text]
----

Canonical Model
↓
Classification Matrix
↓
Follow-Up Projections and Maintenance Views
↓
Controlled Reimport
↓
Canonical Model
---------------
----

== Core Modeling Pattern

The central relation is:

[source,text]
----

Variable × Loan Product
↓
Classification
--------------
----

The classification does not represent the final business data.

Instead, it determines:

* applicability,
* editability,
* required follow-up information,
* visibility in maintenance views.

== Canonical Entities

=== Variables

Variables represent configurable business concepts.

Examples:

* interest rate
* repayment amount
* maturity date
* collateral value
* borrower income

Typical attributes:

* variable_id
* technical_name
* business_label
* description
* data_type
* unit
* display_group
* active

=== Loan Products

Products represent configurable product types.

Examples:

* fixed-rate loan
* floating-rate loan
* annuity loan
* bullet loan

Typical attributes:

* product_id
* name
* description
* active

== Classification Matrix

The primary maintenance surface is a matrix.

[source,text]
----
                fixed_rate     floating_rate     annuity

interest_rate       A+C           A                 A+C
margin              -             A                 -
target_rate         T             -                 T
term                A             A                 A
-----------------------------------------------------
----

Rows represent variables.

Columns represent products.

Cells contain classification codes.

The matrix itself does not contain all business attributes.

The matrix classifies the role of a variable within a product.

== Role Codes

A classification cell may contain one or more role codes.

Examples:

[source,text]
----

A
C
T
A+C
A+C+E
-----
----

Possible meanings:

[cols="1,3"]
|===
| Code | Meaning

| A
| Input value

| O
| Optional input value

| T
| Target value

| C
| Calculated value

| E
| Additional role marker

| -
| Not applicable
|===

The exact semantics remain domain-specific.

The framework should not hardcode meaning into the code values.

== Role Sets

A cell may contain a set of role codes rather than a single code.

Examples:

[source,text]
----

A+C
A+C+E
-----
----

The framework should treat role sets as data.

Interpretation belongs to configuration and projection logic.

The ability to represent multiple codes within a single cell already aligns
with existing compact-cell and legend-based concepts.

== Allowed Combinations

Not every role combination is necessarily meaningful.

Examples:

[source,text]
----

A+C        valid
A+C+E      valid
B+C+E      invalid
------------------
----

The framework should distinguish:

* possible role tokens,
* allowed combinations,
* forbidden combinations.

Allowed values may be derived from:

* explicit configuration,
* legend definitions,
* future combination-generation rules.

Validation should reject impossible combinations.

== Legends

Legends act as explicit code books.

They provide:

* human-readable meaning,
* allowed value definitions,
* validation source,
* workbook documentation.

Legends should remain domain-neutral.

They describe allowed values, not business behavior.

== Follow-Up Projections and Maintenance Views

The most important aspect of this use case is not the matrix itself.

The most important aspect is the set of maintenance views derived from the
classification matrix.

Example:

[source,text]
----

Variable X
Product Y
Role Set = A+C
--------------
----

This may imply:

[source,text]
----

Input Attributes View
→ relevant

Calculated Attributes View
→ relevant

Target Attributes View
→ not relevant
--------------
----

The classification determines which maintenance views become active.

== Example Maintenance Views

=== Input Attributes

Possible fields:

* default value
* validation range
* mandatory flag
* display group
* help text

=== Target Attributes

Possible fields:

* solver settings
* precision
* convergence limits
* target validation

=== Calculated Attributes

Possible fields:

* display precision
* explanation text
* dependency notes

== Product-Specific Applicability

A variable may be relevant for one product and irrelevant for another.

This is expected behavior.

Example:

[source,text]
----

Variable X
Product A → role set A+C
Product B → not applicable
Product C → role set A
----------------------
----

The framework should support this naturally.

== Visibility Versus Applicability

Not applicable does not necessarily mean hidden.

In many situations it is useful to show the full matrix structure.

Therefore:

* visible but editable
* visible but read-only
* visible but not applicable

should be treated as distinct states.

== Workbook UX

The workbook should visually distinguish:

* editable cells,
* non-editable cells,
* not-applicable cells,
* helper columns,
* key columns,
* derived values.

Color coding is expected to be important.

The user should immediately recognize:

[source,text]
----

What can I edit?
What should I not edit?
Why is this cell present?
-------------------------
----

== Follow-Up Projection and Maintenance View Concept

A key architectural requirement is:

[source,text]
----

Classification
↓
Projection Rules
↓
Maintenance Views
-----------------
----

The matrix determines which records appear in downstream maintenance views.

This is distinct from:

* facet splitting,
* matrix stacking,
* compact cell encoding.

The primary requirement is conditional projection driven by classification
membership.

== Validation

Projection logic and business plausibility logic should remain separate.

Projection logic answers:

[source,text]
----

What should appear?
What should be editable?
------------------------
----

Validation answers:

[source,text]
----

Is this combination allowed?
----

Examples:

* forbidden role combinations,
* missing required attributes,
* contradictory classifications.

Validation should be expressed through explicit rules.

== Reimport Safety

Business users should not edit canonical configuration directly.

Recommended workflow:

[source,text]
----

Canonical Data
↓
Workbook Export
↓
User Editing
↓
Review Area
↓
Validation
↓
Promotion
↓
Canonical Data
--------------
----

This reduces accidental corruption of the source model.

== Candidate Framework Capabilities Exercised

This adoption case exercises:

* matrix projections
* compact cell encoding
* legends
* validation
* lookup helpers
* workbook styling
* dynamic frame projection
* dynamic view column targeting
* conditional maintenance views
* controlled reimport

== Open Questions

=== Classification Vocabulary

Which role codes exist?

Which combinations are meaningful?

Which combinations are forbidden?

=== Projection Rules

How should role membership activate maintenance views?

Can projections be expressed declaratively?

Can dynamically generated view columns be addressed declaratively by later
rules?

How much can be derived automatically?

=== Workbook UX

How should editable and non-applicable cells be styled?

Should role meanings be visible directly in the workbook?

=== Configuration Ergonomics

Can common model defaults be inferred?

Can workbook views be generated from higher-level intent?

Can maintenance views be assembled from reusable projection primitives?

== Initial Assessment

The use case appears structurally compatible with the framework.

The primary challenge is not matrix representation.

The primary challenge is the relationship between:

[source,text]
----

Classification Matrix
↓
Conditional Follow-Up Projections
↓
Maintenance Views
-----------------
----

This area should be explored before introducing domain-specific solutions.

The goal is to provide a generic projection toolkit rather than a dedicated
loan-product framework.
