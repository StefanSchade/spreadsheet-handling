= Adoption Progress Plan
:toc:
:toclevels: 3

== Purpose

This document tracks the current adoption path for the framework and records
the strategic evaluation behind the next feature slices.

It is intentionally more than a checklist.

The important question is not merely:

* Which consumer workaround exists?

The important question is:

* Does a proposed feature make strategic sense for the product?

Consumer workaround removal is only desirable when the answer to that question
is yes.

== Phase Plan

=== Phase 1 -- Worldbuilding Consumer

Goals:

* collect findings,
* validate core primitives,
* document adoption experience.

Status:

* in progress,
* largely successful.

=== Phase 2 -- 0.2.0 Release Stabilization

Focus:

* documentation,
* review,
* release readiness.

Guardrail:

* no major feature wave.

=== Phase 3 -- Consumer Ergonomics

Strategic focus:

* reduce adoption friction where the capability clearly strengthens the
  product.

Planned FTRs:

* `FTR-DYNAMIC-FRAME-PROJECTION-P5`
* `FTR-DYNAMIC-VIEW-COLUMN-TARGETING-P5`
* `FTR-MODEL-DEFAULT-CONFIGURATION-P5`
* `FTR-PRESENTATION-DEFAULTS-SEMANTIC-CLASSIFICATION-P5`

Priority note:

* the first three are currently the stronger product levers,
* presentation-default classification remains relevant but is less central to
  immediate consumer simplification.

=== Phase 4 -- Worldbuilding Rework

Goals:

* adopt the new capabilities where they provide a real simplification,
* remove consumer-specific glue where the platform now owns the capability,
* keep only the extensions that remain legitimately consumer-local.

Important note:

* the objective is not "remove all plugins at any cost",
* the objective is "remove only the workarounds that correspond to
  strategically valid platform capabilities".

=== Phase 5 -- Generic Loan Configuration

Role:

* first business-oriented consumer.

Goals:

* classification matrix,
* multiple codes per cell,
* legends,
* follow-up projections,
* editability concept.

Planning note:

* this phase may still surface additional FTRs around conditional projections,
  editability semantics, and role-combination validation,
* such FTRs should only be created once the consumer modeling becomes
  concrete enough to cut a real generic product slice.

=== Phase 6 -- Gap Analysis

Question:

* do the available primitives plus the ergonomics slices now support a serious
  business-style adoption path?

Outcomes:

* if no: targeted additional FTRs,
* if yes: production-oriented adoption path.

=== Phase 7 -- Project Introduction

Goal:

* present the product with evidence from real consumer experience rather than
  only technical capability claims.

== Strategic Evaluation

=== Primary Decision Criterion

The primary evaluation criterion for follow-up features is:

* does this feature make strategic sense in the product?

Secondary criteria such as implementation effort or local convenience matter
less than this question.

=== Current Assessment of the Main FTRs

==== `FTR-DYNAMIC-FRAME-PROJECTION-P5`

Assessment:

* strategically strong.

Reason:

* dynamic editing projections are central to the product identity,
* the pattern recurs across matrices, localization-like views, and maintenance
  workbooks,
* this is not a worldbuilding-specific quirk.

==== `FTR-DYNAMIC-VIEW-COLUMN-TARGETING-P5`

Assessment:

* strategically strong.

Reason:

* dynamically generated workbook views must remain targetable by later concerns
  such as validation, editability, legend-driven constraints, and presentation,
* otherwise consumers must add procedural glue exactly where the product should
  offer stability.

This is adjacent to dynamic projection, but not the same question.

Projection answers:

* how is the view shaped?

Dynamic targeting answers:

* how do later rules address the generated columns of that view?

==== `FTR-MODEL-DEFAULT-CONFIGURATION-P5`

Assessment:

* strategically strong.

Reason:

* it reduces repetitive low-level configuration without changing the execution
  model,
* it directly improves model growth and long-term maintainability,
* it supports both technical and business-facing consumers.

==== `FTR-PRESENTATION-DEFAULTS-SEMANTIC-CLASSIFICATION-P5`

Assessment:

* strategically valid, but lower immediate leverage.

Reason:

* it clarifies architecture and future consistency,
* but it is less directly tied to the main consumer-side glue currently
  observed in adoption work.

== Current View on Consumer Workarounds

=== Workarounds That Point to Strategic Platform Capabilities

These are the most important signals:

* dynamic matrix view shaping,
* reverse payload reconstruction for dynamic matrix views,
* declarative targeting of projected matrix columns,
* repetitive model/configuration boilerplate.

These justify product-facing FTRs.

=== Workarounds That Currently Look Like Smaller Utility Gaps

These should not automatically become major FTRs:

* small frame-set cleanup helpers,
* simple column dropping or frame normalization,
* local roundtrip hardening around multi-header normalization.

They may still deserve follow-up work, but currently look more like:

* quick wins,
* hardening,
* or small generic steps,

than major strategic product slices.

== Conclusion

At the current stage, four follow-up FTRs are enough for the strategic
adoption agenda:

* `FTR-DYNAMIC-FRAME-PROJECTION-P5`
* `FTR-DYNAMIC-VIEW-COLUMN-TARGETING-P5`
* `FTR-MODEL-DEFAULT-CONFIGURATION-P5`
* `FTR-PRESENTATION-DEFAULTS-SEMANTIC-CLASSIFICATION-P5`

Among them, the first three are currently the most important for product
direction.

No further major strategic FTR is required at this moment solely to mirror the
remaining `worldbuilding` plugins.

Additional follow-up should only be created when it strengthens the generic
product story rather than merely absorbing local consumer glue.
