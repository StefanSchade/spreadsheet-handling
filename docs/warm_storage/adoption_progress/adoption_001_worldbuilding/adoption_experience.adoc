= Adoption Review 001 – Worldbuilding Consumer Validation
:toc:
:toclevels: 3

== Purpose

This document captures observations gathered during the first real-world use of
the spreadsheet-handling framework outside the core and demo repository.

The consumer used for this validation is a worldbuilding repository for a
fictional setting. While the domain itself is non-business related, many of the
observed patterns appear transferable to business configuration and metadata
management use cases.

The goal of this review is not to evaluate implementation quality but to
identify adoption barriers, architectural strengths, recurring modeling
patterns, and future improvement opportunities.

== Executive Summary

The validation exercise produced a surprising result.

Most findings were not related to missing transformation capabilities or
architectural defects.

Instead, the majority of findings concerned:

* projection ergonomics,
* workbook usability,
* model growth,
* configuration complexity,
* user onboarding,
* safe review workflows for reimport.

This suggests that the architectural core has reached a level of maturity where
the primary challenges are increasingly located at the boundary between the
framework and its users.

== Initial Assumptions

The framework originally appeared to solve a problem that could be summarized
as:

[source,text]
----
JSON <-> Spreadsheet
----

During practical usage this characterization proved insufficient.

The actual problem being solved is closer to:

[source,text]
----

Canonical Model
↓
Editable Projections
↓
Roundtrip
↓
Canonical Model

----

The spreadsheet itself is not the primary artifact.

The spreadsheet is an editing surface.

== Key Observation: Projection-Centric Architecture

The worldbuilding consumer repeatedly exposed the same pattern.

Canonical structures such as:

* stories,
* characters,
* places,
* regions,
* story participation relations,
* group memberships,

were projected into editing-oriented workbook views:

* cross tables,
* lookup-enriched sheets,
* faceted matrices,
* validation-driven entry forms,
* legend-driven coding surfaces.

Many discovered enhancement requests can be interpreted as projection concerns:

* dynamic frame projection,
* helper aliasing,
* editable area visualization,
* dense matrix handling,
* workbook presentation defaults,
* multi-view projection of one canonical relation.

This suggests that the framework's center of gravity is shifting from file
transformation toward projection management.

== Transferability to Business Contexts

The observed modeling patterns appear directly transferable to business
configuration scenarios.

A representative business pattern can be described as:

[source,text]
----
Entity A × Entity B
↓
Classification
↓
Follow-up Projections
↓
Specialized Maintenance Views
----

Examples include:

* products × variables,
* applications × permissions,
* systems × controls,
* roles × capabilities,
* countries × regulations.

The matrix itself often does not contain the final business data.

Instead, the matrix classifies relationships.

The classification determines which additional attributes become relevant and
which follow-up maintenance views must be populated.

This pattern emerged naturally during worldbuilding and appears applicable well
beyond the original use case.

== Separation of Canonical Model and Editing Surface

A recurring insight was that spreadsheet-based maintenance does not require the
spreadsheet to be the system of record.

The framework allows a separation between:

* canonical model,
* editing projections,
* validation logic,
* transformation logic,
* version history.

This reduces the need to embed business logic into spreadsheets.

The resulting architecture is closer to:

[source,text]
----
Canonical Model
↓
Generated Workbook
↓
User Editing
↓
Controlled Reimport
----

than to traditional spreadsheet-centric systems.

An important practical consequence is that the reimport target does not need to
be the canonical data directory immediately.

In the validating consumer, the safer pattern was:

[source,text]
----
Canonical Model
↓
Generated Workbook
↓
User Editing
↓
Reimport Into Review Area
↓
Semantic Diff
↓
Explicit Promotion
----

This proved valuable for adoption because it allows users to inspect roundtrip
effects before mutating the canonical repository state.

== LLM-Assisted Modeling

The validation exercise also suggests a possible future user profile.

Historically, pipeline authoring would likely have required software
development skills.

With modern LLM assistance, the activity increasingly resembles model
construction rather than programming.

Most consumer work consisted of:

* identifying entities,
* defining relationships,
* designing editing views,
* selecting projections,
* refining workbook ergonomics.

Implementation effort remained comparatively small.

This may indicate that future users could include:

* business analysts,
* domain experts,
* architects,
* power users,

supported by LLM tooling.

== Governance Considerations

An important observation is the separation between platform development and
consumer modeling.

The framework can act as a reusable platform component.

Consumer repositories can focus on:

* data,
* modeling,
* workbook views,
* validations,

without requiring modifications to the framework itself.

This separation may reduce the need for application-specific development and
deployment cycles for certain classes of maintenance workflows.

The worldbuilding consumer also showed that such a repository does not need to
be packaged as a reusable Python distribution.

A minimal setup with:

* local virtual environment,
* editable install of the framework from a sibling checkout,
* local plugin modules inside the consumer repository,

was sufficient for productive use.

This lowers the entry barrier for non-library consumers and should influence
onboarding guidance.

== Consumer Findings

The following themes appeared repeatedly during validation:

=== Visible Cell Typing

Users need a clear understanding of how visible spreadsheet values are treated
during roundtrip operations.

This topic is considered strategically important and requires an explicit
project decision.

=== Dynamic Projection

Adding new attributes frequently requires projection maintenance.

A recurring request is support for:

[source,text]
----
fixed leading columns
+
all remaining columns
----

without explicit enumeration of every column.

This became especially visible when relation matrices required helper columns
such as `title` and `story_id` to stay fixed while the remaining matrix columns
had to grow dynamically with newly added characters or groups.

=== Lookup Ergonomics

Lookup helper generation works technically but exposed demand for:

* user-friendly aliases,
* configurable helper naming,
* projection-oriented lookup preparation.

The canonical FK-helper path itself proved valuable and should remain a
first-class demonstration path.

The observed friction was not a correctness problem but an ergonomics problem:
stable generated helper names were technically acceptable yet less natural than
domain-oriented aliases.

=== Reimport Safety

A major adoption concern was not transformation power but confidence in the
roundtrip.

What helped in practice was a consumer-side workflow that separated:

* reimport into a temporary review area,
* semantic comparison against canonical data,
* explicit promotion only after review.

This suggests that framework adoption is supported not only by pipeline
features, but also by documented review workflows and small consumer-side
utility tooling.

=== Consumer Extensibility

The validation confirmed that small consumer-side plugins are a viable escape
hatch and do not necessarily indicate a framework weakness.

Several real needs were solved with small, local plugins:

* dynamic matrix view preparation,
* workbook payload reconstruction for reimport,
* frame-set cleanup,
* local header normalization.

This is a positive result.

The framework core already provided the important transformation primitives;
plugins mainly compensated for declarative ergonomics gaps.

=== Multi-View Maintenance Over One Canonical Relation

One of the clearest modeling patterns was the use of multiple workbook views
over a single canonical relation.

A canonical tuple relation of the form:

* `story_id`
* `character_id`
* `facet`
* `value`

was projected into several workbook matrices such as:

* roles,
* notes,
* character development,
* crystals.

This pattern is important because it demonstrates that one canonical relation
can legitimately give rise to several specialized maintenance surfaces without
duplicating the system of record.

=== Workbook UX

Several observations concerned workbook usability rather than transformation
correctness.

Examples include:

* editable-area visualization,
* helper-column styling,
* presentation defaults,
* sheet naming constraints.

Another useful result was confirmation that nested object-like structures can be
rendered as multi-row workbook headers while still roundtripping back into flat
consumer-facing field paths.

This is relevant for dossier-like maintenance surfaces where a single entity
contains several coherent attribute groups.

=== LibreOffice Compatibility

Formula-backed helper columns exposed interoperability questions between:

* XLSX,
* ODS,
* LibreOffice Calc,
* Microsoft Excel.

Further investigation is recommended before making portability guarantees.

The practical value of CLI I/O overrides also became visible here.

They made it possible to keep one shared pipeline graph while switching between
XLSX and ODS at invocation time, reducing duplication and lowering the risk of
format-specific pipeline drift in consumers.

== Potential High-Value Direction

A particularly promising direction is reduction of configuration overhead.

Current consumers frequently repeat information already implied by the model.

Examples include:

* id columns,
* label columns,
* helper definitions,
* validation rules,
* workbook view configuration.

A future configuration-expansion layer could allow users to define model
intent once and derive common policies automatically.

This appears to be one of the highest-value opportunities discovered during
consumer validation.

== Conclusion

The worldbuilding exercise did not primarily validate spreadsheet generation.

Instead, it validated a broader hypothesis:

The framework can act as a projection engine that exposes multiple editable
views over a canonical model while preserving roundtrip consistency.

The most significant discoveries were not architectural failures but
opportunities to improve ergonomics, adoption, and model authoring.

This is a positive signal.

It suggests that future effort should focus less on adding new transformation
primitives and more on making existing capabilities easier to discover,
configure, and apply.

== Adoption Issues

Two concrete backlog items emerged directly from the adoption work:

* `BUG-ODS-REIMPORT-REPEATED-CELL-MEMORY-EXPLOSION`
  +
  ODS reimport exposed a parser failure mode where repeated empty cells or
  rows can trigger uncontrolled memory growth. This is a genuine adoption
  blocker for ODS-heavy workflows and should be treated as a correctness and
  robustness issue, not merely as a performance optimization.

* `FTR-CLI-INTERRUPT-ERROR-UX-P5`
  +
  Interrupting a long-running CLI operation currently falls back to raw Python
  `KeyboardInterrupt` behavior. This is not a correctness defect, but it is a
  noticeable usability gap during real adoption and debugging work.
