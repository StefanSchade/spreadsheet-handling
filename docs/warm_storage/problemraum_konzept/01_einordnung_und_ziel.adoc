== Einordnung und Ziel des Problemraumkonzepts

Dieses Dokument ist ein Arbeitsdokument zur fachlichen Einordnung des Projekts
`spreadsheet-handling`.

Es ist nicht selbst das Functional Model. Es dient als vorbereitender Input für
eine spätere Überarbeitung und Vertiefung des Functional Model. Die Überführung
in die bestehende englische Modelldokumentation soll in einem separaten Schritt
erfolgen, idealerweise durch einen Agenten, der den aktuellen Codebestand, die
bestehenden Dokumente, die Backlog-Artefakte und die bisherige semantische
Modellierung vollständig vor Augen hat.

Das Ziel dieses Dokuments ist nicht, neue technische Regeln oder neue
Metadaten-Kategorien festzulegen. Es soll vielmehr den funktionalen
Problemraum so beschreiben, dass spätere Modellentscheidungen wieder stärker
an den tatsächlichen Anforderungen des Projekts ausgerichtet werden können.

=== Rolle der drei bestehenden Modelle

Das Projekt unterscheidet mehrere Modellebenen. Für dieses Dokument sind vor
allem drei davon relevant:

* Functional Model
* Technical Model
* Semantic Model

Das Functional Model beschreibt, welche funktionalen Probleme das System lösen
soll. Es sollte den Anforderungsraum und die fachlichen Nutzungsmuster des
Systems verständlich machen, bevor technische oder semantische Lösungsbegriffe
eingeführt werden.

Das Technical Model beschreibt, wie das System technisch aufgebaut ist. Es
behandelt Architektur, Komponenten, Schnittstellen, Ablaufstrukturen,
Persistenzformen, Rendering, I/O-Adapter, Pipeline-Mechanik und technische
Grenzen.

Das Semantic Model beschreibt, welche Bedeutung bestimmte Informationen im
System haben und welche Entscheidungen der Algorithmus auf dieser Grundlage
treffen kann. Es soll nicht Selbstzweck sein, sondern dem System helfen,
Informationen kontrolliert durch Transformation, Projektion, Bearbeitung,
Rückführung, Validierung und Speicherung zu bewegen.

Diese drei Modelle stehen damit in einer Abhängigkeitsbeziehung:

## [source,text]

Functional Model
beschreibt den funktionalen Problemraum

Technical Model
beschreibt technische Lösungsstrukturen

Semantic Model
beschreibt Bedeutungen, die algorithmische Entscheidungen ermöglichen
---------------------------------------------------------------------

Das Semantic Model sollte seine Begriffe aus funktionalen Notwendigkeiten
ableiten. Wenn diese funktionale Grundlage zu dünn bleibt, besteht die Gefahr,
dass semantische Begriffe aus sich selbst heraus wachsen.

=== Aktuelle Ausgangslage

Im Projekt wurde in den letzten Entwicklungsphasen eine umfangreiche
semantische Begriffswelt aufgebaut. Diese Arbeit war nicht grundlos. Sie
entstand aus realen Anforderungen des Roundtrips, aus konkreten Problemen mit
Projektionen, Hilfsinformationen, Rückimport, Rendering, lokaler
Arbeitsinformation, Validierung und Persistenzgrenzen.

Gleichzeitig ist erkennbar, dass das Semantic Model an Komplexität gewonnen
hat, ohne überall hinreichend stark im Functional Model verankert zu sein.
Dadurch besteht das Risiko, dass neue semantische Unterscheidungen nicht mehr
direkt aus funktionalen Anforderungen entstehen, sondern aus bereits
eingeführten Begriffen heraus weiter differenziert werden.

Ein typisches Muster wäre:

## [source,text]

Ein Widerspruch wird entdeckt.
↓
Ein neuer semantischer Begriff wird eingeführt.
↓
Der neue Begriff erzeugt neue Grenzfälle.
↓
Weitere Begriffe und Unterkategorien entstehen.
-----------------------------------------------

Dieses Muster kann in Einzelfällen berechtigt sein. Es kann aber auch zu
Begriffsvermehrung führen, ohne dass klar bleibt, welche konkrete
algorithmische Entscheidung dadurch besser, sicherer oder überhaupt erst
möglich wird.

Die Vermutung dieses Dokuments ist daher:

[quote]

---

Ein Teil der aktuellen Komplexität im Semantic Model entsteht daraus, dass der
funktionale Problemraum noch nicht deutlich genug als Prüfstein für semantische
Modellentscheidungen formuliert ist.

---

=== Ziel dieses Dokuments

Dieses Problemraumkonzept soll diese Lücke verkleinern.

Es soll beschreiben, welche funktionalen Anforderungen das Projekt tatsächlich
tragen:

* Welche Art von Brücke schlägt das System?
* Welche Prozesse sollen unterstützt werden?
* Welche Rollen spielen maschinennahe Datenbasis und fachnahe Bearbeitungsansicht?
* Welche Arten von Projektionen entstehen?
* Welche Entscheidungen muss der Algorithmus an Prozessgrenzen treffen können?
* Welche Informationen dürfen verloren gehen?
* Welche Informationen müssen in die langfristig weiterverwendbare Datenbasis
  zurückgeführt werden?
* Welche Informationen dienen nur der Bearbeitung, Darstellung oder dem
  laufenden Prozess?
* Welche Varianten gehören zum Spielfeld des Projekts?
* Welche angrenzenden Ideen sollen bewusst außerhalb des Kerns bleiben?

Das Dokument soll damit keine endgültige Lösung vorwegnehmen. Es soll das
Spielfeld abstecken, auf dem bestehende und zukünftige Modellentscheidungen
bewertet werden können.

=== Sprachliche Abgrenzung

Dieses Dokument bleibt bewusst auf Deutsch.

Das ist keine Aussage über die spätere Repository-Sprache. Der bestehende
Codebestand und die stabile technische Dokumentation bleiben weiterhin
englisch ausgerichtet. Die deutsche Sprache dient hier einem anderen Zweck:
Sie schafft Abstand zu bereits etablierten englischen Begriffen des Technical
und Semantic Model.

Dieser Abstand ist wichtig, weil das Dokument nicht mit vorhandenen
Lösungsbegriffen beginnen soll. Es soll die funktionalen Anforderungen so
beschreiben, dass spätere englische Begriffe gezielt und sparsam daraus
abgeleitet werden können.

Die spätere Überführung in das englische Functional Model muss diesen Abstand
erhalten. Sie darf nicht einfach bekannte semantische oder technische
Terminologie über diesen Text legen. Ihre Aufgabe wird sein, die hier
beschriebene Problemraumlogik in die bestehende Projektdokumentation zu
integrieren, ohne den funktionalen Fokus zu verlieren.

=== Beschränkung auf den funktionalen Problemraum

Dieses Dokument beschreibt bewusst nicht den Lösungsraum.

Es legt keine neuen technischen Architekturregeln fest. Es definiert keine
neuen verbindlichen Metadaten-Kategorien. Es entscheidet nicht, welche
bestehenden FTRs geändert, ersetzt oder neu geschnitten werden sollen. Es
beschreibt auch nicht, wie konkrete Implementierungen im Code anzupassen sind.

Stattdessen konzentriert es sich auf Fragen wie:

## [source,text]

Welche Informationen bewegt das System?
Zwischen welchen Darstellungen wechseln diese Informationen?
Welche menschlichen Bearbeitungssituationen sollen unterstützt werden?
Wo entstehen Mehrdeutigkeit, Redundanz oder Konflikte?
Welche Entscheidungen muss der Algorithmus treffen können?
Welche Verluste sind akzeptabel, welche nicht?
----------------------------------------------

Damit bildet dieses Dokument eine Grundlage für spätere Arbeiten an:

* dem Functional Model,
* dem Semantic Model,
* dem Technical Model,
* einzelnen FTRs,
* Architektur-Reviews,
* und der Begriffsbereinigung im Projekt.

Die beabsichtigte Richtung ist evolutionär. Das bestehende Modell soll nicht
durch einen Big Bang ersetzt werden. Stattdessen soll dieses Dokument helfen,
die weitere Entwicklung an den funktionalen Notwendigkeiten des Projekts
auszurichten.
