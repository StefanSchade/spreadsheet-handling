= FK Helper Problem-Space Notes

== Purpose

This note captures the FK-helper problem space before deriving a general
metadata model.

It intentionally does not classify fields as canonical / derived / intent /
runtime yet. Those terms are ambiguous unless tied to a lifecycle station.

The goal is to collect concrete transformation journeys first. Only after
multiple journeys are visible side by side should common metadata attributes
be extracted.

== Core Use Case

A user works with multiple tabular frames that have relations.

Example:

* `employees` has `employee_id` as primary key.
* `departments` or another frame may reference `employees.employee_id`.
* Human editors should not only see the technical FK value, but also
  readable helper values such as `last_name`, `first_name` or
  `birth_date`.

FK helpers are readable, derived projection values next to an identity
reference. They help humans edit or understand a frame, but they are not
canonical payload.

== Table / Column Annotations

Before FK-helper enrichment happens, tables and columns may carry
annotations.

Examples:

* primary-key column of a table;
* foreign-key relation from one column to another table;
* possible helper fields of a target table;
* default helper field of a target table;
* possibly later: data type, editability, validation role.

These annotations are not themselves FK-helper materialisation. They describe
the schema or semantic role of columns.

Open naming question:

* table schema?
* column annotations?
* relation policy?
* semantic column roles?

== Helper Field Configuration

For a target table, the user may define which fields are allowed as helper
fields.

Example `employees`:

* possible helpers:
  ** `last_name`
  ** `first_name`
  ** `birth_date`
* default helper:
  ** `last_name`
* forbidden as helper: (implicit)
  ** `salary`

The allowed-helper list is a schema/configuration constraint. It prevents
sensitive or unsuitable fields from being projected as read-only helper data.

== FK Helper Enrichment

A projection step may enrich a frame by adding helper columns next to FK
columns.

Input:

* a source frame with an FK column;
* relation annotation or FK policy;
* target frame;
* configured helper fields.

Output:

* source frame plus one or more new helper columns;
* metadata/provenance marking these columns as derived helpers.

The helper values may be inserted as literal values or as spreadsheet formulas
such as VLOOKUP/SVERWEIS.

The resulting helper columns are not original data. They should normally be
read-only or visually marked during spreadsheet rendering.

== Reimport / Drop

On reimport, helper columns must not be promoted into canonical data.

If metadata/provenance is present:

* identify helper columns;
* optionally validate helper values;
* drop helper columns;
* keep the FK column as canonical payload.

If metadata/provenance is missing:

* the system cannot safely know that the column is a helper;
* the column is treated as normal payload unless a bootstrap/configuration step
  re-attaches the missing information.

This is deliberate. `_meta` can be lost; the system must not guess silently.

== Heuristic Inference

The original proof of concept used naming conventions:

* `ID` means primary key;
* `ID(<table>)` means foreign key to `<table>`.

This should no longer be implicit magic.

Instead, heuristic inference may exist only as an explicit configuration step.
That step may inspect naming conventions and then write explicit schema /
relation annotations.

Possible future parameters:

* primary-key name pattern;
* foreign-key name pattern;
* target table naming convention.

== Future: Editable Projection Helpers

A future feature may allow helper-like projection columns that are editable.

These are not ordinary FK helpers.

Possible semantics:

* values are copied from original source fields into a projection;
* spreadsheet users edit the projection copy;
* on reimport, edited projection values are written back to the original source;
* write-back is authoritative; no timestamp conflict resolution unless later
  requirements demand it.

This feature has different semantics from read-only FK helpers.

Important distinction:

* FK helper: derived, read-only, dropped on reimport.
* editable projection helper: derived copy, editable, written back on reimport.

== Crosstable Extensions

FK-helper-like projection may also apply to crosstables.

Current / near-term:

* helper columns on the left side of a crosstable row axis.

Future:

* helper rows above a crosstable column axis;
* possibly multiple helper rows before data starts;
* requires multi-row header / grouped header support.

This suggests that the concept should not be named too narrowly as
`helper_columns` in the long term. The general concept may be closer to:

* relation helper projection;
* axis helper projection;
* projection helper marker.

== Lifecycle Matrix Sketch

Instead of classifying metadata globally as canonical / derived / intent /
runtime, classify concrete information across lifecycle stations.

Stations:

. Bootstrap / annotation
. Enrichment
. Render / export
. Persistence
. Reimport
. Cleanup / drop
. Promotion to canonical data

Candidate information rows:

* primary-key annotation;
* foreign-key relation;
* allowed helper fields;
* default helper field;
* materialised helper column;
* helper column provenance;
* workbook/view helper marker;
* heuristic inference result;
* editable projection copy-back marker.

For each cell, decide:

* created;
* read;
* persisted;
* ignored;
* dropped;
* promoted;
* forbidden.

Only after this matrix is filled should common metadata attributes be derived.

== Current Suspicion

The existing FK-helper metadata model likely contains historical layering:

* legacy v1 per-target configuration;
* newer v2 relation policy;
* runtime provenance;
* durable sheet/view helper markers;
* fallback chains for reimport cleanup.

The target model should be smaller.

A plausible target direction:

* one durable relation/schema annotation model;
* one runtime projection provenance model;
* one durable carrier/view marker for persisted helper projections;
* no implicit heuristics outside explicit inference steps;
* no long-term v1/v2 duality.
