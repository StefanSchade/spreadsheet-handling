== 1. Purpose

This note refines the XRef crosstable concept after reviewing the first
Functional Model promotion.

The initial Functional Model wording said, or strongly implied, that XRef row
and column axes are always primary-key based and not label-based. That is too
narrow.

The existing worldbuilding integration deliberately uses human-readable
character names as dynamic matrix columns. This is not an accidental workaround
that should be removed. It is a valuable UX pattern that should remain
supported.

The correction is not:

## [source,text]

## Visible labels are always identity.

The correction is:

## [source,text]

XRef axes are addressed by configured key fields.

Those key fields may be technical primary keys or explicitly configured,
unique, human-readable alternate keys.

## They must not be guessed from visible labels without configuration.

== 2. Existing Worldbuilding Mechanism

Worldbuilding currently uses a pipeline pattern like this:

## [source,yaml]

* step: join_frames
  left: story_cast_role_cells
  right: characters
  left_key: character_id
  right_key: id
  right_rename: { name: character_name }

* step: contract_xref
  relation: story_cast_role_named
  output: story_cast_role_matrix
  row_keys: [story_id]
  column_key: character_name
  value: value
  dense_axes:
  rows_from: { frame: stories, key: id }
  columns_from: { frame: characters, key: name }
  name: story_cast_role_matrix

---

The original relation may use technical identifiers such as `character_id`.
Before the XRef projection, the relation is enriched with `character_name`.
The matrix then uses `character_name` as the visible and configured column key.

That means the XRef axis is not always addressed by the entity primary key.
In this case, it is addressed by a configured human-readable key field.

This is intentional and should remain supported.

== 3. UX Requirement

For many spreadsheet users, visible technical IDs are a usability regression.

A worldbuilding user wants to maintain a matrix over:

## [source,text]

Rexi
Galli
Comphi
Donzo
-----

not:

## [source,text]

CHAR-REXI
CHAR-GALLI
CHAR-COMPHI
CHAR-DONZO
----------

Similarly, a business user may prefer:

## [source,text]

Japan
Germany
United States
-------------

over:

## [source,text]

COUNTRY-JP
COUNTRY-DE
COUNTRY-US
----------

or may prefer product names, product codes, region names, or other
human-readable stable keys.

Therefore the Functional Model must not require visible technical primary keys
in the crosstable.

The UX requirement is:

## [source,text]

XRef should support human-readable axis keys when they are explicitly
configured and unique.
----------------------

This is not weak typing. It is a declared key contract.

== 4. Static and Dynamic Axis Key Lists

Both static and dynamic axis declarations should remain valid.

Static example:

## [source,yaml]

column_axis:
frame: characters
key: name
keys: [Galli, Donzo, Rexi]
--------------------------

Dynamic dense-axis example:

## [source,yaml]

dense_axes:
columns_from: { frame: characters, key: name }
----------------------------------------------

Both variants use a configured key field.

The difference is only where the list comes from:

[cols="1,3"]
|===
| Axis list source | Meaning

| Static list
| Pipeline configuration directly enumerates the visible axis keys.

| Dense/dynamic list
| Pipeline step derives axis keys from a configured frame and key field.

| Future query/filter step
| Separate upstream step derives a key list from a query or filter and passes
that list into XRef.
|===

XRef should not privilege one of these merely because one is more technical.
The important contract is that the configured key field is resolvable and
unambiguous.

== 5. Primary Key vs. Alternate Key

The Functional Model should distinguish:

[cols="1,3"]
|===
| Term | Meaning

| Primary key
| Technical identity field of the entity frame. Preferred for machine stability.

| Alternate key
| Explicitly configured, unique field that may be used to address axis members.

| Display label
| Human-readable helper/display value. Not automatically a key.

| Visible header
| Workbook carrier representation of an axis member. It may be backed by a
primary key, alternate key, or future mapping contract.
|===

The current baseline should allow:

## [source,text]

## axis key = configured key field

not require:

## [source,text]

## axis key = primary key only

A configured key field may be:

* the entity primary key;
* a unique technical code;
* a unique human-readable name;
* a unique business identifier;
* a statically listed key value.

A configured key field must not be:

* inferred by fuzzy matching;
* parsed from a display label without configuration;
* accepted if non-unique;
* silently repaired if ambiguous.

== 6. Required Functional Model Correction

The Functional Model should not say:

## [source,text]

## row-axis and column-axis identities are PK-based, not label-based

Replace that with wording like:

## [source,text]

row-axis and column-axis addresses are based on configured key fields.

The preferred long-term identity model may use technical primary keys with
separate display labels, but the current baseline must also support explicitly
configured alternate keys when they are unique in the axis entity frame.

Human-readable keys such as `name` may be preferable for workbook usability.
This is not implicit label parsing: the field must be declared as the axis key
and must be validated as unique.
--------------------------------

The model should use the term "axis address" or "configured axis key" where it
does not strictly mean primary-key identity.

== 7. Boundary: No Implicit Label Parsing

Preserving the worldbuilding UX must not open the door to heuristic label
parsing.

Allowed:

## [source,text]

## columns_from: { frame: characters, key: name }

because the axis key field is explicitly declared.

Allowed:

## [source,text]

keys: [Galli, Donzo]
key: name
frame: characters
-----------------

because the static values are interpreted against a configured key field.

Not allowed:

## [source,text]

The workbook header says "Galli", therefore XRef guesses which frame and field
that belongs to.
----------------

Not allowed:

## [source,text]

## Two characters have name = "Rexi", and XRef silently picks one.

The fail-fast rule remains:

## [source,text]

declared alternate key is non-unique -> error
visible axis value cannot be resolved -> error
ambiguous axis value -> error
-----------------------------

== 8. Relation Enrichment Before XRef

The worldbuilding pattern often starts from a technical relation and enriches
it before creating the crosstable.

Conceptual flow:

## [source,text]

technical relation
story_id, character_id, value

* join to characters
  character_id -> characters.id

-> named relation
story_id, character_name, value

-> XRef crosstable
row_key = story_id
column_key = character_name
---------------------------

This is a legitimate pipeline composition pattern.

It should not be treated as an anti-pattern merely because the final XRef
column key is not the primary key of the axis entity.

The important requirements are:

* the enrichment step is explicit;
* the resulting XRef relation uses a declared key field;
* the declared key field is unique in the axis frame;
* reverse recomposition writes to the configured target relation frame;
* any later translation back to technical IDs is a separate configured step or
  part of the surrounding pipeline.

== 9. Relationship to Dynamic Projection FTRs

The following FTRs and implementation slices are relevant background and should
be named when updating the Functional Model:

* `FTR-XREF-CROSSTABLE`
* `FTR-XREF-AXIS-MAPPINGS-P4A2`
* `FTR-XREF-GROUPED-COLUMNS-P4A2`
* `FTR-PROJECTED-FRAME-COLUMN-SEMANTICS-P5`
* `FTR-DYNAMIC-FRAME-PROJECTION-P5`
* `FTR-DYNAMIC-FRAME-PROJECTION-IMPL-P5`
* `FTR-DYNAMIC-VIEW-COLUMN-TARGETING-P5`
* `FTR-DYNAMIC-VIEW-COLUMN-TARGETING-IMPL-P5`
* `BUG-CROSS-CARRIER-META-ROUNDTRIP-P4A`

The most important relation is to the dynamic projection line:

`FTR-PROJECTED-FRAME-COLUMN-SEMANTICS-P5`::
Defines the column-role taxonomy consumed by projected workbook views:
`row_identity`, `display_helper`, `matrix_value`.

`FTR-DYNAMIC-FRAME-PROJECTION-P5` and
`FTR-DYNAMIC-FRAME-PROJECTION-IMPL-P5`::
Introduce and implement the `project_by_role` mechanism. This replaced the
worldbuilding consumer plugins for shaping XRef-derived matrices and preserving
the intended editing UX.

`FTR-DYNAMIC-VIEW-COLUMN-TARGETING-P5` and
`FTR-DYNAMIC-VIEW-COLUMN-TARGETING-IMPL-P5`::
Allow downstream rules such as validations to address dynamically generated
matrix columns by role, for example `roles: [matrix_value]`, without hardcoding
the concrete generated column names.

This means the worldbuilding UX is not accidental residue. It is part of the
recent projection architecture and should be preserved by the XRef Functional
Model.

== 10. Relationship to Future Axis Mapping

`FTR-XREF-AXIS-MAPPINGS-P4A2` points toward a richer future axis model.

That future model may separate:

## [source,text]

identity_key
address_key
display_label
order_key
group_key
---------

For example:

## [source,yaml]

axis:
frame: characters
identity_key: id
address_key: name
display_label: name
order_by: name
--------------

This would make the current worldbuilding pattern more explicit:

* stable technical identity can remain available;
* workbook-visible addressing can remain human-readable;
* display labels can be separated from identity and ordering;
* grouped headers can build on the same axis model.

However, this richer mapping is future work. The current Functional Model should
not pretend it already exists.

Current baseline:

## [source,text]

## configured axis key

Future target:

## [source,text]

## axis mapping with explicit identity, address, label, order, and grouping fields

== 11. Grouped Headers

`FTR-XREF-GROUPED-COLUMNS-P4A2` remains relevant for structured axes such as:

## [source,text]

Kredit
Annuitätendarlehen
Forward-Darlehen

Anlage
Festgeld
Depot
-----

or:

## [source,text]

Region
Country
-------

Grouped headers should not force a PK-only model either.

They should build on the same clarified axis vocabulary:

* configured axis key for addressing;
* optional identity key for technical identity;
* display label for workbook readability;
* group fields for classification and layout.

Grouped headers are not part of baseline XRef, but the baseline wording should
not make them harder by overcommitting to PK-only visible axes.

== 12. Static Lists Must Remain Supported

The model should preserve the ability to hardcode visible axis values.

Example:

## [source,yaml]

column_key: character_name
column_keys: [Galli, Donzo, Rexi]
---------------------------------

or future surface:

## [source,yaml]

column_axis:
frame: characters
key: name
keys: [Galli, Donzo, Rexi]
--------------------------

This is useful when a small, stable editing surface is desired.

Static lists should be treated as explicit projection scope, not as second-class
or legacy behavior.

The same rules apply:

* values are interpreted against the configured key field;
* duplicates fail;
* unknown values fail;
* no fuzzy matching;
* no implicit lookup against unrelated labels.

== 13. Functional Model Update Guidance

When processing this note, update the current XRef Functional Model promotion as
follows:

. Apply the earlier cleanup pass:

* avoid `Step:` headings in the Functional Model;
* prefer `XRef Projection Declaration`,
  `Relation to Matrix Projection`,
  `Matrix to Relation Recomposition`;
* remove `drop_at_persistence` and `drop_if_stale` from Functional Model
  wording;
* replace `XRef expansion provenance` with functional wording such as
  `XRef runtime resolution`;
* do not state that runtime resolution is only for same-run use;
* explain that storing, embedding, or reconstructing runtime resolution is an
  implementation question.

. Replace PK-only axis wording:

* use `configured axis key` or `axis address`;
* say that primary keys are preferred for technical stability but not
  required for all current UX;
* explicitly allow unique configured alternate keys;
* explicitly preserve human-readable matrix headers;
* explicitly preserve static axis lists.

. Preserve fail-fast behavior:

* non-unique configured alternate key -> error;
* ambiguous visible axis value -> error;
* unknown configured axis value -> error;
* missing entity frame or key field -> error.

. Keep advanced features deferred:

* grouped headers;
* full axis mappings;
* editable helper writeback;
* composite cells;
* compact multi-value cells.

. Do not create an FTR.
. Do not touch runtime code.
. Do not edit the warm-storage files after using them as input unless explicitly
asked.
. Keep the Functional Model concise and stricter than this note.

== 14. Working Summary

The existing XRef Functional Model promotion is directionally correct, but its
PK-only axis wording is too narrow.

The corrected baseline is:

## [source,text]

XRef axes are addressed by configured key fields.

Those keys may be technical primary keys or explicitly configured unique
alternate keys, including human-readable fields such as `name`.

Visible labels are never parsed heuristically. If a human-readable value is used
as an axis key, the field must be declared and unique.

Requiring technical primary keys in the visible matrix would often be a UX
regression and would contradict the existing worldbuilding projection
mechanism.
----------

This preserves the current product direction:

## [source,text]

human-readable editable matrix

* explicit configured keys
* fail-fast validation
* no hidden heuristics

---
