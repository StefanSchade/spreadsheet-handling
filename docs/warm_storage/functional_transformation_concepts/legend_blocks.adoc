= Legend Blocks Problem-Space Notes

== Purpose

This note captures the Legend Blocks transformation family before deriving or
adjusting the metadata lifecycle model.

It intentionally starts from the user-facing workbook problem, not from the
current `_meta` field layout.

Legend Blocks are not a standalone rendering trick. They are a small but
central support concept for spreadsheet projections where compact coded values
must remain editable, constrained, explainable, and roundtrip-safe.

The goal is to document the functional storyline first:

[source,text]
----
controlled code values
-> spreadsheet validation / dropdowns
-> optional visible legend
-> optional code expansion on reimport
-> reuse in matrix and compact-cell transformations
----

Only after this storyline is clear should concrete metadata paths be
classified as canonical intent, render support, carrier reconstruction data,
or residue.

== Transformation Family Overview

The Legend Blocks family contains a core feature and several related
extension paths.

=== Core Family

The core family is:

* define a named code list;
* restrict values in one or more spreadsheet columns or ranges;
* generate spreadsheet validation/dropdowns from that code list;
* optionally render a visible legend near the affected table;
* preserve enough workbook metadata so generated legend surfaces do not become
  ordinary payload during reimport.

This is the baseline.

=== Related Transformations

The following transformations may consume or extend Legend Blocks:

[cols="1,3,2", options="header"]
|===
| Transformation / concept | Relationship to Legend Blocks | Status in this note

| Validation from legend
| Uses legend tokens as allowed spreadsheet values.
| Core

| Visible legend rendering
| Renders token/label information as a workbook support surface.
| Core

| Code resolution on reverse transformation
| Optionally maps compact codes back to long-form / explicit values when
returning to a structured model.
| Extension, conceptually important

| XRef / crosstable projections
| Applies the same legend to value cells in a matrix area instead of a simple
column.
| Extension path

| Cell Codec / compact multi-axis
| Packs several semantic values into one spreadsheet cell using a separator,
then unpacks them during reverse transformation.
| Neighboring family, dependent on code semantics

| Referenced codebook/source frame
| Supplies legend entries from a structured data table instead of inline
pipeline config.
| Frame-backed source mode; not built today

| Localization / discriminator-dependent legends
| Allows token labels or even token keys to vary by locale/context.
| Explicitly deferred
|===

This note keeps the current slice narrow. It records extension pressure, but
does not pull all neighboring features into the Legend Blocks lifecycle slice.

== Core Use Case: Restrict Values in a Spreadsheet Column

A pipeline author wants a spreadsheet column to accept only a controlled set
of values.

Example structured declaration:

[source,yaml]
----
legend_blocks:
  status_codes:
    entries:
      - token: E
        label: Editable
      - token: R
        label: Read-only
      - token: K
        label: Calculated

constraints:
  - sheet: variables
    column: status
    rule:
      type: from_legend
      legend: status_codes
----

Example workbook result:

[source,text]
----
variables

variable_id   label          status
amount        Amount         E
rate          Interest Rate  R
margin        Margin         K
----

The `status` column receives a spreadsheet validation list / dropdown:

[source,text]
----
E, R, K
----

Optionally, the workbook also contains a visible legend:

[source,text]
----
Status Codes

Token   Meaning
E       Editable
R       Read-only
K       Calculated
----

The central business point is not formatting.

The central business point is controlled compact editing:

* the user edits compact values;
* the spreadsheet prevents or discourages invalid values;
* the visible workbook explains the codes;
* the structured model can remain more explicit than the spreadsheet surface.

== Motivation: Expert Users Prefer Compact Codes

The motivation is not only that long text makes spreadsheets wide.

A more precise observation is:

Expert users often prefer highly compressed, cryptic spreadsheet notation.

They may want to type `E`, `R`, `K`, or even `E-K-S` because that is faster,
denser, and easier to scan in a large matrix.

At the same time, the structured model should not be forced to become equally
cryptic.

The framework therefore needs a bridge:

[source,text]
----
compact expert-oriented spreadsheet notation
<-> explicit generally-readable structured representation
----

Legend Blocks provide the code vocabulary for that bridge.

This explains why reverse resolution from codes to long-form values is an
important conceptual extension, even if the current implementation only uses
the legend for validation and display.

== Terminology

=== Code List / Legend Block

A named vocabulary of allowed compact values.

Each entry has up to three value roles, plus optional fields:

* *code* (token) — the compact value shown and edited in the cell (`N`);
* *label* — the human-readable display meaning shown in the visible legend or
  dropdown (`Numeric`); may be localised;
* *long form* — the explicit value written into the structured tuple
  (`numeric`); stable, machine-meaningful, referenced by persisted data;
* optional group, description, ordering.

The three roles have different lifecycles (see Label, Long Form below). In the
simplest legends they collapse: long form and label default to the code. They
are separated only when the cell value, the display text, and the structured
value genuinely differ.

The name "Legend Block" comes from the rendered workbook surface, but
conceptually the object is closer to a small controlled code list.

The current note keeps the existing term to avoid renaming during a lifecycle
slice.

=== Code / Token

The compact value shown or edited in the spreadsheet cell. "Code" and "token"
are used interchangeably; the rest of this note keeps "token".

Examples:

[source,text]
----
E
R
K
x
Mo
----

=== Label

The human-readable display meaning, shown in the visible legend table or the
dropdown explanation.

Examples:

[source,text]
----
Editable
Read-only
Calculated
Not meaningful
Monday
----

Display-facing. It may become locale-dependent (see Localization). It is *not*
the value persisted in the structured model, so changing it does not break
already-persisted tuples.

=== Long Form

The explicit value written into the structured tuple on the reverse path
(`numeric`, `input`, `Monday`).

Machine-meaningful and stable. Because persisted tuples reference it by value,
changing a long form is a maintenance event (see Legend Evolution Requires
Maintenance), unlike changing a label.

This is the field the draft previously conflated with the label. Separating them
makes the lifecycle difference explicit: the long form is roundtrip- and
maintenance-critical; the label is display-only. In simple legends the two
coincide.

=== Rendered Legend

A workbook-visible support table generated from the Legend Block.

It explains codes to spreadsheet users.

It is not automatically a payload table just because it is visible.

=== Codebook Source Frame

A possible future source of legend entries.

Instead of declaring the entries inline in the pipeline config, a structured
frame may hold the codebook:

[source,text]
----
weekday_legend

value   label
Mo      Monday
Tu      Tuesday
We      Wednesday
----

This source frame is structured data.

It must be distinguished from the rendered legend table, which is a workbook
support projection.

== Source of Legend Entries

=== Current Baseline: Inline Configuration

The normal current form is inline configuration.

The pipeline config declares the entries directly.

This is simple, deterministic, and sufficient for small stable code lists.

=== Frame-Backed: Referenced Source Frame

A Legend Block may reference an existing data frame as the source of entries.
See <<Lifecycle Across the Git Boundary>> for why the source mode is a lifecycle
statement, not an implementation detail.

Example:

[source,yaml]
----
legend_blocks:
  weekdays:
    source:
      frame: weekday_legend
      token_column: value
      label_column: label
----

Conceptually, this is useful when code lists are business-maintained data.

Important boundary:

* the codebook source frame is part of the structured model;
* the rendered visible legend is a projection of that source;
* changing the source frame requires a new pipeline run before the workbook
  changes;
* no live update inside an already-open spreadsheet is implied.

Implementation status (a fact about the code, not the spec): not built today.
The functional model describes it as a peer of the config-backed mode; the
current slice neither builds nor blocks it. This is binary, not a roadmap.

== Core Workflow

=== Forward Path: Structured Model to Workbook

Input:

* ordinary data frames;
* Legend Block declarations;
* constraints or render configuration referencing those Legend Blocks.

Forward behavior:

. The pipeline carries the Legend Block declaration as workbook/projection
intent.
. Validation generation reads the legend entries and creates spreadsheet
validation/dropdowns for the target column or range.
. The render layer may create a visible legend table from the same entries.
. If a visible legend table is rendered, concrete workbook placement may be
computed during render.
. Workbook-embedded metadata may carry reconstruction information needed by the
parser on readback.

The user-facing result is:

* compact editable cells;
* controlled allowed values;
* optional explanatory legend surface.

=== Workbook Editing

The user edits the payload cells.

For the baseline scenario, the user does not edit the visible legend as a way
to change the code list.

Changing code-list definitions remains a model/configuration action, not an
in-workbook editing action.

A future feature could support workbook-authored codebook edits, but that would
require its own precedence and conflict rules.

=== Reverse Path: Workbook to Structured Model

On reimport, the pipeline reads the workbook.

For the baseline scenario:

* edited payload cells are preserved;
* generated legend tables are not imported as ordinary payload frames;
* validation metadata may be read back or ignored depending on backend support;
* concrete render reconstruction hints are transient.

Optional extension:

* compact tokens may be mapped back to long-form / explicit structured values.

Example:

[source,text]
----
spreadsheet value: E
structured value:  Editable
----

or, for a weekday example:

[source,text]
----
spreadsheet value: Mo
structured value:  Monday
----

This mapping must be explicit. The system must not guess from visible labels.

== Lifecycle Across the Git Boundary

This section applies the meta-lifecycle lens to Legend Blocks. The framework
concepts it builds on (carriers, Type 2 events, lifecycle attributes) live in
`docs/functional_model/ch05_problem_space_overview/03_meta_lifecycle_framework.adoc`
and are not restated here.

=== Meta Is Lost or Stale at the App Boundary

Data may arrive from a git repository where someone else persisted it. The
`_meta` sidecar is gitignored: it may be absent, or present but stale relative
to data that changed since it was written.

Consequence for Legend Blocks: no essential legend information may depend on
surviving in `_meta` across the git boundary. Each legend's durable truth must
live somewhere that travels in git.

=== Durable Home Outside Meta

Every legend whose entries are essential has a durable home that is not meta:

* a *data frame* in the structured model -- the codebook is
  business-maintained data; or
* the *pipeline configuration* -- the vocabulary is framework-fixed and not
  part of the data.

What `_meta` carries is only:

* the *resolved working form* of the legend for the current run, and
* *workbook reconstruction hints* (carrier-local).

Both are reconstructible from the durable home and may be dropped or treated as
stale. The legend is therefore re-resolved from its durable source on every
forward run; a prior `_meta` is not trusted, because it may be outdated.

=== Two Source Modes

The durable home determines the source mode. Both are equal functional
concepts; the functional model describes both regardless of what the code
builds today.

* *Config-backed:* entries declared inline in the pipeline config. Durable
  home: the version-controlled pipeline definition.
* *Frame-backed:* entries read from an existing data frame. Durable home: the
  structured data in git.

The choice of mode is a functional statement about where a legend's durable
truth lives -- not an implementation detail.

Implementation status (a fact about the code, not the spec, kept binary rather
than as a future/now gradient): config-backed is implemented; frame-backed is
not. The functional model thinking ahead of the code is normal -- it is a
specification. Whether and when frame-backed is built is decided elsewhere, not
implied by its presence here. (The semantic model and its registry, by
contrast, track the current implementation, because the registry is part of the
implementation.)

=== Worked Example: Variable Inventory per Product

[NOTE]
====
This example is deliberately complex. For illustrating the lifecycle it is more
economical to show one rich use case and let simpler ones follow by omission
than to repeat a trace for each variant -- it avoids redundancy.

That choice is expository, not a claim of completeness. The example illustrates
the target lifecycle; it may have gaps against the current implementation, and
some legs are not built today (see the per-leg note at the end). It is the
storyline, not a guaranteed end-to-end path right now.
====

A variable inventory is maintained per software product (configuration data for
several products built on a shared framework). For each variable, per product,
two facts are recorded: its data type and its input/output role. The target is
a single compact crosstable the business analyst can scan, and a structured
tuple output after roundtrip.

Two legends, with different durable homes:

* `variable_product_data_type_legend` -- `S` = string, `N` = numeric. Present
  in the data as a frame (frame-backed source mode).
* `variable_product_iotype_legend` -- `I` = input, `O` = output. Not in the
  data inventory; supplied in pipeline config (config-backed source mode).

Logically two identical crosstables (variable x product) are spanned -- one
carrying the data type, one the IO role. A cell codec packs the two axis tokens
into one compact cell (`S-I`, `N-O`) so the analyst sees everything condensed on
one table. A combined dropdown list restricts allowed values.

On reverse transformation a compact cell such as `N-I` (numeric, input) is
split back into a tuple:

[source,text]
----
{ variable:  "some_business_field",
  product:   "credit_application",
  IO_type:   "input",
  data_type: "numeric" }
----

This example illustrates the target state. Its data-type leg uses the
frame-backed mode (not implemented today); its IO leg uses the config-backed
mode.

=== Type 2 Trace for This Example

[source,text]
----
app boundary ingress   Data from git; _meta absent or stale -> ignored.
                       data_type legend arrives as a data frame; iotype legend
                       exists only in pipeline config.

setup_legends (T1)     Resolves both legends from their durable source into
                       Frame meta (working form, not sidecar-durable). May be
                       one step or one per legend -- implementation detail.

crosstable + codec(T1) Two logically identical crosstables (data_type / iotype)
                       combined via multi-axis; cell codec packs to S-I etc.;
                       allowed-combination list taken from config, or derived
                       with explosion caution.

tabular write (T2)     Crosstable rendered; dropdown validation set; visible
                       legends optionally rendered; resolved placement written
                       to the workbook-embedded carrier.

[ analyst edits S-I / N-O in one compact table ]

tabular read (T2)      Reconstruction hints re-anchor legend/support tables;
                       payload cells read.

domain egress (T2)     Cell codec splits; multi-axis separates the axes ->
                       {variable, product, IO_type, data_type} to git.
                       Meta projected to persistable subset: resolved form and
                       hints dropped; legends persist in their frame/config home.
----

=== The Second Roundtrip: Reconstructing the View

A roundtrip does not end the story. The tuples persisted to git on the reverse
path become the input of the *next* forward run, which must rebuild the compact
crosstable from them.

That next forward run:

* reads the long-form tuples (`data_type: "numeric"`, `IO_type: "input"`);
* re-resolves both legends from their durable home;
* maps each long-form value back to its token (`numeric -> N`, `input -> I`) --
  the inverse of the egress resolution;
* re-spans the two crosstables and re-packs the compact cell (`N-I`).

Two consequences that were implicit before:

* code resolution must be *reversible* -- token and long-form map both ways
  unambiguously, or the view cannot be rebuilt losslessly;
* the legend must be *stable* across runs -- a changed legend can orphan
  long-form values that no longer map to any token.

The reverse path stores the long-form, not the token, on purpose: the
structured model should be explicit and readable, not cryptic. The cost is that
the long-form in each tuple is effectively a reference to the legend by value,
which makes the next point necessary.

=== Legend Evolution Requires Maintenance

Because the tuples carry the resolved long-form, a change to the legend is not
local. Correcting a typo in a label (`numric -> numeric`) means:

* fixing the legend entry, and
* back-translating every already-persisted tuple that still carries the old
  value.

Editing the legend data inside the canonical data mid-roundtrip is possible but
ugly: it smuggles a migration into an ordinary run.

The cleaner direction is a dedicated *maintenance transformation*, a peer to
dropping or renaming a table or column -- an explicit step that renames a token
or label and cascades the change across the legend and all referencing tuples.
This is the same shape as foreign-key rename reconciliation (cf.
`FTR-RENAME-RECONCILIATION`), applied to legend values.

This slice does not implement the maintenance transformation. It records that
legend evolution is a first-class maintenance concern, not an in-band edit.

=== The Allowed-Combination Set Is Its Own Lifecycle Citizen

The dropdown list for the combined cell (`S/I`, `S/O`, `N/O`, ...) is the
cartesian product of the per-axis tokens. It is not a single legend's entry
list, and it has its own lifecycle:

* it can explode (n x m; a few longer lists become unwieldy);
* it may therefore be supplied explicitly in pipeline config rather than fully
  generated;
* domain constraints may forbid combinations (e.g. `N/I` does not exist) --
  also durable config, not a runtime artifact.

It is named separately under Conceptual Information Pieces.

== Application to Crosstables

The next natural application is a crosstable / matrix.

Instead of applying a legend to one ordinary column, the legend applies to the
matrix value area.

Example:

[source,text]
----
feature      product_a   product_b
currency     E           E
amount       E-R-K       R
obsolete     x           x
----

Here the row/column axes identify the tuple being edited, while the cell value
is a compact legend-coded value.

Benefits are the same as for a simple column:

* matrix cells stay compact;
* dropdowns can restrict allowed values;
* a visible legend can explain the codes;
* reverse transformation can optionally expand the code into explicit values.

This is a more advanced use case than the simple-column scenario and should not
be used as the opening example of the family.

It belongs as the first extension path after the baseline column/range case.

== Relationship to Cell Codec / Compact Multi-Axis

Cell Codec is a neighboring concept.

Legend Blocks define the allowed atomic tokens and their meanings.

Cell Codec defines how several values are packed into one cell.

Example:

[source,text]
----
E-K-S
----

This may represent three separate semantic values that would otherwise live in
three columns or three tuple attributes.

Forward direction:

[source,text]
----
explicit structured values
-> compact spreadsheet cell
----

Reverse direction:

[source,text]
----
compact spreadsheet cell
-> explicit structured values
----

The important distinction:

* Legend Block answers: what does token `E` mean?
* Cell Codec answers: how is `E-K-S` split and mapped?
* Compact Multi-Axis answers: which semantic dimensions do the split parts
  represent?

Therefore Cell Codec should not be folded into the Legend Blocks lifecycle
slice.

But the Legend Blocks model should remain compatible with it.

== Relationship to Localization / Discriminators

There is a possible future interaction with localization or other discriminator
axes.

Example question:

* In a German workbook, does `Mo` resolve to `Montag`?
* In a French workbook, does the key become `lun`, or does only the label become
  `lundi`?
* Does the token remain stable across locales while only the label changes?
* Or is the token itself locale-specific?

These are hard questions because token identity and display language may both
vary.

Current recommendation:

* keep Legend Blocks simple for now;
* do not make localization part of the current Legend Blocks family slice;
* document that labels may later become context/discriminator-dependent;
* avoid hardcoding assumptions that make discriminator-dependent labels
  impossible later;
* do not support locale-dependent token identity without a dedicated FTR.

The code/label/long-form split makes the safe near-term path clearer: only the
label is locale-variable; the code (cell token) and long form (structured value)
stay stable across locales.

[source,text]
----
stable code + stable long form + locale-dependent label   plausible future
locale-dependent code identity                            deferred, much harder
----

The current Legend Blocks slice should not try to solve this.

== Render and Layout Boundary

Legend Blocks may provide render intent:

* a visible legend should exist;
* it belongs to or explains a target table/range;
* it uses the declared token list;
* it may need a title.

Detailed layout behavior is not owned by this problem-space note.

Out of scope here:

* exact placement algorithm;
* multi-table layout policy;
* collision avoidance;
* protection / write-locking of legend cells;
* visual styling beyond the fact that a legend is a support surface.

The only lifecycle-relevant layout point is:

If the renderer computes concrete coordinates for a visible legend, those
coordinates are a render/readback support artifact, not durable model truth.

== Rendered Legend vs Codebook Source Frame

Avoid the misleading rule:

[source,text]
----
Legend Blocks are not data frames.
----

That is too broad.

The correct distinction is:

[source,text]
----
A rendered legend table is not canonical payload.
A codebook source frame may be structured data.
A Legend Block declaration may be derived from that source frame in the
frame-backed mode (not built today).
----

This distinction matters because future workflows may maintain code lists as
ordinary structured data.

In such a workflow:

* `weekday_legend` may be a real source frame;
* the workbook-visible legend may be a rendered projection of it;
* reimporting the visible legend as if it were the source frame would still be
  wrong unless a dedicated editable-codebook feature is defined.

== Conceptual Information Pieces

This section names conceptual information, not final field paths.

=== Legend Entry Set

The declared or resolved list of entries, each carrying code, label, and long
form (see Terminology).

Lifecycle expectation:

* durable projection intent when declared inline (config-backed), or read from a
  data frame (frame-backed; not implemented today);
* code and long form are stable and reference-bearing; the label is display-only
  and may be localised;
* usable by validation, rendering, and code-resolution consumers;
* re-resolved from its durable home each forward run; not trusted from a prior
  `_meta`.

=== Allowed Combination Set

The set of permitted token combinations for a compact multi-axis cell (e.g.
`S/I`, `S/O`, `N/O`), used to drive the combined dropdown.

Lifecycle expectation:

* distinct from any single legend's entry set -- it is the cross-product of
  axis tokens, narrowed by domain rules;
* durable config when explicitly supplied or constrained (e.g. `N/I` forbidden);
* may instead be derived at run time, but then bounded against combinatorial
  explosion;
* not inferred from edited workbook values.

=== Legend Identity

The name used by constraints, render configuration, codecs, or matrix
configuration to refer to a Legend Block.

Lifecycle expectation:

* durable enough for references;
* renaming requires maintenance of references;
* should not be inferred from visible workbook text.

=== Target Usage

A declaration that a legend applies to a column, cell range, or matrix value
area.

Lifecycle expectation:

* durable projection intent;
* consumed by validation/rendering;
* may be expressed through constraints, workbook-view configuration, XRef
  configuration, or future compact-cell configuration.

=== Visible Legend Request

A declaration that the legend should be rendered as a visible support table.

Lifecycle expectation:

* durable workbook/projection intent;
* layout-specific details are delegated to rendering.

=== Rendered Legend Surface

The actual visible legend table in the workbook.

Lifecycle expectation:

* workbook support surface;
* not ordinary payload;
* must not accidentally reimport as a data frame.

=== Code Resolution Rule

Optional rule that says how compact tokens map back to long-form structured
values.

Lifecycle expectation:

* explicit transformation contract;
* must not be guessed from visible labels unless explicitly configured;
* belongs to reverse transformation semantics, not plain rendering.

=== Workbook Reconstruction Hint

Concrete information embedded in the workbook carrier so the parser can
recognize generated support surfaces.

Lifecycle expectation:

* may be workbook-carrier durable for that workbook;
* not sidecar-durable;
* recomputed on the next forward render;
* dropped before writing structured sidecar metadata.

=== Parser Table Classification

The parser-side result that a visible table is a legend/support table rather
than an ordinary data table.

Lifecycle expectation:

* transient read-path result;
* load-bearing only during workbook-to-frames conversion;
* not project metadata.

== Functional Slices

=== Slice: Define Legend Entries

The user declares or provides a named set of compact tokens and meanings.

==== Information Consumed

* Inline entries, or future source-frame configuration.

==== Information Produced

* Legend entry set.
* Legend identity.
* Optional ordering/grouping/description fields.

==== Notes

This slice does not render anything by itself.

It creates a reusable code vocabulary.

=== Slice: Apply Legend to Simple Column or Range

The user applies a Legend Block to a normal spreadsheet column or range.

==== Information Consumed

* Legend identity — essential.
* Legend entry set — essential.
* Target column/range — essential.

==== Information Produced

* Workbook validation/dropdown rule.
* Optional visible legend request if configured.
* No payload transformation by default.

==== Notes

This is the baseline scenario and should be the first example in the
functional model.

=== Slice: Render Visible Legend

The workbook renderer turns a Legend Block into a visible support table.

==== Information Consumed

* Legend entry set.
* Optional title.
* Visible legend request.
* Render/layout context.

==== Information Produced

* Rendered legend surface.
* Optional workbook reconstruction hint.

==== Notes

The exact layout algorithm is outside this concept.

This slice only needs to state that a visible support surface may be rendered
and must remain distinguishable from payload on readback.

=== Slice: Reimport Workbook with Generated Legend Surface

The workbook parser reads a workbook containing data tables and generated
legend surfaces.

==== Information Consumed

* Visible workbook tables.
* Optional workbook reconstruction hints.

==== Information Produced

* Payload frames only.
* Parser classification for legend/support tables.
* No ordinary payload frame for generated legend surfaces.

==== Notes

This is the safety slice that prevents visible support surfaces from corrupting
the structured model.

=== Slice: Resolve Codes on Reverse Transformation

Optional extension.

The reverse pipeline maps compact spreadsheet tokens to more explicit
structured values.

==== Information Consumed

* Payload cell values.
* Legend entry set.
* Explicit code resolution rule.

==== Information Produced

* Structured values using labels, long-form values, or configured target
  fields.

==== Notes

This is not the same as validation.

Validation says which token is allowed.

Resolution says what that token becomes when returning to a more explicit
model.

=== Slice: Apply Legend to Crosstable Value Area

Extension path.

The legend applies to a matrix value area instead of a simple column.

==== Information Consumed

* Legend identity.
* Legend entry set.
* Matrix value-area configuration.
* XRef row/column identity configuration.

==== Information Produced

* Validation/dropdown behavior for matrix cells where supported.
* Optional visible legend.
* Optional reverse code resolution during matrix expansion.

==== Notes

This is a later application of the same concept, not the core starting point.

The matrix example should be used after the simple-column example.

=== Slice: Use Legend Tokens in Cell Codec

Neighboring family.

A compact cell may contain several legend tokens separated by a configured
separator.

==== Information Consumed

* Legend entry set.
* Codec separator and position mapping.
* Target semantic dimensions.

==== Information Produced

* One compact spreadsheet cell in forward direction.
* Several explicit structured values in reverse direction.

==== Notes

This belongs to the Cell Codec / Compact Multi-Axis family.

Legend Blocks provide vocabulary; Cell Codec provides packing/unpacking.

== Why This Slice Is Not Self-Contained

Meta resists being bent into a purely local, per-feature concept. That
resistance is intrinsic to this domain, not an accident of the current code:
realistic use cases couple a legend to the transformations around it (xref,
cell codec, multi-axis, maintenance).

A consequence for method: a feature cannot be built in isolation by dumping
everything that comes to mind about it. The work has to be driven by use cases
of increasing complexity, each pulling in the neighboring transformations it
actually needs. The frame-backed legend is the concrete example -- it does not
exist yet precisely because no use case has yet pulled the surrounding
transformations along to support a frame-sourced legend end to end.

So this note deliberately records extension pressure and cross-family seams
rather than pretending the legend slice closes on its own. The per-family
lifecycle classification stays useful as a lens; the driver of the work is the
use case, not the family boundary.

== Scope Boundaries for the Current Slice

The current Legend Blocks lifecycle/refactoring slice should include:

* inline Legend Block declarations;
* validation/dropdown use;
* visible legend rendering at the concept level;
* readback safety for generated legend surfaces;
* pruning of render/readback reconstruction data before sidecar persistence;
* classification of parser-local residue.

The current slice should not include:

* full Cell Codec redesign;
* grouped XRef design;
* localized/discriminator-dependent token identity;
* editable codebook tables inside the workbook;
* renderer layout policy beyond required metadata handoff;
* write protection or detailed styling of legend tables;
* source-frame-backed legends, except as future-compatible concept pressure.

== Simplification Direction

The implementation should not flatten all Legend Block concerns into one
lifecycle bucket.

There are distinct concerns:

* code vocabulary;
* usage target;
* validation generation;
* visible support rendering;
* code resolution;
* workbook reconstruction;
* parser classification.

The simplification goal is to keep these concerns separated and prevent stale
runtime/readback artifacts from pretending to be durable model intent.

Target direction:

* declared entries and usage intent survive as model/projection intent;
* workbook-specific reconstruction hints remain carrier-local;
* parser-local annotations do not become semantic contracts;
* matrix and cell-codec extensions reuse the vocabulary without forcing the
  baseline feature to carry their full complexity;
* localization remains explicitly deferred.

== Handover to Functional Model

The Functional Model should start with the simple-column use case:

[source,text]
----
I want to restrict a spreadsheet column to a named set of compact codes.
The workbook should provide dropdown validation.
Optionally, the workbook should display the code meanings as a legend.
On reverse transformation, the code may optionally be resolved into a more
explicit structured representation.
----

Then it should show how the same concept extends to:

* matrix value areas;
* compact multi-token cells;
* future source-frame-backed codebooks.

The Functional Model should not start with the most advanced matrix example.

Inventory findings are evidence.

They are not the storyline.
