= Semantic Family Assessment
:toc:
:toclevels: 2

Review date:: 2026-05-27
Scope:: Architecture note on emerging transformation-family semantics after Phase 3.5.
Status:: Warm-storage architecture note. This is not a normative specification and does not define a new semantic framework.

== Executive Summary

Phase 3.5 exposes several emerging transformation families, but the repository does not need a grand unification layer to benefit from them. The useful pattern is narrower: where implementation already has stable primitives, registries, and guards, the project can name the semantic family and use that name to reduce drift.

The clearest stabilized families are FK-helper policy application, tuple/matrix xref conversion, cell-value encoding, compact multiaxis composition, discriminator splitting, workbook projection, and shared finding-frame serialization. These are not all the same kind of transformation, and forcing them into one abstraction would likely create more governance cost than value.

The main architectural opportunity is practical reconciliation: use the glossary terms consistently, keep registry semantics tied to review and drift detection, and slice future backlog around concrete implementation surfaces.

== Already-Generalized Transformation Families

=== FK-helper policy application

FK-helper behavior is now a policy-driven family rather than a naming-convention family.

Implemented anchors::

* `src/spreadsheet_handling/domain/helper_policies.py`
* `src/spreadsheet_handling/domain/transformations/fk_helpers/policy.py`
* `src/spreadsheet_handling/domain/transformations/fk_helpers/enrich.py`
* `src/spreadsheet_handling/domain/transformations/fk_helpers/drop.py`

Current semantic shape::
Producer-side configuration records helper policy/provenance. Primitive transformations consume that policy. The v1 compatibility shape is a lifecycle bridge, not a separate long-term semantic model.

=== XRef tuple/matrix conversion

`xref_crosstable` already expresses a bidirectional tuple/matrix family.

Implemented anchor::
`src/spreadsheet_handling/domain/transformations/xref_crosstable.py`

Current semantic shape::
`expand_xref` converts a matrix/crosstable frame into a long-form relation. `contract_xref` converts a relation into a matrix/crosstable frame. Dense-axis behavior is explicit enough to support future grouped-axis cleanup without inventing unrelated terminology.

=== Cell codec

Cell codec behavior is already a reusable encoding/decoding surface that composes with compact transforms.

Implemented anchor::
`src/spreadsheet_handling/domain/transformations/cell_codec.py`

Current semantic shape::
The codec is a helper transformation family for representing structured cell payloads without making every caller own string parsing semantics.

=== Compact multiaxis composition

Compact multiaxis transformation is already a composition over xref-style matrix semantics and cell codec behavior.

Implemented anchor::
`src/spreadsheet_handling/domain/transformations/compact_multiaxis.py`

Current semantic shape::
This is a concrete composed transformation, not a reason to generalize all compact, grouped, and localized-resource behavior into one abstraction.

=== Discriminator split

Discriminator split is a stable adjacent transformation family.

Implemented anchor::
`src/spreadsheet_handling/domain/transformations/discriminator_split/`

Current semantic shape::
It handles partitioning by discriminator values and has enough package-boundary structure to remain separate from xref tuple/matrix conversion.

=== Workbook projection

Workbook projection is becoming a distinct family around frames, projected sheets, workbook view metadata, and readback mappings.

Implemented anchors::

* `src/spreadsheet_handling/domain/workbook_views.py`
* `src/spreadsheet_handling/rendering/workbook_projection.py`
* `src/spreadsheet_handling/rendering/composer/layout_composer.py`

Current semantic shape::
The domain frame remains separate from the workbook sheet projection. `workbook_view.sheet_mappings` records projection/readback identity rather than asking parsers to infer it from sheet names.

=== Finding-frame serialization

Finding-frame behavior is generalized at the interchange-frame layer.

Implemented anchor::
`src/spreadsheet_handling/domain/finding_frame.py`

Current semantic shape::
This provides shared frame serialization for findings while allowing validation finding objects to retain domain-specific meaning.

== Still-Fragmented or Special-Case Areas

=== Localized-resource composition

Localized-resource flows demonstrate composition across existing primitives, but they do not yet define a general transformation family.

Assessment::
This is acceptable. Treat localized resources as a use-case composition unless repeated implementation drift shows that a stronger abstraction would remove real complexity.

=== Resource overrides, sparse defaults, pivot, and join behavior

These areas are adjacent to tuple/matrix and composition semantics, but they are not currently one implemented family.

Assessment::
Backlog work should compare them where useful, but should avoid defining a broad transformation taxonomy before implementation evidence requires it.

=== Grouped axes and compact xref backlog

Grouped-column, axis-mapping, and compact-xref backlog items overlap with current xref and dense-axis semantics.

Assessment::
This is backlog fragmentation rather than a missing architecture layer. The cleanup need is shared terminology and acceptance criteria.

=== Adapter-boundary constants

Workbook projection and parser behavior still depend on localized constants and string conventions.

Assessment::
This is a practical ownership issue. It should be solved with narrow ownership and guards, not a new semantic framework.

== Tuple-Matrix Semantics

The glossary now provides enough vocabulary for tuple/matrix work:

* a canonical tuple frame is the normalized relation-style representation;
* a matrix frame is a two-dimensional display or interchange representation;
* a crosstable frame is a matrix-style frame with one or more axes encoded in headers;
* `expand_xref` moves from matrix/crosstable to relation;
* `contract_xref` moves from relation to matrix/crosstable.

The implementation and user guide align on this direction. Older architecture text that reverses expand/contract direction should be treated as drift and corrected during documentation cleanup.

Practical implication::
Future xref, compact, grouped-column, and dense-axis work should start from these terms rather than introducing local synonyms.

== Workbook Projection Semantics

Workbook projection now has a workable semantic boundary:

* domain data lives in frames;
* workbook views describe projected workbook layout;
* sheets are output/readback projection artifacts;
* `workbook_view.sheet_mappings` preserves projection identity across readback;
* presentation metadata can be canonical when it represents user-authored workbook facts.

Remaining open questions are concrete:

* conflict precedence when payload metadata and workbook view metadata disagree;
* ownership of workbook option constants;
* ownership of header-flattening separators and readback parsing conventions;
* guard coverage for projection/readback invariants.

These questions are operational and adapter-boundary questions, not a reason to rename the frame/view model.

== XRef, Compact, and Discriminator Relationships

The review suggests a useful relationship map:

XRef::
Bidirectional tuple/matrix conversion over relation and crosstable forms.

Compact multiaxis::
Composition that builds on xref-style axes and cell-codec representation.

Grouped columns and axis mappings::
Likely backlog extensions of xref/dense-axis semantics, not independent vocabulary islands.

Discriminator split::
Adjacent transformation family for partitioning and recombining by discriminator values. It should stay conceptually separate unless future code shows direct shared mechanics.

Practical implication::
Use shared language where the same axis/matrix concepts appear, but keep implementation slices small and family-specific.

== Localized-Resource Composition Observations

Localized-resource workflows are useful evidence that existing primitives can compose:

* relation-style resource data can be projected into workbook-friendly forms;
* compact/cell-codec behavior can help represent structured resource cells;
* workbook view metadata can preserve intended workbook layout;
* validation/finding frames can report issues without becoming resource-specific semantics.

This does not imply a need for a localized-resource framework. It is better treated as an integration scenario that can expose missing seams in existing primitives.

== Operational-Carrier Observations

The operational-carrier vocabulary is helping where adapter behavior previously looked ambiguous.

Helpful distinctions::

* `carrier` describes the medium or backend-specific representation.
* `carrier-shape` describes structural constraints imposed by that medium.
* `operational-carrier` describes metadata needed for operational transport rather than model semantics.
* `presentation_meta` can be canonical when preserving authored presentation facts such as column widths.

Current risk::
The vocabulary can be over-applied. Not every imported workbook fact should trigger a registry-level classification debate. The useful test is whether the fact affects round-trip behavior, user-authored intent, transformation correctness, or adapter drift.

== Overmodeling and Governance-Risk Assessment

=== Where precision is helping

Vocabulary precision is clearly useful in these areas:

* FK-helper primitive behavior, where policy/provenance replaced inference.
* Workbook projection, where frame/sheet/view distinctions prevent readback guesswork.
* XLSX presentation metadata, where column widths can be classified without treating every carrier fact as model data.
* Lifecycle bridges, where temporary compatibility shapes are visible and reviewable.
* Finding frames, where shared serialization no longer requires collapsing validation semantics.

=== Where abstraction could slow implementation

The following risks should be avoided:

* turning registries into runtime rule engines;
* requiring lifecycle classification for every metadata key;
* forcing all transformations into a single family abstraction;
* expanding schema validation before a pilot contract proves value;
* treating every adapter constant as a semantic doctrine issue;
* reopening implemented FK-helper package boundaries without new evidence.

=== Governance posture

The practical governance posture is incremental:

* name semantics where they prevent drift;
* add guards where a contract is already stable;
* keep transitional states explicit;
* prefer backlog reconciliation over broad taxonomy work.

== Practical Guidance for Future Semantic Adoption

Future semantic adoption should remain evidence-driven.

Recommended approach::

* Start with implemented behavior, not an aspirational taxonomy.
* Use glossary terms in backlog and documentation when touching related files.
* Update registry review artifacts alongside runtime changes only when the semantic contract is real.
* Add guards for stable contracts, not for every draft vocabulary distinction.
* Re-anchor stale FTRs before using them as implementation authority.
* Keep workbook projection, tuple/matrix conversion, and helper-policy semantics separate unless a concrete shared implementation surface appears.
* Preserve transitional-state language where compatibility bridges or partial adoption are intentional.

This note should be used as review context for later cleanup work, not as a mandate to redesign the transformation layer.
