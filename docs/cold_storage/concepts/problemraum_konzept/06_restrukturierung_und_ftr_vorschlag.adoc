== Restrukturierungsvorschlag: Functional Model vs. Semantic Model

=== Arbeitshypothese

Die weitere Modellarbeit sollte diese Rollen klarer trennen:

* *Functional Model* erklärt den Problemraum: Arbeitsformen, Motivation,
  Nutzungsszenarien, Prozessgrenzen, fachliche Risiken und Beispiele.
* *Semantic Model* definiert den Vertrag: externally visible vocabulary,
  Garantien, Interpretationsregeln, Producer/Consumer-Verantwortung,
  Precedence, Lifecycle und Guard-Kandidaten.

Diese Trennung ist eine Navigations- und Verantwortungsregel. Sie verlangt
keinen Big Bang und keine neuen Runtime-Konzepte.

=== Konkrete Restrukturierung

Kurzfristig sollte die Dokumentation nur umgruppiert und verlinkt werden:

* Semantic Model mit klarer Reader-Struktur versehen:
  ** Management Summary,
  ** Contract Core,
  ** Explanatory Context,
  ** Adoption And Planning.
* Problem Statement, External Inspirations und Roundtrip Walkthroughs nicht
  löschen. Sie bleiben als erklärender Kontext erhalten.
* Das deutsche Problemraumkonzept als Functional-Model-Input sichtbar machen
  und über eine einzelne Entry Page renderbar bündeln.
* Keine Registry-, Runtime- oder Implementierungsänderungen in diesem Schritt.

Mittelfristig sollte ein separates FTR die stabile englische Functional-Model-
Integration schneiden. Dabei werden Inhalte referenziert statt dupliziert.

=== Content, der referenziert statt dupliziert werden sollte

* Die Grundmotivation aus `docs/semantic_model/03_problem_statement.adoc` sollte
  nicht zusätzlich im Semantic Contract Core wiederholt werden. Der Contract
  Core verweist darauf.
* Die externen Inspirationsquellen aus
  `docs/semantic_model/05_external_inspirations.adoc` bleiben Hintergrund. Die
  normativen Project-Interpretation-Anteile fließen nur dort in Contract-Kapitel
  ein, wo sie konkrete Regeln tragen.
* Roundtrip-Prozessbilder aus
  `docs/semantic_model/10_roundtrip_walkthroughs.adoc` und dem
  Problemraumkonzept sollten nicht doppelt beschrieben werden. Functional Model
  erklärt den Ablauf; Semantic Model definiert, welche Interpretationsregeln an
  den Prozessgrenzen gelten.

=== Content, der unverändert bleiben sollte

* Die FK-helper Pilot Matrix bleibt im Semantic Model. Sie ist bereits ein
  konkretes Contract-Artefakt.
* Registry Role bleibt im Semantic Model. Es beschreibt die Verbindlichkeit der
  Registry als Projektartefakt, ohne sie automatisch zur Runtime-Regel zu
  machen.
* Ambiguity Policy, Conflict/Precedence, Lifecycle, Versioning und Glossary
  bleiben Semantic-Model-Kern.
* Das Problemraumkonzept bleibt zunächst Warm Storage und deutsch, damit der
  Abstand zu bestehenden englischen Lösungsbegriffen erhalten bleibt.

=== Konkreter FTR-Vorschlag

==== FTR-FUNCTIONAL-SEMANTIC-MODEL-SPLIT-P5 -- Stabilize concept docs around Functional/Semantic split

*Status:* Proposal

*Priority:* P5

*Area:* Documentation / models

===== Purpose

Stabilize the model documentation around the distinction:

* Functional Model explains the problem space.
* Semantic Model defines the contract.

===== Scope

* Integrate the German problem-space concept into the English Functional Model
  as explanatory problem-space material.
* Keep Semantic Model contract chapters focused on vocabulary, guarantees,
  interpretation rules, lifecycle, precedence, registries, guards, matrices and
  versioning.
* Replace duplicate explanation in Semantic Model with cross-references where
  possible.
* Preserve the FK-helper pilot matrix and existing Semantic Model intent.
* Keep the current no-big-bang adoption strategy.

===== Non-goals

* No runtime behavior changes.
* No registry schema or registry content changes.
* No model implementation changes.
* No broad rewrite of the Functional, Technical or Semantic Model.
* No retroactive rewrite of historical FTRs or review artifacts.

===== Acceptance

* Functional Model has a clear problem-space entry path.
* Semantic Model has a clear contract-core entry path.
* Every current Semantic Model chapter is classified as contract core,
  explanatory context, adoption/planning or appendix.
* Explanatory content is referenced instead of copied into contract chapters.
* German Warm Storage source remains available as preserved input.
* AsciiDoc entry pages render without mixed Markdown headings or copied block
  artifacts.

===== Validation

* `asciidoctor --failure-level=ERROR docs/cold_storage/concepts/problemraum_konzept.adoc`
* `asciidoctor --failure-level=ERROR docs/semantic_model/semantic_model.adoc`
* `make docs-html`
