= Concept: Collaborative Repository Roundtrip

== Status

Draft concept.

This document explores a collaboration pattern observed during real-world
adoption. It is not currently a mandatory architectural rule.

== Problem

Spreadsheet-handling supports metadata roundtrip.

Metadata may originate from several sources:

* bootstrap/configuration steps,
* transformation results,
* workbook readback.

Examples:

* validations,
* workbook views,
* helper policies,
* column widths,
* text orientation,
* freeze panes,
* styling hints.

When multiple users collaborate through Git, persisted `_meta` can become a
source of merge conflicts.

This is especially problematic when the metadata was generated,
reconstructed, or derived rather than intentionally authored.

== Observation

The workbook itself is not problematic.

Workbook-embedded metadata participates in a single-user roundtrip and does
not normally create repository conflicts.

The problem appears when metadata is persisted into repository-managed JSON
artifacts and becomes a shared collaboration surface.

== Recommended Collaboration Pattern

For collaborative Git-based workflows:

## [source,text]

Canonical Data
+
Checked-in Pipeline Configuration
---------------------------------

## Shared Truth

Persisted `_meta` should be treated as generated state whenever possible.

Required metadata should be reproducible from:

* canonical data,
* checked-in configuration,
* deterministic transformation logic.

Pipelines should be capable of reintroducing required metadata during export.

== Implications

Losing persisted `_meta` should not normally destroy a collaborative project.

A fresh checkout should be able to reconstruct the required workbook behavior
through pipeline execution.

Examples:

* validation rules reintroduced through bootstrap steps,
* helper policies regenerated,
* workbook styling restored,
* presentation defaults re-applied.

== Non-Goals

This concept does not prohibit persisted metadata.

Some deployments may intentionally treat persisted metadata as a canonical
artifact.

The recommendation applies specifically to collaborative repository workflows
where merge safety and reproducibility are primary concerns.

== Open Questions

* Which metadata categories are realistically reproducible?
* Which metadata categories should remain local-only?
* Should the framework provide explicit support for local generated `_meta`
  artifacts?
* How should this pattern interact with the metadata lifecycle model?
