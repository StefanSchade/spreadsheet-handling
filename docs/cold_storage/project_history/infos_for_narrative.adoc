== Era 0 — Proof of Concept (Aug–Sep 2025)

The project started on 2025-08-29 as a quick Python proof of concept:
bidirectional conversion between JSON configuration data and Excel
spreadsheets. The original use case was world-building and structured story
data maintained in spreadsheets and consumed as JSON.

The earliest implementation focused on:

* JSON ↔ XLSX conversion
* preservation of column ordering
* suppression of unwanted pandas index columns
* first smoke tests
* simple CLI flows

The project was still strongly implementation-driven during this phase. The
primary goal was simply to make spreadsheet editing practical for structured
hierarchical data.

There was no explicit IR layer yet, no stable adapter abstraction, and no
formal architectural model. The codebase nevertheless already showed several
themes that would later become central:

* deterministic output
* roundtrip thinking
* metadata preservation
* separation between payload and operational metadata
* spreadsheet-centric editing workflows

== Era 1 — Early Pipeline and Metadata Thinking (Oct 2025)

During October 2025 the project evolved from a proof-of-concept converter into
something closer to a general spreadsheet transformation framework.

This period introduced many of the conceptual foundations that later became
architecturally important:

* pipeline-oriented workflows
* metadata bootstrap concepts
* ordered-list semantics
* nested structure handling
* workbook write-path thinking
* adapter abstraction planning
* demo and publication planning
* roadmap/FTR discipline

The repository also began developing explicit engineering process structure:

* backlog organization
* roadmap grouping
* release notes
* AI usage policies
* architecture and workflow documentation
* release/versioning strategy

The project was still highly exploratory, but the direction was already
becoming recognizable: deterministic spreadsheet roundtrip and transformation
rather than one-off spreadsheet export.

The Git history from this period shows the beginning of explicit architecture
thinking:

* `FTR-BACKEND-ABC` formalized backend abstraction ideas
* `FTR-IR-WRITEPATH-P1` prepared the Render IR direction
* metadata preservation entered the WorkbookIR layer
* the xlsx backend moved into its own subpackage
* project evolution and release semantics became documented explicitly

At the same time, the repository intentionally became AI-friendly:

* concept documents were introduced
* AI workflow guidance appeared early
* review discipline became explicit
* backlog entries captured deferred ideas instead of relying on memory

This was an important enabling decision for later large-scale AI-assisted
refactoring and review work.

== Era 2 — Architectural Expansion and Genericization

The following months were characterized by rapid capability expansion.

The project moved far beyond simple JSON ↔ XLSX conversion and accumulated a
large set of spreadsheet transformation and roundtrip capabilities:

* workbook rendering and reconstruction
* metadata-driven rendering
* workbook views
* foreign-key helper workflows
* compact multiaxis transforms
* crosstable editing flows
* discriminator split transforms
* validation and findings systems
* derived-column handling
* artifact manifest generation
* workbook recomposition
* pipeline orchestration
* backend abstraction layers
* ODS support

This period was intentionally ambitious and exploratory.

The architecture evolved quickly while feature surface expanded almost
continuously. Large modules appeared. Several concepts co-evolved in partially
implicit form:

* canonical vs derived metadata
* workbook reconstruction
* helper provenance
* rendering passes
* parser projection
* operational metadata
* transient vs persisted state
* spreadsheet layout semantics

The project nevertheless avoided several common failure modes:

* no large framework rewrite
* no generic "meta engine"
* no speculative plugin platform
* no external orchestration dependency
* no hidden runtime reflection layer

The architecture stayed pragmatic despite increasing sophistication.

== Era 3 — Architecture Consolidation and Executable Boundaries (Mar–May 2026)

After the earlier feature-heavy expansion waves, the project entered a
consolidation phase.

This was not a pause in development. Functional expansion continued rapidly,
including:

* workbook roundtrip support
* workbook-view recomposition
* compact multiaxis expansion
* sparse crosstable handling
* discriminator split transforms
* generated artifact manifests
* derived-column policy handling
* parser/reconstruction improvements
* demo integration work

However, the center of gravity gradually shifted from feature acquisition
toward architectural stabilization.

Several important structural transitions became defining characteristics of
this era:

* the Render IR became a real abstraction boundary;
* XLSX and ODS converged behind a shared spreadsheet contract facade;
* architecture boundaries became executable guardrails instead of conventions;
* registry ownership became increasingly explicit;
* package modularization became analysis-driven and behavior-preserving.

The repository increasingly adopted the principle that architecture should be
enforced by tests and guardrails rather than relying on contributor discipline
alone.

Representative examples included:

* parser contract guards
* semantic invariant tests
* meta/carrier boundary guards
* transformation package-boundary guards
* package-root API guards
* AST-equivalence based modularization discipline
* registry and layer boundary checks

The `fk_helpers` and `discriminator_split` modularization work became
important reference implementations for the project's refactoring methodology:

* analysis first
* acyclic dependency reasoning
* behavior-preserving moves
* package-root re-export preservation
* architecture guards introduced together with the split
* explicit rejection of speculative abstraction layers

This period also saw the repository become intentionally AI-operable.

The project increasingly assumed that work would continue across many isolated
AI sessions and therefore expanded:

* architecture primers
* concept documentation
* review workflows
* guard inventories
* snapshot tooling
* AI operating guidance
* project narratives and release documentation

The architecture was no longer merely modular. It had become increasingly
self-defending.

== REVIEW-001 — Structural Integrity and Architecture Hardening

REVIEW-001 focused primarily on architecture.

The review identified several important structural concerns:

* incomplete enforcement of hexagonal boundaries
* weakly protected adapter seams
* metadata serialization fragility
* module-size drift
* unclear ownership boundaries in some transformation areas

The resulting work emphasized concrete enforcement rather than broad rewrites.

Major outcomes included:

* executable layer-boundary enforcement
* spreadsheet contract formalization
* deterministic metadata serialization
* adapter isolation hardening
* registry discipline
* behavior-preserving modularization methodology

One of the most important outcomes was the transition:

> from “hexagonal by intention” to “hexagonal by enforcement”.

The ODS backend became especially important because it validated that the IR
and spreadsheet contract represented genuine abstractions rather than
XLSX-specific leakage.

By the end of the REVIEW-001 follow-up work, the project had become
substantially safer to evolve:

* refactorings were repeatable
* package boundaries were executable
* architectural drift became detectable
* adapter abstractions were exercised by multiple real implementations

The project increasingly resembled a stable evolving architecture rather than a
large prototype.

== REVIEW-002 — Semantic Pressure and Meaning Modeling

REVIEW-002 revealed a different class of problem.

The main architecture boundaries were no longer the dominant uncertainty.
Instead, the review exposed growing semantic pressure around:

* metadata meaning
* provenance
* reconstruction semantics
* lifecycle ownership
* canonical vs operational state
* parser observation vs semantic recovery
* transient vs persisted metadata
* layout intent vs layout observation

The project had reached a scale where implicit semantics were no longer
sufficient.

A representative example emerged during the IR-006 render-input mutation fix.
A mutation cleanup removed `legend_blocks[*].resolved` as apparently
render-local state. Independent review then demonstrated that the same field
also acted as persisted reconstruction-support metadata during XLSX readback.

The problem was not primarily architectural.

The meaning and authority of the field had never been modeled explicitly.

This led to an important project-level realization:

> The architecture had become too large for purely implicit semantics.

The next bottleneck was therefore no longer primarily package structure, but
explicit semantic modeling.

The project increasingly required vocabulary and conceptual distinctions such
as:

* canonical metadata
* derived metadata
* observed metadata
* reconstruction-support metadata
* rendering-local transient state
* carrier artifacts
* layout intent
* backend capability limitations
* metadata authority and provenance

The remaining risks were increasingly understood as meaning-modeling problems
rather than raw structural failures.

== Transition Toward Productization and Semantic Stewardship

By mid-2026 the repository had accumulated a sufficiently broad feature set to
function as a realistic spreadsheet transformation and roundtrip framework
rather than merely a proof-of-concept converter.

Capabilities now included:

* JSON/CSV/XLSX/ODS flows
* pipeline orchestration
* spreadsheet roundtrip support
* workbook reconstruction
* metadata-driven rendering
* validation and severity-policy systems
* compact multiaxis transforms
* discriminator split/merge transforms
* artifact-manifest generation
* FK-helper workflows
* workbook-view recomposition
* extensive architecture and semantic guardrails

This reduced the pressure for aggressive feature acquisition.

The near-term project strategy shifted toward:

. demo integration and beta usability
. user-facing documentation and tutorials
. repository cleanup and archival discipline
. explicit semantic-model clarification
. later targeted refactorings after semantic stabilization
. additional feature expansion only after the semantic model became clearer

The project was no longer primarily fighting code chaos.

Instead, it had entered a phase where the central engineering challenge became:

> defining and stabilizing the meaning of the system.

The architectural assessment at this stage was:

> stable-beta architecture with localized, repayable maintainability debt.

The project had not stalled halfway toward its goals.

Instead, it had reached a point where the remaining challenges increasingly
concerned semantic clarity, reconstruction authority, metadata meaning, and
long-term conceptual stewardship rather than raw implementation capability.
