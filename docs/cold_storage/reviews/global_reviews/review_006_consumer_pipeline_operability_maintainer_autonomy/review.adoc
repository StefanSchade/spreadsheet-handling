== Review 006 -- Consumer Pipeline Operability and Maintainer Autonomy

*Status:* Complete (read-only review; no code changes)

*Date:* 2026-07-06

*Baseline:* core `22ed328` (post Review-005 closure `69c4cbc` line). Review 005
is closed and paused; this review deliberately does not continue the
domain-contract/meta-registry work.

*Core question:* How hard is it for a human maintainer or power user to
understand, adjust, and safely operate consumer pipelines without reading
custom plugin code or asking AI?

*Method:* Read-only inspection of pipeline YAML, consumer plugin code,
Makefiles, dependency state, and the framework step surface
(`pipeline/build.py`, `pipeline/registry.py`, `registries/pipeline_step_registry.json`).
No import/export/promote commands were run; no data was read beyond
pipeline/config/plugin files. In `personal_data` only pipeline and plugin
files were inspected; domain data files were not opened, and schema facts
cited here come from pipeline YAML only.

*Prior art honored:* `docs/cold_storage/reviews/topic_reviews/CONFIGURATION-SIMPLIFICATION-FTR-ASSESSMENT.adoc`
(single-consumer-era assessment of the same friction),
`FTR-MODEL-DEFAULT-CONFIGURATION-P5` (open, broad),
`FTR-YAML-SURFACE-NORMALIZATION-P5` (open),
`FTR-DYNAMIC-VIEW-COLUMN-TARGETING-*` (done),
`FTR-DYNAMIC-FRAME-PROJECTION-IMPL-P5` (project_by_role line).

=== 1. Executive assessment

*Yes, with one significant exception and one dogfood caveat: the real consumer
fleet is currently maintainable by a human without AI.*

The dominant real-consumer shape is a 2-3 step forward pipeline
(`validate_references` -> `configure_workbook_view` -> `add_validations`) and a
2-step reverse pipeline (`apply_workbook_view_sheet_mappings` ->
`validate_references`), using only built-in registered steps and zero plugins.
`personal_data` (7 datasets + project workspaces) and `home-lab` (6 datasets)
follow this shape uniformly. A maintainer can read any of these files top to
bottom and know exactly what happens. This is a genuinely good result and
should be recognized as the project's healthy baseline.

The friction is concentrated, not diffuse:

1. *The single largest friction is repeated schema knowledge, not plugin
   opacity.* Every consumer duplicates its primary-key/foreign-key/uniqueness
   rule block verbatim between the forward and reverse pipeline (and
   worldbuilding repeats it a third time mid-reverse). The schema of each
   dataset lives in two or three places that must be synchronized by hand.
   This is the highest-value, lowest-risk fix, and it is the exact narrow
   slice the CONFIGURATION-SIMPLIFICATION assessment said should wait for a
   second real consumer. That second (and third, and fourth) consumer now
   exists.

2. *worldbuilding/dino_insel is the one severe real-consumer pipeline* (39 +
   25 steps, 360 + 210 lines). Its bulk is a 4x copy-pasted facet motif
   (extract -> join -> contract_xref -> join -> project_by_role, mirrored on
   reimport). It is legible but exhausting to audit, and its four plugin calls
   are all generic table operations, not domain logic.

3. *The in-repo dogfood consumers, not the real consumers, are the
   AI-interpreter hotspots.* `registries/domain_contracts` pipelines are 100%
   plugin steps; `project_memory`'s roundtrip pipelines are 3-of-4 and
   1-of-2 plugin steps with reserved `__base_*` frame conventions. Their YAML
   communicates nothing beyond function names; safe operation requires reading
   328-750-line Python modules.

4. *Workflow safety is good.* The export/check/diff/promote Makefile
   convention is uniform across consumers, tmp-staged, and hard to misuse.
   Version drift is a documentation problem (demo's stale pin), not a runtime
   problem: every consumer venv runs the local core checkout editable.

Main friction ranking: (1) duplicated model facts in YAML, (2) one severe
pipeline's copy-paste volume, (3) dogfood plugin opacity, (4) stale demo
surface, (5) missing tiny generic steps (drop columns / select frames) that
three codebases have independently reimplemented as plugins.

=== 5. Main findings

==== F1 -- The standard consumer pattern is healthy and human-operable (positive finding)

*Evidence:* 30 of 32 real-consumer roundtrip pipelines are 2-3 built-in steps,
plugin-free, with a uniform Makefile lifecycle (`export` / `import` / `diff` /
`check` / `promote`) in worldbuilding, personal_data, and home-lab. The
personal_data plugin package is empty.

*Consumer impact:* A maintainer can open any of these YAML files and read the
whole operation. The `promote` step only copies from a tmp reimport dir that
`check` has diffed, which is a safe-by-construction loop.

*Cause:* current framework surface (validate_references /
configure_workbook_view / add_validations / apply_workbook_view_sheet_mappings
cover the whole standard need).

*Direction:* none -- protect this baseline; measure any new feature against
"does it keep the 3-step pipeline a 3-step pipeline".

*Confidence:* high.

==== F2 -- Schema facts are duplicated 2-3x per dataset across pipeline pairs (primary friction)

*Evidence:* The `validate_references` rule block is repeated verbatim between
forward and reverse pipelines in *every* dataset:
`personal_data/pipelines/beruf_meta/json_to_spreadsheet.yaml:11-30` equals
`spreadsheet_to_json.yaml:9-28` (18 rules); same pattern in all 7
personal_data datasets, both project workspaces, all 6 home-lab datasets, and
worldbuilding (which carries the block *three* times:
`json_to_spreadsheet.yaml:20-42`, `spreadsheet_to_json.yaml:19-35` (subset),
and `spreadsheet_to_json.yaml:183-205` (superset)). Fleet-wide this is roughly
16 duplicated rule blocks, several hundred lines whose divergence would be
silent.

*Consumer impact:* Adding a frame or FK means editing 2-3 rule sites per
dataset by hand; a missed site silently weakens reimport validation (the
dangerous direction: canonical data gets promoted after a check that no longer
tests the new constraint). This is precisely the "maintainer must synchronize
several files mentally" failure the review is hunting.

*Stale or current?* Current framework surface: `validate_references` accepts
only an inline `rules:` list (`reference_validations.py:72-79`); there is no
way to name a ruleset once and reference it, although the sibling
`apply_overrides` step already has the `overrides_path` precedent.

*Direction:* the Section 9 slice (shared ruleset file via `rules_path`).

*Confidence:* high (proven in three real consumers).

==== F3 -- dino_insel's faceted-matrix fan-out is the one severe real pipeline

*Evidence:* forward `json_to_spreadsheet.yaml:100-262`: the 5-step motif
extract_frame -> join_frames(name) -> contract_xref -> join_frames(title) ->
project_by_role is copy-pasted 4x (facets `role`, `notes`,
`character_development`, `crystal`), differing in one string; the reverse
mirrors it with 4x project_by_role -> expand_xref -> join_frames and a single
`merge_by_discriminator` (`spreadsheet_to_json.yaml:177`). ~200 of 360 forward
lines are this repetition.

*Consumer impact:* auditing means diffing near-identical blocks by eye; adding
facet #5 means writing ~35 lines in two files that must mirror each other.
However -- as the CONFIGURATION-SIMPLIFICATION assessment already concluded --
the sequence is architecturally legible: every stage (split, label join, dense
axes, title join, projection) is visible and diagnosable from YAML.

*Stale or current?* Current framework surface; not version drift. One concrete
oddity: the *reverse* uses the built-in `merge_by_discriminator`, while the
*forward* hand-rolls the inverse as 4x `extract_frame` even though
`split_by_discriminator` (its registered bijective inverse,
FTR-SPLIT-BY-DISCRIMINATOR-P4A) exists. A consumer-side rewrite could replace
the four extract_frame steps with one split_by_discriminator step today, with
zero framework change (4 steps -> 1; column projection to
`[story_id, character_id, value]` still needed per facet or via value_map
semantics -- to be verified against the step's parameter contract before
migrating).

*Direction:* (a) consumer-side: try split_by_discriminator in the forward
pipeline; (b) framework-side: continue to *withhold* the faceted-matrix macro
(one consumer, one relation -- unchanged since the prior assessment); (c) F2's
ruleset slice removes ~60 of the 570 lines on its own.

*Confidence:* high on the evidence; medium on the split_by_discriminator
substitution (parameter fit not executed, read-only review).

==== F4 -- Three codebases independently reimplement the same two generic frame ops as plugins

*Evidence:* drop-columns exists as `character_views:drop_columns`
(worldbuilding), `queries:strip_concern_event_matrix_context`
(project_memory), and `frame_cleanup:drop_helper_like_columns` (ssdp repos,
filename-level evidence). Select-frames-for-output exists as
`frame_sets:keep_frames` (worldbuilding) and `queries:derived_views_only`
(project_memory). Plus trivial `copy_frame` and `ensure_frame_columns`.

*Consumer impact:* each occurrence forces the reader out of YAML into Python
for an operation whose whole semantics is one line; each consumer must also
re-learn the `dict(frames)` copy discipline and `_meta` preservation rule
(frame_sets:keep_frames explicitly hand-carries `_meta` -- a correctness trap
each plugin author must rediscover).

*Stale or current?* Current framework surface: the registry has no
column-drop or frame-set-selection primitive (extract_frame projects columns
but always writes a new output frame and cannot say "all columns except X";
configure_workbook_view selects sheets for rendering but not frames for
json_dir export).

*Direction:* after the F2 slice lands, a follow-up micro-FTR for `drop_columns`
+ `select_frames` steps is justified by the rule of three. Deliberately *not*
the first slice: it reduces plugin count (a non-goal by itself) more than it
reduces synchronization risk.

*Confidence:* high for the duplication; medium for priority.

==== F5 -- Dogfood pipelines are the actual "AI required as interpreter" cases

*Evidence:* `registries/domain_contracts/pipelines/*.yaml` are 100% plugin
steps (4 dotted calls into 328-line `workbook.py`, zero parameters in YAML).
`project_memory/pipelines/memory/ods_to_json.yaml` is 3-of-4 plugin steps
around one expand_xref, with a reserved `__base_concern_event_xrefs` frame
created by `copy_frame`, consumed via `base_relation`, and cleaned up inside
`finalize_concern_event_xrefs` together with `_meta.xref_crosstable` pruning
-- none of which is visible in the YAML. The concern-ID list is additionally
duplicated between forward `contract_xref.column_keys`
(`json_to_ods.yaml:16-23`) and reverse `expand_xref.value_columns`
(`ods_to_json.yaml:24-31`), and both must track the `concerns` frame content.

*Consumer impact:* these are the pipelines where the maintainer demonstrably
cannot answer "what does my own pipeline do" from config. They are also the
complexity proxy for what real consumers would look like if plugin-first
authoring became the norm.

*Stale or current?* Current, but deliberately so: domain_contracts is Review
005 territory (paused), and project_memory's finalize logic is genuine
roundtrip identity policy. The *generic* fraction (copy_frame, strip context
columns, enrich = join_frames) is the same F4 gap.

*Direction:* no registry/framework work now. Two cheap mitigations when these
files are next touched anyway: (a) replace `enrich_concern_event_matrix` with
the built-in `join_frames` it re-implements, (b) one comment line per plugin
step in the YAML stating its contract ("copies X so expand_xref can diff
against a pre-edit base"). Do not resume Review 005 for this.

*Confidence:* high.

==== F6 -- Forward/reverse pairs require mental synchronization beyond validation rules

*Evidence:* per dataset the maintainer must keep aligned: (a) frame lists --
`configure_workbook_view.sheets` vs
`apply_workbook_view_sheet_mappings.logical_frames` (e.g. home-lab network
6-frame list in both files); (b) xref parameters -- contract_xref vs
expand_xref (relation/matrix names, row_keys, column_key, value, legend names)
in worldbuilding and project_memory; (c) personal_data `projekte/_template` vs
per-project copies (currently byte-identical, guaranteed to drift silently
after the first project-specific edit); (d) Makefile `DATASETS` lists vs
`pipelines/` directories; (e) demo `pyproject.toml` pin vs Makefile-installed
editable core.

*Consumer impact:* every model change fans out to 2-6 hand-edited locations.
Today the fan-out is manageable because datasets are small; it is the growth
axis to watch.

*Stale or current?* Structural consequence of the (sound) explicit two-file
pipeline design; only (c) and (e) are consumer-side hygiene issues.

*Direction:* F2's slice removes the largest single synchronization (rules).
The frame-list pairing (a) is *acceptable* explicitness: the two lists answer
different questions (render vs readback boundary) and inference was already
rejected in the FTR record. Leave (b) explicit per prior assessment. For (c),
a one-line README note in `projekte/_template` ("template is copied, not
referenced; diff against _template after edits") is enough for now.

*Confidence:* high.

==== F7 -- home-lab's entire pipeline surface is untracked in git

*Evidence:* `git status` shows `pipelines/`, `data/`, `scripts/compare_reimport.py`,
`templates/`, `STRUCTURED_DATA.adoc` untracked; Makefile/AGENT.md modified;
last commit 2026-05-10 predates all of it.

*Consumer impact:* the most operations-relevant consumer (homelab inventory /
network / power) has no versioned or recoverable pipeline definitions; a
stray `git clean` or disk issue loses them. This is a pure operability risk
independent of the framework.

*Stale or current?* Consumer-side workflow state, not framework.

*Direction:* commit (or consciously .gitignore data/ while committing
pipelines/). Recommend but do not perform (read-only review).

*Confidence:* high.

==== F8 -- demo is drifting from the current step surface

*Evidence:* demo pins `spreadsheet-handling==0.1.0b6` in `pyproject.toml`
while its venv runs editable core; `demo_product_business_slice.yaml` uses the
`validate` step (registry status: *transitional*, superseded in practice by
`validate_references`) with convention flags (`detect_fk: true`,
`helper_prefix`) that current consumers no longer use; last commit 2026-05-25.

*Consumer impact:* demo is the public teaching surface; a new user copying
demo patterns learns the transitional API instead of the pattern all real
consumers use. Not a runtime failure -- a message failure.

*Stale or current?* Stale consumer state (recorded as such per the review's
caveat rule; not evidence of framework failure).

*Direction:* demo realignment pass after the F2 slice (so demo can teach the
ruleset pattern too). Out of scope here.

*Confidence:* high on facts; medium on urgency.

==== F9 -- Failure-mode diagnosability is decent for built-ins, absent for plugins

Which failures can be diagnosed from YAML alone (spot-check of error paths in
`build.py` and step purposes in the step registry):

* unknown step name -> error lists all registry keys (`build.py:32`);
* non-string YAML key (the `on:` boolean trap) -> targeted hint (`build.py:55-69`);
* validate_references in `mode: fail` -> failure message carries rule facts
  that map 1:1 to YAML lines;
* missing FK helper policy -> error names `configure_fk_helpers` /
  `infer_fk_relations` (per registry purpose entries);
* wrong/renamed frame in extract/join/xref steps -> "Expected DataFrame 'x'"
  traces to a YAML parameter.

Which require opening Python: *any* plugin step failure (raw traceback into
consumer code); *why* `strip_trailing_header_separator` must exist
(flatten/unflatten join-mode artifact); everything in the dogfood pipelines
(F5); `merge_by_discriminator` source_pattern mismatches (the pattern
`story_cast_{value}_cells` is in YAML, but knowing which frames matched
requires runtime knowledge).

*Direction:* no action beyond F2/F4; the built-in error surface is already
maintainer-friendly.

*Confidence:* medium (static reading, not fault injection -- read-only review).

=== 6. What is acceptable plugin usage?

Plugins are the *right* tool, and should remain plugins, when the logic is:

* *external-system adapters* -- e.g. home-lab's `render_adoc.py` style report
  rendering (kept outside the pipeline entirely, even better), a future
  CMDB/API push;
* *domain-specific report generation* -- project_memory's heatmaps, blocker
  joins with scoring bands and interpretation thresholds; the value *is* the
  domain judgment encoded in Python;
* *roundtrip identity policy that is genuinely local* --
  `finalize_concern_event_xrefs` (id synthesis scheme, note preservation);
* *one-off extraction / teaching examples* -- demo's extraction plugins exist
  to demonstrate the extension point;
* *temporary experiments* -- anything expected to be deleted or promoted
  within a release cycle (the `story_cast_views` retirement shows the promote
  path works: plugin -> FTR -> registered step -> plugin deleted).

Operational bar for an acceptable plugin: the YAML step's `dotted:` name plus
`args:` should state *what* it does and *which frames/columns it owns*, even
if the *how* lives in Python; frame-set changes should be visible in args, not
in Python defaults (negative examples: `derived_views_only` and
`strip_concern_event_matrix_context`, whose frame/column lists are Python
default arguments invisible to a YAML reader).

=== 7. What is overgrown configuration?

Patterns observed in this fleet that make YAML hard to maintain:

* *repeated model facts* -- the same PK/FK/unique rules in 2-3 files per
  dataset (F2); the same concern-ID list in forward and reverse xref steps
  (F5);
* *copy-paste step motifs varying in one token* -- the 4-facet fan-out (F3);
  the reader must diff blocks to discover they are identical;
* *implicit lifecycle coupling* -- reserved frames (`__base_*`) created by one
  plugin, consumed by a step three lines later via `base_relation`, and
  destroyed inside a third plugin, with the contract nowhere in the file (F5);
* *hidden joins and hidden column sets* -- joins and drop-lists living as
  Python defaults instead of YAML args (`enrich_concern_event_matrix`,
  `strip_concern_event_matrix_context`);
* *duplicated view definitions across the forward/reverse boundary* --
  contract_xref/expand_xref parameter mirroring without a shared name;
* *plugin step names that hide a generic operation* --
  `strip_concern_event_matrix_context` is drop-columns;
  `derived_views_only` is select-frames; the name suggests domain logic where
  none exists, which *raises* the reading cost;
* *pipelines where safe operation requires reading Python* -- both
  domain_contracts pipelines (100% plugin) are the limit case.

Deliberately *not* overgrown (verbose but correct): the explicit
`helper_columns`/`key_columns` on every project_by_role call (rationale
recorded in FTR-DYNAMIC-FRAME-PROJECTION-IMPL-P5); the paired frame lists in
configure_workbook_view / apply_workbook_view_sheet_mappings (different
boundaries); per-sheet in-list validations in add_validations (domain
vocabulary belongs in the dataset's own file); dense_axes declarations on
contract_xref (they encode a real modelling decision -- dense vs sparse axes).

=== 9. Candidate first implementation slice

*Slice: named shared rulesets for `validate_references` via `rules_path`
(one dataset schema file, referenced by both pipeline directions).*

*Problem statement:* Every consumer duplicates its dataset's structural facts
(primary keys, uniqueness, foreign keys) as identical `validate_references`
rule blocks in forward and reverse pipelines (worldbuilding: three sites).
Model changes require synchronized hand edits; a missed reverse-side edit
silently weakens the reimport check that gates `promote`. The framework
already has the file-reference idiom (`apply_overrides.overrides_path`), so
this adds no new concept -- only the missing symmetry.

*Proposed surface (exact, minimal):*

[source,yaml]
----
# pipelines/beruf_meta/reference_rules.yaml  (one file per dataset)
rules:
  - { type: primary_key, frame: phases, columns: [id] }
  - { type: foreign_key, frame: situations, columns: [phase_id],
      target: phases, target_columns: [id], allow_empty: false }
  # ... unchanged rule schema, plain list ...
----

[source,yaml]
----
# both json_to_spreadsheet.yaml and spreadsheet_to_json.yaml
- step: validate_references
  mode: fail
  rules_path: ./pipelines/beruf_meta/reference_rules.yaml
----

Semantics: `rules_path` loads the YAML file's top-level `rules` list and
validates it with the identical code path as inline `rules`. When both
`rules` and `rules_path` are given, concatenate file rules first, then inline
rules (supports worldbuilding's reverse-pipeline superset case: shared base
from file + post-recomposition additions inline). Paths resolve relative to
the pipeline YAML's directory (same convention as other consumer-relative
paths). No includes-of-includes, no templating, no rule generation.

*Exact affected files:*

* `core/src/spreadsheet_handling/domain/validations/reference_validations.py`
  -- add optional `rules_path: str | None` to `validate_references`
  (signature at line 72), plus a small loader with clear errors (missing
  file, missing/empty `rules` key, non-list);
* `core/src/spreadsheet_handling/pipeline/build.py` -- likely no change
  (kwargs pass through); verify base-path resolution: if step factories do not
  receive the config file's directory, thread it or resolve relative to CWD
  and document that the Makefile convention (`--config` run from repo root)
  makes CWD-relative paths stable -- decide during implementation and record
  the decision in the FTR;
* `core/registries/pipeline_step_registry.json` -- `validate_references`
  entry: add the `rules_path` parameter (`source_kind` matching the
  registry's file-reference convention as used by
  `apply_overrides.overrides_path`);
* `core/docs/user_guide/ch03_transformation_catalog/` -- document the shared
  ruleset pattern (supporting-steps chapter);
* `core/docs/release_notes/release_notes.adoc` -- bullet under the active
  unreleased section (public YAML surface change);
* backlog: record as the narrow refinement slice of
  `FTR-MODEL-DEFAULT-CONFIGURATION-P5` (per the CONFIGURATION-SIMPLIFICATION
  assessment's recommendation to narrow, not compete).

*Likely tests:*

* `core/tests/unit/domain/validations/test_reference_validations.py` --
  extend: rules_path-only; rules_path+inline concatenation (order and
  duplicate tolerance); missing file -> actionable error naming the path and
  the step; file without `rules` key / empty list -> clear error; relative
  path resolution;
* one integration-level test running a two-step pipeline YAML from a tmp dir
  proving both directions can share one ruleset file (mirrors the consumer
  layout).

*Acceptance criteria:*

* a personal_data-shaped consumer can delete both inline rule blocks, add one
  `reference_rules.yaml`, and `export`/`check` behave identically (same
  failures for same violations, same findings frame in warn mode);
* inline `rules` behavior is byte-for-byte unchanged when `rules_path` is
  absent (zero migration pressure);
* worldbuilding's reverse superset case is expressible as file + inline
  without duplicating the base block;
* error text for a bad path/file names the pipeline step and the resolved
  absolute path;
* expanded rule provenance is inspectable: in warn mode findings (or verbose
  logging) it is possible to see that rules came from a file (satisfies the
  prior assessment's inspectability requirement).

*Risks:*

* path-resolution ambiguity (config-relative vs CWD-relative) -- the one real
  design decision; must be documented and tested either way;
* scope creep toward a model-fact DSL ("since we have a schema file, let's
  also generate helpers/views from it") -- explicitly out of scope; the file
  contains literal `validate_references` rules and nothing else;
* a second place to look when reading a pipeline -- mitigated by the file
  being plain rules in the same schema, named in the step that uses it;
* silent divergence if consumers only migrate one direction -- migration note
  should recommend deleting inline blocks in the same change.

*Explicit non-goals:*

* no rule generation or inference from frames/columns/names;
* no shared-ruleset *registry* (a plain consumer-owned file, no core registry
  additions);
* no cross-step reuse (helpers, workbook views, projections stay untouched);
* no changes to `add_validations` (cell-level in-list rules stay inline: they
  are per-sheet domain vocabulary, not duplicated schema facts);
* no include mechanism inside ruleset files;
* no deprecation of inline `rules`.

*Why this slice and not the alternatives:* it is the only candidate that (a)
appears in *all three* real consumers, (b) reduces silent-divergence risk
rather than just line count, (c) has an in-repo surface precedent
(`overrides_path`), (d) was already identified as the safest direction by the
prior topic assessment -- which explicitly waited for the multi-consumer
evidence this review now supplies, and (e) is implementable in one small,
independently shippable change. The runner-up (generic `drop_columns` /
`select_frames` steps, F4) reduces plugin count but not synchronization risk,
and "fewer plugins" is a non-goal by itself.
