== 1. Verdict in Three Sentences

The spreadsheet adapter boundary is *structurally sound and correctly sized*:
`spreadsheet_contract.py` is a genuinely thin facade, both backends delegate
through it under executable import guards, backend formula syntax is truly
confined to adapters, and `RenderPlan`/`WorkbookIR` are backend-neutral and
guarded -- ODS is a real second backend, not an XLSX-shaped accident. The
residual risk is *not* in the contract shape but in three small generic
spreadsheet semantics that leak *sideways as duplicated, unowned, already-drifted
constants* across the adapter boundary (the workbook-option vocabulary, the
`" / "` multi-index header convention, and an XLSX-only carrier->canonical
column-width promotion), none of which is protected by a behavioral parity
guard at the architecture tier. All adapter-boundary risk in this pass is
*accidental and narrowly repayable* sitting on top of genuinely inherent
spreadsheet roundtrip complexity; nothing here justifies a package split, a
generic backend abstraction, or a generic parser framework.

== 3. Concrete Findings

=== P3-01 (MEDIUM, accidental debt) -- XLSX parser promotes a carrier observation into the persisted canonical meta blob; ODS does not; the stated "no carrier promotion" invariant does not cover it

*What.* On the XLSX read path, authored column widths are read from the carrier
(`openpyxl_parser.py:126 _extract_column_widths`, gated on
`dim.customWidth`), merged into a *canonical-looking* location
`workbook_meta["sheets"][sheet]["column_widths"]`
(`openpyxl_parser.py:152 _merge_column_width_meta`), and then the *entire*
workbook meta is re-serialized as JSON back into the hidden `_meta`
(`openpyxl_parser.py:176 _store_workbook_meta`). `workbookir_to_frames`
then returns that blob verbatim as `frames["_meta"]`
(`workbook_projection.py:42-49`). The ODS parser has *no* equivalent path.

*Why it matters.* The parse contract is explicit: "parser-side hints must not
silently override canonical meaning"; "carrier-side hints remain explicit and
quarantined rather than becoming implicit canonical semantics"
(`spreadsheet_parse_contract.adoc`, Boundary Invariants). Column widths *are*
already quarantined correctly as the `__column_widths` hint
(`parser_interpretation.py:90-92`, `openpyxl_parser.py:68`) -- and that part is
right. The *additional* merge into `sheets.<name>.column_widths` + blob
re-store is a second, contradictory action: it promotes a carrier observation
into the persisted canonical payload on read. A read->write XLSX roundtrip
therefore silently *adds* `column_widths` to canonical meta that never
originated from the domain/canonical source; an ODS roundtrip of the same file
does not. This is the single clearest XLSX-centric read-path assumption that
crosses the carrier->canonical boundary the contract forbids.

*Guard gap.* The strongest stated invariant -- "roundtrip restores canonical
meta *without promoting carrier hints*" -- exists in three places
(`test_spreadsheet_parse_contract.py:117`,
`test_spreadsheet_semantic_invariants.py:149`,
`test_meta_carrier_boundary_guards.py:69`) but every fixture writes default
column widths (`customWidth` is False), so `_merge_column_width_meta` returns
early and `_store_workbook_meta` is never reached. The invariant is real and
correctly stated; its fixture coverage has a hole *exactly* at the only code
path that violates it. The cross-adapter parity test has the same hole: it
asserts `xlsx_back["_meta"] == ods_back["_meta"]` only because no fixture
authors a custom width (`test_ods_xlsx_parity.py:202,242,...`).

*Classification.* Accidental architectural debt, bounded to XLSX and to
authored-width inputs. The quarantine principle exists and is followed
elsewhere; this one path bypasses it. Not inherent: nothing about spreadsheet
roundtrip requires the parser to mutate the canonical blob.

*Disposition.* This is a *decision*, not a refactor: either (a) column widths
are not canonical -> stop the blob re-store, keep them only as the
`__column_widths` hint; or (b) they are canonical for XLSX -> document the
asymmetry in the capability profile and add an authored-width roundtrip guard.
Prefer (a); it restores the invariant the project already wrote. Do *not*
generalize column widths into a cross-backend canonical field (capability
profile classifies column-width as XLSX-only/partial).

=== P3-02 (MEDIUM, accidental debt) -- The workbook-option vocabulary is triplicated and already divergent across the adapter boundary

*What.* The set of `meta_canonical` keys that project into
`SheetIR.meta["options"]` is defined three times and is *not* the same set:

* `composer/layout_composer.py:9 _WORKBOOK_OPTION_KEYS` =
  `{freeze_header, auto_filter, header_fill_rgb, helper_fill_rgb, helper_prefix}`
  (5 keys, write/compose path).
* `xlsx/parser_interpretation.py:12 OPTION_HINT_KEYS` and
  `ods/parser_interpretation.py:10 OPTION_HINT_KEYS` = the same 5 *plus*
  `helper_columns` and `protection` (7 keys, read path, byte-identical between
  the two adapters).

*Why it matters.* This is a generic `meta_canonical -> options` projection
vocabulary -- a spreadsheet-neutral semantic, not adapter-local interpretation.
The divergence is already observable: top-level (workbook-scoped)
`protection`/`helper_columns` are surfaced as inheritable options on the *read*
path but are *not* seeded as workbook-level defaults on the *write/compose*
path (`layout_composer.py:251-255`). It is currently masked only because every
fixture places `protection`/`helper_columns` under `meta["sheets"][name]` (the
sheet overlay at `layout_composer.py:296-302`), never at workbook scope (e.g.
`test_ods_xlsx_parity.py:476-492` puts `helper_columns` under
`sheets.Variables`). A workbook-level `helper_columns` would round-trip
asymmetrically today.

*Guard gap.* No guard pins read-vocabulary == write-vocabulary.
`test_sheet_option_precedence...` (`test_spreadsheet_semantic_invariants.py:126`)
exercises only the composer's 5-key subset.

*Classification.* Accidental debt + hidden semantic seam. The *adapter-local
heuristic bodies* in the two `parser_interpretation` modules are deliberate
pragmatism (the parse contract explicitly permits adapter-local interpretation
helpers and forbids a generic parser framework) -- those should stay
duplicated. But the *vocabulary list itself* is generic and should have one
owner.

*Disposition.* Narrow seam hardening: a single shared constant owning the
option vocabulary, referenced by composer and both `parser_interpretation`
modules, plus a one-line guard asserting the three references resolve to the
same set. This is *not* a parser-framework and *not* a package split.

=== P3-03 (MEDIUM, accidental debt) -- The `" / "` multi-index header convention is an unowned magic-string contract duplicated across four modules, with a write/read asymmetry and a drift-inviting comment

*What.* The MultiIndex header join/split separator `" / "` appears literally in
four modules with no shared constant (grep-confirmed: no `HEADER_SEP`-style
symbol anywhere in `src/` or `tests/`):

* `composer/layout_composer.py:88` -- joins *all* tuple parts:
  `" / ".join(map(str, tup))`.
* `xlsx/parser_interpretation.py:126` -- joins *non-empty* parts only.
* `ods/parser_interpretation.py:117` -- joins *non-empty* parts only.
* `workbook_projection.py:30-31` -- splits on `" / "` to rebuild a
  `pd.MultiIndex`.

The composer comment at `layout_composer.py:83` literally says "adjust if you
prefer another joiner" -- an explicit invitation to the exact drift that would
silently break parser/projection roundtrip.

*Why it matters.* This is a generic spreadsheet semantic (header flattening
convention) spanning the adapter boundary and the generic projection, with no
single owner. The practical roundtrip-bearing path is parser->projection (the
*write* path renders multi-level headers from `__header_grid`, not from the
joined string -- `flow.py:127-143`, so live risk is bounded), and the
join-all-vs-join-non-empty asymmetry already makes trailing-empty header levels
lossy -- but that loss is explicitly inside the parse contract's "optional /
incremental layout recovery" zone, so it is *not* a correctness defect today.
The architectural defect is the *unowned, duplicated, drift-invited* magic
string itself.

*Classification.* Accidental debt + hidden semantic seam. Bounded (the lossy
asymmetry is contract-permitted), but the seam is real and unguarded.

*Disposition.* One shared constant referenced by all four sites; delete the
drift-inviting comment. Do *not* build a header-codec abstraction; do *not*
attempt to make trailing-empty levels lossless (that is contract-deferred
layout recovery).

=== P3-04 (LOW, accidental debt) -- ODS renderer silently drops `SetColumnWidth` while it explicitly skips protection ops

*What.* `odf_renderer.py:309-313` skips `SetSheetProtection`/`ApplyCellLock`
with explicit `continue # ODS ... not implemented in this slice` comments.
There is *no* branch for `SetColumnWidth` at all -- it falls off the end of the
`_collect_sheets` dispatch silently (`odf_renderer.py:231-315`). Column-width is
a documented accepted ODS gap (capability profile; FTR-ODS-CONSUMER-FEATURE-
PARITY-P4A), but the capability profile requires accepted gaps to be "explicit
allowed differences instead of ... undocumented skips."

*Why it matters.* The skip is *implicit and uncommented*, unlike the protection
gap which has an explicit branch and a dedicated documented-gap test
(`test_ods_xlsx_parity.py:528 test_protection_is_xlsx_only_ods_documented_gap`).
No equivalent column-width gap test exists; the gap is asserted only in prose.

*Classification.* Accidental, trivially repayable. Acceptable *asymmetry* in
behavior (the gap itself is fine and profiled); the *code expression* of the
gap is inconsistent.

*Disposition.* Add an explicit no-op `SetColumnWidth` branch + comment mirroring
the protection pattern, and a documented-gap test mirroring
`test_protection_is_xlsx_only_ods_documented_gap`. Trivial; no architecture
change.

=== P3-05 (MEDIUM, carry-over IR-006 elevated to adapter scope) -- The shared write facade mutates caller inputs and the multi-backend path concretely reaches it

*What.* The two IR-006 mutations Pass 1 flagged are confirmed live at the exact
lines, and both sit *inside* `build_spreadsheet_render_plan` -- the single
facade *both* adapters call:

* `frame_selection.py:198` -- `value[col] = series.apply(...)` mutates caller
  frames in place (only under a `workbook_view` sheet-rename).
* `layout_composer.py:230` -- `spec["resolved"] = {...}` writes back into the
  caller's `meta` dict (legend resolution; this write is *read back* by both
  parsers' `_legend_table_hints`, so it is load-bearing for the legend
  roundtrip *and* a shared-state mutation).

*Pass 3 contribution.* This is not new, but Pass 3 establishes it is
*concretely reachable at the adapter boundary*: the parity harness
`_write_both` (`test_ods_xlsx_parity.py:101-108`) calls
`ExcelBackend().write_multi(frames)` then `OdsBackend().write_multi(frames)`
on the *same* `frames`/`meta` objects. The first
`build_spreadsheet_render_plan` mutates them; the second consumes the mutated
state. It is currently benign (legend `resolved` re-write is idempotent for
identical placement; the formula-sheet rewrite only fires under view renames),
but it is a latent correctness hazard precisely on the multi-output path this
pass is scoped to.

*Classification.* Accidental debt (input mutation in a pure-by-contract render
path). Consistent with Pass 1/2 disposition: a *surgical purity fix*
(copy-on-write of the selected frames; relocate the `resolved` write off the
caller meta), explicitly *not* a refactor and *not* a modularization target.

=== P3-06 (LOW, deliberate pragmatism -- watch item, not debt) -- ODS hidden-sheet detection hard-codes the carrier name `_meta`

*What.* `odf_parser.py:109-113 _table_is_hidden` treats a table as hidden if it
is styled `display:false` *or* its name is literally `"_meta"`. XLSX uses the
real `ws.sheet_state` (`openpyxl_parser.py:46`).

*Assessment.* This is acceptable, deliberate pragmatism: the capability profile
mandates the hidden `_meta` table as the ODS canonical-meta carrier, so a
name-based fallback is reasonable for first-cut robustness. The only residual
risk is that a *non-hidden* user table genuinely named `_meta` would be
misclassified -- an edge the project already implicitly reserves (`_meta` is a
reserved frame key everywhere: `frame_selection.py:10`,
`layout_composer.py:8`). No action; record as a known carrier coupling.

* `spreadsheet_contract.py` (55) -- correctly minimal; splitting would fragment
  a single facade concept.
* XLSX triad (`xlsx_backend` 113, `openpyxl_parser` 338,
  `parser_interpretation` 284, `openpyxl_renderer` 359) and ODS triad

* The two `parser_interpretation` modules are ~85% structurally parallel, but
  the parse contract *forbids* a generic parser framework and the duplication

  only the *shared constants* (P3-02, P3-03) should get a single owner.

There is *no* package-split pressure and *no* justification for a generic

== 7. Inherent vs Accidental Complexity (the headline distinction)

*Inherent -- do NOT "fix"; the current design is proportionate:*

* Roundtrip fidelity and the `meta_canonical`/`meta_rendering`/carrier split
  (`generic_ir.adoc` invariants 1-4) -- the layered model + JSON blob + ADR is
  the right-sized response, not over-design.
* Multi-backend formula translation (XLSX `=XLOOKUP` vs ODS `of:=XLOOKUP[...]`)
  -- intrinsic format divergence, correctly confined behind `FormulaSpec`.
* The deferred reverse projection (user formatting -> `meta_canonical`) -- the
  contract deliberately leaves this open; the named accepted gaps (freeze
  read-hint, protection, column-width) are inherent format asymmetries, *not*
  debt, and are mostly encoded as explicit gap tests.
* The legend `resolved` round-trip channel -- a genuine cross-direction
  contract; its only flaw is *where* it is written (P3-05), not that it exists.

*Accidental -- repayable with the existing, proven, guard-backed methodology;
no architecture change:*

* P3-01 carrier->canonical column-width promotion (XLSX-only, bounded).
* P3-02 triplicated/divergent option vocabulary.
* P3-03 unowned `" / "` header convention.
* P3-04 implicit ODS `SetColumnWidth` skip.
* P3-05 input mutation in the shared write facade (surgical purity fix).
* G4 parity pinned only at integration tier.

None of the accidental items requires a new framework, a package split, or a
generic abstraction. Each is a *narrow seam-hardening* or an *explicit
decision*, exactly the posture the prompt and ADR-001 prescribe.
