== Executive Verdict

*Yellow.*

The maintainer's core assessment is broadly correct: the architecture is
sound, the major semantic drift is substantially stabilized at the
primitive layer, and the next useful phase really is real use, demo, and
documentation rather than further structural refactoring. The repository
*is* in a state where a non-beta `0.2.0` is realistic in scope -- but
not yet a publishable release without a small, concrete cleanup pass on
the public surface and a refresh of one piece of governance
documentation.

Concretely, this review found:

* zero blockers preventing credible real-world trial use today;
* a small group of *visible*, *cheap*, *pre-0.2.0* friction items
  (stale top-level config scratch, a placeholder `release_notes.adoc`,
  a guard inventory that lags the live suite);
* one *before-low-touch* governance gap (maintainer entrypoints /
  module budgets and a backlog audit against the post-Phase-3.5
  reality);
* the previously known module-size watch list is *unchanged in size
  but unchanged in risk*: nothing has materialized a defect from it
  since REVIEW-002, and the proven analysis-driven split methodology
  remains the right response.

The verdict is *not* Green because the items listed under "Before
0.2.0" are exactly the kind of friction a non-developer + LLM early
adopter -- the explicit target audience in the prompt -- will hit on
the first day of evaluation. None of them require an architecture
change; all of them are within the existing maintainer's small,
reviewable-commit cadence.

The verdict is *not* Red because the runtime works (full default suite
passes; architecture guards pass; CLI launches; the demo entry path is
documented), the hexagonal boundary is executable, both backends are
real, the FK-helper primitive semantics have been stabilized in code
since REVIEW-002, and the IR-006 input-mutation residue documented in
REVIEW-002 has been partially fixed (see Section 5.4 in the table
below).

The verdict is *not* Inconclusive because there *is* enough evidence:
902 collected tests pass, the runtime/manual step-registry pair is
guard-aligned, the CLI/package boundary is exercised, and prior
review/reassessment artifacts provide a calibrated baseline against
which the current HEAD can be compared.

== Findings

=== Blockers Before Real Use

*None.*

Real-world trial use is credibly possible today. Concrete supports:

* `sheets-run --help` launches and exposes a documented argument
  surface (`src/spreadsheet_handling/cli/apps/run.py`); the same is
  true of the two example commands.
* `python -m compileall src/spreadsheet_handling` exits 0; the
  package imports.
* The default test suite (`make test`-equivalent) passes 902/902 in
  under 6 seconds; the architecture suite (106) and unit suite
  (704) and integration suite (92) all green individually.
* The manual step registry
  (`registries/pipeline_step_registry.json`, 38 entries) matches the
  runtime registry exactly; the architecture guard
  (`tests/architecture/dependency_guards/test_pipeline_step_registry.py`)
  prevents drift.
* The demo repo (sibling on disk) carries a first-hour tutorial
  README; it is the documented user entry path and the core README
  links to it.

If a target early adopter cannot get started today, the cause is
documentation packaging or onboarding ergonomics, not a runtime
defect.

=== Before 0.2.0

These should be addressed before removing the beta suffix. They are
cheap, visible-surface items; none of them require architectural
change.

==== F-0.2-1 -- Stale top-level config artifacts misrepresent the public YAML surface

*Observation.* The core repo root contains
`examples/config_example.yaml`, `sheets.yaml`, `pipeline.yml`, and
`input.json`. `examples/config_example.yaml` uses an obsolete shape
(`backend: excel`, `levels: 3`, `sheets: ...`, `refs: ...`) that
does not exist in the current public YAML surface; `sheets.yaml`
references step *factories* (`spreadsheet_handling.domain.validations.unique:make_step`,
`...domain.extractions.select:make_step`) that do not exist in the
source tree. `input.json` is unrelated scratch content (the file
contents are an unrelated AsciiDoc draft). The agent rules
(`docs/ai_info/agent_rules.adoc`) treat `input.json` as non-source,
but a non-developer + LLM evaluating the repo cannot tell that from
the top-level listing.

*Evidence.*

* `examples/config_example.yaml` -- uses `backend`/`levels`/`refs`
  vocabulary; no `io:` / `pipelines:` block; not referenced from any
  current docs file (`grep -rn "config_example.yaml"` finds no
  consumer).
* `sheets.yaml` at repo root -- references
  `spreadsheet_handling.domain.validations.unique:make_step` and
  `spreadsheet_handling.domain.extractions.select:make_step`; neither
  module path exists under `src/spreadsheet_handling/`. The file is
  not referenced from any current doc.
* The same-named filename `sheets.yaml` *inside test fixtures*
  (`tests/integration/pipeline/*.py`, `tests/unit/pipeline/test_app_config.py`)
  uses a different, current shape, which compounds the confusion for
  a reader.
* The package *does* ship in-tree current examples
  (`src/spreadsheet_handling/examples/pipeline_basic.yml`,
  `pipeline_pack.yml`, `pipelines.yml`) using the live `step:` /
  `pipelines:` / `io.profiles:` shape, but those are not what a
  newcomer sees first.

*Concrete risk if ignored.* A non-developer + LLM following the README
(which correctly redirects to the demo, but a curious user will also
browse the core root) copies `examples/config_example.yaml` and asks
their LLM "make this run". The LLM, taking the example at face value,
produces a non-working pipeline; the user concludes the project does
not work. This is the most realistic onboarding failure mode for the
declared target audience.

*Recommended action.* Either remove these four files, or move them
under `tmp/`/`scratch/` and add them to `.gitignore` where applicable.
For `examples/config_example.yaml` specifically, the right replacement
already exists at `src/spreadsheet_handling/examples/pipeline_basic.yml`
-- either point `examples/` at that, or delete `examples/` and update
the (currently absent) doc reference.

*Priority rationale.* Cheap; deletions/moves only; directly aimed at
the prompt's named early adopter; the risk is visible on day one.

==== F-0.2-2 -- `release_notes.adoc` is a placeholder

*Observation.* `docs/release_notes/release_notes.adoc` is an explicit
placeholder: "Detailed historical release notes are intentionally not
reconstructed in this first Pages publishing slice." For a `0.2.0`
release that drops the beta suffix, even a one-section "what changed
since `v0.1.0b6`" is the minimum acceptable documentation surface.
The repository has accumulated 85 commits between `v0.1.0b6` and the
current HEAD, including the FK-helper policy-driven primitives, the
inferred FK relation step, IR-006 mutation-safety fixes, and the
Phase 3.5 semantic foundation work.

*Evidence.*

* `docs/release_notes/release_notes.adoc` lines 22-24: "Detailed
  historical release notes are intentionally not reconstructed in
  this first Pages publishing slice."
* `git log --oneline v0.1.0b6..HEAD | wc -l` -> 85 commits, including
  the FK-helper policy slice (`bc01209`, `7f6c4f5`, `7a06d37`).
* `docs/cold_storage/backlog/ftrs_done/` contains the closed FTRs
  that constitute the bulk of the post-b6 work; release notes can
  source from there rather than be reconstructed from scratch.

*Concrete risk if ignored.* A non-beta `0.2.0` published with an empty
release-notes page is a public signal that the project is not actually
out of beta. It also leaves users who installed `v0.1.0b6` with no
documented migration path for the FK-helper public-surface change
(the v2 relation policy shape).

*Recommended action.* For 0.2.0, source one section per Phase 3a / 3.5
closed FTR from `docs/cold_storage/backlog/ftrs_done/`. Do not
attempt to reconstruct beta-cycle micro-history; document the
behavior-visible changes between `0.1.0b6` and `0.2.0`.

*Priority rationale.* This is the minimum public-surface bar for a
non-beta tag. It is not a runtime concern; it is a release-readiness
concern.

=== Before Maintainer Absence / Low-Touch Period

These should be addressed before the project enters a period of low
maintainer touch. They protect the AI-friendly maintainability claim
(Maintainer Assessment row 6) at the moment it is most needed.

==== F-LT-1 -- Maintainer entrypoints + module budgets are still an open FTR

*Observation.* `FTR-MAINTAINER-ENTRYPOINTS-AND-MODULE-BUDGETS-P5` is
open. The repository ships eight transformation/validation modules
above the ~500-line threshold (`xref_crosstable.py` 801,
`reference_validations.py` 729, `cell_codec.py` 631,
`join_views.py` 567, `enrich_lookup.py` 567, `odf_renderer.py` 540,
`validations/fk_helpers.py` 539, `workbook_views.py` 501). The
proven safe-split methodology exists, but the published
*change-entrypoint map* the FTR is meant to produce does not.

*Evidence.*

* `wc -l src/spreadsheet_handling/domain/transformations/*.py` --
  six >= 412-line transformation modules.
* `docs/backlog/FTR-MAINTAINER-ENTRYPOINTS-AND-MODULE-BUDGETS-P5.adoc`
  lines 1-58 -- status Open; deliverables include "entrypoint map in
  the developer manual" and "module-size warning thresholds in
  `tools/check_source_style.py`".
* `docs/ai_info/architecture_primer.adoc` carries a partial
  "Where new code belongs" table but explicitly notes the table
  records *intended* targets rather than current state for some rows
  (lines 197-210).

*Concrete risk if ignored.* During a low-touch period, the most
likely change driver is an AI agent or a returning collaborator
opening one of the 700-800-line modules cold. Without a published
change-entrypoint map, the agent either over-touches the module
(refactoring "in passing", which CLAUDE.md and `agent_rules.adoc`
explicitly forbid) or under-touches it (missing the registry or guard
that should also be updated). The cost is not catastrophic, but it
compounds.

*Recommended action.* Land the FTR's acceptance items 1 and 4 in a
single small slice (entrypoint map + module-size warning policy).
Defer items 2-5 unless the slice forces them naturally.

*Priority rationale.* Low-touch period mitigation; the FTR is already
scoped; no architectural decision required.

*Concrete risk if ignored.* A low-touch period combined with an open

*Recommended action.* As part of `FTR-CONCEPT-DOCS-CLEANUP-P5`
