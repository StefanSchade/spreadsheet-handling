= Review 005 Closure Assessment After Slice 1d

Date: 2026-07-06

Scope: closure assessment for Review 005 implementation commits
`664297e`, `31aab46`, `690ec69`, `2227511`, `b326f0a`, and `79b0142`.

== Executive Assessment

Review 005 implementation work is stable enough to pause.

The sequence proved the intended operating mode:

review -> narrow slice -> code trace -> registry declaration -> guard/test -> gap disposition

That is no longer only an intended discipline. It happened repeatedly:

* Slice 0/1b turned the registry loop itself into a checked workflow:
  table shape, xrefs, controlled vocabularies, unexpected tables, duplicate
  surface paths, staging freshness, and workbook/export binding are guarded.
* Slice 1/1a/1c used code behavior in `schema_maintenance` as the authority,
  declared the behavior in `meta_reference_surfaces.json`, and added tests that
  fail when code and declaration drift.
* Slice 1d joined the behavior registry to `meta_registry.yaml` without
  copying the inventory. The YAML remains source of truth for meta inventory;
  the domain-contract table records only the schema-maintenance behavior and a
  `meta_registry_entry` join.

The domain-contract registry is still constrained. It has grown, but the growth
has been bounded by executable checks rather than new prose-only tables.

== Pause Decision

Pause Review 005 implementation work here and move to the next strategic
topic.

Reason: the Review 005 loop now has enough proof of method. Continuing would
mostly pull the project into broader metadata ontology work, not into the
maintainer-autonomy bottleneck. The remaining Review 005 gaps are real, but
they are acceptable to leave open because they are documented, scoped, and no
longer blocking the registry/control-loop mode.

== Registry And Bridge Assessment

`meta_reference_surfaces.json` is coherent as a behavior contract for one
capability family: column schema maintenance. It should not become a general
meta inventory.

`meta_registry.yaml` remains coherent as the inventory/ownership source of
truth. It owns meaning, producer/consumer facts, persistence, and lifecycle
classification.

The bridge relation is the right shape:

* surface rows carry `meta_registry_entry`;
* architecture tests parse `meta_registry.yaml` directly;
* pattern rows (`*`, `__*`) intentionally stay unjoined;
* no JSON mirror of `meta_registry.yaml` was introduced.

Slice 1d therefore avoided the mirror failure mode. The residual duplication is
the human-readable `owner` column, which should be treated as commentary unless
a later narrowing slice removes or derives it.

== Remaining Risks

Acceptable risks if Review 005 pauses now:

* `GAP-META-REF-CONVENTION-ESCAPE` remains open for non-conventional key names
  and for unhandled paths in supported roots other than `helper_policies`.
* `GAP-SCHEMA-TABLE-KEY-UNIMPLEMENTED` remains open: table/key maintenance rows
  are candidates, while implemented schema maintenance is column-level.
* `GAP-META-REF-DECLARATIONS-IN-CODE` remains accepted: `meta_refs.py` remains
  runtime authority, and registry consistency is enforced by tests rather than
  generated code.
* `owner` in `meta_reference_surfaces.json` can drift as commentary if people
  start treating it as authoritative.
* The registry can still accrete if future work adds rows without a code trace
  and a guard.

These are not blockers. They are bounded maintenance risks, not reasons to keep
Review 005 as the active implementation stream.

== Next Strategic Track

If the project goal is maintainer autonomy rather than feature velocity, the
next strategic focus should be pipeline operability and configuration
simplification.

This should include plugin reduction only where plugins are compensating for
missing declarative pipeline capabilities. Do not treat plugin removal as a
goal by itself. The goal is that a maintainer can inspect, run, modify, and
debug real pipelines without reading several local plugin modules first.

The first area to inspect is Project Memory, because it is in-repo, active, and
plugin-heavy:

* `project_memory/pipelines/memory/json_to_derived_queries.yaml` - 8 steps, all
  plugin calls.
* `project_memory/pipelines/memory/ods_to_json.yaml` - mixed xref operation and
  plugins around matrix import cleanup/finalization.
* `project_memory/pipelines/memory/json_to_ods.yaml` - xref contraction plus a
  matrix-enrichment plugin.
* `project_memory/plugins/queries.py` - central plugin module behind the three
  memory pipelines.

Secondary areas to inspect:

* `registries/domain_contracts/pipelines/*.yaml` and
  `tools/domain_contracts/workbook.py` - useful comparison case; these plugins
  are thin registry-authoring adapters and may be acceptable.
* `src/spreadsheet_handling/pipeline/build.py`,
  `src/spreadsheet_handling/pipeline/steps.py`,
  `src/spreadsheet_handling/pipeline/registry.py`, and
  `registries/pipeline_step_registry.json` - public pipeline vocabulary,
  plugin escape hatch, and registered declarative alternatives.
* Backlog evidence around `FTR-DECLARATIVE-TABULAR-VIEW-OPS-P4A`,
  `FTR-DECLARATIVE-WORKBOOK-VIEWS-P4A`, and
  `FTR-DYNAMIC-FRAME-PROJECTION-IMPL-P5` - prior decisions about when local
  plugins should become core declarative steps.

== Candidate First Slice

Recommended first slice: a read-only pipeline operability inventory, followed
by one narrow replacement candidate.

Inventory questions:

* Which pipelines are longest and most plugin-heavy?
* Which plugin calls are pure table/view operations that could be declarative?
* Which plugin calls are legitimate domain-specific adapters?
* Which YAML argument blocks are hard to audit because they repeat frame names,
  concern IDs, or column lists?
* Which failures would a maintainer diagnose from YAML alone, and which require
  opening Python plugin code?

Smallest likely implementation follow-up, after inventory: choose one
Project-Memory plugin call that is a thin generic view operation and replace it
with an existing registered step or one small declarative step, with parity
tests. Do not start with the whole memory pipeline.

== Non-Goals

Do not continue Review 005 by expanding `transformation_meta_io` or adding more
metadata-reference rows without a new concrete code trace.

Do not generate `meta_refs.py` from the registry yet.

Do not mirror `meta_registry.yaml` into domain-contract JSON.

Do not try to close every metadata-reference escape before moving on.

Do not remove plugin support globally.

Do not start with a broad pipeline framework redesign, a new DSL, or a registry
ontology for every configuration field.

Do not use domain-contracts as the next operability target; it just received
focused hardening and is now a comparison case rather than the active bottleneck.
