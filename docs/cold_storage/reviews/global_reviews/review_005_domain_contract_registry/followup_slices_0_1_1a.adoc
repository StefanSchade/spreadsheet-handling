= Review 005 Follow-Up: Slices 0, 1, and 1a

Date: 2026-07-06

Scope: adversarial review of commits `664297e`, `31aab46`, and `690ec69`
before the next Review 005 slice.

== Bottom Line

Slices 0/1/1a did prove the intended working mode, but only for a narrow
control loop:

* Slice 0 made normal ODS staging/promote materially safer. The checker now
  validates current table shape, xrefs, controlled vocabularies, and staging
  freshness before promote.
* Slice 1 is useful as a mirror-plus-guard model. It is not just prose:
  `test_meta_reference_surfaces.py` binds root-level policy declarations to
  `meta_refs.py`, and the checker validates the new tables.
* Slice 1a fixed the concrete lookup-helper escape with acceptable behavioral
  scope. Rename updates only the target frame's lookup policy entry; drop
  blocks referenced lookup-policy columns, including prune, which is
  conservative but coherent.

The model is not yet safe enough for broad metadata-reference expansion.
Remaining gaps are small but real.

== Top Risks

. *Promote still trusts an external stamp too much.*
  `domain-contracts-export` writes a canonical fingerprint to
  `domain_contracts.ods.stamp.json`, and `domain-contracts-import` copies that
  stamp into staging. The stamp proves that canonical has not changed since
  export, but it is not bound to the imported workbook content. A stale or
  arbitrary workbook can be imported with a fresh sidecar stamp if the files
  are manually mixed. This is not the common accidental path, but it is a
  remaining adversarial loophole.
. *Promote copies any staging `*.json` file.*
  `domain-contracts-promote` validates the required tables in staging, then
  copies `staging/*.json` into canonical. The checker does not reject extra
  table files, so a rogue JSON table can enter canonical without being governed
  by `TABLE_FIELDS`.
. *The metadata-reference model is root-governed, not path-complete.*
  The new registry structure is tractable, but path-level truth still lives in
  code/tests/evidence rows. Supported roots are exempt from the unknown-root
  scanner, so unhandled paths inside `helper_policies`, `sheets`, etc. remain
  the live class behind `GAP-META-REF-CONVENTION-ESCAPE`.
. *Tests prevent many drifts, not all semantic drifts.*
  The tests catch table schema drift, controlled-value drift, important xrefs,
  root-policy drift, and lookup behavior. They do not yet catch extra canonical
  JSON tables, stamp/workbook mismatch, duplicate `meta_reference_surfaces`
  path declarations, or path-pattern-vs-code mismatch beyond the hand-written
  architecture guard.

== Slice Answers

Slice 0::
Safe enough for the normal registry loop, not fully adversarial-safe. The
freshness guard and checker are a real improvement; the two remaining loopholes
are sidecar-stamp/workbook separation and unvalidated extra JSON files copied
by promote.

Slice 1::
The model is useful and maintainable at the current size. It avoids becoming
pure prose because there are executable guards against `meta_refs.py` root-set
drift and checker-enforced vocabularies/xrefs. It would become another prose
registry if the next step adds many path rows without executable path-level
fixtures.

Slice 1a::
The lookup policy fix is correctly narrow. It uses exact lookup-frame scoping,
preserves scalar/list shapes for `key` and `order.sort_by`, leaves other
frames untouched, and treats malformed lookup policy shape as blocking rather
than guessing. No over-broad user-visible behavior change was found.

New tests::
Sufficient for the proved slices, insufficient as a general anti-drift net.
The next tests should target the two promote loopholes and one path-level
metadata-reference fixture before adding more registry rows.

== Recommended Next Slice

Do *not* take all of `GAP-META-REF-CONVENTION-ESCAPE` as the next code slice.
It is the right theme, but too broad if interpreted as a general convention
engine or full registry-driven metadata reference model.

Smallest safe next slice:

* close the two remaining Slice 0 control-loop loopholes first:
  make staging validation reject unexpected `*.json` tables, and bind the
  export stamp to the workbook content imported into staging; then
* add one narrow executable fixture for `GAP-META-REF-CONVENTION-ESCAPE`: an
  unhandled conventional column reference inside an already supported root
  should block/report instead of silently passing.

That slice keeps the registry loop trustworthy before adding more model mass,
and it proves the escape class with one concrete fixture instead of designing a
large metadata-reference engine.

== Blockers

No blocker to continuing Review 005 work.

Blocker to broad modeling: the promote loop should reject extra staging tables
and stamp/workbook mismatches before more canonical tables or surfaces are
added.

== Fix Before More Modeling

* Add a checker/promote guard for unexpected canonical/staging JSON files.
* Bind `domain_contracts.ods.stamp.json` to the exported workbook bytes, or
  otherwise verify at import time that the workbook being imported is the one
  that was stamped.
* Add one path-level regression around supported-root convention escape before
  expanding `meta_reference_surfaces`.
