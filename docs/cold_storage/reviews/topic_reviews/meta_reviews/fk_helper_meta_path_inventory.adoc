= FK Helper Metadata Path Inventory

Review scope: FK-helper-related `_meta` paths only. +
Inputs read (code evidence only): `domain/helper_policies.py`,
`domain/fk_relations.py`, `domain/transformations/fk_helpers/` (enrich,
drop, policy, provenance), `domain/transformations/derived_column_policy.py`,
`domain/transformations/helpers.py`, `domain/validations/fk_helpers.py`,
`domain/validations/reference_validations.py`, `domain/column_roles/resolver.py`,
`domain/schema_maintenance/meta_update.py`, `domain/workbook_views.py`,
`pipeline/persistence_boundary.py`, and tests under `tests/unit`,
`tests/architecture`, `tests/roundtrip`. +
Prior audit inputs: `audit/fk_helper_lifecycle_and_meta_review.adoc`,
`audit/meta_flattening_rg_observations.adoc`.

== Scope and Purpose

This document is a code-evidence inventory: producer, consumer, persistence
behavior, test coverage, runtime dependency, controlled-consumer dependency,
and suspicion level for every FK-helper-related `_meta` path. Each path
receives one of four classifications:

* *active feature contract* — consumed on every normal pipeline execution;
  removing it would break behavior.
* *runtime helper* — ephemeral cross-step carrier within a single run;
  not persisted; the pipeline architecture requires it.
* *carrier detail* — survives into persistence but is consumed only on a
  fallback path or as a metadata-about-metadata discriminator; narrower
  contract than an active feature contract.
* *legacy residue* — present in the data model but not consumed by any
  runtime code path; exists for backward-compat or because retirement was
  explicitly deferred.

The final section evaluates code-level evidence for and against the
meta-flattening hypothesis stated in `meta_flattening_rg_observations.adoc`.

== Inventory

[[FKMPI-PATH-1-V1-PER-TARGET-DICT]]
=== PATH-1: `_meta.helper_policies.fk.<target_name>` (v1 per-target dict)

==== Producer
`configure_fk_helpers` → `_build_fk_policy` → stored directly into the
`fk` namespace at `fk_policies[policy["target"]] = policy`. +
File: `domain/helper_policies.py:141-145`. +
Shape written per target: `{target, target_sheet, key, allowed_helpers,
default_helpers, helper_prefix, fk_column, [label]}`.

==== Consumer(s)
`_v1_policy_helper_names` in
`domain/transformations/derived_column_policy.py:294-325` — last-resort
cleanup fallback. Iterates the `fk` namespace, matches each per-target
dict to the source frame by checking whether `fk_column` appears in the
source frame's columns, and reconstructs helper column names via
`{helper_prefix}{target_frame}_{field}` for each `default_helpers` entry.
Called from `_safe_policy_helper_names` which is the third and lowest-priority
input to `apply_derived_column_policy`. +
`schema_maintenance/meta_update._handle_helper_policies` at
`meta_update.py:297-348` — treats each non-`schema_version`, non-`relations`
key in the `fk` namespace as an opaque v1 payload; iterates it to emit a
warning but does not rewrite v1 content on rename/drop (only v2 relations
are structurally updated). The v1 dict is therefore NOT kept consistent
with target-column renames when a rename touches `key` or a
`default_helpers` field.

==== Persistence behavior
KEPT. The persistence boundary's `_project_helper_policies` function
(`persistence_boundary.py:148-178`) only removes runtime-produced v2
relation entries under `fk.relations`; v1 per-target dicts are passed
through unchanged.

==== Tests protecting it
`test_keeps_helper_policies_v1_per_target_dicts` in
`tests/unit/pipeline/test_persistence_boundary.py:77` — asserts that a v1
entry survives the boundary. +
`test_helper_cleanup_uses_fk_policy_when_view_helper_metadata_is_absent`
in `tests/roundtrip/test_cleanup_invariants.py:54` — exercises the full
roundtrip where the forward pipeline does NOT write
`_meta.sheets.<frame>.helper_columns`, so cleanup must fall back to the
v1 dict (and/or v2 relations) to identify and drop helper columns.

==== Current runtime dependency
LOW on the primary execution path. `_v1_policy_helper_names` is the
third reader in `apply_derived_column_policy`; it fires only when BOTH
`_meta.derived.sheets.<sheet>.helper_columns` AND
`resolve_v2_fk_relations` return nothing. `configure_fk_helpers` writes
both v1 and v2 simultaneously, so v2 is always present during the same
run. The v1 fallback is only reachable on reimport when the v2 entries
produced by `configure_fk_helpers` have been dropped at the persistence
boundary and the reimport pipeline did not re-run a configuration step.

==== Controlled consumer dependency
The v1 shape is opaque to consumer repos. Consumers configure
`configure_fk_helpers` via YAML; they do not directly read
`_meta.helper_policies.fk.<target_name>`. YAML-level compatibility is
preserved as long as `configure_fk_helpers` keeps writing the v1 dict.

==== Suspicion level
MEDIUM-HIGH. The v1 dict is a parallel encoding of the same intent
already expressed in the v2 relation list produced by the same step in
the same call. The content reachable via v1 (`helper_prefix`,
`default_helpers`, `fk_column`) is redundant once the relation's
`helper_columns[*].column` is recorded. The persistence boundary
docstring explicitly acknowledges this and defers the broader redesign
of `helper_policies`. Schema maintenance does not keep v1 content
consistent with target-column renames, making it subtly stale after
any rename operation.

==== Classification
*carrier detail* — survives persistence but is read only as a last-resort
cleanup fallback; no FK-helper primitive reads it; it can become stale
after schema maintenance without detection.

=== PATH-2: `_meta.helper_policies.fk.schema_version`

==== Producer
`apply_v2_relations` in `domain/fk_relations.py:218`. Always sets
`fk_root["schema_version"] = SCHEMA_VERSION` (the constant `2`) whenever
any v2 relations are merged. Written by both `infer_fk_relations` (via
`apply_v2_relations`) and `configure_fk_helpers` (via
`_v2_relations_from_explicit_policies` → `apply_v2_relations`).

==== Consumer(s)
`resolve_v2_fk_relations` in
`domain/transformations/fk_helpers/policy.py:48` — the single entry
point through which all FK-helper primitives read relations. Returns
`None` (triggering `MissingFkRelationPolicyError`) when `schema_version
!= 2`. Every primitive imports `resolve_v2_fk_relations`; none bypasses
it.

==== Persistence behavior
KEPT when any non-runtime-produced relations survive. DROPPED together
with `relations` when all relations are runtime-produced
(`persistence_boundary.py:174-175`): the orphaned `schema_version` would
be semantically misleading without `relations`, so the boundary removes
both together.

==== Tests protecting it
`test_drops_empty_relations_and_schema_version_after_full_prune` in
`tests/unit/pipeline/test_persistence_boundary.py:142`. +
`test_keeps_relations_without_runtime_produced_marker` at line 92.

==== Current runtime dependency
CRITICAL. Every FK-helper primitive reaches relations through
`resolve_v2_fk_relations`, which returns `None` on version mismatch.
Without this field a v1 dict and a v2 block could be conflated.

==== Suspicion level
LOW.

==== Classification
*active feature contract* — the single discriminator guarding primitive
access to v2 relations; removing it would collapse the v1/v2 distinction
that the entire primitive refactor (`FTR-FK-HELPERS-POLICY-DRIVEN-
PRIMITIVES-P5`) depends on.

[[FKMPI-PATH-3B-CONFIGURE-V2-RUNTIME]]
=== PATH-3b: `_meta.helper_policies.fk.relations[*]` (produced by `configure_fk_helpers`)

Entries written by `configure_fk_helpers` with
`produced_by.step = "configure_fk_helpers"`, `produced_by.mode = "explicit"`.

==== Producer
`configure_fk_helpers` → `_v2_relations_from_explicit_policies` →
`build_v2_relation` → `apply_v2_relations`. +
File: `domain/helper_policies.py:147-149`.

==== Consumer(s)
Same as PATH-3a within the same pipeline run: all FK primitives consume
v2 relations without distinguishing producer.

==== Persistence behavior
DROPPED. `_is_runtime_produced_fk_relation` at
`persistence_boundary.py:181-187` returns `True` for
`produced_by.step == "configure_fk_helpers"`. The persistence boundary
docstring calls this "compiled pipeline state".

==== Tests protecting it
`test_drops_runtime_produced_fk_helper_relations`
(`test_persistence_boundary.py:116`). +
`test_drops_empty_relations_and_schema_version_after_full_prune`
(`test_persistence_boundary.py:142`).

==== Current runtime dependency
WITHIN-RUN ONLY. Enables primitives to consume a uniform v2 shape
regardless of whether declaration came from `configure_fk_helpers` or
`infer_fk_relations`. Dropped before persistence so it never accumulates.

==== Suspicion level
MEDIUM. The docstring explicitly labels it "compiled pipeline state"
(persistence_boundary.py:94). Its presence under the same root as durable
v2 entries is the root cause of the `produced_by` discriminator (PATH-4).
An alternative design would write runtime v2 entries under
`_meta.derived.fk_relations` instead of under `helper_policies.fk`, which
would eliminate both the discriminator and the coexistence confusion.

==== Classification
*runtime helper* — correctly dropped at the persistence boundary;
necessary only to let primitives read a single v2 shape within a run
started with `configure_fk_helpers`.

[[FKMPI-PATH-4-PRODUCED-BY]]
=== PATH-4: `_meta.helper_policies.fk.relations[*].produced_by`

==== Producer
`build_v2_relation` in `domain/fk_relations.py:288`: field
`{"step": produced_by_step, "mode": produced_by_mode}` always present.

==== Consumer(s)
`_is_runtime_produced_fk_relation` in `pipeline/persistence_boundary.py:181-187`
— the sole runtime consumer. Checks `produced_by.get("step") ==
"configure_fk_helpers"`. Nothing else reads this field at runtime.
`_assert_no_relation_conflict` (`fk_relations.py:235`) reads
`produced_by.step` to detect cross-producer rewrites; this is an error-
enforcement path, not a feature path.

==== Persistence behavior
KEPT on surviving relations (those not dropped by the boundary).

==== Tests protecting it
`test_drops_runtime_produced_fk_helper_relations` (the discriminator
value is the exact test fixture). +
`test_keeps_relations_without_runtime_produced_marker` (absence of
`produced_by` on an entry that has none).

==== Current runtime dependency
LOW during pipeline execution. HIGH at the persistence boundary for the
single discriminator call.

==== Suspicion level
MEDIUM. This field exists solely because runtime-produced and durable
entries coexist under `_meta.helper_policies.fk.relations`. If runtime
entries lived under a different root (e.g., `_meta.derived.fk_relations`),
the discriminator would be unnecessary. As-is, the field is a metadata-
about-metadata carrier created by an architecture ambiguity.

==== Classification
*carrier detail* — necessary only to resolve the coexistence ambiguity
at the persistence boundary; would disappear if runtime and durable v2
entries were placed under separate roots.

==== Producer
`build_v2_relation` in `domain/fk_relations.py` stores the `helper_prefix`

==== Consumer(s)
NONE on the runtime execution path. +
`iter_relation_fk_defs` (`fk_helpers/policy.py:119-144`) reads only
`column` and `target_field` from each `helper_columns` entry; it does not

field.

==== Persistence behavior
KEPT when relation survives.

==== Tests protecting it

on a v2 relation. Idempotency tests indirectly preserve it.

==== Current runtime dependency
NONE on the primary path.

already fully recorded in `helper_columns[*].column`. `helper_prefix`

from original configuration) but is never used as such by any consumer.
Tracked as Sonnet C007 in the prior lifecycle review.

==== Classification
*legacy residue* — carried in the v2 shape but never read by any runtime
consumer; the explicit column name in `helper_columns[*].column` makes
it redundant.

[[FKMPI-PATH-10-SHEETS-HELPER-COLUMNS]]
=== PATH-10: `_meta.sheets.<frame>.helper_columns`

==== Producer
`configure_workbook_view` via `_merge_sheet_options`
(`domain/workbook_views.py:444-456`). Written from the
`sheets[*].helper_columns` YAML list when present. Does not materialise
helper columns; only records which columns are helpers for workbook-view
purposes.

==== Consumer(s)
`_safe_durable_helper_names` (`derived_column_policy.py:244-268`) — second-
priority fallback in `apply_derived_column_policy` when
`_meta.derived.sheets.<source>.helper_columns` is absent (e.g., on
reimport after the persistence boundary). Returns names only; cannot
drive value comparisons. +
`_view_helper_columns` (`column_roles/resolver.py:217-232`) — resolves
the `display_helper` role for frame-projection steps. Reads via
`_sheet_for_frame` which first maps the frame name to a sheet name via
`_meta.workbook_view`. +
`_durable_helper_columns` (`validations/reference_validations.py:615-628`)
— the ONLY source for the `no_helper_columns` validation rule. Post-cleanup
assertion reads this list to detect helper columns that survived the cleanup
step; if this list is empty, the rule silently passes without checking. +
`schema_maintenance/meta_update._handle_sheets`
(`meta_update.py:202-204`) — propagates renames and drops to
`helper_columns` lists across sheets.

==== Persistence behavior
KEPT — preserved by the persistence boundary, which explicitly defers the
pruning question to `FTR-FK-HELPER-POLICY-LIFECYCLE-P4A`
(`persistence_boundary.py:74-76`).

==== Tests protecting it
`test_no_helper_columns_validation_reports_bypassed_cleanup`
(`tests/roundtrip/test_cleanup_invariants.py:130`) — the `no_helper_columns`
rule fires only when this path is populated; if absent, the rule silently
passes. +
`tests/unit/domain/column_roles/test_resolver.py` — asserts `display_helper`
role resolution from this path.

==== Current runtime dependency
MEDIUM-HIGH. Three distinct consumers, each with a different purpose and
no single canonical authority. The `no_helper_columns` rule is the load-
bearing contract: it is the recommended post-cleanup assertion and reads
ONLY this path. If this path is absent, the assertion silently passes even
when helpers are present.

==== Suspicion level
MEDIUM. The persistence boundary docstring flags its undecided status: it
can be added by `configure_workbook_view` (a system step), so its presence
does not reliably imply user-authored intent. The fact that the
`no_helper_columns` rule reads ONLY this path (not provenance, not v2
relations) makes it both load-bearing and fragile: a pipeline that calls
`add_fk_helpers` but not `configure_workbook_view` will not populate this
path, and the `no_helper_columns` rule will silently do nothing even if
helper columns survive cleanup.

==== Classification
*active feature contract* — consumed by three distinct code paths
(cleanup fallback, role resolution, post-cleanup assertion); removing it
would break all three without a replacement path.

== Sub-field Inventory: Vestigial Slots

=== SUB-1: `label_field` in `build_v2_target_registry`

`build_v2_target_registry` (`fk_helpers/policy.py:100-104`) sets
`"label_field": target_key` as a default. Code comment: "intentionally a

== Evaluation: Evidence For and Against the Meta-Flattening Hypothesis

The hypothesis (from `meta_flattening_rg_observations.adoc`): +
_`_meta` is a carrier; feature semantics should be local._

[[FKMPI-001-META-FLATTENING-SUPPORTED-MARGINS]]
=== Evidence Supporting the Hypothesis (paths that could be simplified)

*v1/v2 coexistence under one root is an architecture artifact, not a
feature requirement.* `configure_fk_helpers` writes both the v1 per-target
dict and a runtime-only v2 relation entry in the same step. The v1 dict
survives persistence; the v2 entry does not. The result is two shapes
with different lifetimes under one root
(`_meta.helper_policies.fk.<target>` vs `_meta.helper_policies.fk.relations`)
that encode the same user intent. No primitive reads v1; v1 is only a
cleanup fallback. If `configure_fk_helpers` were changed to write only a
single durable v2 entry (without `produced_by.step = "configure_fk_helpers"`,
making it permanent rather than runtime-only), both the v1 dict and the
`produced_by` discriminator would become unnecessary.

*`produced_by` is a discriminator for an architecture ambiguity, not a
feature field.* PATH-4 (`produced_by`) exists because runtime and durable
v2 entries coexist under `helper_policies.fk.relations`. If runtime-only
v2 entries were placed under `_meta.derived.fk_relations` instead, the
discriminator would be unnecessary. The field has no active feature semantics:
nothing reads `produced_by.mode` at runtime; `produced_by.step` is read
only at the persistence boundary.

*`helper_prefix` and `helper_fields` in v2 relations are not consumed.*
Both fields are stored in every v2 relation but no runtime consumer reads
them (PATH-6, PATH-7). `helper_prefix` is computable from any
`helper_columns[*].column` name; `helper_fields` is exactly
`helper_columns[*].target_field`. Schema maintenance renames
`target_field` without updating `helper_fields`, allowing silent staleness.

*The vestigial slots (SUB-1 through SUB-4) are genuine legacy residue.*
`label_field`, `FKFinding`, `del prefix`, and the single-member mode
frozenset each confirm that the shape carries more surface area than the
active use case requires.

[[FKMPI-002-META-FLATTENING-CONTRADICTED-CORE]]
=== Evidence Contradicting the Hypothesis (paths that cannot be made local)

*The pipeline architecture requires `_meta` as a cross-step carrier.*
The pipeline is a chain of pure transformations, each receiving and
returning `frames = {sheet_name: DataFrame, "_meta": dict}`. There is no
mechanism for a step to pass information to a downstream step other than
through `frames`. The following paths are cross-step dependencies that
cannot be made local without changing the pipeline's architectural contract:

. `_meta.helper_policies.fk.relations` — produced by configuration steps
  (`configure_fk_helpers` or `infer_fk_relations`), consumed by primitive
  steps (`enrich_helpers`, `drop_helpers`, etc.) that run separately in
  the pipeline sequence. Localising this would require merging
  configuration and execution into a single step, which the design
  explicitly separates.

. `_meta.derived.sheets.<sheet>.helper_columns` — produced by
  `enrich_helpers` in step N, consumed by `drop_helpers`,
  `apply_derived_column_policy`, and `validate_fk_helpers` in steps N+K.
  This is the textbook case for `_meta` as a cross-step carrier. The
  prior lifecycle review rated this path "clearly necessary."

. `_meta.sheets.<frame>.helper_columns` — the only post-persistence anchor
  for cleanup and the only source for the `no_helper_columns` assertion
  rule. Without this path, a reimport pipeline that does not re-run a
  configuration step has no metadata-driven basis for identifying helper
  columns after the persistence boundary has stripped `_meta.derived`.

*The architecture guard encodes an active design principle.* The test
`test_fk_helper_primitives_consume_policy.py` statically asserts that
primitives must NOT re-derive FK identity from column-name conventions
(i.e., `detect_fk_columns`, `build_registry`, `FK_PATTERN` are banned).
This guard can only be enforced if the v2 policy or derived provenance
is in `_meta`; making them local to the feature module would mean they
are no longer available to independent pipeline steps.

*The persistence boundary's Intent/Resolution split confirms that `_meta`
is the intentional carrier.* The boundary docstring explicitly uses the
terms "Intent" and "Resolution" and preserves Intent while dropping
Resolution. This is not an accident; it is a deliberate architecture
decision that acknowledges `_meta` as the cross-run carrier for user-
authorised declarations. The same pattern is applied consistently to
`legend_blocks` and `xref_crosstable`, confirming a project-wide choice,
not a local inconsistency.

=== Summary Assessment

The meta-flattening hypothesis is *partially supported* at the margins:
the v1/v2 coexistence, the `produced_by` discriminator, `helper_prefix`,
and `helper_fields` are genuine overdesign that a future cleanup can
remove. These are legacy residue or carrier details created by the
transitional state between v1 and v2 models.

The hypothesis is *contradicted at the core*: the three primary cross-step
and cross-run paths (`helper_policies.fk.relations`,
`derived.sheets.<sheet>.helper_columns`, `sheets.<frame>.helper_columns`)
are genuine pipeline carrier paths. They exist because the step-based
pipeline architecture requires `_meta` as the only cross-step communication
channel. Feature-local semantics are already enforced by the architecture
guard; the guard requires that the carrier exist.

The overdesign is not in `_meta` being a carrier — it is in `_meta`
carrying more shapes than the active use case requires. Specifically: one
canonical, durable v2 relation shape (with a single producer, no
`produced_by` discriminator, and no parallel v1 dict) plus one ephemeral
provenance path would cover every currently active consumer. The three
legacy-residue paths (PATH-1 v1 dict as primary shape, PATH-6
`helper_prefix`, PATH-7 `helper_fields`) are the consolidation targets,
not the carrier model itself.
