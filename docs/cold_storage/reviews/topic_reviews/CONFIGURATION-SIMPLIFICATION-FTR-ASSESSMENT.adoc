===== CONFIGURATION-SIMPLIFICATION-FTR-ASSESSMENT -- Configuration simplification FTR assessment

*Status:* Assessment

*Type:* Scope discovery / problem definition. No implementation proposal.

*Inputs reviewed:*::

* `../worldbuilding/pipelines/dino_insel/json_to_spreadsheet.yaml`
* `../worldbuilding/pipelines/dino_insel/spreadsheet_to_json.yaml`
* `../worldbuilding/plugins/{matrix_meta,story_cast_views,frame_sets}.py`

==== Purpose

This document assesses whether the current configuration surface has a
remaining simplification problem that deserves a real FTR.

This is not an implementation task and not a refactor proposal. The question is
whether repeated real consumer usage shows a problem whose smallest useful
solution would preserve architectural clarity.

==== Executive assessment

There is evidence of real configuration friction, but not enough evidence for a
broad new "configuration simplification" FTR.

The strongest friction is not `project_by_role` itself. The repeated
`helper_columns: [title]` / `key_columns: [story_id]` arguments are visible, but
they are intentionally explicit in the current architecture and mostly cost a
few lines. Removing them now would risk hidden coupling to frame ancestry,
pipeline order, or workbook-view metadata.

The largest real friction is the repeated multi-view matrix pipeline around
`story_cast_cells`: four facets are expanded into four workbook matrices and
recomposed on reimport. That repetition is genuine, but a framework shortcut for
it would likely become a macro language over existing pipeline steps unless a
second real consumer confirms the same shape.

The safest FTR-worthy area is narrower: explicit reuse or expansion of model
facts that are already repeated as configuration, especially validation rules
and basic frame/relation facts. This is already partially captured by
`FTR-MODEL-DEFAULT-CONFIGURATION-P5`, but that FTR is currently broad. If a real
FTR proceeds, it should refine or replace that broad idea with a small,
inspectable configuration-expansion slice.

==== Observed friction points

===== 1. Faceted matrix projection is verbose

The worldbuilding outbound pipeline projects the canonical
`story_cast_cells` relation into four specialized workbook matrices:

* `role`
* `notes`
* `character_development`
* `crystal`

Each facet repeats the same conceptual path:

* `extract_frame` from `story_cast_cells`
* `join_frames` to character names
* `contract_xref` into a matrix
* `join_frames` to story titles
* `project_by_role` for workbook-facing column order

Concrete evidence:

* Four `extract_frame` calls at `json_to_spreadsheet.yaml` lines 100, 106, 112,
  and 118.
* Four character-name `join_frames` calls at lines 124, 134, 144, and 154.
* Four `contract_xref` calls at lines 164, 175, 186, and 197.
* Four story-title `join_frames` calls at lines 208, 216, 224, and 232.
* Four `project_by_role` calls at lines 240, 246, 252, and 258.

The inbound pipeline mirrors the same shape:

* Four inbound `project_by_role` calls at `spreadsheet_to_json.yaml` lines 77,
  83, 89, and 95.
* Four `expand_xref` calls at lines 101, 110, 119, and 128.
* Four character-name-to-id `join_frames` calls at lines 137, 147, 157, and
  167.
* One `merge_by_discriminator` at line 177 recomposes the canonical relation.

This is the largest source of visible YAML volume in the current real consumer.
It also represents a real modelling concept: one canonical discriminator-based
relation has several workbook maintenance views.

However, the current explicit sequence is understandable. It shows where the
canonical relation is split, where display labels enter, where dynamic axes are
created, where workbook projection happens, and where reimport recomposes.

===== 2. Reference validation rules are duplicated

The same base validation facts are repeated across the outbound and inbound
pipelines.

Examples:

* Outbound validation at `json_to_spreadsheet.yaml` lines 20-42.
* Inbound pre-transform validation at `spreadsheet_to_json.yaml` lines 19-35.
* Inbound post-transform validation at lines 183-205.

This repetition is not just line noise. The same primary-key and foreign-key
facts must be kept aligned in more than one pipeline phase. If the model grows,
the author must remember each validation site.

Unlike the faceted matrix case, this friction has a small conceptual core:
model facts such as primary keys, unique labels, and foreign keys are declared
multiple times because there is no explicit reusable ruleset or model-fact
expansion surface.

===== 3. `project_by_role` call sites repeat role overrides

Worldbuilding currently has ten `project_by_role` call sites:

* Five outbound calls: `json_to_spreadsheet.yaml` lines 94, 240, 246, 252, and
  258.
* Five inbound calls: `spreadsheet_to_json.yaml` lines 49, 77, 83, 89, and 95.

All ten pass:

[source,yaml]
----
helper_columns: [title]
key_columns: [story_id]
----

This is visible repetition, but it is not currently a strong simplification
candidate. `FTR-DYNAMIC-FRAME-PROJECTION-IMPL-P5` explicitly records why these
overrides are required in worldbuilding: `configure_workbook_view` runs after
outbound projection, and `contract_xref` records row keys on the upstream matrix
frame rather than on the joined `*_matrix_view` frame.

Removing these arguments without a stronger metadata channel would require the
framework to infer ancestry or pipeline intent. That would be hidden coupling,
not simplification.

===== 4. Dynamic matrix-value targeting still uses a consumer plugin

The outbound pipeline still calls
`plugins.matrix_meta:add_from_legend_constraints_for_matrix` for
`story_group_matrix_view` at `json_to_spreadsheet.yaml` lines 311-316.

The plugin computes "all columns except helper and key columns" and emits one
legend constraint per matrix-value column. This is a real ergonomics gap, but
it is already covered by `FTR-DYNAMIC-VIEW-COLUMN-TARGETING-P5` and its decision
artifact: the proposed smallest surface is role-based targeting with
`roles: [matrix_value]` on existing rule target schemas.

This should not be pulled into a new configuration-simplification FTR.

===== 5. Workbook view and sheet mapping are repeated but not dominant

Outbound `configure_workbook_view` declares frame-to-sheet mappings and helper
columns at `json_to_spreadsheet.yaml` lines 296-309. Inbound
`apply_workbook_view_sheet_mappings` declares the logical frames at
`spreadsheet_to_json.yaml` lines 6-7.

This is a moderate source of maintenance friction, but the current example has
only one inbound list and one outbound mapping. It does not justify a framework
feature on its own.

===== 6. Local tidy-up plugins are not strong framework evidence

`ensure_frame_columns`, `strip_trailing_header_separator`, `drop_columns`, and
`keep_frames` are real consumer utilities. The prior dynamic-projection review
already classified most of these as local or degenerate cleanup utilities, not
as load-bearing framework concepts.

They should not drive a configuration simplification FTR.

==== Candidate simplifications

===== A. Named reusable validation/model fact blocks

Candidate idea: allow a consumer to declare common model facts once and reuse or
expand them into existing validation steps.

This preserves clarity if the generated or expanded configuration remains
inspectable and if execution still uses the existing `validate_references`
surface.

This is the strongest small candidate because it addresses real repeated facts:
primary keys, unique labels, and foreign keys. It would reduce maintenance
risk, not only YAML length.

It overlaps with `FTR-MODEL-DEFAULT-CONFIGURATION-P5`, but that FTR is broader
than the current evidence requires. A useful slice would start with reference
validation reuse only, not helpers, workbook views, projection settings, and
presentation defaults all at once.

Assessment: potentially worth a narrow FTR.

===== B. Explicit configuration expansion from compact model declarations

Candidate idea: refine the existing `FTR-MODEL-DEFAULT-CONFIGURATION-P5` into a
mechanism that expands compact model declarations into current explicit
configuration artifacts.

This is architecturally acceptable only if:

* generated configuration is reviewable,
* low-level explicit configuration remains valid,
* the expansion has no runtime heuristics,
* the generated result uses existing pipeline steps and metadata shapes.

Assessment: directionally valid, but too broad unless sliced down. It should not
begin with dynamic projection, workbook views, FK helpers, and validations all
in one FTR.

===== C. Faceted matrix-view declaration

Candidate idea: declare the four `story_cast_cells` facets once and expand to
the repeated `extract -> join -> contract -> join -> project` and reverse
chains.

This would reduce the most YAML in worldbuilding, but it is also the riskiest
candidate. It would hide multiple architectural stages behind one declaration:
discriminator extraction, label enrichment, dense axis creation, workbook
projection, reimport expansion, id recovery, and discriminator merge.

The current evidence is one real consumer with one canonical relation using
four facets. The functional-model docs describe similar future patterns, but
they are not independent working consumers.

Assessment: not ready for an FTR. Track as an observed pattern and wait for a
second real consumer or a second independent relation in worldbuilding.

===== D. Omit `helper_columns` and `key_columns` from `project_by_role`

Candidate idea: let `project_by_role` derive `title` and `story_id` at the
current worldbuilding call sites.

This is mostly syntactic sugar today. The implementation FTR already explains
why the resolver cannot reliably discover those facts at these sites without
explicit overrides.

Assessment: reject for now. It would reduce a few repeated lines but add hidden
coupling to pipeline order or frame ancestry.

===== E. YAML anchors, templates, or pipeline macros

Candidate idea: use generic YAML reuse or a pipeline-local macro mechanism to
remove repeated blocks.

YAML anchors may help a consumer locally, but they are not a framework concept
and do not reduce conceptual load. A framework macro system would be worse: it
would introduce a second execution layer above the existing pipeline.

Assessment: not a core FTR. Documentation can mention YAML-native reuse if
useful, but framework support should not start here.

===== F. Complete role-based dynamic targeting

Candidate idea: replace `add_from_legend_constraints_for_matrix` with
role-targeted validation rules.

Assessment: worthwhile, but already owned by
`FTR-DYNAMIC-VIEW-COLUMN-TARGETING-P5` and its decision artifact. Do not create
a duplicate simplification FTR.

===== G. YAML surface normalization

Candidate idea: normalize parameter names such as `source`, `relation`,
`matrix`, `output`, `target_frame`, `column_key`, and related inverse-step
shapes.

Assessment: worthwhile, but already owned by
`FTR-YAML-SURFACE-NORMALIZATION-P5`. It is adjacent to simplification but not
the same problem as repeated model facts.

==== Risks

===== Hidden coupling

The most tempting simplifications infer meaning from frame names, sheet names,
pipeline ancestry, or conventional column names. These would make YAML shorter
while making behavior harder to explain.

Examples:

* assuming `story_id` is the key because a frame name starts with `story_`,
* assuming `title` is always a display helper,
* tracing `contract_xref` ancestry through later `join_frames`,
* assuming every discriminator-based relation should expand into workbook
  matrices.

These are not acceptable simplifications.

===== A second metadata system

A compact model declaration can become a parallel source of truth if it is not
strictly an expansion source for existing configuration. This is the main risk
around `FTR-MODEL-DEFAULT-CONFIGURATION-P5`.

If the compact declaration owns semantics independently from current pipeline
steps, users must understand both the compact model and the expanded pipeline.
That increases conceptual load.

===== Macro language creep

The faceted matrix pattern is verbose enough to invite a macro. A macro would
make the current worldbuilding YAML shorter, but it would hide the specific
transform stages that are currently valuable for review and debugging.

This is especially risky because the same repeated block crosses both outbound
and inbound paths. A shortcut would have to encode roundtrip semantics, not just
write-time view generation.

===== Syntactic sugar mistaken for simplification

Several candidates reduce line count without reducing concepts:

* omitting `helper_columns` and `key_columns`,
* aliases for one-off step parameters,
* YAML templates for repeated `project_by_role` calls,
* shortcuts for `apply_derived_column_policy` on three named frames.

These are not strong FTR candidates unless tied to a broader, explicit,
reviewable configuration-reuse mechanism.

===== Over-broad first slice

The existing `FTR-MODEL-DEFAULT-CONFIGURATION-P5` names many possible derived
areas: primary keys, labels, foreign keys, helpers, workbook views, validation
rules, and projection settings. Implementing all of those together would blur
the boundary between model declaration, pipeline execution, workbook rendering,
and projection semantics.

==== Recommendation

Do not create a broad new `FTR-CONFIGURATION-SIMPLIFICATION` implementation
FTR.

The project already has targeted FTRs for the two current role/projection
issues:

* `FTR-DYNAMIC-FRAME-PROJECTION-IMPL-P5` for `project_by_role`.
* `FTR-DYNAMIC-VIEW-COLUMN-TARGETING-P5` for role-based rule targeting.

The remaining broad-surface concern is already represented by
`FTR-MODEL-DEFAULT-CONFIGURATION-P5`, but that FTR should not proceed as a broad
implementation slice in its current form.

If a real FTR is justified, the smallest useful slice is:

* *Working scope:* explicit reuse or expansion of reference-validation model
  facts.
* *Source facts:* frame primary keys, unique label columns, and foreign-key
  relations.
* *Expansion target:* existing `validate_references` rule lists.
* *Inspection requirement:* the expanded rules must be observable in logs,
  generated config, dry-run output, or an equivalent reviewable artifact.
* *Compatibility:* existing explicit `validate_references` rules remain fully
  supported.

This slice would address real repetition without touching projection,
workbook-view composition, FK-helper generation, or dynamic targeting.

==== Explicit non-goals for any follow-up FTR

Any follow-up FTR should explicitly exclude:

* implicit inference from frame or column names,
* hidden defaults that change pipeline behavior,
* a general pipeline macro or loop system,
* discriminator-to-matrix view generation,
* changes to `project_by_role`,
* changes to `contract_xref`, `expand_xref`, `join_frames`, or
  `merge_by_discriminator`,
* a new column-role taxonomy,
* replacement of current explicit low-level YAML,
* workbook presentation defaults,
* FK-helper aliasing or naming policy,
* YAML surface normalization already owned by
  `FTR-YAML-SURFACE-NORMALIZATION-P5`,
* role-based dynamic targeting already owned by
  `FTR-DYNAMIC-VIEW-COLUMN-TARGETING-P5`.

==== Alternatives considered

===== Alternative 1: No FTR

This is defensible if the project wants more consumer evidence before changing
the configuration surface.

The downside is that validation/model-fact duplication will continue to grow as
worldbuilding grows. The current repeated rules are already enough to show a
maintenance problem, but not enough to force immediate implementation.

===== Alternative 2: Keep `FTR-MODEL-DEFAULT-CONFIGURATION-P5` as-is

This keeps the strategic direction visible, but it is too broad for the current
evidence. It risks absorbing unrelated concerns under "configuration
simplification."

Recommendation: keep it only as a parent / parking-lot concept, or rewrite it
around the narrow reference-validation slice.

===== Alternative 3: Add a faceted matrix declaration

This attacks the biggest YAML block, but it is premature and high risk. The
current explicit pipeline is long but architecturally legible. A shortcut here
would likely hide too much behavior.

Recommendation: do not proceed until the same pattern appears in another real
consumer or in a second independent worldbuilding relation.

===== Alternative 4: Add defaults to `project_by_role`

This removes repeated lines but does not remove a repeated concept. It would
also contradict the explicit-resolver reasoning recorded in
`FTR-DYNAMIC-FRAME-PROJECTION-IMPL-P5`.

Recommendation: reject for now.

===== Alternative 5: Use documentation and examples only

This is reasonable for faceted matrix patterns and plugin cleanup patterns.
Documenting a recommended explicit pattern may be better than adding a
framework shortcut.

It does not solve duplicated validation facts.

==== Architectural risk estimate

[cols="2,1,3"]
|===
| Candidate | Risk | Reason

| Reference-validation fact reuse / expansion
| Low to medium
| Low if it expands to existing `validate_references` rules and remains
  inspectable. Medium if it grows into a general model metadata layer.

| Broad model-default configuration
| High
| Crosses validation, helpers, workbook views, projection settings, and
  presentation concerns. Easy to become a second configuration authority.

| Faceted matrix declaration
| High
| Hides a multi-step outbound/inbound roundtrip pattern behind one abstraction.
  Current evidence is too narrow.

| `project_by_role` implicit defaults
| Medium to high
| Saves little but couples role resolution to ordering, frame ancestry, or
  naming conventions.

| Dynamic role targeting
| Medium
| Worthwhile, but already bounded by a dedicated FTR and a decision artifact.

| YAML surface normalization
| Low to medium
| Already scoped as naming/alias cleanup with compatibility constraints.
|===

==== Verdict

A real FTR should not exist under the broad title "configuration
simplification" unless it is explicitly an assessment or parent tracking item.

A narrow follow-up FTR is justified only for explicit, inspectable reuse of
model facts in reference validation. That work should refine the existing
`FTR-MODEL-DEFAULT-CONFIGURATION-P5` rather than create a competing umbrella.

No FTR is currently justified for:

* reducing `project_by_role` call-site arguments,
* hiding the faceted matrix projection chains,
* adding a pipeline macro language,
* replacing consumer cleanup plugins wholesale.

The current best path is conservative:

. Finish or retain the already-scoped role-targeting work separately.
. Treat YAML surface normalization separately.
. Narrow `FTR-MODEL-DEFAULT-CONFIGURATION-P5` to one inspectable
   validation/model-fact expansion slice before any implementation work.
. Continue collecting evidence for faceted matrix-view repetition before
   abstracting it.
