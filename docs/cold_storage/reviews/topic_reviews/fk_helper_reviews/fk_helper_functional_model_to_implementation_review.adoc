== Executive Summary

The functional model and the current code are *largely aligned on structure
and lifecycle*. The three-step shape (Declare / Enrich / Drop), the
runtime-provenance carrier dropped at egress, and the "no silent magic"
inference stance are all already implemented. The divergence from the model is
*sediment, not behavior*: parallel encodings of the same intent that survive
from the v1→v2 transition.

[[FKFM-001-V1-DICT-LOAD-BEARING]]
The most important code finding contradicts the cleanup ordering suggested in
`meta_flattening_hypothesis.adoc` (its Step 4 names the v1 per-target dict as
the likely first slice). Code evidence shows the v1 dict
(`_meta.helper_policies.fk.<target>`) is *still load-bearing on exactly one
path*: a `configure_fk_helpers` reimport where derived provenance and
`sheets.<frame>.helper_columns` are both absent. On that path the v1 dict is
the *only* surviving helper-identity source, and `test_helper_cleanup_uses_fk_
policy_when_view_helper_metadata_is_absent` pins it. Removing the v1 write *by
itself* would silently leak helper columns into canonical JSON. So v1 is not
the smallest safe slice.

[[FKFM-002-UNREAD-V2-RELATION-FIELDS]]
The smallest safe slice is removing the two relation-level fields that have
*zero runtime readers and no behavior test* — `helper_prefix` and
`helper_fields` on each v2 relation. This is behavior-preserving, touches one
producer function and a handful of shape assertions, and needs no consumer
migration.

On the document flagged for special attention
(`ch05_problem_space_overview/03_meta_lifecycle_framework.adoc`): its strong
claim — that *all* Type 2 lifecycle handling should become field-agnostic and
"read only meta registry attributes" — *does not survive contact with the
FK-helper persistence boundary*. Three of that boundary's FK-helper decisions
are not expressible as flat per-field attributes. Treated as a premise it would
mandate exactly the "global meta ontology" the project's own meta-flattening
hypothesis (and this task) rule out. See <<lifecycle-premise>>.

*Does the functional model imply code changes now?* Yes, but small and
optional, not urgent, and not on a load-bearing path. The model ratifies a
cleanup direction the code can already absorb; the recommended first slice is a
narrow code change (Slice 1). No documentation-only or no-op outcome is
required, and no broad framework should be built. See <<implication>>.

== Functional Model vs Current Implementation

=== Already aligned behavior

* *Three-step structure.* The model's Declare / Enrich / Drop maps cleanly:
  ** Declare = `configure_fk_helpers` (`helper_policies.py:91`) and
     `infer_fk_relations` (`fk_relations.py:29`).
  ** Enrich = `enrich_helpers` / `add_fk_helpers` (`fk_helpers/enrich.py:33`).
  ** Drop = `drop_helpers` / `remove_fk_helpers` (`fk_helpers/drop.py:27`).
* *"Heuristic inference is explicit, opt-in, not silent magic"* (model §Variant).
  `configure_fk_helpers` *rejects* `targets="auto"` and points the caller at
  `infer_fk_relations` (`helper_policies.py:114-119`). Primitives are barred
  from re-deriving FK identity from column names, enforced statically by the
  guard `test_fk_helper_primitives_consume_policy.py`.
* *Runtime provenance lifecycle = `drop_at_persistence` + `drop_if_stale`*
  (model §Enrich). `_meta.derived.sheets.*.helper_columns` is written by
  `_write_helper_provenance` (`provenance.py:13`) and removed unconditionally
  at egress: `RUNTIME_ONLY_TOP_LEVEL_META_KEYS = {"derived"}`
  (`persistence_boundary.py:101`), pinned by `test_drops_top_level_derived`.
* *Helper column values are data, not meta* (model §Enrich). `enrich_helpers`
  adds columns to the source frame; the literal-vs-formula choice is a
  materialization mode (`helper_value_mode`, `enrich.py:105`), not a separate
  semantics — matching the hypothesis doc's "same meaning, different carrier".
* *Drop precedence: provenance preferred, relation declaration as fallback*
  (model §Drop). `drop_helpers` consumes provenance first and falls back to v2
  relations per source frame (`drop.py:38-47`, `_columns_to_drop_by_sheet`
  `drop.py:73-100`).

=== Gaps (model target not yet reached in code)

* *v1 per-target config is not "treated as version mismatch → dropped"* (model
  §Simplification Note; framework §Meta Versioning, lines 57-65). Code still
  *writes* the v1 dict on every `configure_fk_helpers` run
  (`helper_policies.py:141-145`) and the persistence boundary *preserves* it
  (`_project_helper_policies`, `persistence_boundary.py:148-178`). It is not
  version-gated at all — it is always written and always kept. This is the
  single clearest gap between the stated target and the code.
* *"One durable workbook marker" vs two durable cross-run carriers.* The model
  (§Enrich/§Drop) describes a single *Durable helper column marker*
  (`carrier_takes_precedence`) read by the drop step on reimport. The code has
  *two* persistence-crossing carriers with different readers:
  ** `_meta.helper_policies.fk.relations` (read by `drop_helpers` and by
     `apply_derived_column_policy`'s policy fallback), and
  ** `_meta.sheets.<frame>.helper_columns` (read by
     `apply_derived_column_policy`'s durable fallback `_safe_durable_helper_
     names` `derived_column_policy.py:244`, by the `no_helper_columns`
     validation rule, and by `column_roles/resolver` for the `display_helper`
     role).
  Critically, `drop_helpers` does *not* read `sheets.<frame>.helper_columns`
  at all — so the model's "drop step reads the durable marker on reimport" is
  realized via v2 relations, not via the field the model's wording most
  resembles. This is a real "more shapes than the model needs" condition, but
  it is *load-bearing* (see below) and explicitly out of scope for a first
  slice.

=== Overgrown / historical parts (sediment)

Re-verified against code; all are written-but-effectively-unread or
discriminators created by transitional coexistence:

* `helper_policies.fk.relations[*].helper_prefix` — stored by
  `build_v2_relation` (`fk_relations.py:287`); *no runtime reader*. The
  materialized name already lives in `helper_columns[*].column`.
* `helper_policies.fk.relations[*].helper_fields` — stored by
  `build_v2_relation` (`fk_relations.py:282`); *no runtime reader*. Confirmed
  the only `helper_fields` reader, `core/fk.py:141-142`, reads a *different*
  `defaults` dict on the legacy `detect_fk_columns` path, never the v2
  relation. Redundant with `helper_columns[*].target_field`.
* `produced_by` discriminator — read only at the persistence boundary
  (`_is_runtime_produced_fk_relation`, `persistence_boundary.py:181-187`) and
  by the conflict guard (`_assert_no_relation_conflict`, `fk_relations.py:235`,
  an enforcement path). Exists only because runtime and durable v2 entries
  coexist under one root.
* Vestigial slots: `label_field` "intentionally a no-op default"
  (`policy.py:100-104`); `del prefix` on `drop_helpers` (`drop.py:36`) and
  `del helper_prefix` on `reorder_helpers_next_to_fk` (`helpers.py:219`);
  single-member `_VALID_MODES = frozenset({"naming_convention"})`
  (`fk_relations.py:25`). The two step params are documented in the step
  registry as "Retained for step-binding compatibility only … no longer
  drives … behaviour" (`pipeline_step_registry.json:262`, `:600`).

=== Risky load-bearing paths (do NOT open with a first slice)

* `_meta.derived.sheets.*.helper_columns` — runtime cross-step glue; the
  highest-priority identity source within a run.
* `_meta.sheets.<frame>.helper_columns` — the only post-persistence anchor for
  the `no_helper_columns` validation rule, which reads *only* this path
  (`validations/reference_validations.py:615-628`); also the `display_helper`
  role source.
* `helper_policies.fk.relations[*].helper_columns[*]` — the live per-field
  record every primitive consumes (`iter_relation_fk_defs` `policy.py:132`,
  `drop.py:94`, validations `:387`, registry build `policy.py:112`).
* `helper_policies.fk.schema_version` — the v1/v2 discriminator gating all
  primitive access via `resolve_v2_fk_relations` (`policy.py:48`).

[#lifecycle-premise]
[[FKFM-003-LIFECYCLE-PREMISE-OVERSTATED]]
=== Testing the meta-lifecycle "attribute-driven Type 2" premise against FK code

`03_meta_lifecycle_framework.adoc` §"Current State vs. Target" states:

[quote]
____
Target: Type 2 event code reads only meta registry attributes. A new domain
step declares its attributes at registration time; the framework event code
stays stable.
____

Treated as a hypothesis and tested against the FK-helper persistence boundary,
the *strong* form (field-agnostic, attribute-only) is *overstated*. Three of
the boundary's FK-helper decisions are not reducible to a flat per-field
attribute lookup:

. *Per-entry, content-dependent pruning inside one list.*
  `_project_helper_policies` keeps some entries of
  `helper_policies.fk.relations` and drops others based on a *sub-field value*
  (`produced_by.step == "configure_fk_helpers"`,
  `persistence_boundary.py:166,181-187`). A `drop_at_persistence` attribute
  attached to the *field* `relations` cannot express "drop these list members
  but keep those" — the predicate is over entry content, not field identity.
. *A cross-field invariant.* When pruning empties the list, the boundary also
  removes `schema_version` because it would be a "semantically misleading
  orphan" (`persistence_boundary.py:169-176`). That is a conditional coupling
  between two fields, not an independent per-field rule.
. *Nested-facet pruning.* The Intent/Resolution split removes sub-keys
  *inside* entries (`legend_blocks[*].resolved`,
  `xref_crosstable[*].dense_axes.resolved`, `[*].column_keys`;
  `persistence_boundary.py:190-237`). The unit of decision is a nested path,
  not a registered top-level field.

To make the framework event code truly field-agnostic for these cases, the
attribute language would have to grow predicates over entry content, cross-field
conditionals, and nested-path selectors. That richer language *is* a global
metadata ontology — precisely what `meta_flattening_hypothesis.adoc` warns
against ("the problem … is the ontologization of `_meta`") and what this task
forbids. So the two documents are in tension, and the FK-helper code sides with
the meta-flattening hypothesis.

The *weak* form of the framework's idea is fine and already partly true: the
conceptual labels (`drop_at_persistence`, `drop_if_stale`,
`carrier_takes_precedence`) are useful per-datum reasoning aids, and the one
genuinely field-level rule today (`RUNTIME_ONLY_TOP_LEVEL_META_KEYS`) is a
simple set that *could* be registry-sourced. The recommendation is to keep
those labels as descriptive vocabulary in feature-local contracts and *not* to
adopt "Type 2 code reads only attributes" as a target for the FK-helper slice.
This is recorded as an Open Question for the doc owners, not a code action.

== Candidate Implementation Slices

[#slice1]
=== Slice 1: Drop the unread `helper_prefix` and `helper_fields` from v2 relations

* *Goal*: remove the two relation-level fields that no runtime consumer reads,
  shrinking the v2 relation to what is actually consumed (`source_frame`,
  `source_column`, `target_frame`, `target_key`, `helper_columns[*]`,
  `produced_by`).
* *Files likely touched*:
  ** `domain/fk_relations.py` — `build_v2_relation` (`:259-289`) stops emitting
     `helper_prefix` and `helper_fields`. The function may keep or drop the
     corresponding parameters; callers (`helper_policies.py:179-190`,
     `fk_relations.py:168-178`) already compute `helper_columns` themselves, so
     the values are no longer needed for storage.
* *Expected tests to update* (shape assertions only):
  ** `tests/unit/domain/test_fk_relations_inference.py:48,56` — drop the
     `helper_fields` / `helper_prefix` assertions; keep the surrounding
     behavior assertions.
  ** `tests/unit/domain/test_fk_helper_policies.py:86` — drop the
     `helper_fields` assertion.
  ** `tests/unit/domain/test_fk_relations_inference.py:221,232` — the two
     "same-producer conflict" tests stay *green* (their divergent fixtures also
     change `helper_columns`, so the conflict still fires via the surviving
     fields); rename them so they no longer imply `helper_prefix`/
     `helper_fields` drive the conflict.
* *New tests needed*:
  ** A unit assertion that `build_v2_relation` output keys are exactly the
     consumed set (guards against silent re-growth).
  ** An idempotency assertion that a real divergence in `helper_columns` still
     raises in `apply_v2_relations` after the fields are gone.
* *Risk*: *Low*. Zero runtime readers; column names are unaffected
  (`test_add_fk_helpers_supports_distinct_helper_prefixes_per_relation`
  `test_fk_helper_configuration.py:146` asserts column *names*, which come from
  `helper_columns`, and stays green).
* *Migration impact*: none for consumers. Persisted durable/inferred relations
  from older runs keep the two extra keys harmlessly until the next forward run
  re-bootstraps them; `resolve_v2_fk_relations` never reads them. No YAML
  surface change.
* *Rollback*: revert the `build_v2_relation` edit and the test-assertion edits;
  fully isolated.

== Recommended First Slice

*Slice 1 — drop the unread `helper_prefix` and `helper_fields` from v2
relations.*

Rationale, against the task's preference for "small and behavior-preserving"
and the exclusion of load-bearing paths:

* It is the only candidate with *zero runtime readers and no behavior test* —
  proven by grep over `src/` (only `_v1_policy_helper_names` and the legacy
  `core/fk.py` reader touch these names, neither via the v2 relation) and by
  the fact that helper column names derive from `helper_columns`, not from the
  stored prefix.
* It touches none of the excluded load-bearing paths
  (`derived.sheets.*.helper_columns`, `sheets.<frame>.helper_columns`,
  `relations[*].helper_columns[*]`).
* It reduces sediment immediately and makes the v2 relation shape match what
  primitives consume — a maintainer-understandability win.
* It is strictly smaller and safer than the hypothesis doc's Step-4 guess (v1
  removal), which code evidence shows is *not* behavior-preserving on its own.

It deliberately leaves the deeper consolidation (Slice 2) for a second,
behavior-reviewed step, because that one changes a persistence contract and the
cleanup fallback chain.

[#implication]
== Does the functional model imply code changes now?

*Code changes — small and optional, not documentation-only, not "no change".*

* The model now *names* a target ("one model per concern", v1 as version
  mismatch) that the code does not yet meet. That is a genuine, code-level gap,
  not a doc-only reconciliation.
* But the model does *not* compel an urgent or large change. The runtime
  behavior already matches the model; the divergence is sediment. The smallest
  honest response is Slice 1 (a code change), with Slice 2 as the
  behavior-reviewed follow-up that actually realizes the v1-retirement the model
  describes.
* The model does *not* imply building the registry-attribute-driven Type 2
  framework; per <<lifecycle-premise>> that target is overstated and would
  violate the project's meta-flattening stance and this task's constraints.

== Open Questions

. *Meta-lifecycle doc vs meta-flattening hypothesis tension.* Should
  `03_meta_lifecycle_framework.adoc` be softened so its "Type 2 code reads only
  attributes" target is stated as the *weak* form (descriptive per-datum
  labels) rather than the *strong* form (field-agnostic framework)? The
  FK-helper boundary shows the strong form needs predicates/cross-field/nested
  selectors — i.e. an ontology. Doc-owner decision; no code action proposed.
. *Drop step: error vs silent no-op.* The model §Drop says that with no
  identification source the step "treats all columns as canonical payload …
  does not guess silently." The code instead *raises*
  `missing_fk_policy_error` when both provenance and relations are absent
  (`drop.py:41-42`). Same spirit ("no guessing"), different behavior (hard
  error vs pass-through). Which is the intended contract? Worth a one-line
  model clarification; not part of any slice here.
[[FKFM-005-TWO-DURABLE-CARRIERS]]
. *Two durable carriers.* Long term, should `sheets.<frame>.helper_columns` and
  the durable v2 relation be unified to the single "Durable helper column
  marker" the model describes? This is load-bearing (validation + roles) and is
  explicitly excluded from a first slice; flag for a later, separately-reviewed
  design step.
