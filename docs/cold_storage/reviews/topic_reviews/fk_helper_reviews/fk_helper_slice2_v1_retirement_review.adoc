== Executive Summary

The Slice-1 review's Slice-2 sketch is broadly right but *understated one
hard constraint and over-stated one removal*:

* *Hard constraint (the principal risk):* the persistence-boundary prune of
  `configure_fk_helpers`-produced relations is not incidental — it is the
  shipped containment for `BUG-RUNTIME-META-PERSISTENCE-BOUNDARY-P4A` (the Dino
  `Frame 'groups' must have flat columns` failure). Making configure relations
  *durable* removes exactly that containment. Slice 2 cannot just delete the
  prune; it must *replace* the containment with a control that survives durable
  relations. The carrier-neutral boundary cannot keep a relation for reverse
  (reimport) cleanup while also stripping it on the forward sidecar, so the
  containment must move out of the boundary.
* *Over-stated removal:* `produced_by` must *not* be deleted. Its
  persistence-discriminator role goes away, but `_assert_no_relation_conflict`
  (`fk_relations.py:240`) still reads `produced_by.step` for cross-producer
  collision detection. Slice 2 retires the *discriminator role*, keeps the
  *field*.

With those two corrections the slice is sound: stop writing v1, let
`configure_fk_helpers` relations persist like inferred/user-authored ones,
retain `produced_by` as provenance + conflict key, and add a single
compensating safety control so durable relations cannot reproduce the Dino
crash. The behavior gate is the existing falsification test
`test_helper_cleanup_uses_fk_policy_when_view_helper_metadata_is_absent`, which
must keep passing — *via durable v2 instead of v1*.

== Why is v1 currently load-bearing?

It is load-bearing on exactly one path, and *only because configure-produced v2
is non-durable*:

. `configure_fk_helpers` writes *both* shapes in one call: the v1 per-target
  dict (`helper_policies.py:141-142`, `fk_policies[policy["target"]] = policy`)
  and v2 relations tagged `produced_by.step = "configure_fk_helpers"`
  (`helper_policies.py:147-149` → `_v2_relations_from_explicit_policies:178-188`).
. The persistence boundary *prunes* those configure-produced v2 relations
  (`_is_runtime_produced_fk_relation`, `persistence_boundary.py:181-187`;
  applied in `_project_helper_policies:164-176`) but *preserves* the v1 dict.
  So after the forward write, the persisted / workbook-embedded meta carries v1
  but **not** configure-v2.
. On reimport, runtime provenance `_meta.derived.*` was dropped at egress
  (`RUNTIME_ONLY_TOP_LEVEL_META_KEYS = {"derived"}`,
  `persistence_boundary.py:101`). If the forward run also did not record
  `_meta.sheets.<frame>.helper_columns` (e.g. `configure_workbook_view` was
  called without a `helper_columns` list), then the *only* surviving
  helper-identity carrier is the v1 dict.
. `apply_derived_column_policy` unions its three identity sources
  (`derived_column_policy.py:75-91`: derived provenance ∪ durable
  `sheets.helper_columns` ∪ policy). With the first two empty and configure-v2
  pruned, the only non-empty contributor is `_v1_policy_helper_names`
  (`derived_column_policy.py:294-325`), which reconstructs
  `{helper_prefix}{target_frame}_{field}` from the v1 dict and feeds cleanup.

*Proof / gate:* `tests/roundtrip/test_cleanup_invariants.py::test_helper_cleanup_uses_fk_policy_when_view_helper_metadata_is_absent`
(line 54) builds a forward YAML whose `configure_workbook_view` carries *no*
`helper_columns`, then asserts `_entities_name` is absent from canonical
`items.json`. Today it passes only because v1 reconstructs the name.

*Corollary:* the baseline roundtrip is **not** v1-dependent. The
`minimal_fk_workdir` forward pipeline sets
`configure_workbook_view ... helper_columns: ["_entities_name"]`
(`tests/roundtrip/conftest.py:74-75`), so
`test_helper_columns_absent_after_canonical_reimport` cleans via
`_meta.sheets.items.helper_columns` (PATH-10), not v1. v1's *sole* live role is
the "view-helper-metadata-absent" path above. Remove the reason it is the only
survivor — make configure-v2 durable — and v1's role disappears.

== What exact code change makes v2 durable enough to replace v1?

Three coordinated changes (producer, boundary, cleanup fallback), plus one
compensating safety control (see <<dino>>). None changes the v2 *relation
shape*; `schema_version` is untouched.

=== 1. Producer: write only durable v2 (`domain/helper_policies.py`)

* Remove the v1 write loop in `configure_fk_helpers`
  (`helper_policies.py:141-143`):
+
[source,python]
----
for policy in policy_specs:
    fk_policies[policy["target"]] = policy   # <- delete this v1 write
helper_policies["fk"] = fk_policies          # keep the fk root / lookup ns
----
+
Keep the v2 emission (`_v2_relations_from_explicit_policies` →
`apply_v2_relations`). `_build_fk_policy` and the resolved `policy_specs` are
still needed *internally* (they compute `fk_column`, `target_key`,
`helper_columns`); only the *persisted v1 dict* is dropped. The locals
`helper_prefix` / `helper_fields` inside `_v2_relations_from_explicit_policies`
(`:164-165`) remain — they build `helper_columns[*].column` / `target_field`.

* `configure_fk_helpers` continues to tag relations
  `produced_by.step = "configure_fk_helpers"`, `mode = "explicit"`
  (`:185-186`). The tag is now provenance only (see
  <<produced-by>>); it no longer triggers any prune.

=== 2. Persistence boundary: stop pruning configure relations (`pipeline/persistence_boundary.py`)

Durability is achieved by *removing the special case*, not by adding a new
field. After this, `configure_fk_helpers` relations persist exactly like
`infer_fk_relations` / user-authored relations already do.

* Delete `_is_runtime_produced_fk_relation` (`:181-187`) and the constant
  `_RUNTIME_FK_HELPER_PRODUCER_STEP` (`:109`).
* In `_project_helper_policies` (`:148-178`) remove the relations-pruning block
  (`:164-176`), including the paired empty-`relations`/`schema_version`
  cleanup. With no prune the list never empties on this path, so
  `schema_version` is preserved naturally — *not* by touching the field, but by
  removing the code that used to drop it.
* `_project_helper_policies` then reduces to a near pass-through for
  `helper_policies` (keep the `Mapping` robustness guard at `:155-156`). The v1
  per-target dict is no longer produced; if an *old* sidecar still carries one
  it simply passes through and is inert (its only reader is deleted in step 3).
  Optional hygiene (not required by this slice): strip stray `fk.<target>`
  dicts on write. Recommend deferring that to keep the diff minimal.

This is the one genuinely load-bearing edit: it reverses the narrow
containment from `BUG-RUNTIME-META-PERSISTENCE-BOUNDARY-P4A`. It is safe *only*
together with the compensating control in <<dino>>.

=== 3. Cleanup fallback: drop the v1 reader (`domain/transformations/derived_column_policy.py`)

* Delete `_v1_policy_helper_names` (`:294-325`).
* In `_safe_policy_helper_names` (`:271-291`) remove the v1 call (`:278`) and
  start from an empty set; keep the v2-relation reading (`:279-290`). After
  Slice 2 the durable configure-v2 relation that now survives persistence is
  what this function reads on reimport — the same code path inferred relations
  already use.

=== 4. Schema maintenance: drop the dead v1 branch (`domain/schema_maintenance/meta_update.py`)

* In `_handle_helper_policies` (`:297-349`) remove the v1 blocked-reference loop
  (`:330-345`, "Residual v1 FK helper policy is not rewritten in this slice").
  No producer writes v1 after step 1, so the branch is dead. Keep
  `_handle_fk_relations` (`:352-409`) — it renames `relations[*].source_column`
  / `target_key` / `helper_columns[*].target_field` and blocks unsafe drops;
  that is the durable-v2 rename/drop behavior and is unchanged.

=== Net effect

After these four edits, the durable carrier for the `configure_fk_helpers`
reimport-cleanup path is the v2 relation list — identical in shape and
lifetime to what `infer_fk_relations` already produces. `resolve_v2_fk_relations`
(`fk_helpers/policy.py:32-58`) reads it unchanged. One model per concern, as
`fk_helpers.adoc` §Simplification Note prescribes ("v1 per-target config
treated as version mismatch → dropped, not migrated").

[#produced-by]
== What happens to `produced_by`?

*Kept as a field; demoted from discriminator to provenance.*

* *Removed role:* its only persistence-time consumer,
  `_is_runtime_produced_fk_relation` (`persistence_boundary.py:181-187`), is
  deleted. `produced_by.step == "configure_fk_helpers"` no longer means
  "prune me."
* *Retained role:* `_assert_no_relation_conflict` (`fk_relations.py:240-248`)
  still reads `produced_by.step` to enforce the cross-producer collision rule
  (two configuration steps must not silently overwrite each other on the same
  `(source_frame, source_column)` key). That is a behavior contract independent
  of persistence and must stay. `produced_by.mode` remains diagnostic origin
  metadata.
* *Consequence:* configure-produced relations now persist with
  `produced_by = {step: configure_fk_helpers, mode: explicit}` intact. This is
  fine — once relations are durable, all producers coexist symmetrically and
  the field is honest provenance, not a lifetime signal.

This *refines* the Slice-1 review, which listed "retire the `produced_by`
discriminator." Code shows `produced_by` has a second, behavior-bearing
consumer, so fully deleting the field is out of scope for Slice 2 and would
break conflict detection. Retiring the *discriminator role* is the correct,
bounded change. (Whether the field can ever be fully removed is a separate
question tied to the conflict-detection design — not this slice.)

[#dino]
== The principal risk and its compensating control: Dino replay

`BUG-RUNTIME-META-PERSISTENCE-BOUNDARY-P4A` forensics (that ticket, "Forensic
analysis result") proved the contamination carrier was the v2 `relations` list,
*not* v1: keeping `relations + schema_version` (v1 removed) *reproduced* the
failure; keeping v1 (relations removed) *restored* success. The failure shape:
a promoted `_meta.yaml` carrying a durable relation for
`source_frame=groups, target_frame=places` caused a later forward export to run
`add_fk_helpers` over `groups` while its columns were non-flat, materializing a
tuple-shaped helper and crashing downstream in `join_frames`
(`Frame 'groups' must have flat columns`,
e.g. `domain/transformations/join_views.py:209`).

Slice 2 deliberately makes configure relations durable, which *re-enables that
exact carrier*. The boundary prune was the containment; the carrier-neutral,
post-pipeline boundary cannot both keep a relation (so reverse-pipeline cleanup
can read it) and strip it (so forward-replay can't re-enrich). So the
containment must move.

*Existing partial guards (not sufficient alone):*

* `apply_fk_helpers` skips a helper column that already exists
  (`core/fk.py:308-310`, `if helper_col in first_cols: continue`) — prevents
  *same-name* duplication but not the *non-flat-frame* crash, which happens
  before/around column add (`core/fk.py:335` adds a MultiIndex tuple column).
* `enrich_helpers` skips a relation whose source column is absent
  (`fk_helpers/enrich.py:71-75`, `source_frame_has_column`) — but a stale
  relation whose source column *is* present on a non-flat frame still proceeds.

*Required compensating control (the load-bearing prerequisite of Slice 2):*
make `enrich_helpers` *defensive about frame applicability* so a durable
relation it cannot safely materialize becomes a *safe skip*, not a crash.
Minimal form: before materializing for a sheet, require the source frame to be
in the flat-column state the materialization path assumes (the same precondition
`join_frames` later enforces); if not, skip that relation (optionally emit a
diagnostic) rather than producing a tuple-shaped helper. This converts the Dino
crash into a no-op and is consistent with the functional model's Drop-step
principle ("the system does not guess silently"). It keeps durable relations
safe regardless of promotion.

*Why not the boundary or a load-time sanitizer?* A load-time `_meta` sanitizer
and a general `drop_if_stale` were explicitly *deferred* by the BUG ticket to
`FTR-FK-HELPER-POLICY-LIFECYCLE-P4A` and the meta-lifecycle framework. They are
broader than this slice and would drift toward the global-ontology work this
project is avoiding. The enrich applicability guard is the smallest control
that neutralizes the specific crash Slice 2 reopens.

*Scope note:* this control is in scope for Slice 2 because without it Slice 2
regresses a shipped fix. It is the only addition beyond removing v1.

== Consumer / migration risk

* *No demo regresses.* All three FK-touching demos use `infer_fk_relations`
  (already durable); none uses `configure_fk_helpers`. Making configure durable
  makes it symmetric with the infer path the demos already rely on.
* *Transitional reimport leakage (the real migration risk).* Artifacts produced
  by *pre-Slice-2* code carry the v1 dict (kept) but **not** configure-v2
  (pruned). After Slice 2 deploys, reimporting such an artifact *without a fresh
  forward run* finds: v1 now ignored (reader deleted), configure-v2 never
  persisted, and — if `sheets.<frame>.helper_columns` was not recorded — *no
  cleanup anchor at all* → helper columns leak into canonical output for that
  one transitional reimport.
  ** *Mitigation:* re-run the forward pipeline once under Slice-2 code to
     re-bootstrap durable configure-v2 (this matches the BUG ticket's
     "contamination cycle breaks at the next successful run" model). Consumers
     that already set `configure_workbook_view ... helper_columns` (like the
     roundtrip fixture) are unaffected because they clean via
     `sheets.helper_columns`.
  ** `schema_version` is *not* bumped (per task constraint). A bump would force
     re-bootstrap and eliminate this transitional window, but it is out of
     scope; the window is accepted with the migration note above.
* *Dino replay (controlled by the compensating control).* A consumer using
  `configure_fk_helpers` *and* the "promote `_meta.yaml` into canonical data"
  workflow could otherwise re-trigger the Dino crash on a later export. The
  enrich applicability guard (<<dino>>) neutralizes it; without that guard,
  Slice 2 must not ship. Note the infer path already carries this theoretical
  exposure today, so the guard also hardens the existing durable path.
* *Public surface unchanged.* `configure_fk_helpers` YAML/args are identical;
  only the persisted `_meta` shape shrinks (no v1 dict). Per project stance this
  is a deliberate, migratable change for controlled consumers.

== Non-Goals (Slice 2)

* No Slice 3 work (dead step params, `label_field`, `FKFinding`,
  single-member `_VALID_MODES`).
* No change to `schema_version` value or semantics.
* No change to `derived.sheets.*.helper_columns`,
  `sheets.<frame>.helper_columns`, or `helper_columns[*]`.
* No full removal of the `produced_by` field (conflict-detection consumer
  remains).
* No general load-time `_meta` sanitizer or `drop_if_stale` engine
  (deferred to `FTR-FK-HELPER-POLICY-LIFECYCLE-P4A`); only the narrow enrich
  applicability guard.
* No new global metadata ontology; no generalization to `legend_blocks` /
  `xref_crosstable`.
