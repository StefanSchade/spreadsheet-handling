== Executive Summary

The FK Helper consolidation does what it set out to do. Slice 1 (`cbc3b6e`)
removes the two unread v2 relation fields (`helper_fields`, `helper_prefix`)
and pins the surviving shape with a regression test. Slice 2 (`ba14230`)
retires the v1 per-target producer write, makes `configure_fk_helpers`
relations durable across the persistence boundary, and moves the Dino-shaped
replay containment from a boundary prune to two materialisation-point controls
(`_preserve_source_flatness` and the absent-target skip). The runtime contract
for `configure_fk_helpers` is now coherent: no v1 write, durable v2 relations,
`produced_by` retained only as provenance / cross-producer conflict key. The
registry fix (`8952050`) corrects an unrelated `enrich_lookup` provenance
reference.

The *runtime* implementation is correct and well covered by tests. The
replay/accumulation regression is genuinely closed and proven by an end-to-end
test that exercises the real replay (not just the old boundary prune). On code
alone this would be Green.

It is not clean, however, because several **contract surfaces that this project
treats as operational / oracle truth still describe the retired v1 producer
write as active**. The `configure_fk_helpers` entry in
`pipeline_step_registry.json` still *declares a v1 write* and an *active
bridge*; the canonical `technical_model` bridge section still says "the
residual v1 dict is still written"; `semantic_model` and the `user_guide`
Known-limitation list still describe v1-blocking / dual-write behaviour that
Slice 2 deleted. None of these are caught by the architecture guards (the
guards validate `meta_registry.yaml`, which *was* updated, but not the
pipeline registry's free-text notes or the prose docs). There is also a
documented but undetected migration hole for pre-Slice-2 artifacts.

Verdict: *Yellow*. The durable v2 contract is sound and shippable; the stale
registry/doc surfaces (F1, F2) should be reconciled before the next slice
because the next slice will read them as truth, and the migration risk (F3)
should be confirmed against the controlled consumer.

NOTE: F1/F2 (registry + technical model) and F4/F5 (user guide, release notes,
semantic model) were subsequently implemented (2026-06-14) — see "Follow-up —
F1 / F2 reconciliation", the per-finding RESOLVED status lines, and the updated
Verdict. The verdict remains Yellow solely on the still-open F3; F6–F8 remain
non-gating tracked debt.

No findings in Slice 1.

== Slice 2 Assessment

contract surfaces that were not brought along* (F1, F2, F4, F5).

== Replay / Accumulation Safety

Review question 2 — the old Dino replay/accumulation bug — is blocked.

* *Flatness preservation.* `_preserve_source_flatness`
  (`enrich.py:130-160`) collapses the MultiIndex-padded helper tuples back to
  their first-level label *only when the source frame was flat*. This is the
  real fix: `apply_fk_helpers` adds helpers as `(_helper,) + ("",)*(levels-1)`
  (`core/fk.py:336`), which on a flat source would otherwise yield a mixed
  flat/tuple column index and trip the downstream
  `Frame ... must have flat columns` guard. Empirically validated:
  `test_repeated_run_does_not_replay_stale_relation` runs the *actual* replay
  with `levels=3` over a flat `groups` frame and asserts no tuple columns
  remain.
* *Missing-target skip.* `enrich.py:54-66` filters relations whose
  `target_frame` is not loaded, before `build_id_value_maps`, preventing the
  crash on a replayed relation whose target is absent
  (`test_durable_relation_with_absent_target_is_skipped`).
* *No accumulation.* `apply_fk_helpers` short-circuits
  `if helper_col in first_cols: continue` (`core/fk.py:309`); the repeated-run
  test asserts `count("_places_name") == 1` across two runs.

These controls are sufficient for the documented failure mode. See F6/F7 for
two minor robustness gaps in the skip, and F8 for a missing structured-source
regression test for `_preserve_source_flatness`.

== Reimport Cleanup Assessment

Review question 4 — reimport cleanup is safe for *forward-produced* artifacts.

`apply_derived_column_policy` resolves helper identity from, in order:
transient `_meta.derived.sheets[source]`, durable
`_meta.sheets[source].helper_columns`, and the v2 policy fallback
`_safe_policy_helper_names` (`derived_column_policy.py:271-295`). After Slice 2

in `ba14230` from a v1 dict to a v2 relation). This is exactly the property
that *justifies* making v2 durable, and it holds.

== Migration / Consumer Impact

Review question 5.

* *Old artifacts with v1-only sidecars (F3, MAJOR).* A pre-Slice-2 *forward*
  run persisted the v1 per-target dict but the *old* boundary pruned
  configure-produced v2 relations — so such sidecars carry v1 only and no v2
  anchor, and their transient `derived` was pruned too. Reimporting one with
  the Slice-2 cleanup finds neither a v2 relation nor the deleted v1 fallback,
  so FK helper columns are silently retained as payload. This is documented in
  `release_notes.adoc:166-172` with a manual mitigation ("re-run the forward
  pipeline once"), but there is no read-side detection. The original Dino
  artifacts came from a controlled worldbuilding adoption; confirm those have
  been re-bootstrapped (or are not reimported through the derived-column
  policy) before relying on the new path.
* *Schema-maintenance behaviour change (covered under F4).* Running
  `sheets-schema-maintain` rename/drop over an old artifact that still carries a
  v1 dict referencing the affected frame/column no longer *blocks*; the inert
  v1 dict passes through with a now-stale reference. Harmless at runtime (no
  reader), but it is a silent change from the previously-conservative
  "block what cannot be safely rewritten" behaviour, and the docs still
  advertise the old behaviour.
* *Release-note migration guidance.* Present and placed under the active
  unreleased section (`== unknown -- planned`), satisfying the placement rule.
  Public YAML/CLI surface is unchanged; only the persisted `_meta` shape
  shrinks.

== Risks and Missing Tests

* `_preserve_source_flatness` (review question 7): the structured-source guard
  (`enrich.py:153`) is exercised only on the flat-source collapse path; the
  "genuinely MultiIndex source is left untouched" branch has no test (F8).
  MultiIndex handling, helper-formula behaviour, and ODS/XLSX rendering are
  otherwise unaffected because the collapse triggers only when the source was
  flat (in the normal forward path the source is already MultiIndex, so the
  function returns `enriched_df` unchanged and rendering is unchanged).
* Absent-target skip (review question 8): correct as a no-op per the functional
  model, but silent and not distinguishable from a real mid-pipeline
  misconfiguration (F6); and the skip uses exact name matching where the
  registry resolves by `normalize_sheet_key` (F7).
* No test pins the new schema-maintenance behaviour over a residual v1 dict
  (now a pass-through, formerly a block).

[[FKPI-F6-ABSENT-TARGET-SKIP-SILENT]]

*MINOR.*

*Evidence:* `enrich.py:54-66` filters relations whose `target_frame` is not in

absent target so the no-op is diagnosable; the no-op semantics themselves are
correct per the functional model. Consider a test for the mid-pipeline-drop
case.
