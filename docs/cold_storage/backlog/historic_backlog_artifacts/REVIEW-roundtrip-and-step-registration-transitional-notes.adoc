== From the roundtrip-layout concept — multi-header transitional hybrid

Context: FTR-MULTIHEADER-P2.

* During P2, multi-row headers + merge regions are emitted via IR `RenderPlan`.
* Data cells are still written via pandas first, then IR ops are overlaid on the
  workbook.
* Reason: pandas cannot reliably write `MultiIndex` data in the exact mode
  needed for IR-driven header control (`header=False` + `index=False`) without
  edge-case failures.
* Practical fix in the writer path: for the data-only write, a temporary
  DataFrame copy gets simple numeric columns; values and column order remain
  unchanged; header rows and merge geometry are then written by IR ops
  (`SetHeader`, `MergeCells`).
* Architectural status: this is an *intentional transitional hybrid, not the
  final shape*. It keeps behaviour stable while enabling P2 multi-header parity
  in XLSX output.
* Full single-pass IR write (including data cells as render ops) is tracked
  separately as *FTR-IR-DATA-CELLS*.

Roadmap position (as written at migration time): land after IR parity P1/P2
work, before IR becomes default and legacy removal; related features
FTR-IR-WRITEPATH-P1, FTR-MULTIHEADER-P2, FTR-IR-TYPING-CANONICAL. *Re-check this
ordering against the current backlog before relying on it.*

== From the step-registration concept — first migration candidates

The pipeline-step-registration concept names typical first migration candidates
(steps that thematically belong in domain but currently live in `pipeline`):

* `flatten_headers`
* `unflatten_headers`
* `reorder_fk_helpers`
* `bootstrap_meta`
* `apply_overrides`
* `add_validations`

Fat factories `add_fk_helpers` / `remove_fk_helpers` are thematically wrong in
`pipeline` too, but extracting them requires first moving their domain logic out
— that is the FK-helper domain extraction (see `FTR-DOMAIN-CONSOLIDATION.adoc`).
Once that is done, those factories can use the generic binding path like any
other.
