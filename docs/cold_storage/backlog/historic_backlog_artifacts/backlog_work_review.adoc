==== 1. Dependencies im FTR sind wahrscheinlich zu stark

* `FTR-FORMULA-PROVIDERS`
* `FTR-RAW-FORMULA-BOUNDARY-GUARDS-P4`
* `FTR-ADAPTER-CONTRACT-BOUNDARY-GUARDS-P4`
* `FTR-ARCHITECTURE-GUARD-INVENTORY-P4`

Für den jetzt umgesetzten narrow current-seam Guard braucht ihr eigentlich nur:

* `FTR-ARCHITECTURE-GUARD-INVENTORY-P4`

Die anderen sind eher `Related-to`, nicht echte Blocker.

Empfehlung: Dependencies schwächen, damit das Backlog keine falsche Reihenfolge suggeriert.

==== 2. `CARRIER_OR_RENDERING_META_KEYS` ist hart gegen den aktuellen Registry-Seed

Das ist für einen narrow Guard akzeptabel. Aber der Test setzt voraus, dass jeder dieser Keys existiert:

[source,python]
----
entry = entries[key]
----

Das ist gewollt, wenn diese Keys Teil des Registry-Contracts sind. Falls neue Registry-Versionen Keys entfernen oder umbenennen dürfen, wäre das zu streng.

Aktuell: akzeptabel.

==== 3. Test-Carrier-Änderungen sind wieder im Diff enthalten

Der Test-Carrier-Guard und README-Ergänzungen sind sinnvoll, gehören aber logisch zu `FTR-TEST-CARRIER-BOUNDARY-GUARDS-P4`, nicht zu diesem FTR.

Kein Problem, wenn Codex das bewusst als parallel vorhandenen Worktree-Zustand behandelt. Für Commit-Hygiene wäre aber besser:

* Meta-Carrier-Commit nur mit Meta-Carrier-Dateien
* Test-Carrier-Commit separat
