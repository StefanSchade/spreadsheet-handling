====== Purpose

Introduce a backend-neutral formula representation (`FormulaSpec`) and keep
backend-specific formula syntax isolated to adapter translation.

This FTR establishes the boundary with the smallest useful production slice:
supported list-validation formulas. It prepares later formula providers without
implementing formula-driven FK helper cells or lookup formulas in this ticket.

'''

====== Problem statement

Raw formula strings are an easy short-term path but create long-term coupling:

* generic layers become backend-specific
* tests validate syntax strings instead of semantics
* ODS/XLSX behavior drifts

Formula support must preserve architecture boundaries.

'''

====== Goals

. Represent formula intent structurally above adapter boundaries.
. Translate formula specs to XLSX/ODS syntax only in adapter-facing modules.
. Keep `DataValidationSpec` and `AddValidation` backend-neutral.
. Provide tests that assert semantic equivalence at generic layers.

'''

====== Scope

This FTR includes:

* add a minimal `FormulaSpec` representation for explicit list validations
  produced from canonical `in_list` constraints
* carry that `FormulaSpec` through `DataValidationSpec` and `AddValidation`
  instead of adapter-facing strings
* translate `FormulaSpec` to XLSX/ODS syntax inside concrete adapter modules
* reconstruct supported parser-visible list validations back into `FormulaSpec`
* update tests to separate semantic assertions from backend string assertions

'''

====== Non-goals

This FTR does not:

* define a complete end-user formula language
* guarantee support for all spreadsheet formula functions
* expose backend syntax as a generic API
* implement formula-driven FK helper cells, `XLOOKUP`, or `VLOOKUP`
* introduce a broad named-reference formula language

'''

====== Backlog split decision

The formula work is intentionally split:

* This FTR owns the durable boundary and the first supported validation-formula
  slice.
* `FTR-FORMULA-FK-HELPER-PROVIDERS-P4A` owns later formula-driven FK helper
  cells and lookup formulas once the boundary has proven stable.

'''

====== Acceptance criteria

. References resolve correctly for supported formula use cases.
. No hard-coded worksheet coordinates are required in generic layers.
. Generic-layer tests assert semantic intent (structure/equivalence), not raw
  backend formula strings.
. Backend-specific string assertions are limited to adapter/translator tests.
. A violation is detectable if raw backend syntax is introduced above the
  adapter/formula-translation boundary.
. Supported XLSX and ODS list validations render and parse back to the same
  `FormulaSpec` values.

'''
