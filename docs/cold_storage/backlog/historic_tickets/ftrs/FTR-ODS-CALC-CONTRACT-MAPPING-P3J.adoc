===== FTR-ODS-CALC-CONTRACT-MAPPING-P3J -- Map the stabilized spreadsheet contract to LibreOffice Calc / ODS capabilities

*Depends-on:* <<FTR-SPREADSHEET-PARSE-CONTRACT-P3I#,FTR-SPREADSHEET-PARSE-CONTRACT-P3I>>, <<FTR-META-REGISTRY-HARDENING-P3I#,FTR-META-REGISTRY-HARDENING-P3I>>, <<FTR-SPREADSHEET-BOUNDARY-GUARDS-P3I#,FTR-SPREADSHEET-BOUNDARY-GUARDS-P3I>>, <<FTR-SPREADSHEET-SEMANTIC-INVARIANTS-P3I#,FTR-SPREADSHEET-SEMANTIC-INVARIANTS-P3I>>

*Context:* Phase 3h made the spreadsheet architecture explicit, and Phase 3i is expected to harden the contract. Before implementing a full ODS/Calc adapter, the project should first map that stabilized contract to actual LibreOffice Calc / ODS capabilities and decide where approximation, omission, or alternate carrier choices are acceptable.

*Problem statement:* Jumping straight into an ODS implementation would risk rediscovering capability gaps inside code, where they are harder to review and easier to encode as accidental adapter policy. A capability/contract mapping step is the cheaper place to decide what the second adapter is really expected to support.

*Goals:*

. Derive an explicit ODS/Calc capability profile from the stabilized spreadsheet contract.
. Identify which shared semantics are expected to work directly, which need backend-specific carrier decisions, and which are accepted gaps for an initial adapter.
. Clarify how persisted workbook metadata, hidden/meta carrier choices, validations, freeze/filter behavior, and naming constraints map into ODS/Calc.
. Produce a concrete implementation split for the later adapter ticket instead of leaving Phase 3j as one opaque bundle.

*Acceptance criteria:*

. The docs contain an explicit capability/gap mapping from shared spreadsheet semantics to ODS/Calc behavior.
. Known uncertainty points are called out as decisions or open technical risks before implementation begins.
. The mapping is concrete enough to define the initial supported scope of the ODS adapter and to keep parity tests honest about accepted gaps.
. The work stays architecture-first and does not already smuggle implementation shortcuts into the shared contract.

*Non-goals:*

- No shipping ODS/Calc adapter code in this FTR.
- No promise of full feature parity with XLSX.
- No attempt to redesign the shared spreadsheet contract around ODS-only needs.

*Why now:* A second adapter should validate the contract that Phases 3h/3i stabilized, not become the place where that contract is still negotiated.
