== Review: FTR-GUARDRAIL-DECISION-RECORD-P4

Reviewed diff: `docs/internal_guide/architecture/architecture_guard_inventory.adoc`.

=== Gesamturteil

Der GDR-Slice ist sinnvoll und passt gut an diese Stelle. Er verhindert genau das Problem, das bei den Guard-FTRs droht: Guards werden technisch eingeführt, aber die zugrunde liegende Boundary-Entscheidung bleibt implizit.

Ich würde den Slice akzeptieren, mit zwei kleinen Nachschärfungen.

=== Positiv

* Das Template ist kurz genug, um tatsächlich benutzt zu werden.
* Die Pflicht zur Unterscheidung `durable-invariant` vs. `current-state` ist genau richtig.
* Exceptions müssen eine Sunset- oder Review-Bedingung haben — das ist wichtig gegen dauerhafte Allowlist-Schulden.
* Die Wiederverwendung derselben Guard ID zwischen Design-FTR und Enforcement-FTR ist pragmatisch und verhindert künstliche ID-Vermehrung.
* Der Ansatz bleibt dokumentarisch und führt keine neue Tooling-Schicht ein.

=== Nachschärfung 1: `Owner` kann in kleinen Projekten zu schwer wirken

Das Feld `Owner` ist grundsätzlich sinnvoll, kann aber in einem kleinen oder solo-getriebenen Projekt bürokratisch wirken.

Empfehlung: Begriff leicht entschärfen:

[source,text]
----
Owner / Review role:
----

Oder im Text ergänzen:

[source,adoc]
----
`Owner` may be a role rather than a named person, e.g. `architecture review` or `formula-provider implementation`.
----

=== Nachschärfung 2: `Current Wave Conformance` behauptet Seed-GDRs

Der Abschnitt sagt:

> The current active guardrail backlog already carries seed GDR entries...

Das ist nur dann korrekt, wenn diese vier FTRs tatsächlich bereits ausgefüllte oder explizit angekündigte GDR-Einträge enthalten.

Falls das repo-seitig durch Scan bestätigt wurde, ist es ok. Falls nicht, sollte die Formulierung schwächer sein:

[source,adoc]
----
The current active guardrail backlog is expected to carry seed GDR entries for the main follow-up seams before implementation starts:
----

oder:

[source,adoc]
----
The current active guardrail backlog should be aligned with this template for the main follow-up seams:
----

Damit vermeidet ihr eine falsche Dokumentationsaussage.

=== Nicht blockierend

* Kein Testlauf nötig; der Slice ist rein dokumentarisch.
* Ein AsciiDoc-Build wäre nett, aber nicht zwingend, solange der Include-Kontext bereits funktioniert.
* Die Platzierung im Guard Inventory ist akzeptabel. Langfristig könnte GDR ein eigenes kleines Dokument werden, aber jetzt ist Inline besser.

=== Entscheidung

*Recommendation:* accept `FTR-GUARDRAIL-DECISION-RECORD-P4` with minor wording cleanup.

=== Blocker

Keine.

=== Empfohlene minimale Änderung vor Commit

. `Owner` als Rolle erlauben.
. `Current Wave Conformance` nur so stark formulieren, wie die FTRs tatsächlich bereits GDR-Einträge enthalten.
