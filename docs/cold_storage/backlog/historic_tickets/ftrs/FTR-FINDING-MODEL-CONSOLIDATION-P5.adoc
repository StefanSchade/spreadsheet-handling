===== FTR-FINDING-MODEL-CONSOLIDATION-P5 -- Consolidate domain finding records and finding-frame serialization

*Status:* Done — independent review ACCEPTED 2026-05-18 (see `FTR-FINDING-MODEL-CONSOLIDATION-P5_review.adoc`); archived to `docs/cold_storage/backlog/historic_tickets/ftrs/`
*Priority:* P5
*Area:* Domain validations / diagnostics / maintainability
*Kind:* Behavior-preserving consolidation

==== Purpose

Consolidate the parallel finding record models used by domain validation and diagnostic-producing transformations into one small, shared domain finding representation and one shared finding-frame serialization path.

REVIEW-002 Pass 1 identified `D-005` as a remaining domain finding-model duality. REVIEW-002 Pass 2 refined this into finding-model quadruplication: several modules independently define finding columns, dataclasses or record shapes, `to_record` conversion, `findings_to_frame`, and failure-message formatting.

This FTR exists to remove that accidental duplication before larger validation-module refactors, especially any future split of `reference_validations.py`.

==== Scope

Unify finding representation and tabular serialization across the currently duplicated domain finding producers.

Initial targets:

* `domain/validations/findings.py`
* `domain/validations/reference_validations.py`
* `domain/transformations/resource_overrides.py`
* `domain/transformations/derived_column_policy.py`

The exact implementation shape should be derived from the current code, but the intended direction is a small domain-level finding module that owns:

* canonical finding columns;
* a common frozen finding record or compatible protocol;
* conversion to a record dict;
* conversion from a list of findings to a pandas DataFrame;
* consistent empty-frame shape;
* common failure-message helpers where semantics are truly shared.

==== Non-Goals

* No redesign of validation semantics.
* No change to pipeline step names.
* No change to YAML configuration.
* No change to severity behavior.
* No change to which findings are emitted.
* No change to finding categories unless explicitly required for compatibility.
* No broad validation framework.
* No plugin system.
* No migration to exceptions-as-findings.
* No reference-validation module split in this FTR.
* No resource-overrides or derived-column-policy behavior changes.

==== Behavior-Preservation Requirements

* Existing finding frames must keep their externally observed column order unless a test proves the old shape was internal-only.
* Existing category strings must remain stable.
* Existing mode/severity behavior must remain stable.
* Existing failure messages should remain stable where tests or user-facing diagnostics depend on them.
* Existing empty finding frames must remain schema-stable.
* Existing tests for reference validations, resource overrides, derived-column policy, and FK/helper validations must pass unchanged unless a test only encoded duplicated implementation shape.

==== Design Constraints

* The shared finding model belongs in the domain layer.
* Transformations may produce findings, but they should not each reinvent finding-frame serialization.
* The pipeline may apply configured severity policy, but must not own finding data modeling.
* Keep the consolidation small and concrete.
* Do not introduce a generic diagnostics engine.
* Do not collapse genuinely different diagnostic semantics into one vague record if local fields are semantically necessary.

==== Suggested Approach

. Inventory all current finding dataclasses, constants, `to_record` methods, `findings_to_frame` helpers, empty-frame helpers, and failure-message helpers.
. Identify the stable shared column subset and any module-specific columns.
. Decide whether a single `Finding` dataclass is sufficient or whether a small protocol / adapter function is safer.
. Move only the genuinely shared mechanics into one domain module.
. Convert target modules to use the shared mechanics while preserving emitted records.
. Add focused tests for shared finding-frame serialization.
. Run all affected module tests plus architecture/registry tests.

==== Acceptance

* Finding-frame serialization is no longer independently reimplemented in the four initial target areas.
* Existing observed finding output remains stable.
* Existing validation and transformation behavior remains stable.
* Reference validation tests pass.
* Resource override tests pass.
* Derived-column policy tests pass.
* FK/helper validation tests pass.
* The consolidation reduces duplication without introducing a broad diagnostics framework.
* The result is suitable as a prerequisite for a later `reference_validations.py` package-split FTR.

==== Review Guidance

The independent review should focus on:

* accidental diagnostic-shape drift;
* accidental category or severity drift;
* overgeneralization into a diagnostics framework;
* whether local diagnostic semantics were flattened too aggressively;
* whether the consolidation genuinely reduces duplication;
* whether the result makes a later `reference_validations.py` split easier.

Reject this FTR if it turns into a validation redesign rather than a behavior-preserving consolidation.

==== Delivery Scope (implementation pass — 2026-05-18)

Behavior-preserving consolidation only. Single repo (`spreadsheet-handling`,
branch `dev`); demo untouched.

*New shared module* `src/spreadsheet_handling/domain/finding_frame.py`
(domain-level, mirroring the existing `domain/frame_lifecycle.py` placement and
import pattern — guard-clean, no new package, no diagnostics engine). Owns:
the canonical seven-column `FINDING_COLUMNS`; a frozen `Finding` record with
`to_record`; `row_index_label`; a generic `findings_to_frame(findings, *,
columns=...)` serializer with a schema-stable empty frame; and
`simple_failure_message(findings, *, heading=...)` for the flat uncapped
failure shape.

*Converted producers:*

* `domain/transformations/resource_overrides.py` — removed its local
  `FINDING_COLUMNS`, `ResourceOverrideFinding`, `findings_to_frame`,
  `row_index_label`, `_failure_message`; now uses the shared model.
  `ResourceOverrideFinding` is preserved as a public alias of the shared
  `Finding` (every construction passes `severity` explicitly, so the old
  class-level `"fail"` default is behaviourally irrelevant).
* `domain/transformations/derived_column_policy.py` — removed its local
  `FINDING_COLUMNS`, `DerivedColumnFinding`, `findings_to_frame`,
  `_row_index_label`, `_failure_message`; now uses the shared model.
  `DerivedColumnFinding` preserved as a public alias; `FINDING_COLUMNS`
  re-exported (test imports it).
* `domain/validations/reference_validations.py` — `findings_to_frame` now
  delegates to the shared serializer. `ReferenceFinding`, its nine-column
  `FINDING_COLUMNS` (module-specific `target_frame`/`target_columns`
  extension), `to_record`, `_failure_message`, `_finding_line` are kept
  unchanged: this is a genuinely different diagnostic (raw `row_index`,
  JSON-encoded tuple `value`, capped/named failure message). Flattening it
  would breach the "do not collapse genuinely different diagnostics" boundary
  and is pinned by an existing test asserting integer `row_index` values.

*Deliberately NOT changed:* `domain/validations/findings.py::Finding` +
`apply_severity_policy` — a category/severity-policy model with no tabular
serialization. It is not part of the finding-frame quadruplication; collapsing
it would alter severity behavior (an explicit non-goal). `graph_validations.py`
left untouched (out of scope) — its `ReferenceFinding`/`findings_to_frame`
imports from `reference_validations` still resolve.

*Tests:* new focused `tests/unit/domain/test_finding_frame.py`. FTR-named
target suites, the broader domain validations/transformations slice, pipeline +
architecture, and the full active `make test` slice (*826 passed*) all pass.
Changed source files are ruff-clean; pre-existing ruff/source-style warnings in
untouched files were left alone (no unrelated cleanup).

*Acceptance mapping:* finding-frame serialization is no longer independently
reimplemented in the three frame-producing target areas (all delegate to one
serializer); observed finding output, categories, severity, and failure text
are stable; reference / resource-override / derived-column-policy / FK-helper
validation tests pass; no diagnostics framework introduced; the result removes
the D-005 quadruplication prerequisite blocking a later
`reference_validations.py` package split.
