===== FTR-MODULE-SPLIT-CLEANUP-P3D — Refactor large modules along established architectural boundaries

*Status:* Open
*Epic:* Phase 3d — Code Quality & Maintainability

*Purpose:*::
Reduce oversized and mixed-responsibility modules by splitting them along the architectural boundaries established in Phase 3b, improving readability, maintainability, and long-term evolvability.

*Context:*::
During earlier phases, modules grew organically while responsibilities were still being clarified. With domain, pipeline, and rendering responsibilities now better defined, modules can be split in a way that reflects actual architectural intent.

*Scope:*::
- Identify modules that mix domain logic, orchestration, and rendering concerns.
- Split modules along clear responsibility lines (e.g., transformations, validations, orchestration helpers).
- Ensure that new modules reflect stable architectural boundaries rather than arbitrary size limits.
- Maintain backward compatibility where necessary during transition.

*Non-Goals:*::
- No purely cosmetic splitting based on file length alone.
- No refactoring that reintroduces architectural ambiguity.
- No large-scale renaming without clear responsibility improvement.

*Design Principle:*::
Modules follow responsibility, not size. Splitting is driven by clear ownership boundaries, not formatting preferences.

*Acceptance:*::
- Previously oversized modules are reduced into smaller, responsibility-focused units.
- Each module has a clearly identifiable role within the architecture.
- Cross-module dependencies align with domain -> pipeline -> rendering direction.
- No increase in conceptual fragmentation or indirection.
