===== FTR-RUN-PATH-SHEET-OVERRIDES-P3G — Honor sheet-level rendering overrides in `sheets-run`

*Status:* Implemented

*Context:* During demo-repo reintegration, the curated product slice started using `apply_overrides` plus `add_validations` to show visible spreadsheet behavior. The generated workbook confirms that:

- list validations are applied;
- per-sheet filter differences can be observed;
- but `freeze_header` does not currently take effect through the active `sheets-run -> orchestrator -> pipeline -> xlsx` execution path.

This is not a change request for new behavior. The project already intends sheet-level rendering options such as `freeze_header`, `auto_filter`, and `header_fill_rgb` to flow from workbook/sheet meta into the final XLSX output. The missing freeze behavior is therefore treated as an integration defect in the current run path.

*Problem statement:* The runtime path that starts from `sheets-run` does not reliably carry workbook and per-sheet rendering meta all the way to the writer. Demo-driven verification exposed this specifically for `freeze_header`, while filters and validations continue to work. This indicates an inconsistency in how meta survives and/or is interpreted across the orchestration path.

*Goals:*

. Make sheet-level rendering overrides behave consistently in the `sheets-run` path.
. Preserve reserved non-table payload such as `_meta` safely across generic pipeline steps.
. Ensure demo-driven config files can express visible sheet differences without relying on alternate execution paths.

*Acceptance criteria:*

. A `sheets-run --config ...` workflow that uses `apply_overrides` with per-sheet `freeze_header`, `auto_filter`, and `header_fill_rgb` produces the expected visible sheet differences in the generated XLSX.
. The demo product slice can show:
.. `product` with filter enabled and freeze enabled;
.. `branch` with filter disabled and freeze disabled;
.. distinct header fills driven by sheet overrides.
. Reserved frame keys such as `_meta` are preserved across generic pipeline steps and are not treated as data frames.
. Add focused regression tests at the main-repo level for the active `sheets-run`/orchestrator path, not only for lower-level compose/render helpers.

*Non-goals:*

- No CLI redesign here; `FTR-CLI-CONSOLIDATE-REMOVE-STEPS` remains separate.
- No expansion of the override schema beyond making the existing intended behavior reliable.
- No new rendering feature surface; this is consistency and correctness work.

*Why now:* The project is still in an early beta phase and the demo repository is becoming the benchmark consumer. Fixing this inconsistency now is cheaper than documenting or releasing against a misleading partially-working configuration story.
