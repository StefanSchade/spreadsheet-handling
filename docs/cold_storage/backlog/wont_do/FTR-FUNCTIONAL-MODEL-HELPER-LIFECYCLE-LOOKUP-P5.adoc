===== FTR-FUNCTIONAL-MODEL-HELPER-LIFECYCLE-LOOKUP-P5 -- Model lookup helpers and derived-column cleanup

*Status:* Superseded / dropped (archived 2026-06-24)

Disposition::
The `helper lifecycle` grouping was rejected after Functional Model
simplification analysis. Lookup-helper and derived-column cleanup behavior
should be treated as FK-helper/helper-column sediment and future bridge-DSL
contract work, not as a separate Functional Model concept.

Evidence::
See
`docs/cold_storage/working_context/semantic_model_simplification/helper_lookup_sediment_analysis.adoc`
and
`docs/cold_storage/working_context/semantic_model_simplification/candidate_terms.adoc`.

*Type:* Functional Model documentation.

*Purpose:*::
Clarify the Functional Model coverage for implemented helper-lifecycle
capabilities that sit next to, but are not identical with, FK helpers.

*Context:*::
Derived from the June 2026 transformation/capability family inventory:
`docs/cold_storage/working_context/transformation_family_inventory_2026_06/transformation_family_inventory.adoc`.
The inventory marks `lookup_helpers` / `enrich_lookup` and
`derived_column_policy` as implemented families with partial Functional Model
coverage and classification caveats.

FK helpers already have direct Functional Model coverage. This FTR covers the
adjacent implemented capabilities without merging them into FK-helper semantics
or creating a new Project Memory hierarchy.

*Scope:*::
* Explain lookup-helper enrichment as readable derived context over a lookup
  frame, including the distinction from FK-helper relation policy.
* Explain derived-column cleanup on reimport as a projection/readback safety
  capability.
* Clarify how lookup-helper provenance, FK-helper provenance, and durable helper
  declarations relate at a functional level.
* State where this material belongs relative to
  `docs/functional_model/ch04_use_case_concepts/fk_helpers.adoc` and the
  problem-space overview.
* Reference the cold-storage inventory for evidence and avoid copying its
  family table.

*Non-Goals:*::
* No runtime implementation.
* No changes to `enrich_lookup`, `configure_lookup_helpers`, or
  `apply_derived_column_policy`.
* No new helper policy schema.
* No expansion of FK-helper semantics beyond what existing FK-helper FTRs own.
* No Project Memory entities.

*Acceptance Criteria:*::
* Functional Model material explains lookup helpers as a distinct but adjacent
  helper projection family.
* Functional Model material explains derived-column cleanup as a reimport
  safety/lifecycle capability.
* Boundaries to FK helpers are explicit.
* The June 2026 inventory is cited as provenance without duplicating inventory
  contents.
* The FTR remains documentation-only; no runtime implementation is performed
  unless separately approved.

*Relationship to the inventory and Functional Model:*::
The inventory remains the canonical June 2026 implemented-family reference.
This FTR turns the helper/lookup partial-coverage finding into bounded
Functional Model catch-up work.
