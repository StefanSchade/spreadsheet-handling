===== FTR-DYNAMIC-VIEW-COLUMN-TARGETING-IMPL-P5 -- Role-based rule targeting for dynamic workbook-view columns

*Status:* Implemented; review pending.

*Type:* Implementation slice.

*Depends on:*::

* `FTR-DYNAMIC-VIEW-COLUMN-TARGETING-P5_decision.adoc` -- source of
  truth for the accepted surface: a single optional `roles:` field on
  existing rule target schemas.
* `FTR-DYNAMIC-FRAME-PROJECTION-IMPL-P5` -- implemented foundation for
  the shared `spreadsheet_handling.domain.column_roles` resolver.

*Purpose:*::

Implement the smallest role-based target surface needed to replace the
worldbuilding-style `add_from_legend_constraints_for_matrix` fan-out:
validation / constraint metadata rules can address generated matrix
columns by semantic column role instead of enumerating concrete column
names.

This slice does not reopen the projection architecture decision and
does not introduce a selector language.

*Implemented surface:*::

`add_validations` rule targets now accept optional role targeting:

[source,yaml]
----
- step: add_validations
  rules:
    - target:
        sheet: story_groups
        frame: story_group_matrix_view
        roles: [matrix_value]
      rule:
        type: from_legend
        legend: story_group_codes
        include_empty: true
----

The historical flat rule shape remains valid, so existing rules do not
need to change:

[source,yaml]
----
- step: add_validations
  rules:
    - sheet: stories
      column: status
      rule:
        type: in_list
        values: [seed, sketch, written]
----

For role targets:

* `roles:` is optional.
* Accepted roles are exactly the foundation taxonomy:
  `row_identity`, `display_helper`, `matrix_value`.
* Unknown roles fail with a clear validation error.
* `roles:` is mutually exclusive with explicit `column:` / `columns:`.
* `frame:` is optional when the frame can be resolved from the target
  `sheet:` via workbook-view metadata or when the sheet name is also the
  frame name.
* Expansion delegates to
  `spreadsheet_handling.domain.column_roles.resolve_column_roles`.
  No second role taxonomy or detection path is introduced.

*Implementation placement:*::

Role expansion happens inside
`spreadsheet_handling.domain.validations.validate_columns:add_validations`.
That is the narrowest existing path because it still has access to the
loaded Frames mapping and writes `_meta.constraints`, the same
concrete per-column metadata shape that renderers already consume.

Rendering code is intentionally unchanged: after expansion,
`_meta.constraints` contains ordinary concrete `column` targets.

*Scope boundaries:*::

In scope:

* role-target expansion for `add_validations`;
* nested `target:` normalization for the decision-document shape;
* preservation of existing flat `sheet` / `column` rules;
* focused unit tests, a pipeline-style integration test, and an
  architecture guard for shared-resolver reuse;
* registry, public-surface gate, and release-note updates.

Deferred:

* presentation-target surfaces;
* other future workbook-constraint families if they diverge from
  `_meta.constraints`;
* worldbuilding YAML migration. The plugin remains the rollback path
  until the consumer repository is migrated in a separate commit.

Out of scope:

* new pipeline steps;
* wildcard syntax;
* set algebra;
* selector grammar;
* changes to `project_by_role`;
* changes to the foundation role taxonomy.

*Validation added:*::

* Unit tests for `matrix_value` fan-out, `display_helper` targeting,
  unknown role errors, `roles` + explicit columns rejection, and missing
  frame resolution.
* Integration-style pipeline test for `configure_workbook_view` +
  `add_validations` + `from_legend` fan-out over a neutral matrix
  fixture.
* Architecture guard ensuring validation role targeting imports and
  calls the shared resolver and does not define a parallel role
  vocabulary.
