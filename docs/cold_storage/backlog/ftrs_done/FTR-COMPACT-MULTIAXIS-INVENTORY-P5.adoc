===== FTR-COMPACT-MULTIAXIS-INVENTORY-P5 -- Promote compact_multiaxis into the Functional Model

*Status:* Done (Functional Model chapter added; archived 2026-06-24)

*Type:* Functional Model documentation.

*Purpose:*::
Promote the already implemented `compact_multiaxis` composition family into the
Functional Model now that the repository-wide June 2026 inventory exists.

*Context:*::
The original narrow need for this FTR was an inventory of the
`compact_multiaxis` family before modeling. That prerequisite is now satisfied
by the June 2026 transformation/capability family inventory:
`docs/cold_storage/working_context/transformation_family_inventory_2026_06/transformation_family_inventory.adoc`.

The inventory classifies `compact_multiaxis` as implemented with partial
Functional Model coverage. It also confirms the family is a composition capstone
over `xref_crosstable`, `cell_codec`, and legend metadata. The backlog work is
therefore no longer to create a second inventory; it is to perform bounded
Functional Model catch-up.

*Dependencies:*::
* `FTR-CELL-CODEC-FUNCTIONAL-MODEL-P5` -- `cell_codec` should be modeled before
  the composition capstone is promoted.

*Scope:*::
* Add use-case-anchored Functional Model material for compact multi-axis matrix
  editing.
* Explain the composition relationship to XRef crosstable projection, Cell
  Codec, and Legend Blocks.
* State the current implementation caveat that dense-axis support is deferred /
  rejected at the compact-multiaxis layer and belongs to underlying XRef
  semantics unless separately approved.
* Cite the cold-storage inventory as evidence and avoid copying its table or
  full implementation matrix.

*Non-Goals:*::
* No runtime implementation.
* No module split.
* No new compact-multiaxis behavior.
* No dense-axis implementation.
* No Project Memory entities.
* No duplicate family inventory in the backlog or Functional Model.

*Acceptance Criteria:*::
* The Functional Model has a bounded compact-multiaxis explanation anchored to
  an existing use case.
* The chapter or section states the composition relationship to
  `xref_crosstable`, `cell_codec`, and `legend_blocks`.
* Current implementation caveats are stated without turning them into roadmap
  promises.
* The June 2026 inventory is cited as provenance without duplicating inventory
  contents.
* The FTR remains documentation-only; no runtime implementation is performed
  unless separately approved.

*Relationship to the inventory and Functional Model:*::
The inventory remains the canonical June 2026 implemented-family reference.
This FTR uses that reference to close the Functional Model coverage gap for the
composition family; it does not create a second source of truth.
