===== FTR-ROUTER-DECLARATIVE-P3F -- Replace defensive router imports with direct registration

*Status:* Done +
*Epic:* Phase 3f -- I/O Infrastructure Cleanup

*Purpose:*::
`io_backends/router.py` uses 5 `try/except Exception` blocks to import backend functions, falling back to `None` if unavailable. Each backend is then conditionally registered only if the import succeeded. This pattern made sense when backends were optional or externally provided -- but all 5 backends (CSV, XLSX, JSON, YAML, XML) are first-party modules in this package. A failed import is a genuine bug, not an expected condition.

*Current State (verified):*::
----
try:
    from .csv_backend import load_csv_dir, save_csv_dir
except Exception:
    load_csv_dir = None   # 5x this pattern
...
if load_csv_dir and save_csv_dir:
    LOADERS["csv_dir"] = load_csv_dir   # conditional registration
----
This means: if a backend has a typo or import error, it silently degrades to "not available" instead of failing loudly. That is the opposite of fail-fast.

*Scope:*::
. Replace all 5 `try/except` import blocks with direct imports.
. Remove the `None` fallback variables and conditional `if` registration.
. Register all backends unconditionally in `LOADERS` and `SAVERS` dicts.
. Remove the `_require()` null-check helper (no longer needed -- backends are always present).
. Keep `get_loader()` and `get_saver()` as public API, but they raise `ValueError` for unknown kinds (not `SystemExit` for missing backends).

*Non-Goals:*::
- No plugin mechanism or dynamic backend discovery (that would be Phase 6+ if ever needed).
- No changes to backend implementations themselves.

*Acceptance:*::
- No `try/except` import blocks in `router.py`.
- No `None` sentinel values for backend functions.
- Import errors in any backend surface immediately as `ImportError`.
- `get_loader("unknown_kind")` raises `ValueError`.
- All tests pass.

*Completed:*:: 2026-03-31
