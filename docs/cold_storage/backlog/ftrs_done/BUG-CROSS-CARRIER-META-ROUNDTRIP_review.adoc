===== Review: BUG-CROSS-CARRIER-META-ROUNDTRIP-P4A commit d816d9a

*Verdict:* Yellow

The implementation of the narrow persistence-boundary slice is green: it prunes the three requested Resolution facets, preserves Intent and Provenance, avoids registry/schema/read-side expansion, and the validation matrix is green.

The verdict is Yellow because the new in-repo XLSX integration test is weaker than its name and docstring claim, and the bug document still references a non-existent fixture path in its older sections. The real cross-carrier failure class is nevertheless covered by the worldbuilding fixture simulation run during this review.

==== Scope assessment

The production change is appropriately scoped to `src/spreadsheet_handling/pipeline/persistence_boundary.py`.

The added rules are limited to:

* `legend_blocks[*].resolved`
* `xref_crosstable[*].dense_axes.resolved`
* `xref_crosstable[*].column_keys`

No registry classification changes, schema/versioning, read-side sanitizer, FTR creation, generic meta framework, or consumer-local workaround were introduced.

The existing earlier persistence-boundary behavior remains present: top-level `derived`, top-level `__*`, and runtime-produced FK-helper v2 relation pruning.

==== Correctness assessment

`project_meta_to_persistable_contract` remains a pure projection for this slice. It creates new dicts only where pruning is needed and does not mutate the input.

Malformed values are handled safely:

* non-mapping top-level `meta` returns `{}`;
* non-mapping `legend_blocks` is passed through unchanged;
* non-mapping `xref_crosstable` is passed through unchanged;
* non-mapping entries under those roots are passed through unchanged;
* non-mapping `dense_axes` is not inspected.

Intent and Provenance are preserved. The fixture tests cover `workbook_view`, `constraints`, `cell_codecs`, `compact_multiaxis`, `split_by_discriminator`, `frame_lifecycle`, `source: workbook` presentation provenance, legend intent fields, and xref intent fields.

Regression risk is acceptable for the implemented slice. `legend_blocks[*].resolved` is render-composer layout output and should be recomputed. `dense_axes.resolved` and top-level `xref_crosstable[*].column_keys` are consumed by xref code as stored/reconstructed axis snapshots, but the persisted fixture also retains `dense_axes.columns_from` and `dense_axes.rows_from`, so a normal rerun with axis frames available can recompute them. The main risk would be a consumer relying on persisted `column_keys` without axis frames or explicit step config; that is a real backward-compatibility edge but not contradicted by the captured worldbuilding validation.

==== Test adequacy assessment

Unit coverage is strong. `tests/unit/pipeline/test_persistence_boundary.py` covers all three pruning rules independently, fixture-based pruning, positive preservation checks, malformed root values, and non-mutation.

Architecture guards use the captured fixture meaningfully. They assert fixture preconditions before projection, then verify both pruning and preservation.

The new XLSX integration test is weakened. In `tests/integration/io_backends/xlsx/test_workbook_meta_blob_carrier.py`, the fixture is loaded and projected, but the test then replaces `workbook_view` and removes `legend_blocks`, `xref_crosstable`, `constraints`, `cell_codecs`, `compact_multiaxis`, `split_by_discriminator`, `sheets`, and `frame_lifecycle` before export. The later pruning assertions over `legend_blocks` and `xref_crosstable` are therefore vacuous. This test is still a useful XLSX carrier smoke test for a reduced fixture, but it does not by itself exercise the original cross-carrier failure class.

The worldbuilding active-sidecar simulation compensates materially for that weakness in this review run: using the captured fixture as `../worldbuilding/daten/dino_insel/_meta.yaml`, both XLSX and ODS export/reimport passed.

==== Documentation assessment

`docs/ai_info/agent_rules.adoc` accurately describes the implemented Intent-vs-Resolution slice and avoids presenting it as a complete canonical-meta solution.

The appended implemented-slice section in `docs/cold_storage/backlog/ftrs_done/BUG-CROSS-CARRIER-META-ROUNDTRIP.adoc` is accurate and explicit about non-goals.

However, older references in the same bug document still point to `docs/cold_storage/backlog/ftrs_done/fixtures/BUG-CROSS-CARRIER-META-ROUNDTRIP-P4A/ods_produced_meta.yaml`, while the actual fixture path is `docs/cold_storage/backlog/ftrs_done/fixtures/BUG-CROSS-CARRIER-META-ROUNDTRIP/ods_produced_meta.yaml`.

==== Findings by severity

* Medium: The in-repo XLSX integration regression is too synthetic to prove the stated cross-carrier fixture behavior. It removes the fixture roots that contain the three pruned facets before export, making the pruning assertions vacuous. The worldbuilding active-sidecar validation covers the real class during review, but the core regression should either keep enough of the fixture active to make the assertions non-vacuous or explicitly position itself as a reduced carrier smoke test.

* Low: `docs/cold_storage/backlog/ftrs_done/BUG-CROSS-CARRIER-META-ROUNDTRIP.adoc` has stale fixture paths in its earlier sections. Lines 7, 41, and 114 reference the `...-P4A` fixture directory, which does not exist in this repo.

==== Recommendation

Accept the implementation after addressing the documentation path typo. Treat the XLSX integration-test weakness as a follow-up unless a core-only regression can be made to preserve the fixture roots without standing up the full worldbuilding frame set. The worldbuilding fixture simulation is the stronger validation for the original failure class in this review.

  (``docs/cold_storage/backlog/ftrs_done/BUG-CROSS-CARRIER-META-ROUNDTRIP.adoc``) was updated
