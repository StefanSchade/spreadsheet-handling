== Verdict

Ship. The slice solves the silent-miss drift problem the FTR
identifies (PyPI keeps old release pages online indefinitely; a
`/latest/...` link inside a frozen README silently sends an old
release's reader to documentation that no longer matches the
installed version), without committing to the more invasive
"two commits per release" automated workflow the FTR's
*Risks / open questions* explicitly defers.

The chosen path is *manual substitution on a release branch +
read-only CI guard*. Substitution stays a maintainer decision; the
helper only verifies. The CI guard is wired into the publish job
before `python -m build`, so a missed bake fails the workflow
before the wheel is generated and PyPI never sees a release whose
README points at `/latest/...`.

== Chosen strategy

*Manual substitution on a release branch + read-only CI guard.*

The release runbook *Release-bound README links* section is the
maintainer-facing contract:

. Branch off `main` to a short-lived release branch.
. Run a sed substitution (a copy-and-paste one-liner is documented
  in the runbook and is also printed by
  `readme_links_check.sh --release-tag` on a non-clean README) that
  rewrites every `…/spreadsheet-handling-pages/latest/` URL to
  `…/spreadsheet-handling-pages/versions/v<version>/`.
. Run `make readme-check TAG=v<version>` to verify.
. Commit the substitution; merge or fast-forward into `main`; tag
  from the release-bound commit; push the tag.
. The publish workflow's *README link-versioning guard* step runs
  the same check before `python -m build`; a missed bake fails the
  publish before anything is built.
. On `main`, restore `/latest/...` so the living README is
  unaffected.

The helper does only the *check* half. It does not call `sed -i`,
does not write to the file system, does not git-add or git-commit,
and does not require any release-time secret access.

== Why not the alternatives

* *Automated `prepare-release` / `finalize-release` two-commit
  workflow* (the second candidate the FTR's *Scope (design phase)*
  describes). The FTR's *Risks / open questions* explicitly calls
  out the two-commits-per-release vs single-drift-commit trade-off
  as the most contentious unresolved design point. Implementing it
  here would require either committing to that workflow shape
  without a separate design review, or building both shapes and
  letting the maintainer pick at release time -- both larger than
  the FTR scope. The chosen path produces the same release-time
  correctness guarantee (no `/latest/` URL leaves PyPI for a frozen
  release) without committing to either shape; the maintainer can
  always layer automation on top later by wrapping the documented
  sed substitution in a script.

* *Build-time README substitution* (rewriting the README inside
  `python -m build` or a pre-build step that does the substitution
  and re-points `[project].readme`). Rejected on three grounds:
+
- FTR *Non-goals* prohibits "template engine, Jinja-style
  preprocessing, general-purpose documentation pipeline". A
  build-time substitution that produces a generated readme path
  blurs that line and starts conversations about substitution
  scope, generated-file location, and reproducibility.
- The tagged commit on the release branch would no longer match
  what `python -m build` packages as the long description: a
  reader who clones the tag would see one README; PyPI would
  display another. Surprising and hard to debug.
- It requires `[tool.setuptools]` plumbing (a `dynamic` declaration
  for `readme`, or a `setup.py`-level shim) that changes how
  routine dev workflows interact with packaging.

* *`pyproject.toml [project].readme` redirected to a `build/`
  artifact.* Same drawbacks as build-time substitution, plus it
  splits the maintainer's mental model between "README on `main`"
  and "README packaged from `build/`". The manual-substitution path
  keeps one README per branch and lets git history record which
  shape each commit carries.

* *No tooling; just a manual checklist*. This is the FTR's
  *Interim manual checklist*. It is the right shape for the policy
  documentation but leaves the "operator forgot to bake" failure
  mode silent: a maintainer who skips the sed step still passes
  every existing `release-check` invariant and ships to PyPI. The
  read-only CI guard added in this slice closes that gap with one
  step in the publish job and no risk to the rest of the workflow.

== Residual risks

* **Maintainer skips the bake AND CI somehow does not fire.** The
  CI guard depends on the publish job being part of every release.
  If a maintainer publishes to PyPI from a local `python -m build`
  + `twine upload` (rather than via the tag-triggered workflow),
  the guard does not run. Mitigation: the runbook prescribes the
  tag-push flow and does not describe a manual upload path. If a
  future emergency requires a hand-publish, the maintainer is
  expected to run `make readme-check TAG=vX.Y.Z` first; the runbook
  *Core Release Checklist* bullet makes that explicit.

* **`main` carries an unbaked README between releases by design.**
  That is the policy: the README on `main` is a *living* document
  with `/latest/` links. A reader cloning the repo and reading
  `README.md` on `main` between releases will see `/latest/` URLs
  and follow them to the moving documentation. This is correct for
  that audience -- they are on the moving tip of the project. The
  policy specifically allows `/latest/` for living contexts; only
  frozen contexts (PyPI long descriptions, GitHub release pages)
  must use `/versions/<tag>/`.

* **Maintainer forgets the restore-on-`main` step after tagging.**
  If the release branch is merged back to `main` carrying the
  baked URLs, `main` itself becomes a "frozen" surface pointing at
  `versions/<vX.Y.Z>/...`. Mitigation: the runbook
  *Manual procedure* lists the restore as step 7. The check
  helper's lint mode (no `--release-tag`) provides a non-blocking
  audit: a routine `make readme-check` from `main` after a release
  prints zero `/latest/` URLs if the restore was skipped, which is
  a visible signal during normal development. A stricter lint
  guard (CI on `main` push that requires `/latest/` URLs present)
  is intentionally not added in this slice; that is a different
  policy and belongs in a follow-up if drift in this direction ever
  shows up in practice.

* **Demo-repo drift not solved by this slice.** Per FTR scope and
  per the user prompt's "Do not touch demo or pages repos"
  instruction, the demo `README.md` is not baked here. Demo tags
  (`v<core-tag>-demo`) currently surface a `/latest/` README on
  their GitHub release pages. The runbook *Demo repository*
  subsection describes the one-line adoption path (copy the helper,
  wire a Make target, add the checklist bullet) so the demo
  maintainer can land a parallel slice when convenient.

* **Substitution allowlist is a single literal prefix.** If the
  Pages hostname or path layout changes (e.g. a custom domain),
  the helper's `LATEST_PREFIX` constant and the runbook example
  sed both need a coordinated update. Same constraint that
  `pages_publish_check.sh` already has; documented inline in both
  files.

* **PyPI long-description rendering trust.** PyPI renders Markdown
  with a constrained subset of features. The slice does not change
  the Markdown shape of `README.md`, only the URL inside existing
  links, so the rendered output is identical except for the href
  target. No risk to the existing rendering.
