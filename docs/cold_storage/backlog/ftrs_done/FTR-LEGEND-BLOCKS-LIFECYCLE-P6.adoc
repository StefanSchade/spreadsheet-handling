==== Context

The persistence boundary
(`src/spreadsheet_handling/pipeline/persistence_boundary.py::_project_legend_blocks`)
strips one concrete runtime facet from each legend block:
`legend_blocks[*].resolved`. That projection is narrow; it does not
classify the full legend-block surface and does not state which other
facets are durable intent, runtime resolution, or carrier-side residue.

The meta registry currently classifies `legend_blocks` as
`canonical_meta` with `persistence_behavior` "persisted in the hidden
workbook metadata payload, including resolved legend placement after
compose". That description matches the workbook-embedded carrier but does
not mention that `resolved` is stripped at the structured-data (sidecar)
boundary. Runtime and registry describe two different persistence targets
without naming the distinction.

==== Why this FTR

`legend_blocks` mixes several lifecycle classes in one structure:

* durable user intent -- `entries` (token / group / label), `placement`,
  `title`, `name` / `id`, `target`;
* runtime resolution state -- `resolved` (sheet, frame_name, top, left,
  n_rows, n_cols), produced by the composer, read back by the parsers to
  re-anchor legend tables, stripped at the sidecar;
* carrier-side read-path residue -- `__legend_blocks` on `sheet.meta`,
  whose only registered consumer is "tests and diagnostics".

Without an explicit classification, a future producer cannot tell which
facet may survive which boundary, and the registry/runtime/doc
disagreement stays latent.

==== Scope

* Inventory every facet under `legend_blocks[*]` and the related
  `__legend_blocks` carrier hint: producer, consumer, persistence
  behavior at both the sidecar boundary and the workbook-embedded
  carrier, tests, runtime dependency.
* Classify each facet: active feature contract, runtime helper, carrier
  detail, or legacy residue (same four-way scheme as the FK-helper slice).
* Map each facet onto the lifecycle attribute model from the functional
  model (`drop_at_persistence`, `drop_if_stale`, `carrier_takes_precedence`,
  `required`, `render_consumed`).
* Reconcile the registry `persistence_behavior` wording with the actual
  sidecar-vs-carrier behavior.
* Derive the functional-model concept (warm storage) and promote it to
  `docs/functional_model/ch04_use_case_concepts/legend_blocks.adoc`,
  reusing the framework section in
  `docs/functional_model/ch05_problem_space_overview/03_meta_lifecycle_framework.adoc`.
* Confirm or retire the `__legend_blocks` residue suspicion with evidence.

==== Non-Goals

* Do not redesign the whole `_meta` model -- that remains
  `FTR-META-LIFECYCLE-INVENTORY-P5`.
* Do not change legend rendering behavior or the visible legend table
  contract.
* Do not bind the runtime to the registry; the registry stays a manual
  review artifact.

==== Acceptance

* Every `legend_blocks` facet and the `__legend_blocks` hint carries a
  documented classification, consistent across runtime, registry, and the
  functional model, or its disagreement is named with a chosen resolution.
* The registry `persistence_behavior` distinguishes the sidecar boundary
  from the workbook-embedded carrier.
* The `__legend_blocks` residue suspicion is either confirmed (with a
  removal slice) or refuted (with the consumer that makes it load-bearing).
* The functional-model legend-block chapter follows the step-slice
  convention and cross-references the ch05 framework rather than restating
  it.

==== Completion

*Status: Done.* Slices A-C complete; Slice D deferred.

* *Slice A (registry wording).* `legend_blocks.persistence_behavior` now
  distinguishes the workbook-embedded carrier (`resolved` present, used by
  parsers) from the structured sidecar (`resolved` stripped, intent facets
  kept).
* *Slice B (characterization tests).* Locked current behaviour before
  cleanup: sidecar projection drops `resolved` and keeps intent facets;
  `resolved` is carrier-durable; rendered legend stays `kind="legend"` and is
  not emitted as a payload frame; the second roundtrip rebuilds `resolved`
  from the config-backed source. ODS parity added in the final follow-up.
* *Slice C (residue removal).* Removed the two
  `sheet.meta["__legend_blocks"] = list(hints)` assignments (XLSX and ODS
  parsers) and the obsolete `__legend_blocks` registry entry, with both
  registry-invariant tests updated. The load-bearing artifact is
  `table.kind = "legend"` (set during parser classification), which is
  unchanged; `__legend_blocks` was confirmed residue (no runtime reader,
  never emitted on the write path).
* *Slice D (attribute-driven persistence boundary).* Deferred as
  cross-cutting future work. `_project_legend_blocks` remains field-specific;
  replacing it with registry-attribute-driven projection spans multiple
  transformation families and is out of scope for this FTR.

*Release note:* none. `__legend_blocks` had no runtime reader and was never
emitted as a user-facing / write-path artifact; removing it changes no
user-facing behavior. (The sidecar `resolved` stripping it relied on was
already documented under a prior bug FTR.)

*Technical-model reconciliation:* the meta-layer / carrier / glossary sites
that listed `__legend_blocks` as a live operational-carrier entry were updated
so the technical model no longer advertises it as current registry metadata.
