===== FTR-PAGES-AND-RELEASE-NOTES-P5 -- Versioned Pages and release notes trail

*Status:* Done
*Priority:* P5
*Area:* Documentation / release management

==== Purpose

Define the lightweight publishing path for versioned documentation, demo
walkthroughs, and release notes before adding more public documentation
content.

This FTR creates the repository and directory structure for the public Pages
artifact trail. Later content FTRs, including the transformation catalog, build
on this structure instead of deciding publication placement again.

==== Scope

* Create or define the dedicated `spreadsheet-handling-pages` repository as
  the static artifact host.
* Define the split between source-owning repositories and the generated Pages
  artifact repository.
* Define how release notes are authored in source form and published.
* Define versioned public paths for core docs, demo walkthroughs, and release
  notes.
* Define the minimum index structure that lets the demo README link users to
  the matching published documentation.
* Keep generated HTML and Reveal.js output as build artifacts, not committed
  source in core or demo.
* Connect this with the conservative release/promotion policy rather than
  duplicating release automation decisions.

==== Delivery Slices

Slice 1, implemented in the core repository, covers normal AsciiDoc HTML
documentation built from `spreadsheet-handling` and published after successful
release tags:

* core documentation landing page,
* user guide,
* minimal release-notes entry point,
* GitHub Actions publication to `/versions/<tag>/core/` in
  `spreadsheet-handling-pages`,
* guarded GitHub Actions publication to `/latest/core/` only when the
  triggering tag is the newest visible core release tag.

Reveal.js / `asciidoctor-revealjs` and presentation publishing from
`spreadsheet-handling-demo` are deferred to a later demo/presentation
publishing slice.

==== Site Roles

Core-built documentation source in `spreadsheet-handling`:

* release notes and changelog trail
* user guide entry points
* transformation catalog
* API and YAML surface notes when stable enough
* links to matching demo walkthrough versions

Demo-built documentation source in `spreadsheet-handling-demo`:

* journey index for new users
* generated Reveal.js walkthrough decks
* downloadable or linked demo inputs/outputs where useful
* capability matrix for the demo repository
* links back to the core version the demo was built against

Published output in `spreadsheet-handling-pages`:

* static HTML generated from core documentation sources
* static HTML / Reveal.js output generated from demo documentation sources
* release notes and index pages that expose validated core/demo pairings
* stable versioned paths plus an optional moving `latest` pointer

The demo repository should own its own `DEMO-*` backlog item for the concrete
demo-side build workflow. This core FTR owns the cross-repository policy, Pages
repository shape, and release trail.

==== Preferred Direction

Use a dedicated GitHub Pages artifact repository:

* `spreadsheet-handling` builds core documentation artifacts from release
  source.
* `spreadsheet-handling-demo` builds demo/integration walkthrough artifacts
  from the matching demo tag.
* `spreadsheet-handling-pages` stores the generated static output that GitHub
  Pages serves.

Prefer stable public paths such as:

* `/latest/`
* `/latest/core/` for latest core documentation
* `/latest/demo/` for latest demo documentation
* `/latest/presentations/` for future presentation output
* `/versions/v0.1.0b6/core/` for core documentation
* `/versions/v0.1.0b6/demo/` for demo walkthroughs
* `/versions/v0.1.0b6/presentations/` for future presentation output
* `/versions/v0.1.0b6/index.html` linking the matching core, demo, and
  presentation docs

Generated artifact history belongs to the Pages repository, not to normal
source branches in core or demo.

==== Versioning Policy

Every published beta or release should have a stable documentation snapshot.
`latest` can move, but versioned paths should remain stable.

For coupled beta/demo releases, the release notes should say which demo tag was
validated against which core tag. The pages index should make that pairing
visible so a user can follow a walkthrough that matches the installed library
version.

Generated HTML lives in the Pages artifact repository, not on the normal source
branches of core or demo. Source branches keep AsciiDoc, slide sources, release
note sources, and build configuration only.

The shared release roots (`/latest/` and `/versions/<tag>/`) represent a
validated combined state of core, demo, and future presentation artifacts.
Publishers must replace only their owned subdirectory inside those roots, such
as `core/`, `demo/`, or `presentations/`. They must not delete or rsync-delete
the shared root itself.

==== Non-goals

* No requirement to publish the full Pages setup before the next beta.
* No generated HTML on normal source branches.
* No large documentation site framework before the AsciiDoc and Reveal.js
  source layout has proven itself.
* No requirement to complete the transformation catalog content in this FTR.

==== Acceptance

* The repository policy for generated Pages artifacts is documented and points
  to `spreadsheet-handling-pages` as the artifact host.
* The initial Pages repository/directory structure is defined or created.
* Release notes have a source-of-truth location and a repeatable generation or
  publishing path.
* Demo walkthrough publishing supports versioned snapshots, not only a single
  overwritten latest state.
* Core-built and demo-built docs have distinct source ownership and link to
  each other by version in the Pages output.
* Reveal.js walkthroughs are treated as generated tutorial artifacts, not as
  the source of truth for reference documentation.
* The next beta can link to either source docs or published Pages output without
  ambiguity.

==== Delivery Scope (Slice 1 -- closed)

Implemented in `spreadsheet-handling`:

* `docs/pages/index.adoc` core landing page, `docs/user_guide/user_guide.adoc`,
  and `docs/release_notes/release_notes.adoc` rendered by
  `make docs-pages` via `scripts/build_docs_pages.sh`.
* `docs/pages/root_index.adoc` as the source for the stable Pages root
  navigation page.
* GitHub Actions job `publish-docs` sequenced after PyPI `publish`, gated on
  `v*` tags, with job-level `concurrency: { group: pages-publish }`.
* Publication into `StefanSchade/spreadsheet-handling-pages` on branch `main`,
  served from `/ (root)`, writing only:
** `versions/<tag>/core/`
** `latest/core/` (guarded by a PEP 440 `packaging.version.Version`
   comparator instead of Git version sort, so betas like `v0.1.0b6` and
   future GA tags like `v0.1.0` compare correctly)
** `index.html` (stable root navigation page, rebuilt every release)
* SSH host keys for `github.com` pinned inline with
  `StrictHostKeyChecking=yes` instead of trust-on-first-use `ssh-keyscan`.
* `set -euo pipefail` on every workflow shell block; bounded push retry with
  `git pull --rebase` to survive a concurrent edit on the Pages repo.
* Developer Guide chapter 9 documents the why behind each of the above plus
  the local toolchain stack required for `make docs-pages`.

Deferred to later slices (intentionally not part of this FTR):

* Demo / integration walkthrough publication to `versions/<tag>/demo/` and
  `latest/demo/` (owned by `spreadsheet-handling-demo`).
* Reveal.js / `asciidoctor-revealjs` presentation publication to
  `versions/<tag>/presentations/` and `latest/presentations/`.
* Per-version umbrella index at `versions/<tag>/index.html` linking core,
  demo, and presentation outputs together (cross-publisher coordination).
