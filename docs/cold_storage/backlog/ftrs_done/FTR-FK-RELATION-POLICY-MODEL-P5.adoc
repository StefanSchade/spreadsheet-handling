===== FTR-FK-RELATION-POLICY-MODEL-P5 -- Explicit FK relation policy model

*Status:* Done

==== Delivery scope

Completed as a documentation / model-contract slice. The landed change defines
the planned v2 FK relation policy under the existing
`_meta.helper_policies.fk` root, records `schema_version: 2` as the shape
marker, documents mandatory `target_key`, relation identity by
`(source_frame, source_column)`, producer provenance, helper-column identity,
and the separation between wire-format FK column naming, configuration /
inference, primitive execution, and derived helper provenance.

The slice deliberately does not change runtime code, tests, demos,
`meta_registry.yaml`, or `pipeline_step_registry.json`. Those registries remain
current-state v1 artifacts until the later implementation FTRs update runtime
behavior and the corresponding registry contracts.

Review artifact: `FTR-FK-RELATION-POLICY-MODEL-P5_review.adoc`. Final review
verdict: READY WITH NITS; the nits were addressed before closure.

*Purpose:*::
Define an explicit metadata model for foreign-key relations used by FK helper
steps, so primitive FK-helper transformations no longer need to infer relation
semantics from column names or table names.

*Context:*::
The current FK-helper path still carries an older convention-based model:
primary keys are commonly inferred from configured/default ID column names, and
foreign keys are commonly detected from naming patterns such as an ID prefix or
target-table suffix.
+
That was acceptable during early bootstrap, but it conflicts with the current
architecture direction:
+
* heuristics may propose configuration;
* primitive transformations should execute resolved configuration;
* hidden naming rules must not become implicit domain semantics.
+
The project already distinguishes configuration steps from primitive steps in
the pipeline registry model. This FTR defines the FK relation policy shape that
future FK configuration and primitive steps will use.

*Relationship to existing `helper_policies.fk`:*::
+
`_meta.helper_policies.fk` is already a real, populated metadata root today.
The existing `configure_fk_helpers` step
(`domain.helper_policies.configure_fk_helpers`) writes a per-target shape under
that root, roughly:
+
[source,yaml]
----
helper_policies:
  fk:
    <target_key>:
      target: <target_key>
      target_sheet: <sheet_name>
      key: <id_field>
      allowed_helpers: [...]
      default_helpers: [...]
      helper_prefix: "_"
      fk_column: "id_(<target_key>)"
      label: <optional>
----
+
This FTR replaces that per-target shape with a v2, relation-oriented shape
under the same root (`_meta.helper_policies.fk`). The project keeps a single
FK-helper policy root; no adjacent root is introduced. The v2 shape is keyed
by `(source_frame, source_column)` because *relation identity is per FK
column on a specific source frame*, not per target alone -- two sheets may
reference the same target with different helper choices.
+
The v2 shape carries an explicit `schema_version: 2` marker at the
`helper_policies.fk` level so consumers can detect the shape unambiguously and
so the lifecycle FTR
(`FTR-DERIVED-PROVENANCE-LIFECYCLE-P4A`) and the meta-schema-validation FTR
(`FTR-META-SCHEMA-VALIDATION-P5`) can validate it directly.

*Dependencies:*::

* Existing FK-helper behavior and tests.
* Existing `_meta.helper_policies` metadata root.
* Existing pipeline step categories: `configuration` and `primitive`.
* Existing derived-helper provenance channel under `_meta.derived.sheets`.

*Scope (this FTR is documentation / model only):*::

* Define the v2 relation-oriented FK policy shape stored under the existing
  root `_meta.helper_policies.fk`. The definition lives in this FTR and in
  the technical concept doc
  (`docs/technical_model/ch04_concepts/fk_helper_metadata.adoc`).
* Represent at least:
  ** source frame,
  ** source FK column,
  ** target frame,
  ** target key column (the PK column on the target sheet, `target_key`,
     mandatory),
  ** target display / helper fields,
  ** materialized helper column names or naming policy,
  ** producer metadata sufficient to distinguish explicit / manual
     configuration from later inferred configuration,
  ** provenance fields needed for reimport cleanup and later helper-value
     consistency checks (helper-column identity that survives a reimport
     without re-running inference).
* Carry a `schema_version: 2` marker at the `helper_policies.fk` level so
  consumers can detect the shape unambiguously.
* Replace the existing per-target shape *conceptually* under the same root;
  do not introduce an adjacent FK-helper policy root (no
  `_meta.fk_relations`, no `helper_policies.fk_relations` parallel to the
  current shape).
* Update the agent-facing FK-helper notes
  (`docs/ai_info/concepts_fk_helpers.adoc`) to add a *Target model (v2)
  (planned)* subsection pointing to the technical concept doc and to this
  FTR, without altering the description of current runtime behavior.
* Define the failure principle that later primitive FK-helper execution must
  honor: missing FK relation policy is an error unless a configuration /
  inference step wrote one earlier in the pipeline. Encoding this principle
  in primitive runtime is owned by
  `FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5`.

*Explicitly out of scope for this FTR (handed off to later FTRs):*::

* Modifying `meta_registry.yaml`. The meta registry is a current-state
  manual review artifact (`maintenance_mode: manual_review_artifact`,
  `runtime_binding: forbidden`). Today its `helper_policies` entry
  truthfully describes the v1 per-target shape that
  `configure_fk_helpers` writes and that
  `domain.transformations.fk_helpers.enrich_helpers` reads. Rewriting it to
  reference the v2 relation shape, `infer_fk_relations`, or v2-aware
  primitive consumers while the runtime still writes / reads v1 would make
  the registry misrepresent runtime. The registry entry for
  `helper_policies` is therefore updated together with each runtime change
  that earns it:
  ** producer entry for `configure_fk_helpers` (manual / explicit v2 path)
     and a new entry for `infer_fk_relations` are added by
     `FTR-INFER-FK-RELATIONS-CONFIGURATION-STEP-P5`;
  ** consumer entries for `add_fk_helpers`, `validate_fk_helpers`,
     `remove_fk_helpers`, `reorder_fk_helpers` reading
     `helper_policies.fk` (v2) are added by
     `FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5`.
* Modifying `pipeline_step_registry.json`. Same rationale: the registry is
  a current-state manual review artifact. The description fields for
  `configure_fk_helpers` and for the FK-helper primitives today truthfully
  describe v1 runtime behavior (per-target `helper_policies.fk.*` writes
  and "derived provenance with prefix fallback" cleanup). They are updated
  by the same later FTRs that change the runtime behavior they describe.
* Implementing `infer_fk_relations`. Owned by
  `FTR-INFER-FK-RELATIONS-CONFIGURATION-STEP-P5`.
* Refactoring `add_fk_helpers`, `validate_fk_helpers`, `remove_fk_helpers`,
  `reorder_fk_helpers` to consume the v2 shape. Owned by
  `FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5`.
* Updating the public user-guide transformation catalog to present v2 as
  available runtime behavior. Owned by
  `FTR-FK-HELPER-DOCS-DEMO-REALIGNMENT-P5`, after the primitives FTR has
  landed.
* Removing `FK_PATTERN` or the `id_(<target>)` cell-level wire format
  convention.

*Non-Goals:*::

* No implementation of `infer_fk_relations`.
* No rewrite of `add_fk_helpers`, `remove_fk_helpers`, `validate_fk_helpers`, or
  `reorder_fk_helpers`.
* No demo pipeline migration.
* No backward-compatibility layer for public users (the project is pre-beta
  for this surface; see the public-surface gate below).
* No runtime schema validation framework.
* No generic relation model for all possible joins; this FTR only covers FK
  helper semantics.
* No removal of the FK column convention `id_(<target>)`
  (`core/fk.py:FK_PATTERN`). The convention remains the cell-level wire
  format; this FTR only changes the metadata shape consumed by
  configuration / primitive steps.

*Public-surface gate:*::

The v2 shape replaces a documented YAML / `_meta` surface (the per-target
`helper_policies.fk` shape currently produced by `configure_fk_helpers` and
referenced in the user-guide transformation catalog). Per
`docs/ai_info/interfaces_and_gates.adoc`, structural changes to the
`helper_policies.fk` shape and to the documented `configure_fk_helpers`
parameter contract are gated surface changes. The developer must approve the
v2 shape (and the related primitive-refactor approval) before
`FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5` is implemented.

*Policy sketch (v2, illustrative):*::

The example uses the project's actual FK column convention
`id_(<target>)` (see `core/fk.py:FK_PATTERN` and the
`product_business_slice` demo data) and project-style example frames
(`Products`, `ProductManagers`).

[source,yaml]
----
_meta:
  helper_policies:
    fk:
      schema_version: 2
      relations:
        - source_frame: Products
          source_column: id_(product_manager)
          target_frame: ProductManagers
          target_key: id
          helper_fields:
            - name
          helper_columns:
            - column: _product_manager_name
              target_field: name
          helper_prefix: "_"
          produced_by:
            step: infer_fk_relations
            mode: naming_convention
----

A second, *explicit* relation produced by `configure_fk_helpers` looks the
same except for the `produced_by` block, which records that the relation
came from manual configuration rather than inference:

[source,yaml]
----
        - source_frame: Products
          source_column: id_(product_manager)
          target_frame: ProductManagers
          target_key: id
          helper_fields: [name, region]
          helper_columns:
            - column: _product_manager_name
              target_field: name
            - column: _product_manager_region
              target_field: region
          helper_prefix: "_"
          produced_by:
            step: configure_fk_helpers
            mode: explicit
----

*Relation entry: field schema (v2):*::

Each entry in `_meta.helper_policies.fk.relations` is keyed by
`(source_frame, source_column)`; that pair is the relation identity. Two
sheets may reference the same target frame with different helper choices,
which is why relation identity is per FK column on a specific source
frame rather than per target.

[cols="1,1,1,4"]
|===
| Field | Required | Type | Meaning

| `source_frame`
| yes
| string
| Name of the frame that carries the FK column.

| `source_column`
| yes
| string
| Header of the FK column on `source_frame`. Project convention is
  `id_(<target>)`, but the policy stores the literal column header rather
  than re-deriving it from a pattern.

| `target_frame`
| yes
| string
| Name of the frame the FK resolves into.

| `target_key`
| yes
| string
| PK column on `target_frame` against which `source_column` joins.
  Mandatory so that primitives, validators, and later reimport cleanup
  never need to re-derive it from `build_registry(...)["id_field"]`
  defaults.

| `helper_fields`
| yes (may be empty list)
| list[string]
| Ordered list of columns on `target_frame` whose values are materialized
  next to `source_column` on `source_frame` as helper columns.

| `helper_columns`
| yes
| list[object]
| Explicit `{column, target_field}` pairs, one per materialized helper
  column. Names the on-sheet helper-column header that `add_fk_helpers`
  will write and the target field it reads. Used by primitive execution,
  validation, and reimport cleanup.

| `helper_prefix`
| no
| string
| Optional helper-column naming prefix used by the producer when it
  generated the `helper_columns` entries. Recorded for diagnostics and
  for future naming-policy negotiation; never re-applied as a runtime
  detection fallback. Defaults to the workbook-level `helper_prefix`
  meta key when absent.

| `produced_by.step`
| yes
| string
| Name of the configuration step that wrote the entry. Initially one of
  `configure_fk_helpers` (manual / explicit) or `infer_fk_relations`
  (heuristic).

| `produced_by.mode`
| yes
| string
| Mode within the producing step. For `configure_fk_helpers` the
  conventional value is `explicit`. For `infer_fk_relations` the initial
  value is `naming_convention`; later inference modes register new
  values.
|===

`schema_version: 2` lives at the `helper_policies.fk` level, not on each
relation. The marker exists so that `FTR-DERIVED-PROVENANCE-LIFECYCLE-P4A`
and `FTR-META-SCHEMA-VALIDATION-P5` can validate the shape directly,
without sniffing individual entries.

The exact field names may still be adjusted during implementation, but the
final policy must be explicit enough that:

* helper materialization does not need to derive relation identity from
  visible column names;
* validation can look up target values from `(target_frame, target_key)`
  alone;
* reimport cleanup can identify materialized helper columns from
  `helper_columns` plus `_meta.derived.sheets.*.helper_columns` without
  consulting a naming convention.

*Authority and consumption model:*::

The v2 policy is *canonical configuration* under
`_meta.helper_policies.fk`. It is not derived helper provenance.
Four concerns must stay distinguishable when reading or writing this
metadata:

. *Configuration / inference produces resolved relation policy.* Two
  configuration steps are the only producers of
  `_meta.helper_policies.fk.relations`:
+
** `configure_fk_helpers` writes entries from user-supplied manual
   configuration (`produced_by.mode: explicit`);
** `infer_fk_relations` writes entries derived from bounded heuristics
   over current frames (`produced_by.mode: naming_convention`, later
   possibly additional modes).
+
Both write the same v2 shape under the same root. Both behave as
*configuration* steps in the pipeline registry sense. A pipeline that
runs both must fail clearly when both producers try to write the same
`(source_frame, source_column)` relation key.

. *Primitive FK-helper execution consumes resolved relation policy.* The
  primitive FK-helper steps (`add_fk_helpers`, `validate_fk_helpers`,
  `remove_fk_helpers`, `reorder_fk_helpers`) read
  `_meta.helper_policies.fk.relations` and the derived helper-column
  provenance under `_meta.derived.sheets.*.helper_columns`. They do not
  themselves run inference. Missing relation policy is a clear error
  whose message names the configuration / inference step that the
  pipeline should have run earlier. This contract is encoded in primitive
  runtime by `FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5`; this FTR only
  states it as the model contract.

. *Derived helper provenance is materialized helper-column state.* The
  per-pipeline-run record of which sheet columns are helper columns
  remains under `_meta.derived.sheets.*.helper_columns` (see the
  `derived` entry in `meta_registry.yaml`). It is not the policy; it is
  what the primitive wrote when it last enriched the source frame.
  Producer-shape changes to that channel (notably adding `target_key`
  to each entry so reimport cleanup and value-consistency checks do not
  re-derive it) are owned by
  `FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5`. Lifecycle policy for
  the channel (pruning, stale / orphan handling, schema validation) is
  owned by `FTR-DERIVED-PROVENANCE-LIFECYCLE-P4A`.

. *Wire-format FK column naming is independent of inference and
  execution.* The cell-level convention `id_(<target>)`
  (`core/fk.py:FK_PATTERN`) remains the serialized cross-sheet
  reference shape on XLSX / ODS roundtrips. It is consumed by the
  parser / writer and by `infer_fk_relations` (as one heuristic mode
  among future modes). It is *not* consumed by primitive FK-helper
  execution for relation inference. Removing primitive inference is not
  the same as removing the wire format; this FTR removes neither.

*Acceptance:*::

* The v2 FK relation policy shape is documented in the technical concept
  doc (`docs/technical_model/ch04_concepts/fk_helper_metadata.adoc`),
  including the `schema_version: 2` marker, the mandatory `target_key`
  field, and the explicit `helper_columns` / `helper_fields` /
  `produced_by` fields.
* The metadata authority is clear: the policy is a canonical configuration
  claim under `_meta.helper_policies.fk`, not derived helper provenance.
* The v2 shape replaces the existing per-target shape *conceptually* under
  the same root; no adjacent FK-helper policy root is introduced. The
  decision is recorded in this FTR's *Relationship to existing
  `helper_policies.fk`* subsection.
* The relation policy contains enough information for:
  ** helper materialization (`helper_fields`, `helper_columns`,
     `helper_prefix`),
  ** helper validation including target-key consistency checks
     (`target_key`, `helper_columns[].target_field`),
  ** helper reordering (`source_column`, `helper_columns[].column`),
  ** producer-side derived provenance writing (the derived-channel
     producer-shape change itself is owned by
     `FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5`),
  ** later reimport cleanup (`helper_columns[].column` paired with the
     derived `helper_columns` provenance).
* The policy distinguishes the relation key (`source_frame`, `source_column`)
  and target resolution (`target_frame`, `target_key`) from materialized
  helper columns (`helper_columns`).
* The policy does not rely on visible workbook labels as identity.
* Producer provenance is part of the shape: every relation entry carries
  `produced_by.step` and `produced_by.mode`, so consumers can distinguish
  manual / explicit configuration (`configure_fk_helpers`,
  `mode: explicit`) from inferred configuration (`infer_fk_relations`,
  `mode: naming_convention` and later modes).
* Existing convention-based FK detection inside primitives is documented as
  current-state behavior to be removed by follow-up FTRs; the convention
  itself (`id_(<target>)`) stays as the cell-level wire format.
* No primitive FK-helper step, configuration step, or `_meta` shape is
  changed in runtime by this FTR.
* The agent-facing notes (`docs/ai_info/concepts_fk_helpers.adoc`) carry a
  *Target model (v2) (planned)* subsection pointing to the technical
  concept doc and this FTR, and the current-state rules in that file are
  *not* rewritten to imply v2 is already active.
* `meta_registry.yaml` and `pipeline_step_registry.json` are deliberately
  *not* modified by this FTR. The reason is recorded above under *Scope
  ... out of scope for this FTR*: those artifacts describe current
  runtime behavior, and the v2 shape is a target model; they will be
  updated by `FTR-INFER-FK-RELATIONS-CONFIGURATION-STEP-P5` and
  `FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5` together with the runtime
  changes that earn the updates.
