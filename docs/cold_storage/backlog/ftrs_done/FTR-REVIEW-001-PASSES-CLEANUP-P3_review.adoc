== Pass ordering preserved?

Yes. The old `apply_all` list and the old `flow.py::default_p1_passes()` list were
both `MetaPass → ValidationPass → StylePass → ProtectionPass → FilterPass →
FreezePass → ColumnWidthPass → NamedRangePass`. The new `default_passes()` returns
the same eight instances in the same order. The architecture_primer and adr-006
diagrams previously showed only six of the eight passes (ProtectionPass and
ColumnWidthPass were omitted); both docs are now corrected.

== Pass implementations behaviorally identical?

Yes, verified by line-by-line comparison against the deleted `core.py` for all
eight pass classes. The `apply()` method bodies are character-for-character
identical. Two small decorator additions that are not behavioral:

* `StylePass` and `FilterPass` gained `@dataclass` in the new modules (old `core.py`
  omitted it; both still had annotated class attributes with defaults). Result:
  `__init__`, `__repr__`, `__eq__` are now generated, but `apply()` is unchanged.
  The production path always calls `StylePass()` with no arguments.

== Is `default_passes()` the single canonical source?

Yes. Both call sites now use it:

* `passes/__init__.py::apply_all` — calls `default_passes()` in the loop.
* `rendering/flow.py::default_p1_passes` — returns `default_passes()`.

The old two-list duplication is eliminated. The ordering test
(`test_default_passes_order` + `test_apply_all_order_equals_default_passes`)
locks both against drift. The monkeypatch approach in `test_apply_all_order_equals_default_passes`
is strong: it patches `passes_module.default_passes` and verifies `apply_all`
respects whatever the function returns, making the link explicit.

=== Minor

*1. `IRPass = IRProtocol` indirection is needlessly roundabout.*

`_base.py` introduces an intermediate Protocol name `IRProtocol` and then aliases
it as `IRPass`. The alias is what `passes/__init__.py` re-exports. The old `core.py`
defined `class IRPass(Protocol)` directly and there is no reason to differ here.

Recommended fix — in `_base.py`, rename `IRProtocol` → `IRPass` and remove the alias:

[source,python]
----
class IRPass(Protocol):
    def apply(self, doc: WorkbookIR) -> WorkbookIR: ...
----

Then remove `IRPass = IRProtocol`. The exported name and behavior are unchanged;
the indirection disappears.

*2. Three stub-era test names are stale after the class migration.*

`test_meta_pass_keeps_hidden_meta_sheet`, `test_style_pass_is_noop_for_now`, and
`test_validation_pass_is_noop_for_now` were written against the old module-level
stub `apply()` functions. They now call the real class `apply()` method, which is
correct. However:

* `test_style_pass_is_noop_for_now` — asserts `ir.sheets["S"].styles == []` after
  `StylePass().apply(ir)`. The assertion passes because `StylePass` modifies
  `sh.meta["__style"]` (not `sh.styles`), so the struct field happens to remain
  empty. The test neither exercises real `StylePass` behavior nor documents any
  meaningful invariant. The name "is_noop_for_now" is now false — the real pass is
  not a no-op.

* `test_validation_pass_is_noop_for_now` — asserts `ir.sheets["S"].validations == []`
  after `ValidationPass().apply(ir)`. Passes because the empty sheet has no
  `_p1_validations` or `constraints`. Again, the name implies stub behavior.

* `test_meta_pass_keeps_hidden_meta_sheet` — this one is fine: it meaningfully tests
  that an existing `_meta` sheet is not discarded.

The stale tests are not dangerous (real behavior is covered by `test_style_themes.py`,
`test_validation_pass_infra.py`, and the integration suite), but they are misleading
noise. Recommended fix: rename or replace the two stale tests with minimal assertions
on real class behavior (e.g., that `StylePass` sets `__style` in `sh.meta`, that
`ValidationPass` appends a `DataValidationSpec` for a sheet with a `_p1_validations`
entry).
