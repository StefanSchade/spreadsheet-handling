the Project Semantic Model, overlapping semantic sets, lifecycle/update

preserving the draft/scaffold posture. It does not change runtime code, tests,

No broad rewrite is needed; the follow-up should be a small semantic-model
wording fix that realigns those examples with current registry truth.

=== Minor

==== M-01: FK-helper derived provenance is described as persistent and roundtrip-surviving

`docs/semantic_model/semantic_model.adoc` lines 612-616 and 675-678 classify
`_meta.derived.sheets.*.helper_columns` as *Persistent* and
*RoundtripRelevant*, including the statement that it "survives roundtrip via
the sidecar".

Current registry truth is different. `registries/meta_registry.yaml` lines
790-817 classify the `derived.sheets` namespace as `lifecycle: transient` and
say it is a "transient IR/workbook-meta working payload", "never persisted to
workbook output as canonical metadata", and "not authoritative" for roundtrip.

Risk: the lifecycle chapter's first overlapping-set example teaches the wrong
persistence posture for a central FK-helper metadata path. That can cause the
semantic model to drift from the registry before later adoption slices begin.

Suggested fix: reclassify this example as Derived + RuntimeConsumed +
Provenance + ProducerOwned, and either mark it Transient / RenderConsumed where
appropriate or explicitly distinguish current-run cleanup relevance from
roundtrip persistence.

==== M-02: v1 per-target FK policy is marked RuntimeConsumed after consumer migration

`docs/semantic_model/semantic_model.adoc` lines 691-698 classify
`_meta.helper_policies.fk.<target>` as *RuntimeConsumed* during the transition.

Current registry truth says the opposite for the post
`FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5` state. `registries/meta_registry.yaml`
lines 303-310, 336-340, 357-369, 377-381, and 403-415 state that v1
per-target policy is still written by `configure_fk_helpers` as a residual
producer-side compatibility shim, but runtime consumers read the v2 relation
policy only.

Risk: the transition-bridge example blurs producer-side residual write state
with runtime-consumer authority. That is exactly the kind of lifecycle
distinction this FTR is supposed to preserve.

Suggested fix: describe `_meta.helper_policies.fk.<target>` as Policy +
Persistent + TransitionBridge + historical producer-side residual, but not
RuntimeConsumed in the current state. If historical consumer behavior is worth
mentioning, label it explicitly as historical truth.

== Semantic Coherence Assessment

The vocabulary is mostly coherent and internally consistent. The chapter keeps
the key distinctions visible:

* metadata shape versus metadata meaning;
* semantic precedence versus age/generation/run identity;
* metadata schema version versus semantic contract version;
* source authority versus producer identity;
* current runtime truth versus historical residue.

The glossary terms align with their use in the chapter. The two findings above
are classification errors in examples, not evidence that the vocabulary itself
is contradictory.

== Overlap And Lifecycle Modeling Assessment

The overlapping-set model is correctly presented as non-taxonomic. The chapter
repeatedly states that fields may belong to several sets at once and that the
intersections are where lifecycle obligations live.

Lifecycle phases are descriptive names for observation points, not claims that
the runtime already exposes phase machinery. The text explicitly says each
event is a "candidate observation point" and that the semantic model does not
require all events to be observed.

The main lifecycle risk is example accuracy: the derived-provenance and v1
policy examples currently overstate persistence/runtime-consumer membership.

== Operational Vocabulary Assessment

The vocabulary expansion is useful but now load-bearing enough that example
accuracy matters. Terms such as *authoritative*, *historical residue*,
*semantic drift*, and *bridge* are coherent with prior semantic-model review
artifacts and the FK-helper lifecycle registry entry.

Residual risk: these terms could become doctrine before per-family adoption
slices validate them. The current text reduces that risk by calling the sets
"candidate", "descriptive", and "rule-ready", but the two inaccurate examples
should be fixed so later readers do not inherit stale membership claims.

== Concrete Example Assessment

* *helper_policies.fk v1 -> v2 coexistence:* accurate in broad bridge framing,
  but the v1 per-target shape's current runtime-consumer status is misstated
  in one membership example.
* *derived helper provenance:* accurate as derived provenance consumed by
  helper cleanup/layout/validation, but persistence and roundtrip relevance
  are overstated in the overlap example.
* *render-layer metadata:* acceptable as illustrative; it is marked
  RenderConsumed + Transient and not canonical/persistent.
* *infer_fk_relations precedence posture:* accurate. The chapter keeps
  heuristic inference subordinate to explicit Policy / Configurable values.
* *configure_fk_helpers policy posture:* accurate enough. It is treated as
  explicit policy/configuration and not as heuristic inference.
* *repeated-roundtrip residue examples:* conceptually accurate and grounded in
  the v1 -> v2 bridge, subject to the v1 runtime-consumer wording fix above.
* *registry consumer examples:* accurate as an operational review surface.
  Registries are described as description/test-oracle today, not runtime rule
  input.
* *adapter-readback authority posture:* acceptable. The text frames adapter
  readback as verification of roundtrip determinism, not canonical amendment.

== Residual Risks

* *Operational vocabulary expansion:* manageable, but later FTRs should keep
  vocabulary additions small and check them against existing registry truth.
* *Doctrine drift:* moderate. The chapter uses strong guidance language in
  places, but the draft/scaffold note and "rule-ready, not executable" wording
  keep it from becoming a runtime doctrine.
* *Stale `_meta.*` paths:* active risk, demonstrated by the two minor findings.
  Later semantic-model reviews should continue checking examples against
  `registries/meta_registry.yaml` and `registries/pipeline_step_registry.json`.
* *Adapter-readback authority posture:* low risk. The current wording does not
  claim adapter readback can amend canonical truth.
* *Lifecycle-phase interpretation drift:* low-to-moderate. The phases are
  descriptive today, but future readers may treat them as runtime phases unless
  later FTRs keep the "candidate observation point" framing.
