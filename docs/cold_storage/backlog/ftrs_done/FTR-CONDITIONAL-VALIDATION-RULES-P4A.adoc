===== FTR-CONDITIONAL-VALIDATION-RULES-P4A -- Gate declarative validations with explicit switches

*Status:* Done
*Priority:* P4A
*Area:* Domain validations / workbook configuration
*Prerequisite:* FTR-STANDARD-REFERENCE-VALIDATIONS-P4A

==== Purpose

Allow declarative validation rules to be enabled only when an explicit
condition applies.

Some configuration models have optional modules, optional product features, or
runtime switches. In those models, a relation can be structurally valid only
when a switch is active, and irrelevant when the switch is inactive. Consumer
repositories should not need plugins just to say "run this standard validation
only for rows or configurations where this flag is enabled."

==== Example Shape

[source,yaml]
----
- step: validate_references
  mode: fail
  rules:
    - type: foreign_key
      frame: calculation_targets
      columns: [secondary_target_variable_id]
      target: variables
      target_columns: [variable_id]
      when:
        column: secondary_target_required
        equals: true

    - type: unique_reference
      frame: optional_operation_routes
      columns: [transaction_type_id, operation_id]
      enabled_when:
        frame: feature_switches
        key: operation_routing
        column: enabled
        equals: true
----

==== Goals

* Support a small safe predicate vocabulary for `when` clauses, aligned with
  generic frame extraction where possible.
* Treat `when` as row-local validation gating against the validated frame.
* Treat `enabled_when` as global rule enablement from an explicit
  settings/switch frame.
* Keep inactive validations visible in reports so skipped checks are auditable.
* Reuse the standard validation finding shape.
* Make the condition part of the pipeline contract, not hidden plugin code.

Skipped-rule summaries should be emitted through the same configured findings
or report frame used by `validate_references`, with `severity: skipped` or an
equivalent explicit status.

==== Non-goals

* Do not add general pipeline branching or DAG routing.
* Do not evaluate Python expressions from YAML.
* Do not encode product- or domain-specific switch semantics in core.
* Do not replace domain-specific validations whose predicate itself requires
  business interpretation.

==== Acceptance

* Standard validation rules can be guarded by explicit `when` or
  `enabled_when` configuration.
* Inactive rules produce an auditable skipped-rule summary in warn/report mode.
* Missing switch frames, missing switch keys, and unsupported predicates fail
  with clear diagnostics unless configured as optional.
* Tests cover row-local conditions, global switches, inactive rules, missing
  switch configuration, warn mode, and fail mode.
* The feature is documented as validation gating, not as pipeline control flow.
