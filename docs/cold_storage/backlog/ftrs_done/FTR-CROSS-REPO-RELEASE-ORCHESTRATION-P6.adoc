===== FTR-CROSS-REPO-RELEASE-ORCHESTRATION-P6 -- Reduce operator cognitive load for multi-repo releases

*Status:* Done
*Priority:* P6
*Area:* Cross-repository release orchestration

==== Delivery scope

Implementation landed in `spreadsheet-handling` as a single core
commit naming this FTR. The slice is intentionally documentation-heavy
with one small read-only helper:

* New `tools/release_status.sh` (read-only) -- one-shot cross-repo
  state summary: per-repo branch/HEAD/tag, demo pin extraction, pages
  versioned snapshots, cross-correlation, next-action hint.
* New `make release-status` Make target wrapping the helper.
* Runbook §"Cross-Repo Release Sequence" -- three-stage table
  (core/demo/pages) with explicit dependencies and the per-stage
  verification helper to run.
* Runbook §"Cross-Repo Recovery Scenarios" -- five recovery cases
  (demo tag pushed before main alignment; stale workflow rerun;
  PyPI succeeded but docs failed; demo publish green but CI hygiene
  red; pages publish green but public site not yet visibly updated).
* `repository_boundaries.adoc` §"Release-time data flow between the
  three repositories" -- explicit directional table making
  `core -> PyPI`, `demo -> released core`, `pages <- publish outputs`
  unmistakable, plus the two rules that fall out of it.

No demo or pages commit. No new automation, no semantic-release
tooling, no automatic version propagation. Sibling review note
archived alongside this file.

==== Goal

Reduce operational friction in cross-repository releases involving:

* core library publication;
* demo alignment;
* Pages publication;
* version synchronization.

The `v0.1.0b6` beta release proved that the overall architecture works end
to end, but the process still relied heavily on operator memory and manual
sequencing.

This FTR aims to improve release orchestration clarity without prematurely
over-automating the workflow.

==== Motivation

Observed during the beta release:

* release activities spanned multiple repositories: core, demo, pages;
* ordering mattered significantly (pages bootstrap -> PyPI publish ->
  demo pin update -> demo tag publish);
* operators had to manually reason about branch state, tag ancestry, workflow
  version selection, and which failures were blocking vs incidental;
* several repair actions were valid but non-obvious (fast-forwarding `main`,
  avoiding reruns of stale workflow definitions, distinguishing release
  artifacts from CI hygiene).

The release succeeded, but the cognitive load was high.

==== Scope

This FTR covers:

* release orchestration clarity;
* sequencing visibility;
* cross-repository coordination aids;
* operational ergonomics.

This FTR does not cover:

* full release automation;
* monorepo migration;
* semantic-release tooling;
* automatic version propagation;
* replacing human-controlled releases.

==== Acceptance

* The release process clearly distinguishes authoritative repositories,
  consumer repositories, and publish repositories.
* Release sequencing is explicitly documented.
* Operators can determine the current release stage, the expected next step,
  and the safe recovery actions.
* Common recovery scenarios are documented:
** failed demo publish,
** stale workflow rerun,
** release from non-`main` branch,
** partial publish success.
* Cross-repository dependencies are explicit:
** core -> PyPI,
** demo -> released core version,
** pages <- publish outputs.
* Release operators can perform a release without relying on undocumented
  knowledge.

==== Candidate Improvements

Possible future enhancements:

* lightweight release helper scripts;
* release-state visualization;
* cross-repo release checklist tooling;
* standardized release verification commands;
* release-status summary generation.

==== Notes

The `v0.1.0b6` release demonstrated that the architecture is fundamentally
viable. The primary problem was not technical failure, but operator
cognitive overhead during coordination and recovery. This FTR intentionally
favors clarity and observability over aggressive automation.
