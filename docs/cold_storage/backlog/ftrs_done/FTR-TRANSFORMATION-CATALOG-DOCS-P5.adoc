===== FTR-TRANSFORMATION-CATALOG-DOCS-P5 -- Publish a parameter catalog for pipeline transformations

*Status:* Done (closed 2026-05-23)
*Priority:* P5
*Area:* Documentation / user guidance

==== Purpose

Create a maintained documentation catalog for pipeline transformations and
their YAML parameters after the Pages artifact structure is clear.

The project now has enough generic transforms that users can no longer infer
the right step from examples alone. A catalog should answer practical questions:

* Which transform should I use for this shape?
* What parameters does it accept?
* Which parameters are required?
* What does the inverse path look like?
* Which metadata does the step read or write?
* Which backends and roundtrip guarantees apply?

==== Scope

The first catalog should cover at least:

* `expand_xref` / `contract_xref`
* `decode_cell_values` / `encode_cell_values`
* `expand_compact_multiaxis` / `contract_compact_multiaxis`
* `split_by_discriminator` / `merge_by_discriminator`
* sparse crosstable collapse/expand
* lookup/FK helper projection steps
* validation configuration steps
* workbook view and frame lifecycle steps
* structured output writers

Each entry should document:

* step name and aliases, if any
* purpose and canonical use case
* required parameters
* optional parameters and defaults
* input frames and output frames
* inverse or recomposition step
* metadata read/written
* common failure modes
* minimal YAML example
* link to deeper concept or FTR documentation

==== Goals

* Make the transformation surface discoverable without reading source code.
* Keep examples short but executable in spirit.
* Distinguish canonical transforms, view preparation steps, validations, and
  output writers.
* Explain when plain xref is enough and when compact multiaxis is justified.
* Document backend dispositions where a step affects spreadsheet rendering or
  reimport.
* Prefer generating the catalog from registry metadata where practical, with
  hand-written narrative around it.

==== Non-goals

* Do not rewrite all user documentation in this FTR.
* Do not promise a stable 1.0 public API before the beta line has settled.
* Do not duplicate every historical FTR detail.
* Do not decide the Pages repository or release-publishing structure here; that
  belongs to `FTR-PAGES-AND-RELEASE-NOTES-P5`.

==== Acceptance

* A transformation catalog exists in the documentation tree and is linked from
  the internal guide or user guide candidate path.
* The catalog covers the maintained transformation steps and names their
  required and optional parameters.
* At least one entry demonstrates free-text xref usage and one entry
  demonstrates compact coded-cell usage.
* The catalog links to relevant concept notes and FTRs.
* A lightweight consistency check or documented maintenance path keeps catalog
  step names aligned with the pipeline step registry.

==== Closing note (2026-05-23)

Catalog landed at `core/docs/user_guide/ch03_transformation_catalog/`:

* `01_catalog_overview.adoc` -- inverse-kind taxonomy and registry-alignment rule
  (catalog step names must match `runtime_name` in
  `pipeline_step_registry.json`).
* `02_bijective_transforms.adoc` -- full parameter tables for `expand_xref` /
  `contract_xref` and `decode_cell_values` / `encode_cell_values`; named
  entries (with "full parameter documentation planned" markers) for
  `expand_compact_multiaxis` / `contract_compact_multiaxis`,
  `split_by_discriminator` / `merge_by_discriminator`, `sparse_collapse` /
  `sparse_expand`, `flatten_headers` / `unflatten_headers`.
* `03_supporting_steps.adoc` -- reference tables for FK helpers and lookup,
  validation, configuration, infrastructure and output writers.

Acceptance is met: free-text xref and compact coded-cell entries demonstrate
the two ends of the surface, concept-doc / FTR links are in place, and the
registry-name alignment is documented as the maintenance path. Remaining
"planned" parameter tables for the composite/discriminator/sparse/flatten
entries are tracked in the catalog text itself; no new core FTR is required.
