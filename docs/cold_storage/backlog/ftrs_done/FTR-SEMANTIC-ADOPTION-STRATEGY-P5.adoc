===== FTR-SEMANTIC-ADOPTION-STRATEGY-P5 -- Define no-big-bang adoption and slice template

*Status:* New
*Priority:* P5
*Area:* Semantic model / process

==== Goal

Define the adoption strategy: no big-bang rewrite, rule-ready before
rule engine, and a transformation-family slice template with explicit
slice questions and exit criteria. Produce a tentative adoption order
that later phases can consume when sequencing FTRs.

==== Scope

* Elaborate the Adoption Strategy chapter in
  `docs/semantic_model/semantic_model.adoc`.
* Record the candidate transformation families (FK Helpers, Derived
  Columns, Discriminator Split, Rendering / Layout Metadata, Validation
  Metadata, Pipeline Step Semantics, Roundtrip / Reimport Metadata,
  Adapter Metadata) and a tentative order.
* Define the slice questions (what does the family read / write, which
  semantic sets, lifecycle events, precedence, acceptable ambiguity,
  registry / docs / tests / guards involved) and slice exit criteria.
* Align the slice template with the existing semantic-refactor loop in
  `docs/developer_guide/ch08_git_and_workflow/01_git_and_workflow.adoc`
  ("FTR Slicing for Metadata and Registry Changes") and the FTR Sizing
  Preflight, rather than introducing a parallel workflow.

==== Out of scope

* No execution of a slice. The FK-helper pilot is owned by
  `FTR-SEMANTIC-CONTRACT-MATRIX-PILOT-FK-HELPERS-P5`.
* No new metadata fields, registry roots, or runtime changes.

==== Seed sections elaborated

* Adoption Strategy

==== Acceptance (rough)

* The chapter records the no-big-bang stance and the rule-ready vs
  rule-engine boundary.
* Slice questions and slice exit criteria are listed.
* A tentative adoption order across the candidate families is recorded,
  with the FK-helper slice marked as the first pilot.
* The slice template is consistent with the FTR Slicing / Sizing
  Preflight guidance already in the developer guide.

==== Code and guards

* No production code. No tests. No guards. Documentation only.
