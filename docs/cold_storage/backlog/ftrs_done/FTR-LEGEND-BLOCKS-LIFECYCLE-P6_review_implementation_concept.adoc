The concept is a sound bridge from Functional Model intent to safe, ordered,

===== Medium

*M1 -- "(FM)" lifecycle expectation for `resolved` cites an attribute the ch04
entry does not yet state.*

Quote (section 3, Resolved Legend Working Form):
____
*Lifecycle expectation (FM):* run-local; re-derived each forward run; not
trusted from prior `_meta`; `drop_if_stale` *and* `drop_at_persistence`;
carrier-durable in the workbook-embedded carrier.
____

The current ch04 functional-model entry tags the resolved working form *only*
`drop_if_stale` and omits `drop_at_persistence` -- this is exactly open finding
*M1* in `_review_functional_model.adoc`, which is not yet fixed. The concept
silently resolves that gap and attributes the corrected pair to "(FM)". The
substance is correct (`drop_at_persistence` is the right classification, code-
confirmed), but the lineage now claims the Functional Model says something its
ch04 text does not yet say.

Risk: if the FM ch04 M1 correction is never made, concept and Functional Model
diverge, and the traceability that this whole wave depends on (FM as intent) is
quietly broken. The concept already cross-references review finding *L4* for the
Code Resolution Rule, so the omission of an *M1* cross-reference here is an
inconsistency in its own citation discipline.

Smallest correction: add a one-line cross-reference to functional-model review
*M1*, noting the ch04 entry currently states only `drop_if_stale` and that the FM
correction is a precursor to this concept's lifecycle claim. Optionally list the
FM ch04 M1 fix as an explicit precursor doc-edit before Slice A.

*M2 -- Slice C mislabels the tests that reference `__legend_blocks`.*

Quote (Slice C, Files):
____
... carrier-boundary guard tests referencing `__legend_blocks`.
____

The tests that actually hard-code `__legend_blocks` are registry-classification
assertions: `tests/architecture/semantic_invariants/test_meta_registry.py:89`
and `test_meta_registry_hardening.py:104`. There are no carrier-boundary guard
tests referencing the key.

Risk: a Slice C executor searches for "carrier-boundary guard tests", finds
none, and may not anticipate that removing the registry `__legend_blocks` entry
breaks two semantic-invariant tests that enumerate it. The slice's
"before-and-after green" gate is then mis-scoped.

Smallest correction: name the two registry-invariant tests in Slice C's file
list as the ones to update when the `__legend_blocks` registry entry is removed;
drop the "carrier-boundary guard tests" phrasing.
