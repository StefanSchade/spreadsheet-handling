= FTR-POST-RELEASE-LIFECYCLE-AND-RESET-P5

== Status

Open

== Type

Process Improvement

== Context

The `v0.2.0` release (tagged `cd66fbf`, 2026-06-03) exposed a missing
post-release activity. The release runbook describes how a release is
prepared and published but does not define the transition back into
normal development mode.

Three days of post-release work made the gap concrete. Five feature FTRs
and one ADR-based closure landed between `v0.2.0` and HEAD (22 commits):

* `FTR-TEXT-ORIENTATION-ROUNDTRIP-P5` follow-up (`850dd5b`)
* `FTR-CELL-ALIGNMENT-ROUNDTRIP-P5` (`218a25a`, `0416989`)
* `FTR-PRESENTATION-META-CARRIER-AUTHORITY-P5` (`d01cb63`, `c7effc3`)
* `FTR-VERTICAL-ALIGNMENT-ROUNDTRIP-P5` (`810ccbd`)
* `FTR-VISIBLE-CELL-TYPING-STRATEGY-P5` closed via
  `ADR-VISIBLE-CELL-TYPING-STRING-SUBSTRATE` (`fc577f9`)

These FTRs each added release-note content. The content was added under
the still-present heading `== 0.2.0 -- planned`, with no rename or
section split at tag time. As a result, the live release-notes file now
claims that text orientation, horizontal alignment, carrier authority,
vertical alignment, and the visible-cell-typing decision all shipped in
`0.2.0` — none of them did. The drift this FTR was created to prevent
has already materialised.

The README link bake from release prep (`README.md:29` →
`/versions/v0.2.0/`) also has no companion restore step in the runbook,
so `main` remains in a release-frozen link state.

This FTR records the structural decision needed to repair the current
state and prevent recurrence at the next tag.

A readiness assessment driving this refinement is at
`docs/cold_storage/backlog/ftrs_done/FTR-POST-RELEASE-LIFECYCLE-AND-RESET-P5_ftr_readiness.adoc`.

== Problem

Two distinct gaps share one root cause:

. *No defined boundary between released and unreleased release-note
  content.* When an FTR adds release notes after a tag, there is no
  documented rule for where the addition goes. The current file shape
  (a single heading that doubles as both "in-progress" and "shipped")
  invites the drift observed since `v0.2.0`.

. *No documented post-release sweep.* The runbook covers pre-release
  preparation (link bake, branch topology, version checks) and
  publication validation, but has no "Post-release" section listing
  the activities that must happen *after* the tag to restore
  development-mode state.

The first is structural and must be decided. The second is mechanical
once the structural decision is in place.

== Goal

. Decide the release-notes structural shape: how released and
  unreleased content coexist in the single file the project keeps.
. Record the placement policy that all FTR implementation reviews
  must enforce: where post-release release-note additions belong.
. Document the post-release sweep in the runbook so the next tag does
  not repeat the `v0.2.0` drift.
. Apply a one-time retro-fix to bring the live release-notes file into
  conformance with the new shape — without losing the 74 lines of
  post-tag content already added.

This FTR does not automate any of the above. Automation is the scope of
`FTR-RELEASE-PROCESS-AUTOMATION-P5`.

== Structural Decision

The project adopts the *single file, two sections* shape for release
notes.

Adopted::

* One `docs/release_notes/release_notes.adoc` file.
* Released sections are *frozen* at tag time. Each shipped release line
  occupies one `== 0.X.Y` section whose content matches the
  release-notes file as it existed at the tagged commit. Once frozen,
  a released section is never edited.
* Exactly one *active unreleased section* exists at the top of the
  file, titled `== 0.(X+1) -- planned` (or equivalent next-version
  marker). All in-flight release-note content lives here. This section
  grows during the development cycle and is renamed to
  `== 0.(X+1) (released)` at tag time.
* *Post-release release-note additions belong to the unreleased
  section.* An FTR landing after the most recent release tag adds
  its release-notes bullets under the active `-- planned` heading,
  never under a frozen released section.
* *Git tags remain the authoritative release record.* The release
  notes are a human-readable summary of intent, not the authoritative
  record. `git show <tag>:docs/release_notes/release_notes.adoc` is
  the canonical source for the frozen content of any release section.

Why this shape:

* Matches the existing project preference (concise, focused on the
  current line; history in git; archived versions on Pages).
* Released sections do not grow once frozen, so the file does not
  accumulate indefinitely *per release* — only one short frozen block
  is added per shipped tag.
* No content is deleted on tag; no content is lost during a retro-fix.
  Pre-tag content is recoverable verbatim from
  `git show <tag>:docs/release_notes/release_notes.adoc`.
* Pattern is reversible: a future FTR can introduce per-version files
  or a Pages snapshot-on-tag model without semantic loss.

Rejected alternatives:

* *Per-version file* (`release_notes_v0.2.0.adoc` etc.) — heavier
  tooling cost and runbook discipline than the current project
  benefits from; deferrable to a future automation FTR.
* *Snapshot-on-tag with in-place full reset* (release notes are
  emptied after each tag; Pages keeps the snapshot) — depends on
  Pages snapshot reliability, which is itself the open question in
  `FTR-RELEASE-PROCESS-AUTOMATION-P5`. Adopting it now couples the
  manual procedure to an unsettled publish-side issue.
* *Single growing section* (status quo) — caused the observed drift.

== Placement Policy

For every FTR that lands after the most recent release tag:

* Release-note bullets are added under the *active unreleased
  section*, not under the most recent released section.
* If the active unreleased section does not yet exist (because the
  previous tag's post-release sweep has not run yet), the implementer
  creates it as part of their FTR's commit, before adding their
  bullets.
* Implementation reviewers explicitly verify placement during the
  review pass. A bullet added under a frozen `(released)` section is
  a release-relevant defect.

This policy is mirrored once into the runbook so it is discoverable
from the release-management surface, and once into
`docs/ai_info/review_instructions.adoc` so review agents enforce it
mechanically.

== Scope

The implementation work this FTR commits to is split into two
sub-slices. Only sub-slice A is in scope for the first implementation
commit; sub-slice B may fold in or land immediately after.

=== Sub-slice A — Retro-fix and runbook (in scope, first commit)

. Add a "Post-release" section to
  `docs/developer_guide/ch09_release_management/02_release_runbook.adoc`
  listing the sweep steps:
+
* freeze the active `-- planned` section as `== 0.X.Y (released)` at
  tag time;
* open a new `== 0.(X+1) -- planned` section at the top of the file;
* restore `README.md` `/versions/<tag>/...` links to `/latest/...`
  per the existing version-baking policy;
* verify Pages publication and package publication (cross-reference
  `FTR-RELEASE-PROCESS-AUTOMATION-P5` for the publish-side automation
  question);
* record any release observations as input for future process
  improvements.

. Apply the one-time retro-fix to
  `docs/release_notes/release_notes.adoc`:
+
* the `== 0.2.0` section's content is set to match
  `git show v0.2.0:docs/release_notes/release_notes.adoc` verbatim,
  with the heading renamed from `0.2.0 -- planned` to
  `0.2.0 (released)`;
* the 74 lines added between `v0.2.0` and HEAD move into a new
  `== 0.3.0 -- planned` section placed *above* the released section.
+
No release-note content is deleted in either direction. The retro-fix
is mechanical: pre-tag content is taken from the tagged file; post-tag
content is identified by `git diff v0.2.0..HEAD -- docs/release_notes/release_notes.adoc`.

. Restore `README.md` link state to `/latest/...` per the new runbook
  rule. The specific URL to restore is the `/versions/v0.2.0/...`
  link at `README.md:29`; the baking policy at
  `FTR-RELEASE-README-VERSION-BAKING-P5 § Link-versioning policy`
  remains the authority for what to do at the next tag.

=== Sub-slice B — Review-instructions reminder (optional, may fold in)

. Add a one-line reminder to
  `docs/ai_info/review_instructions.adoc` so reviewers enforce the
  placement policy during the next FTR's implementation review:
+
"Release-note bullets added after the most recent release tag belong
 under the active unreleased section. A bullet under a frozen
 `(released)` section is a release-relevant defect."

Sub-slice B can fold into sub-slice A as one extra line, or ship as a
follow-up doc commit immediately after. It is not blocking.

== Non-Goals

* No automation of the post-release sweep. Automation belongs to
  `FTR-RELEASE-PROCESS-AUTOMATION-P5`. This FTR settles the manual
  procedure that automation will later codify.
* No change to the bake side of `FTR-RELEASE-README-VERSION-BAKING-P5`.
  Only the runbook gains a companion *restore* section; the bake
  policy itself is unchanged.
* No per-version release-notes files. The project keeps the single
  `release_notes.adoc` file.
* No migration of historical release information. `v0.1.0bX` content
  is not reconstructed.
* No changes to already-published releases. The PyPI page and Pages
  archive for `v0.2.0` remain frozen at their tag state.
* No deletion of any release-note content. All 74 post-tag lines
  move; none are dropped.
* No new release-notes file structure beyond the released-frozen /
  unreleased-active split.
* No backlog index or cold-storage movement triggered by this FTR.
  Per-FTR cold-storage movement remains a per-FTR workflow concern.

== Acceptance Criteria

Functional (required, sub-slice A)::

* `docs/release_notes/release_notes.adoc` after sub-slice A contains
  exactly two top-level sections (beyond `== Related Links` and the
  document attributes):
** `== 0.3.0 -- planned` (active unreleased section) at the top;
** `== 0.2.0 (released)` (frozen) below it.
* The content of the `== 0.2.0 (released)` section matches
  `git show v0.2.0:docs/release_notes/release_notes.adoc` verbatim,
  modulo the heading rename from `0.2.0 -- planned` to
  `0.2.0 (released)`.
* The `== 0.3.0 -- planned` section contains the 74 lines added
  between `v0.2.0` and the pre-retro-fix HEAD, with no loss of
  content. Bullet ordering within the section may be preserved
  verbatim or grouped under the same subheadings used pre-retro-fix
  (`=== Spreadsheet backends and roundtrip behavior`,
  `=== Known limitations`).
* `README.md` no longer carries `/versions/v0.2.0/...` links;
  references resolve to `/latest/...` per the runbook's
  post-release rule. (The bake policy still applies at the next tag.)

Process (required, sub-slice A)::

* `docs/developer_guide/ch09_release_management/02_release_runbook.adoc`
  contains a top-level "Post-release" section listing the sweep steps
  enumerated above.
* The runbook references this FTR's *Structural Decision* and
  *Placement Policy* sections as the authority for the released-vs-
  unreleased split.

Optional (sub-slice B)::

* `docs/ai_info/review_instructions.adoc` carries the placement
  reminder. Acceptable as a one-line addition; no broader edit
  required.

Out of scope (deferred to `FTR-RELEASE-PROCESS-AUTOMATION-P5`)::

* A `make release-notes-check` (or equivalent) read-only sweep that
  asserts no post-tag content has accumulated in a frozen section.
* Automated Pages-snapshot reset.
* Automated README link bake/restore.

== Validation

The implementer should run, at minimum:

[source,bash]
----
# Confirm the released section matches the tag verbatim (modulo heading).
diff <(git show v0.2.0:docs/release_notes/release_notes.adoc \
        | sed 's/^== 0.2.0 -- planned/== 0.2.0 (released)/') \
     <(sed -n '/^== 0.2.0 (released)/,/^== /p' \
        docs/release_notes/release_notes.adoc \
        | sed '$d')

# Confirm no v0.2.0 versioned link remains on main.
grep -nE 'versions/v0\.2\.0' README.md && echo "FAIL: v0.2.0 link present"

# Confirm the runbook gained the Post-release section.
grep -nE '^==+\s*Post-release' \
    docs/developer_guide/ch09_release_management/02_release_runbook.adoc

# Confirm the active unreleased section exists at the top.
grep -nE '^==\s*0\.3\.0\s+--\s+planned' docs/release_notes/release_notes.adoc

# Confirm no implementation code or tests changed.
git diff v0.2.0..HEAD --name-only | grep -vE '^docs/'
----

The diff comparison is the central correctness check: a non-empty diff
means the released section drifted from the tag and the retro-fix is
incorrect.

== Risks and Notes

Risks::

* *Structural-decision drift during implementation.* If the
  retro-fix is performed without the runbook section in the same
  commit, an implementer can produce a shape that does not match the
  structural decision recorded here. *Mitigation:* sub-slice A is one
  commit; both the runbook section and the release-notes split land
  together.

* *README bake-vs-restore ambiguity.* The bake side
  (`FTR-RELEASE-README-VERSION-BAKING-P5`) is documented; the restore
  side is what this FTR adds. A naive restore could either preserve
  the bake (drift remains on `main`) or over-restore (breaks the
  next PyPI archive page if the bake is not redone at the next tag).
  *Mitigation:* the runbook's new "Post-release" section is the
  authority; the README change in sub-slice A follows it.

* *Continued misplacement of post-release notes until policy is
  recorded.* Between sub-slice A landing and the next tag, any new
  FTR's review must explicitly check placement. *Mitigation:*
  sub-slice B's one-line reminder in
  `docs/ai_info/review_instructions.adoc`.

* *Scope creep into publish-side automation.* The Pages "Browse by
  version" inconsistency described in
  `FTR-RELEASE-PROCESS-AUTOMATION-P5` is real but distinct.
  *Mitigation:* explicit "See also" cross-reference in this FTR's
  References section; no scope mixing.

* *Subheading preservation.* The 74 post-tag lines were added under
  existing subheadings (`=== Spreadsheet backends and roundtrip
  behavior`, `=== Known limitations`). The retro-fix should preserve
  those subheadings inside the new `== 0.3.0 -- planned` section so
  the move is purely structural, not editorial.

Notes::

* Sub-slice A is reviewable as a single focused documentation commit.
  No code, registry, or implementation tests change.
* The structural decision is *reversible*. A future FTR can introduce
  per-version files or snapshot-on-tag without losing semantic
  meaning, because no content is deleted by the path chosen here.
* The placement policy and the runbook section are both authoring-
  side; the next published Pages snapshot is whatever the runbook
  procedure produces at the next tag, not what is in flight today.
* `git show v0.2.0:docs/release_notes/release_notes.adoc` is the
  *source of truth* for the frozen `0.2.0` section content. Any
  manual reconstruction risks discrepancies; the implementation
  should copy from the tagged file rather than re-deriving from the
  current file by subtraction.
