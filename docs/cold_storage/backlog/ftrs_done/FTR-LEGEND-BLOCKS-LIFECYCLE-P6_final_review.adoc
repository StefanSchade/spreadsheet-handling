*M1 -- Registry-entry removal left technical-model doc sites referencing
`__legend_blocks` as a live registry/carrier entry.*

invariant tests, but the technical model still enumerates it as current. Most

`docs/technical_model/ch05_registries/registry_semantics/02_meta_layer.adoc:144`
-- under a table titled *"Mapping to the live registry"*:
`| operational-carrier | ... | workbook_meta_blob, _hidden, __autofilter_ref,
__legend_blocks.`
____

This is now factually false -- `__legend_blocks` is no longer in the live
registry. Also stale: `02_meta_layer.adoc:107` (describes it as a current
meta-layer entry), `04_carrier.adoc:91`, and glossary `ch09_glossary/01_terms.adoc:183`.

* *Risk:* the FTR's own acceptance criterion is "documented classification,
  consistent across runtime, registry, and the functional model". Removing the
  registry entry while leaving the technical model advertising it recreates
  exactly the registry/doc disagreement this FTR exists to eliminate. A
  technical-model reader will look for `__legend_blocks` in the registry and not
  find it.
* *Smallest correction:* update those four sites -- at minimum the
  `02_meta_layer.adoc:144` "live registry" mapping table -- to drop
  `__legend_blocks` from the current operational-carrier examples (or mark it
  removed).
