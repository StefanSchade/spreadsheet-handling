===== FTR-PAGES-ROOT-ARCHIVE-NAV-P5 -- Pages root as latest banner + version archive portal

*Status:* Done
*Priority:* P5
*Area:* Pages root navigation

==== Delivery scope

Implementation landed in the same editorial slice as
`FTR-PUBLIC-DOCUMENTATION-ENTRYPOINTS-P5`. The Pages-root deliverable
is the rewrite of `docs/pages/root_index.adoc` into a *Latest
release: <tag>* banner with three deep links (user guide, release
notes, product-business-slice walkthrough deck) and a *Browse by
version* table seeded with the v0.1.0b6 row (demo tag
`v0.1.0b6-demo` merged into the same row per the FTR row-merge rule).
The stale "demo / presentations deferred" wording was removed. The
publisher script `scripts/build_docs_pages.sh` is unchanged; no new
build steps were added.

Per-version-list maintenance becomes a release-checklist step: a new
bullet in `docs/developer_guide/ch09_release_management/02_release_runbook.adoc`
*Core Release Checklist* tells the releaser to bump the banner tag
and add a row to the *Browse by version* table at release time.

Sibling review note
`FTR-PUBLIC-DOCUMENTATION-ENTRYPOINTS-P5_review.adoc` covers both
FTRs as one editorial slice. Final verdict: *Ship*; no open findings.

==== Purpose

Make the Pages root a deliberate two-purpose entry point: a prominent
"latest release" banner that links to the current core and demo
artifacts, and a per-version archive list that lets users browse
documentation for any published release.

==== Context

The current Pages root (rendered from
`core/docs/pages/root_index.adoc`) links only "Latest core
documentation" and the documentation for the release that generated the
index page. It explicitly states that demo walkthrough output and
presentation output are *reserved for later publishing slices* -- that
claim is now stale; the demo `publish-docs` workflow writes
`versions/<tag>/demo/slides/` and `latest/demo/slides/`. The root is
also not an archive: older releases become discoverable only by URL
guessing.

==== Scope

Rewrite `core/docs/pages/root_index.adoc`:

* prominent "Latest release" banner with the current core release
  (e.g. *Latest release: v0.1.0b6*) and three deep links:
** *User guide* -> `/latest/core/user-guide/`;
** *Release notes* -> `/latest/core/release-notes/`;
** *Demo visual walkthrough* -> `/latest/demo/slides/...` (link to the
   primary maintained journey deck);
* a *Browse by version* section listing every published
  `versions/<tag>/` entry as a single row. Each row shows the version
  tag and one-line context plus links to the per-version core and (if
  present) demo artifacts;
* a brief note that developer documentation, the FTR backlog, and
  architecture material live in the source repository and are not
  published here by design;
* drop the stale claim that demo / presentation output is deferred.

==== Non-goals

* No JavaScript-driven navigation, no search, no filter UI.
* No automatic crawler over the Pages repo's contents.
* No per-version umbrella page at `versions/<tag>/index.html` -- that
  remains the explicit out-of-scope item from the closed
  `FTR-PAGES-AND-RELEASE-NOTES-P5`.
* No change to `make docs-pages` or the publishing workflow. Build still
  goes through the existing `scripts/build_docs_pages.sh`.
* No promotion of developer / internal documentation to Pages.

==== Acceptance

* The Pages root renders with a clear *Latest release: <tag>* banner
  and three working deep links (core user guide, release notes, demo
  walkthrough deck).
* A *Browse by version* section lists every published
  `versions/<tag>/` entry. Demo tags (`v<core-tag>-demo`) are merged
  into the matching core release row rather than shown as separate
  rows.
* The "demo / presentations deferred" wording is gone.
* A brief note clarifies that developer documentation, FTR backlog, and
  architecture material live in the source repository and are not
  published here.
* `make docs-pages` still succeeds with zero asciidoctor errors; no new
  build steps are added.

==== Dependencies

* Co-ships with `FTR-PUBLIC-DOCUMENTATION-ENTRYPOINTS-P5` as one
  editorial slice. The core README's *Browse documentation by version*
  link expects the per-version list this FTR adds.
* Per-version list maintenance becomes a release-checklist item until /
  unless automation is added later (potentially under
  `FTR-RELEASE-README-VERSION-BAKING-P5` scope).

==== Implementation notes

* Manual per-version list is acceptable for the beta line (one or two
  releases). A short asciidoc list under *Browse by version* is enough.
* If automation is wanted later, `scripts/build_docs_pages.sh` can
  pass a `{versions-list}` attribute derived from `git tag -l 'v*'`,
  filtering out demo-suffix tags and rendering the resulting list via
  an asciidoc template -- not required for this FTR.
* The current `=== Scope` block at the bottom of `root_index.adoc` was
  written for the first publishing slice and is stale once this FTR
  lands; drop or rewrite it.

==== Risks / open questions

* Per-version list drift if maintainers forget to update it on each
  release. Mitigation: add to the Core Release Checklist in
  `developer_guide/ch09_release_management/02_release_runbook.adoc`.
* Whether the *Latest release* banner should also show a small
  "installed a different version? browse by version below" hint to
  help PyPI-page visitors who installed an older release. Recommended
  yes; cheap to add.
