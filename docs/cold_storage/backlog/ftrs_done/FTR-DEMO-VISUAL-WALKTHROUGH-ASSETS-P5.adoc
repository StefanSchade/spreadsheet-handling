==== Purpose

Capture a small set of *manual* screenshots for the product business
slice journey so the demo README's inspect / edit steps can be
supported by visual aids. Keep the first slice narrow: no automation,
no CI capture, no downloadable workbook generation.

Existing Reveal.js publishing remains the primary live visual artifact
the demo README links to; this FTR adds static screenshots as the
*supplementary* aid for the eight-step first-hour tutorial defined by
`FTR-PUBLIC-DOCUMENTATION-ENTRYPOINTS-P5`.

==== Context

The eight-step demo tutorial benefits from a handful of small images
showing what the user should see in their spreadsheet at key points
(workbook overview, helper columns, list validations, a representative
edit). Automatic capture (headless Excel / LibreOffice, CI workflows)
adds significant complexity and is rejected for the first slice.
Manual capture by the maintainer at release prep time is sufficient
for the beta line.

This FTR is deliberately bounded by the approval decisions on the
public-documentation slice: no publishing automation, no downloadable
workbook generation, no screenshots for trails other than the
maintained product business slice.

==== Scope

* Capture (manually) 3-5 screenshots of the product business slice
  journey, covering at minimum:
** generated workbook overview;
** helper columns next to their FK column;
** list validations on `status` or `category`;
** a representative edit (before and / or after);
* Commit screenshots under
  `demo/docs/screenshots/product_business_slice/` as PNG files.
* Add a brief `demo/docs/screenshots/README.md` describing what each
  image shows, when it was captured (release tag / date), and how to
  refresh it.
* Demo `README.md` *may* reference these screenshots via the demo
  repository's relative path (so they render on the GitHub web view)
  if `FTR-PUBLIC-DOCUMENTATION-ENTRYPOINTS-P5` has already landed and
  the README points to local paths; if not, the README stays text-only
  for now and references are added by a follow-up.

==== Non-goals

* No automatic screenshot capture.
* No headless Excel / LibreOffice automation.
* No CI screenshot generation.
* No downloadable workbook generation in this slice.
* No publishing of screenshots to the Pages repo in this slice (that
  is a separate follow-up; see "Open questions" below for the
  placement policy that publishing slice must respect).
* No screenshots for ACME slice 1, ACME slice 3, or the Storyforge
  trail in this slice.
* No video walkthroughs, no PDF rendering of slide decks.
* No new automation around the existing Reveal.js publishing path.

==== Acceptance

* `demo/docs/screenshots/product_business_slice/` exists and contains
  3-5 PNG screenshots that cover the four content categories listed
  in *Scope*.
* Each screenshot is reasonably small (target <= 200 KB; larger only
  with a one-line justification in the directory README).
* The screenshots accurately reflect the demo as it runs against the
  currently-pinned core version at the time of capture.
* `demo/docs/screenshots/README.md` documents what each image shows
  and how a maintainer refreshes the set for a future release.
* No screenshot is referenced from a release-bound or version-frozen
  document with a label that implies it matches that frozen version
  (see "Open questions" on placement policy).

==== Dependencies

* Loosely follows `FTR-PUBLIC-DOCUMENTATION-ENTRYPOINTS-P5`. The
  README structure that screenshots support is owned there; this FTR
  can ship independently but is most useful once the eight-step
  tutorial is in place.
* A later follow-up FTR will address publishing screenshots to Pages,
  including the versioning / labelling decision noted below.

==== Implementation notes

* Manual capture: run the product business slice forward pipeline
  against the currently-pinned core, open the workbook in
  Excel or Calc, take cropped screenshots focusing on the relevant
  region (avoid full-window captures with surrounding chrome).
* File naming: descriptive kebab-case
  (e.g. `workbook-overview.png`, `helper-columns.png`,
  `list-validations-status.png`, `edit-status-before.png`,
  `edit-status-after.png`).
* The directory README should make refresh procedure explicit so the
  next maintainer (or AI agent) can redo the capture during release
  prep without guessing the framing.
* Existing Reveal.js decks under `/latest/demo/slides/...` remain the
  primary linked visual artifact in the demo README; screenshots are
  supplementary.

==== Risks / open questions

* *Drift.* Manual screenshots will drift out of sync with the
  software. Mitigation: a release-checklist item to review screenshot
  accuracy at each release; when reviewing reveals drift, prefer
  *deleting* the stale image over shipping a misleading one.
* *Placement policy for any later publishing slice* (out of this
  FTR's scope, but recorded here so the follow-up doesn't recreate the
  `/latest/` drift problem):
** screenshots placed under a shared `/assets/screenshots/...` path
   are acceptable only for *latest / current* illustrative material;
** any screenshot linked from versioned release documentation (for
   example from inside `/versions/<tag>/core/user-guide/`) must
   either live under `/versions/<tag>/...` or carry a clearly
   "latest / current release" label;
** the publishing slice must pick a model that honours the same
   frozen-context-versioned policy as
   `FTR-RELEASE-README-VERSION-BAKING-P5` (DRAFT).
* Whether to link screenshots from the demo README in this slice at
  all. Two acceptable options: (a) reference them via relative
  paths now so GitHub renders them inline; (b) hold all README image
  references for the publishing slice. Either is consistent with the
  narrowed scope.
