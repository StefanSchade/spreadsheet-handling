== Verdict

ACCEPT WITH FOLLOW-UP.

The Slice-A runtime implementation is safe to use as the explicit sheet-remap
prerequisite for later recomposition work. The only follow-up is a descriptive
registry-note correction: the runtime behavior drops/replaces visible sheet
keys, but `pipeline_step_registry.json` currently understates that frame-set
change.

== Architectural Assessment

`apply_workbook_view_sheet_mappings`
(`spreadsheet_handling.domain.workbook_views`) is implemented in the correct
layer and stays deliberately small. It delegates all identity resolution to
`resolve_workbook_view_sheet_mappings`, so identity remains based only on the
persisted `_meta.workbook_view.sheet_mappings` contract. There is no inference
from visible sheet names, order, labels, normalized names, columns, or other
heuristics.

The step re-keys visible workbook sheet names to logical workbook-view frames,
returns only mapped visible frames plus `_meta`, and preserves `_meta`
unchanged by identity. The optional `logical_frames` parameter strengthens
validation rather than weakening it: the resolver rejects mappings that point
to unknown logical frames and also rejects unknown `canonical_frame`
references when present.

Pipeline layering remains clean. Runtime registration in
`src/spreadsheet_handling/pipeline/registry.py` is thin and consistent with
other `frames_target` steps. Pipeline orchestration remains metadata-agnostic.
Adapters remain untouched and continue only to transport `_meta`.

Scope control is intact. This commit does not implement recomposition,
matrix/crosstable expansion or contraction, conflict detection,
precedence/merge logic, a generic reimport framework, `_meta.derived` use for
view mapping, or adapter-specific behavior.

== Findings

=== 1. SHOULD-FIX -- descriptive pipeline step registry understates frame-set change

Evidence:

* Runtime behavior in
  `src/spreadsheet_handling/domain/workbook_views.py::apply_workbook_view_sheet_mappings`
  constructs a fresh output mapping and copies only mapped logical frames plus
  `_meta`.
* `docs/internal_guide/architecture/pipeline_step_registry.json` records
  `_meta` as preserved, but leaves both `replaces` and `drops` empty for the
  new step.

Risk:

The runtime behavior is correct, but the descriptive review artifact suggests
the visible-sheet frame set survives unchanged. That can mislead later FTR
work, reviews, or architecture notes that consult the registry to understand
frame ownership and frame-set transitions.

Recommended fix:

Update the `frame_contract` entry for
`apply_workbook_view_sheet_mappings` so it explicitly describes that visible
sheet frames are replaced by logical view frames, while `_meta` is preserved.

=== 2. NICE-TO-HAVE -- `logical_frames` parameter meaning could be documented more explicitly

Evidence:

* `apply_workbook_view_sheet_mappings` passes `logical_frames` through to
  `resolve_workbook_view_sheet_mappings`.
* The resolver validates both `mapping.logical_frame` and optional
  `mapping.canonical_frame` against the provided set.
* The pipeline-step registry currently describes `logical_frames` only as an
  optional list parameter.

Risk:

A caller may assume `logical_frames` means "expected remapped outputs only" and
be surprised when a mapping fails because an explicitly recorded
`canonical_frame` is not included. This does not make the step unsafe, but it
does make the stricter contract easier to misuse.

Recommended fix:

Document `logical_frames` as the full allowed set of referenced logical frame
identities, including `canonical_frame` when present.

=== Accepted as-is

* Renamed visible sheets are handled explicitly through persisted
  `sheet_mappings`.
* Sheet order is irrelevant.
* Duplicate or ambiguous mappings are rejected by existing resolver behavior.
* Loud failures on malformed, undeclared, missing, or unknown mappings are
  preserved at the step boundary through resolver delegation.
* `_meta` is preserved unchanged.
* Unmapped visible workbook sheets are not carried through.
* YAML/config callability is tested and works.

from sheet names, order, labels, or columns. Conflict/precedence is explicitly
delegated to `FTR-WORKBOOK-VIEW-PAYLOAD-CONFLICT-PRECEDENCE-P6` and is not
exercised here beyond documenting the deferral.
