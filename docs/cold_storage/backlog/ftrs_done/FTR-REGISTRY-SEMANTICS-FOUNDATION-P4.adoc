===== FTR-REGISTRY-SEMANTICS-FOUNDATION-P4 -- Foundation vocabulary for the meta registry

*Status:* Done (closed 2026-05-23 as the implementation review's disposition)
*Priority:* P4
*Area:* Documentation / semantic foundation

==== Origin

The 2026-05-18 REVIEW-002 (architecture review, pass 3) flagged residual
semantic density around the vocabulary the meta registry rested on:
`canonical`, `derived`, `carrier`, `model`, `presentation`,
`workbook_identity`, and the data-level pair `original` / `derived`. The
terms were load-bearing across the meta registry, the architecture primer,
several concept docs, and AI-facing guidance, but they had never been
defined as a foundation. Each use silently asserted a slightly different
union of criteria, and edge cases (presentation intent that is
backend-restricted; read-path reconstructions that became de-facto
sources; persistable derived caches) were absorbed into the canonical
bucket by accident rather than placed deliberately.

This FTR is filed post-hoc to track the work done in commits `84c9056`
and `221e30e` on the `semantics` branch, which delivered the foundation
text. The FTR is created at the point of the post-publish hygiene pass
because the work pre-dated the FTR-per-commit discipline rebuild in the
P4/P5 era.

==== Goal

Provide a written foundation for the meta-registry vocabulary so that

* `canonical`, `derived`, `operational-carrier` are defined as distinct
  *layer* roles with explicit criteria;
* `model`, `presentation`, `workbook_identity` are defined as distinct
  *classification* roles, orthogonal to the layer axis;
* `original` and `derived` at the *data layer* are distinguished from
  `canonical` / `derived` at the *meta layer*, removing the silent
  cross-level conflation;
* `carrier` is defined narrowly, with `carrier-shape` named explicitly
  as the operational-carrier layer's content;
* the live `meta_registry.yaml` encoding maps onto the foundation
  vocabulary without being re-encoded in this slice.

==== Scope

This FTR delivers documentation only:

* `docs/technical_model/ch05_registries/registry_semantics/_chapter.adoc`
  (subsection intro and includes);
* `docs/technical_model/ch05_registries/registry_semantics/01_data_and_meta.adoc`
  (data layer vs meta layer; original vs derived data);
* `docs/technical_model/ch05_registries/registry_semantics/02_meta_layer.adoc`
  (strict five-criterion canonical; derived; operational-carrier; live
  registry mapping);
* `docs/technical_model/ch05_registries/registry_semantics/03_meta_classification.adoc`
  (model; presentation; workbook_identity; open questions on namespaces
  and straddling entries);
* `docs/technical_model/ch05_registries/registry_semantics/04_carrier.adoc`
  (narrow carrier definition; carrier-shape; historical drift; mapping
  to `carrier_artifact` and `read_path_hint`);
* `docs/technical_model/ch05_registries/_chapter.adoc` (include the new
  subsection ahead of the registries themselves);
* `docs/technical_model/ch04_concepts/metadata_canonical_vs_derived.adoc`
  (forward reference to the foundation, keeping the concept doc as the
  functional-vs-temporal framing entry point);
* `docs/technical_model/ch09_glossary/01_terms.adoc` (add a *category*
  column; add the meta-system foundation terms with cross-references
  into `registry_semantics/`).

==== Non-goals

* Re-encoding `meta_registry.yaml` to expose the foundation layer and
  classification axes directly. The YAML continues to use the older
  two-value layer axis and four-value classification field. Re-encoding
  is deferred to a separate future FTR; the mapping table is documented
  in `02_meta_layer.adoc` and `03_meta_classification.adoc`.
* Any code change. The foundation is conceptual; behavior is unchanged.
* Pruning, lifecycle, or ownership semantics for `_meta.derived`. Those
  are owned by `FTR-DERIVED-PROVENANCE-LIFECYCLE-P4A`. The foundation
  underwrites that FTR's vocabulary but does not pre-empt its work.

==== Acceptance

* The foundation chapter is included from
  `ch05_registries/_chapter.adoc` and renders without warnings as part
  of `technical_model.adoc`.
* The strict five-criterion definition of *canonical* is stated, with
  the historical-drift explanation that motivates each criterion.
* The data-layer (`original` / `derived`) and meta-layer
  (`canonical` / `derived` / `operational-carrier`) vocabularies are
  separated and the cross-level conflation is named.
* The classification axis is named with `model`, `presentation`, and
  `workbook_identity`, with operational consequences (conflict policy,
  roundtrip strictness, adapter portability, producer profile)
  enumerated.
* *Carrier* is defined narrowly and distinguished from IR, the
  canonical role, cell data, and the adapter object model.
  *Carrier-shape* is named as the operational-carrier layer's content.
* Every live registry term used in the project (the entries the
  foundation maps onto) appears in the foundation's mapping tables, so
  no live entry is left without a foundation reading.
* The glossary records the foundation terms with category labels and
  links back to the canonical definitions in `registry_semantics/`.
* No live behavior or YAML encoding is changed.
