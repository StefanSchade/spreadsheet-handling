===== FTR-GENERATED-ARTIFACT-MANIFEST-P4A -- Collect generated output files into a manifest

*Status:* Implemented (minimal slice)
*Priority:* P4A
*Area:* Output generation / release traceability

==== Purpose

Provide a small, generic manifest layer for pipeline steps that generate files
as side effects.

The structured YAML writer already records generated files, and the key-value
resource writer should do the same. Once a pipeline writes several artifact
families, users and CI need one place to see what was produced, which writer
produced it, and which source frames contributed.

Note: `FTR-LOCALIZED-RESOURCE-MATRIX-ROUNDTRIP-P4A` does not generate files.
Manifest coverage starts when resource or configuration writer steps emit
artifact report frames.

==== Example Shape

[source,yaml]
----
- step: write_artifact_manifest
  reports:
    - structured_yaml_files
    - key_value_resource_files
  output: generated_artifacts
  manifest_path: build/generated-artifacts.yml
  checksum: sha256
----

==== Goals

* Define a shared generated-file report shape for file-writing pipeline steps.
* Merge multiple writer report frames into one deterministic manifest frame.
* Optionally write the manifest as YAML or JSON.
* Record relative path, artifact kind, writer step, source frame names, row
  count where available, and optional checksum.
* Validate that reported paths stay inside configured output roots.
* Make generated artifacts visible in CI logs and review workflows.

==== Shared Writer Report Shape

Writer report frames should converge on at least these columns:

* `path`
* `artifact_kind`
* `writer_step`
* `source_frames`
* `row_count`
* `checksum`
* `status`

Paths are relative to the configured output root. Checksums, when enabled, are
computed over file contents and reported with relative paths only, so the
manifest remains stable across workspaces.

==== Non-goals

* Do not publish files to GitHub Pages or another remote destination.
* Do not decide which generated files should be committed.
* Do not clean stale output directories automatically in this slice.
* Do not replace writer-specific validation.

==== Implemented Minimal Slice

*Status:* Implemented (minimal slice)

* Pipeline step: `write_artifact_manifest`.
* Pure domain module: `spreadsheet_handling.domain.artifact_manifest`.
* Supported parameters: `reports`, `output`, `output_dir`, `manifest_path`,
  `checksum`, `name`.
* `reports` accepts plain frame-name strings or annotated dicts with `frame`,
  `writer_step`, `artifact_kind`.
* Manifest frame columns: `path`, `artifact_kind`, `writer_step`,
  `source_frames`, `row_count`, `checksum`, `status`.
* `source_frames` is a list; normalized from the provisional `frame` column
  of existing writer reports as `[<frame>]`.
* Duplicate artifact paths across merged report frames produce a clear
  `ValueError`.
* Optional checksum: `"sha256"` over raw file bytes (platform-independent).
  Requires `output_dir`.
* Optional manifest file output: `.yml`/`.yaml` or `.json` format. Requires
  `output_dir`. `manifest_path` must be relative and must not escape
  `output_dir` — validated with the same path-safety rule as artifact paths.
  Written with UTF-8 encoding and LF newlines.
* Manifest record ordering is fully deterministic: sorted by `(path, writer_step)`.
* Existing writer report frames (`write_structured_yaml`, `write_key_value_resources`)
  remain in their current provisional shapes.  Convergence to the shared shape
  is handled by this step's normalization and is deferred for the writers themselves.

==== Acceptance

* Structured writers can emit report frames that conform to the shared shape.
* A manifest step can merge multiple report frames deterministically.
* Optional checksum generation is stable, content-based, and records only
  paths relative to the configured output root.
* Tests cover multiple report frames, missing report frames, duplicate paths,
  path traversal rejection, deterministic ordering, and YAML/JSON manifest
  output.
* The manifest remains an execution artifact, not a hidden dependency on a
  consumer repository.

==== Closure Hygiene Note

Reported artifact paths are validated for portability even when no `output_dir`
is configured. Absolute paths and `..` path segments are rejected before the
manifest frame is produced, and accepted path separators are normalized to
stable relative manifest paths. When `output_dir` is configured, the stronger
root-containment validation still applies before checksum or manifest-file
output.

==== ODS disposition

*Not applicable.* The manifest is an execution-time YAML/JSON traceability
artifact describing generated files; it is not a spreadsheet workbook and does
not pass through the `WorkbookIR` / `RenderPlan` / backend path. There is no
XLSX-vs-ODS behavior to reconcile. Tracked by
`FTR-ODS-CONSUMER-FEATURE-PARITY-P4A`.
