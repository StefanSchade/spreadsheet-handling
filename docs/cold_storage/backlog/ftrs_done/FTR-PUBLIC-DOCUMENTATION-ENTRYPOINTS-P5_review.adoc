== Residual risks

* **Frozen-context drift on PyPI.** The README rewritten in this slice
  ships to PyPI on the next tag with `/latest/...` deep links. Users
  who install an older PyPI release later will follow those links to
  *today's* latest documentation, not to the documentation of the
  release they installed. This is the gap
  `FTR-RELEASE-README-VERSION-BAKING-P5` (DRAFT) will eventually close,
  either via tooling or via a manual release-checklist step. The
  *Browse by version* portal mitigates this in practice: a user who
  notices the version mismatch can navigate to their installed release
  from the root. The FTR explicitly accepts this trade-off; this slice
  does not need to solve it.

* **Per-version list maintenance is manual.** The *Browse by version*
  table on the Pages root is hand-edited. A releaser who forgets to
  add a new row will leave new releases discoverable only by URL
  guessing. Mitigation landed in this slice: the *Core Release
  Checklist* bullet in `02_release_runbook.adoc` calls out the
  required edit. If drift recurs, the FTR
  `Implementation notes` block sketches an asciidoc-attribute /
  `git tag -l` automation path that can be added later without
  touching the root template.

* **Demo tutorial assumes a sibling core checkout.** Step 2 prefers
  `make setup-lib-local`, with `make setup-lib-pypi` mentioned as the
  fallback for users without one. This matches the *prototype
  consumer* posture of the demo repo but is slightly user-hostile for
  a pure PyPI user. Acceptable for this slice; the trade-off is
  intentional and documented inline.

* **Living-context labels go stale silently.** If a future release
  introduces a breaking change to the workbook layout, the
  *latest release* user-guide link will quietly point at the new
  behavior even though the README on a still-installed older release
  said *latest release*. Same root cause as the frozen-context point;
  same eventual mitigation.

* **Tutorial step 8 uses `diff -ru` and assumes the JSON layout is
  stable.** If the canonical-JSON serializer ever reformats output
  (key order, indent depth), the diff in step 8 will show noise the
  tutorial does not predict. Today both pipelines emit canonical-JSON
  in a stable shape, so this is theoretical; if it ever fires, the
  tutorial wording will need a small adjustment ("expect to see only
  your edited field plus serializer-formatting noise").

=== Resolved finding

*Severity: Medium — demo README verification step overpromised clean
reimport output.*

The README claimed that `diff -ru ./data/product_business_slice
./tmp/product_business_slice_reimported | head -40` would show exactly
the edited business field and that helper columns would not leak into
canonical JSON. Running the documented product-business-slice flow
showed extra `_product_manager_name` / `_branch_name` helper fields plus
the expected `_meta.yaml` / `overrides.yaml` sidecar difference.

Resolution landed in demo commit `f9473f1`: `pipelines/demo_workbook_reimport.yaml`
now runs `remove_fk_helpers`, the README diff excludes the expected
sidecar/config files, and `tests/test_roundtrip_editing_demo.py` asserts
that the helper fields are absent after reimport. After the fix, the
documented diff shows only the edited `product.status` field.
