== Findings

=== High: `validate_fk_helpers` silently accepts missing v2 policy / provenance

`src/spreadsheet_handling/domain/validations/fk_helpers.py` documents and
implements a no-op FK-specific validation path when neither v2 policy nor
derived provenance is available. Lines 6-8 state that FK-specific checks emit
no findings without policy/provenance, and `_expected_helper_columns_by_sheet`
uses `resolve_v2_fk_relations(frames) or []`, which collapses absent policy
into an empty relation set. `validate_fk_helpers` then returns only duplicate-id
findings. The behavior is also pinned by
`tests/unit/domain/validations/test_fk_check.py`, where
`test_no_policy_no_provenance_returns_no_fk_findings` expects no FK findings
for a workbook that still has convention-shaped FK/helper columns.

This conflicts with the reviewed FTR and the updated docs. The FTR scope says
`validate_fk_helpers` should check relation policy and materialized helper
values rather than naming conventions, and the commit body says declared
policy is required for add/reorder/validate paths. The updated technical docs
also state that `add_fk_helpers`, `remove_fk_helpers`, `validate_fk_helpers`,
and `reorder_fk_helpers` raise a clear missing-policy error naming
`configure_fk_helpers` / `infer_fk_relations`.

Recommended fix: make the `validate_fk_helpers` primitive require v2 policy
or derived helper provenance before FK-specific validation, raising the same
actionable `MissingFkRelationPolicyError` used by add/remove/reorder. Keep
`check_duplicate_ids` independently usable if the generic duplicate-id check
needs to remain policy-free.

=== Minor: derived provenance registry/docs are still partly stale

The `derived` entry in `meta_registry.yaml` lists consumers as rendering base
helpers, `drop_helpers`, and `apply_derived_column_policy`, but after this
implementation `domain.transformations.helpers.reorder_helpers_next_to_fk` and
`domain.validations.fk_helpers.validate_fk_helpers` also consume
`_meta.derived.sheets.*.helper_columns`. In addition, the short agent notes
describe derived helper provenance as `{column, fk_column, target, value_field}`
without `target_key`, even though `target_key` is now required by the new
producer contract.

Recommended fix: update the derived metadata consumer list and short
provenance shape summaries so the registry/docs match the post-FTR runtime
contract.

== Notes On Review Focus

* `add_fk_helpers` is policy-driven. It reads only
  `_meta.helper_policies.fk.relations` with `schema_version: 2`, builds
  deterministic `FKDef` rows from relation entries, and writes provenance.
* `remove_fk_helpers` is provenance/policy-driven. Prefix fallback is deleted
  from normal cleanup; the `prefix` parameter is retained only for binding
  compatibility.
* `reorder_fk_helpers` is provenance/policy-driven. It pairs FK/helper columns
  from derived provenance first, then v2 policy when provenance is absent.
* `validate_fk_helpers` is policy/provenance-driven when metadata is present,
  but does not currently enforce the missing-policy error required by the
  FTR/docs.
* No forbidden primitive imports of `detect_fk_columns`, `FK_PATTERN`, or
  `build_registry` were found in the reviewed execution paths.
* `configure_fk_helpers` and `infer_fk_relations` remain the configuration
  producers. The `configure_fk_helpers` frame scan for the configured literal
  FK column is producer-side v1 -> v2 bridge logic, not primitive inference.
* The broad old `unexpected_helper` prefix heuristic was intentionally removed.
  This is acceptable under the FTR, but means arbitrary orphan underscore
  columns are not reported unless their name is declared for another sheet.
* No demo/docs realignment, release machinery, or unrelated runtime surface
  was touched by the implementation commit.

== Residual Risks

* The architecture guard is primarily import/textual/reference based. It is
  useful, and it catches the intended regression class, but it cannot prove
  all future helper pairing logic is semantically non-heuristic.
* Removing broad prefix-based `unexpected_helper` detection is aligned with
  the FTR, but users with undeclared stale helper-looking columns will no
  longer receive that specific warning.
* The producer-side v1 -> v2 bridge in `configure_fk_helpers` remains active.
  It is documented as transitional; a later cleanup should remove the residual
  v1 write when authorized.
