== Verdict

NOT READY.

The implementation lands the basic `infer_fk_relations` step shape and the
happy-path tests pass, but the merge semantics and transition documentation do
not yet match the FTR/model contract. The largest runtime issue is that
same-producer writes to an existing `(source_frame, source_column)` relation
key silently overwrite divergent relation content even though the registry and
review criteria require idempotent same-producer writes only.

== Blocking Findings

. Same-producer divergent relation writes are silently accepted.
+
`src/spreadsheet_handling/domain/fk_relations.py` merges by relation key and
only rejects cross-producer conflicts. When the prior and new relations have
the same `produced_by.step`, line-level logic overwrites the existing relation
without comparing `target_frame`, `target_key`, helper fields, helper columns,
`helper_prefix`, or `produced_by.mode`. The docstring says same-producer
rewrites are idempotent, and `meta_registry.yaml` says the same producing step
may overwrite its own prior entry idempotently, but the implementation accepts
non-identical overwrites.
+
Required fix: if a prior relation exists for the same
`(source_frame, source_column)`, accept only semantic equality. If anything
meaningful differs, raise a clear conflict error naming the relation key and
producer. Add a test that re-runs the same producer with changed
`helper_prefix`, `target_label_fields`, `target_key`, or equivalent and
expects failure.

. The v1-v2 bridge in `configure_fk_helpers` is not authorized or documented
  clearly enough.
+
`configure_fk_helpers` builds v2 entries by taking the resolved v1
`fk_column` and scanning all current data frames for that literal column. This
can write v2 relation entries without explicit `source_frame` /
`source_column` configuration. That may be acceptable as a deterministic
transitional normalization of current v1 policy, but the active FTR currently
says `configure_fk_helpers` writes v2 from user-supplied configuration only
and removes `targets: "auto"` together with related `auto:` parameter shapes.
It does not describe this source-frame discovery bridge or its risks.
+
Required fix: either make explicit v2 source identity part of
`configure_fk_helpers` configuration, or document this as a transitional
compatibility bridge owned by this FTR, with clear handoff to
`FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5`. Add tests for no matching source
frame, multiple source frames with the same configured FK column, and the
bridge coexisting with existing v1 primitive behavior.

. Concept and agent-facing docs remain stale after v2 becomes runtime-produced.
+
`docs/technical_model/ch04_concepts/fk_helper_metadata.adoc` still says the v2
shape is "not current runtime behavior" and is "not produced", and its
current-state section still says `configure_fk_helpers` writes v1 including
`targets: "auto"` until both later FTRs ship. `docs/ai_info/concepts_fk_helpers.adoc`
also says v2 is not current runtime behavior and names `infer_fk_relations` as
planned. That was true for the closed model-only FTR, but this implementation
now produces v2 from `infer_fk_relations` and from `configure_fk_helpers`,
while primitives still consume v1.
+
Required fix: update the technical concept doc, AI notes, and active FTR to
state the actual transition: v2 is now produced by configuration steps,
current FK-helper primitives still consume v1, and v1+v2 coexistence is
temporary until `FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5`.

. Registry hybrid-contract wording is too close to a permanent dual authority.
+
`meta_registry.yaml` describes the `fk` namespace as keyed by FK target and as
"additionally" carrying the v2 relation view, but does not clearly call this a
transitional compatibility state. `pipeline_step_registry.json` similarly
lists both v1 and v2 writes for `configure_fk_helpers` without stating that
v1 is retained only for current primitives and should narrow after the
primitive refactor.
+
Required fix: make the current-state transition explicit in both registries:
v1 exists for current primitive consumers, v2 is the target producer shape,
and the primitive FTR owns removing or narrowing the bridge.

== Non-Blocking Findings

. `_try_build_inferred_relation` has branch complexity 13. Given the current
  shape this is reviewable, but if the blocking merge/bridge fixes add more
  branches, split target matching, target-key resolution, and helper-column
  construction into smaller helpers before commit.

. `helper_policies.py` imports the private `_build_v2_relation` helper from
  `fk_relations.py`. This works, but it creates hidden coupling between the
  explicit policy path and the inference module. Prefer a public relation
  builder/normalizer if both producers own the same shape.

. Tests cover idempotent re-infer only by asserting the relation count remains
  one. They do not assert that the retained relation is unchanged, nor do they
  cover same-producer divergent writes.

== Questions / Risks for Developer Decision

* Is scanning all frames for the configured v1 `fk_column` an approved
  transitional bridge for `configure_fk_helpers`, or should explicit v2
  relation identity require source-frame/source-column configuration?
* Should the `auto` parameter remain on `configure_fk_helpers` as a defaults
  bag despite the FTR wording that says related `auto:` parameter shapes are
  removed? If yes, the FTR and registry should say exactly which auto-derived
  defaults remain and why they are not heuristic relation inference.
* The technical and AI docs need a transition-state vocabulary: "v2 produced,
  v1 consumed by current primitives" rather than "v2 planned / not current".
