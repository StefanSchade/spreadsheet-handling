= Analysis: discriminator_split.py package-split candidacy

Date: 2026-05-18
Kind: Analysis only -- no code/test/registry/doc changes made.
Pattern reference: FTR-DOMAIN-TRANSFORMATION-MODULE-SPLIT-FK-HELPERS-P5
(behavior-preserving package split, accepted with one doc follow-up).

== Summary recommendation

*PROCEED WITH CONSTRAINTS.*

A behavior-preserving package split of
`src/spreadsheet_handling/domain/transformations/discriminator_split.py`
(563 lines, 25 functions, ~10 responsibilities) is *mechanically achievable*:
the public surface is two functions, the registry uses `module:function`
colon-target resolution, and (unlike FK helpers) no test asserts internal
dotted symbol paths -- so a package `__init__` re-export preserves every
runtime/registry/doc contract with zero edits outside the package.

It is, however, *not as cleanly separable as FK helpers*. `split_by_discriminator`
and `merge_by_discriminator` are bijective inverses (`merge(split(df)) == df`)
that share a thick web of value-normalization, pattern/naming, frame-validation
and metadata helpers. The natural cut produces one small, domain-specific
shared core (`values.py`) imported by split, merge and naming. That is
acceptable as the FK-FTR's explicitly-allowed "tiny helper must remain local"
case, but it must be a single narrow module -- not a broad shared utility
package -- and the split must *not* introduce any conflict/precedence seam,
because that design space is owned by
`FTR-WORKBOOK-VIEW-PAYLOAD-CONFLICT-PRECEDENCE-P6`.

Verdict rationale: not `DEFER` (a verbatim move requires no semantic redesign);
not `DO NOT SPLIT` (the file is genuinely large and multi-responsibility);
`PROCEED WITH CONSTRAINTS` because the shared symmetry core raises the bar on
the implementation FTR's behavior-preservation guards relative to FK helpers.

== C. Candidate package shape (derived from the code, not the suggestion)

[source,text]
----
src/spreadsheet_handling/domain/transformations/discriminator_split/
  __init__.py     # re-export: split_by_discriminator, merge_by_discriminator
  split.py        # split_by_discriminator, _split_value_entries
  merge.py        # merge_by_discriminator, _merge_value_entries,
                  #   _meta_value_entries, _source_frame_list,
                  #   _uniform_source_columns, _output_column_order,
                  #   _sorted_entries_for_merge, _merge_sort_key,
                  #   _row_order_positions
  naming.py       # _ensure_pattern, _pattern_parts, _try_value_from_frame_name,
                  #   _value_from_frame_name, _safe_value_segment,
                  #   _valid_frame_name, _value_map_entries,
                  #   _mapped_frame_name, _frame_for_value
  values.py       # _valid_discriminator_value, _plain_value, _is_empty_cell,
                  #   _values_equal, _sort_token  (small shared domain core)
  framecheck.py   # _require_frame, _ensure_column,
                  #   _ensure_source_frames_exist,
                  #   _ensure_no_existing_target_frames,
                  #   _ensure_unique_target_frames
  metadata.py     # _META_KEY, _split_meta, _write_split_meta
----

Notes / deviations from the suggested shape:

* Suggested `validation.py` is intentionally renamed `framecheck.py` and scoped
  to *structural* frame checks only. There is deliberately *no* `conflict.py`
  or `precedence.py`: the current code performs no conflict reconciliation, and
  that design space belongs to `FTR-WORKBOOK-VIEW-PAYLOAD-CONFLICT-PRECEDENCE-P6`.
  Creating such a module now would pre-empt P6 -- forbidden.
* `values.py` is the single allowed shared core (5 tiny domain-specific
  functions), imported by `split`, `merge`, `naming`. This is the FK-FTR's
  explicitly-permitted "a tiny helper must remain local to the package" case --
  it is *not* a broad cross-domain utility package.
* Re-exported names: only `split_by_discriminator`, `merge_by_discriminator`
  (the public/registry/meta surface). No private re-export is required (FK
  needed `_write_helper_provenance` re-exported only because a test asserted
  its dotted path; no such constraint exists here).

Dependency DAG (must be acyclic):
`split -> {naming, values, framecheck, metadata}`,
`merge -> {naming, values, framecheck, metadata}`,
`naming -> values`; `values`, `framecheck`, `metadata` are leaves.

== D. Risk assessment

[cols="1,1,3",options="header"]
|===
| Concern | Level | Notes

| Split/merge shared invariants
| MEDIUM
| `merge(split(df)) == df` relies on consistent `_plain_value`,
  `_values_equal`, `_sort_token` and the metadata payload shape across both
  sides. A verbatim move preserves this; the risk is *future* edits silently
  breaking symmetry once the helpers live in a shared module. Mitigation:
  AST-equivalence guard + keep the existing roundtrip test.

| Implicit ordering semantics
| MEDIUM
| `_merge_sort_key` + `_row_order_positions` + `_sorted_entries_for_merge` +
  `_sort_token` jointly implement row-order restoration. Keep the first three
  co-located in `merge.py`; only `_sort_token` lives in `values.py` (also used
  by split-side dedup ordering). Do not scatter the ordering trio.

| Duplicate / conflict semantics
| MEDIUM
| Current behavior: duplicate target names and frame collisions are *errors*;
  merge does *not* reconcile differing values for the same tuple. The original
  `FTR-SPLIT-BY-DISCRIMINATOR-P4A` explicitly defers that to a "future explicit
  conflict policy". The split must preserve this exact (non-)behavior and add
  no conflict module.

| Overlap with FTR-WORKBOOK-VIEW-PAYLOAD-CONFLICT-PRECEDENCE-P6
| LOW-MEDIUM
| P6 targets *workbook-view recomposition* payload conflicts (a different
  execution path), not discriminator merge, and explicitly forbids heuristic
  resolution / generic reimport frameworks. No code coupling exists today. Risk
  is purely *anticipatory naming*: a premature `conflict/precedence` package
  boundary here would collide with P6's design. Mitigation: hard-forbid such a
  module in the implementation FTR (captured in section C).

| Hidden coupling to reimport / recomposition
| LOW
| `compose_workbook` and `workbookir_to_frames` are registered *consumers of
  the meta key string* `split_by_discriminator`, not importers of the module.
  Preserving `_META_KEY` and the payload dict shape (kept inline in the two
  orchestrators) fully decouples them. No import-graph coupling.

| Metadata lifecycle assumptions
| MEDIUM
| Unlike FK `derived` (transient), `split_by_discriminator` meta is
  `canonical_meta` / persisted / roundtrip-relevant and profile-locked by
  `test_meta_registry_hardening.py`. A verbatim move does not change layer,
  classification, key, or payload -- but any incidental payload reshaping would
  be a behavior change *and* a hardening-test failure. Keep payload literals in
  `split.py`/`merge.py`; `metadata.py` only does the read/write plumbing
  (`_split_meta`, `_write_split_meta`, `_META_KEY`).

| Stable re-exports preserve registry/doc paths
| LOW (resolved)
| Confirmed: colon-target registry resolution + dotted meta-registry references
  resolve through a package `__init__` re-export with zero external edits, and
  no test pins internal submodule paths. Strictly *less* doc-coupling than the
  FK split.
|===

Net: every individual risk is MEDIUM-or-lower and mitigable by a strict
verbatim move plus guards. None forces semantic redesign -> not DEFER. The
aggregate (bijective coupling + canonical/persisted metadata + adjacent P6
design space) is materially higher than FK helpers -> CONSTRAINTS, not a plain
PROCEED.

== F. Proposed implementation FTR outline

ID (suggested): `FTR-DOMAIN-TRANSFORMATION-MODULE-SPLIT-DISCRIMINATOR-P5`
Kind: Behavior-preserving refactoring. Priority: P5.

Purpose: split `discriminator_split.py` into the package in section C without
changing runtime behavior, step IDs, YAML surface, the canonical
`split_by_discriminator` meta key/payload, or split/merge symmetry.

Scope:

* Verbatim function moves into `split/merge/naming/values/framecheck/metadata`.
* `__init__.py` re-exports only `split_by_discriminator`,
  `merge_by_discriminator`.
* No edits to `registry.py`, `pipeline/steps.py`, `meta_registry.yaml`,
  `pipeline_step_registry.json` (paths resolve via re-export; verify, do not
  assume).
* Add the package re-export + registry-resolution guard test.
* Doc touch-ups only if a *live* (non-historic) doc pins the old `.py` file
  path (mirror the FK follow-up that fixed
  `docs/technical_model/ch04_concepts/fk_helper_metadata.adoc`).

Hard boundaries (carry verbatim into the FTR):

* No `conflict.py` / `precedence.py` / any conflict-resolution seam -- that is
  `FTR-WORKBOOK-VIEW-PAYLOAD-CONFLICT-PRECEDENCE-P6`.
* No broad shared utility package; `values.py` is the *single* allowed local
  core.
* No change to step names, YAML keys, `_META_KEY`, metadata payload shape,
  ordering semantics, or merge semantics.
* No generic reimport/recomposition framework; no coupling to
  `compose_workbook` / `workbookir_to_frames` beyond the existing meta key.
* Demo verification only; no demo edits/commits.

Acceptance:

* Package created per section C; `__init__` re-exports the two public
  functions; AST-equivalence holds for all moved functions.
* All section E "must pass" suites green; full `make test` green; demo smoke
  green against local core.
* Registry/meta/doc paths verified still resolving; no external-to-package code
  edits (or only the FK-style live-doc pointer fix).
* Diff is predominantly move/import churn; explicit confirmation that
  split/merge symmetry and the canonical metadata payload are byte-equivalent.

Review focus:

* accidental behavior change hidden in moved bodies (AST diff + roundtrip);
* `values.py` is genuinely tiny/local and not a latent shared-utility seed;
* no anticipatory conflict/precedence boundary introduced;
* `_META_KEY` / payload shape / ordering trio unchanged and cohesive;
* registry colon-target + meta-registry dotted paths still resolve;
* `test_meta_registry_hardening.py` profile for `split_by_discriminator`
  unchanged.
