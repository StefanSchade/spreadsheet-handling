==== Purpose

Assess whether the project needs a pre-`0.3.0` FTR for explicit schema
evolution / maintenance operations over canonical frames.

This assessment deliberately excludes advanced table discovery and workbook
interpretation topics:

* named ranges as table anchors;
* multiple logical tables per sheet;
* legends printed beside tables;
* automatic table-region detection;
* visible cell typing;
* automatic schema inference from arbitrary workbook edits.

The focus is only explicit, user-declared maintenance of canonical frame
columns.

==== Executive assessment

A narrow pre-`0.3.0` FTR is justified.

The reason is not that the current roundtrip is broken. The current direction
is correct: the canonical model is the durable source of truth and workbooks
are editable projections. The gap is that intentional canonical model-shape
changes now lack a first-class workflow.

For worldbuilding and similar consumers, manual JSON edits are still possible
but are poor maintenance ergonomics once frames contain many records and
metadata references. Ad-hoc spreadsheet edits are worse: they blur ordinary
value editing with schema mutation and invite accidental drift between
canonical JSON, `_meta`, validation rules, helper policies, and workbook view
configuration.

The smallest useful FTR should not design a broad migration framework. It
should define explicit one-shot schema maintenance operations for canonical
frames:

* `add_column`
* `drop_column`
* `rename_column`
* `reorder_columns`

These should be treated as maintenance operations, not as automatic roundtrip
behavior and not as dynamic projection.

==== Evidence from current state

===== Worldbuilding has real canonical schema surface

The current worldbuilding canonical data already has non-trivial frame shapes:

* `characters`: `id`, `name`, `kind`, `core_role`, `home_place_id`, `school`,
  `notes`, plus grouped dotted fields such as `identity.age_stage`,
  `persona.crystal`, `persona.favorite_food`, `story.learning_goal`, and
  `story.learning_message`.
* `stories`: `id`, `title`, `status`, `cycle`, `volume`, `sequence_no`,
  `theme`, `summary`, `notes`.
* `groups`: `id`, `name`, `kind`, `home_place_id`, `event_id`,
  `event_significance`, `notes`.
* relation frames such as `story_cast_cells`, `story_locations`, and
  `story_groups`.

This is already beyond a toy schema. Adding a new character dossier field,
renaming a story attribute, or dropping an obsolete field is a plausible
near-term maintenance activity.

The adoption review also records that most consumer work shifted toward model
construction: identifying entities, defining relationships, designing editing
views, selecting projections, and refining workbook ergonomics. It explicitly
states that adding new attributes frequently requires projection maintenance.
That is adjacent evidence for schema-maintenance pressure.

===== Existing pipeline steps are not schema maintenance operations

The registry already contains steps that manipulate columns, but their role is
different:

* `extract_frame` can select, rename, add constants, filter, and sort while
  deriving a new output frame. Its registry entry classifies omitted columns as
  dropped by projection and states that recomposition belongs elsewhere.
* `project_by_role` reorders or retains columns by projected-column role on a
  workbook-facing surface; inbound intentionally drops display helpers.
* `apply_derived_column_policy` drops helper/derived columns from reimported
  payload frames using registered derived provenance.
* `reorder_fk_helpers` moves materialized helper columns next to FK columns for
  spreadsheet readability.

These are not intended to rewrite the canonical model shape. Reusing them as
schema-maintenance primitives would confuse projection, cleanup, and durable
model evolution.

===== Metadata references make manual edits risky

Column names appear in many explicit metadata and configuration surfaces:

* `validate_references` rules (`columns`, `target_columns`);
* `_meta.helper_policies.fk` relation entries;
* `_meta.derived.sheets.*.helper_columns` during a pipeline run;
* workbook-view helper columns and sheet mappings;
* frame lifecycle / projection metadata;
* xref, compact multiaxis, discriminator-split metadata such as row keys,
  value columns, source columns, and column order.

The existing metadata docs stress that canonical metadata is authoritative
model intent, while derived metadata describes materialized working state. A
schema operation must respect that boundary. It cannot safely be "just rename
the DataFrame column" or "just let a spreadsheet header change through."

==== Why ordinary roundtrip editing is different

Ordinary roundtrip editing means:

* the configured model shape is stable;
* users edit values in a declared projection;
* helper/display/derived columns do not silently patch canonical data;
* reimport reverses known projections and validates invariants;
* unsupported arbitrary edits are rejected, ignored, or surfaced as findings.

Schema evolution means:

* the model shape itself intentionally changes;
* canonical frame columns may be added, removed, renamed, or reordered;
* metadata and configuration references may need coordinated updates;
* the operation should be reviewable as a maintenance change, not hidden inside
  normal import/export.

This distinction should be explicit in any FTR. A schema operation is closer to
a migration/edit command over the machine-near data base than to a spreadsheet
roundtrip.

==== Why dynamic projection is different

Dynamic projection columns such as matrix columns in `story_cast_*_matrix_view`
represent *data values from an axis*, not canonical schema fields.

Example: when a new character appears, a story-cast matrix gains a new visible
character column. That column is not a new field in the canonical
`story_cast_cells` schema. It is a projected representation of rows whose
canonical shape remains:

* `story_id`
* `character_id`
* `facet`
* `value`

Schema evolution operates on fields such as `characters.persona.motto` or
`stories.theme`. Dynamic projection operates on axis members such as character
names or group names. Mixing these would recreate the advanced table-discovery
problem in another form.

==== Candidate operations

===== `add_column`

Near-term relevance: high.

Worldbuilding is likely to add new dossier/story fields as the model grows.
Adding a field manually across JSON records is easy for a small file and
tedious or error-prone as the frame grows.

Safe one-shot semantics:

* add a named column to one canonical frame;
* require the column not to exist unless an explicit idempotent mode is chosen;
* set a declared default value or empty value;
* preserve existing row order and column order except for the inserted column;
* optionally insert before/after another column;
* do not infer validations or helper policies automatically.

Metadata handling:

* no metadata is required by default;
* optional metadata changes must be explicit and reviewable;
* if the new column is immediately referenced by validation/helper/view config,
  that should be expressed as separate explicit configuration, not guessed.

Assessment: in scope for the smallest FTR.

===== `drop_column`

Near-term relevance: medium to high.

Models grow and shed fields. Dropping a field manually is easy in one file but
risky when metadata or pipelines still reference it.

Safe one-shot semantics:

* remove a named column from one canonical frame;
* fail by default if the column is referenced by known canonical metadata or
  pipeline configuration;
* support a deliberate prune mode only if the FTR defines exactly which
  metadata references are removed and which remain hard errors;
* never silently drop key columns, FK columns, discriminator columns, or columns
  used by configured validations unless explicitly allowed.

Metadata handling:

* prune or fail on validation rules referencing the column;
* prune or fail on workbook-view helper columns referencing the column;
* fail on FK policy references unless the user explicitly changes the relation;
* derived/transient metadata should generally be discarded or regenerated, not
  maintained as authoritative state.

Assessment: in scope for the smallest FTR, but with conservative fail-first
behavior.

===== `rename_column`

Near-term relevance: high.

Renaming is likely in worldbuilding because names such as
`story.learning_message`, `persona.start_aspect`, or `event_significance` are
domain-language choices that may change as the model becomes clearer.

Safe one-shot semantics:

* rename a column in one canonical frame;
* require source exists and target does not exist;
* preserve values and relative column order;
* update recognized canonical metadata references atomically, or fail if any
  known reference cannot be updated safely;
* emit a clear report of changed data and metadata references.

Metadata handling:

* update validation rule `columns` / `target_columns` where the frame matches;
* update helper policy source/fk/helper references only where the reference is
  unambiguous;
* update workbook-view helper/editability/presentation references where the
  target frame is unambiguous;
* treat derived provenance as stale working state; do not promote it to
  canonical truth.

Assessment: in scope for the smallest FTR, with strict metadata-reference
coverage and fail-fast behavior.

===== `reorder_columns`

Near-term relevance: medium.

Column order is not usually semantic, but it matters for JSON readability,
workbook ergonomics, review diffs, and stable generated workbooks.

Safe one-shot semantics:

* apply an explicit column order to one canonical frame;
* either require a complete order, or support a bounded `listed_first` /
  `listed_last` mode with all omitted columns retained in their prior order;
* preserve values and metadata semantics;
* fail on unknown or duplicate column names.

Metadata handling:

* generally no semantic metadata change;
* update explicit column-order metadata only where such metadata exists and is
  clearly about the same frame;
* do not affect dynamic matrix axis order unless the target is the canonical
  axis frame itself.

Assessment: in scope for the smallest FTR because it is low-risk and useful for
reviewability.

===== `split_column`

Near-term relevance: plausible, but not pre-`0.3.0`.

Splitting a column requires value parsing, error handling, defaults, and
possibly multi-row findings. This is more than schema shape maintenance.

Assessment: defer.

===== `merge_columns`

Near-term relevance: plausible, but not pre-`0.3.0`.

Merging requires value composition rules and conflict handling. It is data
transformation plus schema maintenance.

Assessment: defer.

===== `move_column_between_frames`

Near-term relevance: plausible, but risky.

Moving a column between frames requires join keys, cardinality rules, FK
understanding, and possibly data duplication or loss.

Assessment: defer. This belongs to model migration or normalized-schema
refactoring, not the first maintenance slice.

===== `change_fk_relation`

Near-term relevance: real, but separate.

Changing a FK relation touches `_meta.helper_policies.fk`, validation rules,
helper columns, workbook view ergonomics, and data values. It is not merely a
column operation.

Assessment: defer to an FK policy / model configuration FTR.

==== Pipeline step, CLI command, or both?

The first FTR should prefer *explicit maintenance commands* over normal
roundtrip pipeline steps.

Rationale:

* Schema evolution is usually a one-shot repository maintenance operation.
* Running it on every JSON→workbook or workbook→JSON pipeline invocation would
  be dangerous.
* It should produce a reviewable diff and a report of data and metadata
  changes.
* It should be easy to run against the canonical JSON directory before
  generating a new workbook.

The implementation shape is not decided here, but the scope direction should
say:

* CLI maintenance surface: recommended for the first user-facing slice.
* Domain-level pure operation functions: likely needed underneath so behavior
  is testable and not tied to CLI parsing.
* Pipeline steps: optional later, or allowed only in explicitly named
  maintenance pipelines. They should not be encouraged in ordinary roundtrip
  pipelines.

If the FTR chooses to expose both, the same domain operation and the same
report/finding contract must back both surfaces.

==== Metadata handling principles

Any schema-maintenance FTR should set these principles before implementation:

. *Canonical first.* Operate on canonical frames and canonical/persistent
  metadata. Do not infer from workbook layout.
. *Explicit references only.* Update only metadata references whose frame and
  field target are known. Ambiguous references fail.
. *Fail before silent drift.* If a column is referenced by validation, FK,
  lifecycle, workbook-view, projection, or presentation metadata that the
  operation does not know how to update, fail with a report.
. *Derived is not authority.* `_meta.derived` is transient operational state.
  Schema operations should generally drop, invalidate, or ignore it rather than
  mutate it as if it were canonical.
. *Report every mutation.* The operation should report frame changes and
  metadata changes separately.
. *No heuristic repair.* Do not guess renamed columns by similarity. Do not
  infer intent from spreadsheet header edits.

==== Smallest useful FTR slice

Recommended pre-`0.3.0` FTR:

* Define explicit schema maintenance operations for canonical frames:
  `add_column`, `drop_column`, `rename_column`, `reorder_columns`.
* Treat them as one-shot maintenance operations, not ordinary roundtrip
  behavior.
* Start with JSON-directory canonical data plus `_meta.yaml` sidecar because
  that is the current concrete consumer shape.
* Require dry-run / report semantics in the FTR scope, even if the exact output
  shape is decided later.
* Define how each operation handles known metadata references:
  validation rules, FK helper policies, workbook-view/sheet helper columns,
  frame lifecycle, projection metadata, and transient derived metadata.
* Require fail-fast behavior for unsupported metadata references.
* Demonstrate on a worldbuilding-shaped fixture, not necessarily by mutating
  the worldbuilding repository itself.

This is a real FTR because it changes supported user workflows and likely adds
a public CLI/YAML-adjacent surface. It should pass the public-surface gate if
the implementation later exposes commands or config.

==== Explicit non-goals

Any follow-up FTR should exclude:

* automatic schema inference from workbook edits;
* arbitrary spreadsheet table discovery;
* named ranges as table anchors;
* multiple logical tables per sheet;
* automatic detection of table regions;
* visible cell typing;
* dynamic matrix-value column handling;
* migration of matrix axis members as if they were schema fields;
* split / merge column value parsing;
* moving columns between frames;
* changing FK relations;
* generic metadata migration framework;
* meta contract versioning beyond citing the existing direction;
* long-term compatibility policy for old schema versions;
* automatic update of unknown plugin-owned metadata;
* applying schema changes during normal roundtrip without explicit user action.

==== Required tests and guards for a follow-up FTR

The assessment does not propose implementation, but a real FTR should require
these validation categories:

* Unit tests for each operation on small DataFrames:
  ** success cases;
  ** missing source column;
  ** duplicate target column;
  ** duplicate requested order;
  ** unknown ordered column;
  ** idempotent mode only if explicitly included.
* Metadata-reference tests:
  ** rename updates known validation references or fails;
  ** drop fails or prunes according to explicit mode;
  ** FK policy references are not silently broken;
  ** workbook-view helper references are updated or rejected;
  ** derived/transient metadata is not treated as canonical.
* JSON-directory integration tests:
  ** operation over a directory of canonical JSON frames;
  ** `_meta.yaml` sidecar handling;
  ** stable output order for records and keys where supported.
* Worldbuilding-shaped fixture:
  ** dotted field names such as `persona.motto` and `story.learning_goal`;
  ** relation fields such as `home_place_id`, `story_id`, `character_id`;
  ** validation and helper-policy references.
* Report/dry-run tests:
  ** dry run does not write;
  ** report lists frame changes and metadata changes;
  ** unsupported references appear as clear errors.
* Guardrail tests:
  ** operation does not read workbook layout or sheet geometry;
  ** operation is not wired into normal roundtrip pipelines by default;
  ** advanced table-discovery concepts are not introduced.

==== Recommendation

Create a narrow pre-`0.3.0` FTR for explicit canonical schema maintenance
operations.

The FTR should be small and operational. It should not become a general schema
migration framework and should not reopen workbook parsing, table discovery, or
dynamic projection.

Recommended working title for a real follow-up FTR:

____
*FTR-SCHEMA-EVOLUTION-OPERATIONS-P5* -- explicit one-shot maintenance
operations for canonical frame columns.
____

Recommended first-slice scope:

* `add_column`
* `drop_column`
* `rename_column`
* `reorder_columns`
* canonical JSON-directory frames and `_meta.yaml`
* dry-run/report semantics
* conservative metadata-reference handling

Architectural risk: medium.

The risk is not the DataFrame column operation itself. The risk is metadata
drift and accidental conflation with ordinary roundtrip editing. A narrow FTR
with fail-fast metadata handling and CLI-first maintenance posture keeps that
risk acceptable before `0.3.0`.
