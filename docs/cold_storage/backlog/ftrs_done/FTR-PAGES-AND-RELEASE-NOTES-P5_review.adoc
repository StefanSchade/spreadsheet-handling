===== FTR-PAGES-AND-RELEASE-NOTES-P5 -- Implementation review

*Status:* Resolved (review findings addressed in the closing commit)

==== Scope of the review

Reviewed the first Pages publishing slice end-to-end: the `publish-docs` job
in `.github/workflows/release.yml`, `scripts/build_docs_pages.sh`, the new
`docs/pages/` and `docs/release_notes/` sources, README and AsciiDoc attribute
updates, FTR text changes, and the Developer Guide chapter 9 additions.

The review focused on architectural and release-safety risks rather than
prose or style.

==== Findings and resolutions

[cols="1,3,3"]
|===
| Severity | Finding | Resolution

| High
| Concurrent tag pushes could both clone `spreadsheet-handling-pages` at the
  same `HEAD` and race the final `git push origin main`. One
  `versions/<tag>/core/` snapshot would be silently lost on non-fast-forward
  rejection; the workflow had no retry.
| `publish-docs` now declares
  `concurrency: { group: pages-publish, cancel-in-progress: false }`. The
  publish step also runs a bounded retry loop that calls
  `git pull --rebase origin main` between attempts, so a concurrent manual
  edit on the Pages repo cannot drop a release snapshot either.

| High
| README/PyPI link and the `:pages-url:` attribute pointed at the bare Pages
  root, which the workflow never populated. PyPI keeps old README rendering
  online forever, so every release page would link to a 404.
| Added `docs/pages/root_index.adoc` as a stable navigation page (not a
  redirect). The build pipeline renders it to `build/pages-root/index.html`
  and the workflow copies it to the Pages repo root on every release. The
  page links to `/latest/core/` and to `/versions/<tag>/core/` for the
  release that generated it.

| High
| `git tag -l 'v*' --sort=-v:refname` does not understand PEP 440
  pre-release suffixes. Once a GA `v0.1.0` ships alongside `v0.1.0b6`, the
  guard would treat the beta as newer and pin `latest/core/` to the wrong
  version.
| The `latest_tag` is computed by an inline `python3 - <<'PY'` block that
  parses each `v*` tag with `packaging.version.Version` and selects the max.
  PEP 440 sorts `v0.1.0b1 < v0.1.0b6 < v0.1.0b10 < v0.1.0` correctly.
  Invalid tags are skipped instead of aborting the sort.

| Medium
| `ssh-keyscan github.com` is trust-on-first-use against whatever the
  runner's network path returns. A successful MITM at that moment would be
  silently trusted for the rest of the job.
| `known_hosts` is now populated with GitHub's three published SSH host keys
  (ed25519, ecdsa-sha2-nistp256, ssh-rsa). `GIT_SSH_COMMAND` sets
  `StrictHostKeyChecking=yes` and pins `UserKnownHostsFile`. A future
  GitHub host-key rotation requires a one-line workflow update.

| Medium
| The workflow's run blocks relied on the default `bash -e`. Under unusual
  triggers an unset `GITHUB_REF_NAME` would have expanded `versions/` to a
  literal path and `rm -rf "$pages_repo/versions/core"` would have deleted
  a path that no publisher owns.
| Every shell block in `publish-docs` and `scripts/build_docs_pages.sh` runs
  under `set -euo pipefail`. Unset variables now abort immediately.

| Medium
| The workflow assumed the Pages repo's default branch was `main`, but that
  was nowhere documented for a fresh maintainer.
| Developer Guide chapter 9 documents Pages repo / branch / publishing
  folder / deploy credential explicitly. Runbook reflects the same.

| Low
| `docs/pages/index.adoc` and `docs/release_notes/release_notes.adoc` rely on
  Asciidoctor CLI `-a` attributes provided by `build_docs_pages.sh`. A
  contributor running `asciidoctor` directly on these files would see
  unresolved placeholders.
| Both files now carry a comment block documenting that header attributes
  come from the build script via CLI `-a` and that direct invocation will
  leave them unresolved.

| Low
| FTR text mentioned a per-version `versions/<tag>/index.html` umbrella
  page, but the workflow did not (and intentionally does not) produce one.
| FTR delivery-scope and Developer Guide both call out the umbrella index
  as deferred cross-publisher work. The current slice owns root navigation
  only.

| Low
| Documented dependency list was thin and did not cover what a fresh local
  VM actually needs.
| Chapter 9 adds a "Local Toolchain Reference" table grouped by purpose
  (base dev, Python/build, spreadsheet/office, documentation build,
  AI-agent convenience, VM/operator convenience). Reveal.js is explicitly
  flagged as deferred to a later demo/presentation slice.
|===

==== Boundaries honored

* No generated HTML committed to `spreadsheet-handling` source branches;
  `build/pages/` and `build/pages-root/` are under the gitignored `build/`
  tree.
* No publication of demo or presentation subtrees; this slice writes only
  `versions/<tag>/core/`, `latest/core/`, and the root `index.html`.
* No changes to PyPI publishing logic, package metadata, or release tagging
  convention beyond the documentation passes.

==== Verdict

All review findings resolved within the same FTR scope. Closing the FTR with
the follow-up commit.
