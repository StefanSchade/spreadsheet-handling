== Architectural assessment

* The package split is coherent and matches the intended responsibilities:
  `split.py` keeps split orchestration and split metadata payload writing,
  `merge.py` keeps merge orchestration plus ordering restoration,
  `naming.py` owns pattern/frame-name derivation,
  `values.py` is the narrow shared value core,
  `framecheck.py` is structural-only,
  `metadata.py` owns only `_META_KEY` and meta read/write plumbing.
* The actual import DAG is acyclic:
  `split -> {framecheck, metadata, naming, values}`,
  `merge -> {framecheck, metadata, naming, values}`,
  `naming -> {framecheck, values}`,
  `values/framecheck/metadata -> leaves`,
  `__init__ -> {split, merge}`.
* The recorded `naming -> framecheck` deviation is acceptable. In
  [src/spreadsheet_handling/domain/transformations/discriminator_split/naming.py](/home/stefan/repos/spreadsheet-handling/src/spreadsheet_handling/domain/transformations/discriminator_split/naming.py:20),
  `_value_map_entries` still calls the structural duplicate-name check
  `_ensure_unique_target_frames`; that is a narrow, one-way dependency and not
  evidence of hidden redesign.
* `values.py` remains a tiny local core rather than a shared-utility seed:
  five discriminator-specific helpers only, no cross-package exports, no new
  generic abstraction.
* `framecheck.py` stays structural-only
  ([framecheck.py](/home/stefan/repos/spreadsheet-handling/src/spreadsheet_handling/domain/transformations/discriminator_split/framecheck.py:19));
  no conflict/precedence seam, no recomposition framework, no unrelated
  transformation-family churn.
