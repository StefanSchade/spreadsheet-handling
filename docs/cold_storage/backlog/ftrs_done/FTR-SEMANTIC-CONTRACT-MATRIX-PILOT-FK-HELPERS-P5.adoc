===== FTR-SEMANTIC-CONTRACT-MATRIX-PILOT-FK-HELPERS-P5 -- Pilot the contract matrix on FK helpers

*Status:* New
*Priority:* P5
*Area:* Semantic model / FK helpers

==== Goal

Use FK helpers as the first transformation-family slice. Produce a
concrete contract matrix for the FK-helper primitives that aligns
registry entries, technical / agent documentation, tests, and runtime
behavior on the same semantic contract. Add or refine guards only where
the contract is already stable.

==== Scope

* Author the Contract Matrices chapter in
  `docs/semantic_model/semantic_model.adoc` and a per-primitive matrix
  for `add_fk_helpers`, `remove_fk_helpers`, `reorder_fk_helpers`, and
  `validate_fk_helpers`. Matrix dimensions follow the seed: reads
  metadata, writes metadata, reads configuration, writes canonical /
  derived data, may-infer / must-not-infer, missing-metadata behavior,
  persistence behavior, provenance requirements, public-surface impact,
  lifecycle stage, tests / guards.
* Cross-check the matrix against:
  ** `registries/meta_registry.yaml` and
     `registries/pipeline_step_registry.json` entries for FK helpers;
  ** `docs/ai_info/concepts_fk_helpers.adoc`;
  ** the FK-helper primitive sources;
  ** the existing FK-helper architecture guard
     (`tests/architecture/.../test_architecture_guard_fk_helpers.py`).
* Where the matrix exposes a divergence, prefer documentation /
  registry tightening. Add or refine a guard only when the contract is
  already stable and the divergence would cause silent drift.

==== Out of scope

* No expansion of the FK-helper policy schema (owned by Phase 3a).
* No lifecycle policy for `_meta.derived.sheets.*.helper_columns`
  (owned by `FTR-DERIVED-PROVENANCE-LIFECYCLE-P4A`).
* No generalization of the matrix to other transformation families in
  this FTR.
* No new runtime behavior for FK-helper primitives.

==== Seed sections elaborated

* Contract Matrices (general purpose + FK-helper instance)
* Semantic Guards (FK-helper pilot portion only)
* Roundtrip Walkthroughs (only the FK-helper aspects needed to support
  the matrix)

==== Acceptance (rough)

* A per-primitive FK-helper contract matrix is recorded in the semantic
  model document and is consistent with `registries/`, the agent-facing
  FK-helper notes, and current runtime behavior.
* Any divergence found during cross-check is resolved by tightening
  documentation, registry entries, or -- only if the contract is
  stable -- by adjusting an existing guard or adding a narrowly scoped
  one. Each such change cites the matrix row that motivates it.
* Tests that touch FK helpers continue to pass; no FK-helper runtime
  source files change as part of this FTR.
* The matrix calls out which rows are stable enough for guards and
  which remain documentation-only.

==== Code and guards

* Production code changes are not expected.
* Test or guard refinement is permitted only on the FK-helper
  architecture guard family and only where the contract is already
  stable. Adding new runtime behavior is out of scope; if a matrix row
  would require it, file a follow-up FTR instead.
