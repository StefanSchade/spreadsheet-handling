== Blocking Findings

=== B1: ODS text orientation is rendered on the wrong style element

Priority:: Blocking
Disposition:: Must fix before treating ODS text orientation as supported.

Observation::
The ODS renderer writes `style:rotation-angle` onto a `TextProperties`
element.

Evidence::

* `src/spreadsheet_handling/io_backends/ods/odf_renderer.py`, function
  `_register_cell_style`, lines around 145-150:
+
[source,python]
----
text_props = style.getElementsByType(TextProperties)
if text_props:
    text_props[0].attributes[(STYLENS, "rotation-angle")] = str(ods_rotation)
else:
    t = TextProperties()
    t.attributes[(STYLENS, "rotation-angle")] = str(ods_rotation)
    style.addElement(t)
----
* A local odfpy schema check shows `style:rotation-angle` is allowed on
  `TableCellProperties`, not `TextProperties`. `TextProperties` exposes
  `style:text-rotation-angle` / `style:text-rotation-scale` instead.
* A generated ODS file from this renderer contains:
+
[source,xml]
----
<style:text-properties style:rotation-angle="90"/>
----

Risk if ignored::
Real ODS consumers may ignore the rendered rotation. The project parser can
still read its own generated XML, so the current roundtrip tests can pass while
the workbook is wrong or ineffective in LibreOffice/Calc-compatible consumers.

Recommended action::
Render cell text orientation via `TableCellProperties` using the carrier shape
expected by ODS. Add an ODS carrier-structure test that opens `content.xml` and
asserts the rotation attribute is on the correct style element, not only that
the project parser can read the file back.

=== B2: New ODS metadata extractors bypass repeated-structure parser limits

Priority:: Blocking
Disposition:: Must fix before accepting ODS column-width or text-orientation
metadata extraction as robust.

Observation::
The normal ODS grid parser has a `ParserLimits` defense and avoids materializing
empty repeated cells. The new presentation-metadata extraction functions
iterate repeated columns and repeated rotated cells independently and do not
enforce those limits.

Evidence::

* `src/spreadsheet_handling/io_backends/ods/odf_parser.py`,
  `_extract_ods_column_widths`, lines around 166-176:
+
[source,python]
----
repeat = int(child.attributes.get((TABLENS, "number-columns-repeated"), 1))
...
for i in range(repeat):
    letter = _column_letters(col_index + i)
    result[letter] = {"width": excel_width, "source": "workbook"}
----
* `src/spreadsheet_handling/io_backends/ods/odf_parser.py`,
  `_extract_ods_text_orientations`, lines around 216-233:
+
[source,python]
----
row_repeat = int(child.attributes.get((TABLENS, "number-rows-repeated"), 1))
...
for r_off in range(row_repeat):
    for c_off in range(col_repeat):
        addr = f"{_column_letters(col_index + c_off)}{row_index + r_off}"
        result[addr] = {"rotation": xlsx_rotation, "source": "workbook"}
----
* Existing repeated-cell safety tests live in
  `tests/unit/io_backends/ods/test_ods_parser_repeated_cells.py`, but the new
  metadata extractors are not covered by equivalent pathological repeated
  style cases.
* `make source-style-guards` also flags `_extract_ods_text_orientations` for
  branch complexity, which is consistent with this logic needing tighter
  structure and limit handling.

Risk if ignored::
An ODS workbook with sheet-wide styled columns or repeated styled rotated
cells can recreate the memory/pathological behavior that the existing parser
limits were meant to prevent. This is especially risky because presentation
style fillers are a common LibreOffice pattern.

Recommended action::
Thread `ParserLimits` or an equivalent bounded extraction policy through the
ODS presentation-metadata extractors. Avoid materializing every implied
metadata address for sheet-wide style fillers. Add regression tests with
large `number-columns-repeated` and `number-rows-repeated` values on styled
columns/cells.

== Important Findings

=== I1: Render IR uses XLSX-shaped orientation semantics

Priority:: Important
Disposition:: Clarify or refactor before broadening backend support.

Observation::
`SetTextOrientation` is documented using the XLSX `textRotation` convention:
`0` horizontal, `1-90` counter-clockwise, `91-180` clockwise stored as
`90 + CW_angle`.

Evidence::

* `src/spreadsheet_handling/rendering/plan.py`, `SetTextOrientation`
  docstring.
* `registries/meta_registry.yaml`, `text_orientations` precedence notes.
* `src/spreadsheet_handling/io_backends/ods/odf_parser.py`,
  `_ods_rotation_to_xlsx`.

Risk if ignored::
The render op remains nominally backend-neutral but carries an XLSX carrier
encoding. That is tolerable as a deliberately chosen canonical metadata
contract, but it should be named as such. Otherwise future ODS or other
backend work may keep treating XLSX encoding as domain-neutral presentation
semantics.

Recommended action::
Either document the XLSX-style rotation range as the project canonical
presentation metadata encoding, or introduce a more neutral representation
such as degrees plus direction and keep XLSX conversion adapter-local.

=== I2: ODS column width support may overstate parity

Priority:: Important
Disposition:: Before-release follow-up.

Observation::
ODS column width is converted through an approximate Excel-character/cm factor
and verified mostly through project-generated ODS files.

Evidence::

* `src/spreadsheet_handling/io_backends/ods/odf_renderer.py` uses
  `_EXCEL_CHAR_TO_CM = 0.254`.
* `src/spreadsheet_handling/io_backends/ods/odf_parser.py` converts back via
  `_EXCEL_CHARS_PER_CM = 1.0 / 0.254`.
* `tests/integration/io_backends/ods/test_ods_display_metadata_roundtrip.py`
  asserts approximate values after render/parse and backend roundtrip.

Risk if ignored::
The support may create false parity with XLSX widths. It preserves the
project's own approximate conversion but does not prove faithful preservation
of widths authored by real ODS producers.

Recommended action::
Add at least one hand-authored or carrier-level ODS fixture with explicit
table-column styles, repeated columns, and known width units. Document that
ODS widths are approximate presentation metadata, not exact XLSX width parity.

=== I3: ODS tests are implementation-confirming

Priority:: Important
Disposition:: Before-release follow-up.

Observation::
The ODS tests render with the new renderer and parse with the new parser. They
do not inspect carrier XML and do not use external/hand-authored ODS carrier
shapes.

Evidence::

* `tests/integration/io_backends/ods/test_ods_display_metadata_roundtrip.py`
  uses `render_workbook(...); parse_workbook(...)` for direct ODS tests.
* Backend tests use `OdsBackend().write_multi(...)` followed by
  `OdsBackend().read_multi(...)`.

Risk if ignored::
Parser and renderer can share the same incorrect carrier assumption and still
pass tests.

Recommended action::
Add carrier-structure assertions for generated ODS and parser tests for real
ODS style shapes, including rotation on table-cell properties and repeated
column/cell cases.

== Minor Findings

=== M1: Tuple/list cell addresses are not fully validated

Priority:: Minor

Observation::
`_cell_address_to_row_col` validates string addresses but tuple/list addresses
can return non-positive row or column values.

Risk if ignored::
Invalid metadata supplied through internal or caller paths can become invalid
render operations.

Recommended action::
Apply the same `row >= 1` and `col >= 1` validation to tuple/list inputs.

=== M2: XLSX SetTextOrientation overwrites existing alignment object

Priority:: Minor

Observation::
The XLSX renderer applies `Alignment(text_rotation=op.rotation)` directly.

Risk if ignored::
If future render operations set horizontal/vertical alignment, wrapping, or
other alignment properties before `SetTextOrientation`, this operation can
discard them.

Recommended action::
Copy the existing alignment and change only text rotation when openpyxl allows
that cleanly, or document operation-order expectations.
