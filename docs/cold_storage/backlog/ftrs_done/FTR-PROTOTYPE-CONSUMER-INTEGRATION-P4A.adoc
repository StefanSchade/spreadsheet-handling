===== FTR-PROTOTYPE-CONSUMER-INTEGRATION-P4A -- Run a prototypical consumer workbook flow before beta

*Status:* Done (closed 2026-05-23)
*Priority:* P4A
*Area:* Consumer integration / beta readiness

==== Purpose

Use a separate prototypical consumer repository as the first real integration
pass before the next beta.

The goal is not to turn consumer-specific configuration into a core library
fixture. The goal is to run the current library against a concrete usage path,
discover missing general capabilities early, and keep those gaps visible before
the demo repository and beta release are polished.

The dependency direction stays one-way: the consumer depends on the local
`spreadsheet-handling` checkout for the integration pass; the core repository
does not depend on the consumer repository.

==== Scope

* Bind the consumer workflow to the local `spreadsheet-handling` checkout for
  the integration pass.
* Identify the maintained workbook flow that should represent the beta
  readiness check.
* Run the selected flow end to end where local data and permissions allow it.
* Capture any missing generic library behavior as separate FTRs in this
  backlog.
* Keep consumer-specific changes, fixtures, and generated artifacts in the
  consumer repository.

==== Minimum Feature Set

The first useful consumer gate should not wait for every P4A gap. It needs at
least:

* `FTR-GENERIC-FRAME-EXTRACTIONS-P4A`
* `FTR-STANDARD-REFERENCE-VALIDATIONS-P4A`
* `FTR-DECLARATIVE-WORKBOOK-VIEWS-P4A`

If the selected flow still depends on consumer-local joins, helper styling, or
view readback, those needs should either pull the relevant FTR forward or be
documented as the next blocker.

==== Non-goals

* No private or consumer-owned data is copied into the main library repository.
* No consumer-specific assumptions are hard-coded into generic transforms.
* No broad demo rewrite is bundled into this FTR; the demo follows after this
  consumer pass.

==== Acceptance

* The selected prototype consumer flow can run against the local library
  checkout, or the blocker is documented with a concrete next FTR.
* Any main-library behavior needed by the consumer flow is either implemented
  as a general capability or recorded as a follow-up FTR.
* Scratch outputs, snapshots, and local run artifacts are ignored or removed
  rather than committed accidentally.
* The next demo-integration pass can start from the capabilities proven or
  newly queued here.

===== Result: PARTIAL PASS

* *Binding: PASS.* The demo consumer binds cleanly to the local core checkout
  via its own `setup-lib-local` path; no tracked-file changes.
* *Maintained smoke flows: 6 of 8 pass* against the local `0.1.0b6.dev` core.
  The non-FK demo journeys (multiheader, transforms, verify-steps) work
  unchanged against the expanded core.
* *Maintained product-business-slice flow: FAIL* — `make run` and the two
  dependent smoke tests
  (`test_named_ranges_demo::test_product_business_slice_workbook_contains_named_ranges`,
  `test_roundtrip_editing_demo::test_product_roundtrip_editing_preserves_user_changes`)
  all fail from a *single* root cause:
  `KeyError: Unknown step 'apply_fks'`.
* *In-core beta-readiness proxy: PASS* (15 passed) — unchanged; the core
  capability set itself is intact.

===== Blocker / gap classification

* Root cause: the demo's checked-in `demo_product_business_slice.yaml` uses
  pre-`0.1.0b6` pipeline step IDs `apply_fks` and `reorder_helpers`. The core
  step registry renamed these during the `0.1.0b6` feature expansion; the
  capability now lives under `add_fk_helpers` / `configure_fk_helpers` /
  `reorder_fk_helpers` (verified present in `REGISTRY`).
* Classification: *consumer-specific config drift* caused by a core-side rename
  of consumer-facing pipeline step keys between the demo's `0.1.0b5` pin and
  the local `0.1.0b6.dev` checkout. This is exactly the pre-beta signal this
  gate FTR exists to surface (the demo missed the `0.1.0b6` expansion).
* *Not a missing generic spreadsheet-handling capability* — the FK-helper
  capability is present under the new step names; the rest of the expected
  pre-beta capability set is implemented per the closed/archived P4A FTRs and
  the `FTR-ODS-CONSUMER-FEATURE-PARITY-P4A` parity matrix.
* The demo's `pyproject.toml` still pins `spreadsheet-handling==0.1.0b5`
  (Makefile `LIB_PYPI_VERSION`); the resolver warning under `setup-lib-local`
  is expected and is the same drift symptom.
* Payload conflict / precedence handling remains out of scope, deferred to
  `FTR-WORKBOOK-VIEW-PAYLOAD-CONFLICT-PRECEDENCE-P6`.
