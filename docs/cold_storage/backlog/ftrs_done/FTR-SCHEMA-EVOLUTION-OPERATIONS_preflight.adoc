===== FTR-SCHEMA-EVOLUTION-OPERATIONS -- Implementation preflight

*Status:* Preflight

*Parent FTR:* `docs/backlog/FTR-SCHEMA-EVOLUTION-OPERATIONS.adoc`

*Scope:* First implementation slice only.  This document does not implement
runtime code, add tests, update registries, or change the FTR.

==== Binding architecture summary

The implementation must follow the FTR architecture exactly:

* Domain operations are carrier-neutral and operate on `Frames`.
* Domain operations return `SchemaMaintenanceResult(frames, report)`.
* The report is not a frame and must not be stored in `_meta`.
* A dedicated maintenance CLI app builds a private `BoundStep`
  programmatically.
* The private `BoundStep` adapts `SchemaMaintenanceResult` to `Frames ->
  Frames`.
* The CLI captures the report through a local collector.
* The CLI delegates execution to `orchestrate()`.
* The private schema-maintenance step must not be registered in
  `pipeline.REGISTRY`.
* The private schema-maintenance step must not be added to
  `registries/pipeline_step_registry.json`.
* The private schema-maintenance step must not be addressable from normal
  `sheets-run` YAML.
* Dry-run should use an output-only `discard` saver or equivalent
  orchestrator-compatible sink.
* The first tested storage workflow may be `json_dir -> json_dir` for write
  mode and `json_dir -> discard` for dry-run mode.  Domain code must not
  contain JSON-specific authority assumptions.

No blocking contradiction was found in the FTR.

==== Proposed domain package/module structure

[source,text]
----
src/spreadsheet_handling/domain/schema_maintenance/
  __init__.py
  model.py          # dataclasses/enums and public domain result types
  columns.py        # DataFrame column mechanics, no metadata logic
  meta_refs.py      # scan recognized and blocked metadata references
  meta_update.py    # apply recognized update/prune rules
  operations.py     # orchestration of one operation over Frames

Dependency direction:

* `columns.py` depends only on pandas and `pipeline.types.Frames`-compatible
  typing.
* `meta_refs.py` and `meta_update.py` depend on plain mappings/lists and the
  model dataclasses.  They must not import I/O backends, CLI, pipeline
  registry, or spreadsheet/rendering modules.
* `operations.py` coordinates `columns.py`, `meta_refs.py`, and
  `meta_update.py`.
* The package must not read files, write files, inspect carrier kind, or
  branch on `json_dir`.

The CLI private step should be built in the CLI module, not in
`pipeline/steps.py`:

[source,python]
----
collector: list[SchemaMaintenanceReport] = []

def run(frames: Frames) -> Frames:
    result = apply_schema_maintenance(frames, request)
    collector.append(result.report)
    if result.report.blocked:
        raise SchemaMaintenanceBlocked(result.report)
    return result.frames

step = BoundStep(name="_schema_maintenance_private", config={}, fn=run)
orchestrate(input=inp, output=out, steps=[step])
----

The exact exception type is an implementation detail, but the CLI must avoid a
user-selected output write when blocking failures exist.  Raising from the
private step before returning to `orchestrate()` is the simplest way to prevent
the saver call.

==== Metadata reference roots to support in the first slice

Support means the scanner has structured knowledge of the root and can either
update, prune, or block with a precise report entry.  It does not mean every
reference under the root is rewritable.

===== `_meta.constraints`

Supported references:

* list entries with `sheet` and `column`;
* list entries with `sheet` and `area` only should be treated as recognized
  but not column-name references;
* `rule.from_legend.legend` or similar legend references are not column
  references and should not be rewritten.

Rename:

* update `column` when `sheet` resolves unambiguously to the target frame.

Drop:

* block by default;
* prune the constraint only under explicit `--prune` if its only affected
  column reference is the dropped column.

Reorder:

* no semantic change.

Frame resolution:

* direct match `sheet == target_frame`;
* if `_meta.workbook_view.sheet_mappings` maps the visible sheet to the target
  frame, treat it as recognized;
* otherwise block as ambiguous rather than guessing.

===== `_meta.helper_policies.fk`

Supported references:

* v2 `relations[*].source_frame`;
* v2 `relations[*].source_column`;
* v2 `relations[*].target_frame`;
* v2 `relations[*].target_key`;
* v2 `relations[*].helper_columns[*].column`;
* v2 `relations[*].helper_columns[*].target_field`;
* residual v1 per-target dicts should be detected but blocked unless the
  implementation explicitly scopes a safe rule.

Rename:

* update `source_column` only when `source_frame == target_frame`;
* update `target_key` and `target_field` only when `target_frame ==
  target_frame` and the renamed column is the target key/field;
* helper materialized column names are not schema fields unless they are
  explicitly declared as helper columns; do not fuzzy-rename helper names.

Drop:

* block by default for any relation touching the target column;
* do not prune FK relations in the first slice unless a later implementation
  decision explicitly scopes relation removal.

Reorder:

* no semantic change.

===== `_meta.sheets.<sheet>.helper_columns`

Supported references:

* durable helper-column declarations under sheet-local metadata.

Rename:

* update only when the sheet resolves unambiguously to the target frame and the
  listed column is the renamed column.

Drop:

* block by default;
* prune the helper-column entry only under explicit `--prune`.

Reorder:

* no semantic change.

===== `_meta.sheets.<sheet>.protection.editable_columns`

Supported references:

* sheet-local editability lists produced by workbook view configuration or
  caller-supplied meta.

Rename:

* update only when the sheet resolves unambiguously to the target frame.

Drop:

* block by default;
* prune only under explicit `--prune`.

Reorder:

* no semantic change.

===== `_meta.workbook_view`

Supported references:

* `workbook_view.sheets[*].frame`;
* `workbook_view.sheets[*].sheet`;
* `workbook_view.sheet_mappings[*].sheet` /
  `visible_sheet`;
* `workbook_view.sheet_mappings[*].frame` /
  `logical_frame`;
* `workbook_view.sheet_mappings[*].canonical_frame`.

Column maintenance:

* frame-name references are recognized for resolving visible sheet names to
  target frames;
* these are not column references for add/drop/rename/reorder and should be
  reported as unaffected unless malformed or ambiguous.

===== `_meta.frame_lifecycle`

Supported references:

* top-level keys as frame names;
* entry fields `derived_from` and `superseded_by` as frame-name references.

Column maintenance:

* recognized for authoritative-frame detection and for blocking missing or
  malformed lifecycle shapes;
* not a column-reference root in the first slice.

===== Explicit column-order metadata, if present

Known current roots do not expose a single stable top-level column-order
contract for authoritative frames.  If the implementation finds an explicit
column-order list under a supported root and can prove it targets the same
frame, `reorder_columns` may update it.  Otherwise report it as blocked or
unaffected according to the FTR.

==== Metadata roots that should be detected-but-blocked

Detected-but-blocked roots are known roots with structured frame/column
references, but the first slice should not rewrite them because their contracts
involve transformation semantics, axis reconstruction, codecs, or unresolved
lifecycle questions.

* `_meta.xref_crosstable`
  ** block on structured references to target frame/column in fields such as
     row keys, column key, value column, relation frame, source/target frame,
     or dense-axis configuration.
* `_meta.split_by_discriminator`
  ** block on structured references to source frame, generated frames,
     discriminator column, original column order, or merge/restore fields.
* `_meta.compact_multiaxis`
  ** block on structured row/column key, value column, relation frame, axis,
     codec, sparse, or group-projection references.
* `_meta.cell_codecs`
  ** block on structured references to output frame, value columns,
     allowed-value references, or token columns.
* `_meta.sparse_defaults`
  ** block on structured target columns, defaulted value columns, or frame
     references.
* `_meta.legend_blocks`
  ** block only if a structured frame/column reference is detected.  Do not
     string-search legend labels, display text, or codebook values.
* residual `_meta.helper_policies.fk.<target>` v1 shapes
  ** block when they structurally reference the target frame/column because
     runtime consumers no longer use v1 and a safe rewrite is not worth
     scoping in the first slice.
* plugin-owned roots
  ** block only when a structured frame/column reference is detected and no
     plugin-specific reference contract exists.  No generic string-search or
     heuristic rewrite is allowed.

==== Metadata roots explicitly out of scope

Out-of-scope roots should not be scanned heuristically and should not be
rewritten.  They may appear in the report as unaffected if discovered during
normal structured traversal of supported roots.

* presentation-only sheet-local metadata keyed by cell address or display
  setting, including `column_widths`, `text_orientations`,
  `horizontal_alignments`, `vertical_alignments`, `freeze_header`,
  `auto_filter`, `header_fill_rgb`, and `helper_fill_rgb`;
* carrier-shape or read-path hints such as `_hidden`, `workbook_meta_blob`,
  `__autofilter_ref`, and `__legend_blocks`;
* root-level `__*` keys, which are already stripped by the persistence
  boundary and are not authoritative schema intent;
* arbitrary unknown roots without structured frame/column reference contracts;
* ordinary YAML pipeline files outside the loaded `Frames` payload;
* normal `sheets-run` step specs and public pipeline registry metadata.

==== `_meta.derived` handling

`_meta.derived` is transient `meta_rendering` / `derived_operational_view`
metadata.  It must not be used as authority for add/drop/rename/reorder.

Recommended first-slice behavior:

* ignore `_meta.derived` when deciding whether an operation is allowed;
* report it as `IGNORED_DERIVED` in dry-run if present;
* for write mode, rely on the existing orchestrator persistence boundary to
  remove top-level `derived` before saving;
* optionally remove `derived` from the returned `Frames` before returning from
  the domain operation only if tests prove this does not obscure the
  orchestrator boundary behavior.

Do not update helper provenance under `_meta.derived.sheets.*.helper_columns`
as if it were canonical metadata.  Do not use it to discover old names, helper
relations, or target-frame identity.

==== Blocking failures and output writes

The domain operation should detect all blocking failures before returning
updated frames.  If `SchemaMaintenanceReport.blocked` is true:

* the private `BoundStep` must raise a CLI-owned or domain-owned blocking
  exception before returning frames to `orchestrate()`;
* `orchestrate()` therefore does not reach `_save_frames(...)`;
* no user-selected output write occurs after blocking failures are detected.

This satisfies the FTR's "no partial write" rule.  The first slice does not
need filesystem-transactional replacement.  Stable write behavior for
`json_dir` remains the existing backend's responsibility.

For dry-run:

* the CLI sets output to `{"kind": "discard", "path": "__discard__"}` or
  equivalent;
* the private step still executes and captures the report;
* the discard saver accepts the frames and writes nothing.

==== Risks and open decisions

* *CLI name:* `sheets-schema-maintain` is recommended but still a public
  surface decision.
* *Report encoding:* JSON is recommended for machine-readable reports.  YAML
  can be added later if needed; first slice should avoid multiple report
  formats.
* *Default value typing:* first slice should treat `--default` as a string or
  empty value.  Typed defaults would need explicit parsing rules.
* *Drop prune scope:* pruning should be limited to supported recognized roots.
  FK relation pruning should remain blocked in the first slice unless
  explicitly approved.
* *Residual v1 FK helper policy:* detect and block structured references rather
  than updating a shape runtime consumers no longer use.
* *Frame-to-sheet resolution:* constraints and sheet-local metadata require a
  clear mapping.  Use direct frame/sheet name match or
  `_meta.workbook_view.sheet_mappings`; otherwise block as ambiguous.
* *Output sink naming:* `discard` is the only recommended first-slice sink
  kind.  `null` may be considered later as an alias only if a real
  user-facing need justifies widening the I/O kind surface.
* *Filesystem write atomicity:* not required by the FTR.  If later needed for
  `json_dir`, scope it as a separate backend hardening task.
* *Metadata lifecycle gaps:* `FTR-META-LIFECYCLE-INVENTORY-P5` remains open.
  This implementation should avoid pretending the registry is complete runtime
  truth.

==== Recommended first implementation slice

Start with Slice 1 plus the smallest part of Slice 2 needed to prove the
domain shape:

* add `domain/schema_maintenance/model.py`;
* add `domain/schema_maintenance/columns.py`;
* add `domain/schema_maintenance/operations.py`;
* implement all four DataFrame column operations on in-memory `Frames`;
* return `SchemaMaintenanceResult(frames, report)`;
* include blocking failures for missing frame, missing source column, duplicate
  target column, duplicate reorder columns, and unknown reorder columns;
* include `_meta` pass-through without report storage;
* add the targeted domain unit tests in the same commit;
* do not add CLI, output backend, or registry changes in this first code
  commit unless implementing the full FTR in one review window is explicitly
  chosen.

Then add metadata-reference handling before the CLI.  The CLI should not ship
until metadata blockers exist; otherwise it would provide a maintenance command
that can silently drift metadata.
