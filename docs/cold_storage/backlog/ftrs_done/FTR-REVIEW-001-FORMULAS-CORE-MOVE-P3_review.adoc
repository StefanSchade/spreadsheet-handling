Yes. The module is pure stdlib (`dataclasses`, `typing`). No openpyxl, no odfpy,
no rendering logic, no backend syntax. The four exported types

== `missing='fail'` rejection — justified?

Yes. In formula mode, the lookup formula runs in the spreadsheet engine at
render time. There is no Python-side mechanism to check for unmatched rows
(the formula just returns `missing=""` when no row is found). Raising
`ValueError` is correct and the error message is clear:

----
missing='fail' cannot be enforced with helper_value_mode='formula';
use missing='empty' or value mode
----

The prior behavior was silent: the `missing_mode` parameter was a complete
no-op in formula mode (both branches of the dead ternary produced `""`).
The new behavior is a pre-beta breaking change from "silently accepted" to
"explicit error." See important finding below.

== Findings

=== Blocker

=== Important

*`missing="fail"` with formula mode is a behavioral breaking change not
documented in the FTR.*

The old behavior: `enrich_lookup(missing="fail", helper_value_mode="formula")`
completed without error (missing_mode was silently a no-op). The new
behavior: `ValueError` is raised.

This is the *correct* fix for D-014. However:

* The FTR does not call this a breaking change — it only says "make the two
  branches distinct." A caller who relied on the silent acceptance (intentionally
  or not) will now get a runtime error without any migration path.
* The error surfaces *late* — `_VALID_MISSING_MODES = {"fail", "empty"}`
  still includes "fail", so the top-level mode validation at line 92 passes.
  The `ValueError` only fires inside `_build_formula_enrichment`, which is
  several frames deeper. This is unintuitive: the caller sees that "fail" is
  an accepted mode string, then gets an error later.

*Recommended fix* — raise early in `enrich_lookup` when the incompatible
combination is detected, before dispatching to `_build_formula_enrichment`:

[source,python]
----
# in enrich_lookup, after resolving use_formulas and missing_mode
if use_formulas and missing_mode == "fail":
    raise ValueError(
        "missing='fail' cannot be enforced with helper_value_mode='formula'; "
        "use missing='empty' or value mode"
    )
----

Then remove the duplicate check inside `_build_formula_enrichment`. This
makes the validation visible at the entry point and removes a private function
that silently knows about public-parameter constraints.

The existing test already covers the error; only the location moves.

=== Minor

*1. Guard prefix "rendering" is slightly under-specified.*

The `forbidden_prefixes` tuple includes both `"rendering"` and
`"spreadsheet_handling.rendering"`. The short form `"rendering"` would
spuriously match a hypothetical `rendering_helper.py` file placed in `domain/`
or `core/`. In practice no such file exists and the static scanner is
import-string based (not filename based), so this is not a real issue today.
If the guard is extended in future, using only fully-qualified prefixes
(`"spreadsheet_handling.rendering"`, `"spreadsheet_handling.io_backends"`)
would be more precise.

*2. `architecture_primer.adoc` guardrails section does not reflect L1 closure.*

Line 208 still reads "Architecture tests in `tests/architecture/` enforce layer
boundaries automatically" without enumerating which edges are covered or which
remain open (L2, L3, L5 etc.). This was flagged as a category-(a) correction
in `review-consolidation.adoc` (recommending "enforce *the boundaries enumerated
in `architecture_guard_inventory.adoc`*; known uncovered edges as of 2026-05:
…"). Closing L1 makes this more urgent — the list of open gaps has shrunk by
one and the primer should reflect the current state.

This is pre-existing and not introduced by this FTR; it belongs to the
category-(a) doc-fix backlog from the review. Low immediate impact since
`architecture_guard_inventory.adoc` was correctly updated.
