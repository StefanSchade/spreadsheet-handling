*M1 -- Resolved working form omits `drop_at_persistence`.*

Evidence: in *Step: Define Legend Entries -> Information Produced*, the resolved
working form is tagged only `drop_if_stale`:

____
*Legend entry set* -- durable in its source home; the in-`_meta` resolved
working form is run-local and re-resolved each forward run (`drop_if_stale`),
not trusted from a prior `_meta`.
____

The warm-storage source's own Type 2 trace says the resolved form is *dropped*
at domain egress ("Meta projected to persistable subset: resolved form and hints
dropped"), and the FTR's central fact is that
`persistence_boundary.py::_project_legend_blocks` strips
`legend_blocks[*].resolved` at the sidecar -- i.e. `drop_at_persistence`. The
doc's own *Render Visible Legend* slice already tags the workbook reconstruction
hint `drop_at_persistence`, so the omission is internally inconsistent, not just
implementation-derived.

Risk: the functional model is silent on the exact boundary attribute the FTR
exists to classify; the later semantic/registry reconciliation has a weaker
target and may read the silence as "no persistence-boundary behaviour stated".

Smallest correction: add `drop_at_persistence` alongside `drop_if_stale` for the
resolved working form, ideally as its own *Information Produced* bullet distinct
from the durable *Legend entry set* (the entry set is durable; only its resolved
working form is run-local). No implementation derivation needed -- the source
storyline already states the drop-at-egress.

The warm-storage note explicitly resolves the synonym pair ("'Code' and 'token'
are used interchangeably; the rest of this note keeps 'token'"). The ch04 entry
uses "code(s)" as the primary prose term but "token" in the

chapter standardises on. Minor discipline slip.

Correction: one sentence in the *Terminology* section stating the chapter uses

*L2 -- "What happens if absent" under-filled in several slices.*

The `_chapter.adoc` convention and the fk_helpers precedent state the
absent-behaviour for essential inputs (e.g. "No enrichment possible without it;
step errors or is a no-op"). In *Apply Legend to a Column or Range*, *Render
Visible Legend*, and *Reimport*, inputs are marked "essential" without the short
absent-behaviour clause.

Correction: append the brief absent-behaviour phrase to the essential inputs in
those three slices.

*L3 -- Semantic Model Reference presents pieces as "proposals" though a registry
key exists.*

Each *Semantic Model Reference* block calls its conceptual pieces "proposals for
the semantic model to formalise; not field paths." A `legend_blocks` registry
entry already exists (`docs/technical_model/ch05_registries/...`), and the
sibling fk_helpers ch04 entry references concrete registry names
(`fk_relations`, `helper_policies`). Treating everything as a not-yet-existing
proposal is defensible under the stated process (derive from storyline, defer
registry reconciliation -- which is an explicit FTR scope item), but as written a
reader cannot tell the deferral is deliberate.

Correction: one clause noting that a `legend_blocks` registry entry exists and
its reconciliation is deferred to this FTR's later semantic step. Keeps the
deferral intentional rather than accidental.

*L4 -- Inherited source tension carried into the promotion (watch-only).*

*Resolve Codes -> Information Produced* lists "*Structured values* using
long-form values, labels, or configured target fields", while *Terminology*
defines the label as display-only and "not referenced by persisted data".
Producing a structured value *from* a label sits in tension with that
definition. The tension is present in the warm-storage source and is faithfully
preserved; it is not a promotion defect. Flagged for the later semantic pass.
