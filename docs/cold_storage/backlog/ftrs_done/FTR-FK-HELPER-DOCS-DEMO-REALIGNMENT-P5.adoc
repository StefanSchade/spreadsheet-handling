===== FTR-FK-HELPER-DOCS-DEMO-REALIGNMENT-P5 -- Realign demo, tutorial, and user docs with explicit FK helper policy

*Status:* New

*Purpose:*::
Update the public-facing demo, tutorial, and user documentation after FK helper
semantics move from implicit naming conventions to explicit relation policy and
configuration-step inference.

*Context:*::
Phase 2 created a clearer public entry surface: README flow, Pages root/archive
navigation, and a maintained product-business-slice tutorial path.
+
The FK-helper refactor will change how users should understand and configure
FK helper behavior. If the tutorial and user guide remain convention-oriented,
the freshly improved public surface will become misleading.
+
This FTR realigns the demo and documentation after the FK-helper runtime changes
land.

*Dependencies:*::

* `FTR-FK-RELATION-POLICY-MODEL-P5`
* `FTR-INFER-FK-RELATIONS-CONFIGURATION-STEP-P5`
* `FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5`
* Existing product-business-slice demo and walkthrough.
* Existing user guide transformation catalog.
* Existing Pages documentation build.

*Scope:*::

* Update demo pipelines to use the new FK-helper sequence.
* Decide for the maintained product-business-slice path whether to show:
  ** fully explicit FK relation policy (`configure_fk_helpers` -> `add_fk_helpers`), or
  ** `infer_fk_relations` followed by policy-driven primitives.
* Update demo README tutorial steps if command output, YAML explanation, or
  verification expectations change.
* Update walkthrough text for the product business slice.
* Update user-guide transformation catalog
  (`docs/user_guide/ch03_transformation_catalog/03_supporting_steps.adoc`):
  ** add `infer_fk_relations`,
  ** clarify `configure_fk_helpers` (explicit / manual; v2 relation-oriented
     shape; `targets: "auto"` removed),
  ** remove wording that says `add_fk_helpers` discovers helpers from FK
     conventions,
  ** remove prefix-fallback language for `remove_fk_helpers` and replace it
     with provenance / policy-driven cleanup wording,
  ** update `reorder_fk_helpers` description to use provenance / policy
     rather than name patterns.
* Update technical-model and AI-info references that still describe FK
  helper behavior as convention-driven primitive logic. List of files that
  must be updated:
  ** `docs/technical_model/ch04_concepts/fk_helper_metadata.adoc`
     -- rewrite "What is true today, and what is not" to reflect the v2
     policy shape, provenance with `target_key`, and the removed prefix
     fallback;
  ** `docs/ai_info/concepts_fk_helpers.adoc`
     -- rewrite the "Key rules" and "Where in code" sections; remove
     "prefix-based" wording in normal-path descriptions;
  ** `docs/technical_model/ch06_architectural_layers/architectural_layers.adoc`
     -- update the FK-helper feature-slice trace.
* Re-render relevant local docs and confirm the Pages / public path still
  makes sense.
* Update screenshots only if visible workbook output changes in a way that
  makes existing screenshots misleading.

*Non-Goals:*::

* No further FK-helper runtime behavior changes.
* No new public tutorial path beyond the maintained product-business-slice path.
* No new release.
* No Pages publishing workflow change.
* No README version-baking solution.
* No attempt to document all possible FK inference modes exhaustively.

*Expected demo pipeline shape (illustrative):*::

Examples use the project's actual FK column convention (`id_(<target>)`)
and the demo's actual frame names (`product`, `product_manager`).

The maintained demo pipeline should show the intended architectural split
clearly. For example:

[source,yaml]
----
pipeline:
  - step: infer_fk_relations
    mode: naming_convention
    id_columns: [id]
    fk_patterns:
      - "id_({target})"
    target_label_fields: [name]

  - step: add_fk_helpers

  - step: configure_workbook_view
    # existing workbook view configuration
----

Or, for a fully explicit tutorial path:

[source,yaml]
----
pipeline:
  - step: configure_fk_helpers
    relations:
      - source_frame: product
        source_column: id_(product_manager)
        target_frame: product_manager
        target_key: id
        helper_fields: [name]

  - step: add_fk_helpers
----

The final form should prefer the clearest user-facing story.

*Expected destination wording for `remove_fk_helpers`:*::

The user-guide catalog entry should read approximately:

[quote]
"`remove_fk_helpers` -- Remove materialized FK helper columns using the
derived provenance written by `add_fk_helpers`. Frames without provenance or
explicit policy are an error. Recovery from externally-edited workbooks
without provenance is a separate opt-in concern and is not part of this
step."

Stale wording to remove from existing docs:

* "discovers helpers from FK conventions" (for `add_fk_helpers`),
* "prefix fallback" / "prefix-based" (as the normal `remove_fk_helpers`
  path),
* "detect_fk: true" as part of the documented `add_fk_helpers` parameter
  surface,
* references to `targets: "auto"` on `configure_fk_helpers`.

(Phrases like "convention" remain acceptable when describing
`infer_fk_relations` or the cell-level `id_(<target>)` wire format -- both
are legitimate places for the convention to appear.)

*Acceptance:*::

* The maintained demo product-business-slice pipeline runs successfully with the
  new FK-helper model.
* The reimport pipeline removes helper columns without relying on prefix
  fallback.
* The demo README no longer implies that primitive FK helper steps discover
  relations by naming convention.
* The user-guide transformation catalog accurately describes:
  ** `infer_fk_relations` as configuration / heuristic,
  ** `configure_fk_helpers` as explicit / manual policy configuration writing
     the v2 `helper_policies.fk` shape,
  ** `add_fk_helpers` as policy-driven primitive materialization,
  ** `remove_fk_helpers` as provenance / policy-driven cleanup (no prefix
     fallback as normal behavior),
  ** `reorder_fk_helpers` as provenance / policy-driven reordering (no
     `FK_PATTERN` matching).
* The product-business-slice walkthrough remains executable.
* Existing screenshots are either:
  ** explicitly left unchanged after a documented local re-render against
     the new pipeline and a visual diff confirming no material change, or
  ** updated with a short review note explaining the change.
* Local Pages build succeeds.
* The public entrypoint flow remains coherent after the terminology change.
