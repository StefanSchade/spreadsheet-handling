== Per-question assessment

=== 1. Is the FTR still accurate after the actual post-0.2.0 experience?

Mostly. The *intent* is unchanged and correct. The *framing* understates the
actual urgency: the FTR was written in a "next release prep" mindset, but
the file inconsistency it predicted has already materialised. The FTR's
"Decision Direction" — release notes transition to the next development
cycle — is right; reality has not yet enacted it.

The FTR's "Candidate Post-Release Activities" list is right as far as it
goes, but incomplete given evidence:

* Missing: explicit policy for *where post-release release-note content
  should land* (the entire 0.2.0 → 0.3.0 drift is exactly this).
* Missing: ADR-based FTR closures, cold-storage movement, backlog-index
  reset — these have already happened ad-hoc during post-release work and
  the FTR should name the cadence at which a release-time sweep verifies
  them.
* Missing: an archival snapshot of release notes at tag time (this also
  ties into `FTR-RELEASE-PROCESS-AUTOMATION-P5` Pages publication).

=== 2. Does the FTR need refinement before implementation?

Yes, narrowly. The minimal refinement set:

* *Promote the released-vs-unreleased structural question* from "candidate
  activity" to "decision required." Three viable shapes:
+
[cols="1,3,3", options="header"]
|===
| Shape | Description | Trade-off
| Single file, two sections
| `release_notes.adoc` carries `== 0.X.Y (released)` per shipped line plus
  `== 0.(X+1) -- planned` at the top.
| Lowest tooling cost; matches Keep-a-Changelog convention; one canonical
  surface.
| Per-version file + planned file
| One `release_notes_v0.2.0.adoc` etc. plus a moving `release_notes.adoc`.
| Cleaner archival; needs include glue and more runbook discipline.
| Snapshot-on-tag, in-place reset
| `release_notes.adoc` is fully reset after each tag; Pages keeps the
  pre-reset snapshot per `/versions/<tag>/`.
| Most aligned with the FTR's "concise, current line" goal; depends on
  Pages snapshot reliability (cf. `FTR-RELEASE-PROCESS-AUTOMATION-P5`).
|===
+
The FTR should pick one before implementation begins. The recommendation
in section 6 below is *Single file, two sections* — but the decision
belongs to the FTR, not this assessment.

* *Add a policy bullet:* "Release notes added by post-release FTRs target
  the next unreleased section, not the just-released section." This is the
  rule the current 22-commit window failed to follow.

* *Split implementation into two sub-slices*: (A) one-time retro-fix for
  the inconsistency currently in `release_notes.adoc`, README link
  restoration, runbook addition; (B) ongoing post-release sweep for every
  future tag. Sub-slice (A) is safe to ship now; sub-slice (B) is a
  forward policy that needs the structural decision in place first.

* *Confirm relationship to `FTR-RELEASE-PROCESS-AUTOMATION-P5`*: the
  automation FTR remains separate and later. Manual post-release reset
  must be settled before automation locks in a wrong shape.

=== 3. What should "post-release reset" mean now that 0.2.0 is already tagged?

Two things at different time horizons:

* *Right now (one-time):* retro-fix the `release_notes.adoc` file so the
  `== 0.2.0` section reflects only what was in `cd66fbf` (the tag), and the
  post-tag content moves under a new `== 0.3.0 -- planned` section. Restore
  the README link to `/latest/`. Add a "Post-release" section to the
  runbook documenting the steps so the next release does not repeat the
  drift.

* *Going forward (ongoing):* a post-release sweep, executed once per tag,
  with the steps from the runbook. The sweep should fail loudly if
  release-note content has accumulated under a "released" section after
  the tag.

The two are not the same activity. The FTR as written conflates them; the
refinement should separate them.

=== 5. How should we avoid losing release-note content added by post-release FTRs?

Two complementary measures:

* *Policy* (effective immediately upon FTR landing): "Release-note content
  added by an FTR after the most recent release tag goes under the
  next unreleased section, not the just-released section." Recorded in
  the FTR itself, mirrored into the runbook, and added as a one-line
  reminder in `docs/ai_info/review_instructions.adoc` so review agents
  enforce it during implementation reviews.

* *Sweep*: at release time, the runbook step "freeze released section"
  asserts that no post-tag content remains in the to-be-released section.
  Tool-free check: `git diff <prev-tag>..<this-tag> -- docs/release_notes/`
  should be the entire intended diff. Anything else needs to move to the
  next unreleased section before the freeze.

=== 6. Should reset mean deleting content, moving content, marking a new development cycle, or splitting unreleased notes from released notes?

*Splitting.* Specifically: a single file with a frozen
`== 0.2.0 (released)` section preserved verbatim from the tag, plus a
fresh `== 0.3.0 -- planned` section into which post-release content
moves.

Justifications:

* The FTR's own "Decision Direction" ("concise release notes focused on
  the current release line", "Git tags and repository history as the
  authoritative release record") is satisfied by the planned section
  staying concise.
* Released sections do not grow once frozen; the file therefore does not
  accumulate indefinitely *per release*, only adds one short frozen
  block per shipped tag.
* The pattern is reversible: a later FTR can introduce per-version files
  or snapshot-on-tag without losing semantic meaning.
* No content is deleted in either the retro-fix or ongoing path.

Rejected:

* *Pure deletion* — loses adoption-relevant context the next user needs.
* *Mark a new development cycle without splitting* — does not solve the
  file's current internal inconsistency.
* *Per-version files* — heavier tooling cost than the project currently
  benefits from; deferrable to a future automation FTR.

The FTR should become a *clarified process/architecture decision plus a

== Risks

* *Risk: structural decision drift.* If sub-slice A is implemented without
  first recording the structural decision in the FTR, the retro-fix's
  shape is opinionated by the implementer rather than by the FTR.
  *Mitigation:* sub-slice A step 1 (edit the FTR) lands before steps
  2-4 — or all four in one commit reviewed against the FTR text.

* *Risk: README bake-vs-restore confusion.* The runbook documents the bake
  but not the restore. A naive retro-fix could either over-restore (lose
  the bake for a future PyPI archive page) or under-restore (leave drift
  on `main`). *Mitigation:* the runbook addition in sub-slice A step 2
  is the definitive rule; the README change in step 4 follows it.

* *Risk: post-release FTRs continue to add notes under the released
  section.* The policy bullet only takes effect once it is recorded.
  Between landing the FTR refinement and the next tag, any new FTR's
  implementation review must explicitly check placement.
  *Mitigation:* sub-slice B's one-line reminder in
  `docs/ai_info/review_instructions.adoc`.

* *Risk: conflating in-tree lifecycle with Pages publication.* The
  automation FTR's Pages-side issue is real but distinct. Keeping the
  two FTRs strictly separate avoids the wrong reset shape leaking into
  publish-side automation. *Mitigation:* explicit "See also" reference
  in the FTR's References section without scope mixing.
