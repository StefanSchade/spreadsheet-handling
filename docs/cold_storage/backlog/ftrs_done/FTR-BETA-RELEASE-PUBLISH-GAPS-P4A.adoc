===== FTR-BETA-RELEASE-PUBLISH-GAPS-P4A -- Close residual gaps before the next PyPI push

*Status:* Done (opened and closed 2026-05-23)
*Priority:* P4A
*Area:* Release readiness / cross-repo publishing

==== Purpose

Close the concrete release-readiness gaps surfaced by the 2026-05-23 audit, so
the next PyPI push (v0.1.0b6) actually delivers what the README and the
release runbook promise: a populated Pages site with both core docs and demo
walkthroughs, a demo README that points users at the published assets, a
cross-tables walkthrough that reflects current core functionality, and a
clean legal posture for the reveal.js dependency.

This FTR is intentionally a small, time-bounded punch list. It is not a place
to grow new feature scope; if something is not on the list below it does not
belong here.

==== Context

The 2026-05-23 audit (recorded in
`developer_guide/ch10_project_evolution/01_project_evolution.adoc`,
"Beta Release-Readiness Wave") closed five planning/integration FTRs:

* `FTR-TRANSFORMATION-CATALOG-DOCS-P5`
* `FTR-DEMO-USER-LAUNCH-FLOWS-AND-REVEAL-WALKTHROUGHS-P5`
* `FTR-DEMO-REPO-INTEGRATION-P4A`
* `FTR-PROTOTYPE-CONSUMER-INTEGRATION-P4A`
* `FTR-DOCUMENTATION-PUBLICATION` (superseded by the closed
  `FTR-PAGES-AND-RELEASE-NOTES-P5`)

The audit also identified that the release-readiness assertions advertised by
the core README and the ch09 runbook are not yet end-to-end true. This FTR
tracks the remaining work so it is visible from the core backlog instead of
being scattered across demo notes and review follow-ups.

==== Scope (each item is independent and individually closable)

. *Cross-tables walkthrough on the demo side.*
+
ACME slice 3 (relation / crosstable editing) is missing. Add a paired
forward/reverse pipeline that uses `expand_xref` / `contract_xref` between
exactly two entities (e.g. role x permission, employee x project, or an
equivalently small business example) plus a walkthrough AsciiDoc source under
`demo/docs/walkthroughs/`. Forward writes to `tmp/`, reverse writes to
`tmp/<name>_roundtrip/`. Include a thin smoke test mirroring
`tests/test_acme_01_plain_roundtrip.py`. Tracked on the demo side under
`FTR-DEMO-USER-LAUNCH-FLOWS-AND-REVEAL-WALKTHROUGHS-P5` (cold storage); kept
here for release visibility.

. *Demo -> pages publishing workflow.*
+
Add a demo-side GitHub Actions job (in
`spreadsheet-handling-demo/.github/workflows/`) that, on `v*` tag push,
renders `demo/docs/walkthroughs/*.adoc` to Reveal.js HTML via
`scripts/render_slides.sh`, then writes the output to
`versions/<tag>/demo/slides/` (and to `latest/demo/slides/` only when the
triggering tag is the highest visible `v*` tag by PEP 440 ordering) in
`spreadsheet-handling-pages`. Mirror the SSH-deploy-key, pinned-host-key,
job-level concurrency `pages-publish`, and bounded `git pull --rebase` retry
pattern used by the core `publish-docs` job in
`core/.github/workflows/release.yml`. The demo publisher must only ever
touch its own subtree under the shared release roots (per the layout in
`developer_guide/ch09_release_management/01_ci_cd_and_publishing.adoc`).

. *Demo README link to the published Pages site.*
+
Add a "Published documentation" section to `demo/README.md` that links to
https://stefanschade.github.io/spreadsheet-handling-pages/ (root index)
and to the matching `versions/<tag>/demo/` path once item 2 ships. Without
this the PyPI long description leads users to the core README, the core
README points at the Pages site, but the demo README dead-ends in local
AsciiDoc files.

. *Reveal.js attribution / vendoring decision.*
+
`demo/scripts/render_slides.sh` currently sets
`revealjsdir=https://cdn.jsdelivr.net/npm/reveal.js@4`. Reveal.js and
highlight.js are both MIT-licensed; jsDelivr is a public free CDN. CDN use is
legally fine, but MIT attribution is a best practice when the rendered HTML
is itself published. Decide and implement one of:
+
.. add a small attribution footer (or a `NOTICE` file in
   `spreadsheet-handling-pages`) that names reveal.js, highlight.js, and
   their licenses, and keep the jsDelivr CDN; or
.. vendor a pinned reveal.js version into the pages repo (preferably under
   the per-version `versions/<tag>/demo/slides/_assets/`) and switch
   `render_slides.sh` to a local `revealjsdir`.
+
Either is acceptable; pick one and apply it consistently. The original
demo-side review filed this as L4 ("flag for future Pages publishing /
should vendor a pinned reveal.js").

. *Dry-run `make docs-pages` before the v0.1.0b6 tag.*
+
The Pages publishing workflow has never been exercised end to end on a real
release tag (latest tag is `v0.1.0b5`, predating the publish-docs job; the
pages repo currently holds only the seed `index.html`). Before pushing
`v0.1.0b6`, run `make docs-pages` locally to catch toolchain or asciidoctor
errors. Document the dry-run command in the runbook if it is not already
there.

==== Non-goals

* No new framework features; this FTR is publication and packaging hygiene
  only.
* No retroactive backfill of release notes for older betas; the release-notes
  source-of-truth location is sufficient for the next release.
* No expansion of the core `publish-docs` job. The demo publishing workflow
  is the demo repo's concern, even though the gap is tracked here.
* No legal opinion beyond identifying the attribution / vendoring question;
  pick one of the two listed options and ship it.

==== Acceptance

* Item 1: a cross-tables walkthrough exists in `demo/docs/walkthroughs/`,
  uses `expand_xref` / `contract_xref` between two entities, has paired
  forward/reverse pipelines, and is covered by a smoke test.
* Item 2: a demo-side GitHub Actions workflow publishes Reveal.js HTML into
  `versions/<tag>/demo/slides/` (and `latest/demo/slides/` for the newest
  tag) using the same SSH-deploy-key, concurrency, and rebase-retry pattern
  as the core `publish-docs` job.
* Item 3: `demo/README.md` links to the published Pages site, and the link
  resolves to the demo subtree for the matching version once item 2 has
  published it.
* Item 4: rendered Reveal.js HTML includes either an MIT attribution footer
  (CDN path) or refers to a vendored reveal.js bundle (vendor path); the
  choice is recorded in `developer_guide/ch09_release_management/`.
* Item 5: `make docs-pages` completes successfully on the developer VM
  before the v0.1.0b6 tag is pushed, and the runbook either already names
  this step or has been updated to do so.

==== Out of scope (deferred elsewhere)

* Per-version umbrella index at `versions/<tag>/index.html` linking core,
  demo, and presentation outputs together remains the explicit out-of-scope
  item recorded in the closed `FTR-PAGES-AND-RELEASE-NOTES-P5`. A
  cross-publisher slice will pick it up later.
* Full ACME beta-slice tutorial buildout beyond the cross-tables slice
  (resource fallback/override, key-value resource writer, artifact manifest,
  explicit XLSX/ODS maintained-path tutorial) continues in the demo repo
  under its own backlog.

==== Closing note (2026-05-23)

All five items shipped on the same day:

. *Cross-tables walkthrough.* New demo slice `acme_03_xref` lives at
  `demo/data/acme_03_xref/{products,markets,product_market}.json`,
  `demo/pipelines/acme_03_xref_{forward,reverse}.yaml`, and
  `demo/docs/walkthroughs/acme_03_xref_roundtrip.adoc`. Forward uses
  `contract_xref` to project a `product x market` matrix; reverse uses
  `expand_xref` with `drop_empty: true` to recover the canonical long-form
  relation. Smoke test
  `demo/tests/test_acme_03_xref_roundtrip.py` runs both pipelines through
  the same `spreadsheet_handling.cli.apps.run.main` entry point the
  launchers exercise and asserts pair equality after sorting; the test is
  green against the local core checkout. The new walkthrough is indexed in
  `demo/docs/index.adoc`.
. *Demo -> Pages publishing workflow.* Added
  `demo/.github/workflows/publish-docs.yml`. Mirrors the core
  `publish-docs` job: inline pytest gate, SSH `DOCS_DEPLOY_KEY` with
  pinned host keys, shared `pages-publish` concurrency group, PEP 440
  latest-tag guard (with `-demo` suffix stripping so demo tags shadow the
  matching core tag for ordering), bounded `git pull --rebase` retry, and
  publication into `versions/<tag>/demo/slides/` plus
  `latest/demo/slides/`. The workflow asserts the vendored reveal.js
  bundle exists in the Pages repo and aborts loudly if it does not.
. *Demo README link to Pages.* Added a "Published documentation" section to
  `demo/README.md` that names the site root, the core user guide path, the
  release notes path, and the demo slide paths.
. *Reveal.js vendoring.* `reveal.js@4.6.1` is vendored into
  `pages/assets/reveal.js@4.6.1/` (dist + plugin + LICENSE +
  REVEALJS-README.md + a local NOTICE.md explaining the layout and bump
  procedure). `demo/scripts/render_slides.sh` learned a `REVEALJS_DIR`
  env override; the default stays the jsDelivr CDN for local dev, and the
  publish workflow sets it to
  `/spreadsheet-handling-pages/assets/reveal.js@4.6.1` so published HTML
  loads from the vendored bundle (and ships with its MIT LICENSE
  alongside, satisfying attribution cleanly).
. *Dry-run `make docs-pages`.* Run on the developer VM at HEAD of `dev`:
  build succeeded into `build/pages/` (core landing, user guide, release
  notes) and `build/pages-root/` (root navigation), ~150 KB of HTML
  output. Asciidoctor warnings about section-title sequence are pre-existing
  standalone-vs-include heading-level skew and not blocking.
