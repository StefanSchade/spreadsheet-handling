===== FTR-ODS-CALC-ADAPTER-IMPLEMENTATION-P3J -- Implement the first non-XLSX spreadsheet adapter against the stabilized contract

*Depends-on:* <<FTR-ODS-CALC-CONTRACT-MAPPING-P3J#,FTR-ODS-CALC-CONTRACT-MAPPING-P3J>>

*Context:* Once the shared spreadsheet contract is guarded and the ODS/Calc mapping is explicit, the project can add a real second adapter. This is the phase where the architecture has to prove that `WorkbookIR`, `RenderPlan`, canonical meta, and the read/write seams are genuinely spreadsheet-neutral rather than just well-documented around XLSX.

*Problem statement:* Without an actual second adapter, the generalized spreadsheet architecture remains only partially validated. But implementation should follow an explicit contract and capability profile rather than improvising through ODS-specific pressure.

*Goals:*

. Add an ODS/Calc adapter that participates in the same `Frames <-> WorkbookIR <-> RenderPlan <-> workbook file` architecture as XLSX.
. Implement the agreed write and read responsibilities through adapter-specific code that stays outside generic spreadsheet layers.
. Reuse the stabilized contract, boundary APIs, and semantic guards from Phases 3h/3i instead of introducing adapter-specific bypasses.
. Establish the first real proof that the spreadsheet architecture is not XLSX-only in disguise.

*Acceptance criteria:*

. A concrete ODS/Calc backend exists for the agreed initial scope.
. The adapter uses the intended spreadsheet-neutral seams rather than internal reach-throughs into generic rendering/parser implementation.
. Shared canonical metadata and agreed spreadsheet semantics are carried through the adapter according to the Phase-3j capability mapping.
. The implementation is covered by focused adapter tests and is ready for the broader parity/gap tests of the following FTR.

*Non-goals:*

- No promise that every XLSX feature is supported in the first ODS/Calc cut.
- No broad refactor of the shared architecture unless the new adapter reveals a real contract bug.
- No attempt to solve all future spreadsheet formats at once.

*Why now:* This is the first implementation step that can genuinely validate the Phase-3h/3i architecture under a second spreadsheet format.
