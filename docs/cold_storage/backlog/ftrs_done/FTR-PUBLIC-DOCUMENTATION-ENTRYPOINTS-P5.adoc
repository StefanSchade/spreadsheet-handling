===== FTR-PUBLIC-DOCUMENTATION-ENTRYPOINTS-P5 -- Intent-based README + Pages root entry points

*Status:* Done
*Priority:* P5
*Area:* Public documentation / READMEs

==== Delivery scope

Implementation landed as one editorial slice co-shipped with
`FTR-PAGES-ROOT-ARCHIVE-NAV-P5` across two commits:

* core (`spreadsheet-handling`) -- rewrite `README.md` into the
  orientation -> install -> intent links -> features -> license shape;
  rewrite `docs/pages/root_index.adoc` into a *Latest release* banner
  plus a *Browse by version* table (this is the
  `FTR-PAGES-ROOT-ARCHIVE-NAV-P5` deliverable, shipped in the same
  core commit); add a *Core Release Checklist* bullet in
  `docs/developer_guide/ch09_release_management/02_release_runbook.adoc`
  for maintaining the per-version table; archive this FTR and the
  sibling `FTR-PAGES-ROOT-ARCHIVE-NAV-P5` to `ftrs_done/`.
* demo (`spreadsheet-handling-demo`) -- rewrite `README.md` into the
  eight-step first-hour tutorial on the product business slice plus
  an intent-based *Where to go next* closing block.

Sibling review note `FTR-PUBLIC-DOCUMENTATION-ENTRYPOINTS-P5_review.adoc`
covers both FTRs (their Dependencies block named them as one editorial
slice). Final verdict: *Ship*; no open findings.

==== Purpose

Restructure the core README, the demo README, and the Pages root so the
public user journey is intent-based ("run the tutorial", "view the visual
walkthrough", "read the user guide for this release") rather than
repository- or path-oriented ("go to Pages", "open `latest/core/`").

Pages becomes an implementation detail behind deep links from the
READMEs, and remains a deliberate archive entry point for users who want
to pick a version.

==== Context

The beta publish established the infrastructure -- PyPI publication
works, README links exist, the Pages site exists, core documentation and
demo slide decks are published. What is still missing is an intent-based
public surface: today the READMEs point at Pages as a generic location,
and the demo README is closer to a five-minute taster than a first-hour
tutorial. This FTR turns the existing infrastructure into a usable user
journey, without adding new artifacts or new tooling.

==== Scope

Core `README.md` (living context on `main`):

* one short orientation paragraph (what `spreadsheet-handling` is);
* install snippet;
* a "where to go next" block with three intent-based deep links:
** *Try it locally in a few minutes* -> the demo repository;
** *Read the user guide (latest release)* -> `/latest/core/user-guide/`;
** *Browse documentation by version* -> the Pages root (which carries
   the per-version archive owned by `FTR-PAGES-ROOT-ARCHIVE-NAV-P5`);
* short feature bullet list and the existing license / AI usage note are
  kept;
* any CLI example block on the README must match the current canonical
  surface or be removed.

Demo `README.md` (living context on `main`) -- replace the existing
"Curated journey" two-step block with an eight-step first-hour tutorial
path:

. clone the demo;
. set up the local environment;
. run the product business slice forward pipeline;
. open the generated workbook (Excel / Calc);
. inspect helper columns, validations, and styling;
. edit a single business value;
. re-import the workbook;
. verify the canonical JSON remains clean;
. (closing block, not a numbered step) *where to go next* with
  intent-based deep links to artifacts that *already exist*:
** *View this slice as a short visual walkthrough* -> the existing
   Reveal.js deck under `/latest/demo/slides/...`;
** *Read the user guide (latest release)* -> `/latest/core/user-guide/`;
** *Browse documentation by version* -> the Pages root.

Pages root content is rewritten by the sibling
`FTR-PAGES-ROOT-ARCHIVE-NAV-P5` and is therefore *not* in scope of this
FTR; the two FTRs ship together as one editorial slice.

==== Non-goals

* No new artifacts (no screenshots, no downloadable workbooks, no new
  Reveal.js decks). Links must resolve at the time of commit; do not link
  to assets that do not yet exist.
* No release tooling changes. Frozen-context versioning of README links
  is owned by `FTR-RELEASE-README-VERSION-BAKING-P5` (DRAFT) or a manual
  release-checklist step; until then this FTR uses `/latest/...` with
  explicit "latest release" / "current release" labelling.
* No promotion of developer guide, technical model, AI agent
  instructions, or backlog material to Pages.
* No Pages topology change; no new top-level subtrees.
* No path-style or repository-oriented wording ("go to Pages", "open
  `latest/core`") in the rewritten READMEs.
* No first-hour tutorial for ACME slice 1, ACME slice 3, or the
  Storyforge trail in this FTR; the product business slice is the single
  primary maintained journey here.

==== Acceptance

* Core README is structured as orientation -> install -> intent links ->
  features -> license. The three intent links are deep-linked to
  existing artifacts and labelled with their living-context lifetime
  (e.g. *Read the user guide (latest release)*).
* Demo README walks the eight concrete steps listed above in a form that
  a user can run end to end during their first hour, using only the
  current maintained product business slice and the currently-pinned
  core version.
* All links in the rewritten READMEs resolve at the time the FTR closes
  (verified manually against the live Pages site for the
  `/latest/...` paths and the demo repo URL).
* Living-context links carry an explicit "latest release" or "current
  release" label. No README link uses a path-style or
  repository-oriented label.
* No link refers to a screenshot, downloadable workbook, or other
  artifact that has not yet been published.
* CLI example snippets in the core README, if retained, are verified
  against the current canonical CLI surface; stale snippets are removed
  or replaced rather than carried forward.

==== Dependencies

* Co-ships with `FTR-PAGES-ROOT-ARCHIVE-NAV-P5`. The core README's
  *Browse documentation by version* link expects the Pages root to be a
  usable archive portal.
* Frozen-context link correctness on PyPI is handled by
  `FTR-RELEASE-README-VERSION-BAKING-P5` (DRAFT) or a manual
  release-checklist step before the next tag. Not blocking for this
  FTR: the interim solution is `/latest/...` with explicit "latest
  release" labelling.

==== Implementation notes

* Pure editorial. One or two small commits in core
  (`README.md` plus the Pages root rewrite from
  `FTR-PAGES-ROOT-ARCHIVE-NAV-P5`) and one small commit in the demo
  repository (`README.md`).
* The eight-step demo tutorial body is text-only in this FTR. Visual
  aids land later under the narrowed
  `FTR-DEMO-VISUAL-WALKTHROUGH-ASSETS-P5`; until they ship the demo
  README describes what the user should see in prose.
* Existing Reveal.js publishing is the one visual deep-link target this
  FTR can use without depending on another FTR. The demo README closing
  block references the deck via its published path
  (`/latest/demo/slides/<journey>.html`).

==== Risks / open questions

* Whether the existing core README CLI example block
  (`sheets-run --config sheets.yaml --profile demo`,
  `sheets-example-json-to-xlsx`, `sheets-example-xlsx-to-json`) still
  matches the canonical CLI surface. To verify before the rewrite; fix
  or drop in scope of this FTR.
* The "Try it locally in a few minutes" framing must remain honest --
  setup is two commands but reading and editing a workbook takes longer.
  Prefer "Try it locally in a few minutes" or "Try the first-hour
  tutorial" over a strict five-minute promise.
* Frozen-context drift on PyPI is *not* solved by this FTR; the README
  on the `v0.1.0b6` PyPI release page will continue to send users to
  `/latest/...`. This is the gap that
  `FTR-RELEASE-README-VERSION-BAKING-P5` (DRAFT) or a manual checklist
  must close before the next tag.
