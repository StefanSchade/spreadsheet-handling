===== FTR-KEY-VALUE-RESOURCE-WRITER-P4A -- Write deterministic key-value resource files

*Status:* Implemented (minimal slice)
*Priority:* P4A
*Area:* Output generation / localized resources

==== Purpose

Add a generic deterministic writer for key-value resource files such as
`.properties`-style outputs.

Structured YAML output is useful for nested configuration trees, but localized
labels and messages often need flat key-value files per locale and resource
type.

Note: `FTR-LOCALIZED-RESOURCE-MATRIX-ROUNDTRIP-P4A` stops at recomposed
canonical tuple frames. Export to `.properties` or other key-value artifacts
remains owned by this writer FTR.

==== Scope

* Write one or more key-value files from frame data.
* Support configured key column and value column.
* Support partitioning output files by locale, resource type, or other columns.
* Support deterministic ordering.
* Support configurable escaping/newline policy.
* Return a generated-file report frame.
* Conform to the shared generated-file report shape used by artifact manifests.

==== Example Shape

[source,yaml]
----
- step: write_key_value_resources
  source: localized_resource_values
  output_dir: ./build/resources
  file_pattern: "{resource_type}_{locale}.properties"
  key: resource_key
  value: text
  sort_by: [resource_key]
  encoding: utf-8
  properties_escaping: unicode
----

==== Non-goals

* Do not implement a full template language.
* Do not infer output file names from localized sheet names.
* Do not replace the structured YAML writer.
* Do not perform application-specific fallback resolution unless the input frame
  has already been normalized by the fallback/override semantics.

==== Implemented Minimal Slice

* Pipeline step: `write_key_value_resources`.
* Pure domain module: `spreadsheet_handling.domain.key_value_writer`.
* Supported parameters: `source`, `output_dir`, `file_pattern`, `key`, `value`,
  `sort_by`, `encoding`, `properties_escaping`, `report_frame`, `name`.
* `file_pattern` uses `{column_name}` placeholders; column names extracted from
  the pattern are used as partition columns.  No placeholders → single file.
* Supported `properties_escaping` values: `unicode` (non-ASCII → `\uXXXX`,
  Java `Properties.load()` ISO-8859-1 compatible), `utf-8` (raw UTF-8 output).
* Both escaping modes escape `\`, newlines, tabs; keys additionally escape `=`,
  `:`, `#`, `!`, and leading whitespace.
* Output paths are validated against the configured `output_dir`; absolute paths
  and directory traversal (`..`) are rejected.
* Duplicate keys within the same output file produce a clear `ValueError`.

===== Provisional Report Frame Shape

The report frame uses columns `path`, `frame`, `rows`, `bytes` — consistent
with `write_structured_yaml`.  `FTR-GENERATED-ARTIFACT-MANIFEST-P4A` defines a
richer shared shape (`path`, `artifact_kind`, `writer_step`, `source_frames`,
`row_count`, `checksum`, `status`) but remains open.  Column alignment with the
manifest shape is deferred to that FTR.  `path` is stable across both shapes.

==== Acceptance

* The writer emits deterministic key-value files from a configured frame.
* Output paths cannot escape the configured output directory.
* Duplicate keys in the same output file fail with a clear error.
* Missing required key/value columns fail with a clear error.
* Tests cover partitioned files, ordering, escaping, duplicate keys, path safety,
  and generated-file reporting.
* Encoding defaults and Java `.properties` unicode escaping behavior are
  documented and tested, including the choice to use UTF-8 or Java-compatible
  escaped output.

==== ODS disposition

*Not applicable.* The writer emits deterministic text key-value resource files
(e.g. `.properties`); it never produces a spreadsheet workbook and does not go
through the `WorkbookIR` / `RenderPlan` / backend path. There is no
XLSX-vs-ODS behavior to reconcile. Tracked by
`FTR-ODS-CONSUMER-FEATURE-PARITY-P4A`.
