===== FTR-ODS-CALC-PARITY-TESTS-P3J -- Add cross-adapter parity and accepted-gap tests for shared spreadsheet semantics

*Depends-on:* <<FTR-ODS-CALC-ADAPTER-IMPLEMENTATION-P3J#,FTR-ODS-CALC-ADAPTER-IMPLEMENTATION-P3J>>, <<FTR-SPREADSHEET-SEMANTIC-INVARIANTS-P3I#,FTR-SPREADSHEET-SEMANTIC-INVARIANTS-P3I>>

*Context:* A second adapter is only useful as architectural evidence if the project can say which semantics are expected to match across adapters and which gaps are accepted. Phase 3i should already have clarified the durable spreadsheet invariants; this FTR applies them across XLSX and ODS/Calc.

*Problem statement:* Without explicit parity and gap tests, adapter differences remain anecdotal. Some real regressions would slip through as "format differences," while some legitimate format gaps would keep resurfacing as unresolved ambiguity.

*Goals:*

. Turn the shared spreadsheet contract into cross-adapter tests for the agreed supported semantics.
. Make accepted ODS/Calc gaps explicit in tests or fixtures instead of leaving them as tribal knowledge.
. Verify that roundtrip-relevant canonical metadata and core spreadsheet semantics remain intact across both adapter paths where support is claimed.
. Provide the evidence needed to evaluate whether the spreadsheet architecture generalizes beyond XLSX.

*Acceptance criteria:*

. There is a visible set of cross-adapter tests covering the shared supported semantics.
. Accepted format gaps are documented and encoded explicitly rather than hidden as unexplained failures or skips.
. The tests exercise both write/read behavior and the semantics that Phase 3i classified as durable invariants.
. The resulting suite makes it clear which parts of spreadsheet behavior are contract-level portable semantics and which remain adapter-specific.

*Non-goals:*

- No byte-for-byte or pixel-perfect identity across formats.
- No requirement that unsupported ODS features be faked just to satisfy parity optics.
- No expansion into a generic N-adapter matrix beyond the first second-adapter proof point.

*Why now:* Once ODS/Calc exists, the project needs contract-based evidence for shared semantics instead of relying on qualitative impressions from manual testing.
