== Findings

=== F-001 - Medium - Active core technical doc still says `add_fk_helpers` contains convention detection

*Location:* `docs/technical_model/ch08_architecture_details/pipeline_semantics.adoc:365` and `:774`

The active technical-model pipeline semantics doc still says
`add_fk_helpers` "Currently contains FK convention detection" and
"Materializes helper columns based on FK conventions." That contradicts the
v2 runtime contract reviewed here: FK-helper primitives consume
`_meta.helper_policies.fk.relations` and derived provenance; convention
heuristics belong to `infer_fk_relations`.

Suggested direction: update this section to describe `infer_fk_relations` /
`configure_fk_helpers` as policy producers and `add_fk_helpers` as a policy
consumer, or explicitly mark the table as historical if it is no longer a
current-state description.

=== F-002 - Low - Optional Storyforge walkthrough omits the new producer step

*Location:* `demo/docs/walkthroughs/storyforge_characters_slice.adoc:46`

The optional Storyforge walkthrough says the slice uses the same canonical
steps as the product journey and lists `validate`, `add_fk_helpers`,
`reorder_fk_helpers`, and `flatten_headers`, but omits `infer_fk_relations`.
The actual `demo_storyforge_characters.yaml` pipeline is correct; the doc list
is stale and can mislead tutorial readers about the required producer before
FK-helper consumers.

Suggested direction: add `infer_fk_relations` to the listed sequence and state
briefly that it writes the relation policy consumed by `add_fk_helpers`.
