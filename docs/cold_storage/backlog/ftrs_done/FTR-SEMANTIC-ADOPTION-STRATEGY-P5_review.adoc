==== Findings by severity

===== Minor

====== Registry runtime wording overstates current registry authority

*Location:* `docs/semantic_model/semantic_model.adoc:1533`

The Adoption Strategy says the root-level registry relocation "declared that
registries are consulted at runtime and by guards". That wording is too broad
for the current artifacts. `registries/meta_registry.yaml` says it is a review
artifact, "not a runtime contract", and `runtime_binding: forbidden`.
`registries/pipeline_step_registry.json` likewise says it is not the runtime
source of truth and points to `src/spreadsheet_handling/pipeline/registry.py::REGISTRY`.

The same Adoption Strategy subsection later says registries are not yet runtime
input, which is the correct boundary. The overstatement should be narrowed to
"consulted by guards/review workflows and aligned with runtime truth" or similar,
without implying that these registry artifacts are runtime rule input today.

====== FK-helper pilot status wording is ahead of the backlog state

*Location:* `docs/semantic_model/semantic_model.adoc:1704`

The tentative adoption-order table marks FK Helpers as `pilot (running)`.
The owning FTR, `FTR-SEMANTIC-CONTRACT-MATRIX-PILOT-FK-HELPERS-P5`, is still
`Status: New` in `docs/backlog/`. The surrounding wording that the FTR is
"positioned as" the first execution is accurate, but `running` can be read as
claiming an active/in-progress FTR lifecycle state. Prefer `pilot (planned)`,
`first pilot`, or another non-lifecycle status unless the backlog status is
advanced separately.

==== Scope control

Scope control is otherwise good. The commit changes only
`docs/semantic_model/semantic_model.adoc`, and the diff is confined to the
existing `[[semantic-model-adoption-strategy]]` chapter. It does not change
runtime code, tests, registries, guards, release machinery, backlog FTR status,
or cold-storage state.

The edit does not execute the FK-helper pilot or implement later FTRs. It names
rule-ready outputs, guard evolution, bridge coexistence, and candidate families
as adoption strategy, while leaving execution to per-family FTRs.

==== Adoption-strategy coherence

The chapter implements the expected strategy content:

* keeps the top-level `semantic-model-adoption-strategy` anchor;
* documents no-big-bang adoption;
* names review friction, token pressure, and semantic reconstruction cost as
  operational symptoms;
* distinguishes local fixes from semantic normalization;
* explains why semantic clarification and runtime refactoring should be
  sequenced together in narrow FTRs rather than detached large phases;
* defines transformation-family slices, slice questions, and exit criteria;
* describes rule-ready outputs without introducing a rule engine;
* covers bridge coexistence, migration handling, and guard evolution;
* distinguishes historical truth, producer-side residual, and genuinely
  undecided semantic debt;
* defines pilot-selection criteria and records a tentative adoption order;
* aligns with the developer-guide FTR Sizing Preflight and metadata/registry
  slicing guidance.

The operational symptoms are framed as descriptive signals that drive adoption,
not as a normative checklist every FTR must satisfy.

==== Registry/current-truth wording

The chapter correctly treats registries as current operational truth and test
oracle artifacts for review purposes, and it correctly says promotion to runtime
input would be a behavioral change owned by a per-attribute slice. The only
problem is the broader sentence called out above, which can be read as making
the current root-level registry files runtime input today.

The chapter otherwise distinguishes:

* current runtime truth;
* transitional bridge truth via `contract_lifecycle`;
* historical truth in archived FTRs and review artifacts;
* producer-side residual writes;
* aspirational future runtime/rule-input direction.

==== Residual risks

The implementer-reported residual risks are real but manageable:

* The operational symptom list may duplicate future checklist material; current
  wording keeps it descriptive.
* The tentative order may need revision; current wording explicitly says it is
  tentative and not a roadmap commitment.
* Registry-as-current-truth is load-bearing on registry accuracy; this is
  intentional, but the runtime-input wording needs the minor fix above.
* Developer-guide title references may drift; the titles exist today, and plain
  title references are acceptable despite not being AsciiDoc xrefs.
* Pilot-selection criteria may need extension later; no blocker for this FTR.
* The adoption strategy overlaps with developer-guide workflow, but it
  explicitly defers process authority to the developer guide.
* FK-helper pilot status wording may become stale; it is already slightly ahead
  of the current backlog status and should be softened.
