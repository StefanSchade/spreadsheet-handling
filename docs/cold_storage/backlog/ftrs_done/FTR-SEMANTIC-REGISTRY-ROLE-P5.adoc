===== FTR-SEMANTIC-REGISTRY-ROLE-P5 -- Pin the description / oracle / rule-input registry roles

*Status:* New
*Priority:* P5
*Area:* Semantic model / registries

==== Goal

Make the three possible registry roles -- description, test oracle, and
possible future runtime / rule input -- explicit, and capture the
implications of root-level registry artifacts. The chapter must let
later FTRs decide whether a given registry change is a documentation
change or a behavioral one.

==== Scope

* Elaborate the Registry Role chapter in
  `docs/semantic_model/semantic_model.adoc`.
* Cite the existing root-level artifacts
  (`registries/meta_registry.yaml`,
  `registries/pipeline_step_registry.json`) and the contract-lifecycle
  vocabulary already documented under
  `docs/technical_model/ch05_registries/registry_semantics/`.
* Record the warning that moving from "test oracle" to "runtime / rule
  input" is an architectural step and must not happen accidentally.
* Document the rule-ready vs rule-engine distinction so later FTRs can
  pick the right ambition level.

==== Out of scope

* No movement, rename, or schema change of registry artifacts.
* No new registry entries.
* No proposal of a runtime rule engine.
* No changes to existing architecture guards.

==== Seed sections elaborated

* Registry Role

==== Acceptance (rough)

* The three roles are named, defined, and illustrated with at least one
  existing registry consumer per role where one exists today.
* The "rule-ready, not rule engine" distinction is stated with what
  each side includes and explicitly excludes.
* Cross-references to existing registry / lifecycle documentation are
  present and resolve.

==== Code and guards

* No production code. No tests. No guards. Documentation only.
