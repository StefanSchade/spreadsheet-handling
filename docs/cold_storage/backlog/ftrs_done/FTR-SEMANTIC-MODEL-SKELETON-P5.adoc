===== FTR-SEMANTIC-MODEL-SKELETON-P5 -- Stabilize the semantic-model doc skeleton

*Status:* New
*Priority:* P5
*Area:* Semantic model / documentation

==== Goal

Establish `docs/semantic_model/semantic_model.adoc` as the canonical entry
point for the project semantic model and stabilize its chapter structure,
so later FTRs can elaborate individual sections without renegotiating the
layout.

==== Scope

* Confirm and lock the chapter structure mirroring the seed (Management
  Summary, Relation To Other Models, Problem Statement, Ambiguity Policy,
  External Inspirations, Project Semantic Model, Overlapping Semantic
  Sets, Lifecycle And Update Semantics, Registry Role, Roundtrip
  Walkthroughs, Conflict And Precedence Questions, Contract Matrices,
  Semantic Guards, Versioning And Evolution, Adoption Strategy, Planned
  FTRs, Open Questions, Glossary).
* Ensure the document renders standalone via `make docs-html`.
* Decide whether the seed text stays in a single file or is split into
  per-chapter includes under `docs/semantic_model/`. Any split is
  structural only.

==== Out of scope

* No rewriting of seed prose into final polished form.
* No inclusion in the public Pages bundle
  (`scripts/build_docs_pages.sh`).
* No new doctrine on ambiguity, registries, lifecycle, or matrices --
  those belong to the per-section FTRs.

==== Seed sections elaborated

* Management Summary
* Relation To Other Models (Functional / Technical / Semantic)

==== Acceptance (rough)

* `docs/semantic_model/semantic_model.adoc` renders without errors via
  `make docs-html`.
* Chapter headings match the seed and are stable; later FTRs add content
  under existing headings rather than reshuffling them.
* The document carries an explicit "draft scaffold" marker until the
  per-section FTRs of this phase are complete.

==== Code and guards

* No production code. No tests. No guards. Documentation only.
