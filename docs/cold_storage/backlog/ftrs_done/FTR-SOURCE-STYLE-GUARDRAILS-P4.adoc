====== Purpose

Prevent recurring source-level readability drift in AI-assisted development by combining lightweight mechanical checks with clearer agent instructions.

This FTR complements architecture guardrails. It protects local code maintainability rather than layer boundaries or semantic contracts.

====== Context

The project already has strong architectural guardrails and explicit AI working rules. The remaining risk is more mundane: AI-assisted edits can gradually introduce long procedural functions, dense inline expressions, inconsistent comments, and local cleverness that passes tests but makes later maintenance worse.

These issues should not be handled only by manual review. A small set of repeatable checks can catch the worst drift early without turning style into an excessive bureaucracy.

====== Goals

. Add mechanical checks for the most review-relevant readability limits.
. Keep thresholds pragmatic and initially warning-oriented where strict enforcement would cause noisy churn.
. Prevent German repo-facing comments and messages from reappearing unintentionally.
. Discourage nested conditional expressions and overly dense expression chains.
. Improve AI primer instructions so future generated code prefers named helpers and explicit control flow.
. Document allowed exceptions for intentionally dense or performance-sensitive code.
. Keep source-style guardrails operationally separate from architecture guardrails to preserve transparency in ownership, failures, and reporting.

====== Non-goals

* No full static-analysis framework beyond existing project tooling unless clearly justified.
* No mandatory complexity metric that blocks useful code because of arbitrary numbers alone.
* No mass rewrite of old code as part of adding the guardrails.
* No replacement for human review.

====== Scope

This FTR includes:

* optional `ruff` rule configuration if suitable rules are already available,
* a small repository-local AST guard script if existing tools do not express the desired checks cleanly,
* make targets or CI integration for the selected guard checks,
* updates to AI primer and repo-committing-agent instructions,
* documentation of thresholds and exception handling,
* a separate execution/reporting path from architecture guardrails.

====== Separation from architecture guardrails

To avoid transparency loss, source-style guardrails and architecture guardrails should remain separate tracks:

* separate command entry points (for example, `make source-style-guards` vs. architecture guard commands),
* separate CI job names and logs where practical,
* separate allowlist/baseline files,
* separate FTR ownership and review cadence.

Architecture guardrails protect boundaries and semantics. Source-style guardrails protect readability and maintainability. Both are required, but they should not produce a mixed signal in one opaque "guards" result.

====== Proposed measures

[cols="1,3,3"]
|===
| Measure | Description | Initial stance

| function length guard
| Detect functions above configured line thresholds, with separate handling for production code, tests, and generated files.
| warn first, fail after cleanup baseline is accepted

| branch/complexity guard
| Detect very high branch count or cyclomatic complexity using `ruff`/`mccabe` or a simple AST script.
| warn first

| nested conditional-expression guard
| Detect Python conditional expressions nested inside another conditional expression.
| fail immediately unless explicitly allowlisted

| dense comprehension guard
| Flag comprehensions with multiple `for` clauses or multiple `if` clauses in production code.
| warn only

| German source-text guard
| Search source comments, maintainer-facing diagnostics, and developer documentation snippets for obvious German markers.
| warn first; fail for comments after cleanup; keep false-positive handling explicit

| AI instruction update
| Tell repo agents to avoid nested ternaries, prefer explicit `if`/`elif` for multi-branch logic, split long functions, and add characterization tests before refactors.
| mandatory instruction

| cleanup baseline file
| Record current accepted exceptions with file, symbol, reason, and review date.
| temporary, reviewed periodically
|===

====== Suggested thresholds

These thresholds are deliberately conservative starting points and should be adjusted after the first inventory run.

[cols="2,1,3"]
|===

| Check | Suggested threshold | Notes

| production function length
| 80 logical lines
| Warnings above 60 may be useful, but failing at 60 is likely too noisy initially.

| test function length
| 120 logical lines
| Tests sometimes need setup and expected data; still flag very large tests.

| branch count / cyclomatic complexity
| 12
| Use as a review trigger, not as proof that code is bad.

| nested conditional expression depth
| 1
| Normal ternary allowed; nested ternary rejected.

| comprehension generators
| 2
| More than two generators should usually become an explicit loop or helper.

| comprehension filters
| 1
| Multiple filters are often better named explicitly.
|===

====== Rollout plan

Use a two-stage rollout to reduce noisy churn:

. Stage A (warn-first): function length, complexity, dense comprehensions, and language markers are warnings; nested conditional expressions fail unless allowlisted.
. Stage B (selective fail): after the cleanup baseline is reviewed and accepted, promote selected warnings (at least source comments language drift and agreed function-length ceiling) to fail.

The stage transition should be explicit in backlog notes or guard documentation.

====== Implementation sketch

Add a small tool such as `tools/check_source_style.py` if existing `ruff` rules cannot cover the desired checks cleanly.

The tool should:

. parse Python files under `src/` and selected `tests/` paths,
. report symbol-qualified findings with file and line number,
. support an allowlist file such as `tools/source_style_allowlist.yaml`,
. return non-zero only for checks configured as `fail`,
. avoid network access and external services,
. run quickly enough for normal local development.

Allowlist policy:

* each entry includes file/symbol, rule ID, rationale, owner, and review date,
* temporary exceptions include an expected sunset or re-evaluation condition,
* stale allowlist entries are reviewed periodically.

Suggested make targets:

[source,make]
----
source-style-guards:
	python tools/check_source_style.py

quality: lint source-style-guards test
----

The exact target names should follow the current Makefile conventions.

====== AI instruction update

Update the project AI primer with rules equivalent to:

* Avoid nested conditional expressions in Python production code.
* Use explicit `if`/`elif`/`else` for multi-branch decisions.
* Prefer named intermediate variables over dense expressions when the expression encodes business or rendering semantics.
* Keep functions small enough to review as one behavior unit; extract pure helpers before adding new abstractions.
* Do not mix languages in repository-facing code, comments, maintainer-facing diagnostics, or developer documentation unless the term is intentionally domain-localized.
* When refactoring existing dense code, add or identify characterization tests first.
* Do not satisfy a guard by mechanically splitting code into meaningless helpers.

====== Acceptance

* A source-style guard command exists and can be run locally.
* The selected command is wired into an appropriate Makefile target.
* Source-style guardrails are executed and reported separately from architecture guardrails.
* Nested conditional expressions deeper than the accepted threshold in production Python are either absent or explicitly allowlisted with a reason.
* Overlong-function and complexity checks produce stable, reviewable output.
* The AI primer or repo-committing-agent instructions include explicit readability rules for future generated code.
* A short allowlist policy exists for justified exceptions.
* CI integration is either implemented or deliberately deferred with a clear reason.

====== Implementation Notes

Delivered as a separate Stage-A source-style guardrail path:

* Added `tools/check_source_style.py`, a repository-local AST/tokenize guard that scans `src/spreadsheet_handling`, `tests/unit`, `tests/integration`, and `tests/architecture`.
* Added `tools/source_style_allowlist.yaml` for narrow documented exceptions with `rule`, `path`, `symbol`, `rationale`, `owner`, and `review_date`.
* Added `make source-style-guards` as the separate local command entry point. It intentionally stays separate from `make test-arch` and architecture guardrail tests.
* Added source-style guard documentation in `docs/internal_guide/dev_man/source_style_guardrails.adoc` and included it from the developer manual.
* Updated AI primer and repo-committing-agent instructions to discourage nested conditional expressions, dense semantic expressions, mixed-language repo-facing text, and meaningless mechanical splitting.
* Translated a remaining German CSV backend error message found by the new string scan.

Stage-A behavior:

* Errors: nested conditional expressions in production Python, German repository-facing comments, parse errors, and invalid allowlist files.
* Warnings: overlong functions, branch complexity, dense comprehensions, and possible German repository-facing strings.
* The current guard run reports `0 error(s)`, `38 warning(s)`, and `3 allowlisted finding(s)`.

CI integration is deliberately deferred. The repository currently has only release automation under `.github/workflows/`; there is no general quality CI workflow to extend without broadening the CI surface in this FTR. The supported execution path for now is the separate local Make target.
