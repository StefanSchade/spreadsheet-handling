===== BUG-XLSX-WORKBOOK-VIEW-BLOB-READBACK-P4A -- XLSX readback loses `_meta.workbook_view` and breaks workbook-view roundtrip

*Status:* Done
*Priority:* P4A
*Area:* XLSX carrier / hidden `_meta` sheet / workbook-view readback / roundtrip regression
*Observed via:* `worldbuilding` consumer after `refactor(worldbuilding): adopt FTR-DYNAMIC-FRAME-PROJECTION-P5`

==== Summary

After the `worldbuilding` consumer adopted the new declarative
`project_by_role` path, the XLSX roundtrip regressed while the ODS path
remained green.

The failing path is:

[source,text]
----
make -C worldbuilding export-stories
make -C worldbuilding check-reimport
----

It aborts with:

[source,text]
----
Error: _meta.workbook_view must be a mapping for workbook view readback
----

The same consumer and same data still succeed on ODS:

[source,text]
----
make -C worldbuilding export-stories-ods
make -C worldbuilding check-reimport-ods
----

with:

[source,text]
----
Reimport matches canonical data after normalization.
----

This makes the problem highly specific: it is not the declarative projection
feature in general, but the XLSX `_meta` carrier / readback contract.

and fails during workbook-view sheet mapping resolution:

* `src/spreadsheet_handling/domain/workbook_views.py:420`

Specifically:

[source,python]
----
def _workbook_view_mapping(meta: Mapping[str, Any] | None) -> Mapping[str, Any]:
    if not isinstance(meta, Mapping):
        raise ValueError("_meta.workbook_view is required for workbook view readback")
    view = meta.get("workbook_view")
    if not isinstance(view, Mapping):
        raise ValueError("_meta.workbook_view must be a mapping for workbook view readback")
    return view
----

==== Direct carrier inspection

The generated XLSX does contain a hidden `_meta` sheet.

Inspection of the first rows of `tmp/dino_stories.xlsx` showed:

[source,text]
----
'workbook_meta_blob' -> '<large JSON string>'
'version'            -> None
'exported_at'        -> None
'author'             -> None
----

Crucially:

* there is **no** standalone `workbook_view` key on the hidden sheet;
* the hidden sheet relies entirely on the `workbook_meta_blob` carrier.

When parsed through the current XLSX read path:

[source,python]
----
ir = parse_workbook("tmp/dino_stories.xlsx")
frames = workbookir_to_frames(ir)
meta = frames["_meta"]
----

the result was:

* `meta` is a `dict`
* `meta["workbook_view"]` is missing
* `meta["workbook_meta_blob"]` is present

That means the readback fell through to the fallback branch in:

* `src/spreadsheet_handling/rendering/workbook_projection.py`

rather than recovering the canonical workbook-meta mapping.

==== Stronger structural clue

The `workbook_meta_blob` string in the generated XLSX is JSON-parseable, but
its outer shape is wrong for workbook-view readback.

Observed properties of the parsed outer object:

* it is a `dict`
* it contains:
** `author`
** `exported_at`
** `sheets`
** `version`
** `workbook_meta_blob`
* it does **not** contain:
** `workbook_view`

The nested `workbook_meta_blob` key itself is again a JSON string beginning
with canonical workbook metadata such as:

* `auto_filter`
* `cell_codecs`
* ...

This strongly suggests a carrier-boundary bug:

* the outer payload being written to the hidden XLSX `_meta` sheet is not the
  canonical workbook meta itself;
* instead, a meta sheet representation containing `workbook_meta_blob` has
  been wrapped and persisted again.

The result is a structurally valid JSON blob that is nevertheless unusable for
`apply_workbook_view_sheet_mappings`, because `workbook_view` is no longer at
the expected top level.

==== Why ODS still works

The ODS path remains green in the same consumer run.

Direct check on the parsed ODS frames showed:

* `frames["_meta"]` is a `dict`
* `frames["_meta"]["workbook_view"]` is a `dict`
* `frames["_meta"]` does **not** expose a raw `workbook_meta_blob` fallback

That indicates the declarative `project_by_role` change itself is not the
regression driver. The bug is specific to the XLSX `_meta` carrier read/write
path and how it interacts with the newer presentation-meta / persistence
machinery.

==== Most likely root cause direction

The likely overlap zone is:

* render-time hidden meta construction
* persistence-boundary projection
* XLSX write/read of hidden `_meta`
* workbook-projection recovery from `workbook_meta_blob`

Relevant code paths:

* `src/spreadsheet_handling/rendering/passes/__init__.py`
  - injects `workbook_meta_blob` into the hidden meta sheet when absent
* `src/spreadsheet_handling/rendering/composer/layout_composer.py`
  - also stashes `workbook_meta_blob`
* `src/spreadsheet_handling/rendering/flow.py`
  - serialises hidden meta via `WriteMeta`
* `src/spreadsheet_handling/io_backends/xlsx/openpyxl_renderer.py`
  - writes hidden `_meta` sheet key/value rows
* `src/spreadsheet_handling/io_backends/xlsx/openpyxl_parser.py`
  - reads `_meta` sheet and reconstructs `workbook_meta_blob`
* `src/spreadsheet_handling/rendering/workbook_projection.py`
  - converts parsed hidden `_meta` sheet back into `frames["_meta"]`

The precise fixing point is still for implementation to determine, but the
contract should be:

* the hidden XLSX `_meta` carrier must roundtrip the canonical workbook meta
  as one coherent mapping;
* `frames["_meta"]["workbook_view"]` must be recoverable after XLSX parse;
* carrier-local helper fields such as `workbook_meta_blob` must not shadow or
  structurally replace the canonical mapping expected by workbook-view
  readback.

==== Resolution

Carrier projection is now defensively idempotent on both ends of the
hidden ``_meta`` sheet:

* `canonicalize_workbook_meta` in
  `src/spreadsheet_handling/rendering/workbook_projection.py` unwraps any
  top-level ``workbook_meta_blob`` carrier wrapper from a workbook meta
  mapping. Inner canonical content (``workbook_view``, ``sheets``,
  ``legend_blocks``, ...) wins over the outer sheet-level wrapper; the
  helper is recursive so chained wrappings collapse to a single canonical
  mapping, and a no-op for already-canonical input.
* `workbookir_to_frames` applies the helper to the parsed blob before
  publishing `frames["_meta"]`, so a malformed file whose blob is itself
  a wrapper containing a nested ``workbook_meta_blob`` still recovers
  ``workbook_view`` at the top level.
* `compose_workbook` in
  `src/spreadsheet_handling/rendering/composer/layout_composer.py` calls
  the helper before serialising the new blob, so a caller that passes
  `frames["_meta"]` carrying a stray ``workbook_meta_blob`` string
  (e.g. a sheet-level representation fed back into the writer) cannot
  re-nest it. Both XLSX and ODS write paths route through this composer,
  so the fix is backend-neutral.

Regression coverage:

* `tests/integration/io_backends/xlsx/test_workbook_meta_blob_carrier.py`
  exercises the real XLSX carrier path (not a unit-shaped mock):
  +
** write canonical meta with `workbook_view` and assert top-level
   recovery on parse;
** write a stray-`workbook_meta_blob` representation and assert the

** read a hand-crafted malformed XLSX with a nested wrapper and assert

** end-to-end through `ExcelBackend.read_multi` / `.write_multi`.
