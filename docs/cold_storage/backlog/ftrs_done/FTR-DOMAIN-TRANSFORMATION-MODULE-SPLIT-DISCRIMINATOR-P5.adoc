===== FTR-DOMAIN-TRANSFORMATION-MODULE-SPLIT-DISCRIMINATOR-P5 -- Split discriminator split/merge module without behavior changes

*Status:* Open -- implementation delivered; self-documentation pending independent review (not Done, not archived)
*Priority:* P5
*Area:* Domain transformations / maintainability
*Kind:* Behavior-preserving refactoring

==== Purpose

Split the discriminator split/merge transformation implementation into a small package with clear internal responsibilities, without changing runtime behavior, pipeline configuration semantics, metadata payloads, or public step names.

The current `discriminator_split.py` module contains both directions of a reversible transformation: splitting one canonical frame into discriminator-specific frames and merging those frames back into one canonical frame. The module is large enough to justify a package split, but the split/merge symmetry is semantically important and must be preserved exactly.

This is a maintainability refactor. It must not introduce conflict handling, precedence rules, reimport semantics, or a broader transformation framework.

==== Analysis Basis

This FTR is based on `FTR-DOMAIN-TRANSFORMATION-MODULE-SPLIT-DISCRIMINATOR-P5_analysis.adoc`.

The analysis recommendation is `PROCEED WITH CONSTRAINTS`: a behavior-preserving package split is mechanically achievable, but the implementation must preserve the bijective split/merge invariant, ordering semantics, and persisted `split_by_discriminator` metadata profile.

==== Target Shape

The intended package shape is:

[source,text]
----
src/spreadsheet_handling/domain/transformations/discriminator_split/
  __init__.py
  split.py
  merge.py
  naming.py
  values.py
  framecheck.py
  metadata.py
----

[cols="1,3",options="header"]
|===
| Module | Responsibility

| `__init__.py`
| Re-export only `split_by_discriminator` and `merge_by_discriminator` as the stable package-root public surface.

| `split.py`
| `split_by_discriminator` orchestration and split-side value-entry derivation.

| `merge.py`
| `merge_by_discriminator` orchestration, merge-side value-entry derivation, source-frame selection, column-order resolution, and row-order restoration.

| `naming.py`
| Target/source pattern handling, safe frame-name generation, and discriminator value extraction from frame names.

| `values.py`
| Tiny local shared discriminator-value core: plain-value normalization, empty-cell handling, equality comparison, sort token, and valid discriminator value checks.

| `framecheck.py`
| Structural frame checks only: required frame/column, source-frame existence, target-frame collision, duplicate target-frame rejection.

| `metadata.py`
| `_META_KEY`, split metadata read/write helpers, and no payload-shape redesign.
|===

The dependency graph must remain acyclic:

[source,text]
----
split -> {naming, values, framecheck, metadata}
merge -> {naming, values, framecheck, metadata}
naming -> values
values, framecheck, metadata -> leaves
----

==== Scope

* Move the existing discriminator split/merge implementation into the package shape above.
* Preserve package-root imports through `domain.transformations.discriminator_split`.
* Preserve the registry-visible public functions `split_by_discriminator` and `merge_by_discriminator`.
* Preserve all pipeline step IDs and YAML configuration shapes.
* Preserve the canonical `_meta["split_by_discriminator"]` metadata key and payload shape exactly.
* Preserve split/merge ordering behavior exactly.
* Preserve merge/split error semantics exactly.
* Add a package re-export and registry-resolution guard test.
* Update only live documentation pointers if they pin the old `.py` path.

==== Non-Goals

* No conflict-resolution logic.
* No precedence policy.
* No `conflict.py`, `precedence.py`, or equivalent anticipatory module.
* No generic reimport or recomposition framework.
* No changes to workbook-view or workbook reimport semantics.
* No changes to pipeline step names.
* No changes to YAML keys or config shape.
* No broad shared utility package.
* No refactoring of FK helpers, tabular views, xref, resources, workbook views, or derived-column policy.
* No demo repo changes except verification.

==== Explicit Conflict / Precedence Boundary

This refactor must not create a conflict or precedence seam.

The design space for overlapping editable payloads and explicit precedence belongs to `FTR-WORKBOOK-VIEW-PAYLOAD-CONFLICT-PRECEDENCE-P6`. Discriminator split/merge must retain its current behavior and must not introduce heuristic winner selection, hidden merge policy, or a generic conflict framework.

==== Explicit Behavior-Preservation Rule

Allowed changes:

* moving functions;
* changing imports required by the new module layout;
* adding a package `__init__.py` re-export;
* adding narrow re-export / registry-resolution tests;
* updating live documentation pointers.

Not allowed:

* changing produced frames;
* changing row order;
* changing column order;
* changing generated frame names;
* changing discriminator value normalization;
* changing duplicate/collision error behavior;
* changing `_META_KEY`;
* changing metadata payload shape;
* changing registry keys or pipeline config semantics.

==== Implementation Plan

. Confirm current public surface and registry targets.
. Move functions into the package shape using verbatim moves where possible.
. Re-export `split_by_discriminator` and `merge_by_discriminator` from package root.
. Keep registry and meta-registry paths unchanged if they still resolve through package-root re-exports.
. Add a guard test that verifies both public functions resolve through the package root and through registry colon-target resolution.
. Run focused discriminator tests, integration tests, architecture/meta tests, full core tests if feasible, and demo smoke verification.

If the implementation requires semantic edits rather than mechanical moves, stop and record the blocker instead of broadening the refactor.

==== Review Guidance

The independent review should focus on:

* accidental behavior changes hidden inside moved functions;
* split/merge symmetry drift;
* row-order and column-order drift;
* `_META_KEY` or metadata payload drift;
* registry colon-target resolution through the package root;
* whether `values.py` remains a tiny local package core rather than a broad shared utility seed;
* absence of conflict/precedence module or behavior;
* absence of coupling to workbook-view/reimport logic.

Reject this FTR if it becomes semantic redesign instead of a mechanical package split.
