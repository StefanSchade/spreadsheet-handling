== Implementation Quality

=== Positiv

* **Klare Architektur-Separation**: Eigenes Modul `reference_validations.py` (484 Zeilen), sauber getrennt vom bestehenden `add_validations`/`validate_columns` Rendering-Pfad
* **Findings als Dataclass**: `ReferenceFinding` ist frozen, typisiert, mit `to_record()` für Frame-Konvertierung — guter Contract
* **Findings-Shape explizit**: `FINDING_COLUMNS` als Public-Constant, konsistente Spaltenreihenfolge
* **Defensive Validierung**: Unbekannte Rule-Types, fehlende Frames, fehlende Columns, Duplicate Column-Names — alles geprüft mit klaren Fehlermeldungen
* **Drei Mode-Semantiken klar getrennt**: `warn` → Findings-Frame, `fail` → ValueError, `ignore` → passthrough
* **Composite Keys durchgängig unterstützt**: `_row_keys` + `_key_token` Pattern für Single und Multi-Column
* **Fehlerreporting gut**: `_failure_message` zeigt bis zu 10 Findings mit Frame/Row/Value-Kontext
* **FTR-Dokument ergänzt**: Explizite Abgrenzung zu `add_validations` im Purpose-Abschnitt eingefügt; Findings-Shape-Sektion mit vollem Schema

=== Anmerkungen (kein Blocker)

* [A1] **Performance bei großen Frames**: `_row_keys` iteriert via `iterrows()` und
  baut eine vollständige Liste aller (row_index, key)-Paare. Bei 100k+ Rows
  könnte das langsam werden. Für den aktuellen Consumer-Scope (Konfigurationsdaten
  mit typisch <10k Rows) ist das kein Problem. Bei späterem Bedarf könnte eine
  vectorized pandas-Lösung nachgezogen werden.

* [A2] **`allow_empty` Default bei FK ist `True`**: Das bedeutet, vollständig leere
  FK-Zeilen werden stillschweigend übersprungen. Das ist pragmatisch für
  teilweise gefüllte Konfigurationstabellen, aber der Default sollte in der
  Nutzerdokumentation erwähnt werden, da manche Konsumenten `allow_empty: false`
  erwarten könnten.

* [A3] **`ignore` Mode**: Erledigt durch
  `test_ignore_mode_does_not_write_findings_frame_or_raise`.

* [A4] **`unique_reference` mit `allow_duplicates: true`**: Erledigt durch
  `test_unique_reference_can_be_explicitly_disabled_for_template_rules`.

* [A5] **`pipeline_step_registry.json`-Eintrag**: Nachprüfung: `rules`,
  `mode`, `findings` und `name` sind dokumentiert. Die ursprüngliche
  Review-Notiz zu fehlenden optionalen Parametern war ein Fehlalarm.

== Architektur-Einordnung

Die Implementierung schafft sauber zwei getrennte Validierungs-Pfade:

* **Rendering-Validations** (`add_validations` → `_meta.constraints` → `DataValidationSpec`):
  Für Spreadsheet-Dropdowns und Cell-Constraints
* **Domain-Validations** (`validate_references` → Findings-Frame oder Fail):
  Für relationale Integritätsprüfungen auf Daten-Ebene

Diese Trennung ist architektonisch korrekt und im FTR-Dokument explizit
dokumentiert. Spätere FTRs (CONDITIONAL-VALIDATION-RULES, GRAPH-REFERENCE-VALIDATIONS)
können auf dem Findings-Shape und Mode-Pattern aufbauen.

'''
