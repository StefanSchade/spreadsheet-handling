===== FTR-DECLARATIVE-TABULAR-VIEW-OPS-P4A -- Add small declarative table operations for workbook views

*Status:* Done
*Priority:* P4A
*Area:* Domain transformations / view composition

==== Purpose

Provide a small, explicit set of generic tabular view operations so consumer
repositories do not need Python plugins for ordinary view preparation beyond
the atomic `extract_frame` step.

This is the transformation half of declarative workbook views. Workbook view
composition decides which frames become visible sheets. Tabular view operations
produce the frames that are worth showing.

`FTR-GENERIC-FRAME-EXTRACTIONS-P4A` owns the atomic select/filter/rename/sort
step. This FTR owns higher-level display operations such as bounded pivots and
small registered view builders. `FTR-SIMPLE-JOIN-VIEWS-P4A` owns explicit join
and semi-join behavior when the current consumer path needs it.

==== Layer Boundary Note

Tabular view operations produce frames. Workbook views consume frames. This FTR
sits between frame-level extraction and workbook-level materialization. Its
output is always a named, inspectable frame, not a hidden intermediate in the
workbook-view rendering path.

==== Consumer Trigger

A prototypical consumer workbook view currently performs operations such as:

* select fixed columns from `variables`
* rename `ID` to `variable_id` for mapping views
* sort rows by `sort_key` and `field_id`
* merge variable display columns into mapping views
* pivot mapping names into display columns
* aggregate duplicate display values with a configured separator

These operations are not domain-specific. They are small relational/display
operations over frames.

==== Goals

* Reuse `extract_frame` for selecting/projecting columns, renaming, output
  column order, filtering, and sorting.
* Support a bounded pivot/display operation with configured index columns,
  column key, value column, fill value, and aggregation policy.
* Prefer several small registered steps over one broad query-like
  `tabular_view_ops` step.
* Delegate explicit join semantics to `FTR-SIMPLE-JOIN-VIEWS-P4A` unless
  `add_lookup_helpers` already fits the helper-enrichment use case.
* Write provenance/lifecycle metadata that marks outputs as derived workbook
  views unless configured otherwise.
* Keep operations deterministic and easy to inspect in YAML.

==== Non-goals

* Do not implement arbitrary SQL.
* Do not expose raw pandas expressions in YAML.
* Do not solve nested JSON or hierarchical query semantics.
* Do not replace specialized plugins for truly domain-specific parsing,
  validation, or generation logic.

==== Acceptance

* A consumer repository can express simple entity, axis, and mapping display
  view preparation without a local `workbook_views` plugin.
* The configured operations are available through the normal pipeline registry.
* Tests cover extraction composition through `extract_frame` and bounded pivot
  behavior.
* Pivot semantics define index columns, column key, value column, fill value,
  duplicate handling, and aggregation policy explicitly.
* Errors point to the configured frame/column names when input columns are
  missing or ambiguous.
* ODS and XLSX render/read paths can consume the resulting frames without
  adapter-specific branches.
