== Inspiration-Vs-Doctrine Balance

The chapter opens with an explicit note that it is inspirational, not
normative. It further states that only the project-interpretation blocks are
load-bearing for `spreadsheet-handling`, while the relevant-ideas blocks
summarize external concepts without binding the project.

The project interpretations are appropriately local:

* Specification thinking is reduced to small semantic contracts carried by
  FTR acceptance criteria, review artifacts, and stable architecture guards.
* Data contracts are tied to `_meta.*`, registries, and guards without
  claiming runtime registry enforcement.
* Schema evolution is tied to transition bridges, owners, and removal
  conditions.
* Consumer-driven contracts are tied to registry consumer lists, runtime
  consumers, guards, and review checks.
* Deprecation and migration policy is tied to FTRs, review artifacts, public
  surfaces, and guards where they exist.

The section explicitly rejects heavyweight formal-spec adoption (`no TLA+`,
`no Z`, `no Alloy`) and does not introduce a rule-engine commitment.
