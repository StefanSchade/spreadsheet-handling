===== FTR-DOMAIN-TRANSFORMATION-PACKAGE-BOUNDARY-GUARDS-P5 -- Guard package boundaries for modularized domain transformations

*Status:* New
*Priority:* P5
*Area:* Architecture guards / domain transformations
*Kind:* Architecture guardrail / maintainability

==== Purpose

Add executable architecture guards that protect the newly modularized domain transformation package boundaries from silent erosion.

Two behavior-preserving package splits have now established a maintainability pattern:

* `FTR-DOMAIN-TRANSFORMATION-MODULE-SPLIT-FK-HELPERS-P5`
* `FTR-DOMAIN-TRANSFORMATION-MODULE-SPLIT-DISCRIMINATOR-P5`

Those splits reduced local cognitive load by moving large transformation modules into responsibility-focused packages with stable package-root public APIs. This FTR makes that pattern durable by adding narrow guardrails. The goal is not governance for its own sake; the goal is to prevent future edits from importing internal package modules directly, bypassing public facades, or reintroducing hidden cross-family coupling.

==== Scope

Add a small architecture-test group for domain transformation package boundaries.

Initial guarded families:

* `spreadsheet_handling.domain.transformations.fk_helpers`
* `spreadsheet_handling.domain.transformations.discriminator_split`

The guards should cover only package-split transformation families at first. Other transformation modules remain out of scope until they are intentionally modularized.

==== Guardrail Rules

===== 1. Package-root public API rule

External code must import transformation-family public functions through the package root, not through internal implementation modules.

Allowed example:

[source,python]
----
from spreadsheet_handling.domain.transformations.fk_helpers import enrich_helpers
from spreadsheet_handling.domain.transformations.discriminator_split import split_by_discriminator
----

Disallowed example:

[source,python]
----
from spreadsheet_handling.domain.transformations.fk_helpers.policy import ...
from spreadsheet_handling.domain.transformations.discriminator_split.values import ...
----

Internal modules may import siblings inside the same package when required by the designed dependency graph.

===== 2. Registry public-target rule

Pipeline registry entries must target package-root public transformation functions, not internal submodule helpers.

Allowed example:

[source,text]
----
spreadsheet_handling.domain.transformations.discriminator_split:split_by_discriminator
----

Disallowed example:

[source,text]
----
spreadsheet_handling.domain.transformations.discriminator_split.split:split_by_discriminator
----

The package root is the stable contract. Internal module layout remains an implementation detail.

===== 3. No cross-family internal imports

Transformation families must not import another family’s internal implementation modules.

Allowed:

* importing a public package-root API if a real dependency is explicitly accepted;
* importing local sibling modules inside the same family package.

Disallowed:

* `fk_helpers.*` importing `discriminator_split.values`;
* `discriminator_split.*` importing `fk_helpers.policy`;
* any other transformation code importing a package-split family’s internal modules.

===== 4. Domain transformation layer boundary

`domain/transformations` code must not import from outer/application-facing layers:

* `spreadsheet_handling.io_backends`
* `spreadsheet_handling.rendering`
* `spreadsheet_handling.cli`
* `spreadsheet_handling.application`

The pipeline may bind transformation functions, but it must not interpret internal transformation metadata structures.

===== 5. No generic grab-bag shared module

This FTR must not create or bless broad utility modules such as:

* `shared/utils.py`
* `shared/helpers.py`
* `shared/common.py`
* `misc.py`

If future shared helpers become necessary, they must have named responsibility and a separate FTR or explicit review decision, for example `shared/meta_access.py` or `shared/column_labels.py`. No shared package is introduced by this FTR.

==== AI Workflow Guidance

Update `docs/ai_info` so future agents know that domain transformation package splits must extend or update these boundary guards.

The guidance should say, in substance:

* package-split transformation families expose stable public APIs from their package root;
* future refactors must avoid external imports from internal submodules;
* future refactors must add or update architecture guards when new transformation packages are modularized;
* pipeline registry targets should use stable package-root APIs;
* broad shared utility packages are not allowed without explicit architectural review.

Keep this guidance concise and discoverable. Prefer updating an existing relevant ai_info file over creating a large governance document unless the current structure clearly calls for a new short file.

==== Non-Goals

* No runtime behavior changes.
* No transformation refactoring.
* No new shared utility package.
* No registry target churn unless an existing target violates the new rule.
* No changes to pipeline step names.
* No changes to metadata semantics.
* No edits to demo repository files.
* No broad architecture framework.
* No guardrails for transformation families that have not yet been package-split.

==== Implementation Notes

The guards can be source-level AST tests similar to existing architecture guards.

Recommended implementation approach:

* enumerate package-split transformation families explicitly;
* define internal module prefixes for each family;
* scan `src/spreadsheet_handling` imports;
* allow imports from internal modules only when importer is inside the same family package;
* scan `pipeline/registry.py` and `pipeline_step_registry.json` targets for internal submodule targets;
* scan `domain/transformations` imports for forbidden outer-layer dependencies;
* keep allowlists narrow and documented in the test file.

==== Acceptance

* Architecture tests protect `fk_helpers` and `discriminator_split` internal package modules from external imports.
* Architecture tests verify pipeline registry targets use package-root public APIs for guarded transformation families.
* Architecture tests verify `domain/transformations` does not import from `io_backends`, `rendering`, `cli`, or `application`.
* Tests do not require package splits for transformation families that are still single modules.
* No runtime/source behavior changes are introduced.
* Existing full test suite remains green.
* `docs/ai_info` contains concise guidance that future domain transformation refactors must update these guards.
* The guardrails are narrow enough that future maintainability work can extend them without fighting a broad governance framework.

==== Review Guidance

The independent review should focus on:

* whether the guards enforce real boundaries rather than superficial style;
* whether the allowlists are narrow and understandable;
* whether the tests avoid freezing implementation details that should remain private;
* whether the registry rule correctly protects package-root public APIs;
* whether the ai_info guidance is discoverable without becoming process-heavy;
* whether the FTR avoided creating a broad shared utility or governance layer.

Reject this FTR if it turns into a transformation refactor, shared-utility extraction, or broad architecture-governance framework.
