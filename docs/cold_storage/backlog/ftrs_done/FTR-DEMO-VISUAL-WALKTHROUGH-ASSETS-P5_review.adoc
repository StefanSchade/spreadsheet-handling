== Residual risks

* **The currently-published slide deck at
  `/latest/demo/slides/product_business_slice.html` still carries
  the pre-rewrite "helpers stay understandable on reimport" wording
  until the next demo `publish-docs` run.** Neither slice edits
  generated Pages output. A new demo tag (or a manual publish-docs
  rerun) is required to republish the corrected deck. The CI/CD
  posture is unchanged; the drift simply cleared from source.

* **Published Reveal.js deck does not yet show screenshots.** By
  design: the walkthrough wraps `image::` directives in
  `ifndef::backend-revealjs[]` because the demo publish-docs
  workflow rsyncs only `build/slides/` into the Pages layout and
  does not bundle `docs/screenshots/`. Embedding raw image
  directives in the slide source would produce broken-image icons
  in the published deck. Closing this gap is the FTR's deferred
  publishing-model item from *Risks / open questions*; a later FTR
  must pick a publication model that honours the frozen-context
  policy from `FTR-RELEASE-README-VERSION-BAKING-P5`.

* **Per-image content specification assumes the workbook layout
  stays roughly as it is today.** If a future pipeline change
  renames columns or reshuffles helper placement, the specification
  block in screenshots `README.md` will drift before any captured
  PNG does. Mitigation: the *Demo Alignment Checklist* bullet in
  the core release runbook tells the releaser to review the
  screenshot set against the freshly generated workbook at each
  release and to delete drifted images rather than ship misleading
  ones.

* **Single-frame edit story.** Dropping `edit-status-before.png` in
  favour of relying on `workbook-overview.png` / `helper-columns.png`
  to convey the pre-edit state is a judgement call. If a reader
  jumps directly to the *What Comes Back On Re-import* section
  without reading the *What To Look For In Excel* frames, they see
  only the after-state. Acceptable for now: the walkthrough is
  structured as a top-down read, the README tutorial step 8 frames
  the edit as a `diff` rather than a visual comparison, and the
  *Per-image content specification* documents the choice. A future
  capture pass can add a paired before-frame if reader feedback
  shows the after-only frame reads as ambiguous.

* **Drift on the walkthrough's embedded YAML block.** The
  walkthrough now quotes the literal `remove_fk_helpers` step from
  `pipelines/demo_workbook_reimport.yaml`. If the pipeline shape
  changes (for example a different prefix, or a follow-up step),
  the walkthrough block becomes stale. Acceptable for now: the
  reimport pipeline has been stable across the last several
  releases and the embedded block is a single two-line `step:`,
  not a full pipeline reproduction.
