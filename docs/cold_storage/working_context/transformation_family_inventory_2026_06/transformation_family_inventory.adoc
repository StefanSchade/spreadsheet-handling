= Transformation/Capability Family Inventory -- June 2026

== Purpose and scope

This document is a June 2026 inventory of currently implemented transformation and capability families in `spreadsheet-handling/core`.

It is evidence-based and is not a backlog, roadmap, design proposal, or Functional Transformation Request. It records what is implemented and currently visible through code, registered pipeline steps, tests, and documentation.

The inventory is scoped to transformation/capability families that are implemented in the current repository state, especially registered pipeline steps, domain transformation modules, workbook/view capabilities, validation and writer capabilities, and the cold-storage FK-helper cleanup context.

== Method

Evidence sources inspected:

* Pipeline registry and step factories: `src/spreadsheet_handling/pipeline/registry.py`, `src/spreadsheet_handling/pipeline/steps.py`.
* Domain transformation and capability modules under `src/spreadsheet_handling/domain/`.
* Tests under `tests/unit/domain/`, `tests/unit/pipeline/`, `tests/integration/pipeline/`, `tests/integration/roundtrip/`, and architecture tests.
* User guide transformation catalog and workflow docs under `docs/user_guide/ch02_workflow/` and `docs/user_guide/ch03_transformation_catalog/`.
* Technical Model concepts under `docs/technical_model/ch04_concepts/`, `docs/technical_model/ch05_registries/`, and `docs/technical_model/ch08_architecture_details/`.
* Functional Model use-case concepts under `docs/functional_model/ch04_use_case_concepts/` and problem-space overview under `docs/functional_model/ch05_problem_space_overview/`.
* Warm storage concept/review material under `docs/warm_storage/`, especially `docs/warm_storage/concepts/acme_progressive_tutorial_guide.adoc` and `docs/warm_storage/global_reviews/next_program_sequencing_review.adoc`.
* Cold-storage working context `docs/cold_storage/working_context/domain_cleanup_fk_helper_2026_06/`, especially `transformation_inventory.adoc` and `coupling_map.json`.

Families are grouped by current implementation and usage semantics, not by future plans. Where two names appear to describe overlapping or adjacent behavior, the ambiguity is recorded instead of forcing a future taxonomy.

Functional Model coverage means:

covered:: Has a direct Functional Model use-case concept for the family.
partial:: Appears in Functional Model use cases or problem-space text, but not as a complete direct family concept.
missing:: No meaningful Functional Model coverage found.
unclear:: Coverage exists only indirectly or the implemented capability is classification-ambiguous.

== Initial candidate family list verification

The starting candidate list was broadly consistent with current implementation. The following adjustments are evidence-based:

* `lookup_helpers` and `enrich_lookup` describe the same current family from different surfaces: step/configuration vocabulary vs. domain transform vocabulary.
* `tabular_views` and `pivot_frame` describe the same current family.
* `writer / artifact_manifest` is a grouped output capability; `write_artifact_manifest` aggregates reports produced by writer steps such as `write_key_value_resources` and `write_structured_yaml`.
* `legend_blocks` is implemented as metadata/rendering/readback behavior, not as a currently registered pipeline step.
* `derived_column_policy`, `project_by_role`, `meta_bootstrap`/`yaml_overrides`, `header_shape`, and `identity`/`plugin` are implemented registered capabilities not present in the initial candidate list.
* No evidence was found for treating `join_views` and `tabular_views` as the same family; they are adjacent projection/view derivation families with distinct registered steps and tests.
