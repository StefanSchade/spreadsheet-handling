= Worldbuilding Contract Rewrite Check

== Purpose

Short operational note on whether a contract-first domain/meta rewrite is
expected to support the current `worldbuilding` consumer repository.

This note is temporary working context. Once the relevant contract gaps are
closed and the consumer flow runs, it can become obsolete.

== Current Consumer Usage

The `worldbuilding` repository currently consumes more than the small
Functional Model demo slice. The real Dino Island pipelines use:

* reference validation: primary keys, unique values, foreign keys;
* FK helper configuration, helper generation, and helper reordering;
* header flattening / unflattening;
* `join_frames`;
* `extract_frame`;
* XRef contraction and expansion with dense axes;
* role-based projection for helper/key/payload columns;
* workbook-view declaration and sheet/frame readback mapping;
* workbook validation constraints, including list rules and legend-backed rules;
* compact multiaxis expansion for coded matrix cells;
* discriminator merge for facet-specific relation frames;
* derived/helper column cleanup on reimport;
* override/bootstrap-style metadata input;
* small local plugins for empty-frame normalization, column dropping, header
  cleanup, and final frame selection.

== Expected Rewrite Coverage

A contract-first domain/meta rewrite should be able to cover the consumer flow
if it includes the Functional Model concepts already identified:

* FK helpers and helper cleanup;
* XRef Crosstable;
* Legend Blocks and legend-backed validation;
* Validation and constraints;
* Workbook View Boundary;
* Derived frame views, including joins and extracts;
* Compact Multiaxis as a convenience composition;
* Discriminator split/merge;
* Meta bootstrap or equivalent declarative metadata input;
* output frame selection / persistence scope.

Worldbuilding is therefore a good later real-consumer contract, but not a good
first rewrite slice. The first slice should stay smaller, likely FK helper
roundtrip.

== Operational Gaps To Watch

* Empty relation frame/schema initialization should not require a consumer
  plugin.
* Header flattening and cleanup should be treated as workbook-shape boundary
  behavior, not core domain logic where avoidable.
* Final frame selection should be an explicit output/persistence-scope concept
  rather than a consumer plugin.
* Derived/helper column cleanup should be absorbed into helper return-path
  ownership instead of surviving as a parallel concept.
* XRef matrices with legend-coded scalar cells should be explicit in the
  contract; Worldbuilding currently combines matrix projection with coded cell
  validation.

== Assessment

The rewrite is expected to support Worldbuilding if the above gaps are handled.
Worldbuilding should become a later integration contract because it combines
several real features in one consumer flow.
