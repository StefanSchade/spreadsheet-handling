= Domain Reformation Strategy Inquiry
:toc:
:toclevels: 3

== Purpose

This inquiry records strategic options for reducing accumulated domain and meta
implementation complexity after the Functional Model promotion work.

The question is not yet an ADR decision. The goal is to compare possible
refactoring/rewrite strategies, preserve the reasoning, and prepare a later
decision record.

== Problem Statement

The current implementation has grown through many valid local steps:

* FK helpers began as a concrete helper feature and later accumulated lookup,
  enrichment, derived-column, cleanup, and provenance paths.
* Workbook-view metadata, rendering metadata, helper metadata, and carrier
  metadata overlap in places.
* Some implemented steps express clear functional needs; others encode
  historical integration strategy.
* Some names couple a functional idea to a carrier or implementation detail.
* Older and newer paths sometimes solve adjacent problems without a single
  user-facing model deciding which one owns the requirement.

The result is a knot of domain transformation behavior, meta lifecycle behavior,
rendering boundary behavior, and publishing behavior.

At the same time, the Functional Model is becoming comparatively small and
coherent: prepare frames, project editable workbook surfaces, interpret edited
returns, validate, and optionally publish artifacts. That raises the question
whether the existing domain/meta implementation is now disproportionate to the
functional model it should express.

== Goal

Find a path that:

* preserves working adapter, renderer, parser, CLI, and fixture knowledge where
  possible;
* removes accumulated domain/meta sediment;
* uses Functional Model use cases as executable contracts;
* accepts breaking changes where they reduce long-term complexity;
* avoids rebuilding the same complexity under new names;
* leaves the final decision to a later ADR.

== Shared Direction

All credible strategies should treat the Functional Model as the source of
intent.

[source,text]
----
user intent
-> Functional Model
-> integration use-case contract
-> domain/meta implementation
-> semantic and implementation consistency checks
----

The risky reverse direction should be avoided:

[source,text]
----
old implementation sediment
-> inferred model
-> renamed sediment
----

== Concise Comparison

[cols="1,2,2,2", options="header"]
|===
|Strategy
|Main benefit
|Main cost
|Fit for current project

|Hard Reset / Greenfield Core
|Maximum freedom to design cleanly
|Loses too much embedded adapter, renderer, parser, and edge-case knowledge
|Weak

|Classic Strangler
|Lowest compatibility risk
|Long coexistence can preserve old complexity too long
|Moderate, but probably too conservative

|Contract-First Domain/Meta Rewrite
|Uses Functional Model contracts while allowing breaking internal change
|Requires strong integration contracts before implementation work
|Strong

|Domain/Meta Reset With Adapter Preservation
|Keeps expensive boundary knowledge while replacing the problematic core
|Requires careful boundary design between old adapters and new domain/meta
|Strong

|In-Place Simplification
|Least disruptive and can delete obvious duplication quickly
|May spend effort fighting the existing structure
|Useful as supporting work, weak as main strategy
|===

== Recommendation For Later ADR

The strongest direction is a combination of:

* *Contract-First Domain/Meta Rewrite* as the main strategy;
* *Domain/Meta Reset With Adapter Preservation* as the architectural boundary;
* selective *In-Place Simplification* only where deletion is obvious and low
  risk.

This means:

. first build integration use cases from the Functional Model;
. keep adapters, renderer, parser, CLI, and useful fixtures where possible;
. rebuild the domain/meta interior behind a cleaner boundary;
. allow breaking changes where they remove historical sediment;
. delete old domain/meta paths once the contract scenarios pass.

Hard Reset / Greenfield Core is not recommended because it discards too much
working context. Classic Strangler is safer for mature public APIs, but this
project has little external adoption and can afford cleaner internal breaks.

== Candidate First Slice

The recommended first slice remains FK helper roundtrip because it touches the
problematic boundary between domain behavior, meta behavior, workbook view,
reimport cleanup, and validation.

The first contract should cover:

* explicit PK/FK declaration;
* helper-column derivation for workbook editing;
* workbook-facing editability;
* return-path helper cleanup or ignoring;
* dangling-reference and duplicate-key validation;
* no separate lookup/enrichment concept unless the Functional Model requires
  it outside FK helper behavior.

== Guardrails

* No implementation rewrite without integration contracts.
* No new public concept merely because an old module exists.
* No duplicate user-facing expression for the same requirement unless the
  Functional Model explains the difference.
* No Project Memory modeling of internal families.
* No backlog/FTR creation from this inquiry by itself.
* No final strategy decision in this document; use an ADR.

== Open Questions For The ADR

* What is the minimal first contract suite?
* What should the new internal boundary be called?
* Which current step names should remain for short-term migration?
* Which historical behaviors are real user requirements?
* How much meta is rewritten in the first slice versus adapted at the boundary?

== Disposition

This inquiry should inform a later ADR and subsequent FTRs if a rewrite path is
accepted.
