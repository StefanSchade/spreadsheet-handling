= Conditional Follow-Up Table Views
:toc:
:toclevels: 3

== Status

Functional-model inquiry.

This note refines the existing backlog idea behind
`FTR-CONDITIONAL-FOLLOW-UP-PROJECTION-FLOWS-P5`.

It is not MVP.

It should not be presented as currently supported workbook behavior unless a
later implementation FTR explicitly accepts it.

== Purpose

Some workbook workflows need projections that only become meaningful after an
earlier workbook maintenance round has classified the data.

A user first edits a matrix or compact matrix. The edited values classify
products, applications, packages, markets, or relation tuples. After reimport,
the classification determines which specialized table views should appear in a
later generated workbook.

The key pattern is:

[source,text]
----

base data
-> classification matrix
-> user edits classification
-> reimport / recomposition
-> derived filtered maintenance views
-> user edits specialized details
-> reimport / merge back into full detail data
----

This is not live spreadsheet filtering. The workbook is a static artifact.
The conditional views are rebuilt on the next generation cycle after the edited
classification has been reimported.

== User Problem

A large configuration model may contain a broad set of common base fields and
many category-specific fields.

For example, software packages may be maintained for different markets. A
product-by-market matrix classifies the deployment model:

[source,text]
----

package       | Germany | Japan | Brazil
Billing API   | SaaS    | SaaS  |
Archive Tool  | OnPrem  |       | SaaS
Mobile App    | AppStore|       | AppStore
----

A compact cell may also carry several facts at once:

[source,text]
----

SaaS / JP / OIDC
OnPrem / DE / LDAP
AppStore / EN / SSO
----

After this classification has been maintained, additional information becomes
relevant only for certain subsets.

Examples:

* SaaS packages need hosting-provider, tenant-model, operations-owner, and
  availability-zone fields.
* Japan offerings need local contact, regulatory classification, and language
  support fields.
* AppStore offerings need store account, publishing status, and review-contact
  fields.
* OnPrem offerings may need installation package, supported operating system,
  and patch-delivery fields.

A flat master table may contain all fields, but showing every field for every
record creates a large sparse workbook that is hard to maintain.

The user-facing need is therefore:

* common base fields remain visible in a normal base table;
* specialized fields appear only in filtered detail views;
* the filtered views are derived from the classification maintained in the
  earlier matrix;
* edits in the specialized views merge back into the full underlying detail
  data.

== Example: Software Package Market Maintenance

Assume the system has these source frames:

[source,text]
----

packages
markets
package_market_offerings
package_market_details
----

The classification relation after matrix reimport may look like this:

[source,text]
----

package_id | market_id | delivery_model | language | auth_mode
PKG-BILL   | DE        | SaaS           | DE       | OIDC
PKG-BILL   | JP        | SaaS           | JP       | OIDC
PKG-ARCH   | DE        | OnPrem         | DE       | LDAP
PKG-MOB    | DE        | AppStore       | EN       | SSO
PKG-MOB    | BR        | AppStore       | EN       | SSO
----

The full detail table may contain common and specialized fields:

[source,text]
----

package_id | market_id | owner | lifecycle_status | hosting_provider | tenant_model | local_contact | store_account
PKG-BILL   | DE        | Anna  | active           | Azure            | shared       |               |
PKG-BILL   | JP        | Anna  | active           | Azure            | dedicated    | Tanaka        |
PKG-ARCH   | DE        | Bernd | active           |                  |              |               |
PKG-MOB    | DE        | Clara | draft            |                  |              |               | MSG Store
PKG-MOB    | BR        | Clara | draft            |                  |              |               | MSG Store
----

The generated workbook may expose:

[source,text]
----

base_package_market_details
package_id | market_id | owner | lifecycle_status

saas_package_market_details
package_id | market_id | hosting_provider | tenant_model

japan_package_market_details
package_id | market_id | local_contact

appstore_package_market_details
package_id | market_id | store_account
----

The specialized views are filtered by explicit configured predicates evaluated
against the reimported classification relation.

For example:

## [source,yaml]

view: saas_package_market_details
source: package_market_details
identity: [package_id, market_id]
include_when:
frame: package_market_offerings
match_on: [package_id, market_id]
where:
delivery_model: SaaS
columns:

* package_id
* market_id
* hosting_provider
* tenant_model

---

A record may appear in more than one specialized view. For example,
`PKG-BILL / JP` may appear both in the SaaS view and in the Japan view.

== Functional Pattern

The pattern has two distinct phases.

=== Phase 1: Classification Maintenance

The first workbook exposes a matrix or compact matrix.

The user edits classification data.

The reimport path reconstructs the structured classification relation.

This phase is owned by existing concepts:

* XRef Crosstable owns matrix addressing and relation recomposition.
* Cell Codec owns compact cell decoding when a cell carries several structured
  attributes.
* Legend Blocks or validation may constrain allowed values.
* Compact Multiaxis may compose these capabilities into one workbook surface.

Conditional follow-up views do not replace these concepts. They consume their
result.

=== Phase 2: Derived Specialized Maintenance Views

After the classification relation has been reimported, configured predicates
determine which rows become relevant for which specialized views.

The system generates normal table views for those subsets.

The user edits the specialized views.

On reimport, the specialized views write back into the full detail table by
stable identity.

## [source,text]

classification relation

* full detail table
* configured predicates
* configured view columns
  -> filtered specialized table views
  -> user edits
  -> writeback into full detail table

---

== Transformation 1: Build Conditional Table Views

The forward transformation builds one or more workbook-facing table views from
a full detail frame, using predicates evaluated against a classification frame
or relation frame.

=== Information Consumed

* Full detail frame.
* Classification frame or relation frame.
* Stable identity columns for the detail rows.
* One or more view declarations.
* For each view:
  ** predicate source;
  ** join or match keys;
  ** predicate condition;
  ** visible columns;
  ** optional sort order;
  ** optional lifecycle metadata marking the result as a derived table view.

=== Information Produced

* Derived filtered table views.
* Optional diagnostics if configured predicates reference missing columns,
  non-unique identities, unknown classification values, or ambiguous matches.
* Optional trace metadata that explains why a row appeared in a derived view.

=== Example Output

[source,text]
----

saas_package_market_details:

package_id | market_id | hosting_provider | tenant_model
PKG-BILL   | DE        | Azure            | shared
PKG-BILL   | JP        | Azure            | dedicated
----

The view is not canonical source data. It is a workbook-facing maintenance view
over the full detail table.

== Transformation 2: Merge Edited Conditional Views

The reverse transformation reads edited specialized views and merges their
editable fields back into the full detail frame.

=== Information Consumed

* Full detail frame.
* Edited derived table views.
* View declarations from the forward path.
* Stable identity columns.
* Editable column set per view.
* Relevance-loss policy for rows that no longer match a view predicate.
* Conflict policy for overlapping views.

=== Information Produced

* Updated full detail frame.
* Optional diagnostics for missing identities, duplicate identities, conflicts,
  invalid stale rows, or destructive relevance-loss handling.
* Optional trace metadata identifying which view updated which full-detail
  fields.

=== Merge Rule

A specialized view must not create hidden semantics by itself.

It writes back only:

* to configured editable columns;
* for rows identified by configured identity columns;
* into the configured full detail frame.

If a view contains a row that is no longer relevant according to the current
classification, the configured relevance-loss policy decides what happens.

== Relevance-Loss Policy

A row can lose relevance between workbook generations.

Example:

* `PKG-BILL / JP` was classified as `SaaS`;
* the SaaS detail view contained hosting fields;
* later the classification changes from `SaaS` to `OnPrem`;
* the SaaS detail fields may no longer be relevant.

This must be explicit.

Possible policies:

[source,text]
----

preserve:
Keep existing specialized field values in the full detail table even when
the row no longer appears in the derived view.

clear:
Clear configured specialized fields when the row loses relevance.

archive:
Move or copy the old specialized values into an audit/history frame before
clearing them.

fail:
Abort reimport if previously maintained specialized values would become
irrelevant without an explicit user decision.
----

The safe default should not be silent destructive clearing. Destructive cleanup
must be configured explicitly.

== Overlapping Views And Conflict Handling

A row may appear in several specialized views.

Example:

[source,text]
----

PKG-BILL / JP is:
delivery_model = SaaS
market_id      = JP
----

It may therefore appear in both:

* `saas_package_market_details`;
* `japan_package_market_details`.

This is allowed.

However, overlapping views can edit the same target field. That requires an
explicit conflict rule.

Possible policies:

[source,text]
----

disjoint_columns:
Views are only allowed to edit non-overlapping column sets.

same_value_ok:
If two views write the same value to the same field, accept it.
If they write different values, fail.

explicit_precedence:
A configured precedence order decides which view wins.

fail_on_overlap:
Any overlapping editable target field is an error.
----

The first implementation should prefer `disjoint_columns` or `same_value_ok`
over automatic precedence. Silent winner selection would make the workbook
roundtrip hard to trust.

== Base View And Specialized Views

The workbook may show a base view that contains only common fields:

[source,text]
----

base_package_market_details:
package_id | market_id | owner | lifecycle_status
----

Specialized columns may be hidden from the base view and maintained only in
derived views:

[source,text]
----

saas_package_market_details:
package_id | market_id | hosting_provider | tenant_model

japan_package_market_details:
package_id | market_id | local_contact
----

The full detail frame still contains all fields.

The base view and specialized views are workbook surfaces over the same
underlying detail data. They must be merged through explicit identity and
column-target rules.

== Functional Boundary

Conditional Follow-Up Table Views are not:

* live spreadsheet filters;
* dynamic insertion or removal of workbook sections while the workbook is open;
* a spreadsheet UI framework;
* a generalized workflow engine;
* a replacement for XRef Crosstable;
* a replacement for Cell Codec;
* a hidden business-rule inference system.

They are a deterministic projection pattern between workbook cycles.

The classification is maintained first. The derived views are generated later.

== Relationship To XRef Crosstable

XRef Crosstable answers:

[source,text]
----

## Which relation tuple does this matrix cell address?
----

Conditional Follow-Up Table Views answer:

[source,text]
----

Given the classified relation tuples after reimport,
which detail rows and columns should be exposed in the next workbook?
----

XRef produces or updates the classification relation.

Conditional Follow-Up Table Views consume that relation.

== Relationship To Cell Codec

Cell Codec answers:

## [source,text]

## How does one compact workbook cell encode several structured attributes?

Conditional Follow-Up Table Views may consume attributes decoded by Cell Codec,
for example `delivery_model`, `language`, or `auth_mode`.

It does not parse compact cell values itself.

== Relationship To Derived Table Views

A derived table view is a workbook-facing table surface over a larger full
detail table.

Conditional Follow-Up Table Views are a specific derived-table-view pattern
where row inclusion is controlled by classification data from an earlier
roundtrip.

The implementation should avoid a new projection universe if existing
extraction, filtering, joins, role-based projection, and workbook-view mapping
can express the first useful slice.

== Non-Goals

This concept does not:

* implement live filtering in Excel or Calc;
* rely on protected cells as a security boundary;
* infer relevant rows from labels, layout, or naming conventions;
* infer business rules from existing data;
* create hidden canonical state from a workbook view;
* silently drop maintained specialized values;
* resolve overlapping view conflicts heuristically;
* support nested object or list mapping;
* become part of MVP.

== MVP Disposition

This is not MVP.

For now, it should remain a future / exploratory pattern unless a concrete
consumer workflow proves that existing extraction, filtering, joins, and
workbook-view mapping are insufficient.

The immediate modeling value is boundary clarification:

* XRef and Cell Codec handle the first classification workbook.
* Conditional Follow-Up Table Views handle later specialized maintenance views.
* The workbook remains static during editing.
* The next generated workbook reflects the reimported classification.
* Writeback into the full detail frame is explicit, identity-based, and
  deterministic.
