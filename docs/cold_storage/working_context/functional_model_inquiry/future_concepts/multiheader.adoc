= Nested Object Fields And Multi-Row Headers
:toc:
:toclevels: 3

== Status

Functional-model inquiry.

This note records a future-facing boundary for nested object fields in tabular
workbook surfaces.

It should not introduce a broad new transformation family. The main distinction
is between:

* structural path mapping between nested records and tabular columns;
* workbook header rendering of those paths.

Multi-row headers are primarily a workbook presentation capability. They are
not, by themselves, a domain transformation.

== Purpose

Structured input data often contains nested objects.

Example:

## [source,json]

{
"customer_name": "Max Muster",
"address": {
"street": "Musterstraße",
"city": "Musterhausen"
}
}
-

A spreadsheet surface cannot directly show this nested object shape inside a
normal table without choosing a tabular representation.

The project already supports the simple and important representation:

## [source,text]

customer_name | address.street | address.city
Max Muster    | Musterstraße   | Musterhausen
---------------------------------------------

This flat-path representation must remain supported.

A future workbook presentation may render the same path structure as multi-row
headers:

## [source,text]

customer_name | address       | address
| street        | city
Max Muster    | Musterstraße  | Musterhausen
--------------------------------------------

The underlying information is the same. Only the workbook-facing header layout
changes.

== User Problem

Users may need to maintain data that naturally contains small nested objects:

* customer address;
* contact information;
* product metadata;
* technical endpoint configuration;
* ownership blocks;
* regulatory attributes grouped by topic.

Flat path columns are precise and easy to process, but they become visually
dense when many fields share the same parent object:

## [source,text]

## address.street | address.city | address.postal_code | address.country

A multi-row header can make the grouping visible without changing the structured
meaning:

## [source,text]

address       | address | address     | address
street        | city    | postal_code | country
-----------------------------------------------

The functional need is not a new data model. The need is a readable workbook
surface for structured field paths.

== Functional Boundary

This inquiry separates two concerns.

=== Nested Object Path Mapping

Nested object path mapping is a structural edge transformation between nested
records and flat tabular columns.

## [source,text]

nested object record
<->
flat tabular record with path columns
-------------------------------------

Example:

## [source,json]

{
"customer_name": "Max Muster",
"address": {
"street": "Musterstraße",
"city": "Musterhausen"
}
}
-

maps to:

## [source,text]

customer_name | address.street | address.city
Max Muster    | Musterstraße   | Musterhausen
---------------------------------------------

This is a real data-shape transformation at the boundary between structured
records and tabular frames.

=== Multi-Row Header Rendering

Multi-row headers are a workbook rendering of the same column paths.

## [source,text]

address.street
address.city
------------

may be rendered as:

## [source,text]

address | address
street  | city
--------------

This does not create new domain semantics. It only changes how the existing
path structure is displayed in the workbook.

== Transformation: Flatten Nested Object Records

The forward structural transformation flattens nested object fields into
tabular path columns.

=== Information Consumed

* Nested input records.
* Configured path separator, for example `.`.
* Optional escaping or collision policy for field names that contain the
  separator.
* Optional list of included or excluded paths.
* Optional column order.

=== Information Produced

* Flat tabular records.
* Column names representing object paths.
* Diagnostics for unsupported shapes, path collisions, or ambiguous field names.

=== Example

## [source,json]

{
"customer_name": "Max Muster",
"address": {
"street": "Musterstraße",
"city": "Musterhausen"
}
}
-

produces:

## [source,text]

customer_name | address.street | address.city
Max Muster    | Musterstraße   | Musterhausen
---------------------------------------------

== Transformation: Unflatten Path Columns

The reverse structural transformation reconstructs nested object records from
flat path columns.

=== Information Consumed

* Flat tabular records.
* Configured path separator.
* Optional escaping or collision policy.
* Missing-value policy.
* Empty-object policy.

=== Information Produced

* Nested object records.
* Diagnostics for path conflicts, unsupported shapes, or ambiguous empty
  structures.

=== Example

## [source,text]

customer_name | address.street | address.city
Max Muster    | Musterstraße   | Musterhausen
---------------------------------------------

produces:

## [source,json]

{
"customer_name": "Max Muster",
"address": {
"street": "Musterstraße",
"city": "Musterhausen"
}
}
-

== Workbook Presentation: Multi-Row Headers

A workbook may display flat path columns as multi-row headers.

This is a presentation decision, not a separate domain transformation.

Example path columns:

## [source,text]

customer_name
address.street
address.city
address.country
---------------

may be rendered as:

## [source,text]

customer_name | address | address | address
| street  | city    | country
-----------------------------

The visible workbook groups related fields while preserving the same underlying
path identity.

A read path may reconstruct the flat path columns from the multi-row header,
provided the header structure is unambiguous and the configured separator or
path mapping is known.

== Separator And Collision Policy

The path separator should be configurable.

Examples:

## [source,text]

address.street
address/street
address__street
---------------

A separator policy must define what happens when source field names contain the
separator.

Possible policies:

## [source,text]

fail:
Reject field names that contain the configured separator.

escape:
Escape separator characters in source field names.

explicit_mapping:
Use configured path-to-column mappings instead of relying only on separator
parsing.
--------

The simplest safe default is `fail`.

== Bijectivity Boundary

For nested objects without lists, flattening and unflattening can be
deterministic and reversible under explicit rules.

Required constraints:

* no nested lists;
* no duplicate path columns;
* no ambiguous separator usage;
* deterministic missing-value handling;
* deterministic empty-object handling;
* stable column-path mapping.

Under these constraints, the same nested structure can be converted to flat
columns and reconstructed again.

This does not mean every possible JSON shape is supported. It only covers
records whose nested structure can be represented as one tabular row.

== Lists Are Out Of Scope

Nested lists are not part of this concept.

Example:

## [source,json]

{
"customer_name": "Max Muster",
"addresses": [
{
"type": "private",
"city": "München"
},
{
"type": "work",
"city": "Nürnberg"
}
]
}
-

This cannot be handled by simple multi-row headers alone.

Lists require a separate modeling decision, for example:

* child frame / relation table;
* repeated column groups;
* compact cell representation;
* unsupported shape with fail-fast diagnostics.

That future problem should not be hidden inside the Multi-Row Header concept.

== Non-Goals

This concept does not:

* introduce a general JSON normalization framework;
* support nested lists;
* infer nested structure from visual formatting alone;
* create a new workbook projection model;
* replace XRef Crosstable;
* replace Cell Codec;
* define object identity for nested collections;
* require multi-row headers for all nested fields;
* remove support for flat path headers.

== MVP Disposition

Flat path headers remain the baseline.

## [source,text]

address.street
address.city
------------

Multi-row headers are a future workbook presentation improvement.

The active model should treat nested object path mapping as a small structural
edge transformation and multi-row headers as a rendering/readback capability
over the same path structure.

The immediate modeling value is boundary clarification:

* flat path columns remain supported;
* multi-row headers may improve readability;
* the structural path identity remains the same;
* nested lists are explicitly out of scope.
