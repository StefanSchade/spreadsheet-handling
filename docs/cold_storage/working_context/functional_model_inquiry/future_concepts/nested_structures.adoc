= Repeating Nested Structures In Tabular Workbooks
:toc:
:toclevels: 3

== Status

Functional-model inquiry.

This note records future-facing requirements for nested lists / repeating
structures in structured input data.

It extends the earlier boundary for nested object fields:

* nested objects without lists can be mapped to path columns;
* nested lists require an additional representation decision.

This is not MVP.

The purpose of this note is to keep the future problem visible without forcing
it into the current MVP model or into unrelated concepts such as XRef,
Multi-Row Headers, or ordinary FK helpers.

== Purpose

Structured input formats such as JSON or XML can represent nested repeating
structures naturally.

Example:

## [source,json]

{
"customer_name": "Max Muster",
"addresses": [
{
"type": "private",
"street": "Musterstraße",
"city": "Musterhausen"
},
{
"type": "work",
"street": "Bürostraße",
"city": "München"
}
],
"bank_accounts": [
{
"iban": "DE001",
"bank": "Bank A"
},
{
"iban": "DE002",
"bank": "Bank B"
}
]
}
-

A spreadsheet does not have a native nested-list structure. Therefore the system
must choose an explicit tabular representation strategy.

The chosen strategy must be configured. It must not be inferred from layout,
labels, or accidental repeated values.

== User Problem

Users may need to maintain entities with one or more repeating child
structures:

* a customer has several addresses;
* a customer has several bank accounts;
* a product has several endpoints;
* a package has several deployment targets;
* a configuration item has several localized labels;
* a project role has several assigned people.

Flat path columns work for single nested objects:

## [source,text]

## customer_name | address.street | address.city

They do not work directly for repeating children:

## [source,text]

## customer_name | addresses[].street | addresses[].city

The system must decide how the repeated child rows are represented in a
workbook and how edited workbook data is reconstructed.

== Functional Boundary

Repeating nested structures are not the same as ordinary FK lookup relations.

An owned nested list means:

## [source,text]

## parent record owns several child entries

Example:

## [source,text]

## customer Max Muster owns two address entries

This may be represented as a child table, but the relationship is part of the
source object's nested structure. The child table's parent reference is not
necessarily a primary key. Several child rows may have the same parent
reference.

This is different from an ordinary normalized FK relation where one frame
references another independent entity catalog.

The functional question is:

## [source,text]

How should an owned repeating child structure be exposed and reconstructed
through tabular workbook surfaces?
----------------------------------

== Representation Strategy 0: Cell-Embedded List

The simplest representation stores the list inside one cell.

A first implementation may use a JSON snippet:

## [source,text]

customer_name | addresses
Max Muster    | [{"type":"private","city":"Musterhausen"},{"type":"work","city":"München"}]
-------------------------------------------------------------------------------------------

This is technically simple and works for backends that only support flat rows
and scalar cells.

Possible future variants:

* JSON cell text;
* YAML-like cell text;
* delimiter-based compact notation;
* custom cell codec for simple repeated values.

=== Advantages

* Works in simple tabular formats such as CSV.
* Avoids row multiplication.
* Avoids additional sheets or named ranges.
* Can preserve arbitrary nested list structure if JSON is used.

=== Disadvantages

* Poor spreadsheet usability.
* Hard to validate at cell level.
* Hard to filter, sort, or compare child entries.
* Editing is error-prone unless tooling or validation improves.

=== Boundary

This strategy is suitable as a fallback or early technical representation. It
should not be presented as the preferred human-editing surface for complex
child records.

== Representation Strategy 1: Repeated Parent Rows

A repeating child list may be represented by multiple table rows.

Example:

## [source,text]

customer_id | customer_name | address.type | address.city
C-001       | Max Muster    | private      | Musterhausen
C-001       | Max Muster    | work         | München
----------------------------------------------------

The parent fields are repeated for each child row.

This works reasonably when one repeating child list is expanded in a table.

=== Consistency Problem

Repeated parent fields now appear multiple times.

The system must define what happens when the user edits repeated parent values
inconsistently.

Example:

## [source,text]

customer_id | customer_name | address.type | address.city
C-001       | Max Muster    | private      | Musterhausen
C-001       | Max Mustermann| work         | München
----------------------------------------------------

Possible policies:

## [source,text]

fail_on_conflict:
Parent fields repeated across child rows must have identical values.

first_wins:
The first row is authoritative; later conflicting values are ignored or
warned about.

last_wins:
The last row is authoritative.

readonly_parent_columns:
Repeated parent columns are displayed but not intended for editing.

warn_only:
Inconsistent repeated values produce diagnostics but do not block reimport.
---------------------------------------------------------------------------

The safest baseline is `fail_on_conflict` or `readonly_parent_columns`.

=== Limitation

This strategy is only safe for one expanded repeating list in a given table.

If two independent child lists are expanded into rows at the same time, a
Cartesian-product problem appears.

Example:

## [source,text]

2 addresses
5 bank accounts
---------------

A naive row representation would produce 10 rows if both lists are expanded
together, even though addresses and bank accounts are independent.

This must be avoided unless a concrete use case explicitly requires a
cross-product structure.

== Representation Strategy 2: Repeated Rows With Merged Parent Cells

Repeated parent rows may be visually optimized by merging the parent cells.

Example:

## [source,text]

customer_id | customer_name | address.type | address.city
C-001       | Max Muster    | private      | Musterhausen
|               | work         | München
----------------------------------------

In a spreadsheet, the repeated parent values may be displayed as merged cells.

=== Advantages

* Reduces visual repetition.
* Makes the parent-child grouping obvious.
* Prevents some inconsistent edits in merged parent fields.

=== Disadvantages

* Merged cells can make filtering and sorting worse.
* CSV cannot represent merged cells.
* Some spreadsheet users find merged cells difficult to edit.
* Readback must define how merged-cell values are propagated to child rows.

=== Boundary

Merged parent cells are presentation, not domain logic.

The logical representation remains repeated rows. Merging only changes workbook
display and editing ergonomics.

== Representation Strategy 3: Extracted Child Table

A repeating child list may be extracted into a separate child table.

Example parent view:

## [source,text]

customer_id | customer_name | address_ref
C-001       | Max Muster    | ADDR-C-001
C-002       | Sara Sommer   | ADDR-C-002
----------------------------------------

Example child table:

## [source,text]

address_ref | address_type | street        | city
ADDR-C-001  | private      | Musterstraße | Musterhausen
ADDR-C-001  | work         | Bürostraße   | München
ADDR-C-002  | private      | Sommerweg    | Augsburg
----------------------------------------------------

The reference identifies the owned child collection. It is not a primary key of
the child row. Several child rows may share the same reference.

The child table may appear:

* on the same sheet as a separate named range;
* on a separate sheet;
* in a generated detail workbook;
* in a backend-specific representation.

The placement is presentation. The domain concept is the extracted owned child
table.

=== Advantages

* Avoids Cartesian products between independent child lists.
* Supports several independent child collections.
* Preserves a clean parent row.
* Allows child rows to be edited as normal table rows.
* Scales better than repeated parent rows when many child lists exist.

=== Disadvantages

* Users must navigate between parent and child surfaces.
* The workbook needs clear grouping, labels, or references.
* The system must generate stable collection references.
* It feels more relational than the original nested object.

=== Boundary

This is still an owned nested structure, not necessarily an ordinary normalized
domain relation.

The extracted child table is a workbook-compatible representation of a nested
list. It should not automatically imply that the child records are independent
entities.

== Mixed Strategies

Different repeated fields mayDifferent use different strategies.

Example:

## [source,text]

addresses:
repeated parent rows

bank_accounts:
extracted child table
---------------------

This must be configured by field path, not per individual record.

Allowed:

## [source,yaml]

nested_lists:
addresses:
strategy: repeated_rows
bank_accounts:
strategy: extracted_child_table
-------------------------------

Not allowed:

## [source,text]

Max Muster addresses use repeated rows,
Sara Sommer addresses use embedded JSON,
unless a special migration/import feature explicitly supports that.
-------------------------------------------------------------------

The representation strategy is part of the data-shape contract for a field.

== Backend Capability Boundary

Not every backend can represent every strategy equally well.

Structured backends such as JSON or XML can naturally represent nested lists.

Spreadsheet backends require explicit tabular strategies.

CSV has even stronger limits. It can usually support:

* cell-embedded list;
* repeated parent rows.

CSV cannot represent:

* multiple sheets;
* named ranges;
* merged cells;
* workbook-local presentation structures.

Therefore the system must not claim that every backend can represent every
nested-list strategy.

Unsupported strategy/backend combinations should fail clearly.

Example diagnostics:

## [source,text]

CSV backend does not support extracted child table placement on separate sheets.

CSV backend does not support merged parent cells.

Spreadsheet backend requires a configured representation strategy for nested
list path 'bank_accounts'.
--------------------------

== Multiple Lists And Cartesian Product Avoidance

Multiple independent lists under the same parent must not be expanded into one
table by accident.

Example:

## [source,json]

{
"customer_name": "Max Muster",
"addresses": [
{"city": "Musterhausen"},
{"city": "München"}
],
"bank_accounts": [
{"iban": "DE001"},
{"iban": "DE002"},
{"iban": "DE003"},
{"iban": "DE004"},
{"iban": "DE005"}
]
}
-

The system must not silently generate:

## [source,text]

## 2 addresses x 5 bank accounts = 10 combined rows

unless the user explicitly configured a cross-product representation.

The safe rule:

## [source,text]

A repeated-row representation may expand at most one list path in a table.
Additional independent list paths must use another strategy such as extracted
child table or cell-embedded list.
----------------------------------

This avoids false relationships between unrelated child entries.

== Identity And Ordering

Repeating child entries may need stable identity and ordering.

Possible cases:

* child entries have no explicit ID;
* child entries have a natural key, such as address type;
* child entries have a generated technical ID;
* child entries are ordered lists where position matters.

The representation strategy must define how edited rows are reconstructed:

[source,text]
----

identity:
generated_id | natural_key | row_position | configured_key_fields

ordering:
preserve_input_order | explicit_order_column | sort_by_configured_fields
----

For early scope, an explicit order column or stable child ID is safer than
trying to infer intent from row order.

== Missing, Empty, And Deleted Child Entries

The read path must distinguish:

* no child list present;
* child list present but empty;
* child row present with empty fields;
* child row deleted by the user;
* child reference removed from parent row.

These cases may not all be representable in every backend or strategy.

The concept should require explicit policies rather than relying on accidental
blank-cell behavior.

Possible policies:

[source,text]
----

empty_child_rows:
drop | preserve_as_empty_object | fail

missing_child_table:
treat_as_empty | preserve_existing | fail

deleted_child_row:
remove_from_list | mark_deleted | fail
----

The safe baseline should fail on ambiguous destructive cases.

== Relationship To Nested Object Fields

Nested object fields without lists can be represented by flat path columns or
multi-row headers.

Repeating nested structures require additional list representation strategy.

[source,text]
----

address.street:
nested object path column

addresses[].street:
repeating nested structure requiring list strategy
----

Multi-row headers may improve display of child fields, but they do not solve
the list problem by themselves.

== Relationship To XRef Crosstable

XRef Crosstable owns matrix projection and recomposition for relation tuples.

Repeating nested structures own projection and reconstruction of child entries
inside a parent object.

They may look similar when represented as separate tables, but the semantics
are different:

[source,text]
----

XRef:
relation between two axes

Nested list:
owned repeated child structure of one parent record
----

== Relationship To FK Helpers

FK helpers display referenced entity labels for user convenience.

A nested child table reference such as `ADDR-C-001` is not automatically an FK
to an independent catalog. It may be a generated collection reference used to
group owned child rows.

Therefore FK-helper semantics should not be reused blindly.

== Non-Goals

This concept does not:

* become part of MVP;
* support arbitrary JSON shapes immediately;
* infer list representation from workbook layout;
* silently generate Cartesian products;
* require all backends to support all strategies;
* treat owned child rows as independent FK entities by default;
* require merged cells;
* require named ranges;
* solve full object identity management for all nested structures;
* define a live spreadsheet UI.

== MVP Disposition

This is not MVP.

Near-term support should continue to fail clearly for nested lists unless a
specific representation strategy is implemented and configured.

The functional model should keep the boundary explicit:

* nested objects without lists can use path columns;
* nested lists require an explicit strategy;
* backend capabilities constrain available strategies;
* repeated-row expansion is limited to one list path per table unless a
  cross-product is explicitly configured;
* extracted child tables are likely the most scalable strategy for multiple
  independent lists.

== Nested Repeating Structures

Repeating structures may themselves contain nested repeating structures.

Example:

## [source,json]

{
"customer_name": "Max Muster",
"addresses": [
{
"type": "private",
"city": "Musterhausen",
"bank_accounts": [
{
"iban": "DE001",
"bank": "Bank A"
},
{
"iban": "DE002",
"bank": "Bank B"
}
]
}
]
}
-

This means the concept must not assume that all repeating lists live directly
under the root record.

A nested list path may be:

[source,text]
----

addresses[]
addresses[].bank_accounts[]
addresses[].contacts[]
contracts[].payment_schedules[].rates[]
----

The same representation strategies may apply recursively, but only when the
configuration makes the parent-child ownership path explicit.

The system must not infer such nesting from table names, visual indentation, or
similar labels.

== Validation And Topology Checks

Nested repeating structures require validation beyond ordinary column checks.

At minimum, future support should consider:

[source,text]
----

unknown_path:
configured list path does not exist in the structured source model

unsupported_backend_strategy:
selected representation cannot be rendered by the chosen backend

multiple_repeated_row_lists:
more than one independent list is expanded into the same repeated-row table

ambiguous_parent_reference:
child rows cannot be assigned to one parent collection

duplicate_child_identity:
child identity is not unique within its configured parent collection

missing_required_parent:
child table contains rows whose parent collection reference is missing

cycle_detected:
configured extraction / reconstruction rules form a loop

recursive_depth_exceeded:
nested list expansion exceeds configured depth limits

conflicting_strategy:
two configured strategies target the same list path or overlapping paths
----

Loop checks matter because nested-list configuration can accidentally describe
a reconstruction cycle.

Example:

[source,text]
----

## customers -> addresses -> customers
----

or:

[source,text]
----

A contains B
B contains C
C contains A
----

Such cycles must fail before data is written.

A future implementation should treat nested-list mappings as a configured
topology. The topology should be validated before export or reimport.

== Depth Boundary

The model should allow nested lists conceptually, but implementation slices may
set explicit depth limits.

Example first slices may support only:

[source,text]
----

root record
-> one repeated child list
----

or:

[source,text]
----

root record
-> repeated child list
-> repeated grandchild list
----

A depth limit is acceptable as long as it is explicit and produces a clear
diagnostic when exceeded.

The important rule is that unsupported nesting fails loudly instead of being
flattened into accidental Cartesian products or silently dropped.
