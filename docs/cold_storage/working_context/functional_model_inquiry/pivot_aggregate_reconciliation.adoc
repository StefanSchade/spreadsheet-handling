= Pivot Aggregate  Reconciliation
:toc:
:toclevels: 3

== Purpose

This note records a functional-model inquiry for workflows where users edit
aggregated planning values in a workbook and expect the system to translate the
change into explicit follow-up demand, adjustment rows, or allocation tasks.

The concept is larger than `pivot_frame`.

`pivot_frame` may be one technical view-building primitive inside this workflow,
but it is not the whole functional concept. The functional concept is the
controlled reconciliation of edited aggregate values against a known baseline.

This is intentionally not part of the MVP scope. It is recorded as a future or
exploratory capability so the model can distinguish it from XRef Crosstable and
from ordinary projection-only pivot views.

== User Problem

Planning workflows often expose capacity, effort, budget, or demand as
aggregated values across time periods or other planning dimensions.

A user may not want to edit every contributing detail row. Instead, the user may
need to say:

* this project stream needs more capacity in July;
* this product area needs less effort in Q3;
* this cost block must be increased for a later planning cycle;
* this time period needs an additional unallocated demand item.

== Functional Boundary

Aggregate Planning Reconciliation is distinct from XRef Crosstable.

XRef Crosstable:

## [source,text]

Aggregate Planning Reconciliation:

## [source,text]

In XRef, editing a matrix cell sets, removes, or updates the relation entry for
that address.

In Aggregate Planning Reconciliation, editing a matrix cell changes the target
aggregate. The system computes the difference to the baseline and materializes
that difference explicitly.

The concept is also distinct from plain `pivot_frame`.

Plain `pivot_frame` is a projection or reshaping primitive. It can produce a
human-readable wide view from long-form data. By itself it does not define
editable aggregate semantics, delta creation, resource allocation, or follow-up
planning.

Aggregate Planning Reconciliation may use a pivot-like view, but its functional
meaning is reconciliation, not reshaping.

== Transformation 1: Build Aggregate Planning Projection

The forward transformation derives a workbook-facing aggregate planning view
from source detail frames or existing planning frames.

=== Information Consumed

* Source detail frame or planning frame.
* Grouping dimensions, for example `stream`, `period`, `team`, `product`,
  `market`, or `cost_center`.
* Value field to aggregate, for example `planned_pt`, `capacity_days`,
  `budget_amount`, or `demand_units`.
* Aggregation function, initially likely `sum`.
* Optional configured axis order, period list, stream list, or other scope
  declaration.
* Optional lifecycle metadata marking the generated frame as an aggregate
  planning projection.

=== Information Produced

* Workbook-facing aggregate planning frame.
* Baseline values needed for later reconciliation.
* Optional trace metadata explaining which source frame, grouping dimensions,
  value field, and aggregation policy produced the projection.

== Transformation 2: Reconcile Edited Aggregate Projection

The reverse transformation compares the edited aggregate planning frame against
the baseline and produces explicit reconciliation output.

It must not silently mutate the contributing detail rows.

=== Information Consumed

* Edited aggregate planning frame.
* Baseline aggregate values from the projection step or from a deterministic
  recomputation against the same source frame.
* Grouping dimensions and value field.
* Aggregation function.
* Reconciliation policy.
* Target frame for generated demand, adjustment, or follow-up rows.
* Optional identity-generation policy for newly materialized rows.

=== Information Produced

* Explicit delta rows, adjustment rows, demand rows, or follow-up planning rows.
* Diagnostics for invalid edits, missing addresses, ambiguous axes, or
  unsupported aggregation functions.
* Optional trace metadata linking each generated row to its aggregate address
  and edited value.

== Soft Bijectivity And Determinism

This workflow is not bijective in the strict mathematical sense.

When an aggregate cell changes from `25` to `35`, there are many possible detail
states that could explain the new total:

## [source,text]

The system cannot infer which of these possibilities the user intended.

However, the project sometimes uses a softer operational meaning of
"bijective": given the same input, configuration, and edited workbook, the tool
must always produce the same deterministic, user-explainable result among the
possible valid results.

In that softer sense, Aggregate Planning Reconciliation can be treated as a
controlled roundtrip pattern:

## [source,text]

The output is not the only theoretically possible interpretation of the edit.
It is the configured interpretation.

This is similar to other controlled reconstruction situations where redundancy
is merged. Merging redundant information may also have more than one possible
resolution, but the framework requires a deterministic and explainable policy
so the user can understand what will happen before trusting the roundtrip.

== Relationship To XRef Crosstable

Aggregate Planning Reconciliation should not be folded into XRef Crosstable.

XRef owns matrix addressing and relation recomposition.

Aggregate Planning Reconciliation owns edited aggregate targets and
delta/materialization semantics.

The same visible workbook shape may look similar, but the meaning is different.

[cols="1,2,2", options="header"]
|===
|Aspect
|XRef Crosstable
|Aggregate Planning Reconciliation

|Cell meaning
|One addressed relation tuple
|Aggregate over contributing rows

|Edit effect
|Set, remove, or update relation entry
|Compute delta against baseline

|Reverse path
|Recompose matrix into target relation
|Materialize adjustment, demand, or follow-up rows

|Blank semantics
|Configured matrix blank/delete/no-update semantics
|Requires explicit reconciliation policy

|Detail rows
|Usually not contributing aggregate details
|Remain unchanged unless a later workflow edits them

|Follow-up workflow
|Optional
|Central: allocation, distribution, or resolution of generated demand
|===


== Relationship To Conditional Follow-Up Projection

This concept is related to conditional follow-up workflows.

An edited aggregate value may create follow-up work:

* find additional resources;
* allocate unassigned capacity;
* create task placeholders;
* request approvals;
* split a demand delta across teams or streams.

The reconciliation transform should only materialize the explicit need. The
later allocation or workflow resolution should remain a separate concept unless
a concrete use case proves a tighter coupling is necessary.

== Non-Goals

This concept does not:

* implement resource allocation;
* define a scheduling optimizer;
* replace XRef Crosstable;
* introduce live spreadsheet UI logic;
* infer axes, periods, or grouping dimensions from workbook layout;

== MVP Disposition

This is not MVP.

Aggregate Planning Reconciliation should be recorded as a future functional
pattern that may become relevant once the project explicitly supports planning,
allocation, or aggregate-edit workflows.
