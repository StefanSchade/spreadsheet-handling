= Hard Reset / Greenfield Core

== Description

Create a new core implementation and treat the existing implementation mainly
as a reference source. The new core would redesign domain transformations,
metadata, and possibly adjacent architecture from the Functional Model upward.

== Pros

* Maximum design freedom.
* No need to preserve old internal names or module boundaries.
* Can remove accumulated sediment in one conceptual move.
* Breaking changes are acceptable because the project has little external
  adoption.

== Cons

* Discards too much embedded knowledge from adapters, renderers, parsers,
  fixtures, and edge-case handling.
* Creates high risk of losing context that is expensive to rediscover.
* Encourages two parallel worlds during the transition.
* Artificial intelligence can make refactoring cheaper, but lost context more
  expensive.

== Assessment

This strategy is not recommended as the main path. It may be useful as a
thought experiment, but a full new core would throw away too much working
technical knowledge.
