= Domain/Meta Reset With Adapter Preservation

== Description

Preserve the expensive outer architecture where possible: adapters, rendering
boundary, parsers, CLI orchestration, and useful fixtures. Replace the
domain/meta interior behind a cleaner boundary.

== Pros

* Keeps technical knowledge that is costly to rediscover.
* Focuses the rewrite on the area with the highest conceptual debt.
* Avoids a full greenfield core.
* Allows the new domain/meta model to be tested through existing IO and render
  paths.
* Fits the observation that the Functional Model is simpler than the current
  domain/meta implementation.

== Cons

* Requires a carefully designed boundary between new domain/meta code and old
  adapters.
* Old adapter assumptions may leak into the new model if not actively guarded.
* Some adapter or renderer changes may still be necessary where old meta shapes
  are deeply embedded.
* Needs strong contract tests to distinguish preserved behavior from preserved
  sediment.

== Assessment

This is the best architectural boundary for the recommended path. It should be
combined with contract-first rewrite work rather than treated as a purely
mechanical module replacement.
