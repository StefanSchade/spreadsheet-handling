= Contract-First Domain/Meta Rewrite

== Description

Build executable integration contracts from the Functional Model, then rewrite
domain and meta internals against those contracts. Compatibility is not the
primary goal; preserving intended user behavior is.

== Pros

* Keeps user intent in front of implementation history.
* Allows breaking internal changes where they remove complexity.
* Reduces the risk of rebuilding old sediment under cleaner names.
* Gives each rewrite slice a clear pass/fail contract.
* Well suited to a young project with little external adoption.

== Cons

* Requires disciplined integration-test design before implementation work.
* Initial progress may feel slower because contracts come first.
* Poorly chosen contracts could freeze accidental behavior.
* Needs explicit non-goals so the first slice does not absorb the whole system.

== Assessment

This is the strongest main strategy. It treats the Functional Model as the
source of truth and lets the implementation change aggressively where the
contracts allow it.
