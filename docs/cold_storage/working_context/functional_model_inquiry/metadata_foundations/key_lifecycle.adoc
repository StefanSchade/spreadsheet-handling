== Purpose

This inquiry captures working context for later Functional Model work around
primary-key value lifecycle, projected relation writeback, XRef crosstable
scope semantics, zombie-record prevention, and future key-aware maintenance
operations.

It is intentionally not a Functional Model chapter yet. It records human
problem framing and possible policy axes so later work can promote stable parts
into the Functional Model and derive narrow implementation slices.

No implementation requirements are added by this note. No public YAML, CLI,
metadata schema, persistence-boundary, XRef behavior, FK-helper behavior, or
schema-maintenance behavior is approved by this note.

It asks what happens when values used as entity identity are edited, renamed, or
deleted while other frames, XRef matrices, FK references, relation tuple frames,
or future metadata still address them.

== Background

A primary-key value looks like an ordinary cell value in a spreadsheet, but it
is not ordinary once other data refers to it. Changing it is not merely editing
a scalar value. It may affect every relation, projection, helper, validation,
and metadata entry that uses that value as identity.

In a classic spreadsheet, a user may edit an identifier cell and manually fix
all visible formulas or references. In `spreadsheet-handling`, the spreadsheet
is an editing surface over frames and metadata. Some references may be hidden,
derived, represented in relation tuple frames, or only visible through a
projection such as an XRef crosstable.

Therefore, primary-key value edits require a different mental model from normal
payload edits.

[NOTE]
====
Key values are not ordinary payload values once other frames, projections, or
metadata address them.
====

== Working Vocabulary

=== Primary-Key Value

A primary-key value is a value in a configured or assumed key column that
identifies an entity row inside a frame.

This inquiry does not decide the final metadata shape for declaring primary
keys. It uses "primary key" in the problem-space sense: the value by which
other frames, relation tuples, projections, or metadata identify a record.

=== Key Lifecycle

Key lifecycle covers changes to key values over time:

* creation of a new key value;
* deletion of a key value;
* rename of a key value;
* rewrite of a key family by mapping or pattern;
* split or merge of key values;
* future changes to key structure, such as moving from one-column keys to
  compound keys.

This inquiry focuses mainly on rename and deletion.

=== Projected Relation

A projected relation is a relation whose maintenance surface is not simply the
canonical tuple frame itself.

Examples:

* an XRef crosstable projects relation tuple rows into matrix cells;
* a workbook view projects a selected subset of records;
* a relation maintenance view exposes only selected fields or selected rows;
* a future grouped or multi-axis projection may expose several axes at once.

Projected relations make key lifecycle harder because not every relevant tuple
or key may be visible at the same time.

=== Projection Scope

Projection scope is the set of records, axis members, columns, rows, and
relation addresses that were part of a particular rendered maintenance surface.

Projection scope answers questions such as:

* Which row-axis members did the user see?
* Which column-axis members did the user see?
* Which relation addresses were editable?
* Which addresses were outside this projection and therefore not being
  maintained by this roundtrip?

Projection scope is not the same as durable canonical truth. It may be
runtime/readback evidence needed to interpret writeback safely.

=== Writeback Scope

Writeback scope is the portion of canonical data that a reimport or reverse
pipeline is allowed to update based on the edited projection.

A projection may expose only a subset of records or relation addresses. Data
outside the writeback scope should generally be preserved unless an explicit
maintenance operation or policy says otherwise.

=== Zombie Record

A zombie record is an entity or key value that the user reasonably believes was
removed or renamed, but which is recreated or preserved by a defensive merge
because an old relation or metadata entry still references the old key.

Zombie behavior can happen when the system treats every dangling relation as
proof that the referenced entity must still exist.

== Problem Statement

When a user edits or renames a primary-key value in one visible table, other
relations may still carry the old value. If the reverse path merges those old
relations defensively, the old key can reappear.

The framework must eventually distinguish at least three situations:

* a key was outside the projection and no statement was made about it;
* a key was inside the projection and was intentionally deleted or renamed;
* a key appears in an old relation but no longer exists in the current entity
  table.

Without that distinction, two opposite failures are possible:

* *zombie resurrection* -- old relation data recreates an entity the user
  thought was gone;
* *unintended data loss* -- relations outside the edited projection are dropped
  because they were not visible in the current maintenance surface.

The Functional Model does not yet describe this area sufficiently. The XRef
crosstable work exposed the gap, but the issue is broader than XRef.

== Why Hidden Data Alone Does Not Solve This

One earlier idea was to carry additional original values or merge information in
hidden sheets or hidden metadata so merge conflicts can be avoided.

Hidden information can help with some tasks, especially:

* preserving projection scope;
* preserving original axis members;
* preserving workbook artifact identity;
* distinguishing generated metadata from user edits.

However, hidden information does not solve the core key-lifecycle problem.
If the user is expected to rename primary keys, the relevant original entity
tables and relation-bearing surfaces must remain understandable and editable in
the workflow. Otherwise the user cannot intentionally update all affected
references.

A hidden tuple table may preserve old relation rows, but it can also make key
renames more opaque. If a relation tuple frame still contains the old key while
the visible entity table shows the new key, writeback behavior becomes
surprising unless the system has an explicit key lifecycle policy.

== XRef Crosstable Example

Consider an XRef relation over two entity frames.

[source,text]
----
products
--------
product_id | label
SaaS       | Software as a Service

markets
-------
market_id | label
DE        | Germany

product_market_relations
------------------------
product_id | market_id | value
SaaS       | DE        | supported
----

The XRef crosstable may render:

[source,text]
----
product_id | DE
SaaS       | supported
----

Now the user renames `SaaS` to `Cloud` in the visible products table.

If the relation tuple frame still contains `SaaS`, the reverse path may see:

[source,text]
----
products
--------
product_id | label
Cloud      | Software as a Service

product_market_relations
------------------------
product_id | market_id | value
SaaS       | DE        | supported
----

A defensive merge rule might reason:

[source,text]
----
Relation references SaaS.
SaaS is missing from products.
To avoid data loss, recreate or preserve SaaS.
----

That creates a zombie record:

[source,text]
----
products
--------
Cloud
SaaS   <-- unexpectedly resurrected
----

From the user's perspective, this violates the expected rename/delete result.

== Scoped Anti-Zombie Posture

A plausible default for projected relation writeback is an anti-zombie policy:
inside the writeback scope, current entity/axis membership should control
whether old relation addresses survive.

A stronger but still scoped formulation:

[source,text]
----
Inside the writeback scope, current axis entity frames are authoritative for
whether projected relation addresses may survive.

Outside the writeback scope, absence from the current projection is not a
statement of deletion.
----

This avoids both obvious failure modes:

* old relation rows should not recreate keys that were removed inside the
  maintained scope;
* relation rows outside the maintained scope should not be dropped merely
  because they were not projected.

This note does not approve that default as runtime behavior. It records it as a
human-preferred starting posture for later Functional Model and slice work.

== Projection Scope Is Required For Correct Writeback

For XRef and similar relation projections, the reverse path cannot decide
correctly from the current visible matrix alone.

It may need to know:

* which row-axis members were originally projected;
* which column-axis members were originally projected;
* which row/column addresses were intended to be editable;
* which relation tuples existed before writeback;
* which entity keys currently exist in the source entity frames;
* which axes were intentionally limited to a subset.

This explains why runtime resolution fields such as materialized XRef
`column_keys` or resolved dense-axis members may not be mere sediment. They may
be projection/writeback scope evidence.

At the same time, such fields must not be promoted to durable canonical truth
without an explicit policy. They describe what a particular artifact projected,
not necessarily what the model is forever.

== XRef Missing Cell Versus Missing Row Or Column

The XRef empty-cell policy and the XRef missing-axis policy are separate.

An empty visible matrix cell means there is a matrix address, but its payload is
empty. The existing `drop_empty` policy controls whether such an empty cell
produces no relation row or an empty-valued relation row.

A missing row or missing column is different. It can mean several things:

* the row/column was outside the original projection scope;
* the row/column was part of the original projection and the user deleted it;
* the workbook structure was damaged;
* the corresponding entity was deleted;
* the corresponding entity was renamed;
* a different key was renamed into the missing/appearing position.

Therefore, XRef crosstable writeback needs a policy axis for missing projected
axis members, separate from `drop_empty`.

== XRef Scope Cases

The following cases are working analysis, not final contract text.

=== Axis Member Was Never In Projection Scope

If an axis member was not part of the original projection scope, its absence in
the edited matrix should normally make no statement about relations for that
axis member.

[source,text]
----
original_projection_scope: does not include US
edited_matrix:            does not include US
expected posture:         preserve existing US relations outside this writeback
                           scope unless another operation owns them
----

=== Axis Member Was In Scope And Still Visible

If an axis member was in scope and remains visible, matrix cells for that
member can be interpreted according to the XRef value and empty-cell policies.

[source,text]
----
original_projection_scope: includes DE
edited_matrix:            includes DE
entity table:             DE still exists
expected posture:         visible cells are writeback input for DE relation
                           addresses
----

=== Axis Member Was In Scope But Is Missing On Readback

If an axis member was in scope but is missing on readback, the system needs an
explicit policy.

Possible interpretations:

* user intentionally deleted the axis member from the projection;
* user deleted or renamed the underlying entity;
* workbook structure was damaged;
* projection was manually narrowed;
* the axis member should be treated as no longer having relations.

Possible policy directions:

* fail as structural change;
* warn/report and preserve old relations;
* prune in-scope relations for the missing axis member;
* treat as entity lifecycle event if the entity frame also changed.

=== Relation References Missing Axis Entity

If a relation tuple references a key value that no longer exists in the current
axis entity frame, possible policies include:

* drop the relation with a report;
* fail validation;
* preserve as orphaned relation;
* recreate the missing entity;
* defer to an explicit key-maintenance policy.

The anti-zombie posture prefers not recreating entities implicitly. If data is
dropped, it should be reported, not silently hidden.

=== Possible Rename

If an old key disappears and a new key appears, this may be a rename. It may
also be a delete plus an add.

Without stable identity or an explicit maintenance operation, rename is not
reliably provable.

Therefore, MVP should not depend on heuristic rename detection. Explicit
key-maintenance operations are the safer path when key continuity matters.

== Relation Tuple Tables And User Expectations

When an XRef crosstable is built from a relation tuple frame, the relation
tuple frame remains the recomposition target. If the user changes keys only in
one visible projection while old keys remain in the tuple frame, the reverse
path may violate user expectations.

A workflow can choose to expose the tuple frame, hide it, or treat it as an
internal frame. Each choice has consequences:

* exposing it gives power users a chance to update all references manually;
* hiding it reduces visible noise but can make key lifecycle opaque;
* treating it as internal requires stronger writeback policies and scope
  evidence.

For maintainability, the system should prefer explicit key-maintenance
operations over requiring users to manually synchronize every projected tuple
surface.

== Key Maintenance Operations

The framework should eventually provide explicit maintenance operations for
primary-key value lifecycle.

These operations are distinct from ordinary cell editing and from frame/column
schema maintenance. They operate on record identity and known references.

Candidate operation families:

* rename one key value;
* drop one key value;
* rewrite key values from an explicit mapping table;
* rewrite key values by controlled pattern or regex;
* dry-run/report before mutation;
* optionally cascade, fail, preserve, or report references depending on policy.

A key rename operation might conceptually perform:

[source,text]
----
products.id: SaaS -> Cloud

and update known references in:

* FK columns pointing to products.id;
* XRef relation tuple frames using products.id as an axis;
* schema-relevant metadata that addresses products.id values;
* future record-addressed metadata, if declared compatible with key rewrite.
----

Regex-based rewriting is powerful but risky. It should require explicit
configuration, deterministic reporting, and preferably dry-run support.

== Key Drop Operations

Dropping a key value is not only deleting a row from an entity frame. Other
frames may still reference that key.

Possible reference policies:

* `fail_if_referenced` -- refuse to drop while references remain;
* `drop_references` -- remove known references as part of the operation;
* `preserve_orphans` -- allow dangling references intentionally;
* `report_only` -- produce findings without mutation;
* `custom_policy` -- workflow-specific plugin or later extension.

A safe maintenance CLI may prefer dry-run/report-first behavior before applying
mutation. A later Functional Model should distinguish explicit maintenance
operation defaults from projected writeback defaults.

For example:

* explicit key maintenance may default to fail/report unless cascade is
  explicitly requested;
* projected XRef writeback may default to scoped anti-zombie drop/report.

This inquiry does not decide the final defaults.

== Maintenance While Data Rests In Structured Form

The intended safe path for large key changes is to run maintenance operations
while the data is at rest in structured form, such as JSON/YAML directories,
rather than by manual spreadsheet edits.

Example flow:

[source,text]
----
structured data in Git
-> run key maintenance dry-run
-> inspect report
-> apply key maintenance
-> commit data + updated schema-relevant metadata
-> run next spreadsheet export/import roundtrip
----

This matches the broader schema-maintenance principle: when identity continuity
matters, use explicit maintenance operations so data and metadata can be
rewritten together where possible.

== Metadata Addressing Records By Primary Key

Current metadata may not yet address individual records by primary-key value.
However, future metadata families may need to do so.

Examples:

* record-level highlighting;
* record-level review status;
* record-level comments or annotations;
* record-level protection or locking;
* relation-tuple highlighting;
* data-quality findings attached to a specific entity row;
* workflow state attached to a selected record.

Such metadata participates in key lifecycle. If a primary-key value is renamed
or dropped without rewriting the metadata, the record-attached property may be
lost, orphaned, or applied to the wrong conceptual entity.

Therefore, future metadata families that store record-addressed claims must
make their key-lifecycle behavior explicit.

They should declare whether entries are:

* rewritten on key rename;
* dropped on key drop;
* preserved as orphaned findings;
* ignored if the record is missing;
* cause warnings or validation failures;
* outside the scope of automatic maintenance.

== Developer Documentation Implication

The project should keep this issue visible when extending metadata.

Any future metadata family should be assessed for whether it references:

* frame names;
* column names;
* key columns;
* key values / record identities;
* relation tuple identities;
* workbook/sheet positions;
* projection scope or writeback scope.

If it references key values or record identities, it needs a lifecycle posture
for key rename/drop/rewrite. Otherwise maintenance operations cannot preserve
that metadata reliably.

This may later become a developer-guide checklist, registry field, or semantic
model rule. This inquiry only records the need.

== UX Consideration: Highlight Editable Relation Fields

Hiding relation-supporting data is not always safer. If the user is expected to
maintain keys and relations, hiding relevant identity information can make
rename/drop operations harder to understand.

A possible future UX direction is to highlight the intended editable fields in
relation maintenance workbooks rather than hiding every supporting field.

For an XRef crosstable, a future workbook might visually distinguish:

* row-axis key fields;
* column-axis key fields;
* editable cross cells;
* context-only labels;
* relation tuple fields that are not intended for direct edit.

This is not MVP. It is recorded because it supports transparent maintenance of
relations and key lifecycle.

== Policy Axes For Later Modeling

The following policy axes may be useful when promoting this inquiry into the
Functional Model.

=== Missing Projected Axis Member Policy

Possible values:

* fail as structural change;
* prune relations for the missing in-scope axis member;
* preserve old relations and report;
* ignore as outside scope only when the member was not projected;
* delegate to entity lifecycle policy.

=== Dangling Relation Policy

Possible values:

* drop relation with report;
* fail validation;
* preserve orphan relation;
* recreate missing entity;
* custom workflow policy.

=== Key Rename Policy

Possible values:

* explicit maintenance only;
* mapping-table rewrite;
* regex rewrite;
* heuristic detection later;
* no automatic rename support.

=== Key Drop Policy

Possible values:

* fail if referenced;
* cascade/drop references;
* preserve orphans;
* report only;
* scoped anti-zombie drop during projected writeback.

=== Projection Scope Persistence Policy

Possible values:

* do not persist scope evidence;
* store scope evidence only in workbook/carrier runtime meta;
* store scope evidence in sidecar meta;
* derive scope from explicit axis-domain intent;
* require configured scope for writeback.

This inquiry does not decide which policy values should become public flags or
internal behaviors.

== MVP Posture Candidate

The following is a candidate posture for later review, not an approved
contract:

* Primary-key value edits are identity lifecycle events, not ordinary payload
  edits.
* Key rename/drop should preferably be performed through explicit maintenance
  operations when references must be updated.
* Projected relation writeback should be scoped. Absence outside projection
  scope should not delete data.
* Inside writeback scope, current entity/axis membership should prevent zombie
  resurrection by default.
* Data dropped because of anti-zombie handling should be reported.
* Rename detection should be explicit, not heuristic, in MVP.
* Runtime resolution may be legitimate projection/writeback scope evidence, but
  it must not silently become durable canonical truth.
* Future PK-addressed metadata must declare key-lifecycle behavior.

== Summary

The XRef crosstable discussion exposed a broader record-identity problem.
When primary-key values are renamed or deleted, relation projections cannot be
merged safely unless the system distinguishes absence outside scope from deletion
inside scope.

Without that distinction, the system can either resurrect zombie records or
drop data the user never intended to touch.

The likely direction is explicit key-maintenance operations, scoped projected
writeback, anti-zombie defaults with reporting, and future developer guidance
for metadata that addresses records by primary key.
