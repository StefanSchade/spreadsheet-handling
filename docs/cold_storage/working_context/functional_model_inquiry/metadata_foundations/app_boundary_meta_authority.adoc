== Background

`spreadsheet-handling` is a framework in the sense of a toolkit of generic capabilities. Users can
orchestrate those capabilities into workflows that may be closer to further from
the use cases anticipated by the project maintainer.

It is the responsibility of the framework's users to:::
* assure the correctness implmeentation of the specific use case
* check that each pipeline step receives the necessary input
* reason about possible data inconsinstencies when data is imported over the apps boundary

It is the repsonsibility of the frameworks maintainer to:::
* provide patterns of the intended use of the framework
* provide deterministic buiding blocks that an implmeentation can be composed of.

It is not the responsibility of the framework's maintiner to:::
* prevent any inadeauate use of the framework
* forsee al possible use cases that the framework is used for
* provide perfect guards to prevent the user from harm

the usecases supported by the framework are transient in nature - they perform 
tranformations. the input of these may be generated beyond the application boundary,
it may be the output of the previeous transformation run or it may be the result of
user interaction.

== Working Vocabulary

=== Application Scope

An application scope is one concrete workflow run or roundtrip envelope. It may
include:

* one or more configured pipelines;
* structured input and output data;
* spreadsheet rendering;
* human spreadsheet editing;
* spreadsheet readback;
* reverse or recomposition pipelines;
* optional review and promotion steps.

The application scope is narrower than the universe of all files a user may
bring to the framework. It describes the bounded workflow in which data,
metadata, generated spreadsheets, and user edits are expected to belong
together.

=== Application Boundary

The application boundary is crossed when data or artifacts enter the workflow
from outside the current application scope.

Examples:

* structured data produced by another application;
* data pulled from Git;
* data produced by another instance or version of `spreadsheet-handling`;
* spreadsheet files created or modified outside the framework;

==== Metadata from outside the application boundary

metadata is intended to be used only within the applciation scope but the existence 
of an application boundary means that when taking in data form that boundary the 
application (and more so the framework) can not prevent certain inconsinstencies:

* metadata may actually be from within the application boundary (previeous run) but
the accompaniing data might be out of sync.
* metadata may be imported back and forth across the application boundary - so it might
be corrupted - this may even cause the framework to fail - the user should be made aware of 
this to help with debugigng.
* metadata might be outdated (written by a previous version of the framwork)

metadata cannot be easily edited by the user or third party applications (technically editable, 
but complex and not externally documented) It also is likely to change in any run. this makes it 
difficult for `git` as this implies potetnial merge conflicts that cannot be resoved easily. we 
strongly suggest that users gitignore _meta.yaml.

User designed workflows shoud think about cases where the data that is used as input for a
pipeline can be influenced from beyond the application boudary (e.g. by a `git pull`),
meta sidecars could be lost (fresh clone), stale (not representing the current data anymore) 
or actually consistent (when the data is indeed not changed since it was generated by one of
the users pipelines). In these cases _meta must not be the only source of essential information 
that is relied upon. The application boundary weakens the assumption that data and metadata still
belong together, regardless of whether the metadata was originally generated by
this framework. Incoming metadata must not be treated as authoritative merely because it exists.

These are the possible percautions

* design pipelines in such a way that the application boundary is not crossed
  before intended inverse transformations / recomposition steps of any projection
  have been performed
* gitignore _meta.yaml
* provide defaults for importand fileds by bootstrap transformation after reaading data from the applciation boundary

It also means (relevant for the development of spreadsheet-handling itself), that 
metadata must not store essential parts of the payload data as the only representation. If future 
features would decide to do this (for instance error state of a field stored in meta) we must provide 
the frameworks user with a transformation that extracts this information to proper data (Frames), so 
the user can act with the appropriate caution at the boundary of his application.

== Toolkit Principle

The framework provides composable capabilities. It should make their contracts
deterministic and reviewable, but it does not guarantee that every user-composed
pipeline is semantically appropriate for the user's domain.

The workflow builder owns orchestration responsibility:

* selecting transformation steps;
* supplying required bootstrap/configuration intent;
* deciding which persisted or readback metadata is acceptable;
* deciding how to handle external or stale artifacts at the appliation boundary;
* deciding whether a workflow needs review, promotion, sanitization, or
  validation gates.

The framework should provide the building blocks and make failure modes clear.
It should not silently invent domain policy where the workflow has not supplied
one.

== Input Artifact Classes

=== Structured Data Without Metadata

Structured data may enter through the application boundary without framework
metadata. In the description above we were discussing Git - but there is a range of potential sources:

* Git repositories;
* generated JSON/YAML/XML/CSV directories;
* web services (connected either via glue code or via a future adapter);
* other applications writing to the same file system;
* manual data edits.

Such data can be valid input if the workflow supplies the required intent by
bootstrap or configuration. The framework cannot infer every pipeline's
required metadata from arbitrary structured data.

=== Spreadsheet Data Without Metadata

Spreadsheet data without metadata may be accepted only in constrained cases.

The framework does not claim to read arbitrary externally authored spreadsheets
as semantically valid workflow artifacts. A simple spreadsheet may happen to be
usable as seed input, but that is not the same contract as reading a workbook
created by the framework for a known roundtrip.

A future capability may explicitly support seeding workflows from simple
spreadsheets. That should be modeled separately from the managed roundtrip
contract.

=== Tool-Generated Spreadsheets With Embedded Metadata

A spreadsheet file produced by an earlier run is not automatically compatible
with a later workflow, changed pipeline, or newer framework version. Readback is
safe only when the current workflow still expects the same workbook shape and
the embedded or sidecar metadata is still compatible with that shape.

This is the managed roundtrip case:

[source,text]
----
structured data + metadata/configuration
-> pipeline
-> generated workbook
-> human edit
-> readback
-> reverse/recomposition pipeline
----

Within such a scope, readback metadata may legitimately carry user-preserved
intent and projection-scope evidence.

Additionally, a workflow may contain more than one human editing step. A
multi-stage pipeline with two generated workbooks and two human edits can still
be a valid managed workflow when each readback step has a matching
reverse/recomposition path.

Other supported workflows may not aim to provide a human-editable interface at
all. They may extract data, generate reports, validate structured input, enrich
frames, or prepare projections without a symmetric back-and-forth roundtrip.


== Metadata Forms

=== `_meta.yaml` / Structured Sidecar Metadata

when the payload data frames are persisted in structured format (yaml, json, xml) the meta data frame is persisted as _meta.yaml.

=== Embedded Spreadsheet Metadata

when the payload data frames are persisted in a spreadsheet, the meta data frame is persisted as a hidden sheeet _meta.

=== Frame Schema Metadata

Frame schema metadata describes structural properties of frames, columns, and
relationships that are required to interpret payload frames correctly.

Examples include primary-key columns, foreign-key relationships, column roles,
required/optional markers, and future logical type information.

Frame schema metadata describes the data model. It is not payload data itself,
and it is not automatically authoritative merely because it is present in
`_meta`. A workflow may provide or reseed schema metadata from configuration,
bootstrap transformations, or trusted metadata, depending on the application
boundary and pipeline contract.


=== Presentation Metadata

Presentation metadata expresses display and workbook UX intent. Examples:

* cell formats;
* column widths;
* row heights;
* header styling;
* sheet-level presentation options;
* possibly filters, freeze panes, and related workbook layout state.

Presentation metadata can be written by the domain (especially bootstrap transformations to codify
a default layout) and by the read path of the spreadsheet adapter (to codify user settings as user
intent). conflicts between the two can be avoided as the bootstrap tranformation allows to specify
a priority so the user can communicate "use this format setting if there is no value in _meta already".

=== Model Metadata

Model metadata expresses structural or transformation intent. Examples:

* FK relation policy;
* helper provenance;
* XRef row/column/value configuration;
* validation intent;
* workbook-view configuration;
* projection or lifecycle policy.

Losing or misapplying model metadata can affect data integrity. It requires a
stronger authority model than presentation metadata.

=== Runtime Resolution and Projection Scope Metadata

Runtime resolution is materialized information from a concrete run. Examples:

* resolved XRef column members;
* resolved dense-axis row identities;
* materialized matrix width/order;
* derived carrier positions;
* current-run reconstruction helpers.

Such information is not durable truth. However, it may however be required until
a projection writeback is performed: it can be neccessary for:

* interpret user edits (e.g. an absent column that was originally part of the proejction communicates the intent "delete" while an absent column taht was never part of the projection communicates no intent)
* merge / reconsile data that was edited in the projected form
* reconstruct the projected form to the original form (e.g. delete helper columsn)

Therefore runtime resolution should not automatically be dismissed.
It may be required to interpret a later reimport safely. the inverse step of each projection 
should consume this data so there is no growing pile of outdated runtime resolution data.

if this becomes impossible (because a use case can not forsee if a pice of data is still needed), a dedicated 
prune step might become necessary. this prune would then be called by the pipeline designer who can 
judge the intent behind the use case.


== layout data

in the domain there is only meta as a container of data describing the data in hte frames - but if we output
to a spreadsheet, we pass the render layer (compare techncal model) the idea is to have one place to consistently
perform layout steps that are agnostic of the current  spreadsheet carrier (xls or libreoffice calc). this render
step creates a new bit of data that we call layout. 

although it has the character of metadata we do not call it like that in our application as we want to 
emphasis that it is not to be confused with meta

* layout data is consumend by the write path of the spreadsheet adapter (ods or xlsx respectiviely) and it is not presissted.
* meta data is not used by the write path of the spreadsheet adapter but treated as payload and persisted accordingly
* layout data is derived from metadata by the rendering layer

== Bootstrap Intent

Bootstrap/configuration intent is supplied by the workflow builder.

It can provide:

* required transformation intent;
* default metadata;
* presentation defaults;
* validation defaults;
* workbook-view defaults;
* schema-specific conventions.

Bootstrap intent is not always an override. In many workflows it should be
configured as fill-if-absent so user-carried metadata from a previous roundtrip
can remain effective.


== Entity Lifecycle and Referential Reconciliation Gap

Spreadsheet roundtrips need a clearer Functional Model treatment of entity
lifecycle and referential reconciliation:

* user deletes an entity row in a source/entity frame;
* user renames an entity key;
* user changes a key value that FK helpers or XRef relations reference;
* user deletes an XRef matrix row or column;
* user changes a schema/key structure through schema maintenance;
* relation rows still reference removed or renamed entities;
* helper columns display values for references that no longer resolve.

Open policy choices include:

* preserve dangling references;
* warn;
* hard-fail;
* drop/prune in-scope relations;
* cascade deletes explicitly;
* preserve out-of-scope relations;
* treat workbook structure as damaged;
* require review/promotion before mutation.

The current XRef slice should not absorb this entire topic. XRef owns
relation/matrix addressing within a given projection/update scope. It should not
own global entity lifecycle, schema migration, FK reconciliation, or cascade
policy by itself.

This topic likely requires a follow-up Functional Model concept and one or more
future slices.

=== XRef and Projection Scope Finding

The XRef crosstable case is one concrete example: missing matrix rows or columns
during reimport cannot always be interpreted from cell values alone, because the
workflow may need to know whether they were inside the original projection scope.



== Schema Maintenance And Key-Structure Change Gap

Schema maintenance introduces harder variants of the same lifecycle problem.

Examples:

* adding, dropping, or renaming columns;
* reordering columns;
* changing which column acts as key;
* changing from a single key to a composite key;
* renaming entities through key changes;
* merging or splitting entities;
* changing FK target structure;
* changing XRef axis identity.

These changes can invalidate persisted metadata, helper policies, validation
rules, workbook views, XRef projection scope, and read-path presentation intent.

The Functional Model should eventually define at least the problem categories
and default safety posture. It does not need to solve all migrations in the MVP.

A likely default posture is that silent data loss should be avoided. Cases where
relations reference missing entities or where schema/key structure changed
should produce explicit diagnostics unless a workflow has configured an explicit
policy.
