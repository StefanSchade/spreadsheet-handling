== Purpose

This inquiry captures working context for later Functional Model work around
schema maintenance, structural drift, frame/column lifecycle, metadata
alignment, and validation responsibility in spreadsheet-oriented roundtrips.

It is intentionally not a Functional Model chapter yet. It records human
framing and unresolved design questions so later work can promote the stable
parts into the Functional Model and derive one or more narrow slices.

No implementation requirements are added by this note. No public YAML, CLI,
metadata schema, persistence-boundary, or plugin API change is approved by this
note.

== Relation To Application Boundary Inquiry

This note follows the inquiry:

`application_boundary_metadata_authority_inquiry.adoc`

That earlier inquiry establishes the broader framing:

* data and metadata may enter through an application boundary;
* metadata can be required intent, default intent, user-carried presentation
  intent, projection/writeback scope evidence, runtime resolution, stale input,
  or foreign residue;
* read-path metadata is not automatically sediment, but it is also not
  automatically durable truth;
* arbitrary visible spreadsheet structure edits are not the intended schema
  maintenance interface.

This note narrows that framing to structural edits: frame/sheet additions,
deletions, and renames; column additions, deletions, and renames; and the way
explicit maintenance operations can keep schema-relevant metadata aligned.

== Background

In a classic ad-hoc spreadsheet, a user can rename a sheet, delete a column, or
add a new worksheet and the spreadsheet simply becomes the new reality.

In `spreadsheet-handling`, the visible spreadsheet is only one carrier and
editing surface among several. A structural edit can affect:

* frames and visible data;
* metadata describing schema, keys, helpers, presentation, or projection;
* pipeline configuration;
* plugin assumptions;
* FK and XRef relations;
* validation rules;
* workbook views and rendering defaults;
* readback and recomposition behavior.

The framework cannot prevent all structural drift. Files may be edited outside
the framework, pulled from Git, generated by another tool, changed in Excel, or
changed by a plugin. The correct framing is therefore not "is this edit
allowed?" but:

* Does the following workflow still have the intent it needs?
* Which metadata is now stale, missing, or dangling?
* Which consuming capability can continue with defaults?
* Which consuming capability should report, warn, or fail?
* Was an explicit maintenance operation used to carry metadata along with the
  data change?

== Working Vocabulary

=== Structural Drift

Structural drift is any change to the visible or logical structure of the data
that may leave metadata, pipeline assumptions, or references behind.

Examples:

* a new frame appears;
* an expected frame disappears;
* a frame is renamed;
* a new column appears;
* a column disappears;
* a column is renamed;
* a key column changes;
* a relation still references a removed or renamed frame or key value;
* presentation metadata refers to a sheet or column that no longer exists.

Structural drift may be intentional, accidental, or merely a consequence of data
entering through the application boundary.

=== Schema Maintenance Operation

A schema maintenance operation is an explicit framework-provided operation that
applies a structural change to frames and updates schema-relevant metadata where
possible.

Examples in the current project direction include operations exposed through
`schema_maintain`-style CLI / Makefile entry points rather than arbitrary manual
spreadsheet edits.

The purpose is not to forbid other edits. The purpose is to offer a safer path
when metadata continuity matters.

=== Schema-Relevant Metadata

Schema-relevant metadata is metadata whose meaning is tied to frame or column
identity.

Candidate examples include:

* key / primary-key declarations;
* FK helper policy;
* helper-column allowances or provenance;
* frame lifecycle classification;
* validation intent tied to columns;
* schema descriptions used by maintenance and validation capabilities.

The exact metadata roots are not decided by this inquiry.

=== Pipeline Design Responsibility

The workflow builder remains responsible for composing a pipeline that still
makes sense after structural changes.

A maintenance operation may update data and schema-relevant metadata, but it
does not guarantee that arbitrary downstream pipeline configuration, workbook
views, plugins, or domain-specific projections remain meaningful.

== Principle: Maintenance Does Not Forbid Drift

The framework should not be modeled as a closed application that forbids all
unexpected structural edits.

Instead:

* users may change JSON files, spreadsheets, or plugin outputs;
* files may enter from outside the application boundary;
* plugins may create, drop, or rename frames;
* metadata may be missing, stale, incomplete, or partially incompatible;
* the framework provides explicit maintenance operations, tolerant metadata
  consumption, and validation/reporting tools to help workflow builders manage
  these cases.

Therefore, the Functional Model should avoid asking whether a structural edit is
"allowed" in the abstract. A better question is whether a later capability has
the required intent and references available to operate safely.

== Structural Drift Sources

=== Structured File Drift

Structured data files may be changed outside the framework.

Examples:

* a JSON file is renamed;
* a JSON file is deleted;
* a new JSON file appears;
* a field is renamed inside records;
* a field disappears from some or all rows;
* data is updated by Git pull from another branch or another user.

The framework cannot trigger program logic merely because a user renamed a file
outside the tool. If metadata continuity matters, the workflow should prefer an
explicit maintenance operation instead of direct file manipulation.

=== Spreadsheet Visible-Structure Drift

A user may rename a sheet, delete a sheet, add a sheet, rename a header, delete
a column, or add a column in Excel or another spreadsheet application.

These edits are observable during readback, but the spreadsheet carrier cannot
be assumed to be an authoritative schema-maintenance interface.

A visible sheet rename is especially identity-sensitive: without a stable rename
mechanism, it looks like one sheet disappeared and another sheet appeared. The
same applies to some column renames when metadata or pipeline configuration
references the old column name.

Such edits may be acceptable in a specific builder-owned workflow, but they are
"on your own risk" unless routed through an explicit maintenance capability.

=== Plugin-Produced Drift

Plugins can change the frame set. A plugin may add, drop, or rename frames and
columns.

That is not inherently wrong. Plugins are an escape hatch for workflow builders.
However, if a plugin performs structural maintenance, it may need a future API
or helper function to publish the corresponding structural change into metadata.

This possible plugin API is not MVP. The current note only records that plugin
structural edits are another source of drift.

=== Explicit Maintenance Operations

Explicit maintenance operations are the preferred path when the intent is a
structural schema change rather than ad-hoc data editing.

They can:

* apply the structural edit to frames;
* update schema-relevant metadata where possible;
* leave downstream pipeline/view redesign to the workflow builder;
* provide deterministic behavior that can be tested and reviewed.

They do not make all future workflows valid automatically.

== Frame / Sheet-Level Cases

=== New Frame / New Sheet

A new frame is not inherently an error.

It may come from:

* a newly added JSON file;
* a newly added spreadsheet sheet;
* a plugin output;
* a seed dataset without prior metadata;
* another workflow or tool.

The frame may simply be visible data with no model metadata yet. Whether that
is a problem depends on the following steps.

Examples:

* If no pipeline step consumes the frame, it may remain harmless.
* If FK helpers should use it as a target, the workflow must provide the
  necessary FK relation intent.
* If XRef should use it as an axis source, the workflow must provide the
  necessary axis-domain intent.
* If workbook rendering should apply specific UX settings, bootstrap or
  read-path presentation metadata may provide defaults.

A new frame should therefore not be treated as a process error merely because it
lacks metadata. Missing metadata becomes relevant only when a later capability
requires it.

=== Deleted Frame / Missing Sheet

A missing frame is not inherently a global error either.

It becomes a problem when something still references it:

* pipeline configuration expects the frame;
* FK policy names it as source or target;
* XRef dense-axis configuration names it as an axis source;
* validation references it;
* workbook view mapping expects it;
* presentation metadata refers to the missing sheet;
* relations in other frames point at entities formerly held by it.

The correct response depends on the consuming capability and workflow policy.
A load/read path should avoid failing merely because non-essential stale
metadata references a missing frame. A transformation or validation step that
requires the frame may fail clearly or report according to configured policy.

=== Renamed Frame / Renamed Sheet

A rename is identity-sensitive.

Without an explicit mechanism, a frame or sheet rename degenerates into:

[source,text]
----
old name disappeared
new name appeared
----

This can lose metadata continuity and can make existing references appear
stale.

The preferred maintenance path is an explicit rename operation, for example a
`schema_maintain` frame rename, because it can apply the data change and update
schema-relevant metadata together.

Direct Excel sheet renames are not the intended MVP maintenance interface. They
may be accepted by a builder-owned workflow, but the framework does not promise
that such a rename preserves frame identity or updates metadata.

Possible future mechanisms could include stable frame identifiers, hidden
sheet-level identity markers, or plugin APIs that publish a rename. These add
complexity and are not assumed by this inquiry.

== Column-Level Cases

=== New Column

A new column is not inherently an error.

It may represent:

* a new data field;
* a temporary user note;
* a plugin-produced field;
* a helper-like column;
* a field that future schema metadata should describe.

The framework should generally accept the visible data. Whether the column is
meaningful to later steps depends on pipeline and metadata configuration.

If the new column should become part of schema intent, the builder should use or
add an explicit maintenance/configuration step. If no step consumes it, it may
remain ordinary data.

=== Deleted Column

A deleted column is a structural absence. It should not cause a global read/load
failure merely because some stale non-essential metadata mentions it.

It may become a problem if the column is required by a later capability:

* key/PK validation;
* FK helper relation policy;
* XRef row key, column key, or value column;
* Cell Codec slot configuration;
* validation rules;
* workbook view configuration;
* helper provenance or cleanup.

The consuming step should behave as tolerantly as reasonable for its contract:

* ignore or prune stale non-essential metadata;
* fall back to safe defaults for presentation behavior;
* report clear diagnostics for missing required transformation intent;
* allow workflow builders to choose warning vs failure where validation is the
  appropriate control.

A deleted key column is not automatically a load-time hard error. It is a data
state that will likely be caught by schema, PK/FK, or transformation validation
if the workflow requires that column.

=== Renamed Column

A column rename may work as ordinary data if no metadata or pipeline step
references the old name.

It becomes identity-sensitive when the old column name is referenced by:

* key schema;
* FK helper policy;
* XRef configuration;
* Cell Codec slots;
* validation rules;
* workbook view mappings;
* presentation metadata;
* plugin assumptions.

For continuity, an explicit maintenance rename is the safer path because it can
update schema-relevant metadata alongside the visible data.

Without such a maintenance operation, a column rename may appear as delete plus
add. That is acceptable in builder-owned workflows, but not a general guarantee
of metadata continuity.

== Metadata Behavior Under Drift

=== Missing Metadata

Missing metadata is not automatically a process error.

A seed workflow may start from structured data or a simple spreadsheet without
framework metadata. The workflow then relies on bootstrap/configuration to
provide required intent.

Missing metadata becomes a problem only when a later step needs intent that was
not supplied elsewhere.

=== Stale or Dangling Metadata

Stale or dangling metadata should be classified by consequence.

Possible responses:

* ignore stale non-essential metadata;
* prune or sanitize invalid metadata;
* regenerate defaults;
* report a warning or finding;
* fail clearly when required intent is missing or contradictory.

The important point is that stale metadata should not cause catastrophic failure
when it only affects non-essential presentation or UX. For example, missing
formatting metadata should lead to reasonable defaults, not an unusable workbook
such as white text on white background.

=== Required Transformation Intent

Some steps legitimately require intent.

Examples:

* FK helper materialization requires FK relation policy;
* XRef requires row/column/value configuration;
* validation requires validation rules;
* Cell Codec requires explicit codec intent;
* schema validation requires declared key expectations.

If required intent is missing, the consuming step may fail. The failure should
name the missing producer/configuration requirement rather than producing a
mysterious downstream error.

== What Explicit Schema Maintenance Should Update

Explicit schema maintenance should update:

* the affected frame data;
* schema-relevant metadata owned by the schema-maintenance contract;
* references in that schema metadata when the operation can do so
  deterministically.

Candidate metadata areas include:

* frame/column schema declarations;
* key declarations;
* helper allowances or policy fragments, if owned by the schema contract;
* schema-level validation expectations.

This inquiry does not decide the exact metadata shape.

== What Explicit Schema Maintenance Does Not Guarantee

Schema maintenance does not automatically guarantee that all workflow surfaces
remain meaningful.

In particular, it should not be assumed to redesign or repair:

* arbitrary pipeline YAML logic;
* workbook views;
* XRef projections;
* follow-up projections;
* plugin-specific assumptions;
* domain-specific business policy;
* user-authored spreadsheets that were structurally edited outside the tool.

A maintenance operation may support updating some references in the future, but
that must be explicitly scoped. The workflow builder remains responsible for
ensuring that the pipeline still matches the data model after maintenance.

== Validation As Optional Control

Validation capabilities are the right place for many consistency checks.

Candidate checks include:

* dangling FK references;
* non-unique primary keys;
* missing required key columns;
* relations referencing missing entities;
* metadata references to missing frames or columns;
* helper provenance referring to absent helper columns;
* XRef configuration that names absent frames or columns.

The workflow builder should be able to configure whether validation findings are
warnings or failures where that distinction makes sense.

This keeps the toolkit composable:

* maintenance operations apply structural edits and metadata updates;
* validation operations diagnose whether the resulting model is consistent
  enough for the intended workflow;
* pipeline design decides which validations are required.

== MVP Posture

Candidate MVP posture for later Functional Model promotion:

* New frames and columns are accepted as data unless later steps require missing
  intent.
* Deleted frames and columns are accepted as data absence unless later steps
  require them.
* Missing or stale non-essential metadata should be tolerated, pruned,
  sanitized, or defaulted where possible.
* Required transformation intent remains the responsibility of
  bootstrap/configuration, trusted metadata, or a prior producer step.
* Renames are identity-sensitive; explicit maintenance operations are the safe
  path when metadata continuity matters.
* Direct Excel sheet/header renames are not the intended MVP schema-maintenance
  interface.
* Plugins may transform frame sets; future helper APIs may let plugins publish
  structural changes to metadata, but that is not assumed by this inquiry.
* Pipeline validity remains the workflow builder's responsibility.
* Validation should provide configurable diagnostics for PK/FK consistency,
  dangling references, and missing schema expectations.

== Non-Goals For Immediate Follow-Up

Immediate follow-up should not attempt to:

* forbid users from editing JSON, spreadsheets, or plugin outputs;
* treat arbitrary spreadsheet structure edits as authoritative schema
  maintenance;
* design a full migration engine;
* solve all rename detection heuristics;
* add hidden frame IDs or sheet identity markers without a dedicated slice;
* update arbitrary pipeline YAML or workbook views automatically;
* make `_meta.yaml` a public manually edited schema interface;
* guarantee that every user-composed pipeline remains valid after maintenance.

== Open Questions

* Which metadata roots are schema-relevant and owned by schema maintenance?
* Which metadata roots are presentation-only and should simply default or prune
  on drift?
* Should there be an always-useful meta preflight/reconciliation capability at
  the application boundary?
* Which drift checks belong to load/read, which to validation, and which to the
  consuming transformation step?
* Which validation findings should support warning vs fail policy?
* How should a schema-maintenance operation report references it cannot update?
* Should plugins get a supported API for publishing frame/column rename/drop
  events into metadata?
* Should future spreadsheet artifacts carry stable frame/sheet identifiers for
  rename detection, or is explicit schema maintenance sufficient for MVP?
* How should schema maintenance interact with future key-structure changes such
  as single-key to composite-key migration?
* How much, if anything, should maintenance operations update workbook-view or
  projection configuration?
