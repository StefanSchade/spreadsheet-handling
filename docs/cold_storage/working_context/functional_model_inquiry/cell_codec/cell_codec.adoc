= Cell Codec Functional Model Inquiry
:revdate: 2026-06

== Purpose

This document records the user-side functional intent behind the `cell_codec`
transformation family.

It is intentionally written from the problem-space perspective and should be
treated as an input to Functional Model work, not as a derivation from the
existing implementation.

The goal is to explain why a capability such as `cell_codec` exists and which
user problem it solves before discussing implementation details, metadata, or
pipeline mechanics.

== Core Problem

Business analysts frequently work with large matrix-like structures that must
present many attributes simultaneously.

A common challenge is that a single matrix intersection may carry multiple
independent properties.

Representing each property in its own matrix is technically simple but often
produces poor usability:

* analysts must switch between multiple views,
* consistency checking becomes difficult,
* relationships between attributes become harder to see,
* information density decreases.

The functional requirement is therefore:

[quote]
Compress several related attributes into a single editable cell while preserving
a controlled transformation back into structured canonical data.

The purpose is not compact storage.

The purpose is human comprehension.

== Representative Example

Consider software products sold into different markets.

[cols="1,1,1,1"]
|===
|Product |JP |US |EU

|Home Automation
|SaaS / JP / OIDC
|SaaS / EN / OIDC
|AppStore / EN / OIDC

|Health Tracking
|AppStore / JP / -
|AppStore / EN / OIDC
|AppStore / EN / OIDC

|Corporate Tool
|SaaS / JP / LDAP
|SaaS / EN / LDAP
|SaaS / EN / OIDC
|===

The canonical representation may contain separate attributes:

[source,text]
----
product
market
delivery_model
language
authentication_mode
----

The matrix view compresses several of those attributes into one visible cell.

The analyst sees one matrix and can inspect consistency across products and
markets without switching between multiple independent tables.

== Functional Concept

Cell Codec is a transformation pattern that converts:

[source,text]
----
multiple structured attributes
<->
one compact editable cell value
----

The transformation is reversible.

The compact cell value is a presentation-oriented representation.

The structured attributes remain the canonical representation.

== Relationship to XRef Crosstable

Cell Codec does not create matrix structures.

Cell Codec answers:

[quote]
How are several attributes represented inside a single cell?

XRef Crosstable answers:

[quote]
How is a relation projected into a matrix?

These are separate concerns.

Cell Codec may be used without XRef Crosstable.

XRef Crosstable may be used without Cell Codec.

They are frequently combined because matrix projections are the place where
information density matters most.

== Relationship to Legend Blocks

Legend Blocks explain or constrain allowed values.

Cell Codec transports compact values between projection and canonical forms.

Legend Blocks provide value context.

Cell Codec provides value transformation.

The concepts are related but independent.

== Relationship to Compact Multiaxis

Compact Multiaxis is not a separate motivation.

It is a composition of simpler concepts.

At a high level:

[source,text]
----
Compact Multiaxis
=
XRef Crosstable
+
Cell Codec
+
Legend Support
----

Cell Codec therefore acts as one building block used by higher-level workbook
patterns.

== Encoding Strategies

Several encoding strategies are possible.

The functional requirement is the preservation of compactness and reversibility,
not a specific encoding mechanism.

=== Strategy A: Position-Based Encoding

[source,text]
----
A-B--D-E
----

Meaning is derived from position.

[source,text]
----
category_1 = A
category_2 = B
category_3 = empty
category_4 = D
category_5 = E
----

This is the most explicit strategy.

=== Strategy B: Disjoint Value Sets

[source,text]
----
A B D E
----

Meaning is inferred from the value set to which a token belongs.

This removes some separator overhead but requires non-overlapping value domains.

=== Strategy C: Category-Hinted Values

[source,text]
----
delivery:SaaS
lang:JP
auth:OIDC
----

or

[source,text]
----
a:A b:B limit:2.0
----

Meaning is carried explicitly by category hints.

This becomes useful when values are not naturally disjoint or when numeric
values participate in the compact representation.

== MVP Scope

The first practical implementation does not require all encoding strategies.

An MVP is expected to support:

* whole-cell codes,
* token splitting under explicit configuration,
* configurable separators,
* deterministic roundtrip behavior.

More advanced category-aware representations are future scope.

== Validation and Allowed Values

Validation is related but distinct.

The primary purpose of Cell Codec is transformation.

Validation exists to improve editing quality.

Simple validation may be provided by explicitly configured allowed values or
Legend Blocks.

Future approaches could derive valid combinations from category-specific value
sets and generate temporary validation surfaces for workbook editing.

Such derived validation structures are optional and not required for the core
Cell Codec concept.

== Metadata and Lifecycle Considerations

Validation support structures, generated codebooks, helper ranges, or similar
editing artifacts may exist only inside the system boundary.

A key observation is:

[quote]
Not every editing aid must survive permanently outside the system boundary.

When a workbook leaves the system, such structures may be treated as ephemeral.

This is intentional.

The goal is to avoid assigning long-term authority to internal helper metadata.

The canonical data remains authoritative.

Editing-support structures are allowed to be transient.

== Non-Goals

Cell Codec is not:

* a parser utility,
* a string manipulation feature,
* a matrix projection mechanism,
* a business-semantics engine,
* a metadata-authority framework.

Its purpose is limited and specific:

[quote]
Provide a reversible transformation between structured attributes and compact,
human-editable cell values.

== Expected Functional Model Outcome

The Functional Model should describe Cell Codec as an information-density
pattern for workbook editing surfaces.

Implementation details such as pipeline steps, registry structures, metadata
paths, parser helpers, or configuration fields should appear only after the
functional motivation has been established.
