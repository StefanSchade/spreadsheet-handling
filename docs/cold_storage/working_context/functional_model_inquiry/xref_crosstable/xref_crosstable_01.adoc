== 1. Purpose

This document captures the functional concept behind XRef crosstables.

It is deliberately written before a Functional Model promotion, before an
implementation concept, and before any slice-level FTR. Its purpose is to
preserve the problem-space understanding while the concept is still broad.

The main goal is to avoid deriving the model from implementation residue,
metadata side effects, or AI-inferred consistency patterns. The concept starts
from the intended user-facing transformation:

[source,text]
----
persisted input artifacts
-> transient working model
-> editable crosstable projection
-> edited workbook / projection
-> recomposed relation frame
-> persisted output artifacts
----

The framework does not own a permanent canonical state. It reads persisted
artifacts from a boundary, constructs a transient working model, performs
configured transformations, and writes new artifacts.

The persistence mechanism may be Git, a filesystem directory, generated build
output, CI workspace output, or another integration-owned store. Git is an
important consumer pattern but not part of the XRef core concept.

== 2. Terminology Caution: "Canonical"

The term "canonical" is intentionally avoided where possible.

In this project, "canonical" can accidentally mix several different claims:

* durable storage;
* semantic authority;
* current consistency;
* freshness after a pull or external update;
* completeness;
* reliability of `_meta`;
* system-owned truth.

Those properties do not automatically coincide.

A tuple relation may be the durable source structure for a relation, but it is
not necessarily freshly validated, globally consistent, or protected from
external edits. `_meta` may contain useful projection intent or reconstruction
hints, but it may also be local, stale, gitignored, carrier-derived, or
runtime-only.

This concept therefore prefers narrower terms:

[cols="1,3"]
|===
| Term | Meaning

| persisted input artifacts
| Files or artifacts read at the application boundary.

| durable source structure
| A long-lived representation used as the source shape for a transformation,
usually better suited for storage and machine processing than a spreadsheet
view.

| transient working model
| The in-memory frame inventory produced for one run from input artifacts,
configuration, optional metadata, and pipeline steps.

| projection intent
| The configured intention to create an editing surface from existing frames.

| runtime resolution
| Concrete values computed during a run, such as actual axis keys, row
identities, or materialized column positions.

| editable projection
| A workbook-facing frame intended for user editing.

| recomposition
| Reverse transformation that turns edited projection cells back into relation
rows.

| output artifacts
| Files or artifacts written by the current run.
|===

== 3. Core Concept

An XRef crosstable is an editable matrix projection over a durable tuple
relation.

The tuple relation is the storage- and processing-friendly representation. The
matrix is the human-friendly editing representation.

A simple relation:

[source,text]
----
product_id | country_id | value
P1         | DE         | X
P1         | JP         | X
P2         | US         | X
----

can be projected as:

[source,text]
----
product_id | DE | JP | US
P1         | X  | X  |
P2         |    |    | X
----

The matrix exists to make relation maintenance easier. It is not inherently the
durable source structure.

The important functional point is that XRef is not only a matrix/tuple shape
conversion. It is a scoped editable projection and recomposition problem.

[source,text]
----
entity frames

* optional existing relation frame
* explicit axis scopes
  -> editable matrix frame

edited matrix frame

* carried relation frame
* configured recomposition target
  -> updated relation frame

----

== 4. Baseline Use Case

The baseline use case has:

* two existing entity frames;
* a primary key on each entity frame;
* an optional existing relation frame;
* explicit row-axis and column-axis key lists;
* one editable value per matrix cell;
* recomposition into a configured target relation frame.

Example:

[source,text]
----
## products

product_id | label
P1         | Credit Application
P2         | Insurance Application

## countries

country_id | label
JP         | Japan
US         | United States

## availability

product_id | country_id | value
P1         | JP         | X
P2         | US         | X
----

Projected crosstable:

[source,text]
----
product_id | JP | US
P1         | X  |
P2         |    | X
----

Reverse path:

[source,text]
----
visible matrix cells

* carried availability relation
  -> recomposed availability relation

----

The target relation may already exist. If it does not exist, the reverse
transformation creates it from the crosstable cells.

== 5. Axis Identity

Axis identity is always based on the primary key of the respective entity.

Visible labels may be projected as helper/display values. They may also be used
as configured lookup values if a future feature explicitly supports that. But
visible labels are not silently treated as primary identity.

Baseline rule:

[source,text]
----
## axis identity = configured primary key of the axis entity
----

Examples:

[source,text]
----
country_id = JP      // identity
label      = Japan   // display/helper value
----

A visible label such as `Japan` may be useful for users, but the identity of
the axis member is `JP`.

== 6. Axis Scope

XRef does not decide which entity rows are active, relevant, current, or
eligible.

The baseline accepts explicit axis key lists.

[source,yaml]
----
row_axis:
  frame: products
  key: product_id
  keys: [P1, P2]

column_axis:
  frame: countries
  key: country_id
  keys: [JP, US]
----

A future query/filter step may derive such lists from entity frames.

[source,text]
----
entity frame

* configured query/filter
  -> list of matching primary keys
  -> XRef axis scope

----

That query step is separate from XRef. XRef consumes the resolved list and does
not interpret fields such as `active`, `status`, `valid_until`, or
`business_area` on its own.

This keeps XRef deterministic and composable:

[source,text]
----
select_axis_keys
-> xref_projection
-> workbook rendering
----

instead of:

[source,text]
----
## xref_projection guesses what "active" means
----

== 7. Axis Order

Axis order is a functional concern because it strongly affects workbook
usability.

Baseline ordering should be explicit or deterministic.

Possible order sources:

* explicit list order;
* order from a produced axis-key list;
* later: configured sort field;
* later: grouped axis order;
* later: order from a maintained axis mapping frame.

XRef should not rely on accidental dataframe order unless that order is
explicitly defined by the upstream step.

== 8. Existing Relation Frame and Scoped Update

The existing relation frame is optional input.

If it exists, it must be carried through recomposition so that relation entries
outside the crosstable update scope are preserved.

This is the central baseline rule:

[source,text]
----
An XRef projection owns only its configured update scope.

Inside that scope:
visible matrix cells produce update statements.

Outside that scope:
existing relation rows are carried through unchanged.
----

This distinguishes:

* replacing the whole relation;
* updating only the scoped subset represented by the matrix.

The baseline is scoped update, not whole-relation replacement.

== 9. Cell Presence, Blank Cells, and Missing Addresses

The reverse transformation interprets only present, addressable matrix cells.

Baseline semantics:

[cols="1,3"]
|===
| Input condition | Meaning

| present cell with value
| Set or replace the relation entry for that matrix address.

| present blank cell
| Remove the relation entry for that matrix address inside the update scope.

| missing row
| No matrix address exists for that row; no update statement is produced for
those addresses.

| missing column
| No matrix address exists for that column; no update statement is produced for
those addresses.

| invalid or ambiguous row/column identity
| Fail fast.

| relation row outside the XRef update scope
| Preserve unchanged.
|===

A blank existing cell is therefore different from a missing row or missing
column.

[source,text]
----
blank cell    -> statement: relation entry absent
missing cell  -> no statement
invalid cell  -> error
----

The transformation does not infer user intent from missing workbook structure.
It does not attempt to determine whether a user accidentally deleted a row or
column.

If deletion protection, header validation, or projection-shape integrity checks
are needed, they belong to a cross-cutting spreadsheet projection integrity
concept, not to XRef itself.

== 10. Error Philosophy

The baseline should fail fast rather than apply hidden heuristics.

Errors should be preferred for:

* missing row-axis entity frame;
* missing column-axis entity frame;
* missing configured primary key;
* relation tuple references an unknown row-axis ID;
* relation tuple references an unknown column-axis ID;
* duplicate relation entries in scalar baseline mode;
* visible matrix row cannot be resolved to an axis ID;
* visible matrix column cannot be resolved to an axis ID;
* cell payload cannot be parsed according to the configured value semantics;
* ambiguous identity mapping.

No fuzzy matching, silent repair, or automatic renaming should happen in the
baseline.

If a later real consumer repeatedly hits one of these errors in a legitimate
workflow, a separate explicit policy or helper step can be introduced.

== 11. Precedence

Within the configured XRef update scope, edited XRef cells have precedence over
the carried relation rows.

Outside the configured scope, carried relation rows remain untouched.

Baseline rule:

[source,text]
----
XRef wins inside its scope.
Carried relation rows survive outside its scope.
No merge heuristic guesses ownership outside the configured scope.
----

This is not intended as a general conflict-resolution engine. If two editable
views produce updates for the same relation tuple, that is a separate
overlapping-view precedence problem.

== 12. Relation Table Visibility

The relation frame may remain in the working frame inventory while the XRef
projection is presented to the user.

Whether the relation frame is visible, hidden, protected, or omitted from a
particular workbook view is a workbook-view / UX concern.

XRef core semantics only require that recomposition has access to the carried
relation data when scoped update is intended.

If a consumer exposes both the tuple relation and the XRef matrix as editable
surfaces in the same workbook, then explicit precedence rules are required. XRef
does not solve that by hiding sheets.

== 13. Value Semantics

The baseline XRef concept is independent of concrete cell value interpretation.

A crosstable cell may later represent:

* boolean marker, for example `X` / blank;
* scalar value;
* code from a configured list;
* token resolved through a legend block;
* long text;
* compact multi-value set;
* composite value built from several relation projections.

XRef owns cell addressing:

[source,text]
----
## Which relation tuple does this matrix cell address?
----

Other concepts own cell value interpretation:

[source,text]
----
What does the visible token mean?
How is it parsed?
How is it serialized?
Which values are allowed?
----

Possible neighbours:

* Cell Codec;
* Legend Blocks;
* Compact Multiaxis;
* validation rules.

Baseline XRef should not absorb those responsibilities.

== 14. Advanced Feature: Compact Multi-Value Cells

A later extension may allow one visible cell to represent multiple relation
tuples.

Example:

[source,text]
----
product_id | country_id | deployment_mode
credit     | JP         | SaaS
credit     | JP         | ManagedService
----

Visible cell:

[source,text]
----
## credit x JP = SaaS / ManagedService
----

This changes the cardinality.

Baseline:

[source,text]
----
## (row_axis_id, column_axis_id) -> zero or one payload value
----

Advanced compact multi-value mode:

[source,text]
----
## (row_axis_id, column_axis_id) -> set of payload values
----

This should be treated as an advanced cell-payload feature, probably owned by
Cell Codec / Compact Multiaxis rather than by XRef itself.

== 15. Advanced Feature: Composite Cells over Several Relations

A later extension may combine several relation projections over the same axes
into one visible matrix.

Example:

[source,text]
----
availability:
product x country -> X / blank

billing_model:
product x country -> monthly / annual

key_account:
product x country -> person
----

Composite visible cell:

[source,text]
----
## X / monthly / Herr Schmidt
----

Functionally this is not the same as a multi-value relation. It is a composite
view over several independent relation targets.

Each component relation keeps its own:

* source relation frame;
* target relation frame;
* value parser;
* empty-cell semantics;
* writeback rule;
* precedence rule.

A future implementation might build this from several XRef projections and a
stacking/composition step, or from a single configured composite-XRef step. This
concept does not decide that.

The functional requirement is only:

[source,text]
----
Several relation projections may share one axis scope and be presented as one
combined editing cell, with explicit component boundaries and explicit
writeback targets.
----

This is advanced and not part of the baseline.

== 16. Advanced Feature: Axis Helpers

Axis helper display is a likely high-value ergonomics feature.

Vertical helper columns are straightforward:

[source,text]
----
product_id | product_label       | JP | US
P1         | Credit Application  | X  |
P2         | Insurance App       |    | X
----

Horizontal helper display requires transposed helpers or multi-row headers:

[source,text]
----
```
          JP       US
```

label         Japan    United States
product_id
P1            X
P2                     X
----

or:

[source,text]
----
```
          Japan          United States
          JP             US
```

product_id
P1            X
P2                           X
----

This is not part of XRef baseline semantics. XRef needs only the conceptual
ability to say:

[source,text]
----
axis identity comes from PK
axis display may project additional helper fields
----

How those helper fields are rendered belongs to projection / workbook-view /
rendering concerns.

Editable helper writeback is a further separate feature. It would mean that a
projected helper value can update its source entity frame during reimport.

That must not be implicit. It requires explicit rules such as:

[source,text]
----
source frame: products
identity: product_id
editable helper field: label
writeback target: products.label
conflict policy: fail | visible_wins | source_wins
----

This is not part of the baseline.

== 17. Advanced Feature: Grouped Headers

Grouped headers are relevant for structured axes.

Examples:

[source,text]
----
Kredit
Annuitätendarlehen
Forward-Darlehen

Anlage
Festgeld
Depot
----

or:

[source,text]
----
EMEA
DE
FR

Americas
US
CA
----

Grouped headers are not merely a way to display nested JSON. In XRef they are
primarily a way to display axis classification or multi-level axis structure.

Identity still comes from configured axis keys. Group labels are display,
ordering, and navigation information unless a future grouped-axis contract says
otherwise.

Grouped headers are not part of the baseline. They should build on a clarified
axis model.

== 18. Cross-Cutting Concern: Projection Structure Integrity

Header edits, renamed helper columns, deleted structural columns, or corrupted
projection shape are cross-cutting spreadsheet readback problems.

Example:

[source,text]
----
entity_id
_entity_name     // helper column
----

If a user renames `_entity_name` to `entity_name`, a naive readback may treat it
as ordinary payload. That problem affects:

* FK helpers;
* lookup helpers;
* XRef matrices;
* compact multiaxis views;
* workbook views;
* validation targets;
* schema-maintenance workflows.

XRef should not own this concern.

A separate projection-integrity feature may later validate expected workbook
structure before interpreting visible cells as domain data.

Possible future rule:

[source,text]
----
Structure comes from the pipeline.
Spreadsheet users edit values.
Renaming tabs, structural headers, helper columns, or generated matrix headers
is unsupported unless an explicit maintenance transformation owns that change.
----

Baseline XRef assumes that the readback input has either valid structure or
fails before XRef recomposition.

== 19. Cross-Cutting Concern: Review / Diff / Promotion Workflow

Safe reimport workflows may use:

[source,text]
----
generated workbook
-> user edit
-> reimport candidate
-> semantic diff
-> explicit promotion
----

This is important for adoption, especially because XRef can create, update, or
remove many relation tuples at once.

However, this is not XRef responsibility.

XRef defines how edited matrix cells are recomposed into a relation frame.
Review areas, semantic diffs, promotion into durable storage, Git integration,
approval workflows, or consumer-specific safety rails belong to the integration
or consumer workflow layer.

== 20. Metadata and Lifecycle Pressure

The current implementation already exposes lifecycle pressure around
`xref_crosstable`.

Functional categories likely include:

[cols="1,3"]
|===
| Information | Functional role

| relation frame name
| Projection intent / recomposition target

| matrix frame name
| Projection intent / produced editable frame

| row-axis key fields
| Projection intent

| column-axis key field
| Projection intent

| value field
| Projection intent

| drop-empty / blank handling
| Recomposition rule

| explicit axis key lists
| Projection scope

| resolved row identities
| Runtime resolution

| resolved column keys
| Runtime resolution

| visible workbook headers
| Carrier/readback surface

| helper display fields
| Projection ergonomics

| grouped header labels
| Axis display/classification

| cell token mapping
| Value interpretation, probably owned by Cell Codec / Legend Block
|===

A later Functional Model or implementation concept should classify these
explicitly.

Important initial suspicion:

[source,text]
----
column_key   = axis attribute / identity field
column_keys  = concrete resolved axis members
----

Those are not the same kind of information.

Similarly:

[source,text]
----
dense_axes.columns_from / rows_from = projection intent
dense_axes.resolved.*              = runtime resolution
----

This concept does not propose the final `_meta` shape. It only records the
functional distinction so that later slices do not preserve mixed fields merely
because they exist in current runtime metadata.

== 21. Baseline Scope Summary

Baseline XRef includes:

* two existing entity frames;
* primary-key based axis identity;
* explicit list-based row and column axis scope;
* optional existing relation frame;
* generated editable matrix frame;
* configured target relation frame for recomposition;
* target relation creation if absent;
* scoped update semantics;
* XRef cells win inside their update scope;
* blank present cell means no relation entry;
* missing row/column means no update statement for those addresses;
* out-of-scope relation rows are preserved;
* fail-fast handling for invalid references, missing PKs, duplicate scalar
  tuples, and ambiguous addresses.

== 22. Deferred / Separate Concepts

The following are deliberately not baseline XRef:

* query/filter step producing axis key lists;
* "active" or other entity-attribute semantics;
* horizontal/transposed helper display;
* editable helper writeback;
* grouped headers;
* compact multi-value cells;
* composite cells over multiple relations;
* Legend Block value resolution;
* Cell Codec parsing;
* Compact Multiaxis composition;
* projection structure integrity / header tamper detection;
* workbook hiding/protection UX;
* review/diff/promotion workflows;
* Git-specific lifecycle assumptions.

These may become follow-up concepts or FTRs, but they should not be smuggled
into the baseline transformation.

== 23. Candidate Future Work Items

This concept suggests several possible later slices. The names are placeholders,
not proposed final FTR IDs.

=== XRef Baseline Functional Model Promotion

Promote the baseline concept into the Functional Model:

* crosstable as editable projection over tuple relation;
* scoped recomposition;
* axis identity and explicit axis scope;
* blank/missing/invalid address semantics;
* baseline non-goals.

=== XRef Implementation Concept

Map the Functional Model to current runtime:

* current `expand_xref` and `contract_xref` behavior;
* current `_meta.xref_crosstable` shape;
* current persistence-boundary pruning;
* current test coverage;
* residue vs load-bearing metadata;
* minimal safe cleanup slices.

=== Projection Structure Integrity

Cross-cutting concept/FTR for detecting corrupted spreadsheet projection shape:

* renamed helper columns;
* missing structural headers;
* unexpected generated columns;
* invalid matrix headers;
* distinction between value editing and schema/projection editing.

=== Axis Key List Producer

Separate pipeline step that produces explicit axis key lists from entity frames
using configured filters or queries.

XRef consumes the list; it does not own the query semantics.

=== Axis Helper Display

Projection ergonomics feature:

* vertical helper columns;
* horizontal/transposed helper rows;
* display labels;
* multi-row headers;
* no identity substitution by labels.

=== Grouped Axis Headers

Structured axis presentation:

* product category -> product;
* region -> country;
* group labels as display/classification;
* identity remains PK-based.

=== Composite XRef Cell View

Advanced feature for combining multiple relation projections over the same axes
into one visible cell.

=== Compact Multi-Value XRef Cells

Advanced feature for representing a set of relation tuples inside one visible
cell.

== 24. Open Questions

The concept intentionally leaves several questions open:

* How should explicit axis key lists be supplied in the current YAML surface?
* Should relation frame carry-through be modeled as ordinary frame inventory,
  frame lifecycle metadata, or a recomposition-specific working input?
* Should XRef baseline support row-axis helper display immediately, or defer all
  helper display to a later axis-helper feature?
* How should duplicate tuples be identified when the relation has more than
  row-axis key, column-axis key, and value?
* How should scalar payload mode distinguish blank, empty string, and absent
  tuple, if at all?
* Does current `drop_empty` already encode the baseline blank-cell behavior, or
  is its meaning narrower in implementation?
* Should the terms `expand_xref` and `contract_xref` be preserved as runtime
  step names even if the Functional Model uses clearer directional names?
* Where should recomposition precedence live if multiple editable projections
  target the same relation?
* Which current `_meta.xref_crosstable` fields are load-bearing for consumers
  and which are historical residue?

== 25. Working Summary

XRef is best understood as a scoped editable projection over a tuple relation.

The durable source structure is the long-form relation. The crosstable is the
editing surface. The framework does not own canonical state; it temporarily
builds a working model from persisted artifacts and configuration.

The baseline feature should stay small:

[source,text]
----
two entity axes

* optional existing relation
* explicit axis key lists
  -> editable matrix
  -> scoped recomposition into relation

----

The central rule is:

[source,text]
----
Only present, addressable matrix cells produce update statements.
Blank present cells remove scoped relation entries.
Missing rows or columns produce no update statement.
Invalid or ambiguous addresses fail fast.
Relation rows outside the XRef scope are preserved.
----

Advanced features such as grouped headers, horizontal helpers, compact
multi-value cells, composite cells, legend resolution, and projection-integrity
guards are real and should be recorded, but they should not define the baseline
XRef contract.

== Implementation Note, 2026-06

During the XRef reformation slice, the existing `drop_empty` flag was confirmed
as the explicit empty-cell policy. `drop_empty=True` omits empty matrix cells
from relation recomposition; `drop_empty=False` preserves them as empty-valued
relation rows. This note records the implementation decision and does not
otherwise change the historical inquiry.
