= Project Memory Contract Rewrite Check

== Purpose

Short operational note on whether a contract-first domain/meta rewrite is
expected to support the repository-internal `project_memory` consumer layer.

This note is temporary working context. It is useful while choosing rewrite
contracts and can become obsolete once Project Memory is covered by the new
contract suite.

== Current Consumer Usage

Project Memory is a repo-internal dogfood consumer. It keeps curated project
state as flat JSON, exposes selected relations through an ODS workbook, and
generates derived query/context artifacts.

Observed spreadsheet pipeline steps:

* `contract_xref`;
* `expand_xref`;
* `plugin`.

The ODS workbook roundtrip currently covers:

* `json_dir` canonical input;
* XRef projection of `concern_event_xrefs` to `concern_event_matrix`;
* local plugin enrichment of the matrix with display/context columns from
  `concern_events`;
* ODS workbook export;
* ODS workbook reimport;
* local plugin stripping of matrix context columns before XRef expansion;
* XRef expansion back to `concern_event_xrefs`;
* local plugin finalization that preserves notes/base rows and removes
  projection-only frames;
* `json_dir` staging output.

Project Memory also uses local plugins outside the spreadsheet roundtrip for:

* derived query views;
* release-note candidate views;
* heatmaps and diagnostics;
* current-context AsciiDoc rendering through templates;
* extraction of candidate rows from repository artifacts and Git history.

These local query/render/extraction plugins are consumer tooling, not core
spreadsheet domain requirements.

== Similarity To Personal Data

Project Memory is similar to `personal_data` in that it relies on:

* flat JSON as curated source truth;
* spreadsheet/ODS as an editable review surface;
* reimport to flat JSON staging;
* generated outputs that are not canonical truth.

It is different because it already uses XRef as an editable relation matrix and
uses local plugins to enrich and clean that matrix.

== Expected Rewrite Coverage

A contract-first rewrite should support Project Memory if it preserves:

* JSON directory frame IO;
* ODS spreadsheet rendering and reading;
* XRef contraction and expansion;
* context/display columns around an editable matrix;
* scoped XRef recomposition that preserves base relation rows where required;
* clean separation between canonical JSON frames, projection-only frames, and
  derived query/report outputs;
* the plugin extension boundary, at least for consumer tooling.

Project Memory should therefore be a mid-level contract:

* more demanding than `personal_data`;
* less broad than Worldbuilding;
* especially useful for XRef, projection cleanup, and plugin-boundary behavior.

== Operational Gaps To Watch

* Matrix context columns must remain display-only. Edits to those columns must
  not update the source event frame.
* XRef reimport must preserve existing notes/base relation rows where the
  workflow expects carry-through behavior.
* Projection-only frames such as `concern_event_matrix` must not be promoted as
  canonical JSON.
* The plugin boundary must remain available for Project-Memory-only query and
  rendering tooling, or those tools must move to a separate non-pipeline
  interface.
* Jinja2 and Project Memory rendering helpers must stay out of product runtime
  code.

== Assessment

Project Memory is not just the same as `personal_data`, but it is still a good
rewrite adoption contract.

Recommended order:

. `personal_data` for minimal JSON/spreadsheet/workbook-view/validation
  roundtrip.
. Project Memory for XRef matrix editing, ODS, projection cleanup, and plugin
  boundary behavior.
. Worldbuilding for the broader real-consumer composition of helpers, XRef,
  legends, compact cells, workbook views, and discriminator merge.
