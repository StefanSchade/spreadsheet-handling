
=== Review/Promotion Variant

For safer workflows, reimport should not necessarily overwrite canonical data
immediately.

A safer pattern is:

[source,text]
----
canonical data
-> generated workbook
-> human edit
-> reimport into review area
-> semantic diff
-> explicit promotion
----

This is especially relevant when metadata may be stale or when spreadsheet
edits can affect relations, helper values, or projected views.

== Web Service Use Case

=== Option: Managed Download/Upload Roundtrip

A web application or service produces structured data and invokes a pipeline to
create a spreadsheet.

The spreadsheet is downloaded by the user, edited, and uploaded again.

If the server can ensure that the uploaded workbook is the artifact it created
for that workflow instance, and if the matching metadata is stored server-side
or embedded in the workbook, this can remain a managed roundtrip rather than an
uncontrolled application-boundary crossing.

Possible managed-envelope supports include:

* server-side stored metadata;
* artifact/session identifiers;
* version checks;
* expected pipeline shape;
* upload validation;
* sanitization before readback;
* explicit review and promotion.



=== Risks

Many things can still go wrong:

* users upload the wrong workbook;
* workbook structure is damaged;
* spreadsheet formulas or content may be unsafe;
* metadata may not match the server-side state;
* concurrent changes may have happened since download.

The framework should provide the capabilities needed to build such workflows,
but the workflow builder owns the application-level policy.

Sanitizing uploads against spreadsheet injection or other hostile content may
become an important supporting capability. It is not solved merely by XRef,
Cell Codec, or metadata authority rules.
