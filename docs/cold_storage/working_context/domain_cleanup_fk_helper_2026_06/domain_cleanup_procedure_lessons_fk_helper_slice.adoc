= Domain Cleanup Wave – Lessons Learned from the FK Helper Slice

:toc:
:toclevels: 3

== Purpose

This document records a methodological insight from the FK Helper investigation.

The primary result of the work was not the removal of individual metadata
fields.

The more important result was the discovery of a repeatable approach for
understanding and simplifying mature features without requiring a large-scale
architectural redesign.

This document captures that approach for future transformation slices.

== Initial Situation

The investigation started from a different hypothesis.

A real bug investigation had exposed substantial complexity around FK Helpers,
metadata persistence, runtime provenance and cleanup behavior.

The observed symptoms were:

* high token consumption during bug analysis;
* difficult navigation through metadata structures;
* large registry artifacts;
* uncertainty about ownership and lifecycle of information.

The initial suspicion was that the architecture itself might have become
over-engineered.

Potential suspects included:

* `_meta`;
* the metadata registry;
* lifecycle concepts;
* the overall domain structure.

== What the Domain Audit Found

The documentation-blind domain audit produced a different picture.

The domain contained:

* clear ownership boundaries;
* relatively isolated feature areas;
* successful package formation opportunities;
* stable tests;
* few architectural surprises.

The audit did not support the hypothesis of a fundamentally broken
architecture.

Instead it suggested that the domain was carrying historical layers that had
accumulated over time.

== The FK Helper Investigation

FK Helpers were selected as the first deep slice because they touched many
areas simultaneously:

* relations;
* projections;
* persistence;
* reimport;
* cleanup;
* runtime provenance;
* helper configuration.

The expectation was that FK Helpers would expose architectural weaknesses.

What emerged instead was a distinction between:

* feature complexity;
* implementation sediment.

The functional behavior of FK Helpers could be described in a relatively small
problem-space document.

The implementation complexity came primarily from:

* v1 and v2 representations existing simultaneously;
* persistence-boundary exceptions;
* runtime containment mechanisms;
* compatibility layers;
* historical bug fixes.

The feature itself was smaller than its implementation history.

== Functional Model First

The most important methodological change was to stop reasoning from metadata
categories and start reasoning from feature behavior.

Instead of asking:

* is this canonical?
* is this derived?
* is this runtime?
* is this intent?

the investigation asked:

* what is the feature trying to do?
* which information is required?
* where is that information created?
* where is it consumed?
* where is it persisted?
* where is it removed?

This produced a functional model that was substantially easier to reason about
than the existing metadata structures.

== Inventory Before Refactoring

Before changing code, the metadata paths used by FK Helpers were inventoried.

For each path the investigation recorded:

* producer;
* consumer;
* persistence behavior;
* tests;
* runtime dependency.

This produced a critical result.

Several suspicious structures were discovered to be genuinely load-bearing.

Several other structures turned out to have no active consumers.

Without the inventory these cases would have been difficult to distinguish.

== Evidence Instead of Ontology

One of the major lessons concerns metadata modeling.

The project had gradually accumulated increasingly rich metadata categories and
classifications.

Many of these classifications were reasonable.

However, they encouraged reasoning through abstraction before understanding the
feature itself.

The FK Helper investigation suggests a different approach.

Instead of building a general ontology first:

* understand a concrete feature;
* trace information flow;
* identify active contracts;
* identify historical residue;
* remove only what evidence shows to be unnecessary.

This replaces category-driven reasoning with evidence-driven reasoning.

== The Simplification Loop

The FK Helper work produced a repeatable simplification loop.

Step 1:

Create a functional description of the feature.

Step 2:

Inventory all active information paths.

Step 3:

Identify producers and consumers.

Step 4:

Separate load-bearing structures from historical residue.

Step 5:

Create a narrowly scoped implementation review.

Step 6:

Implement the smallest safe slice.

Step 7:

Validate with tests.

Step 8:

Repeat.

This loop proved capable of removing complexity while preserving behavior.

== Why This Matters

The FK Helper work demonstrated that complexity does not necessarily imply
architectural failure.

A mature feature may contain:

* useful behavior;
* successful abstractions;
* historical compatibility layers;
* obsolete intermediate structures.

Without investigation these become difficult to distinguish.

The simplification loop allows the project to reduce complexity incrementally
without requiring large redesign efforts.

== Implications For Future Slices

Future candidates include:

* legend blocks;
* xref crosstable;
* compact multiaxis;
* workbook views;
* derived column lifecycle.

The recommended approach is:

Do not begin with a new framework.

Do not begin with a new metadata taxonomy.

Do not begin with a new ontology.

Instead:

* build a functional model;
* inventory active information paths;
* identify load-bearing contracts;
* remove confirmed sediment;
* let abstractions emerge from repeated observations.

== Final Assessment

The domain-cleanup wave did not reveal a failing architecture.

Instead it revealed an architecture that appears robust enough to tolerate
incremental simplification.

The main source of friction was not the existence of metadata.

The main source of friction was the accumulation of historical feature-specific
layers whose necessity had not been re-evaluated for some time.

The FK Helper slice demonstrated that these layers can be examined,
challenged, and reduced using evidence from code, tests, and consumers.

This is the primary lesson of the domain-cleanup wave so far.
