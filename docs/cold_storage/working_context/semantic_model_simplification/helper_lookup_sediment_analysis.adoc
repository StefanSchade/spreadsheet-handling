= Helper Lookup / FK Helper Sediment Analysis
:revdate: 2026-06

== Purpose

This note records an analysis observation from Functional Model simplification
work: the current code and backlog history contain a split between FK helpers,
lookup helpers, `enrich_lookup`, helper policies, derived provenance, and
derived-column cleanup. That split may be implementation sediment produced while
generalizing an initially heuristic FK-helper feature.

This is not a backlog, not an FTR, and not a Semantic Model redesign.

== Observed History

The original user-facing need was workbook readability around references:
spreadsheets that show only opaque keys are hard to edit safely, so helper
columns show readable context next to the key while the normalized data remains
authoritative.

The early FK-helper implementation addressed this through fixed naming
heuristics such as convention-based FK columns. Later work generalized the
mechanism by adding explicit lookup helper enrichment:

* `source`, `lookup`, and `on` identify where helper values come from;
* `helpers.fields`, allowlists, and defaults control which helper fields may be
  projected;
* ordering, missing-match behavior, and duplicate-key checks make the
  projection deterministic;
* formula mode lets generated helper cells remain visibly linked to the lookup
  source in workbook renderings;
* derived provenance and cleanup prevent helper columns from becoming canonical
  payload on reimport.

The implementation result is useful, but its naming and documentation now make
several things look like separate concepts: FK helpers, lookup helpers,
`enrich_lookup`, helper lifecycle, helper policies, and derived-column policy.

== Simplification Hypothesis

The Functional Model probably needs one user-facing concept:
helper columns as derived, readable projection context for workbook editing.

Within that concept, there may be implementation variants:

* FK-convention helpers;
* explicit lookup helpers;
* value-backed helper cells;
* formula-backed helper cells;
* cleanup/reimport protection for derived helper columns.

The hypothesis is that `helper lifecycle` should not become a durable
Functional Model or Semantic Model term. `derived_column_policy` should also not
be promoted as a user-facing family unless later evidence shows a distinct user
requirement. It currently reads as a safety mechanism for the helper-column
roundtrip.

== Requirement Shape To Preserve

Before simplifying implementation or vocabulary, the intended requirements need
to be stated independently from the current implementation split.

Likely requirement candidates:

* A workbook view may show helper columns next to a key or relation-bearing
  column so humans can edit with enough context.
* Helper columns are derived context, not canonical payload.
* Users or pipeline authors can explicitly control which helper fields are
  exposed.
* Helper-field allowlists prevent accidental exposure of unwanted lookup fields.
* Missing lookup rows, duplicate lookup keys, and output-column conflicts fail
  clearly unless an explicit tolerant mode is configured.
* Reimport does not persist generated helper columns as source data.
* Helper cleanup does not depend on silent guessing when explicit provenance or
  durable markers are available.
* Formula-backed helper cells are retained only if they remain a real workbook
  readability requirement, not merely because the implementation supports them.

== Requirements-Test-First Simplification Shape

The proposed simplification shape is:

. Finish the Functional Model wording from the user requirement, not from the
  current implementation split.
. Add a narrow integration-test set that expresses those requirements at the
  workflow boundary.
. Use the tests as a safety net while reducing internal duplication,
  terminology, and obsolete heuristic paths.

Candidate integration-test themes:

* Forward workbook projection shows configured helper fields beside the key.
* Reimport of an edited workbook drops helper columns from canonical output.
* Explicit helper-field allowlists reject disallowed fields.
* Duplicate lookup keys and missing required lookup matches fail with clear
  diagnostics.
* FK-style helpers and explicit lookup helpers satisfy the same observable
  helper-column requirements where their behavior overlaps.
* Formula-backed helpers are tested only if retained as a Functional Model
  requirement.

These are not committed backlog items. They are a possible safety-net shape for
future simplification work.

== Non-Decisions

This note does not decide:

* whether `add_lookup_helpers` or `configure_lookup_helpers` should be renamed;
* whether `enrich_lookup` implementation code should be merged into FK-helper
  modules;
* whether `_meta.derived.sheets.*.helper_columns` and
  `_meta.derived.sheets.*.enrich_lookup` should be unified;
* whether any current pipeline step should be removed;
* whether formula mode remains in scope.

Those decisions require Functional Model wording and test coverage first.
