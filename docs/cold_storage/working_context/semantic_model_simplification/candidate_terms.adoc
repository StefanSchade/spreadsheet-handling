= Candidate Semantic Simplification Terms
:revdate: 2026-06

== Purpose

This document records observed candidates for semantic simplification discovered
during Functional Model promotion work.

The entries below are observations, not decisions. They do not propose changes
to the Semantic Model and do not create backlog work.

== Classification Labels

deletion candidate::
  A term or section that a review observed as adding unnecessary semantic
  weight relative to the Functional Model purpose.

review/process artifact::
  Material that may be useful during review but should not necessarily become
  durable Functional Model or Semantic Model content.

local concept::
  A useful chapter-local phrase or structuring concept that does not currently
  need promotion to shared semantic vocabulary.

unresolved::
  A term or pattern where current evidence is insufficient to classify it as
  deletion-worthy or durable shared vocabulary.

=== Compact Editable Cell Contract

Classification: deletion candidate

Observation:

The Functional Model promotion review identified `Compact editable cell
contract` as a coined concept that restated the inquiry's plain meaning: compact
cell values are presentation-oriented workbook editing forms for structured
source attributes. The review observed that the term added semantic weight
without adding functional distinction.

Recorded evidence:

* The Cell Codec inquiry states the plain functional intent: compact cell values
  are presentation-oriented, while structured attributes remain the source
  representation.
* The review recommended dropping the coined term and keeping the plain meaning.
* The revised Cell Codec chapter removed the term.

Status:

Observation recorded only. No Semantic Model change is proposed here.

=== Inline Semantic Consistency Review Sections

Classification: review/process artifact

Observation:

The review identified the inline `Semantic Consistency Review` section in the
Cell Codec Functional Model chapter as semantic-model bookkeeping inside a
Functional Model chapter. It duplicated per-step `Semantic Model Reference`
blocks and pulled the chapter below the intended problem-space altitude.

Recorded evidence:

* The FTR asked for a use-case-anchored Functional Model chapter, with Semantic
  Model references where needed.
* The review recommended deleting or reducing the inline semantic consistency
  review because the established chapter convention defers semantic/registry
  reconciliation to later passes.
* The revised Cell Codec chapter removed the section.

Status:

Observation recorded only. This is not a rule banning all semantic notes in
Functional Model chapters; it records one review finding that inline
reconciliation can become excess when it duplicates existing references.

=== Repeated Neighbor-Concept Distinctions

Classification: deletion candidate

Observation:

The review found that the Cell Codec chapter stated the Cell Codec / XRef
Crosstable / Legend Blocks / Compact Multiaxis boundary in multiple places. The
distinction itself was valid, but repeated statements increased document weight
without adding new functional information.

Recorded evidence:

* The inquiry distinguishes Cell Codec from XRef Crosstable, Legend Blocks, and
  Compact Multiaxis.
* The review recommended keeping one clear distinction section and dropping
  echoes elsewhere.
* The revised chapter keeps a single `Distinction From Neighboring Concepts`
  section.

Status:

Observation recorded only. No shared Semantic Model vocabulary change is
proposed.

=== Cell Codec Transformation Intent

Classification: unresolved

Observation:

The review marked `Cell Codec transformation intent` as borderline rather than a
clear deletion candidate. It has a defensible parallel with existing
Functional Model language such as `XRef projection intent`, but it is still a
chapter-local phrase introduced during promotion.

Recorded evidence:

* The FTR asked the chapter to name the `cell_codecs` registry entry without
  embedding field paths.
* The review recommended keeping the phrase for now, while noting it would be
  the next candidate if a future deletion pass further reduces vocabulary.
* The revised chapter retains the phrase only as a local explanation of the
  declared reversible mapping.

Status:

Observation recorded only. Future promotions may add evidence if similar
`<family> transformation intent` phrases either help alignment or create
unnecessary semantic vocabulary.

=== Encoding Strategy

Classification: local concept

Observation:

The review noted that `Encoding strategy` is not a newly invented semantic term:
the inquiry already uses encoding-strategy language to distinguish reversible
ways of representing several attributes inside one cell.

Recorded evidence:

* The Cell Codec inquiry has an `Encoding Strategies` section.
* The review objected only to treating the term as newly introduced semantic
  vocabulary.
* The revised chapter keeps the functional section because it is directly
  faithful to the inquiry.

Status:

Observation recorded only. This remains a local Functional Model concept unless
future evidence shows it should be treated differently.

=== Implementation Reference Tail

Recorded evidence:

Observation recorded only. This is a Functional Model writing observation, not a

=== Helper Lifecycle

Classification: deletion candidate

Observation:

`Helper lifecycle` appears to be a bundling phrase introduced while turning the
June 2026 transformation-family inventory into follow-up FTRs. The older lookup
FTRs do not establish it as a user-facing concept. They describe helper-column
projection, explicit lookup-helper policy, formula-backed helper cells, derived
provenance, and cleanup/reimport behavior.

Recorded evidence:

* The Functional Model already has `FK Helper Columns` as the user-facing
  concept for readable derived projection values that are dropped on reimport.
* `FTR-EXPLICIT-HELPER-LOOKUP-POLICY-P4` frames explicit lookup enrichment as a
  generic helper lookup/enrichment transform, motivated by generalizing
  convention-based FK helper behavior.
* `FTR-LOOKUP-FK-CONFIGURATION-STEPS-P4` places lookup-helper and FK-helper
  policy under a shared helper-policy direction, even though the FK slice was
  separated for delivery.
* `FTR-DERIVED-PROVENANCE-LIFECYCLE-P4A` discusses lifecycle for the
  `_meta.derived` provenance channel, not a separate user-facing
  `helper lifecycle` requirement.

Status:

Observation recorded only. No Semantic Model or backlog change is proposed
here.

=== Lookup Helpers As A Separate Functional Family

Classification: unresolved

Observation:

The implemented `lookup_helpers` / `enrich_lookup` capability has behavior that
goes beyond the original heuristic FK-helper path: explicit `source` /
`lookup` / `on` configuration, multi-key lookup, allowlists, defaults,
deterministic helper ordering, missing-match policy, and formula mode. However,
the historical FTRs suggest this grew from the desire to generalize FK helpers,
not from a separate user-facing requirement family.

Recorded evidence:

* The older lookup FTRs use names such as `HELPER-LOOKUP-POLICY`,
  `LOOKUP-FK-CONFIGURATION`, and `LOOKUP-HELPER-FORMULA-MODE`.
* No done FTR with `ENRICH` in the filename was found; `enrich_lookup` appears
  to be implementation/runtime naming for the helper lookup capability.
* The user-facing requirement appears to be readable helper columns for workbook
  editing, plus explicit control over where those helper values come from.

Status:

Observation recorded only. Current evidence leans toward treating explicit
lookup helpers as a variant of helper-column projection, but this document does
not decide the Functional Model structure.

=== Derived Column Policy As A User-Facing Family

Classification: deletion candidate

Observation:

`derived_column_policy` looks like a reimport/cleanup safety mechanism for
derived helper columns rather than a user-facing Functional Model family. It is
important operationally, but the user requirement is simpler: helper columns
that were generated for readability must not become canonical payload on
reimport.

Recorded evidence:

* `FTR-DERIVED-PROVENANCE-LIFECYCLE-P4A` describes producers, consumers, and
  cleanup of `_meta.derived` provenance.
* The current helper lookup/FK helper implementation uses derived provenance so
  cleanup and rendering can identify helper columns.
* The Functional Model `FK Helper Columns` chapter already states the core user
  requirement: helper columns are derived, readable projection values and are
  dropped on reimport.

Status:

Observation recorded only. This does not propose deleting implementation code;
it records that the term may be implementation sediment rather than Functional
Model vocabulary.

=== Explicit Helper Projection

Classification: local concept

Observation:

`Explicit helper projection` is a possible local phrase for the common user
requirement behind FK helpers and lookup helpers: show readable, derived helper
values in a workbook projection under explicit user/configuration control. It
may be useful while rewriting the Functional Model, but current evidence does
not justify promoting it to shared Semantic Model vocabulary.

Recorded evidence:

* The old FK-helper path used fixed naming heuristics.
* The lookup-helper work generalized helper enrichment by making source,
  lookup, join key, helper fields, missing behavior, and ordering explicit.
* Both paths produce workbook-visible helper columns that are derived context,
  not canonical payload.

Status:

Observation recorded only. Treat as possible Functional Model wording, not as a
Semantic Model decision.

=== Requirements-First Integration Tests

Classification: review/process artifact

Observation:

The simplification path discussed after the helper/lookup review is not to
model all implementation sediment, but to restate the Functional Model
requirements and then protect them with integration tests before removing or
consolidating technical layers.

Recorded evidence:

* The user's stated direction is Functional Model first: requirements should
  drive simplification, not the current implementation family split.
* Candidate requirements include readable helper columns next to keys,
  explicit helper-field control, no helper leakage into canonical data on
  reimport, deterministic failure for conflicts or missing lookups, and optional
  formula-backed helper cells only if retained as a user requirement.

Status:

Observation recorded only. This is a process observation, not a backlog item or
test plan commitment.
