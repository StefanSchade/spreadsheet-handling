[[semantic-model-contract-matrices]]
== Contract Matrices

=== Purpose

A contract matrix makes semantic expectations compact and reviewable.

It should help avoid contradictions such as:

* docs say missing policy is an error;
* runtime treats missing policy as empty;
* tests accidentally pin the old no-op behavior.

=== Candidate matrix dimensions

For a transformation or primitive:

* reads metadata;
* writes metadata;
* reads configuration;
* writes canonical data;
* writes derived data;
* may infer;
* must not infer;
* missing metadata behavior;
* persistence behavior;
* provenance requirements;
* public surface impact;
* lifecycle stage;
* tests/guards.

=== FK Helper Pilot Matrix

==== Context

The FK-helper family is the first transformation-family pilot for the
contract-matrix machinery because it hit all five pilot-selection
criteria before any other family: recent review pain, bounded primitive
surface, live v2 runtime contract, roundtrip exposure, and existing
registry and guard scaffolding (see the Adoption Strategy chapter for
the full rationale).

The family divides into three participant classes:

* *Policy producers* (`infer_fk_relations`, `configure_fk_helpers`) —
  write the v2 relation policy under `_meta.helper_policies.fk`;
  a pipeline that uses any primitive must run one of these earlier in
  the same pipeline, or rely on persisted policy from a prior run.
* *Primitive consumers* (`add_fk_helpers`, `remove_fk_helpers`,
  `reorder_fk_helpers`, `validate_fk_helpers`) — read the v2 policy
  and the transient derived helper provenance
  (`_meta.derived.sheets.*.helper_columns`); must not perform any
  convention-based FK inference.
* *Transient working metadata* (`_meta.derived.sheets.*.helper_columns`)
  — written by `add_fk_helpers`, consumed within the same pipeline run
  by the other primitives and by the render layer; not persisted as
  canonical roundtrip metadata.

The `FK_PATTERN` / `id_(<target>)` cell-level convention is
wire-format truth used by the IO layer and by `infer_fk_relations` as
its default heuristic mode. Primitive steps do not consult it for
relation identity.

==== Drift found and resolved

Cross-checking the matrix against
`registries/pipeline_step_registry.json` surfaced three stale notes in
the `configure_fk_helpers` entry and one in `infer_fk_relations`.
All were residue from before `FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5`
completed:

. `configure_fk_helpers` meta_contract write notes for the v1 path
  said "consumed by current FK-helper primitives" — stale; updated to
  record it as a residual producer-side write with no current runtime
  consumer.
. `configure_fk_helpers` meta_contract write notes for the v2 relations
  path said "compatibility for the period in which ... primitives still
  consume v1" — stale; updated to record that the runtime consumer
  transition is complete and the bridge is active on the producer side
  only.
. `configure_fk_helpers` purpose said "persist them as canonical v1
  helper policy metadata for current FK-helper primitives" — stale;
  updated to describe v2 as the runtime contract and v1 as a residual
  producer-side write.
. `infer_fk_relations` purpose said "current FK-helper primitives do
  not pick up inferred relations until
  FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5 makes them v2-aware" —
  stale; the FTR has completed and primitives are v2-aware now.

These are documentation-only drift corrections.  The registry
`contract_lifecycle` block in `meta_registry.yaml` for `helper_policies`
was already accurate (`transition_bridge.status: active`,
`transition_bridge.bridge_type: dual_write`) and required no change.
No runtime source files changed.

==== Guard coverage assessment

The primitive no-convention-inference contract is already guarded and
stable:

* `tests/architecture/dependency_guards/test_fk_helper_primitives_consume_policy.py`
  pins that the `fk_helpers` primitive package, `reorder_fk_helpers`,
  and `validate_fk_helpers` do not import `FK_PATTERN`,
  `detect_fk_columns`, or `build_registry` from `core.fk`, and that
  each primitive references at least one policy or provenance token.
* `tests/architecture/dependency_guards/test_architecture_guard_fk_helpers.py`
  guards that `pipeline.steps` does not import FK-helper domain symbols
  or manipulate `_meta` / `derived` / `helper_columns` directly.
* `tests/architecture/dependency_guards/test_fk_helpers_package_reexports.py`
  pins the public dotted-path symbols on the `fk_helpers` package root.

These guards cover the most silent drift risk: a future caller
re-introducing convention-based inference into primitives. No new guard
was added by this matrix pilot because:

* The producer-side residual v1 write in `configure_fk_helpers` is
  already recorded in `meta_registry.yaml` under `contract_lifecycle`;
  a guard against its consumer absence would duplicate the registry
  record without adding test oracle value.
* The `derived.sheets.*.helper_columns` transient lifecycle
  (pruning, orphan handling, schema validation) is deferred to
  `FTR-DERIVED-PROVENANCE-LIFECYCLE-P4A`; a guard here is premature
  while that contract is unsettled.
* The duplicate-id check in `validate_fk_helpers` using
  `defaults['id_field']` for non-declared FK sheets is a stable
  behavior but a narrow, schema-adjacent concern that is adequately
  covered by `test_fk_check.py`.

Rows in this matrix that are stable enough for guard consideration are
marked by already having an existing guard (all four primitive rows).
The two producer rows (`infer_fk_relations`, `configure_fk_helpers`)
are stable on their v2 write contract and could support a guard
verifying that each writes `schema_version: 2`; that is deferred as
a future narrow guard if silent drift surfaces.

==== Deferred concerns

* Provenance lifecycle for `derived.sheets.*.helper_columns` (pruning,
  stale / orphan handling, schema validation) — owned by
  `FTR-DERIVED-PROVENANCE-LIFECYCLE-P4A`; no lifecycle policy beyond
  "transient, recomputed per run" is recorded here.
* Producer-side residual v1 write in `configure_fk_helpers` — bridge
  state is active; removal condition and owning FTR are recorded in
  `meta_registry.yaml` under `contract_lifecycle`.
* Duplicate-id check behavior in `validate_fk_helpers` for non-declared
  FK sheets (`defaults['id_field']`) — stable but outside the core
  FK relation policy contract; deferred to per-validation adoption work.
