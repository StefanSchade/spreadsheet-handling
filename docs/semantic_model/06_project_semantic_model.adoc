== Project Semantic Model

=== Core entities

The semantic model treats the following as first-class entities. Each one
carries semantic obligations beyond its data shape.

* *transformations* -- the primitive and step-level operations that read or
  write frames, metadata, or render state;
* *pipeline steps* -- composed operations the registry and YAML configuration
  describe and orchestrate;
* *metadata attributes* -- fields under `_meta.*` that carry meaning between
  steps and across roundtrips;
* *canonical frame data* -- the project's authoritative dataset
  representation, decoupled from any spreadsheet projection;
* *derived data* -- materialized fields and columns whose authority comes from
  a producer step, not from user edit (for example, FK-helper columns recorded
  under `_meta.derived.sheets.*.helper_columns`);
* *configuration* -- YAML and CLI inputs that select transformations,
  primitives, and named semantic modes;
* *persisted sidecar metadata* -- `_meta`-shaped state that survives between
  runs;
* *render directives* -- metadata that controls workbook presentation and is
  not part of canonical data;
* *workbook-visible cells/columns* -- the public projection a user edits;
* *registry entries* -- `registries/meta_registry.yaml` and
  `registries/pipeline_step_registry.json` records that describe producers,
  consumers, and (in the rule-ready direction) contracts;
* *lifecycle events* -- writes, reads, supersessions, prunings, and
  reimports that change metadata state over time.

[NOTE]
====
Metadata *meaning* is distinct from metadata *shape*. Two fields with the same
JSON shape may carry different semantics if they live in different sets (for
example, a column index in `_meta.derived.sheets.*.helper_columns` is derived
provenance, while the same shape in a configuration block would be input
policy). The semantic model classifies fields by meaning; the schema describes
shape.
====

=== Transformations

A transformation may:

* read frame data;
* write frame data;
* read metadata;
* write metadata;
* derive helper data;
* remove helper data;
* validate data;
* render data;
* prepare data for adapter output.

Questions per transformation (to be answered by the transformation's owning
FTR, registry entry, and -- where stable -- architecture guard):

* What does it read?
* What does it write?
* Does it mutate canonical data, metadata, or only render state?
* Does it consume policy, provenance, config, or inferred structure?
* May it infer semantics, or must it consume explicit metadata? The FK-helper
  primitives, for example, no longer infer relation identity from column
  names and must consume `_meta.helper_policies.fk` (policy) or
  `_meta.derived.sheets.*.helper_columns` (provenance).
* What happens when required metadata is absent? (See the Ambiguity Policy
  for the "must become explicit" criteria.)
* Which registry entry describes it?
* Which tests or guards enforce its contract?

=== Metadata attributes

A metadata attribute is a meaning-bearing field in project metadata. It is
characterized by the semantic sets it belongs to, not by where it lives in
the JSON tree.

Questions per metadata attribute:

* What does it mean (in project terms, not in JSON-schema terms)?
* Who may produce it? Who currently does?
* Who may consume it? Who currently does?
* Is it persistent or transient (in the sense defined under Overlapping
  Semantic Sets)?
* Is it canonical, derived, render-related, operational, or diagnostic?
* Is it user-editable through the workbook surface?
* Is it roundtrip-relevant? Does it survive a canonical-to-workbook-to-
  canonical cycle?
* Does it require provenance, or is the producer obvious from its location?
* Does it have lifecycle/versioning rules (transition bridge, deprecation,
  removal owner)?
* What happens when it conflicts with configuration or newly generated data?
  Which precedence rule applies?
