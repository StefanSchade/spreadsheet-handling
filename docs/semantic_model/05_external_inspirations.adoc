== External Inspirations

[NOTE]
====
This section is inspirational, not normative.

The project borrows useful concepts from each tradition listed below without
importing any methodology wholesale. Only the *project interpretation* blocks
are load-bearing for `spreadsheet-handling`; the "relevant ideas" blocks
summarize the external concept in its own terms and do not bind the project.
====

[NOTE]
====
*Internal split under `FTR-FUNCTIONAL-SEMANTIC-MODEL-SPLIT-P5`.*

This chapter intentionally pairs two kinds of content:

* The *"Relevant ideas"* block in each subsection is *Explanatory
  Context*. It belongs to the Functional Model's problem-space framing
  (see `docs/functional_model/ch05_problem_space_overview/`) and is
  preserved here only because it has historically been read alongside
  the project interpretation. It is not a Semantic Model contract.
* The *"Project interpretation"* block in each subsection is
  *contract-adjacent*. It states what the project does and does not
  adopt from the surrounding tradition and is the load-bearing part
  for the Semantic Model.

Reviewers and agents should treat the "Project interpretation" blocks
as the contract surface of this chapter. The "Relevant ideas" blocks
remain as background and may be relocated into the Functional Model in
a follow-up editorial pass; no new ideas, sets, or rules are added
here.
====

=== Specification thinking

Relevant ideas:

* A specification helps understand a system before building or changing it.
* Writing precise descriptions exposes sloppy thinking earlier than code does.
* The hardest part is choosing the right abstraction, not the notation.
* Important properties can be expressed as invariants that survive change.
* Implementation can be viewed as satisfying or refining an abstract contract.

Project interpretation:

* The project does not adopt a formal specification language (no TLA+, no Z,
  no Alloy). The cost of a heavyweight formalism would exceed the value at
  this stage.
* What the project does adopt is the *habit* of writing small, explicit
  semantic contracts for behavior that affects deterministic transformation
  or roundtrip semantics, before that behavior is wired into runtime.
* The existing carriers for this habit are FTR acceptance criteria, review
  artifacts, and -- where invariants are stable enough -- architecture
  guards under `tests/architecture/`. Each acts as a partial specification
  appropriate to its surface.
* The seed roundtrip statement
  (`canonical -> projection -> edited workbook -> deterministic
  reconstruction`) is the most important project-level invariant; other
  contracts are evaluated by whether they preserve it.

=== Data contracts

Relevant ideas:

* Producers and consumers should agree on what data *means*, not only on its
  shape.
* Breaking changes should be visible before runtime failure.
* Consumers should not silently reinterpret missing or changed fields.
* A contract names producers, consumers, semantics, and a stability posture.

Project interpretation:

* Metadata fields in `_meta.*` (for example `_meta.helper_policies.fk`,
  `_meta.derived.sheets.*.helper_columns`, `_meta.helper_policies.lookup`)
  are data contracts between pipeline steps, not internal scratch state.
* Registries (`registries/meta_registry.yaml`,
  `registries/pipeline_step_registry.json`) can describe the contract surface;
  they do not yet enforce it at runtime, and the semantic model does not
  propose that they should at this phase.
* Architecture guards check stable parts of the contract that are cheap to
  check and high-value to protect (for example, that FK-helper primitives do
  not regress to convention-based inference).
* "Shape" and "meaning" can drift independently: a field whose JSON shape is
  unchanged can still acquire a new semantic role, and the contract must
  follow the meaning, not the shape.

=== Schema evolution

Relevant ideas:

* Schemas evolve; old and new shapes may legitimately coexist for a window.
* Compatibility rules (forward, backward, both) should be explicit and named.
* Deprecation and removal need owners and conditions, not just intentions.
* Migration is a property of the system, not of the schema in isolation.

Project interpretation:

* *Metadata schema version* (the shape of a field) and *semantic contract
  version* (what that field means and who depends on it) are distinct
  concerns. A v2 shape may still carry v1 semantics during a transition, and
  vice versa.
* Transition bridges in this project name five things explicitly: current
  writers, current readers, target readers, removal conditions, and the
  owning FTR. The Problem Statement already references this pattern as the
  remedy for lifecycle ambiguity.
* The v1 -> v2 evolution of `_meta.helper_policies.fk` (per-target shape
  alongside `helper_policies.fk.relations`) is the working example of
  coexistence: both shapes are accepted while the migration is in flight,
  and the owning FTRs track removal of the older shape.

=== Consumer-driven contracts

Relevant ideas:

* Knowing what a producer writes is not enough; the load-bearing question is
  what *consumers* rely on.
* A producer change is only safe if every current consumer either tolerates
  it or has migrated.
* Consumer expectations should be visible to the producer, not hidden in
  consumer-local code.

Project interpretation:

* A metadata field's contract should record not only producers but also
  consumers (registries, guards, pipeline steps, public examples). A field
  with an unknown consumer set has no contract -- only a current behavior.
* Review should ask whether registry consumer lists still match runtime
  consumers after a primitive change. The FK-helper phase showed that this
  check is easy to forget and cheap to perform when surfaced.
* Missing consumer entries are *semantic drift*, not harmless documentation
  drift, when guards, registries, or runtime decisions depend on the field.
  The Ambiguity Policy treats this as ambiguity that must become explicit.

=== API deprecation and migration policies

Relevant ideas:

* Public behavior cannot be changed casually; users coordinate around it.
* Deprecation needs a migration path, a timeline, and removal conditions, not
  only a notice.
* The deprecated path and the new path must coexist long enough for migration
  to complete.

Project interpretation:

* The project's public surface for this purpose is wider than "the Python
  API": YAML pipeline configuration, the CLI, the demo, and the user guide
  are part of the contract that downstream consumers depend on. Public Pages
  bundling is the gate that distinguishes public from internal.
* Internal metadata is not automatically private: when a field survives
  roundtrip or is consumed by multiple steps, it acquires lifecycle
  obligations similar to a public API and needs the same explicit
  deprecation/migration treatment.
* The `prefix` fallback in `remove_fk_helpers` is the working migration
  example: a transitional safety net retained while explicit policy /
  provenance was rolled out, with removal scoped to the owning FTRs rather
  than to ad-hoc cleanup.
* FTRs own the deprecation lifecycle; review artifacts record the cut-over;
  guards (where they exist) prevent regression. No single one of these
  mechanisms is sufficient on its own, but together they form the project's
  practical deprecation policy.
