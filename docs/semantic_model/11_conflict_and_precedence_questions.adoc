== Conflict And Precedence Questions

=== Source classes

Metadata can enter `_meta` from several source classes, each with a different
default authority and a different set of consumers it must be safe for.

[cols="1,3"]
|===
| Source class | Project examples and authority posture

| explicit YAML config
| pipeline configuration, FK relation declarations,
  `_meta.helper_policies.fk.relations` written by `configure_fk_helpers`.
  High authority for Configurable attributes; conflicts against this should
  raise unless the contract explicitly allows override.

| CLI override
| ad-hoc per-invocation overrides. Same authority as YAML config for the
  duration of one run; should not be silently persisted as if it were
  equivalent to a declared policy.

| persisted metadata sidecar
| `_meta` carried across runs. Authoritative for Persistent attributes
  whose producer is the previous run; bridges between runs but does not
  outrank fresh explicit config.

| workbook-embedded metadata
| metadata serialized into the workbook surface. Trusted only insofar as
  the workbook is itself part of the public contract; subject to user edit.

| workbook-visible user edits
| edits to UserEditable cells/columns. Authoritative for Canonical content;
  must not be silently overwritten by Derived or RenderConsumed output.

| bootstrap logic
| reconstruction from existing data when persisted metadata is missing.
  Authoritative only as a fallback and must be visible as such.

| heuristic inference step
| for example, `infer_fk_relations`. HeuristicallyInferred; loses to
  explicit Policy or Configurable values.

| transformation output
| `_meta.helper_policies.fk`, `_meta.derived.sheets.*.helper_columns`, and
  similar. Authoritative within its producer step's scope; ProducerOwned.

| render layer output
| RenderConsumed + Transient. Must not survive as Canonical or Persistent
  unless an explicit promotion rule applies.

| adapter readback
| values reread from the persisted artifact after a write. Used to verify
  roundtrip determinism, not to amend canonical truth.
|===

=== Precedence questions

For each relevant metadata attribute, the owning adoption slice should be
able to answer the following questions. These are *questions* the slice must
resolve, not pre-decided rules.

* Which source class is authoritative?
* Which source may overwrite which other source?
* Is conflict an error, or is one of the sources designated as the winner?
* Is merge allowed? If so, on what key, and how is order resolved?
* Is fallback allowed (use source B if source A is absent), or must A be
  present?
* Is explicit override required (refuse to overwrite without an explicit
  marker)?
* Is the historical value retained as audit, or discarded?
* Does "newer wins" ever apply? Within which authority class?
* Does user-visible workbook content override hidden metadata? (Default for
  Canonical: yes; for RenderConsumed-only: no.)
* Does persisted metadata override regenerated metadata? Or is each run a
  fresh write for that attribute?

The Overlapping Semantic Sets chapter informs default answers (for example,
*Configurable* + *Policy* attributes default to "explicit config wins over
inference"), but per-attribute decisions remain the slice's responsibility.

=== Resolve conflicts early when possible

A conflict that can be detected at write time should be resolved at write
time. Deferring resolution to consumers multiplies the resolution rule
across every consumer and is the most common path to drift.

This does not mean the writer must always be the resolver. It means the
writer is responsible for leaving metadata in a state whose meaning is
unambiguous, even if that requires raising an error rather than silently
writing competing values.

Where genuine late resolution is required (for example, when only the
render layer knows whether a value is needed), the late-resolution rule
itself must be documented and reviewable.

=== No customer identity as semantics

Customer- or context-specific behavior must not enter runtime code as
identity checks. The Problem Statement chapter covers the underlying
anti-pattern; in the precedence context the rule is narrower:

*Source classes are defined by role, not by identity.* "explicit YAML
config" is a source class because it has a defined contract and lifecycle.
"config for customer X" is not a source class; it is an identity-keyed
branch wearing a config-shaped costume.

Bad:

[source,python]
----
if customer == "x":
    behave_differently()
----

Preferred:

[source,yaml]
----
# A named semantic mode is reviewable as a contract.
semantic_mode: explicit_named_mode
----

Each semantic mode needs:

* name;
* contract;
* tests;
* documentation;
* migration/removal ownership if temporary (in which case it is also a
  TransitionBridge).
