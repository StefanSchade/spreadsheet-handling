[[semantic-model-management-summary]]
== Management Summary

=== Why this document exists

`spreadsheet-handling` has reached a point where the main source of complexity
is no longer syntax, layering, or basic transformation mechanics.

The hard part is semantic consistency across:

* canonical data;
* metadata;
* transformation steps;
* registries;
* rendered workbook surfaces;
* reimport / roundtrip behavior;
* documentation;
* tests and architecture guards;
* AI-assisted implementation and review workflows.

The recent FK-helper migration showed that a technically clean architecture is
not enough. Even with a technical model, layered code, registries, guards, and
reviews, feature work can still get stuck on semantic ambiguity.
(the critical review with blocking issues of the past phase testifies this)

=== Main thesis

Deterministic, roundtrip-capable spreadsheet transformations require explicit
semantic contracts.

Ambiguity is not automatically a defect. However, ambiguity that affects runtime
behavior, roundtrip reconstruction, public contracts, metadata lifecycle, or
migration truth must be made explicit.

=== What this document should provide

This semantic model should:

* explain why semantic ambiguity matters in this project;
* define the core semantic concepts used by the project;
* describe the role of registries as first-class project artifacts;
* model transformations and metadata attributes as overlapping semantic sets;
* make lifecycle, precedence, and update questions explicit;
* define a no-big-bang adoption strategy;
* prepare the project for contract matrices, semantic guards, and possibly
  later rule-ready or rule-driven validation.

=== What this document should not become

This document should not become:

* a theoretical essay detached from implementation;
* a second technical model;
* a rule engine specification before the concepts are stable;
* a reason to formalize every harmless ambiguity;
* a mandate for a global rewrite.

[[semantic-model-relation-to-other-models]]
== Relation To Other Models

=== Functional Model

The functional model describes the user-facing problem space and requirements.

It answers questions such as:

* What should the tool help users do?
* Which workflows matter?
* Which user-visible capabilities exist?

=== Technical Model

The technical model describes how the system is structured technically.

It answers questions such as:

* Which architectural layers exist?
* Where do adapters, domain logic, rendering, and pipelines live?
* Which module boundaries and dependency rules apply?

=== Semantic Model

The semantic model describes meaning-bearing contracts between data,
metadata, transformations, registries, and roundtrip behavior.

It answers questions such as:

* What does a metadata field mean?
* Who is allowed to produce or consume it?
* Is it canonical, derived, persistent, transient, user-editable, or
  render-only?
* What does missing metadata mean?
* What happens when configuration, persisted metadata, bootstrap logic, and
  transformation output disagree?
* Which ambiguity must become an error, and which ambiguity may remain
  intentionally unresolved?

[[semantic-model-problem-statement]]
== Problem Statement

=== Architecture reduced structural friction, but semantic friction remains

The project already has substantial structural discipline:

* technical model;
* layered architecture;
* registries (now relocated to `registries/` as operational artifacts rather
  than documentation, per the registry-artifact relocation review);
* FTRs;
* architecture guards;
* review artifacts;
* public/demo documentation gates.

Nevertheless, recent FTRs still required unexpected review and follow-up work.
Large models consumed many tokens not because they could not edit code, but
because they had to reconstruct and reconcile semantic intent across multiple
artifacts that each looked locally consistent.

Concrete examples of semantic friction observed in the FK-helper phase:

* A primitive's runtime behavior did not match the FTR and docs. The
  `FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5` review found that
  `validate_fk_helpers` treated absent v2 policy / provenance as "nothing to
  validate", while the FTR and updated docs stated that missing required
  policy should be reported as a clear error. Three artifacts (code,
  FTR, docs) each described different semantics for the same input.
* Missing metadata had two incompatible meanings -- "nothing to validate"
  and "configuration error" -- and no contract said which one applied to
  which primitive. This is the canonical missing-policy ambiguity that
  motivates an explicit policy contract for FK-helper primitives.
* Registry consumer lists drifted after runtime changes. When primitives
  migrated to `_meta.helper_policies.fk` and derived provenance under
  `_meta.derived.sheets.*.helper_columns`, registry entries describing
  producers and consumers were not always updated in lockstep, so the
  registry described a state that runtime no longer produced.
* Active documentation surfaces still described old convention-based
  FK-helper behavior after primitives had moved to explicit policy /
  provenance. The `FTR-FK-HELPER-DOCS-DEMO-REALIGNMENT-P5` review shipped
  with minor fixes because two active doc surfaces outside the main
  modified set still carried stale convention-oriented wording.
* Lifecycle transition bridges required explicit current writers, current
  readers, target readers, and removal owners. Without that, "v1 vs v2"
  policy state could not be discussed without re-deriving it from code.
* The registry relocation itself was a semantic signal, not just a path
  change: moving registries out of `docs/technical_model/ch05_registries/`
  into `registries/` declares that registries are operational artifacts
  consulted at runtime and by guards, not explanatory prose. Code, docs,
  and guards had to be updated together for that signal to land.

=== Human teams, AI agents, and semantic ambiguity

Human teams and AI agents handle semantic ambiguity differently, and this
difference shifts as the project ages.

==== Early project stage

Human teams begin by debating terminology and forming project-specific
language. This is visible as:

* domain terms;
* in-group project vocabulary;
* architecture decision notes;
* conventions not yet encoded in code.

AI agents can appear surprisingly effective early on because they infer intent
from a broad, relatively homogeneous corpus of software-engineering practice.
At this stage, both humans and agents tolerate ambiguity well because the
system is small enough to keep in working memory.

==== Growing project stage

As the project grows, semantic contradictions accumulate across FTRs, docs,
registries, tests, and code.

Human teams often fail to notice all contradictions because they rely on shared
history, local context, and informal memory. A reviewer who already knows what
`add_fk_helpers` is supposed to do will read past wording that no longer
matches the implementation.

AI agents notice contradictions more aggressively. They compare artifacts
side-by-side without that shared history, and they spend tokens reconciling
intent before they can answer. This can feel overcritical, but each surfaced
contradiction tends to map to an unresolved semantic contract that a human
reviewer would also have to resolve sooner or later. The FK-helper phase
review record bears this out: most blocking findings were not bugs in code,
but disagreements between code, FTR, registry, and docs.

==== Legacy project stage

In legacy systems, ambiguity is often operationalized as customer- or
context-specific behavior:

[source,text]
----
A consumer expected different behavior
-> add a configuration switch
-> add a branch keyed on the consumer's identity
-> preserve old behavior
-> no one remembers what the special case means
----

The problem is not configuration itself. The problem is configuration without
a named semantic mode: a branch that encodes *who* asked for behavior rather
than *what* the behavior actually is.

Avoid identity-keyed switches such as:

[source,yaml]
----
# Anti-pattern: identity-keyed branch with no named semantic mode.
consumer_x_mode: true
legacy_dataset_quirk: true
----

Prefer named semantic modes that describe the contract the runtime should
honor, using the project's existing metadata namespaces (`_meta.helper_policies.*`,
derived helper provenance, roundtrip contracts):

[source,yaml]
----
# Named semantic modes, aligned with current project naming.
_meta:
  helper_policies:
    fk:
      missing_policy: error | ignore | infer_from_convention
      cleanup_strategy: provenance_only | policy_or_provenance
  roundtrip:
    conflict_strategy: fail | prefer_workbook | prefer_canonical
----

Each named mode is a contract the registry, docs, and guards can describe and
review. An identity-keyed branch is not, because its meaning lives in
institutional memory rather than in the artifact.

=== Why this matters more here than in ordinary CRUD systems

Many business systems can tolerate ambiguity longer because meaning is resolved
socially or operationally:

* synchronous deployments;
* manual coordination;
* end-to-end smoke tests;
* implicit team knowledge;
* "the team knows how this field is used".

`spreadsheet-handling` is more sensitive because its core promise is not merely
CRUD, workflow, or reporting.

Its core promise is closer to:

[source,text]
----
canonical data
-> human-readable/editable spreadsheet projection
-> edited workbook
-> deterministic reconstruction of canonical data
----

A deterministic roundtrip cannot safely depend on ambiguous interpretation of
missing metadata, implicit helper conventions, stale lifecycle assumptions, or
consumer-local fallback rules.

[[semantic-model-ambiguity-policy]]
== Ambiguity Policy

Not every ambiguity is a bug. The semantic model exists to distinguish
ambiguity that must become explicit from ambiguity that may remain informal,
not to eliminate every imprecise sentence in the repository.

=== Ambiguity that must become explicit

Ambiguity should become explicit -- in code, registries, guards, or a
contract document -- when it affects any of the following.

* *Deterministic runtime behavior.*
  Example: whether `validate_fk_helpers` treats absent v2 policy / provenance
  as "nothing to validate" or as "missing required policy" determines whether
  a pipeline run silently passes or fails. The
  `FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5` review surfaced this exact
  divergence between code, FTR, and docs.
* *Roundtrip reconstruction.*
  Example: cleanup of materialized helper columns relies on derived helper
  provenance under `_meta.derived.sheets.*.helper_columns`; if it is
  ambiguous whether absent provenance means "no helpers were materialized"
  or "provenance was lost", reimport may delete user columns or leave stale
  helpers behind.
* *Public user-facing contracts.*
  Example: the FK-helper docs/demo realignment review (`FTR-FK-HELPER-DOCS-DEMO-REALIGNMENT-P5_review.adoc`)
  flagged active doc surfaces that still described old convention-based
  behavior after the primitives had moved to explicit policy / provenance.
  Public docs are part of the contract, not a separate stream.
* *Metadata lifecycle.*
  Example: when `_meta.helper_policies.fk` migrates from v1 to v2, the
  status of v1-shaped writers/readers, the cut-over point, and the removal
  owner must be recorded; otherwise neither humans nor agents can tell
  which shape is currently authoritative.
* *Migration / deprecation truth.*
  Example: prefix-based fallback in `remove_fk_helpers` was retained as a
  transitional safety net while explicit policy/provenance was rolled out;
  the deprecation horizon and removal conditions belong in a contract, not
  in tribal memory.
* *Producer/consumer responsibilities.*
  Example: registry consumer lists that drift after primitives change
  producers (or readers) hide the fact that the producer/consumer contract
  itself has changed. Consumer-driven contracts (see External Inspirations)
  treat this as semantic drift, not documentation drift.
* *Conflict or precedence resolution.*
  Example: when configuration, persisted metadata, and freshly inferred
  relations disagree about `_meta.helper_policies.fk.relations`, the
  precedence order must be named and reviewable.

=== Ambiguity that may remain tolerated

Ambiguity may remain tolerated when it has no behavioral consequence the
project commits to upholding.

* *Historical only.*
  Example: past review notes and archived FTRs that describe earlier
  conventions need not be retroactively reconciled with current semantics,
  as long as current docs and registries are coherent.
* *Local to prose wording with no behavioral consequence.*
  Example: phrasing differences between "helper column" and "derived helper
  column" in narrative prose are tolerable until a guard, registry, or
  primitive depends on the distinction.
* *Intentionally deferred and marked as such.*
  Example: this document's own draft-scaffold marker explicitly defers
  hardening individual chapters to per-section FTRs. Deferral is acceptable
  when it is named and owned, not when it is silent.
* *Not used by runtime, tests, registries, or public examples.*
  Example: an internal helper docstring may use informal language as long as
  no guard or contract references it. Once a registry or guard reads the
  same term, the informal language becomes a contract surface.

=== Why this distinction matters

The cost of "make every ambiguity explicit" is paid in artifact churn,
review cycles, and false-positive contradictions. The cost of "tolerate every
ambiguity" is paid in incidents, broken roundtrips, and stale registries.
The policy above is the project's attempt to spend the first cost only where
the second cost would otherwise be unbounded.

[[semantic-model-external-inspirations]]
== External Inspirations

[NOTE]
====
This section is inspirational, not normative.

The project borrows useful concepts from each tradition listed below without
importing any methodology wholesale. Only the *project interpretation* blocks
are load-bearing for `spreadsheet-handling`; the "relevant ideas" blocks
summarize the external concept in its own terms and do not bind the project.
====

=== Specification thinking

Relevant ideas:

* A specification helps understand a system before building or changing it.
* Writing precise descriptions exposes sloppy thinking earlier than code does.
* The hardest part is choosing the right abstraction, not the notation.
* Important properties can be expressed as invariants that survive change.
* Implementation can be viewed as satisfying or refining an abstract contract.

Project interpretation:

* The project does not adopt a formal specification language (no TLA+, no Z,
  no Alloy). The cost of a heavyweight formalism would exceed the value at
  this stage.
* What the project does adopt is the *habit* of writing small, explicit
  semantic contracts for behavior that affects deterministic transformation
  or roundtrip semantics, before that behavior is wired into runtime.
* The existing carriers for this habit are FTR acceptance criteria, review
  artifacts, and -- where invariants are stable enough -- architecture
  guards under `tests/architecture/`. Each acts as a partial specification
  appropriate to its surface.
* The seed roundtrip statement
  (`canonical -> projection -> edited workbook -> deterministic
  reconstruction`) is the most important project-level invariant; other
  contracts are evaluated by whether they preserve it.

=== Data contracts

Relevant ideas:

* Producers and consumers should agree on what data *means*, not only on its
  shape.
* Breaking changes should be visible before runtime failure.
* Consumers should not silently reinterpret missing or changed fields.
* A contract names producers, consumers, semantics, and a stability posture.

Project interpretation:

* Metadata fields in `_meta.*` (for example `_meta.helper_policies.fk`,
  `_meta.derived.sheets.*.helper_columns`, `_meta.helper_policies.lookup`)
  are data contracts between pipeline steps, not internal scratch state.
* Registries (`registries/meta_registry.yaml`,
  `registries/pipeline_step_registry.json`) can describe the contract surface;
  they do not yet enforce it at runtime, and the semantic model does not
  propose that they should at this phase.
* Architecture guards check stable parts of the contract that are cheap to
  check and high-value to protect (for example, that FK-helper primitives do
  not regress to convention-based inference).
* "Shape" and "meaning" can drift independently: a field whose JSON shape is
  unchanged can still acquire a new semantic role, and the contract must
  follow the meaning, not the shape.

=== Schema evolution

Relevant ideas:

* Schemas evolve; old and new shapes may legitimately coexist for a window.
* Compatibility rules (forward, backward, both) should be explicit and named.
* Deprecation and removal need owners and conditions, not just intentions.
* Migration is a property of the system, not of the schema in isolation.

Project interpretation:

* *Metadata schema version* (the shape of a field) and *semantic contract
  version* (what that field means and who depends on it) are distinct
  concerns. A v2 shape may still carry v1 semantics during a transition, and
  vice versa.
* Transition bridges in this project name five things explicitly: current
  writers, current readers, target readers, removal conditions, and the
  owning FTR. The Problem Statement already references this pattern as the
  remedy for lifecycle ambiguity.
* The v1 -> v2 evolution of `_meta.helper_policies.fk` (per-target shape
  alongside `helper_policies.fk.relations`) is the working example of
  coexistence: both shapes are accepted while the migration is in flight,
  and the owning FTRs track removal of the older shape.

=== Consumer-driven contracts

Relevant ideas:

* Knowing what a producer writes is not enough; the load-bearing question is
  what *consumers* rely on.
* A producer change is only safe if every current consumer either tolerates
  it or has migrated.
* Consumer expectations should be visible to the producer, not hidden in
  consumer-local code.

Project interpretation:

* A metadata field's contract should record not only producers but also
  consumers (registries, guards, pipeline steps, public examples). A field
  with an unknown consumer set has no contract -- only a current behavior.
* Review should ask whether registry consumer lists still match runtime
  consumers after a primitive change. The FK-helper phase showed that this
  check is easy to forget and cheap to perform when surfaced.
* Missing consumer entries are *semantic drift*, not harmless documentation
  drift, when guards, registries, or runtime decisions depend on the field.
  The Ambiguity Policy treats this as ambiguity that must become explicit.

=== API deprecation and migration policies

Relevant ideas:

* Public behavior cannot be changed casually; users coordinate around it.
* Deprecation needs a migration path, a timeline, and removal conditions, not
  only a notice.
* The deprecated path and the new path must coexist long enough for migration
  to complete.

Project interpretation:

* The project's public surface for this purpose is wider than "the Python
  API": YAML pipeline configuration, the CLI, the demo, and the user guide
  are part of the contract that downstream consumers depend on. Public Pages
  bundling is the gate that distinguishes public from internal.
* Internal metadata is not automatically private: when a field survives
  roundtrip or is consumed by multiple steps, it acquires lifecycle
  obligations similar to a public API and needs the same explicit
  deprecation/migration treatment.
* The `prefix` fallback in `remove_fk_helpers` is the working migration
  example: a transitional safety net retained while explicit policy /
  provenance was rolled out, with removal scoped to the owning FTRs rather
  than to ad-hoc cleanup.
* FTRs own the deprecation lifecycle; review artifacts record the cut-over;
  guards (where they exist) prevent regression. No single one of these
  mechanisms is sufficient on its own, but together they form the project's
  practical deprecation policy.

[[semantic-model-project-semantic-model]]
== Project Semantic Model

=== Core entities

The semantic model treats the following as first-class entities. Each one
carries semantic obligations beyond its data shape.

* *transformations* -- the primitive and step-level operations that read or
  write frames, metadata, or render state;
* *pipeline steps* -- composed operations the registry and YAML configuration
  describe and orchestrate;
* *metadata attributes* -- fields under `_meta.*` that carry meaning between
  steps and across roundtrips;
* *canonical frame data* -- the project's authoritative dataset
  representation, decoupled from any spreadsheet projection;
* *derived data* -- materialized fields and columns whose authority comes from
  a producer step, not from user edit (for example, FK-helper columns recorded
  under `_meta.derived.sheets.*.helper_columns`);
* *configuration* -- YAML and CLI inputs that select transformations,
  primitives, and named semantic modes;
* *persisted sidecar metadata* -- `_meta`-shaped state that survives between
  runs;
* *render directives* -- metadata that controls workbook presentation and is
  not part of canonical data;
* *workbook-visible cells/columns* -- the public projection a user edits;
* *registry entries* -- `registries/meta_registry.yaml` and
  `registries/pipeline_step_registry.json` records that describe producers,
  consumers, and (in the rule-ready direction) contracts;
* *lifecycle events* -- writes, reads, supersessions, prunings, and
  reimports that change metadata state over time.

[NOTE]
====
Metadata *meaning* is distinct from metadata *shape*. Two fields with the same
JSON shape may carry different semantics if they live in different sets (for
example, a column index in `_meta.derived.sheets.*.helper_columns` is derived
provenance, while the same shape in a configuration block would be input
policy). The semantic model classifies fields by meaning; the schema describes
shape.
====

=== Transformations

A transformation may:

* read frame data;
* write frame data;
* read metadata;
* write metadata;
* derive helper data;
* remove helper data;
* validate data;
* render data;
* prepare data for adapter output.

Questions per transformation (to be answered by the transformation's owning
FTR, registry entry, and -- where stable -- architecture guard):

* What does it read?
* What does it write?
* Does it mutate canonical data, metadata, or only render state?
* Does it consume policy, provenance, config, or inferred structure?
* May it infer semantics, or must it consume explicit metadata? The FK-helper
  primitives, for example, no longer infer relation identity from column
  names and must consume `_meta.helper_policies.fk` (policy) or
  `_meta.derived.sheets.*.helper_columns` (provenance).
* What happens when required metadata is absent? (See the Ambiguity Policy
  for the "must become explicit" criteria.)
* Which registry entry describes it?
* Which tests or guards enforce its contract?

=== Metadata attributes

A metadata attribute is a meaning-bearing field in project metadata. It is
characterized by the semantic sets it belongs to, not by where it lives in
the JSON tree.

Questions per metadata attribute:

* What does it mean (in project terms, not in JSON-schema terms)?
* Who may produce it? Who currently does?
* Who may consume it? Who currently does?
* Is it persistent or transient (in the sense defined under Overlapping
  Semantic Sets)?
* Is it canonical, derived, render-related, operational, or diagnostic?
* Is it user-editable through the workbook surface?
* Is it roundtrip-relevant? Does it survive a canonical-to-workbook-to-
  canonical cycle?
* Does it require provenance, or is the producer obvious from its location?
* Does it have lifecycle/versioning rules (transition bridge, deprecation,
  removal owner)?
* What happens when it conflicts with configuration or newly generated data?
  Which precedence rule applies?

[[semantic-model-overlapping-semantic-sets]]
== Overlapping Semantic Sets

=== Basic idea

Semantic classes are modeled as overlapping sets, not as disjoint categories.

A metadata attribute or a transformation may belong to several sets at the
same time, and the *intersections* are where the load-bearing rules live.
A taxonomy that forced each attribute into exactly one bucket would either
lose information (collapsing distinctions) or invent artificial buckets
(splitting attributes that genuinely overlap).

Concrete project example:

* `_meta.derived.sheets.*.helper_columns` is *Derived* (produced by
  `add_fk_helpers` and other helper-enrichment steps), *Transient*
  (current-run working metadata; never persisted to workbook output as
  canonical metadata), *RuntimeConsumed* within the *current run* (read by
  `remove_fk_helpers`, helper layout, and FK-helper validation),
  *Provenance*, *ProducerOwned*, and *InternalOnly*. It is *not*
  *UserEditable* (workbook edits should not invent or rewrite provenance)
  and *not* authoritative for roundtrip reconstruction: when helper
  provenance is needed in a later run, it is recomputed by helper-
  enrichment steps rather than restored from the sidecar.

The same field carries multiple membership statements simultaneously, and
each statement contributes a different lifecycle obligation. Note that
"runtime relevance within the current run" is not the same obligation as
"authoritative across roundtrip"; an attribute can be the first without
being the second.

=== Candidate sets

The following sets are candidates for classifying metadata attributes and,
where the framing fits, transformations. Each definition is one sentence;
detailed semantics are deferred to per-family adoption FTRs.

* *Canonical* -- treated as authoritative for the reconstructed domain
  output; conflicts against canonical are not silently merged.
* *Derived* -- produced from other data or metadata by a known producer
  step; its authority depends on that producer remaining the source of truth.
* *Persistent* -- survives process boundaries or roundtrip via the sidecar,
  workbook, or another store.
* *Transient* -- exists only during a single run or rendering phase and
  must not be persisted as canonical metadata.
* *RoundtripRelevant* -- its presence, absence, or value can change the
  outcome of a canonical-to-workbook-to-canonical cycle.
* *RenderConsumed* -- read by the render layer to shape workbook
  presentation, not to alter canonical content.
* *RuntimeConsumed* -- read by transformations or validations at runtime
  and therefore affects behavior, not only documentation.
* *UserEditable* -- legitimately modifiable through the workbook projection
  or other public surface; edits are part of the contract.
* *ProducerOwned* -- has a single, identifiable producer step or component
  responsible for its current value; other writers are exceptions, not the
  rule.
* *Configurable* -- supplied or overridden by YAML pipeline config or CLI
  arguments.
* *HeuristicallyInferred* -- computed by an inference step from
  conventions or data shape rather than from explicit declaration (for
  example, `infer_fk_relations`).
* *Policy* -- a declarative input that tells a transformation what relation,
  rule, or behavior to apply (for example, `_meta.helper_policies.fk`).
* *Provenance* -- records how a derived value was produced (for example,
  `_meta.derived.sheets.*.helper_columns`).
* *PublicSurface* -- part of the project's external contract (YAML schema,
  CLI, demo, user-guide), bundled through the public Pages gate.
* *InternalOnly* -- not part of the public surface; may still have
  lifecycle obligations if it is RoundtripRelevant or RuntimeConsumed.
* *Deprecated* -- announced as on its way out; new producers should not
  emit it and consumers must have a migration path.
* *TransitionBridge* -- temporarily coexists with an old or new shape during
  a migration, with explicit current and target behavior.

These sets are descriptive and rule-ready, not yet enforced by code.
Membership is recorded in registry entries and reviewed in FTRs; runtime
enforcement is owned by `FTR-SEMANTIC-CONTRACT-MATRIX-PILOT-FK-HELPERS-P5`
and the per-family adoption slices.

=== Example membership statements

FK-helper provenance (current-run working metadata, not authoritative for
roundtrip):

[source,text]
----
_meta.derived.sheets.*.helper_columns
  in {Derived, Transient, RuntimeConsumed, Provenance, ProducerOwned,
      InternalOnly}
  not in {UserEditable, Canonical, Configurable, Persistent,
          RoundtripRelevant}
  -- RuntimeConsumed here means current-run consumers
     (`remove_fk_helpers` cleanup, helper layout, FK-helper validation).
     The namespace is transient working metadata: it is not persisted as
     canonical workbook metadata, and helper provenance is recomputed by
     helper-enrichment steps in later runs rather than restored from the
     sidecar.
----

FK-helper v2 relation policy:

[source,text]
----
_meta.helper_policies.fk.relations
  in {Policy, Persistent, RuntimeConsumed, RoundtripRelevant, Configurable,
      InternalOnly}
  not in {UserEditable, Canonical, HeuristicallyInferred}
----

v1 per-target helper-policy shape (post-consumer-migration,
producer-side residual):

[source,text]
----
_meta.helper_policies.fk.<target>
  in {Policy, Persistent, TransitionBridge}
  not in {RuntimeConsumed}
  -- Still written by `configure_fk_helpers` as a residual producer-side
     compatibility shim under the same root as the v2 .relations shape.
     After `FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5`, current runtime
     consumers read the v2 relation policy only; historical consumption
     of the v1 per-target shape is historical truth, not current runtime
     truth. Trimming the residual v1 producer write is a follow-up
     clean-up that does not block the runtime consumer contract.
----

Render-layer column-group metadata (illustrative):

[source,text]
----
column-group render metadata
  in {Derived, RenderConsumed, Transient}
  not in {Canonical, Persistent}
  -- regenerated per run; absence in canonical output is intentional.
----

Membership statements are *descriptive* of current understanding. They are
recorded so that future producers, consumers, and reviewers can disagree
explicitly rather than implicitly.

=== Derived rules from intersections

The useful part is not only classification, but rules over intersections.
The rules below are rule-ready guidance, not executable checks.

[source,text]
----
x in Derived and Persistent
=> x needs explicit provenance: who produced it, from what, when.
   (Note: FK-helper provenance under _meta.derived.sheets.* is Derived
   but Transient, not Persistent; it is a working example for
   Derived + RuntimeConsumed within a run, not for this intersection.)
----

[source,text]
----
x in Policy and RequiredByPrimitive
=> missing x is a configuration error, not a no-op.
   This is the resolved FK-helper missing-policy contract.
----

[source,text]
----
x in RenderConsumed and Transient
=> x must not be persisted as canonical metadata, even if persisting it would
   be technically convenient. Persistence is a semantic decision.
----

[source,text]
----
x in UserEditable and Derived
=> conflict behavior must be explicitly defined: prefer-user-edit,
   prefer-derived, or raise. Silent merge is a defect.
----

[source,text]
----
x in Deprecated and RuntimeConsumed
=> a TransitionBridge is required: named current writers, current readers,
   target readers, removal conditions, and owning FTR.
----

[source,text]
----
x in TransitionBridge
=> at most one owning FTR controls the bridge state at a time; ambiguous
   bridge ownership is itself a semantic drift signal.
----

[source,text]
----
x in PublicSurface and (Deprecated or TransitionBridge)
=> deprecation policy from External Inspirations applies: migration path,
   timeline, and removal conditions must be named, not implied.
----

[[semantic-model-lifecycle-and-update-semantics]]
== Lifecycle And Update Semantics

=== Why lifecycle matters

Metadata is not static. The same field may, over the lifetime of a project,
be loaded from persistence, supplied by configuration, inferred from
conventions, overwritten by a transformation, consumed by the render layer,
persisted again, and eventually pruned. The semantic model must describe
what happens at each transition -- not only what each individual step does.

Two failure modes motivate this chapter:

* *Historical residue* -- metadata that survives roundtrips after the
  semantics it described have changed. The runtime accepts it, no consumer
  errors, and yet the value no longer reflects current truth.
* *Semantic drift* -- the gradual divergence between what the runtime
  actually does and what registries, docs, and tests claim it does. The
  FK-helper phase produced concrete instances of both (see the Problem
  Statement chapter).

Lifecycle rules are how the project keeps residue from becoming drift.

=== Lifecycle events

Candidate lifecycle events, grouped by the phase they occur in:

[cols="1,3"]
|===
| Phase | Events

| Load
| load persisted metadata; load configuration; load registry entries.

| Establish
| bootstrap metadata from existing data; infer metadata from conventions
  (for example, `infer_fk_relations`); apply explicit configuration.

| Transform
| transformation reads metadata; transformation writes metadata;
  transformation overwrites or supersedes metadata; primitive emits derived
  provenance.

| Render
| render layer consumes metadata; render layer emits transient render plan;
  render-only metadata is intentionally discarded.

| Persist
| adapter persists metadata; sidecar is written; workbook is emitted.

| Reimport
| workbook is edited; workbook is reimported; metadata is reconstructed;
  helper columns are removed via provenance or v2 policy.

| End-of-life
| metadata is pruned; metadata is deprecated; metadata is removed; transition
  bridge is closed.
|===

Each event is a candidate observation point for a future guard or registry
rule. The semantic model does not require that all events be observed; it
requires that they have *names* so different parts of the project can
discuss the same transition without each inventing its own term.

=== Write-time conflict resolution

Conflicts should preferably be resolved when metadata is written or updated,
not deferred indefinitely to later consumers. Late resolution multiplies
ambiguity: every consumer must reinvent the resolution rule, and the rules
drift apart.

Bad pattern:

[source,text]
----
old persisted value remains
new config value exists
consumer decides ad hoc which value to use
----

Better pattern:

[source,text]
----
on write/update:
  decide allow / reject / supersede / merge / explicit override required
  leave metadata in a coherent post-event state
----

The post-event state must be valid on its own, so that downstream consumers
do not need to know the history to interpret it correctly. If the history
matters, it should be recorded explicitly (provenance, supersession), not
implied by leaving multiple competing values in place.

=== Precedence as update semantics

Precedence is not merely a read rule -- "which value wins when I look it up".
It is part of update semantics -- "what state may exist after a write".

Questions a precedence rule must answer:

* May a new producer overwrite an existing value, or must it call out the
  supersession explicitly?
* Does explicit config override persisted metadata? Always, or only for
  attributes in *Configurable*?
* Does bootstrap inference override explicit config? (For
  `infer_fk_relations` the answer must be "no": explicit policy wins.)
* Does a workbook edit override generated render metadata? Render metadata
  is *RenderConsumed* + *Transient*; workbook edits sit in
  *UserEditable* + *Canonical* and have priority.
* Is an older explicit value stronger than a newer heuristic value?
* Is "newer wins" valid only within the same authority class?
* Should the old value be discarded, retained as audit history, or marked
  as superseded?

These are the questions that adoption slices must answer per
transformation family. The semantic model only requires that each slice
answer them, not that they all give the same answer.

=== Age, generation, and run identity

Age or generation may be useful as a tie-breaker or staleness signal, but
must not silently replace semantic precedence. The project distinguishes
two layers:

* *Semantic precedence* -- which source class is authoritative for an
  attribute (explicit config, persisted metadata, inferred value,
  transformation output, render layer, ...). This is a property of the
  attribute and its membership in the candidate sets above.
* *Age / generation* -- when a value was produced and by which run. This
  is a property of a particular instance of the value, not of the
  attribute.

Newer is not automatically better. A newer heuristic value should *not*
override an older explicit value; a newer render-layer write should *not*
override an older user edit. Age is consulted only within a single
authority class.

Potential fields that could carry generation/run information (none are
required today; this list is rule-ready material, not a schema):

* `produced_by.step`
* `produced_by.run_id`
* `produced_from`
* `generation`
* `source_authority`
* `valid_for_frame_schema_hash`
* `lifecycle_stage`
* `supersedes`
* `stale_when`
* `prune_on`

Possible rule shape:

[source,text]
----
If two values come from the same authority class:
    newer generation may win.

If explicit config conflicts with heuristic inference:
    explicit config wins or conflict is raised, regardless of age.

If generated render metadata conflicts with user-edited canonical data:
    generated render metadata must not silently win.
----

Whether any of these fields should actually be written to `_meta` is an
open question, deferred to per-family adoption slices. The semantic model
records the *distinction* between age and precedence so that the answer
to that question is reviewable.

=== Repeated roundtrip risk

A single roundtrip may appear correct while repeated roundtrips accumulate
stale or contradictory metadata. This is where historical residue most often
becomes semantic drift.

Concrete project example: when `_meta.helper_policies.fk` migrated from a
v1 per-target shape to the v2 `relations` shape, runs during the bridge
window could legitimately write both shapes. Without a removal owner and a
named bridge state, repeated runs would either keep accreting both shapes
indefinitely or silently drop one of them based on writer order.

Questions:

* Can metadata entries become stale (still valid by shape, no longer valid
  by meaning)?
* Do repeated runs accumulate obsolete entries?
* Does each run overwrite, supersede, or prune prior metadata?
* Is there a concept of generation, or does the project rely on
  end-of-pipeline pruning?
* When is pruning safe? Specifically, when can a writer assume that the
  attribute it is about to discard has no other current consumer?
* Which metadata must survive roundtrip (RoundtripRelevant + Persistent)?
* Which metadata must be regenerated every run (RenderConsumed + Transient)?
* Which metadata must never be persisted at all (Transient + InternalOnly)?

The Ambiguity Policy treats any of these questions as "must become explicit"
once the attribute is RuntimeConsumed or RoundtripRelevant.

[[semantic-model-registry-role]]
== Registry Role

=== Registries as first-class project artifacts

Registries are operational project artifacts.

They are not merely documentation embedded in a technical-model chapter.

Current root-level artifacts:

* `registries/meta_registry.yaml`
* `registries/pipeline_step_registry.json`

=== Registry roles

The project should distinguish three possible registry roles.

==== Registry as description

The registry describes semantic intent.

Readers:

* humans;
* reviewers;
* AI agents.

==== Registry as test oracle

The registry is used by tests and architecture guards to verify that code, docs,
and declared semantics do not drift.

Readers:

* architecture tests;
* semantic invariant tests;
* review workflows.

==== Registry as runtime or rule input

The registry may eventually become input to runtime validation or lifecycle rule
checks.

Readers:

* future semantic rule checks;
* possible runtime validation;
* possible migration tooling.

[WARNING]
====
Moving from "test oracle" to "runtime/rule input" is a significant architectural
step. It makes registry changes behavioral changes. This should not happen
accidentally.
====

=== Rule-ready, not rule engine

The near-term goal is rule-ready structure.

That means:

* metadata attributes are classifiable;
* producer/consumer relationships are explicit;
* lifecycle questions are stated;
* precedence questions are visible;
* stable rules can become guards;
* unstable questions remain documented as open.

It does not yet mean:

* a general semantic rule engine;
* full runtime evaluation of the registry;
* automatic enforcement of all lifecycle rules.

== Roundtrip Walkthroughs

[NOTE]
====
This section should eventually contain PlantUML sequence diagrams.
====

=== Forward path

Candidate flow:

[source,text]
----
canonical data
+ persisted metadata
+ configuration
-> bootstrap / configuration steps
-> transformations
-> metadata updates
-> render planning
-> adapter output
-> workbook / persisted files
----

Questions:

* Which metadata is read from persistence?
* Which metadata is generated during bootstrap?
* Which configuration has precedence?
* Which transformations may overwrite metadata?
* Which render directives are persisted?
* Which render-only information is discarded?

=== Reimport path

Candidate flow:

[source,text]
----
workbook
+ embedded metadata / sidecar metadata
+ optional configuration
-> adapter read
-> bootstrap / reconstruction
-> helper removal / validation
-> canonical frames
-> updated metadata
----

Questions:

* Which workbook-visible edits affect canonical data?
* Which helper columns are derived and removable?
* Which metadata is trusted from the workbook?
* Which sidecar metadata wins over workbook-derived metadata?
* What happens when helper provenance is missing?
* What happens when persisted metadata conflicts with visible workbook edits?

=== Repeated roundtrip path

Candidate flow:

[source,text]
----
roundtrip n output
-> roundtrip n+1 input
-> metadata reuse / update / pruning
-> roundtrip n+1 output
----

Questions:

* Does metadata age?
* Are obsolete metadata entries pruned?
* Does new metadata supersede old metadata?
* Are repeated transformations idempotent?
* Can old render directives accidentally affect new output?

[[semantic-model-conflict-and-precedence-questions]]
== Conflict And Precedence Questions

=== Source classes

Metadata can enter `_meta` from several source classes, each with a different
default authority and a different set of consumers it must be safe for.

[cols="1,3"]
|===
| Source class | Project examples and authority posture

| explicit YAML config
| pipeline configuration, FK relation declarations,
  `_meta.helper_policies.fk.relations` written by `configure_fk_helpers`.
  High authority for Configurable attributes; conflicts against this should
  raise unless the contract explicitly allows override.

| CLI override
| ad-hoc per-invocation overrides. Same authority as YAML config for the
  duration of one run; should not be silently persisted as if it were
  equivalent to a declared policy.

| persisted metadata sidecar
| `_meta` carried across runs. Authoritative for Persistent attributes
  whose producer is the previous run; bridges between runs but does not
  outrank fresh explicit config.

| workbook-embedded metadata
| metadata serialized into the workbook surface. Trusted only insofar as
  the workbook is itself part of the public contract; subject to user edit.

| workbook-visible user edits
| edits to UserEditable cells/columns. Authoritative for Canonical content;
  must not be silently overwritten by Derived or RenderConsumed output.

| bootstrap logic
| reconstruction from existing data when persisted metadata is missing.
  Authoritative only as a fallback and must be visible as such.

| heuristic inference step
| for example, `infer_fk_relations`. HeuristicallyInferred; loses to
  explicit Policy or Configurable values.

| transformation output
| `_meta.helper_policies.fk`, `_meta.derived.sheets.*.helper_columns`, and
  similar. Authoritative within its producer step's scope; ProducerOwned.

| render layer output
| RenderConsumed + Transient. Must not survive as Canonical or Persistent
  unless an explicit promotion rule applies.

| adapter readback
| values reread from the persisted artifact after a write. Used to verify
  roundtrip determinism, not to amend canonical truth.
|===

=== Precedence questions

For each relevant metadata attribute, the owning adoption slice should be
able to answer the following questions. These are *questions* the slice must
resolve, not pre-decided rules.

* Which source class is authoritative?
* Which source may overwrite which other source?
* Is conflict an error, or is one of the sources designated as the winner?
* Is merge allowed? If so, on what key, and how is order resolved?
* Is fallback allowed (use source B if source A is absent), or must A be
  present?
* Is explicit override required (refuse to overwrite without an explicit
  marker)?
* Is the historical value retained as audit, or discarded?
* Does "newer wins" ever apply? Within which authority class?
* Does user-visible workbook content override hidden metadata? (Default for
  Canonical: yes; for RenderConsumed-only: no.)
* Does persisted metadata override regenerated metadata? Or is each run a
  fresh write for that attribute?

The Overlapping Semantic Sets chapter informs default answers (for example,
*Configurable* + *Policy* attributes default to "explicit config wins over
inference"), but per-attribute decisions remain the slice's responsibility.

=== Resolve conflicts early when possible

A conflict that can be detected at write time should be resolved at write
time. Deferring resolution to consumers multiplies the resolution rule
across every consumer and is the most common path to drift.

This does not mean the writer must always be the resolver. It means the
writer is responsible for leaving metadata in a state whose meaning is
unambiguous, even if that requires raising an error rather than silently
writing competing values.

Where genuine late resolution is required (for example, when only the
render layer knows whether a value is needed), the late-resolution rule
itself must be documented and reviewable.

=== No customer identity as semantics

Customer- or context-specific behavior must not enter runtime code as
identity checks. The Problem Statement chapter covers the underlying
anti-pattern; in the precedence context the rule is narrower:

*Source classes are defined by role, not by identity.* "explicit YAML
config" is a source class because it has a defined contract and lifecycle.
"config for customer X" is not a source class; it is an identity-keyed
branch wearing a config-shaped costume.

Bad:

[source,python]
----
if customer == "x":
    behave_differently()
----

Preferred:

[source,yaml]
----
# A named semantic mode is reviewable as a contract.
semantic_mode: explicit_named_mode
----

Each semantic mode needs:

* name;
* contract;
* tests;
* documentation;
* migration/removal ownership if temporary (in which case it is also a
  TransitionBridge).

[[semantic-model-contract-matrices]]
== Contract Matrices

=== Purpose

A contract matrix makes semantic expectations compact and reviewable.

It should help avoid contradictions such as:

* docs say missing policy is an error;
* runtime treats missing policy as empty;
* tests accidentally pin the old no-op behavior.

=== Candidate matrix dimensions

For a transformation or primitive:

* reads metadata;
* writes metadata;
* reads configuration;
* writes canonical data;
* writes derived data;
* may infer;
* must not infer;
* missing metadata behavior;
* persistence behavior;
* provenance requirements;
* public surface impact;
* lifecycle stage;
* tests/guards.

=== Example: FK-helper primitive contract

[source,text]
----
primitive              reads policy   reads provenance   may infer   missing metadata
add_fk_helpers         required       no                 no          error
remove_fk_helpers      optional       yes                no          error
reorder_fk_helpers     optional       yes                no          error
validate_fk_helpers    optional       yes                no          error
----

This example should be refined by the FK-helper contract-matrix pilot FTR.

[[semantic-model-semantic-guards]]
== Semantic Guards

=== When a guard is useful

A semantic guard is useful when:

* the contract is stable enough;
* drift would cause runtime or review confusion;
* the check can be precise without becoming a brittle schema framework.

=== When documentation is enough

Documentation may be enough when:

* the concept is still exploratory;
* no runtime behavior depends on it yet;
* the ambiguity is intentionally deferred;
* the cost of a guard would exceed the risk.

=== When runtime rule checks may be justified

Runtime or rule checks may be justified when:

* metadata conflicts can produce silently wrong canonical output;
* missing metadata changes deterministic behavior;
* stale metadata can survive roundtrip and affect future output;
* a registry entry is already treated as operational truth.

[[semantic-model-versioning-and-evolution]]
== Versioning And Evolution

=== Version layers

The project should distinguish:

* data schema version;
* metadata schema version;
* semantic contract version;
* registry artifact version;
* lifecycle bridge state;
* documentation model version.

=== Contract evolution questions

When introducing or changing a semantic contract:

* What was the previous behavior?
* What is the new behavior?
* Is there a transition bridge?
* Who writes both old and new forms?
* Who reads old vs new forms?
* Which FTR owns the transition?
* Which FTR owns removal?
* Which tests guard current truth?
* Which docs describe historical truth only?

=== Historical truth vs current runtime truth

Historical FTRs and review artifacts may preserve commands, paths, and wording
that were true at the time.

Current active docs, registries, and guards must reflect current runtime truth.

The semantic model should make this distinction explicit to avoid rewriting
audit records unnecessarily.

[[semantic-model-adoption-strategy]]
== Adoption Strategy

=== No big-bang adoption

The semantic model is not adopted by a global rewrite. There is no single FTR
that "applies the semantic model to the codebase", and there should not be
one. A global semantic normalization would collide with active runtime work,
balloon review surface beyond what any reviewer can hold in working memory,
and require coordinated changes across the registry, runtime, tests, guards,
docs, and demo simultaneously -- which is the exact failure mode the project
already observed during the FK-helper phase, at a much smaller scale.

Adoption proceeds *transformation family by transformation family*, in narrow
vertical slices that each touch a coherent slice of the runtime, registry,
docs, tests, guards, and -- when relevant -- demo. Each slice is sized to
fit the existing FTR Sizing Preflight (see
`docs/developer_guide/ch08_git_and_workflow/01_git_and_workflow.adoc`,
"FTR Sizing Preflight" and "FTR Slicing for Metadata and Registry
Changes"). This chapter does not introduce a parallel workflow; it describes
*how* the semantic model is applied within that existing workflow.

=== Operational symptoms that drive adoption

The trigger for a semantic adoption slice is not "we have not yet
classified this family". It is operational symptoms that point at semantic
friction in real review work.

The recurring symptoms are:

* *Review friction.* Multiple review rounds spent reconciling FTR, docs,
  registry, and runtime that each look locally consistent. The FK-helper
  primitives phase produced this pattern: each artifact described a
  plausible behavior, and the disagreements only surfaced when the four
  were read side by side.
* *Token pressure.* AI-assisted review consumes large amounts of context
  reconstructing semantic intent from artifacts that do not name it
  explicitly. The cost is not "the agent could not edit code"; it is
  "the agent had to derive the contract before it could act".
* *Semantic reconstruction cost.* Both human and AI reviewers have to
  re-derive what an attribute means each time they touch it, because the
  meaning is implied by code rather than stated by registry or contract.

These symptoms are the project's signal that a family is overdue for a
semantic adoption slice. The Ambiguity Policy already distinguishes
"must become explicit" from "may remain tolerated"; this chapter says how
the project acts on that distinction operationally.

=== Local fixes vs semantic normalization

Two response modes are explicitly distinct:

* *Local fix.* A bug, drift, or doc misalignment in a single artifact.
  Resolved by a targeted change to that artifact. No semantic-model work
  is required. The FK-helper docs/demo realignment is the canonical
  example: the contract was stable, the docs were wrong, the fix was
  bounded.
* *Semantic normalization.* The contract itself is ambiguous, drifting,
  or unevenly understood across producer, consumer, registry, and docs.
  This is an adoption slice. It requires a coherent set of changes
  across runtime, registry, docs, tests, and -- if relevant -- the demo
  and public surface.

Mistaking the second for the first is the most common path to repeated
review rounds. The slice template below exists so that semantic
normalization is recognized as such and routed into the FTR Sizing
Preflight from the start.

=== Semantic clarification and runtime refactoring are not separable

A "modeling slice" that only changes prose and never reaches runtime
produces unenforced doctrine; a "runtime slice" that changes behavior
without naming the new contract produces silent drift. Adoption slices
treat semantic clarification and runtime refactoring as a single concern
sequenced across narrow FTRs, not as two independent streams.

This sequencing is already realized in the existing developer-guide
splitting rule. The Phase 3a FK-helper sequence is the worked example to
copy:

[source,text]
----
FTR-FK-RELATION-POLICY-MODEL-P5            (model: v2 shape + identity)
FTR-INFER-FK-RELATIONS-CONFIGURATION-STEP  (heuristics into configuration)
FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES    (primitives consume v2)
FTR-FK-HELPER-DOCS-DEMO-REALIGNMENT        (public-surface realignment)
FTR-REGISTRY-CONTRACT-LIFECYCLE-MODEL      (lesson-capture, lifecycle model)
----

The lesson-capture FTR was not planned at the start of the phase; it
emerged because the early slices answered the lifecycle-modeling question
informally. The adoption strategy expects this pattern: a slice that
discovers a missing piece of the semantic model is allowed to produce a
follow-up modeling FTR rather than smuggle the model into an
implementation slice.

=== Rule-ready before rule engine

The near-term operational target is:

[source,text]
----
documentation + registries + contract matrices + selective guards
----

not:

[source,text]
----
general semantic rule engine
----

"Rule-ready" means the project's semantic contracts are structured well
enough that *some* of them could become executable checks later, *if* the
benefit justified the cost. It does not mean a rule engine exists, or
that one is planned, or that the registry is yet a runtime input. The
chapter on Registry Role and the Semantic Guards chapter own the
"when does a contract become a runtime check" question; this chapter
only states that the answer is decided per attribute, not by a global
switch.

Rule-ready outputs from a slice are:

* registry entries that name producers, consumers, and contract shape;
* a contract matrix for the family's transformations (see Contract
  Matrices);
* selective architecture guards where the contract is stable enough to
  guard cheaply;
* documented open questions for parts that are not yet stable.

The slice is not expected to produce a runtime rule check unless one of
the conditions in Semantic Guards is independently met.

=== Registry as operational truth

Registries (`registries/meta_registry.yaml`,
`registries/pipeline_step_registry.json`) are operational semantic
artifacts. The registry relocation from
`docs/technical_model/ch05_registries/` to the root-level `registries/`
directory was a semantic move, not a path-only refactor: it declared that
registries are consulted by review workflows, architecture guards, and
semantic consistency checks, and that they are kept aligned against
current runtime truth. Registries are not yet generic runtime rule
input; promotion of registry semantics into runtime rule execution
would be a future per-slice behavioral decision, not a property the
relocation conferred.

Operational consequences for adoption slices:

* A slice that changes producer or consumer behavior must update the
  registry entry in the same FTR, or hand off the registry update to an
  explicit sibling FTR. A slice that ships with a registry entry that
  no longer describes runtime is a drift event, not a successful slice.
* Registry truth is the per-attribute classification (lifecycle,
  producers, consumers, persistence, roundtrip relevance, contract
  lifecycle bridge state). The semantic-model document classifies
  *concepts*; the registry classifies *current runtime instances*.
  When the two disagree, the registry is the current truth for semantic
  review and drift detection, and the semantic-model wording is
  corrected -- as in the
  `FTR-SEMANTIC-SETS-AND-LIFECYCLE-MODEL-P5` follow-up that realigned
  the derived-provenance and v1 FK-policy examples with registry truth.
* Registries are not yet generic runtime rule input. Promotion of a
  registry entry from description / test-oracle to runtime rule input
  is a behavioral change and is owned by the per-attribute slice that
  justifies it.

=== Bridge coexistence and migration handling

Adoption slices are expected to leave the project in a coherent state at
the end of each FTR, not at the end of the slice family. That means
*coexistence periods are normal*: a slice that introduces v2 of an
attribute does not require v1 to be removed in the same commit, and
removing v1 may legitimately be its own follow-up FTR.

What a slice must do during coexistence:

* Use the `contract_lifecycle` block in the registry (see the
  developer-guide checklist for the field set: current state, target
  state, transition bridge, removal condition, identity derivation).
  This is the project's only sanctioned way to record two shapes
  side by side.
* Name *current writers, current readers, target readers, removal
  conditions, and owning FTR* explicitly. This is the
  transition-bridge contract the Lifecycle chapter references.
* Distinguish runtime-consumer narrowing from producer-side residual.
  The v1 per-target `_meta.helper_policies.fk.<target>` shape is the
  worked example: after `FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5`,
  consumers read v2 only, but `configure_fk_helpers` still emits the
  v1 shape as a residual back-compat write. The slice is complete on
  the consumer side and explicitly incomplete on the producer side,
  and that distinction is recorded.

What a slice must not do:

* Resolve coexistence by silently dropping the older shape.
* Encode an identity-keyed branch (per the No customer identity as
  semantics rule under Conflict And Precedence Questions) instead of
  a named bridge.
* Claim "the bridge is closed" when only one side has migrated.

=== Guard evolution strategy

Architecture guards under `tests/architecture/` are the project's
existing mechanism for protecting stable semantic contracts. The
adoption strategy does not invent a new guard framework; it sequences
guards alongside the contract they protect.

A guard is added when:

* the contract is stable enough that a guard is unlikely to flap;
* drift would cause runtime or review confusion;
* the check can be expressed precisely without becoming a brittle
  schema framework (see Semantic Guards for the more general
  criteria).

A guard is *not* added when:

* the contract is still exploratory;
* the family slice has not yet stabilized producer/consumer roles;
* the check would require structures the registry does not yet
  provide.

The FK-helper architecture guard that prevents convention-based FK
detection in primitives is the canonical example: the contract
("primitives must not call `detect_fk_columns`") was stable enough to
guard, drift would have been silently re-introduced by callers, and
the guard could be expressed without inventing a schema layer.
Conversely, guarding "every attribute has a contract matrix" is
explicitly out of scope for this adoption strategy.

=== Semantic debt and legacy coexistence

Semantic debt is the gap between current runtime truth and the
contract the registry / docs / guards describe. Adoption slices reduce
it; coexistence periods temporarily increase it; the goal is not zero
debt, it is *named* debt with an owning FTR.

Three legacy states should be distinguished:

* *Historical truth.* Past behavior recorded in archived FTRs and
  review artifacts. Not rewritten retroactively. The semantic model
  has an explicit "historical truth vs current runtime truth"
  distinction in the Versioning chapter; adoption slices rely on it.
* *Producer-side residual.* A producer still emits an older shape
  for back-compat reasons, but no current consumer reads it. The v1
  FK-policy residual write is the worked example. Recorded in the
  registry's `contract_lifecycle` block; safe to trim in a later
  cleanup FTR.
* *Genuinely undecided.* The project has not yet committed to a
  contract. Recorded in the Open Questions chapter; not silently
  resolved by a slice that happens to need an answer.

A slice may legitimately leave all three states in place. What it may
not do is conflate them: producer-side residual is not historical
truth, and undecided is not "we decided not to do it".

=== Pilot selection criteria

The pilot transformation family for a new piece of semantic-model
machinery (contract matrix, lifecycle model, guard pattern) is chosen
on the following criteria:

. *Recent review pain.* A family that recently caused multi-round
  review reconciliation is the best target, because the operational
  symptoms are fresh and the failure modes are documented.
. *Bounded surface.* A family with a small, identifiable set of
  primitives, producers, and consumers. Adoption work scales poorly
  on diffuse families.
. *Runtime relevance.* The family must already be RuntimeConsumed
  for some attribute, so the slice has something concrete to protect
  against drift. Purely descriptive families wait.
. *Roundtrip exposure.* A family that touches RoundtripRelevant
  attributes makes the lifecycle work load-bearing rather than
  illustrative.
. *Existing artifact gravity.* A family that already has a registry
  entry, documented primitives, and at least one architecture guard
  is cheaper to pilot than one that needs the entire scaffold built
  first.

=== Why FK-helpers became the first semantic pressure point

FK-helpers were not chosen as a pilot a priori; they became the first
adoption target because they hit all five criteria above before any
other family did:

* Recent review pain: the
  `FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5` review surfaced
  three-way disagreement between code, FTR, and docs about
  missing-policy behavior.
* Bounded surface: a small set of primitives (`add_fk_helpers`,
  `remove_fk_helpers`, `reorder_fk_helpers`, `validate_fk_helpers`)
  with a single policy root (`_meta.helper_policies.fk`).
* Runtime relevance: the v2 relation policy is RuntimeConsumed; the
  v1 per-target shape was, before consumer narrowing.
* Roundtrip exposure: helper-column removal on reimport depends on
  derived provenance and v2 policy; ambiguity here breaks roundtrip.
* Existing artifact gravity: registry entries already named
  producers and consumers; a primitive-isolation architecture guard
  already existed.

For these reasons,
`FTR-SEMANTIC-CONTRACT-MATRIX-PILOT-FK-HELPERS-P5` is positioned as
the first planned execution of the adoption-slice template, not as a
representative example of every family. The FTR itself is still new;
FK helpers are the first intended semantic pilot domain because they
already accumulated semantic pressure and migration history, but the
matrix-pilot execution itself is planned / next-up rather than already
underway.

=== Slice questions

For each transformation family, an adoption slice must answer the
following questions. The questions are aligned with the FTR Sizing
Preflight (items 3 and 6 in particular) and with the
"FTR Slicing for Metadata and Registry Changes" checklist; this
chapter does not duplicate those, it states which of the project's
existing questions the slice owns.

* Which transformations belong to the family?
* Which metadata attributes do they read?
* Which metadata attributes do they write?
* Which semantic sets do those attributes belong to (per Overlapping
  Semantic Sets)?
* What are the lifecycle events (per Lifecycle And Update Semantics)?
* What are the precedence rules (per Conflict And Precedence Questions)?
* What ambiguity is acceptable (per Ambiguity Policy)?
* What ambiguity must become an error (per Ambiguity Policy)?
* Which registry entries are affected, and is the slice
  extending, replacing, or introducing a root?
* Is a `contract_lifecycle` block needed (current state, target state,
  transition bridge, removal condition, identity derivation)?
* Which docs are authoritative for the contract after the slice lands?
* Which tests or architecture guards travel with the rule?
* Is runtime refactoring needed, and if so, is it sequenced into the
  same FTR or a sibling slice?
* Is a public-surface gate involved (YAML / CLI / demo / user guide)?
* What is the operational symptom that motivated this slice, and what
  symptom would tell the project the slice succeeded?

=== Slice exit criteria

A semantic adoption slice is complete when all of the following agree
on the same contract:

* runtime behavior;
* registry entries (including `contract_lifecycle` when used);
* technical and semantic-model docs;
* public docs where relevant (YAML schema, CLI, demo, user guide);
* tests and architecture guards;
* the FTR's review artifact.

Additionally, the slice must have:

* named the operational symptom it addressed;
* recorded any producer-side residual or transition bridge in the
  registry rather than in prose;
* fed any new open question into the semantic model's Open Questions
  chapter, rather than left it implicit;
* either added or explicitly deferred guard coverage, with rationale.

Successful completion of a slice does not require zero semantic debt
in the family; it requires that remaining debt is *named* and *owned*.

=== Alignment with the FTR Sizing Preflight

This adoption strategy is a specialization of the existing project
workflow, not a replacement for it. Concretely:

* The Sizing Preflight in
  `docs/developer_guide/ch08_git_and_workflow/01_git_and_workflow.adoc`
  decides whether an FTR needs slicing at all and which slicing
  checklist applies.
* The "FTR Slicing for Metadata and Registry Changes" checklist owns
  the rules for extending, replacing, or introducing registry roots,
  for using the `contract_lifecycle` model, and for routing
  public-surface gates.
* This Adoption Strategy chapter owns the *semantic-model-side*
  questions: which sets the attributes belong to, which lifecycle
  events matter, which ambiguity must become explicit, and what the
  operational symptom is.

If the two ever conflict, the developer-guide workflow is the
authoritative process source; this chapter is corrected in the next
review cycle.

[[semantic-model-open-questions]]
== Open Questions

These questions are recorded, not resolved. Per-family adoption slices will
answer them for their own attributes; the semantic model only requires that
each slice answer them deliberately.

General lifecycle and precedence:

* Should metadata attributes carry generation/run identity, or is end-of-pipeline
  pruning enough?
* Which metadata classes require explicit producer provenance?
* Which metadata classes may be pruned automatically, and under what
  preconditions?
* Are render directives persistent metadata or transient render plan? (Current
  working assumption: RenderConsumed + Transient unless an explicit promotion
  rule applies.)
* How should config interact with persisted metadata when the two disagree on
  a Configurable attribute?
* Should conflicts be represented (kept side-by-side), rejected (raise), or
  resolved at write time (supersede)? The default favors write-time
  resolution, but some attributes may legitimately need representation.
* Can semantic contracts be versioned separately from metadata schema? The
  v1->v2 `helper_policies.fk` example suggests yes; the registry surface
  to express this is not yet decided.
* When does a registry entry become runtime-affecting? Today registries are
  description + test oracle; promotion to runtime input is owned by
  `FTR-SEMANTIC-REGISTRY-ROLE-P5` and adoption slices.
* Which parts of the contract matrix should become executable guards?
* Which ambiguity should remain intentionally informal?

Set membership and intersections:

* Are the candidate sets above exhaustive, or will adoption slices need to
  introduce additional sets (for example, *AuditOnly*, *DiagnosticOnly*)?
* Is *RuntimeConsumed* always paired with at least one of *Policy*,
  *Provenance*, or *Configurable*? If so, that pairing is itself a rule.
* How is set membership recorded today (free-text in FTRs and registry
  prose) versus structurally? The semantic model defers this to
  `FTR-SEMANTIC-REGISTRY-ROLE-P5`.

Transition bridges:

* When a TransitionBridge is closed, how is the historical residue (if any)
  pruned from existing sidecars?
* May an attribute be in multiple TransitionBridges concurrently? (Working
  assumption: no -- ambiguous bridge ownership is a drift signal.)

[[semantic-model-glossary]]
== Glossary

canonical::
  Data or metadata treated as authoritative for reconstructed domain output.

derived::
  Data or metadata produced from other sources and not independently
  authoritative unless explicitly stated.

persistent::
  Survives process boundaries or roundtrip.

transient::
  Exists only during a run or rendering phase.

render::
  Pertaining to the workbook presentation layer; render metadata controls
  how canonical data is projected, not what it means.

authoritative::
  Designated as the source of truth for a given attribute or class of
  attributes; conflicts against an authoritative source are not silently
  resolved in favor of another source.

runtime-consumed::
  Read by transformations or validations at runtime, so changes to its value
  or absence change behavior, not only documentation.

historical residue::
  Metadata that survives roundtrips after the semantics it described have
  changed; still accepted by the runtime, no longer reflects current truth.

semantic drift::
  Gradual divergence between what the runtime actually does and what
  registries, docs, or tests claim it does.

bridge::
  Short form of *transition bridge*: a temporary coexistence of old and new
  semantic forms with named current writers, current readers, target readers,
  removal conditions, and an owning FTR.

policy::
  Declarative metadata that tells a transformation what relation, rule, or
  behavior to apply.

provenance::
  Metadata that records how derived data was produced or what source it
  corresponds to.

producer::
  Step or component that writes a semantic attribute.

consumer::
  Step or component that reads and acts on a semantic attribute.

runtime truth::
  What the current implementation actually reads, writes, or enforces.

target contract::
  The intended future semantic contract after a migration is complete.

transition bridge::
  Temporary coexistence of old and new semantic forms, with explicit current
  and target behavior.

precedence::
  Rule determining which source wins when multiple sources provide competing
  semantic claims.

supersession::
  Explicit replacement of an older semantic value by a newer or more
  authoritative value.

generation::
  Run- or update-related marker that may help detect stale metadata, but should
  not replace semantic precedence rules.

semantic ambiguity::
  A situation where multiple meanings or behaviors are plausible and the system
  has not made the intended contract explicit.

rule-ready::
  Structured enough that stable parts can later become executable checks, but
  not yet implemented as a general rule engine.
