== Management Summary

=== Why this document exists

`spreadsheet-handling` has reached a point where the main source of complexity
is no longer syntax, layering, or basic transformation mechanics.

The hard part is semantic consistency across:

* canonical data;
* metadata;
* transformation steps;
* registries;
* rendered workbook surfaces;
* reimport / roundtrip behavior;
* documentation;
* tests and architecture guards;
* AI-assisted implementation and review workflows.

The recent FK-helper migration showed that a technically clean architecture is
not enough. Even with a technical model, layered code, registries, guards, and
reviews, feature work can still get stuck on semantic ambiguity.
(the critical review with blocking issues of the past phase testifies this)

=== Main thesis

Deterministic, roundtrip-capable spreadsheet transformations require explicit
semantic contracts.

Ambiguity is not automatically a defect. However, ambiguity that affects runtime
behavior, roundtrip reconstruction, public contracts, metadata lifecycle, or
migration truth must be made explicit.

=== What this document should provide

This semantic model should:

* explain why semantic ambiguity matters in this project;
* define the core semantic concepts used by the project;
* describe the role of registries as first-class project artifacts;
* model transformations and metadata attributes as overlapping semantic sets;
* make lifecycle, precedence, and update questions explicit;
* define a no-big-bang adoption strategy;
* prepare the project for contract matrices, semantic guards, and possibly
  later rule-ready or rule-driven validation.

=== What this document should not become

This document should not become:

* a theoretical essay detached from implementation;
* a second technical model;
* a rule engine specification before the concepts are stable;
* a reason to formalize every harmless ambiguity;
* a mandate for a global rewrite.

== Relation To Other Models

=== Functional Model

The functional model describes the user-facing problem space and requirements.

It answers questions such as:

* What should the tool help users do?
* Which workflows matter?
* Which user-visible capabilities exist?

=== Technical Model

The technical model describes how the system is structured technically.

It answers questions such as:

* Which architectural layers exist?
* Where do adapters, domain logic, rendering, and pipelines live?
* Which module boundaries and dependency rules apply?

=== Semantic Model

The semantic model describes meaning-bearing contracts between data,
metadata, transformations, registries, and roundtrip behavior.

It answers questions such as:

* What does a metadata field mean?
* Who is allowed to produce or consume it?
* Is it canonical, derived, persistent, transient, user-editable, or
  render-only?
* What does missing metadata mean?
* What happens when configuration, persisted metadata, bootstrap logic, and
  transformation output disagree?
* Which ambiguity must become an error, and which ambiguity may remain
  intentionally unresolved?

== Problem Statement

=== Architecture reduced structural friction, but semantic friction remains

The project already has substantial structural discipline:

* technical model;
* layered architecture;
* registries;
* FTRs;
* architecture guards;
* review artifacts;
* public/demo documentation gates.

Nevertheless, recent FTRs still required unexpected review and follow-up work.
Large models consumed many tokens not because they could not edit code, but
because they had to reconstruct and reconcile semantic intent across multiple
artifacts.

Examples of semantic friction seen in recent work:

* a primitive's runtime behavior did not match the FTR and docs;
* missing metadata could mean either "nothing to validate" or "configuration
  error";
* registry consumer lists became stale after runtime changes;
* docs still described old convention-based behavior after primitives had moved
  to explicit policy/provenance;
* lifecycle transition bridges needed explicit current/target/runtime truth.

=== Human teams, AI agents, and semantic ambiguity

Human teams and AI agents handle semantic ambiguity differently.

==== Early project stage

Human teams often begin by debating terminology and forming project-specific
language. This is visible as:

* domain terms;
* in-group project vocabulary;
* architecture decision notes;
* conventions not yet encoded in code.

AI agents can appear surprisingly effective early on because they infer intent
from a broad, relatively homogeneous corpus of software-engineering practice.

==== Growing project stage

As the project grows, semantic contradictions accumulate.

Human teams often fail to notice all contradictions because they rely on shared
history, local context, and informal memory.

AI agents tend to notice contradictions more aggressively. They compare FTRs,
docs, registries, tests, and code and try to restore consistency. This can feel
overcritical, but it often points at unresolved semantic contracts.

==== Legacy project stage

In legacy systems, ambiguity is often operationalized as customer- or
context-specific behavior:

[source,text]
----
Customer X expected different behavior
-> add configuration
-> add special branch
-> preserve old behavior
-> no one remembers what the special case means
----

The problem is not configuration itself. The problem is configuration without a
named semantic mode.

Bad:

[source,yaml]
----
customer_x_mode: true
----

Better:

[source,yaml]
----
missing_policy_behavior: error | ignore | infer_from_legacy_convention
helper_cleanup_strategy: provenance_only | policy_or_provenance
roundtrip_conflict_strategy: fail | prefer_workbook | prefer_canonical
----

The project should avoid identity-based behavior branches and prefer named
semantic modes with explicit contracts.

=== Why this matters more here than in ordinary CRUD systems

Many business systems can tolerate ambiguity longer because meaning is resolved
socially or operationally:

* synchronous deployments;
* manual coordination;
* end-to-end smoke tests;
* implicit team knowledge;
* "the team knows how this field is used".

`spreadsheet-handling` is more sensitive because its core promise is not merely
CRUD, workflow, or reporting.

Its core promise is closer to:

[source,text]
----
canonical data
-> human-readable/editable spreadsheet projection
-> edited workbook
-> deterministic reconstruction of canonical data
----

A deterministic roundtrip cannot safely depend on ambiguous interpretation of
missing metadata, implicit helper conventions, stale lifecycle assumptions, or
consumer-local fallback rules.

=== Ambiguity policy

Not every ambiguity is a bug.

Ambiguity should become explicit when it affects:

* deterministic runtime behavior;
* roundtrip reconstruction;
* public user-facing contracts;
* metadata lifecycle;
* migration / deprecation truth;
* producer/consumer responsibilities;
* conflict or precedence resolution.

Ambiguity may remain tolerated when it is:

* historical only;
* local to prose wording with no behavioral consequence;
* intentionally deferred and marked as such;
* not used by runtime, tests, registries, or public examples.

== External Inspirations

[NOTE]
====
This section is inspirational, not normative.

The project should borrow useful concepts without importing a heavyweight
methodology wholesale.
====

=== Specification thinking

Relevant ideas:

* A specification helps understand a system before building or changing it.
* Writing precise descriptions exposes sloppy thinking.
* The hardest part is choosing the right abstraction.
* Important properties can be expressed as invariants.
* Implementation can be viewed as satisfying or refining an abstract contract.

Project interpretation:

* We do not need to introduce a full formal specification language now.
* We do need small, explicit semantic contracts for behavior that affects
  deterministic transformation and roundtrip semantics.

=== Data contracts

Relevant ideas:

* Producers and consumers should agree on what data means, not only on shape.
* Breaking changes should be visible before runtime failure.
* Consumers should not silently reinterpret missing or changed fields.

Project interpretation:

* Metadata fields are data contracts between pipeline steps.
* Registries can describe these contracts.
* Guards can check stable parts of the contract.

=== Schema evolution

Relevant ideas:

* Schemas evolve.
* Old and new shapes may coexist temporarily.
* Compatibility rules should be explicit.
* Deprecation and removal should have owners.

Project interpretation:

* Metadata schema version and semantic contract version are distinct concerns.
* Transition bridges must state current writers, current readers, target readers,
  removal conditions, and owning FTRs.

=== Consumer-driven contracts

Relevant ideas:

* It is not enough to know what a producer writes.
* The important question is what consumers rely on.

Project interpretation:

* A metadata field should record not only producers but also consumers.
* Review should ask whether registry consumers match runtime consumers.
* Missing consumer entries are semantic drift, not harmless documentation drift,
  when guards or runtime decisions depend on them.

=== API deprecation and migration policies

Relevant ideas:

* Public behavior cannot be changed casually.
* Deprecation requires migration path, timeline, and removal conditions.

Project interpretation:

* YAML, CLI, demo, and user-guide behavior are public surface.
* Internal metadata may still need lifecycle rules when it survives roundtrip
  or is consumed by multiple steps.

== Project Semantic Model

=== Core entities

The semantic model starts with the following entities:

* transformations;
* pipeline steps;
* metadata attributes;
* canonical frame data;
* derived data;
* configuration;
* persisted sidecar metadata;
* render directives;
* workbook-visible cells/columns;
* registry entries;
* lifecycle events.

=== Transformations

A transformation may:

* read frame data;
* write frame data;
* read metadata;
* write metadata;
* derive helper data;
* remove helper data;
* validate data;
* render data;
* prepare data for adapter output.

Questions per transformation:

* What does it read?
* What does it write?
* Does it mutate canonical data, metadata, or only render state?
* Does it consume policy, provenance, config, or inferred structure?
* May it infer semantics, or must it consume explicit metadata?
* What happens when required metadata is absent?
* Which registry entry describes it?
* Which tests or guards enforce its contract?

=== Metadata attributes

A metadata attribute is a meaning-bearing field in project metadata.

Questions per metadata attribute:

* What does it mean?
* Who may produce it?
* Who may consume it?
* Is it persistent or transient?
* Is it canonical, derived, render-related, operational, or diagnostic?
* Is it user-editable?
* Is it roundtrip-relevant?
* Does it require provenance?
* Does it have lifecycle/versioning rules?
* What happens when it conflicts with configuration or newly generated data?

== Overlapping Semantic Sets

=== Basic idea

Semantic classes should be modeled as overlapping sets, not as disjoint
categories.

Example analogy:

[source,text]
----
A plastic tile may be:
- red;
- triangular;
- large;
- wooden or plastic.

The set of red tiles is not disjoint from the set of triangular tiles.
----

Likewise, a metadata attribute may belong to multiple semantic sets.

=== Candidate sets

Potential sets include:

* Canonical
* Derived
* Persistent
* Transient
* RoundtripRelevant
* RenderConsumed
* RuntimeConsumed
* UserEditable
* ProducerOwned
* Configurable
* HeuristicallyInferred
* Policy
* Provenance
* PublicSurface
* InternalOnly
* Deprecated
* TransitionBridge

=== Example membership statements

[source,text]
----
target_key in Derived
target_key in Persistent
target_key in RoundtripRelevant
target_key in RuntimeConsumed
target_key not in UserEditable
----

[source,text]
----
helper_policies.fk.relations in Policy
helper_policies.fk.relations in Persistent
helper_policies.fk.relations in RuntimeConsumed
helper_policies.fk.relations in RoundtripRelevant
----

=== Derived rules from intersections

The useful part is not only classification, but rules over intersections.

Examples:

[source,text]
----
x in Derived and Persistent
=> x needs producer/provenance information.
----

[source,text]
----
x in Policy and RequiredByPrimitive
=> missing x is a configuration error.
----

[source,text]
----
x in RenderConsumed and Transient
=> x must not be persisted as canonical metadata.
----

[source,text]
----
x in UserEditable and Derived
=> conflict behavior must be explicitly defined.
----

[source,text]
----
x in Deprecated and RuntimeConsumed
=> a lifecycle bridge and removal owner are required.
----

== Lifecycle And Update Semantics

=== Why lifecycle matters

Metadata is not static.

It may be:

* read from persistence;
* written by bootstrap logic;
* supplied by configuration;
* inferred by configuration steps;
* written by transformations;
* consumed by render logic;
* embedded into or reconstructed from workbooks;
* persisted across multiple roundtrips;
* deprecated or superseded.

The semantic model must describe what happens when metadata changes over time.

=== Lifecycle events

Candidate lifecycle events:

* load persisted metadata;
* load configuration;
* bootstrap metadata from existing data;
* infer metadata from conventions;
* apply explicit configuration;
* transformation reads metadata;
* transformation writes metadata;
* transformation overwrites metadata;
* adapter persists metadata;
* render layer consumes metadata;
* render layer emits transient render plan;
* workbook is edited;
* workbook is reimported;
* metadata is reconstructed;
* metadata is pruned;
* metadata is deprecated;
* metadata is removed.

=== Write-time conflict resolution

Conflicts should preferably be resolved when metadata is written or updated,
not deferred indefinitely to later consumers.

Bad pattern:

[source,text]
----
old persisted value remains
new config value exists
consumer decides ad hoc which value to use
----

Better pattern:

[source,text]
----
on write/update:
  decide allow / reject / supersede / merge / explicit override required
  leave metadata in a coherent post-event state
----

=== Precedence as update semantics

Precedence is not merely a read rule. It should be understood as part of update
semantics.

Questions:

* May a new producer overwrite an existing value?
* Does explicit config override persisted metadata?
* Does bootstrap inference override explicit config?
* Does a workbook edit override generated render metadata?
* Is an older explicit value stronger than a newer heuristic value?
* Is "newer wins" valid only within the same authority class?
* Should the old value be discarded, retained as audit history, or marked as
  superseded?

=== Age, generation, and run identity

Age or generation may be useful, but should not silently replace semantic
precedence.

Potential fields:

* produced_by.step
* produced_by.run_id
* produced_from
* generation
* source_authority
* valid_for_frame_schema_hash
* lifecycle_stage
* supersedes
* stale_when
* prune_on

Possible rule shape:

[source,text]
----
If two values come from the same authority class:
    newer generation may win.

If explicit config conflicts with heuristic inference:
    explicit config wins or conflict is raised, regardless of age.

If generated render metadata conflicts with user-edited canonical data:
    generated render metadata must not silently win.
----

=== Repeated roundtrip risk

A single roundtrip may appear correct while repeated roundtrips accumulate stale
or contradictory metadata.

Questions:

* Can metadata entries become stale?
* Do repeated runs accumulate obsolete entries?
* Does each run overwrite, supersede, or prune prior metadata?
* Is there a concept of generation?
* When is pruning safe?
* Which metadata must survive roundtrip?
* Which metadata must be regenerated every run?
* Which metadata must never be persisted?

== Registry Role

=== Registries as first-class project artifacts

Registries are operational project artifacts.

They are not merely documentation embedded in a technical-model chapter.

Current root-level artifacts:

* `registries/meta_registry.yaml`
* `registries/pipeline_step_registry.json`

=== Registry roles

The project should distinguish three possible registry roles.

==== Registry as description

The registry describes semantic intent.

Readers:

* humans;
* reviewers;
* AI agents.

==== Registry as test oracle

The registry is used by tests and architecture guards to verify that code, docs,
and declared semantics do not drift.

Readers:

* architecture tests;
* semantic invariant tests;
* review workflows.

==== Registry as runtime or rule input

The registry may eventually become input to runtime validation or lifecycle rule
checks.

Readers:

* future semantic rule checks;
* possible runtime validation;
* possible migration tooling.

[WARNING]
====
Moving from "test oracle" to "runtime/rule input" is a significant architectural
step. It makes registry changes behavioral changes. This should not happen
accidentally.
====

=== Rule-ready, not rule engine

The near-term goal is rule-ready structure.

That means:

* metadata attributes are classifiable;
* producer/consumer relationships are explicit;
* lifecycle questions are stated;
* precedence questions are visible;
* stable rules can become guards;
* unstable questions remain documented as open.

It does not yet mean:

* a general semantic rule engine;
* full runtime evaluation of the registry;
* automatic enforcement of all lifecycle rules.

=== Forward path

Candidate flow:

[source,text]
----
canonical data
+ persisted metadata
+ configuration
-> bootstrap / configuration steps
-> transformations
-> metadata updates
-> render planning
-> adapter output
-> workbook / persisted files
----

Questions:

* Which metadata is read from persistence?
* Which metadata is generated during bootstrap?
* Which configuration has precedence?
* Which transformations may overwrite metadata?
* Which render directives are persisted?
* Which render-only information is discarded?

=== Reimport path

Candidate flow:

[source,text]
----
workbook
+ embedded metadata / sidecar metadata
+ optional configuration
-> adapter read
-> bootstrap / reconstruction
-> helper removal / validation
-> canonical frames
-> updated metadata
----

Questions:

* Which workbook-visible edits affect canonical data?
* Which helper columns are derived and removable?
* Which metadata is trusted from the workbook?
* Which sidecar metadata wins over workbook-derived metadata?
* What happens when helper provenance is missing?
* What happens when persisted metadata conflicts with visible workbook edits?

=== Repeated roundtrip path

Candidate flow:

[source,text]
----
roundtrip n output
-> roundtrip n+1 input
-> metadata reuse / update / pruning
-> roundtrip n+1 output
----

Questions:

* Does metadata age?
* Are obsolete metadata entries pruned?
* Does new metadata supersede old metadata?
* Are repeated transformations idempotent?
* Can old render directives accidentally affect new output?

== Conflict And Precedence Questions

=== Source classes

Potential metadata sources:

* explicit YAML config;
* CLI override;
* persisted metadata sidecar;
* workbook-embedded metadata;
* workbook-visible user edits;
* bootstrap logic;
* heuristic inference step;
* transformation output;
* render layer output;
* adapter readback.

=== Precedence questions

For each relevant metadata attribute:

* Which source is authoritative?
* Which source may overwrite which other source?
* Is conflict an error?
* Is merge allowed?
* Is fallback allowed?
* Is explicit override required?
* Is historical value retained?
* Does "newer wins" ever apply?
* Does user-visible workbook content override hidden metadata?
* Does persisted metadata override regenerated metadata?

=== No customer identity as semantics

Customer- or context-specific behavior must not enter runtime code as identity
checks.

Bad:

[source,python]
----
if customer == "x":
    behave_differently()
----

Preferred:

[source,yaml]
----
semantic_mode: explicit_named_mode
----

Each semantic mode needs:

* name;
* contract;
* tests;
* documentation;
* migration/removal ownership if temporary.

== Contract Matrices

=== Purpose

A contract matrix makes semantic expectations compact and reviewable.

It should help avoid contradictions such as:

* docs say missing policy is an error;
* runtime treats missing policy as empty;
* tests accidentally pin the old no-op behavior.

=== Candidate matrix dimensions

For a transformation or primitive:

* reads metadata;
* writes metadata;
* reads configuration;
* writes canonical data;
* writes derived data;
* may infer;
* must not infer;
* missing metadata behavior;
* persistence behavior;
* provenance requirements;
* public surface impact;
* lifecycle stage;
* tests/guards.

=== Example: FK-helper primitive contract

[source,text]
----
primitive              reads policy   reads provenance   may infer   missing metadata
add_fk_helpers         required       no                 no          error
remove_fk_helpers      optional       yes                no          error
reorder_fk_helpers     optional       yes                no          error
validate_fk_helpers    optional       yes                no          error
----

This example should be refined by the FK-helper contract-matrix pilot FTR.

== Semantic Guards

=== When a guard is useful

A semantic guard is useful when:

* the contract is stable enough;
* drift would cause runtime or review confusion;
* the check can be precise without becoming a brittle schema framework.

=== When documentation is enough

Documentation may be enough when:

* the concept is still exploratory;
* no runtime behavior depends on it yet;
* the ambiguity is intentionally deferred;
* the cost of a guard would exceed the risk.

=== When runtime rule checks may be justified

Runtime or rule checks may be justified when:

* metadata conflicts can produce silently wrong canonical output;
* missing metadata changes deterministic behavior;
* stale metadata can survive roundtrip and affect future output;
* a registry entry is already treated as operational truth.

== Versioning And Evolution

=== Version layers

The project should distinguish:

* data schema version;
* metadata schema version;
* semantic contract version;
* registry artifact version;
* lifecycle bridge state;
* documentation model version.

=== Contract evolution questions

When introducing or changing a semantic contract:

* What was the previous behavior?
* What is the new behavior?
* Is there a transition bridge?
* Who writes both old and new forms?
* Who reads old vs new forms?
* Which FTR owns the transition?
* Which FTR owns removal?
* Which tests guard current truth?
* Which docs describe historical truth only?

=== Historical truth vs current runtime truth

Historical FTRs and review artifacts may preserve commands, paths, and wording
that were true at the time.

Current active docs, registries, and guards must reflect current runtime truth.

The semantic model should make this distinction explicit to avoid rewriting
audit records unnecessarily.

== Adoption Strategy

=== No big-bang adoption

The semantic model must not be adopted by a global rewrite.

Adoption should proceed transformation family by transformation family.

=== Rule-ready before rule engine

The near-term goal is:

[source,text]
----
documentation + registries + matrices + selective guards
----

not:

[source,text]
----
general semantic rule engine
----

=== Transformation-family slices

A semantic adoption slice should cover one coherent transformation family.

Candidate families:

* FK Helpers
* Derived Columns
* Discriminator Split
* Rendering / Layout Metadata
* Validation Metadata
* Pipeline Step Semantics
* Roundtrip / Reimport Metadata
* Adapter Metadata

=== Slice questions

For each transformation family:

* Which transformations belong to the family?
* Which metadata attributes do they read?
* Which metadata attributes do they write?
* Which semantic sets do those attributes belong to?
* What are the lifecycle events?
* What are the precedence rules?
* What ambiguity is acceptable?
* What ambiguity must become an error?
* Which registry entries are affected?
* Which docs are authoritative?
* Which tests or guards are required?
* Is runtime refactoring needed?

=== Slice exit criteria

A semantic adoption slice is complete when:

* runtime behavior;
* registry entries;
* technical/semantic docs;
* public docs where relevant;
* tests/guards;
* review artifact

agree on the same semantic contract.

== Glossary Draft

canonical::
  Data or metadata treated as authoritative for reconstructed domain output.

derived::
  Data or metadata produced from other sources and not independently
  authoritative unless explicitly stated.

persistent::
  Survives process boundaries or roundtrip.

transient::
  Exists only during a run or rendering phase.

policy::
  Declarative metadata that tells a transformation what relation, rule, or
  behavior to apply.

provenance::
  Metadata that records how derived data was produced or what source it
  corresponds to.

producer::
  Step or component that writes a semantic attribute.

consumer::
  Step or component that reads and acts on a semantic attribute.

runtime truth::
  What the current implementation actually reads, writes, or enforces.

target contract::
  The intended future semantic contract after a migration is complete.

transition bridge::
  Temporary coexistence of old and new semantic forms, with explicit current
  and target behavior.

precedence::
  Rule determining which source wins when multiple sources provide competing
  semantic claims.

supersession::
  Explicit replacement of an older semantic value by a newer or more
  authoritative value.

generation::
  Run- or update-related marker that may help detect stale metadata, but should
  not replace semantic precedence rules.

semantic ambiguity::
  A situation where multiple meanings or behaviors are plausible and the system
  has not made the intended contract explicit.

rule-ready::
  Structured enough that stable parts can later become executable checks, but
  not yet implemented as a general rule engine.
