[[semantic-model-planned-ftrs]]
== Planned FTRs

[NOTE]
====
Names below are draft names. They should be adjusted to current backlog naming
conventions.
====

=== FTR-SEMANTIC-MODEL-SKELETON-P5

Goal:

* create `docs/semantic_model/`;
* integrate it into docs build/navigation;
* preserve this draft scaffold;
* add management summary and chapter structure;
* do not finalize detailed semantics.

=== FTR-SEMANTIC-PROBLEM-STATEMENT-P5

Goal:

* elaborate the problem statement;
* explain human vs AI vs legacy handling of semantic ambiguity;
* define the ambiguity policy;
* connect deterministic roundtrip requirements to explicit semantics.

=== FTR-SEMANTIC-INSPIRATIONS-P5

Goal:

* add concise external inspirations;
* cover specification thinking, data contracts, schema evolution,
  consumer-driven contracts, deprecation/migration policies;
* extract lessons without importing a heavyweight methodology.

=== FTR-SEMANTIC-SETS-AND-LIFECYCLE-MODEL-P5

Goal:

* define overlapping semantic sets;
* define metadata lifecycle and update-event questions;
* describe precedence, supersession, pruning, generation, and conflict handling
  as rule-ready concepts.

=== FTR-SEMANTIC-REGISTRY-ROLE-P5

Goal:

* define registry roles:
  description, test oracle, future rule input;
* clarify the implications of root-level registry artifacts;
* document when registry changes are semantic changes.

=== FTR-SEMANTIC-ADOPTION-STRATEGY-P5

Goal:

* define no-big-bang adoption;
* define transformation-family slices;
* define slice questions and exit criteria;
* produce an adoption plan.

=== FTR-SEMANTIC-CONTRACT-MATRIX-PILOT-FK-HELPERS-P5

Goal:

* use FK helpers as first pilot;
* define a contract matrix for FK-helper primitives;
* ensure registry/docs/tests/runtime agree;
* add or refine guards only where contract is stable.
