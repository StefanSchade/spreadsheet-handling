=== Python

* `from __future__ import annotations` in every module
* Type hints required on all public functions and methods
* Dataclasses for value objects; prefer immutability unless mutation is domain-relevant
* Composition over inheritance; use ABCs or explicit extension points when needed

Naming:
[cols="1,2"]
|===
| Identifier type | Convention

| Functions and variables
| `lowercase_with_underscores`

| Classes
| `PascalCase`

| Constants
| `UPPERCASE`
|===

* No nested conditional expressions in production code â€” use explicit `if`/`elif`/`else`
* Named intermediate variables over dense expressions when the value carries domain semantics
* Functions small enough to review as one behavior unit; extract meaningful pure helpers
* Do not split code mechanically just to silence a source-style guard

=== Bash and Tools Scripts

* Strict mode in every script: `set -euo pipefail`
* Complex Makefile logic belongs in `tools/` or `scripts/` (see split
  below), not inline recipes
* Document environment variables at the top of each script
* Prefer Python for complex logic (better testability)

==== `scripts/` vs `tools/`

The split is intentional and matters for cross-repo reuse:

[cols="1,3"]
|===
| Directory | Purpose

| `scripts/`
| Repo-local automation tied to core's own build, dev, branch, or release
  helpers (e.g. `build_docs_pages.sh`, `merged_to_dev.sh`). Sibling repos do
  not copy from here.

| `tools/`
| Generic helpers intended to be reused across sibling repos (e.g.
  `repo_snapshot.sh`, `check_source_style.py`, `pip_install_spec.sh`).
  Sibling repos copy what they need rather than depend on a sibling
  checkout. The `spreadsheet-handling-demo` Makefile `snapshot` target is
  the canonical example: it references `tools/repo_snapshot.sh` and tells
  the user to copy it manually from the core repo if absent.
|===

Place new helpers under `scripts/` if they are meaningful only inside
core's own loop; under `tools/` if a sibling repo could plausibly copy and
run them unchanged. If in doubt, prefer `tools/` and keep it
dependency-light.

=== YAML / JSON

* 2-space indentation, explicit booleans (`true`/`false`), avoid `null` unless required
* Comment config keys with expected type and semantics
* Validate syntax before committing

=== Makefile

* Provide `##` comment on every target â€” `make help` auto-generates from these
* Complex multi-line shell logic belongs in `tools/` or `scripts/` (see
  split above), not in inline recipes
* Provide entire targets when modifying; full file if multiple targets change

=== OS and File Portability

Three distinct levels apply:

[cols="1,3"]
|===
| Level | Rule

| Development and automation
| Assumes POSIX/Linux â€” Makefile, `tools/` scripts, CI, agent workflows

| Input file compatibility
| Accept Windows-originated files: CRLF, Excel-generated XLSX/ODS

| Generated artifacts
| Encoding: UTF-8 explicit; newlines: LF (`\n`) explicit, never platform default;
  paths: relative to output root; ordering: fully deterministic
|===

=== Source-Style Guards

Run with `make source-style-guards` (delegates to `tools/check_source_style.py`).
Exceptions are documented in `tools/source_style_allowlist.yaml`.

Active rules:

[cols="1,1"]
|===
| Rule | Severity

| Nested conditional expressions in production code
| error

| German repository-facing comments after cleanup baseline
| error

| Overlong functions, high branch complexity, dense comprehensions
| warning (review trigger, not automatic failure)
|===

Allowlist entries require: `rule`, `path`, `symbol` (if scoped), `rationale`, `owner`, `review_date`.
Temporary entries must name the cleanup condition that removes them.

=== Runtime ``_meta`` and the persistence boundary

Producers may write runtime state into ``_meta`` -- including under
``derived`` and ``__*``-prefixed keys -- for use inside the current
pipeline. Persistence is *not* a producer concern. The orchestrator owns
the boundary between the runtime ``_meta`` view and the persistable
``_meta`` contract; the projection runs once, immediately before the saver,
in ``src/spreadsheet_handling/pipeline/persistence_boundary.py``.

Implementer rules:

* Producers must not strip ``_meta`` themselves before save. Consumer-side
  workarounds are an organisational convenience, not an architectural fix.
* The projection is not exposed as a public YAML step and must not be
  auto-appended as one. If a new runtime-only key needs pruning, extend the
  pure function in ``persistence_boundary.py`` with a named rule and a
  short rationale.
* The projection applies to every output kind. Do not add a JSON/YAML
  special case that diverges from the spreadsheet path -- both backends
  consume the same persistable contract.

The full ``_meta`` lifecycle classification is deferred to
``FTR-META-LIFECYCLE-INVENTORY-P5``. Until then, new producers must
document whether their output is persistable, runtime-only, derived,
carrier-specific, or user-authored. See
``docs/semantic_model/08_lifecycle_and_update_semantics.adoc`` for the
authoritative description of the boundary.
