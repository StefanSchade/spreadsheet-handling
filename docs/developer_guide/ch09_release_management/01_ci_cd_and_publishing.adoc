=== GitHub Actions Pipeline

The workflow is defined in `.github/workflows/release.yml`.

==== Trigger Conditions

[source,yaml]
----
on:
  push:
    branches: [ main ]      # runs the test job on every push to main
    tags: [ 'v*' ]          # additionally runs the publish job on version tags
----

==== Jobs

[cols="1,1,3"]
|===
| Job | Runs when | What it does

| `test`
| Every push to `main` and every `v*` tag
| Installs deps with `pip install -e '.[dev]'`, runs `make test`

| `publish`
| `v*` tag only, after `test` succeeds
| Builds sdist + wheel with `python -m build`, publishes to PyPI

| `publish-docs`
| `v*` tag only, after `publish` succeeds
| Builds core AsciiDoc HTML with `make docs-pages`, then publishes generated
  artifacts to `spreadsheet-handling-pages` via the `DOCS_DEPLOY_KEY` SSH key
|===

==== Why `publish-docs` is sequenced after `publish`

`publish-docs` declares `needs: publish`, so it only runs after PyPI publishing
has succeeded. This is deliberate:

* The documentation site advertises a release version; publishing docs for a
  version that never reached PyPI would mislead users.
* A failed PyPI publish (token problem, sdist build failure, Trusted Publisher
  misconfiguration) must not leak into the public docs trail.
* PyPI publishing is the irreversible side-effect; docs publishing is a
  downstream artifact refresh. Failing fast on PyPI keeps the recovery story
  simple -- re-run the workflow after fixing the underlying problem and the
  docs catch up.

==== Why Pages writes are serialized

The `publish-docs` job sets a job-level concurrency group:

[source,yaml]
----
concurrency:
  group: pages-publish
  cancel-in-progress: false
----

Two release tags pushed in quick succession would otherwise both clone the same
`HEAD` of `spreadsheet-handling-pages`, race the final `git push`, and one of
the two `versions/<tag>/core/` snapshots would be lost on a non-fast-forward
rejection. `cancel-in-progress: false` ensures every tagged release still gets
to write its own immutable `versions/<tag>/core/` snapshot rather than being
preempted by a newer tag.

==== Why the push retries with rebase

The publish step pushes the generated commit with a bounded retry loop. On
each non-fast-forward rejection it runs `git pull --rebase origin main` and
tries again, up to five attempts.

* Job-level concurrency normally prevents conflicting writes, but a manual
  edit to `spreadsheet-handling-pages` (for example a hand-edited root page or
  a typo fix on `latest/core/`) can still advance `main` between clone and
  push.
* Rebasing the single docs commit onto the new tip is safe because each
  publisher only touches its own subtree (`versions/<tag>/core/`, `index.html`,
  optionally `latest/core/`).
* The retry is bounded so a real authentication or network failure surfaces
  instead of looping forever.

=== Documentation Build Stack

The core publishing job builds normal AsciiDoc HTML from the core repository.
Demo walkthrough slide decks are now published by a sibling workflow in
`spreadsheet-handling-demo` (see "Demo Slide Publishing" below); the core
job itself still does not invoke `asciidoctor-revealjs`.

The CI runner and a local VM both need this documentation toolchain:

* `asciidoctor`
* `plantuml`
* `graphviz`
* Java runtime (`default-jre-headless` or OpenJDK 21)
* `ruby-asciidoctor-plantuml`

On the current Debian/Ubuntu-style environment, prepare the stack with:

[source,console]
----
sudo apt-get update
sudo apt-get install -y \
    asciidoctor \
    plantuml \
    graphviz \
    default-jre-headless \
    ruby-asciidoctor-plantuml
----

CI additionally pins Java with `actions/setup-java` instead of relying on the
system JRE:

[source,yaml]
----
- uses: actions/setup-java@v4
  with:
    distribution: temurin
    java-version: '21'
----

The PEP 440 latest-tag guard (see "Pages Artifact Layout" below) runs through
`python3` with the `packaging` library. CI installs it explicitly. Locally,
`packaging` is already pulled in transitively by the project's dev dependencies
through `setuptools-scm`, so no extra step is required for `make docs-pages`.

The public core documentation build is:

[source,console]
----
make docs-pages
----

This renders, into `build/pages/`:

* `docs/pages/index.adoc` -- core landing page
* `docs/user_guide/user_guide.adoc` -- user guide
* `docs/release_notes/release_notes.adoc` -- release notes entry point

And into `build/pages-root/`:

* `docs/pages/root_index.adoc` -- root navigation page for the Pages repo

Both output directories are under `build/`, which is gitignored. *Generated
HTML must never be committed to `spreadsheet-handling`.* Release publishing
copies the output into the separate `spreadsheet-handling-pages` artifact
repository.

=== Pages Artifact Layout

The Pages repository uses shared release roots for a validated combined release
state. Individual source repositories own subdirectories inside those roots:

* `versions/<tag>/core/` -- core documentation built by
  `spreadsheet-handling`
* `versions/<tag>/demo/` -- future demo integration documentation
* `versions/<tag>/presentations/` -- future presentation output
* `latest/core/`, `latest/demo/`, `latest/presentations/` -- moving pointers
  for the latest validated release state

The core workflow only ever replaces the core-owned subtree. It removes and
re-copies `versions/<tag>/core/` for every release tag run. It updates
`latest/core/` only when the triggering tag is the highest visible `v*` tag by
PEP 440 ordering. Older tag reruns therefore refresh their stable versioned
snapshot without rolling back `latest/core/`.

==== Why `/versions/<tag>/core/` is treated as immutable

Each tagged release produces a stable snapshot at `versions/<tag>/core/`. Old
PyPI release pages remain online forever, and their READMEs link into the
Pages site. If those snapshots could shift retroactively, an old PyPI release
page would silently start linking to documentation describing a newer library
version. Treating `versions/<tag>/core/` as effectively immutable per tag
preserves the link contract for users browsing historical releases.

A re-run for the same tag does overwrite `versions/<tag>/core/` -- that is the
intended escape hatch for fixing a broken doc build, not for editing the
release narrative.

==== Why `/latest/core/` is guarded

`latest/core/` is a moving pointer and must only advance, never roll back. The
guard is "only update when the triggering tag is the newest PEP 440 version
among the visible `v*` tags". Re-running an older tag therefore refreshes that
tag's stable `versions/<tag>/core/` snapshot but leaves `latest/core/` pointing
to the actual latest release.

==== Why PEP 440 comparison instead of Git version sort

The earlier implementation used:

[source,console]
----
git tag -l 'v*' --sort=-v:refname | head -n 1
----

Git's version sort does *not* understand pre-release suffixes. For tags
`v0.1.0`, `v0.1.0b6`, `v0.1.0b10` it would sort `v0.1.0b10 > v0.1.0b6 > v0.1.0`
because the trailing `b6`/`b10` is appended after the version number, not
interpreted as a pre-release marker. Once a GA tag like `v0.1.0` ships, the
guard would incorrectly believe a beta is still newer.

The workflow now runs a short Python comparator that parses each tag with
`packaging.version.Version` (stripping the leading `v`) and selects the max.
PEP 440 sorts pre-releases below the matching release, so:

* `v0.1.0b1 < v0.1.0b6 < v0.1.0b10 < v0.1.0` -- expected order
* invalid tags are silently skipped instead of breaking the sort

==== Why the root `index.html` is a navigation page, not a redirect

The Pages root URL is referenced from `README.md` (and therefore from the
README that PyPI freezes onto every release page). PyPI keeps that
long-description rendering online indefinitely. Pointing the root URL straight
at `/latest/core/` via redirect would mean that a user who clicks the link
from an old PyPI release lands on documentation for a later library version,
without any indication of the mismatch.

Instead, the root `index.html` is a stable landing page that names this
release explicitly, links to `/latest/core/`, and links to
`/versions/<tag>/core/` for the release that generated the index. The source
lives at `docs/pages/root_index.adoc`; the workflow rebuilds it on every
release and copies it to the Pages repo root.

==== Scope boundary: per-version umbrella index

A future `versions/<tag>/index.html` (or `latest/index.html`) linking matching
core / demo / presentations versions together is out of scope for this
publishing slice. That umbrella page coordinates output from multiple
publishers and belongs to a later cross-publisher slice. Until then the only
shared page owned at the repo root is the navigation `index.html` described
above; `versions/<tag>/` and `latest/` directories themselves stay flat,
without an umbrella index.

=== Demo Slide Publishing

The demo repository carries its own `publish-docs` workflow at
`spreadsheet-handling-demo/.github/workflows/publish-docs.yml`. It mirrors
the core job's structure but renders Reveal.js slide decks instead of plain
HTML and writes into the demo subtree of the Pages repository.

[cols="1,3"]
|===
| Aspect | Demo `publish-docs`

| Trigger
| `v*` tag push on the demo repo.

| Test gate
| Inline `pytest` job (`test`), same pattern as core.

| Build
| `scripts/render_slides.sh` with
  `REVEALJS_DIR=/spreadsheet-handling-pages/assets/reveal.js@<pinned>`
  so the rendered HTML references the vendored bundle, not a CDN.

| Owned subtrees
| `versions/<tag>/demo/slides/` always; `latest/demo/slides/` only when the
  triggering tag is the newest demo or core tag by PEP 440 ordering. The
  comparator strips a trailing `-demo` suffix so `v0.1.0b6-demo` shadows the
  matching `v0.1.0b6` core tag for ordering purposes.

| Concurrency
| Shared `pages-publish` group with the core job, so two simultaneous tag
  pushes serialize against the single Pages repo `main` branch.

| Deploy credential
| Same `DOCS_DEPLOY_KEY` SSH deploy key shape as the core publisher.
  Configure it as a repository secret on the demo repo (it grants write
  access to `spreadsheet-handling-pages` only, never to core or demo).

| Reveal.js asset
| The workflow does *not* vendor reveal.js itself. It asserts that
  `assets/reveal.js@<pinned>/` already exists in the Pages repo and aborts
  if the bundle is missing. The bundle is committed once to the Pages repo
  during a one-time bootstrap.
|===

=== Reveal.js Vendoring and Attribution

Slide decks ship with a pinned reveal.js bundle vendored into the Pages
repository at `assets/reveal.js@<version>/`, rather than loading reveal.js
from a public CDN. Reveal.js (and the highlight.js plugin bundled with it)
are both MIT-licensed; the bundle ships unmodified with its `LICENSE` file
alongside the distributed code, which satisfies MIT attribution cleanly.

Local development (`demo/scripts/render_slides.sh` without
`REVEALJS_DIR`) still defaults to the jsDelivr CDN at `reveal.js@4` so
authors can preview a deck without a Pages-repo checkout. Only the
published artifacts are pinned to the vendored bundle.

==== Bumping the pinned reveal.js version

. Pick the new version.
. Add a sibling directory `assets/reveal.js@<new-version>/` to the Pages
  repository via `npm pack reveal.js@<new-version>` followed by copying
  `dist/`, `plugin/`, and `LICENSE` out of the extracted package.
. Update `REVEALJS_VERSION` / `REVEALJS_DIR` in the demo `publish-docs`
  workflow to point at the new directory.
. After the first release that uses the new bundle has shipped, optionally
  remove the old bundle. Keeping older bundles alongside lets historical
  `versions/<old-tag>/demo/slides/` snapshots continue to render against
  the reveal.js version they were validated against.

==== Why SSH deploy-key handling is structured the way it is

* *Deploy key, not a personal token*: the credential only grants write access
  to `spreadsheet-handling-pages`. A leak cannot push to core or to any other
  repository.
* *Step-scoped secret*: `DOCS_DEPLOY_KEY` is only exposed to the
  "Configure Pages deploy SSH key" step, written to a private file with mode
  `600`, and never echoed. Subsequent steps reach it only via
  `GIT_SSH_COMMAND` pointing at the file path.
* *Pinned host keys*: instead of trust-on-first-use through
  `ssh-keyscan github.com`, the workflow writes GitHub's published SSH host
  keys (ed25519, ecdsa-sha2-nistp256, ssh-rsa) into `known_hosts` and runs
  `ssh` with `StrictHostKeyChecking=yes`. A man-in-the-middle on the runner's
  network path cannot silently substitute an unexpected host key. Pinning
  costs a one-time update if GitHub ever rotates its SSH host keys.
* *Ephemeral runner*: GitHub Actions runners are discarded after the job, so
  no further cleanup of the key file is required.

=== Dynamic Version from Tag

[source,yaml]
----
- uses: actions/checkout@v4
  with:
    fetch-depth: 0    # CRITICAL — ensures all tags are present
----

Without `fetch-depth: 0` the checkout is shallow and `setuptools-scm` cannot
find the most recent tag, causing the version to fall back to `0.0.0`.

=== PyPI Trusted Publisher

The publish job uses OIDC-based Trusted Publisher authentication — no PyPI
API token stored as a secret.

[source,yaml]
----
permissions:
  id-token: write   # required for OIDC token exchange
  contents: read

- name: Publish to PyPI
  uses: pypa/gh-action-pypi-publish@release/v1
----

The matching Trusted Publisher entry must be configured on PyPI under the
project's publishing settings (publisher: GitHub Actions, repository owner,
repository name, workflow filename).

=== Release Process

. Ensure `dev` is stable: `make test` green, all relevant FTRs done.
. Merge `dev` → `main` (rebase and merge).
. Create and push an annotated version tag:
+
[source,console]
----
git tag -a v1.2.3 -m "Release 1.2.3"
git push origin v1.2.3      # developer executes
----
. GitHub Actions picks up the tag, runs tests, builds the distribution, and
  publishes to PyPI automatically.
. After PyPI publishing succeeds, GitHub Actions builds core documentation and
  publishes generated HTML to `spreadsheet-handling-pages` under
  `/versions/<tag>/core/`, and under `/latest/core/` only for the newest
  release tag.
. Verify the release at https://pypi.org/project/spreadsheet-handling/.
. Verify the published documentation at
  https://stefanschade.github.io/spreadsheet-handling-pages/.

=== GitHub Actions Version Policy and Dependabot Scope

==== Pinned action versions

All workflows in this project (core `release.yml`, demo `ci.yml`, demo
`publish-docs.yml`) pin first-party GitHub Actions to their current
published majors:

[cols="1,1,2"]
|===
| Action | Pin | Notes

| `actions/checkout`
| `v4`
| Current major. `fetch-depth: 0` is set on tag-triggered jobs that need
  the full tag history (PEP 440 latest-tag guard, `setuptools-scm`
  dynamic version).

| `actions/setup-python`
| `v5`
| Current major. Python `3.11` for the demo CI, `3.12` for core
  release/publish.

| `actions/setup-java`
| `v4`
| Used only by core `publish-docs` for the AsciiDoctor build stack.
  Temurin distribution, Java 21.

| `pypa/gh-action-pypi-publish`
| `release/v1`
| PyPA's documented rolling tag for the v1 line. PyPA explicitly
  recommends pinning to `release/v1` rather than a SHA or fixed minor;
  this tracks security patches automatically without manual bumps. Used
  only for the Trusted Publisher OIDC exchange against PyPI.
|===

The pinning policy is "current major from the publisher; let patches
roll in". The Trusted Publisher action is the one intentional exception
that uses a rolling tag because PyPA owns the upgrade story.

==== Node.js runtime deprecation warnings

GitHub Actions emits Node.js runtime deprecation warnings during release
runs. As of this writing every first-party action above is pinned to
its latest published major and those majors still run on Node 20. The
warnings come from GitHub's runner-side notice that Node 20 will be
deprecated in favor of the next LTS once action maintainers publish new
majors.

These warnings are *upstream-driven* and cannot be silenced from a
workflow file today. When `actions/checkout@v5`, `actions/setup-python@v6`,
or equivalents ship targeting the next Node LTS, bump the pins and the
warnings disappear. Until then the warnings are expected noise and do
not invalidate any release.

==== Job-level permissions

Every workflow sets a top-level `permissions: contents: read` default
for `GITHUB_TOKEN` and elevates per job only where needed:

* Core `release.yml`:
** `test` and `publish-docs`: read-only contents (inherited).
** `publish`: explicitly adds `id-token: write` for the PyPI Trusted
   Publisher OIDC exchange. No write access to repo contents.
* Demo `ci.yml`: read-only contents. Pytest and CLI smoke do not write
  to the repository.
* Demo `publish-docs.yml`:
** `test`: read-only contents (inherited).
** `publish-docs`: read-only contents. The push to
   `spreadsheet-handling-pages` runs under the `DOCS_DEPLOY_KEY` SSH
   deploy key, not `GITHUB_TOKEN`.

A leaked `GITHUB_TOKEN` from any of these jobs therefore cannot mutate
the source repository, publish to PyPI, or push to the Pages repository.
This is operational defense-in-depth rather than a response to a known
incident.

==== Dependabot scope (no checked-in configuration)

None of `spreadsheet-handling`, `spreadsheet-handling-demo`, or
`spreadsheet-handling-pages` carries a `.github/dependabot.yml` file. Any
Dependabot activity visible in the repository status page therefore comes
from GitHub repository-level settings, not from version-controlled
configuration in this codebase.

The two GitHub-side knobs that can surface Dependabot signals without a
checked-in config are:

. *Settings → Code security and analysis → Dependabot alerts* --
  vulnerability scanning of installed dependencies. Does not run as a
  workflow; surfaces as security alerts.
. *Settings → Code security and analysis → Dependabot security updates*
  -- automatic PRs to bump vulnerable dependencies. Runs under the
  Dependabot identity. This is the most likely source of the
  `403 Token invalid` failures observed during the `v0.1.0b6` release
  cycle: the bot attempts to author a PR but the per-repo token boundary
  rejects the action (commonly because a private package registry, a
  required token secret, or the per-repo Dependabot permission block
  needs adjustment in the UI).

The release pipeline does *not* depend on Dependabot in any form:

* No release-critical workflow consumes Dependabot output.
* No publish gate is conditioned on a Dependabot job's success.
* No package version pin is rewritten by Dependabot during release.

Dependabot failures are therefore reclassified as
*infrastructure-maintenance noise*, not release-blocking. See
`02_release_runbook.adoc` § Blocking vs Non-Blocking CI Signals for the
operator-side triage shape.

Manual GitHub UI follow-up (cannot be done from a code commit):

. Open `https://github.com/StefanSchade/spreadsheet-handling/settings/security_analysis`
   and inspect the Dependabot status for each of the three repositories
   in turn.
. For each repository, decide explicitly: keep Dependabot alerts /
   security updates enabled, or disable them. Either choice is valid;
   the current ambiguity (signals appearing without a tracked policy) is
   what needs to end.
. If updates remain enabled and continue to 403, capture the failing run
   ID and open a new FTR -- diagnosing the token boundary requires UI
   access this codebase cannot provide.
. If a future `dependabot.yml` is wanted (for example to scope updates
   to a single ecosystem on a monthly cadence), introduce it as its own
   small FTR. Adding one is out of scope for the maintenance-noise
   stabilization slice that motivated this section.
