=== Commit Format

Conventional Commits: `type(scope): summary`

* Use the active FTR ID as scope: `feat(FTR-IR-WRITEPATH-P1): add write-path parity`
* Subject ≤ 72 characters, imperative tone
* Non-trivial commits: short English body explaining the *why*
* Reference GitHub issues in footer: `References: #24` or `Closes: #24`
* One logical change per commit; vague summaries (`misc`, `fix stuff`) are not acceptable

=== Branching Model

* `main` — stable / release-ready
* `dev` — integration branch; default target for all commits

Default: commit directly to `dev`.
Feature branches are reserved for out-of-the-ordinary work (e.g. a major refactor)
and require coordination with the developer before the merge.

=== Merge Policy

* Rebase-only: no merge commits on `dev`
* Rebase feature branch on `dev` before merging
* GitHub PRs: use "Rebase and merge"
* Remote sync (`git pull`, `git push`) is performed by the developer, not by agents

FTRs live in `docs/backlog/`.
Move completed FTRs to `docs/cold_storage/backlog/ftrs_done/`.

=== FTR Workstream Completion / Lifecycle Cleanup

A workstream is not complete until the planning surface is true again. Shipping
the code is not the last step; the last step is leaving `docs/backlog/` an
accurate picture of open work.

Definition-of-done checklist for a completed FTR / BUG workstream:

* confirm implementation, tests, documentation, and review are complete;
* update the FTR / BUG status to `Done`, `Fixed`, or `Superseded`;
* remove the completed items from `docs/backlog/_ftrs_open.adoc`;
* move the completed FTR / BUG files and their companion artifacts
  (inventory, preflight, implementation concept, reviews) to
  `docs/cold_storage/backlog/ftrs_done/`;
* extract residual scope into a small follow-up FTR before archival, if needed;
* prefer archival over deletion;
* run `git diff --check` and the docs include validation;
* commit lifecycle cleanup separately from runtime implementation when possible.

This is the per-workstream rule. The broader, periodic
<<_documentation_cleanup_runbook,Documentation Cleanup Runbook>> handles
accumulated backlog and warm-storage hygiene.

=== Documentation Artifact Roles

The documentation tree contains both stable docs and live working artifacts.
Use the artifact's role, not only its path, when deciding where new material
belongs.

[cols="1,3"]
|===
| Location | Role

| `docs/backlog/`
| Active planning surface. Holds open, new, postponed, or future FTRs and BUGs
  plus their active review notes. It is a living area, not a long-term stable
  reference surface. Keep `_ftrs_open.adoc` focused on active and future work.

| `docs/warm_storage/`
| Active working storage for adoption notes, broad reviews, assessments,
  exploratory concepts, and other material that still informs current work but
  is not part of the stable model documentation. It is not long-term stable,
  but living documents such as backlog items may reference it when it is the
  current evidence source.

| `docs/cold_storage/`
| Archive for completed, superseded, or historical material. Do not delete
  useful history just because it is no longer active. When moving a bundle
  into cold storage, keep the directory structured and add or update local
  `index.json` files for machine-readable discovery.

| `docs/developer_guide/ch10_project_evolution/`
| Curated project narrative and runway. Use it for phase / milestone meaning,
  architectural direction, and why the project moved, not for a per-FTR
  changelog.

| `docs/release_notes/release_notes.adoc`
| User- and release-relevant changes, migration notes, and known limitations.
  Release notes are not the architecture narrative and should not capture every
  internal FTR unconditionally.

| `docs/ai_info/`
| Condensed operational guidance for agents. It may summarize rules and point
  to the Developer Guide, but it should not become a parallel full
  documentation system.
|===

Lifecycle rules:

* Promote facts from warm storage into stable docs when they become current
  model, architecture, or user-facing contract.
* Move completed FTRs, resolved BUGs, and historically closed review bundles
  to cold storage; preserve traceability through paths, review notes, and
  `index.json`.
* Keep project evolution updates at phase or milestone granularity. Do not
  rewrite the narrative for every small FTR.
* Keep release-note entries tied to user-visible behavior, public surfaces,
  migration semantics, or release positioning.

=== Documentation Cleanup Runbook

This runbook is an independent documentation hygiene process. It is often
worth running after a release, but it can also run whenever the active backlog
or warm storage has accumulated enough completed or historical material to
obscure current work.

Do not treat it as part of the mandatory release reset. The release reset
freezes / opens release-note sections and restores release-time links; this
cleanup pass archives completed work and refreshes planning context.

==== When to Run

Run this cleanup when one or more of these is true:

* a release shipped and the backlog still contains completed FTRs or BUGs;
* `_ftrs_open.adoc` no longer reflects the actual active work;
* warm storage contains completed review, assessment, or concept bundles that
  are now historical;
* release notes, Project Evolution, and backlog posture disagree about the
  current cycle.

No FTR is normally required for this runbook. It is documentation
restructuring and archival hygiene unless it changes a product or architecture
contract.

==== Procedure

. *Inventory candidates.*
  List completed FTRs / BUGs, their review artifacts, completed global or
  topic reviews, stale concept notes, and warm-storage bundles whose decisions
  have been incorporated elsewhere. Also list material that must stay warm
  because it still drives active work.

. *Update the curated narrative first.*
  If the cleanup closes a phase, release wave, or boundary-discovery episode,
  update `docs/developer_guide/ch10_project_evolution/01_project_evolution.adoc`
  before pruning the backlog. Keep this at phase / milestone granularity, not
  as a per-FTR changelog.

. *Refresh release notes where relevant.*
  Update `docs/release_notes/release_notes.adoc` only for release-relevant
  user behavior, public surface, migration semantics, or known limitations.
  Do not add release-note bullets merely because an internal FTR moved to
  cold storage.

. *Prune the active backlog.*
  Keep `docs/backlog/_ftrs_open.adoc` focused on open, postponed, and future
  work. Remove includes for completed FTRs / BUGs. Keep active review gaps
  visible until their evidence requirement is satisfied or explicitly descoped.

. *Move completed backlog artifacts.*
  Move completed FTRs / BUGs and their review artifacts to
  `docs/cold_storage/backlog/ftrs_done/`. Move dropped, rejected, or
  superseded items to `docs/cold_storage/backlog/wont_do/`.

. *Move historical warm-storage bundles.*
  Move completed reviews, assessments, concepts, and collaboration notes from
  `docs/warm_storage/` into an appropriate `docs/cold_storage/` area such as
  `reviews/`, `concepts/`, or `collaboration/`. Preserve active adoption
  evidence and current reviews in warm storage.

. *Maintain cold-storage indexes.*
  Every cold-storage directory that receives a meaningful bundle should have
  an `index.json`. Use a JSON array of flat objects so the archive remains
  machine-readable. Keep entries compact and factual; typical fields are:
+
[source,json]
----
[
  {
    "path": "example.adoc",
    "kind": "review",
    "what": "Short content description.",
    "usage": "When this artifact is useful.",
    "status": "historically completed",
    "source_path": "docs/warm_storage/example.adoc",
    "related_refs": "FTR-EXAMPLE-P5"
  }
]
----

. *Rewrite references deliberately.*
  Update references to moved historical material. It is acceptable for living
  backlog or warm-storage artifacts to reference warm-storage files when those
  files are still the current evidence source.

. *Validate the cleanup.*
  At minimum:
+
[source,bash]
----
find docs/cold_storage -name index.json -print | sort | xargs -n1 python3 -m json.tool
git diff --check
asciidoctor -o /tmp/ftrs_open.html docs/backlog/_ftrs_open.adoc
asciidoctor -o /tmp/release_notes.html docs/release_notes/release_notes.adoc
----
+
For large moves, also inspect `git diff --summary` or `git status --short` to
confirm Git detects the moves as renames where practical. `git log --follow`
works best when the moved files keep high textual similarity.

=== When to Create an FTR

Create an FTR when the work has *acceptance criteria worth defining upfront
and checking off later*: new features, significant architectural changes,
anything with user-visible impact.

No FTR needed for: cleanup, refactoring, environment migration, bug fixes,
documentation restructuring.  A good commit message is sufficient.

=== FTR Sizing Preflight

Once you have decided an FTR is warranted (see "When to Create an FTR"
above), run through this six-question preflight *before* writing
acceptance criteria. It exists to catch FTRs that are silently bigger
than they look and to route them into the right slicing discipline,
without inventing a parallel ruleset -- each question points at an
existing rule, gate, or checklist that already governs that concern.

. *Forward export only?* If the change only affects how the system
  emits data and does not touch documented public behavior, a roundtrip
  contract, or any of items 3-6, it is often a no-FTR change per
  "When to Create an FTR" above. A good commit message is sufficient.

. *Must survive reimport / roundtrip?* The FTR's acceptance criteria
  must include a readback / recomposition assertion (or name the
  test that already covers it). Roundtrip-relevant changes do not
  ship without an explicit roundtrip test or a documented gap.

. *Touches identity, helper columns, derived data, metadata,
  registry roots, or lifecycle?* Apply the
  "FTR Slicing for Metadata and Registry Changes" checklist below and
  fill in its copy-paste block. Decide explicitly whether you
  *extend*, *replace*, or *introduce* a root. If the contract's shape
  is in motion between a current and a target form, evaluate the
  opt-in `contract_lifecycle` model in
  `docs/technical_model/ch05_registries/registry_semantics/05_contract_lifecycle.adoc`.

. *Must work across XLSX and ODS?* The change must either reach
  backend parity inside this FTR *or* carry an explicit
  `accepted initial gap` entry in
  `docs/technical_model/ch08_architecture_details/spreadsheet_backend_capability_profiles.adoc`.
  No silent backend asymmetry; no overloading bridge / lifecycle
  vocabulary onto capability-profile semantics.

. *Changes YAML, CLI, demo behavior, public docs, plugin or import
  surfaces?* Public-surface gate per
  `docs/ai_info/interfaces_and_gates.adoc`. The FTR must name the
  gate, list the affected surface, and own (or hand off to a sibling
  FTR) the docs / demo realignment slice. Approval is required before
  implementation.

. *Needs lifecycle, precedence, conflict, fallback, override, or
  reconstruction rules?* This is a *modeling* concern, not a simple
  implementation fix. The FTR must define the rule explicitly and add
  the guard or test that enforces it; deferring it to "we will see
  during implementation" is the failure mode this question catches.

[NOTE]
====
*Splitting rule:* if more than one of items 3-6 fires on the same
FTR, slice it. A modeling slice (rule + guard + docs, no behavior
change) typically comes first, followed by an implementation slice and
then a docs / demo realignment slice. Phase 3a's FK-helper sequence is
the worked example to copy: `FTR-FK-RELATION-POLICY-MODEL-P5` ->
`FTR-INFER-FK-RELATIONS-CONFIGURATION-STEP-P5` ->
`FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5` ->
`FTR-FK-HELPER-DOCS-DEMO-REALIGNMENT-P5`. The intermediate lesson-capture
FTR `FTR-REGISTRY-CONTRACT-LIFECYCLE-MODEL-P5` came out of that loop
precisely because items 3 and 6 had been answered too informally in the
earlier slices.
====

=== Self-Review Checklist

* [ ] Every acceptance criterion met or explicitly deferred with rationale
* [ ] All new code has tests; test count reported
* [ ] Full active test suite passes
* [ ] No unintended changes outside the FTR scope
* [ ] FTR document updated: status set to Done, delivery scope table added
* [ ] FTR moved to cold storage
* [ ] Phase/wave comment in `_ftrs_open.adoc` names the FTR ID and disposition
* [ ] `pytestmark = pytest.mark.ftr("FTR-XXX")` set on all new/modified test files
* [ ] Release-notes checkpoint applied: update
  `docs/release_notes/release_notes.adoc` (including *Known limitations* when
  a limitation's validity, scope, or release relevance changes), or record in
  the FTR review artifact why no release note is needed

=== FTR Review Notes

Every FTR implementation review is captured in a sibling file:

  docs/backlog/FTR-<ID>_review.adoc

It records the verdict, architectural assessment, findings, and final disposition.
When review follow-up changes are made, update the review note before committing.

=== Release-Notes Checkpoint

For every FTR that changes user-facing behavior, public CLI / YAML /
configuration surface, package / import / plugin surface, demo behavior,
documentation entry points, migration semantics, or other release-relevant
behavior, the implementer must either update
`docs/release_notes/release_notes.adoc` or explicitly state in the FTR review
artifact why no release note is needed.

This includes adding, changing, or removing *Known limitations* when the FTR
changes a limitation's validity, scope, or release relevance.

Internal-only refactors, tests, guard-only changes, and documentation-internal
cleanups do not need release-note entries unless they affect public behavior or
release positioning. Do not require a release-note entry for every FTR
unconditionally.

=== FTR Slicing for Metadata and Registry Changes

When an FTR (or a group of FTRs in a phase) proposes new `_meta` fields,
new registry roots, or new pipeline-step categories, use the following
checklist before writing acceptance criteria. The rules below were
extracted from the Phase 3a FK-helper planning review and are intended
as reusable guidance for future metadata / registry slices.

. *Survey before you propose.* Read `registries/meta_registry.yaml`
  and `registries/pipeline_step_registry.json` (explanatory chapters
  live under `docs/technical_model/ch05_registries/`) and grep the source
  tree for the metadata root(s) you intend to touch. An FTR that proposes
  a "new" root which already exists under another name causes downstream
  rework.

. *Decide explicitly: extend, replace, or introduce.*

[cols="1,3"]
|===
| Decision | When to use

| Extend
| The existing shape's intent is correct; you are adding fields. Keep
the root; bump a shape / `schema_version` only if consumers need to
disambiguate.

| Replace
| The existing shape's intent is wrong or has drifted. Keep the *root*
and replace the shape under it; carry a `schema_version` so consumers
can detect the change. State that this is a gated public-surface
change.

| Introduce
| A genuinely new concept with no existing root. Justify why no
existing root fits. Avoid "adjacent" roots with redundant semantics
(e.g. `helper_policies.fk` *and* `fk_relations`).
|===

. *Producer-shape FTR is not the lifecycle FTR.* Separate "what fields
the producers write" from "when entries are pruned, who may read them,
schema validation". Two FTRs, possibly in two phases, with an explicit
handoff note in both. Mixing them produces overlapping authority.

. *Heuristics live in configuration steps.* If your FTR adds inference
behavior, register the producer as a `configuration` step and have it
write *resolved* metadata. Primitive steps consume that metadata.
`docs/technical_model/ch08_architecture_details/pipeline_semantics.adoc`
already mandates this; the FTR enforces it for a concrete case.

. *Primitive steps are deterministic.* A primitive step must read
either inline YAML or resolved `_meta`, and fail clearly when neither
is present. No hidden fallback to a naming heuristic; recovery from
missing metadata is a separate opt-in step or follow-up FTR.

. *Distinguish wire format from inference.* A naming convention
(e.g. `id_(<target>)` as the cell-level FK column header) is the
*wire format* on the serialized side. Inference *uses* a convention
to write resolved policy. Execution semantics *consume* resolved
policy. Removing convention from primitive execution is not the same
as removing the wire format -- say which one you mean.

.  *Do not make current-state registries lie.* Before changing
  `meta_registry.yaml`, `pipeline_step_registry.json`, or another
  registry-like artifact, decide whether the artifact describes current
  runtime behavior, target-state design intent, or both. If it is
  current-state only, do not update it to a not-yet-implemented shape.
  Document the target model in the FTR / concept docs instead, and
  hand off the registry/runtime contract change to the implementation
  FTR that makes the runtime true. A registry entry may describe
  target-state only when that status is explicit and cannot be mistaken
  for implemented behavior.

. *Use the contract lifecycle model when current and target shapes
  coexist.* When a registry entry would otherwise have to describe two
  shapes side by side (current runtime shape, target shape, and the
  temporary bridge between them), use the opt-in
  `contract_lifecycle` block defined in
  `docs/technical_model/ch05_registries/registry_semantics/05_contract_lifecycle.adoc`
  instead of merging the two narratives in prose. The FTR must answer,
  for any contract whose shape is in motion:
+
[cols="1,3"]
|===
| Field | Question

| Current state
| What do runtime producers write and runtime consumers read *today*?
  Name the producers and consumers and the shape(s) involved.

| Target state
| What is the preferred shape after the change, which FTR owns making
  that target true at runtime, and which consumers will read it?

| Transition bridge
| Is a bridge active, planned, retired, or absent? If active, what is
  its `bridge_type` (e.g. `dual_write`, `adapter`), source shape, and
  target shape?

| Removal condition
| Which FTR owns narrowing or removing the bridge (the `removal_ftr`),
  and what runtime / consumer / test condition makes the bridge safe to
  remove? Date-only conditions are not used in this repository.

| Identity derivation
| If the shape change moves the contract's identity model (e.g. v1 is
  per-target, v2 is per source-frame / source-column), how does the
  new identity get derived from the old one? Pick one of
  `explicit_new_fields`, `deterministic_normalization`,
  `temporary_bridge`, `rejection_with_migration_error`, `no_migration`,
  or `not_applicable`, and state the authorization for that mechanism.
|===
+
Lifecycle fields are opt-in. Entries with stable single-shape contracts
do not need to adopt them. Do not use `canonical` as a synonym for
target state, and do not use backend capability-profile vocabulary
(`accepted_gap`, `accepted initial gap`) as a bridge status.

. *YAML / step-surface changes are gated.* Any change to a
documented step name, parameter shape, or `_meta` root shape is a
public-surface change per
`docs/ai_info/interfaces_and_gates.adoc`. The FTR must state the
gate and the developer must approve before implementation begins.

. *Presentation is downstream of identity / provenance.* Optional
rendering features (column groups, styles, outlines) must consume
metadata produced by identity / provenance steps. Presentation must
never *substitute* for identity (e.g. column grouping cannot decide
which columns are helper columns).

. *Architecture guards travel with the rule.* If the FTR adds a new
rule (e.g. "primitives must not call `detect_fk_columns`"), add a
new architecture guard test under
`tests/architecture/dependency_guards/`. "Existing guards still
pass" is not enough; a new rule needs a new guard or an extension
of an existing one.

. *Docs / demo realignment is its own FTR slice.* Update the
user-guide transformation catalog, the relevant technical-model
concept doc, the agent-facing AI-info notes, and the demo
pipelines / walkthrough as a separate FTR after the runtime change
has landed. List the affected files explicitly so the realignment
FTR is reviewable.

Worked example (Phase 3a FK-helper semantics): the phase replaces
the per-target `helper_policies.fk` shape with a v2 relation-oriented
shape under the same root; introduces `infer_fk_relations` as a
configuration step and migrates the heuristic / `targets: "auto"`
path out of `configure_fk_helpers`; refactors the FK-helper
primitives to consume the v2 shape; and realigns the docs / demo in
a fourth slice. Lifecycle of the derived helper-column channel
remains with `FTR-DERIVED-PROVENANCE-LIFECYCLE-P4A` in Phase 4. See
`docs/backlog/phase_review_fk_helpers.adoc` for the reasoning
behind these slicing choices.
