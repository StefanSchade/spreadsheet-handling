=== Commit Format

Conventional Commits: `type(scope): summary`

* Use the active FTR ID as scope: `feat(FTR-IR-WRITEPATH-P1): add write-path parity`
* Subject ≤ 72 characters, imperative tone
* Non-trivial commits: short English body explaining the *why*
* Reference GitHub issues in footer: `References: #24` or `Closes: #24`
* One logical change per commit; vague summaries (`misc`, `fix stuff`) are not acceptable

=== Branching Model

* `main` — stable / release-ready
* `dev` — integration branch; default target for all commits

Default: commit directly to `dev`.
Feature branches are reserved for out-of-the-ordinary work (e.g. a major refactor)
and require coordination with the developer before the merge.

=== Merge Policy

* Rebase-only: no merge commits on `dev`
* Rebase feature branch on `dev` before merging
* GitHub PRs: use "Rebase and merge"
* Remote sync (`git pull`, `git push`) is performed by the developer, not by agents

=== FTR Status Lifecycle

[cols="1,3"]
|===
| Status | Meaning

| New
| Not yet started

| Open
| Acknowledged; may be in progress

| Open (Umbrella)
| Coordination FTR; delegates to sub-features

| Superseded
| Replaced by another FTR — kept for traceability

| Done
| All acceptance criteria met and merged to `dev`
|===

FTRs live in `docs/backlog/`.
Move completed FTRs to `docs/cold_storage/backlog/ftrs_done/`.
Move dropped, rejected, or superseded FTRs to `docs/cold_storage/backlog/wont_do/`.

=== When to Create an FTR

Create an FTR when the work has *acceptance criteria worth defining upfront
and checking off later*: new features, significant architectural changes,
anything with user-visible impact.

No FTR needed for: cleanup, refactoring, environment migration, bug fixes,
documentation restructuring.  A good commit message is sufficient.

=== Per-FTR Workflow

[cols="1,2,2"]
|===
| Step | Activity | Owner

| 1. Plan
| Read FTR, clarify scope, raise questions
| Joint (developer + AI)

| 2. Implement
| Code + tests; commit directly to `dev`
| AI (developer reviews)

| 3. Self-Review
| Verify every AC against actual code; list deviations
| AI

| 4. Merge-Ready Report
| Summary with AC status, files changed, open items
| AI → Developer

| 5. Merge
| Fast-forward to `dev` after developer confirms
| Joint
|===

=== Self-Review Checklist

* [ ] Every acceptance criterion met or explicitly deferred with rationale
* [ ] All new code has tests; test count reported
* [ ] Full active test suite passes
* [ ] No unintended changes outside the FTR scope
* [ ] FTR document updated: status set to Done, delivery scope table added
* [ ] FTR moved to cold storage
* [ ] Phase/wave comment in `_ftrs_open.adoc` names the FTR ID and disposition
* [ ] `pytestmark = pytest.mark.ftr("FTR-XXX")` set on all new/modified test files

=== FTR Review Notes

Every FTR implementation review is captured in a sibling file:

  docs/backlog/FTR-<ID>_review.adoc

It records the verdict, architectural assessment, findings, and final disposition.
When review follow-up changes are made, update the review note before committing.

=== FTR Slicing for Metadata and Registry Changes

When an FTR (or a group of FTRs in a phase) proposes new `_meta` fields,
new registry roots, or new pipeline-step categories, use the following
checklist before writing acceptance criteria. The rules below were
extracted from the Phase 3a FK-helper planning review and are intended
as reusable guidance for future metadata / registry slices.

. *Survey before you propose.* Read
  `docs/technical_model/ch05_registries/meta_registry/meta_registry.yaml`
  and `pipeline_step_registry.json` and grep the source tree for the
  metadata root(s) you intend to touch. An FTR that proposes a "new"
  root which already exists under another name causes downstream rework.

. *Decide explicitly: extend, replace, or introduce.*

[cols="1,3"]
|===
| Decision | When to use

| Extend
| The existing shape's intent is correct; you are adding fields. Keep
the root; bump a shape / `schema_version` only if consumers need to
disambiguate.

| Replace
| The existing shape's intent is wrong or has drifted. Keep the *root*
and replace the shape under it; carry a `schema_version` so consumers
can detect the change. State that this is a gated public-surface
change.

| Introduce
| A genuinely new concept with no existing root. Justify why no
existing root fits. Avoid "adjacent" roots with redundant semantics
(e.g. `helper_policies.fk` *and* `fk_relations`).
|===

. *Producer-shape FTR is not the lifecycle FTR.* Separate "what fields
the producers write" from "when entries are pruned, who may read them,
schema validation". Two FTRs, possibly in two phases, with an explicit
handoff note in both. Mixing them produces overlapping authority.

. *Heuristics live in configuration steps.* If your FTR adds inference
behavior, register the producer as a `configuration` step and have it
write *resolved* metadata. Primitive steps consume that metadata.
`docs/technical_model/ch08_architecture_details/pipeline_semantics.adoc`
already mandates this; the FTR enforces it for a concrete case.

. *Primitive steps are deterministic.* A primitive step must read
either inline YAML or resolved `_meta`, and fail clearly when neither
is present. No hidden fallback to a naming heuristic; recovery from
missing metadata is a separate opt-in step or follow-up FTR.

. *Distinguish wire format from inference.* A naming convention
(e.g. `id_(<target>)` as the cell-level FK column header) is the
*wire format* on the serialized side. Inference *uses* a convention
to write resolved policy. Execution semantics *consume* resolved
policy. Removing convention from primitive execution is not the same
as removing the wire format -- say which one you mean.

.  *Do not make current-state registries lie.* Before changing
  `meta_registry.yaml`, `pipeline_step_registry.json`, or another
  registry-like artifact, decide whether the artifact describes current
  runtime behavior, target-state design intent, or both. If it is
  current-state only, do not update it to a not-yet-implemented shape.
  Document the target model in the FTR / concept docs instead, and
  hand off the registry/runtime contract change to the implementation
  FTR that makes the runtime true. A registry entry may describe
  target-state only when that status is explicit and cannot be mistaken
  for implemented behavior.

. *YAML / step-surface changes are gated.* Any change to a
documented step name, parameter shape, or `_meta` root shape is a
public-surface change per
`docs/ai_info/interfaces_and_gates.adoc`. The FTR must state the
gate and the developer must approve before implementation begins.

. *Presentation is downstream of identity / provenance.* Optional
rendering features (column groups, styles, outlines) must consume
metadata produced by identity / provenance steps. Presentation must
never *substitute* for identity (e.g. column grouping cannot decide
which columns are helper columns).

. *Architecture guards travel with the rule.* If the FTR adds a new
rule (e.g. "primitives must not call `detect_fk_columns`"), add a
new architecture guard test under
`tests/architecture/dependency_guards/`. "Existing guards still
pass" is not enough; a new rule needs a new guard or an extension
of an existing one.

. *Docs / demo realignment is its own FTR slice.* Update the
user-guide transformation catalog, the relevant technical-model
concept doc, the agent-facing AI-info notes, and the demo
pipelines / walkthrough as a separate FTR after the runtime change
has landed. List the affected files explicitly so the realignment
FTR is reviewable.

Worked example (Phase 3a FK-helper semantics): the phase replaces
the per-target `helper_policies.fk` shape with a v2 relation-oriented
shape under the same root; introduces `infer_fk_relations` as a
configuration step and migrates the heuristic / `targets: "auto"`
path out of `configure_fk_helpers`; refactors the FK-helper
primitives to consume the v2 shape; and realigns the docs / demo in
a fourth slice. Lifecycle of the derived helper-column channel
remains with `FTR-DERIVED-PROVENANCE-LIFECYCLE-P4A` in Phase 4. See
`docs/backlog/phase_review_fk_helpers.adoc` for the reasoning
behind these slicing choices.

=== Virtual Env and Dependency Stamps

`.venv` presence does not guarantee a working environment — stamps may be stale.
Safe recovery:

[source,console]
----
make clean-stamps && make setup
----

Do not stage environment artifacts; do not infer test runnability from `.venv` alone.
