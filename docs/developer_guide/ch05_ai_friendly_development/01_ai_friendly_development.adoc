=== Philosophy

This project is *intentionally designed for AI-assisted development*.
AI agents (Claude Code, Codex CLI, and similar tools) are treated as
first-class collaborators at every phase: requirements, design, implementation,
and review.

This is not about automation replacing engineering judgment — human oversight
and review are mandatory.  The design goal is to make AI collaboration
as frictionless and reliable as possible, so that the developer and the AI
can work together without friction caused by missing context.

=== Context File Architecture

The `docs/ai_info/` directory is the machine-readable knowledge base for AI
agents.  It is not incidental documentation — it is a deliberate, maintained
part of the project infrastructure.

[cols="2,3"]
|===
| File | What it covers

| `_ai_info.adoc`
| Navigation index — which file to load for which task

| `project_context.adoc`
| Identity, stack, related repos, AI use policy, practices

| `conventions.adoc`
| Python, Bash, YAML, Makefile rules, source-style guards

| `architecture_primer.adoc`
| Layer map, key modules, IR render-flow, pipeline execution

| `git_and_workflow.adoc`
| Commit format, branching, FTR lifecycle, self-review checklist

| `agent_rules.adoc`
| Scope boundaries, forbidden actions, working style

| `interfaces_and_gates.adoc`
| Consumer-facing surfaces requiring explicit approval before change

| `concepts_fk_helpers.adoc`
| FK helper columns: capability, materialization, provenance shape
|===

An agent starting a new context window reads `_ai_info.adoc` first, then
loads only the files relevant to the current task.  This keeps context lean
and focused.

=== FTR Workflow

Work is structured around *Feature Tickets* (FTRs), stored as AsciiDoc files
in `docs/backlog/`.  FTRs carry acceptance criteria, scope
boundaries, and traceability markers.

The purpose of the FTR structure is to allow an AI agent to pick up a task in
a fresh context window with a precise, unambiguous description of what
"done" means — without requiring a long conversation to reconstruct intent.

Every test file declares its FTR via `pytestmark`:

[source,python]
----
import pytest

pytestmark = pytest.mark.ftr("FTR-EXAMPLE-NAME")
----

This links tests to their originating ticket and enables targeted test runs:

[source,console]
----
pytest -k "FTR-EXAMPLE-NAME"
----

=== Parallel Workflow

The project is developed on a headless Debian VM.  The developer typically
runs Claude Code and Codex CLI in parallel, assigning different tasks or
cross-checking each other's output.  The context files in `docs/ai_info/`
ensure both agents work from the same shared understanding without
contradicting each other.

=== Snapshot Targets

The Makefile provides snapshot targets for capturing the full codebase state
as a text artifact — useful for pasting into a new AI context window when
starting from scratch:

[source,console]
----
make snapshot               # single concatenated snapshot
make snapshot-multi         # split by section (docs, src, tests, infra, tree, loc)
----

Snapshots land in `build/snapshots/` and are gitignored.

=== What This Means in Practice

* AI agents do not need repeated explanation of the project structure — the
  context files provide it.
* Planned changes to outward-facing interfaces must be stated *before*
  implementation — not discovered in a PR.
* Changes to CLI flags, YAML config keys, documented import paths, or plugin
  conventions require explicit developer approval before the AI proceeds.
* All AI-generated artifacts are reviewed, version-controlled, and tested
  before acceptance.
