== Domain Comprehension and Semantic Refactoring Wave (June 2026)

After the `v0.2.0` release the project entered a period that initially
looked like ordinary cleanup work.

The visible trigger was a series of domain-package reviews:

* xref crosstable
* enrich lookup
* cell codec
* compact multiaxis
* legend reconstruction
* FK helpers

The original goal was structural:

* reduce module size;
* improve package boundaries;
* improve navigation;
* make the domain easier for humans and AI agents to understand.

The work started as a conventional package-refactoring exercise.
Several transformations were isolated into dedicated packages and
guarded by package-boundary tests.

However, the FK-helper review exposed a different class of problem.

=== Discovery

The dominant source of complexity was not code structure.

The dominant source of complexity was uncertainty about metadata
meaning.

Several investigations appeared at first to be independent:

* lifecycle reviews;
* metadata inventories;
* persistence-boundary bugs;
* helper-column promotion bugs;
* replay and accumulation defects;
* semantic-model work;
* functional-model restructuring.

All of them converged on the same observation:

Metadata paths were often carrying multiple responsibilities
simultaneously.

Examples included:

* durable configuration;
* runtime resolution state;
* reconstruction authority;
* projection provenance;
* transition-bridge compatibility data.

The immediate temptation was to redesign `_meta` globally.

A competing hypothesis emerged during the review wave:

The architecture itself was not the primary problem.

The larger problem was accumulated historical residue around otherwise
sound architectural boundaries.

=== Method Shift

The project deliberately avoided a global metadata redesign.

Instead a new review-driven methodology emerged.

For a single transformation family:

. inventory metadata paths;
. document the functional problem space;
. derive a functional model;
. compare the model against the implementation;
. identify historical residue;
. remove only complexity that has no demonstrated consumer;
. preserve proven contracts;
. repeat for the next transformation family.

The FK-helper slice became the first complete execution of this process.

Artifacts:

* FK-helper lifecycle review;
* metadata-path inventory;
* functional-model concept;
* implementation review;
* Slice 1 cleanup;
* Slice 2 bridge retirement;
* consolidation reviews;
* closure review.

=== Main Conclusion

The FK-helper investigation changed the understanding of the project.

The review did not prove that `_meta` should disappear.

The review did not prove that the metadata architecture was wrong.

Instead it showed that:

* most runtime contracts were already coherent;
* many architecture guards were correctly protecting those contracts;
* several difficult bugs originated from historical compatibility layers
  and metadata residue rather than from the architecture itself.

The result was a shift from:

"replace the metadata model"

toward:

"recover the intended model slice by slice."

=== Impact on Future Work

Future semantic adoption work should follow the same sequence:

. functional understanding first;
. implementation inventory second;
. targeted simplification third;
. no framework-wide redesign without evidence from multiple slices.

Transformation families such as:

* legend blocks;
* xref crosstables;
* compact multiaxis;

are expected to use the same process.

The goal is not architectural revolution.

The goal is gradual recovery of conceptual clarity while preserving
working behavior and consumer compatibility.

=== Historical Assessment

In retrospect, the project's rapid growth phase successfully produced a
working framework and a real consumer ecosystem.

The cost of that speed was localized semantic sediment and compatibility
layers.

The FK-helper wave demonstrated that these problems can be addressed
incrementally without destabilizing the overall architecture.

This review therefore marks the transition from architecture creation to
architecture comprehension.
