== Architectural Runway

=== Current Phase: 0.2.x Stabilization

The primary objective of the 0.2.x line is not rapid feature growth.

Instead, the focus is on learning from real-world adoption and reducing
friction encountered by actual users.

==== Immediate: Current cleanup and review closure

The first post-`v0.2.0` process gap is closed:
`FTR-POST-RELEASE-LIFECYCLE-AND-RESET-P5` defines how the repository returns
to normal development after a tag. Release automation remains postponed under
`FTR-RELEASE-PROCESS-AUTOMATION-P5`.

Current immediate work is therefore not release reset, but backlog hygiene and
review closure:

* archive closed FTRs and their review artifacts in cold storage;
* close or explicitly amend the Yellow review gap on
  `FTR-DYNAMIC-FRAME-PROJECTION-IMPL-P5`;
* finish the remaining edge in
  `BUG-COLUMN-LEVEL-PRESENTATION-META-READBACK-P5`;
* keep release-notes entries in the `unknown -- planned` section until the next
  tag ships.

==== Phase 1: Adoption-driven boundary hardening

Theme:
Use cases drive priorities; adoption exposes the boundaries that need hardening.

The project already treated real use cases as the steering mechanism. The
post-`v0.2.0` adoption wave clarified what that means in practice: adoption
does not only produce feature requests. It also makes implicit architectural
boundaries visible when consumer workflows cross them in unexpected ways.

Current boundary axes:

* runtime metadata vs persisted project state;
* declarative intent vs computed resolution;
* explicit cell metadata vs column-level presentation defaults;
* projection vs transformation;
* problem-space explanation vs semantic contract;
* consumer contract vs backend carrier representation.

Each adoption-discovered boundary issue should end in an explicit rule, a
guard or regression test where practical, and documentation that states the
contract.

Focus areas:

* additional adoption projects
* reduction of onboarding friction
* closure of adoption-discovered gaps
* roundtrip robustness — user-authored display settings in particular;
  column widths, text orientation, horizontal alignment, vertical alignment,
  and carrier-authoritative clearing are done; column-level readback remains
  the active presentation-meta gap
* semantic-model clarification
* incremental architectural refinement
* selective completion of functionality required by real users

Architectural work is not frozen during Phase 1.

Refactoring, semantic-model clarification, and guardrail maintenance proceed
slice-wise alongside adoption work, tied to real use cases and active tickets
rather than isolated campaigns.

Success indicators:

* fewer surprises during adoption
* decreasing number of corrective FTRs and BUGs
* growing confidence in the semantic model
* smoother project setup experience

==== Phase 2: User-facing experience

Theme:
Make successful usage easier to discover and reproduce.

Focus areas:

* improved documentation structure — clear separation of release-versioned
  core docs from latest-oriented tutorials and integration patterns
  (FTR-USER-FACING-DEMO-AND-INTEGRATION-DOCUMENTATION-P5)
* additional reveal.js walkthroughs
* workflow-oriented tutorials
* AI-assisted pipeline authoring support
* clearer user-facing entry points on Pages
* improved discoverability of common patterns

Success indicators:

* reduced maintainer involvement during onboarding
* users can reproduce common workflows from documentation
* AI systems can assist pipeline authoring from curated context material

=== Next Milestone: target version TBD

Theme:
Adoption-ready boundary contracts and improved user surface.

The goal of the next milestone is not long-term API stability guarantees.

Instead, the goal is that external users can successfully adopt the library
with limited maintainer assistance through a combination of:

* stable core workflows
* adoption-tested patterns
* explicit boundary contracts for the workflows adoption already exercised
* curated documentation
* tutorial material
* AI-friendly guidance artifacts

=== Strategic Principle

Real adoption projects drive roadmap priorities.

New features and capabilities should normally emerge from observed user friction
rather than speculative feature design.

Architectural work — refactoring, semantic clarification, guardrail maintenance —
is not subject to the same gate. It continues incrementally throughout the 0.2.x
line, slice by slice, tied to concrete problems rather than campaigns.
