== Project Evolution Narrative

This chapter explains why the repository looks the way it does.

Use it when:

* a commit references an FTR and the local context is unclear;
* a module shape looks surprising;
* a review finding appears in a commit message;
* an AI agent needs project history without loading the full FTR archive.

Detailed FTR files live in cold storage or the active backlog.
This chapter only preserves the short engineering story.

== Era 0: Proof of Concept (Aug-Sep 2025)

The project started on 2025-08-29 as a small Python converter:

* JSON input.
* XLSX output.
* spreadsheet editing by humans.
* JSON reconstruction for downstream use.

The first domain was structured story/world-building data.
The domain was later removed from the project vocabulary.

Early traits already visible:

* deterministic output mattered;
* column order mattered;
* pandas index leakage had to be suppressed;
* spreadsheet roundtrip was more important than export only;
* metadata and payload needed separation.

Architecture at this stage:

* direct pandas/openpyxl usage;
* no IR layer;
* no backend contract;
* no stable layer model;
* the DataFrame acted as the effective intermediate representation.

== Era 1: Generic Pipeline Direction (Sep-Oct 2025)

The project moved from one converter to a general spreadsheet workflow tool.

Important changes:

* domain-specific names were removed;
* YAML-driven pipeline ideas appeared;
* backend abstraction became a design topic;
* release and backlog discipline started;
* AI-facing project notes became part of normal development.

FTR anchors:

* `FTR-BACKEND-ABC` - backend abstraction direction.
* `FTR-IR-WRITEPATH-P1` - early Render IR direction.
* `FTR-DEV-MARKERS` - development helper markers.
* `FTR-COLUMN-MAP` - column mapping infrastructure.
* `FTR-TEST-HARNESS` - test harness and backend markers.
* `FTR-ROUNDTRIP-SAFE-P1` - first roundtrip-safe path.

Meaning for later work:

* FTRs became memory, not only tickets.
* AI sessions could resume from written context.
* Deferred ideas were recorded instead of kept implicitly.

== REVIEW-001: Architecture Review and Fix Wave (May 2026)

REVIEW-001 was a four-part architecture review.

Review direction:

* domain and core boundaries;
* IR and rendering flow;
* adapters, CLI, and orchestration;
* consolidated release risk.

Main findings:

* one real hexagonal violation: domain imported rendering formula objects;
* fragile `workbook_meta_blob` serialization via Python `repr`;
* duplicated backend dispatch and orchestration paths;
* misleading rendering pass package structure;
* unregistered derived metadata channel;
* weak guard coverage in several boundaries;
* module-size and naming drift.

Fix wave FTRs:

* `FTR-REVIEW-001-QUICK-WINS-P3`
* `FTR-REVIEW-001-FORMULAS-CORE-MOVE-P3`
* `FTR-REVIEW-001-META-BLOB-JSON-P3`
* `FTR-REVIEW-001-META-REGISTRY-DERIVED-P3`
* `FTR-REVIEW-001-PASSES-CLEANUP-P3`
* `FTR-REVIEW-001-BACKEND-DISPATCH-P4A`
* `FTR-REVIEW-001-BACKEND-DISPATCH-P4A-SLICE02`
* `FTR-SOURCE-STYLE-CLEANUP-P4`
* `FTR-SOURCE-STYLE-GUARDRAILS-P4`

Outcome:

* formula intent moved to `core`;
* metadata serialization became deterministic JSON;
* derived metadata became registered;
* rendering passes became real modules;
* backend dispatch was centralized;
* CLI surface cleanup started;
* guardrails became part of normal architecture work.

Useful shorthand:

* before REVIEW-001: hexagonal architecture by intention;
* after REVIEW-001 follow-up: hexagonal architecture by enforcement.

== REVIEW-002: From Structure to Semantics (May 2026)

REVIEW-002 reassessed the architecture after the REVIEW-001 fix wave.

Main conclusion:

* release-relevant REVIEW-001 findings were closed;
* the second backend proved the spreadsheet contract;
* package split methodology was now repeatable;
* test architecture was beta-grade;
* remaining risks were more semantic than structural.

Open or recently addressed items:

* `FTR-FINDING-MODEL-CONSOLIDATION-P5` consolidated accidental finding-frame duplication.
* `FTR-IR-006-RENDER-INPUT-MUTATION-FIX` removed render input mutation risks.
* The later legend reconstruction follow-up showed that some fields looked render-local but also carried reconstruction meaning.

The important lesson:

* not every surprising field is architectural debt;
* some fields encode reconstruction authority;
* metadata meaning must be modeled before more refactoring;
* semantics must no longer remain implicit in implementation details.

This is why the project paused broader refactoring work and started explicit model documentation.

== v0.1.0b6 Beta Release and Release-Check Hardening (May 2026)

The first end-to-end beta release shipped to PyPI on schedule. The
release artifacts were correct, but the tagging sequence was not:
`v0.1.0b6` was first tagged from `dev` rather than `main`, which
produced operator uncertainty, CI noise, and a manual fast-forward
repair of `main` after the fact.

Closed in this wave:

* `FTR-RELEASE-RUNBOOK-BRANCH-AWARENESS-P5` -- introduced
  `tools/release_check.sh` and `make release-check`, a strictly
  read-only pre-tag check that exits non-zero when the current branch
  is not the release branch, when the working tree is dirty, when
  local `main` is out of sync with `origin/main`, or when an
  explicitly named tag is not reachable from `origin/main`. Codex
  follow-up added named-tag validation, advisory-warning shaping, and
  the runbook section that codified "tag from `dev`" recovery. The
  helper makes no fetches and no writes; the developer runs
  `git fetch origin` before invoking it.

Meaning:

* the v0.1.0b6 release artifacts are reachable from `main` after the
  follow-up fast-forward; future releases can detect this class of
  topology drift before pushing the tag;
* CI hygiene gaps surfaced during the release became the motivation
  for the Phase 1 wave that followed (see below).

== Phase 1: Release and CI Stabilization (May 2026)

The v0.1.0b6 release proved the publishing architecture end-to-end
but left operator cognitive load high: which CI signals were
release-blocking, what state was the demo pin in, did Pages actually
receive the snapshot, what was the safe recovery for each common
mistake. Phase 1 closed those gaps without redesigning the publish
flow.

Closed wave (four FTRs, chronological order):

* `FTR-DEMO-CI-SMOKE-AND-LEGACY-TARGET-CLEANUP-P5` -- removed broken
  legacy `make pack/unpack/roundtrip/verify/extract` targets from the
  demo `Makefile` whose underlying entrypoints (`sheets-pack`,
  `sheets-unpack`) and config (`sheets.yaml`) no longer existed on
  the supported core surface. Deleted orphan scripts
  (`scripts/sha.py`, `scripts/unpack_current_sha.sh`,
  `scripts/verify.py`), trimmed dead exports from
  `scripts/dev-local.lib.sh`, replaced the legacy compat block in
  `docs/reference/commands.adoc` with a one-paragraph note, and added
  an explicit three-layer intent header to the demo `ci.yml`
  (install / pytest demo journeys / public CLI smoke).
* `FTR-GITHUB-ACTIONS-MAINTENANCE-P5` -- declared minimal
  `permissions: contents: read` on every core and demo workflow with
  job-level overrides only where strictly needed (`id-token: write`
  on the PyPI Trusted Publisher job; `DOCS_DEPLOY_KEY`-backed
  publishing on the Pages-push jobs). Documented the Dependabot
  classification (no `dependabot.yml` checked in to any of the three
  repos; observed `403 Token invalid` originates from repo Security
  UI settings, requires manual UI inspection, does not gate any
  release-critical workflow). Codex's skeptical follow-up found that
  the initial documentation overstated upstream action-major state;
  pins were bumped to verified current majors
  (`actions/checkout@v6`, `actions/setup-python@v6`,
  `actions/setup-java@v5`, all declaring `node24` runtime) and
  documentation now records the verified-on date.
* `FTR-PAGES-PUBLISH-POSTDEPLOY-VALIDATION-P6` -- added
  `tools/pages_publish_check.sh` and `make pages-check`, a read-only
  structural validator for the Pages working tree. Classifies each
  expected publish section as missing / partial / plausible with
  distinct exit codes (0 / 1 / 2; 3 reserved for setup error). Codex
  follow-up tightened the demo-latest section to flag empty slide
  files symmetrically with the demo-versioned section.
* `FTR-CROSS-REPO-RELEASE-ORCHESTRATION-P6` -- added
  `tools/release_status.sh` and `make release-status`, a read-only
  one-shot cross-repo state summary (per-repo branch/HEAD/tag, demo
  pinned-core, pages versioned snapshots, cross-correlation, likely
  next-action hint). Codex follow-up hardened invalid-path handling,
  visibility-gap reporting, parameter-expansion-safe path stripping,
  and `GIT_OPTIONAL_LOCKS=0` enforcement. Runbook gained the
  three-stage Cross-Repo Release Sequence table and the five
  Cross-Repo Recovery Scenarios (demo tag pushed before main
  alignment; stale workflow rerun; PyPI succeeded but docs failed;
  demo publish green but CI hygiene red; pages publish green but
  public site not yet visibly updated). `repository_boundaries.adoc`
  gained the release-time data-flow direction table making
  `core -> PyPI`, `demo -> released core`, `pages <- publish outputs`
  unmistakable.

Outcome:

* three composable read-only helpers (`release-check`, `pages-check`,
  `release-status`) tied together by a per-question helper table in
  the runbook;
* runbook now distinguishes release-blocking CI from non-blocking
  maintenance noise and documents the six common recovery cases;
* workflow `GITHUB_TOKEN` permissions are minimal-by-default;
* the cross-repo release flow is documented as a three-stage
  sequence with explicit dependencies rather than as scattered
  per-repo checklists.

Non-goals deliberately preserved:

* no release automation (no semantic-release, no automatic version
  propagation, no CI invocation of the new helpers);
* no `dependabot.yml` introduced (Dependabot policy is the
  maintainer's UI decision, not workflow code);
* no demo or pages workflow redesign;
* no change to publish behavior.

One operational follow-up remains, owned by the maintainer rather
than a separate FTR: inspect
`Settings -> Code security and analysis` on each of the three repos
to decide whether to keep Dependabot security updates enabled or to
disable them, and confirm `Settings -> Actions -> General -> Workflow
permissions` defaults to read-only.

== Phase 3.5: Semantic Model Foundations & Adoption Plan (May 2026)

=== Motivation

The FK-helper architecture review (REVIEW-002) surfaced a recurring friction
class: implicit metadata semantics. Multiple attributes — `_meta.derived`,
`helper_columns`, `helper_policies.fk` — were simultaneously "derived",
"policy", "transient", "persistent", and "provenance", with no shared
vocabulary to resolve conflicts between them. Registries (`meta_registry.yaml`,
`pipeline_step_registry.json`) were being used as test oracle and operational
truth, but that role was undocumented. Architecture guards made strong claims
about inference prohibitions without a backing model. Code review and agent
reconstruction of semantics were both slower than they should be.

Phase 3.5 addressed that friction systematically, within a deliberately narrow
documentation-first scope: no runtime refactoring, no general rule engine, no
big-bang adoption.

=== What Was Built

Six FTRs in sequence, each a reviewable slice of the seed scaffold under
`docs/semantic_model/`:

* `FTR-SEMANTIC-MODEL-SKELETON-P5` — integrated the seed scaffold into the
  docs build and stabilized the chapter structure (problem statement,
  inspirations, overlapping sets, lifecycle/update semantics, registry role,
  adoption strategy, contract matrices, semantic guards).

* `FTR-SEMANTIC-PROBLEM-STATEMENT-P5` — elaborated the motivating problem:
  attribute ambiguity, implicit contracts, and the gap between registry truth
  and runtime trust.

* `FTR-SEMANTIC-INSPIRATIONS-P5` — surveyed the design space (schema
  registries, RDF/OWL, CDC, bitemporal models, probabilistic classifiers) and
  recorded what applies and what does not.

* `FTR-SEMANTIC-SETS-AND-LIFECYCLE-MODEL-P5` — defined overlapping semantic
  sets (Derived, Transient, Policy, Persistent, RoundtripRelevant,
  HeuristicallyInferred, TransitionBridge, Deprecated, etc.) and lifecycle
  events (Establish, Transform, Reimport, Serialize, Reconstruct,
  End-of-life). Established the precedence and update-semantics rules that
  resolve ambiguity across set membership.

* `FTR-SEMANTIC-REGISTRY-ROLE-P5` — pinned the registry-role distinctions:
  `meta_registry.yaml` and `pipeline_step_registry.json` are operational
  semantic truth and review oracle, not generic runtime rule input. That
  distinction is now explicit and referenced by architecture guards.

* `FTR-SEMANTIC-ADOPTION-STRATEGY-P5` — defined the no-big-bang slice
  template: adopt the semantic model transformation-family by family in narrow
  vertical slices. FK helpers were selected as the first slice because they
  hit all five pilot-selection criteria before any other family (recent review
  pain, bounded primitive surface, live v2 runtime contract, roundtrip
  exposure, existing guard scaffolding).

* `FTR-SEMANTIC-CONTRACT-MATRIX-PILOT-FK-HELPERS-P5` — executed the first
  adoption slice. Produced the FK Helper Contract Matrix (three sub-tables
  across role/metadata-contract, inference/missing-metadata/lifecycle, and
  persistence/provenance/public-surface/registry dimensions). Cross-checking
  the matrix against registries discovered four stale notes in
  `pipeline_step_registry.json` — all residue from before
  `FTR-FK-HELPERS-POLICY-DRIVEN-PRIMITIVES-P5` completed. Corrected. Extended

== Adoption-001: First External Consumer Validation (May-June 2026)

The framework was exercised against a real external consumer for the
first time: a worldbuilding repository for a fictional setting, used
purely as a productive consumer of `spreadsheet-handling`. The

Strategic finding:

* The framework's center of gravity is shifting from *file
  transformation* (JSON <-> XLSX) toward *projection management*
  (canonical model -> editable projections -> roundtrip -> canonical
  model). The spreadsheet is an editing surface, not the system of
  record. Most observed friction sits at the framework/consumer
  boundary -- projection ergonomics, configuration overhead,
  reimport-safety workflow -- not at the transformation core.

Concrete findings closed during the wave:

* `BUG-ODS-REIMPORT-REPEATED-CELL-MEMORY-EXPLOSION` -- the ODS

== Post-0.2.0 Boundary Discovery (June 2026)

The period after the first non-beta release was not primarily a feature-growth
period. It was a boundary-discovery period.

The same pattern repeated across documentation, adoption, and bug work:

* real consumer usage or focused review exposed surprising behavior;
* the surprise first looked like a local bug or wording problem;
* investigation revealed an implicit boundary that the architecture had relied
  on but not stated strongly enough;
* the project turned the boundary into a named rule, guard, test, or
  documentation contract.

The recurring boundary questions were:

* problem-space explanation vs semantic contract;
* runtime metadata vs persisted project state;
* declarative intent vs computed resolution;
* visible carrier authority vs embedded metadata;
* projection vs transformation;
* consumer contract vs backend carrier representation.

This is why several post-release changes look small in code but large in
meaning. They made implicit ownership rules explicit.

== Functional / Semantic Model Recut (June 2026)

The semantic-model work before `0.2.0` deliberately created a shared
vocabulary for metadata meaning, lifecycle state, registry authority, and
contract review. After the release, that success created a documentation
pressure of its own: explanatory problem-space material, motivating examples,
contract vocabulary, and lifecycle rules were all easy to route through the
Semantic Model.

The project clarified the reader contract:

* Functional Model -- explain the problem space and why workbook projections
  exist.
* Semantic Model -- define interpretation rules, lifecycle guarantees, and
  metadata contracts.

The independent assessment later corrected the blunt version of the diagnosis:
the Semantic Model was not simply "wrong" to contain problem-space framing.
The more accurate problem was structural. The Functional Model had been too
thin to host that framing, so the Semantic Model became the natural place to
put it. The recut improved navigation by giving problem-space context a home,
while keeping semantic contract growth behind the Semantic Model's adoption
and slice rules.

== Post-0.2.0 Presentation-Metadata Series (June 2026)

The first post-release feature wave focused on adoption-driven spreadsheet
display metadata. The work preserved user-authored presentation choices across
XLSX and ODS roundtrips without making display metadata part of domain logic.

The engineering challenge was not just adding more formatting keys. Text
orientation, horizontal alignment, vertical alignment, stale embedded metadata,
and column-level formatting all exposed the same deeper question: which
presentation source is authoritative when workbook carriers and embedded meta
disagree?

The answer adopted by the completed carrier-authority slice is:

* visible carrier state is authoritative on parse;
* non-empty carrier extraction replaces the embedded family entry;
* empty carrier extraction clears any persisted embedded value, so cleared
  formatting is not silently reapplied on the next write.

Column-level formatting is a related but still active edge. It introduces the
fallback question inside the visible carrier itself: explicit cell formatting
should win, column defaults should act only as fallback, and repeated-cell /
sheet-wide styling must remain bounded by parser limits. That rule is not
fully closed for every presentation family yet, which is why
`BUG-COLUMN-LEVEL-PRESENTATION-META-READBACK-P5` remains active.

Closed in this wave:

* `FTR-TEXT-ORIENTATION-ROUNDTRIP-P5` -- preserved vertical / rotated cell text
  under the `text_orientations` metadata family.
* `FTR-CELL-ALIGNMENT-ROUNDTRIP-P5` -- preserved horizontal alignment under the
  `horizontal_alignments` family.
* `FTR-PRESENTATION-META-CARRIER-AUTHORITY-P5` -- made the visible carrier
  authoritative for presentation families and consolidated replace-or-clear
  update semantics.
* `FTR-VERTICAL-ALIGNMENT-ROUNDTRIP-P5` -- preserved vertical alignment under
  the `vertical_alignments` family.

One related read-path defect remains active:

* `BUG-COLUMN-LEVEL-PRESENTATION-META-READBACK-P5` -- column-level formatting
  readback received a narrow implementation, but the review still records an
  unresolved text-orientation fallback edge. It stays in the active backlog
  until the remaining finding is closed or explicitly descoped.

== Dynamic Projection and Workbook Meta Hardening (June 2026)

The worldbuilding adoption then pushed the project from display preservation
into projection ergonomics and cross-carrier metadata robustness.

Projection was the product-side version of the same boundary problem. The old
mental model was readers -> transformations -> validations -> writers. The
worldbuilding consumer needed workbook views that were not merely transforms
and not merely render operations: they were editable projections over a
canonical model.

That evidence made projection a first-class concept:

* projected-frame column roles name what survives across a view;
* `project_by_role` shapes an editable frame by those roles;
* dynamic view-column targeting remains the follow-up question for downstream
  validation / presentation rules over generated columns.

At the same time, workbook-view bugs showed that metadata boundaries were not
only within one backend. The immediate XLSX readback failures were real, but
the deeper failure was that an ODS-produced sidecar could later poison an XLSX
workflow. That reframed `_meta` persistence from "serialize canonical meta" to
"project runtime state into a persistable contract before any carrier writes".

The rule that emerged is:

* persist declarative intent;
* discard computed resolution;
* recompute resolution from current data, layout, and pipeline state.

This rule builds on the earlier runtime-metadata persistence boundary from
Adoption-001: runtime-produced helper policy and reconstruction state must not
quietly become canonical input for the next run.

Closed concept and decision work:

* `FTR-PROJECTED-FRAME-COLUMN-SEMANTICS-P5` -- named the foundation roles
  (`row_identity`, `display_helper`, `matrix_value`) that later projection and
  targeting work consume.
* `FTR-DYNAMIC-FRAME-PROJECTION-P5` -- recorded the decision to introduce a
  role-based projection surface rather than keep the worldbuilding projection
  plugin as the long-term recommended path.

Implemented but still review-marked:

* `FTR-DYNAMIC-FRAME-PROJECTION-IMPL-P5` -- added `project_by_role`, the shared
  role resolver, registry wiring, release notes, and worldbuilding YAML
  migration. The implementation review is Yellow because the migration-parity
  row-order fixture required by the FTR is still missing. The ticket remains
  active until that test gap is closed or the FTR scope is amended.

Closed bug fixes:

* `BUG-XLSX-WORKBOOK-VIEW-BLOB-READBACK-P4A` -- recovered workbook-view
  metadata from the XLSX hidden-sheet blob shape.
* `BUG-XLSX-WORKBOOK-VIEW-CANONICAL-META-LOSS-P4A` -- stripped empty
  hidden-sheet marker keys so canonical workbook meta is not replaced by
  wrapper noise.
* `BUG-CROSS-CARRIER-META-ROUNDTRIP-P4A` -- added the narrow
  Intent-vs-Resolution persistence-boundary slice for the ODS-produced
  sidecar fixture, pruning resolved legend/xref facets while preserving
  declarative intent. Residual lifecycle classification remains owned by
  `FTR-META-LIFECYCLE-INVENTORY-P5`,
  `FTR-FK-HELPER-POLICY-LIFECYCLE-P4A`, and
  `FTR-DERIVED-PROVENANCE-LIFECYCLE-P4A`.

Meaning:

* the framework's active product direction is now projection management:
  canonical model -> role-shaped workbook projection -> controlled reimport;
* role taxonomy and projection shape are no longer speculative;
* targeting dynamic projected columns remains the next open product question;
* cross-carrier bugs exposed that `_meta` lifecycle classification must keep
  moving slice by slice.

== Backend State

Backend roles:

* JSON: primary structured serialization target.
* YAML: structured writer for nested configuration and frame exchange.
* XLSX: primary spreadsheet backend.
* ODS/Calc: second spreadsheet backend proving the contract.
* CSV: structurally limited; useful but not layout-rich.
* XML: legacy PoC-era backend; likely scope-limited or future deprecation candidate.

This asymmetry is intentional.
All backends do not need equal capability.
Spreadsheet backends carry layout and roundtrip semantics that flat formats cannot carry.

== Reading Rules for Future Agents

When a commit references an FTR:

* first locate the FTR in this chapter;
* then load the detailed FTR only if the short context is insufficient;
* prefer review summaries over raw review prompts;
* treat cold storage as archaeology, not default context.

When a refactor looks unexpectedly conservative:

* check whether metadata or reconstruction semantics are involved;
* check whether the related FTR was review-driven;
* avoid "cleanup" that removes fields before their authority is modeled.

When in doubt:

* preserve behavior;
* preserve public import paths;
* preserve registry names;
* add or update guards with the change.
