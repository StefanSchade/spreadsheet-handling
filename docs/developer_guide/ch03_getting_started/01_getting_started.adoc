=== Prerequisites

* Python >= 3.10 on a POSIX/Linux system
* `make`, `git`
* A virtual env is managed entirely by the Makefile — do not create one manually

=== Clone and Bootstrap

[source,console]
----
git clone https://github.com/StefanSchade/spreadsheet-handling.git
cd spreadsheet-handling
make setup          # creates .venv, installs all dev deps from pyproject.toml
----

`make setup` is idempotent and stamp-driven: it only re-installs when
`pyproject.toml` has changed since the last run.

=== Verify the Installation

[source,console]
----
make test           # unit + integration + architecture tests (active suite)
make run            # demo roundtrip on the bundled example
----

If either fails after a clean clone, `make doctor` shows the environment and
stamp state.  A reliable recovery path is:

[source,console]
----
make clean-stamps && make setup
----

=== Develop the Demo Against a Local Core Checkout

Use this only for local development. Release and CI validation of the demo
use the released `spreadsheet-handling` version pinned by the demo project.

Expected sibling layout:

[source,console]
----
workspace/
  core/                         # spreadsheet-handling core checkout
  demo/                         # spreadsheet-handling-demo checkout
----

In the demo repository:

[source,console]
----
make setup
make setup-lib-local LIB_SRC=../core
.venv/bin/python -c "import spreadsheet_handling, inspect; print(inspect.getfile(spreadsheet_handling))"
----

If your sibling path differs, either pass `LIB_SRC=/path/to/core` or adjust the
demo Makefile default locally. Do not commit workstation-specific absolute
paths.

To return to the released dependency:

[source,console]
----
make setup-lib-pypi
----

If the demo `pyproject.toml` is temporarily changed to a local path dependency
for development, restore the released version pin before committing release or
CI-facing changes.

=== Following Runtime Metadata Changes

Use the existing CLI verbosity behavior to inspect structural runtime metadata
movement through a pipeline. Passing `-vv` enables standard DEBUG logging; no
separate trace option or logging level is needed.

After each successfully completed step, the pipeline runner reports which paths
under runtime `_meta` were added, changed, or removed. An unchanged step gets a
compact `meta: unchanged` result. The summary reports structural names, not
metadata values, and is intended to help a maintainer follow metadata
construction during development and review. For the pipeline boundary and the
runtime-versus-persistable metadata distinction, see
link:../../technical_model/ch06_architectural_layers/architectural_layers.adoc[Architectural Layers]
and the repository-local pipeline persistence guidance.

This trace is best-effort. It is neither an audit log nor a canonical semantic
record, and it does not decide whether a metadata change is correct. No stable
machine-readable log format is promised. Values are not logged by the new
summary, but structural names can include project-selected frame, sheet,
column, contract, or plugin terminology, so DEBUG output remains potentially
sensitive operational output.

The existing pre-step configuration DEBUG line is outside the metadata
summary's data-minimisation guarantee. Enabling DEBUG exposes the complete
application debug stream, not only this structural trace.
