===== FTR-SIMPLE-CLI-SHIMS-P6 -- Revisit shortcut commands as intentional simple-user UX

*Status:* Superseded by the pre-beta public-surface decision in
FTR-REVIEW-001-QUICK-WINS-P3 / A-002.

*Context:*::
The historical `sheets-pack` and `sheets-unpack` commands came from an earlier
project phase where the
tool mainly converted a handful of JSON files into a single XLS workbook and
back again. Since then the system has gained a generic run path, multiple
spreadsheet adapters, helper-column enrichment, validation behavior, and a much
broader notion of what a spreadsheet pipeline can do.

The pre-beta API-surface decision has now removed those misleading names
entirely and replaced them with explicit reference shortcut commands:
`sheets-example-json-to-xlsx` and `sheets-example-xlsx-to-json`.

*Problem statement:*::
The current shim commands are no longer clearly named for the system they sit
on top of:

* `pack` / `unpack` describe an old mental model rather than today's richer
  pipeline behavior,
* but the idea of thin convenience commands is still valuable for casual users
  who should not need `config.yaml` plus `pipeline.yaml` for common flows.

The immediate naming problem is resolved. Phase 4 may still add further
features that deserve exposure through a simple CLI layer, but that future work
should start from the shortcut-command pattern rather than from the old names.

*Goal:*::
After Phase 4, reassess the simple CLI surface as a deliberate UX layer:

* decide which high-value simple workflows deserve dedicated commands,
* decide which reference examples should become stable product shortcuts,
* keep any retained shims as thin wrappers over the generic run/orchestrator
  path,
* and document which simple commands are stable entry points for casual users.

*Non-goals for now:*::

- No deprecated aliases for the removed historical names.
- No attempt to make the reference examples the central product UX.
- No removal of convenience entry points in favor of `sheets-run` only.

*Placement rationale:*::
This item is superseded for the old naming question. Future UX shaping belongs
in a fresh FTR once concrete product shortcuts are known.

*Acceptance criteria:*::

* A small, explicit decision is documented for the simple-command layer after
  Phase 4.
* The retained command set reflects actual high-value casual-user workflows
  rather than historical naming accidents.
* No `pack` / `unpack` command names survive as compatibility aliases.
* The simple-command layer remains a thin wrapper over the unified run path
  rather than growing its own orchestration logic.
