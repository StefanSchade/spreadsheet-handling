===== FTR-QUALITY-DASHBOARD-SONAR-PILOT -- Optional read-only quality dashboard

*Status:* Future
*Priority:* Low
*Area:* Quality tooling / observability

==== Purpose

Evaluate whether SonarQube or a comparable quality dashboard adds useful
long-term signal beyond the repository-local tests, source-style guardrails,
and architecture guards.

The project already has high-signal custom checks for its own architecture.
An external dashboard may still be useful later for trend visibility:
complexity, duplication, coverage import, security hotspots, and slow drift
across releases.

==== Trigger

Consider this after the next beta cycle has stabilized and the repo-native
quality commands are boring to run. It should not block the near-term beta or
replace focused project-specific guardrails.

==== Scope

* Create a minimal `sonar-project.properties` or equivalent configuration.
* Exclude generated/build artifacts and internal backlog files where needed to
  avoid noisy results.
* Import test coverage if coverage reporting is already present.
* Run SonarQube as a read-only dashboard first.
* Document which findings are actionable for this project and which are
  intentionally ignored.

==== Non-goals

* Do not make SonarQube a release gate during the pilot.
* Do not weaken repository-specific architecture guards because a generic tool
  reports a green dashboard.
* Do not add server administration requirements to ordinary local development.
* Do not chase low-signal style findings that duplicate Ruff, Black, or the
  source-style guardrails.

==== Acceptance

. A pilot report records whether SonarQube provides useful additional signal.
. Any proposed quality gate is explicitly reviewed before it can become
  blocking.
. The project documentation states that repository-local tests and guards
  remain authoritative for project-specific architecture and behavior.
