===== FTR-DOMAIN-TRANSFORMATION-MODULE-SPLIT-FK-HELPERS-P5 -- Split FK helper transformation module without behavior changes

*Status:* New
*Priority:* P5
*Area:* Domain transformations / maintainability
*Kind:* Behavior-preserving refactoring

==== Purpose

Split the FK helper transformation implementation into a small package with
clear internal responsibilities, without changing runtime behavior, pipeline
configuration semantics, metadata lifecycle, or public step names.

The current FK helper implementation has grown beyond a single-purpose module.
It owns enrichment, helper cleanup, policy resolution, provenance updates, and
formula-provider integration in one file. That made sense while the feature
was evolving quickly, but the surrounding architecture is now stable enough to
make the code easier to review and maintain.

This FTR is a maintainability slice. It should reduce local cognitive load for
future agents and reviewers. It must not be used to introduce new FK helper
semantics.

==== Context

Recent pre-beta work intentionally tightened workbook projection, reimport,
ODS parity, prototype consumer integration, and demo integration. The project
is entering a maturity phase where small, behavior-preserving refactorings are
valuable if they make stable capabilities easier to understand.

FK helpers are a good first candidate because they are important, relatively
self-contained, and already have meaningful unit, architecture, and demo
coverage.

This FTR may serve as the pattern for later modularization of other
transformation modules such as discriminator split/merge or tabular views, but
those are explicitly out of scope for this slice.

==== Target Shape

The intended shape is approximately:

[source,text]
----
src/spreadsheet_handling/domain/transformations/fk_helpers/
  __init__.py
  enrich.py
  drop.py
  policy.py
  provenance.py
  formula_provider.py
----

Exact file names may differ if the code shows a cleaner smaller cut, but the
responsibility split should stay close to:

[cols="1,3",options="header"]
|===
| Module | Responsibility

| `enrich.py`
| Main FK helper enrichment orchestration and public enrichment entry point.

| `drop.py`
| Helper-column removal and cleanup entry point.

| `policy.py`
| Resolution and validation of FK helper policies from `_meta` and inline
  config.

| `provenance.py`
| Writing and cleaning `_meta.derived` helper-column provenance.

| `formula_provider.py`
| Lookup formula provider construction for formula-mode helpers.
|===

The package `__init__.py` should re-export the stable functions used by the
pipeline registry and tests, so callers do not need to know the internal file
layout.

==== Scope

* Move FK helper implementation code from the current single module into a
  package structure.
* Preserve all public function names that the registry, tests, or documented
  extension points currently use.
* Preserve pipeline step IDs and YAML configuration shape.
* Preserve `_meta.derived` helper provenance semantics exactly.
* Preserve helper cleanup behavior exactly.
* Preserve formula/value helper modes exactly.
* Update imports, registry references, architecture docs, and any reviewed
  symbol paths that point at the old module.
* Add only minimal tests if import/re-export behavior is not already covered.

==== Non-Goals

* No new FK helper functionality.
* No helper-policy redesign.
* No pipeline step renaming.
* No compatibility-alias policy changes.
* No changes to workbook-view semantics.
* No changes to `_meta.derived` lifecycle beyond preserving current behavior.
* No derived-provenance persistence/orphan-cleanup work.
* No formula-provider redesign.
* No broad `domain/transformations` package reorganization.
* No discriminator, tabular-view, xref, resource, or workbook-view refactoring.
* No changes to demo workflows except validating that they still pass.

==== Explicit Behavior-Preservation Rule

This FTR is successful only if the runtime behavior is unchanged.

Allowed changes:

* moving code;
* renaming private helpers where necessary for module boundaries;
* updating imports;
* adding narrow import/registry guard tests;
* updating documentation and architecture references to the new module paths.

Not allowed:

* changing produced DataFrames;
* changing helper column names or ordering;
* changing FK policy resolution;
* changing error semantics except where import paths appear in messages;
* changing metadata payloads;
* changing pipeline configuration syntax;
* adding legacy aliases or removing current public symbols.

==== Implementation Plan

The implementation should proceed in three phases inside one focused task:

. Mini inventory
  * Inspect the current FK helper module, registry bindings, tests,
    architecture docs, meta registry references, and demo configs.
  * Produce a short move plan before editing.

. Mechanical split
  * Create the package.
  * Move code by responsibility.
  * Keep public entry points re-exported from `fk_helpers/__init__.py`.
  * Update imports and symbol paths.

. Verification
  * Run focused FK helper tests.
  * Run pipeline registry / architecture guard tests.
  * Run full core test target if feasible.
  * Run demo smoke tests against the local editable core checkout.

If the mini inventory reveals that the split is not mechanical, stop and
record the blocker instead of broadening the refactor.

==== Acceptance

* FK helper code is split into a package with clear internal responsibilities.
* Registered pipeline steps continue to work with the same step IDs and YAML
  config.
* Existing public imports used by the registry and tests continue to work.
* `_meta.derived` helper provenance output is byte-/structure-equivalent for
  existing test fixtures.
* Helper cleanup behavior is unchanged.
* Formula-mode and value-mode helpers behave unchanged.
* Pipeline step registry and architecture documentation refer to the current
  symbol paths where relevant.
* Core FK helper, pipeline, architecture, and relevant integration tests pass.
* Demo smoke tests still pass against the local editable core checkout.
* The diff is predominantly move/import/documentation churn, not logic change.

==== Review Guidance

The independent review should focus on:

* accidental behavior changes hidden inside moved code;
* broken import/re-export contracts;
* stale registry or architecture-doc symbol paths;
* `_meta.derived` provenance drift;
* unintentional broadening into helper-policy redesign;
* any coupling introduced between FK helpers and workbook-view/reimport logic.

A review should reject this FTR if it becomes an architectural redesign rather
than a mechanical module split.

==== Relationship to Future Work

This FTR may establish a pattern for later maintainability slices such as:

* splitting discriminator split/merge into a package;
* splitting tabular view transforms into pivot/unpivot/validation modules;
* extracting shared frame-selection, column-label, and meta-access helpers.

Those future refactorings should be evaluated separately after this slice has
proven that the package-split pattern is safe.
