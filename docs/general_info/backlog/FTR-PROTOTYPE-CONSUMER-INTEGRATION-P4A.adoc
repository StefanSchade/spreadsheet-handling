===== FTR-PROTOTYPE-CONSUMER-INTEGRATION-P4A -- Run a prototypical consumer workbook flow before beta

*Status:* Open
*Priority:* P4A
*Area:* Consumer integration / beta readiness

==== Purpose

Use a separate prototypical consumer repository as the first real integration
pass before the next beta.

The goal is not to turn consumer-specific configuration into a core library
fixture. The goal is to run the current library against a concrete usage path,
discover missing general capabilities early, and keep those gaps visible before
the demo repository and beta release are polished.

The dependency direction stays one-way: the consumer depends on the local
`spreadsheet-handling` checkout for the integration pass; the core repository
does not depend on the consumer repository.

==== Scope

* Bind the consumer workflow to the local `spreadsheet-handling` checkout for
  the integration pass.
* Identify the maintained workbook flow that should represent the beta
  readiness check.
* Run the selected flow end to end where local data and permissions allow it.
* Capture any missing generic library behavior as separate FTRs in this
  backlog.
* Keep consumer-specific changes, fixtures, and generated artifacts in the
  consumer repository.

==== Minimum Feature Set

The first useful consumer gate should not wait for every P4A gap. It needs at
least:

* `FTR-GENERIC-FRAME-EXTRACTIONS-P4A`
* `FTR-STANDARD-REFERENCE-VALIDATIONS-P4A`
* `FTR-DECLARATIVE-WORKBOOK-VIEWS-P4A`

If the selected flow still depends on consumer-local joins, helper styling, or
view readback, those needs should either pull the relevant FTR forward or be
documented as the next blocker.

==== Non-goals

* No private or consumer-owned data is copied into the main library repository.
* No consumer-specific assumptions are hard-coded into generic transforms.
* No broad demo rewrite is bundled into this FTR; the demo follows after this
  consumer pass.

==== Acceptance

* The selected prototype consumer flow can run against the local library
  checkout, or the blocker is documented with a concrete next FTR.
* Any main-library behavior needed by the consumer flow is either implemented
  as a general capability or recorded as a follow-up FTR.
* Scratch outputs, snapshots, and local run artifacts are ignored or removed
  rather than committed accidentally.
* The next demo-integration pass can start from the capabilities proven or
  newly queued here.

==== Pre-beta integration pass — 2026-05-17

_Self-documented integration audit; pending independent review (see

===== Selected consumer flow

The maintained workbook flow chosen to represent beta readiness is the
*configured workbook view reimport roundtrip* described in
`docs/internal_guide/concepts/by_use_case/business_calculation_workbook_roundtrip.adoc`
(a concept note explicitly _derived from a prototype consumer discussion_):

* forward: `contract_xref` -> `configure_workbook_view` (visible sheet renamed)
* spreadsheet write/read across XLSX and ODS
* simulated business edit on one matrix cell of the renamed sheet
* reverse: `apply_workbook_view_sheet_mappings` -> `expand_xref` back to the
  canonical relation frame

This is the runnable in-core representative of the prototype-consumer workbook
flow; its executable form is
`tests/integration/pipeline/test_configured_workbook_view_reimport_roundtrip.py`.

===== Local binding method

No vendoring or copying. The flow runs directly against the local
`spreadsheet-handling` checkout through its public pipeline API
(`spreadsheet_handling.pipeline.build_steps_from_config` / `run_pipeline`) and
the published console entrypoints (`sheets-run`, `sheets-example-*`). Local
checkout version: `0.1.0b6.dev131` (the pre-beta `0.1.0b6` dev line this FTR
gates). For a real external consumer repo the least-invasive binding would be a
local path / editable install of this same checkout.

===== Result: partial pass (in-core proxy) + documented environment blocker

* *In-core beta-readiness proxy: PASS.* The selected configured workbook view
  reimport roundtrip runs end to end against the local checkout across XLSX and
  ODS, including the renamed-sheet remap and the no-edit identity roundtrip. All
  three FTR-named validation targets pass (15 passed).
* *External prototype-consumer-repo binding/run: BLOCKED — environment/access.*
  No separate prototype consumer repository is present or accessible in this
  environment. Only `spreadsheet-handling` (core) and `spreadsheet-handling-demo`
  (explicitly out of scope for this FTR) exist locally; clone history shows no
  other consumer repo was ever provisioned on this VM. The prototype consumer
  was a *design discussion* that already produced the 16 derived FTRs, the
  `REVIEW-prototype-consumer-core-gaps-P4A.adoc` review, and the two
  `by_use_case` concept notes — not a checked-out repository here. There is
  therefore no external consumer command/test to invoke; the consumer-side
  acceptance cannot be physically exercised in this environment.

===== Blocker / gap classification

* External consumer repo absent -> *environment/access blocker* (not a library
  capability gap; no consumer repo provisioned in this VM).
* No *missing generic spreadsheet-handling capability* was discovered. The
  expected pre-beta capability set (generic frame extraction, reference
  validations, declarative workbook views, helper-column styling,
  editable/read-only protection, lookup helper formula mode, resource
  fallback/override semantics, key-value resource writer, generated artifact
  manifest, workbook reimport view mapping, configured workbook view reimport
  roundtrip, ODS parity disposition) is already implemented per the
  closed/archived P4A FTRs and the `FTR-ODS-CONSUMER-FEATURE-PARITY-P4A`
  parity matrix; the proxy flow exercises the core slice and passes.
* Payload conflict / precedence handling remains correctly out of scope and
  deferred to `FTR-WORKBOOK-VIEW-PAYLOAD-CONFLICT-PRECEDENCE-P6`.

===== Follow-up FTRs

None created. No generic library gap surfaced, so no concrete next FTR is
warranted; the blocker is purely the absence of a provisioned external consumer
repository in this environment.

===== Data-handling statement

No consumer data was copied into the core repository (none exists in this
environment to copy). No core `src/` or test code was changed — this is a
documentation-only integration audit. No scratch or generated artifacts were
staged: the proxy flow writes only to pytest `tmp_path`, and both working trees
remained clean.

===== Disposition

The acceptance criterion is satisfiable via its OR clause: the selected flow
runs against the local checkout (in-core proxy passes) *and* the external-repo
step is documented as an environment blocker with no generic next FTR required.
Lifecycle status stays *Open* (not Done): the controlled integration against a
real, separate consumer repository has not been physically performed here and
this record is self-documentation pending independent review. `_ftrs_open.adoc`
is intentionally left unchanged — the lifecycle status is unchanged and no new
follow-up FTR was created.
