===== FTR-CONDITIONAL-FOLLOW-UP-PROJECTION-FLOWS-P5 -- Materialize conditional detail views after workbook roundtrip

*Status:* New
*Priority:* P5
*Area:* Workbook projection / roundtrip workflows
*Kind:* Conceptual follow-up FTR

==== Purpose

Support workflows where an edited workbook matrix determines which additional
detail rows or follow-up views become relevant in a later workbook generation
cycle.

The motivating pattern is a business-facing matrix such as:

* products × markets
* products × countries
* products × deployment modes
* features × product variants

A user edits the matrix first. Certain cell values then imply that additional
detail information must be collected for the affected subset. For example, if
a product is provided as SaaS in Japan, the next workbook generation may need
one compliance-responsible contact per relevant component.

The core insight is that this should not be modeled as live spreadsheet UI
magic. The workbook is a static projection artifact. Conditional detail views
should be derived deterministically after reimport/recomposition, then exposed
in a subsequent generated workbook.

==== Context

The project already supports:

* declarative workbook views
* matrix/crosstable projection and recomposition
* explicit sheet-to-frame readback mapping
* helper/read-only column policy
* reference validations
* extraction, joins, and resource output steps

These capabilities make a two-phase workflow plausible:

. Generate and edit the main matrix workbook.
. Reimport/recompose the edited matrix into canonical relation frames.
. Apply explicit conditional rules to materialize required follow-up rows.
. Generate a second workbook projection that contains the relevant detail rows.

This FTR records the pattern and keeps it out of the current beta/demo scope
unless a concrete consumer flow later proves it is required.

==== Design Direction

The preferred model is a deterministic roundtrip-driven workflow:

[source,text]
----
canonical frames
  -> workbook matrix projection
  -> user edits matrix
  -> reimport/recompose
  -> explicit conditional follow-up rule materializes detail frames
  -> regenerated workbook contains relevant detail views
----

The system should not attempt to dynamically mutate workbook structure while
the user edits the spreadsheet. Spreadsheet formulas, helper columns, filters,
or validation hints may improve usability, but they must not become hidden
canonical semantics.

==== Scope

A minimal future slice should define a small, explicit transformation that:

* consumes canonical relation frames after workbook reimport;
* evaluates explicit configured predicates over those frames;
* materializes deterministic follow-up rows into one or more output frames;
* preserves stable identity columns for generated follow-up rows;
* allows regenerated workbook views to show only the relevant detail rows;
* records clear diagnostics for missing keys, duplicate identities, or invalid
  rule configuration.

A simple first implementation may be closer to filtered/extracted frames than
to a new projection framework.

==== Possible Configuration Sketch

[source,yaml]
----
step: materialize_conditional_followups
source: product_market_matrix_rows
output: japan_saas_component_contacts
identity:
  - product_id
  - market_id
  - component_id
when:
  market_id: JP
  deployment_mode: SaaS
expand_from: product_components
join_on:
  product_id: product_id
fields:
  - product_id
  - market_id
  - component_id
  - compliance_owner
defaults:
  compliance_owner: ""
----

This sketch is illustrative only. The actual configuration should be derived
from concrete use cases and existing transform conventions.

==== Non-Goals

* No live Excel/Calc UI mutation.
* No dynamic insertion/removal of workbook sheets while the workbook is open.
* No hidden synchronization framework.
* No heuristic inference of which follow-up rows are needed.
* No automatic merge/conflict resolution.
* No nested object/list support.
* No generalized workflow engine.
* No domain-specific compliance model in the core library.
* No reliance on protected/read-only cells as a security boundary.

==== Explicit Heuristic Boundary

Execution steps must consume explicit configuration and deterministic metadata.
They must not guess which follow-up rows to create or which business rule
applies.

If heuristic assistance is ever useful, it belongs in a separate opt-in
preprocessing or inference step that proposes or materializes deterministic
configuration. The actual execution step must remain deterministic and
configuration-driven.

==== Simpler Alternatives to Evaluate First

Before adding a new transform, evaluate whether the same user need can be met
with existing mechanisms:

* generate all potential detail rows and use helper columns plus autofilter to
  mark relevant rows;
* use lookup/helper formulas to show why a row is relevant;
* split relevant follow-up rows into a separate workbook view using existing
  extraction/join/filter transforms;
* require a second explicit generation pass after reimport rather than trying
  to make the first workbook self-adapting.

If these simpler approaches are sufficient, this FTR may remain a documented
pattern rather than a new runtime feature.

==== Acceptance

* A main matrix edit can deterministically cause follow-up detail rows to be
  materialized after reimport.
* The materialized rows are based only on explicit rules and canonical frames.
* Regenerating the workbook exposes the relevant follow-up detail view.
* Irrelevant detail rows are absent, filtered out, or clearly marked according
  to explicit configuration.
* Missing or ambiguous identities fail loudly.
* No live workbook UI magic or heuristic row creation is introduced.

==== Demo / ACME Disposition

For the next ACME demo integration, this pattern should be documented as a
future or exploratory capability unless it can be represented cleanly with
existing extraction/filter/helper mechanisms.

The demo may mention the business motivation, but it must not imply that
dynamic conditional workbook sections are already supported.

==== Relationship to Other FTRs

* Builds on workbook-view readback and recomposition.
* Related to, but distinct from, payload conflict/precedence handling.
* Related to future nested/list support only if follow-up rows later become
  owned embedded object collections.
* Does not change ODS/XLSX adapter semantics; any workbook projection remains
  backend-neutral unless a gap is explicitly documented.
