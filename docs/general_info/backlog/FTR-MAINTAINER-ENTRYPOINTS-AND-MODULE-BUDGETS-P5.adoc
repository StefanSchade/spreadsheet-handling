===== FTR-MAINTAINER-ENTRYPOINTS-AND-MODULE-BUDGETS-P5 -- Clarify change paths and module-size budgets

*Status:* Open
*Priority:* P5
*Area:* Internal quality / maintainer onboarding

==== Purpose

Make the project easier to extend without turning the codebase into a maze of
large modules and implicit change paths.

The runtime registry, architecture guards, and source-style guardrails already
protect several important drift points. External review feedback from
2026-05-08 nevertheless highlighted two practical maintainability questions:

* where should a maintainer start when changing a specific kind of behavior?
* how do we notice modules that are growing too large before they become hard
  to review?

This FTR captures the small, concrete hardening slice. It is not a broader
quality-tooling campaign.

==== Scope

* Add a concise internal "change entrypoints" document for common work types:
** adding or changing a pipeline step
** adding a domain transformation
** changing workbook/rendering behavior
** changing spreadsheet adapter behavior
** adding metadata namespaces
** adding validations
** updating user-facing YAML examples
* Include that document from the developer manual.
* Extend `tools/check_source_style.py` with module-size warning thresholds.
* Keep module-size findings warn-first and allowlistable, like the existing
  function-length and branch-complexity findings.
* Document the thresholds and the intended review response.

==== Non-goals

* Do not introduce SonarQube or another external dashboard in this FTR.
* Do not set a project-size limit.
* Do not fail builds merely because a module is large.
* Do not split modules mechanically without improving local cohesion.
* Do not replace the reviewed pipeline step registry or architecture guards.

==== Acceptance

. The developer manual contains an entrypoint map that points maintainers to
  the right source files, tests, and registry/docs artifacts for common change
  types.
. Source-style guardrails emit a warning for modules that exceed the agreed
  production/test module-size thresholds.
. Module-size warnings can be allowlisted with the existing documented
  rationale/owner/review-date policy.
. The source-style guardrail documentation explains that module size is a
  review trigger, not automatic proof that a file must be split.
. Existing source-style tests cover the new module-size warning behavior.

==== Implementation notes

Suggested starting thresholds:

* production module warning: 450 physical lines
* test module warning: 700 physical lines

The thresholds should be tuned after the current Phase-4 feature wave rather
than treated as permanent architecture law.
