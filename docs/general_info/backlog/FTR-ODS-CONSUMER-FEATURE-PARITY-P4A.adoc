===== FTR-ODS-CONSUMER-FEATURE-PARITY-P4A -- Track ODS parity for consumer-driven workbook features

*Status:* Open
*Priority:* P4A
*Area:* Spreadsheet adapters / parity
*Kind:* Cross-cutting tracking FTR

==== Purpose

Keep ODS/Calc support visible while consumer-driven workbook features are
added.

The ODS adapter is not just a secondary output format. It is also the practical
test that the adapter architecture carries the spreadsheet-neutral contract
rather than hard-coding XLSX assumptions into new features.

This is a tracking FTR, not a standalone feature. Each spreadsheet-facing FTR
in this group should document its ODS disposition. This FTR verifies that the
group has no implicit XLSX-only behavior.

==== Scope

For the pre-beta prototype-consumer-derived feature set, maintain a small
parity matrix and focused tests. The scope is split between post-wave items
already implemented (audit only, no rebuild) and feature work that lands
during the active P4A wave (parity must be covered while the feature lands).

===== Audit existing parity

The following capabilities are already implemented in the post-REVIEW-001 wave
state. The task here is to record their ODS disposition explicitly
(implemented, not applicable, or accepted gap), not to re-implement them:

* declarative workbook views (closed by `FTR-DECLARATIVE-WORKBOOK-VIEWS-P4A`)
* explicit helper-column styling (closed by `FTR-HELPER-COLUMN-STYLE-METADATA-P4A`)
* editable/read-only column protection (closed by `FTR-EDITABLE-COLUMNS-AND-PROTECTION-P4A`)
* lookup helper formula mode (closed by the formula-providers / formulas-core-move FTR chain)
* validation dropdowns from legend metadata, where existing ODS parity tests
  do not already cover the case
* column-width behavior, including accepted XLSX-only gaps

===== Coverage during the active wave

The following active P4A FTRs must each document an explicit ODS disposition
(implemented, not applicable, or accepted gap) in their own acceptance text;
this tracking FTR verifies that disposition is present:

* `FTR-LOCALIZED-RESOURCE-MATRIX-ROUNDTRIP-P4A`
* `FTR-RESOURCE-FALLBACK-OVERRIDE-SEMANTICS-P4A`
* `FTR-KEY-VALUE-RESOURCE-WRITER-P4A` (ODS disposition typically "not
  applicable" since the writer emits text resource files, not spreadsheets)
* `FTR-GENERATED-ARTIFACT-MANIFEST-P4A` (same: typically "not applicable")
* `FTR-WORKBOOK-REIMPORT-VIEW-MAPPING-P4A`
* `FTR-WORKBOOK-REIMPORT-DERIVED-COLUMN-POLICY-P4A`
* `FTR-WORKBOOK-VIEW-ROUNDTRIP-RECOMPOSITION-P4A`

===== Post-wave invariants to protect

* Backend dispatch now runs centrally through `io_backends/router.py`. New
  spreadsheet-facing features automatically reach both XLSX and ODS through
  the router; new code paths must not reintroduce parallel dispatch.
* The lazy-optional-import invariant for openpyxl and odfpy must be preserved.
  The regression guard `test_package_import_does_not_load_spreadsheet_backend_modules`
  stays green: bare `import spreadsheet_handling` must not load `openpyxl`,
  `odf`, or the `io_backends.xlsx` / `io_backends.ods` submodules.

==== Non-goals

* Do not force fake ODS parity where the backend model is materially different.
* Do not block a feature forever if the correct outcome is an explicit
  documented adapter gap.
* Do not duplicate every XLSX integration test one-to-one.

==== Acceptance

* Every prototype-consumer-derived pre-beta workbook feature has an explicit
  ODS disposition: implemented, not applicable, or accepted gap.
* At least one ODS integration or roundtrip test exercises the maintained
  declarative workbook path.
* Any accepted ODS gap is documented in the relevant FTR and in a compact
  adapter capability note.
* No new generic workbook feature lands with only implicit XLSX behavior.
