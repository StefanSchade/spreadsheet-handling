===== FTR-LOCALIZED-RESOURCE-MATRIX-ROUNDTRIP-P4A -- Generic discriminator/context matrix roundtrip for localized-resource use cases

*Status:* Open
*Priority:* P4A
*Area:* Domain transformations / generic tuple-frame and matrix-frame roundtrip

==== Purpose

Support the generic domain transformation pattern:

`canonical tuple frame -> discriminator-specific matrix frames -> edited matrix frames -> canonical tuple frame`

The first domain example is localized resources: labels, enum texts, validation
messages, error messages, and similar resource values whose identity is a
resource key plus locale plus optional business context.

The implementation, however, must not become language-specific. `locale` is
only the first concrete discriminator example. The underlying transformation
pattern is generic:

* row identity key(s)
* discriminator axis
* matrix column/context axis
* editable payload value

This same pattern must support other configured discriminator domains, such as:

* deployment mode: SaaS vs on-premise
* customer-specific labels
* product-specific messages
* module-, screen-, or context-specific resource values
* software variant-specific error messages

==== Domain Shapes

The generic canonical shape is:

[source,text]
----
canonical_values
----------------
<row identity key column(s)>
<discriminator column>
<matrix/context axis column>
<editable payload value column>
----

At domain level this is a *canonical tuple frame*: a long-form tabular frame
whose configured key columns identify each editable payload value.

The projected editable shape is a *matrix frame* or *crosstable frame*: row
identity key columns remain as columns, and configured matrix-axis values become
payload columns. Domain transformations operate on frames and metadata only.
Sheets and workbooks are projection/rendering concepts handled by workbook view
configuration, Render IR, and concrete backends.

The localized-resource example uses:

[source,text]
----
localized_resource_values
-------------------------
resource_key
locale
context_id
text
----

The first slice may use `locale` as the discriminator in fixtures and examples.
Production code must remain generic and must use configured column names. No
implementation should hard-code `locale`, `context_id`, `resource_key`, or
`text` except in tests and examples that specifically demonstrate the
localized-resource use case.

The first slice may assume that row keys, discriminator values, and matrix
column/context values are already available as frames or configured axes.

The transform should reuse existing primitives such as `split_by_discriminator`
and `contract_xref` / `expand_xref` through a configured composition.
`merge_by_discriminator` is the inverse recomposition primitive for
discriminator-specific frames. These primitives remain the semantic core.

Do not introduce a second locale-specific matrix framework. Any future wrapper
must be a thin generic composition wrapper over existing frame primitives, not
a localized-resource engine.

==== Generic Roundtrip Pattern

The intended domain transformation is:

* split the canonical tuple frame by configured discriminator into
  discriminator-specific frames
* contract each discriminator-specific tuple frame into a crosstable frame
  using configured row keys, configured matrix column/context axis, and
  configured payload value column
* allow edits only to configured matrix-frame payload cells
* expand each edited matrix frame back into long tuple frames using configured
  row keys and matrix-axis columns
* merge discriminator-specific tuple frames back into the configured canonical
  tuple frame

Optionally, a workbook view may project one or more matrix frames to workbook
sheets for editing. That projection must remain separate from the domain
transformation and must not imply that each discriminator value becomes a
sheet.

==== Goals

* Project canonical tuple frames into one or more discriminator-specific
  matrix/crosstable frames.
* Use configured discriminator split/merge mechanics when discriminator values
  such as locale, deployment mode, customer, product, module, screen, or
  variant are represented in frame names. Workbook view configuration may later
  project selected frames to sheets.
* Use context columns such as `default`, product ids, module ids, screen ids,
  deployment modes, or customer ids as the crosstable axis.
* Expand edited matrix frames back into canonical tuple frames.
* Preserve identity from configured keys and axes, not visible labels or sheet
  name guessing.
* Support helper/context columns for descriptions and resource types.
* Integrate with standard key/reference/uniqueness validations.

==== Non-goals

* Do not judge text quality or translation correctness.
* Do not implement a full i18n framework.
* Do not implement a locale-specific matrix framework.
* Do not hard-code localized-resource column names in production code.
* Do not make domain transformations depend on workbook sheets, XLSX, or ODS
  rendering behavior.
* Do not infer fallback behavior from empty cells; use the fallback/override
  semantics FTR for that.
* Do not write target `.properties` files in this FTR.

The first implementation slice may avoid ambiguous empty-cell cases by using
complete matrix fixtures or by treating empty cells mechanically. Policy-driven
omission, explicit empty text, and default/override collapse belong to
`FTR-RESOURCE-FALLBACK-OVERRIDE-SEMANTICS-P4A`.

==== Acceptance

* A canonical tuple frame can be transformed into discriminator-specific
  matrix/crosstable frames.
* The localized-resource fixture may use `resource_key`, `locale`,
  `context_id`, and `text`, but the production implementation uses configured
  column names for row identity, discriminator, matrix/context axis, and
  payload value.
* Row keys, discriminator values, and matrix/context axes are supplied
  explicitly as frames or configured axes in the first slice.
* Edited matrix frames can be recomposed into canonical tuple frames with
  stable configured identity. In the localized-resource example this identity
  is `(resource_key, locale, context_id)`.
* Tests cover at least two locales and two non-default contexts.
* Helper columns are ignored or checked according to explicit metadata, not
  silently merged as text payload.
* Existing primitives such as `split_by_discriminator`,
  `merge_by_discriminator`, `contract_xref`, and `expand_xref` remain the
  semantic core of the implementation.
* Optional workbook view projection to sheets is explicit and does not change
  the domain transformation contract.
* XLSX and ODS dispositions are explicit at the rendering/readback boundary.
