== Module split clean?

Yes. The single 261-line `registry.py` is decomposed into four single-responsibility modules:

* `pipeline/registry.py` (146 lines) — `REGISTRY` table + `resolve_registration`.
  No build logic, no execution loop, no logger.
* `pipeline/build.py` (87 lines, new) — `build_steps_from_config`,
  `build_steps_from_yaml`, `_ensure_string_parameter_keys`.
* `pipeline/execution.py` (22 lines, new) — `run_pipeline(frames, steps)` only.
  The 5-line execution loop now stands alone with its own logger.
* `pipeline/runner.py` (45 lines) — exposes `run_app(AppConfig)` as the application
  entry; imports from `build` and `execution`, not `registry`.

The orchestrator's import was updated from
`from ..pipeline.registry import run_pipeline` to
`from ..pipeline.execution import run_pipeline`. The frame-level
`run_pipeline(frames, steps)` is now properly owned by `execution.py`, and the
old runner-shadow `run_pipeline(AppConfig)` is gone.

=== Minor

*1. `run_app` does not forward `header_levels` to the loader.*

Slice01 standardized loaders to accept a keyword-only `header_levels: int = 1`.
The orchestrator's `_load_frames` passes `header_levels=header_levels` from
its public parameter. `run_app` calls `loader(inp.path, options=...)` without
forwarding it, so `AppConfig`-driven workflows always read at one header row.
This matches the old `runner.run_pipeline(AppConfig)` behaviour (no regression),
but the asymmetry between the two entry points is worth either fixing or
documenting:

* Cheapest: add `header_levels: int = 1` to `IOEndpoint` (default 1, kept off the
  YAML schema for now) and forward it in `run_app`.
* Alternative: document in `AppConfig` / `run_app` that multi-header input is
  only reachable via `orchestrate(...)`.

Not blocking; mirrors prior behavior.

*2. `run_app` does not wrap `ValueError` like the orchestrator does.*

`_load_frames` in `orchestrator.py` wraps router `ValueError("Unknown loader
kind: X")` into `ValueError("Unsupported input kind: 'X'")` for better
diagnostics. `run_app` calls `get_loader(inp.kind)` directly, so a typo in
`AppConfig.io.inputs[*].kind` surfaces as the lower-level message without the
AppConfig context. One-liner polish:

[source,python]
----
try:
    loader = get_loader(inp.kind)
except ValueError as exc:
    raise ValueError(f"Unsupported input kind: {inp.kind!r}") from exc
----

(same for `get_saver`). Trivial; not blocking.

*3. Legacy YAML dialect rejection is sharp but undocumented at the surface.*

The new `load_app_config` raises `ValueError` if it sees the old
`pipeline: {steps: [...]}` dialect. The error message is good. The
slice02 FTR's Implementation Result notes the change but there's no
end-user-facing release note / CHANGELOG entry; for pre-beta this is acceptable,
but if any consumer outside the repo still ships the old dialect, they'll see
the ValueError without a migration hint pointing at the canonical form.

Cheapest fix: extend the error message to show a 2-line example of the new form:

[source,python]
----
raise ValueError(
    "YAML key 'pipeline' must be a step list using the canonical step: "
    "dialect. Example:\n  pipeline:\n    - step: identity"
)
----

Not blocking; current message already identifies the symptom and the dialect name.
