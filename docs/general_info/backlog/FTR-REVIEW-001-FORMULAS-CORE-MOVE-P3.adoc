===== FTR-REVIEW-001-FORMULAS-CORE-MOVE-P3 — Move `rendering/formulas.py` to `core/` and close hexagonal guard gap L1

*Status:* Done
*Priority:* P3
*Area:* core / rendering / domain
*Kind:* Refactoring / Architecture
*Review source:* REVIEW-001, Cluster C1, Finding IDs: D-001, IR-002, D-014

==== Purpose

`rendering/formulas.py` contains purely backend-neutral value objects
(`ListLiteralFormulaSpec`, `LookupFormulaSpec`, and their factories).
Because it lives in the `rendering/` layer, `domain/transformations/enrich_lookup.py`
and `domain/transformations/fk_helpers.py` must cross the hexagonal boundary to import
it — the only real layer violation in the codebase. Moving the file to `core/formulas.py`
eliminates the violation and allows a guard to enforce the `domain ⊥ rendering` boundary
going forward.

==== Current State (verified)

* `rendering/formulas.py` (72 lines) — defines `ListLiteralFormulaSpec`,
  `LookupFormulaSpec`, `FormulaSpec` (type alias), `make_list_literal_formula_spec`,
  `make_lookup_formula_spec`. Purely data; no rendering or backend logic. (IR-002)
* `domain/transformations/enrich_lookup.py` — imports `from rendering.formulas import
  lookup_formula`; direct `domain → rendering` import with no architecture guard catching
  it (gap L1). (D-001)
* `domain/transformations/fk_helpers.py` — imports from `rendering.formulas` for the
  same reason. (D-001)
* Five `rendering/` modules and four `io_backends/` modules also import from
  `rendering.formulas` (verify with `grep -r 'from rendering.formulas' src/`).
* `domain/transformations/enrich_lookup._build_formula_enrichment` — sets
  `missing = "" if missing_mode == "empty" else ""` — both branches identical;
  `missing_mode` is a no-op in formula mode. (D-014)
* Architecture gap L1: no guard enforces `domain ⊥ rendering` or
  `domain ⊥ io_backends` today.

==== Scope

. Move `rendering/formulas.py` → `core/formulas.py` (use `git mv` to preserve history).
. Update all import sites — change `from rendering.formulas import ...` →
  `from core.formulas import ...`:
  * `domain/transformations/enrich_lookup.py`
  * `domain/transformations/fk_helpers.py`
  * All `rendering/` importers (confirm with `grep -r 'from rendering.formulas' src/`)
  * All `io_backends/` importers (same grep)
. Fix `enrich_lookup._build_formula_enrichment`: make the two `missing_mode` branches
  distinct — e.g. raise `ValueError` for `"fail"`, return `""` for `"empty"` as an
  intentional documented choice. Remove the dead duplicate assignment.
. Add architecture guard `test_domain_modules_do_not_import_rendering_or_io_backends`
  in `tests/architecture/` — assert no module under
  `src/spreadsheet_handling/domain/` or `src/spreadsheet_handling/core/` imports
  from `rendering` or `io_backends`.
. Add the new guard to `docs/internal_guide/architecture/architecture_guard_inventory.adoc`.
. Run full test suite.

==== Non-Goals

* Do not change the public API of `FormulaSpec`, `ListLiteralFormulaSpec`, or
  `LookupFormulaSpec`.
* Do not move any other `rendering/` module in this FTR.
* Do not fix other guard gaps (L2–L8) — those belong to FTR-GUARDRAIL-AUDIT-P5.

==== Dependencies

None. This FTR is the prerequisite for the L1 guard; the guard would start red without
the move. Execute before any refactor that expects the guard to be green.

==== Risk

Low mechanical risk — the move is a file rename; the type objects carry no side effects.
The import-site count (~11 files) is bounded and fully revealed by `grep` before touching
any file. Any missed import site is caught by the test suite.

==== Acceptance

* `grep -r 'from rendering.formulas' src/` returns no matches.
* `grep -r 'spreadsheet_handling.core.formulas' src/` returns ≥11 matches (all
  updated import sites).
* Guard `test_domain_modules_do_not_import_rendering_or_io_backends` exists in
  `tests/architecture/` and passes.
* `enrich_lookup._build_formula_enrichment`: the two `missing_mode` branches are
  distinct and non-redundant.
* `architecture_guard_inventory.adoc` lists the new guard.
* Full test suite passes (`pytest`).

==== Implementation Notes

* `FormulaSpec`, `ListLiteralFormulaSpec`, `LookupFormulaSpec`, and their factories now
  live in `spreadsheet_handling.core.formulas`.
* The old `spreadsheet_handling.rendering.formulas` module was removed rather than kept
  as a compatibility alias, so the L1 guard can enforce `domain/` and `core/` staying
  independent from `rendering/` and `io_backends/`.
* Formula meaning and adapter translation semantics were not changed.
