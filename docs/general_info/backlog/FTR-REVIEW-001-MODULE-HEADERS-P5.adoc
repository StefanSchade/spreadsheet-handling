===== FTR-REVIEW-001-MODULE-HEADERS-P5 — Add Purpose/Layer/Imports header to every `*.py`

*Status:* Open
*Priority:* P5
*Area:* domain / core / rendering / pipeline
*Kind:* Hygiene
*Review source:* REVIEW-001, Cluster C8, Finding IDs: D-012, IR-014

==== Purpose

Most modules in `domain/`, `core/`, `rendering/`, and `pipeline/` have no module-level
docstring. `head -5` on any of these files gives no orientation for a developer or
agent. A lightweight header convention — 1–3 lines stating Purpose, Layer, and allowed
project-internal imports — costs little to write and makes every module self-describing
without reading its body. This is the highest-leverage token-efficiency improvement for
agent-assisted development: `head -5` replaces a full-file read for orientation.

==== Current State (verified)

* `domain/` — most modules have no docstring; a few have one-line summaries of varying
  quality. (D-012)
* `core/` — same gap. (D-012)
* `rendering/flow.py`, `rendering/plan.py`, `rendering/ir.py`, `rendering/config.py`,
  `cli/apps/run.py` — no docstring. (IR-014)
* `pipeline/flow.py`, `pipeline/plan.py`, `pipeline/ir.py`, `pipeline/config.py` — no
  docstring. (IR-014)
* `docs/ai_info/conventions.adoc` — no rule about module headers or module-level
  docstrings. (D-012, IR-014)

==== Scope

. Define the convention in `docs/ai_info/conventions.adoc` (under `=== Python`,
  new sub-section "Module headers"):
  * Every `*.py` starts with a module docstring of 1–3 lines.
  * Line 1: Purpose — one sentence describing what the module does.
  * Line 2: Layer — one of `core`, `domain`, `rendering`, `pipeline`, `io_backends`,
    `cli`.
  * Line 3 (optional): `May import:` — comma-separated allowed project-internal
    packages (derive from the layer map in `architecture_primer.adoc`).
. Add headers to all modules in `domain/`, `core/`, `rendering/`, `pipeline/`. For
  `cli/` modules, add at minimum a one-line purpose docstring.
. (Optional) Extend `tools/check_source_style.py` with a warning (not a failure) for
  any `*.py` in the four source layers that has no module docstring. Wire it into the
  existing warn-and-allowlist mechanism.

==== Non-Goals

* Do not require headers in test files.
* Do not add headers to auto-generated or vendored files.
* Do not fail CI for missing headers — warning mode only if the optional guard is added.
* Do not retroactively add headers to `io_backends/` adapter modules in this FTR;
  include them only if the effort fits.

==== Dependencies

None. Fully independent; can be done in parallel with any other cluster.

==== Risk

Negligible — module docstrings cannot break runtime behaviour. The optional source-style
guard is warn-only and allowlistable.

==== Acceptance

* `docs/ai_info/conventions.adoc` contains a "Module headers" subsection under
  `=== Python`.
* `python -c "import ast, pathlib; [ast.parse(p.read_text()) for p in
  pathlib.Path('src/spreadsheet_handling').rglob('*.py')]"` succeeds (no syntax errors
  introduced).
* `grep -rL '"""' src/spreadsheet_handling/domain/
  src/spreadsheet_handling/core/
  src/spreadsheet_handling/rendering/
  src/spreadsheet_handling/pipeline/` returns no `.py` files (all have a docstring).
* Full test suite passes.
