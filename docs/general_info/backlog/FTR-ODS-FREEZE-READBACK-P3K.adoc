===== FTR-ODS-FREEZE-READBACK-P3K — Reconstruct freeze hint from ODS settings on the read path

*Depends-on:* Phase 3j completion

*Context:* The Phase-3j parity tests explicitly encode freeze readback as an accepted gap: XLSX reconstructs a `__freeze` hint during parsing, but the ODS parser does not. The renderer already writes freeze configuration into `settings.xml` via `odf.config`, so the write side works. The missing piece is purely on the read path.

*Problem statement:* Without freeze readback, the ODS adapter remains asymmetric on this specific semantic: it can write freeze but cannot observe it when reading back. This is the single largest accepted gap in the current parity suite. Closing it would strengthen the cross-adapter parity claim and remove the most visible format-level difference.

*Goals:*

. Parse freeze configuration from ODS `settings.xml` view-state settings.
. Expose the result as a `__freeze` hint on the read path, consistent with how the XLSX parser already does it.
. Convert the current accepted-gap parity test into a positive parity assertion.

*Acceptance criteria:*

. The ODS parser reconstructs `__freeze` hints from settings.xml when freeze configuration is present.
. The parity test `test_freeze_parse_hint_difference_is_an_explicit_accepted_gap` is replaced by or updated to a positive parity check.
. The freeze hint follows the same shape as the XLSX parser (`{"row": N, "col": M}`).
. Workbooks without freeze settings produce no spurious `__freeze` hint.

*Non-goals:*

- No changes to the write-side freeze implementation.
- No expansion of freeze semantics beyond the existing hint-based pattern.
- No change to the canonical `freeze_header` meta treatment.

*Why now:* This is the most visible parity gap remaining from Phase 3j. Closing it before broader feature work removes a known asymmetry and simplifies future parity reasoning.
