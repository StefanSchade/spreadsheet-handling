===== FTR-IO-BACKEND-KIND-USER-INTERFACE-SMOOTHING-P5 — Smooth backend kind user-interface surfaces

*Status:* Open
*Priority:* P5
*Area:* io_backends / application / CLI
*Kind:* User-surface polish / Architecture guard
*Source:* `FTR-REVIEW-001-BACKEND-DISPATCH-P4A_review_slice01.adoc`, minor findings 2 and 3

==== Purpose

The backend-dispatch slice centralized dispatch ownership in
`io_backends/router.py`, but it intentionally preserved the existing kind
vocabulary split:

* `make_backend(...)` accepts backend-family names such as `csv`, `json`,
  `xml`, `xlsx`, `excel`, `ods`, and `calc`.
* `get_loader(...)` / `get_saver(...)` accept file- or directory-oriented names
  such as `csv_dir`, `json_dir`, `yaml_dir`, `xml_dir`, `xlsx`, and `ods`.

That split is technically valid, but it is a user-interface rough edge: CLI
users, config authors, tests, and Python callers see similar words that do not
always work in the same places. This FTR captures the later smoothing step
instead of expanding the backend-dispatch slice.

==== Scope

. Decide and document the intended backend kind vocabulary for user-facing
  surfaces (`sheets-run`, YAML config, and documented Python helpers).
. Align error messages and help text so unsupported aliases explain the right
  accepted surface.
. Either normalize harmless aliases across loader/saver/factory surfaces or
  document why a surface remains intentionally narrower.
. Add an architecture guard that keeps `application/orchestrator.py` routed
  through `io_backends/router.py` instead of importing backend implementation
  modules directly.
. Add focused tests for the chosen kind vocabulary and the application-to-router
  boundary.

==== Non-Goals

* Do not add a new backend framework or registry abstraction.
* Do not change backend read/write semantics.
* Do not broaden the Python public API without an explicit surface decision.
* Do not revisit the pipeline registry / runner cleanup from
  `FTR-REVIEW-001-BACKEND-DISPATCH-P4A-SLICE02`.

==== Acceptance

* The supported backend kind vocabulary is documented for user-facing surfaces.
* CLI/help/config examples do not advertise misleading kind names.
* Focused tests cover accepted and rejected aliases for the chosen surfaces.
* An architecture guard prevents `application/orchestrator.py` from importing
  concrete backend implementation modules directly.
* Full test suite passes.
