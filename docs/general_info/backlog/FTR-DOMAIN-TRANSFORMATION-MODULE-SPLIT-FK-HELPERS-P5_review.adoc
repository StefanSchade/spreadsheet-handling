Date: 2026-05-18

Status: *Independent review completed: ACCEPT WITH FOLLOW-UP.* The original
body below remains the implementation self-documentation for this
behavior-preserving refactoring slice. The independent review verdict accepted

== Scope of original implementation pass

Mechanical, behavior-preserving split of the single
`src/spreadsheet_handling/domain/transformations/fk_helpers.py` module into a
package. Single repo (`spreadsheet-handling`, branch `dev`); demo repo used for
verification only and left unmodified.

== What changed (core repo)

* `fk_helpers.py` -> `fk_helpers/` package: `__init__.py`, `enrich.py`,
  `drop.py`, `policy.py`, `provenance.py`, `formula_provider.py`. Function
  bodies moved verbatim; only module-relative import depth changed (`...` ->
  `....`) and responsibility-scoped imports added between the new submodules.
* `__init__.py` re-exports `enrich_helpers`, `drop_helpers`,
  `_write_helper_provenance`. The first two are the public registry/test
  surface; `_write_helper_provenance` is re-exported because
  `meta_registry.yaml` and `test_meta_registry.py` reference it as the
  `derived`-channel producer at the package-level dotted path.
* Internal dependency DAG (verified acyclic by successful import + tests):
  `enrich -> {policy, provenance, formula_provider}`, `drop -> provenance`.
  `_visible_label` co-located in `provenance.py` (shared by provenance cleanup
  and `drop_helpers`); `drop` imports it from `provenance` -- single
  direction, no cycle. No broad shared utility package introduced.
* New guard test
  `tests/architecture/dependency_guards/test_fk_helpers_package_reexports.py`
  asserts the public + documented symbol paths keep resolving through the
  package (covers the part `test_meta_registry.py` does not: that the dotted
  path actually resolves, not just that the YAML string exists).
* Doc pointers updated: `docs/ai_info/concepts_fk_helpers.adoc` and
  `docs/ai_info/architecture_primer.adoc` (`fk_helpers.py` -> package).

== What deliberately did NOT change

* No edits to `pipeline/registry.py` or `pipeline/steps.py`: their
  `..domain.transformations.fk_helpers` imports resolve unchanged through the
  package `__init__`.
* `meta_registry.yaml` and `pipeline_step_registry.json` unchanged. The
  documented symbol paths (`domain.transformations.fk_helpers.enrich_helpers`,
  `.drop_helpers`, `._write_helper_provenance`) are NOT "old" paths -- they
  still resolve as package attributes and remain accurate, so the FTR's
  "update only if it references old paths" condition does not trigger. Changing
  them to submodule paths would have broken the existing meta-registry contract
  test for no behavioral gain.
* No pipeline step IDs, YAML config syntax, helper column names/ordering, FK
  policy resolution, formula/value-mode behavior, `_meta.derived`
  structure/lifecycle, or cleanup behavior touched. No new FK functionality.
  No compatibility aliases added, no public symbols removed.
* No non-FK transform families touched (`discriminator_split`,
  `tabular_views`, `xref`, `resources`, `workbook_views`,
  `derived_column_policy`, etc.). `derived_column_policy._visible_label` was
  confirmed to be its own local function, not an import from this module.
* `domain/validations/fk_helpers.py` is a separate module, out of scope, left
  untouched.
