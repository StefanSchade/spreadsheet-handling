===== FTR-BACKEND-ABC — Formal backend interface + Table protocol

*Status:* Superseded in practice by <<FTR-SPREADSHEET-BACKEND-CONTRACT-P3H#,FTR-SPREADSHEET-BACKEND-CONTRACT-P3H>> and related Phase 3h work

*Context:*::
This proposal predates the Phase 3h spreadsheet-backend generalization work. It aimed to formalize
adapter contracts across the active backend set by tightening `BackendBase` and introducing a
minimal table protocol.

After reassessment, this is no longer the preferred entry point for adapter-interface work. The more
urgent architectural need is not a broad abstract base class for all backends, but a clearer
spreadsheet-specific contract around `WorkbookIR`, `RenderPlan`, parsers, renderers, and the narrow
adapter-facing API. That concern is now addressed more directly in Phase 3h.

*Decision:*::
Keep this ticket only for traceability.
Its original scope should not be implemented as written.
If a later phase still needs broader non-spreadsheet backend interface cleanup, it should be defined
anew after the spreadsheet architecture and drift-protection work in Phases 3h/3i have settled.

*Original intent (retained for traceability):*::

- Convert `BackendBase` to `ABC` with `@abstractmethod read()/write()` and default `read_multi()/write_multi()` loops.
- Introduce a minimal `Table` Protocol (pandas DataFrame conforms).
- Wrap YAML/CSV helpers into `YAMLBackend` / `CSVBackend` thin classes.

*Why superseded:*::

- The adapter contract question turned out to be most urgent in the spreadsheet path, not uniformly across all backends.
- Phase 3h now treats spreadsheet adapters as a first-class architectural boundary instead of forcing one broad backend abstraction too early.
- Prematurely freezing a universal backend ABC would risk locking in the wrong seam before spreadsheet-specific contracts and invariants are clarified.
