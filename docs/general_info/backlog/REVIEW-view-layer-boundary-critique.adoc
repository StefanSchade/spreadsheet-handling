= Stellungnahme: Abgrenzung SIMPLE-JOIN-VIEWS / DECLARATIVE-WORKBOOK-VIEWS / TABULAR-VIEW-OPS
:author: Review Session 2026-05-07
:toc:
:status: Eingearbeitet

== Anlass

Externe Kritik an der Grenzziehung zwischen den drei view-bezogenen FTRs.
Kernfrage: Kann der Workbook-View-Layer versteckt Daten-Joins ausführen und
damit eine zweite Transformationssprache im Rendering-Pfad etablieren?

'''

== Status nach Einarbeitung

Die Kritik wurde im Rahmen von `FTR-DECLARATIVE-TABULAR-VIEW-OPS-P4A`
eingearbeitet:

* `FTR-SIMPLE-JOIN-VIEWS-P4A` enthält nun eine explizite Ownership Boundary
  für Join-Semantik.
* `FTR-DECLARATIVE-WORKBOOK-VIEWS-P4A` enthält nun ein Composition Principle:
  Workbook Views konsumieren Frames und erzeugen keine neuen Datenrelationen.
  Die Umsetzung über `configure_workbook_view` schreibt nur
  `_meta.workbook_view`, `_meta.frame_lifecycle`, und sheet-nahe Renderoptionen;
  die Auswahl/Renennung läuft über `select_render_frames()`.
* `FTR-DECLARATIVE-TABULAR-VIEW-OPS-P4A` enthält nun eine Layer Boundary Note
  und ist umgesetzt: `pivot_frame` materialisiert ein benanntes, inspizierbares
  View-Frame vor dem Rendering.

Dieses Dokument bleibt als Review-/Decision-Record erhalten. Es beschreibt die
Grenzziehung, die spätere View-FTRs und Reviews respektieren sollen.

'''

== Bewertung: Der Kritik wird zugestimmt -- mit Nuancen

Die Kritik ist **im Kern richtig**, aber die aktuellen FTR-Texte (Stand nach
Codex-Überarbeitung) enthalten die geforderte Grenzziehung bereits
**weitgehend**. Die Situation ist besser als die Kritik vermutet, aber eine
explizite Schärfung hat sich gelohnt.

'''

== Ist-Zustand der FTRs (Post-Codex-Überarbeitung)

[cols="1,3"]
|===
| FTR | Aktuelle Boundary-Sprache

| *DECLARATIVE-WORKBOOK-VIEWS*
| Non-Goals: "Do not implement column projection, row filtering, sorting,
joins, or pivots; those belong to frame extraction and tabular view
preparation steps." +
Goals: "Allow prepared view frames from `extract_frame`, tabular view
operations, or existing transforms to be exposed as sheets." +
Acceptance: "The implementation uses or extends the existing `workbook_view`
policy and `select_render_frames()` path rather than bypassing them."

| *SIMPLE-JOIN-VIEWS*
| Purpose: "This FTR is narrower than a query engine: it should only cover
explicit, deterministic joins." +
Explicit delegation from TABULAR-VIEW-OPS: "FTR-SIMPLE-JOIN-VIEWS-P4A owns
explicit joins, especially semi-join / existence-filter behavior."

| *TABULAR-VIEW-OPS*
| Purpose: "This is the transformation half of declarative workbook views.
Workbook view composition decides which frames become visible sheets. Tabular
view operations produce the frames that are worth showing." +
Goals: "Delegate explicit join semantics to FTR-SIMPLE-JOIN-VIEWS-P4A."
|===

**Fazit:** Die dreigliedrige Architektur ist bereits richtig angelegt:

----
canonical frames
  → extract_frame (atomic projection/filter/rename/sort)
  → tabular_view_ops (bounded pivot, display builders)
  → simple_join_views (explicit joins, semi-joins, filter-joins)
  → workbook_views (Frame→Sheet Zuordnung, Sheet-Name, Order, Visibility)
  → render (XLSX/ODS output)
  → reimport (reverse mapping)
----

'''

== Was fehlte: Explizite Ownership-Klausel

Die Kritik hat recht, dass die _Ownership-Richtung_ nicht scharf genug
formuliert war. Die folgenden Ergänzungen wurden als Zielbild übernommen:

=== Vorschlag 1: In FTR-SIMPLE-JOIN-VIEWS-P4A

Ergänzung unter Goals oder als eigener Abschnitt "Ownership":

[source,asciidoc]
----
==== Ownership Boundary

This FTR owns join semantics and produces materialized, named output frames.
Workbook-view steps may reference these frames as visible sheets but must not
redefine, inline, or duplicate join semantics. If a workbook view needs a
joined frame, the join must be declared as a preceding pipeline step that
produces an explicit output frame.
----

=== Vorschlag 2: In FTR-DECLARATIVE-WORKBOOK-VIEWS-P4A

Ergänzung unter Non-Goals oder Goals:

[source,asciidoc]
----
==== Composition Principle

Workbook views are consumers of frames, not producers of new data relations.
They compose already-available frames into visible workbook sheets and attach
view/lifecycle/render metadata. Non-trivial data reshaping — joins, pivots,
aggregations, relation-changing transformations — must happen in preceding
pipeline steps (`extract_frame`, tabular view operations, or join views) that
produce explicit named frames before workbook-view materialization.

A workbook view step may perform visibility-near projection: sheet selection,
sheet naming, sheet ordering, column subset selection for display, column
label overrides, editability/protection markers, and lifecycle annotations.
It must not perform semantic data joins or row-producing operations.
----

=== Vorschlag 3: In FTR-DECLARATIVE-TABULAR-VIEW-OPS-P4A

Der bestehende Text ist bereits gut. Optional ergänzen:

[source,asciidoc]
----
==== Layer Boundary Note

Tabular view operations produce frames. Workbook views consume frames.
This FTR sits between frame-level extraction and workbook-level materialization.
Its output is always a named, inspectable frame — not a hidden intermediate
in the workbook-view rendering path.
----

'''

== Zur Frage "Inline-Join als Syntactic Sugar"

Die Kritik erlaubt optional: "Falls Inline-Join in Workbook Views trotzdem
gewünscht ist, dann nur als syntactic sugar, das intern auf denselben
Simple-Join-View step normalisiert wird."

**Empfehlung: Nein, nicht jetzt.**

Begründung:

1. Syntactic Sugar, das intern auf einen anderen Step normalisiert, ist
   architektonisch sauber, aber es erzeugt _Pipeline-Config-Ambiguität_:
   Derselbe Join kann an zwei Stellen konfiguriert werden.
2. Das erschwert Reimport-Mapping: Wenn ein Join im Workbook-View-Step
   steckt, muss der Reimport-Path wissen, dass der Sheet-Name nicht
   direkt auf einen Source-Frame mappt sondern auf einen synthetischen
   Join-Frame.
3. Die bestehende Registry-Konvention (ein Step, ein Konzept) spricht
   dagegen.
4. Wenn der Bedarf später entsteht, kann ein Komposit-Step eingeführt
   werden, der _beide_ FTRs zitiert. Das ist aber Post-Beta-Arbeit.

'''

== Zur Beziehung TABULAR-VIEW-OPS ↔ SIMPLE-JOIN-VIEWS

Die Kritik fragt auch implizit: "Ist TABULAR-VIEW-OPS ein Überbau über
SIMPLE-JOIN-VIEWS?"

**Antwort: Nein, sie sind orthogonal.**

* TABULAR-VIEW-OPS: Bounded Pivot, Display-Builder (Index→Columns→Values).
  Input: ein Frame. Output: ein Frame.
* SIMPLE-JOIN-VIEWS: Explicit Left/Inner/Semi-Join. Input: zwei Frames.
  Output: ein Frame.

Keine der beiden Operationen subsumiert die andere. `add_lookup_helpers`
ist die dritte Alternative (Key-basierte Helper-Column-Enrichment, nicht
als View-Frame sondern als Column-Addition).

'''

== Zusammenfassung

[cols="1,3"]

| Punkt | Bewertung

| Kritik berechtigt?
| Ja, im Kern. Die Boundary ist _implizit_ richtig angelegt, aber nicht

| Änderungsbedarf?
| Moderate Ergänzungen in allen drei FTRs (Ownership/Composition/Layer

| Risiko ohne Änderung?
| Gering bis mittel. Der aktuelle Non-Goals-Text in WORKBOOK-VIEWS ist

  wäre ein expliziter Ownership-Absatz hilfreicher.

| Inline-Join als Sugar?
| Empfehlung: Nicht in dieser Iteration. Post-Beta evaluieren.
|===

'''

== Erledigte Aktion

Codex hat die drei oben beschriebenen Ergänzungen in die FTRs eingearbeitet:

1. SIMPLE-JOIN-VIEWS: "Ownership Boundary" Abschnitt.
2. DECLARATIVE-WORKBOOK-VIEWS: "Composition Principle" Abschnitt.
3. TABULAR-VIEW-OPS: "Layer Boundary Note" Abschnitt.

Das war eine reine Dokumentationsschärfung, keine Scope-Änderung. Die
FTR-Semantik bleibt identisch: Transformationen erzeugen Frames, Workbook
Views konsumieren Frames, und Rendering führt keine versteckten Daten-Joins aus.
