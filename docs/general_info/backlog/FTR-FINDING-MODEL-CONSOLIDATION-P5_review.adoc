== Scope of original implementation pass

Behavior-preserving consolidation of the duplicated finding-frame machinery
flagged as D-005 (REVIEW-002 Pass 1) and escalated to finding-model
quadruplication (REVIEW-002 Pass 2 §4.1). Single repo
(`spreadsheet-handling`, branch `dev`); demo repo not touched.

No validation redesign, no diagnostics engine, no pipeline step-name change, no
YAML/config change, no severity-behavior change, no `reference_validations.py`
package split, no unrelated cleanup.

== What changed (core repo)

* New `src/spreadsheet_handling/domain/finding_frame.py`. Domain-level
  placement deliberately mirrors `domain/frame_lifecycle.py` (already imported
  the same way by transformations) so the new shared module needs no new guard
  family and trips none of the existing ones. It owns: the canonical
  seven-column `FINDING_COLUMNS`; a frozen `Finding` dataclass with
  `to_record`; `row_index_label`; `findings_to_frame(findings, *,
  columns=None)` generic over any `to_record()`-bearing record with a
  schema-stable empty frame; `simple_failure_message(findings, *, heading)`.
* `domain/transformations/resource_overrides.py`: removed the local
  `FINDING_COLUMNS`, `ResourceOverrideFinding` dataclass, `findings_to_frame`,
  `row_index_label`, `_failure_message`. `ResourceOverrideFinding` retained as
  a module-level alias of the shared `Finding` (public-name stability; every
  construction site already passes `severity=` explicitly so the old
  class-level `"fail"` default never affected output). Failure path now calls
  `simple_failure_message(..., heading="Resource override normalization
  failed")`; serialization calls the shared `findings_to_frame(...,
  columns=FINDING_COLUMNS)`.
* `domain/transformations/derived_column_policy.py`: same shape — local
  `FINDING_COLUMNS`/`DerivedColumnFinding`/`findings_to_frame`/
  `_row_index_label`/`_failure_message` removed; `DerivedColumnFinding` alias
  retained; `FINDING_COLUMNS` re-exported (its unit test imports it);
  `simple_failure_message(..., heading="Derived column policy failed")`.
* `domain/validations/reference_validations.py`: `findings_to_frame` now
  delegates to the shared serializer with the module's own nine-column
  `FINDING_COLUMNS`. `ReferenceFinding`, that column list, `to_record`,
  `_failure_message`, `_finding_line`, `_format_*` are kept verbatim — this is
  a genuinely different diagnostic shape and was intentionally NOT collapsed.
* New `tests/unit/domain/test_finding_frame.py` (FTR-marked): canonical
  columns, scalar/tuple `row_index_label`, `to_record` rendering, blank
  `row_index`/`value`, default and custom-column serialization, schema-stable
  empty frames (both schemas), and the flat failure message.

== What deliberately did NOT change

* `domain/validations/findings.py::Finding` + `apply_severity_policy`,
  `resolve_severity`, `SeverityPolicy`. This is a category→severity-policy
  model consumed by `pipeline/steps.py` and `domain/validations/fk_helpers.py`;
  it has no `FINDING_COLUMNS`/`to_record`/`findings_to_frame`. REVIEW-002
  Pass 2 §4.1 lists it among "four finding models", but Pass 1 D-005 frames it
  precisely as the *other* model. It has semantically necessary local fields
  and no tabular serialization to consolidate; merging it would be exactly the
  "vague record" / severity-behavior change the FTR forbids. Left untouched
  (it carries pre-existing ruff findings unrelated to this FTR — not cleaned,
  per the no-unrelated-cleanup boundary).
* `ReferenceFinding` semantics: raw (non-stringified) `row_index`,
  JSON/scalar `value` formatting, the `target_frame`/`target_columns`
  columns, and the capped + `name`-parameterised failure message with
  `_finding_line`. Distinct from the resource-override / derived-column flat
  uncapped `"<heading>:\n  - <rule_type>: <message>"` shape. An existing
  reference-validations test asserts integer `row_index` values
  (`[2, 0, 1]`), which routing through the shared labelled renderer would have
  silently changed — concrete evidence the shapes are genuinely different.
* `graph_validations.py` (out of FTR scope): unchanged; its
  `from ...reference_validations import ReferenceFinding, findings_to_frame`
  still resolves (both names still defined there). Its own
  `_failure_message`/`_finding_line` left as-is.
* `domain/validations/__init__.py` surface (`FINDING_COLUMNS`,
  `ReferenceFinding`, `findings_to_frame`, `validate_references`,
  `validate_graph`): unchanged and still resolves.
* No pipeline/registry edits: REVIEW-002 confirmed registry coupling is
  string colon-target only; no static import edges into these modules. The
  finding dataclasses are not registry targets.
* Category strings, severity/mode behavior, emitted findings, empty-frame
  schemas, and user-visible failure text: all preserved.

== Surface note (intentional, importer-verified)

`resource_overrides.row_index_label` and `resource_overrides.FINDING_COLUMNS`
were verified to have no importers anywhere in `src/` or `tests/`
(grep-checked). `FINDING_COLUMNS` is still exposed there (re-export, passed
explicitly at the serialization call site). `row_index_label` was not
re-exported from `resource_overrides`; its canonical home is now
`domain.finding_frame.row_index_label`. This is a zero-importer surface
relocation, not an observed-behavior change; flagged here for the independent
reviewer rather than hidden.

== Reviewer focus (per FTR Review Guidance)

* Accidental diagnostic-shape drift: diff each removed `to_record` against the
  shared `Finding.to_record` — resource-override and derived-column rendering
  were byte-identical pre-consolidation (`", ".join` columns, labelled
  `row_index`, `" | ".join` value); confirm.
* Category / severity drift: confirm no category string changed and that every
  `*Finding(...)` construction passes `severity=` explicitly (so the alias's
  shared default is inert).
* Over-generalization: confirm `finding_frame.py` is a small record + two
  helpers, not an engine; confirm `simple_failure_message` was *not* grown with
  cap/name options to absorb the reference-validations message (kept separate
  on purpose).
* Flattening check: confirm `ReferenceFinding` and `findings.py::Finding` were
  correctly judged genuinely different and left intact, and that the
  integer-`row_index` reference test still pins that distinction.
* Duplication-reduction reality: confirm finding-frame serialization is now
  single-sourced and that this unblocks a later `reference_validations.py`
  split (the D-005 prerequisite called out in REVIEW-002 §2.2 / §6).
