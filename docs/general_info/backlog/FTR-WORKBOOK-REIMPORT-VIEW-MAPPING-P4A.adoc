===== FTR-WORKBOOK-REIMPORT-VIEW-MAPPING-P4A -- Map edited sheets back to logical frames

*Status:* Open
*Priority:* P4A
*Area:* Pipeline / readback semantics

==== Purpose

Add the first reverse-view slice for workbook readback: map visible sheet names
and workbook views back to logical frame names without guessing from labels.

Rendering already uses `_meta.workbook_view` and `_meta.frame_lifecycle` to
decide which frames become visible. Reimport needs the inverse lookup: which
visible sheet corresponds to which logical view frame, and which canonical
source frame that view may patch or recompose.

Note: `FTR-LOCALIZED-RESOURCE-MATRIX-ROUNDTRIP-P4A` closed with explicit frame
names during reimport. Generic renamed-sheet/frame mapping remains owned here.

==== Prerequisites

* `FTR-DECLARATIVE-WORKBOOK-VIEWS-P4A` — *Done* (archived in
  `docs/cold_storage/backlog/historic_tickets/ftrs/`).
* existing `select_render_frames()` / `workbook_view` policy
* existing `frame_lifecycle` metadata

==== Meta-Carrier disposition

The sheet-to-frame mapping is roundtrip-relevant: a workbook rendered with a
given view configuration must be reimportable into the same logical frames
without consulting the original rendering session. The mapping therefore lives
in the *canonical* `_meta.workbook_view` channel, not in the transient
`_meta.derived` channel.

Concretely, this FTR adds (or extends) a sub-key under `_meta.workbook_view`
that records, per visible sheet, which logical view frame it represents and
optionally which canonical source frame the view recomposes into. The exact
sub-key name is implementation-detail of this FTR, but its registration in
`meta_registry.yaml` is part of the acceptance.

This boundary is deliberate. The registered `_meta.derived` channel covers
helper-column provenance and other transient operational data that is removed
by `drop_helpers` before output. Reverse view mapping is a persistent contract
between render and reimport and must survive roundtrips.

==== Scope

* Persist a deterministic sheet-to-frame mapping under `_meta.workbook_view`
  during workbook composition.
* If a new sub-key is introduced under `_meta.workbook_view` for this mapping,
  register it in `docs/internal_guide/architecture/meta_registry.yaml` with
  `producer` and `consumer` references.
* Read that mapping during reimport.
* Fail clearly when a required visible sheet is missing or duplicated.
* Keep physical sheet names separate from logical frame names.
* Support renamed sheets through explicit view metadata.

==== Non-goals

* Do not expand matrices back to relations in this slice.
* Do not drop helper columns in this slice.
* Do not infer canonical frame names from arbitrary sheet names.
* Do not implement conflict resolution.

==== Acceptance

* A rendered workbook carries enough metadata to map each configured visible
  sheet back to a logical view frame.
* The mapping is persisted in the canonical `_meta.workbook_view` channel and
  survives a render/reimport roundtrip.
* If a new sub-key is introduced under `_meta.workbook_view`, it appears in
  `meta_registry.yaml` with non-empty `producer` and `consumer` references.
* A reimport pipeline can select workbook sheets by configured view identity,
  not by hard-coded physical sheet names.
* Missing, duplicated, or unknown required sheets produce clear diagnostics.
* Tests cover at least one renamed sheet and one omitted intermediate frame.
* XLSX and ODS readback dispositions are explicit.
