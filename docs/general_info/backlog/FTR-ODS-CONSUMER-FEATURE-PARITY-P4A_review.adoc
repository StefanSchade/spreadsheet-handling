== Architectural Assessment

No production code changed and no new tests were added. This is deliberate:

* The *implemented* dispositions are each already proven by an existing
  ODS+XLSX test (see the matrix evidence column). Duplicating those would
  violate the FTR non-goal "do not duplicate every XLSX integration test
  one-to-one".
* The *not applicable* items (resource fallback/override, key-value writer,
  artifact manifest, derived-column policy slice 1) are backend-neutral by
  construction: they are domain/in-pipeline or text/JSON-writer steps with no
  `WorkbookIR` / `RenderPlan` / spreadsheet-parse path. Forcing an ODS parity
  test there would be fake parity, which is an explicit hard boundary.
* The *accepted gaps* reflect materially different backend models and are
  documented rather than hidden or simulated.

ODS parity here functioned as an architecture pressure test: the audit found
the post-REVIEW-001 router centralization already routes new spreadsheet
features through both backends, so no XLSX-only drift was discovered.
