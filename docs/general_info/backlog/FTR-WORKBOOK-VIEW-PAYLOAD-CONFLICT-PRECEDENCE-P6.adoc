===== FTR-WORKBOOK-VIEW-PAYLOAD-CONFLICT-PRECEDENCE-P6 -- Detect overlapping editable view payload conflicts under explicit precedence

*Status:* New
*Priority:* P6
*Area:* Pipeline / roundtrip semantics
*Prerequisites:* FTR-WORKBOOK-VIEW-ROUNDTRIP-RECOMPOSITION-P4A

==== Purpose

Split out the conflict/precedence concern that was originally mentioned in
`FTR-WORKBOOK-VIEW-ROUNDTRIP-RECOMPOSITION-P4A`. It is intentionally deferred
out of the beta-critical recomposition path and revisited only when
overlapping editable views create real payload-conflict pain (the same trigger
as `FTR-JSON-SHARDING-BY-PARTITION`).

When two or more editable workbook views recompose into the same canonical
relation, they can produce different values for the same canonical tuple. The
recomposition path proven by `FTR-WORKBOOK-VIEW-ROUNDTRIP-RECOMPOSITION-P4A`
does not detect or resolve this; a duplicated/conflicting tuple currently only
surfaces as a generic primary-key validation failure after merge. This FTR
makes that conflict explicit and gives the operator an explicit, configured
precedence contract.

==== Scope

* Detect overlapping editable view payloads that produce different values for
  the same canonical tuple, across two or more recomposed source frames.
* Support an explicit, configuration-driven precedence policy that
  deterministically selects the winning source for a conflicting tuple.
* Emit clear findings in the existing validation-style findings shape, and
  fail loudly by default when a conflict is found without a configured
  precedence policy.
* Keep the conflict/precedence step inspectable and runnable through the
  existing pipeline config/build/run machinery and `sheets-run`.

==== Critical architectural principle

Execution steps must not use heuristics to decide winners.

* No heuristic conflict resolution inside execution steps.
* No hidden merge logic. A conflict is never silently collapsed; resolution
  happens only through an explicit precedence contract.
* No generic reimport framework. This is a single, narrow detection/precedence
  step composed from existing pipeline machinery.
* No broad synchronization engine and no multi-user merge reconciliation.
* Precedence must be explicit and configuration-driven. The configured
  precedence is an ordered, named list of contributing sources; an unknown or
  unresolved source is a clear error, not a guess.
* Optional future heuristic assistance may only *generate or propose*
  deterministic configuration in a separate opt-in preprocessing/inference
  step. Such a step must materialize an explicit precedence contract for
  review; it must never silently affect execution. Execution/recomposition
  steps consume explicit contracts/configuration only.

==== Non-goals

* Do not require automatic conflict resolution.
* Do not infer canonical targets, winners, or precedence from sheet names,
  order, labels, columns, or any heuristic.
* Do not trust protected/read-only cells as a security or precedence boundary.
* Do not own helper-column mismatch policy; that remains with
  `FTR-WORKBOOK-REIMPORT-DERIVED-COLUMN-POLICY-P4A`.
* Do not implement multi-user Git-style merge reconciliation.

==== Acceptance

* Two editable views recomposing into the same canonical relation with
  differing values for the same tuple are detected.
* Without a configured precedence policy the conflict fails loudly with clear,
  validation-shaped findings naming the conflicting tuple(s) and sources.
* With an explicit precedence policy the winning value is selected
  deterministically by configured source order; an unknown/unresolved source
  is a clear error.
* Focused tests cover: detected conflict without precedence (fail), resolved
  conflict with precedence (deterministic winner), and the non-conflicting
  overlap case (no false positive).
* No heuristic winner selection exists anywhere in the execution path; any
  heuristic assistance, if added later, only proposes configuration.
