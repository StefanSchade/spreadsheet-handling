===== [[FTR-CLI-STEPS-DEPRECATION-P3G]] FTR-CLI-STEPS-DEPRECATION-P3G — Deprecate `--steps` in product messaging while keeping compatibility

*Status:* Open
*Epic:* Phase 3g — Refactoring after Demo Integration

*Context:*::
`steps.yaml` was the first direct configuration path for `sheets-run`.
Later, `--config` plus `--pipeline` added a higher-level abstraction for maintained,
named workflows. Removing `--steps` entirely was evaluated in
<<FTR-CLI-CONSOLIDATE-REMOVE-STEPS#,FTR-CLI-CONSOLIDATE-REMOVE-STEPS>> and rejected:
the removal would reduce real ad-hoc functionality while only modestly simplifying the CLI path.

The actual product concern is narrower. The public CLI narrative has become harder to explain because
multiple entry paths are documented with similar weight, even though the preferred user guidance is now:

* semantic shims for common workflows, or
* `sheets-run --config ... --pipeline <name>` for maintained reusable flows.

That concern can be addressed without removing backward-compatible support for `--steps`.

*Goal:*::
Keep `--steps` functional for compatibility and power-user scenarios, while making it clear that it is
no longer the preferred documented path for new usage.

*Rationale:*::
* Preserves the original direct configuration mechanism for ad-hoc usage.
* Avoids breaking existing consumers for limited technical simplification.
* Achieves most of the desired UX consolidation through messaging rather than capability removal.
* Keeps the migration pressure low while still signaling preferred CLI usage going forward.

*Solution:*::
. Emit a deprecation warning when `sheets-run --steps ...` is used.
. Emit a deprecation warning when the deprecated alias `--pipeline-yaml` is used.
. Keep the underlying behavior unchanged for now.
. Update CLI help and user documentation so that `--config` plus optional `--pipeline` is the primary
  generic path described to users.
. Reduce or remove prominent user-guide walkthroughs that center `--steps` as the normal starting point.
. Keep compatibility notes for `--steps` available where needed for migration or advanced use.

*Non-Goals:*::
* No removal of `--steps` in this feature.
* No removal of `--pipeline-yaml` in this feature unless a later dedicated removal FTR is approved.
* No redesign of the pipeline engine or step factory mechanism.
* No forced migration of all existing callers in the same change set.

*Acceptance:*::
* `sheets-run --steps ...` still works.
* `sheets-run --steps ...` surfaces a clear deprecation warning.
* `sheets-run --pipeline-yaml ...` surfaces a clear deprecation warning.
* Primary user-facing docs no longer present `--steps` as the preferred generic workflow entry point.
* The backlog records that outright removal was considered and rejected for now.

*Follow-up Considerations:*::
* Reassess full removal only if compatibility usage drops enough that the remaining path becomes dead weight.
* If desired later, make `--pipeline-yaml` fail earlier than `--steps`, since it is the older alias of the older path.
