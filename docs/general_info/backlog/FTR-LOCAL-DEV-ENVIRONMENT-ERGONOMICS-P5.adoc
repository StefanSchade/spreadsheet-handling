===== FTR-LOCAL-DEV-ENVIRONMENT-ERGONOMICS-P5 -- Make local development commands predictable

*Status:* Open
*Priority:* P5
*Area:* Internal quality / developer ergonomics

==== Purpose

Reduce repeated local friction for maintainers and coding agents without
hard-coding one developer's workstation into the project.

The project is effectively Linux-first today: CI, dependency tooling, and the
reliable local verification path are aligned with a POSIX environment
(`make`, shell scripts, Python tooling).

The current maintainer setup has shifted: instead of optimizing mainly for
local WSL workflows, the primary development path is now a dedicated headless
Linux VM (Debian) on Proxmox, accessed from a laptop via SSH in VS Code. The
goal is to give coding agents (Codex and Claude) a stable, low-friction,
Linux-native workspace and avoid repeated environment drift.

This FTR captures a narrow post-beta hardening slice: make that remote Linux
workflow explicit, stable, and easy for humans and agents to follow, while
keeping fallback guidance pragmatic.

==== Scope

* Document the repository's execution preference:
** primary: dedicated remote Linux dev VM accessed via SSH (VS Code Remote SSH
   or equivalent)
** secondary: local Linux environments
** tertiary: WSL on Windows when needed
** native Windows without WSL as best-effort only (not a primary target)
* Add stable developer entrypoints for common checks, such as:
** changed-file formatting and lint checks
** focused FTR/unit verification
** source-style guardrails
** architecture guardrails
** full local confidence run
* Prefer `make` targets or small checked-in helper scripts over long ad-hoc
  shell command strings.
* Keep commands deterministic enough that an approving tool can recognize
  recurring safe operations without granting arbitrary shell access.
* Add guidance for Git operations:
** sequence staging, committing, and status checks instead of firing them in
   parallel
** avoid background commands that keep `.git/index.lock` alive unexpectedly
** treat lock files as a symptom to inspect, not as files to delete by default
* Add a short "agent workflow" note to the internal developer docs that assumes
  a Linux shell baseline and remote SSH development as the default maintainer
  path.
* Update existing maintainer-entrypoint documentation if it overlaps with this
  FTR.

==== Non-goals

* Do not require WSL for all contributors.
* Do not remove Windows support from the library or from adapter behavior.
* Do not encode absolute local paths, usernames, hostnames, or company-specific
  workstation details.
* Do not introduce a new task runner if existing `make` plus small Python
  helpers is sufficient.
* Do not turn local convenience checks into release gates without separate
  review.
* Do not broaden tool approvals to arbitrary shell execution.
* Do not spend disproportionate effort on full native-Windows parity for local
  maintainer workflows in this slice.

==== Acceptance

. The developer documentation states the intended execution order with remote
  Linux via SSH as primary, then local Linux/WSL fallback guidance.
. A maintainer or coding agent can run the normal focused verification path via
  one stable command or a very small set of stable commands.
. The stable commands cover changed Python formatting/linting, focused tests,
  source-style checks, and architecture checks at the appropriate confidence
  level.
. The documented workflow explains when to run focused checks versus broader
  checks.
. Git workflow guidance explicitly discourages parallel Git operations and
  explains how to respond to an index lock without deleting it reflexively.
. The implementation avoids machine-local assumptions and remains useful for a
  fresh Linux clone, including a remote SSH-hosted Linux clone.

==== Implementation notes

Good candidate entrypoints:

* `make verify-changed`
* `make verify-ftr`
* `make verify-architecture`
* `make verify-style`
* `make verify-local`

The exact target names can be adjusted during implementation, but the result
should favor stable command shapes over long one-off shell invocations.

If a helper script is added, it should detect the current platform and print a
clear diagnostic when the preferred environment is unavailable. On Windows, the
script may recommend WSL or a remote SSH Linux workspace, but it should not
silently switch execution contexts in a surprising way.

The maintainer baseline for this FTR is Linux shell semantics over SSH to the
dedicated Debian VM. Native Windows execution without WSL is explicitly
best-effort and may remain incomplete as long as core maintainability commands
depend on `make` and shell scripts.
