===== FTR-XREF-DENSE-AXES-P4A2 Review: Minimal XRef Dense-Axis Slice

*Reviewed Commit:* `8ce0c0d`
*Review Date:* 2026-05-14
*Verdict:* ACCEPT (no blocking issues)

==== Summary

The minimal FTR-XREF-DENSE-AXES-P4A2 slice implements dense-axis reconstruction for `contract_xref` and `expand_xref` transforms. The implementation is architecturally sound, correctly addresses the core use case (sparse relations → dense matrices), and cleanly defers composite-transform integration. No blocking issues identified.

==== Architectural Assessment

===== Generic XRef Shape Materialization ✅

The implementation is purely structural: frame/key/value terminology only, no locale/context/resource-specific semantics. Dense axes configuration uses plain mappings (`rows_from`/`columns_from`) referencing external source frames by name and key column(s). No domain assumptions embedded.

===== No Localized-Resource Semantics ✅

The feature is fully generic. Tests use `resource_key`/`context_id` as example labels but the code makes no assumptions about their meaning. Axis source keys are normalized to plain Python types (`_plain_axis_value`), ensuring consistent identity matching without semantic interpretation.

===== Sparse/Non-Dense Behavior Preserved ✅

When `dense_axes` parameter is absent, all dense logic is bypassed. Original behavior (`_ordered_row_identities` from relation only) remains unchanged. Test at line 452 confirms sparse relations without dense_axes produce no `dense_axes` metadata entry.

===== No Second Matrix Framework ✅

Dense-axis reconstruction works within the existing `contract_xref`/`expand_xref` primitives. No new transformation layer, helper, or abstraction introduced. Row/column filling logic (lines 166-174) is a minimal extension of the existing cell-fill loop.

===== Compact_multiaxis Remains Cleanly Deferred ✅

`compact_multiaxis.py` is unchanged. The `dense_axes` parameter remains stubbed with `_reject_dense_axes()` raising NotImplementedError. FTR documents this deferral (line 27, 93-94 of FTR file). No inconsistency created between primitive and composite layers.

==== Correctness Assessment

===== Dense Row/Column Reconstruction from Explicit Sources ✅

* `rows_from`/`columns_from` are required for dense reconstruction; no heuristic inference from labels, positions, or names.
* Arity validation (lines 491-495): axis source key count must match `row_keys`/`column_keys` count.
* Multi-column keys are supported: line 501 constructs multi-element tuples correctly.
* Test coverage includes single-column (lines 260-294) and combined rows+columns (lines 298-330).

===== Sparse and Empty Relations Handled ✅

* Empty relation (no tuples): produces full Cartesian product of configured axes with all cells filled (test lines 334-358).
* Sparse relation (partial tuples): existing tuples are placed in correct cells; missing combinations are filled (test lines 260-294).
* Default `fill_value=""` applied only to missing cells, not confused with empty-cell semantics or fallback behavior.

===== Validation for Duplicate/Missing/Unknown Identities ✅

Error handling is clear and actionable:

* Duplicate axis-source keys: rejected with identity in message (lines 508-511, 544-546).
* Empty keys in axis source: rejected with frame/row number (lines 681-691).
* Missing axis source frame: rejected with frame name (lines 488, 528).
* Unknown rows in relation (not in axis source): rejected with list of offending identities (lines 591-594).
* Unknown columns in relation: rejected with list of offending values (lines 610-614).

All errors include frame name and context for user debugging.

===== Metadata Roundtrip Sufficient ✅

`_dense_axes_meta_payload` (lines 694-731) writes:

* Dense axes configuration (`rows_from`/`columns_from` mappings)
* Resolved axis identities (actual row/column values extracted from sources)

Test at lines 412-448 proves reimport can reconstruct full shape without source frames present, using only stored metadata. This enables deterministic roundtrips across workbook reads/writes.

===== Expand → Contract Roundtrip ✅

Test at lines 362-408 proves the full cycle:
1. Sparse relation (1 of 4 possible tuples)
2. Contract to dense matrix (2×2 = 4 cells, 3 empty)
3. Expand to relation (preserves empty cells because `drop_empty=false` and metadata present)
4. Contract again to dense matrix (full shape restored)

==== Non-Blocking Concerns

===== Multi-Column Dense Row Keys Not Yet Tested

Code supports `rows_from: {keys: ["col1", "col2"]}` syntax (lines 450-462), with proper arity validation (lines 491-495) and multi-column tuple construction (line 501). However, no test covers this scenario. The implementation is correct, but regression-test coverage is incomplete.

*Mitigation:* Add a focused test for multi-column row_keys before broader follow-up work.

===== Column_keys + Columns_from Precedence Wording

Lines 341-348 enforce that explicit `column_keys` must match `columns_from` order when both configured. Error message "must match...order when both are configured" is technically correct but could be clearer about precedence semantics.

*Impact:* Very low; error is actionable and users will rarely configure both explicitly.

===== Metadata Fallback/Reuse Documentation

Functions `_stored_row_identities` and `_stored_column_keys` (lines 631-650, 653-667) accept pre-resolved metadata on reimport without validating it against source frames. This is intentional (fallback when source unavailable) but the design intent could be documented in future.

*Impact:* None; behavior is correct and tested. Documentation is deferred improvement only.

The implementation is ready for production. Architecture is sound, correctness is proven, and deferred work is clearly marked.

A single test for multi-column dense row_keys will close the coverage gap identified above and prevent future regressions:

This test is defensive (code already works) but provides regression protection and documents the multi-column row_keys capability for future maintainers.
