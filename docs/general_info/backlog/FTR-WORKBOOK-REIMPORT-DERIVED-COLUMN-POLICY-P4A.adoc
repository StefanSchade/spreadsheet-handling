===== FTR-WORKBOOK-REIMPORT-DERIVED-COLUMN-POLICY-P4A -- Drop or check derived columns on reimport

*Status:* Implemented (minimal slice 1)
*Priority:* P4A
*Area:* Pipeline / readback semantics

==== Purpose

Make reimport behavior explicit for helper, derived, and read-only columns in
business-facing workbook views.

Generated workbooks often show context columns next to editable data. During
reimport those columns should not silently patch canonical data unless they are
explicitly configured as editable. The first robust policy is to drop known
helper columns from payload recomposition and optionally check that their
edited values still match canonical lookup sources.

Note: `FTR-LOCALIZED-RESOURCE-MATRIX-ROUNDTRIP-P4A` excluded helper columns by
explicit payload extraction in its integration slice. General helper/edit
policy and mismatch findings remain owned here.

==== Prerequisites

* `FTR-WORKBOOK-REIMPORT-VIEW-MAPPING-P4A` — *open*.
* `FTR-HELPER-COLUMN-STYLE-METADATA-P4A` — *Done* (archived in
  `docs/cold_storage/backlog/historic_tickets/ftrs/`).
* `FTR-EDITABLE-COLUMNS-AND-PROTECTION-P4A` — *Done* (archived in
  `docs/cold_storage/backlog/historic_tickets/ftrs/`).

==== Policy Modes

* `drop`: ignore derived/helper columns during payload extraction.
* `warn_on_mismatch`: compare derived values to canonical sources and write
  findings when they differ.
* `fail_on_mismatch`: compare derived values to canonical sources and fail the
  pipeline when they differ.

No edited helper value wins silently in this slice.

These policy-mode names are intentionally policy-specific. They map internally
onto the standard `fail` / `warn` / `ignore` validation convention
(`drop` ≈ `ignore` for value comparison, `warn_on_mismatch` ≈ `warn`,
`fail_on_mismatch` ≈ `fail`) but the public step keeps the reimport-policy
vocabulary because "drop" also denotes the always-applied column removal, which
the standard names do not express.

==== Slice 1 — Implemented Scope and Boundaries

Implemented as the pipeline step `apply_derived_column_policy`
(`spreadsheet_handling.domain.transformations.derived_column_policy`).

* Slice 1 consumes `_meta.derived` **in-memory, before `drop_helpers`
  cleanup**, within a single pipeline run. It does check + drop in one step
  and is intended to run ahead of `drop_helpers` on the reimport path.
* Slice 1 does **not** solve durable file→frame reimport classification.
  Reading derived identity back from a rendered/parsed workbook is not in
  scope here.
* Persisted workbook metadata, sheet→frame view mapping, and the
  derived-provenance lifecycle in rendered workbooks remain **deferred**.
* The current transient classification of `_meta.derived` (per
  `meta_registry.yaml`: removed by `drop_helpers`, not persisted to workbook
  output) is described here as the **current-code boundary for this slice**,
  not as a final architectural claim that derived provenance can never be
  persisted through workbook metadata. A later FTR may revisit
  derived-provenance persistence and lifecycle.
* Helper identity is taken **only** from `_meta.derived.sheets[<frame>]`
  (`helper_columns[*].column` and `enrich_lookup.helper_columns[*]`); no
  column-name heuristics.
* `enrich_lookup` helper columns are owned by this policy. They are not
  underscore-prefixed and are not covered by `drop_helpers`' prefix-based
  cleanup, so this step is genuinely additive for them.
* Value comparison (`warn_on_mismatch` / `fail_on_mismatch`) covers
  **`enrich_lookup` helpers only**, using the self-contained `on` join keys
  recorded in provenance. FK `helper_columns` are dropped but **not**
  value-checked in slice 1, because FK provenance
  (`{column, fk_column, target, value_field}`) does not record the target
  key column required to resolve the canonical value without heuristics.
* `frame_lifecycle` / `workbook_view` editability fusion is **not** part of
  slice 1.

==== Scope

* Read helper-column identity from the registered `_meta.derived` channel:
  ** `_meta.derived.sheets.<frame>.helper_columns` (FK helper provenance)
  ** `_meta.derived.sheets.<frame>.enrich_lookup` (lookup-helper provenance,
     including its own `helper_columns` subfield).
  Do not infer helper identity from column-name heuristics.
* Combine that with explicit editability metadata from `frame_lifecycle` and
  `workbook_view` to distinguish payload columns from derived display columns.
* Drop helper columns from canonical payload frames.
* Optionally check helper values against their source lookup frame.
* Produce validation-style findings for helper/read-only mismatches.
* Keep protected cells as an editing hint, not as a trust boundary.
* Lifecycle ordering (decided): the policy reads `_meta.derived` *before*
  `drop_helpers` runs and performs the column drop itself. If it runs after
  `drop_helpers` (channel already cleaned), it is a documented no-op for
  identity/value purposes. Reading derived identity from a persisted workbook
  `_meta` payload at reimport time is deferred (see Slice 1 boundaries).

==== Non-goals

* Do not implement automatic conflict resolution.
* Do not patch canonical lookup tables from helper columns.
* Do not require spreadsheet protection to be present before reimport.
* Do not detect multi-user Git merge conflicts.

==== Acceptance

* A configured reimport path can produce a payload frame without helper columns.
* Helper-column identity comes from the registered `_meta.derived` sub-keys
  (`helper_columns`, `enrich_lookup.helper_columns`); no column-name heuristics
  are used.
* Mismatched helper/read-only values are ignored, warned, or failed according
  to explicit policy.
* Findings use the same report shape as standard validations where practical.
* The lifecycle ordering between `_meta.derived` consumption and `drop_helpers`
  is documented and tested.
* Tests cover unchanged helper columns, edited helper columns in warn/fail
  modes, and payload extraction from a mixed editable/helper sheet.
* XLSX and ODS readback dispositions are explicit: slice 1 is an in-pipeline
  frame-level step and does **not** parse XLSX/ODS. File-level readback and
  sheet→frame mapping are owned by `FTR-WORKBOOK-REIMPORT-VIEW-MAPPING-P4A`.
  Because `_meta.derived` is currently transient and not persisted to workbook
  output, durable file→frame derived classification is out of scope here.

==== Deferred Concerns

* FK-helper value comparison (slice 1 drops FK helpers but does not check
  their values; provenance lacks the target key column).
* `frame_lifecycle` / `workbook_view` editability fusion with derived
  identity.
* Durable file→frame reimport and persisted workbook `_meta` consumption
  (owned by `FTR-WORKBOOK-REIMPORT-VIEW-MAPPING-P4A`).
* A later FTR should examine durable derived-column provenance in rendered
  workbooks, including orphaned/stale provenance after sheet or column
  edits/renames, and the derived-provenance persistence/lifecycle question
  more broadly.
