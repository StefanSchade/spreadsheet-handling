=== ADR-RENDER-FLOW-VS-DOMAIN-PIPELINE: Terminologie & Trennung — Domain „Pipeline“ vs. IR „Render-Flow“

:date: 2025-10-28
:deciders: dr_stefan_schade@yahoo.com

==== Context
The term “pipeline” is established for the domain transformation flow. The IR/XLSX path also used “pipeline”, which risks ambiguity across layers. A unifying abstraction was considered but the responsibilities (inputs/outputs, invariants, error models, performance) differ substantially.

==== Decision
Use **“Pipeline”** exclusively for the **Domain** layer, and **“Render-Flow”** for the **IR/Rendering** layer. Do not introduce an abstraction that unifies both.

Renames (IR layer):
- Functions: `compose` → `compose_ir`, `run_passes` → `apply_ir_passes`, `render` → `render_xlsx`
- Files: `rendering/pipeline.py` → `rendering/flow.py`; `rendering/passes/p1_core.py` → `rendering/passes/core.py`
- Add a glossary distinction in documentation.

==== Alternatives Considered
. Single abstract “Pipeline” interface across layers — *Rejected.* Would create leaky abstractions and coupling.
. Keep “pipeline” in both layers — *Rejected.* Leads to confusion in PRs, onboarding, docs.

==== Rationale
Clear terminology reduces review friction and aids debugging (stack traces reflect the layer). It keeps domain evolution decoupled from rendering and vice versa.

==== Consequences
- Small refactors (names/imports) without runtime behavior changes.
- Docs and roadmap wording updated (use “Render-Flow”).
