=== ADR-001 — Hexagonal Architecture (Pragmatic Adoption)

**Status:** Accepted

**Context**::
* The `spreadsheet-handling` library should remain modular while keeping overall complexity manageable.
* A classic hexagonal (ports-and-adapters) architecture cleanly separates domain, infrastructure, and I/O layers.
* Python, however is not suitable for a literal adoption:
  ** lack of strict typing
  ** no formal interfaces, no dependency injection containers
  ** no concept of private access

**Decision**::
* Adopt the *principle* of the hexagonal architecture with a great deal of pragmatism:
  ** No mapping between layers:  (Our home brew `meta` and `Panda` 's `DataFrame` type are used outside the "domain" layer)
  ** Minimal ports: In the absence of formal interfaces or DI containers our adapters only implement a tiny surface ('read_multi', 'write_multi').
  ** No type safety: We use a simple dictionary for 'meta' instead of a schema
  ** Strong coupling between domain and adapter layer: 'meta' is encoded by the domain and interpreted by the adapter.

* Each adapter exposes a pragmatic, minimal interface (`read_multi`, `write_multi`).
* Focus on exchangeability and testability, not on formal interface purity.

**Consequences**::
* Structural clarity and replaceability are preserved.
* Complexity remains low compared with a fully abstract implementation.
* Some coupling (e.g., use of Pandas DataFrames inside the domain) is accepted as pragmatic compromise.
