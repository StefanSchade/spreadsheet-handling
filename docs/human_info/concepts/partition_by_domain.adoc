[[concept-partition-by-domain]]
== Partition by Domain — Split / Merge by Component Attribute

_Split large frames into deterministic per-component slices before persistence
and reassemble them losslessly on read, so multiple teams editing disjoint
components hit far fewer Git merge conflicts._

Status: feature concept. Lives in `domain/transformations`; I/O adapters stay
neutral; IR may use the same component info for presentation (optional). Related
backlog: `FTR-PARTITION-BY-DOMAIN`, `FTR-JSON-SHARDING-BY-PARTITION`.

=== Why this matters

When several teams edit one big frame, every change touches the same file and
merges conflict. If rows carry a *mutually exclusive and fully comprehensive*
component attribute (e.g. `component`), the frame can be sliced per component
before persistence — disjoint edits then land in disjoint files and merge
automatically — and recomposed exactly on read. The transform must be reversible
and adapter-agnostic.

=== The two bijective steps

* `partition_by_component(frames, meta)` — split input frames into multiple
  frames keyed by component.
* `merge_by_component(frames, meta)` — recompose the original frames from the
  component-sliced frames.

Scope: applies to frames where rows carry the component attribute directly or
via a join on a lookup table.

=== Requirements

Deterministic naming and layout of partitions (stable across runs); lossless
recomposition (merge restores exactly the pre-partition state, with row order
optionally normalized if configured); configurable lookup of the component
attribute (direct column or join); and rejection or flagging of rows with
missing/unknown components.

=== Heuristic defaults vs. explicit configuration

Best-effort defaults: if a column `component` exists in a frame, use it
directly; otherwise, if a frame named `components` exists with columns
`(variable, component)`, join on `variable`; partitioned frame names default to
`<frame>__<component>`. Explicit configuration (persisted in `meta`):

[source,yaml]
----
partition_by_domain:
  frames: [products, features]        # which frames to split
  component:
    column: component                 # direct column; OR
    via_join:                         # alternative: derive via join
      lookup_frame: components
      source_key: variable
      lookup_key: variable
      result_col: component
  naming:
    template: "{frame}__{component}"  # deterministic naming
  order:
    components: [Core, Pricing, Risk] # optional stable output order
  on_missing_component: error         # error | warn | drop
----

=== Algorithm

*Partition*: for each configured frame, resolve `component` for every row
(direct column or join), validate mutual exclusivity (exactly one component per
row), emit a new frame per distinct component using `naming.template`, preserve
all original columns, optionally normalize row order.

*Merge*: for each original frame name, gather all slices matching
`naming.template`; concatenate in the configured `components` order (or by
sorted component name if none given); restore exactly the original column set
and dtypes; preserve or normalize order per config.

=== Edge cases

* *Missing component*: follow `on_missing_component` (error/warn/drop), record
  to `meta.issues`.
* *Unknown component value*: treat as missing unless `allow_unknown: true`.
* *Overlapping components*: if a join produces > 1 component per row, raise an
  error (violates mutual exclusivity).
* *Empty partitions*: allowed; still create empty frames if `emit_empty: true`
  (useful for stable diffs).

=== Meta persistence

Store the effective configuration (resolved heuristics + user config) under
`partition_by_domain`, and a manifest of produced slices for traceability:

[source,yaml]
----
partition_by_domain_manifest:
  products: [products__Core, products__Pricing, products__Risk]
  features: [features__Core, features__Pricing, features__Risk]
----

=== Concrete example

A `products` frame with rows tagged `Core` / `Pricing` / `Risk` becomes
`products__Core`, `products__Pricing`, `products__Risk` on persistence; a later
read concatenates them back (in `[Core, Pricing, Risk]` order if configured)
into the original `products`, byte-for-byte modulo configured order
normalization. IR may optionally group or color-code tables by component on
render, but the partitioning logic stays in the domain layer.

=== Acceptance

Partition followed by merge yields the original frames byte-for-byte (modulo
configured order normalization); partition naming is deterministic and
documented in `meta`; rows with missing/overlapping components are handled per
policy and logged to `meta.issues`.

=== How it connects

* Independent of the compact-transform chain — does not depend on
  <<concept-compact-transforms,Compact Transforms>>.
* <<concept-metadata-canonical-vs-derived,Canonical vs. Derived Metadata>> —
  the effective configuration is canonical; the slice manifest is derived
  operational state.
* Code: `domain/transformations/` (split-by-discriminator family —
  `domain/transformations/discriminator_split.py`).
