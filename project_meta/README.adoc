= Project Meta

`project_meta` is an experimental dogfood consumer for repository metadata.
It exists to keep the project's own truth in a spreadsheet-friendly shape that
can be edited in ODS and exchanged as flat JSON.

== Execution Environment

`project_meta` intentionally uses the repository's current development
environment and the local editable checkout.

The Makefile targets use the repo-local `.venv/bin/sheets-run`.

Do not install or pin a PyPI `spreadsheet-handling` version into this venv for
`project_meta`.

The point of this path is to detect breakage in the current code before
release. If a breaking change affects `project_meta`, update `project_meta`
together with the core change before release.

Testing released-package compatibility is a separate concern and would require
a separate temporary venv or release-smoke setup. That release-smoke setup is
intentionally out of scope for the current dogfood path.

== Layout

* `canonical/` contains curated project truth.
* `derived/` contains rebuildable query outputs derived from canonical data.
* `staging/` is the review area produced by ODS import.
* `staging/` must not be promoted blindly.

== Data Files

* `ftrs.json` holds FTR header metadata only.
* `ftr_dependencies.json` is the flat xref replacement for list-valued FTR dependencies.
* `reviews.json` stores review records for FTRs, packages, releases, commits, and architecture areas.
* `findings.json` stores review findings over time and separates historical findings from current risks through `status` and `current_relevance`.
* `decisions.json` stores stable project-level choices.
* `releases.json` stores project-wide release or package pin state.

== Extracted Candidates

`project_meta/extracted/` contains machine-generated candidate JSON produced
from ADOC artifacts.

These files are not curated truth. They are conservative hints for later
review and must not be promoted into `canonical/` without manual inspection.

== Derived Query Views

The derived query pipeline can generate read-only views under `derived/`:

* `current_findings`
* `ftr_dependency_edges`
* `ftr_blockers`

These views are generated from canonical data for review and query
exploration. They are not canonical source, must not be edited manually, and
must not be promoted back into `canonical/`.

The raw canonical export pipeline remains available for the source-only
workbook. Query outputs are separate and rebuildable.

== Generated Context Report

`make meta-context` generates
`docs_generated/project_meta/current_context.adoc` from the derived query
outputs.

That report is generated, gitignored, and intended for LLM/review handoff
context rather than stable user documentation.

== Current Status Vocabulary

Findings use this status vocabulary:

* `open`
* `accepted`
* `resolved`
* `superseded`
* `wontfix`
* `historical`

`current_relevance` uses:

* `current`
* `partial`
* `historical`
* `unknown`

== Notes

Queries are intentionally lightweight and repository-local in this slice.
Future query examples may be added later, but no generic query engine or
automatic ADOC extraction is required here.

`make meta-extract` produces:

* `project_meta/extracted/ftr_candidates.json`
* `project_meta/extracted/review_candidates.json`
* `project_meta/extracted/finding_candidates.json`
