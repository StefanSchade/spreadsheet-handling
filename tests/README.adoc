= Test Suite Topology

The active suite is organized by responsibility first and implementation detail second.
When adding a new test, choose the directory that best describes the behavior or boundary it protects.

== Active Layers

* `tests/unit/`: small-scope behavior, helpers, local module collaboration
* `tests/integration/`: real product paths across multiple layers, adapters, or roundtrips
* `tests/architecture/`: dependency guards, contract seams, durable semantic invariants, and explicitly labeled current-state checks
* `tests/legacy_pre_hex/`: quarantined historical material, not part of the default active suite
* `tests/experimental/`: temporary scratch or parked test work, not part of the active collected suite and not a long-term archive

Supporting fixtures and helpers stay under `tests/data/`, `tests/utils/`, and `tests/experimental/`.
These support carriers must not define collected tests.
`tests/experimental/` is temporary parking only and should be revisited through the archive/quarantine rules when work needs to live longer.

== Lifecycle States

Every test belongs to exactly one lifecycle state:

* *Active*: lives under `tests/unit/`, `tests/integration/`, or
  `tests/architecture/`; authoritative for current behavior; included in the
  normal active development suite.
* *Quarantined*: physically separated from the active suite, currently under
  `tests/legacy_pre_hex/`; excluded from `make test`; runnable explicitly with
  `make test-prehex`; must have a status or classification note.
* *Archived*: retired from the executable suite. Prefer documented deletion; if
  retained for history, archived tests must live in a clearly labeled archive
  location and must not be collected or cited as normative behavior.

Allowed transitions are explicit:

* Quarantined -> Active: validate against the current architecture and move into
  the correct active layer.
* Quarantined -> Archived: retire the behavior and document the decision.
* Active -> Quarantined: only for active redesign or investigation, with a
  short rationale.

Do not create ambiguous states such as ignored active tests, undocumented
parking, or collected tests under support carriers.

== Placement Guide

* Put pure helper logic under `tests/unit/`, even when the helper lives in a backend package.
* Put real parser, renderer, adapter, CLI-with-IO, or roundtrip paths under `tests/integration/`.
* Put import-direction checks, AST guards, contract seams, and long-lived semantic rules under `tests/architecture/`.
* Put transitional carrier-shape or current-backend-form assertions under `tests/architecture/current_state/`, not beside durable invariants.
* Do not place historical pre-hex tests into the active tree until they are explicitly promoted or rewritten.

== Integration Slices

Integration tests are grouped by product path under `tests/integration/`.
The current slice map and consolidation rules live in `tests/integration/README.adoc`.

== Architecture Tests

Use the architecture layer when the primary assertion is about system shape rather than product output.
This layer is not only a relocation area: it may also split formerly mixed test files so durable invariants and transitional checks become visibly separate.

* `tests/architecture/dependency_guards/`: import direction and layer boundaries
* `tests/architecture/carrier_boundary/`: test-carrier topology, support-carrier, and path-owned marker guards
* `tests/architecture/formula_boundary/`: raw backend-formula leakage guards above adapter or translator boundaries
* `tests/architecture/spreadsheet_contract/`: write-side adapter and `RenderPlan` boundary guards
* `tests/architecture/parse_contract/`: read-side `WorkbookIR` and projection boundary guards
* `tests/architecture/semantic_invariants/`: durable architecture statements that should survive backend evolution
* `tests/architecture/current_state/`: transitional or current-carrier checks that are intentionally not durable contracts and may remain backend-shaped

Marker and Make-target normalization for architecture tests is handled by the path-first marker policy in `tests/conftest.py` and the documented `make` slices below.

== Markers

Markers are a secondary execution axis. Physical placement stays primary.
Core topology markers are assigned automatically from the file path in `tests/conftest.py`, so new tests normally do not need manual `unit`, `integ`, `arch`, `current_state`, `legacy`, or `prehex` markers.
Those topology markers are path-owned and should not be written manually in test modules.

Use manual markers only when they add execution value beyond the directory:

* `guardrail`: drift-protection tests in the architecture layer
* `xlsx_ir`: XLSX IR-based spreadsheet path
* `ods`: ODS / Calc-specific path
* `smoke`: small confidence checks
* `slow`: valid but intentionally outside the normal default slice

Raw backend formula syntax belongs only in concrete adapters, parser carrier extraction, or explicitly quarantined current-state checks.
Generic layers should represent formula intent structurally rather than by backend-specific strings.

== Make Targets

The expected everyday commands are:

* `make test`: normal active development slice, excluding `legacy`, `prehex`, and `slow`
* `make test-unit`: unit tests only
* `make test-integ`: integration tests only
* `make test-arch`: architecture tests only
* `make test-ir`: XLSX IR-focused slice
* `make test-ods`: ODS / Calc-focused slice
* `make test-smoke`: smoke checks
* `make test-all`: all active tests, including slow tests when present
* `make test-prehex`: explicit quarantined pre-hex slice, or a documented no-tests/deferred outcome when the directory is empty

== Common Examples

* Formula translator helpers: `tests/unit/rendering/` or `tests/unit/io_backends/`
* `read_spreadsheet_frames(...)` and parser-boundary product paths: `tests/integration/spreadsheet_contract/`
* JSON/XLSX/ODS roundtrips: `tests/integration/roundtrip/` or `tests/integration/io_backends/`
* Backend import guards or adapter-boundary seams: `tests/architecture/dependency_guards/`
* Durable meta or render semantics: `tests/architecture/semantic_invariants/`
* Transitional carrier details such as current `WriteMeta` shape: `tests/architecture/current_state/`

== Guardrails

`tests/architecture/` is for boundary and invariant protection, not a dumping ground for awkward end-to-end tests.
If a test mainly proves user-visible behavior through a real path, it belongs in `integration`, even if the motivating bug was architectural.

`tests/legacy_pre_hex/` remains quarantined on purpose.
Classify those tests deliberately as `promote`, `rewrite`, or `retire` in follow-up work instead of mixing them back into the active suite.
